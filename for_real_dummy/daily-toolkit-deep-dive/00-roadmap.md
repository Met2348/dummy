# 日常工具与科研工作流实操 —— 路线图与进度表

> 目标:不是"概念讲清楚",是"手把手教会操作"——IDE、Linux 日常操作、SSH 远程服务器、Git 协作、Python 环境管理、命令行监控调试、LaTeX 论文写作,7 个分类,由浅入深,分批次完成。
> 背景:2026-07-23,老师反馈"学生可能连 IDE/VSCode/Linux/TeX/SSH 远程服务器这些都不会用,这些操作性质的东西都得详细手把手教——不只是这两个,是所有日后职业、研究日常要天天打交道的东西"。这条系列直接响应这条反馈。

## 这条系列和仓库里其他系列的关键差异

**这不是"深挖系列"(numpy/torch/dsa 那种),是"操作系列"。** 深挖系列回答"这个函数/这个机制是什么、为什么这样设计";这条系列回答"这件事具体怎么做、做的时候会看到什么、卡住了怎么办"。类比:深挖系列像教科书,这条系列像一位坐在你旁边、一步步带你操作的学长。

**和已有的两处"操作性"内容的边界(避免重复劳动)**:
- [`04-how-to-practice-with-jupyter.md`](../04-how-to-practice-with-jupyter.md) 已经手把手教过"VSCode 里怎么开 Jupyter notebook、选 kernel、跑 cell、验证结果"——这条系列的 01 号文件(VSCode 日常操作)**不重复这部分**,只处理"写 `.py` 文件、调试器、终端、多文件导航、扩展"这些 notebook 之外的 VSCode 日常操作,并在需要时反向链接回 04 号文件。
- [`rhcsa-bash-deep-dive/`](../rhcsa-bash-deep-dive/00-roadmap.md) 已经用 RHCSA 考试视角教过 Linux 基础操作(文件权限/进程/网络配置等),里面确实包含"SSH 完整登录流程""scp/rsync 传输"这类知识点,**但那是从"考试会不会考"的视角写的,不是"科研日常要用"的视角**——比如没有讲 `~/.ssh/config` 怎么管理多台服务器、没有讲端口转发访问远程 Jupyter/TensorBoard、没有讲 VSCode Remote-SSH。这条系列的 02(Linux 日常操作)、03(SSH 远程服务器)两个文件明确聚焦"科研日常工作流"这个新角度,不重新讲 rhcsa-bash 已经讲过的考试向内容,而是在需要处标注"底层命令语法可参考 rhcsa-bash 对应知识点"。

## 每个知识点的固定讲解结构(6 步,专为"操作类"内容设计,不是深挖系列的 7 步)

深挖系列的"底层机制/为什么这样设计"步骤,在这条系列里被拆成更符合操作类内容特点的结构——不能像 rhcsa-bash 那样把"为什么"压缩成一句话就直接讲危险操作对比(这是 rhcsa-bash 系列在自查中被发现的真实问题:进程/inode/权限模型这些基础概念因此从未被真正建立过,详见本文件底部"已知教训"一节),必须先把"是什么/为什么需要"讲透,再进入操作步骤:

1. **为什么需要这个 / 不会有什么后果**——从零建立动机,不假设读者已经理解这件事解决什么问题(比如"为什么不能直接在 Windows 记事本改远程 GPU 服务器上的代码")
2. **环境要求**——需要装什么、怎么确认自己已经有了(不假设环境已就绪,给出检查命令)
3. **一步步跟着做**——真正的操作核心,每一步都写清楚"输入什么 → 应该看到什么现象/输出",不是命令堆砌;能在仓库 `.venv`/Git Bash/WSL2 真实跑通的步骤,一律真实跑过、贴真实输出,不能真实操作的(比如 VSCode 图形界面点击、需要远程服务器才能验证的部分)如实标注"图形界面操作,已按当前 VSCode 版本核实菜单路径,但未做自动化验证"
4. **背后发生了什么**——简短说明每个关键步骤在底层做了什么(不是装饰性的一句话,要能回答"如果这一步跳过会怎样")
5. **常见坑**——真实会卡住的地方 + 怎么自查,风格类似 [04-how-to-practice-with-jupyter.md](../04-how-to-practice-with-jupyter.md) 第 8 节"遇到报错怎么办"那张表格
6. **自测清单**——3-5 条"如果你能做到以下几件事,说明这一节真的掌握了",让读者自己校验,不是老师说了算

## 验证环境

- **Git/SSH 本地部分/Python 环境管理**:仓库根目录 Git Bash,已确认 `git 2.38.1`、`ssh(OpenSSH 9.0p1)`、`python 3.13.9` 均可用,能真实验证 key 生成/config 语法/venv 操作。
- **LaTeX**:本机已装 MiKTeX(`xelatex`/`pdflatex`/`tex` 均确认可用),能真实编译验证,不是空谈语法。
- **Linux 日常操作(tmux 等)**:复用 rhcsa-bash-deep-dive 已装好的 WSL2 Rocky Linux 环境,具体工具可用性由撰写该章节时现场确认并如实记录(不假设"应该有"就跳过检查)。
- **SSH 远程服务器实操**:没有真实可连接的远程 GPU 服务器,`~/.ssh/config` 语法、密钥生成、端口转发的**语法和用法**可以真实验证(用 `ssh -G` 解析 config、本地回环地址模拟端口转发等手段),但"连上一台真实远程机器"这个完整流程如实标注为"语法已验证,真实连接效果需要读者在自己有权限的服务器上验证",不冒充有一台真机可用。
- **VSCode 图形界面操作**:本机已确认 VSCode 1.129.1 存在,菜单路径/快捷键按当前版本核实,但图形界面点击本身不能像代码一样跑 assert,如实标注这一类内容的验证颗粒度。

## 已知教训(写作前必须记住,避免重蹈覆辙)

2026-07-23 对全仓库 17 条系列做过一次"新手可读性"诊断审计(详见各系列自身改动记录),其中 rhcsa-bash-deep-dive 被发现存在结构性问题:该系列模板第 4 步是"从最容易犯错的做法讲起",这一步默认读者已经理解"进程/inode/权限模型/LVM"这些概念本身,只负责警示常见错误——结果是这些最基础的概念在全系列里从未被真正定义过,直接被当成已知背景使用。**这条新系列必须吸取这个教训:第 1 步"为什么需要这个"必须先把概念本身建立起来,不能跳过"是什么"直接进入"怎么做/别做错"。**
