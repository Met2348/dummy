## 1. 打开项目 / 工作区 —— File > Open Folder,不是一个个打开文件

**为什么需要这个 / 不会有什么后果:**

VSCode 里有两种完全不同的"打开"方式:打开单个文件(File > Open File),和打开一整个文件夹作为**工作区**(File > Open Folder)。很多刚接触 VSCode 的人图省事,想看哪个文件就双击哪个文件,VSCode 也确实允许这么做——但这样打开的文件是"孤立"的,VSCode 不知道它属于哪个项目、项目的边界在哪。

后果很具体:左侧文件树(Explorer)是空的,看不到这个文件所在项目还有哪些其他文件;第 3 节要讲的 Ctrl+P 快速跳转、全局搜索、跳转到定义,全部依赖"VSCode 知道项目根目录在哪"这个前提,单文件模式下这些功能要么用不了、要么范围小到没有意义;第 4 节调试器要用的 `launch.json` 配置文件,也需要一个明确的项目根目录才有地方存放。你继承的这个仓库有几百个文件、几十个子项目,如果只会单个文件地打开,相当于拿到一整本书,却只能一页一页孤立地看,看不出任何一页跟其他页的关系。

**环境要求:**

- VSCode 已安装。检查方法:打开一个终端(还没学怎么开的话,先看第 2 节,或者直接用系统自带的终端/命令提示符),输入 `code --version`。
- 本机真实检查结果(供你核对自己机器上跑出来的是不是类似格式):
  ```
  1.129.1
  8a7abeba6e03ea3af87bfbce9a1b7e48fed567b8
  x64
  ```
  第一行是版本号,第二行是这个版本对应的 commit 哈希,第三行是架构。只要能跑出三行类似格式的内容,就说明 VSCode 装好了且命令行工具可用。

**一步步跟着做:**

1. **操作**:打开 VSCode,点菜单栏 `File`(文件)→ `Open Folder...`(打开文件夹...),或者直接按快捷键 `Ctrl+K Ctrl+O`(先按住 Ctrl 点 K,松开后再按住 Ctrl 点 O,这是 VSCode 里常见的"组合键链"写法,不是同时按四个键)。
   **应该看到**:弹出系统级的"选择文件夹"对话框——注意这个对话框和"选择文件"对话框长得不一样,双击一个文件夹会"进入"这个文件夹而不是"选中"它,要选中你当前所在的这一层文件夹,得点对话框右下角的"选择文件夹"按钮。

2. **操作**:导航到仓库根目录(比如 `E:\Workspace\dummy`),点击"选择文件夹"。
   **应该看到**:VSCode 窗口重新加载,标题栏变成类似 `dummy - Visual Studio Code` 的样式(显示的是文件夹名,不是某个文件名);左侧出现 Explorer 面板,顶部用大写字母显示文件夹名(这是 VSCode 的固定约定,文件夹模式下根节点永远大写显示),下面是完整的目录树——`for_real_dummy`、`learning`、`research`、`.venv` 等仓库里真实存在的顶层目录都应该能看到。

3. **对比操作**(体会两种模式的差异):用 `File` → `Open File...`(`Ctrl+O`,注意和上一步的 `Ctrl+K Ctrl+O` 不是同一个快捷键)单独打开仓库里的某一个文件,比如 `learning/agent-foundations/src/common.py`。
   **应该看到**:编辑器区域出现这一个文件的内容,但左侧 Explorer 面板要么是空的,要么提示"你还没有打开任何文件夹"之类的文字;标题栏只显示这一个文件名;按 `Ctrl+P` 打开快速搜索框,输入仓库里其他任意文件名(比如 `react_loop`),搜不到任何结果——因为 VSCode 压根不知道"仓库"在哪,它只认识你刚刚手动打开的这一个文件。

4. **怎么确认自己现在是"文件夹模式"还是"单文件模式"**:看左侧 Explorer 面板顶部——文件夹模式会显示文件夹名(大写)+ 完整目录树;单文件模式面板是空的,或者只在"打开的编辑器"分组下面孤零零挂着一个文件,没有目录树。也可以看标题栏:文件夹模式是"文件夹名 - Visual Studio Code",单文件模式是"文件名 - Visual Studio Code"。

5. **确认这次打开会被记住**:关闭 VSCode 再重新打开(不用额外操作),应该默认恢复到上次打开的文件夹。也可以用 `File` → `Open Recent` 查看历史记录,里面应该能看到刚才打开的文件夹路径。

**背后发生了什么:**

打开一个文件夹后,VSCode 会把这个文件夹当成"工作区"(workspace)的根,在内部建立一份文件索引(供 Ctrl+P、全局搜索使用),自动检测这个文件夹是不是 git 仓库(是的话左下角会出现分支名,文件树上会出现 M/U 之类的改动标记),并且会去找这个文件夹下有没有 `.vscode/settings.json`(项目级配置,优先级高于用户全局设置)、有没有 `.vscode/launch.json`(调试配置,第 4 节要用)、有没有 `.vscode/extensions.json`(推荐扩展列表)。这些机制全部以"你打开的这个文件夹路径"为根。单文件模式下,VSCode 连"根"在哪都不知道,上面这些功能自然无从谈起——这不是 VSCode 故意阉割单文件模式的功能,而是这些功能本身的实现逻辑就需要一个明确的项目边界才能工作。

如果跳过这一步直接进入后面几节:第 3 节的 Ctrl+P 会是空的,跳转到定义跳不出当前文件;第 4 节的 launch.json 没有合理的存放位置。这就是为什么"打开项目/工作区"排在整篇第一节。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| Ctrl+P 搜不到仓库里明明存在的文件 | 当前是单文件模式,没有 Open Folder,VSCode 不知道项目边界在哪 |
| 左下角看不到 git 分支名 | 同上;或者打开的文件夹本身不是 git 仓库的根目录(打开了某个子目录) |
| 标题栏显示的是文件名而不是文件夹名 | 当前是单文件模式,回到步骤 1 重新 Open Folder |
| 打开了文件夹,但 Explorer 树看起来"缺了一截" | 可能不小心打开了仓库的某个子目录而不是根目录,检查 Explorer 顶部显示的文件夹名对不对 |
| 每次重新打开 VSCode 都要重新选文件夹 | 检查是不是每次都用"新窗口"打开而不是直接启动 VSCode;常用项目可以在 `File > Open Recent` 列表里把它固定(pin)住 |

**自测清单:**

- [ ] 能说出用 "Open Folder" 和 "Open File" 打开同一个仓库,Explorer 面板的表现差异是什么
- [ ] 不看这篇教程,能独立用 `Ctrl+K Ctrl+O` 打开一个文件夹
- [ ] 看一眼窗口(标题栏 + Explorer 面板),就能判断当前是文件夹模式还是单文件模式
- [ ] 能说出为什么后面"多文件导航"和"调试器"两节都要求先用文件夹模式打开仓库

## 2. 集成终端 —— 在编辑器里跑命令,不用切窗口

**为什么需要这个 / 不会有什么后果:**

VSCode 内置了一个"集成终端"(Integrated Terminal),很多新手不知道这回事,习惯另外开一个独立的 Git Bash / PowerShell 窗口敲命令,跑代码时在两个窗口之间来回 Alt+Tab 切换。这样做**不会导致任何功能损失**——独立终端窗口能做的事,集成终端全都能做,两者背后就是同一个 shell 进程,没有能力上的差异。差的是效率:独立窗口需要手动 `cd` 到项目目录(而集成终端默认直接从项目根目录启动)、需要来回切换窗口打断你正在看的代码(而集成终端就在代码下方,眼睛不用离开这块区域)、终端里如果打印出报错信息带着文件路径和行号,独立窗口里那串文字只是纯文本,而集成终端里通常可以直接点击跳转到编辑器对应位置。这些摩擦单次看很小,但科研/工程日常里"改代码 → 跑一下 → 看报错 → 改代码"这个循环一天要走几十上百遍,累积起来的效率差异不小。

**环境要求:**

- 已经按第 1 节用文件夹模式打开了仓库(集成终端默认从工作区根目录启动,单文件模式下没有明确的启动目录)。
- 至少装了一种可用的 shell。本机确认可用:Git Bash、PowerShell、命令提示符(cmd)三种都装了。

**一步步跟着做:**

1. **操作**:按快捷键 `` Ctrl+` ``(Ctrl 加英文输入法下 Esc 键下方那个反引号键),或者点菜单 `Terminal`(终端)→ `New Terminal`(新建终端)。
   **应该看到**:编辑器下方弹出终端面板,自动启动一个默认 shell(具体是哪一种取决于系统配置的 default profile),提示符里显示的当前路径就是你 Open Folder 打开的那个文件夹(比如形如 `E:\Workspace\dummy>` 或者 `ericp@DESKTOP MINGW64 /e/Workspace/dummy` 这样的开头),不需要你手动 `cd` 过去。

2. **操作**:点终端面板右上角、"+"新建终端图标旁边的下拉箭头(v 形图标)。
   **应该看到**:弹出一个下拉列表,列出本机检测到的所有可用 shell profile,比如 "Git Bash"、"PowerShell"、"命令提示符" 等(具体列出哪几项取决于你机器上装了什么)。点其中一项会新开一个用该 shell 的终端标签。

3. **操作**:重复上一步,分别开一个 Git Bash 和一个 PowerShell 的终端标签。
   **应该看到**:终端面板顶部出现多个标签页,每个标签图标能区分是哪种 shell。点哪个标签就切换到哪个终端会话,各自独立、互不影响——在一个终端里 `cd` 到别的目录,不会影响另一个终端的当前路径。

4. **操作**:在某个终端标签上找到面板右上角的垃圾桶图标(或者在终端里直接输入 `exit` 回车)。
   **应该看到**:这个终端标签消失,其余标签不受影响。

5. **操作**:试一下拆分终端——点终端面板右上角的"拆分终端"图标(通常画的是左右两个矩形),或者快捷键 `Ctrl+Shift+5`。
   **应该看到**:当前终端面板被分成左右两栏,可以同时看两个终端会话内容(比如左边跑一个脚本、右边同时开着看日志或者监控 GPU 占用),不用来回切标签。

**背后发生了什么:**

VSCode 集成终端启动的是一个真实的 shell 进程——选 Git Bash 就是真的启动了 `bash.exe`,选 PowerShell 就是真的启动了 `powershell.exe`,VSCode 只是把这个进程的标准输入输出接管到自己的面板里显示,不是某种阉割版或者模拟出来的终端。这也是为什么终端里能跑的任何命令,换到独立窗口里也能跑,两者能力完全等价,差的只是"是否嵌在编辑器界面里"这一点体验。终端启动时的初始工作目录,默认取自当前打开的工作区根文件夹——这也解释了为什么第 1 节强调要先用文件夹模式打开仓库:单文件模式下,VSCode 都不知道该从哪个目录启动这个终端进程。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| 终端打开时提示符路径不是仓库根目录 | 当前是单文件模式(回第 1 节);或者上一次终端会话手动 `cd` 到别处后,新开的标签"继承"了这个路径 |
| 想要的 shell(比如 Git Bash)不在下拉列表里 | 该 shell 没有正确安装,或者 VSCode 没检测到它的安装路径——需要在设置里手动配置 `terminal.integrated.profiles.windows` |
| 关闭 VSCode 或者电脑重启后,之前在终端里跑着的任务全没了 | 集成终端进程跟着 VSCode 窗口的生命周期走,VSCode 关闭时终端进程也会被杀掉——长时间跑的任务(比如远程 GPU 上跑几小时的训练)不能直接依赖本地这个集成终端,03 号文件会讲 `tmux`/`screen` 这类能在连接断开后继续跑的工具 |
| 终端里中文或者特殊字符显示乱码 | 通常出在 shell 自身的编码设置上,`cmd.exe` 比 Git Bash / PowerShell 更容易遇到这个问题 |

**自测清单:**

- [ ] 不看教程,能用快捷键打开/关闭集成终端
- [ ] 能新开一个指定类型(比如 PowerShell 而不是默认 shell)的终端标签
- [ ] 能同时管理 2 个以上终端标签,知道怎么切换和关闭
- [ ] 能说出集成终端和独立终端窗口底层是不是同一个东西(是——只是显示位置和联动能力不同)

## 3. 多文件导航 —— 读懂继承代码库的核心技能

**为什么需要这个 / 不会有什么后果:**

你继承的这个仓库,`learning/` 下有几十个子项目,每个子项目里代码都是"一个函数在 `common.py` 里定义,被另外好几个文件 import 之后调用"这种结构——这是正常、成熟的代码组织方式(不重复写同一段逻辑),但对第一次读这份代码的人来说,意味着你**不可能靠从头到尾顺序读完一个文件就搞懂它**:你会不断遇到"这个函数是从哪冒出来的""这个东西改了之后还有哪些地方会受影响"这类问题。

如果不会用 VSCode 的导航工具,能做的只有：打开一个文件,遇到不认识的函数名,凭名字猜它大概做了什么,或者手动一个个打开旁边的文件用眼睛找定义在哪——小脚本这么做还凑合,几十个文件互相调用的代码库这么做,极容易读错、以及在你自己动手改代码时漏掉某个不知道存在的调用点,悄悄引入 bug。本节要学的几个工具,就是用来解决"这个符号是什么/从哪来/谁在用它"这几个问题的。

**环境要求:**

- 已经用文件夹模式打开仓库(第 1 节)。
- 跳转到定义 / 查找所有引用这些语义级别的功能,并不是 VSCode 编辑器本体内建的——第 5 节会用现场读到的证据说明,VSCode 自带的最基础 Python 语法支持包其实只有语法高亮,不含这些能力。真正提供这些功能的是 Pylance 扩展(`ms-python.vscode-pylance`),本机已确认装了这个扩展。

**一步步跟着做(用仓库里一个真实存在的调用关系做例子):**

这里用到的例子不是编出来的:`learning/agent-foundations/src/common.py` 第 61 行定义了一个函数 `parse_action`,负责把 ReAct 风格的一行文本(形如 `Action: search(query=foo)`)解析成"工具名 + 参数字典"。这个函数在同一目录下的 `react_loop.py` 第 7 行被 import、第 49 行被真正调用。假设你现在读 `react_loop.py`,读到第 49 行 `name, args = parse_action(out)`,但不知道 `parse_action` 具体是怎么解析的——下面一步步用 VSCode 的工具把这件事搞清楚。

1. **Ctrl+P 快速打开文件。操作**:按 `Ctrl+P`,输入 `react_loop`。
   **应该看到**:弹出一个下拉列表,随着你打字实时过滤仓库里所有匹配文件名的文件,列表里应该能看到 `react_loop.py`,路径提示是 `learning/agent-foundations/src/react_loop.py`。回车打开。
   附加技巧:Ctrl+P 支持模糊匹配(不用输完整文件名,比如输 `rctlp` 大概率也能命中),还支持输入 `文件名:行号` 直接跳到指定行,比如 `react_loop.py:49`。

2. **定位到调用点**。文件打开后,滚动或者用 `Ctrl+G`(跳转到指定行,输入 49)找到第 49 行:
   ```python
   name, args = parse_action(out)
   ```
   把光标点在 `parse_action` 这个词中间。

3. **跳转到定义。操作**:右键这个词,菜单里选 "Go to Definition"(转到定义),或者更快——光标停在词上直接按 `F12`。
   **应该看到**:编辑器跳转到 `learning/agent-foundations/src/common.py`,光标落在第 61 行:
   ```python
   def parse_action(text: str) -> tuple[str, dict]:
   ```
   函数体展开在眼前,你能看到它其实是用正则表达式解析 `Action: tool_name(args)` 这种格式的文本,把参数按 `key=value` 或者单个字符串两种情况分别处理。
   附加技巧:如果只是想看一眼、不想真的跳过去打断当前阅读位置,用 `Alt+F12`("Peek Definition",窥视定义),会在当前文件里嵌一个小窗口直接预览定义,按 `Esc` 关闭小窗口就回到原来的位置。

4. **查找所有引用。操作**:停留在 `common.py` 里 `parse_action` 定义那一行,右键选择 "Find All References"(查找所有引用),或者按 `Shift+F12`。
   **应该看到**:弹出一个引用列表面板,列出这个符号在整个项目里被使用的每一处。这段调用关系里,真实存在的引用点一共有 3 处,已经用 Grep 在代码里核实过:`common.py` 自己内部第 123 行、第 126 行(这两处是它的自测函数 `_self_test` 里调用自己做验证),以及 `react_loop.py` 第 49 行(真正的业务调用点)——这三处无论如何都应该出现在结果列表里。(诚实补充一句:不同版本的语言服务器,对"定义那一行本身算不算一条引用"这个细节处理不完全一致,你实际看到的条目总数可能是 3 条也可能是 4 条,这属于第 0 节说明过的、离不开真实 GUI 交互才能确认的细节,不影响这个工具的核心用法。)点列表里任意一条,编辑器直接跳过去并高亮那一行。
   这一步的价值:假如你要修改 `parse_action` 的解析规则(比如想支持一种新的参数格式),Find All References 能让你在动手之前先确认"改了这个函数,会影响到哪些地方",而不是改完之后才发现某个你完全不知道的调用点跟着出错。

5. **Ctrl+Shift+F 全局文本搜索,和 Find All References 做对比。操作**:按 `Ctrl+Shift+F`(或者点左侧活动栏的放大镜图标),搜索框输入 `parse_action`。
   **应该看到**:侧边栏出现搜索面板,列出仓库里所有**包含这串字符**的文件、行号和上下文预览——注意这和上一步的结果范围不一样,全局搜索是纯文本匹配,不理解代码语义,只要文字上出现 `parse_action` 就会被列出来(哪怕是在注释里,或者某篇 `.md` 文档里提到这个名字)。
   两者的本质区别:**Find All References 是"语义符号引用"**(VSCode 理解 Python 的作用域和 import 关系,只找真正引用这个具体符号的代码位置),**Ctrl+Shift+F 全局搜索是"字符串匹配"**(不理解语义,可能有假阳性——比如另一个文件里恰好也有一个同名但完全无关的函数,或者只是注释里提了一句)。两个工具用途不同:想知道"改这个函数影响谁"用 Find All References;想知道"这个词在整个仓库文档/代码里哪里被提到过"用全局搜索。

6. **面包屑导航。操作**:停留在 `common.py`,看编辑器标签页正下方那一条窄栏。
   **应该看到**:一行形如 `learning > agent-foundations > src > common.py > parse_action` 的路径。点其中任意一段(比如点 `src`)会弹出这一层目录下的其他文件列表,可以快速切换到同目录的别的文件;点最后一段(当前光标所在的函数名 `parse_action`)会弹出当前文件里所有顶层函数/类的列表,不用来回滚动就能跳到同文件里的另一个函数。
   如果没看到这一条:按 `Ctrl+Shift+P` 打开命令面板,输入 "Toggle Breadcrumbs" 回车即可切换显示/隐藏(这个命令的 ID 是 `breadcrumbs.toggle`,本机已从当前安装的 1.129.1 版本程序文件里现场确认过这个命令确实存在)。

**背后发生了什么:**

Go to Definition / Find All References 不是简单的文本搜索。VSCode 背后跑着一个专门的 Python 语言服务器——这正是第 5 节要讲的 Pylance——它会解析项目里所有 `.py` 文件的抽象语法树(AST)、追踪每个文件的 import 关系,在后台建立一张"符号 → 定义位置"和"符号 → 所有引用位置"的索引表。你按 F12 / Shift+F12 时,VSCode 问的不是"这个词在别处长得一样的地方还有哪些"(那是 Ctrl+Shift+F 干的事),而是"这个具体的 Python 符号,在语义上对应哪个定义、被哪些地方真正引用了"。这也是为什么 Find All References 比全局文本搜索更精准——它不会把注释里恰好出现的同名字符串,或者别的文件里恰好定义了同名但完全无关的另一个函数,错误地算进结果里。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| F12 跳转没反应,或者跳到一堆不相关的结果 | 没装 Pylance(内建的纯文本级 Python 支持能力弱很多);或者 Pylance 还在后台建索引,留意右下角状态栏是不是有转圈的进度提示,等它转完再试 |
| Find All References 结果是空的,明明代码里确实有调用 | 符号名拼写/大小写不完全一致(Python 区分大小写);或者调用发生在字符串拼接、`exec`/`eval`、反射等动态执行场景里,静态分析天然看不到这种运行时才确定的调用 |
| Ctrl+Shift+F 全局搜索结果里混进一堆 `.venv`、`__pycache__` 里的无关结果 | 可以在搜索面板的"排除"输入框里加 `.venv, __pycache__`;仓库有 `.gitignore` 的话,VSCode 搜索默认会尊重它(受 `search.useIgnoreFiles` 设置控制) |
| 面包屑那一条不见了 | 之前被切换关闭过,命令面板搜 "Toggle Breadcrumbs" 重新打开 |
| Ctrl+P 输入文件名,死活搜不到明明存在的文件 | 回第 1 节确认是不是单文件模式;或者这个文件被 `.gitignore` 排除,同样受 `search.useIgnoreFiles` 影响 |

## 4. 调试器 —— 用工具找 bug,不是用眼睛

**为什么需要这个 / 不会有什么后果:**

没有调试器时,排查 bug 基本靠"到处加 `print`,跑一遍,看输出,删掉 print,再改代码再 print"——这种方法对付简单 bug 够用,但代码"能跑、不报错,但算出来的结果就是不对"这类 bug(没有报错信息可以直接定位出错的行,只能靠你自己一步步验证)用 print 法排查效率很低:每次想多看一个变量,都要改代码、重新跑一遍完整程序;想在某个循环跑到第 3 次的那一刻看清楚状态,只能在 print 里加条件判断。调试器能让程序在你指定的位置真正"暂停"下来,那一刻所有变量的真实值直接摆在你眼前,不用猜、不用改代码插桩,甚至能在暂停的那一刻现场敲代码验证你的猜想对不对。不会用调试器不代表调试不了(print 本身也是一种有效手段),但复杂一点的 bug,两种方法的排查效率差异会很明显。

**环境要求:**

- 已装 Python 扩展(`ms-python.python`)和配套的调试器扩展(`ms-python.debugpy`)——本机已用 `code --list-extensions --show-versions` 现场确认都装好了(`ms-python.python@2026.4.0`、`ms-python.debugpy@2026.6.0`)。
- 仓库根目录有可用的 Python 解释器(本机是 `.venv`,怎么配置环境不是本节重点,后续系列会细讲)。
- 本节要用的练习脚本:[`_assets/01-debug-demo.py`](_assets/01-debug-demo.py),核心逻辑如下(已用仓库 `.venv` 真实跑过,不是纸上谈兵;完整文件开头还有一段说明用的模块 docstring 和一行 `from __future__ import annotations` 导入,为了节省篇幅这里没有重复贴出来):
  ```python
  def average(nums: list[float]) -> float:
      """计算 nums 的算术平均值。"""
      total = 0
      count = len(nums)
      for i in range(count - 1):
          total += nums[i]
      return total / count


  def summarize_scores(scores: list[float]) -> str:
      """对外的入口函数:算出平均分,再包一层格式化文本。"""
      avg = average(scores)
      return f"average score = {avg}"


  def main() -> None:
      scores = [10, 20, 30, 40]  # 手算平均值 = (10+20+30+40) / 4 = 25.0
      result = summarize_scores(scores)
      print(result)


  if __name__ == "__main__":
      main()
  ```

**一步步跟着做:**

1. **打开脚本**。按 `Ctrl+P`,输入 `01-debug-demo`,回车打开。先读一眼:这个脚本算一个分数列表的平均值。

2. **不设断点,先跑一遍,建立"预期 vs 实际"的落差**——这是调试的起点,不是可以跳过的步骤:你得先确认"真的有 bug"、并且知道"正确答案应该是什么",才有的放矢。
   **操作**:右键编辑器空白处,选 "Run Python File in Terminal"(或者点编辑器右上角的三角形运行图标,或者干脆在集成终端里手动敲 `python 01-debug-demo.py`)。
   **应该看到**:集成终端里打印出一行 `average score = 15.0`。
   对照脚本里的注释手算:`scores = [10, 20, 30, 40]`,平均值应该是 `(10+20+30+40)/4 = 25.0`。`15.0 ≠ 25.0`,确认这里面有 bug——而且是"程序正常退出、没有任何报错,但答案是错的"这种更难排查的类型。

3. **设置断点**。
   **操作**:把鼠标移到 `total += nums[i]` 这一行,行号左边(gutter,行号和代码之间那一小条空白)会出现一个淡色的红点预览,点一下(或者光标放这一行按 `F9`)。
   **应该看到**:一个实心红色圆点固定显示在该行行号左侧(不再是只有鼠标悬停才出现的淡色预览)。

4. **配置 `launch.json`**(调试器第一次用需要配置一次,之后会被记住,不用每次都配)。
   **操作**:点左侧活动栏的"运行和调试"图标(三角形加虫子的图案,快捷键 `Ctrl+Shift+D`),面板里如果还没配置过,会看到一个 "create a launch.json file" 链接,点它。
   **应该看到**:弹出一个下拉选择列表,选 "Python Debugger",再选 "Python File"(调试当前打开的文件)。VSCode 会在仓库根目录自动创建 `.vscode/launch.json`,内容大致是:
   ```json
   {
       "version": "0.2.0",
       "configurations": [
           {
               "name": "Python Debugger: Current File",
               "type": "debugpy",
               "request": "launch",
               "program": "${file}",
               "console": "integratedTerminal"
           }
       ]
   }
   ```
   注意 `"type"` 字段是 `"debugpy"`——这是本机已装的 `ms-python.debugpy` 扩展真实注册的调试器类型(现场读过它的 `package.json` 确认过)。如果你在网上搜到的教程里看到 `"type": "python"`,那是几年前的旧写法,后来 Python 调试能力被拆分成独立的 "Python Debugger" 扩展,类型字段也跟着改成了 `"debugpy"`——VSCode 如果检测到旧格式的配置,通常会提示自动迁移,这是正常升级行为,不是报错。

5. **启动调试**。
   **操作**:确保当前编辑器打开的是 `01-debug-demo.py`(因为 `launch.json` 里配的是 `${file}`,也就是"当前打开的文件"),按 `F5`(或者点运行和调试面板里的绿色三角形按钮)。
   **应该看到**:程序开始运行,在你第 3 步设的断点处自动暂停——那一行会被整行高亮(通常是黄色/橙色背景);顶部出现一条悬浮的调试工具条(继续、单步跳过、单步调入、单步跳出、重启、停止几个图标);左侧活动栏切换显示 "VARIABLES"(变量)、"WATCH"(监视)、"CALL STACK"(调用堆栈)、"BREAKPOINTS"(断点)几个面板;底部状态栏通常会变成橙色,提示当前处于调试模式。

6. **看 Variables 面板**。
   **应该看到**:展开 "Locals" 分组,列出当前作用域(此刻是 `average` 函数内部)所有已定义的局部变量和它们此刻的值:`nums`(一个 list,展开能看到 `[10, 20, 30, 40]` 四个元素)、`total`(此刻应该是 `0`——因为断点设在 `total += nums[i]` 这一行,这一行"即将执行但还没执行",所以 `total` 还停在循环开始前的初始值)、`count`(`4`)、`i`(`0`,这是 for 循环的第一次迭代)。

7. **加一个 Watch 表达式**。
   **操作**:切到 "WATCH" 面板,点 "+",输入表达式 `count - 1`。
   **应该看到**:这个表达式被加入监视列表,实时显示计算结果 `3`——这是关键线索:`count` 是 `4`,但循环的上界是 `count - 1 = 3`,配合 Python `range()` 不包含右端点的规则,已经能推测出循环少跑了一次。

8. **反复按 Step Over,观察 `i` 和 `total` 怎么变**。
   **操作**:点调试工具条的"单步跳过"图标,或者按 `F10`。
   **应该看到**:第一次按下后,`total += nums[i]` 这一行被真正执行,`total` 从 `0` 变成 `10`;因为 for 循环体只有这一行代码,下一次停留位置回到 for 语句本身准备判断下一次迭代,Variables 面板里 `i` 变成 `1`。继续按 F10 几次,会看到 `total` 依次变成 `10 → 30 → 60`,`i` 依次变成 `0 → 1 → 2`,然后循环直接结束(因为 `range(3)` 只产生 `0, 1, 2` 三个值),跳到 `return total / count` 这一行——`total` 定格在 `60`,从来没有等于 `10+20+30+40=100`。这就是 bug 的现场:下标为 `3` 的第四个数字 `40`,这个循环从头到尾都没碰过它。

9. **对比 Step Into 和 Step Out**。
   `F10`(单步跳过)在遇到一次函数调用时,会把这个函数当"黑盒"整体执行完,直接停在调用返回之后的下一行,不会带你进函数内部。`F11`(单步调入,Step Into)则会真的跳进被调用的函数内部,一行一行走它的代码。`Shift+F11`(单步跳出,Step Out)是反过来:假设你已经在某个函数内部,懒得再一行行走完剩下的代码,想直接跳回调用它的地方,按一下就会自动执行完当前函数剩下的所有代码,停在返回调用者之后的那一行。
   建议实际操作体会一次:停止当前调试(`Shift+F5`),把断点挪到 `main()` 函数体里的 `result = summarize_scores(scores)` 这一行,重新按 `F5` 启动调试。分别试一次 `F10`(不会进 `summarize_scores`/`average` 内部,直接停在下一行 `print(result)`,此时 Variables 面板里 `result` 已经是算好的完整字符串 `"average score = 15.0"`)和一次 `F11`(会先跳进 `summarize_scores` 函数,停在它内部的 `avg = average(scores)` 这一行,再按一次 `F11` 才会继续跳进 `average` 本身)。

10. **用 Debug Console 交互式验证猜想**。
    **操作**:调试器暂停在断点处时(比如还停在 `average` 函数里某次循环),点底部面板的 "DEBUG CONSOLE" 标签,在输入框里直接输入 `nums[3]`,回车。
    **应该看到**:输出 `40`——证明这个数据本身没问题,列表里确实有第四个数字,问题出在循环没有走到它。再输入 `list(range(count - 1))`,回车。
    **应该看到**:输出 `[0, 1, 2]`——不用改代码、不用重新跑程序,当场证实了 `range` 确实少产生了一个数。这就是 Debug Console 的价值:暂停的这一刻,程序当时的完整状态(所有变量、已导入的模块)都能像在一个 Python 交互式解释器里一样,被你直接查询、做算术、调用函数验证猜想。

11. **修复并验证**。
    **操作**:停止调试(`Shift+F5`),把 `average` 函数里的 `for i in range(count - 1):` 改成 `for i in range(count):`,保存(`Ctrl+S`),不设断点直接运行一遍(`Ctrl+F5`"运行不调试",或者终端里手动 `python 01-debug-demo.py`)。
    **应该看到**:输出变成 `average score = 25.0`,和手算的期望值一致——本机已经真实验证过这个修复方案确实把结果改对了。

**背后发生了什么:**

断点的本质,是告诉背后的调试适配器(debug adapter,这里具体是 `debugpy`)"程序执行到这一行的时候暂停,把控制权交还给我,不要往下执行"。`debugpy` 内部依托 Python 自带的逐行跟踪机制的更高效实现,能在不修改你源代码的情况下,拦截解释器的逐行执行事件。Step Over / Into / Out 都是在"已经暂停"这个基础上,精确控制"继续执行到哪里再暂停"的策略:Over 是把当前这一行(如果是函数调用,连同整个函数体)执行完;Into 是如果当前行是函数调用,进去之后停在函数内部第一行;Out 是把当前函数剩余部分全部执行完,停在返回调用者之后的位置。Variables 面板显示的是暂停那一刻,当前"调用栈帧"里能访问到的所有名字和它们此刻绑定的对象——这也是为什么同一个变量名(比如 `i`),在你 Step Into 别的函数之后,Variables 面板里可能整个消失或者变成完全不同的值:变量的作用域是和调用栈帧绑在一起的,不是全局唯一的一份。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| 按 F5 没反应,或者提示找不到调试配置 | 还没创建 `launch.json`(按步骤 4 做一次,以后会被记住);或者当前编辑器焦点不在一个 `.py` 文件上 |
| 断点显示是空心红圈,不是实心红点 | 断点设置在了没有可执行代码的行上(比如空行、纯注释行),挪到真正有代码的那一行;偶尔也可能是调试器还没准备好这个文件 |
| F5 之后程序直接跑完了,断点完全没有暂停 | 断点所在的代码分支根本没被执行到(比如设在了 `if` 的另一个分支里);或者不小心按了 `Ctrl+F5`("运行不调试",这个命令语义上就是不理会任何断点) |
| Variables 面板显示 "unavailable" 或者停留着上一次调试的旧值 | 调试会话已经结束但没注意到(看底部状态栏是不是还是橙色、调试工具条还在不在),重新 F5 启动一个新的调试会话 |
| Debug Console 里敲的内容没反应或者报 `NameError` | 当前调试会话不在"暂停"状态(程序正在跑或者已经结束)——Debug Console 只有在断点暂停时,才能访问到当时那个作用域里的变量 |

**自测清单:**

- [ ] 不看教程,能独立给一行代码设断点、启动调试、看到程序真的停在那一行
- [ ] 能说清楚 Step Over / Step Into / Step Out 三者的区别,并且知道什么场景该选哪个
- [ ] 断点暂停时,能通过 Variables 面板或者 Debug Console,准确说出某个变量此刻的真实值,而不是靠猜
- [ ] 独立用这一节的方法,在 `01-debug-demo.py` 里定位到 bug 所在的具体行,并且改对
- [ ] 知道 `launch.json` 是干什么用的,以及为什么通常只需要配置一次

## 5. 值得装的扩展

**为什么需要这个 / 不会有什么后果:**

VSCode 本体只是一个通用的文本编辑框架,真正"理解"某种语言、提供智能功能的部分,几乎全部来自扩展。这不是空话——VSCode 自带了一个内置的最基础的 "python" 语言支持包,本机现场读过它的 `package.json`,里面只注册了语法高亮(`grammars`)和文件关联,**没有**任何跳转定义、查找引用、调试相关的能力。也就是说,如果不装任何扩展,VSCode 打开一个 `.py` 文件,智能程度和一个带语法上色的记事本差不太多——第 3、4 两节讲的所有功能,背后都是扩展在起作用。

**环境要求:**

- 能访问 VSCode 扩展商店(Extensions Marketplace,需要联网)。

**一步步跟着做:**

1. **操作**:点左侧活动栏的扩展图标(四个方块、右下角带一个飞出的方块,快捷键 `Ctrl+Shift+X`)。
   **应该看到**:侧边栏出现搜索框,下面分"已安装"等分组。

2. **操作**:搜索框输入 "Python",找发布者精确是 "Microsoft"、名字就叫 "Python" 的那一个(应用商店里同名或者相似名字的扩展不少,认发布者不认图标花不花哨)。
   **本机真实现状**(已用 `code --list-extensions --show-versions` 核实过,不是假设):这台机器已经装了 `ms-python.python`(2026.4.0)、配套的语言服务器 `ms-python.vscode-pylance`(2026.3.1)、调试器 `ms-python.debugpy`(2026.6.0)。如果你在自己的新机器上第一次配置环境,搜到 "Python" 点 "Install" 即可,Pylance 和 debugpy 通常会作为依赖被一并自动装上。

3. **操作**:搜索框输入 "GitLens"。
   **本机真实现状**:确认**还没有装这个扩展**(同样用 `code --list-extensions` 核实过,完整列表里没有任何 `gitlens` 相关条目)。现在可以顺手装上,后续 Git 协作相关的文件会用到它。

4. **确认一个扩展到底装没装,不用只靠肉眼翻列表**。
   **操作**:在终端里跑 `code --list-extensions | grep -i gitlens`(Git Bash 环境)。
   **应该看到**:如果没装,这条命令没有任何输出;装了之后会打印出类似 `eamodio.gitlens` 的一行。这比在扩展面板"已安装"分组里用眼睛翻更快、更不容易漏看。

**各自解决什么问题(一句话说清楚,不用每个都展开讲):**

- **Python**(`ms-python.python`):让 VSCode 认识 Python 代码背后的"语义"而不只是给关键字上色——选择解释器(`.venv` 还是系统 Python 还是 conda 环境)、运行/调试入口、代码格式化,靠的都是它提供的基础设施。
- **Pylance**(`ms-python.vscode-pylance`):真正提供第 3 节用到的"跳转到定义""查找所有引用""函数签名提示""自动补全""类型检查"这些智能功能的语言服务器。没装它,Python 扩展本身的跳转能力非常有限,退化成简单的文本猜测,准确率明显更差。
- **GitLens**:在编辑器里逐行显示"这一行代码是谁写的、什么时候改的、属于哪次 commit"(这种标注叫 blame),不用切到终端敲 `git blame` 再对照行号找。读继承来的代码库时,想搞清楚"这段逻辑当初为什么这么写、是哪次改动引入的",GitLens 能省掉大量来回翻 `git log -p` 的功夫。
- **Remote-SSH**(`ms-vscode-remote.remote-ssh`,本机已确认装了):让 VSCode 能直接"打开"远程服务器上的一个文件夹,像操作本地文件一样编辑、调试远程 GPU 服务器上的代码。具体怎么配置连接、和 `~/.ssh/config` 怎么配合,[03-ssh-and-remote-servers.md](03-ssh-and-remote-servers.md) 会详细展开,这里只提一句它的存在。

**背后发生了什么:**

VSCode 的扩展系统本质上是把"语言智能"这件事外包给独立进程——Pylance 这类语言服务器跑在单独的进程里,通过一套标准协议(Language Server Protocol)和编辑器本体通信,编辑器负责界面和交互,语言服务器负责真正的代码分析。这也是为什么装扩展有时候需要"重新加载窗口"才生效:新进程需要重新启动才能接管。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| 搜"Python"出来一堆扩展,不知道装哪个 | 认发布者不认名字——官方的发布者精确是 "Microsoft" |
| 装了 Pylance 但 Go to Definition 还是不准 | 项目较大,可能还在后台建索引(留意右下角状态栏的进度提示);或者当前不是文件夹模式(回第 1 节) |
| 装了扩展但感觉没生效 | 少数扩展需要重新加载窗口才生效,装完如果没有自动弹出 "Reload" 提示,可以手动用命令面板执行 "Developer: Reload Window" |
| 同一功能出现好几个类似名字的扩展,装了以后互相冲突或重复提示 | 装到了非官方的同名/相似名扩展,卸载多余的,只保留官方发布者那个 |

**自测清单:**

- [ ] 能说出 Python 扩展和 Pylance 各自负责什么(两者是配合关系,不是同一个东西)
- [ ] 能独立在扩展商店里区分"官方"和"同名山寨"扩展
- [ ] 知道 GitLens 解决的是什么问题,即便暂时还没装它
- [ ] 记得 Remote-SSH 这个扩展的存在,以及深入内容要去哪个文件找

## 6. 常用快捷键 + 设置同步

**为什么需要这个 / 不会有什么后果:**

前面几节已经用到不少快捷键(`Ctrl+P`、`F12`、`Shift+F12`、`F9`、`F5`、`F10`/`F11` 等)。这一节汇总一份**真正常用、新手容易漏掉**的清单,目的不是让你背下来,而是"以后遇到类似需求时,想起这里有一份表可以回来查"。不背快捷键不影响任何功能——菜单栏永远能点到同样的操作——纯粹是效率差异:鼠标在菜单栏里翻找,和手不离开键盘直接触发,单次差异很小,但科研/工程日常里这类高频小操作一天要重复几十上百次,累积下来的时间差值得花十分钟记住最常用的这一小撮。

**环境要求:** 无(纯操作复习)。

**一步步跟着做 / 常用清单:**

只列真正高频、新手容易不知道的,不做成上百条的糊弄清单:

| 快捷键 | 作用 | 对应第几节 |
|---|---|---|
| `Ctrl+K Ctrl+O` | 打开文件夹 | 1 |
| `` Ctrl+` `` | 打开/关闭集成终端 | 2 |
| `` Ctrl+Shift+` `` | 新建一个终端标签 | 2 |
| `Ctrl+Shift+5` | 拆分终端 | 2 |
| `Ctrl+P` | 快速打开文件(按文件名模糊搜) | 3 |
| `Ctrl+Shift+F` | 全局文本搜索 | 3 |
| `F12` | 跳转到定义 | 3 |
| `Alt+F12` | 预览定义(不真的跳转过去) | 3 |
| `Shift+F12` | 查找所有引用 | 3 |
| `F9` | 在当前行切换断点 | 4 |
| `F5` | 启动调试 / 继续执行到下一个断点 | 4 |
| `F10` / `F11` / `Shift+F11` | 单步跳过 / 单步调入 / 单步跳出 | 4 |
| `Shift+F5` | 停止调试 | 4 |
| `Ctrl+Shift+X` | 打开扩展面板 | 5 |
| `Ctrl+S` | 保存当前文件 | 全局 |
| `Ctrl+B` | 显示/隐藏侧边栏 | 全局 |
| `Ctrl+Shift+P` | 命令面板 | 全局,见下面单独说明 |

特别提两个容易被忽略、但用起来很顺手的:

- **`Alt+Left` / `Alt+Right`**:回到上一个光标位置 / 前进到下一个。第 3 节里用 F12 跳来跳去之后,经常需要"回到刚才看的那个地方"——不用手动滚回去找,`Alt+Left` 直接原路返回,浏览器的"后退"按钮是同一个逻辑。
- **`Ctrl+Shift+P` 命令面板**:这是所有快捷键都记不住时的兜底方案。任何你知道名字但不知道快捷键的操作,打开命令面板直接搜关键词(中英文都能搜,比如搜"格式化"或者"format"),回车执行——命令面板里每条命令后面还会顺带标注它绑定的快捷键,用得多了自然就记住了高频的那几个,不需要死记硬背这张表。

**背后发生了什么:**

所有快捷键都是可自定义的(`File` → `Preferences` → `Keyboard Shortcuts`,或者命令面板搜 "Keyboard Shortcuts"),如果和其他软件的使用习惯冲突,可以自己改绑定,这里不展开。

**设置同步(简单提一句是什么、为什么有用):**

VSCode 自带一个叫 "Settings Sync" 的功能,能把你的设置、快捷键自定义、已装扩展列表、代码片段,通过登录 GitHub 或 Microsoft 账号同步到云端。对你现在的处境很实用:学校机房电脑、自己的笔记本、以后可能要连的远程服务器,大概率不是同一台机器,每台都从头装一遍扩展、调一遍设置很浪费时间。开一次 Settings Sync,以后换机器基本是登录账号就自动恢复,不用重新配置。

开启方式(图形界面操作,已按 1.129.1 版本核实菜单路径,但未做自动化验证):点左下角的账号图标(人形轮廓),选 "Turn on Settings Sync..."(启用设置同步),勾选想同步的内容(设置 / 快捷键 / 扩展 / 代码片段 / UI 状态,默认全选就行),用 GitHub 或 Microsoft 账号登录授权即可。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| 按了某个快捷键没反应,或者触发了别的东西 | 可能被其他扩展或者系统全局快捷键占用冲突了,命令面板搜 "Preferences: Open Keyboard Shortcuts" 能看到当前生效的完整绑定和冲突提示 |
| 记不住这么多快捷键 | 不需要死记,养成"不确定就 `Ctrl+Shift+P` 搜关键词"的习惯,用多了自然记住最高频的那几个 |
| 换了台电脑,设置和扩展要重新配一遍 | 没开 Settings Sync,回上面最后一段开启 |

- [ ] 不看表格,能默写出这份清单里至少 8 条快捷键和对应作用
- [ ] 遇到不确定叫什么名字的功能,知道用 `Ctrl+Shift+P` 命令面板搜关键词,而不是干瞪眼翻菜单
- [ ] 知道 Settings Sync 是干什么的,以及大致怎么开启
