## 1. 为什么需要 SSH —— 为什么不能在本地编辑器直接改远程 GPU 服务器上的代码,为什么密码登录不够好

**为什么需要这个 / 不会有什么后果:**

你现在写代码,大概率是在自己的 Windows 笔记本上用 VSCode 打开一个文件夹,改一个 `.py` 文件,保存,在本机的终端里直接 `python xxx.py` 跑起来看结果——能这样做,是因为代码、Python 解释器、要用到的数据,全部实实在在就在这台笔记本的硬盘上。

等你开始跑真正的科研实验——训练一个 world model,数据集几十上百 GB,需要一张甚至好几张 A100/3090 连续训练好几个小时到好几天——这些东西根本不可能放在你的笔记本上:你的笔记本可能根本没有能训得动的 GPU,就算有,散热和电费也扛不住连续跑好几天。真正跑实验的地方,是实验室的服务器、学校的计算集群,或者云厂商按小时计费的 GPU 实例——这是一台和你笔记本完全独立的机器,有自己的操作系统、自己的硬盘、自己的进程空间,物理上通过网络才能够到,你的笔记本对它的硬盘上有什么、内存里在跑什么,一无所知,反过来也一样。

具体后果非常直接:你在本地 VSCode 里把训练脚本的一个 bug 改好、按了 `Ctrl+S`——这个改动这一刻**只存在于你笔记本的硬盘上**。远程那台 GPU 服务器的硬盘上,还是改之前的旧代码,它对你刚才的操作一无所知。如果没有某种"把这次改动送过去、把远程机器上跑出来的日志/checkpoint 取回来"的机制,你在本地"写代码"和在远程"跑代码"这两件事,根本没法在物理上分离的两台机器之间衔接起来——你会被卡在"改了但传不过去"或者"想看远程的日志但够不着"这种最基础的问题上,更谈不上后面的调试、监控、迭代。

**SSH(Secure Shell)解决的正是这个"衔接"问题**:它让你在本地敲的命令,变成在远程机器上真正执行的命令——效果上就像有一根线,把远程机器的终端接到了你自己屏幕前面,你打字,回车,远程那台机器上真的在执行,它的输出真的传回来显示在你眼前,而这根"线"本身是加密的,数据在公网上跑一路也不用担心被人截获看懂。后面第 2-7 节要讲的密钥认证、多主机别名管理、端口转发、VSCode 直接编辑远程代码、文件同步、tmux 断线不丢任务,全部是建立在这根"线"之上的具体应用——没有 SSH,这些事一件都做不了。

**那"密码登录"哪里不够好?** 先纠正一个常见的误解:SSH 连接本身,不管你用密码还是用密钥登录,传输过程**都是加密的**——密码不存在"在网络上裸奔被人截获看懂"这个问题,这不是密码认证的真正短板。密码认证真正的问题在别处,而且是三个不同层面的问题:

1. **人选的密码普遍弱、还容易复用**——这不是 SSH 的错,是人性,你没法指望自己或者实验室每个人对每台服务器都设一个足够随机、从不复用的强密码,一旦某个服务的密码库泄漏,凡是复用过这个密码的地方全部遭殃。
2. **没法安全地自动化**——以后你会写脚本,让一台机器自动 SSH 到另一台机器上跑点什么(同步 checkpoint、提交定时任务、CI 流水线里的部署步骤……),脚本要嘛在代码里写死一个明文密码(极不安全,任何能读到这份代码的人都拿到了密码),要嘛没法在无人值守的情况下完成登录——这条路基本走不通。
3. **很多真实的远程服务器,压根不允许你用密码登录**——这不是理论上的担忧,下面就是真实证据。

**这次验证用的 WSL2 Rocky Linux,现场查到的真实配置,正好是这个行业惯例的一个具体例子:**

```
$ sshd -T | grep -E 'permitrootlogin|pubkeyauthentication|passwordauthentication'
permitrootlogin without-password
pubkeyauthentication yes
passwordauthentication yes
```

`permitrootlogin without-password` 这一条,意思是"root 这个账号,允许用密钥登录,但不允许用密码登录"——即使 `passwordauthentication` 整体是开着的(给其他普通账号用),root 账号本身已经被专门挡在密码认证之外。再看这台机器 root 账号真实的密码状态:

```
$ grep '^root:' /etc/shadow
root:*:20186:0:99999:7:::
```

`/etc/shadow` 第二个字段是密码的加密哈希,这里是一个孤零零的 `*`——这是 Linux 的约定写法,意思是"这个账号根本没有设置过可用的密码,密码登录在密码这一步之前就已经不可能通过"。真实发起一次强制只用密码认证(关掉密钥这条路,逼它走密码)的连接尝试,能看到真实的拒绝过程:

```
$ ssh -o PreferredAuthentications=password -o PubkeyAuthentication=no -o ConnectTimeout=5 root@localhost echo "should not print"
Permission denied, please try again.
Permission denied, please try again.
root@localhost: Permission denied (publickey,gssapi-keyex,gssapi-with-mic,password).
```

这不是这台 WSL2 机器故意搞特殊——**很多真实的云 GPU 实例,出厂配置就是这样**:创建实例的时候,你在控制台粘贴一个公钥,root(或者默认账号)从此只能用对应的私钥登录,密码登录整个被关闭。等你以后拿到实验室/云厂商发给你的一台真实远程机器,如果发现"输密码怎么都登不上",不要惊慌以为自己操作错了——大概率就是这个原因,而第 2 节要讲的密钥认证,就是你唯一的登录方式,不是可选项。

**环境要求:**

- 本机 Git Bash 已确认 `ssh` 客户端可用:`ssh -V` → `OpenSSH_9.0p1, OpenSSL 1.1.1q  5 Jul 2022`。Windows 上如果还没有,装 Git for Windows(装 Git Bash 顺带就有 OpenSSH 客户端)或者启用 Windows 自带的 OpenSSH 客户端可选功能均可,这篇不展开装法。
- "远程服务器"角色用的是 rhcsa-bash-deep-dive 已经装好的 WSL2 Rocky Linux,检查方法:`wsl -l -v` 看它是否存在;如果状态是 `Stopped`,跑一条 `wsl -d RockyLinux -u root -- true` 之类的命令会自动把它启动。**如实提醒**:WSL2 这个虚拟机闲置一段时间会自动休眠/重启(02 号文件已经记录过这个特性,和真实远程服务器"你不连它它也一直开着"的行为不一样),第 7 节会再具体讲一次这次真实撞到的情况。

**一步步跟着做:**

这一节的"步骤"更多是建立概念,不是敲一长串命令——但上面**为什么需要这个**里的每一条命令都是真实在这台机器上跑出来的,不是虚构的示例。用一张简单的示意图总结这一节的核心结论:

```
[你的 Windows 笔记本]                      [远程 GPU 服务器]
  VSCode / 终端                              数据、代码、GPU 都在这
  没有大 GPU、没有完整数据集          <——SSH(加密的“线”)——>   真正跑训练的地方
  改代码 → 存盘 → 只在本地生效                真正执行命令、真正读写文件的地方
```

记住这张图里的核心事实:**改代码这件事永远发生在“线”的某一端,要嘛你改的是本地文件(那和远程没关系),要嘛你通过 SSH 直接改的就是远程文件本身(第 5 节 VSCode Remote-SSH 干的就是这件事)——不存在“改一份本地文件,远程自动跟着变”这种隔空生效的操作,中间没有 SSH(或者构建在 SSH 之上的工具)搭桥,两边永远是两份互不知道对方存在的文件。**

**背后发生了什么:**

SSH 协议本身分两层:先建立一条加密的传输通道(基于 Diffie-Hellman 之类的密钥交换,协商出一份只有通信双方知道的会话密钥,之后所有数据用这份会话密钥加密),然后在这条通道之上做**身份认证**(证明"连过来的这个人真的有权限登录这个账号")。密码认证和密钥认证,都是发生在"通道已经加密"之后的认证阶段的两种不同方式——这也是为什么前面强调"密码不是在网络上裸奔",两者的安全性差异不在传输层,而在认证方式本身对"猜测/复用/自动化"这几类风险的抵抗力不同。第 2 节会具体讲密钥认证到底是怎么"证明身份"的。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| 输密码怎么都登不上,反复要求重新输入 | 目标账号可能像本节例子一样根本没设密码,或者服务端配置直接禁止了这个账号的密码登录(`PermitRootLogin without-password`/`prohibit-password` 是很常见的配置)——换密钥认证(第 2 节),不要一直重试密码 |
| 以为"公网传输不安全"所以密码登录比密钥"更麻烦但更安全" | 传输层加密和认证方式是两回事,SSH 的密码登录传输过程一样是加密的——密码认证的真实短板在"人的密码习惯弱"和"没法安全自动化",不是传输不加密 |
| 不理解"为什么改了本地文件远程没反应" | 本地文件和远程文件是两份完全独立的拷贝,中间没有任何自动同步机制,除非显式用 SSH/`scp`/`rsync`(第 6 节)或者 VSCode Remote-SSH(第 5 节,直接在远程编辑,没有"两份拷贝"的问题) |

**自测清单:**

- [ ] 能说清楚"本地改代码"和"远程跑代码"为什么是两个物理上独立的动作,中间靠什么衔接
- [ ] 能解释"SSH 传输加密"和"用密码还是密钥认证"是两个不同层面的事,不要混为一谈
- [ ] 能说出密码认证的三个真实短板(弱密码复用、没法安全自动化、很多远程服务器直接禁止),而不是笼统地说"密码不安全"
- [ ] 拿到一台新的远程服务器,如果密码登录不上,知道第一反应应该是"检查是不是只能用密钥",而不是死磕密码

## 2. SSH 密钥认证 —— 生成密钥对、部署公钥、告别密码

**为什么需要这个 / 不会有什么后果:**

第 1 节已经用真实证据说明:很多远程服务器(包括你以后大概率会用到的云 GPU 实例)根本不给密码登录这条路。密钥认证不是"更安全的可选项",是**唯一的路**。

密钥认证的核心思想,用一个类比理解:密码认证是"你和服务器共享同一个秘密(密码),服务器让你把秘密报一遍,对上了就放行"——这个模式的问题是,这个共享秘密**必须要在某个时刻被'亮出来'给服务器验证**,输入的动作本身、存储密码的地方,都是风险点。密钥认证换了一个完全不同的思路:你生成一**对**数学上相关联的文件——**私钥**和**公钥**。私钥你自己留着,永远不传给任何人、任何服务器;公钥可以随便复制给任何你想登录的服务器,泄露了也不怕别人拿它冒充你(公钥不能反推出私钥,这是这套非对称加密数学机制的基础)。登录的时候,服务器不要求你"报出秘密",而是给你出一道只有对应私钥才能正确回答的"题"(专业说法是"挑战-应答"),你的 SSH 客户端用私钥算出正确答案发回去,服务器用它保存的公钥验证这个答案对不对——**私钥全程没有在网络上传输过一个字节**,这是它比密码认证本质上更安全的地方,也是为什么私钥泄露的后果和密码泄露一样严重(必须妥善保管),但私钥本身**从来不需要主动交出去**这一点,给自动化(脚本免密登录)和多机器管理都打开了空间。

不会有什么后果——如果不会这一节的内容:你要嘛卡在第 1 节那道"密码登录不上"的墙前面,压根进不了远程服务器;要嘛即使某台服务器凑巧还允许密码,以后要同时管理好几台服务器(自己的、实验室的、云上临时开的),每台都要记一个密码、每次都要手动输入,慢且容易搞混,更谈不上写脚本自动化同步代码/checkpoint。

**环境要求:**

- `ssh-keygen`、`ssh-copy-id` 是 OpenSSH 客户端自带的工具,装了 `ssh` 一般就有——本机确认都在:`command -v ssh-keygen` → `/usr/bin/ssh-keygen`,`command -v ssh-copy-id` → `/usr/bin/ssh-copy-id`(这一条不是所有 Windows 环境都有,如果你的机器上没有,本节后半会讲手动等价操作)。
- 一个能连接的目标主机(本文用 WSL2 Rocky Linux 验证)。

**一步步跟着做:**

**第一步:生成密钥对。** 真实做法应该直接生成到 `~/.ssh/id_ed25519`(默认位置,以后 SSH 客户端会自动去这里找);**这次验证为了绝对不碰任何真实已有的密钥文件,故意生成到一个专门的临时目录、用明确不会撞名的文件名**,原理和默认位置完全一样:

```
$ ssh-keygen -t ed25519 -f ./dtd03_id_ed25519 -N "" -C "dtd03-temp-verify-2026-07-23"
Generating public/private ed25519 key pair.
Your identification has been saved in ./dtd03_id_ed25519
Your public key has been saved in ./dtd03_id_ed25519.pub
The key fingerprint is:
SHA256:V0vGGu+ATLrcGY6z7pfVJ9WWyMwNN7fetzf/0IDeDyg dtd03-temp-verify-2026-07-23
```

参数说明:`-t ed25519` 指定密钥算法(ed25519 是目前推荐的默认选择,比老式的 `rsa` 更短、更快、同等安全强度;如果连的是很老的系统不认 ed25519,退回 `-t rsa -b 4096`);`-f` 指定生成到哪个文件(不加的话会交互式问你,并默认建议 `~/.ssh/id_ed25519`);`-N ""` 是给私钥设置的"密码短语"(passphrase,给私钥文件本身再加一层保护,输入空字符串表示不设置——生产实践里更推荐设一个,这里为了脚本化验证方便留空,真实使用时按需权衡);`-C` 是一个纯粹给自己看的注释,通常写"这把钥匙是谁在哪台机器上生成的",不影响任何功能。

**第二步:看清楚公钥、私钥分别长什么样,理解为什么私钥不能给别人、公钥给别人没关系。**

```
$ ls -la
-rw-r--r-- 1 ericp 197611 419 dtd03_id_ed25519
-rw-r--r-- 1 ericp 197611 110 dtd03_id_ed25519.pub

$ head -1 dtd03_id_ed25519
-----BEGIN OPENSSH PRIVATE KEY-----

$ cat dtd03_id_ed25519.pub
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIKK9Mt4HQlNxRTiiRSGcnNwWtCCka1j/egbTn5a1KD29 dtd03-temp-verify-2026-07-23

$ ssh-keygen -lf dtd03_id_ed25519.pub
256 SHA256:V0vGGu+ATLrcGY6z7pfVJ9WWyMwNN7fetzf/0IDeDyg dtd03-temp-verify-2026-07-23 (ED25519)
```

不带 `.pub` 后缀的是**私钥**,内容是一大段 `BEGIN OPENSSH PRIVATE KEY` 包起来的加密数据,这个文件只应该存在于你自己的电脑上,不传给任何人、不传给任何服务器、不上传到 GitHub。带 `.pub` 后缀的是**公钥**,只有一行,格式是"算法名 + 一长串 base64 编码的公钥数据 + 你起的注释"——这个文件的用途就是**复制给你想登录的每一台服务器**,第三步就是干这件事。`ssh-keygen -lf` 能算出这把公钥的"指纹"(fingerprint,对公钥内容做哈希得到的一个短得多的摘要),第一次连接一台新服务器时,SSH 会把服务器的公钥指纹显示给你确认——这是用来防止连接被中间人偷换的机制,和这里"给自己的公钥算指纹"是同一套原理反过来用。

**第三步:光有密钥对不够——连接前必须先测一次"还没部署公钥"的真实失败状态,确认后面第五步的成功不是巧合。**

```
$ ssh -i ./dtd03_id_ed25519 -o BatchMode=yes -o ConnectTimeout=5 root@localhost echo "should not print"
root@localhost: Permission denied (publickey,gssapi-keyex,gssapi-with-mic,password).
```

私钥是真的、格式是对的,但服务器的 `authorized_keys` 里还没有对应的公钥——服务器不认识这把钥匙,自然拒绝。

**第四步:部署公钥——`ssh-copy-id` 的真实使用体验,包含一个值得记住的真实陷阱。** 正常情况下,`ssh-copy-id` 是最方便的部署方式,它本质上是帮你把公钥内容自动追加到远程的 `~/.ssh/authorized_keys` 里。但它内部有一个"先尝试登录一次,如果发现这把钥匙已经能登录了就跳过"的预检查逻辑——这次真实运行,暴露了这个预检查在**没有任何可用登录方式**的情况下会给出一个容易让人误判的结果:

```
$ ssh-copy-id -i ./dtd03_id_ed25519.pub -o ConnectTimeout=5 root@localhost
/usr/bin/ssh-copy-id: INFO: Source of key(s) to be installed: "./dtd03_id_ed25519.pub"
/usr/bin/ssh-copy-id: INFO: attempting to log in with the new key(s), to filter out any that are already installed

/usr/bin/ssh-copy-id: WARNING: All keys were skipped because they already exist on the remote system.
		(if you think this is a mistake, you may want to use -f option)
```

看到这条 `WARNING` 很容易以为"哦,已经装过了,不用管了"——**但这是假的**。立刻用这把钥匙重新尝试登录,验证真相:

```
$ ssh -i ./dtd03_id_ed25519 -o BatchMode=yes -o ConnectTimeout=5 root@localhost echo "REALLY LOGGED IN"
root@localhost: Permission denied (publickey,gssapi-keyex,gssapi-with-mic,password).
```

**还是登不上。** 到 Rocky 那边直接确认 `authorized_keys` 文件根本不存在:

```
$ ls -la /root/.ssh/
total 8
drwx------ 2 root root 4096 .
dr-xr-x--- 9 root root 4096 ..
$ cat /root/.ssh/authorized_keys
cat: /root/.ssh/authorized_keys: No such file or directory
```

真相是:`ssh-copy-id` 的预检查本身需要先用某种方式登录一次才能判断"是不是已经装过了",而这台机器上 root 账号既没有密码、这把新钥匙又还没被装上——预检查连接本身失败的方式,被这个具体版本的 `ssh-copy-id` 脚本误判成了"已存在,跳过"(读过它的源码确认:预检查逻辑是"用新钥匙尝试登录,如果输出里**没有看到** `Permission denied` 字样就当作已经装好了"——这次的失败路径,连接过程没有干净利落地打出这个关键字符串,导致误判)。**这是一个值得记住的真实教训:`ssh-copy-id` 报"已存在/跳过",不代表钥匙真的能用,尤其是目标账号没有任何其它可用登录方式的时候,必须自己再实际测一次登录才能确认。**

**第五步:`ssh-copy-id` 走不通时的手动等价方法——这也是很多云 GPU 实例真实的初始化方式。** `ssh-copy-id` 本质上只是自动化了"把公钥内容追加到远程 `~/.ssh/authorized_keys` 末尾"这一件事;理解了这个本质,任何能碰到远程机器文件系统的手段都能达到同样效果。云厂商的做法通常是:你在创建实例时,在控制台网页表单里粘贴你的公钥内容,实例第一次开机时由云初始化脚本自动写好——这一步压根不经过 SSH,因为这时候你还没有任何方式 SSH 上去。这次验证环境里,等价的"不经过 SSH、直接摸到机器"的通道,是 WSL2 提供的 `wsl -d RockyLinux -u root` 直接进入(不需要密码,Windows 信任当前登录用户),相当于云控制台的"免密初始化"入口:

```
$ wsl -d RockyLinux -u root -- bash -c "
    mkdir -p /root/.ssh && chmod 700 /root/.ssh
    echo 'ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIKK9Mt4HQlNxRTiiRSGcnNwWtCCka1j/egbTn5a1KD29 dtd03-temp-verify-2026-07-23' >> /root/.ssh/authorized_keys
    chmod 600 /root/.ssh/authorized_keys
  "
$ wsl -d RockyLinux -u root -- ls -la /root/.ssh/
-rw------- 1 root root  110 authorized_keys
```

没有 `ssh-copy-id`、也没有任何带外通道的极简系统上,同样的效果还有一种更常见的写法(把本地公钥内容通过管道传给一条远程命令,靠已有的某种登录方式——比如密码——执行追加):`cat id_xxx.pub | ssh user@host "mkdir -p ~/.ssh && chmod 700 ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"`,这次环境没有可用的密码登录,没法采用这条路径,但原理和上面完全一致,记住即可。

**第六步:真正验证密钥登录成功——这是本节最终的证据。**

```
$ ssh -i ./dtd03_id_ed25519 -o BatchMode=yes -o ConnectTimeout=5 root@localhost "whoami && hostname && echo 'key-based auth: SUCCESS'"
root
Eric-12900H-3080Ti
key-based auth: SUCCESS
```

不需要输任何密码,`whoami` 真实返回 `root`、`hostname` 真实返回这台机器的名字——密钥认证完整跑通。

**第七步(如实记录一个 Windows 平台上的真实差异,不是必修步骤但值得知道):私钥文件权限,Linux 管得很严,这次实测 Windows 这一侧的行为不一样。** 在真正的 Linux 系统上,如果私钥文件的权限过于开放(比如同组或其他人可读),SSH 客户端会直接拒绝使用这把私钥,报 `UNPROTECTED PRIVATE KEY FILE!` 并中止——这是 rhcsa-bash 系列已经讲过的经典坑。这次在 Windows Git Bash 上真实复现同样的操作:

```
$ chmod 644 dtd03_id_ed25519
$ icacls dtd03_id_ed25519
dtd03_id_ed25519 ERIC-12900H-308\CodexSandboxUsers:(I)(M)
                 ...(其余若干账号，均带 Modify 或 Full 权限)...

$ ssh -i ./dtd03_id_ed25519 -o BatchMode=yes -vvv root@localhost echo test 2>&1 | grep -iE "warn|permiss|unprotected"
root@localhost: Permission denied (publickey,gssapi-keyex,gssapi-with-mic,password).
```

**真实观察到的结果**:即使私钥的权限位显示成 `644`(且 `icacls` 显示的真实 Windows ACL 也确实把 Modify 权限开给了好几个账号),这次用到的这套 Windows Git Bash + OpenSSH_9.0p1 组合,**没有触发预期中 Linux 上会看到的 `UNPROTECTED PRIVATE KEY FILE` 拒绝**——上面的连接失败是因为这次特意没有重新部署这把改过权限的钥匙,不是因为权限检查生效。这是如实记录的真实现象,不代表所有 Windows 环境都这样(不同 OpenSSH 构建、是否是微软官方 Win32-OpenSSH 移植版还是 Git for Windows 自带的 MSYS2/Cygwin 构建,行为可能不同)——**不要因为这次在 Windows 客户端上没被拦下来就掉以轻心**,私钥文件应该始终保持"只有自己能读",尤其是当你以后用真正的 Linux 客户端(比如实验室工作站、另一台服务器上转发密钥)连接时,这道检查是真实会拦你的。

**背后发生了什么:**

- **非对称加密的"挑战-应答"机制**:私钥和公钥在数学上成对生成,公钥能验证"只有对应私钥才能算出的答案",但公钥本身不能反推出私钥、也不能冒充私钥去回答挑战。这就是为什么公钥可以随便公开、随便复制到任意多台服务器的 `authorized_keys` 里,而私钥只有一份、必须留在你自己手上。
- **`authorized_keys` 文件的本质**:它就是一份纯文本清单,一行一个公钥,`sshd` 收到一次登录尝试时,会用这份清单里的每一个公钥依次验证挑战应答,只要有一个对上了就放行——这也是为什么可以在同一个账号的 `authorized_keys` 里放好几把不同的公钥(比如你的台式机一把、笔记本一把),互不冲突,任何一把都能登录同一个账号。
- **`ssh-keygen` 生成的私钥文件权限,以及 `ssh-copy-id` 的本质**:`ssh-keygen` 在类 Unix 系统上默认就会把私钥文件权限设成 `600`(只有属主能读写),这是设计上的默认保护;`ssh-copy-id` 没有任何"专有魔法",本质就是自动化了"读公钥文件内容 → 通过某种已有的登录方式连上远程 → 追加进 `~/.ssh/authorized_keys`"这几步,理解了这一点,任何等价的手动操作都能达到同样效果,不依赖这个工具是否存在或者是否在特定环境下正常工作。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| `ssh-copy-id` 报"already exist / skipped",但实际登录还是失败 | 本节第四步真实记录的陷阱——`ssh-copy-id` 的预检查在**完全没有可用登录方式**的情况下可能误判,不要相信这条提示,务必自己用密钥再测一次真实登录 |
| 生成密钥对时手滑覆盖了已有的 `~/.ssh/id_ed25519` | `ssh-keygen -f` 如果指向一个已存在的文件,会问你要不要覆盖(默认交互式确认),但脚本化/非交互场景下容易疏忽——不确定的时候先 `ls ~/.ssh/` 看清楚有什么,用一个明确不会撞名的新文件名(本文全程就是这么做的) |
| 部署完公钥,登录还是要求密码 | 检查目标账号的 `~/.ssh` 目录和 `authorized_keys` 文件权限是不是过于开放(`sshd` 出于安全考虑,如果 `~/.ssh` 或 `authorized_keys` 权限不对,可能直接忽略这份文件,静默退回其它认证方式而不是明确报错)——本节部署时特意 `chmod 700 ~/.ssh` + `chmod 600 authorized_keys`,不是可省略的步骤 |
| 私钥权限过开放,Linux 客户端直接拒绝 `UNPROTECTED PRIVATE KEY FILE` | 这是 Linux 上标准、预期的保护行为,`chmod 600 私钥文件` 修好即可;如果你在 Windows 上没看到这个拦截,不代表可以不管,参见本节第七步的诚实记录 |
| 一台服务器要给好几个人/好几台自己的设备登录权限 | 不要共享同一把私钥——给每个人/每台设备单独生成一对密钥,把各自的公钥都追加进同一个 `authorized_keys`(一行一个),这样出问题时能单独吊销某一把,不用全员换钥匙 |

## 3. `~/.ssh/config` 多主机管理 —— 给连接起别名,不用每次打一长串命令

**为什么需要这个 / 不会有什么后果:**

第 2 节验证密钥登录用的完整命令是这样的:`ssh -i ./dtd03_id_ed25519 -o BatchMode=yes -o ConnectTimeout=5 root@localhost`——这条命令能用,但只是"能用",不是"好用"。真实科研场景里,你手上大概率不止一台机器要连:实验室自己的 GPU 服务器一台、学校集群的登录节点一台、云上临时开的实例可能又是一台,每台可能用户名不同、端口不同(有些服务器出于安全考虑把 SSH 端口从默认的 22 改成别的数字)、要用哪把私钥也可能不同(不同机构可能要求你用各自单独生成的密钥)。如果每次都要完整背出"用户名、地址、端口、密钥路径"这一整串,不仅慢,还极容易在切换机器的时候记混——比如把该用密钥 A 登录的机器,手滑打成了密钥 B 的路径,平白多花时间排查"明明装了公钥怎么登不上"。

`~/.ssh/config` 解决的正是这个问题:给每一台你常连的服务器起一个好记的别名,把用户名、地址、端口、密钥路径这些参数都提前写好,以后连接只需要敲这个别名,SSH 客户端自动帮你把剩下的参数补全。

**环境要求:**

- 和第 2 节一样的密钥、一样能连接的目标主机。
- `~/.ssh/config` 是纯文本文件,任何编辑器都能改;如果之前从来没配置过,这个文件很可能还不存在,直接新建即可。**这次验证要特别小心的一点**:这台机器上真实存在一个已经在用的、里面有真实服务器信息的 `~/.ssh/config`(第 0 节已经说明了这次差点覆盖它、后来改为完全绕开真实路径操作的经过)——**动手改这个文件前,先确认清楚自己改的到底是哪一个 `~/.ssh/config`**,如果你的电脑上也已经有别的服务器配置在里面,后面的操作是**追加一段新的 `Host` 块**,不要整个文件覆盖掉。

**一步步跟着做:**

**第一步:写一个 `Host` 块。** 语法是一个 `Host 别名` 起头,后面每一行缩进写一个参数:

```
Host dtd03-rocky-test
    HostName localhost
    User root
    Port 22
    IdentityFile /tmp/dtd03-ssh-verify/dtd03_id_ed25519
```

`Host` 后面这个词是你自己起的别名,和真实的服务器地址无关,想叫什么都行(比如 `lab-gpu`、`hpc-login`、`my-cloud-a100`,起一个自己看得懂、不容易和别的机器搞混的名字);`HostName` 才是真正要连接的地址(域名或 IP);`User` 是登录的账号名;`Port` 是 SSH 端口(默认 22,服务器如果改了端口必须写这里,不然会连不上);`IdentityFile` 指定这台服务器用哪把私钥(前面强调过,不同服务器可能要求用不同的密钥,这一行就是用来区分的)。真实使用时,这一段通常直接写在默认路径 `~/.ssh/config` 里(不带任何 `-F` 参数,SSH 客户端默认就会读这个文件);本文的 `IdentityFile` 之所以写了一个 `/tmp/...` 的临时路径而不是常见的 `~/.ssh/id_ed25519`,只是因为第 2 节这把验证用的密钥本身就没有放在默认位置(前面解释过原因),不影响语法本身。

**第二步:不发起真实连接,只看 SSH 客户端怎么解析这段配置——`ssh -G`。** 这是一个很好用但很多人不知道的调试手段:它只做配置解析,不建立任何网络连接,适合在连之前先确认参数到底对不对:

```
$ ssh -G dtd03-rocky-test | grep -E "^(hostname|user|port|identityfile) "
user root
hostname localhost
port 22
identityfile /tmp/dtd03-ssh-verify/dtd03_id_ed25519
```

四个参数全部按预期解析出来了——如果这里看到的值跟你以为写的不一样(比如 `HostName` 打错了字母),问题在配置文件本身,还没走到网络连接那一步,排查起来更快。

**第三步:真实连接——只用别名,不用再输任何其它参数。**

```
$ ssh -o BatchMode=yes dtd03-rocky-test "whoami && echo 'connected via alias: SUCCESS'"
root
connected via alias: SUCCESS
```

对比第 2 节那条完整命令,现在只需要 `ssh dtd03-rocky-test`——用户名、地址、端口、密钥路径,全部由配置文件自动补上。以后第 4、6、7 节用到的每一次连接,都会直接用这个别名,不再重复写完整参数,这也是为什么把这一节放在端口转发、文件同步、tmux 组合之前讲。

**第四步:真实的多主机场景,只是多写几个 `Host` 块。** 假设你以后同时要管实验室服务器和一台云实例,`~/.ssh/config` 里追加对应的块即可,互不影响:

```
Host lab-server
    HostName 10.0.1.50
    User yourname
    IdentityFile ~/.ssh/id_ed25519

Host cloud-a100
    HostName 203.0.113.42
    User ubuntu
    Port 2222
    IdentityFile ~/.ssh/cloud_key
```

以后 `ssh lab-server`、`ssh cloud-a100`,各自带各自的参数,不用每次切换机器都重新想一遍这台机器的用户名/端口/密钥是什么。

**背后发生了什么:**

`ssh` 客户端每次启动,会按固定顺序读取几处配置来源(命令行参数优先级最高,然后是用户级的 `~/.ssh/config`,再是系统级的 `/etc/ssh/ssh_config`),把同一个 `Host` 块里出现的参数当成这次连接的默认值,命令行上显式指定的参数(比如手动加的 `-p`)会覆盖配置文件里写的值。`Host` 后面的别名支持简单的通配符匹配(比如 `Host *.mylab.edu` 能一次性给一批域名相同后缀的机器设置公共参数),这篇不展开,知道存在即可,用到的时候再查。`ssh -G` 之所以能在不连接的情况下打印出解析结果,是因为它只做到"配置合并"这一步就停下来,不再往后走真正建立 TCP 连接、协商加密、认证的流程——这也是为什么它是一个安全、无副作用的调试手段,可以随便用来检查配置对不对。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| 缩进/格式写错,配置没生效 | `Host` 这一行必须顶格写,下面的参数行必须有缩进(不缩进也能工作,但缩进是约定俗成的可读性写法,强烈建议保持);参数名不区分大小写,但拼写必须准确(`HostName` 不是 `Hostname`,虽然大部分实现对此宽容,不同版本行为可能不一致,照着标准写法抄最保险) |
| 改完 `~/.ssh/config` 不生效 | 确认改的是真的在用的那个 `~/.ssh/config`(尤其是这台机器上有不止一个 SSH 客户端/不止一个 `HOME` 环境的情况下,本文第 0 节的真实教训);也检查有没有多个同名 `Host` 块——SSH 对同一个参数,**只认第一次出现的那个值**,后面重复定义的会被忽略,不是"后写的覆盖先写的" |
| 连接一台新服务器,提示公钥指纹要求确认 | 这是第一次连接的正常提示(防中间人机制,第 2 节讲密钥指纹时提过),确认地址无误后输入 `yes` 即可,以后这台机器的指纹会被记进 `~/.ssh/known_hosts`,下次不会再问 |
| 多个 `Host` 块的地址其实是同一台机器,行为却好像连了不同机器 | 检查 `IdentityFile` 是不是配错了(比如两个别名不小心指向了同一把不匹配这台机器的密钥),或者 `Port` 写错——`ssh -G 别名` 是最快的排查手段,先看解析出来的参数对不对,再怀疑网络问题 |

**自测清单:**

- [ ] 能独立写出一个包含 `Host`/`HostName`/`User`/`Port`/`IdentityFile` 的完整配置块
- [ ] 知道 `ssh -G 别名` 是干什么用的,以及为什么它是一个安全的排查手段(不会真的发起连接)
- [ ] 能说出"同一个参数在配置文件里出现两次,以哪个为准"这条容易踩坑的规则
- [ ] 理解配置里的 `Host` 别名是自己起的名字,和真正的服务器地址(`HostName`)是两回事

## 4. 端口转发(`-L`)—— 在自己电脑浏览器里打开远程的 Jupyter/TensorBoard

**为什么需要这个 / 不会有什么后果:**

训练模型的时候,Jupyter Notebook 和 TensorBoard 这类工具几乎是标配——但它们的一个共同点是:默认都是启动一个本地网页服务,监听在启动它的那台机器的某个端口上(比如 Jupyter 常见的 `8888`,TensorBoard 常见的 `6006`)。你在远程 GPU 服务器上敲 `jupyter notebook` 或者 `tensorboard --logdir=...`,这个网页服务是**监听在远程服务器自己的网络接口上**——而且出于安全考虑,几乎所有教程和默认配置都会让它只绑定在 `127.0.0.1`(也就是"这台机器的自己人"),不对公网暴露,原因很直接:Jupyter 默认配置下几乎等于给了"在这台机器上执行任意代码"的权限,直接暴露在公网上、被任何人访问到,是非常危险的。

后果就是:你没法直接在自己笔记本的浏览器地址栏敲一个远程服务器的 IP 加端口号就打开它——远程服务器的 `127.0.0.1` 和你笔记本的 `127.0.0.1` 完全是两个不同的地址空间,你的浏览器请求根本到不了那台服务器上监听的那个端口。端口转发就是用来解决这个"我能 SSH 上去,但浏览器直接访问不到"的问题:把远程服务器上只有它自己能访问的一个端口,通过已经建立的 SSH 加密隧道,"接"到你自己电脑上的一个端口——之后你在自己电脑浏览器里访问自己的这个本地端口,实际请求会通过隧道转发到远程那个端口,效果就像那个服务本来就跑在你自己电脑上一样。

**环境要求:**

- 一条已经能用的 SSH 连接(本节直接复用第 3 节配置好的 `dtd03-rocky-test` 别名)。
- Rocky 端要有一个监听在它自己 `127.0.0.1` 上的服务用来验证——这次用 Python 自带的 `http.server` 模拟(道理和 Jupyter/TensorBoard 完全一样,都是"绑在远程自己的 loopback 上,外面直接访问不到"这个模式,只是换成一个最简单、不需要额外装包就能验证的服务)。

**一步步跟着做:**

**第一步:在 Rocky 端起一个只监听它自己 `127.0.0.1` 的测试服务,模拟"远程的 Jupyter/TensorBoard"。**

```
$ wsl -d RockyLinux -u root -- bash -c "
    echo 'this is the remote GPU servers Jupyter-like service...' > /root/dtd03-verify/marker.txt
    cd /root/dtd03-verify && nohup python3 -m http.server 8899 --bind 127.0.0.1 > httpserver.log 2>&1 &
  "
$ wsl -d RockyLinux -u root -- ss -tlnp | grep 8899
LISTEN 0  5  127.0.0.1:8899  0.0.0.0:*  users:(("python3",pid=1487,fd=3))
```

注意 `--bind 127.0.0.1`——这一步是故意的,确保这个服务只在 Rocky 内部能访问,和真实的 Jupyter/TensorBoard 默认行为一致。

**第二步:建立转发隧道。**

```
$ ssh -o BatchMode=yes -f -N -L 18899:localhost:8899 dtd03-rocky-test
```

`-L 18899:localhost:8899` 是端口转发的核心语法,拆开理解:`-L 本地端口:目标主机:目标端口`——**"本地端口"(`18899`)是你自己电脑上会新开始监听的端口**;**"目标主机:目标端口"(`localhost:8899`)是从 SSH 服务器(这里是 Rocky)的视角出发,要连去哪里**——这里最容易搞混的一点是,这个 `localhost` 指的是**远程机器自己的 localhost**,不是你本机的 localhost,连接建立后,是 Rocky 自己发起"我连我自己的 8899"这个动作,你的电脑全程不直接碰 Rocky 的网络,只是把请求经过隧道"委托"给 Rocky 去连。`-f` 表示建好隧道之后转入后台(不占着这个终端);`-N` 表示这次连接不需要执行任何远程命令,只是单纯建隧道。

**第三步:验证——用你自己电脑上的 `18899` 端口,取到远程服务的真实内容。**

```
$ curl -s --noproxy '*' http://127.0.0.1:18899/marker.txt
this is the remote GPU servers Jupyter-like service, port forwarding proves this line is real
```

从你自己电脑请求 `127.0.0.1:18899`,拿到的内容是 Rocky 那边 `127.0.0.1:8899` 上真实提供的——这条数据完整走了"你的请求 → SSH 隧道 → Rocky 自己连它自己的 8899 → 原路返回"这整个路径。真实场景下,把这里的 `18899` 换成你自己习惯的端口、把 `8899` 换成 Jupyter/TensorBoard 真正监听的端口,浏览器里直接打开 `http://127.0.0.1:18899`,效果完全一样。

**第四步(如实记录一个真实撞到的环境巧合,不影响结论但必须说清楚):这次的 WSL2 环境,`8899` 端口不需要隧道也能直接访问到。**

```
$ curl -s --noproxy '*' --max-time 3 http://127.0.0.1:8899/marker.txt
this is the remote GPU servers Jupyter-like service, port forwarding proves this line is real
```

这个结果一开始看起来像是"隧道没有意义"——真实原因是这次 WSL2 用的是较新的 `mirrored` 网络模式(第 0 节已经提过,`.wslconfig` 里现场确认 `networkingMode=mirrored`),这个模式下 **Windows 和 WSL2 共享同一份网络接口,包括 loopback**,所以 Rocky 里绑定在 `127.0.0.1` 的服务,巧合地在 Windows 这边的 `127.0.0.1` 也能直接访问到。**这是这次实验环境的特殊性,不是端口转发"不需要"的证据**——真实的远程 GPU 服务器,它的 `127.0.0.1` 只在它自己内部有意义,你笔记本的 `127.0.0.1` 是完全独立的另一个地址空间,中间隔着真实的网络甚至公网,不存在"共享 loopback"这回事,`-L` 隧道会是你唯一的访问路径。

**第五步:更严谨的证据——只看隧道自己开的那个端口(`18899`),证明它的可用性完全跟着隧道进程走,和上面的巧合无关。**

```
$ curl -s --noproxy '*' --max-time 3 http://127.0.0.1:18899/marker.txt   # 隧道还活着
this is the remote GPU servers Jupyter-like service, port forwarding proves this line is real

$ kill 2284   # 杀掉隧道进程(2284 是 ssh -f -N -L 这条命令的真实 PID)
$ curl -s --noproxy '*' --max-time 3 http://127.0.0.1:18899/marker.txt   # 隧道已死
$ echo "curl exit=$?"
curl exit=7

$ ssh -o BatchMode=yes -f -N -L 18899:localhost:8899 dtd03-rocky-test   # 重新建立隧道
$ curl -s --noproxy '*' --max-time 3 http://127.0.0.1:18899/marker.txt   # 立刻又能用了
this is the remote GPU servers Jupyter-like service, port forwarding proves this line is real
```

`18899` 这个端口本身,除了这条 `ssh -L` 隧道,没有任何别的东西会监听它——杀掉隧道进程,这个端口立刻连不上(`curl` 返回码 `7`,表示连接失败);重新建立隧道,立刻又能用。这组"杀掉即失效、重建即恢复"的因果关系,不受第四步那个 loopback 共享巧合的影响,是这次验证里对"隧道本身真的在工作"最直接的证据。

**背后发生了什么:**

`-L`(本地转发,Local forwarding)的工作方式是:你的 SSH 客户端在本地机器上开始监听你指定的本地端口(这里是 `18899`);任何连到这个端口的流量,客户端会把它包进已经加密的 SSH 连接里,发给远程的 `sshd`;`sshd` 收到之后,代替你去连接你指定的"目标主机:目标端口"(这里是从 Rocky 自己的视角连 `localhost:8899`);目标服务的响应再原路通过隧道传回来。整个过程里,真正发起"连 8899"这个动作的,始终是远程的 `sshd` 进程,不是你的本地客户端——这就是为什么"目标主机"那部分的 `localhost` 要以远程视角理解,以及为什么这个机制能够转发到"只有远程自己能访问"的端口:因为发起连接的本来就是远程自己。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| `bind: Address already in use` | 本地端口已经被别的程序占用,换一个没被占用的本地端口(`-L` 左边的数字可以随便挑一个空闲的,不需要和远程端口一致,本文特意用 `18899` 而不是 `8899` 就是为了避免这类混淆) |
| 浏览器打开转发的本地端口,页面加载了但样式/资源加载不全,或者跳转后打不开了 | Jupyter/TensorBoard 这类应用有些资源路径或者 WebSocket 连接是硬编码/动态生成的,可能没有考虑到"被转发到另一个端口"的情况——常见解决办法是转发时让本地端口和远程端口保持一致数字(比如都用 `8888`),减少这类路径不一致的问题 |
| 关掉建隧道用的那个终端窗口,转发跟着失效了 | 如果建隧道时没加 `-f` 转后台,这条 `ssh` 命令会一直占着前台,关掉终端窗口会连带杀死它;加 `-f -N`(本节写法)转后台之后,终端可以正常继续用,隧道进程独立存在,直到你手动 `kill` 掉它或者断网 |
| 以为不建隧道也能直接访问远程端口(把这次 WSL2 的巧合当成了通用结论) | 参见本节第四步——这是 `mirrored` 网络模式下 Windows/WSL2 共享 loopback 的巧合,真实的、物理上独立的远程服务器不会有这个现象,不要以此类推 |
| 隧道建立后一段时间自动断开 | 检查是不是网络不稳定导致底层 SSH 连接本身断了(隧道依附于这条 SSH 连接,连接没了隧道也没了)——第 7 节会讲配合 `tmux` 保护"任务不受连接断开影响"的做法,但要注意隧道这类"当前连接自带"的功能,连接一断本身就没了,和 tmux 里独立运行的任务不是一回事,重新连接后需要重新建立隧道 |

**自测清单:**

- [ ] 能默写出 `-L 本地端口:目标主机:目标端口` 的完整语法,并且解释清楚"目标主机"是从谁的视角出发
- [ ] 能说清楚为什么 Jupyter/TensorBoard 默认只绑定 `127.0.0.1`,以及这为什么导致直接访问不了
- [ ] 知道 `-f -N` 两个参数分别的作用
- [ ] 能设计一个实验,证明"转发出来的本地端口"和"隧道进程是否存活"之间的因果关系(不是想当然,是真的能验证)
- [ ] 理解 SSH 连接断开会导致依附在它上面的端口转发跟着失效,这和 tmux 里独立运行的任务是两回事

## 5. VSCode Remote-SSH —— "远程当本地用"

**为什么需要这个 / 不会有什么后果:**

[01-vscode-editor-workflow.md](01-vscode-editor-workflow.md) 第 5 节已经提过 Remote-SSH 这个扩展的存在("让 VSCode 能直接'打开'远程服务器上的一个文件夹,像操作本地文件一样编辑、调试远程 GPU 服务器上的代码"),这一节把它具体展开。

先想清楚:如果没有这个工具,你有哪些办法在远程服务器上写代码?一种是纯命令行编辑器(`vim`/`nano`),直接在 SSH 终端里改——能用,但你已经在 01 号文件里体会过 VSCode 的跳转定义、查找引用、调试器这些功能带来的效率差异,纯终端编辑器不具备(或者具备得很有限)这些能力。另一种是"本地改完,用第 6 节的 `scp`/`rsync` 传上去"——这样做的问题是,你本地的这份代码和远程真正在跑的那份代码,变成了两份需要你自己手动保持同步的拷贝,忘记同步、同步错方向(把旧版本盖过了新版本)这类事故很容易发生,尤其是来回改的次数一多。

Remote-SSH 提供的是第三条路:**VSCode 窗口本身长得和平时一模一样,左侧 Explorer 面板、集成终端、调试器、Pylance 的跳转定义全部照常能用——但这个窗口此刻编辑的,是远程服务器磁盘上的真实文件,不是本地的拷贝**。你按 `Ctrl+S` 保存,写入的直接就是远程那份;你在集成终端里跑命令,直接就是在远程执行;你设断点、按 F5 调试,调试器真的是在远程那台有 GPU 的机器上跑起来的。没有"两份拷贝需要同步"这个问题,因为自始至终只有一份。

**环境要求:**

- 已装 Remote-SSH 扩展。本机现场核实的真实清单:`ms-vscode-remote.remote-ssh@0.124.0`(核心扩展)、`ms-vscode-remote.remote-ssh-edit@0.87.0`(编辑 SSH 配置文件时的语法高亮/补全支持)、`ms-vscode.remote-explorer@0.5.0`(侧边栏统一管理 SSH/WSL/Tunnel 等各类远程目标的面板)。如果你的机器上还没有,扩展商店搜 "Remote - SSH"(发布者 Microsoft)安装即可。
- 已经按第 3 节配置好 `~/.ssh/config`——Remote-SSH 会自动读取这个文件里的 `Host` 列表,不需要在扩展里重新配置一遍连接信息,这是它和第 3 节内容直接衔接的地方。

**一步步跟着做:**

1. **打开远程主机列表。操作**:点 VSCode 窗口左下角的"><"图标(这是 Remote 相关扩展统一使用的入口,和 `ms-vscode.remote-explorer` 扩展提供的侧边栏是同一套体系),或者 `Ctrl+Shift+P` 命令面板搜 "Remote-SSH"。
   **应该看到**:一个下拉菜单,包含 "Connect to Host..."(连接到主机)这类选项——这个字符串已经从本机安装的 0.124.0 版本扩展包里现场核实存在,不是凭旧版本印象写的。选择后,弹出的主机列表**直接读取自你的 `~/.ssh/config`**,第 3 节配置的每一个 `Host` 别名都会出现在这里(这也是为什么本节要求先完成第 3 节)。

2. **连接到一个主机。操作**:点选列表里的一个别名(比如第 3 节配的 `dtd03-rocky-test`)。
   **应该看到**:VSCode 弹出新窗口(或者复用当前窗口,取决于你的选择),左下角状态栏出现和这台远程主机相关的连接状态提示;首次连接一台新主机,VSCode 会自动在远程主机上下载安装一个轻量的 "VS Code Server" 组件(这是本机从扩展的真实设置项里核实到的机制,下面"背后发生了什么"细讲),这一步需要远程主机能访问网络,连接速度取决于网速。

3. **像本地一样打开远程文件夹。操作**:连接成功后,用平常的 `File > Open Folder`(和 01 号文件第 1 节完全一样的操作),选择远程主机上的一个目录(比如你训练代码所在的目录)。
   **应该看到**:左侧 Explorer 面板出现的是**远程那台机器磁盘上真实的目录结构**,不是本地的;集成终端(`` Ctrl+` ``)打开的是一个真实运行在远程主机上的 shell;01 号文件第 3、4 节讲的跳转定义、断点调试,在这里同样能用——前提是远程主机上也要有对应的语言支持扩展(比如 Python + Pylance),VSCode 会在你打开一个 `.py` 文件时提示是否要在这个远程主机上安装,这是因为语言分析这类功能需要在文件实际所在的那一端运行,不能靠本地扩展跨网络分析远程的代码。

4. **确认自己现在到底在"本地"还是"远程"。** 这是新手最容易迷糊的地方——看两个位置:窗口左下角状态栏最左侧的"><"区域,会明确写着当前连接的主机名(比如 `SSH: dtd03-rocky-test`);标题栏也会带上主机名。不确定自己此刻改的文件到底是本地还是远程时,先看这两处,不要凭感觉猜。

5. **断开/切回本地。操作**:命令面板搜 "Remote-SSH: Close Remote Connection",或者直接 `File > Close Remote Connection`。
   **应该看到**:窗口切回本地模式,左下角"><"区域的主机名提示消失。

**背后发生了什么(这部分不是图形界面操作,是真实读取扩展设置项得到的机制说明):**

VSCode Remote-SSH 的架构,不是简单地"通过 SSH 把远程文件内容拉过来显示"——真正在做事的是一个安装在**远程主机**上的组件("VS Code Server"),本地这边的 VSCode 窗口只负责界面渲染和交互,通过 SSH 连接和远程的这个 Server 组件通信,真正的文件读写、语言分析、调试器执行,全部发生在远程主机上。这解释了几件事:第一,为什么第一次连接一台新主机需要花时间"安装"——那正是在把这个 Server 组件部署到远程;第二,为什么语言扩展(Pylance)需要**在远程主机上单独装一份**,不是本地装了就自动生效——分析代码这件事发生在远程,分析器自然也要在远程。这次现场读取这个扩展真实注册的设置项,能看到不少支撑这套架构的具体开关,比如 `remote.SSH.serverInstallPath`(自定义 Server 组件装在远程的哪个路径)、`remote.SSH.defaultExtensions`(每次连接新主机自动帮你装好哪些扩展,不用每台机器手动装一遍)、`remote.SSH.useLocalServer`;还有一个特别值得知道的设置 `remote.SSH.reconnectionGraceTime`(官方描述原文:"the grace time in seconds to wait before terminating the remote server when the client disconnects",即客户端断开后,远程这个 Server 组件不会立刻退出,会等一段宽限时间,方便短暂断线后重新连上能无缝恢复)——这个机制和第 7 节要讲的 tmux 保护的不是同一层(这个宽限期保护的是 VSCode 自己的连接状态和未保存的编辑器状态,不是你在集成终端里手动跑起来的训练进程),两者不冲突,该用 tmux 保护真正的训练任务这件事不会因为这个设置而改变。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| 远程主机列表里找不到第 3 节配的别名 | 检查 Remote-SSH 读取配置文件的路径设置(`remote.SSH.configFile`,官方描述"The absolute file path to a custom SSH config file",默认空值表示用标准位置)是不是被改到了别的地方;或者确认第 3 节的配置确实写进了 VSCode 实际在用的那个 `~/.ssh/config`(本文第 0 节强调过的"确认到底改的是哪个文件"这个教训,这里同样适用) |
| 连接一台新主机,卡在"正在安装/初始化"很久 | 首次连接需要往远程主机下载部署 VS Code Server 组件,网速慢或者远程主机访问外网受限(很多校内集群、公司内网机器默认不能访问外网)都会导致这一步很慢甚至失败;网络受限的机器,通常需要额外配置离线安装或者内网镜像,这篇不展开 |
| 打开远程的 `.py` 文件,没有跳转定义/自动补全 | 参照 01 号文件第 5 节的结论——语言智能功能来自 Pylance 这类扩展,而且要**装在远程这一端**,本地装了不算数;VSCode 弹出的"是否要在远程安装此扩展"提示不要一直忽略 |
| 不确定当前改的文件是本地的还是远程的 | 参照本节第 4 步,看窗口左下角"><"区域和标题栏,不要凭感觉猜——改错文件(以为在改远程,其实在改一份本地缓存/不存在的路径)是真实会发生的低级失误 |
| 关闭 VSCode 窗口,以为远程主机上的进程也停了 | Remote-SSH 窗口里的集成终端,和第 2 节讲过的普通 SSH 会话有同样的风险——终端跟着连接消亡,直接在里面跑起来的前台进程可能被杀掉;需要长期跑的训练任务,同样应该用第 7 节的 tmux 保护,不要依赖 VSCode 窗口一直开着 |

## 6. `scp`/`rsync` 文件同步 —— 给文件搬家,两种方式怎么选

**为什么需要这个 / 不会有什么后果:**

有了 VSCode Remote-SSH,直接编辑远程文件这件事解决了,但不是所有场景都适合"直接在远程改"——比如你想把本地整理好的一个数据集打包传上去,或者想把远程训练出来的 checkpoint 文件下载回本地分析,这类"整份文件/目录搬家"的需求,`scp` 和 `rsync` 是标准工具。rhcsa-bash 系列已经讲过两者"一句话"层面的区别(全量复制 vs 差量同步),但那边受限于没有真实目标主机,没有验证过真正的差异有多大——这一节用真实数据把这个差异跑给你看。

**环境要求:**

- 和前几节一样能连接的目标主机(继续用第 3 节的 `dtd03-rocky-test` 别名)。
- `scp` 本机确认可用:`command -v scp` → `/usr/bin/scp`。**`rsync` 如实记录一个真实的环境限制**:`command -v rsync` 在这台 Windows Git Bash 上没有任何输出——**没有安装**。这不是个别现象,Git for Windows 的默认安装确实不带 `rsync`(它不是 OpenSSH 的一部分,是完全独立的另一个工具),如果你自己的 Windows 电脑上也遇到"没有 rsync",不是操作错误,需要额外安装(比如通过 WSL 内部的 Linux 环境使用,或者装 cwRsync 之类的移植版)。这次 `rsync` 部分的验证,受限于此,**改成在 Rocky Linux 内部**(它真实装了 `rsync 3.4.4`)针对它自己完成——用一把专门为这次验证生成、验证完删除的独立密钥,让 Rocky 通过真实的 SSH 传输层回环连接自己,命令语法和"本地推到远程"完全一致,如实标注这一点,不冒充是从 Windows 侧真实发起的。

**一步步跟着做:**

**第一步:`scp` 真实上传一个文件,校验和确认内容完全正确。**

```
$ echo "local test file created ..., for scp/rsync verification" > upload_test.txt
$ echo "line 2: original content" >> upload_test.txt
$ md5sum upload_test.txt
990327774a296426817b214995c2a43c *upload_test.txt

$ scp upload_test.txt dtd03-rocky-test:/root/dtd03-verify/upload_test.txt

$ ssh dtd03-rocky-test "md5sum /root/dtd03-verify/upload_test.txt"
990327774a296426817b214995c2a43c  /root/dtd03-verify/upload_test.txt
```

本地和远程的 md5 校验和完全一致(`990327774a296426817b214995c2a43c`),证明传输过程没有任何字节损坏——`scp` 的用法很直接:`scp 本地路径 别名:远程路径`(第 3 节配好的别名同样能直接用在 `scp` 上,不用重新指定用户名/端口/密钥),反过来从远程下载到本地则是 `scp 别名:远程路径 本地路径`,方向对调而已。

**第二步:体会"全量复制"是什么意思——`scp` 不管文件改了多少,永远整份重传。** 这一点不需要专门验证,是 `scp` 的设计决定的:它没有任何"比较新旧版本、只传差异"的逻辑,每次执行,都是把源文件完整读一遍、完整传一遍、在目标端完整写一遍,哪怕只改了一个字节。文件小的时候感觉不出差别,文件是几个 GB 的 checkpoint、只改了训练配置的一行、你只是想重新传一次的时候,`scp` 会让你重新等一遍完整的传输时间——这正是 `rsync` 要解决的问题。

**第三步:`rsync` 差量同步,真实数据对比。** 在 Rocky 内部生成一个 8MB 的测试文件(用随机数据,保证不可压缩,让"传输了多少字节"这个数字有意义,不会被压缩算法干扰):

```
$ dd if=/dev/urandom of=rsync_src/bigfile.bin bs=1M count=8 status=none
$ ls -la rsync_src/bigfile.bin
-rw-r--r-- 1 root root 8388608 bigfile.bin
```

第一次同步(目标端还没有这个文件,预期是一次完整传输):

```
$ rsync -av --stats -e 'ssh -i ~/.ssh/dtd03_rsync_loopback' rsync_src/bigfile.bin root@localhost:rsync_dst/
Number of files: 1 (reg: 1)
Total file size: 8,388,608 bytes
Literal data: 8,388,608 bytes
Matched data: 0 bytes
Total bytes sent: 8,390,756
```

`Literal data`(真正传输的原始数据)等于文件全部大小,`Matched data`(找到匹配、不需要重传的部分)是 `0`——第一次同步,目标端什么都没有,自然只能整份传,这一步和 `scp` 没有本质区别。

现在只改动文件中间一小块(4KB),文件总大小完全不变:

```
$ dd if=/dev/urandom of=rsync_src/bigfile.bin bs=1k count=4 seek=2000 conv=notrunc status=none
$ md5sum rsync_src/bigfile.bin rsync_dst/bigfile.bin
067d089a4ede6f0436c2bb91596d6472  rsync_src/bigfile.bin
edc0dbe5463be6d196d3cfe0931fd385  rsync_dst/bigfile.bin
```

（改动前后校验和不同,确认修改真实生效,两边此刻内容不一致。）再同步一次:

```
$ rsync -av --stats -e 'ssh -i ~/.ssh/dtd03_rsync_loopback' rsync_src/bigfile.bin root@localhost:rsync_dst/
Number of files: 1 (reg: 1)
Total file size: 8,388,608 bytes
Literal data: 5,792 bytes
Matched data: 8,382,816 bytes
Total bytes sent: 17,480
```

**这就是差量同步的真实威力**:文件总共 8,388,608 字节,这次真正通过网络传输的 `Literal data` 只有 5,792 字节(只比实际改动的 4096 字节多一点,多出来的部分是 rsync 算法本身按块比对产生的边界开销),`Matched data` 高达 8,382,816 字节——这部分数据 rsync 判断"目标端已经有完全一样的内容",直接复用,根本不传。`Total bytes sent`(这次同步总共通过网络发出的字节,含 rsync 自己的协议开销)是 `17,480`,只占文件总大小的 **0.2% 左右**。同步完成后校验和重新一致:

```
$ md5sum rsync_src/bigfile.bin rsync_dst/bigfile.bin
067d089a4ede6f0436c2bb91596d6472  rsync_src/bigfile.bin
067d089a4ede6f0436c2bb91596d6472  rsync_dst/bigfile.bin
```

想象一下真实场景:一个几十 GB 的 checkpoint,你只是调了一行训练配置重新跑了几步、想把最新的传回本地做个检查——`scp` 会让你重新等一遍几十 GB 的传输时间,`rsync` 大概率几秒钟就搞定(前提是文件内容大部分没变,checkpoint 场景通常确实是这样,模型权重的变动集中在被更新的那部分参数上)。

**背后发生了什么:**

`rsync` 能做到"只传差异",核心是一个叫**滚动校验和(rolling checksum)**的算法:它把文件切成一个个小块,对目标端已有的文件计算每一块的校验和,源端拿着这份校验和列表,用一种能够高效处理"哪怕数据整体往前/往后挪了几个字节,也能重新对齐找到匹配块"的滑动窗口算法,逐段判断源文件的哪些部分能在目标端的旧版本里找到完全一样的内容——找得到的部分,只需要告诉目标端"复用你已有的第几块"(几个字节的指令,不是完整数据),真正需要传输的,只有那些在旧版本里根本找不到匹配、只能算作全新数据的部分(也就是上面看到的 `Literal data`)。这一整套比对和传输逻辑,同样是通过 SSH 建立的加密通道完成的(`-e ssh` 这个参数就是显式声明"用 ssh 作为底层传输方式"),所以 `rsync` 同时具备"和 scp 一样加密安全"以及"比 scp 省流量省时间"两个优点,唯一的代价是双方都需要先算一遍校验和,对于本来就很小的文件,这点计算开销可能比省下来的传输时间还不划算——这也是为什么"小文件用 scp 更省事,大文件/需要反复同步用 rsync 更划算"是两者更贴切的选择标准,不是"rsync 全面优于 scp"这么简单。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| Windows 上找不到 `rsync` 命令 | 参照本节"环境要求"如实记录的情况——Git for Windows 默认不带,需要额外安装或者改用 WSL 内部的 rsync;这不是配置错误 |
| `rsync` 同步一个目录,结尾要不要加 `/` 结果差很多 | `rsync 源目录/ 目标目录/`(源路径带斜杠)是"把源目录**里面的内容**同步进目标目录";`rsync 源目录 目标目录/`(源路径不带斜杠)是"把源目录**这个目录本身**同步进目标目录,变成目标目录下的一个子目录"——这是 `rsync` 最容易踩的一个语法细节,不确定的时候先用 `-n`(dry-run,只打印会做什么,不真的执行)确认一遍 |
| 第一次同步没感觉到 `rsync` 比 `scp` 快 | 符合预期——第一次同步目标端什么都没有,`rsync` 和 `scp` 一样只能整份传,差异只在"后续重复同步已经大部分相同的内容"时才会体现出来 |
| 传输中断,文件传了一半 | `scp` 默认没有断点续传能力,中断了通常要重新完整传一次;`rsync` 由于会重新比对两边内容,即使中断过,重新执行同一条命令,已经传过的部分大概率会被识别成"已匹配",不会整个重来(但这不是专门的"断点续传"功能,是差量比对的自然结果) |
| 忘记给 `rsync` 加 `-e ssh` 或者等价配置 | 不同发行版/不同安装方式的 `rsync` 默认传输方式可能不同(有的会尝试用不加密的 `rsh` 或者需要单独跑一个 rsync 守护进程),科研场景里几乎总是应该显式用 `-e ssh`(或者依赖大多数现代发行版把 SSH 设成默认值),确保走的是加密传输,不要想当然 |

**自测清单:**

- [ ] 能说出 `scp` 和 `rsync` 最核心的行为差异(全量复制 vs 差量同步),并且能举一个"哪种场景该用哪个"的具体例子
- [ ] 见过真实的 `rsync --stats` 输出,能看懂 `Literal data`/`Matched data`/`Total bytes sent` 各自的含义
- [ ] 知道 `rsync` 源路径结尾加不加 `/` 的行为差异,遇到不确定的情况会先用 `-n` 试跑确认
- [ ] 理解 `rsync` 之所以能省流量,靠的是滚动校验和比对已有内容,而不是什么"压缩得更好"
- [ ] 记得这次 Windows 客户端没有 `rsync` 这个真实环境限制,知道该怎么应对(装,或者改用 WSL 内部环境)

## 7. tmux + SSH 组合收尾 —— 断开重连,训练还在跑

**为什么需要这个 / 不会有什么后果:**

[02-linux-daily-workflow.md 第 3 节](02-linux-daily-workflow.md) 已经完整讲过 `tmux` 本身的原理:SSH 连接断开,内核会给这次连接开出来的终端上的前台进程发 `SIGHUP`,默认处理方式是终止——所以直接在 SSH 终端里跑起来的训练任务,笔记本合盖、网络抖动断线、SSH 客户端意外关闭,都可能让它陪葬,而且大概率没有保存 checkpoint,几个小时的 GPU 时间白跑。`tmux` 的解决办法是让任务跑在一个不依附于任何一次具体连接的常驻会话里,连接断了,会话和里面的任务不受影响。

这一节不重新证明这个原理(链接到 02 号文件),而是把这套机制放回**这篇文章的主题——SSH 本身**的场景里,走一遍研究者真实会经历的完整闭环:**通过 SSH 连上远程服务器 → 在 tmux 里启动一个长时间任务 → 断开这次 SSH 连接(不是在 tmux 里手动 detach,是这次连接本身彻底结束,模拟真实的断网/合盖)→ 过一段时间,用一条全新的、和上一次毫无关系的 SSH 连接重新连上去 → 确认任务全须全尾地跑完了**。这是全文最重要的一节,因为它是前面所有节省下来的时间和减少的麻烦,最终服务的那个目标:**保证你的实验不会因为一次连接问题就白跑**。

**环境要求:**

- 和前几节一样能连接的目标主机,`tmux` 已确认装好(02 号文件已经验证过这台 Rocky Linux 上 `tmux`/`screen` 都有)。
- 复用第 3 节配置好的 `dtd03-rocky-test` 别名,不需要额外配置。

**一步步跟着做:**

**第一步(连接 1):建 tmux 会话,塞进一个模拟训练的循环,看一眼刚开始的进度,然后让这次连接彻底结束。**

```
$ ssh dtd03-rocky-test "
    tmux new-session -d -s dtd03train
    tmux send-keys -t dtd03train 'for i in 1 2 3 4 5 6; do echo \"step \$i/6 \$(date +%H:%M:%S)\" >> ~/dtd03-verify/logs/train_progress.txt; sleep 2; done; echo TRAIN_DONE > ~/dtd03-verify/logs/train_done.txt' Enter
    sleep 2
    tmux list-sessions
    cat ~/dtd03-verify/logs/train_progress.txt
    cat ~/dtd03-verify/logs/train_done.txt
  "
dtd03train: 1 windows (created Thu Jul 23 08:42:32 2026)
step 1/6 08:42:32
cat: ~/dtd03-verify/logs/train_done.txt: No such file or directory
```

这条命令本身跑完,SSH 连接就自然结束了(和你手动敲了几条命令后关掉终端窗口是同一回事)——此刻这个"假装训练"的循环总共只跑完 1/6,`train_done.txt` 这个完成标记文件确认还不存在,连接到这里彻底断开。

**第二步:完全不连接任何东西,真实等待一段时间(比 6 步循环需要的总时长更长)。**

```
$ sleep 11
```

这 11 秒里,没有任何 SSH 连接、任何终端,和这台远程机器保持着联系——如果训练任务是直接跑在裸终端里的,这个时间点它应该早就随着上一步连接结束被杀掉了。

**第三步(连接 2):一条全新的、和连接 1 毫无关系的 SSH 连接,检查真相。**

```
$ ssh dtd03-rocky-test "
    tmux list-sessions
    cat ~/dtd03-verify/logs/train_progress.txt
    cat ~/dtd03-verify/logs/train_done.txt
    ps -ef | grep -E 'tmux|dtd03train' | grep -v grep
  "
dtd03train: 1 windows (created Thu Jul 23 08:42:32 2026)
step 1/6 08:42:32
step 2/6 08:42:34
step 3/6 08:42:36
step 4/6 08:42:38
step 5/6 08:42:40
step 6/6 08:42:42
TRAIN_DONE
root  1766  1  0 08:42 ?  00:00:00 tmux new-session -d -s dtd03train
```

**这就是结论的真实证据**:`dtd03train` 这个会话,在两次完全独立的连接之间**依然存在**;进度日志显示全部 6 步都真实跑完了(时间戳从 `08:42:32` 到 `08:42:42`,严格按每步 2 秒的节奏推进);`train_done.txt` 完成标记确认存在,内容是 `TRAIN_DONE`;`ps -ef` 显示这个 tmux 会话进程的父进程 ID 是 `1`(也就是 `init`)——它不挂在任何一次登录会话下面,连接 1 结束之后、连接 2 建立之前那段完全没有人连着的空档期里,这个"训练循环"自己安安静静地跑完了全部 6 步。

**第四步:真实的日常用法——`attach` 接入看实时进度,这一步和 02 号文件一样,是真实的按键交互,没法在这里自动化验证。** 连接 2 里如果不想只是用脚本查日志,而是想像"打开一个窗口"一样看到 tmux 会话里实时发生的一切:`ssh -t dtd03-rocky-test "tmux attach -t dtd03train"`——`-t` 参数在这里是必须的(强制分配一个伪终端,不加的话 `tmux attach` 这种需要交互式终端的命令会出问题,这一点和普通跑一条命令不太一样,SSH 默认在你显式提供了一整条远程命令时不会自动分配伪终端)。接入之后,该会话里发生的一切(包括已经滚动过去的历史输出,取决于 tmux 的回滚缓冲区设置)都会显示出来;`Ctrl+B` 然后 `D` 分离,回到你自己的终端,会话继续跑,和 02 号文件讲过的完全一致。

**第五步:收尾——不需要的会话记得清理。**

```
$ ssh dtd03-rocky-test "tmux kill-session -t dtd03train"
```

**背后发生了什么:**

这一节没有引入 02 号文件没讲过的新机制——tmux 会话为什么能在连接之间存活,02 号文件"背后发生了什么"那部分已经讲透(server/client 分离架构,SIGHUP 传不到 tmux 管理的进程上)。这一节真正做的,是把"连接"这个变量从"同一个终端里手动 detach"换成了"两次物理上独立、中间完全没有任何连接存在的真实 SSH 会话"——更贴近你真实会遇到的场景:你不是在电脑前手动按 `Ctrl+B D` 主动分离,而是笔记本没电关机、宿舍网络突然断了、出门在外换了个网络环境——这些情况下,你根本来不及"手动 detach",tmux 会话是被动地因为连接消失而失去了当前这个 client,而不是你主动操作的结果。这次验证特意设计成"连接 1 直接跑完退出、不做任何手动 detach 动作",就是为了让这个对照尽量接近真实的意外断线,而不是一次刻意配合的操作演示。

**如实记录这次验证时间线上的一个插曲**:采集完上面全部证据之后,在做清理工作时,发现 WSL2 虚拟机又一次自动重启了(`tmux list-sessions` 一度报 `no server running`,随后 `wsl -l -v` 确认这台机器的状态经历了重新拉起)——这和 02 号文件记录的"WSL2 虚拟机闲置一段时间会自动休眠/重启"是同一个特性,这次即使专门起了一个后台进程试图保持虚拟机不休眠,依然没能完全避免。这个现象出现在证据**已经完整采集之后**,不影响上面的验证结论,但值得再次强调:**这是 WSL2 作为轻量虚拟机的自身特性,真实的远程 GPU 服务器不会因为你没连着就自动关机重启**——如果你自己在用 WSL2 练习这套操作时也撞到会话突然消失,先怀疑是不是这个原因,不代表 tmux 或者你的操作有问题。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| 重新连接后,`tmux list-sessions` 显示 `no server running` | 两种可能:要么是真的还没建过任何会话;要么像本节如实记录的那样,底层虚拟机/服务器本身重启过——前者用 `tmux new-session` 建一个,后者需要确认任务是不是也跟着没了(真实远程服务器不会无故重启,如果发生了,通常意味着更严重的问题,比如服务器真的宕机维护,这时候任务大概率也没了,和 tmux 本身无关) |
| `ssh 别名 "tmux attach -t 会话名"` 报错或者显示异常 | 大概率漏了 `-t` 参数强制分配伪终端;确认会话名拼写正确(`tmux list-sessions` 先看一眼真实存在哪些会话) |
| 分不清这一节和 02 号文件第 3 节的区别,内容看起来很像 | 02 号文件讲的是"tmux 本身为什么能扛住断线"这个原理,验证手段也用到了 SSH,但重点在 tmux;这一节是这篇 SSH 专题的收尾,重点是"这一整套 SSH 日常操作——密钥、别名、端口转发——最终要为一件事服务:你能安心断开连接,不用守着终端不敢走开",两者不是重复,是同一个机制在两篇不同重点的文章里各自被引用了一次 |
| 忘记给不用的 tmux 会话做清理,攒了一堆 | 和 02 号文件提的坑一样,`tmux list-sessions` 定期看一眼,`tmux kill-session -t 名字` 清掉不需要的,尤其是这类"故意模拟断线"的验证性会话,用完随手清理是好习惯 |
