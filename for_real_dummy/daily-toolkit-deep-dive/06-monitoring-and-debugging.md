## 1. `nvidia-smi` 日常操作 —— 利用率 / 显存 / 温度,一眼看出训练是不是卡住了

**为什么需要这个 / 不会有什么后果:**

你现在手头就有一块真实的 GPU(这台机器的 RTX 3080 Ti Laptop GPU),以后大概率还要连到实验室或者云上的远程 GPU 服务器。不管是本机还是远程,"训练脚本跑起来之后,GPU 到底在不在干活"这件事,肉眼是看不出来的——屏幕上除了 log 一行行往下滚,没有任何直接证据告诉你"GPU 真的在算,还是其实卡在某个地方空转"。

不会看 `nvidia-smi` 会有什么具体后果?举两个真实会撞上的场景:
- 训练脚本跑起来了,log 也在正常打印,但其实是数据加载卡住了,GPU 全程 0% 利用率——纯看 log 输出看不出这一点(该打印的东西一样打印,只是每一步间隔变得异常长),你可能要过了很久才后知后觉发现"这几个小时 GPU 根本没干活",白白浪费机时。
- 显存被之前一次没有正常退出的训练进程占着没释放,新开一次训练直接报 `CUDA out of memory`,但你不知道去哪查是谁占着——`nvidia-smi` 的 Processes 表就是用来回答这个问题的。

`nvidia-smi`(NVIDIA System Management Interface)是 NVIDIA 驱动自带的命令行工具,只要装好了官方驱动就有,不需要额外安装。

**环境要求:**

- 已经装好 NVIDIA 官方驱动(不是随便什么显卡驱动,集成显卡/非 NVIDIA 显卡没有这个工具)。
- 检查命令:`nvidia-smi --version`。本机真实输出:
  ```
  NVIDIA-SMI version  : 595.97
  NVML version        : 595.97
  DRIVER version      : 595.97
  CUDA Version        : 13.2
  ```
  三个版本号一起打出来,说明驱动装好了,`nvidia-smi` 可用。
- 本机真实确认这块 GPU:`nvidia-smi -L`
  ```
  GPU 0: NVIDIA GeForce RTX 3080 Ti Laptop GPU (UUID: GPU-e7007fb5-ab7f-4348-73de-5a621ebf861b)
  ```

**一步步跟着做:**

1. **最基础的用法:不加任何参数,直接敲 `nvidia-smi`。** 本机真实输出(GPU 当时完全空闲,没有任何进程在用):
   ```
   Thu Jul 23 08:26:26 2026
   +-----------------------------------------------------------------------------------------+
   | NVIDIA-SMI 595.97                 Driver Version: 595.97         CUDA Version: 13.2     |
   +-----------------------------------------+------------------------+----------------------+
   | GPU  Name                  Driver-Model | Bus-Id          Disp.A | Volatile Uncorr. ECC |
   | Fan  Temp   Perf          Pwr:Usage/Cap |           Memory-Usage | GPU-Util  Compute M. |
   |                                         |                        |               MIG M. |
   |=========================================+========================+======================|
   |   0  NVIDIA GeForce RTX 3080 ...  WDDM  |   00000000:01:00.0 Off |                  N/A |
   | N/A   72C    P0            751W /   80W |       0MiB /  16384MiB |      0%      Default |
   |                                         |                        |                  N/A |
   +-----------------------------------------+------------------------+----------------------+

   +-----------------------------------------------------------------------------------------+
   | Processes:                                                                              |
   |  GPU   GI   CI              PID   Type   Process name                        GPU Memory |
   |        ID   ID                                                               Usage      |
   |=========================================================================================|
   |  No running processes found                                                             |
   +-----------------------------------------------------------------------------------------+
   ```

   逐列说明你日常会用到的这几个:
   - **Driver-Model**:`WDDM`。Windows 上有 `WDDM`(Windows Display Driver Model,普通桌面/笔记本显卡默认模式,显卡同时管显示输出)和 `TCC`(Tesla Compute Cluster,专门给纯计算用、不接显示器的显卡准备的模式,常见于服务器)两种,Linux 服务器上的显卡驱动是另一套(没有 WDDM/TCC 的区分)。这台笔记本是 `WDDM`——第 2 步你会看到这个模式下 Processes 表里显存这一列有个已知的真实限制。
   - **Fan / Temp / Perf**:风扇转速(这台笔记本显示 `N/A`,笔记本的风扇由笔记本自己的散热系统控制,不通过这个接口暴露)、温度(`72C`,这是 GPU 完全空闲时的真实读数——笔记本 GPU 散热空间小,空载温度比桌面/服务器 GPU 通常看到的数字要高,不代表有问题)、性能状态(`P0` 是最高性能状态,数字越大性能状态越低;这里空闲时仍然显示 `P0` 是这块笔记本 GPU 真实观察到的行为)。
   - **Pwr:Usage/Cap**:当前功耗 / 功耗上限。如实说明:本机这一行读出来的 `751W / 80W` 数字本身不太合理(用电量远超上限,物理上说不通),大概率是这块笔记本 GPU 在 WDDM 模式下功耗上报方式本身有些古怪,不是操作错误,也不是本文重点,不深究——这也提醒你,不是 `nvidia-smi` 打印出来的每一个数字都必须严丝合缝地"讲得通",遇到明显不合理的读数,先怀疑是不是这块具体硬件/驱动组合的怪癖,而不是怀疑自己操作错了。
   - **Memory-Usage**:显存占用,`已用/总量`,这里是 `0MiB / 16384MiB`(16GB 显存,当前完全空闲)。**这是判断"显存是不是快满了"最直接看的一个数字。**
   - **GPU-Util**:GPU 利用率,当前 `0%`。**这是判断"训练是不是卡住了"最直接看的一个数字**——但"背后发生了什么"一节会讲这个数字容易被误解的地方。
   - **Compute M.**:计算模式,`Default` 表示多个进程可以同时往这块 GPU 上提交任务(还有 `Exclusive`/`Prohibited` 等模式,日常科研机器上几乎都是 `Default`,了解有这个字段即可)。
   - **MIG M.**:是否开启了 MIG(Multi-Instance GPU,把一块物理 GPU 切成几个独立小 GPU 的技术)。这里 `N/A` 表示不支持——消费级显卡(包括这块 RTX 3080 Ti)不支持 MIG,这是数据中心卡(比如 A100/H100)才有的功能,提一句避免你以后在别的机器上看到这一行有值时一头雾水。
   - **Processes 表**:当前所有正在用这块 GPU 的进程,这里显示 `No running processes found`,和上面 `0MiB` 互相印证——没人在用。

2. **让 GPU 真的有负载,再看一遍——不能对着一块空闲 GPU 讲操作,也是"怎么看哪个进程占显存"的真实演示。** 用 [`_assets/06-gpu-load-demo.py`](_assets/06-gpu-load-demo.py)(反复做 4096×4096 矩阵乘法,吃 GPU 算力,默认跑够时间自动退出,不需要手动 `Ctrl+C`):
   ```
   $ .venv/Scripts/python.exe for_real_dummy/daily-toolkit-deep-dive/_assets/06-gpu-load-demo.py 30 > gpu_load.log 2>&1 &
   ```
   3 秒后再跑一次 `nvidia-smi`,真实输出:
   ```
   Thu Jul 23 08:26:29 2026
   ...
   |   0  NVIDIA GeForce RTX 3080 ...  WDDM  |   00000000:01:00.0 Off |                  N/A |
   | N/A   71C    P0             33W /   80W |     457MiB /  16384MiB |      0%      Default |
   ...
   +-----------------------------------------------------------------------------------------+
   | Processes:                                                                              |
   |  GPU   GI   CI              PID   Type   Process name                        GPU Memory |
   |        ID   ID                                                               Usage      |
   |=========================================================================================|
   |    0   N/A  N/A           29548      C   ..._qbz5n2kfra8p0\python3.13.exe      N/A      |
   +-----------------------------------------------------------------------------------------+
   ```
   两处变化:
   - **`Memory-Usage` 从 `0MiB` 变成 `457MiB`**——显存确实被占用了。
   - **Processes 表多了一行**,`PID 29548`,`Type` 是 `C`(Compute,计算任务;还有 `G` 表示图形任务,`C+G` 两者都有),`Process name` 是这个进程的可执行文件路径。**这一行的 PID 就是"占显存的到底是哪个进程"这个问题的答案**——GPU 显存告急时,先看这张表,再拿 PID 去 `ps aux`(第 2 节)查这个进程具体是什么。
   - **如实记录一个真实限制**:`GPU Memory Usage` 这一列这里显示的是 `N/A`,不是具体数字——这不是 bug,是 **WDDM 驱动模式下的已知限制**:Windows WDDM 模式下 `nvidia-smi` 经常没法精确汇报单个进程占了多少显存,只能给出 GPU 整体的 `Memory-Usage`。真 Linux 服务器上这一列通常会显示具体的 MiB 数字。这台机器上想知道"大概占了多少",看顶部整体的 `Memory-Usage`(`457MiB`),结合 Processes 表只列出了这一个进程,反推它大概占了这么多。
   - 这一次的 `GPU-Util` 还是 `0%`——"背后发生了什么"一节会解释这不代表没在工作。

3. **持续监控:`-l`(loop),以及一个本机真实撞到的参数误区。** 网上不少教程会写 `nvidia-smi -l 1 -c 5`,意思是"每 1 秒刷新一次,只刷 5 次就自动退出"——但这是**过时或者版本不对的写法**。本机现场核实这个驱动版本(595.97)的 `nvidia-smi -h`:
   ```
   -l,   --loop=               Probe until Ctrl+C at specified second interval.
   ...
   -c,   --compute-mode=       Set MODE for compute applications:
                                0/DEFAULT, 1/EXCLUSIVE_THREAD (DEPRECATED),
   ```
   `-c` 在这个版本里其实是 `--compute-mode`(设置计算模式的,完全是另一件事),不是"刷新次数"。`-l`(`--loop`)本身的说明写得很直白:**"Probe until Ctrl+C"——不给它设一个外部的时间限制,它会一直刷,只能手动按 `Ctrl+C` 停**。凡是要写进脚本自动跑、或者像本文这样要在有限时间内跑完做演示,直接用 `nvidia-smi -l 1` 会导致命令永远不退出。正确做法是外部套一层 `timeout`(GNU coreutils 自带,大多数 Linux/Git Bash 都有):
   ```
   $ timeout 6 nvidia-smi -l 1
   ```
   意思是"最多跑 6 秒,时间到了 `timeout` 自己发信号让 `nvidia-smi` 退出"。真实跑出来的前后两帧(GPU 负载脚本还在后台跑着):
   ```
   Thu Jul 23 08:26:29 2026
   ...
   | N/A   75C    P0             67W /   80W |     457MiB /  16384MiB |     99%      Default |
   ... [Processes 表同上,PID 29548]

   Thu Jul 23 08:26:34 2026
   ...
   | N/A   78C    P0             79W /   80W |     457MiB /  16384MiB |     98%      Default |
   ... [Processes 表同上]
   ```
   `GPU-Util` 这次稳定在 `98%~99%`,和第 2 步刚启动时看到的 `0%` 形成对比("背后发生了什么"一节会解释这个落差)。温度也从 `71C` 一路涨到 `78C`,是矩阵乘法持续吃算力的真实反映。

4. **`--query-gpu` + CSV 格式:适合写进脚本、存成文件、画图的输出方式。** 上面的表格格式给人看很直观,但不好用脚本解析。需要把 GPU 状态记下来(比如训练脚本旁边配一个监控脚本,每隔几秒把状态追加写进一个 CSV 文件,训练完之后画一张利用率曲线)时,用 `--query-gpu` 指定要哪几列,`--format=csv` 输出成逗号分隔:
   ```
   $ timeout 4 nvidia-smi --query-gpu=timestamp,utilization.gpu,memory.used,memory.total,temperature.gpu --format=csv -l 1
   timestamp, utilization.gpu [%], memory.used [MiB], memory.total [MiB], temperature.gpu
   2026/07/23 08:26:35.740, 99 %, 457 MiB, 16384 MiB, 78
   2026/07/23 08:26:36.747, 98 %, 457 MiB, 16384 MiB, 78
   2026/07/23 08:26:37.760, 99 %, 457 MiB, 16384 MiB, 79
   2026/07/23 08:26:38.770, 99 %, 457 MiB, 16384 MiB, 79
   ```
   同样套了 `timeout` 避免无限跑。这条命令产出的每一行都是干净的 CSV,直接 `>> gpu_usage.csv` 追加到文件里,训练结束后用 pandas/Excel 打开画图,比自己解析表格线框的输出省事得多。`--query-gpu=help` 能列出这个驱动版本支持查询的全部字段(温度、功耗、时钟频率、PCIe 带宽等几十项),这里只挑了最常用的四项。

5. **精确定位"到底是哪个进程在用 GPU",比翻系统进程列表更安全的做法:`--query-compute-apps`。** 第 2 步 Processes 表已经能看到 PID,但如果要写一个脚本"自动找到所有在用 GPU 的进程",更精确、更不容易误伤无关进程的做法是直接问 `nvidia-smi` 本身,而不是去操作系统层面按名字/命令行内容模糊搜索(下面"常见坑"讲一次本文撰写过程中真实因为后一种做法翻车的教训):
   ```
   $ nvidia-smi --query-compute-apps=pid,process_name,used_memory --format=csv
   pid, process_name, used_gpu_memory [MiB]
   ```
   （这次 GPU 已经空闲,所以只有表头、没有数据行——这本身也是一种确认"当前没有任何进程占着 GPU"的方式。)如果这时候 GPU 上正跑着东西,这里会精确列出 PID、进程名、显存占用,只包含**真的在用这块 GPU** 的进程,不会有任何误伤风险。

6. **确认训练真的结束了、显存真的释放了。** 杀掉第 2 步启动的负载进程,等几秒,再看一次:
   ```
   $ nvidia-smi
   ...
   | N/A   74C    P0             34W /   80W |       0MiB /  16384MiB |      0%      Default |
   ...
   |  No running processes found                                                             |
   ```
   `Memory-Usage` 精确地掉回了 `0MiB`,Processes 表也重新变回空的——这是"进程真的退出、显存真的还给系统了"的确认方式。如果杀掉进程之后显存数字长时间不掉,说明有别的东西还占着,需要再排查。

**背后发生了什么:**

`nvidia-smi` 本身不直接读硬件,它是 NVML(NVIDIA Management Library)这个 C 库的一个命令行包装——驱动常驻在后台,持续从 GPU 的硬件计数器里轮询温度、功耗、利用率这些指标,`nvidia-smi`/NVML 只是把驱动已经采集好的数字读出来展示,这也是为什么这个命令本身跑起来几乎不消耗额外资源、不会干扰你正在跑的训练。

有一点容易被误解、值得单独说清楚:**`GPU-Util` 这个百分比,衡量的是"时间占比",不是"算力占比"**。它的准确含义接近"过去一个采样周期里,GPU 上有没有 kernel 在跑"的时间比例——哪怕你的 kernel 写得很差、只用到了这块 GPU 一小部分的计算单元(SM),只要它一直在跑、没有空闲,`GPU-Util` 照样能显示接近 100%。换句话说,`GPU-Util` 99% 完全可能同时伴随着"这块 GPU 的算力实际只用上了三成"这种情况——**这一列数字只能回答"卡住了没有"(0% 附近且持续不变,大概率是卡住了或者在等 CPU 端的数据),回答不了"我的 kernel 效率高不高"这个更深的问题**。真的要深挖后面这个问题(SM 到底占用了多少、有没有被寄存器/共享内存卡住),需要看 SM occupancy 这一层,这正是 kernel-gpu-deep-dive 系列 [01-gpu-hardware-and-memory.md](../kernel-gpu-deep-dive/01-gpu-hardware-and-memory.md) 第 4 节的内容,这篇不重复。

第 2 步里 `GPU-Util` 一开始读到 `0%`、紧接着第 3 步就变成 `99%`,是真实抓到的现象,原因也和"时间占比"这个本质有关:GPU-Util 是"上一个采样周期"的回顾性统计,矩阵乘法之间有极小的间隙,或者 Python/CUDA 刚初始化、第一个 kernel 还没真正提交,偶尔一次查询就会撞上这个间隙,不代表进程没在工作——这时候更早、更稳定的信号其实是 `Memory-Usage` 已经涨到了 `457MiB`,显存占用比 `GPU-Util` 更早、更持续地反映出"这个进程已经在用这块 GPU 了"。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| 抄网上教程写 `nvidia-smi -l 1 -c 5`,结果 `-c` 好像没起到"刷 5 次就停"的作用 | 本文第 3 步已经现场核实:至少在本机这个驱动版本(595.97)上,`-c` 是 `--compute-mode`(设置计算模式),不是"刷新次数"。不要凭记忆/旧教程假设某个参数的含义,先用 `nvidia-smi -h` 看这台机器这个版本真实支持什么;需要限制运行时长,用外层 `timeout` 更保险,不依赖某个具体版本才有的参数 |
| `nvidia-smi -l 1` 敲下去,终端"卡住不动了" | 没卡住,是它在按文档说的"Probe until Ctrl+C"持续刷新——`Ctrl+C` 手动停,或者一开始就套 `timeout` |
| Processes 表里 `GPU Memory Usage` 一列显示 `N/A`,查不到这个进程具体占了多少显存 | 本文第 2 步已确认:这是 Windows **WDDM 驱动模式**下的已知限制,不是操作错误。看顶部整体的 `Memory-Usage` 数字反推;真 Linux/TCC 环境这一列通常会给出具体数字 |
| GPU 正在跑重负载,`GPU-Util` 却显示 `0%` 或者数值一直在跳 | 两种可能都要考虑:① 单次查询恰好撞上两次 kernel 之间的间隙(配合 `Memory-Usage` 一起看,或者连续 `-l` 观察几秒钟的趋势,不要只看一次快照);② 真的是训练卡在 CPU 端(数据加载/预处理),GPU 确实没活干——第 2 节会讲怎么用 `ps aux` 一起排查这种情况 |
| 想写一个脚本"自动找到所有在用 GPU 的进程再处理",直接用操作系统级别的命令行内容模糊搜索(比如 Windows 上 `wmic process where "CommandLine like '%关键词%'"`) | **本文撰写过程中真实撞到的教训**:在 Git Bash 里给 `wmic` 的 `LIKE` 查询做双引号/百分号转义,很容易因为多层转义出岔子,搜出来的结果比预期宽泛得多(实测一次错误查询返回了 8 个毫不相关的 PID,而不是预期的 1 个),如果不假思索直接对这些 PID 批量 `taskkill //F` 后果可能是误杀无关进程。更安全的做法是用 `nvidia-smi` 自己的 `--query-compute-apps` 精确拿到"真正占着这块 GPU"的 PID 列表(见第 5 步),而不是依赖操作系统层面的模糊字符串匹配 |

## 2. `ps`/`htop` —— 监控 CPU、内存和自己的训练进程

**为什么需要这个 / 不会有什么后果:**

`nvidia-smi` 只回答"GPU 这一侧"的问题。但很多"训练是不是卡住了"的真实根因出在 CPU 这一侧——数据加载、预处理、tokenize 这些步骤跑在 CPU 上,如果这部分变成瓶颈,你会看到 GPU-Util 很低,但这不代表 GPU 坏了或者代码有 bug,而是 CPU 端在拖后腿。`ps`(process status)和 `htop` 就是用来看"CPU、内存这一侧,我的训练进程到底是什么状态"的工具。

不会用的具体后果:进程占用内存一直涨(常见于数据没有正确释放、缓存越攒越多),不盯着看的话,直到系统内存耗尽被强制杀掉之前,你可能完全没意识到有问题;进程明明"看起来在跑"(log 还在滚),但其实已经卡死在某个系统调用上,`ps` 的 `STAT` 列能帮你分辨这两种情况。

**环境要求:**

- 这台机器是 Windows,`ps`/`htop` 严格来说是 Linux/Unix 世界的工具——**不能想当然假设它们在你的终端里等价于 Linux 原生版本**,下面第 1 步会现场验证这一点,不是凭印象跳过。
- 复用 [02-linux-daily-workflow.md](02-linux-daily-workflow.md) 已经装好、已经验证过的 WSL2 **Rocky Linux** 环境。本次现场额外确认:`htop 3.3.0`、`ps from procps-ng 4.0.4`、`python3 3.12.13`,通过 `/mnt/e/...` 路径能直接访问这个仓库的文件。

**一步步跟着做:**

1. **先在这台机器的 Git Bash 里直接试一下 `ps aux`,不要想当然。** 真实敲出来:
   ```
   $ ps aux
         PID    PPID    PGID     WINPID   TTY         UID    STIME COMMAND
        1839    1836    1836      34288  ?         197611 08:22:03 /usr/bin/bash
        1838    1836    1836      57732  ?         197611 08:22:03 /usr/bin/ps
        1836       1    1836      25136  ?         197611 08:22:03 /usr/bin/bash
   $ ps --version
   ps (cygwin) 3.3.6
   ```
   两个真实问题一起暴露出来:第一,**这个 `ps` 是 Cygwin 移植版**,不是 Linux 原生 `ps`——列名完全不一样(没有 `%CPU`/`%MEM`/`RSS` 这些最常用的列,多出 `WINPID`/`PGID` 这种 Windows 特有的东西);第二,**它只看得到当前这个 Git Bash/MSYS 会话内部的进程**,系统里其他真实在跑的 Windows 进程(包括你可能刚启动的训练脚本,如果不是从这个 Git Bash 窗口启动的)根本不出现在列表里。`aux` 这三个参数本身(`a`=所有用户的进程,`u`=user-oriented 格式带 %CPU/%MEM,`x`=没有控制终端的进程也显示)是 Linux `ps` 的经典组合,Cygwin 这个移植版形式上认识这几个字母、不报错,但给出的东西和真正的 Linux `ps aux` 是两回事。**结论:这台 Windows 机器上,想要"教科书标准"的 `ps aux`/`htop` 体验,得去一个真正的 Linux 环境——下面切到 WSL2。**

2. **切到 WSL2 Rocky Linux,启动一个真实在吃 CPU 的"训练进程"。** 用 [`_assets/06-fake-training-loop.py`](_assets/06-fake-training-loop.py)(纯 CPU 空转,默认跑够时间自动退出,不需要手动 `Ctrl+C`,专门为这一节准备的):
   ```
   $ wsl -d RockyLinux
   [进入 WSL2]
   $ nohup python3 /mnt/e/Workspace/dummy/for_real_dummy/daily-toolkit-deep-dive/_assets/06-fake-training-loop.py 20 > train.log 2>&1 &
   disown
   ```
   （`nohup ... &` 加 `disown` 是 02 号文件第 4 节讲过的组合,这里直接用,不重新展开语法。)

3. **`ps aux | grep python`,真正的 Linux `ps aux` 长什么样。** 等 2 秒钟让它跑起来,真实输出:
   ```
   $ ps aux | head -1
   USER         PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
   $ ps aux | grep python | grep -v grep
   root         190  0.0  0.1 131232 45372 ?        Ssl  08:20   0:00 /usr/bin/python3 -sP /usr/sbin/firewalld --nofork --nopid
   root        1487  0.0  0.0  26836 19724 pts/2    S+   08:30   0:00 python3 -m http.server 8899 --bind 127.0.0.1
   root        1518 98.5  0.0  12920  8680 pts/4    R+   08:34   0:01 python3 /mnt/e/Workspace/dummy/for_real_dummy/daily-toolkit-deep-dive/_assets/06-fake-training-loop.py 20
   ```
   （最后 `grep -v grep` 是很常用的小技巧——`ps aux | grep 关键词` 这条命令里 `grep` 自己的参数也带着这个关键词,会把自己也匹配进去,`| grep -v grep` 把这一行反向排除掉。)

   **一个真实、值得记住的现象**:这次 `grep python` 一次性匹配到了**三个**互不相关的进程——系统自带的 `firewalld`(它内部用 Python 实现)、一个之前别的操作留下的 `http.server`、以及我们真正要找的 `06-fake-training-loop.py`。**这说明"关键词能匹配到"不等于"这就是你要找的那个"**,真正定位到自己的训练进程,还是要看 `COMMAND` 这一列的完整内容(脚本路径、参数)对不对得上。

   逐列说明:
   - **USER**:进程属于哪个用户(这里都是 `root`,因为 WSL2 这个环境整体以 root 会话运行)。
   - **PID**:进程号,后面 kill 一个进程、或者去 `/proc/PID/` 下面查东西,用的都是这个号。
   - **%CPU**:这一列是重点——目标进程显示 `98.5`,证实它确实在拼命占用 CPU(纯 CPU 空转的脚本,符合预期);另外两个系统进程是 `0.0`,说明它们目前空闲。**这就是"我的训练脚本是不是真的在跑,还是卡住了"这个问题在 CPU 这一侧的答案**——一个应该在计算的进程,`%CPU` 长期是 `0.0`,大概率是卡住了。
   - **%MEM**:占用物理内存的百分比。**长时间盯着一个训练进程,如果这个数字持续、单调上涨不回落,是内存泄漏的信号**。
   - **VSZ / RSS**:两个都是内存大小(单位 KB),但含义不同——`VSZ`(Virtual Size)是这个进程**申请**了多少虚拟地址空间,`RSS`(Resident Set Size)是**真正**占用了多少物理内存。`VSZ` 通常比 `RSS` 大很多(很正常);真正该盯的是 `RSS`(或者上面的 `%MEM`,两者说的是同一件事)。
   - **TTY**:这个进程关联的终端,`?` 表示没有控制终端(常见于后台/守护进程)。
   - **STAT**:进程状态,排查"卡住了没有"很有用。这里 `R+` 表示"Running(正在被调度到 CPU 上执行)+ 前台进程组";常见的其他状态还有 `S`(Sleeping,在等某个事件,正常空闲,不是卡死)、`D`(不可中断的睡眠,通常在等磁盘 I/O,长时间停留在这个状态可能真的有问题)、`Z`(Zombie,进程已经结束但父进程还没回收它)。
   - **START / TIME**:进程启动的时间点,和它**累计**用掉的 CPU 时间(不是挂钟时间)。
   - **COMMAND**:完整命令行,最终确认"这一行是不是我要找的进程",要落到这一列。

4. **`htop`——同样的信息,可视化、实时刷新的版本。** `ps aux` 是"拍一张快照",`htop` 是一个持续刷新的全屏仪表盘,人坐在终端前盯着看的时候比反复敲 `ps aux` 舒服很多。真实版本:`htop -V` → `htop 3.3.0`。正常交互式启动就是敲 `htop` 回车,占满整个终端,`q` 退出。

   **诚实说明**:`htop` 是一个全屏的 ncurses 程序,靠不断在终端里移动光标位置重绘画面来实现"实时刷新",这决定了它的原始输出**没法像 `ps aux` 那样干净地重定向到一个文件里**——你会得到一堆终端控制字符,而不是一段可读文本(这一点和 01 号文件里"图形界面点击本身没法自动化验证"是同一类限制,只是这次连"截图"这个退路都没有,因为这是纯文本终端环境)。下面这段是**真实**启动 `htop`(用它自带的 `-n 1` 参数,表示"只刷新 1 帧就自动退出",配合 `-C` 禁用颜色,拿到一次确定性的快照),再把裹在文字外面的终端控制字符去掉之后的结果——数值本身是真的,排版是为了能贴进这篇文档手动整理过的:
   ```
   CPU 核心表(19 个逻辑核心,大部分 0.0%):0[0.0%] 3[0.0%] 6[0.0%] ... 19[100.]

   Mem[||#@$ 1.22G/23.3G]   Tasks: 101, 30 thr, 0 kthr; 0 running   Swp[0K/6.00G]
   Load average: 0.74 0.27 0.09   Uptime: 00:05:57

      PID USER PRI  NI  VIRT   RES   SHR S  CPU%-MEM%   TIME+  Command
     1314 root  20   0 12920  8644  5496 R 166.7  0.0  0:02.07 python3 /mnt/...06-fake-training-loop.py
        1 root  20   0 22948 14252 10316 S   0.0  0.1  0:00.74 /sbin/init
        2 root  20   0  3180  2208  2072 S   0.0  0.0  0:00.01 /init
        ...
   ```
   排在第一行的正是目标进程,`CPU%` 显示 `166.7`——**超过 100% 是正常的**,`htop` 的 `CPU%` 是相对**单个核心**的 100% 封顶,一个进程如果调度上跨用了不止一个核心的时间片,数字就会超过 100(这台机器是多核,`htop` 采样的具体时刻也会有一些波动,不用纠结每一位数字)。

   `htop` 界面从上到下依次是:CPU 每个核心各自的利用率(比 `nvidia-smi` 那种单一汇总数字更细,能看出是不是只有一个核心在忙、其他核心闲着)、内存/交换分区占用条、任务总数、系统负载(Load average,过去 1/5/15 分钟的平均负载)、开机时长,下半部分是和 `ps aux` 信息基本对应的进程表,只是加上了排序(默认按 `%CPU` 降序)、颜色高亮、以及可以直接用键盘操作的交互能力。

5. **只看自己关心的进程:`-p` 参数和 `F4` 过滤。** 进程表几十上百行,不想每次都在里面找,`htop` 支持:
   - 启动时用 `-p PID1,PID2,...` 只显示指定的几个 PID(比如已经知道自己训练脚本的 PID,`htop -p 1314` 就只跟踪这一个)。
   - 交互式界面里按 `F4`,输入一个关键词,实时按名字过滤——这是本机现场核实过、`htop --help` 里真实列出的功能(`-F --filter=FILTER  Show only the commands matching the given filter`),图形交互部分本身沿用上面第 4 步的诚实说明(没法自动化验证按键这个动作本身,但确认这个功能真实存在、参数真实生效)。

6. **组合诊断:GPU-Util 低 + CPU 占用高,大概率是数据流水线瓶颈,不是训练卡死。** 把第 1 节和这一节串起来用:如果 `nvidia-smi` 显示 `GPU-Util` 持续接近 0%,同时 `ps aux`/`htop` 显示你的训练进程(或者它 fork 出来的 DataLoader worker 子进程)`%CPU` 很高、状态是 `R`(Running),这组合起来通常意味着——GPU 一直在"等米下锅",瓶颈出在 CPU 端的数据加载/预处理/tokenize 太慢,不是训练"卡死"了(卡死的话对应进程的 `%CPU` 应该接近 0,状态可能停在 `D` 或者长期 `S` 不动)。这个组合诊断思路,比单独看某一个工具的输出更可靠。

**背后发生了什么:**

`ps`/`htop` 显示的所有信息,最终来源都是 Linux 内核暴露的 `/proc` 这个虚拟文件系统——每一个进程在 `/proc/PID/` 下面都有一份实时更新的状态文件,`%CPU`/`%MEM`/`RSS` 这些数字都是从这些文件里读出来、按公式算出来的,不是某种"猜测",这也是为什么这些数字能做到实时、精确。`htop` 相比 `ps` 多做的事,是**周期性地重新读一遍 `/proc`、计算差值、重绘屏幕**,本质信息来源完全一样,`htop` 只是体验上更友好、多了排序/过滤/键盘交互这些便利。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| 在这台 Windows 机器的 Git Bash 里敲 `ps aux`,结果和网上教程截图完全对不上,没有 `%CPU`/`%MEM` 列 | 本文第 1 步已确认:Git Bash 的 `ps` 是 Cygwin 移植版,不是 Linux 原生 `ps`,只能看到当前会话内的进程,列也不一样——需要标准 `ps aux` 体验,去 WSL2/真实 Linux 服务器 |
| `ps aux \| grep 关键词` 匹配到好几个不相关的进程 | 本文第 3 步真实撞到的现象——关键词只要是子串就会匹配,不能假设"匹配到的就是我要找的",必须看 `COMMAND` 列的完整内容确认 |
| `htop` 敲下去屏幕花掉、或者提示打不开终端 | 大概率是这次调用的标准输出没有连着一个真实终端(比如被重定向进了文件或者管道)——`htop` 是需要真实终端的全屏程序,和第 3 节要讲的 `tail -f`(纯文本输出,可以随便重定向)是完全不同的两类工具,不要混着用 |
| `htop` 里某个进程 `CPU%` 显示超过 100 | 不是 bug,`htop` 的 `CPU%` 以单核 100% 为封顶单位,进程用了不止一个核心的时间片就会超过 100 |
| 盯着一个训练进程的 `%MEM`/`RSS`,数字一直涨不回落 | 内存泄漏的典型信号(常见于 DataLoader 没有正确释放、日志/中间结果缓存越攒越多),需要去代码里排查,但这是 `ps`/`htop` 能帮你**发现**问题的地方 |

## 3. 训练 log 实时追踪 —— `tail -f` 配合 `grep` 过滤

**为什么需要这个 / 不会有什么后果:**

02 号文件第 5 节已经完整讲过 `grep`/`awk` 处理训练 log 的语法组合拳,包括 `tail -f` 配合 `--line-buffered` 的写法和原理,这里不重复语法。但那篇文章受限于篇幅,只是**描述**了"`tail -f` 能实时看到新写入的内容"这个结论,没有真正给你看一次"追着一个正在被写入的文件,你会亲眼看到什么"——这一节要补的就是这个:一次真实的、有时间线的、"某一行是在写入的那一刻才出现"的完整演示。

不会用 `tail -f` 的具体后果:训练还在跑,你想看最新进度,只能反复手动重新打开这个 log 文件;更实际的问题是,**你没法一边跑着训练一边"守着"某个特定的异常信号**(比如"一旦出现 loss=nan 立刻通知我"),只能等训练结束或者定期手动检查,可能错过第一时间发现问题的机会。

**环境要求:**

- 任意 bash + GNU `tail`/`grep`(Git Bash 自带,现场确认版本:`tail (GNU coreutils) 8.32`、`grep (GNU grep) 3.0`)。
- 需要 `timeout` 命令来保证下面的实时演示不会挂住不退出(Git Bash 自带:`timeout (GNU coreutils) 8.32`)。**这一点在这一节格外重要**——`tail -f` 这个命令本身的设计就是"一直不退出,持续等新内容",如果直接裸跑它,会导致脚本永远卡在这一步,必须像上一节的 `nvidia-smi -l 1` 一样,外面套一层 `timeout`。

**一步步跟着做:**

1. **准备一个会被持续写入的"训练 log"。** 用一个简单的后台循环模拟(每秒写一行,一共写 8 行,第 4 行故意模拟一次梯度异常警告,写完自动退出,不需要手动干预):
   ```bash
   $ ( ( for i in 1 2 3 4 5 6 7 8; do
           ts=$(date +%H:%M:%S)
           if [ "$i" -eq 4 ]; then
             echo "$ts [WARNING] gradient norm exceeded threshold: 12.4 > 10.0, clipping"
           else
             echo "$ts [INFO] step=$((i*847)) loss=2.5 lr=0.0010 throughput=140.2tok/s"
           fi
           sleep 1
         done ) > live_train.log & )
   ```

2. **裸 `tail -f`,限时观察,亲眼看着行数增加。** 等 1 秒钟让文件先被创建出来,然后:
   ```bash
   $ timeout 9 tail -f live_train.log
   ```
   （效果就是每隔 1 秒钟多冒出一行,9 秒之后 `timeout` 自动让它退出,不需要 `Ctrl+C`。)

3. **配合 `grep --line-buffered` 过滤,这才是日常真正会用的写法——只看你关心的异常,不看正常的进度行刷屏。** 这是本节的核心演示,真实、完整跑出来的效果:
   ```bash
   $ timeout 9 tail -f live_train.log | grep --line-buffered -E "WARNING|ERROR"
   08:41:57 [WARNING] gradient norm exceeded threshold: 12.4 > 10.0, clipping
   ```
   **这一行输出,是在第 4 秒左右、`[WARNING]` 那一行被真正写进文件的那一刻才打印出来的**——前 3 秒钟,终端里什么都不显示(因为前 3 行都是 `[INFO]`,被 `grep` 过滤掉了),不是命令卡住或者没生效,是真的"没有你关心的东西"。事后用 `cat` 核对完整的原始 log,确认这次一共写了 8 行,`grep` 精确地只放行了这一行 `[WARNING]`:
   ```
   $ cat live_train.log
   08:41:54 [INFO] step=847 loss=2.5 lr=0.0010 throughput=140.2tok/s
   08:41:55 [INFO] step=1694 loss=2.5 lr=0.0010 throughput=140.2tok/s
   08:41:56 [INFO] step=2541 loss=2.5 lr=0.0010 throughput=140.2tok/s
   08:41:57 [WARNING] gradient norm exceeded threshold: 12.4 > 10.0, clipping
   08:41:58 [INFO] step=4235 loss=2.5 lr=0.0010 throughput=140.2tok/s
   08:41:59 [INFO] step=5082 loss=2.5 lr=0.0010 throughput=140.2tok/s
   08:42:00 [INFO] step=5929 loss=2.5 lr=0.0010 throughput=140.2tok/s
   08:42:01 [INFO] step=6776 loss=2.5 lr=0.0010 throughput=140.2tok/s
   ```
   这就是"实时追踪 + 过滤"真正的使用体验:开一个终端标签(或者一个 tmux 面板,见第 5 节),挂上这条 `tail -f xxx.log | grep --line-buffered -E "关键词"` 命令,让它一直开着,你去做别的事,一旦训练 log 里出现你关心的关键词,这个终端立刻会冒出一行——不用人工盯着刷屏的正常输出。

4. **如果不确定该抓哪些关键词,先看真实项目里常见的几类信号。** 结合前面几篇和这篇的例子,值得用 `tail -f | grep` 常驻盯着的典型模式:`ERROR|WARNING`(框架自己标出来的问题)、`loss=nan|loss=inf`(02 号文件第 5 节强调过,很多框架的 `nan` 不会主动升级日志级别,必须显式抓这个字符串,不能只依赖级别标签)、`CUDA out of memory`(显存溢出,配合第 1 节的 `nvidia-smi` 一起排查)、`Traceback`(Python 报错栈的起始行,一旦看到基本意味着进程要挂了)。

**背后发生了什么:**

`tail -f` 的原理是持续监视这个文件的大小变化(轮询文件的元数据,或者在支持的系统上用 `inotify` 这类文件系统事件通知机制),一旦发现文件变长了,就把新增的那部分内容读出来打印,如此循环,直到收到终止信号(这里是 `timeout` 发的)。这也是为什么它能实时响应"文件被追加内容"这个事件。

`grep --line-buffered` 这个参数在 02 号文件里提过一句,这里再强调一次为什么它对这个场景是必需的:`grep` 默认会判断自己的输出对象是不是一个真实终端——如果是,它会立刻把匹配到的每一行输出;但如果输出被接进了一个管道,`grep` 出于效率考虑,默认会**攒一批数据再一次性写出去**。对着一个静态文件跑,这个差异感觉不到;但对着 `tail -f` 这种"细水长流"的场景,不加 `--line-buffered` 就意味着"明明已经匹配到了,但因为攒批的缓冲区还没攒够,迟迟不显示在你眼前"——这条命令的价值就是"第一时间看到",攒批攒没了这个价值。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| 直接裸敲 `tail -f xxx.log`,忘了加 `timeout` 或者别的限制手段,命令"卡住不动" | `tail -f` 设计上就是不会自动退出的——这是正常行为,不是 bug。交互式使用时故意让它一直开着(比如单独占一个 tmux 面板)完全没问题;但凡是写进自动化脚本、或者要在有限时间内跑完的场合,必须套 `timeout` |
| `tail -f \| grep 关键词`,程序明明在正常写日志,过滤后的终端却半天没反应 | 忘了加 `--line-buffered`,`grep` 在管道里默认攒批输出,不是没匹配到,是匹配到了但还没显示出来 |
| 只监控 `ERROR`/`WARNING` 这两个关键词,结果真正的问题(比如 `loss=nan`)完全没被发现 | 02 号文件第 5 节已经用真实例子证明过:很多训练框架的 `nan` 不会主动把日志级别升到 `WARNING`/`ERROR`。**过滤关键词要包含你真正关心的具体异常值本身**,不能只依赖级别标签 |
| 想同时监控好几个不同的信号(比如 `nan`、`OOM`、`Traceback`),重复开好几个 `tail -f` 太麻烦 | 用 `-E` 加 `\|` 在一条 `grep` 里组合多个模式(本节示例的 `"WARNING\|ERROR"` 就是这么写的),不需要为每个关键词单独开一条命令 |

- [ ] 能独立写出一条 `timeout N tail -f 文件 | grep --line-buffered -E "模式1|模式2"` 命令,并解释每个部分的作用

## 4. `pdb` —— 没有图形界面时,命令行怎么调试

**为什么需要这个 / 不会有什么后果:**

[01-vscode-editor-workflow.md](01-vscode-editor-workflow.md) 第 4 节已经教过 VSCode 的图形化调试器——打断点、看 Variables 面板、Step Over/Into/Out。**这一节讲的是同一件事的另一种界面**:当你 SSH 到一台远程 GPU 服务器、只有一个终端窗口、没有(或者不方便配置)图形界面连接时,VSCode 那一整套图形化操作全部用不了,但调试这件事本身——让程序在某一行真正停下来,亲眼看当时每个变量的值——同样需要,而且科研日常里"只能连终端"的场景只多不少。`pdb`(Python DeBugger)是 Python 标准库自带的命令行调试器,不需要额外安装,任何装了 Python 的地方都能用。

不会用命令行调试的具体后果:退化回"到处加 `print`"这一种手段——01 号文件已经论证过,这种方法对付"能跑、不报错,但结果就是不对"这类 bug 效率很低。在一台只有终端的远程服务器上,这个效率差距会更明显。

**环境要求:**

- 任意装了 Python 3.7+ 的环境(`breakpoint()` 这个内建函数从 3.7 开始才有;更早的版本要手写 `import pdb; pdb.set_trace()`,效果一样)。本机用仓库 `.venv` 的 `python 3.13.9` 验证。
- 本节要用的练习脚本:[`_assets/06-pdb-demo.py`](_assets/06-pdb-demo.py),和 01 号文件的 `01-debug-demo.py` 是**同一类问题的命令行版本**——一个滑动平均窗口函数 `running_average`,`for` 循环从 `1` 开始而不是 `0`,永远漏掉窗口内第一个数字,但除数还是按完整窗口大小除的,所以算出来的平均值会比真实值偏低,而且不会报错、不会崩溃。核心逻辑:
  ```python
  def running_average(history: list[float], window: int) -> float:
      """计算 history 最近 window 个数字的平均值(模拟 loss 的滑动平均)。"""
      recent = history[-window:]
      total = 0.0
      for i in range(1, len(recent)):  # bug: should start at 0, this skips recent[0]
          total += recent[i]
      return total / window
  ```
  在 `main()` 里,训练跑到第 4 步时会命中一个 `breakpoint()`(第 41 行),从这里开始用 `pdb` 一步步走。
- **一个真实撞到、已经修复的坑,提前说明**:这个脚本原本在 `breakpoint()` 那一行和 bug 所在的 `for` 循环那一行各带一句中文行内注释。真实用 `pdb` 的 `l`(list)命令查看源码时发现,这两行注释在这台机器的默认编码设置下会显示成乱码(不影响断点/变量这些核心调试功能,纯粹是那一行注释文字本身显示不出来)——已经把这两行注释改成了纯 ASCII 内容。下面"常见坑"会再具体说一次这个现象。

**一步步跟着做:**

1. **不设断点,先跑一遍,建立"预期 vs 实际"的落差**(和 01 号文件的调试流程完全一致,这是固定起点)。
   ```
   $ .venv/Scripts/python.exe .../06-pdb-demo.py
   step=1 loss=0.9000 avg3=0.0000
   step=2 loss=0.7000 avg3=0.2333
   step=3 loss=0.5000 avg3=0.4000
   step=4 loss=0.4000 avg3=0.3000
   step=5 loss=0.3500 avg3=0.2500
   step=6 loss=0.3000 avg3=0.2167
   ```
   手算第 4 步的 `avg3`(最近 3 步 `[0.70, 0.50, 0.40]` 的平均值):`(0.70+0.50+0.40)/3 = 0.5333`,脚本算出来的是 `0.3000`——确认这里有 bug。

2. **正常运行,程序会在第 4 步命中 `breakpoint()`,自动进入 `pdb` 交互环境。** 下面每一条 `(Pdb)` 开头的输出都是本机真实敲出来的。先用 `l`(list)看一眼当前停在哪:
   ```
   $ .venv/Scripts/python.exe .../06-pdb-demo.py
   step=1 loss=0.9000 avg3=0.0000
   step=2 loss=0.7000 avg3=0.2333
   step=3 loss=0.5000 avg3=0.4000
   > ...06-pdb-demo.py(41)main()
   -> breakpoint()  # pdb starts here (see doc section 4)
   (Pdb) l
     36    def main() -> None:
     37        losses = [0.90, 0.70, 0.50, 0.40, 0.35, 0.30]
     38        for step, loss in enumerate(losses, start=1):
     39            history = losses[:step]
     40            if step == 4:
     41    ->          breakpoint()  # pdb starts here (see doc section 4)
     42            log_step(step, history)
     43
     44
     45    if __name__ == "__main__":
     46        main()
   ```
   `l` 列出当前行前后各 5 行源码,`->` 标出来的就是当前停留的位置。

3. **`p`(print)打印变量,确认当前上下文。**
   ```
   (Pdb) p step
   4
   (Pdb) p history
   [0.9, 0.7, 0.5, 0.4]
   ```

4. **`s`(step into)一路走进 `running_average` 内部。** `s` 和 VSCode 的 Step Into 是同一个概念:下一行是函数调用的话,`s` 会真的跳进这个函数内部,不像 `n`(next)会把整个函数调用当"黑盒"一次执行完。连续 `s`,从 `main` 走进 `log_step`,再走进 `running_average`:
   ```
   (Pdb) s
   > ...06-pdb-demo.py(42)main()
   -> log_step(step, history)
   (Pdb) s
   --Call--
   > ...06-pdb-demo.py(30)log_step()
   -> def log_step(step: int, loss_history: list[float]) -> None:
   (Pdb) n
   > ...06-pdb-demo.py(32)log_step()
   -> avg = running_average(loss_history, window=3)
   (Pdb) s
   --Call--
   > ...06-pdb-demo.py(21)running_average()
   -> def running_average(history: list[float], window: int) -> float:
   ```
   `--Call--` 是 `pdb` 明确告诉你"刚刚发生了一次函数调用、现在停在被调用函数的入口"的提示。

5. **`n`(next)在 `running_average` 内部单步,配合 `p` 亲眼验证 bug 现场。**
   ```
   (Pdb) n
   > ...06-pdb-demo.py(23)running_average()
   -> recent = history[-window:]
   (Pdb) p recent
   *** NameError: name 'recent' is not defined
   ```
   **这个报错本身就是一次有价值的教学**:`pdb` 显示的"当前行"是**即将执行、还没执行**的那一行——这时候 `recent` 确实还不存在,`p` 报 `NameError` 完全正常。再 `n` 一步,让这一行真正执行完:
   ```
   (Pdb) n
   > ...06-pdb-demo.py(24)running_average()
   -> total = 0.0
   (Pdb) p recent
   [0.7, 0.5, 0.4]
   ```
   现在 `recent` 已经被赋值,确认是 `[0.7, 0.5, 0.4]`——数据本身完全正常,问题还没出现。继续单步走进 `for` 循环:
   ```
   (Pdb) n
   > ...06-pdb-demo.py(25)running_average()
   -> for i in range(1, len(recent)):  # bug: should start at 0, this skips recent[0]
   (Pdb) p i
   *** NameError: name 'i' is not defined
   (Pdb) n
   > ...06-pdb-demo.py(26)running_average()
   -> total += recent[i]
   (Pdb) p i
   1
   (Pdb) p total
   0.0
   ```
   **这就是 bug 现场**:循环第一次执行,`i` 已经是 `1`,不是 `0`——`recent[0]`(也就是 `0.7`)整个循环从头到尾都不会被碰到。继续走完这个循环:
   ```
   (Pdb) n
   > ...06-pdb-demo.py(25)running_average()
   -> for i in range(1, len(recent)):  # bug: should start at 0, this skips recent[0]
   (Pdb) p total
   0.5
   (Pdb) n
   > ...06-pdb-demo.py(26)running_average()
   -> total += recent[i]
   (Pdb) p i
   2
   (Pdb) n
   > ...06-pdb-demo.py(25)running_average()
   -> for i in range(1, len(recent)):  # bug: should start at 0, this skips recent[0]
   (Pdb) p total
   0.9
   ```
   `total` 最终定格在 `0.9`(只累加了 `recent[1]=0.5` 和 `recent[2]=0.4`,`recent[0]=0.7` 全程没有参与),而不是正确的 `0.7+0.5+0.4=1.6`。

6. **`c`(continue)让程序跑完剩下的部分,验证这个 bug 确实是这次错误结果的根因。**
   ```
   (Pdb) n
   > ...06-pdb-demo.py(27)running_average()
   -> return total / window
   (Pdb) n
   > ...06-pdb-demo.py(33)log_step()
   -> print(f"step={step} loss={loss_history[-1]:.4f} avg3={avg:.4f}")
   (Pdb) p avg
   0.3
   (Pdb) c
   step=4 loss=0.4000 avg3=0.3000
   step=5 loss=0.3500 avg3=0.2500
   step=6 loss=0.3000 avg3=0.2167
   ```
   `avg` 确实是 `0.3`(`0.9 / 3`),和第 1 步跑出来的 `avg3=0.3000` 完全对得上——**用 `pdb` 亲眼确认了这次结果错误,根因就是 `running_average` 里那一行 `range(1, ...)` 的索引错误**,不是猜的。`c` 之后程序不会再命中新的断点,剩下的第 5、6 步正常跑完退出。

**背后发生了什么:**

`breakpoint()` 是 Python 3.7 引入的内建函数,默认行为是调用 `sys.breakpointhook()`,这个 hook 默认指向 `pdb.set_trace()`——效果就是"程序执行到这一行,把控制权交给一个 `pdb.Pdb` 实例"。`pdb` 本身构建在标准库的 `bdb`(基础调试框架)和 `cmd`(命令行交互框架)之上:`bdb` 利用 Python 解释器自带的逐行跟踪机制(`sys.settrace`,和 01 号文件提到的 `debugpy` 底层依托的机制是同一套),在每一行代码即将执行前把控制权交还给调试器;`cmd` 负责解析 `l`/`p`/`s`/`n`/`c` 这些命令,分发到对应的处理函数。这也是为什么本文能用 `printf '命令1\n命令2\n...' | python 脚本.py` 这种方式**非交互式地**驱动整个调试过程——`pdb` 归根结底就是一个从标准输入一行行读命令的循环,不关心这些命令是人在键盘上敲出来的,还是从一个管道里流进来的。

`s`(step into)和 `n`(next)的区别,和 01 号文件 VSCode 调试器里 Step Into / Step Over 完全对应。VSCode 的图形化调试器本质上就是通过 `debugpy` 这个适配层,把同一套 `sys.settrace` 机制包装成了带断点红点、Variables 面板的图形界面——**命令行 `pdb` 和图形化调试器,是同一个底层能力的两种呈现方式,不是两套不同的技术**。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| 用 `l` 查看源码时,某一行的中文注释显示成一堆乱码 | **本文撰写过程中真实撞到**:在这台 Windows 机器的具体编码环境下,`pdb`/`linecache` 重新读取源文件、显示给你看的时候,用来解码文件字节的编码和文件实际保存用的 UTF-8 对不上,中文注释就会花掉——不影响断点、单步、变量打印这些核心功能,只是那一行注释文字本身显示不出来。断点附近的行内注释尽量保持纯 ASCII 是最省心的绕开方式(本节用的 `06-pdb-demo.py` 已经这么改过) |
| `p 变量名` 报 `NameError`,明明代码里下一行就要用到这个变量 | 本文第 5 步已经现场证明过:`pdb` 显示的"当前行"是**即将执行、还没执行**的那一行,这一行如果正是给这个变量赋值的那一行,查询它当然还不存在——这是预期行为,`n` 一步让这行真正执行完,变量就出现了 |
| 用 `printf '命令序列' \| python 脚本.py` 这种管道方式驱动 `pdb`,命令数量没对齐,后面全部错位/提前 `continue` | 这种非交互方式下,`pdb` 拿到的就是一整条命令流,没有任何容错——**每一次 `s`/`n` 具体会停在哪一行,必须实际跑一遍确认,不能凭空在脑子里模拟**(本文撰写过程中真实第一次就算错了步数,见第 0 节)。想复现一段调试过程,最可靠的做法是先真的跑一遍观察输出,再据此调整命令序列 |
| 没有图形界面的服务器上,想用 Tab 补全变量名/命令历史,发现 `pdb` 好像不太顺手 | 这是管道/非交互场景的真实限制——真正坐在一台远程服务器终端前交互式使用 `pdb` 时,Tab 补全、上下箭头翻历史这些和普通 shell 类似的编辑体验是有的,只是没法在非交互管道里演示出来 |
| 忘了当前停留在哪一层调用栈,`p` 出来的变量值和预期对不上 | 用 `l` 随时确认当前位置(顶部会显示 `函数名()`),或者用 `w`(where)打印完整调用栈——不同栈帧里同名变量互不相干,这一点和 01 号文件 VSCode 调试器 Variables 面板"离开当前函数变量就消失"是同一件事 |

## 5. 远程训练任务断线重连 —— 监控工具在这个流程里怎么用

**为什么需要这个 / 不会有什么后果:**

这是这篇文章最后一节,把前面 4 节讲过的工具(`nvidia-smi`/`ps`·`htop`/`tail -f`)和 02 号文件已经讲过的 `tmux`、03 号文件将会讲的 SSH,全部串成一个你以后会反复用到的完整流程:**SSH 到远程 GPU 服务器,在 tmux 里启动一个长时间训练任务,断开连接(合笔记本盖/网络抖动/主动下班),过一段时间重新连上——重新连上之后,怎么用监控工具确认"训练到底还活不活着、跑到哪了、有没有出问题"。**

不搞清楚这个流程会有什么后果:02 号文件第 3 节已经用真实实验证明过,`tmux` 能保证任务在断线期间继续跑——但"任务在继续跑"和"你知道它现在具体是什么状态"是两回事。重新连上之后如果不知道该按什么顺序检查,容易出现两种典型的低效场景:要么心里没底,反复 `attach` 进去用眼睛盯着屏幕从头看到尾;要么盲目相信"没报错就是没事",错过了训练早就在几小时前静默卡死或者显存溢出的信号。

**环境要求:**

- 无新增环境要求——这一节完全复用前面 4 节 + 02 号文件已经验证过的工具和结论,不引入任何新工具。
- 需要你已经看过 02 号文件第 3 节(tmux 的 `new-session`/`attach`/`detach`)。SSH 连接本身怎么建立、`~/.ssh/config` 怎么配置多台服务器,以 03 号文件(进度见 [00-roadmap.md](00-roadmap.md))为准,这里不重复。

**一步步跟着做(一份你可以直接照着走的检查流程):**

把整个"断线重连"流程拆成断线前、重连后两段,每一步写清楚"用哪个命令、期待看到什么、如果不对说明什么"——这份清单本身不需要重新跑一遍真实的远程服务器(没有真实可连接的远程 GPU 服务器这条验证边界,03 号文件已经说明过),但每一个环节用到的具体命令,前面几节和 02 号文件都已经真实验证过,这里只是把它们按正确的顺序串起来。

**断线前(开始训练的时候就该做的准备,不要等断线了才后悔):**

1. **确认自己在 tmux 会话里,不是在裸终端上直接跑训练**(02 号文件第 3 节)。习惯性地先 `tmux new-session -s train`(或者 `tmux attach -t train` 接回一个已经存在的会话),训练命令一定敲在这个会话里面,不是 SSH 登录之后的裸终端。
2. **训练命令本身,把标准输出重定向进一个你知道路径的 log 文件**,比如 `python train.py 2>&1 | tee train.log`(`tee` 能做到"屏幕上照常看得见输出,同时又存了一份文件",比只输出到屏幕、断线后没有历史记录可查更稳妥)。这个 log 文件就是第 3 节 `tail -f`/`grep` 要用的对象。
3. `detach`(`Ctrl+B` 再按 `D`)离开这个 tmux 会话,可以安心断开 SSH 连接了——02 号文件第 3 节已经用真实的两次独立 SSH 连接证明过,这一步之后训练不会因为连接断开而被杀掉。

**重连后(不管是主动重新连接,还是网络抖动之后自动恢复):**

4. **重新 SSH 连上服务器**(03 号文件详细讲怎么配置连接,这里假设你已经连上了)。

5. **`tmux list-sessions`,确认会话还在。** 如果这条命令报"找不到会话",说明服务器本身可能重启过、或者会话被什么原因清理掉了——这是最先要排除的可能性,后面几步都建立在"会话还在"这个前提上。

6. **在 `attach` 进去、亲眼看一大堆滚动输出之前,先用不需要真正接入会话的方式做一次快速体检——这正是这篇文章前 4 节的价值所在:**
   - `nvidia-smi`(第 1 节):GPU 现在还有负载吗?`GPU-Util` 是不是接近 0(可能已经跑完了,或者卡住了)?`Memory-Usage` 正常吗?
   - `ps aux | grep python`(第 2 节):训练进程的 PID 还在列表里吗?`%CPU` 是不是接近 0(如果训练该算的时候 `%CPU` 却是 0,大概率卡住了)?进程启动时间(`START` 列)距现在过去了多久,和你预期的训练总时长对不对得上?
   - `tail -n 50 train.log`(先看最后 50 行,不是从头 `cat` 整个文件):最新的几行 log 停留在哪一步、什么时间戳?这个时间戳和"现在"差多久——如果最后一行日志的时间戳是几个小时前,但进程还显示在跑,这组合起来是"卡住了但进程没死"的强烈信号。

7. **确认训练看起来健康之后,再用 `timeout N tail -f train.log | grep --line-buffered -E "WARNING|ERROR|loss=nan"`(第 3 节)挂一会,确认没有正在发生的异常。** 这一步不是必须的(如果第 6 步已经看到明显问题,应该直接进 tmux 排查),但如果一切正常、只是想"多确认几秒钟",这是比 `attach` 进去干等着更轻量的方式。

8. **真的需要动手排查(改代码、手动重启、或者训练已经在 `input()`/断点等着交互)的时候,才 `tmux attach -t train` 真正接入会话。** 前面几步的价值就在于:大部分"随手看一眼"的场景根本不需要走到这一步,只有真的发现问题、需要动手干预时才需要接入会话本身。

**背后发生了什么:**

这个顺序不是随便排的,背后有一条明确的"成本从低到高"的逻辑:`nvidia-smi`/`ps aux`/`tail -n 50` 这几个命令都是**一次性的快照**,几乎不消耗时间,而且不会打断/影响正在运行的训练;`tail -f | grep` 稍微重一点,但仍然是被动监听,不需要真的"进入"训练进程所在的终端环境;只有 `tmux attach` 这一步,才是真正把你的终端接进正在运行的会话——这一步本身没有风险(02 号文件已经解释过,`attach`/`detach` 不影响会话里的进程),但如果目的只是"确认还活着",直接跳到这一步、对着可能几千行的滚动输出大眼瞪小眼,是这条流程里最没效率的做法。前面几步用最低的成本,尽可能提前排除"一切正常"的情况,把真正需要人工介入的场景才升级到成本最高的一步。

**常见坑:**

| 现象 | 大概率原因 |
|---|---|
| `tmux list-sessions` 报错"找不到会话"或者提示没有 server 在跑 | 02 号文件第 3 节"常见坑"已经列过:可能服务器本身重启过(会话本身跟着 tmux server 一起消失,tmux 只能保证"这次 SSH 连接断开"不影响任务,保证不了"整台服务器重启"不影响任务);也可能会话名字打错了,先 `tmux list-sessions` 不带 `-t` 参数看看现在到底有哪些会话 |
| `nvidia-smi` 显示 GPU 空闲,但 `tmux attach` 进去一看,训练进程分明还在跑 | 可能训练本身就正常结束了(比如提前触发了 early stopping),GPU 空闲是符合预期的——要结合 log 的最后几行判断训练是"正常跑完"还是"意外卡住" |
| `ps aux` 看到训练进程,`%CPU` 却是 `0.0`,但 `nvidia-smi` 显示 GPU 还有显存占用、利用率也不是 0 | 这其实是**正常**的:训练主进程大部分时间可能在等 GPU 算完一个 batch(这部分等待在 CPU 层面看起来就是空闲/低占用),真正的计算发生在 GPU 上——不要看到 CPU 端 `%CPU` 低就断定"卡住了",要综合 GPU 那一侧的信号一起看 |
| log 文件最后一行的时间戳明明是很久之前,但 `ps`/`nvidia-smi` 都显示进程/GPU 一切正常有负载 | 可能是这一步训练本身单步就是很慢(比如验证集评估、保存一个很大的 checkpoint),log 只在每个 epoch/固定间隔才打印一行,不代表卡住——进程真的在吃 CPU 或者 GPU 有负载,通常就没有真的死掉,只是这一步恰好比较慢 |
| 重新连上之后忘了自己训练脚本 log 文件的路径,翻了半天 | 呼应本节第 2 步的建议:训练一开始就把 log 明确写到一个你记得住、找得到的固定路径,不要依赖"应该在某个地方"的记忆 |
