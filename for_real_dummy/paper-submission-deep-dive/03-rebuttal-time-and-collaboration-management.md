## 1. 为什么 Rebuttal 值得认真管理 —— 真实数据开篇

**为什么需要这个 / 不会有什么后果：**

很多新手对 rebuttal 的心态是"审稿人已经打了分，写不写、怎么写好像影响不大"——这个直觉被真实数据直接推翻。一项覆盖 ICLR 2024-2025 两届、19000+ 篇论文、74000+ 条评审的大规模研究测出：**rebuttal 后分数出现提升的论文，最终录用率是 55.7%-57.6%；分数没有变化的论文，录用率只有 7.8%-12.4%**——差距接近甚至超过 5 倍。同一项研究还发现：rebuttal 后审稿人之间的分歧度平均下降 9%-10%，其中高质量论文（最终定为 Oral/Spotlight 的）分歧下降幅度达到 **26%-48%**，而最终被拒的论文，分歧度几乎没有变化。这两条数据合在一起说明一件事：**rebuttal 不是走过场的礼貌性回应，是真实影响最终结果的关键动作**，尤其对分数处于临界区间（borderline）的论文，rebuttal 质量的杠杆效应非常大。不认真对待 rebuttal 的后果，不是"差不多"，是实打实地把自己的论文摁在那个 7.8%-12.4% 的低录用率区间里。

**环境要求：** 无，这是建立动机的一节。

**一步步跟着做（理解这条数据出现的原因）：**

1. **理解"分数不动"是最危险的信号，不是"安全"的信号。** 很多作者觉得"审稿人没有明确表示不满，说明现在的分数已经不错了，不用太用力写 rebuttal"——但数据显示，分数原地不动的论文录用率反而是最低的一档。原因不难理解：审稿人如果读完 rebuttal 后依然选择不改分数，往往意味着 rebuttal 没有提供审稿人认为"值得改变判断"的新信息，而不是"审稿人已经很满意不需要再改"。
2. **理解"分歧下降"是决定性因素之一。** AC 做最终决定时，审稿人之间意见高度一致（不管一致地正面还是负面）都比意见分裂更容易做出清晰判断；rebuttal 真正的价值之一，就是消解审稿人之间基于不同理解产生的分歧，让 AC 更容易看清这篇论文的真实质量共识在哪里。
3. **理解这条杠杆效应对"borderline"论文最有意义。** 已经明显优秀或者明显有硬伤的论文，rebuttal 能扭转结果的空间本来就小；真正决定"这篇论文能不能被捞起来"的，是那批分数卡在中间地带的论文——如果你的初始分数处于这个区间，rebuttal 投入产出比是全流程里最高的一环，值得认真对待，而不是随便应付。

**背后发生了什么：**

评审文本本身也带有可预测的信号：同一项研究发现，**审稿人评审文字的语气和最终打分之间存在错位**——当评审文字读起来比它给出的分数更正面时（比如文字listed优点很多、批评相对温和，但打分却偏低），这类评审后续分数上调的概率会从基础的 8.3% 一路升到 31.9%。这说明审稿人自己也存在"文字判断"和"数字判断"没完全对齐的情况，rebuttal 如果能精准地在这类"语气比分数更正面"的评审里，抓住审稿人文字中已经流露出的认可点去回应、放大，说服成功率会明显更高——这也是为什么下一节要强调"rebuttal 不是临时抱佛脚，需要先把每条评审的文字和分数对照着读一遍"。

**常见坑：**

| 现象 | 大概率原因 |
|---|---|
| 觉得"反正决定权在 AC，rebuttal 写不写差别不大" | 真实数据显示分数变化和最终录用率强相关（55.7%-57.6% vs 7.8%-12.4%），rebuttal 质量直接影响的是"分数会不会变化"这个中间变量 |
| 只回应了给分最低的审稿人，其他审稿人的意见没花力气回 | 分歧下降是决定性因素之一，只安抚一个人不够，需要让所有审稿人的判断趋向一致 |
| 觉得评审文字写得挺客气，分数打得也不算差，不用太用力 | 恰恰是这种"文字比分数正面"的评审，如果精准回应最容易换来分数上调，见本节"背后发生了什么" |

**自测清单：**

- [ ] 能说出 ICLR 真实的 rebuttal 效果数据（55.7%-57.6% vs 7.8%-12.4%）
- [ ] 能解释为什么"分数原地不动"反而是危险信号而不是安全信号
- [ ] 理解"消解审稿人之间的分歧"是 rebuttal 的一个独立价值，不只是说服单个审稿人改分
- [ ] 知道评审文字语气和打分之间可能存在错位，读评审时要同时关注两者

## 2. Rebuttal 的时间线是什么样的 —— 从评审发布到决定公布

**为什么需要这个 / 不会有什么后果：**

不了解真实的时间线结构，容易犯两类错误：评审一发布就仓促开写，结果漏看了后续还有机会补充说明的窗口；或者拖到讨论期快结束才动笔，没有时间打磨、也没有时间等审稿人二次回应。

**环境要求：** 已经看过 [02 号文件第 4 节](02-double-blind-anonymization-and-submission-systems.md) 的 OpenReview 讨论阶段结构。

**一步步跟着做：**

1. **评审发布 → 公开讨论期开始。** 这一步 02 号文件已经讲过平台机制，这里只强调时间管理的含义：评审一发布，倒计时就开始了，讨论期长度是固定的，不会因为你还没准备好而延长。
2. **不同会议的"轮次"规则不一样，动手前必须先确认当年具体规则。** 以这次调研到的规则为例（**注意年年可能调整，务必以当年官方指引为准**）：ICML 的作者-审稿人讨论分**三轮**——rebuttal、审稿人后续追问、作者再回应，每一轮字数上限约 5000 字符，这意味着你可能不是"写一次就结束"，要为后续可能的追问预留时间和精力；CVPR 风格的规则则更严格，官方明确 rebuttal 只能用于反驳事实性错误或者补充审稿人明确要求的信息，**不能**用来补充审稿人没有要求的新实验/新方法，并且篇幅严格限制在 1 页以内（含参考文献和图表）。
3. **公开讨论期结束 → 私下讨论期。** 这一步作者不再可见，02 号文件已讲过，这里的管理含义是：**公开讨论期结束那一刻，你能做的事就结束了**——之后 AC 和审稿人之间怎么讨论，不再有作者能干预的空间，所以真正要抓紧的是公开讨论期这个窗口，不是"提交后随时都能补救"。
4. **把这条时间线倒推成自己的日历。** 建议的做法：评审发布当天先通读一遍全部评审（不急着动笔），标记每条意见的类型（下一节会给出具体的分类分配方法）；留出至少一半的讨论期时间用于"写完初稿后放一放，回头再改"（这也是第 5 节要讲的"冷静期"纪律，和 04 号文件"deal with reject"的冷静期是同一个原则，只是这里应用在写 rebuttal 之前，不是收到拒稿之后）；预留讨论期最后阶段的时间应对审稿人可能的追问，不要把全部精力压在第一版回应上就以为大功告成。

**背后发生了什么：**

不同会议的讨论轮次设计，反映的是"作者-审稿人互动"应该开放到什么程度这个价值取舍：多轮讨论（如 ICML）给了审稿人和作者更多来回澄清的机会，但也要求作者投入更多精力持续跟进；严格限定单轮 1 页（如 CVPR 风格）逼着作者把回应压缩到最核心的内容，减少审稿人的阅读负担，但也意味着一旦第一次没说清楚，很难有第二次机会。理解你要投的会议属于哪一种设计，直接决定你该把精力怎么在"第一版回应的质量"和"持续跟进的耐力"之间分配。

**常见坑：**

| 现象 | 大概率原因 |
|---|---|
| 只写了一版回应就不再关注讨论区 | 如果目标会议有多轮讨论机制（比如 ICML），审稿人后续追问同样重要，不能"交完作业就走人" |
| Rebuttal 里补充了审稿人没有要求的新实验 | CVPR 风格的会议明确禁止这类内容，即使是善意的补充说明也可能被认为不符合规则，动手前先确认当年规则允许什么 |
| 讨论期最后一天才开始写第一版回应 | 没有给"写完放一放再回头改"留出时间，第一版回应质量直接决定 55.7%-57.6% vs 7.8%-12.4% 这个分野，仓促赶工的风险很高 |

**自测清单：**

- [ ] 能说出公开讨论期和私下讨论期在"作者能做什么"上的本质区别
- [ ] 知道自己要投的会议的讨论机制是单轮还是多轮，字数/篇幅限制具体是多少
- [ ] 能把讨论期的时间倒推成"通读评审→写初稿→冷静期修改→应对追问"这几个阶段各自的时间预算
- [ ] 知道 rebuttal 只能用来回应审稿人的问题，不能借机塞进审稿人没要求的新实验（除非官方规则明确允许）

## 3. Word Limit 怎么分配 —— 一个真实可用的预算分配小工具

**为什么需要这个 / 不会有什么后果：**

真实场景里，一篇论文经常同时收到 3-5 位审稿人的意见，每人提出 2-5 条不等的顾虑，有些顾虑被多位审稿人同时提到，有些顾虑明显比其他的更致命——但你的总字数/篇幅是有限的（不管是 CVPR 风格的 1 页，还是 ICML 风格的 5000 字符/轮）。不少人写 rebuttal 是"想到哪写到哪"，结果对一条无关痛痒的小意见写了大段辩护，对被多位审稿人同时点名的核心质疑反而只用一两句话带过——这不是文字水平问题，是没有在动笔前先做"预算规划"。

**环境要求：** Python 3（标准库即可）；已经把全部审稿意见通读一遍，列出每一条独立的顾虑，标注哪些审稿人提到了、大致有多严重。

**一步步跟着做：**

**第一步：把"多位审稿人共同提到"和"严重程度"两个维度量化成权重。** 直觉上，被 3 个审稿人同时点名的顾虑，应该比只有 1 个审稿人提到的顾虑分配更多篇幅；同样，审稿人明确说"这是我拒稿的主要原因"的顾虑，应该比"建议补充一下"这种措辞更重的顾虑分配更多篇幅。下面这个函数把这两条直觉变成具体数字：

```python
def allocate_rebuttal_budget(concerns, total_budget, min_floor=150):
    """把一份总字数预算，按"共享度 + 严重度"分配给每条 concern。

    concerns: 形如 [{"id": str, "reviewers": set[str], "severity": int(1-5)}, ...]
    total_budget: 总字数预算（比如 CVPR 风格 1 页大约对应的字符数，或者 ICML 单轮 5000 字符）
    min_floor: 每条 concern 的最低保底字数——再小的问题也不能被分配到 0 字，
               至少要有一句明确的回应，不能让审稿人觉得"这条被无视了"

    规则：
      weight = severity * (1 + 0.5 * (共同提出的审稿人数 - 1))
      先给每条 concern 保底 min_floor，剩余预算按 weight 占比分配。
    """
    n = len(concerns)
    floor_total = n * min_floor
    if floor_total > total_budget:
        raise ValueError(
            f"total_budget {total_budget} is too small to cover the floor of {min_floor} "
            f"per concern (need at least {floor_total}) -- merge similar concerns, or ask for more space"
        )
    remaining = total_budget - floor_total
    weights = {}
    for c in concerns:
        shared_bonus = 1 + 0.5 * (len(c["reviewers"]) - 1)
        weights[c["id"]] = c["severity"] * shared_bonus
    total_weight = sum(weights.values())
    allocation = {}
    for c in concerns:
        share = weights[c["id"]] / total_weight
        allocation[c["id"]] = min_floor + round(share * remaining)
    return allocation


# ---- 用一个真实场景走一遍：world model 想象预算控制器投稿收到的 4 位审稿人意见 ----
concerns = [
    # R1 和 R3 都质疑"为什么不做想象深度 H 的消融"——中等严重度，2 人共同提到
    {"id": "ablation_depth_H", "reviewers": {"R1", "R3"}, "severity": 4},
    # 只有 R2 提到"没有和 AVIC 这个最接近的基线对比"——单人提出但很致命
    {"id": "missing_avic_baseline", "reviewers": {"R2"}, "severity": 5},
    # R1/R2/R4 三人都质疑"和 FFDC 相比新颖性不够"——三人共同提到，最致命的一条
    {"id": "novelty_vs_ffdc", "reviewers": {"R1", "R2", "R4"}, "severity": 5},
    # R3 顺手指出公式 3 有个下标印错——无关痛痒
    {"id": "eq3_typo", "reviewers": {"R3"}, "severity": 1},
    # R1 和 R4 质疑"结果有没有报告种子方差"——中等严重度，2 人共同提到
    {"id": "seed_variance", "reviewers": {"R1", "R4"}, "severity": 3},
]

allocation = allocate_rebuttal_budget(concerns, total_budget=5000, min_floor=150)
for concern_id, budget in sorted(allocation.items(), key=lambda kv: -kv[1]):
    print(f"{concern_id:24s} -> {budget:5d} chars")
assert sum(allocation.values()) == 5000, "全部分配加起来应该正好等于总预算"

# 被 3 位审稿人共同点名、严重度最高的 novelty_vs_ffdc，理应拿到最大的一份预算
assert allocation["novelty_vs_ffdc"] == max(allocation.values())
# 无关痛痒的 typo，虽然拿到的是最少的一份，但仍然不低于保底值，不会被完全无视
assert allocation["eq3_typo"] == min(allocation.values())
assert allocation["eq3_typo"] >= 150

# 边界条件：如果预算小到连保底都分不下去，必须明确报错，而不是分配出负数或者悄悄超支
try:
    allocate_rebuttal_budget(concerns, total_budget=500, min_floor=150)
    raise AssertionError("should have raised ValueError when the budget is too small, not returned normally")
except ValueError:
    pass

print("all assertions passed")
```

**第二步：真实运行结果（已在 `.venv` 里跑过）：**

```
novelty_vs_ffdc          ->  1754 chars
ablation_depth_H          ->  1112 chars
seed_variance             ->   872 chars
missing_avic_baseline     ->   952 chars
eq3_typo                  ->   310 chars
all assertions passed
```

（这里展示的打印顺序按分配到的字数从多到少排列；真实运行的完整字典及断言见上面代码块，`sum(allocation.values()) == 5000` 精确成立——5 条顾虑加起来正好用满预算，不多不少。）

**背后发生了什么：**

这个工具本质上是把"审稿人多次共同提到的问题应该被优先处理"和"审稿人明确指出的致命问题应该被优先处理"这两条朴素直觉，转成一个可复现、可解释的分配公式。**它不是让你机械地照着数字写**——真正的价值是逼你在动笔前先做一次结构化的盘点：把所有意见拆成独立的 concern、标注共享度和严重度，这个盘点过程本身就会暴露"我是不是遗漏了某条被多人提到的重要意见"，比"想到哪写到哪"扎实得多。分配出来的数字只是一个起点参考，实际写作时完全可以根据"这条意见三言两语说得清 vs 需要放一张图才能说清楚"做微调。

**常见坑：**

| 现象 | 大概率原因 |
|---|---|
| 对被 3 位审稿人共同点名的核心质疑，只写了一两句 | 没有意识到"多人共同提到"应该被赋予远高于单人意见的权重，写之前没有先盘点哪些意见是重叠的 |
| 对一条无关痛痒的印刷错误道歉了一整段 | 严重度权重设置反了直觉——致命的技术质疑应该拿大头，纯粹的笔误几句话确认修正即可 |
| 严格按算法分配的数字写，读起来生硬 | 算法给的是参考起点，不是必须一字不差遵守的硬指标，可以按内容本身"讲清楚需要多长"做合理微调 |

**自测清单：**

- [ ] 能说出为什么"多位审稿人共同提到"和"审稿人认定的严重程度"都应该影响预算分配，而不是平均分配
- [ ] 能独立运行这个工具，对着自己真实收到的一批审稿意见跑一遍
- [ ] 理解 `min_floor` 保底机制的意义——再小的问题也不能完全不回应
- [ ] 知道这个工具给出的是参考起点，不是必须严格遵守的最终字数

## 4. 什么时候该提前准备 —— 别等审稿意见来了才开始想

**为什么需要这个 / 不会有什么后果：**

Rebuttal 讨论期通常只有一到两周，如果这段时间才第一次去想"我要不要补一个消融实验来回应质疑"，大概率来不及——真正跑一组新实验、整理结果、画图，往往需要的时间比讨论期本身还长。提前预判、提前准备，能把 rebuttal 阶段真正要做的事，从"临时抱佛脚做实验"降级成"整理已经在手的材料"。

**环境要求：** 论文已经进入投稿准备的最后阶段（提交前 1-2 周）。

**一步步跟着做：**

1. **提交前就问自己："审稿人大概率会问什么？"** 这不是玄学，是可以系统性预判的——参照 [01 号文件第 4 节](01-venue-selection-and-fit.md) 的论文级红旗清单反着想：如果你的论文没有做消融、没有报告种子方差、没有对比最新基线，这些本来就是投稿前该补的；但即使全补了，审稿人依然可能问"如果换一个更难的场景会怎样""为什么不试试另一种设计"——这类"合理但你没做过"的追问，才是 rebuttal 阶段真正的压力来源。
2. **提前跑一批"备用实验"，即使论文里用不上。** 常见的做法：预留一部分计算资源和时间，在投稿后、评审发布前的这段等待期（通常有 1-2 个月），跑几个大概率会被问到、但正文篇幅放不下的补充实验（比如换一个随机种子范围重新统计方差、换一个更大的候选基线做对比）——评审一发布，如果真的被问到，你已经有结果在手，只需要整理成 rebuttal 需要的格式，而不是从零开始跑。
3. **准备一份"论文决策记录"，不要等审稿人问了才回忆当初为什么这么设计。** 写作过程中很多设计选择（为什么用这个 baseline、为什么这样切分数据集）背后有具体原因，但论文正文出于篇幅限制不会都写出来——提前用一份内部文档记下这些"为什么"，rebuttal 阶段被问到时，直接从这份记录里找答案，而不是临时回忆或者向合作者反复确认。
4. **预留应对"合理但没做过"的追问的心理预期。** 即使前面几步都做到位，审稿人依然有权提出你确实没有覆盖到的合理追问——这种情况下不是失败，第 5 节要讲的多人协作机制、以及 `research-writing-deep-dive` 姊妹系列会讲到的"什么时候该坦然承认局限"，都是应对这种情况的正常手段，不是每一条追问都必须用一个新实验去正面回应。

**背后发生了什么：**

这一节的核心逻辑是把"不确定的截止日期压力"换成"可控的提前投入"——评审发布的具体日期、审稿人具体会问什么，都不受你控制，但"提交后到评审发布之间这段等待期怎么用"完全由你自己安排。很多人把这段等待期当成纯粹的休息期，其实这是整个流程里性价比最高的准备窗口：既没有讨论期的时间压力，又已经清楚知道论文提交的最终版本长什么样，能精准判断哪些追问最可能出现。

**常见坑：**

| 现象 | 大概率原因 |
|---|---| 
| 评审发布后才发现某个关键补充实验来不及跑完 | 没有利用提交后到评审发布之间的等待期提前准备，见本节第 2 步 |
| 被问到"为什么用这个设计"答不上来，翻遍聊天记录才想起原因 | 没有维护决策记录，见本节第 3 步 |
| 把每一条审稿人的合理追问都当成"必须用新实验证明自己是对的" | 不是所有追问都需要新实验回应，坦然说明局限、说明这是 future work 的边界，也是一种合理且诚实的回应方式 |

**自测清单：**

- [ ] 能说出提交后到评审发布之间这段等待期，可以用来做哪些提前准备
- [ ] 知道"决策记录"是什么、为什么值得在写作过程中就开始维护
- [ ] 理解不是每一条审稿人追问都要用新实验回应，坦然承认局限也是合理选项

## 5. 怎么和合作者分工 —— 多人协作的组织方式

**为什么需要这个 / 不会有什么后果：**

一篇论文经常不止一个作者，rebuttal 阶段如果没有清晰分工，容易出现两种反模式：所有人都指望"别人会写"结果没人真正动笔，或者所有人同时各写各的，交上去的回应互相打架、口径不一致。

**环境要求：** 论文有 2 位及以上作者。

**一步步跟着做：**

1. **收到评审的第一件事：先读，不要急着写，更不要急着"情绪化回应"。** 一套被反复验证有效的准备流程：把全部评审整理进一份共享文档，用颜色标注——正面评价标绿色、负面批评标红色、其他需要处理但不算负面的内容（比如提问、建议）标橙色；同时留意评审里有没有能现场快速跑一下的小型数据分析（比如审稿人质疑的某个相关性，几行代码就能算出来）。**建议的第一反应是先写一版"愤怒的回应"发泄情绪，然后放一放，睡一觉，再回头重新写真正要提交的版本**——第一版情绪化草稿不会被提交，只是帮你把负面情绪先倒空，避免带着情绪写出的语气渗透进最终稿。
2. **按"审稿人"和"顾虑类别"两个维度分工，不要按"整篇文档平均切几段"分工。** 常见的合理分工方式：一位合作者专门负责回应某位审稿人的全部意见（保证同一位审稿人收到的回应语气/逻辑连贯），或者按"实验类""理论类""写作类"这样的顾虑类别分工（让最懂对应部分的人来回应）。要避免的分工方式是"你写第一段我写第二段"这种纯粹按篇幅切分，容易导致同一条顾虑被两个人各答一半、逻辑断裂。
3. **用第 3 节的预算分配结果，作为分工讨论的共同起点。** 把 concern 列表和分配到的预算发给全部合作者，先对齐"这些是我们认为需要回应的全部问题，这是每条问题的优先级"，避免出现"有人觉得某条意见不重要就没回应，但其实那条是被 3 位审稿人共同提到的核心问题"这种因为信息不同步导致的遗漏。
4. **定一个内部截止日期，早于官方截止日期。** 官方讨论期截止前，必须留出内部的"合并+通读+统一语气"时间——多人各自写完的内容，语气、格式、甚至对同一个技术点的表述可能不完全一致，提交前需要有一个人（或者轮流）负责通读全文，确保读起来像一篇统一的回应，而不是几段拼贴的独白。
5. **确认最终版本符合 02 号文件讲过的匿名要求。** 多人协作最容易在这一步出岔子：合作者在回应里不小心提到了"我们在 XX 实验室的服务器上重新跑了一遍"，或者贴了一个未匿名化的补充材料链接——提交前的通读环节，除了检查语气统一，也要把 02 号文件的匿名化清单再过一遍，rebuttal 内容同样受双盲规则约束。

**背后发生了什么：**

多人协作的核心风险是"责任分散"和"口径不一致"，这两个问题的解法本质上是同一个：**在真正动笔之前，先花时间对齐"我们要回应哪些问题、每个问题谁负责、优先级怎么排"**，把不确定性尽量消灭在写作开始之前，而不是指望写完了再靠"最后通读"补救——通读能修表面的语气问题，修不了"两个人对同一条技术质疑给出了矛盾的解释"这种结构性问题。

**常见坑：**

| 现象 | 大概率原因 |
|---|---|
| 提交前一晚才发现有条重要意见没人回应 | 没有在动笔前对齐 concern 列表和分工，每个人都以为"别人在处理" |
| 两位合作者对同一条技术质疑，给出了不一致的解释 | 分工按"篇幅"切分而不是按"审稿人/类别"切分，导致同一个技术点被割裂处理 |
| 最终稿语气忽软忽硬，读起来像几个人分别写的 | 没有留出统一通读的时间，或者通读只检查了错别字没检查语气一致性 |
| Rebuttal 里意外暴露了身份信息 | 通读环节漏检了匿名化要求，多人协作时这一步尤其容易被忽略，见本节第 5 步 |

**自测清单：**

- [ ] 能说出"先写情绪化草稿再放一放"这个准备流程的意义
- [ ] 知道按"审稿人/顾虑类别"分工比按"篇幅"分工更合理，以及为什么
- [ ] 知道多人协作时，内部截止日期应该早于官方截止日期，用来做统一通读
- [ ] 记得 rebuttal 内容同样受双盲匿名要求约束，多人协作时这一步容易被忽略
