## 0. 这篇文章是怎么验证的（先说清楚）

这一篇的内容性质和后面几篇不完全一样：没有命令可以敲、没有代码可以跑，核心是"判断力"。验证方式对应调整：

- **真实数据**：JMLR/IEEE TNNLS 的影响因子与 h5-index 对比、ML 领域"会议优先"文化的历史成因、workshop/journal track 的真实运作方式、predatory conference 的真实识别信号，全部来自这次真实执行的 WebSearch 调研，来源见文末"参考来源"。
- **真实项目案例**：第 5 节用 [`research/world-model-imagination-controller/`](../../research/world-model-imagination-controller/) 的真实技术内容做 venue-fit 判断力练习——已实地读取该项目的 `01-meeting-briefing.md`，其中"AAAI 已排除、按 ICLR 时间线倒推约 2 个月"是真实记录在案的决定，但文档本身没有展开排除 AAAI 的具体理由，本文不会编造未记录的内部理由，第 5 节会如实标注这一点，只是借这个真实项目的技术特征练习"如果自己来判断，该看哪些依据"这套方法本身。
- **无法验证的部分**：某个具体 venue"社区评价怎么样"这类判断本质上是软性共识，没有唯一客观真值，本文呈现的是调研到的、有具体来源支撑的普遍看法，不代表所有子领域的人意见完全一致。

## 1. Venue 的基本图景 —— 投哪、投什么形式，为什么这本身是个需要判断力的决定

**为什么需要这个 / 不会有什么后果：**

如果你只写过课程作业，"交作业"这件事是没有选择的——只有一个提交入口。科研论文完全不是这样：同一份工作，理论上可以投给几十个不同的地方，而且这些地方彼此差异极大——**会议（conference）** 和 **期刊（journal）** 在 ML 领域几乎是并行的两条线，会议之下还分 **main track**（完整工作）、**workshop track**（进行中的工作，通常 2-6 页的短文）；期刊之下一些会议甚至开了 **journal track**（会议+期刊的混合体）。选错了会有很具体的后果：把一份还在探索阶段的初步结果硬投 main track，大概率因为"结果不够完整"被拒，白白耗掉一轮 3-4 个月的评审周期；把一份已经很扎实的工作投进一个和你的技术内核关系不大的 venue，审稿人可能压根不熟悉你用的方法，提出的意见"文不对题"，你也很难在字数有限的 rebuttal 里把审稿人从零教会。

更隐蔽的后果是**打乱你自己的时间线**：ML 头部会议（NeurIPS/ICML/ICLR，业内常称"三大"）截稿日期一年只有固定的几个窗口，一旦错过某一轮，下一次机会可能是几个月后——[`research/world-model-imagination-controller/01-meeting-briefing.md`](../../research/world-model-imagination-controller/01-meeting-briefing.md) 里这份真实项目的时间线规划就是一个例子：文档第 5 节明确记录"AAAI 已排除，时间线按 ICLR 倒推约 2 个月"，这条决定直接决定了后续每一周该做什么实验、什么时候必须收尾——venue 选择不是投稿前一天才需要想的事，它会往回影响你研究阶段的时间分配。

**环境要求（这里指"判断前你需要先确认的信息"）：**

- 你大致知道自己论文属于哪个技术方向（比如：这份 world model 想象预算控制器的工作，核心方法论落在 model-based RL / 测试时计算 / 决策时搜索这个交叉地带）。
- 你知道自己论文现在的成熟度——是完整的方法+系统性实验，还是阶段性的初步发现。

**一步步跟着做：**

1. **先分清"会议"和"期刊"在你的领域里分别意味着什么。** 和很多传统工科/理科领域不同，ML 领域是**会议优先**的文化：NeurIPS、ICML、ICLR 这几个头部会议的论文，业内认可度普遍不低于甚至高于大多数期刊论文——这不是 ML 特有的"不正规"，而是这个领域公认的、独特的发表文化（下一节会展开这背后的历史原因）。也就是说，你不能直接套用"期刊比会议档次高"这种在别的学科成立的经验。
2. **确认自己的论文成熟度，对应到 track。** 完整的方法+系统性实验（消融、多个 baseline、多个 benchmark）投 main track；还在进行中的初步发现、demo、位置性观点投 workshop（下一节详细展开）。不要把 workshop 当成"main track 的备胎"随便对待，也不要把明显还没做完的工作硬塞进 main track。
3. **列出 2-3 个候选 venue，逐一对照"技术内核匹配度"。** 不是看这个 venue 名气大不大，是看你方法论的技术根子在哪——如果你的贡献核心是"测试时计算怎么分配"，那和这个概念强相关的会议（近几年 NeurIPS/ICML/ICLR 都有大量 test-time compute、world model、decision-time planning 相关工作）会比一个泛泛的"人工智能大会"更合适，因为审稿人池子里更可能出现真正懂这个子领域的人。
4. **确认截稿日期和你的时间线能不能对上**，倒推出"从现在到投稿还剩多少周，这些周要做完哪些事"——这一步经常被新手忽略，导致临近截稿才发现某个关键实验根本来不及跑完。

**背后发生了什么：**

ML 领域"会议优先"这个文化不是历史包袱，而是这个领域发展速度决定的：期刊评审周期动辄半年到一年以上，而 ML 技术迭代速度极快，会议评审周期（提交到公开决定通常 3-4 个月）更适应这个节奏，逐渐形成了"重要成果先投顶会"的路径依赖，头部会议也因此聚集了这个领域最活跃的审稿人群体和最新鲜的评审标准。这也是为什么下一节要专门拆解"影响因子"这个在很多学科通用的质量代理指标，在 ML 语境下经常不是你以为的那个意思。

**常见坑：**

| 现象 | 大概率原因 |
|---|---|
| 论文投了一个"名气很大"但和自己方法论关系不大的会议，评审意见普遍"没 get 到点" | 审稿人池子和你的技术方向不匹配，参见第 4 节"红旗信号"和第 5 节的 fit 判断案例 |
| 明明工作还在早期阶段却坚持投 main track，被拒后才反应过来 | 混淆了"这个方向很有前景"和"这份工作已经完整到可以投 main track"——workshop 不是低人一等的选择，是匹配当前阶段的正确选择 |
| 临近截稿才发现关键实验做不完 | venue 选择没有和研究时间线一起倒排，截稿日期应该在研究规划阶段就成为一个已知约束，不是投稿前才查 |

**自测清单：**

- [ ] 能说清楚在 ML 领域"会议优先"和其他学科"期刊优先"这个文化差异，以及为什么
- [ ] 能区分自己手头的工作现在更适合 main track 还是 workshop
- [ ] 知道 venue 的截稿日期应该多早开始纳入研究时间线规划
- [ ] 能列出至少 2-3 个和自己研究方向技术内核匹配的候选 venue，而不是只知道"三大"这种笼统答案

## 2. 影响因子迷思 —— 为什么 JMLR 比影响因子更高的 IEEE TNNLS 在 ML 圈里可能更受认可

**为什么需要这个 / 不会有什么后果：**

"影响因子"（Impact Factor, IF）是很多学科评价期刊质量的标准指标，很多新手（包括不少跨领域的评委、导师）会下意识用"IF 高不高"来快速判断一个 venue 值不值得投。这个直觉在 ML 领域会系统性地失灵，如果不了解这一点，可能会做出两类错误决定：**为了追一个数字更高的期刊放弃一个在本领域内公认更权威的选择**，或者**误判别人论文的分量**（比如低估一篇发在 JMLR 上但影响因子不算最高的论文）。

**环境要求：** 无（这是一次调研结论的讲解，不需要额外准备）。

**一步步跟着做（真实数据对照）：**

1. **先看两本期刊的原始指标。** 这次调研查到的真实数字：IEEE Transactions on Neural Networks and Learning Systems（TNNLS）的 JIF 是 **9.7**，Google Scholar 的 h5-index 是 **149**；JMLR（Journal of Machine Learning Research）的 JIF 是 **6.8**，h5-index 是 **117**。单看这两个数字，TNNLS 全面领先。
2. **再看社区里的真实评价。** 调研结论明确指出：尽管 TNNLS 数值上占优，很多 ML 研究者对一篇纯方法论文，仍然会更偏好投 JMLR，原因是"这个圈子知道哪些期刊是由 ML 研究者自己办、为 ML 研究者服务的"——这不是空洞的说法，是一条可以追溯到具体历史事件的判断依据（见下一步）。
3. **理解 JMLR 的历史成因。** JMLR 创刊于 2001 年，起因是当时 *Machine Learning* 期刊（一本历史更久的老牌期刊）的 **40 位编委集体辞职**，抗议"在互联网时代还让研究者继续在收费、封闭获取的期刊上发表论文"不合理，随后自己创办了 JMLR——一本同行评审、开放获取的期刊。这段历史直接决定了 JMLR 在 ML 社区里的身份：它不只是"一本期刊"，是这个领域自己人为了开放科学理念专门办起来的阵地。
4. **理解"该投哪本"取决于论文类型，不是单纯比 IF。** JMLR、*Machine Learning*、IEEE TNNLS 都能发方法类深度工作，但把一篇偏工程应用的论文投给以方法论/理论见长的 JMLR 通常不合适（编辑会认为方向不对口，不是质量问题）；反过来，一篇纯方法论文如果只把 IEEE TNNLS 定位成"应用/工程导向、读者群更偏工程"的期刊,也可能错过更匹配的受众。
5. **把这条结论放回第 1 节的大背景里看。** 别忘了 ML 领域本身是会议优先的：JMLR 和 TNNLS 相互competing的同时，两者都要和 NeurIPS/ICML/ICLR 这些头部会议竞争"研究者的第一选择"——多数人会说，真正的"prestige"排序里，头部会议不比这两本期刊低。

**背后发生了什么：**

影响因子本质上衡量的是"这本期刊过去两年发表的论文，平均每篇被引用多少次"——这是一个纯粹的引用计数统计量，不衡量"这本期刊是不是这个具体子领域的人认可的地方"。一个期刊如果覆盖面更广（比如 TNNLS 覆盖神经网络和学习系统的更广泛应用），基数大、引用来源多元，容易堆出更高的 IF；一个定位更聚焦、由本领域核心研究者主导的期刊（比如 JMLR 聚焦方法论本身），哪怕 IF 数字不占优，在真正懂行的人眼里权威性可能反而更高。这就是"影响因子迷思"的本质：**IF 衡量的是引用规模，不衡量社区认同**，两者经常相关但绝不等价，尤其在 ML 这种年轻、发展快、社区认同感强的领域，后者的权重经常被低估这条指标本身的局限性的人给忽略。

**常见坑：**

| 现象 | 大概率原因 |
|---|---|
| 只用 IF 数字排序候选期刊，选了数字最高的那个 | 没有意识到 IF 是引用规模指标，不是社区认同指标；同样的逻辑也适用于会议之间比较"录用率"数字——录用率低不天然等于质量高，也可能是投稿量虚高 |
| 把一篇应用导向的论文投给方法论导向的期刊被拒，觉得是质量问题 | 很多时候是方向不对口，编辑/审稿人判断的是"这不是我们要的这类论文"，不是"这篇论文不够好" |
| 跨学科评委/导师用别的学科的 IF 直觉评价 ML 论文分量 | ML 领域的发表文化本身比较特殊（会议优先+社区自办期刊传统），直接套用其他学科的经验容易系统性误判 |

**自测清单：**

- [ ] 能说出 JMLR 和 IEEE TNNLS 各自真实的 IF/h5-index 数字，以及谁的数字更高
- [ ] 能解释为什么数字更低的 JMLR，在部分场景下反而可能被认为更"权威"
- [ ] 能说出 JMLR 创刊的历史背景（2001 年编委集体辞职事件）
- [ ] 知道"该投哪本"取决于论文类型（方法论 vs 应用），不是单纯比 IF 数字

## 3. Track 怎么选 —— Main Track / Workshop Track / Journal Track 三方对比

**为什么需要这个 / 不会有什么后果：**

第 1 节已经提过"workshop 不是备胎"，这一节把这句话具体展开成可操作的判断依据。选错 track 的后果是双向的：把不够完整的工作硬塞进 main track，大概率被拒，浪费一整轮评审周期；把已经很扎实、够格投 main track 的工作过度谨慎地丢进 workshop，则会白白损失本该有的正式发表记录和更广的受众——workshop 论文通常不算严格意义上的同行评审正式发表，在很多场合（比如求职/申请材料）分量不如 main track。

**环境要求：** 已经完成第 1 节，知道自己论文当前的成熟度。

**一步步跟着做：**

1. **Main Track（完整工作）**：适用于方法+系统性实验都完整的论文——有明确的贡献、有和现有方法的系统对比、有消融实验支撑关键设计选择。这是大多数人默认的目标，也是唯一被视为"正式、完整同行评审发表"的选项。
2. **Workshop Track（进行中的工作）**：定位是"完成度大概在 50%-70%"的工作——初步但有价值的发现、还在验证阶段的想法、位置性论述。历史上 ICLR 甚至专门拆分过独立的 Workshop Track（区别于 Conference Track 单独评审），workshop 论文通常篇幅更短（2-6 页级别的短文/扩展摘要），评审标准也相应更宽松。**workshop 的真实价值**：给还不够成熟但方向对的工作一个曝光和拿反馈的机会，对博士生/研究早期阶段的人是很自然的早期发表渠道，不代表工作没价值，只代表"还没做到能扛住 main track 严格程度的评审"。
3. **Journal Track（少数会议提供）**：一些会议（比如 ACML）提供期刊道——论文走期刊式的评审流程（往往有多轮修改机会，不是"一次提交定生死"），录用后发表在对应期刊的特刊上，同时仍然可以在会议上展示。ECML PKDD 也有类似的期刊道设计。这类选项适合"工作本身扎实，但可能需要不止一轮反馈才能打磨到位"的论文——本质上是把期刊"可以大修重投"的宽容度和会议的展示曝光度结合起来。
4. **决策依据小结成一张表：**

| 维度 | Main Track | Workshop Track | Journal Track（少数会议提供） |
|---|---|---|---|
| 适用完成度 | 完整（方法+系统实验+消融） | 50%-70%，进行中 | 完整，但可能需要多轮修订才能定稿 |
| 篇幅 | 通常 8-10 页级别 | 通常 2-6 页级别 | 期刊篇幅规范，通常比会议正文更宽松 |
| 评审严格程度 | 最严格 | 相对宽松 | 严格但允许大修重投（不是一次定生死） |
| 是否算正式同行评审发表 | 是 | 通常不算严格意义的正式发表 | 是（按期刊标准） |
| 典型使用场景 | 求职/申请的主力材料 | 早期反馈、占位曝光 | 需要多轮打磨的复杂工作 |

**背后发生了什么：**

这三种 track 本质上是"评审严格程度"和"给作者的修订空间"两个维度上的不同权衡：main track 严格但一次性（reject 就要等下一轮）；workshop 宽松但曝光度和分量有限；journal track 试图把"多轮打磨"的宽容度也纳入会议生态。选择哪一种，本质是在问"我这份工作现在最需要的是什么——严格的同行认可，还是先拿到反馈再迭代，还是需要不止一轮的打磨空间"。

**常见坑：**

| 现象 | 大概率原因 |
|---|---|
| 一篇明显不完整的工作坚持投 main track | 混淆了"这个想法有价值"和"这份材料的完整度够格" |
| 把 workshop 发表等同于 main track 发表写进简历/申请材料 | 大部分场合会区分对待，过度包装反而可能在后续核实时显得不诚实 |
| 不知道自己领域的会议是否有 journal track 可选 | 这是相对小众的选项，容易被忽略，值得在选择 venue 时顺手查一下当年 CFP 有没有提供 |

**自测清单：**

- [ ] 能说出 main track 和 workshop track 各自适用的完成度区间
- [ ] 知道 workshop 发表和 main track 发表在"是否算正式同行评审发表"上的差异
- [ ] 知道 journal track 这种选项的存在，以及它解决的是什么问题
- [ ] 能对着自己手头的工作，判断现在更适合哪个 track，并说出判断依据

## 4. 识别真实红旗信号 —— 论文级红旗 vs Venue 级红旗

**为什么需要这个 / 不会有什么后果：**

这一节要分清两类完全不同的"红旗"：**论文级红旗**（你的论文本身有什么特征容易被拒，这是自我审查的依据）和 **venue 级红旗**（你打算投的这个"会议"本身是不是有问题，这是保护自己不被骗、不浪费版面费和时间的依据）。不识别第一类，容易带着侥幸心理投一篇实际上经不起推敲的稿子；不识别第二类，后果可能更严重——真金白银交了版面费，成果却发在一个学界根本不认可、甚至会倒扣你学术信誉的地方。

**环境要求：** 无。

**一步步跟着做：**

**A. 论文级红旗（审稿人一眼就能识别的常见问题模式）：**

1. **摘要声称"新模型"，方法部分却是标准 transformer/CNN 换了个数据集**——审稿人对这个模式极其敏感，"换皮"式的贡献很容易被一眼看穿。
2. **结果表没有消融实验**——没法证明性能提升到底来自你声称的核心创新，还是某个无关的调参技巧。
3. **没有和当前最新的基线对比**——用几年前的旧基线证明自己领先，说服力大打折扣。
4. **报告结果没有种子方差**——单次跑出来的数字，审稿人无法判断这是真实提升还是随机波动。
5. **贡献陈述和实际验证的场景不匹配**——比如声称"通用方法"，却只在一个非常窄的合成环境里验证过。

**B. Venue 级红旗（判断一个"会议"本身是不是有问题）：**

1. **邀请邮件本身可疑**：群发式的"Dear Researcher"开头（不是你的真实姓名）、过度吹捧你是"知名专家"（哪怕你只是在读研究生）、要求你几天内就决定投稿、发件邮箱是 Gmail/Yahoo 这类免费邮箱而不是机构域名。
2. **审稿速度快得不合理**：真实的同行评审需要以周为单位，如果一个"会议"承诺投稿后 48 小时甚至几小时内就给"录用"通知，或者所有投稿看起来都被接收，说明它根本没有实质性的评审。
3. **组委会/演讲嘉宾名单可疑**：预防性做法是挑 2-3 个列出来的知名学者，去他们自己的主页确认是否真的提到过这个会议——很多"挂名"是组织方单方面盗用名字，本人根本不知情。
4. **收费结构可疑**：presenter（要展示论文的人）被收取远高于一般与会者的注册费，是一种典型的"以发表为生意"的盈利模式信号。
5. **过度承诺检索/发表保证**：任何"保证被 Scopus/Web of Science 收录"的承诺都值得怀疑——是否被这些数据库收录，从来不是会议主办方能单方面保证的事，这个决定权在数据库方。
6. **网站质量差、主题大杂烩**：网站有大量模板痕迹、拼写错误、和其他会议网站高度雷同；会议主题标榜"跨学科"却覆盖了完全不相关的一堆领域，这种"来者不拒"往往是为了扩大投稿量收取更多版面费。

**背后发生了什么：**

论文级红旗背后的共同逻辑是"审稿人在用一套隐含的最低标准过滤":贡献是否经得起推敲（不是换皮）、结论是否有实验支撑（消融+当前基线+方差）。Venue 级红旗背后的逻辑是识别"以营利为目的、包装成学术会议的商业活动"——正规同行评审需要真实的时间和专家投入，这些成本没法被压缩到几乎为零，一旦"评审"快到不合理、门槛低到人人能过，唯一合理的解释就是它根本没有认真评审。

**常见坑：**

| 现象 | 判断 |
|---|---|
| 论文投出去大概率被拒，但自己心存侥幸 | 提交前对照 A 类红旗自查一遍，能提前发现的问题不要留给审稿人替你发现 |
| 收到一封"热情邀请"投稿的陌生会议邮件 | 先查：邀请是不是群发模板？审稿承诺是不是快得不合理？组委会名单是不是能在本人主页上验证？ |
| 已经投了一个后来被发现是可疑会议 | 及时止损，不要因为已经交了钱/投了稿就硬着头皮参加，学术声誉的长期损失比这笔沉没成本大得多 |

**自测清单：**

- [ ] 能说出至少 3 条论文级红旗，并对照检查自己手头的论文有没有踩中
- [ ] 能说出至少 3 条 venue 级红旗（预警信号）
- [ ] 知道怎么用"查作者/演讲嘉宾自己的主页"这个方法验证一个可疑会议的真实性
- [ ] 理解"审稿快到不合理"本身就是最强的红旗之一，不需要等到其他证据都齐全才警觉

## 5. 结合真实项目案例，走一遍 venue-fit 判断流程

**为什么需要这个 / 不会有什么后果：**

前面 4 节讲的都是判断依据，这一节把它们串成一次真实的练习——不是编一个抽象例子，是用 [`research/world-model-imagination-controller/`](../../research/world-model-imagination-controller/) 这个真实在研项目（用户在导师 Weikai Lin 指导下的测试时想象预算自适应分配控制器研究）的真实技术内容走一遍。**如实说明**：这份项目文档（`01-meeting-briefing.md`）记录了"AAAI 已排除、按 ICLR 时间线倒推约 2 个月"这个真实决定，但没有展开排除 AAAI 的具体理由——本节不编造未记录的内部理由，只是借这个项目真实的技术特征，练习"如果自己从零判断，应该看哪些依据"这套方法本身。

**环境要求：** 已经读过第 1-4 节。

**一步步跟着做：**

1. **看技术内核落在哪里。** 这份研究的核心问题是"world model 做测试时决策（该不该想象、想几个候选、什么时候停）的计算预算该怎么自适应分配"，方法论上牵涉 value-of-computation 理论、序贯决策、不确定性估计，实证对比对象是 AVIC（arXiv:2602.08236）、FFDC（arXiv:2605.06222）、Video-T1（arXiv:2503.18942）这几篇 2025-2026 年的工作，背景理论基础是 DreamerV3、MuZero、TD-MPC2 这条 model-based RL 脉络。这些关键词——test-time compute、world model、决策时搜索/规划——近几年在 NeurIPS/ICML/ICLR 都有持续增长的相关工作和专门的 session，说明技术内核和"三大"的审稿人池子高匹配。
2. **对照第 1 节"技术内核匹配度"这条判断依据。** 如果换一个泛化程度更高、不特别聚焦 ML 方法论本身的综合性 AI 会议，审稿人抽到真正懂 model-based RL/test-time scaling 这个细分方向的概率会更低，出现"审稿人没 get 到点"的风险相应更高——这正是第 1 节强调"技术根子往哪投"的具体应用。
3. **对照第 3 节的 track 判断。** 项目当前阶段（按 `04-sharpened-recommendation.md` 记录）还在"pilot 验证阶段"——已经在一个 32 状态的合成小环境里验证了几条核心假设（比如"想象和基线共享同源信息时，多算是无用功"有完整数学推导支撑），但还没有搬到真实高维 world model（DreamerV3/TD-MPC2）上验证。这意味着：如果现在就要投稿，需要先把 pilot 阶段的发现扩展成一套完整的方法+系统实验（对照第 3 节 main track 的完成度要求），或者考虑这个阶段的初步发现是否适合先投一个 workshop 探路——这正是 `01-meeting-briefing.md` 里"idea 10（诊断性研究）是独立成一篇论文，还是作为方法论文的第一部分"这条开放问题所对应的真实判断场景。
4. **对照第 4 节做一次自查。** 有没有消融？（pilot 阶段已经做了扫想象深度 H、扫候选数 K 的对照实验，这是消融的雏形）有没有当前基线？（AVIC/FFDC/Video-T1 都是 2025-2026 年的最新工作，不是过时基线）有没有种子方差？（pilot 结果确实报告了 5 个随机种子的均值±标准差，符合规范）——可以看到，这份项目从 pilot 阶段就已经在按论文级红旗自查的标准做事，这不是巧合，是"先建 eval protocol"这个研究纪律本身带来的副产品。
5. **时间线校验。** `01-meeting-briefing.md` 的 Gantt 图显示"会议后"阶段（baseline 复现 14 天+方法开发消融 21 天+写作投稿准备 8 天）总计约 6 周，加上会前已完成的文献调研+pilot 阶段，整体和"按 ICLR 时间线倒推约 2 个月"这条约束对应——这是第 1 节"venue 截稿日期倒推时间线"的真实应用。

**背后发生了什么：**

这个练习真正想说明的是：venue 选择不是一次孤立的决定，是贯穿研究全过程的一条约束线——从"技术内核匹配"决定投哪个圈子的审稿人池子，到"当前完成度"决定投 main track 还是先探路，到"红旗自查"决定这份材料经不经得起最严格的评审标准，到"截稿日期倒推"决定研究阶段每一周该做什么。四个判断维度不是投稿前一天才需要的 checklist，是从项目一开始就该心里有数的坐标。

**范围声明：** 本节呈现的是"如果自己套用这套判断框架会看到什么"，不代表这就是该项目团队实际决策过程的完整还原——真实的方向拍板按项目文档记录是会议讨论决定的事,本节没有也不会杜撰会议内容。

**常见坑：**

| 现象 | 大概率原因 |
|---|---|
| 只看"这个方向火不火"就锁定 venue，不看自己论文当前完成度 | 混淆了"技术方向匹配"和"完成度匹配"，两者要同时满足 |
| 时间线规划只留出"写论文"的时间，没有留出"如果审稿人要求补实验"的缓冲 | 后面 03/04 号文件会详细展开这一点，这里先建立"venue 时间线要留缓冲"的意识 |

**自测清单：**

- [ ] 能把第 1-4 节的判断依据，套用到自己手头正在做的任何一份工作上做一次自查
- [ ] 能说出"技术内核匹配度"和"完成度匹配 track"是两个独立但都必须满足的维度
- [ ] 知道 venue 选择应该多早开始纳入研究规划，而不是投稿前一周才想
