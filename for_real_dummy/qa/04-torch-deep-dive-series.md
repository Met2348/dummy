## 做了什么

建立了第三条深挖系列:**[torch-deep-dive/](../torch-deep-dive/00-roadmap.md)** —— 100 个知识点,11 个分类,覆盖 tensor 内存模型 → autograd → nn.Module 内核 → 层的数学推导 → 损失函数 → 优化器 → 训练循环(混合精度/梯度累加/checkpoint)→ 内存性能 → 分布式训练 → 序列化部署 → 调试报错精解,难度顺序由浅入深,专门避开和 numpy 系列重复的基础 tensor 操作,只讲 torch 独有、且是面试重灾区的部分。

## 怎么做的(这次过程比前两条系列更曲折,方法论价值也更高)

- **分工:** 01(tensor 内存模型)我自己手写打样,验证模板可行;02/03/04/05/06/08/09 用并行 subagent 分工写;07/10/11 因为下面这个原因改成我自己写。
- **中途遭遇真实的 API 5 小时限额耗尽**,6 个 subagent 在不同完成阶段被打断:04/05/06/08/09 已经写完文件但自己的最终校验环节被打断,07/10/11 连文件都没来得及写。用户一句"意外中断继续"确认了继续推进的方向——没有重新派发 agent(限额窗口内大概率立刻再次失败),而是我自己接手完成剩下的工作。
- **独立复验每一个文件,不因为"文件已存在"就默认它对**:哪怕是 agent 在被打断前已经写完的文件,也当成"自检未完成"来对待,逐节抽取代码重新跑一遍。过程中真的抓到了几处需要修正的问题,不是走流程:
  - **01(我自己写的)**:两处浮点数/stride 数值错误——自己复制粘贴示例时没针对新场景重新算。
  - **06(agent写的)**:一处硬编码的、依赖未设种子随机数的"示例数值",重新运行对不上。
  - **08(agent写的)**:一处实验设计缺陷——把不必要的矩阵乘法重新计算放进了计时循环内,导致噪声(GPU频率波动)把要测的真实信号(PCIe传输时间差)完全淹没,复验时5轮里只有1轮方向对;拆开重新设计实验后,5轮全部同向,信号从"勉强1.05倍、时有时无"变成"稳定20~40倍、清晰可见"。
  - **09(agent写的)**:一处 API 使用错误——按"空格拼接"的整句去匹配一段官方 docstring,但源码里那句话实际是手动折行(中间是`\n`不是空格),直接照抄会匹配失败。
  - **11(我自己写的)**:一处过于精确的性能数字断言——`detect_anomaly()`的开销倍数在单次测量里从2.9倍到7倍都出现过,原始写法断言死一个"6倍"级别的阈值,复验时就因为噪声不达标失败;改成更宽松、如实反映"数倍、量级稳定但精确倍数不固定"的表述和阈值。
- **08 篇额外发现一个真实的 PyTorch/cuBLAS 行为**:同一进程里只要调用过一次 matmul,cuBLAS 会留下一块约 8MB 的持久 workspace,`del`+`empty_cache()`都不会释放它——导致`memory_allocated()`的"零点"不再是绝对的0。这不是文档的错(文档每一节单独跑确实是0),而是我自己"整个脚本一次性跑完11个小节"的验证方式暴露出的一个真实细节,已经补记进 08 篇正文。

## 结论

11 个文件全部完成并逐条独立验证,合计精确 100 个知识点(12+14+10+11+7+10+7+8+6+6+9)。这条系列比前两条系列的核心区别不是"更长",是每个知识点都要求现场读官方源码/docstring(不是转述印象)、用对比实验把"为什么这样设计"证到底(比如 06 篇验证 AdamW 和 Adam+L2 在梯度尺度差异大的参数上实际衰减力度能差 1000 倍),并且明确准备"面试追问链"而不是只回答表面问题。

最值得记住的不是任何一个 PyTorch 知识点,是这次反复验证出的方法论教训:**float32 别用 `==`、随机数据要设种子、测小信号时先把无关的重计算挪出计时循环、跨小节共享同一个进程时库的持久化状态会污染"从零开始"的假设**——这几条在 numpy 系列、python-advanced 系列里也出现过雏形,这次因为深度更高、验证更严格,被更完整地暴露和记录了下来。

三条深挖系列(numpy-deep-dive + python-advanced + torch-deep-dive)现在合计约 240 个知识点,全部完成并验证,阶段性告一段落。
