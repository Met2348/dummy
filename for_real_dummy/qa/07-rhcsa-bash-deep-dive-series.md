# Q&A —— RHCSA/Linux 系统管理深挖系列(独立新领域)的建立与验证(2026-07-11)

## Q:TensorFlow 系列做完之后,再做一条 Linux/bash 系列,面向 AI 博士研究场景的运维需求,RHCSA 认证深度

延续前五条系列的方法论,但这条系列的性质和前面完全不同:前五条都是"补 Python/ML 相关能力",这条补的是**完全独立的新领域**——Linux 系统管理 + bash 脚本编程,对标 Red Hat 官方 RHCSA(EX200)认证的技能范围,明确排除已经转向 Ansible 自动化的 RHCE(EX294)。RHCSA 官方基准已从 RHEL 9 切换到 RHEL 10,分类和内容撰写都对齐这个新基准。

## 做了什么

建立了第六条深挖系列:**[rhcsa-bash-deep-dive/](../rhcsa-bash-deep-dive/00-roadmap.md)** —— 100 个知识点,9 个分类,覆盖必备工具与文本处理 → 进程与系统运行 → 本地存储与 LVM → 文件系统与权限 → 软件与系统部署 → 用户组管理 → 网络配置 → 安全(SELinux/防火墙)→ bash 脚本编程本身,固定七步讲解结构(不采用 torch/tensorflow 系列的"面试怎么问"环节——RHCSA 是纯上机操作考试,没有面试环节)。

**环境:** WSL2 Rocky Linux 10.2(Red Quartz,RHEL 10 下游二进制兼容发行版),真实 systemd 作为 PID 1,默认 root 会话,1TB 可用磁盘。

## 怎么做的(这次过程比前几条系列都更曲折,核心原因是环境本身先坏了)

- **环境先天故障,排查耗费大量精力:** taxonomy 定稿后,发现 Rocky Linux WSL 实例连 `root` 用户都解析不出来(`getpwnam(root) failed 2`),`wsl -l -v` 显示实例存在但 VHDX 只有 134MB(正常装好的系统会有几百 MB 到几 GB)、UNC 路径完全访问不到文件系统——诊断确认这不是配置错误,是这个实例从未真正完成过初始化。找到官方修复路径(注销坏实例,从 `dl.rockylinux.org` 官方 CDN 下载正式的 `Rocky-10-WSL-Base` 镜像重新导入)并准备执行时,发现用户那边**另一个并发的 Claude Code 会话已经独立完成了同样的修复**——两边分头排查、殊途同归定位到了同一个官方镜像源。
- **意外发现的并发协作:** 修复 WSL 之后,发现 `rhcsa-bash-deep-dive/` 目录和 taxonomy 早就已经存在(那个并发会话已经先行完成了 taxonomy 定稿和 09 类目),而在我按常规流程派发 02/04/05 三个类目的 agent 之后,同一批文件又被那个并发会话独立完成/覆盖了一轮(我自己派发的 agent 有的因为限额中断、有的和对方的版本"撞车"),01/02/03/04/05/06 最终都是那个并发会话产出的版本。**处理原则:不因为"是别人写的"就跳过独立复验,也不因为自己的 agent 被撞车就重新抢着写**——检测到并发冲突后,立刻停止重试自己的 agent(避免用陈旧上下文覆盖对方已经写好的新内容),转为对磁盘上实际存在的最终版本逐一独立复验,验证通过就正常收编、提交,这比争抢"谁的版本"更重要。
- **共享可变环境导致的真实"环境污染"案例(值得记录的方法论教训):** 独立复验 02 类目"rsyslog/logrotate 默认未装"这条结论时,亲自在 WSL 里重跑发现 rsyslog **确实装上了**,与文档描述矛盾。查 `dnf history`/`rpm --qf INSTALLTIME` 定位到:02 类目写完(13:47)之后、05 类目做包管理测试期间(13:51),rsyslog 被作为**另一个类目验证过程的副作用**装了上去。结论:02 类目的原始记录在撰写当下是真实、准确的(反映了 Rocky Linux 10 的真实默认状态),只是共享 WSL 实例后续被其他并发类目的 dnf 操作"弄脏"了——**不能因为"现在重跑对不上"就认定原始内容错误,必须先定位是内容本身的错误,还是共享可变环境事后漂移导致的**,这是多 agent 共享同一个可变系统时特有的验证陷阱,原有内容予以保留,不做无意义的"纠正"。
- **安全边界:多 agent 共享一个可变 WSL 实例,必须显式约束破坏性操作。** 给自己派发的 agent 明确约束:绝对不能真的执行 `reboot`/`shutdown`/`poweroff`(会连带打断其他并发 agent 的工作)、所有临时资源加分类编号前缀避免撞名、LVM/存储实验一律用 `dd`+`losetup` 构造的 loop device(不碰真实设备)、涉及改 IP/路由的知识点(07 类)全部在独立创建的 dummy 虚拟网卡上操作、全程确认真实 `eth0` 连接零影响。这几条约束后来被证明确实必要且被遵守。
- **WSL2 架构性限制,三次遇到,三次都选择"深挖到底 + 如实标注"而不是回避:**
  1. **GRUB2(05类第5项)**:WSL2 根本不经过 BIOS/UEFI→GRUB→内核这条引导链路。`grub2-mkconfig` 能装、能跑、不报错,甚至能生成 UUID 精确匹配真实磁盘的"看起来无懈可击"的配置——但用 `/proc/cmdline` 里真实生效的 WSL 专有启动参数、和当前运行内核在 `/boot` 下找不到任何对应文件这条决定性证据,证明这份配置从生成那一刻起就是孤立的、不会被任何引导流程读取的产物。
  2. **VDO(04类第13项)**:`dm-vdo` 内核模块在 WSL2 定制内核里没有编译进去,`lvcreate --type vdo` 因此必然失败,但如实验证了工具链完整、`vdocalculatesize` 规划工具能真实运行、`modprobe dm-vdo` 现场确认报错文本,把"失败原因是内核限制不是命令语法错误"这件事证到底。
  3. **SELinux(08类第1-7项)**:三层递进排查——配置文件写 `enforcing` 但 `getenforce` 返回 `Disabled`(根因是禁用与否现在由内核启动参数 `selinux=0` 决定,不是这个文件);尝试用 `grubby` 移除内核参数、重启验证无效(WSL2 根本不读 BLS 配置,和 GRUB2 那条同根同源);深挖发现 `/sys/fs/selinux` 存在但残缺(`policyvers` 文件缺失),策略加载失败。但没有因此把整个 SELinux 章节都判定"不可用"——精确验证出 `semanage`(操作策略数据库本身)不依赖内核真正 enforcing,可以完整验证,而 `getsebool`/`setsebool`(操作运行时活跃状态)确实不行,两者分属不同层次,不能一概而论。
  4. 另有 podman rootless(05类第8项)因 `newuidmap` 缺 `CAP_SETUID` 失败,完整诊断到"很可能是这个 WSL 根文件系统当初从 OCI 镜像层导出解包时丢失了文件能力扩展属性"这一步,**但没有执行 `setcap` 修复**——因为这会永久修改共享环境的系统安全属性,未经明确授权不应该对共享环境做持久性变更,如实记录诊断结果、不擅自修复。
- **我自己的独立复验:** 全部 9 个文件通读一遍(不只是抽查),并挑选每篇里最关键、最反直觉的结论亲自在 Rocky Linux WSL 里重新跑一遍,包括:`ln -s` 在无权限环境下静默降级成假拷贝(01)、chmod 在 NTFS 上的权限坍缩(01)、`pvremove` 顺序错误退出码是 5(03)、`xfs` 最小 300MB 门槛(04)、账户锁定是原哈希前加 `!`(06)、`network.service` 在 RHEL 10 已不存在(07)、SELinux 配置文件与 `getenforce` 的真实落差、firewalld `--reload` 会清除仅运行时的规则(08)——全部现场复现,结论与文档记录一致。

## 结论

9 个文件全部完成并逐条独立验证,合计精确 100 个知识点(14+12+12+13+10+10+10+10+9),192 个代码块(01:14/02:24/03:23/04:26/05:20/06:19/07:20/08:19/09:27)100% 通过。这条系列最大的方法论价值,不是任何一个具体的 Linux 命令,而是**在"多个 agent 共享同一个可变系统"这种此前五条系列从未遇到过的场景下,把"独立验证"这个原则又往前逼近了一层**——不仅要验证"某段代码写得对不对",还要能分辨"验证当下环境的真实状态"和"文档记录的历史状态"之间的落差到底是内容错误、还是环境被并发操作污染,这是纯 Python 系列(每次都在干净的进程里跑)完全不会遇到的新问题。
