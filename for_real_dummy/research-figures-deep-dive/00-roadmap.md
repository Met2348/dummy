## 这个系列是什么、不是什么

**是什么**:面向"完全没有画过投稿级学术图表"的读者,讲清楚"这张图该用什么类型""这个颜色为什么不能用""
这几个子图该怎么排版""这个架构图该用什么工具""图注该写什么""常见的坑长什么样"——每一条判断背后都有
真实可查的调研依据(色觉研究、Tufte 的原始论述、真实的"图表罪案"案例、matplotlib 官方文档),不是凭个人
审美泛泛而谈。

**不是什么**:不是 matplotlib API 用法大全(不会逐一介绍每个函数的每个参数),不是设计美学鉴赏课。全篇
判断力最终都要落到两件事上:①能不能找到真实调研依据 ②能不能用真代码把这个判断"跑出来"看到差异——
后者是这个系列区别于"科研写作""投稿评审"等系列的地方:那几个系列的判断力大多没有"运行代码"这个抓手,
这个系列几乎每一条判断都能直接画两张图对比着看。

**和 `research/world-model-imagination-controller/`(用户真实在研、即将投稿 ICLR 的项目)的关系**:
这是用户在导师 Weikai Lin 指导下的真实研究项目,测的是"世界模型测试时想象预算怎么自适应分配"。撰写本
系列前已实地读取该目录下 `01-meeting-briefing.md`(会前简报,含真实 pilot 数据:想象深度 H、候选数 K
扫描出的决策改变率/命中率表)和 `02-deep-gap-analysis.md`(二次深挖,含"想象要划算必须有信息优势"这条
核心发现)。[07 号教程体文件](07-build-a-mini-publication-figure.md)的核心结果图借用这个项目**真实的
场景设定**(想象预算 vs 任务表现、"有没有任务相关信息优势"决定想象是否划算)来设计一张"这类论文真会画
出来"的图,但图上的具体数字是本系列自己生成的合成数据(有明确随机种子、形状经过 sanity check,不是凭空
乱画),不是抄项目里的真实实验数字——项目自己的真实数字留给项目自己的论文用,教学案例只借场景,不借
结论,这是设计文档"跨系列共同纪律"第 2 条的要求。

## 验证方法论差异声明(必须读,这是本系列和其余"深挖系列"最大的不同)

仓库里已有的 12+ 条"深挖系列"(torch-deep-dive、numpy-deep-dive、kernel-gpu-deep-dive……)全部是**纯
文本输出**的验证方式:每个可运行例子最终用 `assert` 精确核对一个数值、一个字符串、一个数据结构——因为
它们的"产出"本身就是数字/对象。

这个系列是仓库第一次系统性地产出**图片文件**(PNG/PDF/SVG)而不是纯文本。这带来一个必须诚实说清楚、
不能含糊带过的方法论差异:

**做不到的事**:不能像其余系列那样 `assert 生成的图和期望的图像素级相同`。原因不是偷懒,是这个 assert
本身没有意义——同一段 matplotlib 代码在不同操作系统、不同字体渲染引擎(FreeType 版本、有没有装某个
字体)、甚至不同 matplotlib 补丁版本下,渲染出的像素可能有细微差异(抗锯齿、字体 hinting),但图表表达
的信息完全没有变化。锁死像素级 assert 会让这个系列每次换机器跑都可能"假失败",这是错误的验证纪律,
不能为了"看起来和其它系列一样严格"就硬装。

**做得到、本系列实际采用的三层验证**:
1. **文件真的生成了**——`os.path.exists(path)` + `os.path.getsize(path) > 0`(拿到的不是一个空文件/
   写入失败的残骸)。
2. **尺寸/格式元数据符合预期**——位图(PNG)用 `PIL.Image.open(path)`,核对 `.size`(像素宽高,精确
   等于 `figsize(英寸) × dpi` 换算出的整数)和 `.info["dpi"]`(核对确实按目标 DPI 导出,不是不小心用了
   默认的屏幕 DPI);矢量图(PDF/SVG)没有"像素尺寸"这个概念,改成核对**文件格式签名**(PDF 文件开头
   精确是 `b"%PDF-"`,SVG 文件开头包含 `<?xml`/`<svg` 标签)+ 文件大小合理(不是几字节的空壳)。
3. **能用真代码计算的"感知/设计"结论,单独算出数字来 assert**——比如"jet 的感知亮度不是单调的,viridis
   是"这条色彩科学结论,不满足于引用文献,自己把两个 colormap 采样 20 个点、按 ITU-R BT.709 相对亮度
   公式算出亮度序列、`assert` viridis 的亮度序列单调不减、jet 有多处不满足单调——这是"设计判断"和"仓库
   assert 验证纪律"少有的能真正对上号的地方,能算的一定真算,不能算的(比如"这张图好不好看")才交给
   人眼对比截图。

**关于 DPI 元数据的一个真实发现(撰写时实测,不是凭空写的注意事项)**:`fig.savefig(path, dpi=300)`
生成的 PNG,用 `PIL.Image.open(path).info["dpi"]` 读回来,实测结果是 `(299.9994, 299.9994)`,不是精确
的 `(300.0, 300.0)`——因为 PNG 文件格式的 `pHYs` chunk 内部存的是"每米像素数"这个整数字段,300 dpi 换算
成像素/米、再换算回 dpi 存在浮点圆整误差。这意味着但凡要在自己的代码里断言 DPI,必须用 `round()` 或者
容差比较,不能用 `== 300`,下面 [03 号文件](03-multi-panel-layout-engineering.md)会带着这个真实数字
展开讲。

**这条声明本身也回答"这个系列的可运行例子到底在验证什么"**:不是验证"这张图长得好不好看"(这件事本身
无法用 assert 回答,最终要靠人眼看截图判断,本系列撰写时每张关键图都用 Read 工具真的看过一遍,不是盲
生成),而是验证"这段能生成图片的代码本身没有写错、图片真的按预期规格产出了、以及一部分设计判断能被
量化的地方,量化的结论和文献结论一致"。这三件事合起来,是这个系列诚实的验证颗粒度。

## 环境声明

运行环境:仓库根目录 `.venv`(Windows 原生)。撰写时实测版本:

| 包 | 版本 |
|---|---|
| Python | 3.13.9 |
| matplotlib | 3.10.9 |
| numpy | 2.4.6 |
| Pillow(PIL) | 12.2.0 |

三个包都是这个 `.venv` 里已经装好的依赖,不需要额外 `pip install`。全篇代码纯 CPU、秒级跑完,不需要
GPU。所有生成的图片文件统一存到本系列目录下的 `_assets/` 子目录(不写进仓库根目录,也不写进
`for_real_dummy/` 顶层,保持仓库整洁)——每个可运行例子的代码块里都会显式 `os.makedirs(ASSETS, exist_ok=True)`
再保存,这是因为 [`_verify_md.py`](_verify_md.py) 把每个 ` ```python ` 代码块单独拎出来起一个新的
Python 子进程独立执行(块与块之间不共享任何变量,包括"目录已经建好了"这件事),所以每个需要落盘的代码块
都要自己保证目录存在,不能依赖前一个块已经建过。

## 知识点结构模板(七步,和 torch-deep-dive/kernel-gpu-deep-dive 完全一致,第 6 步换了名字)

1. 签名/是什么
2. 一句话
3. **底层机制/为什么这样设计**(这里具体是**视觉感知原理**——人眼怎么处理颜色/亮度/角度/透视,不是
   "调库参数是什么")
4. AI 研究场景(具体到"写 ICLR/NeurIPS 这类论文时,这条判断会在什么地方用上")
5. 可运行例子(真在 `.venv` 里跑过,产出真实图片文件,验证方式见上面"方法论差异声明")
6. **审稿人/读者会怎么挑刺**(替换掉其余系列的"面试怎么问"——这里的读者是审稿人,不是面试官,追问链
   问的是"这张图会不会被 reviewer 抓到把柄",不是"这个概念背不背得出来")
7. 常见坑

## 收尾阶段的一处修复(诚实记录)

撰写这条系列的 agent 在最后一轮"重新扫描确认干净"的自查途中撞上 API 5 小时限额被切断——但独立核实
后发现全部 7 个内容文件 + roadmap + 全部图片资产实际上已经写完,只差这最后一步自查。核实时用主会话
自己独立重新执行,发现自查本身是对的:`02-color-and-perception.md`、`03-multi-panel-layout-engineering.md`、
`04-architecture-and-flow-diagrams.md`、`06-anti-patterns.md` 四个文件里,一共 22 处 `print()` 语句的
字符串字面量混用了中文说明文字——这类内容按仓库既定纪律("Windows-GBK print() 必须纯 ASCII",数值/
状态类的 `print()` 输出如果混了中文,在这台机器的 GBK 控制台编码下会显示成乱码,即使 `_verify_md.py`
本身因为只检查有没有 Python 异常、不检查文字是否可读而依然判定"[ok]")本该在撰写阶段就避免,这次是
在最终提交前由主会话核实时发现并直接修复(22 处 `print()` 全部改成纯 ASCII 英文说明,数值断言逻辑
本身完全没有改动,改前改后 `_verify_md.py` 复测结果一致)。这条经验和同批次 `research-release-deep-dive`、
`paper-submission-deep-dive` 两个系列在自己撰写阶段发现的同一类问题是同一件事的三次独立复现,值得
任何以后在这台机器上给本仓库写"真实可执行 Python 代码块"的人提前知道:**只要代码块会被直接执行
(不只是 `_verify_md.py`,也包括读者自己复制出去跑),`print()` 语句本身的字符串字面量必须是纯 ASCII,
中文解释放在代码块外的正文里。**
