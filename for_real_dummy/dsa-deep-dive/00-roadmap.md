## 复杂度验证方法论(本系列独有,区别于其余六条系列的核心方法论差异)

**核心原则:复杂度不是断言出来的,是真的测出来的。** 用 `time.perf_counter()` 在多组输入规模上真实计时,而不是像大多数教材/题解那样写一句"这是 O(n log n)"就结束。

**输入规模选择需要匹配复杂度量级**(否则要么测不出趋势,要么等太久):
- O(n) / O(n log n) 类:用较大规模(数万到数十万),避免 Python 解释器固定开销掩盖真实增长趋势。
- O(n²) 类:用中等规模(数千),避免单次验证耗时过长。
- O(2^n) 指数级(如朴素子集枚举、朴素回溯无剪枝版本):用很小规模(15~20)。

**容差要足够宽。** 比如 n 增大 5 倍,O(n) 允许耗时增长落在 2~10 倍这个区间,不追求精确复现理论系数——Python 解释器本身有真实、不可忽视的计时噪声(GC 时机、其他进程抢占 CPU、JIT warm-up 都会引入波动)。**如实标注这个方法论局限:本系列的复杂度验证只验证"增长趋势的量级方向"(比如确认 O(n²) 的实际耗时增长明显快于 O(n log n)),不是精确复现渐进复杂度的具体系数。** 这和 huggingface-deep-dive 系列"显存测量必须交叉核对 `torch.cuda.max_memory_allocated()` 和 `nvidia-smi`"是同一种"不轻信单一理论说法、用真实测量交叉验证"的纪律,只是验证对象从"库的真实行为"换成了"算法的真实复杂度表现"。

**验证脚本**:每类知识点写完后,用复用自 [huggingface-deep-dive/_verify_md.py](../huggingface-deep-dive/_verify_md.py) 的模式(独立提取每个 ` ```python ` 代码块、单独 subprocess 执行、汇总 pass/fail)做最终验证。本系列纯 Python 标准库,不需要 `PYTHONUTF8` 这类环境变量前置声明(没有 trl 那种编码坑)。

**全库终验阶段追加的真实教训(容差宽度不是唯一变量,采样质量同样关键):** 01 类知识点 1、8 的计时断言在全库终验阶段(单文件复测 20~30 次)里暴露过间歇性失败——根因不是容差不够宽,而是被测操作本身规模选得太小(如"链表头部命中"、字符串拼接 n=2000),真实耗时落在个位数微秒级,任何一次系统调度噪声(哪怕只有几十微秒)都足以让耗时膨胀几倍甚至几十倍,再宽的容差区间也扛不住这种量级的相对噪声。修复方式不是继续放宽阈值(那样会削弱断言本身的区分力度,失去验证意义),而是引入 `best_of(fn, trials=3~5)`(多次采样取最小值——噪声只会让耗时偏高,取最小值能有效逼近真实下界)和/或把过小的输入规模适度调大。**方法论结论:规模选择本身也要满足"绝对耗时不能小到被解释器噪声淹没"这个隐性前提,不是只有"增长比例的容差区间"这一个自由度可以调;确认"稳定"不能只复测 1~3 次就下结论,本系列最终对每个文件做了 5~20 次不等的重复验证才捞出这两处间歇性问题。**
