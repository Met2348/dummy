B. Mathematical Modeling of the Multi-Agent System
LLMs power a wide variety of applications (Cao et al., 2026; Zeng et al., 2025). Specifically, in a formal sense, we
characterize an autonomous LLM-powered entity as a quintuple E = ⟨B, P, A, Ω, Γ⟩. Here, B represents the core linguistic
brain (the LLM), P signifies the perceptual manifold of the target environment, and A denotes the repertoire of executable
operations (tools). The mapping function Ω : P → A acts as the decision logic, while Γ encapsulates the persistent memory
state. The operational trajectory of such an entity follows a recursive observe-deliberate-act cycle: at any discrete temporal
index k, the entity ingests a sensory input pk ∈ P, leverages B to synthesize a strategic intent based on Ω, and dispatches
an intervention ak ∈ A to modulate the system state. To tackle the intricate state spaces of consensus implementations,
we extend this individual paradigm into a collaborative collective G = {E1 , E2 , . . . , Em , X }. In this formulation, each Ei
represents a specialized functional unit, and X defines the coordination medium that facilitates information exchange and

                                    Protocol Constraints & BFT Attack Model

   Protocol Type Constraint: BFT (Byzantine Fault Tolerant)
   This model assumes an environment capable of tolerating Byzantine faults, where nodes may behave maliciously or
   arbitrarily.
   I. CFT-type Attack Vectors (Crash Faults):

      • Availability: Node crash, recovery, and frequent restarts.
      • Network: Partitions, significant latency, packet loss, and message reordering.
      • Logic: Boundary condition exploits and duplicate message utilization.

   II. BFT-specific Attack Vectors (Malicious Behavior):
      • Equivocation: Sending conflicting votes/proposals for the same round/height to different peers.
      • Payload Tampering: Injecting invalid, malformed, or tampered protocol messages.

      • Selective Broadcast: Distributing messages to a subset of nodes to induce state inconsistency.
      • Strategic Withholding: A malicious leader intentionally delays or withholds proposals to stall liveness.
      • State Forgery: Crafting fraudulent certificates, votes, or cryptographic signatures.
      • Timing Attacks: Exploiting vulnerabilities within the protocol’s timing and timeout assumptions.

   III. Attack Mindset & Objectives: Assuming control of up to f malicious nodes (f < n/3), the attacker aims to:



                                                            15
           Agora : Toward Autonomous Bug Detection in Production-Level Consensus Protocols with LLM Agents


     1. Safety Violation: Can honest nodes be coerced into accepting an invalid state?

     2. Liveness Violation: Can the system be prevented from reaching consensus indefinitely?
     3. Consistency Violation: Can different honest nodes be led to perceive divergent “truths”?

   Note: Effective testing must simulate a clever attacker who strategically exploits the protocol’s specific logic
   weaknesses rather than just random fault injection.




                                     Protocol Constraints & CFT Fault Model

   Protocol Type Constraint: CFT (Crash Fault Tolerant)
   This model assumes an environment that can only tolerate crash faults. All nodes are assumed to be honest; they
   either follow the protocol correctly or stop functioning entirely.

   I. Allowed Attack Methods (Non-Malicious Faults):
      • Node Dynamics: Random node crashes (immediate stop) and restarts (recovery with potential loss of in-memory
        state).
      • Network Adversity: Network partitions, extreme communication delays, message loss, and packet reordering.

      • Implementation Limits: Boundary conditions (overflows, null values, extreme inputs) and duplicate resource
        utilization.

   II. Prohibited Attack Methods (Byzantine Behavior):

      • No Malicious Messages: Sending forged, tampered, or intentionally malformed payloads is strictly prohibited.
      • No Equivocation: Double voting or sending conflicting proposals in the same round is not allowed.
      • No Selective Broadcast: Intentional inconsistency through partial message distribution is excluded.

      • No Identity Theft: Actions involving signature forgery or impersonating other nodes are disallowed.

   III. Core Assumptions: In a CFT environment, the testing focus shifts from ”malice” to ”reliability”:
     1. Binary State: Nodes are either ”Correct” (following the protocol) or ”Failed” (stopped). There is no ”half-dead”
        or Byzantine behavior.

     2. Honest Participation: All participating nodes are assumed to act in good faith to reach a common consensus.
     3. Recovery Consistency: The system must maintain safety and eventually regain liveness after network partitions
        heal or nodes recover.

   Note: Testing for CFT protocols should exhaustively simulate unstable infrastructure and edge-case logic rather than
   adversarial intent.

     CmdA . Final state: x = 1.

     visible to the user.
   • Risk Level: High in high-concurrency environments with non-persistent deployments.
   • Feasibility: Highly likely if network latency is asymmetric and nodes do not use synchronous disk writes for
     every PreAccept.

   • Persistence: Enforce stable storage for all metadata changes to prevent state loss upon reboot.

   • Implicit Trust: The function assumes the QuorumCert provided is internally consistent, even though the
     View field in the QC is not cryptographically bound to the signature in this specific implementation.

  2. Acquisition: A Byzantine node obtains a valid QC certifying Block A from View v-k.

  3. Tampering: The attacker constructs a fakeQC with the original valid signature for Block A but an inflated
     View: v + 1000.
  4. Replay: The attacker sends this fakeQC to an honest node.

     The node now erroneously believes the highest certified view is v + 1000, causing it to reject legitimate
     messages from the actual current view.

   • Category: CWE-20: Improper Input Validation (Cross-field consistency check missing).

     if qc.View() != newBlock.View() {
     return false, fmt.Errorf("QC view mismatch")
