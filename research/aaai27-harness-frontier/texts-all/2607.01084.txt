A. Sandbox Environment
Since live APIs and MCP-based tool ecosystems are often unstable, version-dependent, and difficult to control during
evaluation, we build a local code-running sandbox to simulate tool calls. Specifically, we create a Python sandbox
environment class that includes database, tools implementation, and tool call module. Detailed information is as follows:

A.1. Database
We built our experimental database using public Point of Interest (POI) data from Beijing. Each POI entry has five key
attributes: Phone Number, Address Name, Adname (Administrative Name), Coordinates (Latitude/Longitude), and Category.
To strictly prevent data leakage and ensure the model relies entirely on tool usage rather than parameter memory to answer,
we adopted an anonymization protocol. Specifically, all Address Names and Phone Numbers are replaced with unique,
randomly generated alphanumeric strings and non-repeating number sequences, respectively. In addition, we strictly separate
the training and test databases to avoid the model guessing the answers correctly by chance. Examples of the database data
are shown in Table 2.

A.2. Logical Architecture and Function Classification of Tools
We designed the logical architecture of the tools in accordance with the logic diagram (Figure 8) and the following rules:

  3. For the distance calculation function: we only allow latitude and longitude as input parameters;

  4. For non-key information queries: we allow any three key information to be used as input.

A.3. Tool Call Module
We set up the different tool call modules for RL training and evaluation testing in two different ways:

   • For RL training, we use the Verl Tool framework. The core idea is to run the code sandbox as a service on the server
     and open an external port. This port receives tool names and corresponding parameters, then sends back the tool call
     results. During each RL training run, we use the terminator <tool call >to detect tool call commands. Specifically,
     when the model outputs this keyword, it stops generating text and triggers the tool detector. After processing all data in
     the same batch, the tool detector accesses the port uniformly to get results. These operations ensure only one tool call

                                                                  16
               Can Agents Generalize to the Open World? Unveiling the Fragility of Static Training in Tool Use

     command is formed per conversation turn. The rest of the reinforcement learning process is the same as normal RL
     training.

   • For evaluation testing, we simulate multi-turn conversations and also detect tool calls via the terminator, which ensures
     that only one tool call is triggered in a single turn. If the model outputs </answer >, it means the response is finished.

The final dataset comprises a total of 6,930 samples, divided into a training set (N = 6, 050) and a test set (N = 880). The
dataset characteristics are designed to ensure rigorous evaluation:

                  • Complexity Distribution: Both the training and test sets follow an identical distribution regarding the length of tool
                    chains (e.g., 1-hop, 2-hop, ..., k-hop). This ensures that the test set accurately reflects the structural complexity of the
                    training data.

                  • Database Isolation: To eliminate the possibility of data leakage, the training and test sets are synthesized from disjoint
                    subsets of the underlying POI database. Consequently, the model must generalize its reasoning capabilities to unseen
                    entities rather than recalling specific POI attributes seen during training.

Regarding task complexity distribution, the dataset employs a stepped logical chain design covering a range from one to
six steps; two-step tasks serve as the core backbone (accounting for 28.4%), while nearly 40% of the samples consist of
high-difficulty reasoning tasks involving three or more steps, ensuring depth in the model’s learning of long-range tool
invocation paths.
In terms of tool and scenario distribution, the dataset exhibits a high degree of domain focus and balance: the core
tool caller id lookup represents 24.4% of invocations, while ticket price calculation (adult/child cost) and geographic
measurement (coords/distance) tools work in close coordination to form the foundation of task execution. This distribution
is further corroborated by the source scenarios, where price-related (34.1%) and distance-related (28.4%) contexts dominate,
providing the model with abundant material for numerical reasoning and logical alignment.
Overall, this dataset covers basic attribute queries (e.g., POI names, phone numbers, and administrative divisions) and
utilizes complex step distributions to force the model into semantic grounding rather than simple symbolic matching during
extreme testing, such as anonymization or logical inversion. With its scientific tool sampling proportions and diverse
business scenarios, this dataset serves as an ideal benchmark for training agents equipped with efficient path planning and
robust generalization capabilities. In addition, we also provide the tools that can be used in Table 3.

B. Experimental Details
We adopt Qwen2.5-7B-Instruct as the backbone model and perform full-parameter fine-tuning based on the LLaMA-Factory
framework. The AdamW optimizer is employed, with the learning rate and batch size set to 3e-5 and 2, respectively. The
training process runs for a total of 800 steps, with a model checkpoint saved every 200 steps. For the RL paradigm, training
leverages the GRPO tool calling algorithm implemented on the Verl-Tool platform. Notably, we use the normal GRPO
to keep comparison fairly. The AdamW optimizer is also used here, with the learning rate set to 1e-6, the number of
deployments configured to 8, and the total training steps set to 280. The PAFT algorithm differs from SFT only in the

training dataset. The reward function was designed as follows:

C. Implementation Details of OpenAgent Evaluation Task
C.1. Tier 1: Perception Generalization
C.1.1. I NSTRUCTION ROBUSTNESS .
• Instruction Blurring: We employ API calls to rewrite user queries, introducing complexities through syntactic variations
  (e.g., colloquialisms, rhetorical questions) or semantic ambiguities (e.g., implicit references, entity omissions).
• Information Redundancy: We employ API calls to rewrite user queries by embedding core instructions within irrel-
  evant contexts or lengthy background details, thereby simulating real-world informational noise. Please refer to the
  Appendix E.1.1 for further details.

C.1.2. S CHEMA A DAPTABILITY.
We alter the parameter dictionary key from the standard args to arguments, as shown in Appendix E.1.2.

C.1.3. S EMANTIC G ROUNDING .
Synonymous Rewriting We assess the model’s capability to decouple functional intent from specific symbolic labels by
replacing tool identifiers with semantically equivalent alternatives. This requires the model to move beyond simple string
matching and demonstrate a deep understanding of the underlying natural language semantics.

                                        Table 4. Tool Mapping for Synonym Rewriting

                 Original Tool Name                      →     Synonymous Variant
                 calculate distance by coords            →     calculate separation via positions
                 search map coordinates                  →     lookup location latlong
                 get adult cost by name                  →     fetch grownup price using name



Noise Injection While maintaining the integrity of the tool schema, we introduce subtle syntactic perturbations into the tool
names by inserting articles or determiners. We utilize this setup to evaluate the model’s tolerance to minor fluctuations in
API naming and its stability when encountering non-standardized inputs.

                                           Table 5. Tool Mapping for Surface Noise

                  Original Tool Name                      →        Noisy Variant
                  calculate distance by coords            →        calculate the distance by coords
                  search map coordinates                  →        search for map coordinates
                  get adult cost by name                  →        get 1 adult cost by name



Semantic Adversarial We designed a “description trap” where the familiar tool names are retained, but their functional
descriptions are modified to explicitly forbid their previous usage. We require the model to overcome “path dependency” on
familiar names, recognize the obsolescence of the original tool based on the updated description, and accurately invoke a
new tool with matching functionality.

randomly generated UUIDs. In this extreme setting, we force the agent to perform reasoning based exclusively on the

C.2. Tier 2: Interaction Generalization
C.2.1. F ORMAT A DAPTABILITY.
We diversify the structure of tool return values, transitioning from strictly structured JSON to less constrained formats such

C.2.2. E RROR C ORRECTION .
   • Error Return: We simulate an explicit failure where the primary tool returns a specific error message (e.g., “This tool
     is unavailable now”) or a traceback. This scenario evaluates whether the agent can interpret explicit error signals and
     proactively switch to a redundant tool to resolve the blockage.

   • Null Return: We introduce an implicit failure where the primary tool returns a null value or an empty set (∅) without
     any diagnostic warning. This tests the agent’s capability to detect anomalies based on the absence of content, rather
     than the presence of an error signal, and infer the necessity of a fallback execution.


                            Table 9. Mapping of Tool Failure Modes and Expected Agent Responses

                         Null Return                              ""



C.2.3. DYNAMIC A LIGNMENT.
• Value Redirection: We simulate a scenario where the provided parameter logic is deprecated, and the environment returns
  a message including new, valid parameters.
• Tool Redirection: We simulate a scenario where the primary tool returns an error explicitly stating it is obsolete while
  simultaneously suggesting an alternative tool.
Importantly, we feed the interaction history into the API model (DouBao) to enable it to judge whether the target model
refuses to answer.

we kept the tool interface unchanged, but significantly adjusted the calculation ratio in the Prompt in the test set to examine
whether the model relies on the memory from the training stage or can perform real-time reasoning based on immediate
instructions.

C.3.2. PATH P LANNING .
Trace Simplification We implement Tool shortcut to evaluate the agent’s ability to optimize its reasoning trajectory when

Logic Inversion We simulate causal inversions in the dependency figure 8. During the testing phase, we intentionally
reverse the information acquisition order described in the documentation, effectively inverting the preconditions of the tools.

the output of Tool B.

                caller id lookup: Tel → Name                  →      search tel by name: Name → Tel

C.4.1. ACTIVE R EFUSAL

from being “inevitably solvable” to “deterministically unsolvable.” We utilize this setup to assess the model’s capa-
bility to recognize task boundaries—specifically, whether it can choose to proactively refuse the request rather than
generating hallucinations when faced with information deficiency or execution deadlocks. The error tools include
calculate distance by coords, search map coordinates, and get adult cost by name.

F. Real-World API Validation
To provide direct empirical evidence, we replace the sandbox tool calculate distance by coords with a real-world
driving-distance API provided by Amap 1 , substituting both training and test data with real API traces, and retrain both SFT
and RL models under identical protocols. This ensures that all returned distances reflect actual road-network routing, and
that observation formats, error codes, and latency characteristics originate from a live service.
Figure 10 presents the full results. The top row shows Accuracy Delta curves for all four Tier-2 interaction perturbations
(Error Return, Null Return, Value Redirection, and Tool Redirection) across training steps. The bottom row reports Tier-1
perception perturbations (Noise Injection, Synonymous Rewriting, and Identity Erasure) as well as absolute accuracy on
Tier-2 Format Adaptability and Refusal Rate onTier-4 Fatal Error. Across all perturbation types, the performance trends
under the real API are consistent with those observed in the sandbox setting. Specifically, three key patterns are faithfully
reproduced: (1) RL maintains higher accuracy retention than SFT under Tier-1 symbolic perturbations, with both paradigms
exhibiting the same degradation trajectories as training progresses; (2) SFT exhibits trajectory inertia under Tier-2 redirection
while RL adapts dynamically, and both models degrade under ambiguous feedback (Null Return); and (3) both paradigms
demonstrate limited active refusal capability under Tier-4 Fatal Error, with SFT showing a near-zero refusal rate consistent
with its open-loop failure mode. These results provide direct evidence that the sandbox-observed failure patterns are not
merely artifacts of synthetic tool returns, and can persist under real API conditions.
