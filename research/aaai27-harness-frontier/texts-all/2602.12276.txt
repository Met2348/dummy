A. Agent Design Decisions
A.1. Action Space
Our agents use a minimal action space with 8 tools designed for web navigation:

   • click(element id): Click on a DOM element by its unique identifier
   • type text(element id, text): Enter text into an input field without submitting
   • hover(element id): Hover over an element to reveal tooltips or dropdown menus
   • scroll(direction): Scroll the page up or down to reveal more content
   • select dropdown option(element id, value): Select an option from a dropdown menu
   • search(element id, text): Type text and submit (equivalent to type + Enter)
   • go back(): Navigate to the previous page in browser history
   • exit(message): Terminate the episode with a final answer or status message

All element-targeting tools require an element id parameter that corresponds to an injected integer id attribute in the
DOM. This simplifies element identification compared to CSS selectors or XPaths.

A.2. Error Handling and Validation
The agent framework includes several error checks that provide feedback during the retry loop:

   • MustCallExactlyOneToolCheck: Ensures exactly one function call per step
   • InvalidActionSchemaCheck: Validates that function arguments match the expected schema
   • ElementMustExistCheck: Verifies that referenced element IDs exist in the current DOM
   • MustProvideReasoningCheck: Requires the agent to provide reasoning in the analysis channel
   • RepeatingActionLoopCheck: Detects and prevents repeated ineffective actions

When a check fails, the agent receives an error message and has up to 5 retry attempts to produce a valid action.

A.3. Conversation Format
Each agent turn follows a structured format with the GPT-OSS Harmony interface:

  1. System message: Establishes agent role and channel usage
  2. Developer message: Provides detailed instructions and constraints
  3. User message: Contains the task, current HTML state, and any error feedback
  4. Assistant response: Analysis in the analysis channel, function call in commentary
  5. Tool observation: Result of the action with updated HTML state

The conversation history preserves prior reasoning and actions but only includes full HTML in the most recent tool
observation to manage context length. This allows the agent to reference its own reasoning trajectory while grounding
decisions in the current page state.

B.4. External Planner (Plan-and-Act)
For the Plan-and-Act baseline, we use a two-agent system where an external planner generates high-level plans and a
separate executor agent follows them. The planner’s system prompt establishes its role:

    You are a web-navigation agent that normally completes browser tasks by thinking
    in an analysis channel and then emitting exactly one function call (using the
    provided tools) per step.
    In this role, you are acting as the Planner in a two-agent system:
    - You see the overall User task, your earlier plans, the executed tool calls, and
     the latest HTML snapshot (inside the most recent tool observation).
    - A separate Executor agent will later read your plan and actually perform the
     function calls in the browser.
    For this planner turn:
    - Use the analysis channel for deep, step-by-step reasoning and scratchpad.
    - Use the final channel to output a clear, concrete plan for the next actions.
    - Do not actually call tools in this turn; only describe how the tools should be
     used in the future steps.

The developer prompt provides additional guidance on plan quality:

    Planner reasoning guidance:
       • Think step by step and deeply in your own natural style.
       • Focus on the next several actions and how to verify progress on the page.
       • Ground your plan in: the user task, prior plans, executed tool calls and their
         results, the latest HTML.
       • Avoid unsupported assumptions about elements or states that you have not
         observed.
    Recovery and loop-avoidance:
       • Do not repeat plans or actions that failed or produced no state change.
       • When something did not work, explicitly include recovery steps: different id,
         scroll, search, hover, go back, or another sensible control.
       • Clarify how the Executor should adapt if the page does not look as expected.
    Referencing future actions:
       • Refer to tools by their names (for example: click, type text, hover, scroll,
         select dropdown option, search, go back, exit), as they appear in the tool
         schemas.
       • Describe how the Executor should pick the target element: use the integer
         id from the latest HTML when known, otherwise reference robust cues (labels,
         nearby text, attributes, structural context).
       • Predict how the DOM or URL should change after key actions.
       • Specify what the Executor should look for to confirm success vs. failure (new
         headings, result lists, modals, disappearance of buttons, etc.).


C. Benchmark Characteristics and LLM-as-Judge Reliability
Benchmark difficulty comparison. WebArena-Lite and the GoBrowse-style benchmark differ substantially in task
difficulty and evaluation methodology:

  • WebArena-Lite (165 tasks): Uses programmatic success checks that verify specific page states or database entries.
    Tasks often require longer trajectories with precise multi-step reasoning. Average trajectory length is approximately
    8–12 steps, and success rates are generally lower (40–47% in our experiments).
  • GoBrowse-style (341 tasks): Uses an LLM-as-judge evaluation protocol. Tasks tend to be shorter and more straightfor-
    ward, with average trajectory lengths of 4–6 steps. Success rates are substantially higher (86–90% in our experiments),
    reflecting easier task specifications.

The higher baseline performance on GoBrowse means there is less room for improvement from inference-time scaling
methods. Nevertheless, CATTS achieves consistent gains on both benchmarks, suggesting the underlying principles
generalize across difficulty levels.

LLM-as-judge reliability. For the GoBrowse-style benchmark, we employ the LLM-as-judge evaluation protocol intro-

While LLM-as-judge evaluation introduces some noise compared to programmatic checks, the 90% agreement rate suggests

pattern observed on both benchmarks (WebArena-Lite with programmatic evaluation and GoBrowse with LLM-as-judge)
provides additional confidence in our findings.

J. Failure Node Examples
This section provides a concrete example from our experiments where the arbiter’s override of high-consensus decisions led
to task failure. Theis example illustrate the core failure mode discussed in Section 4: when the vote distribution already
exhibits strong agreement (high margin ∆t , low entropy Ht ), arbitration introduces override risk rather than improving
decision quality.
Figure 7 presents an example from a WebArena-Lite task requiring the agent to find “Meat Substitutes” in an online grocery
store. At a critical decision point, the sampled candidates show 90% consensus for “Scroll Down”, the correct action, since
the target category lies below the current viewport. However, the arbiter overrides this strong consensus and selects “Click
Pantry Staples,” a minority action that navigates to an incorrect category and derails the trajectory.
This failure pattern is particularly instructive because the vote distribution provides a clear signal: with 9 out of 10 candidates
agreeing on “Scroll,” the normalized entropy is low (Ht ≈ 0.3) and the margin is high (∆t = 0.6). Under CATTS, this step
would be classified as high-confidence and the arbiter would be bypassed entirely, preserving the consensus action. The
example thus demonstrates both (i) why uniform arbitration can be harmful, and (ii) how vote-derived uncertainty enables
selective intervention.
