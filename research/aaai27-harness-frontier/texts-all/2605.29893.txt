   Redundancy Type         Explanation
   Abnormal Step           The step invokes a tool, but the tool call fails due to accidental issues
                           such as network error.
   Duplicated Step         The step invokes a tool whose name, input arguments, and output are
                           identical to those of a previous tool call, thereby providing no additional
                           information gain.
   Incorrect Step          The step invokes a tool that is irrelevant to the task objective or cannot be
                           executed properly. For example, when a user requests a flight cancellation,
                           the agent calls a tool for checking mobile data usage; or the agent invokes
                           a user-information tool before obtaining the required user ID from the user.
   Exploratory Step        The step invokes a tool as part of an exploration process toward task
                           completion, but the call does not directly contribute to achieving the final
                           objective. For example, when the user requests cancellation of flight
                           FA-1003, the agent queries both FA-1002 and FA-1003, where querying
                           FA-1002 serves only as an exploratory step.

A   Annotation Guideline
In Figure 4, we present our annotation guidelines used in trajectory annotation procedure. This guideline
defines different redundancy categories, necessary behaviors, and boundary cases, which provides a
unified framework for annotating redundant steps in agent trajectories.



                                                       Annotation Guideline

     • Rules
     1.   Each case contains an original trajectory file (including the task description and the agent–user interaction trajectory) and an
          automatically annotated file (including the indices and types of redundant steps in the trajectory).
     2.   Please revise the automatic annotations based on the original trajectory data and classify each step as either redundant or necessary.
          Note that only agent tool-calling steps should be considered; dialogue messages and user tool-calling steps should be ignored.
     3.   If a step (i.e., an agent tool call) is redundant, label the current step as redundant, record the indices of both the current step and the
          following step (the tool execution result), and include them in the annotation file. If a step is necessary, label it as necessary and
          remove the indices of the current step and the following step (the tool execution result) from the annotation file (i.e., do not record
          them).
     4.   If you are uncertain whether a step is redundant or necessary, label it as redundant and set the confidence level to “uncertain”.



     • Category Definitions
     1. Redundant Steps
          Steps that consume computational resources but do not contribute to task completion.
              ① Abnormal Tool Call
                   The step invokes a tool, but the tool call fails due to external issues such as network errors.
              ② Repeated Tool Call
                   The step invokes a tool whose name, input arguments, and output are identical to those of a previous tool call, thereby
                   providing no additional information gain.
              ③ Incorrect Tool Call
                   The step invokes a tool that deviates from the task objective or cannot be executed properly.
                   For example:
                    a) The user requests a flight cancellation, but the agent calls a tool for checking mobile data usage.
                    b) The user requests a flight cancellation, but before obtaining the required user ID, the agent calls a user-information
                         query tool that requires the user ID as input.
              ④ Exploratory Tool Call
                   The step invokes a tool as part of an exploratory process toward task completion, but the call does not directly contribute to
                   the final objective. For example, if the user requests cancellation of flight FA-1003, but the agent queries both FA-1002 and
                   FA-1003, then querying FA-1002 is considered an exploratory step.

     2. Necessary Steps
          Steps that directly contribute to task completion.
              ① Necessary Tool Call
                   The step invokes a tool that is required for completing the task.
                   For example, when the user requests a flight cancellation, the agent calls a tool to query user information.
                   You may refer to the ground-truth actions provided in the task information of the original trajectory file, although these
                   actions may be incomplete or contain errors.
              ② Goal-Aligned Tool Call
                   The step invokes a tool that is consistent with the task objective.
                   For example, if the task requires transferring the user to a human agent when the request cannot be handled automatically,
                   then calling the transfer_to_human_agent tool should be considered necessary.

     3. Boundary Cases
              ① If a step invokes two tools, where one tool call is necessary and the other is redundant, the entire step (together with its
                execution result) should be labeled as redundant.
              ② If a tool call appears in the ground-truth actions but some tool arguments do not exactly match (e.g., book_reservation
                or transfer_to_human_agent), while the execution result is still correct, the step should be labeled as necessary.
                This is because:
                 a) some tools have default arguments,
                 b) some arguments may be omitted by the agent,
                 c) and some arguments are only descriptive and do not affect execution.
              ③ If the trajectory skips certain steps, resulting in missing necessary tool calls or execution results while subsequent steps still
                execute normally, the entire data sample should be removed.
                This may indicate data leakage, where the LLM has already seen the sample during training.

B     Algorithm Details
We define the following notation used in the algorithms. Let Q denote the query provided to the system.
T = {t1 , t2 , . . . , tn } denotes the trajectory consisting of n steps where each entry ti specifies the utterance
or action taken at time step i by one agent. R ⊆ {1, 2, . . . , n} denotes the set of redundant steps identified
by the model.

B.1    Details of One-to-One

Algorithm 1 One-to-One Strategy
Require: Task query Q, trajectory T = {t1 , t2 , . . . , tn }
Ensure: Redundant step index set R
 1: Initialize R ← ∅
 2: for i = 1 to n do
 3:     Provide the task query Q and target step ti to the LLM
 4:     Obtain redundancy prediction for ti
 5:     if ti is predicted as redundant then
 6:          R ← R ∪ {i}
 7:     end if
 8: end for
 9: return R


B.2    Details of Window-to-One

Algorithm 2 Window-to-One Strategy
Require: Query Q, trajectory T = {t1 , t2 , . . . , tn }, window size k
Ensure: Redundant step set R
 1: Initialize R ← ∅
 2: for i ∈ {1, 2, . . . , n} do
 3:     Construct window Wi ← {tmax(1,i−k) , . . . , ti , . . . , tmin(n,i+k) }
 4:     Provide Q and Wi to LLM
 5:     if LLM indicates ti is redundant given context Wi then
 6:          R ← R ∪ {i}
 7:     end if
 8: end for
 9: return R


B.3    Details of All-to-All

Algorithm 3 All-to-All Strategy
Require Query: Q, trajectory T = {t1 , t2 , . . . , tn }
Ensure: Redundant step set R
 1: Provide Q and full trajectory T to LLM
 2: LLM jointly predicts labels y1 , y2 , . . . , yn for all steps
 3: R ← {i | yi = redundant, 1 ≤ i ≤ n}
 4: return R
