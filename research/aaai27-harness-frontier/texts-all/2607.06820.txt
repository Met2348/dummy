A. Experimental Details
A.1. Implementation Details
SageMath Sandbox. All SageMath execution is performed in an ephemeral Docker container. For each tool call, a
generated Sage program is launched through a small wrapper. This wrapper imports sage.all, runs Sage’s preparser to
standardize Sage syntax, executes the script and extracts the designated result variable. It also captures standard output and
returns a JSON-friendly representation of the result to the host.
For each tool call, the generated code snippet is executed in a fresh interpreter. The execution is limited to 2 CPU cores, 4 GB
of memory, a 45s wall-clock timeout, and a 50s CPU-time limit. The executed code has no network access. The host also
rejects outputs larger than 262,144 bytes before parsing the returned result to prevent excessively large intermediate outputs.
Failure handling is normalized into a small number of categories. User-code exceptions are returned as code error
together with the captured traceback. Docker startup failures, such as an unavailable daemon or missing image, are classified
as docker error. Container crashes or abnormal exits are classified as runtime crash, and wall-clock overruns are
classified as timeout; in the timeout case the host forcibly removes the container.

A.2. Dataset Construction
We begin with the dataset of 633 problems obtained from the RealMath benchmark (Zhang et al., 2025). We focus on this
dataset because it is derived from the contemporary mathematical literature. Each problem from this dataset contains the
question, context and the answer, each of which is represented in a LaTeX format. We also enrich the dataset enriching it
with the arXiv categories that were associated with the papers from which the corresponding problems were scraped via the
RealMath extraction pipeline. If the paper had several categories, only the first one was used.
We automatically retain only those tasks whose final answers can be normalized into executable SymPy (Meurer et al.,
2017) objects suitable for automatic verification. Specifically, we want the answers to be normalized into either a concrete
numerical value, or a symbolic expression that might consist of several free variables or use the widely used mathematical
notation (such as π for 3.14159..., e for 2.71828..., Γ for the Gamma function, etc). In order to do this, we employ
LLM-assisted normalization procedure, under which the original ground truth answer, given in the free LaTeX form, is
transformed into SymPy-parsable string. Under this procedure we reduce the problem size in half, resulting in 300 problems.
Since the context field in RealMath often contains excessive or irrelevant information, we use an LLM to compress the
context, retaining only the relevant definitions and incorporating them directly into the question. We automatically annotate
the answer type as one of the following: a number is a concrete numerical value, an expression is a symbolic expression in
some free variables, a formula is an equality “named invariant = expression” in which the left-hand side is a label rather than
a checkable quantity.
As a next step, we employ an LLM-assisted annotation stage to estimate whether symbolic experimentation or computational
verification is feasible within SageMath. In order to do this, we define several criteria for a problem to be suitable for our
study: a task is suitable if

  1. Objects mentioned in the question can be represented by the means of SageMath,
  2. Candidate answers can be explored with the help of computational experiment,
  3. Intermediate computation steps can be executed in SageMath,

This stage reduces the problem set to 219. From this, we sample 133 problems attempting to reduce the skewness of the
distribution across arXiv categories. Figure 6 illustrates the distribution of the selected problems across arXiv categories.

C. Function usage by arXiv category
Figure 16 reports the most frequent computational functions invoked by the tool-augmented agents, grouped by arXiv
primary category, focusing on the mathematical actions that were performed by the agents when dealing with the problems
from these categories. We exclude boilerplate calls such as object constructions, printing, iterations, and list manipulations.
Function names are reported as normalized display labels, such as matrix.rank or polynomial.factor.
The most frequent operation overall is arith.gcd, with 1274 traces, of which 1042 occur in math.NT. Thus, the
dominance of this row is largely a number-theoretic effect rather than a uniform pattern across all areas. Similarly,
itertools.combinations appears 1041 times overall, with 601 occurrences in math.CO, 160 in math.GR, and
92 in math.SG. This suggests that a substantial fraction of tool use in these areas consists of finite search over candidate
subsets or configurations. The related row itertools.permutations is even more concentrated: 285 of its 360
occurrences are in math.CO.
Linear algebra operations are more broadly distributed. The row matrix.rank appears 895 times overall and has
substantial mass in math.RA, math.AG, math.AT, math.CA, and math.SG. Likewise, modules.dimension and
modules.basis are especially prominent in math.RA, where they appear 302 and 179 times respectively, but also occur
frequently in math.AG. This indicates that agents often reduce problems in representation theory, algebraic geometry, and
related algebraic categories to rank, dimension, basis, or kernel computations. Polynomial operations form another distinct
cluster. The row polynomial.factor appears 422 times, mainly in math.NT and math.AG. The more specialized
operation polynomial.groebner basis appears less frequently overall, but is concentrated in algebraic categories,
especially math.AG and math.AT. This is consistent with the overall practice of Groebner basis computations being used
for ideal membership, elimination, or solving polynomial systems.
Analysis-oriented categories show a different computational profile. The rows scipy.integrate.solve ivp,
scipy.integrate.quad, mpmath.quad, calculus.derivative and calculus.integrate are concen-
trated in math.AP, math.PR, math.CA, and math.DS. For instance, scipy.integrate.solve ivp appears 92
times in math.DS and 72 times in math.CA, while quadrature routines are concentrated in math.AP and math.CA. This
separates continuous-analysis problems from the enumerative and algebraic patterns seen in the other categories. Finally,
methods for symbolic manipulations such as symbolic.solve, symbolic.expand, symbolic.simplify, and
symbolic.simplify full are less tied to a particular mathematical domain, but often appear as intermediate steps
used to solve equations, compare coefficients, expand polynomial identities, or simplify expressions before producing a final
answer. Notably, symbolic.solve is especially frequent in math.CO, while symbolic.expand is concentrated
in math.AG; simplification routines are more broadly spread across algebraic, analytic, probabilistic, and geometric
categories.

D. Model-specific observations
GPT-5.5. The dominant configuration: tools lift its solve rate from 67.7% to 75.2%, the best result overall, at the lowest
cost (34k tokens and 3.8 Sage calls per problem, median trace of 1). It combines an 85% call-success rate with the best
post-failure recovery (77%), self-finalizes 91% of runs, and contributes the largest single-model coverage (100/133) to the
ensemble analysis.

Opus 4.8. The second-best solve rate overall (73.7%, up from 66.2% without tools), reached with a compact agentic tool
call traces (3.6 calls per problem, median trace of 2, 89% self-finalization). Its one weakness is timeouts: 22.6% of its Sage
calls exceed the wall-clock limit, the highest share among the 4 leading models, which decreases its raw call-success rate
(66%) without noticeably hurting its accuracy.

Opus 4.7. Tool access improves solve rate from 61.7% to 72.9%, 16 problems gained. It has the cleanest tool-interaction
profile in the study: the fewest Sage calls overall (435; 3.3 per problem), an 84% call-success rate, zero NameErrors, 92%
self-finalization, and the shortest sandbox time per problem (29 s).

Fugu-Ultra. Tied with Gemini 3.1 Pro for the second-strongest tool-free baseline (66.9%, behind GPT-5.5). Fugu-Ultra
reaches 72.9% with tools: short, clean code (median 662 characters), an 85% call-success rate, and no interpreter-state
errors. Its token multiplier under tool access is the smallest of any model (4.5×, 70k tokens per problem), placing it second
on the accuracy-cost plane among frontier models after GPT-5.5.

Qwen 3.7 Max. The largest gain from CAS access, jumping from 42.1% to 69.9% (+27.8 pp). The cost is a ≈ 99×
increased token usage (230k tokens per problem), so dominated by reasoning that the sandbox accounts for only 4% of its
solve time.

Gemini 3.1 Pro. The most stable configuration (66.9% → 69.9%): only 7 problems gained and 3 lost under tool access.
Its agentic statistics 8.1 calls per problem, 78% call success, 66% self-finalization. Gemini-3.1 also has the strongest
symbolic-checker accuracy among the models at 51.1%, suggesting better answer normalization or cleaner exact outputs.

Sonnet 5. Moves from 59.4% to 66.9% despite a problematic execution profile. It writes the shortest code among
non-degenerate models (median 437 characters) but launches honest, computationally infeasible enumerations: 41% of
its Sage calls time out (the largest share of any model), its mean sandbox time (142 s) is among the three highest, and its
post-failure recovery (16%) is the lowest, since it retries variants of the same infeasible computation.

Gemini 3.5 Flash. A solid mid-tier agent (61.7% → 66.9%) whose failure profile is dominated by NameErrors (99),
reflecting frequent loss of interpreter state across the stateless Sage calls; recovery is nonetheless strong (71%), and its
overall call-success rate (74%) remains respectable at 9.3 calls per problem.

DeepSeek V4-Pro. Improves its accuracy from 52.6% to 63.2%: 1,285 Sage calls at an 81% success rate (the best among
the heavy tool users), combined with active documentation use (33 Context7 lookups). This throughput is not cheap:
240k tokens per problem, the second-largest budget after MiniMax, and nearly half of its runs (48%) terminated by forced
finalization.

GLM 5.2 Gains +9.0 pp (53.4% → 62.4%) but prefers heavy symbolic computations that frequently exceed the wall-clock
limit (27% of calls; mean sandbox time 147 s), recovers poorly (26%), and self-finalizes only 53% of runs.

Kimi K2.7. This model is persistent but inefficient: 1,421 Sage calls with a median trace of 14, a call-success rate below
50% split between code errors (26%) and timeouts (25%), the longest mean sandbox time (161 s per problem), and 205k
tokens per problem - all for a +1.5 pp gain (57.1% → 58.6%). It is also the only model whose traces contain a call to a
nonexistent tool name, which the orchestrator rejected.

MiniMax M3. A substantial tool gain (38.3% → 55.6%, +17.3 pp) achieved at extreme token usage: 556k tokens per
problem, by far the largest budget we observed, driven by very long reasoning traces, with a third of runs terminated by
forced finalization rather than by the model’s own decision.

Question.    A permutation π = π1 . . . πn is called reverse alternating if

                                             πi > πi+1       if and only if      i is odd,

i.e.,
                                                     Des(π) = {1, 3, 5, . . .},
where Des(π) denotes the set of descents (indices i such that πi > πi+1 ). A permutation is an involution if π = π −1 . Let
RAI2n (1234) denote the set of reverse alternating involutions of length 2n that avoid the pattern 1234, i.e., no subsequence
of four elements is order-isomorphic to the increasing sequence 1234. Let Mn denote the n-th Motzkin number, which
counts the number of Motzkin paths of length n. A Motzkin path of length n is a lattice path from (0, 0) to (n, 0) using up
steps U = (1, 1), down steps D = (1, −1), and horizontal steps H = (1, 0), never going below the x-axis. The Motzkin
numbers satisfy
                                           M0 = 1,       M1 = 1,        M2 = 2,
and the recurrence or generating function as standard. Express the cardinality |RAI2n (1234)| in terms of Motzkin numbers
Mk for appropriate k.

Answer.
                                                    Mn+1 − 2Mn−1 + Mn−3 .

Solution’s strategy. GPT-5.5 first directly enumerated involutions of length 2n for small n, filtered them by the reverse-
alternating condition and by 1234-avoidance, and obtained the sequence

                                                      1, 2, 6, 14, 35, 89, . . . .

It then inspected the corresponding Robinson–Schensted shapes and implemented a dynamic program over standard Young
tableau growth states. The dynamic program enforced the prescribed alternating descent pattern and restricted row lengths
to at most three. Finally, the agent generated a longer sequence, compared it against linear combinations of shifted Motzkin
numbers, and verified agreement with
