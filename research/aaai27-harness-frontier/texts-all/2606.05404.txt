   Case Study C.1: T IME C LAW on Context-is-Key: How Textual Context Steers the Forecast

   Core Task. We pick a pair of CiK tasks from the bivariate linear SVAR family at the same seed (seed=5): M INIMAL -
   C AUSAL C ONTEXT B IVAR L IN SVAR and F ULL C AUSAL C ONTEXT E XPLICIT E QUATION B IVAR L IN SVAR. The two
   records share identical 128-step past history and 32-step ground-truth future of the target variable X1 , and share the
   same step-change schedule for the covariate X0 over the forecast window (0.028 → 0.111 → 0.223). The only difference
   is what the textual context reveals about the data-generating process. This isolates a clean variable: context → forecast.
   Prompt: See Appendix I for the full prompt template.
   Visualization: We provide Figure 9 to better illustrate this case study.

   Memory Retrieval Stage.

     1. Two-stage retrieval. T IME C LAW embeds the task’s background + scenario + constraints block with
        text-embedding-3-small, takes the top-Ktext records by cosine similarity from the training bank, then re-ranks
        the survivors by L2 distance on the 20-dim numerical fingerprint of past_time. With k = 1 on this task both
        stages converge on the same family (F ULL C AUSAL C ONTEXT I MPLICIT E QUATION B IVAR L IN SVAR) seen during
        training.

     2. Reference compression. The retrieved trajectory is compressed by summarize_trajectory into the trainer’s
        analytic spine: a context→forecast explanation, together with the sequence of MCP tool calls and their truncated
        responses. This summary is then injected before the test prompt as a REFERENCES FROM PRIOR TRAINING block,
        surfacing the transferable rule that the scenario’s lagged linear parent effects should be handled by iterating the
        SVAR recursion over the known future X0 schedule.

   Execution Stage (Full Context).

     1. Context parsing. The agent reads the explicit linear SVAR equation from the prompt (X1t = 1.322 X0t-1 −
        0.604 X1t-1 + 0.926 X0t-2 + 0.763 X1t-2 − 0.851 X0t-3 + 0.623 X1t-3 ) and the piece-wise constant schedule for X0
        over the 32-step horizon.

     2. Tool-grounded historical inspection. Guided by the retrieved spine, the agent invokes the MCP analysis server:
        series_overview() → channel_stats("1") → compute_acf("1", max_lag=20) → channel_values("1",
        120, 128), confirming the recent X1 levels needed to seed the recursion.

     3. Closed-form forecast. With the equation, the recent lagged values, and the future X0 schedule all known, the
        agent unrolls the recursion deterministically, producing a 32-step forecast that tracks the ground-truth ramp from
        0.035 → 1.03 almost exactly.

   Counterfactual: Minimal Context, Same Series. On the matched M INIMAL-context variant, the prompt only states
   that “parents for X1 at lag k ∈ {1, 2, 3} are X0 , X1 ,” while the six SVAR coefficients are withheld. Given the same
   retrieved memory and tool budget, the agent cannot reconstruct the recursion and instead falls back to extrapolating a
   small monotone drift in [0.036, 0.054], missing the X0 step changes that drive the true ramp.
   Evaluation. On the full-context variant, T IME C LAW achieves sMAPE 0.44% (MSE ≈ 10−6 ), compared with sMAPE
   124% (MSE 0.254) on the minimal-context variant. This yields a ∼ 280× sMAPE gap for forecasting the same
   numerical series. Since the two variants share identical histories, retrieved memory, and tool budget, the gap is explained
   by the ∼ 400-token difference in textual context. This demonstrates that T IME C LAW’s gains arise from context
   comprehension rather than distributional pattern matching on the history alone. Figure 11 overlays both forecasts on the
   shared past and ground truth.

Case Study C.2: T IME C LAW on TSRBench: How Retrieved Memory Steer the Decision

Core Task. A multiple-choice causal-discovery question from TSRBench (causal_reasoning, Elbe-River-Flood).
The agent is shown 64 days of runoff measurements from five river sensors and must pick the 5×5 binary adjacency
matrix that captures the causal structure (columns = cause, rows = effect). Ground-truth answer is D: an upstream-to-
downstream cascade in which small tributaries (rivers R77, R546, R1071) feed the large downstream river R176, which
in turn feeds R175.
Prompt: See Appendix I for the full template.
Visualization: We provide Figure 10 to better illustrate this case study.

Memory Retrieval Stage.

  1. Two-stage retrieval. T IME C LAW embeds the test prompt with text-embedding-3-small and selects the top-
     Ktext records from the training bank by cosine, then re-ranks the survivors by L2 distance on the fingerprint of the
     loaded series. Then T IME C LAW retrieves k = 3 records.

  2. Rule extraction via summarize_trajectory. Each retrieved trajectory is compressed to a tool-call spine plus a
     context_to_action block written during training. One block of the retrieved action in memory reads verbatim:
    “the mechanism mapping is upstream-to-downstream edges.” This is the transferable decision rule that primes the
     test agent.

  3. Prompt injection. The compressed references are concatenated as a REFERENCES FROM PRIOR TRAINING block
     in front of the test prompt, only analytic spine and decision rules.

Execution Stage.

  1. Targeted tool use. Primed by the retrieved rule, the agent runs only the inspections needed to confirm direction:
     list_channels → series_overview → five compute_acf probes, one per river. It skips the exhaustive per-
     channel statistics, peak-finding, and periodicity-detection that the no-memory agent runs through.

  2. Mapping the rule onto the new river set. The agent matches its own observations to the retrieved rule: the
     channels with mean flow ∼ 660–680 (R175, R176) form the downstream sinks; the three with mean flow < 4 (R77,
     R546, R1071) are upstream tributaries. The rule then determines option D, in which the sink row R176 is densely
     populated with parents from the tributaries and the tributaries themselves are source rows.

  3. Answer realisation. The agent emits the final letter <answer>D</answer>, with a brief justification citing the
     retrieved cascade rule.

Counterfactual: Same Series, No Memory. Re-running the identical task at k = 0 removes only the REFERENCES block.
Without memory, the agent resorts to brute-force inspection, issuing 22 tool calls across channel_stats, compute_acf,
find_peaks, and detect_periodicity for every channel. It then selects option A, which encodes the physically
reversed cascade (R176 caused by R175; R77, R546, and R1071 each caused by R176). Without the retrieved rule, the
agent lacks an anchor for causal direction and defaults to a topology inconsistent with the observed lead-lag patterns.
Evaluation. On the same instance, T IME C LAW changes from wrong (A) at k = 0 (without memory) to correct (D)
at k = 3 (with memory), while reducing the tool-call budget from 22 to 7 (3.1× fewer calls). The retrieved memory
provides a transferable decision rule. This demonstrates memory improves reasoning by carrying compact rules that
redirect the agent’s search process, rather than by matching numerical histories alone.

   Core Task. TSAIA’s finance split presents the agent with portfolio and market time-series panels and asks it to answer
   quantitative multiple-choice questions. At the beginning, the agent’s toolbox only contains generic time-series inspection
   tools. It does not include finance-specific primitives for portfolio evaluation or market-factor regression.
   Prompt: See Appendix I for the full template.

   Tool Evolution.

     1. Routine extraction via summarize_trajectory. During training, three recurring analytical routines appear
        across successful trajectories: portfolio risk-adjusted return estimation, portfolio risk estimation, and market-factor
        regression. Each successful trajectory is summarized into a compact routine description that records the input
        conventions, intermediate computations, and final decision rule. These summaries reveal reusable computation
        patterns.

     2. Evolving finance-specific tools. From these recurring routines, T IME C LAW evolves three finance-specific tool
        schemas and adds them to the agent’s reusable toolbox:
           • portfolio_sharpe(channels, weights, risk_free, period_per_year): compute risk-adjusted return
             for each portfolio option and compare the returned ratios.
           • portfolio_var(channels, weights, horizon=H, alpha=α, method="parametric"): compute port-
             folio risk using task metadata and compare the resulting losses across options.
           • capm_regression(asset_channel, market_channel): estimate market-factor statistics {α, β, r2 } and
             select the option whose target statistic matches the returned value.

   Evaluation. Across the finance test split, evolved tools improve both accuracy and tool-use efficiency, especially when
   manual arithmetic is unreliable. On MarketAB-beta tasks, the average number of capm_regression calls per task
   increases to 0.83, and accuracy rises by +16.7. On VaR tasks, accuracy improves by +12.0, while the average tool

   Prompt Template B.3: Memory-Retrieval Prefix and Training-Mode Suffix

   Augmentation blocks. Two cross-cutting blocks compose with every builder above. The retrieval prefix is inserted
   at test time when the memory bank returns nearest-neighbor trajectories; the training suffix is appended at memory-
   bank-construction time so the agent produces a justifying analytic trace plus a transferable <context_to_action>
   explanation.

        Retrieval prefix (test time, when k > 0 neighbors are retrieved):
        === REFERENCES FROM PRIOR TRAINING ===
        Below are similar prior tasks (same task family where possible) and how they were solved, including which
        tools were called, the key numbers returned, and the final answer. Use them as a guide for the analytic process.
        The correct answer for the current task may differ; do not copy blindly.

        Reference 1:
        [family=<family_key>]
        <trainer’s transferable explanation>
        tool_name(args) → <truncated_response>
        tool_name(args) → <truncated_response>
        ...

        Reference 2: ...

        === END REFERENCES ===

        Training-mode suffix (memory-bank construction):
        TRAINING MODE

        Do TWO things before emitting your final answer:
        1. Inspect the series with whatever tools are available and walk through the reasoning step by step.
        2. Output a <context_to_action>...</context_to_action> block (≤3 sentences) that explicitly states:
           • which sentences in the context (background / scenario / constraints / question / options) drive the answer’s
             shape, AND
           • the rule that maps those sentences to that shape (e.g. “scenario says ‘heat wave for 2 hours’ → ground
             truth has a 4× spike at the stated start → because air-conditioning load scales with cooling demand”).
        Make this block transferable, NOT series-specific. Future test-time agents will read it to map their own
        context, so avoid statements like “the mean is 850” and prefer statements like “scenario X implies pattern Y
        because of mechanism Z”.

        Then produce your final answer in the format requested above.
