A     Extended Related Work
Anticipating future need. The cost of mispredicting need is compute heavy: under recency or LRU,
entries are discarded shortly before reuse and must be recomputed (Pan et al., 2026). This has motivated
proactive systems. At the serving level, KVFlow and PBKV (Pan et al., 2026; Zheng et al., 2026) predict
which agent or cache block will fire next from workflow structure and prefetch accordingly; at the agent
level, AgentFold (Ye et al., 2025) learns to fold and consolidate its own trajectory as it runs. These
establish that anticipation beats reaction, but they anticipate computation from workflow graphs, or rewrite
history through further LLM calls. We work along anticipating which past content an upcoming query
will require, cheaply and without invoking an LLM.
Systems-level context management. KV systems such as PagedAttention (Kwon et al., 2023) remove
memory fragmentation and enable cross-request reuse, raising throughput. Such caching lowers the cost
of re-processing a seen context, but doesn’t relieve the cost of attending to that context, which becomes
error-prone once critical content is buried within a long window (Liu et al., 2024).
Self-supervised relevance without labels. Training a scorer cheaply requires supervision that costs
nothing to obtain. Hindsight relabeling (Andrychowicz et al., 2017) reuses a failed trajectory by relabeling
it with the goal it did reach for RL, and contrastive pretraining (Izacard et al., 2021) learns dense retrievers
with no relevance labels at all. We draw on this idea to derive retention labels directly from what later
turns reused, at no annotation cost.

B     Benchmark Utility
B.1    LoCoMo: Query-Agnostic Conversational Eviction
LoCoMo(Maharana et al., 2024) serves as the primary benchmark for evaluating query-agnostic relevance
prediction in conversational context eviction. The dataset contains 5,882 dialogue turns across 10
conversations, where each conversation forms a distinct evaluation group.
   1. Data structure. Each sample corresponds to a dialogue turn ui , and all turns within a conversation
form a group g ∈ {1, . . . , 10}. The task is to assign a relevance score to each turn without conditioning
on a specific downstream query. About 24% of dialogue turns are considered truly relevant (gold evidence
across questions)
   2. Label construction. We consider two labeling regimes. In the supervised setting, a turn is labeled
positive (yi = 1) if it appears in the gold evidence set of any question associated with its conversation.
Because these labels are pooled across all the questions tied to the conversation, it yields a query-agnostic
relevance signal. The model learns to identify turns that are generally load-bearing or informative for
the conversation, rather than those that just match a specific query. In the self-supervised setting, a turn
is labeled positive if it shares at least 40% token overlap with an eventual answer for training without
explicit annotations. The choice of token overlap percentile is discussed in Section D.1
   3. Splitting and leakage-free training. We use a Leave-One-Group-Out (LOGO) protocol over the 10
conversations. In each fold, one full conversation is held out for evaluation while the remaining nine are
used for training. To prevent information leakage, the vectorizer and StandardScaler are fitted strictly
on training data within each fold. A deduplication guard removes any exact-text overlaps between training
and test folds. In LoCoMo, this has minimal effect, removing only 8 training instances corresponding to 3
generic utterances (e.g., “Bye!”, “Take care!”).
   4. Feature representation and scoring. Each unit is mapped to a feature vector combining: (i) a
TF-IDF representation of the text, and (ii) six trajectory features: position, recency, length, digit count,
question-marker presence, and capitalized-word count (Table 6). A logistic regression model is trained to
estimate relevance scores pi = P (yi = 1 | ui ) using these features. At inference time, the model scores
all units in the held-out conversation without access to downstream queries.
   5. Evaluation and downstream usage. The model is evaluated both intrinsically and extrinsically.
For intrinsic evaluation, we compute ROC-AUC over held-out folds (Section E). For extrinsic evaluation,
retained dialogue turns are passed verbatim into a downstream QA model. Performance is measured using
token-level F1, as suggested by Maharana et al. (2024).

B.3   AppWorld: Agentic Long-Horizon Evaluation
AppWorld is a long-horizon tool-use benchmark in which an agent writes Python against application
APIs (Venmo, Spotify, Gmail, and others) to complete a natural-language task, graded by hidden unit
tests. We evaluate on the test_normal split (168 tasks: 57 Easy, 48 Medium, 63 Hard), following (Kang
et al., 2025) and train the agent scorer on the 90 tasks of the disjoint train split. The agent and, where
applicable, the compressor are gpt-4.1-mini.
   1. Trajectory generation and training data. We first execute an unevicted baseline agent ("nocomp")
on the AppWorld training split consisting of 90 tasks. This produces full execution trajectories containing
interleaved Python code actions and environment observations, which are used as training data for the
LRE scorer.
   2. Self-supervised labeling via quotation reuse. Since no human relevance annotations are available,
we construct a zero-supervision label using a hindsight reuse criterion. Let zi denote an identifier
introduced at step i. The label is defined as:

   • yi = 1 if zi is reused in at least 3 subsequent steps,

   • otherwise yi = 0.

This identifies load-bearing identifiers that persist across the execution trajectory.
   3. Scorer training. The logistic regression scorer (s90 ) is evaluated under a Leave-One-Task-Out
(LOGO) protocol.
   4. Inference and eviction loop. During evaluation on test task splits, the agent operates in a ReAct-style
loop. When the context length exceeds the token budget (B = 2048), the eviction module is triggered.
LRE performs CPU-based scoring of historical units. Units are ranked by score-per-token and retained
greedily. This ensures chronological and verbatim reconstruction of the compressed context and preserves
exact API identifiers and execution-relevant symbols.
   5. Evaluation and Task Goal Completion (TGC). AppWorld evaluates success using Task Goal
Completion (TGC), computed via hidden unit tests that operate on the final environment state.
   A known failure mode is self-reported success bias, where task success is determined from the agent’s
own completion signal rather than from the environment state. This is problematic because the agent can
declare success without having actually produced a state that satisfies the underlying task constraints,
leading to an empirical inflation of success rates by approximately 2× (table 5).

D     Justification of Self-Supervised Label Design Choices
We evaluate label quality using two quantities: (i) AUC-vs-gold, which measures how well self-supervised
labels align with human-annotated gold relevance labels (higher is better; 0.5 corresponds to random
alignment), and (ii) positive rate, which denotes the fraction of units labeled as positive under a given rule.
For agentic settings, we additionally report LOTO AUC (Leave-One-Task-Out AUC), which measures
how well a classifier trained on these labels generalizes to unseen tasks.

D.1    Conversational Overlap Threshold (τ = 0.4)
For conversational self-supervision, a turn is labeled positive if its content overlaps with a down-
stream response above a threshold τ . We select τ = 0.4 based on a joint sweep across LoCoMo
and LongMemEvalS , evaluating AUC-vs-gold and label sparsity.
   Table 7 summarizes the results.

                    τ    LoCoMo AUC       LongMemEvalS AUC       LoCoMo +      LongMemEvalS +
                   0.2       0.742               0.647              0.94             0.47
                   0.3       0.754               0.686              0.85             0.38
                   0.4       0.769               0.698              0.66             0.29
                   0.5       0.764               0.682              0.61             0.22
                   0.6       0.758               0.679              0.39             0.13
                   0.8       0.754               0.652              0.30             0.11

                               Table 7: Sweep over response-overlap threshold τ .

    Why τ = 0.4 is chosen: First, it is the joint maximizer of AUC-vs-gold across both datasets. Second,

                                                         18
it lies in a flat optimum region (τ ∈ [0.3, 0.5]), where performance varies by at most 0.016 AUC. This
indicates stability.
    Why not other values: Lower thresholds (τ = 0.2, 0.3) produce excessively high positive rates on
LoCoMo (0.85–0.94). This collapses the ranking problem into near-uniform positives and weakens
discrimination. Such a near-degenerate label distribution does not reflect real-world deployment nuances.
Higher thresholds (τ ≥ 0.5) reduce coverage too aggressively (down to 0.11–0.61 positive rate) and
degrade alignment with gold annotations.

D.2    Agentic Reuse Breadth (b = 3)
For agentic trajectories, a step is labeled positive if an identifier it introduces is reused at least b times in
subsequent steps. We evaluate b ∈ {1, 2, 3, 4, 5} using Leave-One-Task-Out evaluation.

                                     b   Pos-rate   # Positives   LOTO AUC
                                     1    0.811       1269          0.878
                                     2    0.582        911          0.830
                                     3    0.432        676          0.828
                                     4    0.335        524          0.835
                                     5    0.271        424          0.847

                        Table 8: Sweep over reuse breadth b for agentic self-supervision.

   LOTO AUC (Leave-One-Task-Out Area Under the ROC Curve) measures how well a classifier trained
on these labels ranks truly relevant steps above irrelevant ones in unseen tasks. A value of 0.5 corresponds
to random ranking, while 1.0 corresponds to perfect separation.
   Why b = 3 is chosen: Unlike the conversational case, LOTO AUC is relatively flat across all values
(0.83–0.88). We therefore select b = 3 based on semantic and operational criteria.
   First, b = 1 is overly permissive, marking 81% of steps as positive. This collapses the eviction task
into a near-majority-class problem, where ranking becomes dominated by trivial structural correlations
rather than load-bearing utility. Second, b = 3 yields a moderate positive rate (≈ 0.43), which aligns with
the expected compression domain (Kang et al., 2025) in which only a subset of trajectory state can be
retained.
   Why not other values: Values b ≥ 4 become too selective, reducing the positive set to 27–34%, which
harms coverage of reusable context. b ≤ 2 over-labels transient identifiers that are only briefly reused. We
choose a parameter value that reflects real deployment nuances and therefore fix b = 3, a semantic choice
that captures repeatedly-reused, load-bearing state while maintaining a stable and informative training
signal.

E     Metrics: definitions and recovery
This appendix defines every reported score in both domains, then describes how the agent metric and its
costs are recovered honestly.

E.1    Conversational relevance scores
ROC-AUC (per group). Each method assigns a real-valued keep-score si to every unit ui of a group,
scored against the binary gold label yi ∈ {0, 1}. Let P = {i : yi =1} and N = {i : yi =0} be the relevant
and irrelevant units, with P = |P|, N = |N |. The area under the receiver-operating-characteristic curve
equals the probability that a random relevant unit outscores a random irrelevant one, computed exactly as
the normalized Mann–Whitney statistic (Mann and Whitney, 1947)

                                      1 XX                               
                          AUC =             1[ si > sj ] + 21 1[ si = sj ] .
                                     PN
                                           i∈P j∈N

AUC = 1 is a perfect ranking (every relevant unit above every irrelevant one), 0.5 is chance (random
ordering), and a value < 0.5 means the score is anti-correlated with relevance. AUC is threshold-free and

                                                       19
invariant to the positive rate, so it measures the ranking quality an eviction policy needs (what to keep
first) without committing to a budget. A group is scored only if it contains at least one relevant and one
irrelevant unit (otherwise AUC is undefined).

Macro-AUC. The per-group AUCs are averaged with equal weight, macro-AUC = |G|                  g∈G AUCg ,
so a large group cannot dominate a small one; the groups are conversations on LoCoMo and questions on
LongMemEvalS .

Recall@budget. A retention policy keeps the K = ⌈b n⌉ highest-scoring of the n units at budget
fraction b ∈ {0.1, 0.2, 0.4}. Recall@b = |kept ∩ P|/P is the fraction of relevant units that survive,
averaged over groups.

Budget@recall (iso-accuracy efficiency). The inverse statistic: the smallest budget fraction b at which
recall@b reaches a target (e.g. 0.8), averaged over groups, reported as the saving relative to recency at the
same recall.

E.2   Conversational answer-quality scores (downstream)
Token-F1 (LoCoMo). With the answerer reading only the retained context, we compare its answer A to
the gold answer G as bags of normalized content tokens. With prec = |A∩G|/|A| and rec = |A∩G|/|G|,

                                                      2 prec · rec
                                         token-F1 =                ,
                                                      prec + rec

the standard extractive-QA overlap, averaged over questions.

LLM-judge (LongMemEvalS ). Free-form answers are graded by a model stronger than the answerer
(gpt-4o judging gpt-4.1-mini, to avoid self-grading): the judge sees the question, the gold answer, and the
candidate, and returns a binary correct/incorrect vote; the score is the fraction judged correct. The judge
was validated on crafted correct, wrong, and paraphrase cases.

E.3   Conversational serving-cost scores
Throughput, neural calls, dollar cost. Units per second is the measured rate at which a loaded scorer
scores resident units (hardware recorded). Neural forward passes per unit counts the encoder or classifier
passes a method performs per scored unit.

E.4   Agent scores
Task Goal Completion (TGC). The all-or-nothing success metric on AppWorld: a task scores 1 only if
every hidden unit test passes (partial credit such as 3 of 4 tests counts as 0), and TGC% over a difficulty
bucket is the mean over its tasks. How we recover true TGC (rather than the inflated self-report) is in §E.5.

Cost axes. cLLM is compressor language-model calls per task (the number of compression events).
Input tokens per request (in/req) is the mean prompt size per generation, our peak-token proxy since the
true per-step peak is not logged. Total tokens per task sums agent and recovered compressor tokens.

E.5   Recovering the agent metric and costs
Recovering true TGC by replay. The agent framework ships with its environment evaluation disabled,
so the run’s self-reported completion flag overstates success and is not Task Goal Completion. We recover
true TGC at zero model cost: because a freshly constructed environment resets task state, we replay each
task’s logged action sequence into a fresh environment and then call the environment’s evaluation, which
passes only when all hidden unit tests pass. The replay reconstructs the final state deterministically rather
than reporting the empty-environment artifact. All reported TGC values use this recovered metric, never
the self-report. Algorithm 12 gives the procedure.

J   Extended limitations
This appendix expands the limitations stated in the main text. We group them by scope: evaluation,
method, supervision, measurement, and generalization. Each limitation lists what we did to mitigate it
and, where relevant, what a deployment would need to verify before relying on LRE.

Single-seed agent comparison. The AppWorld results are reported at one seed (s1). The host agent
(gpt-4.1-mini) is non-deterministic, so per-task outcomes carry run-to-run noise that a single seed cannot
characterize.

In-fork ACON baseline. ACON’s published fork ships the unoptimized compression prompt and we
used it as it is. The fair comparison that licenses our claim is the matched in-fork one (identical agent,
identical seed, identical evaluation harness).

Benchmark coverage. We evaluate on three benchmarks: LoCoMo and LongMemEvalS for conversa-
tional memory, AppWorld for long-horizon agents. These cover dialogue and tool-use coding agents but
do not include web-browsing agents, multimodal agents, or open-domain coding agents (e.g., SWE-bench).
The Pareto-efficiency claim is supported within these regimes.

Subset of AppWorld used. We evaluate on test_normal (168 tasks). We do not evaluate on
test_challenge (417 tasks).

Budget-bounded performance. On tasks whose required state exceeds the operating budget (AppWorld
Hard at B = 2048 being the clearest case) no-compression wins because retaining everything is the only
way to keep state that does not fit. LRE therefore complements, rather than replaces, unbounded retention
when the deployment can afford it. The right deployment question is whether the budget is binding; where
it is, LRE is Pareto-efficient.

Eviction unit granularity. Our agent unit is the (action, observation) pair; our conversational unit is the
dialogue turn or session. We do not evict at the token level inside a single forward pass, which is what
KV-cache methods such as H2 O (Zhang et al., 2023), SnapKV (Li et al., 2024), Selection-p (Chung et al.,
2024), and TRIM-KV (Bui et al., 2025) do. Token-level methods are complementary: they operate inside
the model and require model-internal signals; LRE operates outside the model and requires only logged
behavior. A deployment could plausibly combine them (LRE for unit-level retention, token pruning inside
each retained unit), but we do not evaluate that combination here.

Host-model coupling of training data. We do not test transfer to other host models (different tokenizers,
different action styles, different failure modes). The scorer is small enough to retrain per host model in
minutes from logged behavior, so the limitation is operational rather than fundamental, but a deployment
switching host models should retrain.

Feature-set dependence. The learned scorer operates on a compact set of causal features that we
designed to be inexpensive, model-agnostic, and available at decision time. While these features transfer
across the benchmarks studied here, we do not claim that they are universally optimal. Different deploy-
ments may expose different signals, interaction patterns, memory units, or notions of future utility. In
practice, a production system should treat our feature set as a reference implementation rather than a fixed
specification, and should measure and validate its own candidate features from logged behavior before
deployment. The broader claim of LRE is that future utility can be learned from causal signals available
at write time, not that any particular feature set is universally sufficient.

Fixed self-supervised threshold. The conversational self-supervised label uses a fixed answer-recall
threshold of τ = 0.4, jointly optimal on LoCoMo and LongMemEvalS . In deployment, however, the right
rule is to set τ implicitly (through a target positive rate on a sample of logged exchanges) rather than to
inherit our value. The τ = 0.4 choice is justified empirically (Table 7); we do not claim it is universal.
