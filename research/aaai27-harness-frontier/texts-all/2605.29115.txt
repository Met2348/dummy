                                                Unix competence is the ability to use shell and operating-system primitives as first-class tools,
                                                not merely to write programs through a terminal. Current terminal benchmarks tend to blur
                                                this distinction: a solver fluent in Python but weak in Unix can pass a substantial fraction of
                                                Terminal-Bench 2.0, while the reverse skill profile is rarely exercised. We make the distinction
                                                operational and build a training surface for the Unix component. unix-ctf is a procedural
                                                generator of capture-the-flag tasks for shell agents. Each task hides a short token (a ﬂag of
                                                the form flag{a3b1c9...}) inside a fresh Linux container using a single Unix feature,
                                                and the agent must recover it. Tasks are produced by an LLM-assisted synthesis pipeline that
                                                generates candidate hiding techniques, rewrites them into parameterized hide-and-find script
                                                pairs, and filters them with a bidirectional contract: the hide script must leave no plaintext trace
                                                of the flag on disk, and the find script must recover the flag in a fresh directory. Because the
                                                LLM only writes the planting and recovery steps (the container, layout, and grading harness
                                                are fixed), the pipeline lands 656 of 750 raw attempts as portable, reusable variants (87.5%).
                                                Our reproduction of Endless Terminals’ full-container-generation approach lands only 17.4%
                                                under the same checks. The 656 variants canonicalize to 155 distinct techniques. Fine-tuning
                                                Qwen3-8B with LoRA using GRPO on this surface lifts solve rate from 11.6% to 43.6% on a
                                                15-skill multi-family holdout (𝑛=225), redistributes which InterCode-CTF tasks the model
                                                solves, and produces a +33 pp gain in Forensics while reaching 32/100 on InterCode-CTF.
                                                These results suggest that Unix competence is separable, trainable, and best evaluated directly
                                                rather than folded into programming-through-a-shell.



                                         1 INTRODUCTION
                                         Terminal interaction has become a standard setting for evaluating language-model agents. InterCode (Yang
                                         et al., 2023), Terminal-Bench (Merrill et al., 2026), AgentBench-OS (Liu et al., 2024), NYU CTF Bench (Shao
                                         et al., 2024a), and Cybench (Zhang et al., 2024) are now common evaluation targets, and large-scale training
                                         pipelines such as Endless Terminals (Gandhi et al., 2026) and Nemotron-Terminal (Pi et al., 2026) have
                                         followed. This landscape is usually described as measuring a single capability: using a shell to accomplish a task
                                         on a computer. We argue that this framing collapses two different skills into one. One is general programming
                                         performed through a terminal; the other is Unix competence itself. This paper isolates the latter and builds a

We use the capture-the-ﬂag (CTF) format inherited from security competitions (Shao et al., 2024a; Zhang
et al., 2024). Each task hides a short token, or ﬂag, of the form flag{a3b1c9...} inside a fresh sandboxed
environment; the agent must find it and print it back. The format is useful because the reward is mechanical
and unambiguous, and because a planted flag can be tied to a single Unix feature. Recovering the flag then
requires using that feature rather than solving a loosely specified programming task.

1.1 UNIX-COMPETENCE VERSUS CODING-IN-A-TERMINAL
We define a task as testing Unix competence when its solution depends on an OS, shell, or file-format feature
with no clean analogue in ordinary general-purpose programming. Concretely: a flag stored as the value of a
POSIX extended attribute (user.secret) on a decoy file is invisible to ls -l, stat, file, or a recursive
grep of the file’s bytes, because xattrs live in a side-channel inode namespace; the canonical recovery is a
single getfattr -n user.secret decoy.txt. Using a programming language like Python does not
remove the dependency: os.getxattr is just a wrapper over the getxattr(2) syscall. The OS-specific
access path is irreducible. Other library instances (the GNU build ID inside an ELF .note.gnu.build-id
section, files identified by pre-epoch mtime, custom OIDs inside an X.509 certificate) have the same shape;
six worked exemplars are in Appendix D.
We define a task as coding-in-a-terminal when the deliverable is a non-trivial program, such as a small neural
network, an interpreter, a distributed-training primitive, or a statistical sampler, and the shell is only the
medium through which the program is created, compiled, and tested. In this regime the shell contributes little
to success: a solver fluent in Python but weak in Unix can still perform well.
Both skills are legitimate, but they lie on substantially different axes. Treating them as one obscures what a
benchmark measures and what a training pipeline teaches. We quantify the resulting bias in current terminal
benchmarks in §2 (Figure 2) and target the underweighted Unix-competence half.

Endless Terminals seeds scenarios such as “developer organizing files” or “DevOps engineer debugging logs,”
which are Unix-flavored but shallow: an agent trained on this material sees grep, awk, and find, but much
less of the filesystem metadata, binary formats, kernel interfaces, and service-configuration surfaces that matter
in real Unix administration. Nemotron-Terminal builds on a primitive-skills taxonomy, such as file I/O, string
manipulation, and graph traversal, that is explicitly cross-language. The most distinctive shell capabilities are
therefore underrepresented in both evaluation and training data, and progress on terminal benchmarks can
track programming proficiency more than Unix fluency. This is the gap we target.
The taxonomy above motivates how we position unix-ctf against four adjacent strands of prior work, in
increasing order of structural overlap with our setting.

Fixed Linux benchmarks. The most direct evaluation yardsticks for shell and operating-system agents
are InterCode, InterCode-CTF, Terminal-Bench 2.0, AgentBench-OS, NYU CTF Bench, Cybench, and
OSWorld (Xie et al., 2024). They are hand-curated and fixed in size (Terminal-Bench 2.0 has 89 tasks, Cybench
40, NYU CTF Bench 200), so none exposes a generative task surface that an RL loop can draw new instances
from.

Computer-use benchmarks beyond the terminal. A parallel line, including WebArena (Zhou et al.,
2023), WorkArena (Drouin et al., 2024), OSWorld (Xie et al., 2024), AndroidWorld (Rawles et al., 2024),
and Windows Agent Arena (Bonatti et al., 2024), also evaluates agents in executable software environments
with programmatic success checks. The substrate is web, desktop, or mobile control rather than the Unix shell.
unix-ctf keeps the executable-environment / mechanical-reward shape but restricts the surface to text-only
Linux interaction and Unix-specific OS features.

Procedural generation in the shell and CTF space. Our closest precedents share both the procedural-
generation goal and the shell substrate. Endless Terminals (Gandhi et al., 2026) constructs 3,255 tasks using

3 THE TECHNIQUE LIBRARY
Each technique enters the library through an offline pipeline that alternates language-model generation with
mechanical verification. The result is a reusable library of 155 canonical technique identifiers ( 𝑘 = 441
deduplicated variants), summarized in Figure 3.

3.1 HARVEST PIPELINE
The five stages are designed to keep the language model on the layer it can validate, while letting mechanical
checks enforce the task contract. (1) Exploration: Claude Opus 4.7 (Anthropic, 2026) runs inside a fresh
ctf-base Docker container and is asked to hide a flag of the form flag{<8–16 hex bytes>} using a
target technique. It emits shell commands turn by turn for up to ten turns; we record both the command
sequence and a candidate recovery command. (2) Mechanical veriﬁcation: in the same container, a recursive
search for the literal flag must return no occurrences on disk, and the candidate recovery command must
print the flag and exit zero. Any failure drops the attempt. (3) Synthesis: Claude Haiku 4.5 (Anthropic,
2025) rewrites the validated exploration into two parameterized scripts, plant.sh (target directory, flag) and
recovery.sh (target directory), as a pure textual rewrite with no shell execution. (4) Portability validation:
the synthesized scripts are re-run against a fresh temporary directory with a fresh flag. plant.sh must exit

3.2 PIPELINE FACTORING AND THE BIDIRECTIONAL CONTRACT
The 87.5% yield is driven by two choices: what the LLM produces, and how the output is filtered. Endless
Terminals asks the LLM to generate the Dockerfile, setup script, planting logic, and tests for each task
(essentially a complete container). In our 𝑛=120 reproduction (17.5% portable), 71.7% of failures occur at
the Docker-build or test-validation stage, and only 10.8% at the LLM task-generation step. The model is
competent at inventing hiding techniques but rarely succeeds at the surrounding infrastructure re-derived from
scratch each time. We therefore lift infrastructure out of the per-attempt budget: a pre-debugged ctf-base
image handles the system layer, and the LLM produces only the parameterized plant.sh / recovery.sh
pair. On top of that we apply two complementary filters: a recursive search of the filesystem must not find the
flag in plaintext, and a re-run of recovery.sh on a fresh directory with a fresh flag must recover it. Upstream
pipelines use only one half: Endless Terminals filters on whether at least one of sixteen OpenAI o3 (OpenAI,
2025) samples solves; SWE-smith and R2E-Gym filter on whether a generated change invalidates a unit test.
Both check solvability without checking non-triviality.

3.4 PRODUCTION RATES
On the most recent pipeline run, 712 of 750 raw harvest attempts survived stage 1 (exploration), 700 survived
stage 2 (mechanical verification), 693 survived stage 3 (synthesis), and 656 survived stage 4 (portability),
giving an end-to-end portability rate of 87.5%. The 0.85-similarity deduplication in stage 5 collapses these 656
variants to 441, which aggregate into 155 canonical technique identifiers (median 2 variants per ID, long
tail to 16). Full per-stage survival numbers are in Appendix B.

3.5 COVERAGE OF THE UNIX-COMPETENCE SURFACE
Post-hoc taxonomy assigns the 155 IDs to sixteen families spanning filesystem metadata, binary and ELF
internals, the shell environment, system services, shell-level cryptographic primitives, processes and IPC, logs,
text-processing tools, databases and serialization formats, media and network formats, and encoding and
steganographic devices (Figure 3a). Configuration files (∼22 IDs), text tools (∼22), and text encoding and
ciphers (∼16) are the largest general-purpose Unix surfaces. The smaller families are the distinctive ones the
library is designed to exercise: filesystem metadata (∼12), processes and IPC (∼12), binary/ELF (∼8), and
crypto/cert/token formats (∼9). Every canonical entry has a canonical solution of one to five shell commands
invoking an OS- or format-specific feature. Six worked exemplars (with plant.sh/recovery.sh pairs)
are in Appendix D; two end-to-end solver trajectories illustrating the agent behavior the surface elicits are in
Appendix E; the full list of 155 identifiers is in Appendix N; the family table with representative IDs is in
Appendix C.

4 ENVIRONMENT GENERATION
4.1 MULTI-FLAG CONTAINERS
Given a list of eight techniques and a seed, the environment generator instantiates a pre-built ctf-base
container (Ubuntu plus standard tools). It can also dress the filesystem as one of seven plausible server roles:
webserver, database, devbox, cicd, mailserver, monitoring, or gateway. Each role supplies
role-appropriate users, hostnames, service configurations, fictitious logs, and shell histories from templates; for
example, an agent in the webserver role sees www-data’s crontab, nginx configuration, and SSL certificates.
For each of the eight techniques, the generator samples a target directory from /home/user/, /var/tmp/,
/opt/app/, /srv/, and /tmp/, generates a fresh flag, base64-encodes the technique’s plant.sh into
the container, runs it with the chosen target directory and flag, and records its exit status and output. The
plant script is then deleted, so it cannot be recovered from disk. The routine returns a MultiFlagEnv
containing the container name, seed, and manifest of planted flags.
We plant eight flags per container rather than one. With a single flag, per-rollout reward is binary, the policy
receives signal only on the (initially rare) successful rollouts, and group-relative methods like GRPO lose most
batches to constant-reward filtering. Spreading reward across multiple sub-tasks per episode is a standard
densification move in sparse-reward RL (Andrychowicz et al., 2017; Ng et al., 1999), and we adopt it as a
practical training choice for validating the library under RL rather than a claim about the optimal 𝑛flags . The


                                                           6
trade-off is that each container consumes eight techniques rather than one, which the pool turnover below
partially compensates for.

Reward. Per-rollout reward is points_earned − turn_cost × turns_used, with turn_cost = 1 (a maximum
penalty of about −18) and per-technique points defaulting to fifteen. Each incorrect flag submission costs
one point. We do not clip reward at zero; negative-reward rollouts still provide within-group gradient signal.
The shape encodes two objectives directly (find the flag, and find it without burning the turn budget) and
was chosen, not swept; we leave reward-shape sensitivity to future work.

4.2 STICKY-POOL SKILL SCHEDULING
The scheduler maintains a rotating pool of fifty techniques from the library. Each environment samples eight
techniques uniformly from the current pool. After each batch, _rotate_pool drops the top rotation_rate ×
pool_size = 15 pool members by per-technique solve rate, so mastered techniques graduate out, plus one
guaranteed random removal for exploration. Replacements come from the ∼105 techniques currently outside
the pool, preferring those with the fewest prior attempts. Each batch exposes the model to 8 groups × 8 flags =
64 technique-instances, and each rotation exchanges fifteen techniques. Across the forty-batch runs we report,
the scheduler cycles most of the 155-technique library through the pool at least once.

5 EVALUATION PROTOCOL
5.1 TRAINING AS A PROBE
We fine-tune Qwen3-8B (Qwen/Qwen3-8B) (Yang et al., 2025a) with rank-32 LoRA adapters. Training is
used as a probe of the generated surface, not as a claim that this recipe is an optimal RL pipeline.

Optional SFT format pass. Before GRPO, we optionally run a brief supervised pass on 492 successful solver
trajectories from an earlier Haiku-solver run against the library. The purpose is to align the model’s output
style and tool-call format with the unix-ctf harness (∼92 gradient updates; full hyperparameters in Appendix I).
We treat this stage as harness adaptation rather than capability transfer; one of the two GRPO initializations
skips it entirely. The empirical claims concern the environment’s pedagogical content: whether GRPO on the
unix-ctf surface produces measurable transfer. They are not claims about the intrinsic properties of Qwen3-8B.
We use Qwen3-8B because it produces well-formed tool calls under our harness’s tool specification without
adaptation. Preliminary attempts with Llama-3.1-8B-Instruct, Kimi-K2.5, and Nemotron-3-Super-120B-
A12B exposed harness-renderer compatibility issues: their native tool-call serializations did not round-trip
cleanly. We therefore treat model selection as a precondition for measurement rather than a variable to ablate.
The SFT format pass extends the same harness-alignment discipline for one of the two GRPO initializations.

GRPO. From either the SFT-formatted checkpoint or the unmodified base model, we run forty batches of
group-relative policy optimization with 8 groups × 8 rollouts (64 trajectories per batch), up to 18 turns per
rollout, temperature 1.0, KL penalty coefficient 0, and remove_constant_reward_groups enabled.
Fifteen technique families spanning file encoding, filesystem, archives, databases, logs and system state, user
accounts, text processing, checksums, and process control are held out of the training pool via the technique-
blacklist mechanism (§6.1); the full list appears in Appendix K. Each batch runs eight concurrent Docker
containers on a single n2-highmem-16 GCP instance.

5.2 VARIANTS AND BENCHMARKS
We compare three variants: Base Qwen3-8B (untrained), GRPO from base init (40 batches of GRPO
from the unmodified base), and GRPO from SFT init (40 batches of GRPO from the SFT-formatted
checkpoint). The two GRPO runs share hyperparameters and training-budget settings but use independent
random orderings drawn by the sticky-pool scheduler (§4.2); the principal intentional difference is the brief
format-adaptation pass that precedes one of them. All evaluations use the final checkpoint.

6 EMPIRICAL VALIDATION
We use training as an experimental probe of the generated surface, not as a tuned RL recipe. The principal
table is in Appendix A.

6.1 THE SURFACE IS LEARNABLE
We fine-tune Qwen3-8B with rank-32 LoRA (Hu et al., 2022) adapters using GRPO (Shao et al., 2024b),
then evaluate on the same 15-skill held-out set (full list in Appendix K). Base solves 11.6%; GRPO from base
init reaches 27.6%; GRPO from SFT init reaches 43.6%. Both GRPO initializations transfer substantially
above base, and the SFT-formatted initialization adds another +16.0 pp over the base initialization on this
multi-family set. On timeout_race alone, a dedicated 𝑛=30 run gives GRPO-from-SFT-init 20/30 versus
base 8/30 (Fisher’s exact 𝑝 ≈ 0.004); details are in Appendix K.

6.2 TRANSFER TO INTERCODE-CTF
On the external InterCode-CTF benchmark (100 tasks, picoCTF-2019), the GRPO variants change which
categories the model solves more than they change the aggregate. The largest category-level transition
is GRPO from base init on Forensics: 1/15 → 6/15, a +33 pp gain (Figure 5b). Reverse Engineering
also improves. GRPO from SFT init shows the same qualitative redistribution despite no aggregate
IC-CTF gain (Figure 5a). The training surface therefore appears to raise solve rate preferentially on the
Unix-competence-adjacent subset of IC-CTF. Full task-level decomposition and the per-category table are
in Appendix F.1. A secondary RL-pipeline observation is that the two GRPO initializations diverge between
training reward and IC-CTF transfer. GRPO-from-SFT-init peaks at +27.0 batch reward but reaches only
23% on IC-CTF, while GRPO-from-base-init never exceeds +1.45 and reaches 32%. The ranking inverts;
per-batch curves and discussion are in Appendix G.
Two GRPO initializations diverge across target distributions. GRPO from base init reaches 27.6%
on the multi-family holdout and 32% on IC-CTF; GRPO from SFT init reaches 43.6% on multi-family but
only 23% on IC-CTF. The ranking inverts. We use the SFT pass only for harness format adaptation and
do not isolate its mechanism; practitioners should treat format-adaptation data as an experimental variable
rather than a fixed step. Each variant is single-seed; per-skill rates are noisy (𝑛≈15); we do not ablate 𝑛flags , the
scheduler, per-role rates, or non-final checkpoints; IC-Bash answer_similarity is identically zero across
variants (Appendix J).

D.1 XATTR: STORING THE FLAG IN A POSIX EXTENDED ATTRIBUTE ON A DECOY FILE
 # plant.sh
 echo "nothing to see here" > "$target_dir/decoy.txt"
 encoded=$(echo -n "$flag" | base64)
 setfattr -n user.secret -v "$encoded" "$target_dir/decoy.txt"


 # recovery.sh
 getfattr --only-values -n user.secret "$target_dir/decoy.txt" | base64 -d

The decoy file prints “nothing to see here” under cat, and standard inspection commands such as stat,
ls -l, file, strings, and a recursive grep for the flag prefix do not surface the flag. The data lives
in the user.* extended-attribute namespace, a POSIX/Linux filesystem feature with no direct analogue
in ordinary programming; the canonical access path is getfattr -n, or equivalently the getxattr(2)
syscall. Solving the task requires knowing that this namespace exists and identifying the attribute name to
query, or dumping all user-namespace attributes on every file. A solver who uses Python still has to invoke
os.getxattr, which is precisely the Unix knowledge being tested.

D.2 ELF_GNU_BUILD_ID: EMBEDDING THE FLAG AS THE .NOTE.GNU.BUILD-ID OF A
    STOCK ELF
 # plant.sh (abbreviated)
 flag_desc=$(echo -n "$flag" | xxd -p)
 python3 <<PYEOF
 import struct
 name = b"GNU\x00"
 desc_raw = bytes.fromhex("$flag_desc")
 desc_padded = desc_raw + b"\x00" * ((4 - len(desc_raw) % 4) % 4)
 note = struct.pack("<III", len(name), len(desc_raw), 3) + name + desc_padded
 with open("buildid.note", "wb") as f: f.write(note)
 PYEOF
 cp /bin/ls ./hidden_elf
 objcopy --update-section .note.gnu.build-id=buildid.note ./hidden_elf ./
     hidden_elf_marked
 rm -f buildid.note hidden_elf
 mv hidden_elf_marked hidden_elf


 # recovery.sh
 readelf -n "$target_dir/hidden_elf" 2>/dev/null \
     | grep -oE ’Build ID: [0-9a-f]+’ | awk ’{print $3}’ | xxd -r -p

The file utility identifies hidden_elf as a 64-bit ELF executable, while strings surfaces nothing
because the flag is hex-encoded. Solving the task requires familiarity with the ELF note-section convention
and with readelf -n.

E WORKED SOLVER TRAJECTORIES
To make the kind of agent behavior the library elicits concrete, we reproduce two successful solver trajectories
from a Claude Haiku 4.5 reference run. Lines beginning with > are the agent’s narration before each tool call;
lines beginning with $ are the bash invocations the agent issued; following lines are the captured stdout. Tool
calls and tool returns are abridged for space; the full message logs are recorded in the project artifact directory.

Exemplar 1: passwd_shadow (3 turns). The instruction primes the agent for “a system administrator
account [with] hidden information embedded in its metadata.” The agent recognizes the GECOS convention
of /etc/passwd (the comma-separated metadata in the fifth field) and extracts a base64-encoded flag.
[turn 1]

> administrator account metadata.
$ whoami

$ cat /etc/passwd | grep -E "root|admin"

[turn 2]

> sysadmin2024 has what looks like base64-encoded text in the GECOS field
> (the comment field): ZmxhZ3syM2FiZGM0NGI5ODliMjJjfQ==. Let me decode it.
$ echo "ZmxhZ3syM2FiZGM0NGI5ODliMjJjfQ==" | base64 -d

[turn 3]
> Perfect -- the flag was hidden in the GECOS field of the system
> administrator account. The encoded string decoded to reveal the flag.
submit_flag(flag{23abdc44b989b22c}) -> Acknowledged.


The Unix-specific knowledge here is the structure of /etc/passwd (seven colon-separated fields, with the
fifth carrying free-form GECOS metadata as comma-separated subfields) and the convention of stashing

carved images, filesystem metadata, and steganographically concealed data. The +33 pp improvement is
consistent with the training distribution teaching these skills, and is our most direct evidence that the library’s
coverage transfers to a third-party benchmark.

M SKILL-CATEGORY CLASSIFICATION RUBRIC (FIGURE 2)
The same judge model (gpt-5.4-nano (OpenAI, 2026)) classifies each task into one of four skill categories.
The per-task input is the instruction text, canonical solution, and grading specification. The output is
{"category": "<one of four>", "reason": "<one sentence>"}.
Discriminating question for unix_competence vs the others: "Could a competent
Python programmer who knew nothing about Unix internals beyond subprocess.run
solve this task by reading the prompt and writing Python?" If yes, NOT
unix_competence.

CATEGORY: unix_competence
The canonical solution invokes a NON-OBVIOUS OS-, kernel-, or format-internal
feature that most engineers would not know to look for without being told. The
Python-stdlib path either does not exist or requires invoking the same OS
feature via syscall/ctypes.
Examples: getfattr, readelf -n, debugfs/find -inum, systemctl in non-trivial
ways, git plumbing (cat-file, pack files, reflog format), POSIX IPC (named
pipes, signals, fifos), filesystem metadata (sparse files, ACLs, capabilities),
pre-epoch mtimes, X.509 custom OIDs, kernel /proc internals.
NOT unix_competence: sqlite3, tar, gzip, awk, sed, find, git status/log/commit,
mkdir, chown, chmod, formatted-text generation, CSV/JSON manipulation -- these
all have clean Python equivalents.

CATEGORY: shell_flavored_coding
Glue pipeline of core text utilities (find, grep, awk, sed, sort, cut, tr, head,
tail, uniq) and/or familiar binaries (sqlite3, tar, gzip, jq, curl) for which
a Python-stdlib (or pandas/sqlite3 module) equivalent of comparable length exists.
Shell is convenient but not load-bearing.

CATEGORY: general_programming
Deliverable is a non-trivial program; success depends on programming proficiency
in a general-purpose language. Multi-hundred-line solve scripts, custom
algorithms, ML training, parsers/interpreters, network services.

CATEGORY: other
Not primarily about shell, OS features, or programming. Pure cryptographic
puzzles solvable with paper-and-pencil, web exploitation, single-string decoding,
trivia.
