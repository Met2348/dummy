1. Introduction

As AI agents move from screen-bound workflows toward the broader world around users, they are expected
to spend increasing amounts of time participating in everyday life as general-purpose assistants. To do so,
they must handle multi-modal information from the embodied environments, especially video and audio.
This immediately raises a question: Is native video and audio understanding necessary to an agent that is
expected to process video or audio in real-world tasks?
In practice, however, many commonly-adopted agents nowadays, such as OpenClaw (OpenClaw, 2026) and
Hermes (Nous Research, 2026), are still text-only or text+image systems, while extending them to handle
video and audio natively requires substantial additional modeling and engineering effort. This creates a
large gap between real-world agent deployment and the dominant paradigm for video-audio processing,
namely native omni-modal models that directly ingest audio and video (Qwen Team, 2026, The Gemini Team,
2026b). At the same time, modern coding agents already make highly effective use of terminal environments
for search, editing, execution, and long-horizon task solving (Yang et al., 2024a, Anthropic, 2026a, Chen,
2026). This raises a straightforward question:


 Can coding agents, which are not designed for native multimodal perception, solve video and audio
      tasks by exploiting existing tools like ffmpeg, ASR, or new tools they write in a sandbox?


We investigate this problem using the most native Codex and Claude Code harnesses, without any human-
provided tools or skills. Text+image coding agents, namely GPT-5.4 high/xhigh and Claude Opus 4.6
(OpenAI, 2026, Anthropic, 2026b,a), achieve strong performance on video-audio benchmarks such as
OmniGAIA, SocialOmni, and LVOmniBench (Li et al., 2026c, Xie et al., 2026, Tao et al., 2026). Codex with
GPT-5.4 xhigh reaches 75.0% on OmniGAIA without directly perceiving raw video or audio, outperforming
the strongest native omni-modal model in our study, Gemini 3.1 Pro (The Gemini Team, 2026b), by more
than 8%, while Claude Opus 4.6 exceeds it by 2.5%. Trajectory analysis shows that this advantage does
not come from end-to-end perception. Instead, coding agents selectively decompose video and audio into
transcripts, sampled frames, and metadata through staged tool use, which also consumes substantially fewer
tokens than packing raw streams into model context.
To push coding agents on omni content further, we ask where they still fail and whether such failures can
be repaired without retraining the base model. We sample 200 trajectories across models and benchmarks
and propose an Omni Terminal Failure Taxonomy together with a process-level evaluation, which reveals
six recurring failure modes ranging from premature exploration of multimodal content to brittle tool and
environment infrastructure (Kim et al., 2025, Yang et al., 2025b, Liang et al., 2025). Guided by this taxonomy,
we then study three skill-injection settings inspired by recent work on reusable agent skills and harness
optimization (Li et al., 2026b, Lee et al., 2026): human-in-the-loop skills that encode expert workflows,
calibration-set self-iteration where the agent revises its skills using only binary correctness feedback, and
log-driven self-distillation where a separate agent mines reusable skills from raw execution traces. All three
improve over the no-skill baseline, raising average OmniGAIA accuracy from 61.4% to 76.7% in the best
setting, with execution-trace distillation outperforming both binary self-iteration and human-written skills.
The above results are obtained with closed-source frontier models, which raises a natural follow-up question:
can the same capability be elicited on open-source models through targeted post-training? As an exploratory
baseline, we propose Code-X, a complete recipe for many-modality terminal agents. We construct Om-

                                                                                                             2
niCoding, a 6,035-example trajectory dataset spanning video, audio, image, and cross-modal tasks, and
train Qwen-3.5-9B and Qwen-3.6-27B with a supervised warm-start followed by GSPO-based reinforcement
learning under a process-aware verifiable reward that jointly penalizes modality-skipping, malformed tool
calls, and answer mismatches. Evaluating on four challenging benchmarks, Code-X yields substantial gains
over direct MLLM baselines and achieves state-of-the-art results among open-source models, with the 27B
model reaching 43.3% on OmniGAIA, 60.0% on LVOmniBench, where it surpasses all existing coding agents,
and more than doubling performance on VideoZeroBench.
Given the maturity of many-modality understanding, we believe the meaningful frontier lies in many-modality
processing: editing videos, auditing image databases, producing spec-compliant multimodal outputs, where
coding agents enjoy a structural advantage over end-to-end omni models. To draw attention to this second
half of many-modality, we introduce TerminalBench-O, the first process-level benchmark for coding
agents on omni-modal processing, with 50 real-world tasks across video, audio, image, document, and
cross-modal categories, each evaluated along weighted dimensions over intermediate artifacts and final
output compliance. Despite strong omni-understanding results, these tasks remain hard for current agents
as they demand long-horizon planning, reliable tool chaining, cross-modal grounding, artifact verification,
and strict output adherence; even the strongest GPT-5.5 Codex reaches only 24% on TerminalBench-O.
We hope our analysis, training recipe, and benchmark open new directions for many-modality processing.

2. Analysis of Sandboxed Agent Performance on Omni-Modal Tasks
Preliminary: Adapting Sandboxed Agents for Omni-Modal Tasks. We represent each omni-modal task as
x = (q, ℱ, 𝒴), where q is a natural-language instruction, ℱ is a set of input files, and 𝒴 is the set of acceptable
final answers (formal definitions and the full rollout formulation are deferred to Appendix C). A native
omni-modal model and a sandboxed agent differ mainly in where omni-modal processing happens:
                                                 pack                   Mθ
                         native omni:     (q, ℱ) −−−→ model context −−→ ŷ,
                                                 stage                                      Ans
                                          (q, ℱ) −−−→ workspace −−−−−−−−−−−−−→ τ −−−→ ŷ.
                                                                    terminal interactions
                   sandboxed agent:

A native omni-modal model Mθ packs all files into model context via media-specific encoding and sampling,
treating video and audio as first-class model inputs to generate answers ŷ. A sandboxed agent, by contrast,
stages ℱ into an isolated workspace and is given only the instruction and file paths; it then inspects files,
invokes tools such as ffmpeg, ffprobe, OCR, and ASR, writes and executes scripts, produces intermediate
artifacts (transcripts, sampled frames, OCR text, timestamps, tables, crops), which compose a trajectory
τ, and submits a final answer ŷ parsed from τ. This view treats raw video, audio, images, and documents
as environment state rather than mandatory model context, so the agent can solve omni-modal tasks by
adaptively reducing raw media into compact evidence that fits the native reasoning interface of a general-
purpose coding model.
Agent Harness and Models. We use Claude Code (Anthropic, 2026a) and Codex (Bolin, 2026, Chen,
2026) as our two primary agent harnesses. Under CC, we evaluate Claude Opus 4.6 and Claude Sonnet 4.6
(Anthropic, 2026b,c), as well as MiniMax M2.7 (MiniMax, 2026) and Kimi K2.5 (Kimi Team et al., 2026).
Claude Opus 4.6 and Claude Sonnet 4.6 both use the default max setting. Under Codex, we evaluate GPT-5.4
with low, medium, high, and xhigh reasoning effort settings (OpenAI, 2026). Throughout evaluation, we
follow the official harnesses as closely as possible in order to approximate realistic end-user usage.


                                                                                                                 3
For native omni models, we evaluate Gemini 3.1 Pro (The Gemini Team, 2026b), Gemini 3 Flash (Doshi,
2025), Qwen 3.5-Omni (Qwen Team, 2026), and MiMo-V2-Omni (Xiaomi, 2026). We further include two

using high reasoning effort (OpenAI, 2026).
Evaluation Protocol. We use four benchmarks with complementary emphases: OmniGAIA for open-ended,
tool-mediated factual and multi-hop omni-modal reasoning (Li et al., 2026c), SocialOmni for audio-visual
social perception and identity recognition (Xie et al., 2026), LVOmniBench for long-form audio-video
understanding (Tao et al., 2026), and VideoZeroBench for challenging video-centric reasoning (Meng et al.,
2026). Given their scale, we evaluate the first 100 examples per level on SocialOmni and LVOmniBench, and
on SocialOmni we report only Level-1 and Level-2 Q1 accuracy, omitting the aggregate Level-2 score since it is
computed only over Level-2-eligible examples and introduces substantial selection bias. For VideoZeroBench,
questions on the same video are asked in a grouped manner. Unless otherwise noted, all models are evaluated
with pass@1, and the agent receives only the task instruction and filesystem path(s) to the input files, with
no pre-built Skills.
Environments. All sandboxed-agent experiments run in a controlled workspace based on Ubuntu 22.04.3
LTS with Python 3.13.2 and standard Unix utilities. By default, models may access web search and
download additional models or dependencies; we inspect trajectories to detect answer leakage and evaluator
hacking. The environment provides common multimedia and scientific tooling (e.g., ffmpeg/ffprobe,
Tesseract OCR, ImageMagick, OpenCV, Librosa), and agents may freely install packages and author task-
specific scripts for video and audio processing.


2.1. Main Results and In-depth Analysis

From Table 1, coding agents prove highly competitive with native omnimodal LLMs across all four benchmark
suites. On OmniGAIA, GPT-5.4 xhigh (75.0) clearly outperforms Gemini 3.1 Pro (66.1), especially on
medium- and hard-level tasks; on VideoZeroBench, the best coding agent (27.6) also exceeds Gemini 3
Flash (17.8). Native omni models lead on Social Omni Level-1, though Claude Opus 4.6 slightly edges out
the best native score on Level-2 Q1. On LVOmniBench, Gemini 3.1 Pro remains the strongest overall, but
coding agents stay competitive on the easy and medium subsets. These patterns suggest that benchmarks
emphasizing fine-grained temporal perception and audio–visual alignment favor end-to-end omni modeling,
while tool-oriented benchmarks such as OmniGAIA align more naturally with the coding-agent paradigm.

 Finding 1: Coding agents with only text+image capability can perform competitively with, and in some
 cases outperform, state-of-the-art native omnimodal LLMs on omnimodal tasks.

Across the GPT-5.4 variants under the same Codex harness, increased reasoning effort yields clear gains on
omnimodal performance. On OmniGAIA, average accuracy rises steadily from 55.0 (low) to 56.4 (medium),
61.4 (high), and 75.0 (xhigh), with improvements appearing across easy, medium, and hard subsets. The
same trend is visible on LVOmniBench, where higher-effort settings outperform lower-effort ones, though
not strictly monotonically. Since the model family and harness are held fixed, this isolates the contribution of
inference-time computation alone.

Token Usage Analysis. Given the concern of using more tokens for better performance, we calculate the
avg. token input and accuracy across four benchmarks. As shown in Figure 3, surprisingly, coding agents

use, the model selectively retrieves only the content it needs, substantially reducing the multimedia footprint
in its context and thereby eliminating a significant amount of redundant tokens.

benchmarks: many tasks can be solved through text or shallow perception (e.g., OCR) rather than deep

Given that closed-source coding models outperform native omni-modal models in both performance and

3.2. Method and Training Infrastructure

We train the terminal agent with supervised warm-start followed by reinforcement learning from verifiable
rewards. All training examples follow the schema x = (q, ℱ, 𝒴), where multiple-choice tasks use option-
letter answer sets and open-ended tasks are assigned curated answer sets covering common aliases, units,
abbreviations, capitalization, and number formats. The agent submits its final answer inside <answer>.
Supervised warm start. For examples with demonstration trajectories, we fine-tune on assistant-generated
tokens only, masking terminal outputs, tool responses, and environment observations from the loss. This
stage teaches the basic interaction protocol: inspect files, run commands, create intermediate artifacts,
recover from tool failures, and submit answers in the required format.
Reward design. We design a process-aware verifiable reward that goes beyond binary answer matching,
addressing three failure patterns: (1) the agent guesses without engaging the workspace, (2) it ignores
the relevant modality (e.g., answering a video question from text alone), and (3) it emits malformed,
escape-attempting, or otherwise broken tool calls. For each rollout τ we compute

Training infrastructure. Our RL system decouples sandboxed rollout generation from policy optimization.
Rollout workers run the current policy in isolated workspaces, execute terminal commands, collect trajectories,
parse answers, compute reward components, and return old log-probabilities and masks; the trainer consumes
these logs and applies GSPO without executing tools or touching raw media. This separation is useful because
omni-modal rollouts involve non-differentiable operations, while the optimizer only needs token sequences,
masks, old log-probabilities, and scalar rewards. Sandboxes are hosted on a Slurm cluster with CPU/GPU
nodes and storage, the trainer runs on cloud servers, and the two are connected via a Cloudflare tunnel for
trajectory collection.

only a 4% pass rate and an average score of 0.55. Overall, the results show that TerminalBench-O
remains challenging across different model and harness configurations, especially for tasks requiring reliable
long-horizon tool use.

5. Related Works
Omnimodal Models and Evaluation Recent multimodal foundation models have advanced from single-
modality extensions to omni-modal systems that jointly process speech, audio, images, and video (Chu et al.,
2024, Sun et al., 2024, Xu et al., 2025, Qwen Team, 2026, The Gemini Team, 2026a). As models push
toward unified vision-and-speech interaction, evaluation has correspondingly shifted from task-specific video
or audio benchmarks (Li et al., 2023, Liu et al., 2024, Fu et al., 2024a, Wu et al., 2024, Li et al., 2022) to
omni-modal suites that more explicitly target joint reasoning over visual, auditory, and textual cues, with
growing attention to long videos, grounded interaction, and real-world complexity (Li et al., 2024b, Hong
et al., 2025, Zhang et al., 2025c, Li et al., 2026c, Radwan et al., 2026, Tao et al., 2026, Xie et al., 2026). Yet
our audit shows that many supposedly audio-visual tasks still allow strong shortcuts or rely only weakly on
audio (Kim and Seo, 2025), motivating more careful benchmark construction.
Multimodal Agentic Tool-Use and Sandboxed Agents. Tool use has become a central mechanism for
extending LLM agents beyond their parametric knowledge (Yao et al., 2022, Schick et al., 2023, Qin
et al., 2023, Patil et al., 2025, Yao et al., 2024, Wang et al., 2025), and increasingly improves multimodal
understanding via active search, code generation, and tool orchestration without retraining a monolithic
model (Hu et al., 2024, Zhao et al., 2025, Zhang et al., 2025b, 2024a, 2025a, Tao et al., 2025b, Lin et al.,
2025, Li et al., 2026a). The most mature form of sandboxed agency has emerged in software engineering,

                                                                                                                                       15
where models resolve real issues inside an execution environment (Jimenez et al., 2023, Yang et al., 2024a,
Wang et al., 2024c, Kwa et al., 2025, Cheng et al., 2026), but existing work primarily targets natively textual
or computational tasks. Our work takes a step beyond: we show, for the first time, that a model can use
a sandboxed terminal with self-constructed tool use to solve non-native multimodal tasks, outperforming
many specialized omni models and suggesting that sandboxed agency is a general mechanism for broader
many-modality problem-solving.


6. Conclusion

We study offline omnimodal understanding through the lens of sandboxed coding agents in both training-free
and fine-tune settings. Across multiple benchmarks, these agents solve a surprisingly large fraction of video-
audio tasks by selectively retrieving and processing evidence from raw inputs, often rivaling or surpassing
native omnimodal systems. We hope this work motivates a deeper study of multimodal intelligence at the
model–tool–environment interface.

A. Limitations

Our study has several limitations. First, due to computing-resource constraints, the Code-X training recipe is
only validated at the 9B and 27B scales (Qwen-3.5-9B and Qwen-3.6-27B), and we have not yet characterized
how the recipe scales to frontier-size open-source models. Second, due to API credit constraints, several
closed-source coding agents could not be evaluated on the full benchmark suite. In particular, Claude Opus
4.6 results on VideoZeroBench and Claude Opus 4.7 results across all four benchmarks are incomplete
in the current draft, which means the comparison between coding agents and native omni-modal models
is somewhat uneven on the closed-source side. Third, our scope is restricted to offline omni-modal tasks,
where files are staged into a workspace ahead of time; conclusions do not transfer directly to streaming or
interactive settings.


B. Broader Impacts

Our findings establish a more compute- and token-efficient pathway for offline omni-modal understanding,
showing that text+image coding agents with sandboxed tool use can match or surpass native omni-modal
models on many video- and audio-centric tasks while consuming fewer tokens. This lowers the resource
barrier for omni-modal applications and broadens access for researchers and users who lack dedicated
omni-model APIs, while our open-sourced Code-X recipe, OmniCoding dataset, and TerminalBench-O
benchmark further support reproducible research at the model–tool–environment interface. As with any
tool-using agent that processes video and audio, downstream deployments should incorporate standard
safeguards such as content provenance, access controls, and user consent; we note that our agents operate
only on user-staged files within an isolated workspace and do not autonomously take real-world actions.


C. Adapting Sandboxed Agents for Omni-Modal Tasks

We represent each omni-modal task as

where q is a natural-language instruction, ℱ is a set of input files, and 𝒴 is the set of acceptable final answers.
We use an answer set rather than a single reference string so that aliases, option letters, numerical formats,
and unit variants can be handled uniformly in both evaluation and reinforcement learning.

interacts with the workspace through a terminal interface: it can inspect files, invoke tools such as ffmpeg,
ffprobe, OCR, or ASR systems, write and execute scripts, create intermediate artifacts, and finally submit
an answer.

D. Discussion and Future Works
Do non-native modalities really require native end-to-end perception? Our results suggest that non-native
modalities do not always require native end-to-end perception. Even in the simplest Codex and Claude Code
settings, coding agents solve a surprisingly large fraction of video-audio tasks, and on OmniGAIA they can
even outperform strong native omnimodal baselines. The key is not that the base model suddenly becomes
an audio-video perceiver, but that the sandbox lets it decompose raw multimedia into artifacts that match
its native reasoning interface, such as transcripts, key frames, timestamps, OCR, metadata, and compact
summaries. In this sense, multimodal processing becomes analogous to protocol-based software access in
text-domain agents: the model delegates non-native operations to the environment and then reasons over
the returned evidence. This perspective is consistent with the broader shift from static function calling
to environment-mediated problem solving in tool-use and terminal-agent evaluation (Wang et al., 2025,
Merrill et al., 2026a, Cheng et al., 2026). It also resonates with recent tool-augmented perception work,
where explicit visual-tool policies and tool-derived perceptual signals both improve fine-grained multimodal
reasoning (Zhou et al., 2025, Wei et al., 2026). A natural next question is therefore not only whether a model
is natively omni, but which capabilities should remain externalized, which can be compiled into reusable
skills, and how agents should detect and recover from imperfect tool outputs. Our failure taxonomy and
process-level benchmark point in this direction, suggesting that the central bottleneck is often tool selection,
verification, and repair rather than the mere absence of raw multimodal encoders.
Shortcuts in current video-audio benchmarks. Are current video-audio benchmarks measuring what they
claim to measure? Our findings suggest only partially. When a text+image coding agent can answer many
ostensibly omnimodal tasks by selectively extracting transcripts, sparse frames, or metadata, benchmark
accuracy no longer cleanly measures native end-to-end audio-video understanding. This does not make such
benchmarks uninformative, but it does mean that they frequently conflate at least three capabilities: native
perception, external evidence acquisition, and general knowledge-based reasoning. Our results are aligned
with recent audits and diagnostic studies showing that many video or audio-visual benchmarks admit strong

                                                                                                             26
visual, textual, or temporal shortcuts (Kim and Seo, 2025, Lim et al., 2026). They are also consistent with
benchmark-design efforts that explicitly enforce stronger audio dependence or reduce text-based shortcut
channels (Wei et al., 2026). Going forward, we believe benchmark design should standardize at least
three controls: reporting text-only, text+image, and agentic baselines; auditing modality dependence with
masking or filtering tests; and minimizing leakage from subtitles, transcripts, metadata, and external retrieval
whenever the target claim is native multimodal understanding. Conversely, when the target is multimodal
agentic competence, retrieval, tool orchestration, and evidence processing should be treated as first-class
evaluation targets rather than accidental shortcuts. Making this distinction explicit would help separate
backbone perception from system-level problem solving, which are both important but not interchangeable.
Can the same sandbox move from understanding to generation? We believe this is the most promising
next step. Once an agent can inspect raw video, audio, and documents, compress them into structured
evidence, and maintain task-relevant state over long horizons, it can in principle also plan edits, retrieve
assets, invoke generative models, and validate outputs in a closed loop. Early multimedia-generation and
agentic video-editing systems already point in this direction, showing that multimodal agents can coordinate
text, images, video, music, and external tools under explicit planning and feedback loops (Zhang et al.,
2026, Zhao et al., 2026). The next frontier is to connect that generative loop with persistent multimodal
memory, so that an assistant can build private memos, recap videos, highlight reels, or even broadcast-ready
drafts grounded not only in a single uploaded file but in a user’s longer-term audiovisual history (Liu et al.,
2026). Doing so will require stronger memory management, preference modeling, provenance tracking,
and process-level verification, since generated content must remain faithful both to source media and to
user intent. More broadly, this points to a different picture of future omni assistants: not a single monolithic
model that passively consumes everything in one shot, but a sandboxed system that can inspect, remember,
transform, and generate across modalities with the right mix of reusable skills and external tools. Under this
view, offline omnimodal understanding is not the endpoint of this line of work, but the substrate for practical
multimedia copilots.

E. Full Related Works

E.1. Multimodal Models and Evaluation

Recent progress toward omni-modal assistants reflects a shift from modality-specific perception modules to
unified models that can process speech, general audio, images, and video within a single system. On the
model side, audio-language systems such as Qwen2-Audio (Chu et al., 2024) and video-centered models
such as VideoLLaMA 2 (Cheng et al., 2024) and video-SALMONN (Sun et al., 2024) established strong
foundations for audio-visual understanding. More recent omni-modal systems, including VITA (Fu et al.,
2024b), OpenOmni (Luo et al., 2025), Baichuan-Omni (Li et al., 2024a), Moshi (Défossez et al., 2024),
and Qwen2.5-Omni (Xu et al., 2025), move toward unified and often real-time interaction across speech
and vision. These systems substantially improve the practicality of omni interaction, but they also make it
harder to separate intrinsic model ability from gains due to decoding strategies, longer context, or external
processing pipelines.
Benchmarking has evolved in parallel. Earlier suites mainly targeted either video understanding, such
as MVBench (Li et al., 2023), Video-MME (Fu et al., 2024a), and MLVU (Zhou et al., 2024), or audio /
audio-visual understanding, such as AIR-Bench (Yang et al., 2024c), MMAU (Sakshi et al., 2024), and
AV-Odyssey (Gong et al., 2024). While these benchmarks advanced multimodal evaluation, many still
under-represent audio, weakly couple audio with video, or emphasize short-context perception more than
long-horizon multimodal reasoning.
Recent work therefore moves toward explicitly omni-modal and long-form evaluation. Benchmarks such as
OmniBench (Li et al., 2024b), OmniEval (Zhang et al., 2025c), OmniGAIA (Li et al., 2026c), LVOmniBench
(Tao et al., 2026), SocialOmni (Xie et al., 2026), MMOU (Goel et al., 2026), and OmniPlay (Bie et al.,
2025) target more realistic settings that require integrating visual, auditory, and textual evidence over longer
horizons. At the same time, recent audits show that several audio-video benchmarks admit strong visual
shortcuts (Kim and Seo, 2025), suggesting that progress in omni-modal evaluation depends not only on
stronger models but also on shortcut-resistant benchmark design and explicit measurement of true modality
dependence.


E.2. Agentic Tool-Use

The literature on tool-augmented language models treats external tools as a way to expand the action space
of models rather than as simple add-ons. Foundational work such as ReAct (Yao et al., 2022) and Toolformer
(Schick et al., 2023) showed that reasoning can be interleaved with actions and that models can learn when
to invoke tools. Later efforts, including Gorilla (Patil et al., 2023), ToolBench (Xu et al., 2023), ToolLLM
(Qin et al., 2023), BFCL (Patil et al., 2025), and τ-bench (Yao et al., 2024), scaled this paradigm to large
tool collections and more interactive, stateful environments. Across these works, the central shift is from
static capability elicitation to environment-mediated problem solving.
Executable code has emerged as one of the most expressive forms of tool use. CodeAct (Wang et al., 2024b)
argues that executable programs provide a general interface for acting in digital environments, allowing
agents to manipulate files, call libraries, and compose multi-step behaviors more flexibly than fixed action
vocabularies. In multimodal settings, Visual Sketchpad (Hu et al., 2024) and MLLM-Tool (Wang et al., 2024a)
show that models can externalize perception into sketches, programs, and structured tool calls. PyVision
(Zhao et al., 2025) and Thyme (Zhang et al., 2025b) further demonstrate that models can synthesize and

                                                                                                             28
execute Python-based image operations on demand, blurring the boundary between model capability and
environment capability.
This perspective is increasingly visible in video and omni-modal reasoning. OmAgent (Zhang et al., 2024a),
OmniAgent (Tao et al., 2025b), Agent-Omni (Lin et al., 2025), and LensWalk (Li et al., 2026a) all treat
multimodal understanding as an active process of decomposition, search, and selective inspection rather
than passive end-to-end perception. LongShOTBench (Kurpath et al., 2025) explicitly evaluates tool use and
iterative refinement in long-video settings. More broadly, LLM-in-Sandbox argues that rich environments can
elicit capabilities not apparent in static chat settings, reinforcing the view that multimodal competence may
emerge from the interaction between a strong base model and an expressive tool substrate (Cheng et al.,
2026). Together, these works motivate viewing multimodal intelligence as a property of a model–environment
system rather than of a standalone backbone.


E.3. Terminal Coding Agents and Benchmarks

The most mature form of sandboxed agency has so far emerged in software engineering. SWE-bench (Jimenez
et al., 2023) established the canonical issue-resolution setting by grounding evaluation in real GitHub issues
and executable repository states. Systems such as SWE-agent (Yang et al., 2024a), AutoCodeRover (Zhang
et al., 2024b), Agentless (Xia et al., 2024), and OpenHands (Wang et al., 2024c) explored different agent
designs for this setting, ranging from search-heavy repair to lightweight execution-driven pipelines. This line
of work reframes coding as long-horizon environment interaction rather than static code generation.
Benchmarks have expanded accordingly. SWE-bench Verified (OpenAI, 2024) improves reliability through a
filtered subset, while SWE-bench Multimodal (Yang et al., 2024b) extends evaluation to visually grounded
software issues. Terminal-Bench (Merrill et al., 2026a) and LongCLI-Bench (Feng et al., 2026) move beyond
repository patching toward broader command-line tasks and longer-horizon CLI interaction. Other recent
benchmarks, including SetupBench (Arora et al., 2025), SWE-Bench Pro (Deng et al., 2025), and SEC-bench
(Lee et al., 2025), further broaden evaluation to environment setup, longer-horizon issue resolution, and
security workflows.
The ecosystem has also expanded beyond benchmark construction into self-improvement and broader auton-
omy evaluation. SWE-smith (Yang et al., 2025a) studies scalable data generation for software engineering
agents, Live-SWE-agent (Xia et al., 2025) explores self-evolution during deployment, and SWE-Fuse (Wen
et al., 2026) uses trajectory learning to improve agent performance. Beyond software-specific tasks, RE-Bench
(Wijk et al., 2024), HCAST (Rein et al., 2025), and TheAgentCompany (Xu et al., 2024) evaluate agents
on longer-horizon work in realistic environments. At the same time, recent meta-analyses caution that
benchmark realism, contamination, and task validity remain open concerns, and that high scores do not
automatically imply broader engineering competence (Martinez and Franch, 2026, Prathifkumar et al.,
2025). These critiques are especially relevant when interpreting terminal-agent performance as evidence of
general agentic ability.

overlays, and auxiliary labels needed to make evaluation precise.


Preprocessing. Each source is normalized into a stable workspace representation before task packaging.
We also prepare closed vocabularies, masks, catalogs, WCS headers, timestamp maps, and source-to-fixture
mappings for downstream grading.


Construction. Step 1: Candidate mining. We first select source segments whose native structure supports
a realistic operation task, such as eventful video intervals, speaker turns, action clips, score-aligned music
excerpts. Candidate segments are filtered for duration, visibility, signal quality, annotation availability, and
non-triviality before fixture generation.
Step 2: Task-specific transformation. We then impose an operation or creation objective on the selected
source. Some tasks add controlled corruptions, hidden mappings, decoys, overlays, splice events, synchro-
nization errors, or multi-source audio compositions. Others keep the media unchanged but require structured
deliverables such as ledgers, catalogs, action datasets, OMR alignments, reading-order tables, or redaction
reports.
Step 3: Hidden reference synthesis. For each instance, we write hidden references that specify the answer
geometry used by the grader. These references include event windows, speaker maps, transcripts, quote
spans, crop paths, clean media, masks, stem targets, timestamp curves, object tracks and closed-vocabulary
labels. They are withheld from the agent but used to score both final outputs and intermediate artifacts.
Step 4: Output-surface packaging. Finally, we expose each task as a small working directory rather than a
question-only prompt. The visible files may include media, manifests, closed vocabularies, policy snippets,
PDFs, CSVs, point clouds, FITS files, or auxiliary metadata, and the expected outputs are ordinary files
such as MP4/WAV/PNG artifacts, CSV/JSON ledgers, subtitles,plots, and reports. This format keeps the
benchmark native to terminal tools, ffmpeg, OCR/ASR, numerical Python, and file-system coordination.


Filtering. We apply filtering before and after packaging. Source candidates are removed if they are too short,
ambiguous, silent, weakly annotated, or trivially solvable. Packaged tasks are then checked for complete
schemas, aligned hidden references, active closed vocabularies, decoy coverage, and resistance to shortcuts
such as pass-through media, empty manifests, constant labels, or copied templates.



                                                                                                              30
Evaluation. Each task uses a dedicated grader with weighted dimensions and gating conditions. Determin-
istic checks cover temporal IoU, event recall, closed-vocabulary accuracy, WER/CER, speaker attribution,
ASR preservation, signal metrics, SSIM/PSNR, mask IoU, timestamp error, catalog matching, photometric
error, geospatial overlap, table consistency, and schema validity. For naturally open-form artifacts such as
memos, overlays, or semantic summaries, some graders add constrained LLM/MLLM checklist axes. The
final score therefore measures process-level completion: media quality, structured intermediate artifacts,
format compliance, and cross-document consistency.

  conditioned on content that diverges from the true recording.
• Video Perception and Extraction Error. The visual stream is misread at the source via blank or off-key
  frame sampling, OCR mistakes, or inverted event ordering, leaving later stages with incorrect evidence.
• Insufficient Exploration of Modal Content. With multimodal inputs available, the agent answers from

Step reward rubric. Only assistant messages are scored; user and tool messages serve as evidence. Each
assistant step is judged using only information visible up to that step (no hindsight), and the three-level

                                                                                                              32
label is defined as follows:

• +1 (correct and effective). The step is factually correct given the available evidence and clearly advances
  the task, e.g., appropriate tool use, evidence-supported interpretation of a tool output, an effective decision,
  or an explicit correction of a prior mistake.
• 0 (neutral or exploratory). The step is reasonable but yields limited or unclear progress, e.g., sensible
  exploration, tentative planning, redundant recap, or a reasonable tool attempt that fails for external
  reasons such as a 404, timeout, or tool limitation.
• −1 (incorrect or harmful). The step is factually wrong, misreads or fabricates evidence, violates system
  or policy constraints, repeats a failed pattern without a meaningful strategy change, or otherwise moves
  the trajectory away from success.

We additionally apply error propagation: once a step is labeled −1, dependent later assistant steps remain
−1 until the agent explicitly corrects the mistake or starts an independent subtask no longer relying on the
bad premise. We provide our annotation interface in Figure 11.

Bad-tool fail categories. The numerator of Pbad = (nunparseable + ndisallowed + nescape + nsyntax-fail )/(1 + ntool )
aggregates four categories:

• Unparseable: malformed JSON in tool call, missing or non-dict arguments, or missing tool name.
• Disallowed: tool name outside the allowed set {execute_commands, image_read, task_complete},
  or arguments matching network primitives (curl, wget, ssh, scp, rsync, nc, telnet, http(s)://).

• Escape: arguments matching gold-answer patterns (ground_truth, answer_key, gold_answer,
  *.gold, all_final.jsonl, rl_train.jsonl, sft_train.jsonl), or absolute paths resolving out-
  side the workspace. Any escape event additionally triggers a hard zero on correctness.

I.1. Omni-terminal Harness

This appendix details our terminal-agent harness, adapted from Terminus-KIRA (KRAFTON AI and Ludo
Robotics, 2026) and designed around a single operation: turning raw multimodal files into small, checkable
evidence. Each episode begins with an isolated workspace containing the question and the staged media
files. Rather than packing media into the model context, files remain on disk, and the agent must decide
how to inspect them: extracting audio, sampling frames, running OCR or ASR, reading metadata, writing
scripts, and verifying intermediate results before answering. The agent operates through a small action
space (execute_commands, image_read, task_complete) and expresses higher-level media operations
through code and command-line tools (ffmpeg/ffprobe, ASR, OCR, Python, OpenCV, NumPy, and self-
authored scripts) rather than benchmark-specific APIs. Beyond execution, the harness is centered on trajectory
collection: every tool call, terminal observation, image observation, final answer, exit reason, and reward
field is serialized so the same rollout can be reused for SFT filtering, RL replay, and failure analysis.
Concretely, the harness has four properties that are important for training:
• Filesystem-grounded media processing. Raw video, audio, images, and documents are workspace state.
  The agent learns to transform them into transcripts, key frames, OCR text, timestamps, tables, and other
  compact evidence.
• Persistent terminal interaction. Commands run in a stateful bash session with marker-based completion,
  bounded observations, timeout recovery, and support for scripts written during the episode.
• Replay-faithful multimodal observations. image_read inserts the actual image into the main conversa-
  tion, rather than replacing it with a caption from a separate model. The saved rollout therefore matches
  the observation distribution used for multimodal SFT/RL.
• Qwen-compatible rollout collection. The tool schema is flat, malformed tool-call blocks are conservatively
  recovered, and requests are routed through sticky SGLang endpoints to preserve prefix-cache locality

Episode state. Each task starts from an isolated workspace
                                                 s0 = (q, W0 , U0 ),
where q is the task instruction, W0 contains the staged input files, and U0 contains the available command-line
tools and Python libraries. The model does not receive the raw video or audio stream as context. Instead, it in-
teracts with files through the terminal. During a rollout, the workspace may accumulate derived artifacts such
as extracted frames, audio clips, ASR transcripts, OCR outputs, metadata files, temporary scripts, and numer-
ical tables. The final prediction is accepted only from the benchmark-normalized <answer>...</answer>
wrapper.
Tool interface. The model sees three tools. The small tool set is deliberate: the terminal is the compositional
interface, and the agent decides how to combine existing utilities with code it writes during the episode.
Agent loop. At each step, the harness sends the current conversation and tool schema to the model. If
the model calls execute_commands, the command is executed in the persistent shell and the bounded
output is returned as a tool observation. If it calls image_read, the harness first records a short tool
acknowledgement and then appends the image itself as a multimodal observation in the main conversation.
If it calls task_complete, the harness checks whether a wrapped answer already appears in the trajectory.
The loop stops on successful completion, step limit, no-tool-call exhaustion, or runtime error.
Shell runtime. The shell is a persistent bash subprocess. This preserves state across turns, including the
current directory, environment variables, installed packages, and files created by the agent. Command
completion is detected by appending a hidden marker after each command and polling until the marker
appears. Terminal observations are capped with head–tail truncation, which keeps both the beginning of long
outputs and the end where errors or final answers often appear. When a command times out, the harness
terminates the direct child processes while keeping the shell alive. If the shell itself dies, it is restarted and
the next observation warns the model that shell-local state has been lost.
Native multimodal replay. The main change from a standard terminal harness is the native image_read
path. A caption-only image reader is acceptable for evaluation, but it is not acceptable for multimodal
policy training: the saved trajectory would contain text produced by a separate model rather than the pixels
observed during rollout. Our harness instead stores image observations as multimodal content blocks in
the main conversation. For Qwen-style chat templates, these messages are folded into the provider-specific
tool-response format only at send time. The saved artifact remains provider-neutral and can be replayed for
SFT or RL without changing the observation sequence.

Qwen serving and tool-call robustness. Open-weight models served through SGLang can emit partially
malformed tool-call blocks, especially when shell commands contain quotes, heredocs, or escape characters.
We therefore use a flat execute_commands schema and a conservative recovery layer. If the backend
returns no structured tool call but the assistant text contains a recognizable <tool_call> block, the harness
reconstructs the call only when the function name is in the declared tool list and the arguments can be
parsed without inventing content. Recovered calls are marked in the trajectory, so downstream data filtering
can keep or drop them explicitly.
Trajectory artifacts. Each rollout stores the full message sequence, parsed tool calls, shell outputs, image-
observation references, final extracted answer, exit reason, and evaluation metadata. The same artifact
supports three downstream uses. First, successful trajectories can be filtered into SFT examples. Second, RL
training can replay the same observation distribution seen during rollout. Third, failures can be inspected at
the process level: whether the agent sampled the wrong frames, trusted a bad ASR transcript, stopped too
early, chose the wrong external fact, or made a calculation error.
Prompt organization. The prompt is factored into reusable pieces. A static system prompt defines the tool
contract. A shared workspace prefix defines sandbox, network, and answer-format rules. Benchmark-specific
prompt bodies supply the actual question, options, files, and output schema. Runtime reminders are used
only when the model stops without a tool call or calls task_complete before producing a wrapped answer.
