   Data Augmentation Prompt
   You are an expert AI data augmentation and reasoning-chain generation specialist. Your mission is to take an existing simple dialogue (the seed query)
   and perform two subtasks:

   TASK 1: Query Expansion & Toolset Enrichment
   Transform the given simple user query into a more complex, realistic, and analytically challenging query — one that naturally requires the
   introduction of new tools to be solvable.
   • The new query must:
      • Require multiple reasoning steps and multiple tool calls to complete.
      • Stay coherent, natural, and goal-driven (avoid artificial or random complexity).
      • Introduce interdependent subtasks when appropriate (e.g., data lookup → processing → comparison → conclusion).
      • Be sufficiently rich that solving it demands capabilities beyond the current toolset, thereby motivating the creation of new tools.
   • You may invent new tools to support the expanded task, but:
      • Each invented tool must be essential for completing the new query (not optional or redundant).
      • Each invented tool must have a clear purpose, realistic input/output, and be plausible in an AI assistant environment.
      • New tools should fill well-defined capability gaps (e.g., new reasoning, analysis, or synthesis functions) that existing tools cannot address.
      • Avoid arbitrary or trivial tool creation; tools must emerge naturally from task needs.

   TASK 2: Trajectory Generation and Tool Documentation
   Using the new complex query and enriched toolset:
   1. Simulate a complete reasoning and execution trajectory that solves the query.
   2. Ensure each assistant step contains concise reasoning ("thought") and EXACTLY ONE tool call.
   3. Include realistic tool outputs as tool steps.
   4. End with a final assistant step calling the Finish function (or providing the final answer).
   5. Collect all tools used (including pre-existing and newly created ones) and output them with complete, generic descriptions.

   CRITICAL: Tool Calling Constraints
   • Each assistant turn MUST call EXACTLY ONE tool (not zero, not multiple).
   • ALL tool calls MUST succeed and return valid results — no failures, errors, or exceptions.
   • Tool outputs must be realistic, specific, and directly usable by subsequent steps.
   • Tools should be designed to be reliable and deterministic, producing successful results every time.

   CRITICAL: Tool Description Requirements
   • Tool descriptions must read like standalone documentation: clearly state the tool’s purpose, enumerate every input parameter with name/type/defaults/-
   constraints, outline the returned structure, and include usage notes or best practices.
   • Keep descriptions task-agnostic but specific enough that another engineer could confidently reuse the tool in unrelated contexts; avoid referencing the
   current task and never leak problem-specific data.
   • Follow a polished textual style (e.g., "Does [X]. Parameters: ’param1’ (string, required) - . . . Returns . . . Notes: . . . ") so each entry matches standard
   documentation.

   AVAILABLE TOOLS (you can use these or invent new ones):
   {List of Existing Tools}

   ORIGINAL TASK CONTEXT:
   {Original Question/Dialogue and Metadata}

   OUTPUT FORMAT (CRITICAL)
   You must output ONLY a valid single JSON object with this structure:
   {
        "trajectory": [
             {"role": "user", "content": "expanded complex query"},
             {"role": "assistant", "thought": "brief reasoning", "tool_calls": [{"type": "function", "function": {"name": "
                    function_name", "arguments": {}}}]},
             {"role": "tool", "name": "function_name", "content": "successful result with specific data"},
             ...
             {"role": "assistant", "tool_calls": [{"type": "function", "function": {"name": "Finish", "arguments": {"mode": "
                    give_answer"}}}]}
        ],
        "tools": [
             {"tool_name": "ExistingTool1", "description": "..."},
             {"tool_name": "InventedToolA", "description": "..."}
        ]
   }
   REQUIREMENTS
   • The output must be a single valid JSON object.
   • Every assistant turn MUST have exactly ONE tool call.
   • Every tool call MUST be followed by a successful tool response.
   • Tool responses must contain realistic, specific, and usable data.
   • Every tool used must be documented in the "tools" list.
   • The process is irreversible — you cannot revise earlier steps.
   • Target a specific number of reasoning steps (e.g., significantly more than the original).
   • The new query must remain in the same domain but achieve higher complexity.
   • The model should reason step-by-step WITHOUT knowing the final answer in advance.

   You are an expert at reconstructing reasoning processes from execution trajectories. Your task is to extract a first-person problem-solving plan and the
   corresponding tool usage chain based on the given trajectory.
   Task Objective
   From the provided trajectory, extract:
   1. plan — A concise, first-person summary describing how I planned to approach and solve the problem step by step.
   2. tool chain — An ordered list of tools I used during the process, with each ‘reason‘ explaining why I used that tool in the problem-solving context.
   Extraction Rules
   • This is a faithful extraction task — do not rewrite or invent reasoning not present in the trajectory.
   • Use first-person narration (e.g., "I first analyzed...", "I planned to use the search tool to find...") rather than meta references like "the model decided
   to..." or "the agent used...".
   • The ‘plan‘ should describe the planning and reasoning process only — do NOT include actual results, outputs, or findings from tool executions.
   • Focus on the intended approach and strategy, not on what was discovered or returned by the tools.
   • The ‘plan‘ should be a coherent first-person narrative of the problem-solving strategy.
   • The ‘tool_chain‘ should include:
       • ‘name‘: the exact tool name from the trajectory.
       • ‘reason‘: a brief, first-person explanation of why I invoked this tool or what problem it solved at that stage.
   • Preserve the original order of tool invocations.
   • Only include tools listed in the provided tool list.
   • Output only a JSON object in this format:
   {
         "plan": str,
         "tool_chain": [
              {"name": str, "reason": str},
              ...
         ]
   }
   Do not include any commentary or text outside the JSON object.

   {query}

   {trajectory_str}

def construct_visual_missing_case ( sample ) :
    """
    Constructs a ' Visual Information Inaccessible ' unsolvable case by stripping visual context .

    """
    # 1. Preserve the original query
    # We deliberately keep the query unchanged ( e . g . , " Describe the image ")
    # to test if the agent can correctly identify that the image is missing .
    query = sample [ " query " ]

    # 2. Remove Visual Information
    # The core logic is to empty the file list in the background context ,
    # effectively simulating a system error where the image failed to load
    # or was not provided .
    if " background " in sample :
         sample [ " background " ][ " files " ] = []

    # 3. Mark as Unsolvable
    sample [" unsolvable_type " ] = " visual_missing "

    return sample

   You are an expert in designing Agent Planning Benchmarks. Your goal is to REWRITE the ‘tool_bank’ for a given task to test ‘Plan Efficiency’.
   Core Logic:
   You must provide a tool set where multiple paths exist to solve the problem, but one path (using Composite Tools) is significantly cheaper (lower total
   cost) than others.
   Instructions:
   • Original Tools (Mandatory): Keep ALL original tools. Assign them a reasonable base cost (e.g., integers between 1 and 5).
   • ADD 5 to 10 NEW Tools (Strict Constraint): Generate exactly 5 to 10 new tools divided as follows:
     • Type A: Efficient Composite Tools (3 to 6 tools): Combine 2 to 3 consecutive steps from the Ground Truth into one single tool. The cost MUST
        be LOWER than the sum of the original tools it replaces (e.g., if Tool A cost 2 and Tool B cost 2, Composite Tool AB should cost 3).
      • Type B: Inefficient/Trap Tools (2 to 4 tools): Tools that look relevant but are suboptimal: Overpriced (simple job, very high cost), Redundant
        Steps (breaks simple action into unnecessary sub-steps), or Slightly Off-target (retrieves partial information, forcing more tool calls).
   • NO Magic Tools: Do not create a single tool that solves the whole query in one step.
   • Output Format: Return a SINGLE valid JSON list containing [All Original Tools + All New Tools]. Each tool object must have: name, description,
      parameters, and cost (integer).
   User Message:
   User Query: {query}
   Original Ground Truth / Steps: {tool_chain or plan}
   Original Tool Bank: {original_tools}
   Requirement: Generate a JSON list of tools (original + composite + traps) with costs. Output PURE JSON only.

    Step-wise Scoring Rubric
   Scoring Levels (with examples):
   1.0 (Perfect): No errors. The predicted step is correct, logical, and will progress toward the goal successfully.
   • Example: Correctly identifies a valid next step with proper tool usage that moves the task forward.
   • Requirement: Zero error flags (E1-E6 all False).
    • Is Correct: YES
    • Note: The step does NOT need to be "optimal" or "most efficient" - it just needs to be correct and reasonable.
    0.8 (Very Good): At most ONE very minor issue (e.g., only E5 with trivial impact) that doesn’t affect correctness.
    • Example: Uses slightly different but equally valid search terms, or chooses a different valid page range.
    • Requirement: At most one minor error flag, and it must be E5 only with minimal impact.
    • Is Correct: NO (any error present)
    0.6 (Acceptable): One minor error (E4 or E5) that doesn’t fundamentally break the solution.
    • Example: Minor parameter issue or small dependency oversight that can be easily recovered in subsequent steps.
    • Requirement: Only one error flag, and it must be E4 or E5.
    • Is Correct: NO (any error present)
    0.4 (Problematic): Multiple minor errors (2+ of E4/E5) OR one moderate error (E2).
    • Example: Skips a necessary step and has tool usage issues.
    • Requirement: Two or more minor error flags OR one E2 flag.
    • Is Correct: NO
    0.2 (Severe): Major errors (E1, E3, or E6) OR multiple moderate errors.
    • Example: Goal misalignment or calling non-existent tools.
    • Requirement: At least one major error flag (E1/E3/E6) OR multiple E2/E4/E5 flags.
    • Is Correct: NO
    0.0 (Failed): Critical errors making the step completely incorrect or unsolvable.
    • Example: Complete misunderstanding of the task combined with multiple errors.
    • Requirement: Multiple major error flags OR complete logical breakdown.
    • Is Correct: NO
    EVALUATION CRINCIPLES:
    • Evaluate based on CORRECTNESS and REASONABLENESS, not optimality or efficiency
    • DO NOT penalize for being "suboptimal" if the approach is valid and will work
    • DO NOT compare with reference to judge "better or worse" - only check if prediction is valid
    • Different valid approaches should NOT be penalized even if they differ from the reference
    • The reference is just ONE possible correct solution, not the only correct one
    • A step is correct if it: (1) aligns with the goal, (2) uses tools properly, (3) has sound logic, (4) will successfully progress the task
    • Only mark errors when there are actual problems (wrong tools, missing data, logical flaws, violations)
    • CRITICAL: ANY error (E1-E6) automatically means is_correct = NO. ONLY perfect predictions with zero errors can be marked as correct.

   • Criteria: The agent explicitly states it cannot proceed due to the broken tool or inability to solve the task.
   • The agent gives up without attempting any alternative solution.
