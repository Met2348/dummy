Security: The core mechanism of HyperTool relies
on the agent dynamically generating and executing
Python code blocks. Without strict isolation, this
exposes the host environment to Remote Code Ex-
ecution (RCE) vulnerabilities, whether from mali-
cious prompt injections or unintended infinite loops.
Deploying HyperTool in real-world systems strictly
necessitates robust sandboxing (e.g., Dockerized
environments or restricted interpreters) and read-
only API access constraints. 2) Interpretability
Trade-offs: By internalizing complex procedural
logic into local blocks, HyperTool inherently hides
intermediate execution states from the main rea-
soning trace. While this prevents context infla-
tion, it makes diagnosing subtle algorithmic errors
(e.g., an incorrect sorting condition within an Ag-
gregate block) more challenging, as the agent only
observes the final output. 3) Reliance on Code
Generation Capability: The efficacy of Hyper-
Tool is bottlenecked by the base model’s inherent
coding proficiency. While our SFT pipeline ef-
fectively activates this capability in smaller mod-
els like Qwen3-8B, deploying this framework to
zero-shot, non-code-optimized models may result
in frequent syntax errors or schema hallucinations,
necessitating robust local repair modules.

E.1    Prompt Template of HyperTool Agent

  You are a helpful assistant. Put your final answer in <answer></answer> part.
      HyperTool Strategy
  IMPORTANT: To maximize efficiency and reduce latency, you should prioritize HyperTool
  Strategy over sequential turn-based calls whenever the workflow is predictable.
  When to use HyperTool (Single code block)
    • The task can be decomposed into a clear sequence of tool calls (A → B → C).
    • The required data flow between steps is straightforward and can be handled with simple logic
      (e.g., filtering, extraction, loops).
    • The structure of tool outputs is either:
         – Known from prior experience, OR
         – Can be reliably inferred, OR
         – Not critical to control flow (i.e., minor parsing errors will not break the pipeline).
  When NOT to use HyperTool (Call tools one by one)
    • The next step depends heavily on interpreting the semantics of previous outputs (non-trivial
      reasoning).
    • The structure of tool outputs is unclear and critical for downstream steps.
    • The workflow may require significant adaptation after observing intermediate results.
  Execution Standard
  Important:
    • If the logic is deterministic, do not wait for the system to respond between steps. Combine the
      calls, process the data locally within the block, and return the final consolidated result.
    • Do not include any reasoning content in the code block, including as comments (e.g., lines
      starting with #).
  Inference Key Rules:
   1. The only tool you can call directly is HyperTool. Other tools can only be called through
      HyperTool by using call_tool(server_name, tool_name, {"param": "value"})—note
      double braces in format string.
   2. You must carefully read the tool descriptions and the tool output format, and use the exact
      argument names as specified without deviation.
   3. Do not introduce unsupported facts; use reasoning to interpret tool results, but ensure
      conclusions are evidence-grounded. If a tool returns empty, None, or unexpected output, you
      MUST diagnose and retry with a corrected approach.
   4. The variables in different HyperTool code blocks are not reusable.
   5. Carefully follow the tool output format and implement simple data processing logic within the
      HyperTool code block to extract the required information. Only use fields explicitly provided
      in the tool output, and eliminate any irrelevant or noisy data.
   6. Remember each HyperTool code block must use "result" as the variable name to store the
      final result. "result" must contain the actual data returned by tool calls (or
      processed/extracted values derived from it), never a hand-written status string like "done" or
      "completed". Do not use print statements to print the result.

Evaluation Steps
Step 1: Reconstruct the final reasoning chain
Trace backward from the final answer to find the minimal set of tool calls, code executions, and
reasoning steps that directly support it. Ignore earlier abandoned or corrected attempts.
Step 2: Verify each step in the final chain
For each step, check:
  • Is the Python code logically correct for its stated purpose?
  • Does the agent correctly read and interpret the tool output?
  • Are intermediate conclusions consistent with the tool outputs?
Step 3: Check for selective reporting / post-hoc rationalization
This is critical. Ask:
  • Did the agent compute multiple variants of a metric (e.g., different date ranges, open vs. close,
    close vs. intraday high) and then selectively report only the variant that supports its
    conclusion?
  • Does the agent’s chosen metric definition match the standard interpretation implied by the task
    wording? For example:
       – “price increased from date A to date B” → should use closing prices on those two specific
          dates, not open-to-intraday-high or any other cherry-picked combination.
       – “revenue in Q2” → should use the Q2 figure, not an annualized estimate.
    If the agent uses a non-standard definition without explicit justification, treat this as a failure
    even if the arithmetic is correct.
  • If the agent tried multiple definitions and only one of them satisfies the task condition, but the
    agent presents that one as the answer without acknowledging the others failed, this is selective
    reporting → score 0.
Step 4: Verify the final answer
  • Does the final answer directly follow from the last valid reasoning step?
  • Is the stated conclusion actually supported by the data, under the standard metric definition
    (not a cherry-picked one)?
Step 5: Check completeness
  • Does the final answer directly and completely address every sub-question in the task?
  • Is the answer specific enough (concrete values, names, conclusions)?
Scoring Rules
Score 1 only if ALL of the following hold:
 • The final reasoning chain is logically valid end-to-end
 • The agent correctly interprets all tool outputs in the chain
 • The final answer is consistent with the chain’s conclusion
 • The metric / method / source used matches the standard interpretation implied by the task (no
   cherry-picking of favorable definitions)
 • No evidence of selective reporting: the agent did not try multiple metric variants and report
   only the one that fits
 • Any task-specified method or source is visibly followed
 • The final answer is complete and directly addresses the task
Score 0 if ANY of the following:
 • A logical or computational error exists in the final reasoning chain
 • The agent misreads a tool output (wrong key, wrong type, off-by-one, etc.)
 • The final answer contradicts the reasoning chain
 • A clearly stated required method / source was ignored or substituted
 • The agent uses a non-standard metric definition without justification (e.g.,
   open-to-intraday-high instead of close-to-close for a price change claim)
 • The agent computed multiple metric variants and selectively reported only the one supporting
