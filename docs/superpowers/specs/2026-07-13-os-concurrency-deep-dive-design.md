# 操作系统与并发深挖 —— 设计文档

> 定位:`for_real_dummy/` 第 13 条独立技能系列,是"职业发展与需求"四部曲(操作系统与并发→计算机网络→数据库原理与实战→系统设计)的第 1 部。四部曲替代了原先"算法/系统设计/统计三选一"里笼统的"系统设计"一项——2026-07-13 真实调研(WebSearch,覆盖国内校招/实习后端面试、欧美 new grad 面试趋势、招聘市场需求信号三个角度)显示,操作系统/计算机网络/数据库这"计算机基础三件套"在国内大厂校招八股文体系里是和系统设计并列、覆盖面同样广的独立必考项,不是系统设计的附属内容,值得各自成篇。

## 1. 目标与规模

约 **80 个知识点**(±10~15% 浮动,和 dsa-deep-dive/statistics-deep-dive 同样的先估后核流程,最终以全库自查的精确计数为准),5 大板块,11 个分类文件 + 1 篇模拟终面 capstone。深度对标本仓库已完成系列(dsa-deep-dive 140 点、statistics-deep-dive 116 点)的"技术终面 1 小时持续面试级别的深度广度",不是教科书搬运。

**每个文件明细(撰写时按此展开,允许在保持知识点独立完整的前提下微调个别点的合并/拆分)**:

- **01 进程与线程基础**:PCB与进程状态转换、fork/exec/wait语义与写时复制COW、线程模型(1:1/N:1/M:N)与Python GIL的位置、协程与用户态调度(有栈vs无栈)、线程私有数据TLS、进程vs线程vs协程选型决策、僵尸进程与孤儿进程、Windows进程创建模型对比(CreateProcess/spawn vs fork)。
- **02 CPU调度算法**:调度目标与指标(吞吐量/周转时间/响应时间/公平性)、FCFS、SJF与饥饿、时间片轮转RR、多级反馈队列MLFQ、CFS完全公平调度器(红黑树+vruntime)、实时调度与优先级反转/优先级继承、多核调度与CPU亲和性/cache一致性代价。
- **03 基础同步原语**:竞态条件本质与检测、临界区与互斥锁、自旋锁vs互斥锁的忙等待代价、信号量与生产者消费者问题、读写锁(读者优先vs写者优先)、条件变量与虚假唤醒、屏障Barrier同步、内存屏障与可见性(happens-before初步)。
- **04 高级并发模式与无锁编程**:CAS与ABA问题、无锁队列初步、线程池设计(核心线程数/队列策略/拒绝策略)、协程调度器实现思想、并发容器设计模式、乐观锁到数据库场景的映射(为后续数据库系列埋伏笔)。
- **05 死锁**:死锁四个必要条件、资源分配图与检测算法、银行家算法、死锁预防策略(破坏四条件)、死锁避免vs预防vs检测的取舍、活锁与饥饿。
- **06 虚拟内存与地址转换**:虚拟地址空间与逻辑/物理地址、MMU与地址转换机制、多级页表与空间开销权衡、TLB与地址转换加速、分段机制、分段与分页对比、写时复制COW的实际应用、大页Huge Page。
- **07 内存分配与页面置换**:malloc内部机制(空闲链表法)、伙伴系统与slab分配器、页面置换算法FIFO/LRU/Clock、最优置换Belady定理与Belady异常、mmap内存映射、Swap与抖动Thrashing、内存碎片(外部碎片vs内部碎片)。
- **08 IO模型演进**:阻塞IO vs 非阻塞IO、select多路复用与局限、poll改进、epoll与水平触发vs边缘触发、异步IO(AIO)与io_uring简介、Reactor模式、Proactor模式、C10K/C10M问题的历史与解法演进。
- **09 进程间通信IPC**:匿名管道Pipe、命名管道FIFO、System V消息队列、POSIX共享内存及同步问题、Socket作为IPC手段、信号Signal机制、IPC方式选型对比。
- **10 文件系统基础**:inode/超级块/目录项结构、日志文件系统journaling与崩溃一致性、page cache与fsync语义、硬链接软链接的文件系统层本质、文件系统一致性保证机制、稀疏文件与文件空洞。
- **11 现代系统专题**:容器隔离namespaces原理、cgroups资源限制原理、eBPF基础原理、NUMA架构对高性能/训练任务的影响、GPU任务调度与OS调度器的关系(需实际核查kernel-gpu-deep-dive内容确认真实关联,不编造)、协程实现原理深化(有栈vs无栈性能对比)、云原生场景下的资源隔离与调度。
- **12 模拟终面capstone**:综合运用前11类知识点,场景设定为"高并发服务出现偶发死锁+内存暴涨,给一段真实模拟日志要求诊断根因",结构参照 statistics-deep-dive 21 类capstone(候选人初版汇报→面试官多轮追问→最终诊断真实异常数据)。

## 3. 知识点结构模板(七步,复用 dsa-deep-dive 模板)

1. **签名/是什么** 2. **一句话** 3. **底层机制/为什么这样设计**(数学零星出现时——Amdahl定律、Little's Law、Belady异常的反直觉证明、LRU竞争比——直接放在这一步,不单独拆"数学推导"步骤;OS的数学是零星出现,不像统计学科每点都需要独立推导) 4. **AI研究/工程场景** 5. **可运行例子**(assert验证,注明验证环境) 6. **面试怎么问+追问链**(五轴方法论,见下节) 7. **常见坑**

不采用 statistics-deep-dive 的独立"数学推导"第八步——已核实 OS 知识点的数学成分是零星点缀而非每点必需,勉强拆分会造成大量"数学推导:无"的空洞步骤。

## 4. 五轴追问链方法论落地(从第一天融入,不事后补)

| 轴线 | 含义 | OS/并发学科例子 |
|------|------|------|
| 规模递增轴 | 小规模→大规模→极限行为 | 单线程→多线程→NUMA多路服务器/海量连接C10M |
| 工程约束递增轴 | 单机→并发→分布式 | 本地锁→跨进程锁→分布式锁(为系统设计系列埋伏笔) |
| 方案批判迭代轴 | 面试官连续指出具体缺陷逼换方案 | 全局锁→细粒度锁→无锁结构,每次因具体性能/正确性缺陷被打回 |
| 决策依据追问轴 | 不纠错,只逼问选择依据 | "为什么用epoll不用多线程阻塞IO" |
| 真实性验证轴 | 简历"做了性能优化"被追问到具体数字 | "具体解决了什么并发问题,怎么测的,改动前后数字是多少" |
| 诊断真实数据(新题型) | 给日志/监控指标,要求诊断而非套公式 | 服务偶发卡顿,给thread dump要求定位死锁/锁竞争根因 |

每个知识点挑 1~2 条最自然的轴线走 2~3 层深,不强行凑满 5 轴。12 类capstone 是唯一要求同时用 3 条以上轴线的文件。

## 5. 验证环境与纪律

**双环境,逐点标注**:
- **默认环境**:仓库根目录 `.venv`(Windows 原生 Python),用于纯算法/数据结构类模拟——调度算法模拟、页面置换算法模拟、锁与竞态的 `threading` 演示、IPC 的 `multiprocessing` 演示、内存分配器模拟等,这些不依赖真实内核语义,Windows 原生 Python 完全够用。
- **WSL2 Rocky Linux**(2026-07-13 已确认可复用,`Python 3.12.13`,复用 rhcsa-bash-deep-dive 已装好并验证过的环境,不新建):用于真实 Linux 语义强相关的知识点——`fork()` 真实写时复制行为、`epoll`(Windows 无此系统调用)、真实 `signal`(如 `SIGCHLD`)、`namespaces`/`cgroups` 探查等。调用方式复用 rhcsa-bash-deep-dive 已记录的规范(Git Bash 调 `wsl.exe` 前需 `export MSYS_NO_PATHCONV=1`)。
- 每个知识点的"可运行例子"步骤开头必须显式标注验证环境(`.venv` 或 `WSL2 Rocky Linux`),延续 rhcsa-bash-deep-dive"逐条如实标注验证颗粒度"的纪律,不笼统一句"已验证"带过。

**并发/计时类断言纪律**(复用 dsa-deep-dive 01 类的教训 + 新增一条本系列特有的安全纪律):
- 计时对比断言用 `best_of(fn, *args, trials=N)` 取多次采样最小值,规模不能小到被系统调度噪声淹没。
- **竞态条件/死锁复现类断言,要求多次运行确认稳定复现,不满足于运气好复现一次**——写作阶段至少重复运行 5~10 次确认现象稳定,不是复现一次就当作已验证。
- **死锁 demo 必须带超时安全网**(如 `threading.Timer` 或进程级 timeout wrapper):真实构造死锁是这个知识点教学价值所在,但不能让验证脚本永久挂起——demo 需要在"真实触发死锁现象"和"脚本本身能确定性退出"之间都做到,不能二选一。这是本系列相对此前系列新增的一条纪律,此前系列没有遇到"正确行为是挂起"这种情况。

## 6. 与仓库已有内容的边界(已核实,不重复)

- **vs rhcsa-bash-deep-dive**:已读取该系列全部 `00-roadmap.md` 及 `02-process-and-boot.md`、`07-networking.md` 的完整知识点列表核实——那一条是纯操作层面(`systemctl`/`nmcli`/`ps`/`kill` 怎么用,面向 RHCSA EX200 上机操作考试,该系列明确声明不采用"面试怎么问"框架),这一条是 OS 内部机制是什么、为什么这样设计、技术面试怎么问,操作层面 vs 理论/机制层面,两者互不重复,可以并存。
- **vs 后续计算机网络系列**:本系列 08 类"IO模型演进"里的 `select`/`poll`/`epoll` 只讲"操作系统怎么实现 IO 事件通知"这个机制角度;TCP/IP 协议细节、HTTP、DNS、拥塞控制、负载均衡等协议层内容留给下一个子系列"计算机网络"专门展开,避免边界重叠。
- **vs kernel-gpu-deep-dive**:11 类"现代系统专题"计划提及"GPU任务调度与OS调度器的关系",执行阶段落笔前必须实际读取 kernel-gpu-deep-dive 相关文件确认真实技术关联,不能凭听起来相关就编造联系(同 G8 纪律)。
- **vs learning/ 48 个模块**:未发现任何 OS/并发理论专题模块,不构成重复。
