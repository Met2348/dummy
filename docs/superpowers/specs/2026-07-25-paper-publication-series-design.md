# for_real_dummy 新增"论文发表系列"(5 个新系列)设计文档

## 背景与动机

2026-07-25,用户转达导师(与 [[worldmodel-imagination-controller-research]] 记忆记录的同一位导师 Weikai Lin)两轮反馈:

**第一轮**:要求补充科研写作、学术绘图、论文投递相关的"手把手教程",要求非常详细面向新手,拆分多个专题,由浅入深最终抵达化境。

**第二轮**(用户追加转达):导师要求再加 rebuttal(强调这一项)、参会时的各种指南、oral、camera ready、release code、poster、presentation、hugface release、interact with community、deal with reject。

用户同时明确:①案例素材可以直接取材于 `research/world-model-imagination-controller/`(用户真实在研、即将投稿的 ICLR 项目)已有的真实材料,不需要刻意脱敏("其实你完全可以用个真实的,不用担心,马上就要投稿了");②计划清楚后直接开始做,不需要等待逐项批准。

这是 for_real_dummy 里第一次出现"研究判断力/学术生存技能"类内容——和"深挖系列"(函数/机制是什么)、"操作类系列"(工具怎么操作)都不同,回答的是"论文这件事从写到发表到被社区看见,应该怎么做"。

## 总体架构

不采用"N部曲"锁定数字的命名(避免后续再扩展时和命名冲突),README 里新增独立小节**"论文发表系列(从写作到社区回响)"**,包含 5 个独立系列:

| # | 系列 | 目录名 | 定位 | 分类数 |
|---|------|--------|------|--------|
| 1 | 科研写作 | `research-writing-deep-dive/` | 论文怎么"论证" | 8 |
| 2 | 学术绘图 | `research-figures-deep-dive/` | 论文里的图表怎么做 | 6 |
| 3 | 投稿与评审 | `paper-submission-deep-dive/` | 怎么走完投稿评审全流程 | 5 |
| 4 | 学术演讲与参会 | `academic-presentation-deep-dive/` | 怎么在会议上现场展示 | 5 |
| 5 | 成果发布与社区 | `research-release-deep-dive/` | 论文之后怎么发布、怎么和社区互动 | 3 |

**为什么是 5 个独立系列而不是揉进已有的 3 个**:导师第二轮反馈涉及的内容(oral/poster/slides/Q&A/会议指南、代码发布/HuggingFace/社区互动)在技能类型上和"写论文""画图""投递"都不同——前者是**临场表达能力**(和书面写作节奏完全不同,SPJ 本人也把"How to Give a Great Research Talk"单独开一个姊妹讲座,不和"How to Write a Great Research Paper"合并),后者是**论文发表之后的事**(时间线上在投稿评审结束之后)。硬塞进"投递系列"会让该系列臃肿到丢失"由浅入深"的层次感,所以拆成独立系列。

**不强行凑知识点数字**:这 5 个系列不会像 dsa-deep-dive"140 个知识点"那样报一个大总数——写作/演讲类内容天然是原则+范例结构,不是原子化的函数级知识点,统一按"分类数"呈现,延续 numpy-deep-dive 进阶深度追加"材料相对薄弱、诚实收敛不强行凑"的先例。

## 各系列详细设计

### 系列 1:科研写作 `research-writing-deep-dive/`

**分类(8 个,由浅入深)**:
1. 论文整体叙事结构——Motivation-Gap-Contribution 框架、Simon Peyton Jones"先写论文再做研究"的哲学(idea → write paper → do research,而非传统的 idea → research → write)、标题与摘要的电梯演讲写法
2. Introduction 段落级写作与 Related Work 定位——每段该承担什么功能、怎么站在前人肩膀上定位贡献而不是列一堆引用了事
3. Method/实验部分的呈现逻辑——怎么让审稿人 5 分钟内 get 到核心贡献,结果先行 vs 方法先行的取舍
4. 句子层面的学术英语写作——主动语态、"参考文献是注释不是名词"(SPJ 原话:"Castelli and Brown [3] showed..." 而不是 "[3] showed...")、常见学术套话反例
5. 局限性/风险自曝的诚实写法——不自毁但也不回避
6. 多轮修改方法论——含 AI 辅助润色的边界在哪(哪些能用 AI 润色语法,哪些必须自己把关论证逻辑)
7. 审稿人视角精读——常见拒稿理由拆解(参考调研发现的真实红旗:摘要声称新模型但方法部分是标准 transformer/CNN 换个数据集、结果表没有消融/无当前基线/无种子方差)
8. **Rebuttal 写作技巧**(响应导师第二轮反馈里被点名强调的一项)——逐条回应的具体写法:怎么开头总结改动、怎么处理"数据不够"类质疑、怎么在字数限制内说服可能只读 rebuttal 不读全部 review 的 Area Chair、什么时候该坦然承认而不是硬辩

**格式模板(新设计,原因见下)**:这类内容核心是"文字论证判断力",没有函数签名,也不是操作步骤,不能照搬仓库现成的两套模板。设计一个新的**六步写作判断力模板**:
1. 常见误区/反面例子(真实典型的差写法)
2. 逐处修改对照(同一段内容改好后的版本,标注每处为什么改)
3. 可操作检查清单(读者下次自己写时能直接套用的具体规则)
4. **能量化的部分写真实代码验证**(比如统计一段文字的被动语态占比、句长分布、参考文献密度——用真实 Python 正则/规则跑出统计结果,延续仓库"assert 验证"纪律,能跑代码的地方不空谈;写作判断力本身不可 assert,但支撑判断的统计信号可以)
5. 审稿人会怎么挑刺 + 反驳链(复用 dsa-deep-dive 20 类已验证的"五轴追问链"方法论,这里是"评审对抗"版本)
6. 常见坑

**Capstone**:模拟"审稿意见 + rebuttal 攻防"——给一段论文摘要(取材真实项目场景,细节可以改编但问题设定要真实),生成几条典型审稿意见(基于调研到的真实拒稿模式设计,不是虚构套路),演示逐条回应怎么写。

### 系列 2:学术绘图 `research-figures-deep-dive/`

**分类(6 个)**:
1. 图表类型选择——折线/柱状/散点/热力图,什么数据用什么图,Table vs Figure 怎么选
2. 颜色与感知设计——Okabe-Ito 色盲友好调色板、为什么不能用 jet/rainbow colormap(不是感知均匀,会制造假的等高线边界)、色彩冗余编码(色盲读者不能只靠颜色区分类别,要加形状/线型)
3. 多面板排版工程——matplotlib `gridspec`、字体大小/线宽/DPI、矢量图(PDF/SVG)vs 位图(PNG)导出场景
4. 架构图与流程图——draw.io/TikZ/PowerPoint 各自适用场景(不要求读者学会 TikZ 语法本身,讲清楚"什么时候该选哪个工具"这个判断)
5. 图注写作规范——一图一个故事,不是塞满信息;caption 要能独立于正文被理解
6. 常见反模式——Tufte 的 chartjunk、双 y 轴陷阱、3D 饼图、跨 panel 配色不一致

**格式模板**:复用"深挖系列"标准 7 步(签名/是什么→一句话→底层机制/为什么这样设计[这里是视觉感知原理]→AI 研究场景→可运行例子→常见坑→"面试怎么问"替换为"审稿人/读者会怎么挑刺"),因为这是仓库里少数真正有"可运行 matplotlib 代码"支撑的写作/设计类内容,套用现成模板不是偷懒,是内容本身天然适合。

**验证方式差异声明(必须写进 00-roadmap.md)**:这是仓库第一次系统性产出图片文件而非纯文本 assert。验证方式改成"检查文件真实生成 + 尺寸/DPI 元数据符合预期"(比如 `os.path.exists` + `PIL.Image.open(...).size` + 检查 dpi 参数),不能像文本一样 assert 像素内容,这个方法论差异必须诚实写清楚,不能含糊带过说"已验证"。

**Capstone(教程体)**:三个系列里唯一真正适合"教程体"(动手敲代码)格式的一个——手把手画一张完整的、多面板的、投稿级别的核心结果图。写作/投递/演讲/发布四个系列**不会**强行凑一篇教程体(没有可敲代码硬凑会违反仓库"诚实收敛不强行凑"纪律)。

### 系列 3:投稿与评审 `paper-submission-deep-dive/`

**分类(5 个)**:
1. Venue 选择与 fit 度判断——影响因子迷思(调研发现:很多 ML 研究者会更看重 JMLR 而非影响因子更高的 IEEE TNNLS,因为前者是"本社区自己的期刊")、track 怎么选
2. 双盲匿名化与投稿系统操作——OpenReview 操作流程 + **真实可用的匿名化检查脚本**(扫描 LaTeX 源码找作者名/机构/未匿名 GitHub 链接/致谢段落,这个要做成读者真的能拿去用的小工具,不是纸上谈兵)
3. Rebuttal 时间与协作管理——和系列 1 的"怎么写"互补,这里管"怎么组织这件事"(调研发现的真实数据:ICLR 2024-2025 分数在 rebuttal 后提升的论文录用率 55.7%-57.6%,不提升的只有 7.8%-12.4%,这条数据作为本节开篇动机);word limit 怎么分配、什么时候该提前准备、怎么和合作者分工
4. **Deal with Reject 深挖**(响应导师第二轮反馈,独立分类而不是被"decision 分支"这种笼统说法盖住)——怎么理性拆解拒稿理由(审稿人真的没看懂 vs 真的有硬伤)、resubmit 策略(改投什么档次的venue、需要补什么实验)、心态调整
5. Camera-ready 与 Supplementary 组织——checklist 式收尾,Supplementary material 怎么组织不喧宾夺主

**格式模板**:复用 daily-toolkit-deep-dive 已验证的 6 步操作类模板(为什么需要/不会有什么后果 → 环境要求 → 一步步跟着做 → 背后发生了什么 → 常见坑 → 自测清单),"被拒了怎么办"这类场景化知识点借用 rhcsa-bash-deep-dive 的"故障排查语序"(现象→判断→行动→原因)。

**Capstone**:叙事体"投稿全流程复盘"(类似 os-concurrency/database 系列的事故复盘 capstone),建议设计成"首投被拒 → 冷静拆解审稿意见 → 调整方向重投 → 录用"这条完整线索,自然演示"deal with reject"而不是单独干讲。

### 系列 4:学术演讲与参会 `academic-presentation-deep-dive/`(新)

**分类(5 个)**:
1. Oral Presentation 结构与讲述节奏——调研依据:SPJ 有专门姊妹讲座"How to Give a Great Research Talk",和论文写作是两套独立方法论
2. Slides 设计原则——区别于系列 2 的论文图表:slide 是"给耳朵听"的,信息密度要求完全不同(一页一个要点,不是塞满数据)
3. Poster 设计与摆摊讲解——6 英尺外要看得清的字号、30 秒电梯讲解怎么组织、怎么应对路人深浅不一的提问(有人只想扫一眼,有人想深聊 20 分钟)
4. 现场 Q&A 应对技巧——和系列 1 的 rebuttal 形成有趣对照(rebuttal 是书面从容准备,这里是即时反应),被问住了怎么体面接住
5. 会议参会实用指南——日程规划、怎么 networking、poster session 作为"参观者"的礼仪、怎么找到该聊的人

**格式模板**:复用系列 1 设计的"六步写作判断力模板",但把"审稿人会怎么挑刺"换成"听众/评委会怎么问"。可运行部分:用 matplotlib/PIL 真实渲染 poster/slide 版式的好坏对照(信息层级示意图,比如"一页塞 8 个要点" vs "一页 1 个要点"的视觉密度对比),延续"能跑代码的地方不空谈"纪律。

**Capstone**:叙事体——从摘要被 poster/oral 接收,到准备 slides,到真实 Q&A 被问到犀利问题当场怎么接。

### 系列 5:成果发布与社区 `research-release-deep-dive/`(新)

**分类(3 个,规模诚实偏小,不硬凑)**:
1. 开源代码发布规范——README/License/复现说明该长什么样。**和 daily-toolkit-deep-dive 的 Git 协作章节划清边界**:那边讲"怎么用 git 这个工具",这里讲"论文代码仓库作为一个成品应该长什么样"(比如需要固定随机种子并写明、需要给出跑通复现结果的确切命令、License 怎么选)
2. HuggingFace Release——model card / dataset card 怎么写、怎么让人愿意用你发布的东西。**已确认本机没有可用的真实 HF 账号/token**(`huggingface_hub.HfApi().whoami()` 返回 `LocalTokenNotFoundError`),按 daily-toolkit SSH 章节的先例诚实处理:语法和本地可验证部分(如构造 model card 内容、`huggingface_hub` 库的本地/dry-run 参数校验、文件结构检查)真实跑,真实上传如实标注"需要读者自己的账号/token,验证颗粒度仅到本地语法层面",不冒充已经真实上传验证过
3. 学术社区互动——论文公开后怎么写一条得体的宣传帖(不夸大不模糊)、怎么回应 GitHub issue 和 follow-up 邮件、怎么面对"在我这不 work"这类反馈

**格式模板**:复用 6 步操作类模板(和系列 3、daily-toolkit 一致)。

**Capstone**:规模小,**不强行配一篇同等重量的叙事 capstone**,收尾用一份可勾选的"发布清单串讲"(把 3 个分类的关键动作串成一份读者能真的照着走一遍的 checklist)。这个规模判断如果后续发现内容其实很丰富,可以再补,不提前假设。

## 跨系列共同纪律

1. **"过来人经验"的来源**:每篇的"常见坑"基于真实调研(WebSearch 真实建议来源、真实数据,已完成第一轮:SPJ 写作七条建议、真实 rebuttal 效果统计、色觉友好绘图研究、venue 选择红旗信号),不是泛泛而谈。执行阶段每个系列还需要针对自己的具体分类补充调研(比如系列 4 需要专门查 SPJ 的"How to Give a Great Research Talk"具体内容、poster 设计的具体研究)。
2. **真实项目素材使用**:涉及案例设计的地方可以直接取材 `research/world-model-imagination-controller/` 已有材料(`00-brainstorm-10-ideas.md`/`01-meeting-briefing.md`/`02-deep-gap-analysis.md`/`03-novelty-competitive-landscape.md`/`04-sharpened-recommendation.md`/`05-full-technical-briefing.md`/`06-world-model-primer.md`/`07-baseline-reproducibility-audit.md`),用户已明确不需要刻意脱敏。但仍然保持基本判断力:不需要逐字复制大段真实结论作为"标准答案"抄进教学文件,借用其真实的论证结构/场景设定来设计范例即可,教学内容的重点是"方法论怎么用",不是"复述这篇论文写了什么"。
3. **不重复已有内容**:系列 5 的代码发布章节明确不重复 daily-toolkit 的 git 操作章节;系列 3/4 涉及的会议/投稿操作若和 daily-toolkit 已有内容(如 LaTeX 编译)重叠,标注交叉引用而不是重写。
4. **诚实的验证颗粒度**:不能真实自动化验证的内容(比如"图形界面操作""需要真实第三方账号的上传"),如实标注验证颗粒度,不冒充和代码同等的验证标准——这是 daily-toolkit-deep-dive/rhcsa-bash-deep-dive 已经确立的纪律,直接复用。
5. **交叉引用**:5 个系列之间互相引用(系列 1 讲"图注怎么写"链接到系列 2;系列 2 的"main result figure"呼应系列 4 的 slides 设计;系列 3 的"Rebuttal 时间管理"和系列 1 的"Rebuttal 写作技巧"互相引用、明确分工不重复)。

## 执行方式

**范围边界(明确排除,避免执行时跑偏)**:
- 不逐一覆盖每个具体会议的详细格式模板(ICML/NeurIPS/ICLR 各自的页数/格式要求年年变,写死了很快过期),只讲**判断框架和检查清单该怎么用**,格式细节交给读者自己查当年 CFP。
- 系列 5 的 HuggingFace 真实上传不做端到端网络验证(无可用账号),如实标注。
- 不代写用户真实论文的具体内容,只用真实项目材料做**教学案例的场景灵感来源**。

**执行机制**:5 个系列相互独立(不同目录、不同文件),适合并行派发。参照 [[tutorial-format-rollout-2026-07-24]] 已验证有效的并行 Agent 派发纪律,每个系列派发给一个独立 agent 承担"该系列的完整研究+撰写+验证",而不是按"分类文件"切得更碎——原因是这类内容需要系列内部统一的方法论声音和难度递进节奏,拆得太碎(比如每个分类单独一个 agent)容易导致同一系列内部风格/深度不连贯。

**统一硬性纪律(写进每一份 agent 派发 prompt)**:
1. 只新增文件,不改动仓库任何已有系列的已有文件(除了自己系列内的新文件)
2. 不碰根 `README.md`(由本人统一收尾更新)
3. 不自行 git commit(由本人核实后统一提交)
4. 不派发嵌套子 agent
5. 可运行代码写进 markdown 前必须真实在 `.venv` 里跑出真实输出
6. 引用其余系列的交叉引用前先实地核实目标文件真实编号,不凭空猜测
7. 不能自动化验证的内容如实标注验证颗粒度,不冒充
8. 涉及真实项目素材(`research/world-model-imagination-controller/`)时先实地读取该目录下的真实文件,不凭空编造项目细节

## 收尾步骤

1. 每个系列独立完成后,本人独立核实(git diff 核对范围、重跑可运行代码、检查文件完整性),不采信 agent 自报
2. 全部 5 个系列验证完成后,统一更新 `for_real_dummy/README.md` 新增"论文发表系列"小节
3. 更新 memory(新增本次的项目记忆,记录 5 系列清单、导师反馈原文、执行过程)
4. git 提交(先各系列独立提交,最后 README 统一提交,和 [[tutorial-format-rollout-2026-07-24]] 的提交节奏一致)
