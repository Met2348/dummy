## 现状盘点(仓库里已有的相关格式,避免重复造轮子)

`for_real_dummy/` 目前的内容分三种风格,分工不同:

| 风格 | 代表 | 特点 |
|------|------|------|
| 知识点参考卡片 | 01-18 号文件,140 个知识点 | 七步模板(签名→一句话→机制→场景→例子→追问→坑),逐点独立,查阅式 |
| 场景化叙事 | 19 号 `mock-interview-capstone`、20 号 `advanced-interview-depth` | 串联多个已有知识点讲一个完整场景(终面 1 小时 / 5 个多级追问链案例),**读者是旁观者**,跟着案例走一遍推理链条,不亲手写代码搭建东西 |
| 入门教程 | 根目录 01-04 号 | 面向"第一次接触"的新手,按顺序读 |

**缺口**:没有一种格式是"读者跟着教程一步步亲手敲代码、每写一段就跑一次看到效果、最终独立组装出一个完整可用的小工具"——这是用户说的"手把手"区别于"场景化叙事"的核心特征:19/20 号文件读者是旁观者(看面试官和候选人的对话推理),而"手把手教程"读者是动手者(自己一步步构建)。

## 范围决策:先做单系列试点,不是 22 条系列全铺开

**为什么不直接铺给 22 条系列**:这是一种全新格式,还没有被验证过读者是否真的觉得"手把手"、深度和篇幅是否合适。本仓库处理"新格式"有过明确先例——`advanced-interview-depth` 这个格式最初只在 `dsa-deep-dive` 试点(见 memory `dsa-deep-dive-complete`),验证质量合格后才在后续 memory `queued-interview-depth-upgrade` 中推广到其余 11 条系列。这次沿用同一策略:**先在一条系列试点一篇,不预先承诺推广范围**,试点产出后留一条明确的"后续推广是独立决定"记录,而不是趁 AFK 窗口把 22 条系列的量都做掉——这不是偷懒省事,是因为格式本身没验证过,批量铺开的风险(方向不对、返工成本)比先出一篇质量过硬的试点更大。

## 试点系列选择:dsa-deep-dive

三点理由(不是随便挑的):
1. **该系列已经是本仓库"新格式先试点"的默认选点**——`advanced-interview-depth` 格式就是在这里首次验证的,复用同一条系列意味着复用户已经熟悉的语境。
2. **零外部依赖**——纯 Python 标准库,`.venv` 直接跑,不需要 WSL2/GPU/数据库引擎,适合在无人盯着的窗口里独立完成并自我验证到底,不会卡在环境问题上。
3. **"手把手搭一个东西"这个格式和 DSA 知识点的组合天然契合**——多个数据结构组合起来正好能拼出一个真实可用的小工具,不像纯数学/统计类知识点那样难以拼装成"动手产物"。

## 新增内容设计:`dsa-deep-dive/21-build-a-mini-search-engine.md`

**选题:从零手写一个迷你搜索引擎。** 不是发明新知识点,是把三个已有知识点串成一个读者从空文件开始、分阶段增量敲出来的完整小工具:

| 阶段 | 目标 | 建立在哪个已有知识点之上 |
|------|------|------|
| 阶段 1 | 倒排索引(word → 命中文档集合),支持多词 AND 查询 | [01 类](../for_real_dummy/dsa-deep-dive/01-complexity-and-python-builtins.md) dict 均摊 O(1)、[02 类](../for_real_dummy/dsa-deep-dive/02-arrays-and-strings.md) 字符串处理 |
| 阶段 2 | 前缀自动补全 | [12 类](../for_real_dummy/dsa-deep-dive/12-trie-and-string-matching.md) Trie 树 |
| 阶段 3 | 按词频相关度排序,只出 Top K | [07 类](../for_real_dummy/dsa-deep-dive/07-heaps-and-priority-queues.md) 堆/TopK |
| 阶段 4 | 组装成一个 `MiniSearchEngine` 类,端到端 demo | 阶段 1-3 全部拼装 |
| 收尾 | 可以怎么继续扩展(只指方向,明确不实现,避免画蛇添足) | 编辑距离(拼写纠错)、TF-IDF、持久化(database-deep-dive) |

**格式规则(和 01-20 号文件的区别)**:
- 不用七步模板,每个阶段是"这一步要让程序多会一件事 → 现场写代码 → 跑起来 → 用 assert 证明它真的按预期工作 → 下一步在这基础上继续加"的节奏。
- 每个 ` ```python ` 代码块必须能被 `_verify_md.py` 独立执行通过(该脚本把每个代码块单独起一个子进程跑,块之间不共享状态)——意味着每个阶段的代码块要么自包含,要么显式重新粘贴前面阶段需要复用的部分,不能依赖"上一个块已经跑过"。
- 复杂度/性能类断言延续 dsa-deep-dive 已确立的纪律(见 [00-roadmap.md](../for_real_dummy/dsa-deep-dive/00-roadmap.md) "复杂度验证方法论"一节):真测不空谈、规模不能小到被噪声淹没、必要时 `best_of` 多次采样取最优。
- 不计入"140 个知识点"统计,和 19/20 号文件同挂,在 roadmap 里单独记一行。
