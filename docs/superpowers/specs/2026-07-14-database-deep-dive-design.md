> 定位:`for_real_dummy/` 新一条独立技能系列,是"职业发展与需求"四部曲(操作系统与并发→计算机网络→数据库原理与实战→系统设计)的第 3 部。第 1 部 os-concurrency-deep-dive(79 点)、第 2 部 computer-networking-deep-dive(80 点)均已完成。四部曲总体顺序已在系列 1 立项时和用户确认,本次不重新讨论顺序,但范围/板块/环境策略按本仓库纪律重新走一轮独立 brainstorming(2026-07-14):用户依次确认①范围广度选择"关系数据库加重",②双真实引擎(PostgreSQL + MariaDB)而非单引擎或纯模拟。

## 1. 目标与规模

约 **65 个知识点**(±10~15% 浮动,先估后核,最终以全库自查的精确计数为准),6 大板块,8 个分类文件 + 1 篇模拟终面 capstone。深度对标本仓库已完成系列(os-concurrency-deep-dive 79 点、computer-networking-deep-dive 80 点)"技术终面 1 小时持续面试级别"的深度广度。

**范围决策(用户 2026-07-14 已确认)**:关系数据库加重——关系模型/索引/事务/存储引擎做到面试深度,分布式数据库与 NoSQL 作为收尾板块点到面试常考程度,不铺开讲(深水区留给四部曲最后一部"系统设计")。这是知识点总数比前两部(79/80)略少的直接原因,但事务/存储引擎相关知识点因为要给出双引擎真实对比证据,篇幅深度不会打折扣。

**环境决策(用户 2026-07-14 已确认)**:真实客户端-服务器数据库引擎装两个——**PostgreSQL 16.14** + **MariaDB 10.11.15**(均已核实可通过 WSL2 Rocky Linux 的 dnf 直接安装),涉及隔离级别/MVCC/锁/复制这类深水区机制的知识点,一律给出两个引擎的真实对比证据,而不是只选一个或纯模拟。这比原计划(单引擎或纯 sqlite3 模拟)工作量更大,是本次 brainstorming 里唯一偏离"效率优先"默认推荐的决策,用户明确选择了最全面的选项。

**每个文件明细(撰写时按此展开,允许在保持知识点独立完整的前提下微调个别点的合并/拆分)**:

- **01 关系模型与SQL基础**:关系模型三要素(关系/元组/属性/域)、主键/外键/唯一约束/检查约束、DDL基础(CREATE/ALTER/DROP的语义与代价)、DML基础(INSERT/UPDATE/DELETE的原子性)、四种JOIN类型(INNER/LEFT/RIGHT/FULL)与执行逻辑、NULL的三值逻辑陷阱、DISTINCT与集合运算(UNION/INTERSECT/EXCEPT)。
- **02 范式与进阶SQL**:1NF/2NF/3NF/BCNF递进式反例驱动讲解、业务场景下的反范式权衡(读多写少场景的性能vs一致性)、GROUP BY/HAVING与聚合函数求值顺序、窗口函数(ROW_NUMBER/RANK/LAG-LEAD)、子查询vs JOIN的等价改写与优化器行为差异、CTE(WITH子句)与递归查询。
- **03 索引结构与查询优化器**:B+树索引结构与磁盘IO复杂度分析(为什么是B+树不是二叉树/跳表)、聚簇索引vs非聚簇索引(InnoDB主键即聚簇索引 vs PostgreSQL堆表+索引)、联合索引最左前缀原则、覆盖索引避免回表、哈希索引适用场景与局限、EXPLAIN/EXPLAIN ANALYZE真实解读(双引擎输出格式对比)、基于代价的查询优化器基础(统计信息/选择率估算)。
- **04 事务与隔离级别**:ACID四性质逐条拆解、四种隔离级别(READ UNCOMMITTED/READ COMMITTED/REPEATABLE READ/SERIALIZABLE)、四种异常现象(脏读/不可重复读/幻读/丢失更新)在PostgreSQL和MariaDB上的真实触发对比(相同隔离级别名字,不同引擎的实际语义差异——尤其REPEATABLE READ,PG是快照隔离天然阻止幻读,MySQL/InnoDB额外靠间隙锁阻止幻读,机制不同结果趋同)。
- **05 MVCC与锁机制**:MVCC快照隔离原理(PostgreSQL的xmin/xmax可见性判断 vs InnoDB的undo log版本链,两种真实实现对比)、锁类型与粒度(行锁/表锁/意向锁)、InnoDB间隙锁与next-key lock、真实死锁复现与检测(两个真实连接构造真实等待图成环,观察引擎自动kill哪一方)、乐观锁vs悲观锁工程选型。
- **06 存储引擎内部机制**:WAL(Write-Ahead Log)为什么能保证持久性、redo log与undo log分工(重做已提交vs回滚未提交)、checkpoint机制与恢复时间权衡、崩溃恢复流程(ARIES三阶段思想简化版:分析/重做/回滚)、缓冲池与页面置换(交叉引用dsa-deep-dive 05类LRU缓存,聚焦"脏页"这个数据库特有维度不重新推导LRU本身)、LSM-tree结构与compaction(对比B+树,为板块VI的NoSQL存储做铺垫)。
- **07 复制与分布式基础**:主从复制原理与binlog/WAL传输、同步复制vs异步复制vs半同步复制的一致性权衡(真实搭建一主一从观察复制延迟)、CAP定理(用真实网络分区场景推导而非死记硬背)、分库分表策略(垂直拆分vs水平拆分,水平分片交叉引用dsa-deep-dive 20类一致性哈希不重复推导)、两阶段提交2PC点到为止(为什么是分布式事务的经典解法但工程上很少直接用,深水区留给系统设计系列)。
- **08 NoSQL与缓存模式**:Redis五种核心数据结构(string/hash/list/set/zset)与底层编码优化、持久化机制RDB vs AOF权衡、缓存模式(cache-aside/read-through/write-through/write-behind)真实对比、缓存一致性问题(双写不一致场景真实复现)、数据库连接池在数据库场景下的特殊性(交叉引用computer-networking-deep-dive 11类连接池通用机制,聚焦"连接是有状态事务边界"这个数据库特有约束)。
- **09 模拟终面capstone**:综合运用01-08类知识点,场景设定为"一次索引失效引发的慢查询雪崩,叠加隔离级别误用导致的幻读业务bug"双线索排查,结构参照 os-concurrency-deep-dive 12类capstone(候选人初版汇报→面试官多轮追问→最终基于真实EXPLAIN输出和真实隔离级别复现数据诊断根因)。

## 4. 五轴追问链方法论落地(从第一天融入,不事后补)

| 轴线 | 含义 | 数据库学科例子 |
|------|------|------|
| 规模递增轴 | 小规模→大规模→极限行为 | 单表百万行索引优化→千万行分表→亿级数据分布式分片 |
| 工程约束递增轴 | 单机→跨机→全球分发 | 单机事务→主从复制读写分离→跨地域多活(为系统设计系列埋伏笔) |
| 方案批判迭代轴 | 面试官连续指出具体缺陷逼换方案 | 全表扫描去重→加唯一索引仍慢(索引本身膨胀)→分区表→分区键选择错误导致热点→复合分区键,每次因具体缺陷被打回 |
| 决策依据追问轴 | 不纠错,只逼问选择依据 | "为什么这里用REPEATABLE READ不用SERIALIZABLE","为什么选乐观锁不选悲观锁" |
| 真实性验证轴 | 简历"优化了慢查询"被追问到具体数字 | "具体加了哪个索引,EXPLAIN的执行计划改动前后分别是什么,扫描行数从多少降到多少" |
| 诊断真实数据(新题型) | 给真实EXPLAIN输出/慢查询日志,要求诊断而非套公式 | 给一段真实EXPLAIN ANALYZE输出,要求判断走没走索引、瓶颈在排序还是回表 |

每个知识点挑 1~2 条最自然的轴线走 2~3 层深,不强行凑满 5 轴。09 类capstone 是唯一要求同时用 3 条以上轴线的文件。

## 5. 验证环境与纪律

**双环境,逐点标注**:

- **默认环境**:仓库根目录 `.venv`(Windows 原生 Python)。stdlib `sqlite3` 覆盖:关系模型/DDL/DML/JOIN/范式反例/聚合与窗口函数/CTE(SQLite 3.35+ 支持窗口函数与递归CTE,版本已在.venv核实)、EXPLAIN QUERY PLAN基础用法、B+树磁盘IO次数的纯算法模拟、LSM-tree compaction的纯算法模拟。
- **WSL2 Rocky Linux(新增真实双引擎,复用 rhcsa-bash-deep-dive 已装好的系统)**:新装 **PostgreSQL 16.14**、**MariaDB 10.11.15**(dnf直接可用,已核实)、**Redis**(需先 `dnf install epel-release` 再装,已核实epel-release当前未装,属于本系列新增的标准环境搭建步骤)。用于:真实EXPLAIN/EXPLAIN ANALYZE输出对比、隔离级别异常真实触发(双引擎对比)、MVCC快照真实观察、锁竞争与死锁真实复现、真实WAL/redo log文件与checkpoint行为观察、真实主从复制搭建、Redis真实数据结构操作与持久化。涉及真实数据库连接的例子通过 `psycopg2`(PG)/`pymysql`(MariaDB)/`redis`-py 从 WSL2 内部连接本地实例(Unix socket或127.0.0.1),一律用 `python-wsl2` 围栏(不在Windows `.venv` 执行,因为Windows端无法直接连接WSL2内部服务的驱动依赖链)。
- 每个知识点的"可运行例子"步骤开头必须显式标注验证环境与引擎(比如"以下例子需在 WSL2 Rocky Linux 验证,依赖 PostgreSQL 16.14 + MariaDB 10.11.15 均已启动"),不笼统一句"已验证"带过。

**并发/隔离级别/真实引擎类特有验证纪律**:

- 隔离级别异常复现涉及两个真实数据库连接的操作时序控制(比如连接A开事务读取,连接B提交,连接A再读取观察是否变化),存在时序敏感性,需要用显式同步点(而非sleep猜时间)控制两个连接的操作顺序,并至少重复5~10次确认现象稳定复现,不满足于运气好复现一次。
- **版本锁定**:PostgreSQL 16.14 / MariaDB 10.11.15 / Redis(版本待装时核实并记录)——凡是例子里断言了具体的EXPLAIN输出格式/系统表结构,注明版本,避免未来引擎版本升级后断言失效被误判为代码错误。
- **不连接外部/生产数据库**:所有数据库实例监听本地(Unix socket 或 127.0.0.1),不连接任何远程/云数据库服务;测试数据全部是脚本内合成的最小数据集,不使用真实业务数据或从外部下载的数据集(类比 computer-networking-deep-dive 的"不引入外部网络依赖"纪律)。
- **需要真实构造死锁/长事务阻塞的 demo 必须带确定性退出的超时安全网**,延续 os-concurrency-deep-dive 死锁 demo 的纪律。

## 6. 与仓库已有内容的边界(已核实,不重复)

- **vs rhcsa-bash-deep-dive**:已用 Grep 核实全部 10 个文件,**完全没有数据库安装配置/运维内容**(05类包管理部署章节不涉及MySQL/PostgreSQL/MariaDB)。这是和前两部最大的结构性差异——不需要专门划一节"操作层vs机制层"边界,本系列在这个方向上是全新领域。
- **vs dsa-deep-dive**:已用 Grep 核实 08类(树)10个子标题,覆盖二叉树族(遍历/BFS/BST/AVL红黑树"思想"/LCA/序列化/树形DP/路径框架/完全二叉树与线段树关系),**完全没有B+树**(磁盘导向的多叉平衡树)——本系列03类可以放心从零讲B+树,不重复。05类已有LRU缓存knowledge point(链表+哈希表实现),本系列06类存储引擎缓冲池章节交叉引用其淘汰策略机制,只聚焦"脏页/WAL先行"这类数据库特有约束,不重新推导LRU本身。20类进阶深度追加案例1已有真实一致性哈希负载均衡实测代码,本系列07类分库分表提及水平分片时交叉引用,不重新写一遍。
- **vs os-concurrency-deep-dive**:03类同步原语(互斥锁/信号量)和05类死锁(死锁四条件/检测算法/资源分配图)讲的是**操作系统层面的进程/线程同步与死锁理论**——通用、与具体应用无关。本系列05类"MVCC与锁机制"讲的是**数据库引擎内部实现的行锁/表锁/间隙锁/next-key lock**,死锁检测是数据库引擎自己维护的等待图(在真实PG/MariaDB上真实触发观察引擎的自动kill行为),两者的"死锁"是同一理论根源(等待图成环)但应用domain和具体实现完全不同层面——已在05类文件头部声明这个边界,不重新推导OS死锁检测算法本身,只引用其"等待图成环"的核心判据。
- **vs computer-networking-deep-dive**:11类已有连接池的通用机制知识点(阻塞/复用的真实证明)。本系列08类提到数据库连接池时交叉引用,只聚焦"数据库连接是有状态事务边界(不能像HTTP连接一样无脑复用,必须在归还前RESET状态)"这个数据库特有约束,不重新推导连接池阻塞/复用机制本身。
- **vs inference-serving-deep-dive**:02类RadixAttention前缀缓存是LLM推理领域的树结构prompt缓存,和数据库缓冲池/查询缓存是完全不同的应用domain,不构成重复,不需要专门边界声明。
