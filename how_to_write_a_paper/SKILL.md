# 顶会论文写作:叙事先行方法论

这个技能把一篇能拿到 strong accept 的论文,拆成**一条灵魂(叙事)**和**一组器官(各部件)**。所有器官的存在理由,只有一个——服务于同一条叙事。这是普通研究者和顶会作者最根本的分野。

> 使用顺序:**先读本文件的"核心方法论"和"审稿人视角检查清单",建立全局世界观;再按任务需要打开 `references/` 下对应的细则文件。** 任何一个部件的写作,都不能脱离核心方法论单独进行。

---

## 一、核心方法论:叙事先行(最高优先级)

**投顶级会议是叙事先行。所有实验、图表、公式都是围绕讲故事服务的。**

1. **顺序是反的。** 不是先做实验再看能讲什么,而是先确定要讲一个什么故事,再让所有材料去支撑它。叙事一旦确定,就一切围绕它服务。

2. **一个实验不支持,不代表叙事崩塌。** 当某个实验结果不支持叙事,正确做法**不是**修改叙事(美其名曰"诚实"),而是**抛弃这个结果**,去思考还有什么实验和证据最有利于支持这个叙事。从其他侧面寻找支持的证据。

3. **证据链是选择性呈现的。** 现实世界永远复杂、矛盾,任何主张都能找到反例。你不可能在一篇文章里讲清现实的全貌——只能挑一个侧面深刻描写。论文展示的是一条**闭合的证据链**:一个事实如果不支持叙事,它有趣,但不属于这篇文章,它是另一篇 follow-up 的事。

4. **严禁主动展示负面内容。** 不要不断汇报所谓"诚实"的负面结果。在研究语境里,把矛盾证据混进同一叙事,会让文章不可读、让审稿人困惑、让主张面目模糊——这不是美德,是弄巧成拙,亲手凿穿自己的证据链。

5. **笃定地写,如同主张已被彻底证实。** 缺数字可以留 TODO 待补,但绝不凭空捏造。语气要有底气,犹豫的措辞("可能""也许")会削弱整篇说服力。

6. **现在很大程度是 AI 审稿。** 你在稿件里露出的任何破绽,都会被精准抓住攻击。证据链必须天衣无缝。

### 策展 vs 学术不端(必须分清的红线)

- **合法的策展**(我们做):用**真实的数字**讲一个**聚焦、干净**的故事;被略去的是真实但偏离主线的支线。把一团乱麻里最清晰的那条线挑出来给人看。
- **学术不端**(绝不碰):伪造数据、数据泄露、没跑基线就宣称超越、benchmark 上做暗手脚、没有 CI/没跑简单基线。这是无中生有,与策展性质完全不同。

## 二、架构与品味基线

- **架构必须 FANCY、复杂、精巧。** 不要教科书式的东西。核心机制至少是 Transformer 或图神经网络起步(仅为例)。
- **真 fancy 是内容,假 fancy 是符号。** 精巧体现在机制本身真有巧思,而不是把朴素机制用唬人的符号包装。
- **绝对禁止的 trivial 设计:** 多层感知机、二元语法、有限状态机、循环神经网络、长短期记忆网络等——凡是已广泛出现在本科教科书上的方法,写上去等于直接拒稿。尤其严禁多层感知机出现在任何地方。
- **必须强模型、强基线、强证据链。**

## 四、审稿人视角检查清单(交付前必过)

把自己切换成一个**挑剔的、想拒掉这篇文章的审稿人**,逐条自检。凡是你能挑出的破绽,审稿人一定能挑出。

**叙事层**
- [ ] 整篇是否只论证**一个**核心主张?读完能否用一句话复述出来?
- [ ] 标题、摘要主张句、引言洞察段,是否是同一件事的三种长度版本?口径完全一致?
- [ ] 引言开头许下的那个缺陷,结尾的数字是否精确地填上了?(首尾闭环)
- [ ] 有没有把不支持主张的、暧昧的、相反的结果主动摆进正文?(若有,删掉——它属于另一个故事)

**逻辑链层(无跳跃)**
- [ ] 方法里每一个"我们用了什么",读者能否就地找到"因为什么"?(每步都有桥)
- [ ] 结果里每个数字,能否回溯"实验设置→方法→这个数字"的来路?
- [ ] 任何术语、符号、设定,是否在第一次"被使用"前已经"被介绍"过?
- [ ] 结论宣称的,有没有超出实验证明的?(过度宣称=送刀给审稿人)

**一致性层(无矛盾)**
- [ ] 符号:同一个量是否全篇始终用同一个符号?
- [ ] 术语:核心方法/模块是否全篇同一个叫法?
- [ ] 数字:正文、摘要、表、图里的同一个数是否完全一致?(最致命的一类不一致)
- [ ] 图文:架构图里画的每个模块,正文都提到了吗?四方(方法-图-公式-消融)是否锁定?

**赢得干净层**
- [ ] 比较是否公平?(同接口、同训练数据、同任务设定、基线被正确复现)
- [ ] 基线是否够强?(领域里最新、最被认可的方法都在对比表里)
- [ ] 提升是否可信?(不是单次运气,最好有方差/多次运行)
- [ ] 消融能否堵死"提升来自无关因素"和"换个简单设计也行"这两句话?

**可读性层**
- [ ] 有没有"该交代却没交代"导致读者卡壳的地方?(破解"知识的诅咒":假设读者对项目一无所知)
- [ ] 有没有术语堆砌、故作高深?(讲不清=没搞懂)
- [ ] 抽象概念后是否紧跟具体例子或直觉?

**终极判断:** 把论文给一个只读了标题、摘要、看了图、扫了主结果表的怀疑型审稿人。两分钟后他应能准确复述:你解决什么问题、凭什么洞察、怎么做的、拿什么证明了——且这四件事咬合成一个无懈可击的闭环。做到,叙事就成功了。

## 五、强制写作约定(语言与术语)

- 任何新术语第一次出现,先用一整段大白话讲清"它是什么",再跟两个形象的例子。
- 英文缩写第一次出现必须跟全称与中文。除数据集名、方法名外,尽量不用英文缩写。
- 严禁自己发明"字母+数字"代号(如某个字母代指一个任务)指代不清的内容。
- 能用平实语言讲清的,就不要堆术语。讲不清,说明自己还没搞懂。
