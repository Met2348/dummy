# Software Engineering Mastery 使用说明

> **先读 `01-textbook.md`**:那是老手要求补充的本科课件式完整教材(七章、连贯教科书叙述),本页只是DeepPoint/ScenarioPoint追问链的使用说明,建议在读完教材之后再来这里练习追问链。

## 为什么这次多了一个 `explain` 字段

老手指出用户 EE 转 NLP/LLM 背景,缺少科班 CS 学生默认已掌握的软件工程/设计模式/数据库/网络/操作系统基础知识,特别嘱咐"因为没有系统学过,所以要讲仔细"。这和此前 `foundation-model-mastery`/`federated-learning-mastery`/`diffusion-mastery` 三个专题的性质不同——那三个专题用户本身已有研究生级基础,只需要把已懂的东西整理成"能被追问三层还接得住"的面试话术;这五个 CS 基础专题是**从零学**,所以每个知识点先给一段系统性的 `explain` 讲解,再接追问链或场景判断,练习时**不要跳过 explain 直接看 chain**——追问链的第二三层经常假设你已经理解了 explain 里讲的机制。

## 三层怎么练

1. **Tier 1(浅,`tier1_shallow/`)**:先建立"软件是怎么被组织开发出来的"这套基础认知——SDLC模型、敏捷/Scrum/Kanban这些团队协作方法论,以及"代码怎么被多人协作管理"(git分支模型、code review、CI/CD)。这一层几乎不需要预备知识,但要注意敏捷不是"绝对正确"的方法论,瀑布模型在特定行业(强监管的航天/医疗)仍然有其合理性,面试追问经常考"你能不能讲清楚适用边界"而不是"你是不是无脑信奉敏捷"。
2. **Tier 2(深,`tier2_deep/`)**:方法论深水层,三个文件分别对应"怎么保证代码质量"这个大问题的三个角度——测试方法论(怎么验证代码是对的)、架构与模块化(怎么组织代码让它易于维护)、代码质量与重构(怎么持续改善已经写出来的代码)。这三者互相支撑:好的架构让测试更容易写,好的测试让重构更安全,重构又让架构持续保持健康,建议练习时留意这条互相支撑的关系,而不是把三个文件当成互相独立的知识点。
3. **Tier 3(社招级别,`tier3_social_hire/`)**:无标准答案的资深判断题。第一个文件考"技术债务和工程文化怎么权衡"(重写vs重构、优先级判断、怎么推动一个团队开始写测试),第二个文件考"交付和发布策略怎么选"(灰度vs蓝绿、DORA指标、故障复盘文化)。这两个文件的 `explain` 字段是建立判断所需的背景知识框架,不是给出标准答案——真正的资深判断力体现在"知道要考虑哪几个维度"而不是"背出正确选项"。

## 一个容易被忽视的陷阱

这五个 CS 基础专题的知识点大多是几十年沉淀下来的共识(SOLID原则、测试金字塔、CAP定理等),看起来"背下来就行",但面试追问链的第二三层往往考的是"为什么",而不是"是什么"。比如"为什么rebase不能用在已推送的共享分支上""为什么100%覆盖率不代表测试到位""为什么正方形继承矩形违反里氏替换原则"——如果只背了结论没理解机制,追问到第二层就会露怯,这正是 `pitfall`/`trap` 字段反复提醒的地方。
