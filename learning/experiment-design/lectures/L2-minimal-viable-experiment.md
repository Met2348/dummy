# L2 · 最小验证实验 (MVE): 一周拿到 go/no-go, 而不是三个月后才发现方向错了

> 30-min lecture · 目标: 学会设计**最小验证实验 (Minimal Viable Experiment)** —— 用最少的算力和时间, 先验证假设的「核心机制到底存不存在」, 再决定要不要 scale。

---

## 0. 新手最贵的错误: 一上来就跑全套

有了假设, 新手的冲动是: 把消融矩阵全排满 (5 方法 × 4 规模 × 3 数据集 × 5 种子 = 300 次训练), 提交, 等三天, 然后发现 —— 核心机制根本不成立, 这 300 次全白跑。

> **研究里最贵的不是算力, 是「在错误方向上花的时间」。** 一个博士被卡住的最常见原因, 不是不够努力, 是**在一个不 work 的 idea 上努力了太久才放弃**。MVE 是这个风险的解药: 用最便宜的实验, 尽早拿到「这条路通不通」的信号。

类比创业的 MVP (最小可行产品): 别先造一辆完整的车, 先造个滑板车验证「人们到底想不想要代步工具」。MVE 是 MVP 在研究里的版本。

---

## 1. MVE 的设计原则: 砍到只剩「核心机制」

MVE 的核心问题是: **我的假设依赖的那个「核心机制」, 用最小的设置能不能看到它的影子?**

我们的假设是「Robust-DPO 在高噪声下更鲁棒」。它的核心机制是: **噪声会伤 DPO, 而 Robust-DPO 的额外机制能挡住这个伤害。** 那么 MVE 只需要回答: **在一个最小设置下, 高噪声真的伤 DPO 吗? Robust-DPO 真的挡住了吗?**

砍枝叶, 留核心:

```
   全套实验 (300 次)                  MVE (≤10 次)
   ─────────────────                 ──────────────
   5 个方法                    →      2 个: DPO vs Robust-DPO (假设直接涉及的)
   4 个模型规模                →      1 个: 最小能跑的 (如 1B 或更小)
   3 个数据集                  →      1 个: 最标准的那个
   噪声 0/10/20/30/40%        →      2 个极端: 0% 和 40% (看交互的两端就够)
   5 个种子                   →      3 个 (够算个粗方差, L5)
   = 300 次训练               →      2×1×1×2×3 = 12 次, 还能更少
```

逐条交代砍的逻辑:
- **方法**: 只留假设直接对比的两个。其它方法 (KTO/ORPO) 是「锦上添花」, MVE 阶段不碰。
- **规模**: 用**最小能复现机制**的规模。机制 (噪声伤害) 如果真实, 在小模型上也该有影子。规模化 (L4 泛化) 是机制确认**之后**的事。
- **噪声**: 只取**两端** (0% 和 40%)。交互效应的存在与否, 看两端的差距是否拉开就够; 中间点 (10/20/30%) 是确认机制后才补的「剂量曲线」。
- **种子**: 3 个 (够算个粗 std 判断「差距是否淹在噪声里」, L5)。

---

## 2. MVE 的产出: 一个 go / no-go 决策

MVE 跑完, 你要得到一个**明确的三岔决策**, 而不是「嗯好像有点用」:

```
                   MVE 结果
                      │
      ┌───────────────┼────────────────┐
   ✅ GO            🔁 PIVOT          ❌ NO-GO
   核心机制确认       机制部分对,        机制不存在,
   (40%下差距明显)    但不是我想的样子    (噪声没伤DPO/
   → scale 到全套     → 修正假设再 MVE    Robust没挡住)
   (L4 消融矩阵)                        → 放弃或换方向
```

> 关键: **MVE 的每个结局都是有价值的**, 包括 NO-GO。一个一周就否掉错误方向的 NO-GO, 帮你省下三个月。新手害怕 NO-GO (觉得是失败), 但**快速、廉价地证伪自己的 idea, 正是高效研究者的核心能力**。你的目标不是「让 MVE 成功」, 是「让 MVE 给你一个可信的信号」。

---

## 3. MVE 的反模式: 别把它做成「缩水的全套实验」

MVE 容易跑偏成两种样子, 都要避免:

| 反模式 | 症状 | 纠正 |
|---|---|---|
| **伪 MVE (其实是全套)** | 「我先小规模跑一下」结果还是排了 100 个设置 | 问: 验证核心机制**最少**需要几次? 通常 ≤10 |
| **测错了东西** | MVE 测了一堆边角指标, 没测「核心机制在不在」 | MVE 只测假设最依赖的那**一个**信号 |
| **没有判据** | 跑完看着数发呆, 不知道算 go 还是 no-go | 跑之前就写好 go/no-go 的**数值阈值** (接 L1 的「幅度」) |

> 最后一条最关键, 它直接连回 L1: 你的假设里那个「**幅度** (≥5 点)」和「**无噪声时无差异**」, 就是 MVE 的 go/no-go 判据。**先写阈值, 再跑实验** —— 否则你会不自觉地把任何结果都解读成「有希望」(确认偏误)。

---

## 4. MVE 怎么接全套实验

MVE 是 GO 之后, 后面三讲依次接管:

```
   MVE (本讲): 2 端噪声 × 2 方法 × 3 种子, 确认机制存在 (GO)
        │
        ▼
   L3 控混淆: 把 baseline 做到真公平 (同数据/同算力/同调参预算), 否则差距可能是假的
        │
        ▼
   L4 消融矩阵: 补全噪声剂量曲线 (0/10/20/30/40%) + 加方法 (KTO/ORPO),
                看交互效应的完整形状
        │
        ▼
   L5 方差显著性: 加种子到足够, 报 error bar + p 值 + 效应量, 让结论站得住
```

> 给博0 的现实节奏: **第一篇论文的第一个月, 应该全花在 MVE 循环上** —— 提假设 → MVE → go/pivot/no-go → 修假设 → 再 MVE。在这个阶段反复、快速地试错很便宜。等机制确认了 (GO), 再投入算力做 L4 全套消融。**把贵的实验留到最后, 把便宜的证伪放到最前。**

---

## 5. 本讲小结 + 通往 L3

- 别一上来跑全套; 先用 **MVE** 验证「核心机制存不存在」, 一周拿信号。
- MVE = 砍到只剩假设直接涉及的核心: 最少方法、最小规模、只取极端条件、够算粗方差的种子。
- 产出是 **go / pivot / no-go** 决策; NO-GO 也有价值 (省你三个月)。
- 判据 (go/no-go 阈值) 必须**跑之前**写好 (就是 L1 假设里的「幅度」)。

> **下一讲 L3「baseline 与对照」**: MVE 给了 GO 信号, 但「Robust-DPO 看起来更好」可能是假的 —— 也许只是它偷偷多调了参、多用了数据。L3 教你怎么把对照做到**真公平**, 排除混淆变量, 否则你的差距经不起审稿人 (9.3 攻击清单 Q1) 一问。

**动手**: 拿你 L1 写的假设, 用 `templates/experiment-protocol.md` 设计一个 ≤10 次运行的 MVE: 写清「砍掉了什么、为什么砍、go/no-go 阈值是多少」。然后去 N1, 在模拟器上把这个 MVE 真跑一遍。
