# L3 · baseline 与对照: 让「差距」经得起审稿人的追问

> 35-min lecture · 目标: 学会设计**公平的 baseline** 和**受控对照**, 识别并消除混淆变量 (confounders) —— 让你测到的差距真的来自你的方法, 而不是来自「你给自己开了小灶」。

---

## 0. 一个让你警醒的事实

ML 论文里, 「我的方法 beat baseline」这句话**经常是假的**, 而且作者往往不是故意的。最常见的原因不是数据造假, 是:

> **对照不公平。** 你的新方法多调了 50 次超参, baseline 用了别人论文里的默认值; 你的方法用了更多数据 / 更长训练 / 更大 batch。差距是真的, 但它来自**这些隐藏的不公**, 而不是来自你的方法本身。

回忆 9.3 攻击清单的 Q1: 「baseline 是不是被故意调弱了 (sandbagging)?」—— L3 就是教你**不被这条攻击命中**。审稿人最爱问的就是「你的提升会不会只是因为调参/算力不对等」, 而这个问题十有八九能问倒新手。

---

## 1. 混淆变量 (confounder): 差距的真正来源可能不是你以为的那个

**混淆变量**: 一个同时影响「你的自变量」和「因变量」的第三者, 让你误以为是自变量造成了效果。

```
   你以为:     method (DPO→Robust)  ───────►  win-rate ↑

   实际可能:   method (DPO→Robust)
                     │ (你顺手给 Robust 多调了参)
                     ▼
              超参质量 ────────────────────►  win-rate ↑
              (真正的原因是这个混淆变量, 不是 method)
```

经典 ML 混淆变量清单 (设计实验时逐个排查):

| 混淆变量 | 它怎么骗你 | 怎么控制 |
|---|---|---|
| **调参预算不对等** | 新方法调 50 次, baseline 用默认 | 给两边**同样的调参预算** (如各 random search 20 次) |
| **算力/训练量不对等** | 新方法训更久 / batch 更大 | 固定 total compute / steps / tokens |
| **数据不对等** | 新方法偷偷多用了数据 | 完全相同的训练集 |
| **评测不对等** | 在对新方法有利的子集上测 | 同一个固定测试集, 测之前定好 |
| **实现质量不对等** | 自己方法精心实现, baseline 随便抄 | baseline 用官方实现 / 强复现 |
| **种子挑选** | 报新方法最好的种子 vs baseline 最差的 | 同样的种子集, 全报 (L5) |

> 控制混淆的总原则一句话: **除了你要研究的那一个变量, 其它一切都保持相同 (ceteris paribus)。** 这就是「控制变量法」。听起来基础, 但 ML 里做到出奇地难, 也出奇地少人认真做 —— 这反而是你做出**可信**结果的机会 (回忆 9.3 的复现 gap: 严肃的公平对照本身就是贡献)。

## 2. 三种 baseline, 你至少要有前两种

| baseline 类型 | 它回答 | 例 |
|---|---|---|
| **平凡基线 (trivial)** | 「比啥都不做强吗?」 | 不对齐的原始模型 / 随机 / 多数类 |
| **直接竞争者 (SOTA prior)** | 「比现有最好的方法强吗?」 | 普通 DPO (你声称要超越的那个) |
| **消融自身 (ablated self)** | 「我方法里那个新组件真的有用吗?」 | Robust-DPO 去掉鲁棒项 = 退化成 DPO |

逐个交代为什么需要:
- **平凡基线**: 防止你的方法其实「还不如不做」。听起来荒谬, 但很多花哨方法在公平对照下连平凡基线都打不过。它是你的**地板**。
- **直接竞争者**: 这是你假设里的那个 baseline (普通 DPO)。**对照必须和它公平** (上一节的混淆控制全用在这)。它是你的**靶子**。
- **消融自身**: 把你方法里「新东西」拿掉, 看效果是否消失。这是证明「**是我的新组件**带来提升, 而不是别的」的关键 —— 直接通往 L4 消融。

> 注意第三种 (ablated self) 和 L4 的关系: 「Robust-DPO 去掉鲁棒项 = DPO」意味着**你的 baseline 本身就是一次消融**。好的实验设计里, baseline 和 ablation 常常是同一件事的两面。

---

## 3. 公平对照的「同等努力」原则

最难也最重要的一条: **给 baseline 和你的方法投入同等的努力。**

新手的天然偏心 (大多无意识):
- 自己的方法: 精心实现、反复调参、debug 到完美、挑好种子。
- baseline: 抄个默认配置、跑一次、用上。

这种「努力不对等」是 ML 可复现危机的头号元凶。一个广为人知的现象: **很多声称超越 baseline 的新方法, 在 baseline 被认真调参后, 优势就消失了** (经典案例: 一批 GAN/推荐系统/对齐方法被「公平再评估」后打回原形 —— 这正是 9.3 说的复现 gap 富矿)。

> 自检问题 (跑完实验问自己): **「如果我把花在自己方法上的调参努力, 同样花在 baseline 上, 差距还在吗?」** 如果你心虚, 说明对照不公平。诚实的研究者会**主动把 baseline 调到最强**, 然后才声称超越 —— 因为打败一个弱 baseline 没有意义, 骗到的发表会在复现时崩塌, 砸的是自己的招牌。

## 4. 在我们的运行例子里控混淆

回到 Robust-DPO vs DPO。要让对照公平, 必须固定 (这正是 `experiment.py` 模拟器的设计):

```
   两个方法之间, 唯一不同的是 method 本身; 以下全部相同:
   ✓ 同一份偏好数据 (同样注入 40% 噪声, 同样的噪声 mask)
   ✓ 同样的模型规模 / 初始化
   ✓ 同样的训练步数 / batch / 学习率搜索预算
   ✓ 同一组种子 {0,1,2,...}, 两边都跑、都报
   ✓ 同一个固定测试集算 win-rate
   ───────────────────────────────────
   → 这样, 若 Robust-DPO 在 40% 噪声下更高, 唯一合理解释就是 method 差异
```

> 模拟器里 `run_experiment(method, noise, seed)` 对两个方法用**完全对称**的接口和同样的种子 —— 这不是偷懒, 这是「受控对照」的代码化身。真实实验里你要靠 config 管理 (9.5) 来保证这种对称。

---

## 5. 本讲小结 + 通往 L4

- 「beat baseline」经常是假的, 元凶是**对照不公平** (混淆变量), 不是数据造假。
- 排查混淆: 调参预算 / 算力 / 数据 / 评测 / 实现质量 / 种子, **除研究变量外一切相同**。
- 至少要有**平凡基线** (地板) + **直接竞争者** (靶子); 最好加**消融自身** (证明是新组件起效)。
- **同等努力原则**: 主动把 baseline 调到最强再声称超越; 否则复现时崩塌, 砸自己招牌。

> **下一讲 L4「消融矩阵」**: baseline 公平了, 现在要系统地拆解「我方法的哪个部分、在什么条件下起作用」。L4 教你排**消融矩阵 (factorial design)**, 并讲一个新手几乎都忽略、却最有信息量的东西 —— **交互效应** (Robust-DPO 的好处为什么「取决于噪声」)。

**动手**: 在 N1 里, 故意制造一次「不公平对照」(比如只给一个方法多跑种子挑最好的), 看差距怎么被夸大; 再恢复公平对照, 对比两者。**亲手把混淆变量制造出来一次, 你就再也不会无意中犯它。**
