# L4 · 实时多模态约束: 流式、低延迟、边界

> 25-min lecture · 目标: 理解实时多模态 (尤其语音/视频) 的工程约束, 收口 M10.5, 接回 harness-engineering。

---

## 0. 从「离线批处理」到「实时流式」

前面的 token 化都假设「整个输入到齐了再处理」(离线)。但 omni 模型最热的应用 —— **实时语音对话、实时视频理解** —— 要求**边来边处理、边想边说** (流式)。这引入一整套新约束, 是 2026 omni 工程的核心战场 (GPT-4o 实时语音、实时 agent)。

---

## 1. 约束一: 延迟预算 (latency budget)

实时语音对话, 人能忍受的「开口延迟」约 200-500ms。这个预算极其紧, 要切分给:

```
   你说完 → [语音→token 编码] → [模型生成响应 token] → [token→语音 声码器] → 你听到
            ~?ms                ~?ms (自回归, 慢!)        ~?ms
   ───────────────────────────────────────────────────────────────
   全程必须 < ~300ms, 否则对话「卡顿」感明显
```

最大瓶颈是**自回归生成慢** (M10.4-L2 软肋): 一个个生成 token, 一句回应几百 token, 串行累积延迟。解法接你已有的专题:
- **投机解码** (你的 speculative-decoding 专题): 小模型猜、大模型验, 加速生成。
- **流式生成**: 不等整句生成完, 边生成边播 (生成一部分语音就开始播)。
- **更小/更快的模型 + 量化** (你的 quantization 专题)。

> **你的推理优化专题在实时 omni 里全部变现。** 实时多模态本质是「推理延迟工程」的极限挑战, 你 M5 的知识 (vLLM/投机/量化) 直接是这里的武器。

---

## 2. 约束二: 流式处理 (streaming)

输入边来边处理, 不能等全部到齐:
- **流式编码**: 语音/视频一帧帧来, 编码器要能增量处理 (不能等整段)。
- **因果性**: 实时模型只能看「过去」, 不能看「未来」(离线模型可以双向看全部)。这限制了架构 (要因果/单向)。
- **打断处理**: 用户说话时打断模型 (barge-in), 模型要能停下来听。这是对话系统的核心交互。

> 流式 + 因果 + 可打断, 把模型从「函数 (输入→输出)」变成「持续运行的进程」。这正是你 harness-engineering 专题的世界 —— **实时 omni 模型需要一个 harness 来管理流式状态、打断、生成调度**。M10.5 在这里和你的 harness 工程接上。

---

## 3. 约束三: 时序边界与同步

多模态实时还有「对齐」难题:
- **音视频同步**: 嘴型和声音要对上 (lip sync), 差几十 ms 就「出戏」。
- **跨模态时序**: 「用户指着图说话」时, 语音和视觉指向要在时间上对齐。
- **缓冲 vs 延迟**: 缓冲多一点更稳 (能看更多上下文) 但延迟大; 缓冲少延迟小但易抖。

> 这些是实时系统的经典权衡 (你 EE/系统背景熟悉)。在 omni 模型里, 它们决定了「实时多模态体验」流不流畅。

---

## 4. M10.5 收口 + M10 主线回顾

M10.5 把 token 化扩到时序模态, 并触及实时:
```
   L1 视频 = 图+时间 (时空 patch, 压缩+编码运动)
   L2 音频 = 波形→mel 谱→token (你的 EE 本行)
   L3 omni = 所有模态进同一条 token 流, 统一 transformer
   L4 实时 = 流式/低延迟/同步 (接 harness + 推理优化)
```

整个 M10 (10.1-10.5) 的主线: **从「给 LLM 装眼睛」(VLM) 到「给 LLM 装所有感官 + 实时」(omni)**。核心不变: token 是通用货币, 统一架构是归宿, 而工程上要和「token 数爆炸」「实时延迟」斗争。

---

## 5. 本讲小结 (M10.5 收口) + 通往 M10.6

- 实时 omni (语音/视频对话) 要求**流式 + 低延迟 + 同步**, 是 2026 工程战场。
- **延迟预算**紧 (~300ms); 瓶颈是自回归慢 → 用**投机解码/流式生成/量化** (你 M5 专题变现)。
- **流式处理**: 增量编码 + 因果 + 可打断 → 模型变成持续进程, 需 **harness** 管理 (接你 harness-engineering)。
- **时序同步**: 音视频对齐、缓冲 vs 延迟权衡 (你 EE/系统本行)。

> **下一专题 M10.6「VLM 评测与幻觉」**: 模型能读能画能听能说了, 但它**说得对吗**? VLM 有独特的评测难题 (视觉幻觉: 图里没有却说有)。M10.6 教你评测 VLM、探测视觉幻觉, 接你的评测安全专题 + 9.6 出图。
