# L3 · 方法示意图: 论文 Figure 1 的设计

> 30-min lecture · 目标: 学会设计**方法示意图** (schematic / architecture diagram) —— 不展示数据, 而是讲清「你的方法怎么运作」。它通常是论文的 Figure 1, 是审稿人理解你贡献的第一道门。

---

## 0. 两类图, 两种功能

论文里的图分两大类, 别混:

```
   数据图 (data figure)              示意图 (schematic / method figure)
   ──────────────────               ─────────────────────────────────
   柱/线/散点/热力图                  框 + 箭头 + 模块 (pipeline/架构)
   展示: 结果 (实验数据)              展示: 方法 (你的系统怎么运作)
   工具: matplotlib (L1/L2)          工具: draw.io/Excalidraw/TikZ/PPT
   位置: 结果章节                    位置: 通常是 Figure 1
```

L1/L2 讲了数据图。这一讲讲示意图。**为什么 Figure 1 这么关键**:

> 审稿人翻到你的论文, 第一个认真看的图往往是 Figure 1 (方法图)。它要在 **30 秒内**让审稿人「看懂你的方法在做什么」。一张好的 Figure 1 能让审稿人带着善意读下去; 一张糊的, 会让他从一开始就困惑、烦躁。**Figure 1 是你方法的电梯演讲的视觉版** (呼应 9.8)。

---

## 1. 一张好方法图的三个层次

设计方法图, 按这三层从粗到细:

### ① 信息流 (从哪到哪)

方法图的骨架是**数据/信息怎么流动**: 输入 → 各处理模块 → 输出。先把这条主干理清:

```
   输入 (偏好数据)  →  [加噪声]  →  [Robust-DPO 损失]  →  策略模型  →  输出 (对齐质量)
                                        ↑
                                   [鲁棒加权] ← 你的创新组件 (要突出)
```

- 主流向用**粗箭头**, 一眼能跟。
- 你的**创新组件**要视觉突出 (颜色/位置/标注), 因为审稿人要一眼看到「新在哪」。
- 别画成一团乱麻 —— 如果箭头互相交叉缠绕, 说明你的信息流没理清 (或方法本身没想清)。

### ② 抽象层级 (画多细)

最难的判断: **画多详细?** 太粗 (三个大框) 说明不了方法; 太细 (每个张量维度都标) 没人看得下去。

> 原则: **抽象到"读者能复述你方法的核心机制"的程度, 不多不少。** 一个张量的具体 shape 不该进 Figure 1 (那是正文/附录的事); 但「鲁棒加权这一步发生在损失计算前」这种**核心机制**必须画出来。问自己: 「删掉这个细节, 读者还懂我方法的关键吗?」不懂 → 留; 懂 → 删。

### ③ 视觉一致性

- 同类东西用同样的视觉 (所有「数据」用一种形状, 所有「模型」用另一种)。
- 颜色和你的数据图**统一** (Robust-DPO 在示意图和结果图里是同一个颜色)。
- 字号够大 (同 L2, 缩小后要读得清)。

## 2. 工具选型

| 工具 | 形态 | 优点 | 适用 |
|---|---|---|---|
| **Excalidraw** | 手绘风网页 | 快、免费、风格友好 | 草图 / 不太正式的图 |
| **draw.io (diagrams.net)** | 框图网页/桌面 | 免费、模块丰富、导出 SVG | 大多数方法图, 推荐 |
| **TikZ / PGF** | LaTeX 代码 | 矢量完美、和论文风格统一、可版本控制 | 追求极致 + 愿意学 LaTeX |
| **PowerPoint / Keynote** | 演示软件 | 人人会用、导出 PDF | 应急 / 也能出不错的图 |
| **matplotlib (本专题)** | Python 代码 | 可复现、进 git、改数据重生成 | 简单 pipeline/flow |

> 给博0 的建议: **方法图起步用 draw.io** (够用、免费、导出矢量 SVG/PDF)。如果你的图是简单的 pipeline/flow 且想要**可复现** (改一处不用重画), 用本专题 `schematic.py` 的代码方式。复杂架构图 (多分支、嵌套) 仍建议 draw.io / TikZ —— 别硬用代码画复杂图, 得不偿失。

## 3. 代码化简单示意图: schematic.py 的演示

复杂架构图用专业工具, 但**简单的横向 pipeline** 完全可以代码化, 好处是和数据图一样可复现 (9.5):

```python
import schematic as sc
import matplotlib.pyplot as plt
fig, ax = plt.subplots(figsize=(11, 1.8))
sc.draw_pipeline(ax, ["读论文找gap", "可证伪假设", "最小验证", "消融+方差", "出版级图", "写论文"])
```

> 这演示一个观念: **图是代码产物, 不是手工艺品。** 能代码化的图 (pipeline/flow/简单架构), 代码化的回报是: 改一个 stage 名不用重画整张、进 git 可 diff、和论文一起版本化。这呼应 9.5 的「结果/图都该能由代码重建」。

## 4. 方法图的常见病

| 病 | 症状 | 治 |
|---|---|---|
| **一团乱麻** | 箭头交叉缠绕, 看不出主流向 | 重排布局, 主流向走直线, 支流退到边 |
| **抽象层级混乱** | 一个框是「整个模型」, 旁边一个框是「一次矩阵乘」 | 同一张图里各模块抽象层级要一致 |
| **创新点淹没** | 你的新组件和标准组件长得一样, 看不出新在哪 | 用颜色/边框/标注突出创新组件 |
| **和正文脱节** | 图里的名字 (模块 A) 正文里从不提 | 图和正文用同一套术语, 互相引用 |
| **纯装饰** | 画了一堆好看的图标, 不传递方法信息 | 每个视觉元素都要有信息职责 (data-ink 的示意图版) |

## 5. 本讲小结 + 通往 L4

- 论文两类图: **数据图** (L1/L2, 讲结果) vs **示意图** (本讲, 讲方法)。Figure 1 通常是后者, 是审稿人理解贡献的第一道门。
- 好方法图三层: **信息流** (主流向清晰、突出创新) → **抽象层级** (抽象到"能复述核心机制") → **视觉一致** (同类同形、配色统一)。
- 工具: draw.io (推荐起步) / TikZ (极致) / matplotlib (简单 pipeline 可复现)。
- 常见病: 一团乱麻 / 层级混乱 / 创新淹没 / 与正文脱节 / 纯装饰。

> **下一讲 L4「图的诚实性 + 整合论文」**: 图能清晰, 也能**误导** (截断 y 轴把 0.3% 吹成"巨大提升")。L4 讲可视化的诚实红线 (9.3 攻击清单的可视化版)、图注怎么写成 self-contained、图与正文怎么呼应。

**动手**: 用 draw.io (或 `schematic.py`) 给你某个复现方法画一张 Figure 1 草图。先只画**信息流**主干 (输入→模块→输出), 再标出「如果这是我的创新, 哪个框要突出」。
