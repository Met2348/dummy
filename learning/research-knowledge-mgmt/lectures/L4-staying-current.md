# L4 · 信息流系统: 如何追 arxiv 而不被淹没

> 35-min lecture · 目标: 建一条**可持续**的信息流 —— 用每周固定的少量时间, 把 arxiv 洪流变成「本周 3-5 篇精读清单」, 而不是天天焦虑、囤积、最后一篇没读。

---

## 0. 这一讲治的病

NLP/LLM 是 arxiv 上更新最快的领域之一, cs.CL + cs.LG 每天上百篇。新手必经的三种病:

1. **FOMO (错失焦虑)**: 怕错过重要工作, 于是什么都想看, 结果什么都没看进去。
2. **囤积**: 开了 40 个标签页 / 收藏夹存了 200 篇, 一篇没读, 还更焦虑了。
3. **被动**: 只看推上谁火转谁, 信息源被算法绑架, 错过冷门但和你方向高度相关的工作。

> **根本问题: 把「追踪」当成了「阅读」。** 追踪 (scan) 和阅读 (read) 是两件事, 用两套时间预算。混在一起, 就会既没追全也没读深。L4 教你把它们分开。

---

## 1. 核心模型: 三层漏斗 (scan → triage → read)

```
   ┌──────────────────────────────────────────────────────────┐
   │ 第 1 层 SCAN (扫描)   每天/隔天 5 分钟, 只看标题            │  数百篇 → 圈出 ~20
   │   信息源: arxiv RSS / Semantic Scholar alerts / 选定的人  │
   ├──────────────────────────────────────────────────────────┤
   │ 第 2 层 TRIAGE (分诊)  每周 30 分钟, 看摘要 + 打分         │  ~20 → 排出本周 3-5 篇
   │   工具: src/arxiv_triage.py (按你的关键词权重打分排序)     │
   ├──────────────────────────────────────────────────────────┤
   │ 第 3 层 READ (精读)   每周固定时段, 三遍读法 (9.3)         │  3-5 篇 → 文献卡 + 永久笔记
   │   产出: 进第二大脑 (L1/L2), 可能触发 idea (L3)            │
   └──────────────────────────────────────────────────────────┘
```

三层各有独立的时间预算, 这是关键:
- **SCAN**: 极廉价, 只读标题。目的是「不漏掉可能相关的」, 不是读懂。**5 分钟看 200 个标题, 圈出 20 个。**
- **TRIAGE**: 中等成本, 读摘要 + 用工具打分。目的是「从 20 个里挑出最该深读的 3-5 个」。**这是 arxiv_triage.py 干的活。**
- **READ**: 昂贵, 走完整三遍读法 (9.3)。**只给通过分诊的少数。**

> 为什么分层省命: 如果你对每篇都用 READ 的强度, 一天读不完 5 篇; 如果对每篇都只 SCAN, 又什么都没读懂。**分层让你把昂贵的注意力只花在最值的地方。**

---

## 2. 信息源选型 (SCAN 层)

别只靠刷推 (算法会喂你热闹而非相关)。搭一个**主动拉取**的多源系统:

| 源 | 用途 | 怎么配 |
|---|---|---|
| **arxiv RSS / API** | 主力, 拉特定分类 (cs.CL/cs.LG) 每日更新 | RSS 订阅 `arxiv.org/rss/cs.CL`; 或本专题 `arxiv_triage.fetch_live()` |
| **Semantic Scholar alerts** | 追「关键论文的新引用」「关键作者的新作」 | 在 semanticscholar.org 关注论文/作者, 邮件提醒 |
| **Google Scholar alerts** | 追关键词 / 追你自己论文被谁引 | Scholar → 创建快讯 |
| **选定的 5-10 个人** | 你方向最活跃的学者, 看他们读/转什么 | X / 个人主页 / 关注他们的 Semantic Scholar |
| **会议 proceedings** | ACL/EMNLP/NeurIPS 出 accepted list 时集中扫一轮 | 每年几次, 看 accepted papers 标题 |

> 给博0 的建议: **源不是越多越好, 是越准越好。** 刚入门时配 arxiv RSS (你的 1-2 个子分类) + 3-5 个关键人就够。源太多 = 又一种信息过载。

## 3. TRIAGE: 用 arxiv_triage.py 把打分自动化

人工读 20 篇摘要再排序很累, 而且每次标准会漂移。`src/arxiv_triage.py` 把它变成一条**可重复、可解释**的打分:

$$\text{score}(p) = \sum_{k}\ w_k \cdot \big(\text{count}_{\text{abs}}(k) + \beta \cdot \text{count}_{\text{title}}(k)\big)$$

逐项交代每个符号:
- **$p$**: 一篇论文 (有 title 和 abstract)。
- **$k$**: 你关心的一个关键词 (如 `reasoning` / `dpo` / `reproducibility`)。
- **$w_k$**: 你给关键词 $k$ 的**权重** (你最在意的给高权重)。
- **$\text{count}_{\text{abs}}(k)$ / $\text{count}_{\text{title}}(k)$**: 关键词 $k$ 在摘要 / 标题里出现的次数。
- **$\beta$**: 标题加权倍率 (代码里 `TITLE_BONUS=2.0`)。命中标题比命中摘要值钱 —— 标题是作者认为最重要的词, 所以乘一个更大的系数。
- **$\text{score}(p)$**: 这篇对**你**的相关分。高分进精读队列, 0 分直接归档。

工具的两个出口:
- `triage(papers, weights)`: 给全部论文打分排序 (含每篇命中了哪些词, 可解释)。
- `reading_list(papers, weights, top_k=5)`: 只返回分数 > 0 的前 5 篇 —— 你的**本周精读清单**。

> 设计意图: 把「读不读」从**情绪决策** (今天心情/FOMO) 变成**参数决策** (我的关键词权重)。权重写在一个地方, 你随研究方向演进去调它, 而不是每天重新纠结。**这就是「系统」相对「靠意志力」的优势: 决策一次, 复用一年。**

## 4. weekly review: 让整套系统活起来的心跳

前面 L1-L3 + 本讲的所有部件 (文献库、永久笔记、idea inbox、triage), 如果没有一个**固定的回看节奏**, 都会退化成墓地 (回忆 L2)。

**weekly review (每周回顾, ~45 分钟)** 是整个第二大脑的心跳。用 `templates/weekly-review.md`, 固定做四件事:

```
   每周固定时段 (如周五下午):
   ┌─────────────────────────────────────────────┐
   │ 1. 清 inbox   把本周捕获的闪念逐条处理:        │
   │               双链 / 升级永久笔记 / 丢弃        │  ← 喂养 L3 孵化
   │ 2. 分诊 arxiv  跑 arxiv_triage, 定下周精读清单 │  ← L4 TRIAGE
   │ 3. 翻新笔记   随机翻 3 条旧永久笔记, 补链接      │  ← 防 L2 笔记墓地
   │ 4. 推进 idea  看 idea 库, 哪个该升级/该砍/该做  │  ← L3 筛选/立项
   └─────────────────────────────────────────────┘
```

逐项交代为什么是这四件:
- **清 inbox**: 不清, 捕获的念头就烂在入口, 流水线断在第一关。
- **分诊 arxiv**: 把追踪集中到每周一次, 其余时间不被信息流绑架 —— **治 FOMO 的核心是「定期且有限」**。
- **翻新笔记**: 主动把旧笔记翻出来, 是对抗笔记墓地、激活知识复利 ($r>0$) 的唯一办法。
- **推进 idea**: 让 idea 流水线持续往前流, 不积压。

> 一句话: **weekly review 是把「一堆工具」变成「一个活系统」的那根心跳线。** 没有它, 你只是装了一堆 app; 有它, 你有了一个会自我维护、持续复利的研究知识系统。这是本专题最该养成的一个习惯 —— 比任何工具配置都重要。

## 5. 整个 9.1 收口: 你现在拥有了什么

```
   ┌─────────── 第二大脑 (9.1 建成) ───────────┐
   │                                            │
   │  L1 文献层   Zotero + bib_to_cards          │
   │       ↓ (读完写批注)                        │
   │  L2 笔记层   永久笔记 + 双链网络 (Obsidian)   │
   │       ↓ (笔记末尾的开放问题)                 │
   │  L3 idea层   inbox → 孵化 → 筛选 → idea卡    │
   │       ↑ (进料)                              │
   │  L4 信息流   arxiv triage → 精读 → 入库       │
   │                                            │
   │  ❤ weekly review 每周让以上全部转一圈        │
   └────────────────────────────────────────────┘
            │ 立项的 idea 卡交棒给
            ▼
        9.4 experiment-design (做实验)
```

你现在有了一个**会复利的容器**。后面 9.2-9.9 产生的每张卡、每个 gap、每个 idea, 都落进这个容器, 而不再是一次性消费。**这就是地基专题的意义。**

---

## 6. 本讲小结

- 把「追踪」和「阅读」分开: **scan (看标题) → triage (看摘要+打分) → read (三遍读法)** 三层漏斗, 各有独立时间预算。
- 信息源要主动拉取 (arxiv RSS / Semantic Scholar / 选定的人), **准比多重要**。
- 用 `arxiv_triage.py` 把「读不读」从情绪决策变成参数决策 (关键词权重一次配, 复用一年)。
- **weekly review 是整个第二大脑的心跳**: 清 inbox / 分诊 / 翻新笔记 / 推进 idea, 缺它整套系统会退化成墓地。

**动手**: 去 `N2-arxiv-triage.ipynb`, 用你自己的研究关键词 (如 `{reasoning:3, rl:2, alignment:2}`) 对一批论文打分, 生成你的第一份「本周阅读清单」。然后把 `weekly-review.md` 设成你日历里每周五的固定 45 分钟。
