# L2 · 推荐信策略 (Recommendation Letter Strategy)

> 30-min lecture · 目标: 把"求推荐信"从一件临到投递前几天才慌忙张口的尴尬事, 变成一套**提前数月经营、有明确素材交付**的长期关系管理动作。
> 关键区分: **推荐信的质量, 90%取决于你在"请"这个动作发生之前几个月甚至几年做了什么, 只有10%取决于你请的措辞好不好。** 把重心放在措辞上, 是在错误的地方努力。

---

## 0. 为什么这是个真问题

`career-pathways`(9.10)L1讲过, 学术求职材料初筛阶段, **推荐信的分量往往比你自己写的所有材料加起来还重**——招聘委员会读了几十份材料, 常常靠推荐信里那句"这是我带过最好的几个学生之一, 具体体现在……"来做第一轮筛选。但绝大多数博士生第一次认真对待推荐信这件事, 是投递前两三周才发邮件问导师"能不能帮忙写一封"——这时候推荐人手里往往只有一个模糊印象, 写出来的信大概率是"该学生学习刻苦、能力扎实"这类可以套在任何人身上的模板话, 而不是能让委员会记住的具体细节。

问题的根源不是推荐人不愿意花时间, 而是**推荐人手里没有能写出具体细节的素材**。一封强推荐信之所以强, 是因为它能讲出具体的、可核查的事: "在XX项目里, 他/她自己发现了baseline的一个隐藏假设问题, 独立设计了验证实验"——这种细节, 推荐人不会凭空记得, 除非你在过去几个月/几年里**持续、具体地**让对方了解你在做什么。这正是本讲要处理的核心: 怎么让"请推荐信"这个动作, 建立在早就存在的、扎实的信息基础上, 而不是临时抱佛脚地现造细节。

```
                  临时抱佛脚模式                          长期经营模式(本讲)
──────────────    ─────────────────────────────         ─────────────────────────────
时间点            投递前2-3周才第一次开口                  入学起持续沟通, 请求只是"正式确认"这个动作
推荐人手里的素材   模糊印象, 只能写套话                     具体项目细节+目标岗位描述+你想强调的点
信的内容          "刻苦、能力强、综合素质好"类套话          具体事例+可核查的贡献+和岗位的匹配点
推荐人的心理负担   突然被要求交付一份重要文档, 压力大         提前预告+分阶段确认, 负担分散在数月里
```

这一讲和`research-life`(9.9)L2"导师/合作沟通"(见`research-life` L2 `L2-mentor-collaboration.md`)直接相关——那一讲教你怎么开有明确ask的meeting、怎么让导师持续了解你的进展。**推荐信策略本质上是那套"向上管理"技能的一个具体应用场景**: 你平时meeting里养成的"让导师清楚知道你做了什么、卡在哪里"的习惯, 恰恰就是几年后推荐信能不能写具体的决定性因素。如果平时meeting只是被动汇报进度、从不主动突出自己的具体贡献, 到了要请推荐信的时候, 才发现导师对你的印象停留在"这个学生还行"这种模糊层面。

## 1. 三个阶段: 选人 · 喂料 · 请求

### 阶段① 选推荐人: 能讲具体细节的人, 而不是头衔最大的人

很多人默认"找最资深/最有名的人写信最有分量"——这个判断不完全错, 但**如果这个人对你的了解仅限于一次课或一次简短交流, 写出来的信必然空洞**, 空洞的大牛推荐信有时反而不如一封具体、有细节的普通合作者推荐信有说服力。选人时优先考虑: 这个人能不能具体讲出至少一到两件你做过的、有技术含量的事; 其次才考虑头衔和名气。

### 阶段② 喂料: 提前给推荐人一份"素材包", 而不是等对方自己回忆

这是整个策略里最核心、最常被忽略的一步。在正式请求写信之前(甚至可以在没有具体求职计划时就开始), 定期(比如每半年)主动向合作过的导师/合作者做一次简短的进展同步, 内容包括: 你最近做了什么具体的技术决策、遇到了什么具体的困难、怎么解决的。到了真正需要推荐信的时候, 给对方一份具体的"素材包": 你的最新CV、目标岗位的具体描述、以及你希望对方**重点提及的1-3个具体事例**(而不是笼统地说"随便写写就好")。这呼应`application_kit_audit.py`工具`recommenders`一节的核心检查点: **提前沟通、给推荐人素材包, 而不是临时才开口。**

**一个具体可执行的辅助习惯**: 维护一份"成果记录文档"(brag document), 每次完成一件值得一提的事(独立解决了一个技术难题、主动补了一个关键实验、帮同门排查出一个隐藏bug), 当场花一分钟记一句话, 而不是等半年后凭记忆回想。这份文档可以直接挂在`research-knowledge-mgmt`(9.1)的"第二大脑"体系里(见`research-knowledge-mgmt` L1 `L1-why-second-brain.md`), 作为你个人知识管理系统的一个专门条目——它的价值不只是给推荐人喂料, 也是`L5`要讲的CV版本化管理的原始素材来源, 半年后写CV或者给推荐人素材包时, 直接从这份记录里摘取具体事例, 远比临时回忆靠谱。

### 阶段③ 请求: 给足时间, 明确说清用途和截止日期

正式请求时, 至少提前4-8周, 明确告知这封信用在哪个具体岗位/项目、截止日期是什么时候、投递系统怎么操作。如果对方犹豫或者语气不够肯定, **认真对待这个信号**——一封勉强答应但不够上心的推荐信, 有时反而不如换一个更愿意认真写的人。

### 特殊情况: 推荐人要求你自己先写一份草稿

一个很多博士生第一次遇到时会措手不及的情况是: 推荐人(尤其是很忙的资深教授)在答应写信之后, 反过来说"你先写一版草稿, 我改一改"。这在学术界是相当普遍的操作方式, 并不是推荐人不负责任或者敷衍——很多资深教授同时要写十几封推荐信, 让被推荐人先起草能大幅提高效率, 也让内容更贴近被推荐人真正想强调的重点。**遇到这种情况, 最忌讳的反应是觉得"这不该是我做的事"而抗拒或应付了事**——既然本讲的核心论点是"推荐信质量取决于素材是否具体", 那么亲自起草恰恰是把这个原则发挥到极致的机会: 你比任何人都清楚自己做过的具体细节, 完全可以写出一版远比推荐人凭记忆写出来更扎实的草稿。

起草时要注意几个具体要点: **用第三人称写**("该学生在……中展现了……", 而不是"我在……中展现了……"), 语气要符合推荐人平时说话的分寸(不要写得比推荐人平时的评价夸张很多, 否则推荐人改起来会很别扭), 并且要给出至少两三个具体的、可核查的事例而不是空泛的赞美。写完之后**明确告诉推荐人这只是一份草稿供参考**, 完全欢迎大幅修改甚至推翻重写——这个动作的目的是帮推荐人省时间、提供具体素材, 不是替对方"决定"信里要写什么。

## 2. 一个完整走查示例

场景: 你博士期间和一位合作者一起做过偏好优化相关的复现改进工作(对应你已有的`dpo-family`专题积累), 现在准备投递学术界和工业界两类岗位。

```
选人: 除了导师, 还选了这位具体合作过、能讲清楚你独立设计实验细节的合作者,
      而不是只找系里最有名但只见过一次面的教授。
喂料: 过去一年里每隔几个月主动同步过一次进展, 这次求信前一周,
      给对方发了CV+目标岗位描述+3个希望重点提及的具体事例
      (独立定位实验设计缺陷/快速迭代出可行方案/主动补充关键消融实验)。
请求: 提前6周发出请求, 明确说明这封信同时会用在1个学术界岗位和1个工业界岗位,
      询问是否需要针对不同岗位写两个侧重略有不同的版本。
```

这个例子里没有一步依赖"临场发挥的好运气"——每一步都建立在过去持续沟通积累的具体信息之上, 这正是本讲想要传达的核心态度: **推荐信写得好不好, 早在你正式开口请求的那一刻之前就已经被决定了大半。**

## 3. 常见误区对照表

| 误区 | 具体表现 | 修正方式 |
|---|---|---|
| 只找头衔最大的人 | 推荐人对你了解有限, 信写得空洞 | 优先选能讲具体细节的人 |
| 临到投递前才开口 | 推荐人只能凭模糊印象写套话 | 提前数月持续同步进展, 分阶段确认 |
| 不给具体素材 | 让推荐人"随便写写", 信缺乏细节 | 提供CV+岗位描述+希望强调的具体事例 |
| 只求信不维护关系 | 平时只在需要时才联系 | 把喂料当成长期习惯, 而非求信前的临时动作 |

## 4. 经典参考

- **Peter Feibelman, 《A PhD Is Not Enough: A Guide to Survival in Science》**: 书中讨论导师/合作者关系维护的章节强调, 良好的推荐信建立在长期、真实的工作关系之上, 而不是求信当下的临时沟通, 这和本讲"喂料先于请求"的核心论点一致。

## 5. 本讲小结 + 通往 L3

- 推荐信质量的大部分决定因素发生在"请求"这个动作之前的数月甚至数年, 而不是请求措辞本身。
- 三阶段策略: 选能讲具体细节的推荐人 → 提前定期喂料(素材包+具体事例) → 给足时间明确请求。
- 和`research-life` L2"向上管理"的关系: 平时meeting里养成的具体沟通习惯, 就是几年后推荐信能不能写具体的决定性因素。

> **下一讲 L3「设计Job Talk」**: 你现在有了扎实的推荐信支撑, 走到了校园面试/on-site这一步——下一讲处理这个环节里最容易翻车的部分: 怎么设计和演练一场"说服评委录用你"导向的research talk, 它和常规会议报告的修辞目标完全不同。

**动手**: 列出2-3位可能为你写推荐信的人, 对每个人写下: 上一次你主动向他们同步具体进展是什么时候、内容是什么。如果超过半年没有主动同步过, 现在就发一条简短的进展更新。
