# L2 · 算力资源规划与申请 (Compute Resource Planning)

> 30-min lecture · 目标: 把"需要更多算力"这句在预算表和组会上都最容易被随口带过的话, 拆成一套可执行的算力规划动作: 分清三种算力来源各自的申请流程、学会估算云计算预算、并准备好组内多人抢卡时能站得住脚的谈判材料。
> 关键区分: **本讲不教你怎么用Slurm提交作业**——那是`training-orchestration`已经讲过的集群调度技术(见该专题`01-slurm.md`)。本讲教的是**在提交作业之前**, "要多少算力、什么时候用、找谁申请、申请不到时的备选方案是什么"这件规划性的事, 这是`training-orchestration`完全没有覆盖的一层, 也是`funding_plan_audit.py`工具`compute_plan`一节要检查的内容。

---

## 0. 为什么这是个真问题

预算论证(L1)里最容易被写得笼统的一项就是算力开销——"申请50万元用于GPU计算"和"申请50万元用于开展本研究"犯的是同一个错误。更隐蔽的问题是: 算力不像人力或差旅那样有统一的申请渠道, 而是同时存在**至少三种来源**, 每种的申请流程、时间尺度、谈判对象完全不同, 新手常常把它们混为一谈, 结果要么申请材料准备不充分被拒, 要么临到真正要跑实验时才发现批下来的资源根本不够。

```
来源                    典型场景                       申请动作                                  时间尺度
────────────────────   ────────────────────────────  ────────────────────────────────────────  ──────────
组内/校内共享集群         日常实验、小规模消融               找组内wiki/系统管理员开账号+提交quota申请工单       数天
外部大型算力(如NSF        大规模训练/多机多卡长跑             写allocation request(分级申请, 见第1节)         数周-数月
ACCESS-CI等国家级资源)
云计算(AWS/GCP/Azure)    弹性峰值需求/排队来不及时的兜底方案     预算审批+成本估算(GPU-hour单价×时长+存储+网络)     依预算审批周期而定
```

## 1. 三种算力来源与对应的申请流程

### ① 组内/校内共享集群
流程最短但资源有限: 通常需要联系系统管理员开通账号, 再针对具体项目提交quota申请工单, 说明预计需要的存储空间和排队优先级。这一层的"申请"更多是行政流程, 真正的技术操作(排队策略/backfill/预留窗口)属于`training-orchestration`讲过的调度知识, 本讲不重复。

### ② 外部大型算力资源
以美国NSF资助的**ACCESS**(Advanced Cyberinfrastructure Coordination Ecosystem: Services & Support, 2022年接替此前运行了11年的XSEDE项目)为例, 它把资源申请按规模分成若干档: **Explore/Discover**档面向评估资源是否适用、研究生课题、小规模基准测试, 门槛低、审批快; **Accelerate**档面向有经验的用户、中等规模需求、多经费项目的整合使用; **Maximize**档面向需要大规模计算资源的项目, 审批周期最长、需要最详细的资源使用论证。**核心设计原则是让申请材料的详细程度和你实际需要的资源规模匹配**——不需要为了申请一个小规模的benchmark测试就准备一份Maximize档才需要的详尽材料, 反之亦然。国内的国家超算中心/教育网算力平台通常也采用类似的分级申请逻辑, 具体名称和门槛以当年官网公布的申请指南为准。

### ③ 云计算资源
弹性最大但成本最直接: 需要在预算审批阶段就把成本估算清楚(见第2节), 而不是先用了再算账。云计算的另一个隐藏优势是**可以作为前两种资源排队/申请不到时的兜底方案**——这正是`funding_plan_audit.py`工具`compute_plan`一节要求写清楚的"备选方案"。

---

## 2. 云计算预算估算方法

估算公式的骨架: **总成本 ≈ GPU-hour单价 × 预计使用小时数 × 实例数 + 存储成本 + 网络传输成本**。三个最容易被低估的坑:

- **spot/抢占式实例 vs on-demand(按需)实例**: spot价格通常显著低于on-demand, 但随时可能被回收——只适合能容忍中断、有checkpoint机制的训练任务, 不适合对连续性要求高的实验, 需要在预算里同时评估"用spot省下的钱"和"被抢占导致重跑的隐性成本"。
- **网络传输(egress)费用**: 把训练结果或大规模数据从云端下载出来, 很多云服务商单独计费, 且往往比计算本身的账单更容易被忽略, 预算论证阶段就要问清楚这一项。
- **存储的持续计费**: 训练数据和checkpoint如果长期挂在云端存储, 是按时间持续计费的, 需要在预算里明确"用完之后是否及时清理"这一环节, 否则会在项目周期结束后仍持续产生费用。

## 3. GPU资源battle的谈判技巧

组内多人同时抢卡是几乎所有实验室都会遇到的场景。**最没有说服力的谈判方式是"我更需要"**——这是主观判断, 无法被第三方核实, 也无法被系统管理员或PI用来做优先级排序。更有效的做法:

- **用具体job profile代替笼统请求**: 提前给出显存需求、精度设置(fp16/bf16/fp32)、预计wall-time, 让排优先级的人能核实"这个请求是否合理", 而不是仅凭印象判断谁更急。
- **把资源请求和项目里程碑绑定**: "这次实验对应下周三的组会汇报节点"比"我这周需要跑一个实验"更容易获得优先排队权, 因为它把资源分配和整个团队都认可的时间节点挂钩, 而不是个人主观的紧迫感。
- **主动提出备选方案**: 与其单纯申请更多资源, 不如同时给出"如果排不到8卡, 我可以先用4卡跑较小规模的验证版本"这类降级方案, 这既降低了系统管理员/PI拒绝你的心理成本, 也呼应了本讲第0节强调的"申请不到时的备选方案"这一核心要求。

## 4. 一个走查示例: 算力规划从模糊到具体

```
反例(模糊, 站不住脚):
  "本项目需要更多GPU资源来完成大规模实验, 现有算力不够用。"

  问题诊断:
  · 没说清楚"更多"是多少——审查者/系统管理员无法核实合理性。
  · 没有区分该走哪种来源(组内集群/外部大型资源/云计算), 无法判断该走哪条流程。
  · 完全没有备选方案, 一旦申请被拒就没有退路。

改写(具体, 可核实):
  "本项目第2-4月需要8×A100持续运行约6周(约合8000 GPU-hours), 优先申请组内
   共享集群的quota; 若组内集群排队超过2周, 备选方案是通过ACCESS的Accelerate
   档申请外部算力(已核实项目规模落在该档要求区间); 若两者都无法在实验窗口期
   内到位, 最后备选是用云计算spot实例补齐缺口, 预算已按第2节公式估算出上限。"

  为什么这版能站住脚:
  · 具体数字(8×A100、6周、8000 GPU-hours)让审查者能核实合理性。
  · 明确了申请顺序(先组内, 再外部大型资源, 最后云计算兜底), 对应第1节三种来源。
  · 每一层都配了触发条件("若排队超过2周"), 不是空泛地说"需要更多"。
```

## 5. 常见误区

1. **把三种算力来源混为一谈** —— 症状: 用申请外部大型算力的详尽材料去申请组内集群的小额quota, 或反过来用一句"需要更多算力"去申请国家级资源。解法: 先判断量级, 再匹配对应流程。
2. **只估算计算本身的成本, 忽略存储和网络** —— 症状: 预算表只写GPU-hour成本, 上线后才发现存储和egress费用远超预期。解法: 套第2节的三项估算公式, 逐项列出。
3. **资源battle里只讲主观紧迫感** —— 症状: 反复强调"我很急", 却拿不出具体job profile或里程碑依据。解法: 第3节的两条谈判技巧。
4. **完全没有备选方案** —— 症状: 一旦申请不到理想规模的资源就完全卡住, 没有降级方案。解法: 备选方案应该在申请前就想好, 而不是被拒之后临时现想, 参照第4节的分层触发条件写法。

## 6. 可参考的官方指南

- **ACCESS(access-ci.org)官方文档**: NSF资助的国家级先进计算资源协调生态, 公开说明了Explore/Discover/Accelerate/Maximize分级申请制度, 是理解"资源申请材料应该和资源规模匹配"这一原则最直接的官方案例, 也是L1提到的经费申请分级思路在算力这个具体场景下的对应版本。
- 也见本仓库`training-orchestration/01-slurm.md`——那一讲讲的是集群任务调度的技术操作(排队策略/backfill/预留窗口), 是本讲"申请到资源之后怎么用"的技术基础, 本讲不重复, 只聚焦"资源怎么被申请到"这一前置环节。

## 7. 本讲小结 + 通往 L3

- 算力有至少三种来源(组内集群/外部大型资源/云计算), 各自申请流程、时间尺度、谈判对象都不同, 不能混为一谈。
- 云计算预算估算要同时覆盖计算/存储/网络三项, spot实例的省钱要和被抢占的隐性成本一起权衡。
- GPU资源battle最有说服力的谈判方式是具体job profile+里程碑绑定+主动备选方案, 而不是"我更需要"这类主观陈述。

> **下一讲 L3「数据管理规划(DMP)」**: 算力规划解决的是"计算资源怎么申请", 但很多资助机构同样强制要求你写清楚"数据本身怎么管理"——存在哪、谁能访问、论文发表后是否公开。L3 讲这份常被当作走过场附件、实则容易被打回的文档该怎么写。

**动手**: 打开 `templates/funding-plan-worksheet.md` 的「算力资源规划」小节, 针对你手头(或设想中)的一个项目, 判断它该走哪种算力来源, 并写出一条具体的备选方案。
