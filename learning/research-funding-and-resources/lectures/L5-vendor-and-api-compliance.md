# L5 · 供应商/API/第三方合规评估 (Vendor & API Compliance Review)

> 30-min lecture · 目标: 把"这个API/数据集能不能这么用"这个工业界研究和使用第三方资源的学术研究里都极常被忽略、直到论文投稿或产品上线前才暴露的问题, 变成一套提前核查的合规评估清单: 数据处理协议是否签、模型/API使用许可是否覆盖你的实际用途、数据集许可证是否允许你打算的发表或商用形式。
> 关键区分: L3讲的是"你自己产生或采集的数据"该怎么管理, 本讲讲的是"你使用的第三方资源"(供应商的API、别人发布的模型、别人整理的数据集)本身附带的使用条款——这是完全不同方向的合规问题: L3问"我的数据能不能被别人用", 本讲问"别人的资源能不能被我这样用", 两者都属于资助/资源规划里容易被当作"技术问题"随意处理、实则是合规问题的环节。

---

## 0. 为什么这是个真问题

工业界研究团队(以及越来越多依赖商业API做实验的学术研究者)有一个共同的隐患: **"公开可访问" 不等于 "可以随意这样用"**。一个API可以被任何人调用, 但它的服务条款可能明确禁止把返回结果用于训练竞品模型; 一个数据集可以被免费下载, 但它的许可证可能只允许非商业研究使用, 不允许你把基于它训练的模型商用或作为产品的一部分发布。这类问题最危险的地方在于它**不会在项目早期报错**——你可以顺利完成实验、写完论文, 直到投稿或产品上线前的合规审查环节才发现许可证根本不覆盖你的用途, 此时往往已经没有时间重新收集数据或更换模型。

---

## 1. 三类常见合规缺口

### ① 数据处理协议(Data Processing Agreement, DPA)
当你把数据(尤其含个人身份信息PII的数据)交给第三方服务商处理时(比如调用第三方API做标注、调用云端服务做推理), 欧盟GDPR(General Data Protection Regulation)第28条明确要求数据处理者(processor)必须通过具有法律约束力的书面协议(即DPA)接受数据控制者(controller)的约束, 明确处理范围、安全措施和责任划分。即使你的研究不直接面向欧盟用户, 只要处理的数据涉及欧盟居民信息, 或者你的合作机构/资助方要求参照GDPR标准, 这条规则就可能适用——**核心提醒是: 把含PII的数据交给第三方处理之前, 先确认是否需要一份正式DPA, 而不是假设"我们只是调用一下API, 不算正式的数据处理关系"**。

### ② 模型/API使用许可核查
各大模型/API提供商(如OpenAI、Anthropic等)都公开发布自己的使用政策(usage policy)和服务条款(terms of service), 但具体条款逐年更新、逐家不同, 常见需要核查的问题包括: 能否用API的输出结果去训练/微调其他模型(很多厂商明确禁止用自家API输出训练竞品)、能否把跑出来的benchmark分数写进论文发表(通常允许, 但部分厂商对可比性声明和披露方式有具体要求)、商用和非商用场景的条款是否不同。**这类条款以厂商官网当年发布的版本为准**, 本讲不替你总结具体条款内容(避免过时或误导), 但强调的原则是: 项目立项阶段就该把这类核查列为待办事项, 而不是等论文写完了才想起来看条款。

### ③ 数据集许可证覆盖范围
训练/评测数据集本身通常带有明确许可证声明(如CC-BY、CC-BY-NC、自定义的"仅限非商业研究使用"许可), 需要核实你打算的用途(发表论文/开源代码/商用产品)是否落在许可证允许的范围内。"仅限非商业研究"类许可证是最容易被忽略的限定——工业界研究团队如果直接使用这类数据集训练即将商用的模型, 即使论文本身没有问题, 也可能构成许可证违规。

---

## 1.5 为什么工业界研究场景尤其容易踩这个坑

学术界的数据集使用大多发生在"发论文"这一单一终点, 合规风险相对单一(主要是能不能引用/能不能公开代码); 工业界研究天然多一层复杂度: 同一份实验产出, 既可能被写成论文, 又可能被直接并入产品迭代, 还可能被用作和其他团队合作的素材。这意味着**同一个第三方资源(API或数据集), 在"发论文"这个用途下可能完全合规, 换到"商用"这个用途下就可能违规**——而工业界的项目节奏往往比学术界更快, 合规审查很容易被当成"上线前最后再补"的手续性环节, 而不是立项时就该问的问题。

```
用途                        论文发表         内部研究复现        商用产品
────────────────────────   ─────────────    ─────────────      ─────────────
"仅限非商业研究"数据集         通常允许          通常允许            通常不允许
禁止训练竞品的API输出         需核查具体条款     视条款而定          需核查具体条款
含PII的第三方数据处理          需评估是否需要DPA  同样需要评估          几乎肯定需要DPA
```

同一份资源在不同用途栏位下的合规结论可能完全不同, 这正是第0节强调"公开可访问≠可以随意这样用"的具体体现——**核查的对象不是资源本身, 而是"资源+具体用途"这个组合**。

## 2. 一个走查示例: 核查清单

```
场景: 工业界研究团队打算调用某商业模型API生成一批合成数据, 用来微调自己团队
      正在训练的下一代模型, 并计划把结果写成论文投稿、后续模型也计划商用。

核查清单(立项阶段, 而非投稿前):
  □ 该API的使用政策是否允许把输出结果用于训练其他模型?
    （很多厂商的条款对此有明确限制, 需要在数据生成前而非生成后确认）
  □ 如果计划把评测分数写进论文, 该厂商条款是否对比较性声明有具体要求?
  □ 涉及的用户数据(如果有)是否需要签署DPA?
  □ 如果后续模型计划商用, 用到的任何第三方数据集是否都是商用友好许可
    (而非"仅限非商业研究")?

反例(常见踩坑路径):
  团队先做完实验、写完论文、准备商业化上线前才第一次认真读API条款,
  发现条款明确禁止"使用输出结果训练可能构成竞品的模型"——此时数据已经
  生成、模型已经训练、论文已经在投稿流程中, 补救成本极高。

正确顺序: 核查清单应该在立项/写经费申请的资源规划阶段就过一遍
(呼应`funding_plan_audit.py`工具`vendor_compliance`一节), 而不是当作
项目收尾阶段才想起来的"最后一步"。
```

## 3. 常见误区

1. **把"公开可访问"等同于"可以随意用于研究发表"** —— 症状: 因为一个API/数据集任何人都能访问, 就默认没有使用限制。解法: 访问权限和使用许可是两件事, 前者只说明"能不能拿到", 后者才规定"拿到之后能不能这样用"。
2. **忽略条款里"仅限非商业研究"这类限定** —— 症状: 学术研究阶段随意使用, 完全没考虑后续如果转商用或产业合作, 许可证是否仍然允许。解法: 立项阶段就问清楚"这个项目未来有没有可能商用", 如果有, 从一开始就核查商用友好的替代资源。
3. **不签DPA就处理第三方的PII数据** —— 症状: 觉得"只是调用个API, 没有正式签合同"。解法: 涉及PII处理的第三方服务, 主动确认是否需要DPA, 而不是假设不需要。
4. **把合规核查放在项目收尾阶段才做** —— 症状: 论文写完、模型训完才第一次认真读使用条款。解法: 套第2节的核查清单, 在立项/资源规划阶段就过一遍。

---

## 4. 可参考的官方规定

- **GDPR(General Data Protection Regulation)第28条**——欧盟关于数据处理者(processor)必须签署书面数据处理协议(DPA)、明确处理范围与责任的官方法律条款, 是理解"为什么需要DPA"这一制度背景的权威来源。
- 主要模型/API提供商(如OpenAI、Anthropic等)公开发布的usage policy/terms of service——具体条款逐年更新, 使用前请以厂商官网当年发布的最新版本为准, 本讲不代为总结具体条款内容以避免过时。
- 也见本仓库`data-curation`专题`01-data-overview.md`——那一讲讲数据集本身的技术特性和常见许可类型, 是本讲第1节③"数据集许可证覆盖范围"判断的技术背景, 本讲聚焦"许可证是否覆盖你的实际用途"这一合规判断, 不重复数据集技术细节。

---

## 5. 本讲小结(本专题收官)

- "公开可访问"和"可以随意这样用"是两回事, 供应商/API/数据集的使用许可需要在使用前而非事后核查。
- 三类常见合规缺口: 数据处理协议(DPA, 尤其涉及PII时)、模型/API使用许可(能否训练衍生模型/能否发表benchmark)、数据集许可证覆盖范围(尤其"仅限非商业研究"这类限定)。
- 合规核查应该在立项/资源规划阶段就完成, 而不是等论文写完或产品准备上线才想起来。

> **本专题(research-funding-and-resources)收官**: L1经费申请全流程 → L2算力资源规划 → L3数据管理规划 → L4多机构合作管理 → L5供应商/API合规评估, 完整走完了一个研究项目在"钱和资源"这个维度上、从申请到执行、从内部资源到外部第三方关系需要覆盖的全部实务环节。用`funding_plan_audit.py`的`audit()`和`reviewer_focus()`可以随时自检一份经费/资源计划是否具备这五块骨架, 而不是只有一句"需要更多算力"。

**动手**: 打开 `templates/funding-plan-worksheet.md` 的「第三方/供应商合规审查」小节, 对你手头(或设想中)用到的一个API或数据集, 按第2节的核查清单逐条核实, 写出结论和依据。
