# 04 · 评估指标（面试高频度：中高）

评估指标这一类在面试里很少单独占一整轮，但几乎每一轮都会被"顺手"问到——你汇报一个模型的效果时说"准确率95%"，面试官大概率会立刻追问"正负样本比例是多少"；你说"用了BLEU评测生成质量"，对方立刻会问"BLEU对同义改写敏感吗"。换句话说，这类问题考的不是背公式，而是**你有没有具备"选对指标"的判断力**——尤其是国内大厂的算法岗/推荐/风控方向，以及任何需要处理不平衡数据(欺诈、点击率、医疗)的岗位，这是几乎必问的送分题也是送命题：说错场景会让对方怀疑你有没有真的上线过模型。对 NLP/LLM 方向而言，perplexity、BLEU/ROUGE 是绕不开的基本功。

## 追问链深挖 1：precision/recall 到底该看哪个、不平衡时怎么办

**Q1（基础问题）**：precision 和 recall 的区别是什么？

**A1**：precision(查准率) = TP/(TP+FP)，衡量"我说是正例的里面，有多少真是正例"，代表预测的可信度；recall(查全率) = TP/(TP+FN)，衡量"真正的正例里，我找到了多少"，代表覆盖程度。二者通常此消彼长——放宽判正的阈值能找到更多正例(recall升)，但混进来的误报也会变多(precision降)。

**面试官追问 Q2**：如果只让你汇报一个数字总结模型好坏，你会怎么选？

**A2**：单独看 accuracy 在不平衡数据下会失真，所以优先考虑 F1 = 2·precision·recall/(precision+recall)——用调和平均而不是算术平均，是因为调和平均对较小值格外敏感，只要 precision 或 recall 有一个塌陷，F1 就会被明显拉低，逼着模型不能靠"精度和召回一高一低"的偏科策略刷分。但要提醒一句：F1 本身是在**某个固定阈值**下算出来的，如果业务场景阈值可以调整，只看单点 F1 会掩盖模型在其他阈值下的表现。

**面试官再追问 Q3**：如果正负样本比例是 1:1000（比如欺诈检测），F1 还够用吗？该看什么？

**A3**：这种极端不平衡下，我会转向 PR-AUC（precision-recall 曲线下面积）而不是 ROC-AUC。原因是 ROC-AUC 依赖假正例率 FPR = FP/(FP+TN)，分母里的 TN 在正例稀少时天然巨大，FPR 会显得很低、ROC 曲线因此"虚高"，掩盖模型在少数正类上的真实表现；PR-AUC 只关注 precision 和 recall，不被大量真负例稀释，更能反映模型在检测这 0.1% 正例上的真实能力。同时我还会看具体阈值下的 precision@recall=X 这种业务对齐的切片指标，而不是一个笼统的曲线下面积了事。

## 追问链深挖 2：模型校准（calibration）——不只是分对，还要"分得心里有数"

**Q1（基础问题）**：什么是模型校准(calibration)？

**A1**：校准衡量的是模型输出的概率和真实发生频率是否一致——如果模型说一组样本是正类的概率是 0.8，这组样本里真实为正的比例应该也接近 80%。常用可靠性图(reliability diagram)可视化，或用 ECE(期望校准误差：把预测按置信度分桶，算每桶预测概率与实际频率之差的加权平均)给出一个数字。现代深度网络(尤其是层数深、用了强正则/温度低的模型)普遍存在**过度自信**的问题，输出的 0.99 未必真的对应 99% 的正确率。

**面试官追问 Q2**：校准差的模型在实际业务里有什么危害？光看 accuracy/AUC 看不出来吗？

**A2**：accuracy/AUC 只关心"排序对不对"，一个模型即使把所有样本的概率都压到 0.9 附近，只要相对排序正确，AUC 依然很高，但这时候概率数值本身已经没有意义了。危害体现在下游**依赖概率数值做决策**的场景：比如风控里用预测概率乘以金额算期望损失去做阈值分级、推荐系统里用点击率预测值去做 ECPM 排序和竞价、RLHF 奖励模型的分数被当作绝对信号去加权采样——这些场景如果模型过度自信或校准漂移，即使排序对，下游的期望值计算和阈值切分也会系统性出错。

**面试官再追问 Q3**：怎么修？temperature scaling 具体怎么做，有什么局限？

**A3**：temperature scaling 是最常用也最简单的后校准(post-hoc calibration)方法：在验证集上学习一个标量温度 T，把模型输出的 logits 除以 T 再做 softmax，T>1 会让分布变得更平滑(降低过度自信)。因为除以同一个标量不改变各类别 logits 之间的相对大小关系，所以它**不改变模型的排序**，也就不影响 accuracy/AUC，只是重新拉伸了概率的绝对刻度。局限在于：它只能做全局的、单一维度的缩放，如果模型在不同子群体/不同置信度区间上的校准偏差方向不一致（比如高置信度区间过度自信、低置信度区间又欠自信），一个标量 T 无法同时修正两头；这时需要更精细的方法（如 Platt scaling 按类别分别拟合、isotonic regression 做非参数校准），并且校准是在验证集分布上学的，一旦部署后出现分布漂移，校准效果也会跟着失效，需要定期重新校准。

## 其余题目速览

完整标准答案见 `src/ai_qa/qbank_metrics.py`（id 前缀 `ai-met-`）。

- **混淆矩阵**：TP/FP/FN/TN 是所有分类指标的地基，先画表再推公式，别死记。
- **accuracy 何时误导**：类别极不平衡时，全预测多数类也能刷高 accuracy，必须看 F1/PR-AUC。
- **F1 score**：precision 和 recall 的调和平均，惩罚"偏科"策略。
- **ROC-AUC vs PR-AUC**：均衡数据看 ROC-AUC，正例稀少(欺诈/检索)看 PR-AUC。
- **困惑度 perplexity**：= exp(平均交叉熵)，语言模型对每 token 的平均"犹豫程度"。
- **BLEU**：n-gram precision + brevity penalty，机器翻译常用，对同义改写不敏感。
- **ROUGE**：n-gram/LCS 的 recall 导向指标，摘要生成常用，和 BLEU 的 precision 导向互补。
- **macro-F1 vs micro-F1**：macro 平权关注小类别，micro 按样本数加权、被大类别主导。
- **决策阈值选择**：不是拍脑袋用 0.5，要结合 precision/recall 的业务代价挑阈值。

## 常见误区清单

1. **把 accuracy 当万能指标**——不平衡数据下第一反应应该是先问"正负比例是多少"。
2. **F1 只是一个阈值点的快照**——阈值可调的场景下，光报 F1 会漏掉模型在其他工作点的表现，应该配合 PR 曲线一起看。
3. **把 ROC-AUC 和 PR-AUC 混用**——极度不平衡数据下 ROC-AUC 会"虚高"，这是最容易被追问抓住的点。
4. **校准和判别能力(discrimination)是两件事**——AUC 高不代表概率数值可信，二者要分开评估，别用一个指标替代另一个。
5. **BLEU/ROUGE 当成绝对真理**——它们是表面词形匹配的代理指标，对语义等价但用词不同的输出会打低分，报告时要说明这一局限。
