# be-02 · 计算机网络（面试高频度：几乎每家都问的门槛项）

计算机网络和操作系统一起构成了国内大厂后端面试里最基础、覆盖面最广的两大门槛类别——TCP三次握手/四次挥手、拥塞控制、HTTP版本区别这几道题几乎是"标配必问"，很多面试官甚至会把它当作面试的第一道热身题，用来快速判断候选人的基础是否扎实。这个类别的特点是"细节极其容易记错"（比如握手挥手的次数、拥塞控制各阶段cwnd的具体变化），一旦被追问到具体数值或者具体状态名答不上来，很容易让面试官怀疑你是不是真的懂、还是只背了一个大概轮廓。本讲义先深挖两条经典追问链：TCP三次握手（一路问到SYN洪泛攻击）、TCP四次挥手（一路问到TIME_WAIT为什么要等2MSL），再速览其余题目，最后列出最容易失分的误区清单。

## 追问链深挖 1：TCP三次握手——从流程问到攻击面

**Q1（基础问题）**：TCP三次握手的完整过程是怎样的？

**A1**：（1）客户端发送SYN报文（seq=x），进入SYN_SENT状态；（2）服务器收到后回复SYN+ACK报文（seq=y，ack=x+1），进入SYN_RCVD状态，这个报文同时确认客户端的SYN、并带上自己的SYN；（3）客户端收到SYN+ACK后回复ACK报文（ack=y+1），自己进入ESTABLISHED状态，服务器收到这个ACK后也进入ESTABLISHED状态，连接正式建立。三次握手本质上完成了一次双向确认：客户端确认"我能发、服务器能收"，服务器也确认"我能发、客户端能收"。

**面试官追问Q2**：为什么不能用两次握手就完成连接建立？

**A2**：两次握手只能让服务器确认"我收到了客户端的包"，却没法让客户端确认"我发的确认包，服务器真的收到了"——可靠性只被单向验证了。经典反例是"已失效的连接请求"：客户端第一个SYN因网络延迟滞留很久，客户端超时后重发SYN、正常握手通信、然后关闭连接；这之后如果那个迟到的旧SYN终于到达服务器，在两次握手的设计下服务器会立刻回复确认并直接建立连接，误以为这是一个新请求——但客户端根本没有再次发起连接，不会理会这个意外确认，服务器就会一直等待一个永远不会到来的数据，白白浪费资源。三次握手把"连接到底建不建立"的最终确认权交给客户端，能够识别并丢弃这种已失效的连接请求。

**面试官再追问Q3**：既然握手机制要处理这种"被恶意利用"的边界情况，那具体的攻击方式SYN洪泛（SYN Flood）是怎么打的？怎么防？

**A3**：攻击者伪造大量不同的源IP向服务器发送SYN，服务器每收到一个都要回复SYN+ACK并进入SYN_RCVD状态，在半连接队列里占一个位置等待第三次ACK——但源IP是伪造的，真实主机不会回应，服务器只能等超时重传，期间大量伪造请求迅速占满半连接队列、耗尽内存和文件描述符资源，导致正常用户的SYN被丢弃，这是一种资源耗尽型的拒绝服务攻击。最经典的防御手段是SYN Cookies：服务器收到SYN后不立即分配资源、也不放入半连接队列，而是用双方IP、端口、时间戳等信息算出一个特殊的初始序列号（cookie）放进SYN+ACK返回，只有收到携带正确cookie的第三次ACK才真正分配连接资源——即使遭受海量伪造SYN攻击，服务器也没有为这些半连接保存任何状态，自然不会被打垮。工程上还会配合调大半连接队列、缩短SYN_RCVD超时时间、以及在网络边界用防火墙/流量清洗设备过滤异常流量。

## 追问链深挖 2：TCP四次挥手——从流程问到TIME_WAIT

**Q1（基础问题）**：TCP四次挥手的完整过程是怎样的？

**A1**：（1）主动关闭方发送FIN（seq=u），进入FIN_WAIT_1状态；（2）被动关闭方收到后先回ACK（ack=u+1）确认，进入CLOSE_WAIT状态，主动方收到ACK后进入FIN_WAIT_2状态；（3）被动方数据发送完毕后再发自己的FIN（seq=v），进入LAST_ACK状态；（4）主动方收到这个FIN后回ACK（ack=v+1），进入TIME_WAIT状态，等2MSL后才真正关闭，被动方收到最后这个ACK后直接关闭。

**面试官追问Q2**：为什么挥手要四次，握手却只要三次？

**A2**：握手时，服务器收到SYN后要同时做"确认收到SYN"和"发出自己的SYN"这两件没有先后依赖的事，可以合并成一个SYN+ACK报文一次发出，三次就够。但挥手时，被动关闭方收到FIN后虽然能立刻ACK确认，但自己这一侧往往还有数据没发完，没法在同一时刻就发出自己的FIN——ACK和FIN这两个动作出现了时间差，没法像握手那样合并，只能拆成"先ACK确认收到对方FIN"和"自己数据发完后再单独发FIN"两个独立步骤，一来一回就比握手多了一次，变成四次。

**面试官再追问Q3**：主动关闭方发完最后一个ACK之后为什么不直接关闭，非要进入TIME_WAIT等2MSL？

**A3**：两个原因。第一，保证最后一个ACK能可靠到达对方：如果这个ACK在网络中丢失，处于LAST_ACK状态的被动关闭方会超时重传FIN，主动方如果还在TIME_WAIT就能收到这个重传的FIN并重新发出ACK；如果主动方已经直接关闭了，收到这个"迟到"的FIN时早已没有对应连接状态，只能用RST去响应，导致被动方无法正常关闭。第二，防止旧连接的滞留报文污染新连接：2MSL（一个报文段在网络中可能存活的最长时间的两倍）足以保证这次连接残留在网络中的所有报文段都已消失或过期作废，这样之后用同样的IP和端口这个四元组建立新连接时，不会把上一次连接残留的报文误当成新连接的数据。副作用是如果服务器主动发起大量短连接的关闭，会在服务器上堆积大量TIME_WAIT状态占用端口和内存，实践中常通过SO_REUSEADDR、调整内核参数、或者让客户端而不是服务器主动关闭来缓解。

## 其余题目速览

完整标准答案见 `src/backend_qa/qbank_network.py`（id 前缀 `be-net-`）。

- **TCP拥塞控制：慢启动+拥塞避免**：cwnd从小值指数增长到ssthresh后转为线性增长（每RTT加1个MSS），超时丢包则ssthresh减半、cwnd打回1，这是AIMD思想。
- **快重传和快恢复**：收到3个重复ACK就立即重传，不必等超时；cwnd只减半（设为ssthresh+3）而不是打回1，恢复速度比超时重传快得多。
- **滑动窗口与拥塞窗口**：滑动窗口（rwnd）做流量控制、防止压垮接收方；cwnd做拥塞控制、防止压垮网络；实际发送量取min(rwnd,cwnd)。
- **TCP和UDP的区别**：TCP面向连接、可靠、有拥塞控制，适合网页/文件传输；UDP无连接、不可靠、无拥塞控制，适合DNS/音视频/实时游戏。
- **HTTP和HTTPS的区别**：HTTPS=HTTP+TLS，握手阶段用非对称加密协商出对称密钥，正式传输用对称加密，兼顾安全与性能。
- **HTTP/1.1、HTTP/2、HTTP/3核心区别**：HTTP/2用多路复用解决应用层队头阻塞，但仍受TCP层队头阻塞拖累；HTTP/3换成QUIC（基于UDP），每个stream独立，彻底解决传输层队头阻塞。
- **DNS解析过程**：浏览器缓存->系统缓存->本地DNS（递归查询）->根域名服务器->顶级域服务器->权威域名服务器（迭代查询）逐级拿到IP。
- **负载均衡算法**：轮询实现简单但不看负载，最少连接动态反映实时负载，一致性哈希让节点增减时只有约1/N的请求需要重新分布，配合虚拟节点让负载更均匀。
- **CDN原理**：把内容缓存到离用户最近的边缘节点，靠DNS调度就近分流，命中缓存就不用回源，降低延迟、减轻源站压力。
- **Cookie和Session的区别**：Cookie存在客户端、Session存在服务端，分布式部署下Session需要额外做会话共享（比如统一存Redis）。

## 常见误区清单

1. **"HTTP/2彻底解决了队头阻塞"——不准确。** HTTP/2只解决了应用层的队头阻塞，因为底层仍然是单个TCP连接，一旦丢包，TCP的严格有序交付会把所有多路复用的流一起卡住，这是HTTP/3用QUIC要解决的传输层队头阻塞，两层队头阻塞不能混为一谈。
2. **把"拥塞窗口cwnd"和"滑动窗口/接收窗口rwnd"当成同一个东西**——前者是发送方对网络状况的自我限速（拥塞控制），后者是接收方对自己处理能力的限速（流量控制），实际发送量是两者的较小值，考点经常在于分清楚"谁在限速、为了防止压垮谁"。
3. **"三次握手是为了防止SYN洪泛攻击"这个因果关系说反了**——三次握手的设计目的是防止"已失效的连接请求"被服务器误认为有效连接，SYN Flood是利用了握手过程中服务器必须为半连接分配资源等待第三次ACK这一机制上的漏洞进行的攻击，不是握手本身用来"防"这个攻击的。
4. **快重传触发条件记错**——是收到3个重复ACK（也就是总共4个相同的ACK），不是1个或2个；混淆这个数字在追问时很容易露怯。
5. **认为TIME_WAIT状态是"浪费"、应该尽量消除**——TIME_WAIT是保证连接可靠关闭、防止旧连接数据污染新连接的必要设计，工程上要优化的是"减少不必要的主动关闭方"（比如让客户端而不是服务器主动关闭），而不是消灭TIME_WAIT本身。
6. **把"负载均衡的一致性哈希"和"普通哈希取模"的效果搞混**——普通取模hash(key)%N一旦N变化几乎所有映射全部失效，一致性哈希增删节点时只有环上相邻一小段（约1/N）的请求需要重新分布，这是它被广泛用在分布式缓存/负载均衡里的核心原因。
7. **"UDP不可靠所以不能做可靠传输"是过度引申**——UDP本身不提供可靠性保证，但完全可以在应用层基于UDP自己实现确认重传、拥塞控制等机制，QUIC（HTTP/3底层协议）就是这么做的，不代表UDP不能承载可靠的应用。
