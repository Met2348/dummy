# 06 · Tokenizer 与数据（面试高频度：中，NLP/LLM 方向尤其高）

Tokenizer 和预训练数据这一类，在通用算法岗面试里出现频率不算最高，但只要面试官知道你是 NLP/LLM 方向出身（简历上写了预训练/微调相关经历），这几乎是必问模块——因为它是"你是不是真的碰过大模型训练全流程"的试金石：会用 `transformers.AutoTokenizer` 不代表懂 BPE/WordPiece/SentencePiece 的原理区别，会调 `datasets.load_dataset` 不代表理解数据配比和去重为什么重要。字节跳动、月之暗面这类自研基座模型的公司，以及任何提到"预训练""数据工程"岗位方向的面试，这类问题会被问得很细；而如果你的目标是纯研究岗，这类问题也是"你懂不懂训练数据如何决定模型能力上限"这个大问题在面试中的具体切片。

## 追问链深挖 1：BPE -> WordPiece -> SentencePiece，一路问下去的分词算法谱系

**Q1（基础问题）**：BPE(Byte Pair Encoding) 训练阶段具体是怎么构造词表的？

**A1**：先把语料切成最基本的符号单元（字符或字节），词表初始就是这些基本符号；然后统计语料里所有相邻符号对(pair)的出现频率，把频率最高的那一对合并成一个新符号、加入词表；重复"统计频率 -> 合并最高频 pair"这个过程若干轮，每轮词表新增一个 token，直到达到预设的词表大小。因为合并顺序严格按频率从高到低进行，最终词表会自然保留高频完整词，把低频词拆成更细的子词组合；编码新句子时，就按训练时记录下来的合并规则顺序重新执行一遍。

**面试官追问 Q2**：WordPiece 也是子词合并算法，它和 BPE 的核心区别是什么？

**A2**：区别在于"选哪一对来合并"的打分标准。BPE 是纯计数——谁频率高就合并谁；WordPiece 选择的是"合并后能让训练语料的似然提升最多"的那一对，具体用一个类似互信息的公式：score(pair) ≈ freq(pair)/(freq(first)·freq(second))，衡量这两个符号共同出现的频率相对于它们各自独立出现频率的"意外程度"。这意味着两个本身都极高频、但共现纯属巧合的符号不会被优先合并；而像 "play" 和 "##ing" 这种共现频率明显高于各自独立频率之积的组合，即便单独看频率不是全局最高，也会被优先合并——这让 WordPiece 切出来的子词边界更贴近有意义的词根/词缀，对生僻词的处理也更保守。BERT 用的就是 WordPiece。

**面试官再追问 Q3**：BPE 和 WordPiece 都要求先做预分词(pre-tokenization)，那 SentencePiece 解决了什么它们都没解决的问题？

**A3**：BPE、WordPiece 传统实现都假设文本已经先用空格/标点切成"词"，再在词内部做子词合并——这对英语这类空格分词语言没问题，但中文、日文这类没有天然空格边界的语言就很尴尬，需要先接一个语言相关的分词器。SentencePiece 直接把原始文本(包括空格本身)当作一个统一的字符流喂给 BPE 或 Unigram 算法，用一个特殊符号 ▁(U+2581) 显式表示空格、把它当成普通字符一起参与合并/建模，这样整个流程完全语言无关、不需要任何预分词步骤，解码时把子词拼接、再把 ▁ 还原成空格即可无损还原原文。这也是为什么 T5、LLaMA 这些需要处理多语言或不依赖空格分词的语言的模型普遍选择 SentencePiece 作为底层框架（可以在其中选择跑 BPE 还是 Unigram 算法）。

## 追问链深挖 2：数据去重与记忆——"重复"是怎么一步步伤害模型的

**Q1（基础问题）**：预训练数据去重(deduplication)为什么重要？

**A1**：预训练语料里常常有大量重复或近似重复的文档——转载、镜像站、模板化网页、爬虫重复抓取。如果不去重，模型会在同一段文本上被反复训练很多遍，浪费算力和 token 预算，同时增加模型死记硬背这些片段而不是学习可泛化模式的风险。因此数据去重(exact dedup 精确去重 + fuzzy dedup 近似去重)已经是 FineWeb-Edu、Dolma、RedPajama 这类现代预训练 pipeline 的标配步骤。

**面试官追问 Q2**：具体是怎么"伤害"模型的？有没有实证研究？

**A2**：Anthropic 的一项研究(Scaling Laws and Interpretability of Learning from Repeated Data)发现，哪怕只有很小一部分数据(比如 0.1%)被重复了上百次，也会显著拖累模型效果，甚至出现类似 double descent 的非单调性能下降——性能会先随训练变差、越过某个临界点后才重新变好。机制上的解释是：模型会把大量容量用来记忆(memorize)这些反复出现的片段，而这部分被"占用"的容量原本可以用来学习更通用的模式；后续的可解释性研究还发现，这种记忆行为会挤占 induction head 等对泛化很关键的内部结构，相当于用死记硬背替换掉了原本用于举一反三的电路。

**面试官再追问 Q3**：那实践中怎么做？光靠去重就够了吗？

**A3**：光去重不够，去重要和数据质量过滤配合做。去重解决的是"同一段内容出现了几次"，质量过滤解决的是"这段内容本身值不值得学"——常见手段包括基于规则的启发式过滤(去掉过短/过长文档、异常符号比例、语言检测)、用小分类器给文档质量打分(比如判断是否类似维基百科/教科书式的高质量文本，FineWeb-Edu 就是这么构造的)，以及对已知低质量/有害域名的黑名单过滤。这两件事叠加起来，往往比单纯堆数据规模对模型能力的提升更明显——这也是为什么"预训练数据工程"本身正在变成一个和"模型架构设计"同等重要的研究方向。

## 其余题目速览

完整标准答案见 `src/ai_qa/qbank_tokenizer_data.py`（id 前缀 `ai-tok-`）。

- **为什么用 subword**：word-level 有 OOV 问题，char-level 序列太长语义太碎，subword 是二者的折中。
- **BPE 编码(推理)阶段**：严格按训练时记录的合并规则顺序重新执行一遍，保证切词方式和词表构造一致。
- **Unigram 语言模型分词法**：和 BPE/WordPiece 方向相反，自顶向下从大词表裁剪到目标大小，支持 subword regularization。
- **byte-level BPE**：在 UTF-8 字节(256 个基础符号)上做 BPE，从根本上杜绝 UNK，GPT 系列采用。
- **词表大小的权衡**：词表越大序列越短但 embedding/输出层参数越多，是序列长度和参数量之间的权衡。
- **预训练数据配比**：不同来源(代码/书籍/网页)对能力的贡献不对称，配比选不好会拖累特定能力或整体质量。

## 常见误区清单

1. **"BPE 和 WordPiece 都是按频率合并"——错。** WordPiece 用的是似然/互信息式打分，不是纯频率，这是二者被追问时最容易答错的点。
2. **"SentencePiece 是一种和 BPE 平行的新算法"——不准确。** 它是一个可以内部跑 BPE 或 Unigram 的框架/工程实现，核心创新是不依赖预分词，而不是发明了新的合并算法。
3. **把"数据去重"和"数据质量过滤"当成一回事**——去重管的是"重复了几次"，质量过滤管的是"内容本身好不好"，二者需要配合而不是互相替代。
4. **"重复数据只是浪费算力，不会真的伤害效果"——错。** 有实证的 double descent 式性能下降，且这个副作用在小比例、高重复次数的数据上反而更明显。
5. **词表越大越好的直觉**——大词表能缩短序列，但会让 embedding 和输出层参数量线性增长，对小模型不成比例地不划算，需要按模型规模权衡。
