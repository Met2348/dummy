# 09 · MoE 架构

## 这类题在面试里的分量

MoE 是 2024 年之后大模型架构面试的高频新增考点——Mixtral、DeepSeek-V2/V3、Qwen-MoE 等主力开源/商用模型陆续采用 MoE，几乎所有"大模型架构""LLM infra""预训练"相关岗位都可能问到"MoE 相比稠密模型的核心权衡是什么""路由怎么保证不塌缩"。这类题的陷阱在于容易和"分布式训练里怎么把专家切到不同 GPU(专家并行/all-to-all 通信)"混为一谈——本讲义和对应的 `qbank_moe.py` 只聚焦 MoE **架构本身**的路由数学和负载均衡机制，分布式通信部分留给"分布式训练"类，面试时也要注意分清面试官问的到底是架构层还是系统层的问题。

## 深挖追问链：负载均衡机制

**Q1（面试官）**：MoE 训练里为什么容易出现少数专家垄断路由的问题？

**A1**：根源是一种自我强化的正反馈：训练初期哪怕只是随机波动让某几个专家被略多地选中，这几个专家就会拿到更多梯度更新、参数学得更快更好，门控网络下一轮就更倾向于把 token 也路由给这些"看起来更强"的专家，进一步挤压其它专家获得训练信号的机会。放任不管，多数专家会逐渐"饿死"——几乎没有 token 路由过去、参数几乎不更新，模型的有效容量退化成只用了一小部分专家的效果，白白浪费了大部分参数量。这就是"路由坍缩"。

**面试官追问 Q2**：那怎么解决？负载均衡辅助损失具体是怎么算的？

**A2**：主流做法（Switch Transformer/GShard 式）是给主任务损失加一项负载均衡辅助损失 L_aux=N·Σ_i f_i·P_i，N 是专家数，f_i 是实际被路由到专家 i 的 token 比例（基于 top-k 的离散硬计数），P_i 是门控网络对专家 i 的路由概率在整个 batch 上的平均值（连续可微的 softmax 输出）。因为 f_i 本身基于离散选择、不可微，没法直接反传梯度；引入连续可微的 P_i 和它相乘，就把"让各专家分到的 token 比例、以及门控给各专家的平均置信度都尽量均衡"这个目标变成一个可以反向传播优化的损失项。完全均衡时 f_i=P_i=k/N 与 1/N，L_aux 取得理论最小值 k（和专家数 N 无关，只和每个 token 激活几个专家的 k 有关）。

**面试官再追问 Q3**：辅助损失是个"软"的引导项，如果它没完全生效、训练中还是出现了瞬时的不均衡，会有什么后果？有没有硬性的保护机制？

**A3**：有，就是专家容量(expert capacity)。辅助损失只是通过梯度慢慢把路由训练得更均衡，本质是"软引导"，不保证任何一个 batch 里绝对均衡；专家容量是"硬兜底"——给每个专家在一个 batch 里设一条硬上限 capacity=capacity_factor×(batch内token数/专家数)，capacity_factor 通常取 1.25~1.5。不管辅助损失有没有起作用，都用这个固定的计算预算把每个专家单批次最多处理的 token 数量硬性封顶，超出容量的 token 直接被丢弃——不经过任何专家计算，通过残差连接直接跳过 FFN 部分往下传，相当于该层对这些 token 只做了恒等变换。这两道机制一软一硬配合：辅助损失负责"长期把路由训练均衡"，专家容量负责"保证任何时刻计算图/显存的形状可预测、不会被某个专家瞬时挤爆"。

## 其余题目一览

完整答案见 `src/ai_qa/qbank_moe.py`（`ai-moe-01/02/03/08/09/10`，除上面深挖用到的 `ai-moe-04/05/06/07`）：

- **MoE 核心思想**（`ai-moe-01`）：条件计算——总参数量(容量)和推理时激活参数量(算力)解耦，是 MoE 能"变大又不变慢"的根本原因。
- **门控路由怎么算**（`ai-moe-02`）：token 隐藏向量过线性层算 logit，softmax 出概率分布，选 top-k 个专家加权求和。
- **为什么大多选 top-2 不选 top-1**（`ai-moe-03`）：top-1 最省算力但容易"赢者通吃"训练不稳定，top-2 相当于对路由做了一点集成，性价比更高。
- **稀疏 MoE vs 稠密模型的成本对比**（`ai-moe-08`）：同总参数量比，MoE 算力/延迟更低但显存相当甚至更高；同激活参数量比，MoE 效果通常更好，代价是显存和工程复杂度。
- **为什么 MoE 比稠密模型更容易训练不稳定**（`ai-moe-09`）：多引入了一个离散路由决策变量，路由和专家参数相互影响、构成正反馈，稠密模型没有这个额外的不稳定来源。
- **共享专家(shared expert)机制**（`ai-moe-10`）：额外设置所有 token 必经的固定专家承担通用知识，让路由专家更专注学差异化知识，减少参数冗余，DeepSeekMoE 的关键改进点。

## 易错点 / 常见误区清单

1. **把负载均衡辅助损失和专家容量说成是同一件事**——一个是训练时加进 loss 里的"软"引导项（作用于梯度），一个是推理/训练时对 batch 形状的"硬"上限（不涉及梯度，直接丢 token），两者互补但机制完全不同，回答时要分开讲清楚各自解决什么问题。
2. **算错辅助损失的理论最小值**——很多人以为均衡时 L_aux 应该趋于 0，但正确的理论最小值是 k(每 token 激活的专家数)，因为 f_i≈k/N、P_i≈1/N，N·Σ(k/N·1/N)=k，不是 0。
3. **把"专家"理解成完全独立、互不相关的子模型**——实际上专家之间通过共享的门控网络、共同的负载均衡损失相互耦合，路由决策会同时影响多个专家的训练信号分配，不是简单的"独立并行训练几个小模型"。
4. **把 MoE 的分布式通信细节(all-to-all、专家并行组切分)和这里的路由/负载均衡数学混在一起讲**——面试官问"MoE 怎么保证负载均衡"时，答案应该是辅助损失+专家容量这套架构层机制，而不是"因为 GPU 之间做了 all-to-all 通信"，后者是另一个层面的系统工程问题。
5. **忽略被丢弃 token 的实际处理方式**——超容量的 token 不是"报错"或"随便丢弃丢失信息"，而是通过残差连接跳过该层 FFN、做恒等变换继续往下传，模型不会因此崩溃，只是该层对这些 token 的有效计算深度打了折扣。
