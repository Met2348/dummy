## 这块在国内大厂后端面试里的分量

Linux 基础很少作为面试的主战场被单独深挖，但几乎必然出现在**系统设计题的收尾追问**和**运维/SRE 方向的岗位追问**里——面试官问完"你这个服务怎么设计"之后，经常会接一句"如果上线之后 CPU 突然飙高/接口突然变慢，你怎么排查"，这类问题考的不是背命令参数，而是**有没有一套排查思路的顺序感**：先看什么指标、再用什么工具收窄范围、最后怎么定位到具体代码。背熟单条命令的用途只是地基，真正拉开差距的是能不能把多个命令串成一条完整的排查流程讲出来。

## 追问链深挖 1：CPU 占用高怎么排查（实战流程题）

**Q1（面试官）：线上一台机器 CPU 占用突然从 20% 涨到 95%，你怎么排查？**

**A1**：第一步先用 `top`（或 `htop`）看两件事：一是顶部 CPU 整体占用的构成，user 高说明是应用代码在算东西，system 高说明是内核态开销大（比如频繁系统调用），iowait 高说明其实是在等磁盘/网络 IO 而不是真的在算；二是下方按 CPU% 排序的进程列表，找到占用最高的那个进程的 PID。

**Q2（追问）：定位到进程之后呢？总不能只知道"是这个进程"就完事了吧？**

**A2**：确实不够，还要继续下钻到"进程内部是哪个线程、哪段代码"。如果是 Java 服务，用 `top -Hp <PID>` 能看到这个进程内部各个线程的 CPU 占用，找到占用最高的线程 TID（这是十进制的）；然后用 `printf "%x\n" <TID>` 把它转成十六进制，因为 Java 的线程 dump（`jstack <PID>`）里线程栈上打印的 `nid`（native thread id）是十六进制的，两者要对上才能在一大堆线程栈里找到占用 CPU 那个线程当前具体执行到了哪一行代码——常见结果是死循环、正则回溯爆炸、或者频繁 Full GC 导致 GC 线程占用高。如果不是 Java 进程，没有 jstack 这类工具，可以用 `perf top` 直接看当前系统级别的热点函数，或者用 `strace -p <PID>` 看这个进程在疯狂执行什么系统调用。

**Q3（追问）：如果 jstack 里发现是频繁 GC 导致的，和"业务代码死循环"导致的 CPU 高，处理思路一样吗？**

**A3**：不一样，这也是为什么"定位到具体代码"比"知道是这个进程"重要得多。如果是死循环或者不合理的重试逻辑，需要回去改业务代码本身的逻辑 bug；如果是频繁 Full GC，说明可能是内存分配速率太高、堆内存设置不合理、或者存在内存泄漏导致老年代持续增长触发频繁 GC，这时候需要结合内存排查的思路（见 `be-linux-10`）而不是死抠 CPU 数字；如果 CPU 高只是因为流量确实涨了、每个请求都在正常范围内工作，那可能压根不是 bug，而是需要扩容或者限流的容量问题。所以排查的终点不是"CPU 为什么高"，而是"CPU 高背后具体是哪一类问题"，这决定了后续的处理方式完全不同。

这条追问链完整对应 `qbank_linux_ops.py` 里的 `be-linux-09`，本质上是"工具链使用顺序 + 分类讨论根因"两件事的组合，建议练习时不要只背"用 top 和 jstack"，而是练成一口气讲出完整流程的能力。

## 追问链深挖 2：kill 信号、nohup、systemd 是怎么串起来的

**Q1（面试官）：`kill` 一个进程和 `kill -9` 一个进程有什么区别？**

**A1**：`kill` 命令本质是给进程发一个信号，不是"直接终止进程"这么简单粗暴的操作。不带参数的 `kill PID` 默认发送的是 `SIGTERM`（信号编号 15），这是一种"礼貌的"终止请求，目标进程可以捕获这个信号，先做清理工作（关闭打开的文件、释放连接、把内存里的状态落盘）再自己退出，也可以选择直接忽略它；`kill -9 PID` 发送的是 `SIGKILL`（信号编号 9），这是内核层面的强制终止，进程完全无法捕获也无法忽略，会被立即结束，不会有任何清理机会。

**Q2（追问）：那是不是应该一律直接用 `kill -9`，反正肯定能杀掉？**

**A2**：不建议。正常流程应该是先用 `kill`（`SIGTERM`）尝试优雅终止，给进程一个自行清理的机会，只有等了足够长时间（比如几秒到几十秒，Docker 停止容器的默认策略就是先发 SIGTERM、等一段时间后仍未退出再发 SIGKILL）进程还没退出，才用 `kill -9` 强制结束。直接一律用 `-9` 的风险是进程可能正在写文件、持有数据库连接或者维护某个中间状态，被强制杀死之后可能留下没清理干净的临时文件、没提交完的事务、或者没释放的资源句柄，排查起来会更麻烦。

**Q3（追问）：那 `SIGHUP` 和 `nohup` 又是什么关系？为什么有的服务收到 `SIGHUP` 不会退出反而会重新加载配置？**

**A3**：`SIGHUP`（信号编号 1，hang up）最初的含义是"控制终端断开连接"——以前你在一个终端里跑一个前台程序，关掉终端窗口，这个程序默认会收到 `SIGHUP` 并退出。`nohup`（no hang up）这个命令的作用就是让它启动的程序忽略这个信号，这样即使你退出 SSH 连接或者关掉终端，程序也能继续在后台跑，这也是为什么临时在服务器上跑一个长任务经常会用 `nohup ./run.sh &`。而很多现代守护进程（比如 nginx）把 `SIGHUP` 这个信号"借用"成了另一层含义：收到 `SIGHUP` 不退出，而是重新读取配置文件、平滑地应用新配置，不需要重启整个进程、不会丢失当前正在处理的连接。但这种"临时挂后台不被中断"的方式毕竟简陋——没有统一的日志管理、进程意外崩溃不会自动重启、机器重启后也不会自动把它拉起来，这些能力更规范的做法是交给 `systemd`：把服务写成一个 unit 文件，声明启动命令、重启策略（比如 `Restart=on-failure`）、依赖关系，用 `systemctl enable` 设置开机自启，日志统一由 `journald` 收集，通过 `journalctl` 查看，这也是为什么线上服务的标准做法是接入 `systemd` 而不是长期依赖 `nohup` 手动挂后台。

这条追问链完整对应 `qbank_linux_ops.py` 里的 `be-linux-06` → `be-linux-07` → `be-linux-08`，体现的是"一个基础概念（信号）如何被上层工具（nohup）复用、又如何被更完善的方案（systemd）取代"这条演进逻辑，面试官喜欢顺着这条线一路问下去看你是不是真的理解还是只背了三个名词。

- **grep**：按行按模式（含正则）过滤文本，命令行排查问题最常用的过滤工具。完整答案见 `qbank_linux_ops.py`（`be-linux-03`）。
- **awk**：在 grep 的"过滤"基础上，还能按字段做提取、运算、统计聚合。完整答案见 `qbank_linux_ops.py`（`be-linux-04`）。

- **内存泄漏定位**：用 `jstat -gcutil` 看老年代 Full GC 后是否降不下来，再用 `jmap` 导出堆转储配合 MAT 分析 GC Roots 引用链。完整答案见 `qbank_linux_ops.py`（`be-linux-10`）。
- **磁盘 IO 高排查**：`iostat -x` 看 %util/await 确认瓶颈，`iotop` 定位到具体进程，结合业务判断是不是不合理的 IO 使用方式或 swap 拖累。完整答案见 `qbank_linux_ops.py`（`be-linux-11`）。
- **网络问题排查**：netstat/ss 看本机监听和连接状态，telnet 测端口连通性，curl 测应用层响应，三者由浅入深组合排查。完整答案见 `qbank_linux_ops.py`（`be-linux-12`）。
- **日志排查思路**：先定位时间范围和关键字，分布式场景靠 traceID 串联跨机器调用链，优先看最早出现的异常而不是报错最多的地方。完整答案见 `qbank_linux_ops.py`（`be-linux-13`）。

## 易错点 / 常见误区

1. **把"CPU 占用高"和"load average 高"当成一回事**：load average 反映的是"正在运行 + 等待运行 + 处于不可中断睡眠（比如等磁盘 IO）"的进程数量，即使 CPU 使用率不高，如果大量进程卡在等磁盘 IO，load average 一样会很高，这种情况瓶颈其实在磁盘而不在 CPU。
2. **看到 `kill` 不生效就立刻上 `-9`**：跳过 `SIGTERM` 给进程清理机会这一步，容易导致资源没释放干净、数据没保存完整，正确顺序是"先温和、再强制"。
3. **以为 `nohup` 能替代 `systemd`**：`nohup` 只解决"终端断开导致进程被杀"这一个问题，进程崩溃不会自动重启、没有统一日志、开机不会自启，长期运行的服务不应该只靠 `nohup` 管理。
4. **只会背命令，说不出指标该看多高算异常**：比如 `iostat` 里 `%util` 到 100% 不一定就是真瓶颈（如果是并行度高的 SSD，`%util` 的含义和传统机械盘不完全一样），面试官更想听到"结合 `await` 等待时间、结合业务是否真的变慢"这种交叉验证的判断方式，而不是死记一个阈值。
5. **日志排查只搜"报错最多"的地方**：分布式系统里下游一个异常经常会在上游层层放大出一堆报错日志，报错数量最多的地方往往是"受害者"而不是"元凶"，应该优先顺着 traceID 找最早出现异常的那一环。
6. **把 CPU 高、内存泄漏、磁盘 IO 高三类问题的排查工具串错**：这三类问题第一步用的工具不同（`top`/`jstat`/`iostat`），混着用会导致排查方向一开始就跑偏，建议先明确"症状属于哪一类资源"，再选对应的工具链往下钻。
