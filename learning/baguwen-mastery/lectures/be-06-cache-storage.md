# be-06 · 缓存与存储

## 这一类在大厂后端面试里的分量

只要项目里用到 Redis（几乎所有后端项目都会用），"缓存穿透/击穿/雪崩"和"缓存与数据库一致性怎么保证"几乎是必问题，属于"用过 Redis 但没深想过"和"真正理解过 Redis 的人"之间分界线最清晰的一类问题。这类问题的特点是概念名字很像、容易被面试官拿来连环追问区分能力——答不清楚穿透和击穿的区别，基本可以断定面试官会追加"那你线上出过缓存问题吗，怎么处理的"这种更难的场景题。另外 Redis 持久化机制、淘汰策略、为什么快这几道，考的是"你是不是真的看过 Redis 的设计文档"，是简历写了"熟悉Redis"之后最容易被现场验证真伪的地方。

## 追问链深挖 1：缓存穿透 / 击穿 / 雪崩三连问

**Q1（面试官）：什么是缓存穿透？怎么解决？**

**A1**：缓存穿透指查询一个缓存和数据库里都不存在的数据（比如恶意用一个不存在的ID反复查询），由于缓存里没有这个key，请求每次都会穿透缓存直接打到数据库上。解决方案：一是缓存空值——即使数据库查出来是空，也在缓存里存一个短TTL的空结果，拦截后续对同一个不存在key的查询；二是布隆过滤器——提前把所有存在的key放进布隆过滤器，请求先过滤一遍，判断为一定不存在的直接拒绝，连缓存和数据库都不用查。

**追问 Q2：那缓存击穿呢？听起来跟穿透很像，区别是什么？**

**A2**：区别在于"数据存不存在"。缓存穿透查询的数据在数据库里本来就不存在；缓存击穿针对的是数据本身是存在的，只是某一个访问量极高的热点key在某一时刻恰好过期失效，导致大量并发请求同时穿过缓存打向数据库来重建这一个key的缓存，是个"单点"问题。解决方案也不同：一是用互斥锁，只让第一个发现key过期的请求去查数据库重建缓存，其余请求阻塞等待或短暂重试；二是让热点数据设置逻辑过期而不是物理TTL，缓存永不真正过期，由后台线程异步刷新。

**再追问 Q3：那缓存雪崩呢？三个概念放一起怎么快速分清？**

**A3**：缓存雪崩指大量不同的key在同一时间集中过期（或者Redis实例整体宕机），导致海量请求同一时间全部打向数据库——区别于击穿是"一个热点key失效"，雪崩是"大批量key同时失效"或"缓存整体不可用"这种更大范围的问题。解决方案是给过期时间加随机值把大量key的过期时间打散、给缓存做高可用部署（主从+哨兵/集群）、并在数据库前加限流熔断降级作为最后一道防线。三者快速区分口诀：**穿透看"数据存不存在"（不存在）；击穿看"范围"（单个热点key）；雪崩看"范围"（大批量key或缓存整体）**。三者共享的兜底思路都是"别让所有请求同时绕过缓存直接砸向数据库"，只是触发原因和防护点不同。

## 追问链深挖 2：Cache Aside 一致性——先更新数据库还是先删缓存

**Q1（面试官）：Cache Aside 模式下更新数据时，应该先更新数据库还是先删缓存？**

**A1**：标准做法是"先更新数据库，再删除缓存"，而且是删缓存，不是直接更新缓存。

**追问 Q2：那"先删缓存，再更新数据库"具体会出什么问题？能举一个真实的并发时序吗？**

**A2**：会导致缓存长期脏读。举个时序：T1 线程A先把缓存删了；T2 这时线程B发起一次读请求，发现缓存未命中，转去读数据库——但线程A此时还没来得及把新值写进数据库，所以线程B读到的是旧值；T3 线程B把这个旧值写回缓存；T4 线程A才完成对数据库的更新（写入新值）。结果是：数据库里是新值，缓存里却缓存着旧值，而且这个脏缓存会一直留在那里，直到下一次有写操作再次触发删除，才有机会被修复——这个不一致的时间窗口可能长达很久，是不可接受的。

**再追问 Q3：那"先更新数据库，再删缓存"就完全没问题了吗？这个方案要怎么进一步兜底？**

**A3**：不是完全没问题，只是问题窗口窄得多。理论上仍然存在一个概率很低的并发场景：读请求在数据库更新完成之前读到了旧值，然后在"删除缓存"这个动作执行之后才把这个旧值写回缓存，同样会导致短暂的脏缓存。但这个窗口需要"读请求恰好横跨在更新数据库和删除缓存这两步之间还没写回"，发生概率远低于"先删缓存"方案的窗口（后者是常规大小的时间窗口，前者需要读写操作精确地交错），工程上通常认为可以接受。想进一步兜底可以用：延迟双删（删一次缓存 -> 更新数据库 -> sleep 一小段时间 -> 再删一次缓存，把那个小概率窗口内可能写回的脏数据清掉）；或者订阅数据库的 binlog（如阿里的 canal），数据库变更后异步、可靠地触发一次删缓存操作，彻底把"删缓存"这个动作和业务代码的执行时序解耦。

## 其余题目点行（完整答案见 `src/backend_qa/qbank_cache_storage.py`）

- **be-cache-01** 数据结构总览：五大结构 String/Hash/List/Set/ZSet，ZSet 底层是哈希表+跳表的组合。
- **be-cache-02** String 场景：KV缓存/计数器(INCR)/分布式锁/session共享/位图统计。
- **be-cache-03** Hash 场景：对象字段聚合存储，省 key 元数据开销，支持局部字段更新（HINCRBY）。
- **be-cache-04** List 场景：轻量消息队列(LPUSH+BRPOP) + 时间线(LPUSH+LTRIM)。
- **be-cache-05** Set 场景：去重 + 社交关系运算(交/并/差) + 标签系统。
- **be-cache-06** ZSet 场景：排行榜/延迟队列/滑动窗口限流，底层跳表比平衡树实现更简单。
- **be-cache-07** RDB：全量二进制快照，恢复快但两次快照之间的数据修改可能丢失。
- **be-cache-08** AOF：写命令日志，appendfsync 三档策略(always/everysec/no)权衡性能与丢失窗口。
- **be-cache-09** 混合持久化：RDB 做 base + AOF 做增量，Redis 4.0+ 后的生产推荐配置。
- **be-cache-10** 淘汰策略：maxmemory-policy 下 LRU/LFU/random/ttl 共 8 种组合。
- **be-cache-11** LRU vs LFU：按最近访问时间 vs 按访问频率，Redis 用采样近似而非精确维护链表。
- **be-cache-16** Redis 为什么快：纯内存操作 + 单线程无锁竞争 + IO 多路复用(epoll)。
- **be-cache-17** 布隆过滤器：位数组 + K 个哈希函数，不会漏判但可能误判，且不支持删除元素。

## 易错点 / 常见误区清单

1. 混淆缓存穿透（数据不存在）和缓存击穿（热点key过期）——这两个词几乎是面试官考察"是不是真的理解而不是死记硬背"的标准探针，一定要能秒答区别。
2. 以为 RDB 和 AOF 只能二选一——生产环境的推荐做法是 Redis 4.0+ 的混合持久化，二者互补而非互斥。
3. 只知道 LRU，不知道它的弱点——一次大范围低价值的批量扫描会"污染"LRU链表，把真正的热点数据挤出去，这正是 LFU 存在的意义。
4. Cache Aside 模式下更新顺序讲反——"先删缓存再更新数据库"会导致长时间脏读，这是这道题最容易被问倒的地方，务必记牢"先更新数据库、再删缓存"这个顺序和背后原因。
5. 以为"先更新数据库再删缓存"就绝对安全——理论上仍有一个概率很低的并发窗口，进阶追问要能提到延迟双删或 canal 订阅 binlog 这类兜底手段。
6. 把布隆过滤器当成"精确判断是否存在"的工具——它只能保证"判断不存在"是百分百准确的（不会漏判），"判断存在"是有误判率的，而且不支持删除操作。
7. 以为 Redis 是纯单线程所以网络 IO 也慢——Redis 6.0+ 之后网络读写可以由多线程处理，只有执行命令的核心逻辑仍然是单线程。
8. 说 Redis 快只答"因为是内存数据库"——完整采分点应包含内存操作、单线程避免锁竞争与上下文切换、IO多路复用三个层面，只答一点容易被追问追穿。
