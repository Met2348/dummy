# ai-15 · 系统设计(AI 向)

## 这一类在真实面试里的分量

系统设计题在算法岗/应用研究岗面试里通常是压轴题，考的不是某个公式的推导，而是"你有没有在真实约束(延迟/算力/数据规模/线上线下一致性)下把一个 ML 系统落地过的直觉"。**推荐系统召回+排序架构、大模型在线推理服务设计**是出现频率最高的两个子话题，几乎是应用型 AI 岗(推荐/搜索/大模型平台)的必考题；**training-serving skew、模型监控/数据漂移、A/B 实验设计**则是"你有没有真正上线维护过模型"的试金石，容易被追问到细节；CV 目标检测/图像分割这两题因为你的方向是 NLP，通常只会被浅问"知道基本概念吗"，不会深挖，本篇也只按这个深度处理。

## 追问链深挖：3 道最有代表性的题

### 深挖 1：推荐系统召回+排序(`ai-sysd-01` + `ai-sysd-02`)

**Q1(面试官)：** 工业级推荐系统为什么普遍采用"召回+排序"两阶段架构，而不是直接对所有物品打分排序？

**A1：** 线上一个请求要从千万甚至上亿量级的物品库里选出几十个展示给用户，如果对每个物品都跑一遍复杂的排序模型，单次请求的算力和延迟都吃不消。于是拆成两阶段：召回先用代价很低的方法从全量物品里快速筛出几百到几千个候选(保证"好东西没漏掉"，高召回率优先)；排序再对这一小批候选跑更精细的模型精确打分(保证"排在前面的确实是用户想要的"，高精度优先)，用递减的候选规模换取可接受的整体延迟。

**追问 Q2：** 召回阶段常见的多路召回都有哪些具体方法？

**A2：** 常见的有多路并行召回后再合并去重：①协同过滤(基于用户或物品的历史行为相似度)；②向量召回——把用户和物品都编码成向量(双塔模型/embedding)，线上用近似最近邻(ANN，如 HNSW/Faiss)快速找到最相似的一批；③规则/统计类召回，如热门榜单、同类目召回、地理位置召回；④基于图的召回。多路召回各自覆盖不同信号来源，合并后统一送入排序阶段打分，兼顾准确性和多样性。

**追问 Q3：** 召回和排序各自的评价指标一般用什么？

**A3：** 召回阶段更看重"有没有把真正相关的物品捞进候选集"，常用离线指标是 recall@K(候选集里覆盖了多少真实相关物品)、hit rate(用户最终点击/购买的物品是否出现在候选集里)，因为候选量大、直接算 AUC 意义不大。排序阶段直接对齐业务目标，用 AUC/LogLoss 衡量 CTR 预估的区分度和校准程度，某些场景还会看 NDCG/MAP 这类衡量"排序位置质量"的指标。但离线指标涨了不代表线上一定有感——最终都要靠 A/B 实验验证线上 CTR/CVR/时长等业务指标是否真的提升。

### 深挖 2：大模型在线推理服务设计(`ai-sysd-06`)

**Q1：** 设计一个大模型在线推理服务，需要重点权衡哪些因素？

**A1：** 核心是延迟和吞吐的权衡：延迟关注首 token 延迟(TTFT)和逐 token 生成速度，吞吐关注单位时间能处理多少请求/token。常见手段：动态批处理(continuous batching)、KV cache 管理(如 PagedAttention)、模型量化/蒸馏/投机解码降低单步计算量、多副本+负载均衡做水平扩展。上线新模型还要通过 A/B 实验或影子流量验证效果，再灰度放量。

**追问 Q2：** continuous batching 具体怎么解决不同请求生成长度不一致的问题？

**A2：** 传统静态 batching 要求同一批请求一起开始、一起结束，如果批里有的请求只需要生成 10 个 token、有的需要生成 500 个，短请求会被拖到和最长请求同时结束，GPU 在等待期间做了大量无意义的 padding 计算，浪费吞吐。continuous batching(也叫 in-flight batching)不再以"批"为固定单位，而是在每个解码 step 粒度上动态调整批的组成：哪个请求生成结束了就立刻把它替换出去，腾出的槽位马上塞进新来的请求，让 GPU 在每一个时间步都尽量满载运行不同长度、不同生命周期的请求，既提高吞吐，又不让短请求被长请求拖慢排队时间。

**追问 Q3：** 灰度放量的流量比例一般怎么设计，出问题怎么快速回滚？

**A3：** 通常从很小的比例开始(比如 1% 真实流量，或先跑纯观察、不影响用户的影子流量)，对比新旧模型在延迟、报错率、核心业务指标上的差异，确认没有明显异常后再逐步放大(1% -> 5% -> 20% -> 50% -> 100%)，每一步设观察窗口和明确的"自动回滚阈值"(比如报错率或延迟 P99 超过某个界限就立刻回滚)。回滚要做到能一键切回旧版本——流量路由层面只是改一个开关/权重，而不是重新部署；旧版本在放量期间要持续热备而不是直接下线，这样出问题时才能真正做到"快速"回滚而不是临时抢救。

### 深挖 3：training-serving skew(`ai-sysd-09`)

**Q1：** 什么是 training-serving skew？为什么它经常是线上效果不如离线评估的元凶？

**A1：** training-serving skew 指模型训练时用的特征计算逻辑/数据管道和线上服务时不一致，导致同一个样本在两条链路里算出来的特征值有系统性差异。常见成因是训练用离线批处理管道、serving 用实时管道，两边代码各写一遍不同步。它是"线下 AUC 很好、线上效果却拉胯"的常见元凶，因为离线评估用的是训练那套(正确的)特征管道，掩盖了线上管道的问题。

**追问 Q2：** feature store 具体怎么保证训练和 serving 用的是同一套计算逻辑？

**A2：** feature store 的核心思路是把特征的计算逻辑(比如"过去 7 天点击次数"这类统计特征怎么算)只注册一份统一定义，离线批量计算(生成训练数据)和线上实时计算(serving)都调用同一份定义/同一套代码路径，而不是两个团队分别用两套语言各写一遍。架构上通常会有离线 store(存历史特征用于生成训练样本)和在线 store(存最新特征供低延迟查询)，但背后的特征转换逻辑共享同一份代码或配置，从根本上避免"两边逻辑不同步"这一类 skew 的来源。

**追问 Q3：** log-and-wait 这种方式有什么代价和局限？

**A3：** log-and-wait 是指直接把 serving 时用到的实时特征值原样记录下来，等真实标签到达后(标签通常有延迟，比如用户是否点击/转化)，把"serving 时的真实特征 + 后来的标签"拼成训练样本，这样训练用的特征天然和 serving 时一致，从根本上消除计算逻辑不一致这类 skew。代价是：①需要等待标签到达，训练数据生产链路有延迟，想快速用最新数据重新训练模型会受限；②需要额外的存储和关联机制把某次请求的特征记录和之后到达的标签对应起来，工程复杂度和存储成本上升；③如果 serving 时某个特征本身有 bug(比如被错误地写死成默认值)，这个 bug 会连同错误特征原样带进训练数据，让下一次训练"学会"这个错误模式，持续存在直到人工发现。

## 易错点 / 常见误区清单

1. **只说"召回快、排序准"，说不出为什么必须拆成两阶段**——要能落到"候选集规模 × 单次打分算力 = 总延迟"这个直觉，而不是背两个名词。
2. **把 A/B 实验说成"随便分两组看看数据"**——一定要提分桶要正交(避免多实验互相污染)、要看统计显著性(避免样本量不足瞎下结论)、要设护栏指标(避免为了涨核心指标损害了别的东西)。
3. **只监控最终业务指标，不监控特征分布**——业务指标衰退往往是最后才能看到的信号，滞后性强；真正成熟的监控体系要在特征分布/预测分布层面提前预警。
4. **把 training-serving skew 简单理解成"线上数据分布变了(concept drift)"**——skew 强调的是计算逻辑/管道不一致这个工程问题，即使数据分布完全没变，两套代码算出不同的特征值也会造成 skew，这是和 concept drift 的关键区别。
5. **设计大模型推理服务时只谈"上更多 GPU"，不谈调度/批处理层面的优化**——continuous batching、KV cache 管理这些软件层面的优化往往比单纯堆硬件更能体现你对推理系统的理解深度。
6. **讲 CV 检测/分割时试图讲得比 NLP 内容还深**——你的方向是 NLP，这两题面试官期待的是"基本概念清楚、术语用对"，讲太深反而显得准备重点跑偏，点到为止即可，把时间留给核心方向的问题。
