# 12 · Agent 与 RAG 与工具调用（2026 新趋势重点）

## 这一类在面试里的分量 + 2026 新趋势

这是本专栏里变化最快、也最能体现"你有没有跟上行业最新动向"的一类。**2026 年面试重心已经从纯理论转向 AI 应用落地**——几年前面试官可能满足于你会背 Transformer 结构、知道什么是 attention，但现在越来越多的面试（不管是研究岗还是工程岗）会直接问"你有没有做过 Agent/RAG 系统""知不知道 MCP""怎么设计一个可靠的工具调用"，因为这些已经是把大模型真正产品化、变成生产力工具的核心拼图，而不再是纯研究话题。尤其 MCP（Model Context Protocol）这个 2024 年底才提出的协议，到 2026 年已经被写进大量一线大厂的 JD 里，成了"是否了解 Agent 工程最新实践"的一个试金石问题——背不出 MCP 解决的是什么问题，很容易被面试官判定为"知识停留在一年多以前"。这一类的另一个特点是概念之间联系紧密：RAG 的检索质量问题 -> 为什么需要 rerank -> Agent 怎么用 ReAct/Plan-and-Execute 编排多步任务 -> 工具调用怎么设计 schema、怎么处理失败重试 -> MCP 怎么把这些工具统一接进来，是一整条能讲通的落地链路，而不是互相独立的知识点。

## 追问链深挖：MCP 协议到底是什么，解决了什么问题

**Q1（面试官）：你简历上提到用过 MCP，能讲讲 MCP 是什么、解决了什么问题吗？**

**A1**：MCP（Model Context Protocol，模型上下文协议）是 Anthropic 在 2024 年 11 月提出的一个开放协议，用来标准化 AI 应用（模型/Agent）和外部工具、数据源之间的连接方式。它要解决的核心问题是所谓的"M×N 集成问题"：在 MCP 之前，每个 AI 应用要接入每一种外部工具/数据源（文件系统、数据库、Git、内部知识库……）都得各自定制一套对接代码，M 个应用对接 N 种工具最多需要 M×N 套互不通用的实现。MCP 定义了统一的 client-server 协议（基于 JSON-RPC），工具那一方只需要按标准实现一次"MCP server"，任何遵循 MCP 的 AI 应用都能直接连接调用，把 M×N 变成了 M+N。

**Q2（追问）：这跟我们平时说的 function calling 有什么区别？MCP 是不是把 function calling 取代了？**

**A2**：不是取代关系，而是不同层面的问题。function calling 解决的是"模型和单次工具调用之间"的接口问题——模型怎么理解一个工具的 schema、怎么决定调用哪个工具、传什么参数，这个机制本身在 MCP 出现之前就已经存在，且 MCP 内部的"工具调用"这一层能力，实现上仍然依赖模型具备 function calling 的能力去决定调什么、传什么参数。MCP 解决的是更上一层的问题——"一个 Agent/应用怎么统一发现、连接、管理成百上千个不同来源的工具/数据源"，也就是**工具生态的集成和分发**问题，而不是"模型怎么调用某一个具体工具"这个问题。可以理解成：function calling 是"车怎么开"，MCP 是"车和加油站/充电桩之间用统一的接口标准，不用每个品牌的车配一套专属加油枪"。实际系统里，Agent 通过 MCP client 连接到某个 MCP server 之后，server 暴露出来的能力（工具列表）最终还是要走 function calling 那一套机制被模型理解和调用，两者是配合关系，不是替代关系。

**Q3（再追问）：现在有的团队接入了几十上百个 MCP server，会不会有什么新问题？**

**A3**：会，而且这是 2025-2026 年 MCP 生态发展起来之后才逐渐暴露的新挑战——**上下文效率问题**。如果一次性把所有已连接 MCP server 暴露出来的工具定义（每个工具的名字、参数 schema、描述文字）都塞进模型的 prompt 上下文，随着连接的 server 越来越多，光是工具定义本身就可能占用大量 token，还没开始真正对话就先"挤占"了上下文窗口，也会拖慢模型做决策的速度和准确性（可选项太多，模型更容易选错工具）。而且工具调用返回的原始结果（比如一次数据库查询返回几千行数据）如果也整个塞回上下文，同样会迅速消耗掉可用的 token 预算。Anthropic 工程团队后续提出的应对方向包括：按需加载工具定义（不是一次性把所有工具都摆出来，而是让模型先看到"有哪些类别的能力"，需要时再展开具体工具的详细 schema）、以及让 Agent 能在一个隔离的代码执行环境里直接处理工具返回的原始数据（比如写一段代码对查询结果做过滤/聚合），只把处理后的精简结果送回模型上下文，而不是把整个原始结果都灌进去。这也说明 MCP 本身仍在快速演进，面试时如果只停留在"MCP 解决了 M×N 集成问题"这一层，容易显得没有跟进它最新的工程挑战。

## 其它高频题速览（完整答案见 `src/ai_qa/qbank_agent_rag.py`）

- **RAG 基本流程** —— 检索(embedding+相似度)+生成，解决模型知识截止和幻觉两个硬伤（`ai-rag-01`）。
- **向量数据库/embedding 检索原理** —— 语义相似度转化为向量空间距离，配合 HNSW 等 ANN 索引在大规模语料下做到近似最优速度（`ai-rag-02`）。
- **为什么需要 rerank** —— 向量召回（bi-encoder，宽召回低成本）之后用 cross-encoder 精排，弥补双塔结构缺乏查询-文档交互信息的短板（`ai-rag-03`）。
- **function calling schema 设计要点** —— 命名清晰无歧义、类型/必填精确定义、能用 enum 就不用自由文本、说明副作用边界（`ai-rag-04`）。
- **工具调用失败重试** —— 把失败转成结构化反馈（具体哪个字段/哪个环节），配合重试上限和"可重试 vs 不可重试"的区分（`ai-rag-06`）。
- **ReAct vs Plan-and-Execute** —— 走一步看一步的灵活循环，vs 先全局规划再逐步执行+按需 re-plan，实践常组合使用（`ai-rag-07`）。
- **RAG 失效模式：不相关检索 + lost in the middle** —— 检索错了是一类问题，检索对了但关键信息在长上下文中间位置被"忽视"是另一类问题（`ai-rag-08`）。
- **chunking 策略** —— 切太碎丢上下文，切太粗混主题，语义边界切分+overlap 是常见折中（`ai-rag-09`）。
- **Agent 记忆机制** —— 短期记忆=上下文窗口本身，长期记忆=持久化+类 RAG 检索注入，写入策略是核心权衡（`ai-rag-10`）。
- **多智能体协作** —— 分工聚焦+并行+交叉审阅换来协调开销和沟通复杂度的代价，不是所有任务都值得引入（`ai-rag-11`）。
- **RAG 评测指标** —— 检索侧 recall@k/precision@k/MRR，生成侧 faithfulness/answer relevance 常靠 LLM-as-judge（`ai-rag-12`）。
- **Agentic RAG 趋势** —— 把"要不要检索、检索够不够、要不要再查一轮"都交给模型自主决策，是检索从固定管道升级成 Agent 一项可调用能力的体现（`ai-rag-13`）。

## 易错点 / 常见误区清单

1. **把 MCP 说成"就是 function calling 的新名字"**——两者解决的问题层面不同，MCP 是工具生态的连接协议，function calling 是模型和单个工具之间的调用机制，MCP 内部仍然依赖 function calling。
2. **认为 RAG 能完全消除幻觉**——RAG 只是把"知识来源"换成了外部可控的文档，如果检索错了、或者模型不忠实于检索内容依然编造（faithfulness 低），幻觉照样存在，RAG 降低但不能杜绝幻觉。
3. **把向量检索和 rerank 的角色搞反**——向量检索负责"快、广"的粗筛，rerank 负责"准"的精排，如果指望向量检索一步到位做到高精度，通常召回质量会不够。
4. **只知道"lost in the middle"这个词，说不出应对办法**——面试官问到这题往往是想看你知不知道"检索对了不等于生成对了"，以及知不知道把关键证据往 prompt 开头/结尾摆放这类工程缓解手段。
5. **工具调用失败处理只会说"重试"，不区分可重试和不可重试的错误**——权限不足、资源不存在这类错误重试没有意义，应该立刻如实反馈而不是让 Agent 空转浪费 token。
6. **把 Agentic RAG 等同于"RAG + 多轮对话"**——Agentic RAG 的核心是检索本身变成可以被模型自主决策、迭代、和其它工具组合调用的能力，而不是简单地多问几轮。
