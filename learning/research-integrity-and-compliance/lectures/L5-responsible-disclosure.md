# L5 · 安全与负责任披露 (Responsible Disclosure for AI Safety Research)

> 30-min lecture · 目标: 搞清一个AI安全/red-teaming类研究发现真实漏洞或危险能力之后, 该走怎样的协调披露(coordinated disclosure)流程、时间线通常怎么谈, 以及公开发表时该保留多少技术细节。
> 关键区分: **`red-team-jailbreak`专题教的是攻击技术本身**(GCG/PAIR/AutoDAN/Crescendo/prompt injection等具体越狱手法怎么运作, 用教学mock代码理解攻击原理以便造好防御)**; 本讲教的是你已经找到一个真实有效的漏洞之后, 该走什么流程告诉厂商/社区, 而不是这个漏洞本身怎么被制造出来。** 前者的读者姿态是"学习攻击原理", 本讲的读者姿态是"我手上已经有一个真实发现, 现在怎么负责任地处理它"。

---

## 0. 为什么这是个真问题, 而不是`red-team-jailbreak`的重复

`red-team-jailbreak`专题一开篇就写明"所有代码均为教学mock, 不针对真实生产模型, 不输出有效jailbreak prompt"——这个克制本身就是一种负责任披露的实践, 但那个专题没有展开讲**这种克制背后遵循的是什么原则、这套原则从哪来、真的在现实中发现问题该走什么流程**。现实中, 一名做red-teaming研究的人迟早会面对这个场景: 你(或你的团队)对一个真实部署的商业模型做红队测试, 意外发现了一个可靠绕过安全训练、能稳定诱导出危险输出的手法——这时候"我该现在就发一篇论文/发一条推文把这个手法公之于众"还是"先私下告诉模型厂商", 是两条完全不同后果的路, 而多数研究者从未被系统教过这个决策该怎么做。

```
                     read-team-jailbreak 关注的                     本讲 L5 关注的
对象                  攻击技术本身怎么运作(算法/机制)                   发现真实漏洞之后的处理流程(告诉谁/什么时候/披露多少)
产出物                理解攻击原理, 用于构建防御(呼应Topic 6)           一份符合协调披露规范的报告+披露时间线安排
风险敞口              教学mock, 明确声明不针对真实模型                 真实发现: 如果处理不当, 可能被恶意利用造成实际伤害
决策难度              技术理解型问题(怎么运作)                         伦理/流程型问题(该不该公开、公开多少、给厂商多久)
```

## 1. 三种披露模型: 协调披露为什么是默认选择

网络安全领域早已为"发现漏洞之后怎么办"这个问题积累了成熟的制度设计, 三种模式的对比是理解为什么"协调披露"成为默认规范的关键:

```
①完全公开披露(Full Disclosure)   发现即刻全文公开, 不给厂商任何修复窗口
   优点: 逼迫厂商必须立刻行动, 透明度最高
   风险: 攻击者获得利用窗口的速度和防御者一样快, 甚至更快(攻击往往比修复容易)

②完全不披露(Non-Disclosure/私藏)  发现后不告诉任何人(自己知道就行, 或只在小圈子流传)
   风险: 漏洞持续存在, 其他人(善意或恶意)迟早独立发现, 用户/公众持续暴露在风险里
         而不自知

③协调披露(Coordinated Disclosure)  先私下报告给厂商, 给一段修复窗口, 窗口到期或修复完成后
   (现行主流默认模式)              再公开细节(通常伴随致谢)
   优点: 兼顾"厂商有时间修" 和 "问题最终会被公开、督促行业整体改进"两个目标
```

指导这套流程的公开标准是**ISO/IEC 29147(信息技术安全技术——漏洞披露)**, 它定义了从接收报告到协调发布的一整套角色和流程规范; **CERT/CC(卡内基梅隆大学软件工程研究所下属的计算机应急响应中心)**长期运营着被广泛参照的漏洞披露政策实践, 是这套流程在现实中长期运转的具体范例。这两者原本是为传统软件安全漏洞设计的, 但其核心逻辑——"给修复方合理时间窗口, 同时不让问题被无限期掩盖"——同样适用于AI安全领域发现的模型漏洞或危险能力。

## 2. 时间线规范: "90天"是怎么成为默认参照的

时间线是协调披露里最具体、也最容易引发争议的部分——**Google Project Zero团队公开采用的"90天"默认披露窗口**是业界最广为人知的具体范例: 报告漏洞给厂商后, 默认给90天修复时间, 到期后无论是否修复完成都会公开(如果厂商正在积极修复且有正当理由, 窗口可协商延长; 如果厂商完全不回应, 也不会无限期等待)。这套具体数字不是放之四海皆准的法律义务, 而是一种被广泛参照的**行业默认惯例**, 目的是在"给厂商足够时间"和"不让问题被无限期掩盖"之间找一个可执行的平衡点。

把这套逻辑迁移到AI安全研究领域时需要注意: 主要AI实验室目前普遍设有针对模型行为问题的报告渠道(类似安全漏洞的举报入口), 但相比传统软件安全领域, AI安全披露的行业规范整体上还处于更早期、更不统一的阶段——不同实验室对"什么算需要私下报告的严重发现""愿意给多长修复窗口"的具体政策差异较大。这意味着研究者面对一个真实发现时, 往往需要**参照协调披露的基本逻辑(先私下报告, 给合理窗口, 窗口到期后再考虑公开细节的方式和范围), 而不能假设存在一套像ISO 29147那样被全行业统一遵守的AI专用标准**——诚实承认这一点, 比假装有一套现成规则更负责任。

## 3. 公开发表时该保留多少细节

协调披露到期之后, "公开"本身也不等于"和盘托出全部可复现的攻击细节"。传统安全领域的类比是: 一份CVE(公开漏洞编号)通常会描述漏洞的类别、影响范围和严重程度, 但**不一定**附带一份可以直接复制粘贴使用的完整攻击代码(exploit PoC)——描述"这一类问题存在且影响多大"和提供"照抄就能用的攻击配方"是两个不同深度的公开。`red-team-jailbreak`专题从一开始就选择用教学mock代码而非真实有效的越狱prompt, 正是这一原则在教学场景下的具体实践: 让人理解攻击的机制类别(比如GCG如何通过梯度优化搜索对抗后缀), 但不提供一份可以直接拿去攻击真实部署模型的现成"配方"。发表一份负责任的AI安全披露时, 同样需要在"让同行理解问题的严重性和机制"与"避免直接提供一份现成的攻击配方"之间做取舍, 通常的做法是: 描述漏洞类别/影响范围/复现所需的大致条件, 但对能直接复制使用的具体提示词/参数细节做适度保留或延迟公开。

## 4. 一个完整走查示例

```
场景: 一名研究者在对一个已上线的商业LLM做red-teaming测试时, 发现一种可靠绕过安全训练、
      诱导模型输出危险合成信息的手法, 成功率很高且方法本身并不复杂。

①先不公开     不在社交媒体/博客/论文预印本上立刻公开具体手法, 即使"技术上很有意思值得马上分享"
              的冲动很强烈。
②私下报告     通过该模型厂商的安全/漏洞报告渠道提交详细报告, 说明复现步骤、影响范围和严重程度。
③协商窗口     参照协调披露的默认逻辑, 给厂商一段合理的响应和修复时间, 保持沟通, 视厂商响应
              情况判断是否需要延长或(厂商长期不回应时)考虑更早走向部分公开。
④公开发表     窗口到期或确认修复上线后, 发表时描述漏洞类别、成功率、影响范围和背后的机制原理,
              但对可以直接复制使用的具体提示词细节做适度保留, 而不是把"照抄即用"的配方全文公开。
```

## 5. 常见误区

1. **"我没有恶意, 直接公开分享技术上很酷"** —— 意图良善不改变客观后果: 立刻公开等于把可利用信息同时交给了防御者和攻击者, 而攻击往往比修复容易, 时间差对防御方不利。
2. **"厂商不回应我就该无限期等下去, 免得给人扣'不负责任'的帽子"** —— 协调披露不等于无条件配合厂商无限拖延, 合理宽限期后走向公开是督促修复的必要机制, 长期沉默的厂商不应该被无限期等待。
3. **"公开=必须交出全部可复现细节, 否则不算真诚"** —— 描述问题类别和影响范围, 与提供一份可以直接复制使用的攻击配方, 是两个不同深度的公开, 保留必要细节不等于隐瞒问题存在。
4. **"这只是学术研究, 不适用工业界那套披露流程"** —— 一旦发现的是能实际影响真实部署系统的问题, 无论产出形式是论文还是博客, 客观风险敞口和工业界安全漏洞并无本质区别。

## 6. 本讲小结 + 本专题(9.15)收官

- 协调披露(先私下报告给厂商, 给合理修复窗口, 到期后再公开)是发现真实漏洞/危险能力后的默认默认选择, 优于立刻全公开或完全不披露两个极端。
- 时间线没有放之四海皆准的法律标准, Google Project Zero的90天默认窗口是业界广泛参照的具体范例, 但AI安全领域整体规范仍在早期, 需要诚实认识到这一点而非假装有统一标准。
- 公开发表时"描述问题类别/影响范围"和"提供可直接复制使用的攻击配方"是两个不同深度, 负责任披露通常止步于前者。

> **本专题(research-integrity-and-compliance)收官**: L1不端调查与authorship仲裁 → L2 IRB/伦理审查 → L3知识产权与成果转化 → L4国际合作合规 → L5安全与负责任披露, 完整走完了科研诚信与合规这条常年隐身、直到出事才被想起的深水线。这五讲共同的教训是同一句话的五个变体: **"没人提醒你就压根不会想到要处理"的合规盲区, 代价往往要等真正出事时才第一次显形——本专题的作用就是提前替你把这些盲区列出来。**

**动手**: 打开 `templates/compliance-checklist.md` 的第5小节(`disclosure_plan`), 假设你(或设想中)的研究发现了一个真实的安全问题, 写下你会先报告给谁、打算给多久的响应窗口、以及公开发表时你会保留哪些细节不直接公开。写完之后运行 `src/compliance_checklist.py` 里的 `audit()`, 检查这份自检五块骨架是否都填完整。
