# L02 · 何时**不**该造 Agent

## 最重要的设计原则:Start Simple

Anthropic、OpenAI 的 agent 指南异口同声的第一条:

> **先用最简单的方案。只在简单方案被证明不够时,才增加复杂度。**

复杂度阶梯,从下往上爬,**能停在哪层就停在哪层**:

```
复杂度 ↑   成本/不可控 ↑
┌─────────────────────────────────┐
│ 4. Autonomous Agent  开放循环     │  ← 仅当 1-3 都不够
├─────────────────────────────────┤
│ 3. Workflow          多 LLM 编排  │
├─────────────────────────────────┤
│ 2. Augmented LLM     单次+检索/工具│
├─────────────────────────────────┤
│ 1. Single LLM call   一次调用      │  ← 90% 任务从这开始
└─────────────────────────────────┘
```

## 不该上 agent 的信号

| 信号 | 说明 | 该用什么 |
|------|------|---------|
| 步骤**固定可枚举** | "翻译再润色" | workflow / 单次调用 |
| 任务**只有一步** | 分类、抽取、改写 | 单次 LLM call |
| **延迟敏感** | 用户等不起循环 | workflow(可估算) |
| **成本敏感** | 高频调用 | 越往下越好 |
| **错误代价高** | 误删数据、转账 | 强约束 workflow + 人审 |
| 调一次就够准 | 不需要迭代 | 别加 evaluator 循环 |

## 三个权衡轴

每往上爬一层,你在拿**可预测性**换**灵活性**:

| 轴 | Workflow | Agent |
|----|----------|-------|
| **成本** | 可估算(调用数固定) | 不确定(循环可能 5 步可能 50 步) |
| **延迟** | 可估算 | 不确定 |
| **可靠性** | 高(路径写死) | 低(轨迹涌现,可能跑偏) |

agent 唯一的好处是**灵活性**——能处理你事先想不到形状的任务。如果任务形状你想得到,就别为灵活性付那笔不可控的账。

## 反向例子:什么时候 agent 真的值

- **SWE 任务**:"修复这个 bug"——要改哪些文件、改几处,事先不知道。
- **深度研究**:要查几轮、查什么,取决于前一轮查到啥。
- **计算机操作**:点哪、等多久,依赖屏幕实时状态。

这些的共同点:**子任务的数量和顺序是数据驱动的,无法预先画成图**。

## 决策一句话

> 你能把任务画成一张固定的流程图吗?
> **能 → workflow。不能 → 才考虑 agent。**
