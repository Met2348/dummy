## 一、渐进分析分治贪心基础自测

**Q1. 请写出 f(n) = Θ(g(n)) 的完整数学定义,并说明为什么这个定义等价于"f(n) = O(g(n)) 且 f(n) = Ω(g(n))"同时成立。**

A: f(n) = Θ(g(n)) 的形式化定义是:存在正常数 c₁、c₂ 和 n₀,使得对所有 n ≥ n₀,都有 0 ≤ c₁g(n) ≤ f(n) ≤ c₂g(n)。这个定义本质上是"上下夹逼":右边的不等式 f(n) ≤ c₂g(n) 正是 O 记号的定义,左边的不等式 f(n) ≥ c₁g(n) 正是 Ω 记号的定义,两者对同一个 n₀ 同时成立。所以 Θ 记号并不是一个独立于 O/Ω 的新概念,而是要求 f(n) 和 g(n) 同阶增长——既不会渐进地更快也不会更慢。这也是为什么 Θ 常被称为"确界记号",只有当上下界能收紧到同一个函数量级时才能使用它。

**Q2. 主定理 T(n) = aT(n/b) + f(n) 的三种情形分别对应递归树里代价集中在哪一层?请分别说明。**

A: 情形一(叶子主导):当 f(n) 相对 n^(log_b a) 渐进更小时,递归树里绝大部分代价集中在叶子层,总代价为 Θ(n^(log_b a))。情形二(各层均衡):当 f(n) = Θ(n^(log_b a) · log^k n) 时,递归树每一层的代价大致相当,共有 log_b n 层,结果多一个对数因子,为 Θ(n^(log_b a) · log^(k+1) n)。情形三(根主导):当 f(n) 相对 n^(log_b a) 渐进更大,且满足正则性条件 a·f(n/b) ≤ c·f(n)(存在常数 c<1)时,代价集中在根节点,总代价为 Θ(f(n))。三种情形的直觉都是"看递归树里代价被哪一层主导"。

**Q3. 用代入法证明递归式上界时,为什么直接假设"T(n) ≤ cn"去验证归纳步骤,经常会在代数上推不出来?应该怎么修正?**

A: 直接假设 T(n) ≤ cn 去做归纳验证时,展开一层往往会多出一个低阶余项(比如 T(n) ≤ 2c(n/2) + n = cn + n),这个多出来的 n 会让归纳假设"卡"在推不出 T(n) ≤ cn 的位置上,这是代入法里典型的"归纳假设强度不够"问题。标准修正方法是减去一个低阶项做更强的归纳假设,比如改设 T(n) ≤ cn - b(b>0 为待定常数),这样展开后多余的正项恰好被减去的低阶项吸收,归纳步骤就能闭合。这个技巧说明代入法的关键不在于"猜对量级",而在于"猜一个足够强、能让归纳自洽"的具体表达式。

**Q4. 如果递归式是不平衡分治,比如 T(n) = T(n/3) + T(2n/3) + n,用递归树法怎么估算总代价?**

A: 这种不平衡递归树两条分支的深度不同:沿着 n/3 那条最短路径大约 log₃n 层就到达边界,沿着 2n/3 那条最长路径大约 log_{3/2}n 层才到达边界,所以整棵树不是规整的完全树,层数介于两者之间。但关键观察是:只要还没到达任意一条路径的边界,这一层所有节点上 f(·) 的值加总起来都恰好是 n(因为每层子问题规模之和等于上一层,而 f 是关于规模线性的),所以每一层的代价上界都是 O(n)。由于树的深度是 O(log n)(由较长的那条路径决定),因此总代价上界是 O(n log n),与完全平衡的 T(n)=2T(n/2)+n 同阶。

**Q5. 势能法计算均摊代价的公式是什么?势能函数 Φ 需要满足的两个条件分别保证了什么?**

A: 势能法定义摊还代价 ĉᵢ = cᵢ + Φ(Dᵢ) - Φ(D_{i-1}),其中 cᵢ 是第 i 次操作的实际代价,Φ(Dᵢ) 是第 i 次操作后数据结构的势能。势能函数需要满足:(1) Φ(D₀) = 0(初始势能为零,通常自然满足);(2) 对所有 i 都有 Φ(Dᵢ) ≥ 0(势能始终非负)。这两个条件共同保证了对任意 n 次操作序列,总的实际代价之和 ≤ 总的摊还代价之和,因为 Σĉᵢ = Σcᵢ + Φ(Dₙ) - Φ(D₀) = Σcᵢ + Φ(Dₙ) ≥ Σcᵢ(由条件2),所以摊还代价的总和构成了真实总代价的一个有效上界,可以用来证明均摊复杂度。

**Q6. 分治法的三个步骤具体是什么?归并排序和快速排序在"合并"这一步的角色分配上有什么本质不同?**

A: 分治法三步是分解(Divide,把原问题划分成若干规模更小的同类子问题)、解决(Conquer,递归求解子问题,规模足够小时直接求解)、合并(Combine,把子问题的解组合成原问题的解)。归并排序的"分解"是平凡的(直接从中点切开,不需要计算),真正的工作量都在"合并"步骤——合并两个有序子数组需要 O(n) 的线性扫描,这是它整个算法复杂度的主要来源。快速排序则相反:"分解"步骤(用主元把数组划分为两部分)承担了全部实质工作,是 O(n) 的划分过程,而"合并"步骤是平凡的——两个已排序的子数组和主元本身放在一起就已经整体有序,不需要额外操作。这是理解两者复杂度分析切入点不同的关键。

**Q7. 快速排序最坏情况 O(n²) 具体是在什么输入和主元选择策略下发生的?为什么随机化主元选择能解决这个问题?**

A: 当主元选择策略是固定的(比如永远选第一个或最后一个元素),并且输入数组恰好是已经有序或逆序排列时,每次划分都会把数组分成一个空(或大小为1)子数组和一个大小为 n-1 的子数组,递归深度达到 Θ(n),总代价退化为 Θ(n²)。这种情况的危险之处在于,攻击者只要知道你的主元选择策略,就能构造出这种对抗性输入(比如系统对外接受排序请求时)。随机化主元选择(每次从当前子数组中均匀随机选一个元素作为主元)让最坏情况的出现不再依赖于输入本身的顺序,而是依赖于随机数,因此对任意固定输入,期望运行时间都是 O(n log n)——这是用随机性把"对抗性输入总能找到"的问题转化为"期望复杂度有保证"的问题。

**Q8. 贪心选择性质的精确数学含义是什么?为什么说贪心算法要求的条件比动态规划的最优子结构更强?**

A: 贪心选择性质是指:通过做出局部当前看来最优的选择,一定存在某个整体最优解包含这个局部选择,即"进行一次贪心选择后剩下的子问题的最优解,加上这次贪心选择,构成原问题的一个最优解",且这个性质在每一步递归地成立而不需要考虑其他候选选择。动态规划只要求问题具有最优子结构(原问题的最优解由子问题的最优解组合而成),但需要枚举所有可能的选择/划分点,通过比较才能确定哪个子问题组合是最优的。贪心算法则进一步要求"不需要枚举比较,每步只做一次局部最优选择就一定通向全局最优",这是明显更强的假设,因此适用范围比 DP 窄得多——绝大多数问题只满足最优子结构而不满足贪心选择性质,这也是为什么"贪心是 DP 的特例"这句话成立(能用贪心的问题一定也能用 DP,反之不然)。

## 二、动态规划图算法网络流深水自测

**Q1. 证明最优子结构时常用的"剪切-粘贴"(cut-and-paste)论证,它的核心逻辑框架是什么?**

A: 剪切-粘贴论证是一种反证法:假设原问题的某个最优解 S 中,某个子问题部分的解不是该子问题的最优解,那么就把这部分"剪切"掉,"粘贴"上该子问题真正的最优解,构造出一个新的解 S'。因为子问题部分变得更优(或至少不差),且这个替换不影响解的可行性,所以 S' 整体不会比 S 差,甚至严格更优。如果 S' 严格优于 S,就与"S 是最优解"的假设矛盾;因此原问题最优解中,每个子问题的部分都必须是该子问题的最优解——这正是最优子结构的定义。这个论证框架是几乎所有经典 DP(如最短路径、背包、编辑距离)证明最优子结构成立的标准手法。

**Q2. 0/1背包问题的DP实现中,如果把内层遍历容量的循环方向从"从大到小"误写成"从小到大",会产生什么错误?为什么?**

A: 0/1背包要求每个物品最多使用一次。如果用一维滚动数组 dp[j] 表示容量为 j 时的最优价值,内层循环必须从大容量往小容量遍历(逆序),这样在计算 dp[j] 时,dp[j-w] 用的还是"上一个物品"处理完之后的旧值,保证每个物品只被计入一次。如果误写成正序遍历,dp[j-w] 可能在本轮循环中已经被同一个物品更新过,相当于同一个物品被无限次重复使用去填充更大的容量,这实际上把0/1背包问题错误地计算成了完全背包问题(每个物品可重复选取)的结果,导致价值被高估。这个bug的隐蔽之处在于,小规模测试数据下如果物品重量恰好较大,循环方向错误可能不会暴露,只有在数据规模足够触发重复选取时才会出错。

**Q3. 割性质(cut property)是什么?它如何被用来证明Kruskal算法(或Prim算法)得到的是最小生成树?**

A: 割性质陈述:给定图的一个割(cut,即把顶点集合划分成两个非空子集 S 和 V-S),如果某条边 e 是横跨这个割的所有边中权重最小的(且该最小值唯一,或者算法在权重相等时按固定规则打破平局),那么一定存在某棵最小生成树包含边 e。证明思路是反证法:假设某棵最小生成树 T 不包含 e,那么加入 e 后 T 必然形成一个环,这个环上一定还有另一条边 e' 也横跨同一个割(因为环从 S 到 V-S 必须至少跨越两次),用 e 替换 e' 后,由于 w(e) ≤ w(e'),新生成树的总权重不会增加,从而得到一棵同样最优(或更优)、包含 e 的生成树。Kruskal 每次选当前权重最小且不成环的边,Prim 每次选连接已选顶点集合与外部的最小权重边,两者的每一步都恰好符合割性质的条件,因此归纳可证它们每一步的选择都存在于某棵最小生成树中,最终得到的就是一棵最小生成树。

**Q4. Dijkstra算法为什么不能正确处理带负权边的图?请给出一个能让Dijkstra算出错误结果的具体反例。**

A: Dijkstra依赖的核心不变量是:一旦某个顶点被从"未确定"集合移入"已确定"集合(即它的最短距离被"定案"),后续的松弛操作再也不会缩短它的距离,因为算法保证之后处理的顶点的当前距离估计只会更大或相等。这个不变量成立的前提是所有边权非负,一旦存在负权边,一个已经被"定案"的顶点仍然可能通过后续发现的负权边被进一步缩短,而算法已经不会再去更新它。反例:考虑三个顶点 A、B、C,边 A→B 权重为 1,A→C 权重为 4,B→C 权重为 -3。Dijkstra从A出发,会先确定 dist(B)=1(因为1<4),如果在处理B之前就已经把C定案为距离4,就会漏掉之后"通过B到C"这条更短的路径,而真实的最短距离 dist(C) 应该是 1+(-3)=-2,比错误定案的4小得多。

**Q5. Bellman-Ford算法要跑n-1轮全边松弛,用归纳法怎样证明"第k轮结束后"具体保证了什么不变量?**

A: 归纳假设(不变量):对任意顶点 v,如果从源点 s 到 v 存在一条最多使用 k 条边的最短路径,那么在Bellman-Ford跑完第 k 轮全边松弛之后,dist[v] 就已经等于这条最多k条边路径的真实最短距离。基础情形 k=0 时,只有源点自身的最短路径(0条边、距离0)成立,dist[s]=0 在初始化时即满足。归纳步骤:假设第 k-1 轮后,所有最多 k-1 条边的最短路径都已经正确求出,那么对于某个最多 k 条边的最短路径 s→...→u→v(其中 s→u 部分最多 k-1 条边),由归纳假设 dist[u] 已经正确,第 k 轮遍历到边 (u,v) 时松弛操作 dist[v] = min(dist[v], dist[u]+w(u,v)) 就会把 dist[v] 更新为正确值。由于任意简单最短路径最多包含 n-1 条边(图中最多n个顶点),所以跑满 n-1 轮就足以保证所有最短路径都被正确求出。

**Q6. 完整陈述最大流最小割定理,并说明Ford-Fulkerson算法终止时为什么当前流一定是最大流。**

A: 最大流最小割定理指出:在一个流网络中,最大流的值恰好等于最小割的容量,即以下三个命题等价:(1) f 是一个最大流;(2) 残量网络 Gf 中不存在从源点 s 到汇点 t 的增广路径;(3) 存在某个割 (S,T),使得 f 的值等于该割的容量 c(S,T)。Ford-Fulkerson算法终止的条件正是残量网络里找不到增广路径,此时令 S 为从 s 在残量网络中可达的所有顶点集合、T 为其余顶点,可以证明所有从 S 跨到 T 的原图边都是满流(否则残量网络里还有正向剩余容量,S会更大)、所有从 T 跨到 S 的边流量都是0(否则残量网络里有反向边,S也会更大),于是流的值恰好等于这个割 (S,T) 的容量。而任意流的值都不会超过任意割的容量(弱对偶性),所以当流的值等于某个割的容量时,这个流必然是最大流,对应的割也必然是最小割。

**Q7. Edmonds-Karp算法相比朴素的Ford-Fulkerson具体改进了什么?为什么"用BFS选最短增广路径"这个改进能保证整体是多项式时间?**

A: 朴素Ford-Fulkerson对增广路径的选择没有任何限制,如果每次都恰好选到容量很小的路径,迭代次数可能正比于流值的大小,当容量是较大的整数时退化为伪多项式时间。Edmonds-Karp的改进是:每次都用BFS在残量网络里找一条边数最少(最短)的增广路径。这个改进能保证多项式时间的核心引理是:可以证明在整个算法执行过程中,每条边(u,v)作为增广路径上的"瓶颈边"(即该路径上剩余容量最小、决定了这次增广流量的边)最多出现 O(V) 次——因为每次(u,v)成为瓶颈边被"用尽"后,u到s(或v到t)的最短距离会严格增加,而这个距离最多能增加O(V)次。由于图中共有O(E)条边,所以总的增广次数是 O(VE),每次BFS找路径需要O(E)时间,总时间复杂度是 O(VE²),与流值大小无关,是严格多项式时间。

**Q8. Hall定理是怎么陈述的?它和最大流最小割定理之间有什么内在联系?**

A: Hall定理(婚姻定理)陈述:给定二分图 G=(X∪Y, E),存在饱和 X 的完美匹配(即X的每个顶点都被匹配)当且仅当对 X 的任意子集 A,其邻居集合 N(A)(A中所有顶点在Y中相邻顶点的并集)满足 |N(A)| ≥ |A|(称为Hall条件)。这个定理和最大流最小割定理的联系在于:把二分图匹配问题转化为网络流问题(源点连向X所有顶点、Y所有顶点连向汇点、X-Y原边都设容量为1)之后,最大匹配数等于最大流值,由最大流最小割定理,最大流值又等于最小割容量。可以证明,这个网络里的最小割恰好对应"选取某个X的子集A使得N(A)不能全部满足匹配"的最坏情况,一旦Hall条件对某个A失败(|N(A)|<|A|),就能构造出一个容量小于|X|的割,导致最大流(即最大匹配数)小于|X|,完美匹配不存在——这正是Hall定理充分必要性背后网络流层面的证明机制。

## 三、算法范式选择与工程权衡判断自测

**Q1. 面对一个新的优化问题,如果怀疑贪心算法可能有效,应该按什么步骤去验证这个判断,而不是仅凭直觉下结论?**

A: 系统性的验证步骤是:第一步,先尝试严格证明贪心选择性质是否成立——即证明"存在一个包含当前贪心选择的整体最优解",通常用交换论证(假设存在一个不包含贪心选择的最优解,证明可以把它替换成包含贪心选择且不会变差的解);第二步,验证问题是否具有最优子结构,做完贪心选择后剩下的子问题的最优解与原问题的解拼接是否一致;第三步,如果第一步的证明找不到严谨的论证路径,应该主动去尝试构造反例(尤其是边界情况和极端输入),因为贪心算法一旦选择性质不成立,几乎总能找到具体反例。仅凭"跑了几个随机测试用例结果都对"是不充分的,因为随机测试很难覆盖贪心失效的特定结构化反例,必须有理论证明或者穷尽性论证(比如覆盖所有情况的分类讨论)兜底。

**Q2. 分数背包问题用贪心算法(按单位重量价值排序)可以得到最优解,但0/1背包问题同样的贪心策略却不行,这是为什么?**

A: 分数背包允许物品被任意比例地拆分放入背包,因此按单位重量价值从高到低排序、优先装性价比最高的物品,直到背包装满(最后一件可以只装部分)必然是最优的——因为这里没有"物品不可分割"的约束,任何非贪心的分配方案都可以通过局部调整(把一部分低性价比物品换成等重量的高性价比物品)变得更优,这符合严格的交换论证。0/1背包则要求每个物品要么整个装入要么不装,不能拆分,这时"性价比最高"不再等价于"对整体最优贡献最大",因为可能出现"性价比稍低但恰好能利用完剩余容量"的组合明显更优、而贪心选出的高性价比物品组合恰好留下大量容量浪费的情况。这个对比是理解"贪心是否成立与问题的约束结构直接相关"的经典例子:0/1背包本质上需要动态规划枚举"选或不选"来搜索所有可能的容量利用组合。

**Q3. "找零钱时每次贪心地选择不超过剩余金额的最大面值硬币"这个策略,为什么不是对所有币值系统都成立?请举出使它失效的具体反例。**

A: 这个贪心策略只在特定的"标准"币值系统下(比如美元的1/5/10/25分,或者更一般地满足特定结构条件的币值系统)才保证最优,并不是对任意币值系统都成立的定理。反例:假设币值系统是 {1, 3, 4},要凑出总金额6。贪心策略会先选面值4(剩2),再选面值1两次(剩1,剩0),一共用了3枚硬币(4+1+1);但实际最优解是用两枚面值3的硬币(3+3=6),只需要2枚硬币。这说明贪心算法必须针对具体的币值系统单独验证贪心选择性质是否成立(通常需要证明该币值系统满足某种"每种面值都是下一档面值的整除数"或类似的结构条件),不能想当然地认为贪心对找零钱问题普遍适用。

**Q4. 团队里有人主张"贪心算法在我们跑的20个随机测试用例上结果都正确,所以这个贪心策略是对的",作为负责验证的人应该如何反驳或补充论证?**

A: 随机测试用例通过并不能构成正确性证明,因为贪心算法失效的反例通常具有特殊的结构性质(比如找零钱例子里的非标准币值组合、或者活动选择问题里刻意构造的时间冲突模式),而随机生成的测试数据很难覆盖到这类边界结构,大概率会"侥幸"全部通过。正确的做法是:要么给出严格的数学证明(交换论证或反证法,证明贪心选择性质在这个问题上成立),要么如果暂时证明不出来,就应该主动尝试构造针对性的对抗测试用例(比如极端不均匀的输入、边界重叠的输入、或者已知类似问题失效模式的变体),用这些有针对性的用例去压力测试贪心策略,而不是依赖随机测试的通过率。在没有理论证明支撑之前,"随机测试全过"只能算是弱的经验证据,不能当作正确性的充分保证,尤其是要上生产环境的场景。

**Q5. 为什么理论上渐进复杂度最优的算法,在实际工程实现中不一定是最佳选择?请列举几个具体原因。**

A: 常见原因包括:(1) 常数因子和低阶项被渐进记号忽略,理论复杂度更优的算法可能有远大于简单算法的常数因子,只有当n足够大时优势才会显现,而实际场景n可能一直较小;(2) 缓存友好性和内存局部性,渐进复杂度分析通常假设统一内存访问代价,但实际硬件中顺序访问数组比复杂数据结构(树/链表)的指针跳转快得多,理论上更优的算法可能因为频繁的cache miss实际更慢;(3) 实现复杂度和可维护性,复杂度更优的算法(比如Fibonacci堆、复杂的平衡树变种)实现难度和出错率更高,长期维护成本可能超过性能收益;(4) 并行化/向量化的友好程度,某些理论上次优的简单算法反而更容易利用现代硬件的SIMD或多核并行加速。因此工程决策要综合考虑实际数据规模、硬件特性和维护成本,而不能只看渐进复杂度这一个维度。

**Q6. 为什么对于长度在几十以内的短数组,插入排序(O(n²))在实测中经常比理论上更优的O(n log n)排序算法(如快速排序、归并排序)更快?**

A: 核心原因是常数因子和额外开销:插入排序的内层循环结构极其简单(比较+移动),没有递归调用开销、没有函数调用栈的建立与销毁、没有额外的辅助空间分配,而且对几乎有序的数据表现接近O(n)。相比之下,快速排序和归并排序在小规模输入上,递归划分/合并带来的函数调用开销、额外内存分配(归并排序的辅助数组)在小n时占比很高,可能远超过它们理论上"更少比较次数"带来的收益。当n很小时,O(n²)和O(n log n)之间的渐进差距本身就很小(比如n=30时,n²=900,而n log n≈147,差距在常数因子面前不构成决定性优势),这也是为什么很多高性能排序库(如C++ STL的introsort)在递归到小规模子数组时会主动切换成插入排序作为"底层优化"。

**Q7. 如果攻击者能够构造大量哈希碰撞的输入,让一个理论上平均O(1)的哈希表查找退化到O(n),从工程角度应该如何应对这种风险?**

A: 这是一种针对哈希表实现的算法复杂度攻击(algorithmic complexity attack),典型应对方法包括:(1) 使用带随机种子的哈希函数(每次进程启动时用随机seed初始化哈希函数,比如多数现代语言运行时对字符串hash做的随机化),让攻击者无法离线预先计算出会导致大量碰撞的输入,因为不知道具体的种子;(2) 对哈希冲突链设置退化保护机制,比如Java的HashMap在JDK 8之后,当同一个桶内的链表长度超过阈值(8)时会自动将该桶转换为红黑树,把最坏情况从O(n)兜底降低到O(log n);(3) 在受信任程度较低的输入场景(比如直接暴露给公网的API)对输入做限流和合理性校验,减少攻击者能提交的极端规模输入的机会。这几种手段本质上都是给"平均情况快但最坏情况可被攻击者主动触发"的数据结构增加一层最坏情况的兜底保证。

**Q8. 面试中候选人说"我用贪心算法解决了这个NP难问题,而且所有测试用例都通过了",作为面试官应该如何追问和评判这个回答?**

A: 首先需要指出一个逻辑上的警觉信号:如果一个问题被公认是NP难的(即不存在已知多项式时间的精确算法,除非P=NP),而候选人声称用一个通常运行在多项式时间的贪心算法"解决"了它,那么要么候选人对问题的NP难性质有误解,要么这个贪心算法实际上只是一个启发式算法/近似算法,并不能保证在所有输入上都得到最优解。面试官应该追问:(1) 这个贪心算法是得到精确最优解还是近似解?如果是近似解,能否给出近似比的理论保证(比如是否是2-近似)?(2) 能否构造出一个贪心算法给出次优解的具体反例?(3) "所有测试用例都通过"具体指的是通过、还是恰好得到了最优值,这两者对NP难问题而言是完全不同的两件事,前者可能只是测试用例设计得不够刁钻。一个合格的候选人应该能清楚区分"启发式方法在实践中效果好"和"该算法在理论上保证最优"这两个完全不同层次的论断。
