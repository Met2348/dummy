# L03 · 怎么设计一个抗刷的基准

## 1. 抗刷的四条原则（本模块各对应一处）

1. **held-out 必须存在且保密**：判分用候选没见过的数据。`task.py` 把 `HELDOUT` 和 `VISIBLE`
   分开，评分器私藏 held-out——这是 `heldout` rubric 抗刷的根。
2. **永不信任自报指标**：候选 print 的任何数字都不算数。`trust_print` 的灾难就是反面教材——
   它不仅被刷，还奖励说谎者。
3. **真执行 + 行为检查**：不只看一个聚合指标，还查行为/不变量（Hands-on 让你加"对称性"检查）。
   单一指标最好刷，行为约束难刷。
4. **独立 verifier**：重算分数的逻辑要和出题/答题方独立（呼应 9.2 角色分离、9.4 忠实度核查）。

## 2. 但"强 rubric"也不是终点

`heldout` 在本模块够用，是因为任务**有明确地面真值**且 held-out 充足。真实研究里：

- **开放问题没有 held-out**：MLR-Bench 那类开放研究任务，"对不对"本身就难定义。
- **任务太简单也是漏洞**：如果候选能反推出规则结构，held-out 也能被"猜"过去（Hands-on #1）。
- **held-out 会泄漏**：一旦基准公开、被训练数据吸收，held-out 就不再 held-out（数据污染）。

> **没有一劳永逸的抗刷基准。** 评测和被评测是一场军备竞赛——
> 这正是 9.7（自我改进会 game fitness）和 9.8（红队）要继续打的仗。

## 3. 把评测当成"会被攻击的系统"来设计

这是 9.6 通向 9.8 的桥：与其问"我的 rubric 对不对"，不如问"**一个想刷分的 agent 会怎么攻击它**"。
本模块的三个候选就是三种攻击：记忆攻击（hardcode）、自报攻击（print-fraud）、越权攻击（malicious）。
设计基准时，先把这些攻击列出来，再逐个堵——这就是红队思维。

## 4. 动手 / 通往毕业 capstone

1. 实现 Hands-on #2（加对称性等行为检查）和 #3（加独立 verifier 重算），
   把 `heldout` 升级成"held-out + 行为 + 独立复算"的三重 rubric。
2. 预告 9.8：把本模块的 `safe_exec` 和"抗刷 rubric"搬去给 9.5 的 mini-AI-Scientist 当守卫——
   让一个**独立的 verifier** 重算它报告里的数字，而不是信它自己的 `review`。
   这就是系列毕业 capstone"可信 mini AI-Scientist"的评测半边。
