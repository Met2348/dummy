## 1. 封面 · TL;DR

- **作者/出处**：Wenlin Zhang, Xiaopeng Li, Yingyi Zhang, Pengyue Jia, Yichao Wang, Huifeng Guo, Yong Liu, Xiangyu Zhao（通讯）；港城大 + 大连理工 + 华为诺亚方舟实验室；2025-08，arXiv 2508.12752，投稿 ACM。
- **一段话**：大语言模型 (large language model, LLM) 虽强，但受困于「内部知识边界 (internal knowledge boundary)」——训练截止后的、动态的、专业的知识它不知道。检索增强生成 (retrieval-augmented generation, RAG) 让模型被动地「读」检索到的内容，但只能被动消费，不参与更深的探索与推理。随着 LLM 获得规划、工具调用、反思推理等 agentic 能力，范式从被动检索转向主动、目标驱动的交互，催生「智能体搜索 (agentic search)」；其中更进一步、把规划+迭代检索+综合写作合为一体的，就是本篇主角 **deep research（深度研究）**。本综述把 deep research 流水线拆成**四个互联阶段**（见原文 Figure 1）：**规划 (Planning) → 提问 (Question Developing) → 网络探索 (Web Exploration) → 报告生成 (Report Generation)**，逐段分析关键技术挑战、归类代表方法，再汇总优化技巧与专用基准，最后给出开放挑战与路线图。
- **三条带走的结论**：
  1. **一条流水线、四段能力**：deep research = 规划（拆子目标）→ 提问（把子目标变查询）→ 网络探索（去网上取证据）→ 报告生成（把碎片证据合成忠实长文）。每段都有自己的定义、瓶颈、代表方法（原文 §2–§5，四张分类表 Table 1–4）。
  2. **「能力为中心」是这篇的差异化卖点**：作者明说（原文 §1 末）——前作 Huang et al.[26] 偏「系统架构 + 路线图」、Xu et al.[84] 偏「任务与工具的枚举式罗列」；**本篇的分类法以「能力的形成与整合」为核心**，想揭示方法间更深的联系与设计范式演化。why > how 的综述。
  3. **评测远未就绪、流水线远未打通**：11 个主流基准里只有 **2 个（DeepResearch Bench、DeepResearchGym）真正覆盖到「报告生成」**（原文 Table 5）——大多数基准止步于「搜索式问答」，报告生成被系统性低估；而四段能力目前多被**孤立优化**，跨段联合优化、长程一致性、多模态、个性化都还是空白（原文 §8）。

> 主讲提示：把第 2、3 条钉死——这是综述的「观点」而非「系统宣称」。第 2 条回答「凭什么再写一篇 deep research 综述」，第 3 条是它留给后人的 to-do list。

## 2. 问题与动机（why —— 本篇最该讲透的一节）

**为什么需要 deep research？三级递进的「能力台阶」。** 论文在 §1 用一条清晰的演进线索说明动机，建议组会上照这条线讲：

1. **纯 LLM 的天花板**：GPT-4、Qwen3、DeepSeek-R1 等让 LLM 成为通用语言 agent，但**受限于内部知识边界**——动态、专业、训练截止之后的知识它根本不知道（原文 §1 开篇）。
2. **RAG 的局限**：RAG 给模型接上外部语料，检索 + 条件生成提升了事实性与适应性；但它**把模型限制成「被动消费检索内容」**，不参与更深的探索或推理（原文 §1）。这是关键转折：RAG 是「单次、被动」的。
3. **agentic search → deep search → deep research**：当 LLM 有了规划、工具用、反思推理，交互从「被动检索」变成「主动、目标驱动」，是为 **agentic search**；其中一个突出方向是 **deep search**——agent 动态地做规划、提问、网络探索来迭代检索。但 deep search **仍不足以完成需要综合与推理的复杂研究任务**；为补这个缺口，**deep research** 把「探索」与「结构化分析 + 迭代生成」结合，能把检索到的证据**转化为连贯、忠实、目标对齐的输出**，满足高层研究工作流（原文 §1）。

> 直觉：把这三步想成「会背书的学生 (LLM) → 会查资料的学生 (RAG) → 会主动设计调研、边查边想的研究助理 (deep research)」。台阶越高，**自主性 (autonomy) 与综合能力 (synthesis)** 越强。

**为什么是「四段能力」而不是「一个系统」？** 论文 §1/§4 给的论证：deep research 系统天然是**多阶段任务 + 大量信息处理**，对 agentic 能力提出**逐级演进 (progressively evolving)** 的要求（原文 Figure 1 的副标题就是这个意思）。所以作者主张：**与其枚举一个个完整系统的 pipeline，不如解剖底层的模块化能力 (modular competencies)，分析各自的技术瓶颈与协调挑战**（原文 §1 倒数第二段）。这正是「why > how」：先问「每段能力难在哪」，再看「代表方法怎么解」。

**不做会怎样？** 论文隐含的代价：
- 没有统一坐标系，社区会陷入「我的 end-to-end 系统比你的强」的不可比竞赛，难以定位**到底是哪一段能力拖了后腿**。
- 评测会继续偏向「搜索式问答」（容易刷分），而**真正困难的「报告生成」被回避**——Table 5 里 9/11 基准不评报告生成，就是这个病的实证。

> 主讲提示：这一节是 why 的核心。把「知识边界 → RAG 被动 → agentic/deep search → deep research」四步台阶讲清；再强调「能力为中心 vs 系统罗列」是本篇方法论的灵魂。后面的分类表都只是这套世界观的展开。

## 3. 研究问题 / 核心 intention（形式化成一句话）

把综述要回答的问题压成一句：

> **如果把「自主做一份基于网络证据的深度研究报告」拆成可独立优化、又必须联合整合的若干「能力模块」，那么——每个模块的精确定义是什么、各自的核心挑战是什么、代表方法如何归类、用什么基准评测、整条流水线还差什么？**

它隐含的**假设 / 立场**：
- (a) **可分解假设**：deep research 的复杂性可以被「规划 / 提问 / 网络探索 / 报告生成」四段干净地切开，每段有清晰的输入输出契约（这点由原文 §2–§5 的四个 Definition + Eq.(1)–(4) 落实）。
- (b) **能力优先于架构**：理解「能力如何形成与整合」比理解「某系统的具体连线」更能揭示设计范式的演化（原文 §1 明确对比 [26][84]）。
- (c) **单/多智能体只是实现选择**：同一套四段能力，既可由单 agent 端到端承担，也可拆给多 agent 分工（原文 §6.1），架构是正交维度，不是分类主轴。

> 主讲提示：强调 (b)——这是它和同期 [26]（架构为主轴）最根本的分工。记住这句，第 17 节讲三篇综述互补时会回扣。

## 5. 方法总览（big picture，先直觉后数学）

整篇的骨架就是原文 **Figure 1** 的一张流水线图：用户给一个研究问题，四个 agent 接力，每段对应一组「agentic 能力」。

```mermaid
flowchart LR
  U["用户<br/>Research Question q0"] --> P
  subgraph CORE["Deep Research 四段流水线（原文 Figure 1）"]
    direction LR
    P["§2 规划 Planning<br/>拆成子目标计划 P"] --> Q["§3 提问 Question Developing<br/>子目标→查询集 Qi"]
    Q --> W["§4 网络探索 Web Exploration<br/>查询→文档集 D"]
    W -. "Iterative / Deep Search<br/>迭代回灌证据 E" .-> Q
    W --> R["§5 报告生成 Report Generation<br/>证据→忠实长报告 Y"]
  end
  P -. "对应能力" .- PA["结构化世界知识<br/>/ 可学习规划"]
  Q -. .- QA["奖励优化<br/>/ 监督驱动"]
  W -. .- WA["浏览器Agent (API)<br/>/ 搜索引擎API"]
  R -. .- RA["结构控制<br/>/ 事实完整性"]
```

**直觉（四段各像什么）**：
- **规划**像「研究设计」：把一个宽泛/含糊的大问题，拆成**有结构、可解释、面向目标**的子任务路线图——决定查什么、按什么顺序查、中间信息如何支撑下游综合（原文 §2 引言）。
- **提问**像「把研究设计变成一句句具体的检索 query」：一个子目标可能要好几个**抽象度/具体度/粒度**各异、互补的查询，且要随证据演化而自适应（原文 §3，对比静态问句 [29]）。
- **网络探索**像「跑图书馆 / 上网查」：选工具、发查询、解析结果、过滤噪声与冲突，**迭代**地深挖稀疏分散的证据（原文 §4，区别于传统 RAG 的「单次检索」）。
- **报告生成**像「把一堆笔记写成结构化报告」：多源融合、篇章级规划、任务特定格式，**既要结构连贯、又要忠于证据**（原文 §5，超越抽取式/浅层生成）。

注意那条**回灌虚线**（W→Q）：deep search 的精髓是「边查边改问」——网络探索拿到的证据 $\mathcal{E}$ 会喂回提问阶段，驱动下一轮查询（原文 §3 Eq.(2) 里的 $\mathcal{E}$、§4 的「iterative, agent-driven」）。这就是它和「单次 RAG」的分水岭。

> 主讲提示：讲完这张图，整篇就「会读」了。强调两点：① 四段是**串行契约 + 迭代回灌**（不是纯线性）；② 每段下面挂的「能力对应」就是 §2–§5 四张分类表的列名。

## 7. 阶段一 · 规划 Planning（§2）：把大问题拆成可执行计划

### 7.1 定义与公式（直觉→符号→式→读出什么）

> 直觉：科研第一步不是「马上去查」，而是「想清楚怎么查」。反应式 (reactive) 的逐步提示容易短视、试错、重复；**显式规划**先生成一张「任务感知的路线图」，让检索与推理更系统，减少 trial-and-error 失败（原文 §2 引言）。

记号（先定义，见 §6 表）：$q_0$ 初始研究问题；$\mathcal{K}$ agent 上下文；$\mathcal{M}^{\text{plan}}$ 规划模型，参数 $\theta$；$\mathcal{P}$ 输出计划。原文 **Def 2.1 / Eq.(1)**：

$$ \mathcal{P} \;=\; \mathcal{M}^{\text{plan}}(q_0,\ \mathcal{K};\ \theta),\qquad \mathcal{P}=[s_1,s_2,\dots,s_n]. $$

读出什么：规划就是「**问题 + 上下文 → 子目标/工具步骤的有序序列**」。$\mathcal{P}$ 是一个**序列**（有先后），它是「高层目标」与「低层执行」之间的桥（原文 §2 末）。计划是**动态生成**的，可随任务复杂度、资源、用户意图自适应。

### 7.2 分类法（原文 Table 1）：规划方法的两大族

原文把规划方法按「计划从哪来」分两大类、各若干子类：

| 一级类 | 二级子类 | 代表工作（原文编号）|
|--------|----------|------|
| **结构化世界知识规划**<br/>Planning with Structured World Knowledge | 经仿真规划 (via simulation) | WebDreamer [20], Simulate Before Act [21] |
| | 经模块化规划 (via modularity) | WebPilot [96], WKM [52], Plan-and-Act (Erdogan) [17] |
| | 经适应规划 (via adaptation) | Thought of Search [32], MPO [83] |
| **可学习的规划过程**<br/>Planning as a Learnable Process | 经空间探索 (via space exploration) | Agentsquare [62], Agent-E [4] |
| | 经自训练 (via self-training) | Patel et al. [45], InSTA (Trabucco) [71] |
| | 经偏好建模 (via preference modeling) | MindSearch [11], SimpleDeepSearcher [69], Search-in-the-Chain [85], WEPO [37], MPO [83] |

**why 两族的根本区别**（原文 §2.1 vs §2.2）：
- **结构化世界知识**：把「世界结构」当外部支架——LLM 可作**隐式世界模型** (implicit world model)（[20]）；Simulate Before Act 先在脑内 rollout 候选动作轨迹、评估可行性再执行，提升鲁棒性 [21]；WebPilot 用「战略规划器 + 子 agent」分布式规划 [96]；Qiao 等用外部知识图谱把推理路径**接地** (knowledge-grounded) [52]；Erdogan 等对长程任务「初始计划 + 迭代精化」保持长程一致 [17]；Katz 等量化推理的计算开销、提出「效率感知规划」在**深思深度与资源占用间权衡** [32]；Xiong 等的 **MPO（meta-plan optimization）** 用元学习跨环境自适应调规划策略 [83]。一句话：**用先验结构换「有远见、又算得起」的规划**。
- **可学习过程**：把**规划机制本身**当学习对象，靠搜索、反馈、大规模训练迭代精化决策流程。Agentsquare 在巨大配置空间里**搜架构**自动拼装任务自适应的规划流水线 [62]；Agent-E 从导航任务里蒸馏可复用组件 [4]；Patel 等证明 LLM agent **纯靠部署经验**自我提升、无需人工干预 [45]；Trabucco 等做「互联网规模」训练把规划行为泛化到海量任务分布、转向**终身适应** [71]；MindSearch 模拟人类搜索深化轨迹 [11]，SimpleDeepSearcher 合成多跳推理路径 [69]，Search-in-the-Chain 用中间检索结果**动态更新动作选择** [85]，WEPO 在 UI 元素级优化动作序列 [37]。一句话：**把规划当可进化的能力，强调持续自我改进与任务驱动适配**。

### 7.3 讨论：规划的强项与三大软肋（原文 §2.3）

- **强项**：① 涌现出**显式结构化输出**（推理链、搜索命令、子目标草图），帮 agent 避免短视/重复；② agent 开始**能学会并精化**规划策略（交互反馈 / 元学习 / 大规模任务分布）。
- **软肋（埋批判）**：① 当前 LLM 生成的计划**脆弱 (brittle)**，对含糊/欠定问题不鲁棒，即使有结构化格式，**计划内部一致性也无保证**，幻觉步骤会把错误向下游传播；② 规划模块的**评测粗糙**，多靠「终任务准确率」而非「计划质量本身」——难以诊断规划失败、难以比较策略；③ 多数系统**把每个研究问题当孤立问题**，不复用共享结构或可迁移策略，**积累不了通用规划知识**。

> 主讲提示：把「计划脆弱 + 评测只看终任务 + 不可迁移」这三条软肋记牢——它们在 §3.4、§5.3、§8 会以不同面孔反复出现，是全篇的「批判主旋律」。原文还点名两个解法方向：标准基准 DeepResearch Bench [16]、强化学习框架 DeepResearcher [98]。

## 8. 阶段二 · 提问 Question Developing（§3）：把子目标变成查询

### 8.1 定义与公式

> 直觉：有了子目标，不能只发**一句静态查询**就指望搜全。复杂研究要**动态生成更有针对性、上下文化、可分解**的查询，去「问得更聪明」——超越关键词匹配，走向任务感知的信息检索（原文 §3 引言）。

记号：$\mathcal{P}$ 计划、$s_i$ 当前子目标、$\mathcal{E}$ 累积证据、$\mathcal{M}^{\text{ask}}$ 提问模型（参数 $\theta$）。原文 **Def 3.1 / Eq.(2)**：

$$ \mathcal{Q}_i \;=\; \mathcal{M}^{\text{ask}}(\mathcal{P},\ s_i,\ \mathcal{E};\ \theta),\qquad \mathcal{Q}_i=\{q_{i,1},q_{i,2},\dots\}. $$

读出什么：每个子目标 $s_i$ 被转成**一组**查询 $\mathcal{Q}_i$（不是一条）；生成同时**条件于**整体计划 $\mathcal{P}$、当前子目标 $s_i$、**和已检索到的证据 $\mathcal{E}$**——正是 $\mathcal{E}$ 让提问能「随检索进展自适应」，产生 §5 那条 W→Q 回灌。（产生并更新 $\mathcal{E}$ 的检索过程见 §4。）

### 8.2 分类法（原文 Table 2）：按「怎么优化查询」分两族

| 一级类 | 二级子类 | 代表工作（原文编号）|
|--------|----------|------|
| **奖励优化方法**<br/>Reward-Optimized Methods | 面向格式与准确率的奖励 (rewards for format and accuracy) | DeepResearcher [98], EvolveSearch [93], R1-Searcher [66], Search-R1 [31], ZeroSearch [68], MaskSearch [80], DeepRetrieval [30] |
| | 多维奖励 (multi-dimensional reward) | InForage [51], OTC-PO [74], IKEA [28], AutoRefine [65], R-Search [97], MMSearch-R1 [77], VRAG-RL [75] |
| **监督驱动方法**<br/>Supervision-Driven Methods | 多智能体系统 (multi-agent systems) | ManuSearch [25], Search-o1 [35], SearchAgent-X [89] |
| | 监督优化 (supervision optimization) | ReasonRAG [95] |

**why 两族的根本权衡**（原文 §3.2 vs §3.3）：
- **奖励优化（RL）**：不要标注的「查询-响应」对，让 agent **试错探索**查询策略、按终任务结果（答案准确率、检索覆盖、效率）自适应「何时查、查什么」。
  - **单纯把奖励定义在「输出格式正确 + 最终答案准确」**（把检索与推理当黑盒）：简化奖励、易在噪声/模拟环境里规模化训练。DeepResearcher[98]/EvolveSearch[93] 用「二元格式奖励 + token 级 F1 答案奖励」经 GRPO 训多 agent 与真实/模拟 API 交互；R1-Searcher[66] 分阶段（先奖格式与搜索使用、再并入 F1 答案质量）；Search-R1[31] 强制 `<think><search><information><answer>` 模板、用 PPO/GRPO 优化精确匹配奖励；ZeroSearch[68] 用**学习出来的搜索模拟器**替真 API、**零真实查询成本**做课程式 RL；DeepRetrieval[30] 直接用 Recall@K、NDCG@K 当奖励对齐检索性能；MaskSearch[80] 用「掩码片段恢复」把格式与准确率合进统一奖励做预训练。
  - **多维奖励**（反映交互推理的复杂度，不止「答对没」，还看「怎么得到的」）：InForage[51] 加**信息增益** (information gain，对真值知识的覆盖) + **效率惩罚**（罚冗余推理跳数）；OTC-PO[74] 的**成本感知奖励**罚过度外部调用、保正确；IKEA[28] 的**知识边界感知奖励**——「简单题用内部知识答对」给正反馈、罚不必要的外部搜索；AutoRefine[65] 不只评最终答案、还评中间精化的完整度；R-Search[97]/MMSearch-R1[77] 把答案正确+证据质量+格式合规组合、支持跨模态结构化推理；VRAG-RL[75] 把这套搬到视觉-语言，用细粒度奖励优化「图选择/视觉注意一致/最终答案」的视觉搜索动作。
- **监督驱动（SFT / 规则 / 多agent）**：避开 RL 的复杂与不稳，用人类示范、任务启发式、或架构优化来引导查询策略——**更可控、更可解释、低资源更稳**。ManuSearch[25] 用透明开源的多 agent（规划子问题 / 外部检索 / 抽 HTML 证据各司其职）；Search-o1[35]、SearchAgent-X[89] 做系统级增强（自适应检索调度、高效批处理），不改底座模型行为；ReasonRAG[95] 用**蒙特卡洛树搜索 (MCTS)** 探推理轨迹 + **直接偏好优化 (DPO)** 排序，用更少样本学会「检索与推理如何交织」。

### 8.3 讨论：提问的三大失败模式（原文 §3.4）

- **过度依赖子目标清晰度**：一旦子目标含糊，查询质量**断崖式下降**，检索跑偏。
- **缺上下文连贯**：很多系统不会有效整合历史查询/任务背景 → 重复、冗余的问。
- **开放问题上太窄**：倾向生成过窄查询，**抓不住宽广的潜在信息**，限制探索性。

> 主讲提示：注意这三条与 §2.3 的「计划脆弱」是连锁的——**坏计划 → 坏子目标 → 坏查询**。这正好说明 §8（全篇结论）为什么把「跨段联合优化」列为头等大事：四段不能各自为政。

## 9. 阶段三 · 网络探索 Web Exploration（§4）：去网上把证据取回来

### 9.1 定义与公式

> 直觉：相关材料**稀疏地散落在无数网页**里。要么派个 agent 递归跟着超链接爬、过滤内容，要么调搜索引擎 API 拿排好序的结果——核心难点是「**准、且可信**」地定位与抽取最有用的内容（原文 §4 引言）。

记号：$\mathcal{R}$ 网络检索器、$\mathcal{Q}_i$ 查询、$\mathcal{H}$ 开放网络语料、$\mathcal{M}^{\text{web}}$ 网络 agent（参数 $\theta$）。原文 **Def 4.1 / Eq.(3)**：

$$ \mathcal{D} \;=\; \mathcal{M}^{\text{web}}(\mathcal{R},\ \mathcal{Q}_i,\ \mathcal{H};\ \theta). $$

读出什么：$\mathcal{D}$ 是「最终检索到的文档集合」。与传统 RAG 的**单次 (single-shot)** 检索不同，这里是**迭代、agent 驱动**的，能对稀疏/分散证据做更深覆盖（原文 §4 末）。

### 9.2 分类法（原文 Table 3）：按「信息来源」分两族

| 信息源 | 二级子类 | 代表工作 / 工具（原文编号）|
|--------|----------|------|
| **基于网页**<br/>Web-based | 网页抓取与爬取 (scraping & crawling) | Scrapy [59], BeautifulSoup [55] |
| | 浏览器型 web agent (browser-based) | WebGPT [41], Selenium [60] |
| | 多模态 web agent (multimodal) | WebVoyager [24], MM-ReAct [90], WebArena [101] |
| **基于 API**<br/>API-based | 工业搜索引擎 (industrial search engines) | Bing [1], X posts [3], Google [2] |
| | 领域专用搜索引擎 (domain-specific) | Reportify [54], YanXueZhiDe [12], CNKI [13], DuckSearch [67], BraveSearch [7], Bocha [6] |

**why 两族各有所长**（原文 §4.1 vs §4.2 vs §4.3 讨论）：
- **网页型 web 检索 agent**——从「简单抽取器」进化到「能导航复杂交互内容的多模态系统」：
  - **浏览器型自主 agent**：不同于静态爬虫，它**动态导航**网页界面、对「该探哪条路径」做上下文决策。WebGPT[41] 首创——把 HTML 转成结构化表示，让模型发 `Find keyword` 这类高层命令取相关段落；更进阶的用 Selenium[60] 直接查 DOM 或跑自定义 JS；Agent-E 构造**可访问性树 (accessibility tree)** 保留语义结构、去冗余元素。这类「靠文本表示感知 + 编程式执行动作」对文字密集任务有效，但**关键信息出现在视觉布局/内嵌可视化时就失灵**——这正是转向多模态的动机。
  - **多模态 web agent**：集成视觉感知，能读图表、识别界面模式、理解空间布局；对「分析数据可视化 / 在视觉复杂的研究数据库里导航」是刚需。两条路线：专门的提示框架（MM-ReAct[90] 把截图直接嵌进推理链、按视觉反馈迭代执行动作）、和完全集成的视觉-语言 agent（WebVoyager[24] 用 GPT-4V 把**截图分析 + HTML 处理**结合，在真实基准上拿到 **59% 任务成功率**，显著超纯文本基线）。Sightseer[34] 从截图重建 HTML/CSS；WebArena[101] 提供完整评测框架。
- **基于 API 的检索**：暴露标准化端点，把成熟工业搜索引擎**无缝**接进流水线，结果**可靠可信**。OpenAI DeepResearch[44] 用微软 Bing 的 web-search 基建发查询、抽段落、派生跟进查询、排候选文档、甚至跑沙箱代码；Grok DeepSearch[81] 自己跑爬虫 + 特权访问 X posts[3]；Gemini DeepResearch[18] 调 Google 自有搜索栈；Perplexity DeepResearch[46] 用「Bing 式 web 索引 + 自家 Sonar API（BM25/关键词/稠密向量重排）」的混合方案。再窄到领域：Reportify[54] 接持牌市场研报与权威财经新闻、YanXueZhiDe[12] 在 CNKI[13] 上做学术检索；开源框架则多依赖 DuckSearch[67]/BraveSearch[7]/Bocha[6] 等第三方引擎。

### 9.3 讨论：两族的死穴与「混合架构」未来（原文 §4.3）

- **网页型**：高度可适应交互/未索引内容，但**资源密集**、且**缺实时可信度评估**。
- **API 型**：快、可靠、拿预索引数据，但常**漏掉小众/动态内容**。
- **共同缺陷**：两者**孤立运作**，与上游「问题构造 + 证据规划」脱节，难以完全贴合具体研究需求。
- **未来 = 混合架构**：系统化整合两者长处——专门的证据抽取、正确性验证、内容质量评估模块；先「快速初检」再「深度交互分析」；多模态处理 + 实时验证。

> 主讲提示：这一节唯一的硬数字是 WebVoyager 的 **59% 任务成功率**（原文 §4.1，引自 [24]）。除此之外**全篇没有自己的实验数字**——这是综述的本性，组会上若被问「有没有横向跑分对比」，直说「原文未给出，它是定性分类型综述，量化对比留给 §7 的基准」。

## 10. 阶段四 · 报告生成 Report Generation（§5）：把碎片证据写成忠实长文

### 10.1 定义与公式

> 直觉：deep research 的产出不是「答一句」，而是一份**结构连贯、逻辑有序、忠于证据**的分析报告。挑战远超传统 QA：要多源融合、篇章级规划、任务特定格式（原文 §5 引言）。

记号：$q_0$ 初始问题、$\mathcal{P}$ 计划、$\mathcal{Q}$ 查询、$\mathcal{D}$ 检索文档、$\mathcal{M}_\theta$ 报告生成模型。原文 **Def 5.1 / Eq.(4)**：

$$ \mathcal{Y} \;=\; \mathcal{M}_\theta(q_0,\ \mathcal{P},\ \mathcal{Q},\ \mathcal{D}). $$

读出什么：$\mathcal{Y}$ 是最终报告，**条件于全链路**（原始问题 + 计划 + 查询 + 文档）。这意味着报告质量**天然耦合**上游所有阶段——这点在 §5.3 被反复强调为「孤立优化报告生成」的根本局限。

### 10.2 分类法（原文 Table 4）：结构控制 vs 事实完整性

报告生成被切成**两个正交目标**：**结构控制 (Structure Control)**（组织得好不好）与**事实完整性 (Factual Integrity)**（忠不忠于证据）。

| 一级类 | 二级子类 | 代表工作（原文编号）|
|--------|----------|------|
| **结构控制**<br/>Structure Control | 规划式生成 (planning-based) | Agent Laboratory [58], AI Scientist v2 [87], LongEval [78], LongWriter [5], LongDPO [49] |
| | 约束引导生成 (constraint-guided) | WebThinker [36], Suri [47], Wan et al. [73] |
| | 结构感知评估 (structure-aware evaluation) | Long2RAG [50], ExPerT [56], Long et al. [38], Kim et al. [33], Huang et al. [27] |
| **事实完整性**<br/>Factual Integrity | 忠实建模 (faithful modeling) | RAGSynth [63], BRIDGE [15], Zhou et al. [102], Shi et al. [64] |
| | 冲突推理 (conflict reasoning) | FaithfulRAG [94], DRAGged [8], Yuan et al. [92], Ying et al. [91] |
| | 事实性评估 (factuality evaluation) | Face4RAG [86], SFR-RAG [42], Wallat et al. [72], FaithJudge [70], RAG-QA Arena [23], MT-RAIG [61] |

**why 这两类必须同时抓**（原文 §5.1 vs §5.2）：
- **结构控制**：长文跨多节、动辄上千 token，必须规划 + 主题对齐 + 布局一致。三条路：
  - **规划式生成**：先用大纲/层级分解组织内容。Agent Laboratory[58] 做段落级规划 + 结构感知提示；**AI Scientist v2[87] 做递归树状规划维持全局布局一致**（注意：这正是本库的旗舰系统，被本综述归到「结构控制/规划式」一格）；LongEval[78]/LongWriter[5] 把生成分解成「高层大纲规划 + 节级合成」，跨千 token 维持连贯；LongDPO[49] 在每步加「批判增强监督」让局部补全对齐全局结构。
  - **约束引导生成**：解码时强制特定格式/风格/内容要求。WebThinker[36] 用「节感知分解」桥接规划与生成；Suri[47] 研究多重布局/风格约束下的指令微调；Wan 等[73] 提「类人修订」框架模拟约束下真实写作动态。
  - **结构感知评估**：给训练与事后对齐提供信号。Long2RAG[50] 用关键点召回 (key-point recall) 评「检索证据是否被充分纳入长答案」；ExPerT[56] 评布局特定的完整性与可解释性；Kim 等[33] 在长上下文里测**逐字保真 (verbatim fidelity)** 以发现结构漂移；Huang 等[27] 提校准方法纠正输出的结构不一致。
- **事实完整性**：RAG 时代仍难保证「忠于证据」。三条路：
  - **忠实建模**：RAGSynth[63] 在已知事实变体下造合成监督信号提升鲁棒与可信；BRIDGE[15] 在检索与生成间插「验证层」评事实充分性；Shi 等[64]、Zhou 等[102] 用**上下文感知解码**优先采信高置信证据片段、减幻觉。
  - **冲突推理**（检索来源自相矛盾时）：FaithfulRAG[94] 做**事实级冲突建模**促与一致事实对齐；DRAGged[8] 用**基于熵的解码 + 干预**识别并缓解源间冲突、按证据不确定性自适应；Ying 等[91] 研究 LLM 在冲突提示下的行为、靠行为微调提升冲突意识。
  - **事实性评估**：Face4RAG[86] 用细粒度指标评中文 RAG 的归因一致性；SFR-RAG[42] 在段落级测对齐；Wallat 等[72] 区分「表面正确」与「真源忠实」；FaithJudge[70]、RAG-QA Arena[23] 建**演进式排行榜**、MT-RAIG[61] 把事实性评估扩到**多表 (multi-table)** 结构化数据洞察。

### 10.3 讨论：报告生成仍处早期（原文 §5.3）

- **孤立优化**：多数方法只攻报告生成的某个子技能，**缺与上游（规划/提问/网络探索）的联合优化**——而有效报告合成本质上耦合上游，这个脱节是**主要局限**。
- **结构僵化**：结构控制常依赖固定大纲/静态规划，缺乏对任务复杂度、证据分布、推理流的灵活适配。
- **一致性短板**：少数方法增强局部 grounding，但**极少能跨多文档/多跳建模一致性**——而科研场景恰恰是长输入 + 冲突来源，这是最痛的缺口。

> 主讲提示：本节是全篇最有「批判味」的一节。一句话立 flag——**「报告生成是 deep research 真正的硬骨头，也是被评测最少的环节」**（接 §11 的 Table 5：只有 2/11 基准评它）。这把「分类法」与「评测缺口」缝在了一起。

## 11. 优化与评测（§6–§7）：单/多 agent、三种训练、基准覆盖矩阵

### 11.1 工作流：单 agent vs 多 agent（原文 §6.1）

> why：四段能力既可塞进一个模型端到端学，也可拆给多个 agent 分工。这是**正交于分类法的实现维度**。

- **单 agent 系统**：一个 LLM 全包——任务分解、查询生成、检索（工具/API）、最终合成全在同一模型内部管理，常**端到端用 RL 训**。代表：DeepResearcher[98]、WebThinker[36]、Search-R1[31]（单体顺序工作流）。
- **多 agent 系统**：把流水线拆给专职 agent——规划 agent 管分解与子目标调度、查询 agent 管多样化上下文查询、检索 agent 管外部工具取证、写作 agent 管结构化合成。代表：AgentRxiv[57]、AI Scientist[39]、OpenResearcher[99]。好处：**每个组件可独立专门化与优化，便于并行与灵活**。

### 11.2 参数优化：三种训练范式（原文 §6.2）

| 训练范式 | 直觉（在解决什么）| 代表 |
|----------|------|------|
| **对比学习 (contrastive learning)** | 用「成功 vs 失败轨迹」对比，教 agent 区分有效/无效行为（尤其工具使用：检索/搜索/摘要），提升多步推理精度 | Avatar [79] |
| **强化学习 (reinforcement learning)** | 用「检索准确率 + 终产物质量」奖励，微调长程规划与决策 | Search-O1 [35], Learning-to-Search [9] |
| **课程训练 (curriculum training)** | 分阶段：先学基本技能、再上完整工作流编排，应对渐进复杂任务 | AI Scientist-v2 [87], SimpleDeepSearcher [69] |

**讨论（§6.2 末）**：优化常与架构选择互补——**单 agent 偏用 RL 做端到端奖励对齐，多 agent 更受益于模块化训练 + agent 级反馈**；AgentLab[58] 等开始探索「符号规划 + 搜索增强 + 人在环反馈」的**混合优化**。

### 11.3 基准覆盖矩阵（原文 Table 5）——本篇评测部分的灵魂

> why：要诊断「四段能力哪段弱」，得有能**分模块**评测的基准。作者把 11 个主流基准按「覆盖哪几段（P/QD/WE/RG）」列成矩阵，一眼看出**报告生成被系统性回避**。

P=Planning、QD=Question Developing、WE=Web Exploration、RG=Report Generation。原文 Table 5：

| 基准 | P | QD | WE | RG | 任务 | 评测指标 |
|------|---|----|----|----|------|----------|
| Mind2Web 2 [19] | ✓ | ✓ | ✓ | ✗ | Web search | Success rate, Partial Completion |
| BrowseComp [76] | ✓ | ✓ | ✓ | ✗ | Web search | Accuracy, Calibration Error |
| WebArena [101] | ✓ | ✓ | ✓ | ✗ | Web search | Success Rate |
| GAIA [40] | ✓ | ✓ | ✓ | ✗ | Multi-step assistant tasks | EM |
| Humanity's Last Exam [48] | ✓ | ✓ | ✓ | ✗ | Multidomain reasoning | Accuracy, Calibration Error |
| BrowseComp-Zh [100] | ✓ | ✓ | ✓ | ✗ | Web search in Chinese | Accuracy, Calibration Error |
| MedBrowseComp [10] | ✓ | ✓ | ✓ | ✗ | Medical web search | Accuracy |
| GPQA [53] | ✓ | ✓ | ✓ | ✗ | Graduate-level QA | Accuracy |
| InfoDeepSeek [82] | ✗ | ✓ | ✓ | ✗ | Open-domain QA | Accuracy, Information Accuracy |
| **DeepResearch Bench [16]** | ✓ | ✓ | ✓ | **✓** | Research report generation | Pairwise Agreement Rate, Overall Pearson Correlation |
| **DeepResearchGym [14]** | ✓ | ✓ | ✓ | **✓** | Research task sandbox | KPR, KPC, Precision, Recall, Clarity, Insight |

**两类基准（原文 §7）**：
- **搜索导向 (search-oriented)**：聚焦信息检索（WE + QD），常在交互/语言特定浏览场景。Mind2Web 2 评「重构查询 + 动态导航网页」；BrowseComp/BrowseComp-Zh 用准确率 + 校准误差评多语种浏览；WebArena 加了需要简单 Planning 的结构化任务。**这些基准很少涉及最终内容生成 → 报告生成被低估**。
- **研究导向 (research-oriented)**：超越搜索，评长程推理、综合、结构化产出，需**四模块全链路协调**。DeepResearch Bench[16] 与 DeepResearchGym[14] 是代表，从 Planning 子任务分解 → QD 子查询 → 实时 WE → 最终研究式报告，全覆盖，并给细粒度指标（clarity、知识精确/召回 KPR/KPC、agreement）。MedBrowseComp[10] 把这条线专门化到生物医学；GAIA[40] 评通用助手任务的全模块工作流；而 Humanity's Last Exam[48]、GPQA[53] 强调「最小可检索性」下的 Planning + QD（模拟开卷但不可搜的 QA）。

> 主讲提示：这张矩阵是整篇评测章的「金句图」。**RG 列只有两个 ✓**——把它放大讲：deep research 号称要产「报告」，但 9/11 主流基准只评「搜得准不准」，不评「写得好不好、忠不忠实」。这就是 §5.3「报告生成处早期」的量化注脚，也是组会最易引发讨论的点。

## 12. 局限与未来方向（§8）：五个还没解决的硬骨头

作者在 §8 诚实列出五大开放挑战（这是综述给后人的「待办清单」）：

1. **多工具集成 (Multi-Tool Integration)**：现有系统**几乎只把传统搜索引擎当唯一外部工具**，严重限制知识源。真实研究要查 API、解析结构化数据库、导航代码仓库、从文档/表格/图表取信息。未来 agent 必须能在**多个异构工具上动态编排**、每个推理步灵活决定调哪个。（注：本篇仅点到方向；MCP 等机制的细化见前作 [26]。）
2. **事实性 (Factuality)**：多源合成/多步推理时易引入事实错误、过时断言、无依据论断，尤其聚合不一致来源时。需显式 grounding（源归因）、事实性感知奖励、事后验证模块。
3. **多模态推理 (Multimodal Reasoning)**：当前流水线**几乎纯文本**，不适合科学/医学/工程里需视觉理解的任务（图像、图表、扫描文档）。让 deep research 处理图像/PDF/结构化数据**基本还是空白但关键**。
4. **工作流设计与模型优化 (Workflow Design & Model Optimization)**：早期靠静态提示 + 硬编码工作流；近期转向自适应/agentic，但**仍缺「跨任务泛化、随新目标演化」的工作流学习机制**；优化侧因高算力成本 + 稀疏奖励难对齐。方向：参数高效微调、课程/分阶段预训练、人/自动反馈引导的 RL。
5. **个性化 (Personalization)**：现有系统缺持久用户建模与动态适配，常把个性化当次要。需可扩展、隐私感知、持续学习的用户模型，并应对过拟合与公平性。

> 主讲提示：把这五条与四段流水线对应着讲——多工具/多模态打的是「网络探索 §4」的源与模态、事实性打的是「报告生成 §5」、工作流优化打的是「规划 §2 + 跨段联合」、个性化是横切全链路的新维度。**它们共同的潜台词是同一句：四段不能各自为政，要联合优化 + 拓宽模态与工具。**

## 16. 局限与批判（诚实 —— 本课的灵魂）

**原文自陈（散见各 §x.3 讨论 + §8）**：规划脆弱、提问过窄、网络探索两族脱节、报告生成孤立优化、五大开放挑战未解（已在 §7/§12 详列）。

**作为综述方法论本身的可批判处（社区视角，原文未必承认）**：
1. **无量化横评**：全篇靠**定性归类**，几乎没有「系统 A 在基准 B 上得分 X」的硬数据（唯一数字 WebVoyager 59% 还是引用来的）。所以它能告诉你「有哪些方法、各属哪类」，但**不能告诉你「哪个真的更好」**。组会引用须守住这条边界。
2. **分类边界有重叠**：同一系统被放进多格（如 AI Scientist v2、Agent Laboratory、MPO[83] 同时出现在多张表/多类），说明「四段 + 各自二分」的切法**并非互斥穷尽 (MECE)**，更像「视角」而非「硬分区」。
3. **盲点：记忆与多轮状态**：四段流水线**几乎不建模「跨步记忆 / 长程状态管理」**——而这恰是更晚的 2512.02038 单列一章补的（§15）。对长程 deep research，这是结构性缺失。
4. **时效与覆盖**：v1 截至 2025-08，引文以 2024–2025 arXiv 预印本为主（大量未经同行评审），分类法可能随领域快速演化而过时；作者也未承诺像 2512.02038 那样持续更新。
5. **「能力可分解」是强假设**：把研究干净切成四段，可能**低估了四段间的强耦合**——而原文自己在 §3.4/§5.3 又反复说「子目标坏则查询坏」「报告耦合上游」，某种程度上**自承了分解的局限**。

> 主讲提示：把第 1 条（无横评）和第 5 条（分解是强假设、且原文自承耦合）当批判主线。一句话——**「它是一张好地图，但地图不是领土：能定位方法，不能替你跑分。」**

## 17. 组会讨论问题（5–8 个，可引发讨论）

1. 本篇把 deep research 切成「规划→提问→网络探索→报告生成」四段，且自承「子目标坏则查询坏、报告耦合上游」。那么**四段到底该联合优化还是分段优化**？若联合，奖励/监督信号怎么设计才能不让某段塌掉？（接 §3.4 / §5.3 / §8）
2. Table 5 里 **9/11 基准不评报告生成**。这是「报告质量难自动评」的客观困难，还是社区「挑容易刷的搜索分」的惰性？要让 DeepResearch Bench / DeepResearchGym 成为主流，缺什么？
3. 提问阶段的「奖励优化 vs 监督驱动」对应「探索更广但不稳 vs 可控但可能过窄」。**对一个面向科研的高风险场景，你押哪边？**IKEA[28] 的「知识边界感知奖励」（简单题用内部知识、罚不必要外搜）是不是更优雅的折中？
4. 本篇几乎不谈「跨步记忆」，而 2512.02038 把「记忆管理」单列一等组件。**长程 deep research 真的需要显式记忆模块吗？**还是足够长的上下文 + 证据集 $\mathcal{E}$ 回灌就够了？
5. 网络探索从「文本浏览器型」走向「多模态」（WebVoyager 59%）。但多模态又贵又不稳。**在「能读图表」和「可靠/可负担」之间，deep research 的甜点在哪？**API 型 + 混合架构（§4.3）是答案吗？
6. 本篇与 [26]、2512.02038 的分工是「体检表 vs 蓝图 vs 最新版+内存条」。**如果让你写第四篇 deep research 综述，你会选哪条新主轴**（如「安全/对齐」「成本-效益」「人机协同」）才不重复这三家？
7. 综述全程无量化横评。**该不该要求 deep research 综述附一个统一 leaderboard？**还是说「定性分类 + 指向基准」已是综述应有的边界，跑分是基准论文的活？

> 主讲提示：1、2、4 是最容易擦出火花的三问——分别戳「分解 vs 联合」「评测缺口」「记忆盲点」，正好是本篇三处最深的张力。

## 18. 一页速记（汇报当天速览）

- **是什么**：以「能力为中心」的 deep research（自主研究 agent）综述；把流水线拆成**四段**：规划 Planning（§2）→ 提问 Question Developing（§3）→ 网络探索 Web Exploration（§4）→ 报告生成 Report Generation（§5），逐段给定义、分类表、代表系统、基准。
- **四个定义式**：Eq.(1) $\mathcal{P}=\mathcal{M}^{\text{plan}}(q_0,\mathcal{K};\theta)$ 拆计划；Eq.(2) $\mathcal{Q}_i=\mathcal{M}^{\text{ask}}(\mathcal{P},s_i,\mathcal{E};\theta)$ 子目标→查询（含证据回灌）；Eq.(3) $\mathcal{D}=\mathcal{M}^{\text{web}}(\mathcal{R},\mathcal{Q}_i,\mathcal{H};\theta)$ 迭代取文档；Eq.(4) $\mathcal{Y}=\mathcal{M}_\theta(q_0,\mathcal{P},\mathcal{Q},\mathcal{D})$ 合成报告（耦合全链路）。
- **四张分类表的主轴**：规划=结构化世界知识 / 可学习过程（Table 1）；提问=奖励优化 / 监督驱动（Table 2）；网络探索=网页 / API（Table 3）；报告生成=结构控制 / 事实完整性（Table 4）。
- **评测关键图**：Table 5 的「P/QD/WE/RG 覆盖矩阵」——**11 基准里仅 DeepResearch Bench[16]、DeepResearchGym[14] 评到报告生成**（其余只评搜索）。
- **唯一硬数字**：WebVoyager 真实基准 **59% 任务成功率**（§4.1，引用 [24]）；其余全为定性分类，**无自做横评**。
- **三家分工**：[26]（2506.18096）给系统架构蓝图；**本篇给四段能力体检表 + 评测矩阵**；2512.02038 给最新全景且补「记忆管理」组件。
- **五大开放挑战（§8）**：多工具集成 / 事实性 / 多模态推理 / 工作流设计与优化 / 个性化——共同潜台词：**四段别各自为政，要联合优化 + 拓宽工具与模态**。
- **一句话结论**：**它不证明任何系统更强，而是给整条 deep research 赛道画了一张「四段能力 × 评测覆盖」的坐标系——能定位瓶颈，不能替你跑分。**

> 主讲提示：结尾回到一句钩子——**「读完这篇，你不会多会写一个 agent，但你第一次知道该问它哪四个问题、该用哪两个基准、还差哪五块拼图。」**
