# 组会汇报 · SciCode：科学家出题的科研代码基准

> 主讲提示：这是 E 组（评测）里最「硬核」的一篇——它不是用模型生成题，而是请 **16 个细分领域的真科学家**，把自己日常用的科研代码改写成题。一句话定调：**当模型已经超过普通人，要测它「难」，就得让真专家从真科研里出题；而出题方式不是一道大题，而是「主问题→子问题」的可验证分解。**

---

## 1. 封面 · TL;DR

- **作者/出处**：Minyang Tian, Luyu Gao, Shizhuo Dylan Zhang, Xinan Chen 等 30+ 作者（UIUC / Argonne 国家实验室 / CMU / MIT / Harvard / U Chicago / Princeton / Stanford …），2024-07-18，arXiv 2407.13168 v1，标注 "Preprint. Under review."。数据/代码/榜单：`https://scicode-bench.github.io/`。
- **一段话**：随着语言模型 (Language Model, LM) 在许多任务上已超过普通人，「出难题」变得越来越难，且很多既有基准已被刷穿 (saturated)。SciCode 的解法是请**自然科学家**把他们**日常科研工作流里真实用到的代码**改写成题：每道**主问题 (main problem)** 被分解成多个更小、更易处理的**子问题 (subproblem)**，每个子问题要求实现一个 Python 函数，再把这些函数**整合 (integrate)** 成主问题的完整解。全集含 **80 道主问题、338 个子问题**，覆盖**数学、物理、化学、生物、材料**五大领域共 **16 个细分领域 (subfield)**；每题都配**科学家标注的金标准解 (gold-standard solution)** 与**测试用例**，可自动评测；并提供「**带/不带科学背景知识**」「**用金标准/用模型生成的前序子问题解**」等多种可切换设置。
- **三条带走的结论**：
  1. **真难**：在最贴近科学家真实用法的「标准设置」下（不给背景、前序子问题用模型自己生成的解），表现最好的 Claude3.5-Sonnet **只解出 4.6% 的主问题**；Claude3-Opus、GPT-4o 只有 **1.5%**；最好的开源模型 DeepSeek-Coder-v2 也只有 **3.1%**（见 §3.2 / 原文 Table 2）。
  2. **背景知识有用但远不够**：把科学家写的背景知识喂给模型后，所有模型都**一致且显著**提升，但即便如此，最好的模型主问题也只到 **12.3%**（原文 §1 / Table 3）——说明短板既有「科学知识」也有「代码整合」。
  3. **子问题 ≫ 主问题**：子问题级 pass@1 普遍在 **14%–26%**，但主问题级骤降到个位数——**模型能零散解对一些子问题，却几乎无法把它们整合成一个完整、正确的主问题解**（这是 SciCode 最核心的难度信号）。

> 主讲提示：开场就把「4.6% 这个数字」抛出来，并立刻解释它来自**哪种设置**（标准设置 = 不给背景 + 前序解自己生成）。整篇汇报的重点就是：**这道分数为什么这么低、低在哪一环。**

## 2. 问题与动机（why —— 本篇最该讲透的一节）

**缺口一：模型太强，旧基准被刷穿。** 原文开篇 (§1) 指出，评测与模型是**协同进化**的，好的基准极大推动了模型进步。但因为 LM 现在在很多任务上**已超过除领域专家外的多数人**，许多成熟基准**很快饱和**，导致「模型被感知的能力」与「真实能力」之间出现**偏差 (discrepancy)**。要继续推进模型，就必须造出**真实、高质量、且有挑战性**的评测。

**缺口二：合成难题不够「真」。** 一条流行的应对路线是**用模型来构造评测样本**（从已有基准里挑模型解不出的子集、或让模型扩写出更难的题）。但原文明确质疑：这种做法**是否真能反映真实世界应用与模型在实际场景中的表现，尚不清楚**。换言之，「模型自己出的难题」可能难得不真实、不对口。

**这篇的赌注（核心动机）**：与其让模型出题，不如让**真科学家**出题——题目直接来自他们**日常科研工作流**里反复用、且很多已用于已发表论文的代码。这样得到的题：(a) **真实**（对得上真实科研需求）；(b) **高质量**（至少两名同领域资深研究者交叉验证）；(c) **抗污染**（这些代码大多只在实验室内部使用、很少开源、标注差，因此在当前 LM 训练数据里**曝光极少**，原文称其与公开数据集**零重叠**，§2.1）。

**为什么要「分解成子问题」而不是出一道大题？** 这是本篇最关键的设计直觉，原文 §2.2 给了科学动机：科学家在真实工作流里**本就习惯**把一个复杂问题拆成若干更小、更可控的部分，**为每一部分写相对独立的代码**，再整合成对主问题的完整解。SciCode 把这种**天然的分解结构**标准化下来。这样做的好处是：
- **能更细粒度地评测**：既能看主问题成败，也能逐个子问题看模型「卡在哪一步」；
- **可验证**：每个子问题都有金标准解和测试用例，能可靠地自动判分；
- **可调难度**：可以选择给模型**前序子问题的金标准解**（少累积误差）还是**模型自己生成的解**（更真实但误差会累积）。

**为什么现在做、不做会怎样**：若继续用饱和基准，我们会**高估**模型的科研能力，把「会写通用代码」误当成「能做科研」。SciCode 想给出一把**贴着真实科研、且短期内刷不穿**的尺子，并进一步**激励「用 AI 加速科学研究」这一方向的方法研究**（原文 §1 结尾：这一方向因缺乏商业激励而受益于 LM 进步较少）。

> 主讲提示：这一节是 why 的核心。三句话讲透——**①旧基准饱和→要新难题；②模型自造难题不够真→请真科学家从真科研出题；③出题方式是「主问题→子问题」的可验证分解，既真又能细粒度判分还能调难度。**

## 3. 研究问题 / 核心 intention（形式化成一句话）

把要解决的问题压成一句：

> **能否构造一个「由科学家从真实科研代码出题、把复杂主问题分解为可验证子问题」的代码生成基准，使它同时满足「真实、高质量、抗数据污染、对前沿模型足够难」，并能在多种设置下细粒度地诊断模型的科学知识与代码整合能力？**

它隐含的**假设**：
- (a) 科学家日常科研代码所对应的任务，**比通用编程题更需要深层科学知识 + 推理 + 把复杂问题分解再整合的综合能力**（原文称 SciCode 测的是「comprehensive and all-around capabilities」，§2.1）；
- (b) 这些代码因「内部使用、少开源、标注差」而在 LM 训练数据中**曝光低**，故能近似规避数据污染（§2.1，脚注 2 还补充：除抗污染外，**很多题对最好的模型都太难**，所以常**简化设定、补更多背景**）；
- (c) **分解 + 金标准 + 测试用例**这套结构，足以把「主观的科研能力」转化为**可自动、可靠判定**的 pass/fail。

## 5. 方法总览（big picture，先直觉后数学）

SciCode 不是「模型 pipeline」，而是「**数据构造 + 评测协议**」。整体可看作三段标注流程 + 一套可切换评测设置（原文 §2.3 / §2.4）：

```mermaid
flowchart TB
  subgraph BUILD[数据构造 §2.3：每题过三关]
    P1[① 问题选择<br/>科学家从真实科研挑<br/>需重知识+推理的题] --> P2[② 评测设计<br/>写金标准解(只用 NumPy/SciPy/SymPy)<br/>+ 数值测试 + 领域专属测试]
    P2 --> P3[③ 三轮验证<br/>同域2人交叉 + 跨域1人审 + GPT-4 试解纠偏]
  end
  subgraph STRUCT[结构 §2.2：主问题→子问题]
    M[主问题 main problem<br/>含 docstring 规定输入输出] --> S1[子问题1<br/>实现1个函数]
    M --> S2[子问题2 ...]
    M --> Sk[子问题k<br/>可选附背景知识]
    S1 & S2 & Sk --> INT[整合 integrate 成主问题完整解]
  end
  BUILD --> STRUCT
  STRUCT --> EVAL[评测 §2.4：多设置可切换]
  EVAL --> E1[带/不带 科学背景]
  EVAL --> E2[前序子问题用 金标准/模型生成 解]
  EVAL --> E3[在 主问题级/子问题级 判分]
```

**直觉**：把「做一段科研代码」想成博士生的日常——先把大任务拆成几步，每步写个函数（必要时查背景知识），再拼起来跑通、对上已发表的数值/物理结果。SciCode 把这套流程**冻结成题**：每一步都有标准答案和测试，于是「能不能做科研代码」第一次变得**可自动、可逐步判分**。原文 Figure 1 给了一个 Haldane 模型 Chern 数的例子：主问题 = 「扫参数生成 Chern 数阵列」，被拆成「写哈密顿量 → 算 Chern 数 → 扫参数成阵列」三个子问题。

## 6. 符号与术语表（后文统一用）

| 记号 / 术语 | 含义（中英对照） |
|------------|------|
| 主问题 (main problem) | 一个完整科研编码任务，定义总体目标，docstring 规定输入/输出格式（§2.2） |
| 子问题 (subproblem) | 从主问题分解出的更小任务，要求实现**一个** Python 函数；可选附背景知识 |
| 整合 (integrate) | 把各子问题函数拼成主问题的完整解 |
| 金标准解 (gold-standard solution) | 科学家写的、只用 NumPy/SciPy/SymPy 等成熟库的参考实现 |
| 数值测试 (numerical test) | 给定输入-输出对，检查生成代码输出是否与 ground truth 一致 |
| 领域专属测试 (domain-specific test case) | 额外阶段引入，检查解是否**重现已发表结果 / 符合解析推导**（如相变温度、表面等离激元色散） |
| 背景知识 (scientific background) | 科学家为某子问题写的、解题所需的知识与推理步骤；可选「带/不带」 |
| 标准设置 (standard setup) | **不给背景** + 前序子问题用**模型自己生成**的解（最贴近真实用法，§2.4） |
| pass@1 | 单次生成（采样 1 个解）即通过全部测试的比例（本文主指标） |
| $N_{\text{main}}, N_{\text{sub}}$ | 主问题数 (80)、子问题数 (338) |
| 开发/测试划分 | 15 主问题(50 子问题) 作开发 (development) split；65 主问题(288 子问题) 作测试 (test)（§2.1） |

## 7. 方法细节 ① 数据来源与领域覆盖（setting 之一）

**why 选「真实科研代码」**：原文 §2.1 强调，这些代码**大多用于一篇或多篇已发表论文**，因此其**正确性与稳健性已被验证**；但它们**主要是内部使用、很少开源、标注也差**，于是在通用 LM 的训练数据里**曝光少**——这给了一个**评测模型「向陌生情境泛化」**的独特机会。

**覆盖了哪些学科（原文 Table 1，5 大领域 / 16 子领域 / 各自主问题数）**：

| 领域 (Field) | 子领域 (Subfields) 及主问题数 |
|------|------|
| 数学 Mathematics | 数值线性代数 (8)、计算力学 (5)、计算金融 (1) |
| 物理 Physics | 凝聚态物理 (13)、光学 (10)、量子信息/计算 (6)、计算物理 (5)、天体物理 (2)、粒子物理 (1) |
| 化学 Chemistry | 量子化学 (5)、计算化学 (3) |
| 生物 Biology | 生态学 (6)、生物化学 (1)、遗传学 (1) |
| 材料 Material Science | 半导体材料 (7)、分子建模 (6) |

**分布（原文 Figure 2）**：按主问题，**物理 46% / 数学 18% / 材料 16% / 化学 10% / 生物 10%**；按子问题，**物理 49% / 材料 17% / 化学 16% / 数学 10% / 生物 8%**。**物理占比最大**（凝聚态 13 道为单子领域之最）。

**规模与划分（§2.1）**：每道主问题子问题数**中位数 3、最多 15**；保留 **15 道主问题（50 子问题）作开发集**、其余 **65 道（288 子问题）作测试集**；15 道开发题**覆盖全部五个领域**，且其中过半各自**少于 4 个子问题**（便于 few-shot 演示）。

**为什么含「诺奖方法」**：原文 §2.3.1 特意纳入若干**复现诺奖工作**的题，以体现前沿科研趋势：
- **自洽场 (SCF) 方法**用于密度泛函理论 (DFT)（1998 化学诺奖）；
- **PMNS 矩阵**用于物质中的中微子振荡（2015 物理诺奖）；
- **Haldane 模型**用于反常量子霍尔效应（2016 物理诺奖，即 Figure 1 / 附录 A.3 的例子）；
- **光镊 (optical tweezer)** 模拟用于微观热力学（2018 物理诺奖）；
- **replica 方法**用于自旋玻璃（2021 物理诺奖）。

> 主讲提示：强调两点——①题目「**真**」到能复现诺奖结果；②**物理近半**，所以这把尺子对「物理/材料类科研代码」信号最强，对生物（仅 8% 子问题）覆盖相对薄。

## 8. 方法细节 ② 主问题→子问题的分解结构（本篇重点之一）

**why 这么分**：§2.2 的科学动机已在第 2 节讲过——科学家本就这么干。这里看**标准化后的格式**：

- **主问题 (Main Problem)**：定义研究的总体目标、统领所有子问题；用一个 **docstring 块**写清**要求的输入**与**期望的输出**。主问题定义清楚后，科学家就有足够指引去解题。
- **子问题分解 (Subproblem Decomposition)**：子问题聚焦**从主问题派生**出来的问题，把复杂主问题拆成更小、更可控的部分；每个子问题有**详细 docstring** 描述所需输入与期望输出，保证清晰、利于精确代码生成。这种结构化分解**简化解题**，并支持对模型科学编码能力做**更细粒度**的评测。

**以原文 Figure 1 / 附录 A.3 的 Haldane 例子看分解长什么样**（这是理解全篇的「标尺样例」）：

> 主问题 (A.3.1)：「为六角晶格上的 Haldane 模型，通过扫描 (on-site 能量 / 次近邻耦合比 $m/t_2$) 和相位 $\phi$，生成一个 Chern 数阵列。」给定晶格常数 $a$、近邻耦合 $t_1$、次近邻耦合 $t_2$、布里渊区离散步长 $\delta$、扫描点数 $N$。函数签名 `compute_chern_number_grid(delta, a, t1, t2, N)`，返回 `(N,N)` 的 Chern 数阵列。依赖只允许 `numpy / cmath / from math import pi,sin,cos,sqrt`。

它被拆成三个子问题（A.3.2）：
1. **子问题 1**：`calc_hamiltonian(kx,ky,a,t1,t2,phi,m)`——写出六角晶格 Haldane 哈密顿量（$2\times2$ 矩阵）。
2. **子问题 2**：`compute_chern_number(delta,a,t1,t2,phi,m)`——用上一步的哈密顿量，在离散布里渊区上算 Chern 数。
3. **子问题 3**：`compute_chern_number_grid(delta,a,t1,t2,N)`——扫 $m/t_2\in[-6,6]$、$\phi\in[-\pi,\pi]$ 各 $N$ 点，拼成 2D Chern 数阵列（= 主问题）。

**子问题 1 的「带背景」长什么样（原文 Table 6）**：科学家会把**解题必需的物理**写进背景，例如给出 B 格点到三个近邻 A 格点的矢量 $\{\mathbf a_i\}$、次近邻矢量 $\{\mathbf b_i\}$，以及哈密顿量的 Pauli 矩阵展开。直觉：先告诉你「这道题需要的物理公式」，再让你把它翻译成代码。

记号（先定义，后用式；均取自原文 Table 6）：
- $\mathbf k=(k_x,k_y)$：动量空间波矢；$a$：晶格常数；$t_1,t_2$：近邻/次近邻耦合；$\phi$：次近邻跃迁相位；$m$：on-site 能量；
- $\{\mathbf a_i\}$：B 格点到三个近邻 A 格点的矢量；$\{\mathbf b_i\}$：次近邻距离矢量；
- $\sigma_1,\sigma_2,\sigma_3$：Pauli 矩阵；$I$：单位矩阵；$d_0,d_1,d_2,d_3$：哈密顿量在 $\{I,\sigma_i\}$ 上的展开系数。

$$ H(\mathbf k)=d_0 I + d_1\sigma_1 + d_2\sigma_2 + d_3\sigma_3 $$

其中（原文 Table 6 给出全式，此处示意前两项）
$$ d_0 = 2t_2\cos\phi\sum_i \cos(\mathbf k\cdot \mathbf b_i),\qquad d_1 = t_1\sum_i \cos(\mathbf k\cdot \mathbf a_i),\quad\dots $$

读出什么：**「带背景」= 把这套展开式直接给模型**，于是题目从「考科学知识 + 代码」退化为更偏「**照着公式写对代码**」；**「不带背景」**则要求模型**自己回忆出**这套物理再写代码——这正是 §2.4 两种设置的本质区别。

> 主讲提示：用这个 Haldane 例子把「分解」讲活——主问题=最终目标函数，子问题=一步一个函数，背景=可选的「公式小抄」。强调**子问题 3 直接就是主问题函数**，所以「整合」体现在「后一步必须正确调用前几步」。

## 9. 方法细节 ③ 数据标注三关：质量与抗污染怎么保证（why 可信）

**why**：基准的生命线是**正确**与**不可刷**。原文 §2.3 用**三阶段、共三轮验证**来加固：

**关 ①（问题选择, §2.3.1）**：科学家从研究周期里**会真实出现**的编码需求挑题（数据处理、拟合、绘图…），并**专挑需要密集科学知识与推理**的，以同时覆盖前沿科研的**广度与深度**。重点选三类：
- **数值方法 (Numerical methods)**：复杂系统拿不到解析解，需推导数值模型/算法；
- **系统模拟 (Simulation of systems)**：基于理论原理与经验数据写模拟，反映对系统的深刻洞察；
- **科学计算 (Scientific calculation)**：数据后处理/可视化时，按科学公式做大量变换以得到感兴趣的物理可观测量。

**关 ②（评测设计, §2.3.2）**：标注者写金标准解时**只用广泛采用、文档完善的库**（NumPy / SciPy / SymPy；原文 Figure 4 实测：**numpy ~100%** 的主问题用到，scipy ~35%，其后是 itertools / math / cmath / sympy）。测试套件有**两部分**：
1. **数值测试**：列输入-输出对，比对生成代码输出与 ground truth；
2. **领域专属测试**（额外阶段）：检验解是否**符合科学家的实际需求**、能否在各自领域里**重现已发表论文结果或解析推导**。原文给的实例：2D 方格 Ising 模型在 $kT/J\approx 2.269$ 处出现相变、推导 2D 层状电子气的表面等离激元模、验证光镊中布朗粒子的弹道运动等。**目的是验证代码不仅能跑，且确实刻画了底层科学问题。**

**关 ③（问题验证/质控, §2.3.3）—— 三轮**：
1. **域内验证 (In-domain)**：至少两名同领域科学家交叉检查**题面设计、解代码、领域测试**，并据反馈迭代；
2. **跨域验证 (Out-of-domain)**：一名其他领域科学家审**题面**，确认信息**精确且充分**到能解题（如所有科学常数都给齐），以发现「圈外人看不懂的隐含假设」；
3. **GPT-4 验证**：把已验证子问题喂 GPT-4 生成解，科学家**对其错误做分析**，必要时**重设计数值测试以防假阳性 (false positive)**；并据 GPT-4 的解再**第三次修订整套工作流**消歧义。

> 主讲提示：这三关是「为什么我信这套题」的答案。特别点出**关③用 GPT-4 反向找漏**——不是用模型出题，而是用模型**给测试用例挑毛病、防假阳性**，这是它和「模型造难题」路线的关键区别。脚注 2 也很诚实：很多题对最好模型都太难，于是**主动简化、补背景**，所以分数低**不等于**题出歪了。

## 10. 方法细节 ④ 多种评测设置（本篇重点之二：四组开关）

**why**：同一套题，拨不同开关就能测不同能力、适配不同强弱的模型（§2.4）。四组「开关」：

1. **不带 vs 带 科学背景 (Without vs. with scientific background)**：
   - 不带：考模型**自身的科学知识 + 推理 + 代码**综合力；
   - 带：把科学家写的知识/推理步骤给模型，评测重心**移向代码生成与指令遵循**；
   - 结论预告（§3.3）：给背景后**所有模型都大幅提升**，反证它们在自然科学上**知识与推理能力不足**。
2. **金标准 vs 模型生成 的前序子问题解 (Gold vs. generated solutions to previous subproblems)**：
   - 用金标准：每步都拿到正确前序解，**只考当前步**；
   - 用模型生成：更真实，但**误差会沿子问题链累积 (error accumulation)**，更难。
3. **主问题级 vs 子问题级 (Main vs. subproblem levels)**：
   - 主问题级：当且仅当**所有子问题解都对、且整合解也对**，才算解出主问题；
   - 子问题级：可**独立评测**某个子问题（不依赖它的其它子问题或主问题）。
4. **由此定义「标准设置 (standard setup)」**：**不带背景 + 前序子问题用模型生成的解**——原文明言这**最贴近科学家用 LM 的真实场景**，故定为标准设置；并指出此设置**对今天最好的模型都极具挑战**（Claude3.5-Sonnet 主问题仅 4.6%）。为让弱模型也有信号，实验里也跑了**更简单的设置**。

**评测主指标 pass@1（精确定义）**：

> 直觉：科学家真正用 LM 时，往往**只取一次输出就用**。所以本文不追求 pass@k 的多次采样，而用「**生成一次就全对**」的严格口径。

记号（先定义）：对某一评测单元（一个主问题或一个子问题）$i$，设其测试集合为 $T_i$；模型对 $i$ **采样 1 个解** $\hat s_i$；指示函数 $\mathbb{1}[\cdot]$ 在条件成立时取 1、否则 0；$\text{pass}(\hat s_i)=\mathbb{1}[\hat s_i\ \text{通过}\ T_i\ \text{中全部测试}]$。在含 $M$ 个评测单元的集合上：

$$ \text{pass@1} \;=\; \frac{1}{M}\sum_{i=1}^{M}\text{pass}(\hat s_i)\;=\;\frac{1}{M}\sum_{i=1}^{M}\mathbb{1}\big[\hat s_i\ \text{通过全部}\ T_i\big]. $$

读出什么：分子是「**一次生成即通过全部测试**」的单元数，分母是单元总数。**主问题 pass@1** 要求整条子问题链 + 整合全部正确，**子问题 pass@1** 只看该子问题——这就是为什么**主问题分数会比子问题低一大截**。原文用 **zero-shot** prompt（§3 Prompts），所以这是**零样本、单次**口径。

> 主讲提示：把四组开关画成「2×2×2 + 标准设置」一张表讲。强调**标准设置 = 最难且最真实**，4.6% 这个招牌数字就出自这里；而**子问题 vs 主问题**这组开关，正是「难度信号」的来源。

## 12. 主要结果（数字 + 解读，别只贴表）

**结果一：标准设置下极难（原文 Table 2，pass@1，标准设置 = 不带背景 + 前序解模型生成）**

| 模型 | 子问题 Pass@1 | 主问题 Pass@1 |
|------|------|------|
| GPT-4o | 25.0 | 1.5 |
| GPT-4-Turbo | 22.9 | 1.5 |
| **Claude3.5-Sonnet** | **26.0** | **4.6** |
| Claude3-Opus | 21.5 | 1.5 |
| Claude3-Sonnet | 17.0 | 1.5 |
| Gemini 1.5 Pro | 21.9 | 1.5 |
| Llama-3-70B-Instruct（开源） | 14.6 | 0.0 |
| Mixtral-8x22B-Instruct（开源） | 16.3 | 0.0 |
| DeepSeek-Coder-v2（开源） | 21.2 | 3.1 |

**读出什么**：
- **子问题级 14%–26%、主问题级跌到 0%–4.6%**：模型**能零散解对一些子问题，却几乎拼不出完整正确的主问题解**（原文 §3.2 明说「开源模型也展现出『能解一些子问题、却解不出任何主问题』的能力缺口」）。这是 SciCode 最核心的难度信号——**瓶颈不在单步，而在「整条链 + 整合」**。
- **Claude3.5-Sonnet 最强**（子 26.0 / 主 4.6），且是主问题分数明显高于他者的唯一模型；**DeepSeek-Coder-v2 是最好的开源**（主 3.1，甚至超过多数专有模型在主问题上的 1.5）。
- **两个开源模型主问题 0.0%**（Llama-3-70B、Mixtral-8x22B）：能解对一两成子问题，但**没有一道主问题被完整解出**。

**结果二：给背景后一致提升，但仍远不够（原文 Table 3，带背景 + 前序解模型生成；Δ = 相对标准设置的增量）**

| 模型 | 子问题 Pass@1 | Δ子 | 主问题 Pass@1 | Δ主 |
|------|------|------|------|------|
| GPT-4o | **35.4** | 10.4 | 9.2 | **7.7** |
| GPT-4-Turbo | 33.7 | **10.8** | 9.2 | 7.7 |
| Claude3.5-Sonnet | **35.4** | 9.4 | **12.3** | 7.7 |
| Claude3-Opus | 26.7 | 5.2 | 4.7 | 3.0 |
| Claude3-Sonnet | 25.7 | 8.7 | 4.7 | 3.0 |
| Gemini 1.5 Pro | 30.6 | 8.7 | 7.7 | 6.2 |
| Llama-3-70B-Instruct | 20.8 | 6.3 | 1.5 | 1.5 |
| Mixtral-8x22B-Instruct | 20.8 | 4.5 | 3.1 | 3.1 |
| DeepSeek-Coder-v2 | 27.1 | 5.9 | 4.6 | 1.5 |
| Qwen2-72B-Instruct | 22.2 | 5.2 | 4.6 | 3.1 |

**读出什么**：
- **所有模型在子问题和主问题上都因背景而显著提升**（原文 §3.3 斜体强调）。子问题级 GPT-4o 与 Claude3.5-Sonnet 并列最高 **35.4**（GPT-4-Turbo 增幅最大 **+10.8**）；说明**短板很大一块是「科学知识/推理」**，给了公式就能写不少代码。
- **但天花板仍很低**：主问题级 Claude3.5-Sonnet 最高 **12.3**，GPT-4o/GPT-4-Turbo 紧随 **9.2**；**即便喂了背景，最好的模型也只解出 ~12% 主问题**——原文据此断言：**SciCode 即使把重心从「考科学知识」移到「考代码生成」，仍然很难**（短板不止知识，还有**复杂代码整合**）。
- **开源提升小于专有**：原文推测开源模型**指令遵循 (instruction following) 偏弱**，故吃不下长背景；有趣的是 **Llama-3-70B 借背景追平了 Mixtral-8x22B**（与标准设置下的相对关系不同）。

**结果三：前序步给越多金标准解、子问题越好（原文 Figure 3，不给背景）**

> 直觉：后面的步骤能利用前面步骤的金标准解作为「in-context 示范」，看到越多正确前序解、对题理解越深，当前步越容易写对；聚焦靠后步骤，能更纯地测「模型自身能力」。

- 趋势：**四个模型大体随「可见的前序金标准解数」增多而提升**（GPT-4o、GPT-4-Turbo、Claude3.5-Sonnet、Claude3-Opus；从 ~35–42% 升到峰值，GPT-4o 峰值约 63%）。
- **两个例外**（原文明确点出）：**条件在第 7 个前序金标准解时出现一处显著回落**；且当**前序解超过 9 个**后，性能**开始下降**——作者归因于**长上下文管理变难 (managing long contexts)**。

> 主讲提示：把三张表/图串成一条故事线——**①标准设置主问题 4.6%（招牌数字，瓶颈在整合）；②给背景子问题能上 35%、主问题才 12%（知识是短板之一，但整合仍是硬骨头）；③前序金标准给越多越好、但 >9 个反而被长上下文拖累。**

## 13. 消融与分析（哪个因素贡献多少）

把上面三组结果当「消融」来读，可分离出各因素的贡献（均据 §3.2–3.3 / Table 2,3 / Fig.3）：

- **背景知识的边际贡献**：主问题级最强模型 **4.6 → 12.3**（Claude3.5-Sonnet），子问题级 **26.0 → 35.4**——**背景对「子问题」帮助更直接**（补的就是该步公式），对「主问题」帮助有限（整合难题仍在）。
- **前序解从「生成」换成「金标准」的贡献**：Figure 3 显示子问题 pass@1 可从 ~35% 抬到 50–60%+（视模型与可见步数）——说明**误差累积**是标准设置变难的重要来源之一：一旦前面步骤错，后面跟着错。
- **「子问题 vs 主问题」的落差本身就是最大信号**：无论哪种设置，**主问题分数都远低于子问题**（标准设置下 26.0 vs 4.6）。这把短板**定位到「整合 + 端到端正确性」**，而非「单个函数写不出」。
- **模型族差异**：专有 > 开源（主问题上尤其明显，两个开源在标准设置下为 0.0%）；专有内部 **Claude3.5-Sonnet > 其余**。
- **长上下文敏感性**：>9 个前序金标准解后子问题性能下降（Fig.3），提示**上下文越长不一定越好**，长上下文管理是独立瓶颈。

> 主讲提示：一句话总结消融——**「背景补知识、金标准前序解防误差累积，两者都能抬子问题分；但主问题的『整合天花板』很硬，谁都没真正突破。」**

## 14. 局限与批判（诚实）

**原文自陈/隐含的局限**：
- **领域不均衡**：物理占主问题 46%、子问题 49%；生物仅占子问题 8%（Table 1 / Fig.2）——**对物理/材料信号强，对生物/化学覆盖薄**，跨域结论需谨慎。
- **为「可解」而简化**：脚注 2 坦言——因很多题对最好模型都太难，**常简化设定、在修订时补更多背景**。这意味着「最终题面」已被人为调易，**真实科研可能更难**。
- **指标单一**：只报 **pass@1**（zero-shot 单次）；**未报 pass@k**、多次采样方差、置信区间。低分到底是「能力上限」还是「单次采样的运气下限」，原文未做区分实验。
- **成本/算力未列**：与 E 组其它评测不同，原文**未给统一温度、种子数、单题推理成本/时延**等（正文与附录均无）。
- **测试用例的充分性**：执行式判分依赖测试覆盖；虽用「领域专属测试 + GPT-4 反向找假阳性」加固，但**测试不可能穷尽**，仍可能放过「碰巧过测试但物理不对」的解（这是所有执行式基准的共性风险）。
- **抗污染是「近似」**：靠「少开源、曝光低」论证零重叠，属**合理但非严格证明**；随时间推移与数据扩张，未来仍可能被污染。

**社区视角可追问的**：
- 「主问题 4.6%」很震撼，但**部分源于严格口径**（整条链全对 + 单次采样）。换 pass@k 或允许少量重试，分数会显著变化——**招牌数字对口径敏感**。
- 子问题靠「前序金标准解」当 in-context 示范能涨分，这**有点像把答案的一部分喂回去**；它测的更接近「会照葫芦画瓢」，与「从零独立做完整科研」之间仍有距离。

> 主讲提示：批判主线——**「4.6% 是真信号，但要带着三个星号读：①题被人为简化过、②口径是 zero-shot 单次、③领域偏物理。」诚实地区分『论文宣称的难』与『口径/采样带来的难』。**

## 15. 在 auto-research 版图的位置

- **阶梯定位**：在 Tool→Analyst→Scientist 阶梯里，SciCode 是**给整条链「打地基」的尺子**——它不构建科研系统，而是**界定「科研代码能力」的真实水位**。它回答的是：在让 AI「做科研」之前，AI **连科学家日常的一段代码都还远没写利索**（标准设置主问题 4.6%）。
- **与本库其它论文的关系**：
  - ↔ **The AI Scientist (2408.06292)**：那篇证明「端到端闭环能跑通、且评审接近人类」，但**结果靠自评**；SciCode 提供的是**外部、客观、可执行**的能力刻度——可用来**戳破**「闭环能跑通 ≈ 科研能力强」的错觉。两者一个测「**流程能不能转**」，一个测「**单环代码到底行不行**」。
  - → 对「用 AI 加速科研」的方法研究是**靶场**：原文明言希望 SciCode **激励该方向**；任何声称能做科研代码的 agent，都该先在 SciCode 上证明自己能把主问题从个位数抬上去。
  - ↔ 与 SciBench [92] / GPQA [72] 的区别：那些测**科学问答/推理**，SciCode 测**可执行的科研代码合成**——更贴「真做事」。

> 主讲提示：把 SciCode 定位成「**照妖镜**」——任何系统类论文吹完闭环，都该问一句「它在 SciCode 这种客观执行尺上能解出几个主问题？」

## 16. 复现与可用性

- **开源**：数据、代码、leaderboard 在 `https://scicode-bench.github.io/`（原文首页脚注 1）。附录 A.1 给了**完整 prompt 模板**（Table 4 不带背景 / Table 5 带背景），A.3 给了一道**完整样例**（Haldane Chern 数，含主问题、三个子问题、两份背景 Table 6/7、领域测试图 Fig.9）。
- **能不能在单卡跑**：SciCode **本身是评测数据集**，不需训练；金标准解只依赖 **NumPy/SciPy/SymPy 等 CPU 科学库**（Fig.4），**判分在普通 CPU 即可**。真正成本在**调用被评模型的推理**（10 个强模型，多为 API），而非 GPU 训练。
- **坑**：
  1. **口径要对齐**——务必区分「标准设置 / 带背景 / 用金标准前序解 / 主问题级 vs 子问题级」，否则数字无法横比（很多复述只引「4.6%」却不写设置）。
  2. **测试用例是判分核心**——领域专属测试需要能复现已发表数值，环境/数值精度要稳。
  3. **长上下文**：前序解 >9 个会拖累表现（Fig.3），设计 agent 时注意上下文裁剪。
  4. 弱/小模型几乎全 0（CodeLlama-7B 主设置 0.4%），调试期建议先用**带背景 + 金标准前序解 + 子问题级**这种最易设置拿到非零信号。

## 17. 组会讨论问题（5–8 个，能引发讨论）

1. **招牌数字的口径**：4.6% 来自「zero-shot 单次 + 整条链全对」。若改用 pass@5 或允许执行报错后重试一次，你预计主问题分数会涨到多少？这是「能力上限」还是「采样下限」？该设计什么实验区分？
2. **整合 vs 单步**：子问题 26% 而主问题 4.6%——这个落差到底是「长程规划/整合」缺陷，还是「误差累积」？用 Figure 3（金标准前序解）能在多大程度上把两者拆开？
3. **背景知识的双刃**：给背景后子问题冲到 35%、主问题才 12%。这说明「知识」和「代码整合」各占多少短板？若给**更强的背景检索**（RAG）而非人写背景，能否复刻这份提升？
4. **领域偏置**：物理占近半、生物仅 8%。当前「最难基准」结论是否被物理主导？若按领域归一化再排名，模型排序会变吗？
5. **抗污染的可持续性**：靠「少开源」近似零重叠。一旦 SciCode 成为热门榜单、题面与解被广泛讨论，如何防止它像旧基准一样被刷穿？需要「滚动更新题库」吗？
6. **执行式判分的盲区**：领域专属测试 + GPT-4 找假阳性能挡多少「过测试但物理错」的解？能否构造一个「碰巧过测试却物理不对」的反例？
7. **对 AI Scientist 类系统的启示**：那些自评「接近人类」的闭环系统，若拉到 SciCode 标准设置上跑，主问题能解出几个？SciCode 该不该成为这类系统的**强制外部门槛**？
8. **子问题分解本身**：分解是科学家手工做的。若让模型**自己分解**主问题再解，分数会更高还是更低？「会分解」是否本身就是一项该被单独评测的科研能力？

## 18. 一页速记（汇报当天速览）

- **是什么**：科学家从**真实科研代码**出题的代码生成基准；**80 主问题 → 338 子问题**，覆盖**数学/物理/化学/生物/材料**五域 16 子领域（物理近半）。每题配**金标准解 + 数值测试 + 领域专属测试**，执行式判分。
- **核心结构**：**主问题→子问题分解**——每个子问题实现一个 Python 函数（可选附背景），再**整合**成主问题解。15 主(50 子)开发 / 65 主(288 子)测试。
- **四组开关**：①带/不带科学背景；②前序解用金标准/模型生成；③主问题级/子问题级判分；④**标准设置 = 不带背景 + 模型生成前序解**（最真实最难）。
- **主指标**：**pass@1**（zero-shot、单次采样、通过全部测试）。
- **关键数**：标准设置——**Claude3.5-Sonnet 主 4.6% / 子 26.0%**，Opus & GPT-4o 主 1.5%，最好开源 DeepSeek-Coder-v2 主 3.1%；**带背景**——最好模型主问题也仅 **12.3%**，子问题最高 35.4%；前序金标准解 **>9 个**后子问题反而下降（长上下文）。
- **三句话结论**：①真专家从真科研出题、可执行判分（**真且难**）；②**子问题能零散解对、主问题几乎拼不出**（瓶颈在整合，难度信号在此）；③给背景能补「知识」短板但补不平「整合」短板——**离「会做科研代码」还很远**。
- **在课里的位置**：客观「照妖镜」，给整条 auto-research 阶梯立**真实水位**；正面校准 AI Scientist 类系统的自评幻觉。

> 主讲提示：结尾回到一句话——**「当模型已经超过普通人，真正的难题得让真科学家从真科研里出；SciCode 用『主问题→子问题』的可验证分解，量出了今天最强模型在真科研代码上仍只是个位数主问题正确率。」**
