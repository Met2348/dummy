# L2 · 招募筛选合作者/实习生 (Recruiting Collaborators and Interns)

> 30-min lecture · 目标: 把"招个人来帮忙"从一句模糊的"越强越好"或"谁先联系我就要谁", 变成一套具体可执行的两步法——写清楚"必须项 vs 可现学项"的标准, 再用一个有边界的试做任务(trial task)真正验证候选人, 而不是只靠一场对话式面试。
> 关键区分: **本讲解决的是"谁能进这个团队"的筛选问题, `team-leadership-for-researchers` L1「指导低年级学生/实习生」解决的是"进来之后怎么带"的问题——招募决定了你带教的起点在哪一级, 筛选标准写得越具体, L1 的任务难度阶梯才越能准确地从第一天就对上候选人的真实水平, 而不是靠带教过程去弥补招募阶段没看清楚的落差。**

---

## 0. 为什么这是个真问题

大多数博士生第一次需要招募本科生intern或合作者时, 完全没有标准可依——要么是"越强越好"式的空洞高要求(简历要求写得像招聘正式研究员, 结果没人敢投, 或者投来的人和真实需求完全不匹配), 要么是完全没有标准, 谁先发邮件表达兴趣就收谁。两种极端都会在几周后暴露问题: 前者可能吓退了本来合适、只是简历不亮眼的候选人; 后者更常见也更麻烦——招进来之后才发现对方连最基本的前置技能都不具备, 而这时候已经投入了带教成本, 退回去比一开始筛清楚代价高得多。

```
                没有标准的招募                          有标准的招募
筛选依据        谁先联系/简历关键词是否好看                 必须项(现在就要有) vs 可现学项(进来再补) 分层
面试方式        聊聊研究兴趣, 凭印象判断                    对话 + 一个有边界的试做任务(trial task)
筛选后果        招进来才发现前置技能缺失, 带教成本骤增        提前暴露落差, 在投入带教成本之前就筛掉不合适的
标准的可复用性  每次招人都重新凭感觉判断一遍                 标准写一次, 下次招人可以复用和迭代
```

## 1. 标准怎么写: 必须项 vs 可现学项

核心动作是把"这个岗位需要什么"拆成两类完全不同的清单, 而不是笼统地列一堆"优秀候选人应具备的素质":

```
必须项 (must-have)                        可现学项 (can-learn-on-the-job)
──────────────────────                    ──────────────────────
进来当天就要具备, 没有它无法开始工作          进来之后, 在这次合作的时间窗口内可以补上
例: 能读懂并修改一份PyTorch训练脚本          例: 某个具体子领域的背景知识
例: 每周能保证X小时以上、持续Y个月           例: 某个具体工具/框架的使用经验
例: 具备基本的实验记录习惯(不瞎跑不记录)     例: 论文写作/画图的熟练程度
```

判断一项能力该归入哪一类的具体检验方法: **问自己"这项能力, 在这次合作的时间窗口内, 能不能通过带教补上"。** 如果答案是"能, 只是需要一点时间和一次好的code review", 归入可现学项; 如果答案是"补不上, 或者补上的时间成本高到不划算(比如"看得懂并独立调试深度学习框架"这种需要长期积累的能力), 归入必须项。**必须项要少而具体, 可现学项不必在招募阶段筛, 甚至写进岗位描述本身可以起到吸引作用**("不要求已经会X, 我们会带你上手")——这和"越强越好"的空洞高要求正好相反: 后者把可现学项也堆进了门槛, 结果吓退了本来合适、只是还没学过某个具体工具的候选人。

Geoff Smart 与 Randy Street 在《Who: The A Method for Hiring》里提出的 **scorecard(记分卡)方法**——先写清楚这个岗位的具体产出目标(outcomes)和支撑这些产出所需的具体能力(competencies), 再据此设计筛选流程, 而不是先看简历再临时决定标准——本质上就是"必须项/可现学项"拆分的更完整版本, 核心思想相通: **标准要在见到候选人之前写好, 而不是见一个人调整一次门槛。**

## 2. 面试/试做任务设计: 一个有边界的 trial task

对话式面试(聊聊研究兴趣、问几个背景知识问题)只能验证候选人"说得出来什么", 无法验证"做得出来什么"——而后者才是实际合作中真正重要的能力。一个设计得当的**试做任务(trial task)**——一段有明确边界、贴近真实工作内容的小任务——是目前对实际表现最有预测力的筛选方式(本质上是一种work-sample测试)。设计原则:

```
原则                做法                                    为什么
────────────      ──────────────────────                  ──────────────
有边界            控制在2-4小时内可完成, 明确写出预计耗时      不能变相占用候选人大量免费劳动时间
贴近真实工作        用"读这段论文复现一个数字"/"给这个现成      通用算法题验证不了实际研究/工程能力,
                  仓库加一个功能"这类具体任务, 不用通用算法题   贴近真实场景的任务预测力才高
标准先行           评分标准在看到提交结果之前就写好              防止"halo effect"——先对某人有好感,
                                                            再倒着找理由证明提交合格
```

一个具体例子: 对于"读论文能力"是必须项的岗位, trial task可以是"给你一篇指定论文的其中一节, 用半天时间复现里面报告的一个具体数字, 写清楚复现过程中遇到的任何偏差"——这个任务同时检验了读论文能力、动手能力、以及是否诚实报告偏差(而不是硬凑出"完全吻合"的假结果)三件事, 而且耗时可控、贴近这个岗位实际会做的事。

## 3. 一个完整走查示例

场景: 招募一位暑期研究intern, 协助跑一批长上下文推理的消融实验。

```
必须项: 能读懂并修改现有PyTorch训练脚本; 每周保证15小时以上、持续8周;
        会用git做基本的版本管理(不要求熟练, 但要求"用过")。
可现学项: 长上下文推理这个具体子领域的背景知识(可以入职后通过精读2-3篇论文补上);
        具体的实验跟踪工具使用经验。

面试流程: 先15分钟对话了解基本背景和时间投入意愿, 不深挖学术背景;
        再布置trial task——"在现有仓库基础上, 给一个已有的评测脚本加一个新的
        输出字段, 3小时内完成, 附一份简短说明", 评分标准提前写好(能否跑通/
        代码风格是否遵循仓库现有约定/说明是否清楚), 不看候选人简历亮不亮眼。

结果: 两位候选人简历都不算突出, 但trial task表现出明显差异——一位顺利跑通并
      写了清楚的说明, 另一位卡在环境配置上超时未完成。选择前者, 而不是回头
      看谁的GPA或学校更好看。
```

## 4. 常见误区

- **只筛学校/GPA, 不直接测必须项本身。** 学校声誉和GPA是间接代理指标, 和"能不能读懂并修改这份代码"这个具体的必须项之间相关性有限——直接测试必须项, 比依赖代理指标可靠得多。
- **面试问题越刁钻越显得筛选严格。** 故意设计超纲的追问题, 只能测出候选人临场应变能力, 测不出实际工作场景下的真实表现, 而实际工作场景恰恰不是临场压力测试。
- **把trial task设计成变相的免费劳动。** 任务边界模糊、耗时无上限、甚至变成"帮我们把这部分真实工作先做完再谈要不要录用你", 这是对候选人时间的剥削, 也会让真正优秀的候选人望而却步——边界(第2节"有边界"原则)必须清楚写明并且严格遵守。

## 5. 经典参考

- **Geoff Smart & Randy Street, 《Who: The A Method for Hiring》**(Ballantine Books, 2008): scorecard(记分卡)方法的系统阐述——先定义岗位的具体产出目标和支撑能力, 再据此设计筛选流程, 是本讲第1节"必须项/可现学项"拆分的直接方法论来源。
- 也见 `team-leadership-for-researchers` L1「指导低年级学生/实习生」——那一讲的任务难度阶梯(Level 1-4)假设你已经知道候选人现在站在哪一级; 本讲的必须项/可现学项标准和trial task, 正是回答"招募阶段怎么判断这个起点在哪"的前置步骤, 两讲首尾衔接, 不重复。

## 6. 本讲小结 + 通往 L3

- 标准拆成必须项(现在就要具备)和可现学项(进来后可以补), 后者不该出现在筛选门槛里, 甚至可以反向用来吸引候选人。
- 对话式面试测不出实际表现, 一个有边界(2-4小时)、贴近真实工作、评分标准提前写好的trial task预测力更高。
- 招募标准写得越具体, 后续 `team-leadership-for-researchers` L1 的任务难度阶梯才能越准确地从第一天开始匹配。

> **下一讲 L3「远程/异步跨时区科研团队协作」**: 招到人之后, 如果这个人和你不在同一个时区、甚至从未见过面(暑期intern远程协作、跨国合作者), 怎么建立协作规范, 是完全不同的下一个问题——L3 从"招进来"转向"招进来之后怎么远程一起工作"。

**动手**: 打开 `templates/team-ops-plan.md`, 为你现在(或未来可能)要招募的一个具体角色, 写出必须项/可现学项两张清单, 再设计一个2-4小时的trial task和对应评分标准——不是抽象地想"招个靠谱的人", 是具体写出这两样东西。
