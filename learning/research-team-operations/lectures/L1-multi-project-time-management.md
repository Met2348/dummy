# L1 · 多项目并行时间管理与优先级排序 (Multi-Project Time Management)

> 30-min lecture · 目标: 用**时间块分配法**(time blocking)把"手头几个项目该怎么分时间"从一句凭感觉的"哪个催得紧就先做哪个", 变成一张可执行、可复盘的每周时间骨架, 从结构上避免"救火式"(firefighting)被动切换吃掉大部分产出。
> 关键区分: **`research-life` L4「可持续科研」教的是个体尺度的节奏管理——你怎么避免 burnout、怎么维持一个人长期稳定的输出; 本讲是结构尺度的分配问题——你手头同时挂着好几个项目, 时间该怎么在它们之间切分。前者关心你的精神状态, 后者关心你的日历本身怎么画格子, 是两个完全不同粒度的问题, 缺一不可但不能互相替代。**

---

## 0. 为什么这是个真问题

一个博士生或年轻研究者, 手头很少只有一个项目: 自己的主线课题、一个合作者的项目、审稿任务、偶尔还有教学助教工作。没人教过"这些项目之间的时间该怎么分", 于是默认策略几乎总是同一种: **谁催得最紧, 就先做谁的。** 合作者发来一封"这周能不能出个结果"的邮件, 你就把这周排给他; 审稿截止日快到了, 你就熬夜赶审稿意见。这套策略看起来很"负责任"——你确实在响应每一个紧急请求——但它有一个致命盲点: **你自己的主线课题几乎从不会主动"催"你**, 它没有外部截止日期天天提醒你, 于是在这套"谁喊得响先做谁"的分配规则下, 它永远是被牺牲的那一个。

```
                救火式分配 (没有本讲的团队)              时间块分配 (有本讲的团队)
分配依据        谁最近催得最紧, 谁先做                     每周固定骨架, 默认谁用哪个时间段
主线课题命运    从不主动催促, 长期被挤到"有空再做"          有专属保护时段, 不因为没人催就被吃掉
切换成本        频繁在项目间跳, 每次都要重新进入状态          同一时段只处理一个项目, 减少反复进入成本
复盘难度        说不清这周时间到底流向了哪, 只记得"很忙"      每周骨架本身就是一份可复盘的记录
```

**这个坑之所以顽固, 是因为它伪装成"响应迅速""合作可靠"**——你确实每次都及时回应了催促,看起来是个尽职的合作者。但半年后回头看, 你会发现自己主线课题的进度几乎为零, 而这半年你并不闲, 只是时间全部流向了"最近喊得最响"的那个项目。**时间没有消失, 是分配规则本身有偏向。**

---

## 1. 时间块分配法 (Time Blocking)

核心工具来自 Cal Newport 在《Deep Work: Rules for Focused Success in a Distracted World》里系统阐述的方法: **不用一张不断变长的待办清单去应对所有项目, 而是提前把一周的日历切成固定的时间块, 每个块默认归属一个具体项目**, 而不是留白等着被临时任务填满。

```
                周一        周二        周三        周四        周五
上午 (深度块)    项目A       项目B       项目A       项目B       项目A
                (主线课题)   (合作项目)   (主线课题)   (合作项目)   (主线课题)
下午 (弹性块)    留给突发协作 / 审稿 / 回复消息 / 补救上午没做完的部分
```

拆开看两条设计原则:

**① 先保护主线课题的深度块, 再分配救火时间, 顺序不能反过来。** 第0节的"救火式"分配之所以总是牺牲主线课题, 根本原因是分配顺序反了——先看这周有什么紧急的, 剩下的时间才轮到主线课题。时间块分配法要求反过来: **先把主线课题(通常是最重要但从不主动催促的那个)的深度时段钉死在日历上, 当作不可挪用的默认占用, 剩下的时段(通常是下午, 精力相对不那么适合深度工作的时段)才拿来处理合作项目、审稿、回复消息这类反应性任务。**

**② 同一个时间块只处理一个项目, 不在块内部来回切换。** 这条原则的依据是组织行为学家 Sophie Leroy 提出的**注意力残留**(attention residue)现象: 从任务A切换到任务B之后, 一部分注意力仍然滞留在A上, 需要一段时间才能完全转移, 这段时间里你在B任务上的表现是打折的。如果你在同一个上午在项目A和项目B之间来回切换几次, 每次切换都会留下残留, 累积起来你会感觉"忙了一上午但什么都没推进"——这正是很多人抱怨"多项目并行效率低"的真实机制, 不是项目太多本身的问题, 是切换频率的问题。**一个块只归一个项目, 是对抗注意力残留最简单的结构性做法。**

**③ 骨架的比例, 按"重要但不紧急"优先分配, 而不是按项目数量平均切分。** 一个常见的误操作是"我手头3个项目, 那就一周3等分"——这忽略了不同项目对紧迫外部截止日期的依赖程度天差地别。主线课题几乎永远属于"重要但不紧急"(没有人天天催, 但长期看它才是决定你产出的核心); 合作项目和审稿往往属于"紧急且有明确截止日期"。**分配比例应该向前者倾斜**(比如主线课题占深度时段的六成以上), 而不是简单按项目个数平均分, 否则你只是把"救火式"的偏向从时间维度换成了项目数量维度, 本质没变。这一比例判断和 `research-direction-proposal` L2「项目级可行性评估」讨论的资源时间线判断是同一枚硬币的两面: 那一讲评估的是"一个研究方向本身在数年尺度上撑不撑得住", 本讲评估的是"这一周/这一月的时间, 在已经确定要做的几个项目之间怎么切", 前者是要不要接这个项目的判断, 后者是接了之后怎么分时间的判断, 尺度不同。

## 2. 一个完整走查示例

场景: 一位博士生同时挂着三件事——自己的主线课题(长上下文推理效率)、和另一个组的合作项目(帮忙跑一批消融实验)、以及本学期的助教工作。

```
没有时间块分配的一周:
周一 合作组发消息"这周能出个初步结果吗", 全天都在跑他们的消融
周二 助教改作业+答疑占满上午, 下午合作组又追问进度, 继续跑消融
周三 主线课题的想法在脑子里转了一整天, 但没有整块时间坐下来推进
周四 审稿deadline提醒邮件到了, 临时插入审稿, 挤掉了原计划补主线课题的时间
周五 合作组结果终于跑完发出去, 一周结束, 主线课题一行代码没写
                ↓ 这一周并不闲, 但主线课题被"没人催"这件事惩罚了

用时间块分配法重新排:
周一/三/五 上午 = 主线课题专属深度块(雷打不动, 除非真正的紧急情况)
周二/四 上午 = 合作项目消融块
每天下午 = 弹性块, 处理助教/审稿/临时消息, 消化不完的顺延到下一个弹性块而不是侵占深度块
                ↓ 一周结束, 主线课题有3个上午的实质推进, 合作项目和助教也都按时完成
```

两周对比里, 总工作时长几乎一样, 差别完全在**分配的默认规则**: 前者是"谁催就先给谁", 后者是"主线课题先占位, 其余按固定骨架轮转"。

## 3. 常见误区

- **把时间块当成绝对不可变的铁律, 一点弹性都不留。** 真正紧急的情况(比如导师临时约谈、审稿真的火烧眉毛)当然需要打破骨架, 但"打破"应该是明确的例外决策, 而不是默认状态——如果你发现自己几乎每周都在打破骨架, 说明骨架设计本身脱离了实际负荷, 需要重新规划, 而不是继续假装骨架存在。
- **在同一个块里"顺便"处理另一个项目的小事。** "反正上午在写主线课题的代码, 顺手回一下合作组的消息"——这正是注意力残留的来源, 看起来省事, 实际上拖慢了两边的进度。弹性块的存在就是为了集中处理这些"顺手"的事, 不需要在深度块里插空处理。
- **排完一次骨架就当作一劳永逸, 不再检查。** 项目所处的阶段会变(合作项目跑完消融进入写作阶段、主线课题进入攻坚期), 骨架需要跟着定期(比如每两周)重新审视和调整, 而不是排一次用一学期。
- **按项目数量平均切分时段, 而不是按"重要但不紧急"程度倾斜。** 三个项目就三等分, 看似公平, 实则把原本该向主线课题倾斜的时间稀释掉了——公平的不是时长本身相等, 是长期产出的优先级得到了对应的时间保护。

## 4. 经典参考

- **Cal Newport, 《Deep Work: Rules for Focused Success in a Distracted World》**(Grand Central Publishing, 2016): 时间块分配法(time blocking)与注意力残留(attention residue, 引自 Sophie Leroy 的研究)的系统阐述来源, 是本讲第1节两条设计原则的直接理论依据。
- 也见 `research-life` L4「可持续科研」——那一讲讲的是个体节奏与burnout自查, 是"这一个人怎么跑得久"的问题; 本讲讲的是"手头几个项目之间怎么切分时间"的结构问题, 两讲粒度不同, 互补而非重复。

## 5. 本讲小结 + 通往 L2

- "救火式"分配(谁催得紧先做谁)的致命盲点是: 主线课题从不主动催你, 于是永远被牺牲。
- 时间块分配法两条原则: 先保护主线课题的深度块再分配救火时间; 同一时间块只处理一个项目, 减少注意力残留。
- 骨架需要保留应对真正紧急情况的弹性, 也需要随项目阶段定期重新审视, 而不是排一次就一劳永逸。

> **下一讲 L2「招募筛选合作者/实习生」**: 时间分配解决的是"手头已有的项目怎么分时间"; 但项目数量本身也在变——你迟早会需要招人扩大产能, 而"该招什么样的人、怎么筛"同样是一门此前完全没人教过的具体技能。L2 从"分配时间"转向"扩充人手"这第一步。

**动手**: 打开 `templates/team-ops-plan.md`, 把你现在真实手头的2-3个项目, 按第1节的骨架画一张属于你自己的每周时间块表——不是抽象地想"我应该更合理分配时间", 是具体填出"周几上午归哪个项目"。
