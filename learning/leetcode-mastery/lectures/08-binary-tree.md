## 二叉树递归题的通用心法

刷二叉树题最容易陷入的误区是"每道题都是全新的，要重新想一套思路"。实际上绝大多数二叉树递归题都在回答同一个问题：

> **这个函数对一棵子树做什么？它需要子树（递归调用）返回什么信息？拿到这些信息之后，怎么和当前节点组合出"这一层该返回的东西"？**

想清楚这句话里的两件事——**"递归调用返回什么"** 和 **"怎么组合"**——几乎就写完了整道题。举几个本 lecture 里的例子对照着看：

| 题目 | 递归函数对子树回答什么问题 | 怎么组合 |
|---|---|---|
| LC104 最大深度 | "这棵子树多深？" | `1 + max(左深度, 右深度)` |
| LC226 翻转二叉树 | "把这棵子树翻转好，给我翻转后的根" | 先递归翻转左右子树，再交换 `left`/`right` |
| LC236 最近公共祖先 | "这棵子树里有没有 p/q？有的话把找到的节点/LCA 报上来" | 左右子树都非空 → 当前节点是 LCA；否则谁非空传谁 |
| LC124 最大路径和 | "这棵子树能贡献给父节点的最大链和是多少？" | 顺带用两条腿的和更新全局答案，但只把一条腿返回给父节点 |

带着这张表的思维方式，再看下面 7 道题会发现它们不是 7 个孤立的技巧，而是同一套心法在不同"组合方式"上的变体。

## 深挖 1：LC105 从前序与中序遍历序列构造二叉树

### 题意

给一棵二叉树的前序遍历序列 `preorder` 和中序遍历序列 `inorder`（节点值互不相同），把这棵树重建出来。

### 两条性质，缺一不可

**为什么前序遍历第一个元素永远是当前子树的根：** 前序遍历定义是"根 → 左 → 右"，不管子树多大，第一个被访问、被写进序列的必然是这棵（子）树的根节点本身。所以只要知道"当前正在处理 preorder 序列的哪个位置"，那个位置的值就是当前要构造的这棵子树的根。

**为什么中序遍历能把"根左边"和"根右边"切开：** 中序遍历定义是"左 → 根 → 右"，一旦知道了根节点的值，在 inorder 序列里找到这个值出现的位置 `mid`，那么 `mid` **左边的一段值，恰好就是整棵左子树所有节点的中序遍历结果**；`mid` 右边的一段，恰好是整棵右子树的中序遍历结果。这是因为中序遍历不会把左右子树的节点交叉排列——左子树的所有节点，不管多深，一定在中序序列里连续地排在根的左边；右子树同理排在右边。

### 组合起来的递归框架

1. 从 `preorder` 当前游标处取一个值，当作**当前子树的根**（性质 1）。
2. 在 `inorder` 里查这个值的位置 `mid`，把 `inorder` 切成 `[in_left, mid-1]`（整棵左子树的中序序列）和 `[mid+1, in_right]`（整棵右子树的中序序列）（性质 2）。
3. **游标前进一步**，递归构造左子树；递归返回后，游标已经跳过了整棵左子树在 preorder 里占据的所有位置，此时游标指向的下一个值，正好是右子树的根——递归构造右子树。

```python
index_of = {val: i for i, val in enumerate(inorder)}   # 值 → 下标，O(1) 定位根
pre_idx = [0]                                            # 可变游标（模拟 nonlocal）

def helper(in_left, in_right):
    if in_left > in_right:
        return None
    root_val = preorder[pre_idx[0]]
    pre_idx[0] += 1
    root = TreeNode(root_val)
    mid = index_of[root_val]
    root.left = helper(in_left, mid - 1)     # 必须先递归左子树
    root.right = helper(mid + 1, in_right)   # 游标才会自动落在右子树根上
    return root
```

用哈希表预处理 `inorder` 的值到下标的映射，把"在 inorder 里查根的位置"从线性扫描的 O(n) 降到 O(1)，避免整体退化成 O(n²)——这是这题效率上容易被忽略的一环。

### 易错点

- `pre_idx` 必须是"可变的外部状态"（这里用长度为 1 的 list 模拟），不能用普通参数传值——如果传值，左子树递归内部对游标的修改，不会体现在紧接着的右子树递归里，会导致右子树用错误的 preorder 位置取根。
- 递归终止条件必须是 `in_left > in_right`，写成 `>=` 会在只剩一个节点时提前判定为空，丢掉这个节点。

## 深挖 2：LC124 二叉树中的最大路径和——两个容易搞混的量

### 题意

"路径"指沿着节点间的边走的一个序列，同一节点在一条路径里最多出现一次，路径不要求经过根节点。求所有路径的节点值之和的最大值。

### 这题最容易搞混的地方

新手写这题最常见的错误是把两个不同的量混成同一个东西：

1. **"以某节点为拐点、经过该节点的最大路径和"**——这条路径可以同时向左、向右两个方向延伸，形状像 `⋀`。这是"最终候选答案"，每个节点都要算一次，取全局最大值。
2. **"某节点能贡献给它父节点的最大链和"**——链的定义是只能往**一个方向**延伸，不能同时往左又往右。为什么不能两边都要：因为如果父节点把"我"当作它自己路径的一部分往上继续延伸，"我"这里只能提供一条腿接上去，两条腿都占用的话路径会在"我"这里分叉，就不再是一条合法的简单路径了。

**这两个量必须分开处理**：递归函数**返回**的是量 2（只能一条腿，因为要喂给父节点继续往上拼接）；但在计算的过程中，**顺手**用量 1（两条腿都算上）去更新一个全局最大值。

```python
best = float("-inf")

def gain(node):
    nonlocal best
    if node is None:
        return 0
    left_gain = max(gain(node.left), 0)    # 负贡献不如不要这条腿
    right_gain = max(gain(node.right), 0)
    best = max(best, node.val + left_gain + right_gain)   # 量 1：两条腿都算，更新全局答案
    return node.val + max(left_gain, right_gain)           # 量 2：只留一条腿，返回给父节点
```

子树的 `gain` 如果是负数，还不如直接不要这条腿（对总和只有负贡献），所以子树的 gain 要先和 `0` 取 `max` 再使用——这对应"路径完全可以只往一个方向延伸、甚至只用当前节点自己"。

### 易错点

- 把"返回值"和"更新 best 时用的值"写成同一个表达式（都用两条腿的和），会把"经过某节点之后又分叉成两支"的不合法路径当作合法的链继续传给父节点，导致父节点那一层算出不存在的路径。
- 忘记把子树的负 gain 截断成 0，遇到全是负数的树时结果会算错——正确答案应该退化成"树中权值最大的单个节点"。

## 小结

7 道题看起来分散，但都能装进"递归函数对子树回答什么问题 + 怎么组合"这一句话里。遇到新的二叉树题，先别急着写代码，先把这句话在纸上填空填清楚：**返回值是什么类型/含义？空节点（base case）返回什么？非空节点怎么用左右子树的返回值算出当前层的返回值？** 这三个空填完，代码基本就是直译。

### 深挖：LC297 二叉树的序列化与反序列化——为什么是"前序 + 显式 None 占位"

#### 题意

设计一种算法把二叉树编码成字符串，再从字符串还原出**结构完全一致**的树。

#### 为什么"前序遍历 + 显式空节点占位"能唯一还原一棵树

只给一个前序遍历序列（不带空节点占位）是**不够**的——比如 `[1, 2]` 既可能是"1 的左孩子是 2"，也可能是"1 的右孩子是 2"，光看前序值本身无法区分。缺的信息正是"每个节点的左右孩子到底存不存在"。

解法是序列化时**显式地把每一个空位也写进序列**（本文件用 `"#"` 占位）。这样，反序列化时只要按照和序列化完全相同的顺序（前序：根 → 左 → 右）消费这个序列，每次读到一个值就能立刻确定："这是一个真实节点吗？还是一个空位？"——不需要额外信息就能唯一确定去哪里放下一个孩子。用一段推理串起来：

1. 前序遍历第一个 token 永远是当前（子）树的根（或者是 `#`，说明这棵子树是空的）。
2. 因为**每个真实节点都恰好消费两个"孩子槽位"**（左槽位、右槽位），而每个槽位最终要么被一个真实节点占据、要么被一个 `#` 占据，所以整个序列可以被无歧义地、递归地解析：读到根值 → 递归解析左子树（会自己消费掉属于左子树的若干 token）→ 递归解析右子树。序列里没有任何"该往哪走"的选择余地，每一步都是确定的。

#### 为什么中序遍历做不到

中序遍历是"左 → 根 → 右"。即使也给中序序列加上 `#` 占位，问题在于：**光看中序遍历本身，不知道该从哪个位置切出"根"**——前序/后序天然有"第一个/最后一个 token 就是根"这条免费的性质，但中序遍历里根可能出现在序列的任意位置，无法单独靠中序序列的顺序确定谁是根、谁是它的左右子树边界。（这也解释了为什么 Part I 的 LC105/本文件的 LC106 都需要"前序或后序 + 中序"两条序列搭配，单独一条中序永远不够。）前序/后序自带"根在哪"的信息，中序不带，这就是二者的本质差异。

```python
class Codec:
    def serialize(self, root):
        values = []
        def dfs(node):
            if node is None:
                values.append("#")
                return
            values.append(str(node.val))
            dfs(node.left)
            dfs(node.right)
        dfs(root)
        return ",".join(values)

    def deserialize(self, data):
        values = iter(data.split(","))
        def dfs():
            val = next(values)
            if val == "#":
                return None
            node = TreeNode(int(val))
            node.left = dfs()
            node.right = dfs()
            return node
        return dfs()
```

#### 易错点

- 反序列化时如果用 `list.pop(0)` 从头部逐个取值，每次取值都是 O(n)，整体退化成 O(n²)；用 `iter(...)` + `next(...)` 才能保持每次取值 O(1)。
- 构造左右子树的顺序必须和序列化时写入的顺序严格一致（先左后右），顺序写反会把树镜像错误地还原出来。

### 深挖：LC987 二叉树的垂序遍历——同一坐标多节点时，为什么必须按值排序打破平局

#### 题意

把二叉树画在一个二维网格上：根节点位于 `(row=0, col=0)`；某节点的左孩子位于
`(row+1, col-1)`，右孩子位于 `(row+1, col+1)`。要求按列从左到右、同一列内
按行从上到下，返回每一列的节点值列表。**如果同一列同一行出现多个节点（多个
节点的坐标完全相同），这些节点必须按节点值升序排列**——这条"平局规则"正是
本题最容易踩坑的地方。

#### 为什么会有"同一坐标多节点"这种情况

看这样一棵树：

```
        1
       / \
      2   3
       \ /
        4
```

（这里画的是逻辑关系：`2` 的右孩子和 `3` 的左孩子都是同一个坐标位置——因为
`2` 位于 `(1, -1)`，它的右孩子位于 `(2, 0)`；`3` 位于 `(1, 1)`，它的左孩子
位于 `(2, 0)`。）在 LeetCode 987 的实际用例里，这种"同一坐标撞在一起"的
节点是两个**不同的节点对象**（比如示例 `root = [1,2,3,4,6,5,7]` 里，节点
`5` 和 `6` 分别是 `2` 的右孩子和 `3` 的左孩子，两者的坐标都是
`(row=2, col=0)`）——它们只是恰好落在同一个网格坐标上，值不同、身份也不同，
必须按某种规则决定谁排在前面。题目规定的规则是：**值更小的排在前面**。

#### 为什么"先按列分组，组内再按行排序"这种朴素写法会漏掉平局规则

一种常见但不完整的写法是：用字典把节点按 `col` 分组，组内先按 `row` 排序，
排序时如果 `row` 也相同就……很多人到这里会漏掉再按 `val` 排序这一步，
或者干脆用一个只存 `(row, node)` 的列表、依赖 Python 排序的"稳定性"来决定
平局时的顺序——但稳定排序保留的是**插入顺序**（也就是 DFS 先访问到哪个
节点的先后），而不是"值更小的在前"，这和题目要求的规则是两回事，很容易在
"两个节点先访问哪个"恰好和"值大小"不一致的用例上出错（比如示例 3
`root = [1,2,3,4,6,5,7]`，如果先访问到值更大的节点，稳定排序会把它错误地
排在前面）。

#### 正确做法：把 `(col, row, val)` 三元组一起排序

```python
items = []

def dfs(node, row, col):
    if node is None:
        return
    items.append((col, row, node.val))
    dfs(node.left, row + 1, col - 1)
    dfs(node.right, row + 1, col + 1)

dfs(root, 0, 0)
items.sort()
```

这里的关键设计是：**把 `val` 也放进要排序的元组里，作为第三个比较字段**，
而不是排序完 `(col, row)` 之后再单独处理平局。Python（以及几乎所有语言）
对 tuple/元组的默认比较规则是**按字典序**——先比较第一个元素，如果相等
再比较第二个，再相等才比较第三个。这恰好和题目"先按列分组，同列内按行
排序，行也相同再按值排序"这套三级规则完全对应，`items.sort()` 这一行代码
同时满足了三条规则，不需要额外写平局处理逻辑。

排序之后，只需要顺序扫一遍 `items`，"col 值和上一个不同"就说明进入了新的
一列，开启一个新的分组：

```python
result = []
prev_col = None
for col, row, val in items:
    if col != prev_col:
        result.append([])
        prev_col = col
    result[-1].append(val)
```

#### 易错点

- 元组的字段顺序必须是 `(col, row, val)`，不能写成 `(row, col, val)` 或
  只存 `(row, val)`——`col` 必须排在最前面，才能保证排序的第一优先级是
  "按列分组"；漏掉 `val` 这个字段，或者把它放在排序范围之外，都会在
  "同一坐标多节点"的用例上得到和官方顺序不一致的错误答案。
- 不要试图用"先访问左子树就排在前面"这类基于 DFS 顺序的规则替代"按值
  排序"——这两者在大多数简单用例上恰好给出相同的结果，容易被误认为是
  等价的，但题目的规则明确是"按值排序"，只有在构造出"同一坐标、DFS 访问
  顺序和值大小顺序相反"的用例（比如官方 Example 3 那样，特意交换了两个
  孩子节点）时，两者才会给出不同的、可验证对错的结果。
