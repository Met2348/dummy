# L08 · 二叉树遍历与构造 —— 递归的通用心法

对应代码：`src/problems/p08_binary_tree.py`

## 二叉树递归题的通用心法

刷二叉树题最容易陷入的误区是"每道题都是全新的，要重新想一套思路"。实际上绝大多数二叉树递归题都在回答同一个问题：

> **这个函数对一棵子树做什么？它需要子树（递归调用）返回什么信息？拿到这些信息之后，怎么和当前节点组合出"这一层该返回的东西"？**

想清楚这句话里的两件事——**"递归调用返回什么"** 和 **"怎么组合"**——几乎就写完了整道题。举几个本 lecture 里的例子对照着看：

| 题目 | 递归函数对子树回答什么问题 | 怎么组合 |
|---|---|---|
| LC104 最大深度 | "这棵子树多深？" | `1 + max(左深度, 右深度)` |
| LC226 翻转二叉树 | "把这棵子树翻转好，给我翻转后的根" | 先递归翻转左右子树，再交换 `left`/`right` |
| LC236 最近公共祖先 | "这棵子树里有没有 p/q？有的话把找到的节点/LCA 报上来" | 左右子树都非空 → 当前节点是 LCA；否则谁非空传谁 |
| LC124 最大路径和 | "这棵子树能贡献给父节点的最大链和是多少？" | 顺带用两条腿的和更新全局答案，但只把一条腿返回给父节点 |

带着这张表的思维方式，再看下面 7 道题会发现它们不是 7 个孤立的技巧，而是同一套心法在不同"组合方式"上的变体。

## 7 题清单（点睛）

- **LC94 二叉树的中序遍历（Easy）**：递归版 `_inorder_recursive` 最直白地体现"左-根-右"的定义；迭代版用显式栈模拟"一路往左压栈，走不动了弹出访问，再转向右子树"——本质是把递归调用栈换成显式栈，工程上更常用（不受递归深度限制）。
- **LC104 二叉树的最大深度（Easy）**：`1 + max(左, 右)`，最短的递归范例，用来建立"信任递归、只管组合"的直觉。
- **LC226 翻转二叉树（Easy）**：递归翻转左右子树后交换引用（不是交换值）。测试时用自己写的 `tree_to_level_list` 把结果转回层序数组比较，而不是逐节点比较对象引用。
- **LC102 二叉树的层序遍历（Medium）**：BFS 的标准写法之上多一步——每轮循环开始先记录 `level_size = len(queue)`，只处理这么多个节点就是"这一层"，这一步是从"普通 BFS"升级到"分层 BFS"的关键。
- **LC105 从前序与中序遍历序列构造二叉树（Medium）**：深挖，见下。
- **LC236 二叉树的最近公共祖先（Medium）**：深挖，见下。
- **LC124 二叉树中的最大路径和（Hard）**：深挖，见下。

## 深挖 1：LC105 从前序与中序遍历序列构造二叉树

### 题意

给一棵二叉树的前序遍历序列 `preorder` 和中序遍历序列 `inorder`（节点值互不相同），把这棵树重建出来。

### 两条性质，缺一不可

**为什么前序遍历第一个元素永远是当前子树的根：** 前序遍历定义是"根 → 左 → 右"，不管子树多大，第一个被访问、被写进序列的必然是这棵（子）树的根节点本身。所以只要知道"当前正在处理 preorder 序列的哪个位置"，那个位置的值就是当前要构造的这棵子树的根。

**为什么中序遍历能把"根左边"和"根右边"切开：** 中序遍历定义是"左 → 根 → 右"，一旦知道了根节点的值，在 inorder 序列里找到这个值出现的位置 `mid`，那么 `mid` **左边的一段值，恰好就是整棵左子树所有节点的中序遍历结果**；`mid` 右边的一段，恰好是整棵右子树的中序遍历结果。这是因为中序遍历不会把左右子树的节点交叉排列——左子树的所有节点，不管多深，一定在中序序列里连续地排在根的左边；右子树同理排在右边。

### 组合起来的递归框架

1. 从 `preorder` 当前游标处取一个值，当作**当前子树的根**（性质 1）。
2. 在 `inorder` 里查这个值的位置 `mid`，把 `inorder` 切成 `[in_left, mid-1]`（整棵左子树的中序序列）和 `[mid+1, in_right]`（整棵右子树的中序序列）（性质 2）。
3. **游标前进一步**，递归构造左子树；递归返回后，游标已经跳过了整棵左子树在 preorder 里占据的所有位置，此时游标指向的下一个值，正好是右子树的根——递归构造右子树。

```python
index_of = {val: i for i, val in enumerate(inorder)}   # 值 → 下标，O(1) 定位根
pre_idx = [0]                                            # 可变游标（模拟 nonlocal）

def helper(in_left, in_right):
    if in_left > in_right:
        return None
    root_val = preorder[pre_idx[0]]
    pre_idx[0] += 1
    root = TreeNode(root_val)
    mid = index_of[root_val]
    root.left = helper(in_left, mid - 1)     # 必须先递归左子树
    root.right = helper(mid + 1, in_right)   # 游标才会自动落在右子树根上
    return root
```

用哈希表预处理 `inorder` 的值到下标的映射，把"在 inorder 里查根的位置"从线性扫描的 O(n) 降到 O(1)，避免整体退化成 O(n²)——这是这题效率上容易被忽略的一环。

### 易错点

- `pre_idx` 必须是"可变的外部状态"（这里用长度为 1 的 list 模拟），不能用普通参数传值——如果传值，左子树递归内部对游标的修改，不会体现在紧接着的右子树递归里，会导致右子树用错误的 preorder 位置取根。
- 递归终止条件必须是 `in_left > in_right`，写成 `>=` 会在只剩一个节点时提前判定为空，丢掉这个节点。

## 深挖 2：LC124 二叉树中的最大路径和——两个容易搞混的量

### 题意

"路径"指沿着节点间的边走的一个序列，同一节点在一条路径里最多出现一次，路径不要求经过根节点。求所有路径的节点值之和的最大值。

### 这题最容易搞混的地方

新手写这题最常见的错误是把两个不同的量混成同一个东西：

1. **"以某节点为拐点、经过该节点的最大路径和"**——这条路径可以同时向左、向右两个方向延伸，形状像 `⋀`。这是"最终候选答案"，每个节点都要算一次，取全局最大值。
2. **"某节点能贡献给它父节点的最大链和"**——链的定义是只能往**一个方向**延伸，不能同时往左又往右。为什么不能两边都要：因为如果父节点把"我"当作它自己路径的一部分往上继续延伸，"我"这里只能提供一条腿接上去，两条腿都占用的话路径会在"我"这里分叉，就不再是一条合法的简单路径了。

**这两个量必须分开处理**：递归函数**返回**的是量 2（只能一条腿，因为要喂给父节点继续往上拼接）；但在计算的过程中，**顺手**用量 1（两条腿都算上）去更新一个全局最大值。

```python
best = float("-inf")

def gain(node):
    nonlocal best
    if node is None:
        return 0
    left_gain = max(gain(node.left), 0)    # 负贡献不如不要这条腿
    right_gain = max(gain(node.right), 0)
    best = max(best, node.val + left_gain + right_gain)   # 量 1：两条腿都算，更新全局答案
    return node.val + max(left_gain, right_gain)           # 量 2：只留一条腿，返回给父节点
```

子树的 `gain` 如果是负数，还不如直接不要这条腿（对总和只有负贡献），所以子树的 gain 要先和 `0` 取 `max` 再使用——这对应"路径完全可以只往一个方向延伸、甚至只用当前节点自己"。

### 易错点

- 把"返回值"和"更新 best 时用的值"写成同一个表达式（都用两条腿的和），会把"经过某节点之后又分叉成两支"的不合法路径当作合法的链继续传给父节点，导致父节点那一层算出不存在的路径。
- 忘记把子树的负 gain 截断成 0，遇到全是负数的树时结果会算错——正确答案应该退化成"树中权值最大的单个节点"。

## 小结

7 道题看起来分散，但都能装进"递归函数对子树回答什么问题 + 怎么组合"这一句话里。遇到新的二叉树题，先别急着写代码，先把这句话在纸上填空填清楚：**返回值是什么类型/含义？空节点（base case）返回什么？非空节点怎么用左右子树的返回值算出当前层的返回值？** 这三个空填完，代码基本就是直译。
