# L04 · 字符串 —— 双指针原地处理 + 中心扩展找子串性质

> 对应代码：`src/problems/p04_strings.py`。先读完这篇 lecture 再去看代码/写代码，尤其是两道深挖题。

## 这一类的两大母题

字符串题看起来五花八门（反转、找前缀、找回文、转数字……），但新手能从这 6 道题里先抓住
两个反复出现的母题，后面遇到没见过的字符串题也能先往这两个方向想：

### 母题一：双指针原地处理

当题目要求"原地修改"或者"从两端向中间收敛"时，用两个下标 `left`/`right` 分别指向
待处理区间的两端，每一步做点什么（交换、比较、收缩），再向中间靠拢，直到相遇。
`reverse_string`（LC344）就是最纯粹的例子：交换首尾、指针内移，循环到 `left < right`
不成立为止。这个骨架在数组、链表题里会反复出现（比如判断回文串、盛水容器）。

### 母题二：中心扩展 / 用一次扫描找出子串的某种"性质"

很多字符串题问的是"某个具有特定性质的子串"（最长回文、最长公共前缀、转换后的排布……），
暴力解法通常是"枚举所有子串再逐一检查"，是 O(n^2) 甚至 O(n^3)。优化的核心思路往往是：
**能不能不重新扫一遍整个子串，而是从上一次已知的结果增量推出这一次的结果？**
最长回文子串的中心扩展法就是这个思路的典型代表——扩展失败就立刻停止，不会把已经验证
过的字符再验证一遍。

## 深挖 1：最长回文子串（LC5，Medium）—— 为什么中心扩展要分奇偶两种中心

**先想清楚"回文"的结构**：一个字符串是回文，当且仅当它关于某个"中心"左右对称。
问题是，这个中心可能是：

- **一个字符**（奇数长度回文，如 `"aba"`，中心是下标 1 的 `'b'`）；
- **两个字符之间的空隙**（偶数长度回文，如 `"abba"`，中心在下标 1、2 之间，
  这个"中心"不对应字符串里任何一个具体下标）。

如果你只写了"以下标 i 为中心向两边扩展"这一种情况，`"cbbd"` 这个样例就会算错——
它的答案 `"bb"` 是偶数长度回文，中心根本不落在任何一个字符下标上，纯奇中心扩展永远
找不到它。

**解决办法**：对每一个下标 `i`（一共 n 个下标），把它当作两种"中心候选"分别扩展一次：

- `expand(i, i)` —— 把下标 `i` 本身当奇中心；
- `expand(i, i + 1)` —— 把下标 `i` 和 `i+1` 之间的空隙当偶中心。

一共产生 `2n - 1` 个"候选中心"（n 个字符中心 + n-1 个空隙中心），覆盖了所有可能的
回文中心位置，不会漏掉任何一种长度奇偶性的回文。

```python
def expand(left: int, right: int) -> str:
    while left >= 0 and right < len(s) and s[left] == s[right]:
        left -= 1
        right += 1
    return s[left + 1:right]      # 注意：循环退出时 left/right 已经多走一步
```

**为什么退出循环后要用 `s[left+1:right]` 而不是 `s[left:right]`**：`while` 循环
是"先检查条件，条件成立才继续扩"，所以循环退出的那一刻，`left`/`right` 已经指向了
一个**不满足回文条件（或者越界）**的位置——也就是说，上一轮真正有效的回文区间是
`[left+1, right-1]`（闭区间），写成切片就是 `s[left+1:right]`。如果直接写
`s[left:right]`，要么会多算一个不属于回文的字符，要么在 `left=-1` 时用负数下标
切片得到错误的结果（Python 里 `s[-1:right]` 不会报错，但语义完全错了，这个坑很隐蔽，
必须靠"想清楚循环不变量"而不是"跑一下试试"来发现）。

**复杂度**：n 个中心，每个中心最坏扩展 O(n)，总体 O(n^2) / O(1) 额外空间。这不是
最优解（最优是 Manacher 算法 O(n)），但对新手来说，中心扩展法"正确性直观、代码量小"，
是这道题的第一层解法，足够应付大多数场景。

## 深挖 2：字符串转换整数 atoi（LC8，Medium）—— 为什么"看起来简单"最容易漏条件

`my_atoi` 这道题算法本身没有任何技巧——从头到尾顺序扫描一遍字符串。它真正的难度在于：
**题目文字描述里藏着五六条隐藏规则，每漏掉一条，代码在某个边界样例上就会答错**。这正是
工业界代码审查里最常见的一类 bug 来源：不是算法错了，是"没想全需求"。把所有边界条件
列全，比想出巧妙算法更重要：

| # | 边界条件 | 出错时的典型表现 |
|---|---|---|
| 1 | 空字符串 / 纯空格字符串 | 跳过空格后下标越界，或者忘记检测直接返回 0 |
| 2 | 前导空格之后允许一个 `+`/`-`，但只能有一个 | 把 `"+-2"` 错误地解析出数字 |
| 3 | 数字中间/之后出现非数字字符要立即截断 | 把 `"4193 with words"` 误解析成 0 或报错，而不是截断在 4193 |
| 4 | 只有符号没有数字（`"+"`、`"   -"`） | 误当作有效输入返回非 0 值，应该判定为无效输入返回 0 |
| 5 | 结果超出 32 位有符号整数范围 `[-2^31, 2^31-1]` | Python 大整数不会自动溢出，必须手动检测并截断，否则会返回一个"正确但不该出现"的超大数 |

**按题目描述的顺序，一步步"消耗"字符串**，就不会漏掉任何一条：

```python
i, n = 0, len(s)
while i < n and s[i] == " ":      # 第 1 步：吃掉前导空格
    i += 1
if i == n:                        # 边界条件 1：吃完空格已经到头
    return 0

sign = 1
if s[i] in "+-":                  # 第 2 步：吃掉一个符号（最多一个）
    sign = -1 if s[i] == "-" else 1
    i += 1

num = 0
has_digit = False
while i < n and s[i].isdigit():   # 第 3 步：吃数字，遇到非数字立刻停（自然实现边界条件 3）
    has_digit = True
    num = num * 10 + int(s[i])
    i += 1
    if num > 2**31:                # 第 4 步（溢出早停）：边扫描边检测，不用等读完整串
        break

if not has_digit:                 # 边界条件 4：只有符号没数字
    return 0

num = sign * num
return max(-2**31, min(num, 2**31 - 1))   # 边界条件 5：最后夹紧到 32 位范围
```

**为什么溢出检测要"边读边测"而不是"读完整串再测"**：如果输入是 200 个 `'9'` 拼成的
字符串（LC8 允许输入长度较长），先把这 200 位数字全部转成 Python 大整数再比较，虽然
Python 不会报错，但这是在"没有意义地做大量无用功"——数字位数只要超过 11 位左右
就已经必然溢出 32 位范围，没必要继续往后累加。更重要的是，这个思路能迁移到那些
真的会整数溢出的语言（C/C++/Java）——在那些语言里，"读完再判断"这件事本身就会
因为溢出而得到错误的中间结果，"边读边判断，一旦确定溢出就提前退出"才是唯一正确的写法。

## 其余 4 题点睛

- **LC344 反转字符串**：母题一的直接应用，双指针交换到中间相遇，`_self_test` 里验证
  的是"调用后原列表被修改"而不是返回值，这是"原地"类题目的通用考点。
- **LC14 最长公共前缀**：本课用 `zip(*strs)` 做"纵向扫描"——把字符串数组看成按位置
  对齐的多行文本，逐列比较是否全部相同，一列不一致就立刻停。这个技巧比手写双重循环
  更简洁，还顺带处理了"字符串长度不一"（`zip` 遇到最短的字符串自动停止）和"空数组"
  （`zip()` 无参时直接是空迭代器）两个边界，不需要额外写 `if`。
- **LC151 反转字符串中的单词**：坑不在"怎么反转"，而在"怎么处理多余空格"——Python
  的 `s.split()`（不传参数）会自动按任意长度空白分割并丢弃空字符串，直接消掉了这道题
  一半的边界处理，比手写状态机可靠得多。
- **LC6 Z 字形变换**：不需要真的建二维网格，只需要一个"当前行号"加一个"当前方向"
  两个状态变量，行号碰到顶或底就反向——这是"用状态变量模拟结构，而不是真的构造结构"
  的常见技巧，`num_rows == 1` 时必须单独特判（不存在"折返"这回事）。
