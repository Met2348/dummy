# L16 · 单调栈与单调队列

> 对应代码：`src/problems/p16_monotonic_stack.py`

## 这一类到底在解决什么问题

单调栈（monotonic stack）和单调队列（monotonic queue/deque）表面上是两种不同的数据结构用法，但解决的都是同一类问题的两个方向：

> **对数组里的每一个位置，找"左边"或"右边"第一个比它大（或比它小）的元素。**

暴力解法是对每个位置都往目标方向线性扫一遍——O(n) 每个位置，n 个位置总共 O(n²)。这里面的浪费在哪？想一想：如果位置 `i` 往右扫的过程中路过了位置 `j`（`j > i`），发现 `nums[j] < nums[i]`，于是继续往右扫——但当未来某个位置 `k` 要往右找"第一个更大的元素"时，`j` 这个点会不会被**重新扫一遍**？会。`j` 被 `i` 扫过一次，可能还会被 `i` 左边更多的位置重复扫到。

单调栈/单调队列的核心思想是：**维护一个内部"单调"（只递增或只递减）的候选序列，让每个元素最多被压入、弹出各一次**。一旦某个元素被弹出，就说明它的答案已经被"结算"完毕，以后不会再被任何人扫到——这样均摊下来，n 个元素总共只有 O(n) 次入栈/出栈操作，整体降到 O(n)。

这类问题的识别信号：题目里出现"下一个更大/更小元素""某个区间的最值""向左/向右能扩展到多远"这类字眼，且需要对**每个位置**都求一次答案时，先考虑单调栈/队列，而不是对每个位置都暴力扫一遍。

## 深挖例题一：每日温度（为什么这样扫能保证每个下标只处理一次）

**题目**：给一串每日气温，对每一天求"还要等几天才会出现比今天更暖和的气温"，等不到记 0。

**从暴力开始想**：对第 `i` 天，往右扫，找到第一个 `temperatures[j] > temperatures[i]` 的 `j`，答案是 `j - i`。这是 O(n²)，最坏情况（温度严格递减）每天都要扫到底。

**找哪里在重复计算**：假设温度序列是 `[73, 74, 75, 71, 69, 72, 76, 73]`。当我们处理到下标 3（温度 71）时，其实我们并不知道"71 后面第一个更大的数在哪"——但当我们后面扫到下标 5（温度 72，比 71 大）时，就能立刻确定：下标 3 的答案就是"等到下标 5"。**这个"确定"的时刻，是由右边出现的一个更大的数触发的，而不是由下标 3 自己主动往右找到的。**

这正是单调栈要做的事：**把"还没找到答案的日子"暂存到栈里，等右边出现一个能同时解决它们疑问的更大温度时，一次性结算掉。**

具体做法：
1. 从左到右扫描，栈里存**下标**（不是温度值——因为答案需要"等待天数"，也就是下标之差，光有温度值算不出天数）。
2. 维护栈内下标对应的温度从栈底到栈顶**递减**。
3. 来一个新的下标 `i`，温度 `t`：只要栈顶的温度比 `t` 低，说明栈顶那天的疑问被解决了——弹出栈顶下标 `j`，`res[j] = i - j`，重复弹出直到栈顶温度不低于 `t`（或栈空）。
4. 把 `i` 压入栈。

**为什么每个下标只处理一次**：一个下标 `j` 被压入栈之后，只有在被弹出时才会"处理"一次（结算答案），而一旦弹出就再也不会被压回去。压入 n 次、弹出最多 n 次，总操作数 O(n)。栈内递减这个性质保证了：只要新元素比栈顶大，栈顶就一定能被立刻结算——不会出现"该结算但没结算"的遗漏。

对照代码 `daily_temperatures`：`while stack and temperatures[stack[-1]] < t:` 这一行就是"连续结算所有能被结算的栈顶"，不能写成只判断一次就 break——一次温度骤升可能同时解决好几天的疑问。

## 深挖例题二：柱状图中最大的矩形（为什么必须用单调栈而不是暴力两层循环）

**题目**：一排柱子，宽度都是 1，求能画出的最大矩形（矩形高度必须等于某根柱子的高度）。

**暴力思路**：对每根柱子 `i`，以 `heights[i]` 为高度，分别向左、向右扩展，直到遇到比它矮的柱子为止，算出这根柱子能撑起的最大矩形宽度。这是对的，但每根柱子的"向两边扩展"都是独立的一次线性扫描——O(n) 每根柱子，O(n²) 总共。

**为什么不能简单优化成"记住上一根柱子扩展到哪"**：因为柱子高度忽高忽低，柱子 `i` 能扩展到的左边界和柱子 `i-1` 能扩展到的左边界之间没有简单的递推关系——除非我们能快速知道"柱子 `i` 左边第一个比它矮的柱子在哪"。

**这正是单调栈的强项**：柱子 `i` 能向左扩展到的边界 = "左边第一个比 `heights[i]` 矮的柱子的位置 + 1"；能向右扩展到的边界 = "右边第一个比 `heights[i]` 矮的柱子的位置 - 1"。这是两个"第一个更小的元素"问题，和"每日温度"的"第一个更大的元素"是同一种母题,只是方向反过来。

具体做法：
1. 维护一个下标栈，栈内对应的高度从栈底到栈顶**递增**（注意方向和"每日温度"相反,那题找的是"更大",这题找的是"更矮",所以栈的单调方向也相反）。
2. 新柱子 `i` 的高度如果比栈顶矮，说明栈顶那根柱子的**右边界**被确定了（右边第一个更矮的就是 `i`）——弹出栈顶 `top`,这时候新的栈顶就是"左边第一个比 `heights[top]` 矮的柱子的下标"，于是宽度 `= i - stack[-1] - 1`（如果栈为空,说明左边没有更矮的,宽度就是 `i` 本身）。
3. 用 `heights[top] * width` 更新答案。
4. 为了让"高度一路递增到数组末尾"的柱子也能被结算，在数组末尾**补一个高度 0 的哨兵**——这样所有柱子最终都保证会被弹出、结算一次，不需要在循环结束后再单独处理栈内剩余元素。

**为什么两层循环不行、单调栈可以**：两层循环里,"柱子 i 的左右边界"每次都要重新扫一遍已经扫过的区域;而单调栈利用了"栈内单调递增"这个不变量——一旦某根柱子在栈里,就说明它还没找到比它矮的柱子,它可以放心地留在栈里等右边的答案自动送上门,不需要主动去扫。每根柱子入栈一次、出栈一次,整体 O(n)。

## 点睛：滑动窗口最大值

`max_sliding_window` 会在下面"单调栈 vs 单调队列"里详细对比,这里先给结论：它和前三题的本质区别是**需要支持"窗口滑动时,移除队首那些已经滑出窗口范围的过期候选"**——这个"从另一端移除"的操作,单调栈（只能从一端进出的 LIFO 结构）做不到,必须换成双端队列。

## 单调栈 vs 单调队列：什么时候用哪个

| | 单调栈（LIFO） | 单调队列（双端队列） |
|---|---|---|
| 数据结构 | 普通栈,只从一端进出 | `collections.deque`,两端都能进出 |
| 典型问题 | "左边/右边第一个更大/更小的元素"（每日温度、柱状图最大矩形、下一个更大元素） | 需要"窗口"概念的最值问题（滑动窗口最大值） |
| 为什么够用/为什么不够用 | 这类问题只需要"新元素来了,把栈顶不满足单调性的元素弹出结算",不需要从另一端移除任何东西 | 窗口每滑动一格,不仅要在队尾维护单调性,还要检查**队首**的候选是否已经滑出窗口范围,需要从队首移除——这是纯栈结构做不到的操作 |

一句话记忆：**问题里如果有"窗口"（有固定大小、会滑动的区间）,候选值有可能因为"太老、滑出窗口了"而失效,就必须用双端队列；如果只是单纯地"找左边或右边第一个满足条件的元素",没有"窗口过期"这个概念,用普通栈就够,还更省心（不需要判断队首过期）。**

## 小结

1. 单调栈/单调队列的通用降复杂度手法:维护内部单调性,让每个元素只入队/入栈、出队/出栈各一次,把 O(n²) 的暴力压缩到 O(n)。
2. 找"下一个更大/更小元素"用单调栈,方向(递增栈找更小边界、递减栈找更大元素)要想清楚再动笔。
3. 涉及"滑动窗口"且要维护窗口内最值时,用单调双端队列,多一步"移除队首过期下标"的检查。
4. 栈/队列里存**下标**还是**数值**,取决于答案是否需要下标信息(等待天数、宽度都需要下标之差;下一个更大元素的值本身够用,存数值更直接)。

### 深挖：最大矩形——为什么二维题能"降维"成多次一维柱状图最大矩形

**题目**：给一个 `'0'`/`'1'` 组成的二维矩阵，求只由 `'1'` 构成的最大矩形面积。

**关键转化：把每一行都看成一张柱状图的"地面"**。对矩阵的第 `r` 行、第 `j`
列，定义"高度" `heights[j]` = 从第 `r` 行往上数，连续有多少个 `'1'`——如果
第 `r` 行第 `j` 列本身是 `'0'`，高度直接归零（因为地面出现了缺口，往上的
`'1'` 再多也不可能和这一行的地面连成一个矩形）；如果是 `'1'`，高度在上一行
的基础上 `+1`（累积"连续为 1"的计数）。逐行更新这个 `heights` 数组，每处理
完一行，`heights` 就代表"以这一行为底"的柱状图——这正好是 84 题"柱状图中
最大的矩形"的输入，直接复用那道题的单调栈解法（维护高度递增的下标栈，柱子
`i` 的左右边界分别是"左边/右边第一个比它矮的柱子"）算出这一行能撑起的最大
矩形，用哨兵 0 保证栈最终清空结算。

**为什么"逐行求一次一维最大矩形"能覆盖二维矩阵里的所有答案**：原矩阵里任何
一个全 `1` 矩形，一定存在一条"最底下的行"（矩形的下边界），而这一行对应的
`heights` 数组，在这个矩形的列范围内，高度必然都 `>=` 矩形的高度（因为矩形
本身要求这些位置往上连续为 1 达到矩形高度）——也就是说，这个矩形一定会被
"以矩形底边所在行为地面"的那次柱状图最大矩形计算捕捉到（柱状图算法本身就是
在找"每根柱子能向左右扩展、高度不超过它自己的最大矩形"，覆盖了所有可能的
子矩形）。所以只要对每一行都算一次柱状图最大矩形，取全局最大值，就不会漏掉
任何一个二维全 1 矩形——这是"把二维问题拆解成多个独立的一维子问题，各自求解
后取最优"的典型手法，之后遇到"二维网格 + 某种一维上已经会做的结构"的题目
（比如"矩阵中的最大正方形"要求的高度换成正方形约束）时，都可以先想一想能不能
做类似的逐行/逐列降维。

### 深挖：拼接最大数——为什么要拆解成"单调栈选最大子序列"+"两个有序子序列的最大归并"两步

**题目**：给定两个数组 `nums1`、`nums2`（各表示一个数字的各位数字），和
整数 `k`，从两个数组里一共选出 `k` 个数字（可以只选一个数组的，也可以两边
都选），必须保持各自数组内部的相对顺序，使拼出的整体数值最大。

**为什么不能直接暴力枚举**：如果直接枚举"从 `nums1`、`nums2` 里各选哪些
下标"，组合数是指数级的，完全不可行。需要把问题拆解成两个规模更小、各自
可解的子问题。

**拆解第一步：单一数组内"选固定长度的最大子序列"，这是单调栈的经典应用**。
如果只考虑一个数组，要从中选出长度为 `L` 的子序列（保持相对顺序）使其代表
的数值最大，做法和 Part II 的"移掉K位数字"（14章）、"最具竞争力的子序列"
（16章 Part II）同源：维护一个"从底到顶非递增"的单调栈，遇到新数字时，
只要栈顶数字比它小、且删除后剩下的元素还凑得够长度 `L`，就弹出栈顶——这样
留下来的子序列在"保持相对顺序"的约束下是数值最大的。这一步保证了：不管
最终答案怎么从两个数组分别拿走多少个数字，从单个数组里拿走的那部分，一定
应该是"这个数组在这个长度下能选出的最大子序列"（否则换成这个最大子序列，
结果只会更好不会更差）。

**拆解第二步：把两个"数组各自的最大子序列"归并成一个长度为 k 的最终结果，
但这不是普通的归并排序**。假设从 `nums1` 选出了长度 `i` 的最大子序列 `a`，
从 `nums2` 选出了长度 `k-i` 的最大子序列 `b`，现在要把 `a`、`b` 交织合并成
一个长度为 `k` 的序列，且各自保持内部顺序，使整体数值最大——这看起来像
"归并两个有序列表"，但关键区别在于：**每一步该从 `a` 还是 `b` 取当前元素，
不能只比较 `a[i]` 和 `b[j]` 这两个数字本身，而要比较"两个子序列从当前位置
开始的剩余后缀，谁的字典序更大"**。原因是：如果 `a[i] == b[j]`，选哪一边
完全取决于"再往后看，谁的后续数字更能占便宜"——比如 `a` 后面紧跟着一个 9、
`b` 后面紧跟着一个 3，即使 `a[i]` 和 `b[j]` 现在相等，也应该先取 `a[i]`
（因为紧接着的 9 更大）。这个"比较剩余后缀字典序"的归并规则，正是这道题
和普通归并排序最大的不同点。

**为什么要枚举所有的切分方式 `i`**：由于无法提前知道"最优解到底应该从
`nums1` 拿几个",只能枚举 `i` 从 `max(0, k-n)` 到 `min(k, m)` 的每一种可能
切分，每种切分都独立完成"两次单调栈选最大子序列 + 一次归并"，最后在所有
候选结果里取字典序最大的一个。

  第一阶段只负责收集候选、不求解，第二阶段才真正配对求解。
- **LC1124 表现良好的最长时间段**：把"劳累日/摸鱼日"转化成 +1/-1 的前缀和，
  问题变成"找最长的区间使前缀和之差 > 0"——本题严格来说是"前缀和 + 哈希表"
  而非单调栈，但因为前缀和每步只变化 ±1 这个特性，使得只需要查 `s[i]-1`
  而不是任意更小的历史值，这个"利用相邻步长有限"的技巧和单调栈"只需要维护
  局部单调性、不需要全序"的精神是相通的，因此归入本章一起对比学习。
- **LC1499 满足不等式的最大值**：单调队列优化的经典应用——把要最大化的式子
  拆成 `(yi - xi) + (xj + yj)` 两部分，用一个单调递减的双端队列维护"窗口内
  `yi - xi` 的最大值"，队首过期（超出 x 的窗口范围）就弹出，这是"滑动窗口
  最大值"（Part I）在窗口条件从下标距离换成坐标距离之后的变体。
- **LC907 子数组最小值之和**：单调栈的"贡献法"应用——不枚举子数组，而是
  枚举每个元素，用两次单调栈算出它能"称霸"（作为最小值）的左右边界范围，
  贡献次数 = 左边界宽度 × 右边界宽度，注意左右两侧的比较符号必须"一边严格、
  一边不严格"，否则重复元素会被重复计数。

## 练法（Part III）

拼接最大数建议先脱离代码，自己动手用两个很短的数组（比如长度 2 和长度 3）
手推一遍"单调栈选最大子序列"和"归并两个子序列"这两步分别做了什么，尤其要
构造一个"两个子序列当前位置数字相等、但后续数字不同"的例子，体会为什么归并
时必须比较"剩余后缀"而不能只比较当前数字——这是这道题和普通归并排序唯一
但也是最容易出错的区别。
