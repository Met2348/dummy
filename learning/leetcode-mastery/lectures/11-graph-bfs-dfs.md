# L11 · 图 BFS/DFS

> 对应代码：`src/problems/p11_graph_bfs_dfs.py`。图题给新手的第一印象是"不知道从哪
> 下手"，因为题目往往不会明说"这是一道图题"。本讲先建立两条选择依据，再挑两道题
> 深挖"图到底藏在哪里"。

## 选择依据 1：BFS 还是 DFS

- **BFS（广度优先）**：一层一层向外扩展,天然保证"第一次到达某个节点时,走过的步数
  就是从起点到它的最短步数"（前提是每条边权重相同,都算"一步"）。凡是题目问
  **"最短路径""最少步数""最少操作次数"**,第一反应应该是 BFS,而不是 DFS+剪枝。
  本类里的腐烂的橘子（最少分钟数）、单词接龙（最短变换序列长度）都是这个模式。
- **DFS（深度优先）**：沿一条路径走到底,走不通再回退,适合**"判断连通性""穷举
  所有路径/所有解""给图打标记"**这类不关心"最短",只关心"能不能到达"或者
  "有哪些"的问题。本类里的岛屿数量（求连通块个数）、克隆图（遍历并复制整张图）
  都是这个模式；courses 系列虽然也可以用 DFS 判环,但本讲统一用 BFS 版本的
  Kahn 算法,因为它和"最短路"系列的 BFS 直觉是一致的,更容易迁移。

## 选择依据 2：先想清楚"图长什么样"

真正决定这道题好不好下手的,往往不是"BFS 还是 DFS",而是**图的表示方式**。图有
三种常见出场方式,认出属于哪一种,后面写代码只是体力活：

| 表示方式 | 例子 | 怎么找"邻居" |
|---|---|---|
| 网格（隐式的图，每个格子是节点） | 岛屿数量、腐烂的橘子 | 上下左右四个方向 |
| 显式邻接表（节点带 `.neighbors`） | 克隆图 | 直接读节点自带的邻居列表 |
| 编号 + 边列表（需要自己建邻接表） | 课程表、课程表 II | 遍历边列表，建 `dict[节点]->[节点]` |
| 完全隐式（节点/边都要现算） | 单词接龙 | 每个单词是节点；边是"改一个字母能不能变成词表里的另一个词"，运行时现枚举 |

越往下表格里的"图"越抽象、越不像传统意义上的图，也越容易让新手看不出"这是图题"。

## 深挖 1：课程表——"能否学完所有课程"为什么等价于"图里有没有环"

**题意翻译成图论语言**：`prerequisites[i]=[a,b]` 表示"先学 b 才能学 a"，把这读
成一条有向边 `b -> a`。于是"能不能安排出一个学习顺序，把所有课程都学完"，翻译成
图论问题就是——**这张有向图存在一个拓扑序吗？而"存在拓扑序"和"图中无环"是完全
等价的两件事**。

直觉解释为什么"有环 = 学不完"：假设 a 依赖 b、b 依赖 c、c 又依赖 a，这三门课两两
互相牵制，无论从哪一门开始都会发现"它的前提还没学"，形成死锁——没有任何一门课
能作为"最早可以学"的起点。反过来，如果整张图无环，一定存在至少一个"没有任何前提"
的课程（入度为 0 的节点），可以以它为起点，学完它之后再看它解锁了哪些课……

**Kahn 算法就是把这个直觉翻译成 BFS**：

```python
graph: dict[int, list[int]] = defaultdict(list)
indegree = [0] * num_courses
for a, b in prerequisites:      # b -> a
    graph[b].append(a)
    indegree[a] += 1

q = deque(i for i in range(num_courses) if indegree[i] == 0)   # 一开始就能学的课
visited = 0
while q:
    cur = q.popleft()
    visited += 1                # "学完"这门课
    for nxt in graph[cur]:
        indegree[nxt] -= 1      # 它解锁的后续课程，前提少了一个
        if indegree[nxt] == 0:  # 前提全部满足，现在可以学了
            q.append(nxt)
return visited == num_courses
```

**为什么"入度归零就入队"能正确工作**：一门课的入度，代表"它还有多少个前提没有
被学完"。当它的入度减到 0，意味着它所有前提课程都已经在之前的迭代里被处理过、
入过队、出过队——这正是拓扑序的定义（每个节点排在它所有前驱之后）。

**为什么最后要比较 `visited == num_courses` 而不是直接看"队列一开始有没有东西"**：
如果图里有一个环，环上所有课程的入度永远不可能被减到 0（环上每门课都在等环上的
另一门课，谁都排不到"最早"），所以它们永远不会入队、永远不会被处理。`visited`
最终会小于 `num_courses`，这正是"图中有环"这一事实在代码里的体现——**不需要额外
写一个"判环"的函数，Kahn 算法的副产品就是环检测**。

`find_order`（课程表 II）是同一套算法，只是把 `visited += 1` 换成
`order.append(cur)`——BFS 出队的顺序本身就是一个合法拓扑序。

## 深挖 2：单词接龙——图是"隐式"的，为什么 BFS 还能保证最短

这题最容易让新手卡壳的地方是：**题目里根本没有"节点"和"边"这两个词，甚至没有
一个显式的图结构可以打印出来看。** 但只要做一个思维翻译：

- **节点**：`word_list` 里的每一个单词（外加起始单词 `begin_word`）。
- **边**：两个单词之间，如果恰好只有一个字母不同，就存在一条边（对应"一步变换"）。

一旦想清楚这一层翻译，"求从 `begin_word` 到 `end_word` 的最短变换序列长度"就变成
了标准的"无权图最短路"问题——而 BFS 正是无权图最短路的标准解法，原因和课程表里
"入度归零就入队"的直觉类似：**BFS 逐层扩展，同一层里的所有节点到起点的步数完全
相同（都等于当前层数），所以 BFS 第一次碰到 `end_word` 的那一层，步数必然是所有
能到达 `end_word` 的路径里最短的那个**——不可能存在一条更短的路径还没被发现，
因为如果真的存在，它一定会在更早的某一层就把 `end_word` 加入队列。

**这张图从来没有被真正"建出来"过**，因为如果真的枚举每两个单词是否只差一个字母，
边的数量是 O(N²)（N=词表大小），代价很高。更聪明的做法是**现算现用**：拿到当前
单词后，对它的每一位（`len(word)` 个位置）尝试替换成 a-z 里的其他 25 个字母，
拼出一个新字符串，查它是否在词表的 `set` 里——如果在,就说明这条边存在,直接把
新单词当成邻居入队：

```python
for i in range(len(word)):
    for ch in "abcdefghijklmnopqrstuvwxyz":
        if ch == word[i]:
            continue
        nxt = word[:i] + ch + word[i + 1:]
        if nxt in word_set and nxt not in visited:
            visited.add(nxt)
            q.append((nxt, steps + 1))
```

这种"边不预先存储、需要时现枚举"的写法在图论题里很常见（尤其是节点空间巨大、
边的判定规则简单的场景），值得作为一种通用技巧记下来：**图不一定是一份数据结构，
它也可以是一条"怎么从当前状态生成下一个状态"的规则。**

## 其余 4 题点睛

- **岛屿数量 `num_islands`**：扫描到未访问的陆地就计数 +1，然后 DFS 把这片陆地
  全部"淹没"（原地改成 `'0'`），防止同一个连通块被重复计数。淹没操作让 visited
  集合变得多余，省一份额外空间，但代价是会修改传入的网格——如果调用方还需要原始
  数据，记得传拷贝。
- **克隆图 `clone_graph`**：图和树最大的区别是"可能有环"，用一个"原节点 val →
  克隆节点"的哈希表在 DFS 前先查表，查到就直接返回已克隆的节点、不再往下递归——
  这一步"先查表再决定要不要递归"，正是遍历有环图和遍历树最核心的区别。
- **课程表 II `find_order`**：算法与课程表完全一样，只是把"计数"换成"记录出队
  顺序"。注意拓扑序不唯一，校验时要检查"每条依赖边的相对顺序"，不能死板比较某个
  具体序列。
- **腐烂的橘子 `oranges_rotting`**：**多源 BFS**——把所有初始腐烂的橘子同时作为
  第 0 层入队（而不是只从一个橘子开始），因为它们是同时开始腐烂邻居的。BFS 每
  扩展一层，恰好对应过去了一分钟。

## 深挖：太平洋大西洋水流问题——为什么"反过来想"比正向判断简单得多

**正向思路**（大多数人第一反应）：对网格里每一个格子，分别判断"水从这个格子出发，
沿着'流向更低或相等的相邻格子'这条规则，能不能走到太平洋边界；能不能走到大西洋
边界"。这个思路完全正确，但代价很高——每个格子都要独立做一次 DFS/BFS，R·C 个
格子各花 O(R·C)，总时间 O((R·C)²)，网格稍微大一点就会超时。

**反向思路**（本题真正的考点）：仔细想"水从格子 A 流到太平洋边界"这句话，它和
"从太平洋边界出发，反着水流方向走，能走到格子 A"其实是**同一件事**——只是描述
的视角不同。既然如此，与其对每个格子单独问一次"你能不能流到边界"，不如反过来
一次性问：**"从边界出发，能反向走到哪些格子？"** 这样只需要从太平洋边界的所有
格子、大西洋边界的所有格子分别做一次多源 BFS，就能同时得到"所有能流到太平洋的
格子"和"所有能流到大西洋的格子"，两个集合取交集就是答案——总时间降到 O(R·C)。

反向搜索最容易出错的地方是**移动规则也要跟着反过来**：

```python
# 正向水流规则：从当前格子流向"更低或相等"的相邻格子
# heights[nxt] <= heights[cur]

# 反向搜索规则：从边界向内走，能走到"更高或相等"的相邻格子
# 因为如果 heights[nxt] >= heights[cur]，就说明正向水流时，
# 水是可以从 nxt 流向 cur 的——我们只是"倒着放电影"而已
if heights[nxt] >= heights[cur]:
    ...
```

一句话理解：**如果正向规则是"A 能走到 B 当且仅当 height(B) <= height(A)"，那么
"反过来从 B 出发找所有能到达的 A"，规则就是"height(A) >= height(B)"——本质上是
同一个不等式，只是问的方向换了**。这是网格题里"正向想很难、反向想很简单"的经典
案例，之前的**被围绕的区域**（先从边界反向 DFS 标记"安全"的 O，而不是对每个内部
O 分别判断是否连到边界）也是同一种反向思维，可以对照着理解。

## 其余 11 题点睛

- **被围绕的区域 `solve_surrounded_regions`**：和"太平洋大西洋"是同一种反向
  思维——先从边界出发 DFS 标记"安全"的 O（用第三态 `#` 区分"安全"和"待处理"），
  再把没被标记的 O 一律改成 X，最后把 `#` 复原成 O。
- **岛屿的最大面积 `max_area_of_island`**：和 Part I 的 `num_islands` 是同一套
  连通块遍历骨架，区别是 DFS 函数要返回"这个连通块的大小"而不是只做计数 +1，
  外层扫描时用 `max()` 而不是简单累加。
- **二进制矩阵中的最短路径 `shortest_path_binary_matrix`**：和"腐烂的橘子/
  单词接龙"一样是 BFS 求无权最短路，唯一变化是移动方向从 4 个变成 8 个（加上
  4 个对角线方向），别忘了特判起点/终点本身是障碍的情况。
- **矩阵中的最长递增路径 `longest_increasing_path`**：DFS + 记忆化的标准写法——
  "以某个格子为起点的最长递增路径长度"是一个固定值，和"怎么到达这个格子"无关，
  第一次算出来后存进 `memo`，之后任何路径再次经过直接查表，避免指数级重复计算。
- **01 矩阵 `update_matrix`**：和"腐烂的橘子"是同一个多源 BFS 模式，把所有 0
  同时作为第 0 层入队；用结果矩阵本身的"是否等于 -1"兼职当 visited 标记，省掉
  额外的集合。
- **蛇梯棋 `snakes_and_ladders`**：BFS 求最短路没有难度，难点在"棋盘编号是
  之字形排列的一维序列，要先转换成二维坐标才能查蛇/梯子"——写一个
  `get_cell(label)` 专门做这层坐标转换，之后 BFS 的状态永远是"处理完蛇梯效果
  之后的最终编号"。
- **最短的桥 `shortest_bridge`**：DFS 找到第一座岛屿的所有格子作为起点集合，
  再做多源 BFS 扩展去找第二座岛屿——是"DFS 探测起点 + BFS 求最短距离"的组合，
  区别于"腐烂的橘子/01矩阵"里源头直接从输入读出，这里的源头要先算出来。
- **所有可能的路径 `all_paths_source_target`**：DAG（有向无环图）上穷举所有
  路径，因为无环所以不需要 visited 标记，是本类里少数"不用担心死循环"的 DFS。
- **寻找图中是否存在路径 `valid_path`**：最基础的连通性判断，BFS/DFS/并查集
  都可以，作为"建图+遍历"的最小可行版本，后面两题都是在它基础上叠加信息。
- **重新规划路线 `min_reorder`**：把有向边当无向边建图，但给每条边打上
  "cost"标签（顺着原方向走 cost=1，逆着走 cost=0），DFS 从首都出发累加 cost，
  就是最少需要反转的边数——"把方向信息编码进边的附加数据"是本题的核心技巧。
- **找出星型图的中心节点 `find_center`**：不需要建图和遍历，利用"中心节点在
  每条边都出现"这个结构特性，只看前两条边共享的端点即可，O(1) 解决。

| 尽量减少恶意软件的传播 `min_malware_spread` | 并查集分量（非 BFS） | 每个连通分量里 initial 节点是否唯一 |
| 不邻接植花 `garden_no_adj` | 贪心染色（非 BFS） | 度数上限 3 < 颜色数 4，保证贪心必成功 |
| 最小体力消耗路径 `min_effort_path` | 改造版 Dijkstra | 松弛用 max 而不是 + |
| 迷宫中离入口最近的出口 `nearest_exit` | `(r, c)` | 出口需在边界且不能是入口本身 |
| 网格中的最短路径-有障碍物消除 `shortest_path_with_obstacle_elimination` | `(r, c, remain)` | 剩余可消除障碍次数 |
| 获取所有钥匙的最短路径 `shortest_path_all_keys` | `(r, c, keys)` | 已收集钥匙集合（位掩码） |
| 推箱子 `min_push_box` | `(box_r, box_c, player_r, player_c)` | 箱子位置 + 玩家位置 |
| 从盒子里获得的最大糖果数 `max_candies` | 多源 BFS（非坐标） | 拥有盒子集合 + 拥有钥匙集合双向解锁 |

## 深挖：获取所有钥匙的最短路径——状态压缩 BFS 到底在压缩什么

**先看"只用坐标做 visited"错在哪**：网格里有钥匙 `'a'`、锁 `'A'`。假设起点附近
有一道锁 `'A'`，如果先走到锁的门口、这时候手上还没有钥匙 `'a'`，只能绕开继续走；
绕了一大圈之后在别处捡到了 `'a'`，理论上应该可以折返回锁 `'A'` 门口、这次直接
推门而入。**但如果 visited 只记录坐标 `(r, c)`**，锁门口这一格第一次经过（没
钥匙）时就已经被标记为"访问过"了，第二次带着钥匙折返，会被误判为"这格子已经
访问过"而拒绝再次入队——于是这条本该更短（或者唯一可行）的路径就被错误地剪掉了。

**根本原因**：坐标相同，不代表"处境"相同。同一个坐标，"带着钥匙 a 路过"和
"不带钥匙 a 路过"是两种完全不同的后续可能性——前者能穿过锁 `A`，后者不能。BFS
的 visited 去重，本质上去掉的应该是"处境完全相同、继续搜索不会有任何新发现"的
重复状态,而不是简单地"来过这个坐标"。

**状态压缩（bitmask）怎么解决**：把"携带哪些钥匙"这个信息压缩成一个整数——
钥匙最多 6 把（`'a'`~`'f'`），用一个 6 位的二进制数表示,第 `i` 位是 1 就表示
"已经拿到第 `i` 把钥匙"（`'a'` 对应第 0 位）。BFS 的状态从 `(r, c)` 扩展成
`(r, c, keys)`，`visited` 也变成对这个三元组去重：

```python
if ch.isupper() and not (keys >> (ord(ch) - ord("A")) & 1):
    continue                          # 没有对应钥匙，这道锁打不开
nkeys = keys | (1 << (ord(ch) - ord("a"))) if ch.islower() else keys
state = (nr, nc, nkeys)
if state not in visited:
    visited.add(state)
    q.append((nr, nc, nkeys, steps + 1))
```

这样"带着钥匙集合 `{a}` 路过锁门口"和"不带任何钥匙路过同一个锁门口"是两个不同
的三元组，各自都能独立地被访问、独立地继续扩展，不会互相干扰。用"整数位掩码"
而不是"钥匙的集合/列表"来表示这份额外信息，是因为**位掩码可以直接作为元组的
一个字段参与哈希和相等比较**，而集合本身不可哈希，没法直接放进 `visited` 这个
哈希集合里；而且钥匙最多 6 把，`2^6=64` 种组合，用一个 6 位整数表示比维护
一个可变集合更紧凑、判断"是否已访问"也只是一次哈希查找。

**这就是"状态压缩"这个说法的字面含义**：把原本需要用一个集合/列表才能表达的
"额外信息"（钥匙组合），压缩编码进一个可以直接参与去重比较的整数里，让 BFS
的状态空间从"坐标"扩大到"坐标 × 额外信息的所有组合"，但又不至于因为额外信息
本身太复杂（比如直接存一个 Python `set`）而没法高效去重。

## 深挖：推箱子——当额外状态不是"压缩位掩码"而是"另一个实体的完整坐标"

推箱子和获取所有钥匙都是"扩展状态空间"，但值得对比它们扩展的方式不一样：
获取所有钥匙的额外信息（钥匙集合）可以压缩进一个小整数，因为钥匙种类有限
（最多 6 种，天然适合位掩码）；**推箱子的额外信息是"玩家的完整坐标"，没有
办法压缩成几个比特，只能老老实实地把它作为状态的一部分**——状态从"箱子在
哪"扩展成"箱子在哪 + 玩家在哪"这样一个四元组 `(box_r, box_c, player_r,
player_c)`。

为什么箱子的位置不够、必须再加上玩家的位置：箱子能不能被推动、往哪个方向推，
取决于玩家能不能先走到箱子的"背后"（推动方向的反方向那一格）——玩家站在箱子
上方就只能往下推，站在箱子左边就只能往右推。如果状态只记录箱子的位置，会把
"玩家在箱子上方"和"玩家在箱子左边"这两种能推动的方向完全不同的处境合并成
同一个状态，某些必须先绕到箱子另一侧才能推动的解法就会被漏掉。

代码结构上，这道题是"BFS 里嵌套一次子 BFS/DFS"：外层 BFS 按"推动次数"分层
扩展箱子的位置；每次尝试把箱子往某个方向推之前，先用一次独立的可达性搜索
（`can_reach`）判断玩家能不能在不穿过箱子的前提下，走到箱子背后那一格。这种
"父问题的每一步转移，都依赖一次子问题的完整搜索"的结构，在原地推箱子这类
"两个实体协同移动"的题目里很常见，值得作为一种模式记住：**当状态包含多个
实体的位置时，"某个实体能不能移动"往往要先问一句"另一个实体能不能先移动到
指定的位置"，这一步天然构成了一次嵌套搜索。**

一句话对比：**获取所有钥匙是"额外信息种类少、可以压缩成位掩码"的状态扩展；
推箱子是"额外信息本身就是另一个实体的坐标、压缩不了、只能原样塞进状态元组"
的扩展——两者都是"扩展状态空间"这同一条主线索的具体应用，区别只在于扩展出来
的这部分信息能不能被压缩。**

## 其余 7 题点睛

- **颜色交替的最短路径 `shortest_alternating_paths`**：状态从 `node` 扩展成
  `(node, last_color)`，因为"经过红边到达"和"经过蓝边到达"决定了下一步只能走
  哪种颜色的边，是两种不能合并的处境；起点 0 要同时以两种虚拟颜色入队，因为
  第一步红边蓝边都可以走。
- **尽量减少恶意软件的传播 `min_malware_spread`**：本类里少有的"并查集主导"
  的题——感染会传满整个连通分量，只有"独占一个分量"的 `initial` 节点被移除
  才有收益，收益等于分量大小；多个节点都不独占分量时按题目要求返回下标最小者。
- **不邻接植花 `garden_no_adj`**：建图之后不需要 BFS/DFS 遍历，贪心染色即可——
  题目保证度数不超过 3，而颜色有 4 种，贪心一定能找到合法颜色，不需要回溯。
- **最小体力消耗路径 `min_effort_path`**：目标函数从"路径权重和最小"变成
  "路径最大边权最小"，标准 Dijkstra 的松弛公式要从 `cur + weight` 换成
  `max(cur, weight)`，其余框架（小根堆 + 每个格子维护当前最优值）不变。
- **迷宫中离入口最近的出口 `nearest_exit`**：标准 BFS 求最短路，考点全在"出口"
  的定义——必须是边界格子且不能是入口本身，只检查"新扩展到的邻居"是否满足这
  两条，天然就不会把起点自己误判成出口。
- **网格中的最短路径-有障碍物消除 `shortest_path_with_obstacle_elimination`**：
  状态扩展成 `(r, c, remain)`，`remain` 是剩余消除次数；当 `k` 足够大
  （`>= rows+cols-2`）时直接返回曼哈顿距离，不需要真的跑 BFS。
- **从盒子里获得的最大糖果数 `max_candies`**：多源 BFS 的变体，状态不是坐标而是
  "拥有盒子集合"和"拥有钥匙集合"的双向解锁关系——新获得盒子或者新获得钥匙都要
  重新尝试打开，因为两者满足的先后顺序不确定，任何一边先到位都可能让之前打不开
  的盒子突然可以打开了。
