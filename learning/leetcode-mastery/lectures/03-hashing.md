# L03 · 哈希表

对应代码：`src/problems/p03_hashing.py`。

## 通用心法：看到"快速查找/去重/计数"就该想哈希表

哈希表能解决的问题本质上只有一件事：**把"这个东西之前有没有出现过 / 出现了几次 / 对应着什么"这类查询，从线性时间降到常数时间**。它的代价是要多花 O(n) 的空间去建一张表，换来的是 O(1) 的查找——这是一笔几乎总是划算的交易，因为大多数场景里内存比时间充裕得多。

具体到题目里，看到下面几种诉求，就该条件反射地想到哈希表：

- **"是否出现过"**：比如判断数组里有没有重复元素（LC217）、判断某个字符是否在窗口里出现过（`02-sliding-window.md` 里的 LC3）。对应哈希 **set**。
- **"出现了几次 / 两个集合的次数是否完全一致"**：比如判断字母异位词（LC242 只看两个字符串的字符计数是否相等）、Top-K 高频元素。对应哈希 **计数表**（`Counter` 或 `dict[key, int]`）。
- **"这个值对应着什么，能不能 O(1) 找到"**：比如两数之和里"target 减当前值是否出现过、出现在哪"（LC1）、字母异位词分组里"这个字符串该归到哪一组"（LC49，key 是排序后的字符串）。对应哈希 **dict（值→附加信息）**。
- **"能不能把一个天然要重复计算的暴力解，替换成边扫描边查表"**：这是哈希表用法的本质抽象——**任何暴力解里"对每个元素都要重新扫一遍别的元素来确认某个条件"的地方，都值得问一句"这个条件能不能预先存进一张表里，查表代替扫描"**。

哈希表本身没有"顺序"和"范围查询"的能力（这是它和有序结构、二分查找的分工边界），所以当题目需要"下一个更大的""某个范围内的最值"时，哈希表通常不是主角，需要换成单调栈、前缀和等结构（这些留到后面模块）。

下面挑 LC128 最长连续序列做深挖——这是本类目里最容易被想复杂、也最能体现"用 set 换判断条件"这个技巧的一题。

## 深挖：LC128 最长连续序列 —— 为什么"只从序列起点出发"能把 O(n log n) 砍到 O(n)

### 第一步：暴力解怎么想

题目要找数组里"数值连续"的最长序列的长度（比如 `[100,4,200,1,3,2]` 里 `1,2,3,4` 是长度 4 的连续序列）。最直接的想法：

1. **排序**：把数组排序后，连续的数值就会排在相邻位置，只需要线性扫一遍，遇到 `nums[i] == nums[i-1] + 1` 就把当前序列长度 +1，否则重新计数，记录过程中的最大值。

这个做法完全正确，但复杂度是 O(n log n)（卡在排序这一步），而题目明确要求 O(n)。

### 第二步：为什么排序这一步是"可以被替换掉的"

回想一下，排序在这里唯一的作用是让"连续的数值互相挨在一起"，这样才能"一次线性扫描"就找到所有连续段。但排序其实是"杀鸡用牛刀"——它顺便把**所有**数值都按大小整理了一遍，而我们真正需要的信息只是"某个数的下一个数（x+1）在不在数组里"，这是一个纯粹的**查找**问题，不需要真的把所有数排好序。

**关键转念**：把数组丢进一个哈希 set 里，"`x+1` 在不在数组里"就变成了 O(1) 的查找（`x + 1 in num_set`），完全不需要排序。

### 第三步：只把"查找"换成 O(1) 还不够——为什么必须只从"序列起点"出发

如果只是把排序换成 set，但对**每一个**数字都执行"从它开始不断查 `x+1, x+2, ...` 直到不连续为止"，粗看好像是 O(1) 查找 * n 次，但仔细想：如果数组是 `[1,2,3,4,5]`，从 `1` 出发要走 5 步，从 `2` 出发又要走 4 步，从 `3` 出发走 3 步……总的步数是 `5+4+3+2+1 = O(n^2)`——**每个数字被重复访问了很多次**，把好不容易换来的 O(1) 查找又在总量上抵消掉了，复杂度不降反升。

这就是本题真正的 insight，也是最容易被漏掉的一步：**一个长度为 L 的连续序列，只需要被"数一次"就够了，应该只从这个序列的起点出发数，而不是每个成员都各自数一遍**。而"某个数 `x` 是不是一个序列的起点"，判断条件非常简单：**`x - 1` 不在集合里**（如果 `x-1` 在集合里，那么 `x` 就不是起点，它属于以 `x-1` 或更早的数开头的那个序列，会被那个更早的起点处理到，轮不到 `x` 重复处理）。

加上这一句 `if x - 1 not in num_set` 的门槛判断之后，回顾复杂度：**每个数字最多在两种情况下被访问**——一次是外层 `for x in num_set` 遍历到它本身，另一次（如果它是某个序列的成员而非起点）是被前面某个起点的内层 `while` 循环访问到、但那种情况下它不会再触发自己的内层循环。所以内层 `while` 循环的**总执行次数**，加起来不会超过 `n`（每个数字最多被内层循环"数过"一次）。整体摊销下来是 O(n)。

**这就是这题的核心教训**：拿到"看似需要排序"的问题时，先问自己"排序真正提供的是什么性质——是顺序本身重要，还是只是需要相邻查找的能力？" 如果只是后者，很可能可以用哈希 set 替换掉排序这一步；但换完之后还要再检查一遍"是否会导致重复计算"，只优化查找、不优化"访问了几次"，复杂度可能不降反升——这正是本题比表面看起来更容易"想复杂"（或者优化了查找却漏了访问次数这一关）的地方。

### 第四步：一步步编码

1. 签名先行：`longest_consecutive(nums: list[int]) -> int`。
2. `num_set = set(nums)`（去重 + 换取 O(1) 查找），`best = 0`。
3. `for x in num_set`：先判断 `if x - 1 not in num_set`（起点门槛），只有通过门槛才继续。
4. 通过门槛后，`y = x`，`while y + 1 in num_set: y += 1`，同时累计长度，最后更新 `best`。
5. 返回 `best`。

对照 `p03_hashing.py` 里的 `longest_consecutive` 实现核对每一步，尤其是第 3 步的门槛判断——这一行删掉的话，代码依然"能跑出正确答案"，只是复杂度会退化，用一个明显更大的数组测试时才会暴露出来（这也是为什么这类"正确但复杂度不对"的 bug 特别容易被漏测）。

## 其余 4 题点睛

**LC217 存在重复元素**：哈希表最基础的用法——一边遍历一边往 set 里塞，塞之前先查一遍。它和 LC128 的联系在于：两题都用到了"用 set 做 O(1) 查找"这个最底层的能力，但 LC128 在此之上还叠加了"只从起点出发"这个避免重复计算的技巧，LC217 可以当作理解这个底层能力本身的最小例子。

**LC242 有效的字母异位词**：哈希表的"计数"用法（而不是"是否存在"）——两个字符串互为异位词，等价于两张字符计数表完全相等。这是 LC49 字母异位词分组的基础：先想清楚"怎么判断两个字符串算不算异位词"（这题），再想"怎么把这个判断关系变成一个能当哈希 key 的规范形式"（LC49）。

**LC49 字母异位词分组**：在"计数比较"的基础上更进一步——不是两两比较，而是给每个字符串算出一个**规范形式**（这里用排序后的字符串）当作哈希表的 key，"是否异位词"就变成了"key 是否相同"，从而把分组问题变成一次遍历 + 分桶。这一步"设计一个哈希 key 使等价的东西映射到同一个 key"的思路，在很多问题里都会复现，值得反复体会。

**LC380 O(1) 时间插入删除获取随机元素**：这题是哈希表和数组"取长补短"的组合题——哈希表擅长 O(1) 查找但不支持随机访问下标，数组擅长随机访问下标但删除中间元素是 O(n)。用 `dict`（值→下标）配合 `list`（存值本身），删除时把待删元素和末尾元素交换再 `pop`，把"数组删除中间元素"的 O(n) 降到 O(1)。这题提醒我们：哈希表不是万能的，它解决"查找"问题，但"顺序访问"和"随机访问"要靠数组补上，实际系统设计里经常是多种结构组合使用。

### 深挖：LC149 直线上最多的点数 —— 为什么斜率必须用"最简分数"而不是浮点数

这题要找平面上共线点数最多的一条直线。思路本身不难想：枚举每个点当"参考点"，对其余点计算相对这个参考点的斜率，斜率相同的点必然和参考点共线，用哈希表把"斜率 → 点数"分组统计，取最大值。**真正的难点不在算法框架，而在于"斜率"这个东西该怎么当哈希表的 key**。

第一反应是直接算 `dy / dx` 这个浮点数当 key。这在数学上没问题，但在计算机里会踩到一个经典陷阱：**浮点数运算存在精度误差**，两个理论上完全相等的斜率（比如 `1/3` 和 `2/6`），因为除法的浮点表示方式不同，算出来的结果可能是 `0.3333333333333333` 和 `0.33333333333333337`（多一位精度尾差）——这两个浮点数在哈希表里会被当成**两个不同的 key**，导致本该分到同一组的共线点被错误地拆到了两个桶里，答案偏小。这个 bug 尤其危险，因为它不会报错、不会崩溃，只会在某些特定的输入（分数约不尽、除法产生长尾小数）上悄悄给出错误答案，很难在小规模手工测试里发现。

解决办法是**完全避开浮点数**，用一对互质的整数（最简分数）表示斜率：把 `dx = points[j][0] - points[i][0]`、`dy = points[j][1] - points[i][1]` 都除以它们的最大公约数 `g = gcd(dx, dy)`，得到的 `(dx/g, dy/g)` 就是这条方向向量的最简整数形式——两条斜率相同的直线，无论具体的 dx、dy 数值差多少倍，约分后一定得到完全相同的整数对，可以放心当哈希表的 key 做精确比较，不存在任何精度误差。

这里还有一个容易漏掉的细节：**符号归一化**。方向 `(1, -2)` 和 `(-1, 2)` 描述的其实是同一条直线（只是"从 A 指向 B"还是"从 B 指向 A"的方向相反），如果不统一符号，会被误判成两个不同的 key。所以约分之后要强制"钦定" dx 为非负（如果 dx 是负的，就把 dx、dy 都取反）；当 dx 恰好是 0（垂直线）时，再强制钦定 dy 为正。这样任意两点之间的方向向量，不管算的时候用哪个点减哪个点，归一化后都会落到同一个 key 上。

**这题真正的教训**：涉及"比较两个由除法/浮点计算得到的值是否相等"时，第一反应应该是"能不能转化成整数比较"（约分成最简分数、或者交叉相乘比较 `dy1 * dx2 == dy2 * dx1`），而不是想当然地直接比较浮点数——浮点误差是那种"平时测试都过、上线偶发出错"的隐蔽 bug 类型，值得在写代码的当下就杜绝，而不是等出了 bug 再排查。

### 其余 6 题点睛

**LC205 同构字符串**：哈希表"双射校验"的最基础形式——同时维护两个方向的映射表（`s→t` 和 `t→s`），任何一个方向出现"同一个 key 映射到不同 value"就说明不是双射。只查一个方向是这题最典型的漏洞。

**LC290 单词规律**：和 LC205 是完全同一个双射校验模型，只是把"字符↔字符"换成了"字符↔单词"，代码结构几乎可以直接照抄，唯一多出的一步是先把字符串按空格切分成单词列表。

**LC350 两个数组的交集 II**：哈希表"计数"用法在"求交集"场景下的应用——如果用 `set` 求交集会丢失重复次数信息，必须用 `Counter` 统计后"命中就收集并减一"，这个"减一"操作天然保证了结果里每个值最多出现"两边计数取较小值"那么多次。

**LC202 快乐数**：本质是"检测序列是否进入循环"，用哈希 set 记录见过的中间结果，一旦重复出现就能判定陷入循环——这是哈希表"记录见过的值"这个最基础能力（和 LC217 存在重复元素同源）在"判环"场景下的应用；如果要求 O(1) 空间，也可以像 LC287 寻找重复数那样改用快慢指针。

**LC387 字符串中的第一个唯一字符**：哈希表"计数"用法的最直接应用——先统计整个字符串的字符频次，再按原顺序遍历一次找到第一个计数为 1 的字符。和 LC242 有效的字母异位词一样，都是"先建计数表，再利用计数表回答问题"的两阶段套路。

**LC454 四数相加 II**：把"四数之和为 0"通过"哈希表预处理一半、查表另一半"的技巧，从暴力的 O(n^4) 降到 O(n^2)——先用哈希表统计前两个数组两两组合的和出现的次数，再遍历后两个数组的两两组合，查"负的这个和"出现了多少次并累加。和 LC1 两数之和是同一个"用哈希表把查找一半问题变成 O(1)"的思路，只是这里预处理的是"和的计数"而不是"值的下标"。
