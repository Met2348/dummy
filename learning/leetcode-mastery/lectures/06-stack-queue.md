# L06 · 栈与队列 —— 栈是处理"嵌套/最近匹配"结构的天然工具

> 对应代码：`src/problems/p06_stack_queue.py`。

## 为什么栈总是和"括号、表达式、单调性"这几类题绑在一起

栈（后进先出，LIFO）看起来只是"限制访问方式的数组"，但它精确对应了现实里一类
非常常见的结构：**"最近发生、还没结束的事情，必须最先被处理/关闭"**。具体到不同题目：

- **括号匹配**：最近打开的括号，必须最先被闭合（不能先关外层再关内层）；
- **表达式求值**：运算符的优先级导致"最近出现、还没算完的子表达式"要先算完，才能
  继续算外层（无论是括号嵌套，还是乘除法相对加减法的优先级）；
- **单调栈**（本课未展开，但值得知道）：维护"目前为止还有可能成为答案的候选"，
  新元素一来，栈顶那些已经不可能再是答案的候选就被弹出——这也是一种"最近的、
  还有效的候选，比更早的候选优先被处理"。

这三类问题的共性是：**当前正在处理的东西，依赖于"最近一个尚未处理完的状态"**，而不是
"最早一个"或者"任意一个"。数组/列表如果要模拟这种"只看最近未处理状态"的访问方式，
天然就该用栈；反过来，一旦你在题目描述里读出"最近的……优先……"这种语感，应该第一
反应就是"这题大概率用栈"。

## 深挖 1：有效的括号（LC20，Easy）—— 为什么栈能天然表达"最近打开的括号最先关闭"

**先想清楚合法括号串的递归结构**：一个合法的括号串，要么是空串，要么是
`(合法括号串)` 整体外面套一层匹配的括号，要么是"两个合法括号串顺次拼接"。不管
哪种情况，任何时刻你从左往右扫描，"当前还没被关闭的左括号"总是形成一个**后进先出**
的序列：最后一个被打开、还没关闭的左括号，必须是下一个被关闭的（因为如果它不是
下一个被关闭的，中间就会插入另一个跨越它边界的括号，破坏了"合法嵌套"这个前提）。

这就是为什么栈天生适合这道题：**栈顶元素，就是"当前最近一个还没被关闭的左括号"**。
扫描过程中：

- 遇到左括号：入栈，表示"多欠了一个待关闭的右括号"；
- 遇到右括号：检查栈顶——如果栈顶正好是它对应的左括号类型，说明这次匹配合法，
  栈顶出栈（这笔"债"还上了）；否则（类型不对，或者栈已经空了）直接判定整个字符串
  无效。

```python
pairs = {")": "(", "]": "[", "}": "{"}
stack: list[str] = []
for ch in s:
    if ch in pairs:
        if not stack or stack.pop() != pairs[ch]:
            return False
    else:
        stack.append(ch)
return not stack
```

**两个容易漏掉的边界，都源于"只顾着写主逻辑，忘了栈本身也可能不满足假设"**：

1. 遇到右括号时，如果此时栈是空的（`not stack`），说明这个右括号根本没有对应的
   左括号——如果不做这个检查，直接对空栈调用 `.pop()`，Python 会抛出
   `IndexError: pop from empty list`。这对应输入形如 `")("`——第一个字符就是
   右括号，栈还是空的。
2. 整个字符串扫描完之后，**必须再检查一次栈是否为空**。像 `"(("` 或 `"([" `
   这种输入，扫描过程中不会触发任何一次 `return False`（因为一直在正常入栈，
   没有遇到"类型不匹配"或者"空栈时来了右括号"的情况），但最终栈里还留着没被
   关闭的左括号——这说明字符串本身不完整。忘记这最后一次检查，是这道题最容易
   漏掉的一类 bug：只处理了"多余的右括号"，没处理"多余的左括号"。

## 深挖 2：基本计算器 II（LC227，Medium）—— 为什么乘除法优先级只需要操作栈顶

**先想清楚为什么不需要真的建表达式树**：四则运算里，乘除法优先级更高，直觉上你
可能觉得需要先把整个表达式解析成一棵树（乘除法节点在下层，加减法节点在上层），
再按后序遍历求值。但仔细观察乘除法的性质：**乘除法只会和"紧挨着它前面的那一个数"
发生运算，不会影响再往前、已经被加减法"确定下来"的部分**。比如 `3 + 2 * 2`，
乘法只跟它前面的 `2` 结合，跟最前面的 `3` 完全无关——`3` 早就已经"入账"了。

这意味着：只要能在扫描到运算符的那一刻，**用一个栈存"目前为止每一个已经确定符号、
不会再被后续运算修改的数字"**，乘除法就只需要修改栈顶这一个元素，不需要回溯更早
的历史。具体做法：

- 维护 `prev_sign`（上一个遇到的运算符，初始视为 `'+'`）和正在累积的 `num`；
- 每当遇到新的运算符（或扫到字符串末尾），就"结算" `num`：
  - `prev_sign == '+'`：把 `num` 原样入栈；
  - `prev_sign == '-'`：把 `-num` 入栈（负号在这一步就处理掉了，栈里存的都是
    "待相加"的数，最后直接 `sum(stack)` 就是答案）；
  - `prev_sign == '*'` 或 `'/'`：弹出栈顶（"紧邻的前一个数"），跟 `num` 做
    乘/除，把结果**重新压回栈**——这就是"只操作栈顶就地修正优先级"，栈里更早
    的元素完全不受影响，因为乘除法压根不需要碰它们。

```python
stack: list[int] = []
num = 0
prev_sign = "+"
for i, ch in enumerate(s):
    if ch.isdigit():
        num = num * 10 + int(ch)
    if ch in "+-*/" or i == len(s) - 1:
        if prev_sign == "+":
            stack.append(num)
        elif prev_sign == "-":
            stack.append(-num)
        elif prev_sign == "*":
            stack.append(stack.pop() * num)
        else:
            stack.append(int(stack.pop() / num))
        prev_sign = ch
        num = 0
return sum(stack)
```

**为什么这样做是对的，本质上是把"运算符优先级"转化成了"栈顶操作 vs 追加操作"的
区别**：加减法的"证据"（这个数带什么符号）已经足够，直接追加到栈末尾；乘除法的
"证据"还不完整——它需要跟前一个数结合之后才能算出"最终对总和的贡献"，所以要把
前一个数从栈里取出来、结合、再放回去。全部扫描完之后，栈里剩下的所有数已经是
"确定不会再被修改"的了，直接求和。

**两个容易漏的点**：

1. Python 的整数除法 `//` 对负数是**向下取整**（`-7 // 2 == -4`），而这道题
   要求的是**向 0 截断**（`-7 / 2` 截断后应该是 `-3`），必须写成
   `int(a / b)`，不能直接用 `//`；
2. "结算"这个动作必须在"遇到新运算符"和"到达字符串末尾"两种情况下都触发——如果
   只在遇到运算符时触发，字符串最后一段数字永远不会被结算进栈，代码里用
   `i == len(s) - 1` 把"到达末尾"也当成一次触发信号来处理。

## 其余 3 题点睛

- **LC232 用栈实现队列**：两个栈分工——`in_stack` 只管接收新元素，`out_stack`
  只在自己为空时才从 `in_stack` 整体倒一次（倒两次栈，先进先出的顺序就恢复了）。
  每个元素一生只会被"倒腾"一次，均摊下来每次操作 O(1)，但要注意"只在
  `out_stack` 为空时才倒"——每次都倒会打乱已经排好序的元素。
- **LC155 最小栈**：`get_min` 要 O(1)，靠的是额外维护一个和主栈严格同步增减的
  "最小值栈"，每次 push 时把 `min(val, 当前最小值)` 也压进去，栈顶就是"到目前为止
  的历史最小值"——不能只用一个全局变量记录最小值，因为它一旦被 pop 掉就找不回
  "次小值"了。
- **LC224 基本计算器**：括号嵌套是"挂起当前状态、先算内层、再恢复外层"的模式，
  用栈存 `(外层已有的 result, 外层的 sign)`，遇到 `(` 就压栈冻结、重置状态去算
  内层；遇到 `)` 就弹栈，把内层结果按外层符号"揉回"外层的累加和——本质上和
  深挖 2 的"栈顶操作代替表达式树"是同一种思维方式，只是这里挂起/恢复的是"整个
  子表达式"而不是"单个数字"。

### 深挖：字符串解码（LC394，Medium）—— 为什么要用两个栈分别存数字和字符串

**先看这题和"有效的括号""基本计算器"有什么不一样**：括号匹配只需要判断"合不合法"，
基本计算器的括号里装的是"一个数值"；而这题的括号里装的是**一整个字符串**，并且
外面还套着一个"重复次数"。也就是说，每当遇到 `[`，都有两样东西需要临时"挂起"：
**当前已经拼到一半的字符串**（外层还没来得及和内层结果拼接）和**刚刚读到的这个
重复次数**（数字读完就没别的地方存了，如果不存下来，扫描完 `[` 内部的内容之后就
不知道要重复几次）。这正是本课开头强调的通用范式的又一次应用：**遇到"进入更深一层
嵌套"的信号时，把"当前层还没处理完的所有状态"一次性打包压栈冻结，转去处理更深的
一层；遇到"退出这一层"的信号时，弹栈恢复冻结的状态，并把刚算完的内层结果按照
冻结时的语境揉回去**。

具体到代码，"当前层还没处理完的所有状态"就是 `(cur_num, cur_str)` 这一对——
`cur_num` 是这一层刚读完、还没被消费的重复次数，`cur_str` 是这一层到目前为止已经
拼好、但还不完整的字符串。用两个独立的栈（`num_stack` 和 `str_stack`）分别存，
本质上和"存一个元组构成的栈"是等价的，只是拆成两个数组更符合 Python 里"数字栈"
和"字符串栈"分开操作更直观的写法：

```python
num_stack: list[int] = []
str_stack: list[str] = []
cur_num = 0
cur_str = ""
for ch in s:
    if ch.isdigit():
        cur_num = cur_num * 10 + int(ch)
    elif ch == "[":
        num_stack.append(cur_num)
        str_stack.append(cur_str)
        cur_num = 0
        cur_str = ""
    elif ch == "]":
        prev_num = num_stack.pop()
        prev_str = str_stack.pop()
        cur_str = prev_str + cur_str * prev_num
    else:
        cur_str += ch
return cur_str
```

**遇到 `]` 时的合并顺序 `prev_str + cur_str * prev_num` 是整道题最容易搞混的
一步**：`cur_str` 是内层刚刚拼好的完整内容，`prev_num` 是这一层对应的重复次数，
`cur_str * prev_num` 就是"内层展开后的结果"；而 `prev_str` 是外层在遇到 `[`
之前已经拼好、被冻结起来的前缀，必须拼在"内层展开结果"的**前面**。如果颠倒顺序
写成 `cur_str * prev_num + prev_str`，字母的先后次序就会整体错位。

**为什么这是处理嵌套结构的通用范式，而不只是这一题的技巧**：无论是这里的
"数字+字符串"，还是基本计算器里的"(result, sign)"，还是编译原理里"函数调用时
保存调用者的局部变量和返回地址"，本质都是同一件事——**当前状态在"进入更深一层"
之前，必须完整地保存下来，因为深入一层之后，外层状态在语法上暂时不可见，但退出
这一层时又必须原样恢复并结合内层的结果继续使用**。栈的"后进先出"完美契合这种
"最近挂起的状态，必须最先被恢复"的时序关系。

**易错点**：1) 数字可能不止一位（比如 `"12[a]"`），必须持续累积到 `cur_num`，
不能只读一位就当成重复次数；2) 遇到 `[` 或 `]` 之后，`cur_num` 和 `cur_str`
都必须重置/替换成新值，如果忘记重置 `cur_num`，下一层的重复次数会被污染成
"外层数字 * 10 + 新数字"这种错误的累积结果；3) 这题的字符串可以嵌套任意深，
所以必须用栈而不是只用两个变量手动保存"上一层"的状态——嵌套深度超过一层时，
两个普通变量根本不够用。

### 其余 7 题点睛

- **LC150 逆波兰表达式求值**：后缀表达式的运算符右边操作数在栈里"后入栈、先出栈"，
  栈的顺序天然对应求值顺序，遇到运算符弹出两个数运算再压回去即可；注意弹出顺序是
  先弹右操作数、后弹左操作数（`a op b` 里 `b` 先出栈），减法/除法写反顺序结果就
  错了，Python 整数除法还要用 `int(a / b)` 做向零截断。
- **LC946 验证栈序列**：不需要枚举，直接用一个辅助栈严格模拟"按 pushed 顺序入栈，
  每入栈一个就贪心地尽量多弹（只要栈顶等于 popped 的下一个值）"，最后栈清空说明
  合法——贪心弹出不会更差，因为现在不弹以后也不会有更好的弹出时机。
- **LC1249 移除无效的括号**：一次扫描分类"待删除"下标——扫到 `)` 时栈空说明它
  没有左括号可配、记为待删除；扫描结束后栈里剩下的下标是没等到右括号的左括号，
  同样要删除。两类"多余括号"都要处理，只处理一类是这题最容易漏的地方。
- **LC636 函数的独占时间**：调用栈的直接应用，栈顶永远是当前正在执行的函数；
  `"end"` 日志的时间戳是闭区间的，结算时要 `+1`，且结算完之后 `prev_time` 也要
  设成 `time + 1`，否则外层函数恢复执行后会重复计入这一个单位时间。
- **LC1472 设计浏览器历史记录**：不需要真的用两个栈模拟"后退栈/前进栈"，一个列表
  存整条历史 + 一个当前位置指针就够了——`visit` 截断指针右边的旧历史再追加新页面，
  `back`/`forward` 只需要在 `[0, len-1]` 范围内移动指针。
- **LC1381 设计支持增量操作的栈**：如果每次 `increment` 都真的遍历 k 个元素会
  退化成 O(size)；用一个和主栈等长的"待结算增量"差分数组，`increment` 只需要
  修改一个下标（`min(k, size) - 1` 处），`pop` 时把这份增量结算给栈顶元素、
  再下推给下面的元素，均摊 O(1)——是"差分数组"思想在栈上的应用。
- **LC682 棒球比赛**：`"+"`/`"D"`/`"C"` 都只依赖"最近的一到两条有效记录"，栈天然
  维护这种"只关心最近状态"的结构；`"C"` 是纯粹的弹栈撤销，`"+"` 是把栈顶两个数相加
  后**追加**一条新记录（不是替换掉原来的两条）。
