# L21 · 设计题 —— 给一组接口复杂度要求，考验能否选对底层数据结构组合

> 对应代码：`src/problems/p21_design.py`。

## 设计题考的是什么

设计题（Design）不像前面 20 个分类那样各自围绕一种算法技巧，它考的是**工程能力和
算法能力的交叉点**：题目会给出一组接口（比如 `get`、`put`、`insert`、`search`），
以及每个接口必须满足的复杂度要求（通常是 O(1) 或 O(log n)），你需要判断"选哪几种
数据结构组合在一起，才能让每一个接口都达标"。单独用哈希表、单独用数组、单独用堆，
往往只能满足接口列表里的一部分要求——真正的难点是想清楚"每种结构各自负责哪一部分
职责，组合起来怎么保持它们之间的数据一致性"。

## 深挖：LRU 缓存（LC146，Medium）—— 为什么"哈希表 + 双向链表"是唯一组合

**先想清楚为什么单独一种结构都不够**：LRU 缓存要求 `get`、`put` 都是 O(1)，并且
每次访问都要能立刻知道"接下来淘汰谁"（最久未使用的那个）。

- 如果只用**哈希表**：`get(key)` 能 O(1) 找到值，但哈希表本身不记录"访问的先后
  顺序"——你没法知道当前所有 key 里谁是"最久未使用"的那个，除非额外遍历一遍
  （O(n)），不满足要求。
- 如果只用**双向链表**（按访问顺序排列，最近访问的挪到一端）：链表天然维护了
  "最近使用"的顺序，链表头/尾的 O(1) 增删也能满足"挪动"和"淘汰"的需求，但问题是
  ——**给定一个 key，怎么 O(1) 找到它在链表里对应的节点？** 链表本身不支持随机
  访问，找一个节点需要遍历，又不满足要求。

**这两种结构的短板恰好互补**：哈希表擅长"给定 key，O(1) 定位"，但不记录顺序；
双向链表擅长"O(1) 调整顺序、O(1) 删除任意已知节点"，但不支持随机定位。把两者
组合起来——**哈希表存 `key -> 链表节点的引用`，双向链表维护访问顺序**——就同时
拿到了两边的优势：`get(key)` 先用哈希表 O(1) 找到节点，再把这个节点在链表里挪到
"最近使用"的一端（O(1)，因为双向链表删除/插入一个已知节点不需要遍历）；`put`
超出容量时，直接删除链表另一端（最久未使用）的节点，同时从哈希表里也删掉对应
的 key。

Python 的 `collections.OrderedDict` 内部就是用哈希表 + 双向链表实现的，直接拿来用
不需要手写链表节点：

```python
class LRUCache:
    def __init__(self, capacity: int) -> None:
        self.capacity = capacity
        self.cache: OrderedDict[int, int] = OrderedDict()

    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1
        self.cache.move_to_end(key)   # O(1) 挪到"最近使用"的一端
        return self.cache[key]

    def put(self, key: int, value: int) -> None:
        if key in self.cache:
            self.cache.move_to_end(key)
        self.cache[key] = value
        if len(self.cache) > self.capacity:
            self.cache.popitem(last=False)   # O(1) 淘汰"最久未使用"的一端
```

`move_to_end` 对应"挪到最近使用的一端"，`popitem(last=False)` 对应"淘汰最久未
使用的一端"——这两个操作正是双向链表相对哈希表单独使用时补上的那块短板。

**两个容易漏掉的点**：

1. `get` 命中之后必须 `move_to_end`，否则这个 key 明明刚被访问过，下一次淘汰时
   却可能被误判成"最久未使用"；
2. 淘汰要用 `popitem(last=False)`（弹出最早插入/最久未移动的一端），默认参数
   `popitem()` 弹出的是**最近**插入的一端，正好和需求相反。

## 其余 5 题点睛

- **LC460 LFU 缓存**：和 LRU 的核心区别是多了一维"频率"，所以要在"哈希表 + 双向
  链表"的基础上再加一层——`频率 -> 该频率下的 LRU 队列（OrderedDict）`。淘汰时永远
  从 `min_freq` 桶里弹出最久未使用的那个（`min_freq` 需要额外维护：只有当某个 key
  从当前的 `min_freq` 桶挪走、且挪走后该桶变空时，`min_freq` 才 +1）。可以理解成
  "LRU 是 LFU 在所有 key 频率都相同时的特例"。
- **LC355 设计推特**：每条推文打一个全局递增的时间戳，`get_news_feed` 要从"自己 +
  关注的人"这些候选推文里找时间戳最大的前 10 条——这是标准的"多个列表里找整体
  Top-K"场景，`heapq.nlargest` 内部就是维护一个大小为 10 的堆（呼应 15 章"堆/
  Top-K"的技巧），比把候选推文全部拼起来整体排序更省。
- **LC1146 快照数组**：给每个下标单独维护一条"`(快照编号, 值)` 的修改历史"，只有
  真正 `set` 时才追加一条记录（同一快照内多次 set 同一下标要原地覆盖，不能重复
  追加）。查询时用 `bisect_right` 在这条历史里二分查找"最后一个快照编号 <= 目标"
  的位置，不需要真的给整个数组拍快照。
- **LC622 设计循环队列**：固定大小的数组 + `head` 指针 + `count` 计数器（队尾下标
  用 `(head + count) % capacity` 算出来，不用单独维护，减少一个容易和 head 失去
  同步的状态量）。判空/判满必须靠 `count` 和 `0`/`capacity` 比较，不能用
  `head == tail` 这种朴素写法——队列全空和全满时两个指针经常重合，没法靠位置
  区分。
- **LC705 设计哈希集合**：不能直接包一层 Python 的 `set()`，要真正模拟哈希表的
  实现原理——固定数量的桶 + 链地址法处理冲突（每个桶是一个普通列表，`key % 桶数`
  决定进哪个桶，桶内线性查找/插入/删除）。这道题的教学目的是理解哈希表本身怎么
  实现，用现成的 set 走捷径虽然能通过测试，但没有练到考点。
