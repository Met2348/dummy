# L33 · 高级数据结构（Phase 3 竞赛级新分类）

> 对应代码：`src/problems/p33_advanced_data_structures.py`。

## 这一类到底在讲什么

这一章和前面大多数分类不太一样：前面每一类基本围绕"一种算法技巧"（前缀和、
单调栈、并查集……），这一章是**更专精的数据结构组件**——它们未必会在面试题里
原封不动地出现（除了 4 道真实 LeetCode 设计题），但经常是解决其他难题时的
"底层零件"：离线算法回退状态要用可撤销并查集，静态区间最值查询要用稀疏表，
"判断一个东西大概率有没有出现过"要用布隆过滤器。学这一章的目的不是"记住某道题
的解法"，而是理解**每个结构"为什么是这个形状"——它用什么代价换取了什么操作
的加速**，以及"什么时候该选哪一个"。

## 深挖：跳表（Skip List）—— 为什么"多层链表 + 大约一半元素跳跃"能做到期望 O(log n)

**跳表要解决的问题**：普通有序链表的查找是 O(n)——即使数据整体有序，也只能
从头开始一个个比较，因为链表不支持"随机访问中间元素"（这是数组能做二分查找、
链表不能的根本原因）。跳表的目标是：**不改变"链表"这种支持 O(1) 插入/删除的
结构，同时让查找也能接近二分查找的 O(log n)**。

**核心想法——加"高速公路"**：如果在原来的链表（第 0 层，包含所有元素）之上，
再建一条"稀疏"的链表（第 1 层），只包含大约一半的元素，查找时可以先在第 1 层
"跑得快"（一次跳过两个元素），跑不动了（下一个元素比目标值大）再降到第 0 层做
精细查找。如果再在第 1 层之上建第 2 层（大约四分之一的元素）、第 3 层（大约
八分之一）……层层叠加，就像"高速公路 → 城市主干道 → 小区内部道路"这样一级一级
从粗到细定位，最终整个查找路径长度期望是 O(log n)——每往上一层，候选元素数量
大约减半，层数自然是 O(log n)，每一层最多横向走几步（因为下一层已经把范围
缩小了）。

**为什么"大约一半"就够了，需要精确一半吗**：不需要。跳表的精妙之处在于它不要求
"精确的一半"，用**每个元素独立地、以固定概率 1/2 抛硬币决定要不要出现在上一层**
（抛到正面就继续留在上一层再抛一次，抛到反面就停），最终每一层的元素数量在
*期望*上是上一层的一半，层数期望是 O(log n)——这是一个概率数据结构：**不保证
每一次操作都是 O(log n)，但从概率分布上看，期望是 O(log n)**，这也是为什么它
被称为"用空间换时间的概率数据结构"的代表：多存了 O(n) 的额外指针（每层的
"跳跃"指针），换来了期望 O(log n) 的查找/插入/删除，而且比红黑树、AVL 树这类
严格平衡树的实现简单得多（不需要旋转、不需要维护平衡因子）。

**本实现的关键改动——用确定性规则替代抛硬币**：标准跳表用真随机数决定每个新
节点的层数，这样"层数分布期望是几何分布"这件事完全依赖随机性，测试时很难
复现（同样的插入序列，每次跑出来的内部结构可能不同）。`SimpleSkipList` 按
任务要求换成完全确定的规则：**第 k 次插入（从 1 开始计数）的层数 = k 的二进制
表示里"末尾连续 0 的个数" + 1**。这个规则并不是拍脑袋想的，它精确复刻了"抛硬币"
的概率分布在整数序列上的确定性版本——在 1 到 k 这些数里：

- 恰好一半是偶数（二进制末尾至少 1 个 0）→ 恰好一半的插入会升到第 2 层；
- 恰好四分之一是 4 的倍数（末尾至少 2 个 0）→ 四分之一升到第 3 层；
- 以此类推，"末尾 0 的个数分布"和"抛硬币抛到反面为止的次数分布"在数量级上
  完全对应。

也就是说，这个规则用"数一个数的二进制末尾 0 的个数"这个完全确定的算术操作，
**模拟**出了标准跳表依赖随机数才能达到的概率分布效果，换来的是"同一个插入
序列，任何时候重新跑，内部层数分配完全一致"的可复现性——这对写单元测试、
调试都极其友好，代价是：如果输入的操作序列恰好是针对这个确定性规则构造的
"最坏情况"（比如故意让很多元素都分到层数 1），跳表会退化，这是用确定性换
可复现性必须承认的权衡（在文档里也明确写了这一点，而不是假装它和真随机版本
完全等价）。

**`insert`/`search`/`erase` 共享的核心定位逻辑**：从跳表当前最高层开始，在
这一层尽量往右走（只要下一个节点的值还小于目标值就继续走），走不动了（下一个
节点值 >= 目标值，或者已经是链表尾）就降一层，重复这个过程直到第 0 层——这时
候要么找到了目标值（`search`/`erase` 命中），要么定位到了"目标值应该插入的
位置"（`insert`）。三个操作用的是同一套"从高到低、逐层逼近"的移动逻辑，区别
只在于"找到之后做什么"。

## 其余点睛

> LC895"最大频率栈"和 LC432"全 O(1) 的数据结构"这两道真实设计题，已经在 21 类
> `p21_design_iii.py` 里实现（分别是 `FreqStack`/`AllOne`），本章不重复实现，
> 只在下面提一句它们和本章结构的联系，详细讲解见 [`21-design.md`](21-design.md)。

**LC981 基于时间的键值存储**：因为同一个 key 的多次 `set` 天然按 timestamp
严格递增的顺序调用，每个 key 的历史记录本身就是有序数组，`get` 用
`bisect_right` 二分定位"最后一个 timestamp <= 查询值"的位置即可，和 21 类
"快照数组"的查询是完全同构的写法。

**`SparseTable`（稀疏表）**：和 27 类的线段树/树状数组放在一起对比会更清楚
它的定位——线段树支持修改（O(log n) 更新 + O(log n) 查询），稀疏表**不支持
修改**（数组一旦变化要整表重建），但换来了 O(1) 查询，是"静态数组、查询远
多于修改（甚至压根不修改）"场景下比线段树更快的选择。它能做到 O(1) 查询的
关键是利用了"区间最值允许查询区间重叠"这个特殊性质，这是求和类问题不具备的。

**`UndoableUnionFind`（可撤销并查集）**：和 17 类的标准并查集对比着看最有
启发——标准并查集用路径压缩换 amortized O(α(n))，但路径压缩会"顺手"修改很多
节点，没法简单撤销；可撤销版本放弃路径压缩、只用按秩合并，换来"一次 union 只
修改一个节点"，从而能被一条历史记录精确撤销，代价是 find 变慢到 O(log n)。
这个取舍在离线算法（比如按时间倒序处理"删边"操作，转换成正序"加边"）里很常见。

**`MonotonicQueue`（手写单调队列）**：16 类已经用 `collections.deque` 的现成
接口实现过"滑动窗口最大值"，这里的练习价值是理解 `deque` 本身是怎么做到 O(1)
首尾操作的——用双向链表而不是 Python 原生 list（`list.pop(0)` 是 O(n)）。
"单调"性质本身（push 新元素时从队尾弹出所有更小的旧元素）和 16 类完全一样，
只是这次自己搭好了底层的双端队列。

**`SqrtDecomposition`（分块）**：和 27 类的线段树解决的是同一个问题（区间批量
加 + 区间和），复杂度 O(sqrt(n)) 比线段树的 O(log n) 慢，但实现更朴素、常数
因子更小——分块是"当问题不方便写成线段树那种二叉树式的合并逻辑"时的常见退路，
也是很多线段树选手一开始用来验证思路是否正确的"参照系"（先写一个分块暴力核对
线段树的答案）。

**`XorTrie`（01-字典树）**：20 类的 `Trie` 按字符（26 叉）组织，这里换成了
按二进制位（2 叉）组织——本质上是同一种"用树的路径表示前缀共享"的思想，应用到
了"求两个数最大异或值"这个数值问题上。贪心策略（每一位都尽量走和当前位相反的
分支）成立的原因是"二进制高位的权重永远压倒所有更低位之和"，不存在需要回溯
的情况。

**`SimpleBloomFilter`（布隆过滤器）**：这是本章唯一一个"有代价的"数据结构——
它不追求 100% 准确，用固定大小的空间换取"绝不漏判、小概率误判"的近似成员
判断，是"用空间换时间"这个母题在**概率**意义上的延伸（而跳表是"用空间换时间"
在**期望**意义上的延伸）。工程上大量用在"先用布隆过滤器快速排除肯定不存在的
key，避免每次都去查真正昂贵的数据库/磁盘"这类场景。

## 小结

1. 这一章的结构没有共同的"一招鲜"技巧，但有共同的思考框架：**每个数据结构都
   是在某个操作上做了让步，换取另一个操作的加速**——跳表让步"确定性"换查找
   速度；稀疏表让步"可修改性"换 O(1) 查询；可撤销并查集让步"路径压缩"换
   "可撤销"；布隆过滤器让步"绝对准确"换极小的空间占用。
2. 遇到新场景先问"这个场景真正需要哪些操作、放弃哪些操作是可以接受的"，再
   决定选哪个数据结构，而不是背题型对号入座。
3. 真实 LeetCode 设计题（21 类的 895/432/1206 + 本类的 981）的共同点是"给一组
   接口复杂度要求，反推需要哪些数据结构组合"——这一点和 21 类"设计题"是同一个
   母题的延伸，这里的组合更精细（双向链表 + 分层索引、按秩合并且不压缩路径等），
   适合放在学完前面所有基础结构之后再看。
