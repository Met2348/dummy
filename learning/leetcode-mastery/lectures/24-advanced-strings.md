# L24 · 高级字符串算法 —— KMP / 中心扩展 / Rabin-Karp 滚动哈希

> 对应代码：`src/problems/p24_advanced_strings.py`。

## 这几题为什么算"硬核"

Phase 1 的字符串章节（04 · 字符串）处理的题目大多可以用"暴力 + 一点小聪明"（双
指针、滑动窗口）在可接受的复杂度内解决。这一章的几道题不一样——想要做到线性或
近线性复杂度，**必须**依赖专门设计的字符串算法（KMP、Manacher、Rabin-Karp），
单纯的暴力优化到头也甩不掉平方级的复杂度。这是字符串算法里真正需要"背算法"而不是
"临场想技巧"的部分。

## 深挖：KMP 实现 strStr（LC28，Medium）—— 失配时为什么能跳过一大截

**先想清楚朴素做法在浪费什么**：暴力字符串匹配的做法是，对 haystack 的每一个
起点，都从头开始逐字符比较 needle，一旦发现某个字符不匹配，就把起点往后挪一位、
重新从 needle 的第 0 个字符开始比较。这个"退回起点"的动作，正是浪费所在——
匹配失败之前，我们其实已经知道"刚刚被匹配上的这一段"内容是什么（它等于
`needle[0..j-1]`，j 是失配时 needle 指针的位置），但朴素做法直接把这个信息扔掉了。

**KMP 的核心 insight**：如果 `needle[0..j-1]`（刚匹配上的这一段）自身存在一个
"长度为 L 的前缀，同时也是它的后缀"，那么这个前缀必然已经在 haystack 里出现过——
就是刚匹配上的那一段的**开头部分**。既然这部分已经确认匹配，下一次比较完全可以
直接从 `needle[L]` 继续，而不必把 needle 的指针整个退回 0 重新开始。这正是
**lps 数组**（最长相同前后缀长度数组）要存的信息：`lps[i]` 表示 `pattern[0..i]`
这一段里，"最长的、既是真前缀又是真后缀"的长度。

```python
def _build_lps(pattern: str) -> list[int]:
    n = len(pattern)
    lps = [0] * n
    length = 0
    i = 1
    while i < n:
        if pattern[i] == pattern[length]:
            length += 1
            lps[i] = length
            i += 1
        elif length != 0:
            length = lps[length - 1]
        else:
            lps[i] = 0
            i += 1
    return lps
```

主匹配过程中，一旦发生失配（`haystack[i] != needle[j]` 且 `j != 0`），把 needle
的指针 `j` 跳到 `lps[j-1]`——这一步跳过的，正是"已经证明不可能匹配"的那部分位置，
而 **haystack 的指针 `i` 完全不需要回退**，因为 haystack 里已经扫过的字符不会
因为 needle 指针的调整而需要重新比较。

```python
i = j = 0
while i < len(haystack):
    if haystack[i] == needle[j]:
        i += 1
        j += 1
        if j == len(needle):
            return i - j
    elif j != 0:
        j = lps[j - 1]
    else:
        i += 1
return -1
```

这就是 KMP 相比朴素算法的本质提升：**haystack 的指针只会前进、不会回退**，整个
匹配过程摊还下来是 O(n+m)，而不是朴素算法最坏情况下的 O(n·m)。

**容易出错的地方**：失配时如果图省事直接把 `j` 重置为 0（退化成朴素算法），虽然
结果依然正确，但完全丧失了 KMP 的加速效果——这是初学者写"看起来像 KMP、其实是
暴力"代码的常见误区，必须确认失配分支写的是 `j = lps[j-1]` 而不是 `j = 0`。

## 其余 3 题点睛

- **LC459 重复的子字符串模式**：判断字符串能否由更短的子串重复拼接而成。最简洁
  的写法是 `s in (s+s)[1:-1]`，但更能体现"进阶理解"的是复用 LC28 的 lps 数组：
  取 `s` 自身 lps 数组的最后一个值 `border`（s 的最长相同前后缀长度），若
  `border > 0` 且 `len(s) % (len(s) - border) == 0`，说明 s 存在一个长度为
  `len(s) - border` 的最小循环节——这是因为"最长相同前后缀"和"字符串的最小周期"
  在数学上是等价的概念。`border == 0` 时不能直接套这个公式（`len(s)-border`
  等于 `len(s)` 本身，取模恒为 0，会被误判为存在循环节）。
- **LC647 回文子串**：中心扩展法——每个回文串都有一个对称中心（奇数长度是一个
  字符，偶数长度是相邻字符间的空隙），枚举所有 n 个字符中心和 n 个"空隙"中心，
  从每个中心向两边同时扩展统计能扩展出多少个回文串。复杂度 O(n²)，对面试场景
  已经足够；如果需要 O(n)，要用 Manacher 算法（利用回文串关于中心对称的性质
  复用已算出的信息），但其实现复杂度在真实面试里通常不要求手写，了解存在即可。
- **LC1044 最长重复子串**：二分答案（子串长度）+ Rabin-Karp 滚动哈希。二分
  是因为"是否存在某个长度的重复子串"具有天然的可判定性，可以用二分查找答案；
  对每个固定长度，用滚动哈希把"判断某个长度的子串是否出现过"降到均摊 O(1)——
  窗口滑动一位时，新哈希值可以用旧哈希值通过"减去滑出字符、乘 base、加上滑入
  字符"的公式 O(1) 递推。取模的 mod 必须是足够大的质数（代码里用 Mersenne 质数
  `2^61-1`），且哈希相同不代表子串真的相同，发现哈希匹配后必须用真正的字符串
  切片比较做二次确认，避免哈希碰撞导致误判。

### 深挖：最短回文串（LC214，Hard）—— 怎么把"求最短回文"转化成"求最长相同前后缀"

**先想清楚这道题在问什么**：只能在字符串 s **前面**添加字符，把它变成回文串，
求最少需要添加多少字符（等价于求"添加后的最短回文串"）。如果 s 前面添加的
那一段记作 `prefix`，那么 `prefix + s` 是回文串，意味着 `prefix` 这一段，
其实必然等于 "`s` 后面那一段没有被"利用"上的内容，反过来拼在前面、镜像对称"
的结果——直觉上，`prefix` 越短，说明 `s` 自身"从头开始能构成回文"的那一部分
越长。于是问题精确地转化成：**求 s 从下标 0 开始的最长回文前缀长度 L**，
知道 L 之后，`prefix` 就是 `s[L:]` 反转的结果，答案是 `reverse(s[L:]) + s`。

**怎么求"从头开始的最长回文前缀"**：如果 `s` 的某个前缀 `s[:L]` 是回文，
那么 `s[:L]` 反过来读还是它自己，也就是说 `s[:L]` 同时也是 `reverse(s)` 的
**后缀**（把整个 s 反转之后，原来在开头的这一段回文前缀，反转后依然在同样的
内容，只是出现在反转字符串的末尾）。所以"s 从头开始的最长回文前缀"，就等价于
"`s` 的前缀"和"`reverse(s)` 的后缀"，两者能重合的最大长度——这正是 KMP 的
lps 数组要解决的"最长公共前后缀"问题，只不过这里的"前缀"和"后缀"分别来自
两个不同的字符串（`s` 和 `reverse(s)`），而不是同一个字符串内部。

**具体构造**：把 `s` 和 `reverse(s)` 用一个 s 中不会出现的分隔符 `#` 拼接成
一个新串 `combined = s + '#' + reverse(s)`，对 `combined` 求 KMP 的 lps 数组，
取最后一个值 `lps[-1]`：

```python
rev = s[::-1]
combined = s + "#" + rev
lps = _build_lps(combined)
overlap = lps[-1]
return rev[: len(s) - overlap] + s
```

`lps[-1]` 表示 `combined` 这个新串"最长的、既是真前缀又是真后缀"的长度——因为
`combined` 是 `s` 和 `reverse(s)` 拼在一起的，它的"真后缀"必然落在 `reverse(s)`
这一段里，"真前缀"必然落在 `s` 这一段的开头，所以这个"相同前后缀"实际上就是
"`s` 的前缀"和"`reverse(s)` 的后缀"的最大重合部分，正是我们要找的 L。

**为什么必须插入分隔符 `#`**：如果不插入分隔符，直接拼接 `s + reverse(s)`，
求 lps 时有可能让"前缀"和"后缀"这两段在中点附近发生重叠——比如前缀已经越过
了 `s` 和 `reverse(s)` 的分界点，跑到了 `reverse(s)` 内部，这样算出来的
"相同前后缀"长度可能会超过 `len(s)`，不再对应"s 的前缀"这个原始含义。插入
一个双方都不会出现的字符 `#` 之后，任何"前缀"最多只能延伸到 `#` 之前（也就是
`s` 的范围内），任何"后缀"最多只能从 `#` 之后开始（也就是 `reverse(s)` 的
范围内），就不会发生这种越界重叠。

**易错点**：1) 忘记插入分隔符，或者分隔符选用了 s 中可能出现的字符（比如
只含小写字母的字符串里不能用字母做分隔符）；2) 算出 `L = lps[-1]` 之后，
容易直接把 `s[L:]` 原样拼到前面，而不是**反转后**再拼——原样拼接得到的字符串
不是回文；3) 容易把这里的做法和 1392 题"最长快乐前缀"的做法弄混——1392 题
是对 `s` 自身求 lps，不需要任何拼接和分隔符，因为那里问的是"同一个字符串
自己的相同前后缀"，和这里"两个不同字符串之间的相同前后缀"是本质不同的两个
问题，只是恰好都能用同一个 `_build_lps` 函数计算。

### 其余 3 题点睛

- **LC1392 最长快乐前缀**：和 LC214 用的是同一个 `_build_lps` 函数，但完全不需要
  拼接分隔符——直接对 `s` 自身求 lps 数组，`lps[-1]` 的定义本身就是"整个 s 里
  最长的、既是真前缀又是真后缀的长度"，恰好就是"最长快乐前缀"的定义，是 lps
  数组最直接、不需要拐弯的应用。容易犯的错误反而是"想多了"——误以为也要像
  214 题一样拼接 `s + '#' + reverse(s)`，这里完全不需要。
- **LC1316 不同的循环子字符串数量**：枚举重复片段长度 L，用一对相隔 L 的双指针
  滑动，维护"当前连续匹配了多少个字符"的计数器，一旦累计匹配满 L 个就找到了
  一个长度为 2L 的回声子串（`a+a` 形式），存入哈希集合去重；关键细节是命中后
  计数器要减一而不是清零，否则会漏掉紧接着重叠的下一个回声子串。这是 Rabin-Karp
  滚动哈希那一类"双指针 + 增量维护匹配状态"思路的变体，只是这里数据范围小，
  直接用字符串切片去重也能通过，不强制要求手写滚动哈希。
- **LC2223 字符串的Score和**：Z 函数（Z-array）的教科书应用——`z[i]` 定义为
  "s 从下标 i 开始的后缀，和整个 s 的最长公共前缀长度"，本题要求的每个后缀的
  score 恰好就是对应位置的 Z 值，把 `z[1]` 到 `z[n-1]` 全部加起来、再加上 n
  （字符串自己和自己的最长公共前缀，Z 函数按惯例不在下标 0 处定义，需要单独
  加上）就是答案。Z 函数维护一个"已知最靠右匹配窗口"的技巧和 KMP 的 lps 数组
  是同源的 insight，只是给出的信息形式不同（每个后缀 vs. 每个前缀内部），适合
  和 214/1392 放在一起对比着理解"前缀函数"这一整个家族。
