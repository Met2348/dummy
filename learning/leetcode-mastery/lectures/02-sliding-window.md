# L02 · 滑动窗口

对应代码：`src/problems/p02_sliding_window.py`。

## 通用心法：滑动窗口是特化的双指针

`01-arrays-two-pointers.md` 里讲的双指针，`left`/`right` 各自独立移动、根据大小关系决定移动哪一个。滑动窗口是双指针的一个特化版本，专门处理"连续子数组/子串"这一类问题，心法可以概括成一句话：

**右指针负责扩张窗口去"找一个可行解"，左指针负责收缩窗口去"在可行解里找最优解"。**

具体展开：

1. `right` 不断向右移动，每移动一步就把新元素纳入窗口的统计信息（累加和、字符计数等）。
2. 每纳入一个新元素后，检查窗口现在是否**违反了某个约束，或者已经满足了某个目标**——这个检查条件因题而异，是这一类题目里唯一需要"想"的地方。
3. 如果窗口不满足约束（违反了"不能有重复字符"）或已经满足了目标（凑够了"必须覆盖的字符"），就让 `left` 向右收缩，直到窗口重新回到"合法"或者"不再满足目标（因而不能再收缩）"的状态，收缩过程中更新答案。
4. `right` 继续前进，重复上面的过程。

因为 `left` 和 `right` 都只会单调向右移动、不会回退，两个指针合起来最多移动 2n 步，所以整体是 O(n)——这是滑动窗口相比"暴力枚举所有子串再检查"（O(n^2) 或 O(n^3)）能提速的根本原因：**每个元素只被 `right` 纳入一次、被 `left` 移出一次，而不是被每个子串各自重新扫描一次**。

真正决定一道滑动窗口题目难度的，从来不是"要不要用滑动窗口"，而是**这一步 2 里的收缩条件到底怎么写**。下面挑两道收缩条件写法完全不同的题目做深挖对比。

---

## 深挖 1：LC3 无重复字符的最长子串 —— "窗口内不能有重复"的收缩条件

### 第一步：暴力解怎么想

枚举所有子串 `s[i:j]`，对每个子串检查是否有重复字符（可以用 set，看 `len(set(s[i:j])) == j - i`），记录最长的合法长度。枚举子串是 O(n^2)，每次检查又要 O(n)，总共 O(n^3)（用滚动 set 优化检查能降到 O(n^2)）。

### 第二步：这里的收缩条件是什么

这道题的"违反约束"很具体：**窗口里出现了两个相同的字符**。所以收缩条件写成：只要 `s[right]` 这个新字符在窗口里已经出现过，就要把 `left` 移动到"重复字符上次出现的位置 + 1"，一次性跳过所有导致重复的旧字符。

注意这里的技巧点：不需要用"一步步挪 `left` 直到窗口里没有重复"这种循环写法（虽然那样也对），而是可以**直接算出 `left` 该跳到哪**，因为我们用一个哈希表记录了每个字符"最近一次出现的下标"，查一次表就知道该跳多远。这是"窗口内不能有重复"这类约束的特点：违反约束的原因是**唯一确定的**（就是这一个重复字符），所以能一步到位，不需要多次收缩。

### 第三步：为什么双指针能替代"每个子串重新检查"

关键单调性：如果 `[left, right]` 这个窗口内没有重复字符，那么把 `left` 固定、只把 `right` 缩小（`[left, right')`，`right' < right`）得到的子窗口，**一定也没有重复字符**（少了字符只会更"干净"）。反过来，如果 `[left, right]` 出现了重复，那么无论怎么进一步右扩 `right`，重复只会更多不会消失——**这个单调性保证了"违反约束"这件事只取决于当前窗口，不需要每次都重新扫描整个窗口去判断**，只需要判断"新加入的这一个字符"有没有制造重复。

### 第四步：一步步编码

1. 签名先行：`length_of_longest_substring(s: str) -> int`。
2. 哈希表 `seen: dict[str, int]` 记录字符→下标，`left = best = 0`。
3. `for right, ch in enumerate(s)`：如果 `ch` 在 `seen` 里**且**它的下标 `>= left`（这个"且"很关键，见下面易错点），把 `left` 更新为 `seen[ch] + 1`。
4. 更新 `seen[ch] = right`，更新 `best = max(best, right - left + 1)`。
5. 返回 `best`。

**为什么第 3 步一定要检查下标 `>= left`**：`seen` 表记录的是"这个字符历史上任意一次出现的位置"，但如果这次出现是在**当前窗口左边界之前**（已经被滑出窗口了），就不该被当作"窗口内重复"处理。比如 `"abba"`：处理到第二个 `a`（下标 3）时，`seen['a']` 里存的是下标 0，但此时 `left` 已经因为中间的 `b` 重复被推到了 2——如果不检查 `0 >= left`（`0 >= 2` 为假），就会误以为 `left` 要跳到 `0+1=1`，但 `left` 已经在 2 了，等于凭空往回跳，是错的。加上这个检查，`left` 才能保证只增不减。

## 深挖 2：LC76 最小覆盖子串 —— "窗口内必须覆盖所有目标字符"的收缩条件

### 第一步：暴力解怎么想

枚举所有子串，检查每个子串是否包含 `t` 的所有字符（含重复次数），记录满足条件里最短的一个。枚举 O(n^2)，每次检查覆盖关系又要扫一遍字符计数，更慢。

### 第二步：这里的收缩条件和 LC3 完全相反

对比一下就能看出这两题的收缩条件本质不同：

| | LC3 无重复子串 | LC76 最小覆盖子串 |
|---|---|---|
| 窗口右扩的目的 | 找一个"合法"的窗口（不违反约束） | 找一个"可行"的窗口（满足覆盖目标） |
| 什么时候需要收缩左边界 | 窗口**违反**了约束（出现重复）时——**必须**缩 | 窗口**已经满足**目标（覆盖了 t）时——**可以尝试**缩，看能不能更短 |
| 收缩到什么时候停 | 缩到窗口重新合法为止 | 缩到窗口**刚好**要变得不满足目标为止（再缩一步就不覆盖了） |

这是新手最容易搞反的地方：LC3 是"违反了才缩"，LC76 是"满足了才缩、缩到临界点为止"——两者收缩的**触发条件**方向是相反的，但都遵循"右扩找可行解、左缩找最优解"这同一个心法。

### 第三步：怎么判断"窗口是否已经覆盖了 t"——不需要每次都扫一遍

如果每次都完整比较窗口的字符计数表和 `t` 的字符计数表是否"逐项 >= "，会比较慢。技巧是维护一个 `missing` 计数器，表示"t 里还有多少个字符的需求量没被满足"：

- 用 `need` 记录 `t` 里每个字符还需要多少个（初始就是 `Counter(t)`）。
- 每次窗口新纳入一个字符 `ch`：如果 `need[ch] > 0`（说明这个字符正是当前还缺的），`missing -= 1`；然后无条件 `need[ch] -= 1`（即使这个字符本来不缺，也记录"多余"，用负数表示，为收缩阶段做准备）。
- `missing == 0` 就意味着窗口已经覆盖了 `t` 的所有字符——这是一个 O(1) 的判断，不需要重新扫描。

### 第四步：收缩阶段为什么要"先吐出多余字符，再判断能不能再缩"

代码里的收缩循环是 `while need[s[left]] < 0`：`need[s[left]]` 是负数说明这个字符在窗口里是"多余"的（超过 `t` 需要的量），可以放心地把 `left` 右移、`need[s[left]] += 1` 去掉一个多余的量，不会破坏覆盖关系。当 `need[s[left]]` 不再是负数（等于 0，说明这个字符的量恰好等于 `t` 需要的量），就不能再缩了——再缩这个字符的需求就得不到满足，`missing` 会变回正数。

之后再显式地把 `left` 处的字符"吐出去"（`need[s[left]] += 1; missing += 1; left += 1`），让窗口重新回到"不满足"的状态，继续让 `right` 前进找下一个可行窗口。

### 编码要点对照

1. 签名先行：`min_window(s: str, t: str) -> str`。
2. `need = Counter(t)`, `missing = len(t)`, `left = 0`, `best_len = len(s)+1`（哨兵值，表示"还没找到"）。
3. `for right, ch in enumerate(s)`：更新 `need[ch]` 和 `missing`；`missing == 0` 时先用 `while need[s[left]] < 0` 吐出多余字符收缩，再结算当前窗口长度是否刷新 `best_len`，然后主动把 `left` 处字符吐出、`missing += 1`、`left += 1`，让窗口回到未覆盖状态。
4. 最后按 `best_len` 是否被更新过决定返回切片还是空字符串。

对照 `p02_sliding_window.py` 里的 `min_window` 实现逐行核对。

## 其余 3 题点睛

**LC209 长度最小的子数组**：收缩条件是"窗口和已经 `>= target`"——和 LC76 的"已经满足才缩"是同一种方向（都是"可行了就尝试收缩找最优"），只是判断可行性的方式更简单（一个数字比较，而不是字符计数覆盖）。这题依赖"元素全为正"这个前提保证了"窗口和"的单调性，是滑动窗口能用在求和类问题上的典型前提条件。

**LC438 找到字符串中所有字母异位词**：这题和 LC3、LC76 都不同——窗口大小是**固定**的（等于 `len(p)`），不需要"扩张找可行、收缩找最优"这套动态伸缩逻辑，而是"定长窗口平移"：每次移动只需要移出最左一个字符、移入最右一个字符，属于滑动窗口里最简单的变体。可以把它看作 LC76 的"简化版"：同样是比较字符计数是否匹配，但因为长度固定，连"要不要收缩"这个判断都省掉了。

**LC567 字符串的排列**：本质和 LC438 是同一个模型（定长窗口 + 字符计数比较），区别只是这题只需要判断"存在性"（找到一个满足的窗口就立刻返回 True），而 LC438 要收集所有满足的起始下标。写代码时能明显感觉到这是同一套模板的两种收尾方式——先把 LC438 的模板吃透，LC567 几乎是抄一遍加一个提前返回。

## 进阶补充（Part II）

对应代码：`src/problems/p02_sliding_window_ii.py`。这一节的 7 道题延续"右扩找可行解、左缩找最优解"这同一个心法，**不重新讲框架**，而是集中扩展"收缩条件"本身的多样性——从"最多容忍几个 0"到"恰好包含 k 种不同值"再到"多个数组联合窗口"。挑其中最反直觉的一道题做深挖，其余点出和 Part I 或彼此的同源关系。

### 深挖：LC992 K 个不同整数的子数组 —— "恰好 K 个 = 至多 K 个 - 至多 (K-1) 个"

这题要统计"恰好包含 K 种不同整数"的连续子数组个数。第一反应容易直接照搬 LC3、LC76 的收缩循环模板，去维护一个"当前窗口是否恰好有 K 种"的判断——但很快会卡住：**"恰好等于 K"这个条件不具备滑动窗口需要的单调性**。对比一下"至多 K 种"：窗口越缩小，distinct 种类数只会不增不减地变小或不变，这个方向是单调的，可以用"违反了就收缩"的标准写法维护。但"恰好 K 种"不是这样——一个满足"恰好 K 种"的窗口，去掉最左边一个字符后，distinct 种类数可能不变（这个字符在窗口里还有别的位置），也可能减少（这是窗口里这个字符的最后一次出现）——**不存在一个统一的收缩方向能一直维持"恰好等于 K"这个条件**，没法直接写一个左指针收缩循环来处理。

这里的转化技巧来自最基础的组合数学容斥思路：**"恰好包含 K 种不同整数的子数组个数" = "至多包含 K 种的子数组个数" - "至多包含 K-1 种的子数组个数"**。直觉上理解：所有"至多 K 种"的子数组，去掉所有"至多 K-1 种"（也就是 distinct 数 <= K-1）的子数组，剩下的必然是 distinct 数恰好等于 K 的子数组——因为不存在 distinct 数大于 K 但被"至多 K 种"这个集合计入的情况。

这个转化的价值在于：它把一个"性质不单调、不好直接滑窗"的问题，转化成了两次"性质单调、可以直接套模板"的滑窗问题相减。代码上只需要写一个 `_at_most_k_distinct(nums, k)` 辅助函数（标准的"违反了才收缩"模板：`while len(count) > k: 收缩`），调用两次做差即可，完全不需要为"恰好"单独设计一套逻辑。这是滑动窗口题里最反直觉、但一旦见过就会反复用到的一招——**任何"恰好满足某计数条件"的子数组/子串统计题，都值得先问一句"能不能转化成两个'至多'相减"**。

### 其余 6 题点睛

**LC1004 最大连续 1 的个数 III**：收缩条件是"窗口内 0 的个数超过 k"，和 LC3"出现重复字符就要缩"是同一个方向（违反了才缩），只是把"是否重复"换成了"计数是否超限"，是这批题里最基础的一个变体。

**LC1493 删掉一个元素以后全为 1 的最长子数组**：直接复用 LC1004 的模板（把阈值从 k 改成 1），但因为题目要求"必须删除一个元素"，即使数组全是 1 也要强制减 1——这是本题和 LC1004 唯一的区别，也是最容易漏掉的一步。

**LC424 替换后的最长重复字符**：收缩条件"窗口长度 - 最大频次字符次数 > k"和 LC1004 是同一种"计数是否超限"的思路，只是这里的"计数"是"窗口内出现次数最多的字符"而不是"0 的个数"。实现上有个不直觉的技巧：`max_freq` 不需要在收缩时主动下调，因为算法只关心"历史最大可行窗口"，不会因此产生错误的更大答案。

**LC30 串联所有单词的子串**：和 LC438、LC567（Part I）是同一个"定长窗口 + 计数比较"模型，只是把"字符"换成了"单词"作为比较的基本单位——窗口长度固定为 `单词数 * 单词长度`，对每个候选起点切出若干个单词比较词频，是这一类模型里"基本单位变粗"的典型代表。

**LC632 最小区间**：把"窗口"从单个数组上的连续下标区间，泛化成了"同时在 k 个有序数组上各选一个下标"的组合窗口，用堆维护当前窗口的最小值和一个变量维护最大值，收缩逻辑变成"弹出最小值所在列表的下一个数"。这是滑动窗口思想在"多个有序序列"场景下的推广，如果之后学到"合并 K 个有序链表"（15-heap-topk 模块），会看到几乎一样的堆维护套路。

**LC643 子数组最大平均数 I**：定长窗口平移的最基础形式，和 LC438 是同一类"窗口大小固定、移进一个移出一个"的模型，唯一区别是这里比较的是"和"而不是"字符计数表"，是这批题里最简单的热身。

### 深挖：LC1839 所有元音按顺序排布的最长子字符串 —— 先转化问题表示，再套用简单扫描

这题最有意思的地方是"想清楚窗口该长什么样"比"套用窗口模板"更难。直接维护左右边界很难处理"一旦出现降序就必须整体作废重来"这种非局部的失效条件——`aeiou` 要求五种元音各自连续出现且顺序不能乱，普通的"扩张-收缩"窗口逻辑无法直接表达"顺序错了就整体作废"这种非局部约束。

转化思路：先把字符串按"连续相同字符"分组做**游程编码**（比如 `"aaeeiioouu"` 编码成 `[('a',2),('e',2),('i',2),('o',2),('u',2)]`），问题就降维成——在这个分组序列里，找最长的一段连续分组，满足：字母序列恰好是 `a,e,i,o,u` 这五个字母按顺序各出现一次。这样一来，原来"字符级别、非局部"的合法性判断，变成了"分组级别、局部"的简单检查（当前分组字母是否是前一个分组字母的下一个元音，且前一个分组字母不是 `u`），可以用一次线性扫描（本质是一个隐式的滑动窗口：分组级别的连续段）解决，长度直接用分组的字符总数累加。

这个思路的普适性在于：**当窗口的"合法性"条件是非局部的（依赖窗口内部的顺序结构，而不只是"计数是否超限"这种局部可增量维护的量）时，先想办法把问题转化到一个更粗的、局部性更好的表示上，而不是死磕在字符级别上强行维护复杂状态**。

**LC1423 可获得的最大点数**：窗口不是单向平移，而是"两头同时变化"（左边界持续右缩、右边界持续左扩）——把"从两端取牌"反过来想成"中间剩下的窗口最小"，这个转化思路本身比编码实现更重要，一旦想通，代码只是 k+1 次加减法。

**LC1652 拆炸弹**：定长窗口在**环形数组**上平移，所有下标都要过一遍取模；k 为正和为负对应两套方向相反的"移出/移入"下标公式，是这批题里最容易在下标运算上出小错、但一旦对拍出错例就能迅速定位的题。

**LC795 区间子数组个数**：和 Part II 深挖过的 LC992 K 个不同整数的子数组是同一个转化思路——"范围限定"的计数问题不具备滑动窗口需要的单调性，但拆成两个"上界限定"的计数问题相减，每个子问题都能用最基础的"违反了就清零"扫描解决。两题放在一起对比着记，能体会到"恰好/范围"类计数问题的通用降维方式。

**LC187 重复的DNA序列**：定长窗口配合哈希 set 判重的最直接应用，和 LC438 属于同一大类"窗口平移 + 用哈希做 O(1) 查重/计数"，区别只是这里窗口内容整体当作一个哈希 key（子串本身），而不是拆成计数表。
