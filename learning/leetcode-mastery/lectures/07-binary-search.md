# L07 · 二分查找 —— 维护"答案所在区间"的不变量

对应代码：`src/problems/p07_binary_search.py`

## 二分查找的本质不是背模板

新手学二分查找最常见的误区是背一套 `while (lo <= hi) { mid = ...; if (...) lo = mid+1 else hi = mid-1 }` 的模板，然后每道变形题都往里套，套不上就懵。

更靠谱的心法是：**二分查找的本质是维护一个不变量——"答案一定在当前区间里"，每一步用中点的信息把区间砍掉一半，区间缩小到不能再缩小时，剩下的就是答案。** 只要你在写代码之前，把这句话对应到具体的区间开闭写法上想清楚，代码就是这句话的直接翻译，不需要死记硬背。

### 区间开闭写法要统一，别中途混用

二分查找最容易出 bug 的地方不是"算法思路错了"，而是**区间开闭写法中途不统一**：一会儿把 `hi` 当成"最后一个可能的下标"（闭区间），一会儿又当成"第一个不可能的下标"（开区间），循环条件和收缩语句对不上，轻则死循环，重则漏掉边界。

本 lecture 和代码里统一约定：

| 写法 | 区间语义 | 循环条件 | 收缩语句 |
|---|---|---|---|
| 左闭右闭 `[lo, hi]` | lo 和 hi 都是候选下标 | `while lo <= hi` | `hi = mid - 1` / `lo = mid + 1` |
| 左闭右开 `[lo, hi)` | lo 是候选，hi 不是（"哨兵"） | `while lo < hi` | `hi = mid` / `lo = mid + 1` |

**推荐优先用左闭右开**，因为它有一个好处：循环结束时 `lo == hi`，这个值本身就是答案（比如"第一个满足某条件的下标"），不需要循环结束后再判断一次"到底是 lo 对还是 hi 对"。LC704（精确查找、找不到返回 -1）用闭区间写更直觉一点，本文件两种写法都保留，方便对照体会区别。

## 深挖 1：LC33 搜索旋转排序数组——怎么想清楚"哪一半有序"

### 题意

一个原本升序的数组，在某个位置被"掐断，前后段对调"了，比如 `[0,1,2,4,5,6,7]` 从下标 4 处旋转，变成 `[4,5,6,7,0,1,2]`。在这样的数组里用 O(log n) 找 target。

### 思路推导过程（不是结论，是怎么一步步想到的）

拿到这题，第一反应可能是"数组不是整体有序的，二分还能用吗？" 这里要先画图观察：任取一个 mid，把数组切成 `[lo, mid]` 和 `[mid, hi]` 两段。

```
[4, 5, 6, 7, 0, 1, 2]
 lo           mid    hi
```

不管旋转点在哪，**这两段里必然有一段是完全有序的**（另一段可能仍然是被旋转过的、不是有序的）。为什么？因为整个数组只被"掐断"了一次，掐断点只可能落在 `[lo, mid]` 或者 `[mid, hi]` 其中一段里，另一段里没有掐断点，自然是从头到尾升序的。

想清楚这点之后，剩下的问题变成两步：

1. **先判断哪一半有序。** 比较 `nums[lo]` 和 `nums[mid]`：如果 `nums[lo] <= nums[mid]`，说明左半 `[lo, mid]` 没有被掐断、是有序的；否则说明左半被掐断了，那么右半 `[mid, hi]` 一定是有序的。
2. **再判断 target 落在有序那一半的值域里吗。** 如果落在有序半的值域内，往那一半收缩；如果不在，说明 target 只可能在另一半里（不管另一半是不是整体有序，target 如果存在就一定在那里），往另一半收缩。

这里的核心转变是：普通二分"看 nums[mid] 和 target 谁大谁小"来决定往哪收缩；这题因为数组整体无序，没法直接比，所以**先退一步，问"这一半是不是有序的"，再问"target 在不在这一半的值域里"**——是把"能不能比大小"这件事情，拆成了两层判断。

```python
lo, hi = 0, len(nums) - 1
while lo <= hi:
    mid = lo + (hi - lo) // 2
    if nums[mid] == target:
        return mid
    if nums[lo] <= nums[mid]:            # 左半 [lo, mid] 有序
        if nums[lo] <= target < nums[mid]:
            hi = mid - 1
        else:
            lo = mid + 1
    else:                                  # 右半 [mid, hi] 有序
        if nums[mid] < target <= nums[hi]:
            lo = mid + 1
        else:
            hi = mid - 1
return -1
```

### 易错点

- `nums[lo] <= nums[mid]` 里的等号不能丢——当 `lo == mid`（区间只剩一两个元素）时，左半天然只有一个元素，也该判定为"有序"。
- 判断 target 是否落在有序半值域里时，边界的 `<=`/`<` 要和"这一半是否包含 mid 自身"对齐，否则会在 target 恰好等于边界值时把方向判错。

## 深挖 2：LC4 寻找两个正序数组的中位数——为什么不能真的合并数组

### 题意

给两个各自升序的数组 `nums1`（长度 m）、`nums2`（长度 n），求它们"假装合并成一个有序数组"之后的中位数，**要求时间复杂度 O(log(m+n))**。

### 为什么合并数组这条路走不通

看到"中位数"，最直觉的想法是"合并两个有序数组再取中间"——归并两个有序数组是 O(m+n)，这是能想到的最简单正确解法。但题目明确要求 O(log(m+n))，这个复杂度要求是个强烈的信号：**这题的解法一定不是"真的把数据合并/扫描一遍"，而是某种能跳过大部分数据的二分**。看到 log 级复杂度要求、数据本身有序，应该条件反射想到"二分"，但二分的对象是什么？不是直接二分"中位数的值"（值域可能很大且没有单调可判定的谓词），而是二分一个**位置**。

### 转化成"切割点"问题

关键的转念：**不需要真的合并数组，只需要找到一种"切法"，把 nums1 和 nums2 分别切成左右两半，使得"两个数组的左半部分拼起来"整体小于等于"两个数组的右半部分拼起来"，且左半总元素个数等于总长度的一半。** 一旦找到这个切割点，中位数就能直接由切割点附近的最多 4 个数字算出来（奇数总长取左半最大值，偶数总长取左右分界处两个数的平均值），完全不需要生成那个"合并后的数组"。

```
nums1:  ... a  |  b ...
nums2:  ...   c  |  d ...
             ↑ 左半到这里切          ↑ 右半从这里开始
要求：max(a, c) <= min(b, d)
```

### 为什么能对着"切几刀"来二分

设 nums1 切出 `i` 个元素放进左半，nums2 就必须切出 `half - i` 个元素（`half` 是左半应有的元素总数，由 m+n 决定，是个已知常数）——**i 和 j = half - i 是绑定的**，只要二分 i，j 自动确定，不需要对两个数组分别二分。

`i` 从 0 增大到 m 的过程中，`nums1` 左半的最大值（切割点左边界）单调不减。这个单调性正是二分能用的前提：如果发现"nums1 左边界的值 > nums2 右边界的值"，说明 `i` 切多了（nums1 左半拿了太大的数），应该把 i 往小调；反过来 i 往大调。这样每次都能把候选的 i 排除一半，直到找到满足 `max(左边界) <= min(右边界)` 的切割点。

为了让复杂度真正是 `O(log(min(m, n)))`（而不是 `O(log(m+n))` 或更差），二分永远在**较短的那个数组**上进行——较长数组的切割点由 `j = half - i` 直接算出，不需要额外二分。如果反过来在较长数组上二分，虽然结果依然正确，但会偏离题目对复杂度的要求，也是这题一个隐藏的"效率坑"。

```python
if len(nums1) > len(nums2):
    nums1, nums2 = nums2, nums1        # 保证在较短数组上二分
m, n = len(nums1), len(nums2)
half = (m + n + 1) // 2

lo, hi = 0, m
while lo <= hi:
    i = lo + (hi - lo) // 2
    j = half - i
    left1  = nums1[i-1] if i > 0 else float("-inf")
    right1 = nums1[i]   if i < m else float("inf")
    left2  = nums2[j-1] if j > 0 else float("-inf")
    right2 = nums2[j]   if j < n else float("inf")
    if left1 <= right2 and left2 <= right1:
        ...  # 找到合法切割点，直接算中位数
    elif left1 > right2:
        hi = i - 1
    else:
        lo = i + 1
```

### 易错点

- `i` 或 `j` 取到数组边界（0 或数组长度）时，对应的"左边界值/右边界值"要当成 `-inf`/`+inf` 处理，忘记这一步会直接数组越界或比较错误。
- 总长度是奇数还是偶数，中位数的取法不一样，别漏掉其中一支。
- 一定要先判断哪个数组更短再二分——这不是"优化"，是题目复杂度要求的硬性前提。

## 小结

拿到任何一道二分题，先问自己两个问题：

1. 我要维护的"答案所在区间"具体是什么？开闭写法选定之后，全程不要中途换写法。
2. 中点的信息能不能直接和 target 比大小？不能的话（比如 LC33 的旋转数组、LC4 的双数组切割），能不能通过"先分类讨论一下所在的区间/子问题"把它转化成能比大小的形式？

这两个问题想清楚之后，代码只是把想法翻译成 `lo`/`hi`/`mid` 的更新语句。

## 进阶补充（Part II）

> 对应代码：`src/problems/p07_binary_search_ii.py`。

这批题里最重要的新东西是引入**"二分答案"**这个子类型：前面所有题目二分的都是
"数组下标"（在数组里找一个位置），而这批题里有三道（LC875/LC1011/LC410）二分的
是**"答案本身可能的取值范围"**——候选答案不再是下标，而是"速度""运载能力""子数组和
上限"这类数值，配合一个贪心或模拟写成的 `check(候选答案)` 函数，判断这个候选答案
是否可行，再用二分不断收缩可行/不可行的边界。一旦看到"最小化最大值""最大化最小值"
这类"最优化一个满足某种可行性条件的数值"的题目描述，就该条件反射想到"二分答案"。

### 深挖：爱吃香蕉的珂珂（LC875，Medium）—— "二分答案"的完整思考过程

**题意**：珂珂要在 `h` 小时内吃完 `n` 堆香蕉，每小时只能选一堆、以速度 `speed`
根/小时进食；如果这一小时吃不完选中的这一堆，剩下的留到下一小时接着吃（不会去吃
另一堆）。求能在 `h` 小时内吃完所有香蕉的最小速度。

**第一步：为什么会想到二分**。题目要求的是"满足某个时间限制的**最小**速度"。直觉
上可以枚举 `speed = 1, 2, 3, ...`，对每个 `speed` 算一遍需要多少小时，第一个
使得"所需小时数 <= h"成立的 `speed` 就是答案——这是一个正确但可能很慢的思路
（`speed` 的取值范围可能很大）。要把这个枚举加速成二分，必须验证一件事：**随着
`speed` 增大，"吃完所有香蕉需要的总小时数"是不是单调不增的？** 想清楚这一点：
`speed` 越大，每一堆吃完花的时间 `ceil(pile / speed)` 只会更短或不变，所有堆的
时间加总自然也只会更短或不变。这个单调性保证了"能否在 h 小时内吃完"这个布尔谓词，
随 `speed` 从小到大扫描，一定是先若干个 `False`、然后全部变成 `True`，中间不会
来回跳变——这正是二分查找能够工作的前提（在一个单调的 0/1 序列里找第一个 1 出现
的位置）。

**第二步：check 函数怎么写**。`check(speed)` 就是"用这个速度吃完所有香蕉要几个
小时"：

```python
def hours_needed(speed: int) -> int:
    return sum((pile + speed - 1) // speed for pile in piles)
```

每一堆需要 `ceil(pile / speed)` 小时（哪怕只剩 1 根也要花掉一整个小时，因为
"吃不完这一堆也不会去吃别的堆"这个规则决定了不存在"匀给下一堆"的可能）。整数运算
里 `ceil(a / b)` 要写成 `(a + b - 1) // b`，直接用 `a // b` 会漏掉除不尽时
多出来的那一小时。

**第三步：二分的区间和收缩方向**。速度的取值范围下界是 `1`（不可能是 0，
一是语义上不合理，二是做除法会报错），上界是 `max(piles)`（哪怕最大的一堆也是
一小时吃完，再快也不会更节省时间，所以没必要超过这个值）。二分的目标是"**在所有
使得 `check(speed)` 为真的候选里，找最小的那一个**"，这是标准的 lower_bound
写法：

```python
lo, hi = 1, max(piles)
while lo < hi:
    mid = lo + (hi - lo) // 2
    if hours_needed(mid) <= h:
        hi = mid          # mid 可行，尝试更小的速度
    else:
        lo = mid + 1       # mid 不可行，必须增大速度
return lo
```

`check(mid)` 为真（能在 `h` 小时内吃完）时，`mid` 是一个可行答案，但也许还有
更小的可行答案，所以把 `mid` 留在候选区间里、收缩 `hi = mid`；`check(mid)`
为假时，说明 `mid` 太小，任何比 `mid` 更小的速度只会更慢，肯定也不行，必须
`lo = mid + 1`。这和 Part I 里 LC35 搜索插入位置的 lower_bound 骨架是完全
一样的写法，只是"数组下标"换成了"speed 的取值"，"`nums[i] >= target`"这个
谓词换成了"`check(speed)` 为真"。

**这道题带来的通用心法**：二分答案的三个要件缺一不可——1) 候选答案要有一个明确
的取值范围（这里是 `[1, max(piles)]`）；2) 存在一个可以在多项式时间内计算的
`check` 函数，判断某个候选是否可行；3) `check` 函数关于候选答案必须是单调的
（这里是"speed 越大越容易满足时间限制"）。只要能确认这三点，就可以把"直接算出
答案"的难题，转化成"猜答案 + 验证"的二分问题。

### 其余 7 题点睛

- **LC74 搜索二维矩阵**：矩阵每行首元素大于上一行尾元素，等价于整体拉平成一维
  升序数组，二分骨架不变，只是用 `mid // cols` / `mid % cols` 换算出行列下标。
- **LC240 搜索二维矩阵 II**：行列值域可能重叠，不能像 LC74 一样拉平二分；改用
  从**右上角**出发，利用它"本行最大、本列最小"的性质，比 target 大就左移一列，
  比 target 小就下移一行，每步排除一整行或一整列，复杂度是 O(m+n) 而不是
  O(log(mn))。
- **LC153 寻找旋转排序数组中的最小值**：比较 `nums[mid]` 和 `nums[hi]`——大于
  说明最小值在右半（`lo = mid + 1`），小于等于说明 mid 到 hi 这段没被掐断、
  最小值在 mid 或其左边（`hi = mid`，注意不是 `mid - 1`）。
- **LC162 寻找峰值**：只要求任意一个局部峰值（不是全局最大），靠"往数值上升的
  方向走一定能碰到峰值"这条单调性二分——比较 `nums[mid]` 和 `nums[mid+1]`，
  这和 LC153 的比较对象（`nums[mid]` vs `nums[hi]`）是两套完全不同的单调性
  依据，不要混用。
- **LC1011 在 D 天内送达包裹的能力**：和"爱吃香蕉的珂珂"是同一个二分答案模型，
  二分的是运载能力 capacity；下界必须是 `max(weights)`（否则最重的一件永远
  装不下），上界是 `sum(weights)`，check 函数用贪心模拟"按顺序尽量多装、超了
  就开新的一天"。
- **LC410 分割数组的最大值**：本质上和 LC1011 是同构的题目——把"最小运载能力"
  换成了"最小化后的最大子数组和"，二分的是这个上限值，check 函数同样是贪心
  统计"按顺序累加，超过上限就新开一段"需要几段。
- **LC154 寻找旋转排序数组中的最小值 II**：相比 Part I 的 LC33 和上面的
  LC153（都假设无重复元素），这题**多出的那一步特殊处理**是：允许重复后，
  `nums[mid] == nums[hi]` 时无法判断最小值在哪一半（两种完全不同形状的旋转
  数组可能在 mid、hi 处取到相同的值），只能保守地把 `hi -= 1` 收缩一格——因为
  `nums[mid]` 这个"替身"和被丢弃的 `nums[hi]` 值相等，不会丢失最小值所在的
  信息。也正是这一步"退而求其次、一格一格收缩"，让最坏复杂度从 O(log n)
  退化成 O(n)（比如数组里全是重复元素时）——这是"允许重复元素"这个条件带来的
  本质代价，不是实现上的缺陷。
