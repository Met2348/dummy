# L05 · 链表 —— 哨兵节点、快慢指针、以及"画图比脑内模拟更可靠"

> 对应代码：`src/problems/p05_linked_list.py`。链表是"理论都懂、一写就乱"的重灾区——
> 因为链表操作本质是在改一堆相互指向的引用，脑子里同时追踪三四个指针非常容易出错。
> 这篇先讲三条心法，把心法用熟了，再碰下面 6 道具体题。

## 三大心法

### 心法一：哨兵节点（dummy node）消除头节点特判

链表题里最常见的一类 bug，是"删除/插入操作如果发生在头节点，代码逻辑就得单独写一份"。
比如"删除倒数第 n 个节点"，如果 n 恰好等于链表长度，要删的就是头节点本身，返回值
也要变成 `head.next` 而不是 `head`——这种"头节点要不要特判"的分支非常容易漏掉
或者写错。

解决办法：在真正的头节点 `head` 前面**假想再放一个节点** `dummy`，让 `dummy.next = head`。
这样"删除头节点"就和"删除链表中间某个节点"变成了同一种操作（都是"删除某节点的
`next`"），不需要再单独判断"是不是头节点"。函数最后返回 `dummy.next` 而不是
原来的 `head`（因为头节点本身可能已经被删掉、替换掉了）。`merge_two_lists`、
`remove_nth_from_end`、`add_two_numbers`、`merge_k_lists` 这 4 道题的实现里
都用了这个技巧，用熟之后你会发现很多"这一定要单独处理头节点"的题，套上哨兵节点
就自动消失了。

### 心法二：快慢指针找中点 / 找环 / 找倒数第 N 个

"慢指针走一步、快指针走两步"这一个动作，配合不同的起点和终止条件，能解决好几类
看起来不相关的问题：

- 快指针到达链表末尾时，慢指针恰好在链表**中点**；
- 如果链表有环，快慢指针必然在环内相遇（`has_cycle`，LC141）；
- 让快指针先走 n 步，之后快慢指针同步前进，快指针到达末尾时，慢指针恰好停在
  **倒数第 n+1 个节点**（即待删除节点的前一个）—— `remove_nth_from_end`（LC19）用的
  就是这个变体。

这三个应用的本质都是：**用两个不同速度的指针在一趟遍历里维护住某种相对位置关系**，
不需要先遍历一遍算出链表长度再遍历第二遍——把两趟遍历压缩成一趟，是链表题里"空间
换时间"之外的另一种常见优化方向（这里其实是"趟数换空间/时间"）。

### 心法三：画图比脑内模拟更可靠

链表题的正确性几乎全部依赖"指针在某一步指向谁"，人脑很难在心里同时追踪 3-4 个指针
变量的实时状态而不出错。强烈建议新手在纸上画出节点方块和箭头，把 `prev`/`cur`/
`next_tmp`（或者 `slow`/`fast`）标在对应节点旁边，**每写一行代码就在纸上挪一次
对应的指针标签**，尤其是反转链表和合并 K 链表这种改动指针方向的题。这不是"偷懒的
新手才需要"，很多有经验的工程师面对复杂链表题时依然会先画图——这是一种能大幅降低
出错率的正规做法，不是能力不足的表现。

## 深挖 1：反转链表（LC206，Easy）—— 三个变量到底怎么挪

`reverse_list` 只有 5 行核心代码，却是新手最容易在"指针方向"上犯迷糊的题，原因是
一步要同时处理"读旧方向"和"写新方向"两件事。逐行拆开看：

```python
prev = None
cur = head
while cur is not None:
    next_tmp = cur.next    # ① 先把"旧方向"存起来，这一步之后 cur.next 要被改写了
    cur.next = prev        # ② 把 cur 的指向翻转：从指向"下一个"改成指向"上一个"
    prev = cur             # ③ prev 前移一位，跟上 cur
    cur = next_tmp         # ④ cur 前移一位，用的是①存的旧值，不是已经被改写的 cur.next
```

**为什么①必须在②之前**：②这一步会把 `cur.next` 改写成 `prev`，如果不先在①把原来的
`cur.next`（也就是链表里"下一个待处理的节点"）存到 `next_tmp`，这个引用就永远丢失了——
链表在这里断开，后面的节点全部访问不到。这是反转链表最典型的 bug：不是逻辑错，而是
**赋值顺序错了导致提前覆盖了还需要用的信息**。

**用纸笔画一遍会发生什么**：假设链表是 `1 -> 2 -> 3 -> None`。

| 步骤 | prev | cur | next_tmp | 链表当前状态（箭头方向） |
|---|---|---|---|---|
| 初始 | None | 1 | - | `1 -> 2 -> 3 -> None` |
| 第一轮①②③④ | 1 | 2 | 2 | `1 -> None`，`2 -> 3 -> None`（还没连起来） |
| 第二轮①②③④ | 2 | 3 | 3 | `2 -> 1 -> None`，`3 -> None`（还没连起来） |
| 第三轮①②③④ | 3 | None | None | `3 -> 2 -> 1 -> None` |
| 循环结束（`cur is None`） | 返回 `prev`，即 `3` | | | 新头节点是 3 |

把这张表在纸上画成"节点方块 + 箭头"，每一轮画完对比一下箭头方向变没变，比脑内推演
可靠得多——尤其是当你在白板面试、没有编辑器自动补全的场合，这张手绘表本身就是
"讲清楚思路"的最好工具。

**空链表 / 单节点为什么不需要特判**：`head=None` 时，`cur = None`，循环条件
`cur is not None` 直接不成立，循环一次都不执行，`prev` 保持 `None`，直接返回 `None`——
这正是空链表反转后应有的结果，不需要额外写 `if head is None: return None`。单节点
同理，循环执行一次后 `cur` 变成 `None`，`prev` 就是那唯一的节点。**想清楚"为什么
不需要特判"，而不是心虚地加一层判断**，是检验你是不是真的理解了这个循环不变量的
标志。

## 深挖 2：合并 K 个升序链表（LC23，Hard）—— 为什么两两合并是 O(nk)、换成堆能到 O(n log k)

**先分析两两合并（naive 做法）的复杂度**：假设有 k 条链表，节点总数为 n。如果按顺序
把第 1 条和第 2 条合并，再把合并结果和第 3 条合并，以此类推：

- 第 1 次合并：两条链表节点数加起来假设是 `n/k * 2`；
- 第 2 次合并：已经合并出的链表（大约 `2n/k` 个节点）再和第 3 条（`n/k` 个节点）合并；
- ……以此类推，第 i 次合并的代价大约是 `O(i * n/k)`；
- 总代价是 `sum(i * n/k for i in range(1, k))` ≈ `O(n * k)`。

问题出在哪：每一次合并，都要把"已经合并出来的、老早就排好序的一大段"重新扫一遍
（虽然只是线性扫，不是重新排序），但这一大段本身的有序性没有被有效利用——你其实只
需要知道"k 条链表里，当前最靠前的那个节点是谁"，不需要每次都把一整条已经合并好的
链表重新过一遍。

**换成堆之后发生了什么**：维护一个大小最多为 k 的最小堆，堆里始终装着"每条尚未耗尽的
链表中，当前最靠前（尚未被消费）的那个节点"。每一步只做两件事：

1. 弹出堆顶——这就是全局所有链表当前候选节点里最小的那个，直接接到结果链表尾部；
2. 如果这个节点还有下一个节点，把它的下一个节点压入堆，替补上刚才弹出的位置。

堆的大小**永远不超过 k**（因为每条链表任意时刻最多贡献一个节点在堆里），而每个节点
一生只会被 push 一次、pop 一次，push/pop 单次代价是 `O(log k)`（堆大小为 k），
所以总代价是 `O(n log k)`——比两两合并的 `O(nk)` 快了一个 `log k` 因子。当 k
很大（比如几千条链表）时，这个差距非常显著。

**为什么堆里要存 `(val, 唯一序号, node)` 三元组而不是 `(val, node)`**：Python 的
堆（`heapq`）在比较两个元组时，是逐个位置比较——先比第一个元素，相等了才继续比
第二个。如果只存 `(val, node)`，一旦两个节点的 `val` 相同，`heapq` 就会尝试比较
`node` 这两个 `ListNode` 对象本身，而 `ListNode` 类没有实现 `__lt__`（小于号）
这个方法，Python 会直接抛出 `TypeError: '<' not supported between instances of
'ListNode' and 'ListNode'`。加一个全局递增、两两不同的 `counter` 作为第二个
比较字段，就保证了"值相等时比 counter，counter 永远不同，比较到这里就能决出胜负，
根本不会走到比较 node 这一步"——这是"给不可比较的对象配一个可比较的替代品塞进
堆/排序"的通用技巧，以后遇到"这个对象不能直接扔进堆"的场景都可以用。

## 其余 4 题点睛

- **LC21 合并两个有序链表**：哨兵节点消除"第一个节点该接谁"的特判，双指针各自扫描，
  谁小接谁，某条耗尽后把另一条剩余部分整体接上（不需要逐个比较，因为剩余部分本身
  已经有序）。
- **LC141 环形链表**：Floyd 判圈法，深挖 2 提到的"快慢指针"思想的最简应用——循环
  条件要同时检查 `fast` 和 `fast.next`，避免对 `None.next` 取值报错。
- **LC19 删除链表的倒数第 N 个结点**：心法二 + 心法一的组合应用——快指针先走
  `n+1` 步（不是 n 步，多这一步是为了让慢指针停在待删节点的前一个，而不是待删节点
  本身），配合哨兵节点统一处理"删的是不是头节点"。
- **LC2 两数相加**：链表逆序存储数字，从头逐位相加正好对应从个位到高位相加，和竖式
  加法的进位逻辑完全一致；坑点是循环终止条件必须包含"进位还没清零"（`carry` 非 0
  也要继续），否则 `999+1=1000` 这种最高位产生新进位的情况会漏掉最后一位。
