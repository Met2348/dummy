# L20 · Trie 前缀树 —— 大量字符串的公共前缀查询

> 对应代码：`src/problems/p20_trie.py`。

## Trie 解决的是什么问题

Trie（前缀树）解决的是"**大量字符串的公共前缀查询**"这一类问题。它的结构很简单：
每个节点代表一个字符，从根节点走到某个节点的路径，拼起来就是一个前缀。多个字符串
如果共享同一个前缀，它们在 Trie 里会**共用同一段路径**，只在前缀分叉的地方才真正
分成不同的分支。

这个结构带来一个关键性质：插入一个单词、查询一个单词/前缀是否存在，复杂度都是
**O(单词长度)**，和 Trie 里已经存了多少个单词完全无关。相比之下，如果用一个普通的
列表/集合存所有单词，"是否存在某个前缀"这种查询需要遍历所有单词逐一判断，复杂度
和单词数量成正比。只要题目出现"大量字符串"+"前缀相关的判断/统计"，就应该第一反应
想到 Trie。

## 深挖：单词搜索 II（LC212，Hard）—— 为什么建一棵 Trie 能让"共享前缀只走一次"

**先想清楚朴素做法的浪费在哪**：如果 79 题"单词搜索"能在网格上判断一个单词是否
存在，那么把 words 列表里的每个单词都单独跑一次 79 题的 DFS，理论上也能解出这道
题。但代价是：如果 words 里有 "oath"、"oat"、"oaths" 这些共享前缀 "oat" 的单词，
朴素做法会让网格上从某个 'o' 开始、沿着 "o"->"a"->"t" 这条路径的 DFS **被重复走
好几遍**——每个以 "oat" 为前缀的单词都要独立地把这条路径探索一次，纯粹的浪费。

**Trie 如何消除这种浪费**：把全部 words 一次性建成一棵 Trie（单词的结尾节点上，
直接存整个单词字符串而不是普通的 `True` 标记，这样找到时能直接拿到答案，不需要
再拼接路径上的字符）。然后对网格做**一次**遍历：从每个格子出发做 DFS，但每一步都
同时在 Trie 上往下走——如果 Trie 当前节点没有这个方向对应的字符分支，说明"沿着
这条路径走下去，不可能拼出 words 里的任何单词"，直接剪枝，不再往这个方向探索。

```python
def dfs(r: int, c: int, parent: dict) -> None:
    letter = board[r][c]
    curr = parent[letter]
    word = curr.pop("$", None)
    if word is not None:
        found.append(word)

    board[r][c] = "#"
    for dr, dc in ((1, 0), (-1, 0), (0, 1), (0, -1)):
        nr, nc = r + dr, c + dc
        if 0 <= nr < rows and 0 <= nc < cols and board[nr][nc] in curr:
            dfs(nr, nc, curr)
    board[r][c] = letter

    if not curr:
        parent.pop(letter)
```

这正是 Trie 的核心价值：**"从当前路径出发，还有没有单词可能以它为前缀"这个判断，
被 Trie 变成了一次 O(1) 的字典查找**（`board[nr][nc] in curr`）。多个单词共享的
前缀部分（比如 "oat"），在网格上只会被走一次——因为这一次走过之后，Trie 上对应的
节点已经"记住"了后续还有哪些分支可能拼出合法单词，不需要为每个单独的目标单词
重新走一遍相同的路径。

**两个额外的剪枝优化，让搜索越搜越快**：

1. 找到一个单词后，立刻用 `curr.pop("$", None)` 清除 Trie 节点上的单词标记——
   防止同一个单词因为网格上存在多条路径而被重复加入结果，同时也让这个分支"变得
   更空"，为下一个优化做铺垫；
2. 如果某个 Trie 节点被清空（`not curr`，不再指向任何未完成的单词），就把它从
   父节点里删除。这样后续的 DFS 在这个方向上会直接因为"不在 Trie 里"而提前终止，
   搜索的分支数会随着已经找到的单词增多而**越来越少**。

网格 DFS 本身仍然需要"淹没当前格子防止重复使用、递归返回后恢复原字符"这套标准
套路（和 79 题一致），但因为多了 Trie 剪枝，实际探索的节点数远少于"逐词单独
DFS"的朴素方案。

## 其余 4 题点睛

- **LC208 实现 Trie**：最基础的 insert/search/starts_with，内部用嵌套 dict 模拟
  节点的 children，终点节点用哨兵键 `"#"` 标记"这里是一个完整单词"。search 和
  starts_with 唯一的区别就是要不要检查这个哨兵键——一条路径存在不代表路径尽头
  是一个完整插入过的单词，也可能只是别的单词的前缀。
- **LC211 添加与搜索单词**：结构和 LC208 一样，区别在查询时要处理 `.` 通配符——
  遇到 `.` 就把当前节点的所有子分支都当作候选，用 DFS 逐一尝试，只要有一条分支
  能匹配到底就算成功，本质是在 Trie 上做了一次"回溯搜索"而不是单纯的路径匹配。
- **LC677 键值映射**：如果每次 `sum(prefix)` 都遍历所有 key 判断前缀再求和，复杂度
  和 key 总数成正比。更好的做法是把"前缀和"直接维护在 Trie 节点上——每个节点存一个
  `_sum` 字段，insert 时把这次操作的"增量"（新值减旧值，处理"同 key 覆盖不累加"）
  沿路径加到每个节点的 `_sum` 上，`sum(prefix)` 直接读对应节点的 `_sum`，不需要
  遍历任何其他 key。
- **LC1268 搜索推荐系统**：容易联想到 Trie，但排序 + 二分更直接——products 排序后，
  "所有以某前缀开头的商品"在数组里是连续一段，用 `bisect_left` 定位这一段的起点，
  取最靠前的 3 个（已经是字典序最小的，因为整个数组有序），比建 Trie 更省事。
