# L23 · 矩阵与模拟 —— 没有高深算法，考的是"代码严谨性"

> 对应代码：`src/problems/p23_matrix_simulation.py`。

## 这类题在考什么

这一章的题目没有一个用到什么高深的算法或数据结构——旋转、螺旋遍历、原地修改、
状态编码，全是"把边界条件、下标偏移、原地修改的约束想清楚再动笔"。这恰恰是电话面
（phone screen）和一面里非常常见的考察点：面试官想看的不是"你懂不懂算法"，而是
"给你一个看起来简单的模拟题，你会不会在边界条件上摔跟头"。这类题目的正确率反而
比一些"中等难度算法题"更能体现代码的严谨程度。

## 深挖：矩阵置零（LC73，Medium）—— 用矩阵自身当标记位的陷阱

**先想清楚不限制空间时的朴素做法**：如果不要求 O(1) 额外空间，最直接的想法是用
两个集合分别记录"哪些行含有 0"、"哪些列含有 0"，扫完整个矩阵后按这两个集合把
对应的行、列置零。这个做法完全正确，只是空间复杂度是 O(m+n)。

**把空间压缩到 O(1) 的 insight**：与其用两个新的集合存这些信息，不如**借用矩阵
自身的第一行和第一列**当作标记位——`matrix[i][0]` 被标记成 0 就表示"第 i 行需要
整行置零"，`matrix[0][j]` 被标记成 0 就表示"第 j 列需要整列置零"。这样不需要任何
额外的数据结构，标记信息就"寄生"在矩阵自己身上。

**但这里有一个致命的陷阱**：第一行、第一列自己本身也是矩阵数据的一部分，一旦
开始用它们记录"其他行列是否需要置零"的信息，它们原本"自己是不是含有 0"这个
信息就会被覆盖、混淆。举个具体例子：如果第一行原本就有一个 0（比如
`matrix[0][3] == 0`），当后续处理第 3 列时，会往 `matrix[0][3]` 这个位置写 0
来标记"第 3 列需要置零"——这次写入和"第一行本身原来就有一个 0"这两件事叠加在
一起，事后已经无法区分"这个 0 是原来就有的，还是后来被当作标记写进去的"。

**解法是提前备份**：在开始使用第一行第一列做标记之前，先用两个独立的布尔变量
把"第一行本身有没有 0"、"第一列本身有没有 0"记录下来，之后才开始借用它们：

```python
first_row_has_zero = any(matrix[0][c] == 0 for c in range(cols))
first_col_has_zero = any(matrix[r][0] == 0 for r in range(rows))

for r in range(1, rows):
    for c in range(1, cols):
        if matrix[r][c] == 0:
            matrix[r][0] = 0
            matrix[0][c] = 0

for r in range(1, rows):
    for c in range(1, cols):
        if matrix[r][0] == 0 or matrix[0][c] == 0:
            matrix[r][c] = 0

if first_row_has_zero:
    for c in range(cols):
        matrix[0][c] = 0
if first_col_has_zero:
    for r in range(rows):
        matrix[r][0] = 0
```

三步走：1) 先备份两个布尔变量；2) 扫描矩阵**除第一行第一列之外**的部分，把 0 的
位置"记"到对应的第一行/第一列格子上；3) 根据这些标记（同样跳过第一行第一列自己）
置零其余格子；4) 最后单独处理第一行、第一列自己该不该被置零，用步骤 1 备份的
布尔变量判断——这时候第一行第一列已经完成了它们"标记位"的使命，可以放心地用
备份的信息决定它们自己的最终状态。

**这是这道题最容易出错的地方**：如果不先备份就直接开始标记，或者标记/置零两个
阶段的循环范围不小心从下标 0 开始（碰到了第一行第一列自己），都会让"数据"和
"标记"这两件事互相污染，产生错误结果。

## 其余 5 题点睛

- **LC54 螺旋矩阵**：维护 `top/bottom/left/right` 四个边界，按"右→下→左→上"
  循环收缩。走完"上边界"和"右边界"之后，走"下边界"和"左边界"之前，必须分别
  检查 `top <= bottom`、`left <= right`——矩阵只有一行或一列时，不加这个检查会
  把同一行/列重复走一遍。
- **LC48 旋转图像**：把"顺时针旋转 90 度"拆成两个更简单的操作——先沿主对角线
  转置（`matrix[i][j]` 和 `matrix[j][i]` 互换），再逐行反转，两步叠加的净效果
  正好是顺时针旋转。转置时内层循环必须从 `i+1` 开始，否则每一对会被换回来等于
  白做。
- **LC289 生命游戏**：原地更新的难点是"计算新状态需要用到邻居的旧状态"，但不能
  用额外矩阵存旧状态。解法是用多出的编码状态把新旧状态压缩存在同一个格子里——
  2 表示"旧值是活、新值是死"，3 表示"旧值是死、新值是活"，判断"邻居原本是否
  是活的"时要判断值是否在 `(1, 2)` 里，而不是只判断 `== 1`，否则会漏掉那些"原本
  活、但已经算出新状态是死"的邻居。
- **LC59 螺旋矩阵 II**：和 LC54 是完全对称的逆过程——不是"按螺旋顺序读出已有
  矩阵"，而是"按螺旋顺序把 1 到 n² 依次写进去"，边界收缩、边界检查的坑和 LC54
  完全一致。
- **LC36 有效的数独**：三组各 9 个集合分别记录每行/每列/每个 3x3 宫已出现的数字，
  宫的编号用 `(r // 3) * 3 + c // 3` 计算——必须先做整数除法压缩到宫格坐标系，
  直接用 `r*3+c` 之类的公式是错的；扫描时要跳过 `.`，且判断冲突要在真正加入集合
  之前完成，避免把数字自己和自己判定为冲突。

### 深挖：重新排列后的最大子矩阵（LC1727，Medium）—— 列可以重排时，排序为什么能替代单调栈

**先想清楚"列的顺序不能变"时这题怎么做**：如果不能重排列，这其实就是"最大矩形"
（LC84/85）的姊妹题——先把 01 矩阵预处理成"高度矩阵"（`matrix[i][j]` 表示以第
i 行为底、第 j 列这根柱子往上能连续摞多高），再对每一行跑一次单调栈，找出每根
柱子能往左右扩展多宽，用"高度 × 宽度"求最大矩形面积。单调栈的存在，本质上是在
解决"这几根柱子在矩阵里的相对位置是固定的，必须老老实实地看它们能不能连成一片"
这个约束。

**但这一题的条件是"列可以任意重排"**——这个额外的自由度直接把"单调栈"这个子
问题连根拔掉了：既然任意几列都可以挪到一起排列，那么"宽度为 k 的矩形能取到的
最大高度"根本不需要关心这几根柱子原来挨不挨着，只需要知道"这一行里，任意选 k
根柱子，它们的高度最小值最大能是多少"——把这一行的高度值**降序排序**之后，
"任选 k 根、高度最小值最大"这个值，就是排序后第 k 位（1-indexed）的高度，
因为前 k 个已经是这一行里最高的 k 个，其中最矮的那个自然就是这 k 个能撑起的
矩形高度上限。

```python
heights = [row[:] for row in matrix]
for r in range(1, rows):
    for c in range(cols):
        if heights[r][c]:
            heights[r][c] = heights[r - 1][c] + 1
best = 0
for row in heights:
    sorted_row = sorted(row, reverse=True)
    for width, height in enumerate(sorted_row, start=1):
        best = max(best, width * height)
```

**这道题真正的 insight 在于"用排序替代单调栈"这个替换本身**：单调栈解决的是
"位置固定、找左右边界"的问题；一旦允许自由重排，"找边界"这个动作就失去了意义
——直接排序、按秩取值，用一个 O(n log n) 的排序换掉了单调栈的 O(n)，是"多了一个
自由度、少了一个约束"之后的直接推论，而不是碰巧凑效的技巧。这也是这道题被列为
竞赛级补充而不是基础篇的原因：需要先识别出"重排"这个条件在算法层面等价于"去掉
了位置约束"，才能想到用排序代替单调栈，而不是照搬 84/85 题的框架生搬硬套。

**易错点**：高度预处理必须严格按行从上到下递推（`heights[r][c] = heights[r-1][c] + 1`）；
排序后"宽度 k 对应第 k-1 个（0-indexed）下标"这一步的 `+1` 很容易漏掉，导致面积
算少了一整列的宽度；这个做法的复杂度 O(m·n·log n) 比朴素单调栈解法（O(m·n)）
略高，这是"列可重排"这个额外自由度换来的简化代价，是一种常见的"用更简单的
数据结构、承受稍高复杂度"的权衡。

### 其余 4 题点睛

- **LC885 螺旋矩阵 III**：和 LC54/59 的"边界收缩"螺旋不是同一套路——这里没有
  现成的矩形边界，而是"以给定起点向外扩张"的螺旋，步长序列是 `1,1,2,2,3,3,...`
  （每两个方向切换一次才 +1），方向循环"东→南→西→北"。故意允许走到网格外
  （不计入结果，但仍要继续走），换来的是不需要对"起点靠近边缘"写任何特判；
  终止条件必须用"已收集坐标数 == rows*cols"，不能用步数作为停止信号。
- **LC1329 将矩阵按对角线排序**：同一条"从左上到右下"对角线上的元素，`r - c`
  是常数（注意不是 `r + c`，那是方向相反的反对角线）。按 `r - c` 分组收集、
  排序、再写回——排序时用降序 + `pop()`（弹出末尾即最小值）能省掉引入双端队列
  的麻烦，但必须保证排序方向和弹出位置严格匹配，否则写回顺序会整体颠倒。
- **面试题 311 稀疏矩阵的乘法**：把两个矩阵都压缩成"每行只记录非零 (列号,值)"
  的稀疏表示，乘法时只枚举非零元素对——只要有一方是 0，这次乘法天然被压缩表示
  跳过，不需要显式 if 判断。真正稀疏时运算次数远小于朴素的 O(m·k·n)；结果矩阵
  仍需要用稠密数组存储（乘积不一定继续稀疏），累加而不是覆盖写入，因为同一个
  位置可能被多个不同的 k 命中。
- **LC1275 找出井字棋的获胜者**：不需要维护真正的字符棋盘，给每行、每列、两条
  对角线各开一个整数计数器，A 走一步 +1、B 走一步 -1，绝对值达到 3 就是这条线
  被占满。副对角线的判断条件 `row + col == n - 1` 容易和主对角线的 `row == col`
  搞混，一步棋（尤其是正中心格）可能同时命中两者，需要分别检查；平局判断要用
  `len(moves) == 9`（棋盘下满），而不是"moves 遍历完"就直接下结论。
