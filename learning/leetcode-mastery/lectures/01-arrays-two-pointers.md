# L01 · 数组与双指针

对应代码：`src/problems/p01_arrays_two_pointers.py`（先读 `00-how-to-solve.md` 的通用四步法，再看这篇）。

## 什么时候该想到双指针

双指针不是一个孤立的技巧，而是"暴力解里有一层可以被省掉的内层循环"这件事的具体表现。看到下面几个信号里的任意一个，就该往双指针上想：

- **数组本身有序，或者可以先排序**。有序意味着"往左走值变小、往右走值变大"是确定的，不需要遍历所有组合去试，两个指针根据当前和与目标的大小关系就知道该往哪边挪——这是"从两端向中间逼近"这一类的核心前提（LC15 三数之和、LC11 盛水容器、LC42 接雨水都属于这一类）。
- **需要同时看"两头"的信息来决策**，比如"两条边界中较矮的那条""还没配对的最左/最右元素"。暴力解要枚举所有 (i, j) 组合，但如果观察到"往中间收缩一定不会丢掉最优解"，就能把一层循环省掉。
- **原地修改数组，且是"删除/去重"类操作**。这时双指针的分工变成 slow/fast 或 left/right：一个指针负责"读"，一个指针负责"写下一个该保留的值"，用 O(1) 额外空间完成本来看起来需要"新建数组"的操作（LC26 删除有序数组重复项、LC27 移除元素属于这一类）。
- **反例提醒**：如果数组无序、且元素之间没有"排序后单调"的性质（比如两数之和 LC1，target 减某个数不保证在已排序位置附近），双指针的"往哪边挪"就失去了判断依据，这时候应该退回到哈希表（见 LC1，以及 `03-hashing.md`），而不是硬套双指针模板。

下面挑三数之和、接雨水两题，把"从暴力解到双指针"的完整思考过程摊开讲；其余四题给出更精简的点睛，重点连回上面这几类信号。

## 深挖 1：LC15 三数之和 —— 排序为什么能把内层循环从 O(n) 判重问题里解放出来

### 第一步：暴力解怎么想

题目要找所有 `nums[i] + nums[j] + nums[k] == 0` 的三元组（下标互不相同）。最朴素的想法：三层循环枚举所有 (i, j, k)，判断和是否为 0，是就收集下来。这是 O(n^3)。

但这里立刻会撞上一个比复杂度更烦的问题：**去重**。数组里数值可以重复（比如 `[0,0,0]`），同一个"值三元组" `[0,0,0]` 可能被不同的下标组合命中很多次，如果只是"暴力收集再对结果去重"，你需要判断两个 list（比如 `[0,0,0]` 和 `[0,0,0]`）是否算重复——这本身又要排序或者用集合，非常绕。

### 第二步：为什么能排序——排序不会丢解，还能让"去重"变简单

三数之和这个问题的答案是"和为 0 的三元组集合"，**这个集合的定义完全不依赖原数组的顺序**——把数组打乱、排序都不会改变"哪些数值组合能凑出 0"。既然排序不影响是否有解、有哪些解，那就可以先排序，用排序换来一个非常重要的好处：

**排序之后，相同的值一定挨在一起**。这意味着"要不要跳过这个候选，因为它和前一个产生的结果重复"，只需要判断"这个位置的值是否和上一个位置的值相同"——O(1) 的判断，而不需要去比较已经生成的三元组本身。这就是排序在这题里最核心的作用：**把"三元组去重"这个复杂问题，转化成了"相邻元素去重"这个简单问题**。

### 第三步：为什么双指针能替代内层循环

排序之后，固定最外层的 `nums[i]`，问题变成："在 `nums[i+1:]`（这一段也是有序的）里找两个数，使它们的和等于 `-nums[i]`"——这正是"有序数组两数之和"（LC1 的有序变体），可以用双指针夹逼：

- `left = i+1`, `right = n-1`。
- 若 `nums[left] + nums[right] < target`，说明和小了，只有把 `left` 右移才可能变大（`right` 左移只会让和更小），所以移动 `left`。
- 若和大了，同理只移动 `right`。
- 若相等，记录答案，同时 `left` 和 `right` 都要往中间挪一步（并跳过重复值）。

**为什么这样不会漏解**：假设最优解是 `(L, R)`，当前 `left < L` 且 `right > R` 是不可能同时成立又和小了的情况——因为如果 `left` 还没走到 `L` 就已经算出"和小了"，说明当前 `left` 位置的值配合任何 `right <= 当前right` 都不够大，继续减小 `right` 只会更小，所以让 `right` 不动、`left` 右移是唯一能变大的方向，不会跳过 `L`。这样，双指针相当于把内层"两数之和"的 O(n^2) 枚举，替换成了 O(n) 的单调夹逼——本质上是利用了有序性排除了大量不可能的组合，而不是真的少做了搜索。

### 第四步：一步步编码

1. 签名先行：`three_sum(nums: list[int]) -> list[list[int]]`。
2. 排序：`nums = sorted(nums)`。
3. 外层遍历 `i`，先处理"跳过重复"：`if i > 0 and nums[i] == nums[i-1]: continue`。
4. 剪枝（可选但常见）：`if nums[i] > 0: break`（排序后从这里开始只会更大，不可能凑出 0）。
5. 内层双指针 `left, right = i+1, n-1`，按上面的三分支处理，命中时收集答案并对 `left`、`right` 各自做"跳过重复"处理。
6. 返回 `res`。

对照 `p01_arrays_two_pointers.py` 里的 `three_sum` 实现，逐行都能在上面五步里找到对应。

## 深挖 2：LC42 接雨水 —— 从"两个数组"到"两个变量"的空间优化

### 第一步：暴力解怎么想

对每一根柱子 `i`，它头顶能积的水只取决于两件事：**它左边最高的柱子** `left_max[i]`，和**它右边最高的柱子** `right_max[i]`——水面高度不可能超过两边里较矮的那个"墙"，否则水会从矮的那边流走。所以每个位置的积水量是：

```
water[i] = max(0, min(left_max[i], right_max[i]) - height[i])
```

暴力解：对每个 `i` 都各自向左、向右扫一遍求最高，O(n) 每个位置，总共 O(n^2)。

### 第二步：第一次优化 —— 预处理数组，O(n) 空间换 O(n) 时间

注意到 `left_max[i]` 其实就是 `left_max[i-1]` 和 `height[i]` 的较大值——它是可以从左到右递推算出来的，不需要每次都重新扫描。同理 `right_max[i]` 可以从右到左递推。于是：

1. 从左到右扫一遍，算出 `left_max` 数组。
2. 从右到左扫一遍，算出 `right_max` 数组。
3. 再扫一遍，套公式求和。

这样是 O(n) 时间，但用了两个长度为 n 的额外数组，O(n) 空间。

### 第三步：为什么双指针能把空间降到 O(1)

关键问题：**算 `water[i]` 真的需要精确知道 `left_max[i]` 和 `right_max[i]` 的具体值吗？** 仔细想一下：如果我们已经知道 `left_max[i] < right_max[i]`，那么 `min(left_max[i], right_max[i])` 就等于 `left_max[i]`——这时候 `right_max[i]` 具体是多少根本不重要，只要它比 `left_max[i]` 大就行，我们不需要提前算出它的精确值。

这就是双指针能用在这里的核心 insight：维护两个指针 `left, right` 从两端向中间走，同时维护 `left_max`（`left` 左侧包括自身的最大值）和 `right_max`（`right` 右侧包括自身的最大值）。**每一步只处理 `left_max` 和 `right_max` 中较小的那一侧**：

- 如果 `left_max < right_max`：这说明对于当前的 `left` 位置，它右边（不管具体在哪、不管 `right` 现在在哪）一定存在一个 `>= right_max > left_max` 的柱子挡着，所以 `water[left] = left_max - height[left]` 这个结论已经可以确定，不需要知道右边最高到底是多少。于是结算 `left`，然后 `left += 1`。
- 反之同理，结算 `right`，然后 `right -= 1`。

这样只需要两个变量 `left_max`、`right_max`（而不是两个数组），把空间从 O(n) 降到 O(1)。这正是双指针的一个重要模式：**不是所有信息都需要精确知道，只需要知道"够不够"**。

### 第四步：一步步编码

1. 签名先行：`trap(height: list[int]) -> int`，空数组特判返回 0。
2. `left, right = 0, len(height)-1`，`left_max, right_max = height[left], height[right]`，`total = 0`。
3. `while left < right`：比较 `left_max` 和 `right_max`，移动较小一侧对应的指针，更新对应的 max，累加 `total`。
4. 返回 `total`。

对照 `p01_arrays_two_pointers.py` 里的 `trap` 实现核对每一步。

## 其余 4 题点睛

**LC1 两数之和**：这题看似和双指针相关，但数组**无序**，双指针的"往哪边挪"没有依据——正确解法其实是哈希表（一次遍历同时记录和查询），见 `p01_arrays_two_pointers.py` 的 `two_sum` 和 `03-hashing.md`。把它放在这个文件里是为了对比：同样是"两数之和"，有序数组能用双指针（`patterns.py` 里的 `two_sum_sorted`），无序数组该用哈希表——**先看数组有没有序，再决定用哪种技巧**。

**LC26 删除有序数组中的重复项**：这是"原地覆盖式修改"这一类双指针的最简形式：`slow` 指向下一个该写入的位置，`fast` 负责往前找不同的新值。因为数组有序，重复元素连续出现，"是否要写入"的判断只需要看和上一个保留值是否相同。是 LC27 的"预备练习"——先掌握 slow/fast 分工，再去处理 LC27 里"不要求保序"的变体。

**LC27 移除元素**：同样是 slow/fast 的原地修改，但因为不要求保持相对顺序，指针分工变成了 `left`（负责读/写当前坑位）和 `right`（负责从末尾"借"一个还没检查过的值来补坑），本质是把"整体搬移"优化成了"局部交换"。和 LC26 对比着学：**要不要保序，决定了你是"顺序搬移"还是"末尾交换"**。

**LC11 盛最多水的容器**：和接雨水系出同源，都是"两端向中间收缩"，但这题只求一个全局最大值，不需要 LC42 那种"每个位置都要结算一次"的逐格计算。移动策略也类似：移动较矮的一侧，因为矮的一侧才是当前面积的瓶颈，移动它才有可能变好。可以把这题当作接雨水的"简化热身"：同样的双指针骨架，判断逻辑更直接。
