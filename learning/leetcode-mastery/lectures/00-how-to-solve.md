# L00 · 通用解题方法论（先读这篇，再碰任何一道题）

> 写给"理论都懂、但一道 OJ 题没提交过"的人。你缺的不是知识，是**把知识翻译成能跑的代码**这道手续——这个手续是可以练出来的固定流程，不是灵感。

## 通用四步法

拿到任何一道新题，不要立刻想"用什么算法"，按顺序走：

### 1. 读题排歧义

先别想解法，先把题目里**没写清楚但影响代码的细节**问一遍（面试里也要大声问出来）：
- 输入范围多大？（`n≤20` 和 `n≤10^7` 决定你能不能用指数级/暴力解）
- 有没有重复元素？有没有负数？会不会是空输入？
- 结果如果有多个合法答案，题目要哪一个（任意一个 vs 特定的一个）？
- 原地修改还是返回新结构？

新手最容易吃的亏：题目条件都在，但读题时跳过了，写到一半才发现漏了个边界，返工。

### 2. 先写暴力解，保证"正确"优先于"快"

哪怕是 O(n²) 甚至 O(2ⁿ)，先在草稿里/脑内把"最笨但一定对"的解法过一遍。这一步的作用：
- 给你一个**正确性的锚点**——之后优化时,可以拿暴力解的结果对拍，而不是凭感觉。
- 面试里，暴力解本身就是分——空手都答不出比说错一个"聪明"解法更糟。

### 3. 找规律优化：能不能拿空间换时间

这是"手感"真正的核心。看暴力解里**哪一步是重复计算/重复查找**，然后问：
- 重复"查找某个值是否出现过"→ 哈希表，O(1) 查找换 O(n) 空间。
- 重复"找区间内最值/和"→ 前缀和 / 单调栈 / 滑动窗口，把 O(n) 重算换成 O(1) 增量维护。
- 子问题会被重复计算（比如 `f(i)` 依赖 `f(i-1), f(i-2)`）→ 动态规划，用一个数组/几个变量存"已经算过的答案"。
- 需要"有序地扩展所有可能"→ BFS（找最短/层数）或回溯（穷举所有解）。
- 数据本身有序或可以排序 → 二分查找，把 O(n) 查找砍到 O(log n)。

**这一步不需要你"想出很聪明的点子"**——它是一张对照表：看到"重复计算什么"，就去查"用什么结构换"。19 个分类的 lecture 就是把这张表拆开精讲。

### 4. 编码 + 手工对拍

- 先写函数签名和几个关键变量名，再填循环体——签名先行能逼你想清楚输入输出契约。
- 写完不要立刻信任它，拿题目给的官方样例**手动带入跑一遍**（心算或纸笔），尤其是边界样例（空输入、单元素、全部相同的输入）。
- 每道题的 `_self_test()` 就是这个"手工对拍"习惯的代码化——你闭卷写完之后，运行它，比自己心算更可靠。

## 没有自动补全的"白板"写法建议

真实技术面试常常没有 IDE 自动补全，甚至是共享文档/白板：
- **签名先行**：先写 `def solve(nums: list[int], target: int) -> list[int]:`，把输入输出想清楚再写实现，避免中途改类型。
- **变量名先想清楚**：`left/right`、`slow/fast`、`i/j` 这种约定俗成的名字提前定好，减少中途改名的混乱。
- **循环不变量先想好再动笔**：比如双指针题先想清楚"循环开始前 `[0, i)` 已经处理完"这句话成不成立，想清楚了再写循环体，而不是写一半发现不变量不对。

## 新手高频踩坑清单

- **off-by-one**：`<` 还是 `<=`，`[lo, hi)` 还是 `[lo, hi]`——写之前先在纸上画一下区间的开闭。
- **空输入 / 单元素边界**：`nums = []` 或 `nums = [1]` 时你的循环还跑得对吗？
- **忘记去重**：题目要求"不重复的组合/子集"时，排序后跳过相邻相同值是常见手法，容易忘。
- **忘记显式 return**：递归函数里某个分支忘了 return，导致返回了 `None`。
- **原地修改 vs 返回新对象**：改错了引用类型（比如改了传入的 list 却被外部当作没变化）。

## 复杂度速查表（面试里判断"这个解法能不能过"）

| 数据规模 n | 能接受的复杂度 |
|---|---|
| n ≤ 20 | O(2ⁿ)、O(n!) 都可能可以（暴力搜索/回溯） |
| n ≤ 1000 | O(n²) |
| n ≤ 10^5 | O(n log n) |
| n ≤ 10^7 | O(n) |
| 更大 / 只给数值范围 | O(log n) 或 O(1)（二分答案、数学公式） |

拿到题目先看 n 的范围，就能倒推"面试官期待什么复杂度"，避免在暴力解上死磕，也避免过度设计。

## 怎么用这套方法论配合本 track

1. 按 `01` → `19` 分类顺序学（已经按由易到难排好）。
2. 每类先读 lecture 里的"深挖例题"——完整看一遍从暴力到优化的思考过程。
3. **合上答案**，自己在编辑器里重写一遍该类目的其余题目（哪怕效率低于参考解，先保证跑通）。
4. 跑 `problems/pNN_*.py` 里的 `_self_test()` 对拍，红了就对照参考解看哪一步想岔了。
5. 用 `tracker.py`（见 `20-review-strategy.md`）把做过的题排进复习节奏，而不是做完就忘。
