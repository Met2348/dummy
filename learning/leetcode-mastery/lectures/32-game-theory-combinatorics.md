# L32 · 博弈论与组合数学

> 对应代码：`src/problems/p32_game_theory_combinatorics.py`。Phase 3 竞赛级新分类。

## 核心范式："必胜态"与"必败态"

博弈论题的核心是"先手是否必胜"这类问题，通常能通过"当前状态是必胜态还是必败态"
的记忆化搜索/DP 解决：

- 一个状态是**必胜态**，当且仅当存在**至少一种**走法，能让对手陷入必败态。
- 一个状态是**必败态**，当且仅当**无论怎么走**，对手都能陷入必胜态（也就是
  "所有走法都通向对手的必胜态"）。

这是一个从终局状态往回推的递归定义——先确定"最简单、最终局"的状态是必胜还是
必败（比如"没有石子可取了"是必败态），再层层往外传播。多数题目直接用记忆化
搜索实现（LC294 翻转游戏 II、LC464 我能赢吗都是这个模式）；有些题目能进一步
把博弈过程转化成"两人轮流做最优决策、最大化/最小化分差"的**区间 DP**（LC877
石子游戏、LC486 预测赢家、LC1406/LC1690 石子游戏系列）；极少数题目甚至能跳过
搜索，直接用数学归纳/组合分析得出闭式结论（LC292 Nim 游戏、LC1927 Sum Game）。

## 深挖：LC913 猫和老鼠 —— 当博弈需要"逆向"思考

**为什么不能直接往前搜索**：LC913 引入了"平局"这个第三种结局——如果双方都
不能强迫对方进入自己必胜的局面，游戏会无限循环下去（位置重复），判定平局。
如果试图用"从初始状态往前递归、记忆化"的方式做，会立刻碰到麻烦：状态之间
可能存在**循环依赖**（状态 A 的胜负依赖状态 B，状态 B 的胜负又依赖状态 A），
朴素的前向递归没法处理这种"互相等待对方结果"的情况，而且"平局"本身就是
"无限循环、永远分不出胜负"的产物，前向搜索天然不知道什么时候该放弃、判定平局。

**逆向拓扑传播(retrograde analysis)**：换一个方向思考——不是"从起点往前搜出
结果"，而是"从所有已知结果的终局状态出发，反过来推导它们的前驱状态"。终局
状态很好确定：老鼠在洞里(节点 0)——不管是谁的回合、猫在哪，老鼠必胜；猫和
老鼠在同一个节点——猫必胜。

```python
color: dict[tuple[int, int, int], int] = {}
degree: dict[tuple[int, int, int], int] = {}
for m in range(n):
    for c in range(n):
        degree[(m, c, MOUSE_TURN)] = len(graph[m])
        degree[(m, c, CAT_TURN)] = len(graph[c]) - (1 if 0 in graph[c] else 0)
```

**"前驱"的定义需要反过来想**：一个状态 `(m, c, t)` 的前驱，是"上一步谁移动、
从哪走到这里"的状态。如果当前轮到老鼠走（`t == MOUSE_TURN`），说明上一步是
**猫**在移动——前驱是 `(m, c_prev, CAT_TURN)`，其中猫从 `c_prev` 走到了 `c`
（`c_prev` 必须是 `c` 的邻居，且不能是洞）。反过来，如果当前轮到猫走，前驱
就是"老鼠从 `m_prev` 走到了 `m`"的状态 `(m_prev, c, MOUSE_TURN)`。这个"反着
定义前驱"的过程，本质上是把原图的边"反过来看"——和 22 类"找到最终的安全
状态"用反图 + 拓扑排序传播"安全性"是同一种思想的推广，只不过这里同时有
两个角色（猫、老鼠）在图上移动，状态空间从"一个节点"变成了"(鼠位置, 猫位置,
轮到谁走)"的三元组。

**传播规则的两种情形**：

1. 如果某个前驱状态"该走的那个玩家"，恰好就是已确定的后继状态的赢家——那
   这个玩家可以**主动选择**走向这个必胜的后继状态，前驱直接染上同样的颜色。
2. 否则，说明"该走的玩家"被迫面对一个对自己不利的后继。只有当这个前驱状态
   的**所有**可能走法都被证明"对当前该走的玩家不利"时（用 `degree` 数组
   统计"还有多少条边尚未被证明不利"，每次不利的传播就把某条边标记掉、
   `degree` 减一，减到 0 说明**所有**出路都已确认不利），才能反过来确定
   这个前驱状态归属于对手。

没有被染色的状态，最终就是平局——它们既没有"能主动走向己方必胜"的边，
`degree` 也没能减到 0（永远有至少一条边的胜负未定），对应着"双方都不会
主动送对方一个必胜局面，也没被逼到绝路"的僵持状态。

**一个容易被忽视的坑**：猫不能走到洞（节点 0），这个约束必须在两个地方
都体现——统计 `degree[(m,c,CAT_TURN)]` 时要把"走向节点 0"这条边排除掉
（否则会误以为猫还有一条没被证明不利的路，实际上那条路根本走不通）；构造
"猫方状态的前驱"时也要过滤掉"猫来自节点 0"这种不可能出现的历史。

## 其余 8 题点睛

- **LC292 Nim 游戏**：必胜态呈现 4 的周期规律——n 是 4 的倍数就是必败态
  （无论取 1~3 颗，对手总能把剩余数量重新拉回 4 的倍数），这是少数能不用
  显式搜索、直接用数学归纳得出 O(1) 结论的博弈题。
- **LC294 翻转游戏 II**：`win(state)` 记忆化搜索，枚举每个可翻转的位置，
  只要存在一种翻转让 `win(下一个状态)` 为 False，当前状态就是必胜态；用
  字符串本身作为记忆化的 key。
- **LC375 猜数字大小 II**：极小化极大的区间 DP——枚举第一次猜的 pivot，
  这一步花费是 `pivot + max(左边最坏情况, 右边最坏情况)`；"要保证赢"对应
  取最坏情况（max），"选择最优的第一步"对应在所有 pivot 里取最小值(min)。
- **LC877 石子游戏 / LC486 预测赢家**：同一个区间 DP 模型——`dp[i][j]`
  表示当前玩家面对区间能比对手多拿的分差，`dp[i][j] = max(nums[i] -
  dp[i+1][j], nums[j] - dp[i][j-1])`。两题唯一的区别是最终判断：LC877 用
  `> 0`（总数为奇数、不会平局），LC486 用 `>= 0`（允许平局也算先手赢）——
  说明"求分差"的模型本身和"具体怎么判胜负"是解耦的。
- **LC1406 石子游戏 III**：区间 DP 的推广——每次可以连续取 1~3 堆，且堆值
  允许为负，`dp[i]`（后缀分差）代替 `dp[i][j]`（区间分差），因为"总是取
  从当前位置到末尾"这个后缀结构比双端都能取的完整区间更简单。
- **LC1690 石子游戏 VII**：也是区间 DP，但这一步的"得分"是**剩下堆的总和**
  （而不是拿走的那一堆的值）——转移公式因此要用预处理的前缀和先算出
  区间总和，再减去被拿走的那一堆，容易和 LC877/LC486"直接拿走的值就是得分"
  的写法搞混。
- **LC1927 Sum Game**：唯一一道纯数学分析、不需要任何搜索或 DP 的题——
  "?" 总数为奇数时 Alice 必胜（可以留到最后一步打破平衡）；为偶数时，
  Bob 能赢当且仅当已知数字的和差恰好等于 `9 * (两侧问号数之差) / 2`，这是
  双方用最优策略互相拉扯后能到达的必然结果，可以直接用一个公式判断。
