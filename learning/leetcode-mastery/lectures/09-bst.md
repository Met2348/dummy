# L09 · 二叉搜索树（BST）—— 一条不变量推出所有技巧

对应代码：`src/problems/p09_bst.py`

## BST 的核心不变量

二叉搜索树（BST）看起来技巧五花八门（构造、验证、找第 K 小、找 LCA……），但它们全都是同一条性质的推论：

> **对 BST 的任意节点，中序遍历的结果一定是严格升序的**（等价地说：对任意节点，它的值严格大于左子树里所有节点的值、严格小于右子树里所有节点的值——而且这个约束对**所有祖先**都成立，不只是直接父子）。

本 lecture 的 4 道题，每一道都能追问"这题用到了上面这条性质的哪一部分"，追问清楚之后，解法自然就出来了，不需要死记"BST 专属模板"。

## 深挖：LC98 验证二叉搜索树——最容易踩的经典陷阱

### 题意

给二叉树根节点，判断它是否是一棵合法的 BST。

### 为什么"只比较父子节点"是错的

几乎每个第一次做这题的人都会写出类似这样的代码：

```python
def is_valid_bst_WRONG(node):
    if node is None:
        return True
    if node.left and node.left.val >= node.val:
        return False
    if node.right and node.right.val <= node.val:
        return False
    return is_valid_bst_WRONG(node.left) and is_valid_bst_WRONG(node.right)
```

这段代码只检查了"每个节点和它的直接孩子"之间的大小关系。看这个反例：

```
        5
       / \
      1   4
         / \
        3   6
```

节点 4 和它的直接孩子 3、6 之间，"左 < 父 < 右"（3 < 4 < 6）是满足的。但整棵树不合法——因为节点 4 整体是根节点 5 的**右子树**，右子树里的每一个节点都必须严格大于 5，而 3 < 5。问题出在：**3 只跟它的直接父亲 4 比较过，从来没有跟更高层的祖先 5 比较过。** "只比较父子"这种写法，天生就无法捕捉"某个值相对更高层祖先"的约束，这正是这题的陷阱所在。

### 正确做法：传递上下界区间，而不是只看父子

给每个节点递归传一对 `(lower, upper)`，含义是"这个节点的值必须落在开区间 `(lower, upper)` 里"。这个区间不是凭空发明的，而是**所有祖先节点的约束层层收紧后叠加出来的结果**：

- 往左子树递归时，新的上界收紧为当前节点的值（因为左子树所有值必须小于当前节点）——下界不变，继续沿用当前的下界（因为左子树同样要满足"比更高层的祖先大"这个更早施加的约束）。
- 往右子树递归时，新的下界收紧为当前节点的值——上界不变，同样沿用当前的上界。

```python
def is_valid_bst(root, lower=float("-inf"), upper=float("inf")):
    if root is None:
        return True
    if not (lower < root.val < upper):
        return False
    return (is_valid_bst(root.left, lower, root.val)
            and is_valid_bst(root.right, root.val, upper))
```

回到反例：根节点 5 递归进入右子树 4 时，区间收紧为 `(5, inf)`；4 递归进入左子树 3 时，上界收紧为 4（变成 `(5, 4)`），但下界仍然带着"必须 > 5"这个更早的约束——`3` 显然不满足 `5 < 3 < 4`，直接判为不合法。**区间是一路带着所有祖先的约束往下传的，不是只看眼前这一层父子**，这正是修好"只比较父子"这个陷阱的关键。

### 易错点

- 只比较父子节点，漏掉更高层祖先的约束——本题最典型、最容易通过小样例但线上出错的写法。
- 边界用 `<=` 而不是 `<`：BST 定义里节点值互不相等，如果写成 `lower <= root.val <= upper`，会把"值相等"这种非法情况误判为合法。

## LC235 vs LC236：普通二叉树 LCA 和 BST LCA 的复杂度差异

LC236（普通二叉树的最近公共祖先，见 `08-binary-tree.md`）因为不知道任何值的大小关系，只能老老实实用后序遍历，递归检查**左右两侧子树**分别有没有 p/q，两侧都非空才能确定当前节点是 LCA——最坏情况要访问树中的每一个节点，时间复杂度 O(n)。

LC235 因为是 BST，多了一条 LC236 没有的信息："左子树所有值 < 当前节点 < 右子树所有值"。这条性质带来一个直接推论：**不需要检查两侧子树**，只需要比较 `p.val`、`q.val` 和当前节点值的大小：

```python
node = root
while node is not None:
    if p.val < node.val and q.val < node.val:
        node = node.left      # 两者都更小，LCA 只可能在左子树
    elif p.val > node.val and q.val > node.val:
        node = node.right     # 两者都更大，LCA 只可能在右子树
    else:
        return node            # 一大一小（或当前节点正好是 p/q 之一）→ 找到 LCA
```

如果 `p`、`q` 的值都比当前节点小，那么它们的最近公共祖先绝不可能藏在右子树里（右子树全都比当前节点大，不可能同时装下两个都更小的值所在的子树），所以可以完全不看右子树，直接往左走；反之同理。这样整个过程只沿着一条从根到 LCA 的单一路径往下走，不需要像 LC236 那样两侧都递归。

| | LC236 普通二叉树 LCA | LC235 BST LCA |
|---|---|---|
| 用到的信息 | 只有树的形状（父子关系） | 树的形状 + 每个节点的大小关系 |
| 需不需要检查两侧子树 | 需要（两侧都递归，再看谁非空） | 不需要（一次比较就知道该往哪一侧走） |
| 时间复杂度 | O(n)（最坏遍历整棵树） | O(h)（只走一条路径，平衡树是 O(log n)） |

这组对比也回应了本 lecture 开头那句话：**同样是"求 LCA"，BST 版本能比普通二叉树版本快一个数量级，唯一的原因是多了"中序遍历升序"这条不变量带来的有序性**——性质越强，能省下的搜索空间就越大。

## 小结

拿到任何一道 BST 题，先问："这题能不能靠'中序遍历升序'这条性质，把原本需要遍历/比较的一部分工作省掉？" LC98 提醒你这条性质必须严格地、带着所有祖先约束地检查；LC108 是反向利用它来构造平衡树；LC230 是直接利用它取第 k 个；LC235 是利用它把 LCA 的搜索范围从"两侧都要看"收窄成"只看一侧"。四道题，一条不变量。

### 深挖：LC99 恢复二叉搜索树——用"中序遍历找逆序对"定位被交换的节点

#### 题意

一棵 BST 中恰好有两个节点的值被错误地交换，导致它不再合法。要求不改变树的结构（不能新建/删除节点，只能修改 val），找到这两个节点并换回正确的值。

#### 为什么"中序遍历找逆序对"能定位到被交换的两个节点

核心不变量说："正常 BST 的中序遍历结果严格递增"。反过来想：如果中序遍历不是严格递增的，那些"不递增的地方"（相邻两个数里前一个反而更大，称为一个"逆序对"）必然就发生在被交换的节点附近——因为原本正确的序列是严格递增的，把两个位置的值互换后，被换的这两个位置本身、以及它们各自和"中序序列里的前驱/后继"之间的大小关系，就是唯一可能出问题的地方，其他位置的相对顺序完全没被影响。

被交换的两个节点在中序序列中的位置关系分两种情况：

- **不相邻**：会产生两个逆序对。比如原序列 `1,2,3,4,5` 把 `2` 和 `4` 换成 `1,4,3,2,5`——逆序对出现在 `(4,3)` 和 `(3,2)` 两处。第一个逆序对里"位置更靠前但值更大"的那个（`4`）就是需要往回换的第一个节点；第二个逆序对里"位置更靠后但值更小"的那个（`2`）就是第二个节点。也就是说：**第一次出现逆序时，记录靠前的那个节点为 `first`；每次出现逆序都更新靠后的那个节点为 `second`**——这样即使中间隔着若干正常递增的节点，最终 `first`、`second` 也能准确定位到这两个被交换的节点。
- **相邻**：只会产生一个逆序对，比如 `1,2,3,4,5` 把相邻的 `3,4` 换成 `1,2,4,3,5`，只在 `(4,3)` 这一处出现逆序，`first`（更靠前的 4）和 `second`（更靠后的 3）在同一次逆序判断里就都确定了。

用一次中序遍历，边走边维护"上一个访问的节点" `prev`，一旦发现 `prev.val > node.val` 就是发现了一处逆序：

```python
first = second = prev = None

def dfs(node):
    nonlocal first, second, prev
    if node is None:
        return
    dfs(node.left)
    if prev is not None and prev.val > node.val:
        if first is None:
            first = prev       # 只在第一次逆序时记录 first
        second = node          # 每次逆序都更新 second
    prev = node
    dfs(node.right)

dfs(root)
first.val, second.val = second.val, first.val
```

`first` 只在第一次逆序时赋值（之后被 `if first is None` 挡住不会再变），`second` 每次逆序都会被更新——这个不对称的写法正是"两处逆序共用同一个 `first`、只有 `second` 需要挪到最后一次逆序的位置"这条推理的直接翻译。最后把 `first.val` 和 `second.val` 交换，树就恢复合法了。

#### 易错点

- 第二次逆序出现时如果把 `first` 也重新赋值，会丢掉第一次逆序里那个真正需要交换的节点——`first` 全程只应该被赋值一次。
- 比较用的 `prev` 是"中序遍历顺序上的前一个节点"，不是"父节点"——这两者在 BST 里绝大多数情况下不是同一个节点，必须用一个显式变量在递归间传递（这里用 `nonlocal`），不能想当然地去比较父子。

- **LC96 不同的二叉搜索树（Medium）与 LC95 不同的二叉搜索树 II（Medium）——"计数"与"枚举"的对比**：两题是同一个"枚举谁是根、左边小值右边大值"框架的两种实现方式。LC96 只要**数量**，可以用 DP 压缩：`dp[k]` 表示 k 个值能构成多少种 BST，`dp[k] = sum(dp[j] * dp[k-1-j] for j in range(k))`（卡特兰数递推），不需要真的构造出树，效率是 O(n²)。LC95 要求返回**具体的树**，DP 的"只存数量"就不够用了，必须改用回溯——递归求出左区间、右区间各自能构成的所有子树列表，再做笛卡尔积拼出每一种完整的树；空区间要返回 `[None]` 而不是 `[]`，否则笛卡尔积会把"某一侧为空"的合法情况全部漏掉。这组对比也是很多"计数 DP"和"构造回溯"题的通用关系：**问"有多少种"往往可以压缩状态、只存数量；问"每一种具体是什么"就必须老老实实地枚举展开。**

### 深挖：LC1382 将二叉搜索树变平衡——先摊平、再对折

#### 题意

给一棵 BST（可能因为插入顺序问题严重退化，比如整棵树蜕化成一条只往一个
方向延伸的链），返回一棵值集合相同、但**高度平衡**的 BST（任意节点左右
子树深度差不超过 1）。答案不唯一，返回任意一个满足条件的即可。

#### 为什么"原树的具体形状"完全不重要

拿到这题，直觉上可能会想："要不要在原来这棵退化的树上做旋转操作（类似
AVL 树的左旋/右旋）把它转平衡？"这条路当然存在，但对退化到底的树（比如一条
长度为 n 的链）需要做 O(n) 次旋转、每次还要维护旋转后的平衡因子，实现复杂
且容易出错。

更好的转念是：**既然最终只关心"值的集合和相对顺序"，那就没必要在原来的树
形状上做手术，可以先把这棵树"摊平"成一个只保留顺序信息、完全丢弃原始形状
的中间产物，再在这个中间产物上重新盖一棵树。** 这里"摊平"的工具正是本
lecture 反复用到的核心不变量——**中序遍历给出严格升序的值序列**，而且这个
序列只取决于"树里有哪些值、大小关系是什么"，和树具体长成什么形状（一条链、
一棵完美树，还是任何中间形态）完全无关。

```python
values = []

def dfs(node):
    if node is None:
        return
    dfs(node.left)
    values.append(node.val)
    dfs(node.right)

dfs(root)
```

不管输入的 `root` 退化成什么样子，`values` 都会是同一个升序数组。

#### 第二步：对折重建——直接复用 Part I 的 LC108

一旦有了这个升序数组，"从一个有序数组构造一棵高度平衡的 BST"这个子问题
在 Part I 的 LC108（将有序数组转换为二叉搜索树）里已经解决过：递归地取
区间**正中间**的元素当根，两侧元素数量差不超过 1，递归下去每一层都保持
这个性质，天然保证高度平衡。

```python
def build(lo, hi):
    if lo > hi:
        return None
    mid = (lo + hi) // 2
    node = TreeNode(values[mid])
    node.left = build(lo, mid - 1)
    node.right = build(mid + 1, hi)
    return node

return build(0, len(values) - 1)
```

#### 这道题真正教会你的事

LC1382 表面上是一道"重新平衡一棵树"的新题目，但拆开来看，它其实是
**LC530/LC538 这类"中序遍历取出有序信息"的技巧，和 LC108"有序数组建平衡
树"的技巧，做了一次直接拼接**——没有引入任何新的算法思想。这提示一种
通用的解题直觉：**当一道题声称要处理"一个被破坏/退化的结构"时，先问自己
"能不能先摊平成一个更简单的、只保留必要信息的中间形式，再用已经会的技巧
在这个中间形式上重建"**，而不是执着于"在原始退化结构上直接修补"。LC99
恢复 BST（Part II 深挖）是同一种思路的另一个例子：不在树结构上直接找被
换的两个节点，而是先想"中序遍历本该长什么样"，再从"实际序列和理想序列的
差异"里反推答案。

#### 易错点

- 第一步的中序遍历必须原样保留所有值（包括重复值，如果输入允许的话），
  不能在这一步提前做任何过滤或去重——摊平这一步的唯一职责是"保留顺序
  信息"，不应该顺带做其他事情。
- 重建时取中点的方式（偏左 `(lo + hi) // 2` 还是偏右 `(lo + hi + 1) //
  2`）两种写法都能保证平衡，但一次递归调用里必须**从头到尾用同一种取法**，
  中途切换会破坏"两侧数量差不超过 1"这个不变量。
- 不要试图跳过"摊平"这一步、直接在原树上做旋转调整——对高度退化的输入
  （比如一条长度为 n 的链），基于旋转的做法实现复杂度和出错概率都远高于
  "摊平 + 重建"这个两步法，且后者的时间复杂度同样是最优的 O(n)。

### 其余 6 题（点睛）

- **LC653 两数之和 IV - 输入 BST（Easy）**：用哈希集合边遍历边记录"已经
  见过的值"，对每个新节点检查 `k - node.val` 是否已出现过——这个做法对
  任意二叉树都成立，提醒"输入是 BST"不代表"必须靠有序性解题"，有时候更
  朴素的哈希做法反而更直接。
  和 LC1 两数之和是同一个"用哈希集合记录已访问过的补数"的思路，只是把
  "数组线性扫描"换成了"树的 DFS 遍历"。
- **LC501 二叉搜索树中的众数（Easy）**：利用"允许重复值时，中序遍历里
  相同的值必然连续排在一起"这条推论，一次中序遍历边走边维护"当前值连续
  出现的次数"，破纪录就清空众数列表重新开始、打平纪录就追加进列表。
- **LC530 二叉搜索树的最小绝对差（Easy）**：三角不等式的直接推论——最小
  差一定出现在中序序列相邻的两个数之间，不需要比较任何"隔着至少一个数"
  的组合，一次中序遍历边走边和"上一个值"比较即可。
- **LC669 修剪二叉搜索树（Medium）**：利用有序性做剪枝而不是逐节点判断
  再重新拼接——值太小就整体丢弃当前节点和左子树、递归右子树；值太大就
  丢弃当前节点和右子树、递归左子树；只有落在区间内才保留当前节点并对
  左右子树分别递归。
- **LC1008 前序遍历构造二叉搜索树（Medium）**：不需要像普通二叉树那样
  额外配一份中序序列——BST 的"值小于根往左、大于根往右"这条定义本身就
  足够切分，用一个"栈顶到栈底值递减"的单调栈一次扫描完成构造，新值比
  栈顶小就是栈顶的左孩子，比栈顶大就要不断弹栈、最后弹出的那个是新值的
  父节点（新值是它的右孩子）。
- **LC1305 两棵二叉搜索树中的所有元素（Medium）**：分别对两棵树中序
  遍历得到两个有序数组，像 LC21 合并两个有序链表一样用双指针归并成一个
  有序结果，不需要把所有值收集起来再整体排序一次。
