# L10 · 回溯（Backtracking）

> 对应代码：`src/problems/p10_backtracking.py`。回溯题在新手眼里千差万别，但拆开看，
> 六道题的骨架代码长得几乎一模一样，区别只藏在两个地方："做选择"具体选什么、
> "剪枝"具体剪什么。把这两点想清楚,回溯就从"玄学"变成了体力活。

## 统一模板：做选择 → 递归 → 撤销选择

所有回溯代码都可以写成这个形状：

```python
def backtrack(状态):
    if 满足终止条件:
        res.append(记录当前解)
        return
    for 候选 in 所有当前合法的候选:
        做选择(把候选加入路径/标记为已用)
        backtrack(更新后的状态)
        撤销选择(把候选移出路径/取消标记)     # 关键：不撤销，状态就会"泄漏"给兄弟分支
```

三步为什么必须按这个顺序：
- **做选择**：把"这一步决定选谁"落到某个可变的容器（`path` 列表、`board` 网格、
  `used`/`cols` 等标记集合）上,这样递归下去的时候,更深层的调用能看到"目前为止
  已经选了什么"。
- **递归**：进入下一层,子问题的规模比当前小一点（少一个可选元素、少一格棋盘、少
  一个字符要匹配……）。
- **撤销选择**：递归返回后,把第一步做的修改精确地复原,这样"兄弟分支"（这一层
  for 循环的下一次迭代）看到的状态和"没做过这次选择"完全一样。**新手最容易漏掉
  的就是这一步**——漏掉撤销,后面的分支会莫名其妙地看到"不属于自己"的状态残留。

六道题的差异只在"候选是什么""怎么判断候选合法（剪枝）"：

| 题目 | 每一步的候选 | 剪枝/去重规则 |
|---|---|---|
| 子集 `subsets` | 选或不选当前元素 | 下一层从 `start+1` 开始，避免重复子集 |
| 全排列 `permute` | 任意一个"未使用"的元素 | 用 `used` 数组，禁止重复用同一个下标 |
| 组合总和 `combination_sum` | 任意一个候选数（可重复选自己） | 下一层从 `start` 开始（不是 `start+1`） |
| 括号生成 `generate_parenthesis` | 放 `(` 或 `)` | 剩余左右括号数决定能不能放 |
| 单词搜索 `exist` | 四个方向里字符匹配的格子 | 原地标记 `#`，四方向都失败才回退 |
| N 皇后 `solve_n_queens` | 当前行里某一列 | 列/两条对角线集合判重复 |

## 深挖 1：组合总和——为什么下一层的起始下标和子集/排列不一样

这是新手最容易在"子集、全排列、组合总和"这三题之间搞混的地方，因为它们的回溯代码
长得极像，唯一的差别就是递归调用那一行传的下标：

```python
# 子集/组合类（每个元素最多用一次）：下一层从 start+1 开始
backtrack(i + 1)

# 组合总和（同一个数字可以重复选无限次）：下一层从 start 本身开始
backtrack(i, remain - candidates[i])
```

想清楚**为什么**：子集要求"每个元素最多出现一次"，所以一旦在这一层选中了下标 `i`
的元素，下一层就不能再选下标 `i`（否则同一个数字进了同一个子集两次），只能往后
看 `i+1, i+2, ...`。而组合总和明确允许"同一个数字被选任意多次"（比如 `target=7`,
`candidates=[2,3,6,7]` 时，`2+2+3=7` 里 `2` 被用了两次），所以下一层递归必须允许
再次考虑当前这个下标 `i` 本身——如果误写成 `i+1`，程序会在"每个数字最多用一次"的
假设下运行,直接漏掉 `[2,2,3]` 这个解（这恰好是 LeetCode 40 号"组合总和 II"的写
法，两题极容易记混）。

同时要注意，**下标依然只能往后走、不能回头选更靠前的数**（比如选过下标 2 之后，
不能在下一层又跳回去选下标 0 或 1）。这一条保证了"选数的顺序永远和下标顺序一致"，
所以 `[2,2,3]` 只会被生成一次，不会因为选数顺序不同（比如先选 3 再选两次 2）又
生成一个"重复"的 `[3,2,2]`。`remain`（还差多少凑够 `target`）代替每次重新求和,
`remain<0` 时直接剪掉这条分支（不可能再凑出 `target` 了，继续选只会越选越负）,
`remain==0` 时收集当前 `path` 作为一个合法答案。

一句话总结这条规则：**"能不能重复选同一个元素"决定了递归传的是 `start` 还是
`start+1`；"要不要保持选数顺序防止重复组合"决定了下标必须单调不减（不能回头）。**

## 深挖 2：N 皇后——剪枝为什么能用集合做到 O(1)

暴力想法是：每放一个皇后之前，重新扫一遍棋盘上所有已放置的皇后，检查是否同行/
同列/同对角线冲突。这样每一步剪枝检查是 O(n)，n 步下来剪枝本身就要 O(n²)，还没
算搜索树本身的开销。

更好的做法：**按行放置皇后**（每行必须且只能放一个，天然满足"不同行"），并且
额外维护三个集合，一边放置一边更新：

```python
cols: set[int]    # 已经被占用的列下标
diag1: set[int]   # 已经被占用的"主对角线"标识：同一条 \ 对角线上，r-c 是同一个常数
diag2: set[int]   # 已经被占用的"副对角线"标识：同一条 / 对角线上，r+c 是同一个常数
```

为什么 `r-c` 能代表一整条对角线：沿着 `\` 方向（行号和列号同时 +1）移动时，
`r` 和 `c` 同时增加相同的量，所以 `r-c` 保持不变——这条对角线上所有格子的 `r-c`
都相等,天然是这条对角线的一个"身份证号"。同理，沿 `/` 方向移动时 `r` 增大、`c`
减小，`r+c` 保持不变，是另一条对角线的身份证号。

于是每次尝试在第 `row` 行放第 `c` 列时，合法性判断从"扫一遍棋盘"降到"三次集合
查询"：

```python
if c in cols or (row - c) in diag1 or (row + c) in diag2:
    continue          # 冲突，跳过这一列
```

放置后把 `c`、`row-c`、`row+c` 分别加入三个集合（O(1) 均摊）；这一行的这一种放法
的所有子树搜索完、要回退时，把三个值分别移除（**别忘记这一步**——忘记移除，后续
兄弟分支会被这一行早已撤销的选择错误地判定为"冲突"，导致搜出的解偏少）。

用一个 `queens` 列表记录"每一行皇后在哪一列"，等递归到第 `n` 行（说明 n 个皇后
都放完了）才把这个列表转换成棋盘字符串——**转换成字符串只在收集答案这一刻做一次**，
而不是每放一个皇后就重新拼一次棋盘，省掉大量无意义的字符串操作。

## 其余 4 题点睛

- **子集 `subsets`**：`res.append(path[:])` 要写在递归**入口**（每个节点都是答案），
  不是叶子——子集长度可以是 0~n 任意值，这一点和"必须凑满 n 个"的排列/必须凑够
  target 的组合总和都不一样。别忘了 `path[:]` 拷贝，直接存 `path` 引用会导致所有
  历史答案共享同一个列表，最后被后续的 pop 全部改空。
- **全排列 `permute`**：核心是每一层都要"从头扫到尾"而不是"从 start 开始"——因为
  排列关心顺序，`[1,2]` 和 `[2,1]` 是两个不同答案，靠下标去重（像子集那样）反而
  会把大部分排列漏掉。用 `used` 数组标记"当前路径里谁已经用过"。
- **括号生成 `generate_parenthesis`**：维护"剩余可放的左括号数 `left`、剩余可放的
  右括号数 `right`"（初始都是 `n`）。放 `(` 的条件是 `left>0`；放 `)` 的条件是
  `right>left`（即目前为止放的左括号比右括号多，放一个右括号才合法）。这一步剪枝
  让生成过程里的每一个中间字符串都自动合法，完全不需要生成完之后再挨个检查。
- **单词搜索 `exist`**：网格 DFS + 回溯的关键是"原地标记复原"——把当前格子临时改
  成 `#` 表示占用，四个方向递归完（不管成功还是失败）都要改回原字符，否则其他起点
  的搜索会发现这一格"凭空消失"。

### 深挖：解数独——为什么要维护"行/列/宫已用数字"的集合，而不是每次重新扫描棋盘

暴力想法是：每次尝试在某个空格填一个数字之前，重新扫一遍它所在的整行、整列、
整个 3x3 宫，检查这个数字有没有已经出现过。这样做当然能得出正确答案，但每一次
"合法性检查"都要花 O(9+9+9)=O(27) 的时间去扫描，而数独有大量空格、每个空格还要
尝试最多 9 个候选数字，暴力扫描的开销会被反复叠加很多次。

更好的做法和 Part I 的 N 皇后是同一个技巧：**用集合把"已经出现的状态"缓存下来，
一边填数一边增量维护，而不是每次都从棋盘上重新读取**。

```python
rows = [set() for _ in range(9)]   # rows[r]：第 r 行已经出现的数字
cols = [set() for _ in range(9)]   # cols[c]：第 c 列已经出现的数字
boxes = [set() for _ in range(9)]  # boxes[b]：第 b 个 3x3 宫已经出现的数字
```

预处理阶段只需要完整扫描棋盘一次，把所有已经填好的数字分别塞进对应的行集合、
列集合、宫集合。之后每次尝试在 `(r, c)` 填 `val`，合法性判断就退化成三次 O(1)
的集合查询：

```python
if val in rows[r] or val in cols[c] or val in boxes[b]:
    continue   # 冲突，换下一个候选数字
```

填入之后把 `val` 同时加进三个集合（O(1) 均摊）；如果这个分支最终没有走出一个
完整解、需要回溯，再把 `val` 从三个集合里精确移除——**这一步"移除"和 N 皇后回溯
时移除 `cols`/`diag1`/`diag2` 是完全一样的道理**：不移除，后续兄弟分支（同一个
空格尝试的下一个候选数字，或者递归返回后上一层空格的下一个候选）就会被这一步
"已经撤销"的选择错误地判定为冲突,导致该有的解被漏判。

数独和 N 皇后还有一处关键差异：**数独只需要找到一个解就够了，不需要像 N 皇后
那样收集所有解**。所以 `backtrack` 的返回值是 `bool`，找到完整解（`idx ==
len(empties)`）就一路返回 `True`；调用方（上一层递归）看到 `True` 也立刻
`return True`,不再尝试当前空格剩下的候选数字——这个"提前返回、不继续横向枚举"
的写法,是数独和"要收集所有解"的回溯题（比如 N 皇后 `solve_n_queens`、子集、
全排列）之间最容易被忽略的结构性区别：**如果误把 `if backtrack(idx + 1):
return True` 写成不检查返回值就继续 for 循环，即使已经找到了一个解,后面的
候选还是会继续尝试,把已经填对的棋盘改错**。

一句话总结：**用集合做增量维护而非重新扫描,是把"合法性检查"的常数因子从 O(27)
降到 O(3) 的关键；而"找到一个解就让 True 一路穿透返回",是数独区别于"收集所有
解"类回溯题的关键。**

### 其余 11 题点睛

- **子集 II / 全排列 II / 组合总和 II（`subsets_with_dup` / `permute_unique` /
  `combination_sum2`）三题共享同一个"排序 + 同层跳过重复"技巧**，但写法细节不能
  照抄照搬，这正是最容易混淆的地方：
  - 子集 II、组合总和 II 都是"下标只能往后走"的结构（子集下一层 `i+1`，组合总和
    II 每个下标只能用一次也是 `i+1`），同层去重条件是 `i > start and
    nums[i] == nums[i-1]`——`start` 是这一层 for 循环的起始下标。
  - 全排列 II 是"每层从头扫到尾选一个未用下标"的结构，**没有 `start` 这个概念**，
    同层去重条件要换成 `i > 0 and nums[i] == nums[i-1] and not used[i-1]`——多出
    来的 `not used[i-1]` 是因为排列里同一个值可能出现在路径的不同深度（纵向），
    必须和"同一层横向重复"区分开，否则会把合法的排列也剪掉。
  - 这三题和 Part I 对应的子集 / 全排列 / 组合总和的核心区别：Part I 的候选数组
    不含重复元素，天生不需要去重；一旦候选可能重复，就要先排序、再加上面这套
    "同层跳过"的判断，两者的回溯骨架（做选择 → 递归 → 撤销选择）完全不变。
- **组合总和 III `combination_sum3`**：候选固定是 1-9、天然无重复，不需要去重，
  这题的考点是"恰好选够 k 个数"——终止条件从"remain==0 就收集"变成
  "len(path)==k 且 remain==0 才收集"，两个条件缺一不可。
- **电话号码的字母组合 `letter_combinations`**：本类里最简单的回溯，没有任何
  去重或剪枝，纯粹是"每一位独立地做笛卡尔积展开"，别忘了 `digits==""` 要特判
  返回 `[]` 而不是 `[""]`。
- **复原 IP 地址 / 分割回文串（`restore_ip_addresses` / `partition_palindrome`）
  是同一种"在字符串上找切割点"的回溯**：每一层试着切下一段，只有这一段满足
  各自的合法性条件（前者是数值范围+无前导零，后者是回文）才允许继续切剩下的
  部分，字符串切完时才收集答案。
- **目标和 `find_target_sum_ways`**：和 N 皇后 II 一样是"只计数不收集具体方案"
  的回溯，每个数字只有 +/- 两种选择，没有任何剪枝（当前和超过 target 不代表
  不可能凑出解，因为后面还可以取负号），是一棵完整的满二叉递归树。
- **删除无效的括号 `remove_invalid_parentheses`**：这是本类里唯一一道换成 BFS
  写的题——因为要求"删除次数最少的所有解"，BFS 按删除次数分层天然满足这个要求，
  第一层出现合法解就停止扩展，不需要像回溯那样额外维护一个全局最小删除数。
- **找出不同的二进制字符串 `find_different_binary_string`**：回溯逐位构造，
  凑够 n 位后查是否在 `nums` 的哈希集合里；作为对比，更快的"对角线法"
  （`result[i]` 故意和 `nums[i][i]` 不同）只需要 O(n)，本题选择回溯写法是为了
  呼应"每一位二选一"这一贯穿本类的主题。
