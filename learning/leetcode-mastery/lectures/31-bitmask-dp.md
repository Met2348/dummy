# L31 · 位运算与状态压缩DP(Bitmask DP)

> 对应代码：`src/problems/p31_bitmask_dp.py`。Phase 3 竞赛级新分类。

## 当"子集"本身就是状态时

前面章节的 DP 状态大多是"一个位置""一个数量""一个区间端点"，本质上是一个数字
或几个数字的组合。但有一大类问题的状态天然是一个**集合**：

- "哪些城市已经被访问过"
- "哪些人已经被分配了任务 / 哪些技能已经被覆盖"
- "哪些数字已经被用过"
- "这一行哪些座位坐了人"

当这个集合的规模 n 不大（通常 n <= 20）时，可以把集合编码成一个整数的二进制
位——第 i 位是 1 表示"第 i 个元素在集合里"。这样一来：

- **判断元素 i 是否在集合里**：`mask & (1 << i)`
- **把元素 i 加入集合**：`mask | (1 << i)`
- **从集合里去掉元素 i**：`mask & ~(1 << i)`
- **枚举一个集合的所有子集**：`sub = mask; while sub: ...; sub = (sub-1) & mask`
  （这个子集枚举的总操作次数是 O(3^n)，来自组合数恒等式 `sum(C(n,k) * 2^k) = 3^n`）

dp 的下标也就从"一个数"变成"一个整数 + 一个位掩码"——`dp[i][mask]` 或
`dp[mask]`。这是状态空间随 n 指数增长的问题的标准应对方式：既然状态数是
2^n 级别，不如干脆用一个整数直接表示整个状态，配合数组或哈希表做记忆化。

## 深挖：LC847 访问所有节点的最短路径 —— 为什么状态必须是 (节点, 集合)

**朴素想法的失败之处**：这道题要在无向连通图上找一条最短路径，访问过所有
节点。如果只用"当前所在节点"做 BFS 的状态，会立刻发现这行不通——比如现在
站在节点 3，"从节点 3 出发的最短路径"这个信息本身依赖于"之前已经走过哪些
节点"：如果之前已经把节点 0、1、2 都访问过了，那么现在只需要再访问节点
4、5；但如果之前只访问过节点 0，现在还需要访问 1、2、4、5。**同一个"当前
节点"对应着完全不同的"接下来该怎么走"，因为"已经完成了多少目标"是两种情况
里完全不同的**——如果 BFS 状态里没有记录"已访问集合"，算法根本没法判断
"现在算不算已经达成目标"，也没法避免同一个节点被重复地、不知道意义地反复
访问。

**必须把"已访问集合"塞进状态**：既然"接下来该怎么走"依赖"已经走过哪些节点"，
这个信息就必须是状态的一部分，而不能是"隐藏在历史里、只影响过程但不出现在
状态定义里"的东西。所以正确的状态是 `(当前节点, 已访问节点集合的位掩码)`——
两个状态"当前节点相同、已访问集合不同"，被算法当成完全不同的两个状态来对待，
分别扩展、分别判断是否需要继续搜索。

```python
queue: deque[tuple[int, int, int]] = deque((1 << i, i, 0) for i in range(n))
visited = {(1 << i, i) for i in range(n)}
while queue:
    mask, node, dist = queue.popleft()
    if mask == full_mask:
        return dist
    for nxt in graph[node]:
        new_mask = mask | (1 << nxt)
        if (new_mask, nxt) not in visited:
            visited.add((new_mask, nxt))
            queue.append((new_mask, nxt, dist + 1))
```

**为什么用"多源 BFS"（同时从所有节点出发）**：题目允许"从任意节点出发"，
如果只从节点 0 开始搜索，会漏掉"从节点 5 出发反而更短"这种情况。与其对每个
起点各跑一次完整的 BFS（浪费很多重复计算），不如把所有 `(1<<i, i, 0)` 这些
"起始状态"一次性全部放进同一个队列——BFS 天然按"层"（也就是按距离）扩展，
第一次有某个状态的 mask 等于 `(1<<n)-1`（全集）时，对应的 `dist` 就是全局
最短答案，不需要区分它究竟是从哪个起点出发的。

**状态去重必须是 (mask, node) 二元组**：如果只用 mask 去重，会把"已访问
{0,1,2}，现在站在节点 0"和"已访问 {0,1,2}，现在站在节点 2"这两个本质不同
的状态（它们的"下一步能走到哪里"完全不同）错误地合并，导致遗漏某些本应
继续探索的分支。

## 其余 9 题点睛

- **LC464 我能赢吗**：博弈论问题，但"哪些数字已经被选走"是状态压缩 DP 的
  核心——`dfs(remaining, used_mask)` 记忆化搜索，"存在一种选法让对手在新
  状态下必败"就是当前状态必胜，这是"必胜态 = 存在走法让对手陷入必败态"的
  标准递归。
- **LC698 划分为 k 个相等的子集**：用 `dfs(mask, current_sum)` 表示"已用
  掉 mask 里的数字，当前桶已装 current_sum"，用取模操作把"装满这一桶就自动
  换下一桶"编码进同一个递归，不需要显式传"当前是第几个桶"。
- **LC1125 最小的必要团队**：集合覆盖问题的标准状压 DP——`dp[state]` 是
  覆盖 state 这个技能集合所需的最少人数，转移是 `dp[state|person_masks[i]]
  = dp[state] + 1`；用 `choice`/`prev_mask` 两个辅助数组回溯出具体名单，
  而不是重新猜测。
- **LC1349 参加考试的最大学生数**：行状态压缩——`mask & (mask<<1) == 0`
  判断"同一行内部无相邻冲突"，上一行的 mask 左移/右移一位再和这一行做按位
  与判断"无对角冲突"，把"同一行怎么坐"整体打包成一次状态转移。
- **LC1494 并行课程 II**：`available` 集合（先修课都满足、还没修的课程）
  枚举所有满足 `size <= k` 的子集作为"这一学期修的课"。**这里有一个非常容易
  踩的坑**：子集枚举如果不小心把"空集"也当成一种合法选择（"这一学期一门课
  都不修"），会转移到 `dp(taken_mask | 0) == dp(taken_mask)`——自己调用
  自己、状态完全没有变化，在记忆化结果写回之前就发生自环，直接导致无限递归。
  正确写法必须用 `while subset:`（`sub = (sub-1) & available` 迭代到 0 就
  停止，不处理 0 本身）排除空集。
- **LC1655 分配重复整数**：把"哪些客户已被满足"作为位掩码，对每个数字的
  频次分组，从大到小尝试用一个频次组同时满足"待满足客户集合"的某个子集，
  子集需求总量不超过这个频次才合法。
- **LC526 优美排列**：回溯法 + 状态压缩记忆化的结合——枚举每个位置放哪个
  还没用过的数字，只有整除关系成立才递归，虽然本质是枚举全排列，记忆化让
  同一个 (位置, 已用集合) 不会被重复展开。
- **LC1723 安排工作以达到最大收益的最小值**：`dp[i][mask]` 表示"前 i 个
  工人恰好分到 mask 这些工作时，最大工作时间的最小可能值"，转移是"取 max
  再取 min"——新的最大值是"之前已确定的最大值"和"新工人自己的工作量"两者
  的较大值，要在所有分配方式里选出这个较大值最小的方案。
- **LC1986 完成任务的最少工作时段数**：`dp[mask]` 表示完成 mask 这些任务
  所需的最少时段数，枚举 mask 的每个非空子集作为"最后一个时段"，转移到
  `dp[mask ^ sub] + 1`，是"枚举最后一步决策"这一 DP 范式在子集上的具体化。
