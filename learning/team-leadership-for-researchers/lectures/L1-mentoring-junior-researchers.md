# L1 · 带教入门:任务难度阶梯与新手向 Code Review (Mentoring Junior Researchers)

> 30-min lecture · 目标: 把"带一个新手"从凭直觉复刻自己被带教的方式(甚至复刻"完全没有方式"), 变成一套可执行的三件事——按难度阶梯分配任务、用新手接得住的方式做 code review、识别并避免自己变成团队瓶颈。
> 关键区分: **带教不是把自己做的事重新做一遍讲给对方听, 是设计一条对方能独立走完、并且下次能自己走的路径。** 你自己做得好, 不等于你能教得会——这是两种完全不同的能力, 前者是执行力, 后者是"把执行力拆解成对方接得住的台阶"的能力。

---

## 0. 为什么这是个真问题

大多数博士生第一次带本科生 intern 或师弟师妹, 完全没有受过任何带教训练——你唯一的参照系是"自己当年是怎么被带的"。这个参照系往往是坏样本: 如果你的导师是放养型 (`research-life` L2 提到的"hands-off"风格), 你可能会把"自己摸索"当成唯一正确的带教方式, 直接把新手扔进一个你自己都花了半年才搞懂的代码库, 附一句"你先看看"; 如果你自己曾经被微观管理到窒息, 你又可能反向过度补偿, 事事都要过一遍自己的手才放心。两条路都通向同一个结果: **新手要么在没有阶梯的陡坡前退出, 要么因为你不肯放手而永远学不会独立判断。**

```
                没有本讲的带教方式                       有本讲的带教方式
任务分配        凭感觉给, 要么太难要么太简单               按难度阶梯分级, 每级都有明确的"过关标准"
code review     指出问题就算完成, 不解释为什么              示范一次思考过程, 教会而不只是纠错
自己的角色      要么全程盯着(微管理), 要么完全撒手         主动设计"新手能独立完成"的边界, 逐步后撤
结果            新手要么挫败退出, 要么学不到真本事           带教者从"唯一能干活的人"变成"能复制自己的人"
```

带教这件事的特殊之处在于: 它不是"你会做研究"这项技能的自然延伸, 而是一项**独立的、需要单独练习的技能**——本讲连同 L2-L5, 就是把这项此前完全没人教过的技能拆开来练。

## 1. 任务难度阶梯 (Task Difficulty Ladder)

给新手的第一个任务, 决定了他接下来几周是建立信心还是彻底放弃。核心原则: **任务难度必须按阶梯上升, 每一级都要有明确、可核对的"过关标准", 而不是凭感觉给一个"差不多"的任务。**

```
Level 1  复现已有 baseline          给现成代码+运行说明, 只要求跑通并对齐已知数字
   │     过关标准: 复现出的指标和你给的参考数字在合理误差内一致
   ▼
Level 2  改动单一受控变量           改一个超参数/换一个数据子集, 预期结果范围你心里有数
   │     过关标准: 能解释"改了这个变量之后, 结果为什么变成这样"
   ▼
Level 3  独立设计一个受限消融        给定研究问题, 自己决定具体怎么拆解、消融哪个组件
   │     过关标准: 消融设计本身有没有漏掉明显应该对照的一组
   ▼
Level 4  独立提出并验证一个小 idea   全流程独立(呼应`critical-reading-gap` L4的idea生成), 你只做方向把关
         过关标准: 是否具备可证伪假设和最小验证实验(`critical-reading-gap` L4 的标准)
```

新手最常见的挫败, 是被直接空投到 Level 3-4, 却连 Level 1 的代码库结构都还没摸熟——这不是新手能力不够, 是**阶梯被跳过了**。反过来, 一个已经能独立设计消融的师弟, 你还在让他跑 Level 1 的复现任务, 同样是浪费——阶梯要跟着人走, 定期重新评估他现在站在哪一级, 而不是把第一次分配的难度当成永久标签。

> **自问清单(分配任务前)**: 这个任务对我来说是几级? 对他来说呢(不是同一个数字, 这正是带教要解决的落差)? 我给的"过关标准"是不是清楚到他自己就能判断"我做完了没有", 而不必每次都来问我?

## 2. 新手向 Code Review: 目标是教会, 不是挑错

同行之间的 code review (假设对方懂全部上下文, 目标是挑出问题、对齐规范) 和带教场景下对新手的 review 是两件事——**新手 review 的首要目标是"教会他下次自己发现这类问题", 挑错只是副产品。** 把同行 review 的习惯直接搬过来, 是新手带教最常见的失败模式之一。

具体做法:
- **一次不要列超过 3-5 条问题。** 新手认知负荷有限, 一次性甩出 20 条批注, 对方唯一学到的是"我很差", 而不是任何具体的东西。挑最关键的几条, 剩下的留到下一轮。
- **解释"为什么", 不只说"改哪里"。** "第 47 行的 seed 没固定" 是纠错; "第 47 行的 seed 没固定, 所以你三次跑出的结果对不上, 这是实验不可复现的典型根源" 才是教学——对方下次自己就能识别同类问题, 而不是只会改这一行。
- **示范一次你自己的思考过程。** 找一次机会, 完整讲一遍"我看到这段代码时脑子里在想什么、按什么顺序检查", 而不是只给结论。这比十次"这里不对"更有教学价值。

本仓库 `software-engineering-mastery` 专题的代码质量深水材料 (`src/tier2_deep/dp_se_code_quality_refactoring.py`) 里提到过两个关于 code review 有效性的实证发现: **评审规模效应**(一次性提交的改动量越大, 评审者发现问题的能力反而下降)和**评审速度效应**(评审速度过快基本等同于走过场)。这两条对新手带教场景的适用性甚至更强——一次性丢给新手一大堆改动意见, 或者你自己十分钟内扫一眼就打发, 都会让本该有的教学效果彻底落空。

## 3. 避免自己变成瓶颈: 三种失败的委派方式

带教做得不好, 最终代价往往不是新手没学会, 而是**你自己变成了团队里唯一能干活的人**——这是最容易被忽视的风险, 因为它看起来"我很负责"。

```
微观管理型                  撒手不管型                    单点故障型
──────────────            ──────────────              ──────────────
事事要过自己的手才放心      任务丢过去就不再过问          某个关键脚本/流程只有你会
新手永远学不会独立判断      新手卡住几天不敢来找你        你请假/忙别的项目, 团队就停摆
你自己没时间做研究          进度看起来在推进, 实则失控    这是最隐蔽的一种, 你自己往往
                                                        意识不到, 直到你不在场那一刻
```

前两种是两个极端, 大多数人对它们有一定警觉; **第三种(单点故障)最容易被忽视, 因为它伪装成"我很可靠"。** 一个可核查的自检方法: 拿 `experiment-ops-repro` L4 讲的仓库卫生清单 (repo hygiene checklist) 去考核你自己带的项目——一个新加入的成员, 能不能只靠这份清单和现有文档, 独立把环境跑起来、复现出你上次汇报的结果, 完全不用来问你? 如果答案是"不能, 得我亲自带一遍", 说明你把关键知识留在了自己脑子里, 而不是团队的共享资产里——**这正是单点故障的定义**, 也是"避免自己变成瓶颈"这件事最具体的检验标准。

## 4. 一个完整走查示例

场景: 一位新 intern 入组第一周, 你要给她安排第一个任务, 并做第一次 code review。

```
任务分配:  给定 Level 1 任务——"跑通 dpo-family 专题现成的 baseline 复现脚本,
           对齐 README 里记录的参考指标"。过关标准写清楚: 三个随机种子的均值
           落在参考值 ±0.5 以内即算通过, 不要求她理解每一行代码的原理。

第一次review: 她提交了改动, 你发现5处小问题, 只挑最关键的2条:
  ① "第23行 batch_size 从32改成了8没有说明原因——这类改动如果没写清楚理由,
     下次review的人(包括你自己三个月后)都不知道这是有意为之还是笔误。"
  ② "第60行漏了set_seed调用——这正是上次讲义提过的可复现性根源, 你现在
     跑三次结果对不上,原因就在这里。"
  其余3处留到下一轮, 附一句"这两条先改, 改完我们对齐一次思路再看下一批"。
```

这个例子里, 任务难度匹配了她此刻的水平 (Level 1), 过关标准清楚到她自己就能判断有没有做完, review 挑了最关键的两条并解释了原因, 而不是甩出全部 5 条问题让她无所适从。

## 5. 常见误区

- **"我当年就是这么被扔进去的, 挺过来了。"** 幸存者偏差——你只看到了挺过来的自己, 看不到因此退出的其他人。任务难度阶梯的意义, 正是不再依赖"能不能扛"这种筛选方式。
- **Review 只给结论不给原因。** 对方能改完这一处, 但下次遇到同类问题还是不会。
- **舍不得把关键脚本/流程交出去。** 表面上是"这个我熟, 我来快", 长期看是在给自己埋单点故障的雷。

## 6. 经典参考

- **Camille Fournier, 《The Manager's Path: A Guide for Tech Leaders Navigating Growth and Change》**(O'Reilly, 2017): 这是一本工程管理书籍, 面向的是软件团队里从个人贡献者成长为技术带教者/管理者的路径, 概念需要迁移到科研团队场景使用——书中关于"怎么带一个新人上手"和"怎么做建设性 code review" 的章节, 是本讲第1-2节框架的重要参考来源。

## 7. 本讲小结 + 通往 L2

- 带教是一项独立技能, 不是"你会做研究"的自然延伸; 复刻自己被带教的方式(尤其是坏样本)是最常见的第一个坑。
- 任务难度阶梯(复现→改动→消融→独立idea)让新手的挫败感和你的信心, 都建立在明确过关标准上, 而不是凭感觉。
- 新手向 code review 以"教会"为目标: 少而精、讲清"为什么"、必要时示范自己的思考过程。
- 避免自己变成瓶颈的三种失败委派方式里, "单点故障"最隐蔽——用新人能否只靠文档独立跑通作为自检标准。

> **下一讲 L2「向下管理与建设性反馈」**: 任务分配和 code review 解决的是"怎么让新手做对事"; 但新手做完之后, 你怎么告诉他做得怎么样——尤其是做得不够好的时候——是完全独立的一门功课。`research-life` L2 教你怎么向上管理导师, L2 要教的是方向完全相反的技能: 怎么向下给出具体、可执行、有时效性的反馈, 而不是一句"继续努力"就把人打发了。

**动手**: 找一个你现在(或未来可能)带教的对象, 用第1节的四级阶梯给他现在正在做的任务打一个等级, 再检查你上一次给他的反馈或 code review 是不是符合第2节"少而精、讲清为什么"的标准。
