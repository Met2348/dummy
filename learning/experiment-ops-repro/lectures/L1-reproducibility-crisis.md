# L1 · 可复现性危机 + 三个层级 + 不确定性的来源

> 30-min lecture · 目标: 理解 ML 的可复现性危机、区分三个常被混淆的层级 (repeatable / reproducible / replicable), 并列清「结果为什么每次不一样」的所有来源。

---

## 0. 一个尴尬但普遍的场景

你三个月前跑出一个漂亮结果, 写进了论文初稿。今天导师说「再跑一次给合作者看」。你回到代码, 跑出来的数**不一样了**。你慌了: 哪个是对的? 当时是怎么跑的? 用的哪版代码、哪个超参、哪个数据?

> **这不是你菜, 这是整个领域的系统性问题。** ML 有公认的「可复现性危机 (reproducibility crisis)」: 大量已发表结果无法被独立 (甚至被原作者自己) 重现。原因不是造假, 是**实验过程没有留下足够的痕迹**, 加上 ML 本身充满**不确定性来源**。

9.4 教你把实验**设计**对; 9.5 教你把实验**可靠地执行并留痕**, 让结果半年后还站得住、别人能重现。这是从「能跑出一个数」到「能负责任地报告一个数」的跨越。

---

## 1. 三个层级, 别混为一谈

社区 (尤其 ACM) 把「可复现」拆成三个精确的层级, 难度递增:

```
   Repeatable (可重复)          Reproducible (可复现)        Replicable (可复刻)
   ─────────────────           ──────────────────          ─────────────────
   同一个你, 同代码同数据,        别人, 用你的代码+数据,        别人, 用自己的实现,
   重跑得到同结果               重跑得到同结果               重新验证你的结论
        │                           │                           │
   考验: 确定性                 考验: 留痕完整                考验: 结论稳健
   (seed/环境固定)             (代码/数据/config 都给)       (不依赖你的实现细节)
```

逐层交代:
- **Repeatable (可重复)**: 最低要求。**你自己**, 同样的代码、数据、环境, 重跑能拿到**同一个数**。做不到这点 (比如没固定 seed), 说明你的实验有不受控的随机性, 一切无从谈起。
- **Reproducible (可复现)**: **别人**拿你公开的代码 + 数据 + config, 能跑出你的数。这要求你**留痕完整** (本专题 L2-L4 的核心)。
- **Replicable (可复刻)**: 最高。**别人用自己的独立实现**, 能验证你的**结论** (不一定是同一个数, 但同样的趋势/显著性)。这考验你的结论是否**真稳健**, 而非依赖你某个实现细节/bug。

> 给博0 的优先级: **先保证 Repeatable (对自己负责), 再做到 Reproducible (对社区负责)。** Replicable 是结论本身的稳健性, 靠 9.4 的多种子/公平对照来支撑。很多新手连第一层都没过 (重跑自己的实验数都对不上), 就急着投稿 —— 这是定时炸弹。

## 2. 不确定性从哪来: 「为什么每次不一样」清单

要做到 Repeatable, 先得知道随机性藏在哪。ML 实验的不确定性来源 (逐个排查):

| 来源 | 例 | 怎么控制 |
|---|---|---|
| **显式随机** | 权重初始化、数据 shuffle、dropout、采样 | 固定 `seed` (numpy/torch/random 都要设) |
| **数据顺序** | DataLoader 的 shuffle、多 worker 顺序 | 固定 seed + `worker_init_fn` |
| **GPU 非确定性** | cuDNN 卷积算法、atomic 加法的浮点不结合律 | `torch.use_deterministic_algorithms(True)` (有性能代价) |
| **并行/分布式** | all-reduce 的归约顺序、多卡时序 | 固定算法; 接受小幅波动 |
| **环境漂移** | 换了 CUDA / torch / numpy 版本 | 锁依赖 (lockfile), 记环境指纹 (L4) |
| **硬件** | 不同 GPU 型号的浮点行为 | 记录硬件; 跨硬件只求结论一致 |

> 一个重要的现实: **GPU 上 100% 比特级确定性很难且很贵** (开 deterministic 算法常慢 2-10 倍)。所以实践中的目标分两层: ① CPU 端逻辑 (数据划分、评测) 必须完全确定; ② GPU 训练接受**小幅波动**, 但用**多种子 + 报方差** (9.4-L5) 来让结论不依赖单次运行。**可复现不等于消灭所有随机, 而是让随机被量化、被控制。**

---

## 3. 「seed everything」: 第一道、也是最容易漏的关

最基础的动作, 却最常漏。一个完整的 seed 设置 (PyTorch 例):

```python
def seed_everything(seed: int):
    import random, os
    import numpy as np
    random.seed(seed)
    np.random.seed(seed)
    os.environ["PYTHONHASHSEED"] = str(seed)   # ← 注意这个!
    try:
        import torch
        torch.manual_seed(seed)
        torch.cuda.manual_seed_all(seed)
    except ImportError:
        pass
```

特别注意 **`PYTHONHASHSEED`** 那行 —— 它正是 9.4 我们踩过的坑:

> Python 对字符串的内置 `hash()` 默认带**进程级随机盐**, 所以用 `hash("DPO")` 派生种子, 会让你**每次重跑得到不同结果** (连 Repeatable 都过不了)。9.4 的 `experiment.py` 当时就中招, 我们改成了手工确定性编码。教训: **任何进入随机数种子的东西, 都必须自己是确定性的。** 这是个真实、隐蔽、教科书级的可复现 bug。

---

## 4. 本讲小结 + 通往 L2

- ML 有系统性**可复现危机**; 原因是留痕不足 + 固有不确定性, 不是造假。
- 三层级: **Repeatable** (自己重跑一致, 考验确定性) → **Reproducible** (别人用你代码一致, 考验留痕) → **Replicable** (别人独立实现验证结论, 考验稳健)。
- 不确定性来源: 显式随机 / 数据顺序 / GPU 非确定 / 并行 / 环境 / 硬件; 分两层控 (CPU 全确定, GPU 容波动 + 多种子报方差)。
- **seed everything** 是第一关, 别漏 `PYTHONHASHSEED` (9.4 踩过的坑)。

> **下一讲 L2「配置管理」**: 要做到 Reproducible, 第一步是把「这次跑了什么」完整、显式地记下来。L2 教你 **config as code**: 一个 run = 一份 config + 一个 seed, 杜绝硬编码超参, 让参数扫描和复现都变成「换 config」。

**动手**: 去 `N1-config-and-tracking.ipynb` 之前, 先在你任意一个已有复现专题里找一处**硬编码的超参** (如代码里直接写 `lr=1e-4`), 体会一下「半年后我怎么知道当时用的是多少」—— 这就是 L2 要解决的。
