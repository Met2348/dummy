# 从 Agent 循环到结构化图：一个调度理论视角的 LLM Agent 执行框架

> **本篇属 agent-harness 库 B 组（控制循环 / L 层）**。它和标杆 Harness-Bench(2605.27922) 是一对互补物：
> Harness-Bench 用实验证明"换 harness 分数摆 23.8 分"，本文则从**理论**上回答"harness 的循环层到底有哪些
> 可设计的自由度、每个自由度换来什么、失去什么"。阅读时请把它读成一份**控制循环的设计规格书**——
> 它不给数字（自陈无实验），但给公式、给状态机、给两条带证明的定理，把"朴素 while 循环"和"图执行引擎"
> 放到同一根调度坐标轴上。

## §1　TL;DR（一页讲清这篇在干嘛）

> 主讲提示：开场先把"Agent Loop 其实是一台退化的调度器"这句话钉在黑板上，全篇都在展开它。

一句话：**当今 agent 的主流范式是 Agent Loop——一个 LLM 读着越滚越长的上下文窗口、每步决定下一步做什么的迭代循环**（Abstract）。本文指出它有三处结构性弱点（§1）：(1) **步骤间依赖是隐式且不可验证的**——"先改代码再跑测试"这个依赖只存在于上下文里，没有结构护栏防止乱序；(2) **失败恢复没有有界语义**——LLM 自主决定重试/跳过/重规划，没有上界，可能无限重试；(3) **执行计划会被静默改写**——LLM 中途改计划，原计划被覆盖，事后无法重建"是哪版计划管了哪步"的审计链。

作者的核心动作分两步：
- **先统一**：把执行系统形式化为一个五元组 $\mathcal{E}=(\mathcal{S},\mathcal{U},\mathcal{P},\mathcal{O},\Delta)$（Def 3.1），证明 **Agent Loop 本质是一台"单就绪单元调度器"**——任一时刻至多一个可执行单元就绪（$|\mathcal{U}|\le 1$），而"下一步选谁"是不透明 LLM 推理的输出、不是可检查的策略（§3.2）。这一步把 Agent Loop 和图执行引擎放到**同一根语义连续轴**上，可精确比较。
- **再抬升**：提出 **Graph Harness（结构化图 harness，下文简称 GH/SGH）**——把控制结构从隐式上下文**抬升为一张显式的静态有向无环图（DAG）**，做三个设计承诺（§1、§4）：① 执行计划在一个"计划版本"生命周期内**不可变**；② 规划/执行/恢复**三层分离**、各有清晰接口；③ 恢复遵循**严格逐级升级协议**（retry → patch → replan），杜绝无界重规划。

- **属于 harness 的哪一层（Θ1）**：本篇打的是 **L 层（Loop / 控制循环）**——它的全部机械都在"下一步派发谁、失败了升级到第几级、计划能不能改"这套循环逻辑上；对 **C 层（上下文）** 有强依赖（靠"执行上下文 vs 诊断上下文"的隔离，Def 5.3）、对 **V 层（验证）** 有接口（靠输出契约 $\kappa$ 把关 running→executed，§6.1）。
- **回扣全库论点（Θ2）**：本文是 `Agent = Model + Harness` 的**理论侧支撑**——它把"harness 的循环层"从一坨 `while True:` 提炼成**一个有 5 个可调自由度的调度器**（就绪集基数、策略显式度、策略确定性、计划可变性、恢复协议），从而解释了"为什么换 harness 会换出能力差"：因为这些自由度的取值本就决定了可并行性、可恢复性、可审计性。
- **够新但要诚实（Θ4/Θ5）**：2026-04 预印本，相对 ReAct(2023) 朴素循环推进了"把控制流显式化 + 给出终止性/可靠性证明 + 把恢复做成协议"；但作者**反复自陈这是 position paper，无实现、无实验**（Abstract 末、§8 Note、§9.5 Limitation 1、§11 末），所有性能主张（$G_{graph}>0$ 等）都是"**理论预测**"而非已验证结论。这条诚实边界必须贯穿全文。

## §2　问题与动机：为什么"把循环改成图"值得做

> 主讲提示：这一页用 Why 三连的"问题层"，讲清 Agent Loop 的三处硬伤，且每条都引原文。

**Why（问题层）——不解决会卡住什么？**
LLM 已经催生了一类新软件——**LLM agent**：自主拆解任务、调用工具、迭代求解（§1 开篇）。主流做法是 Agent Loop，简单且被广泛采用，但随任务变大，三处弱点越发明显（§1，逐条引原文）：

1. **依赖隐式且不可验证**："当 agent 必须『改代码，然后跑测试』时，第二步依赖第一步这个事实**只存在于上下文窗口里**；LLM 必须在推理时记住这个依赖，没有任何结构护栏阻止乱序执行"（§1 原文转述）。
2. **失败恢复无有界语义**："失败时，LLM 自主决定重试/跳过/重规划——没有明确契约规定哪种失败类型可用哪种恢复，也**没有尝试次数上界**"（§1）。
3. **执行计划可被静默改写**："若 LLM 中途改计划，原计划在上下文里被覆盖；执行后再也无法重建一条忠实的审计轨迹，说清是哪些动作被哪版计划管辖"（§1）。

作者强调这**不是边缘个案**：对 70 个开源 LLM agent 项目的分析显示 **60%（42/70）采用 Agent Loop**（§1、Table 7、附录 A.5）；近期增强（planner 增强循环、图编排、多智能体分解）改善了局部，但**没有根治**结构问题——控制流仍是隐式的，执行缺乏稳定承诺（§1）。

> **读出什么**：本文的动机不是"再造一个更强 agent"，而是"给 agent 的**控制循环**补一套调度理论的语言和护栏"。这正是 B 组（控制循环）作为方法论层的意义——它管的不是"模型多聪明"，而是"这台循环让模型的聪明能不能被有序、有界、可审计地兑现"。

## §3　核心贡献（论文 §1 末，四条）

1. **调度器统一框架**：把 LLM agent 执行系统刻画为**由就绪集基数 $|\mathcal{U}|$ 与策略显式度参数化的调度器**，从而能精确比较 Agent Loop、图执行器与中间形态，并生成可检验的性能假设（$G_{plan}>0$、$G_{scaffold}>0$ 等）。
2. **四原则设计方法论**：由 70 系统调查提炼，显式刻画可控性/表达力/可实现性三者的权衡；每条原则都用调查里的定性观察佐证（例如"失败循环"行为在图/流编排系统里常见、在状态机系统里罕见）。
3. **形式化执行模型**：规划/执行/恢复**三层分离** + 不可变计划版本方案 + **带终止性与可靠性证明的节点状态机**（在显式公平性假设下）。
4. **可归因实验协议**：七组设计，逐组只加一个特征，从而把每个设计决策的贡献单独分离（$G_{plan},G_{scaffold},G_{graph},G_{patch},G_{replan}$）。**该协议只是方法，执行留待未来工作**（§1、§8）。

> **诚实标注（Θ5 前置）**：贡献 1/4 的所有"增益 > 0"都是**逻辑推论 / 待验证假设**，不是实测（§3.4 "Important caveat"、§8 Note）。全文凡见 $G_{\cdot}$ 一律按"预测"读。

## §4　统一框架：把执行系统写成一个五元组

> 主讲提示：这是全篇的地基公式。先给直觉（"一台调度器"），再逐符号定义，最后说"读出什么"。

**直觉**：把任何 agent 执行系统都想成一台**调度器**——它盯着一堆"可执行单元"（工具调用/子任务/推理步），每一拍从"当前就绪、依赖已满足"的单元里挑一些派发出去，拿到结果后更新状态、再算下一拍谁就绪。Agent Loop 和图引擎的差别，不在"是不是调度器"，而在**每拍能挑几个**、**挑的规则透不透明**。

**定义（Def 3.1，执行系统五元组）**——先定义每个符号再用：
$$\mathcal{E}=(\mathcal{S},\mathcal{U},\mathcal{P},\mathcal{O},\Delta)$$
- $\mathcal{S}=\{(v,s_v)\mid v\in V, s_v\in\Sigma\}$：**节点状态集**。$V$ 是节点（可执行单元）集合，$\Sigma$ 是节点状态集（见 Def 6.1）。
- $\mathcal{U}:\mathcal{S}\to 2^{V}$：**就绪集函数**——把全局状态映射到"依赖已满足、可派发"的节点集：
  $$\mathcal{U}(\mathcal{S})=\{v\in V\mid s_v=\texttt{ready}\ \wedge\ \forall(u,v)\in E:\ s_u=\texttt{executed}\}$$
  （$E$ 是表依赖的边集；一个节点当且仅当所有前驱都已 executed 才 ready。）
- $\mathcal{P}:2^{V}\to V$：**调度策略**——从就绪集里选出下一个要派发的节点的**确定性函数**；在非确定系统里 $\mathcal{P}$ 退化为**关系**而非函数。
- $\mathcal{O}=\{\text{success},\text{failure},\text{retry},\text{escalate}\}$：节点执行的**结果空间**。
- $\Delta:\mathcal{S}\times V\times\mathcal{O}\to\mathcal{S}$：**状态转移函数**——按结果更新该节点状态，并为新全局状态重算 $\mathcal{U}$。

> **读出什么**：这套记号本身**不新**（作者明说建立在经典 DAG 调度文献 Topcuoglu 2002 / Cormen 2009 之上，§1"Relationship to classical scheduling theory"）；**新在把它用到 LLM agent**，并识别出经典调度器不面对的 **LLM 特异挑战**（非确定节点输出、语义验证、推理型错误）。用**关系而非函数**来刻画 $\mathcal{P}$ 是关键设计——它让同一个框架能同时装下确定性调度器（下一步唯一确定）和非确定调度器（同一状态可能选不同），而 Agent Loop 恰恰属于后者。

### §4.1　两个判别定义 + 一个"实锤"观察

**Def 3.2（单就绪单元调度器）**：若在每个可达状态 $s$ 都有 $|\mathcal{U}(s)|\le 1$，则 $\mathcal{E}$ 是单就绪单元调度器。此时 $\mathcal{P}$ 变得**空洞**——至多一个单元可选，不存在有意义的调度决策，"选谁"实际由"产生下一个 $\mathcal{U}$ 元素的过程"决定；在 Agent Loop 里，那个过程就是一次 LLM 推理（§3.2）。

**Def 3.3（多就绪单元调度器）**：若存在某可达状态 $s$ 使 $|\mathcal{U}(s)|>1$，则为多就绪单元调度器。此时调度关系成为**真正的设计决策**（拓扑序、优先级、并行派发会产生不同轨迹）。这是图执行器所处的区间。

**核心观察**：**Agent Loop 是一台非确定的单就绪单元调度器**（§3.2）。理由直接来自其设计——每步 LLM 恰好产出一个动作 $a$（故 $|\mathcal{U}|=1$），但同一上下文状态在不同调用下可能产出不同动作（故 $\mathcal{P}$ 非函数）。

> **Why（设计层）——"并行工具调用"能不能救 Agent Loop？** 有人会说 OpenAI 的 `tool_calls` 一次能装多个函数并发执行，那 $|\mathcal{U}|$ 不就 >1 了？作者专门反驳（§3.2 "Parallel tool calls"）：这对应 $|\mathcal{U}|>1$ 但 $\mathcal{P}$ **仍非确定**——哪几个操作并行是 **LLM 决定的**，不是显式调度策略。所以 Agent Loop 的短板"**不只是没有并行，而是没有结构性并行保证**"：这一轮 LLM 可能并发调两个工具、下一轮又串行，全看它内部推理。Graph Harness 的优势不是"能并行"，而是"**显式并行**"——就绪集由 DAG 拓扑算出，只要依赖允许就保证并行派发（§3.2 末）。这一层区分是本文最锋利的地方，务必讲透。

## §5　调度器连续谱：从"朴素循环"到"Graph Harness"的一根轴

> 主讲提示：这是本篇的"一图流"，把五种系统排在一条轴上。这也是 Why 三连"设计层"最集中的地方。

**框架观察（§3.4，调度器连续谱）**：LLM agent 的执行系统可沿三根轴排布——
1. **就绪集基数**：$|\mathcal{U}|=1$（单）↔ $|\mathcal{U}|\ge 1$（多）；
2. **策略显式度**：隐式（LLM 靠上下文推理）↔ 提示级（结构化提示约束输出）↔ 状态机级（对形式状态的显式策略）；
3. **策略确定性**：非确定（$\mathcal{P}$ 是关系）↔ 确定（$\mathcal{P}$ 是函数）。

**Figure 2（连续谱）** 把五类系统排成一排（沿"策略显式度与确定性"递增）：

| 位置 | 系统 | $\|\mathcal{U}\|$ | 策略 |
|---|---|---|---|
| ① Sequential | **Naive Loop**（朴素循环） | $=1$ | 非确定 |
| ② Parallel calls | **Parallel Loop**（带并行工具调用的循环，如 `tool_calls`） | $\ge 1$ | 非确定 |
| ③ Planner | **Planner Loop**（加了 planner 但执行仍单步） | $=1$ | 非确定 |
| ④ Scaffold | **Structured Loop**（脚手架约束计划遵从） | $=1$ | 半确定 |
| ⑤ Graph | **Graph Harness** | $\ge 1$ | **确定** |

> **Why 三连（把"朴素 while 循环 → 显式执行图"讲成设计必然，Θ 硬性要求）**：
> - **问题层**：朴素做法就是 `while not done: action = llm(context); observe(action)`（②③④都仍是这台循环的变体）。它会因为三件事卡住——**并行是 LLM 拍脑袋**（$|\mathcal{U}|\le 1$ 或非确定），**依赖靠上下文记忆**（长窗口下易"忘记 analyze 需要两个文件都读完"，§3.6.1 Turn 5 注），**重试无界**（Turn 7→8 "换个 fix"是一次没有协议的 ad-hoc LLM 决定，§3.6.1）。
> - **设计层**：一个关键陷阱是——"**给循环加个 planner** 就够了吗？" 作者明确回答（§3.4 末）：**加 planner 只让系统沿"策略显式度"轴右移，不改变就绪集基数**——Planner Loop 仍是 $|\mathcal{U}|=1$（③）。**只有图执行器能到达多就绪单元区间。** 这就说明了为什么"显式执行图"不是"循环+更好提示"能替代的：并行性、依赖追踪、有界恢复在图里是**结构属性**（DAG 拓扑天然给出），而在循环里永远是 **LLM 决策**（§3.6.2 末"Structural differences"、§3.7.1"deterministic progress"）。
> - **结果层**：作者用一个 bug-fix 任务把两条路并排跑（§3.6）。**Agent Loop 用 11 个串行 turn**（Turn 1–2 两个搜索串行、Turn 7 fix 失败后 Turn 8 ad-hoc 换 fix、Turn 8–10 docs 与 fix/test 串行，§3.6.1）；**Graph Harness 用 10 次派发 / 6 个调度轮**，其中 4 轮并行（搜索 $|\mathcal{U}|=2$、读 $|\mathcal{U}|=2$、fix/docs $|\mathcal{U}|=3$），且 `fix_A` 瞬时失败触发 Level-1 恢复、但因 `any_of` join 已被 `fix_B` 满足而**跳过重试不浪费预算**（§3.6.2）。机制上，墙钟时间随可并行度成比例下降（§3.6 末"Step count"）。**注意作者对这个例子的诚实**：Figure 1 脚注明说这是"理想条件"（planner 正确识别全部并行、依赖在规划期全知、每节点正确执行），真实任务未必这么干净——见 §9.6 讲 planner 失败。

## §6　四条设计原则：每条都是"牺牲什么换什么"

> 主讲提示：这一页是本文的"价值观"。每条原则先给一句箴言，再讲牺牲/收获，最后落到 Table 8。

作者由 70 系统调查（Table 7）提炼：任何 agent 执行系统都在三个目标间权衡——**可控性**（可预测、可验证、可审计）、**表达力**（能表示的任务结构范围）、**可实现性**（工程复杂度与风险）。Graph Harness **刻意不站在表达力前沿**，而是"最大化可控性、只保留足够表达力去覆盖『可验证工程任务』——那些依赖结构在执行前就能articulate清楚的任务"（§4.1）。四原则（§4.2–4.5）：

| # | 原则（箴言） | 牺牲 | 收获 |
|---|---|---|---|
| P1 | **可控性优先**：收益不确定时，优先执行过程的可预测/可验证 | 竞争性并行(`first_of`)、递归子图展开、父链回滚 | 可预测执行、可验证轨迹 |
| P2 | **稳定执行承诺**：计划一旦生成并验证，执行期不得改结构 | 动态改计划 | 可审计计划史、可靠归因 |
| P3 | **有界恢复**：恢复动作须有明确触发、有界范围、严格升级协议 | ad-hoc LLM 驱动恢复 | 确定性升级、杜绝失败循环 |
| P4 | **副作用分级**：可执行单元按副作用画像分类，策略须尊重该分类 | 无约束并行派发 | 安全保证、风险有界调度 |

**每条原则的"设计层 Why"（作者都从调查取证，且都诚实标注了主观性）**：
- **P1**：调查观察到"失败循环"（无限/无界恢复环）**在图/流编排系统里常见（4 个里 3 个），在状态机系统里罕见（7 个里 0 个）**（§4.2）。作者明说**没有正式量化**、分类主观、不同评审可能不同结论——但跨多次代码/issue 检视，该模式一致。→ 灵活无结构的控制流虽表达力强，但更易陷入恢复病态；工程任务里"失控执行（静默数据损坏、不可复现失败、不可审计决策）的代价往往超过表达更复杂结构的收益"。
- **P2**：调查里**带版本历史的系统（如 LangGraph 的线程作用域状态）比可变历史系统更好调试**（§4.3）——计划被版本化而非就地改写时，开发者能更可靠地重建执行史。作者同样标注这基于人工 issue 检视、"版本化 vs 可变"本身是连续谱。→ **不可变的计划版本是问责的最小单位**。
- **P3**：调查发现 Agent Loop 实现**普遍缺乏任何恢复上界**（§4.4），表现为 (a) 无限重试（LLM 死磕失败路径）或 (b) 过早放弃（瞬时错触发不必要重规划）。→ 有界恢复把失败处理**从一次 ad-hoc LLM 决定变成一套带可审计升级规则的协议**。
- **P4**：并非所有动作同等不可逆——只读 API 可随意重试，数据库写不可（§4.5）。→ 没有副作用分级，调度器就有"投机式并行执行危险操作"的风险，且该风险随就绪集基数增大而增大。

> **读出什么**：这四条不是拍脑袋，是**从调查里的失败模式反推出的护栏**；而且每条都老老实实写了"我牺牲了灵活性"。这种"牺牲—收获"对账（Table 8）本身就是可直接抄的设计文档模板（见 Inspires-Us b）。

## §7　形式化执行模型：不可变计划 + 三层分离 + 上下文隔离

> 主讲提示：这一页讲清"计划是一份不可变合同""规划/执行/恢复各管一段""执行上下文和诊断上下文互不见面"。

**Def 5.1（执行计划）**：执行计划是一个元组
$$\Pi=(id,\ version,\ V,\ E,\ \sigma,\ \kappa)$$
$V$ 是节点集，$E\subseteq V\times V$ 是有向边集，$\sigma:V\to\text{NodeConfig}$ 给每个节点配置（动作、重试策略、副作用等级），$\kappa$ 是**输出契约**（规定计划完成时必须产出什么）。

**Def 5.2（计划不变式）**：在计划版本 $v$ 的整个生命周期内，结构 $(V,E)$ **不可变**；产生不同结构的**唯一机制**是经 replan 协议创建新版本 $v+1$。这条不变式形式化了 P2，带来三个理论后果（§5.1）：① **可归因性**——每条轨迹能唯一系到某 $(id,version)$；② **可预测性**——任意状态 $s$ 的就绪集 $\mathcal{U}(s)$ 完全由固定拓扑 $(V,E)$ 与当前节点状态决定，执行期不会冒出结构惊喜；③ **可验证性**——计划可在执行前校验（查环、查不可达、查不可满足依赖），不必担心结构中途变。

**为什么偏偏选静态 DAG（§5.2，Why 设计层）**：作者考虑并否决了三个更有表达力的替代（Table 9：递归展开图 / 父链回滚图 / 动态拓扑图，三者 Predictability 与 Verifiability 都 Low）。静态 DAG 由构造保证三性质——**无环依赖**（保证就绪集总能推进，至少存在一个全前驱满足的节点）、**有限执行**（节点数固定，执行步数有上界）、**拓扑调度**（拓扑序给出可被策略细化但绝不违背的规范调度）。

**三层分离（§5.3，Figure 3）**：
- **Planner Layer**：收任务意图 → 产出满足计划不变式的、经校验的静态 DAG $\Pi$。planner 可以是 LLM、模板生成器或二者混合。
- **Runtime Layer**：拿 $\Pi$ 执行、**绝不改 $(V,E)$**；维护每节点状态、算就绪集、施加 $\mathcal{P}$、记录观察；**只上报失败、不决定怎么处理**。
- **Recovery Layer**：收失败报告 → 诊断根因 → 选恢复动作。**它被建模为独立于执行循环**——其决策基于一个对 runtime 执行上下文**不可见**的诊断上下文。

**Def 5.3（上下文隔离）**——本文对 **C 层**的关键接口：系统维护两个**不相交**上下文——
- **执行上下文 $\mathcal{C}_{exec}$**：输入、可见产物、运行时状态、预算；执行期节点可访问。
- **诊断上下文 $\mathcal{C}_{diag}$**：失败史、planner 注记、历史计划版本；**只有 recovery/planner 层可访问**。

约束是 $\mathcal{C}_{exec}\cap\mathcal{C}_{diag}=\varnothing$（节点执行期间）；诊断信息只能**经 planner（作为新计划版本的一部分）**流入 $\mathcal{C}_{exec}$。

> **Why（设计层）——为什么要把失败史挡在执行上下文外？** 朴素做法是把"上次失败了、报错是 XXX"直接塞进下一步的提示。→ 这会造成一个隐蔽病态：**用失败史当后续执行步的隐式输入，把恢复决策耦合进执行逻辑**，同时毁掉可预测性与可审计性（§5.4）。隔离两个上下文，就把"执行"和"复盘"彻底解耦——这也是本文对我们最有迁移价值的一招之一（见 Inspires-Us a）。

## §8　节点状态机与两条定理：终止性 + 条件可靠性

> 主讲提示：这是全篇理论密度最高的一页。先看状态机图（直觉），再逐条读定理"它保证了什么、它的前提是什么"。

**Def 6.1（节点状态）**：每个节点 $v$ 的状态取自
$$\Sigma=\{\texttt{pending},\texttt{ready},\texttt{running},\texttt{waiting\_human},\texttt{blocked},\texttt{executed},\texttt{failed\_retryable},\texttt{failed},\texttt{cancelled},\texttt{skipped}\}$$
终止状态 $\Sigma_{term}=\{\texttt{executed},\texttt{failed},\texttt{cancelled},\texttt{skipped}\}$。**Figure 4** 给出完整转移；关键两条：`running→failed_retryable` 发生在**瞬时错**（网络超时、限流，重试可能成功）；`running→failed` 发生在**结构错**（缺依赖、非法计划，重试同一节点无救）。完整转移表见附录 Table 15。

**Def 6.2（有界执行）**：每节点有超时 $\tau_v\in\mathbb{R}^+$ 与重试预算 $b_v\in\mathbb{N}$；任何 `waiting_human` 节点另有有限超时 $T_{human}$。`running→failed` 当且仅当：(1) 已耗时 $>\tau_v$，或 (2) 重试次数 $>b_v$。→ 保证没有节点永远停在 running。

**终止状态吸收性**：终止状态无任何出边（Table 15），一旦进入永不离开——这是就绪集计算正确性的基础（一旦某前驱 executed，调度器可**永久**依赖这个事实）。

**Proposition 6.1（进度保证）**：在有界执行的有限 DAG 上，若至少一个节点处于非终止态，则要么有节点能转入 `ready`、要么有节点能转入某终止态。**证明思路**：有限 DAG 里总存在一个"所有前驱都已终止"的节点（由无环性 + 有限性）；再对该节点按 join 类型分情况推进（all_of 全 executed→ready；any_of 至少一个 executed→ready，其余 pending 前驱 skip）。blocked 节点因无环不会死锁成环，其阻塞依赖终会解除 → pending。

**Theorem 6.2（有界终止）**：给定合法 DAG（有限、无环、所有节点从入口可达）在有界执行下（有限 $\tau_v,b_v$），**Graph Harness 主循环以概率 1 终止**：每个节点在有界时间内到达终止态。**证明思路**：每节点至多 $\tau_v$ 停在 running、至多尝试 $b_v+1$ 次；由 Prop 6.1，主循环每一拍至少发生一次状态转移；由终止吸收性，终止态不回退；$|V|$ 有限、每节点非终止状态数有限，故转移总数有上界 $|V|\cdot|\Sigma\setminus\Sigma_{term}|$。总时间上界
$$\sum_{v\in V}\tau_v\cdot(b_v+1)\ +\ \sum_{v:\,s_v=\texttt{waiting\_human}}T_{human}(v).$$

> **读出什么**：这条定理是本文相对 ReAct 朴素循环最硬的增量——**它把"循环会不会停"从经验祈祷变成了带上界的定理**。但代价写在前提里：它需要**每节点有限超时与预算**、且 `waiting_human` 也有超时（否则等人可以等到天荒地老）。这正是"有界恢复"(P3) 的形式回报。

**契约验证语义**：转移 `running→executed` 被**输出契约把关**——节点只有满足输出契约 $\kappa_v$ 才能进 executed，否则转 `failed_retryable`/`failed`。故"所有节点 executed"**由构造**蕴含"所有节点输出契约通过"$\bigwedge_v\sigma(v)=\texttt{executed}\Rightarrow\bigwedge_v\kappa_v(o_v)$。

**Theorem 6.3（验证可靠性下的条件可靠性 / "验证间隙"）**：设 $p_v\in(0,1]$ 为节点 $v$ 的契约验证**正确判定通过输出**的概率（假阳率 $1-p_v$）。若所有节点达 executed 且各验证独立，则**所有节点输出真正正确的概率至少**
$$\Pr[\text{all outputs correct}]\ \ge\ \prod_{v\in V}p_v.$$

> **读出什么（这是本文最诚实、也最该被记住的一条）**：这式子把 **"验证间隙"（validation gap）** 讲透了——**即便所有节点都"通过了契约验证"，系统正确性仍被验证方法本身的可靠性所上界**。语法验证（字段存在、类型、格式，由确定性代码执行）$p_v\approx 1$，界几乎白给；**语义验证（"这个修复对不对？"，尤其用 LLM 当裁判）$p_v$ 可能显著 < 1**（§6.1）。作者明说 Graph Harness **不能消除**这个间隙，只能让它**可观测**（审计轨迹记录用了哪种验证）和**可控**（设计者按节点选验证方法），并靠三招缓解：P4 要求高影响节点尽量用代码验证、`waiting_human` 给关键步留人审、恢复协议在下游依赖边界兜住语义错（下游节点若输入语义错会失败）。**这条直接呼应我们自己的收口隐忧"谁来验证验证者"**（见 Inspires-Us d）。

## §9　三级恢复协议：把"重试还是重规划"变成一部电梯

> 主讲提示：这一页讲清"恢复只能一级一级往上升，不许跳级"，且这条不变式是机械可强制的，不是口头规范。

**Def 6.3（恢复动作，三级）**：
$$\mathcal{R}=\underbrace{\{\texttt{local\_retry}\}}_{\text{Level 1}}\ \cup\ \underbrace{\{\texttt{local\_patch}\}}_{\text{Level 2}}\ \cup\ \underbrace{\{\texttt{request\_replan}\}}_{\text{Level 3}}$$

**Table 10（触发与范围）**：

| 级别 | 触发 | 范围 |
|---|---|---|
| `local_retry` | 瞬时错（网络、超时） | 仅当前节点；计划结构不变 |
| `local_patch` | 契约违规、鉴权错 | 当前节点配置；计划结构不变 |
| `request_replan` | 缺依赖、非法计划结构 | 整个计划版本；生成新 $(V',E')$ |

**Proposition 6.4（升级不变式）**：恢复动作遵守严格升级序——**必须先耗尽 Level $i$（达重试上限或遇不可 patch 错）才能触发 Level $i{+}1$；禁止跳级**。这形式化了 P3：系统总是先试最不具破坏性的恢复，full replan（最贵、最不可预测）是最后手段。

**机械强制（§6.2，这点很重要）**：升级不变式**不是靠自觉**，而是靠每节点一个恢复状态计数器
$$recovery\_state[v]\in\{\texttt{pristine},\texttt{retried},\texttt{patched}\}$$
强制。recovery 层只暴露三个入口 `attempt_retry(v)` / `attempt_patch(v,cfg)` / `request_replan(reason)`，并把升级序作为**前置条件**：`attempt_patch` 除非 $recovery\_state[v]\ge\texttt{retried}$ 否则被拒；`request_replan` 除非所有 failed 节点 $recovery\_state[v]\ge\texttt{patched}$ 否则被拒。**绕过 API 直接改节点状态**的实现，落在 Graph Harness 的保证之外——作者类比为类型安全语言里的 `unsafe` 块（§6.2 末）。

**Def 6.4（诊断）**：诊断是元组 $d=(\varphi,c,r,\alpha)$——$\varphi$ 观察到的失败，$c$ 根因假设，$r\in\mathcal{R}$ 推荐恢复动作，$\alpha\in[0,1]$ 诊断置信度。诊断器**只在 $\mathcal{C}_{diag}$ 上工作、不碰 $\mathcal{C}_{exec}$**——保证诊断推理不泄进执行路径（可审计性关键）。

> **Why（设计层）——为什么不许跳级？** 朴素做法是让 LLM 一看失败就自己决定"要不要直接重规划"。→ 这正是 §4.4 那两个病态的温床：要么无限重试、要么瞬时错就过早 full replan。逐级电梯把"诊断"（LLM 可参与，产出 $d$）和"动作"（沿固定升级梯，机械强制）**解耦**——LLM 可以*建议*，但*不能跳级执行*。这条是本文对"我们自己重试逻辑"最直接的处方（见 Inspires-Us e）。

## §10　Join 语义：支持 all_of / any_of，刻意排除 first_of

> 主讲提示：join 决定"一个节点什么时候进就绪集"。讲清两种支持的、一种被砍的，以及为什么砍。

**Def 7.1（all_of join）**：节点 $v$ 的 all_of join 覆盖前驱集 $P$，当且仅当**每个**前驱都达终止成功态才进就绪集：$ready_{\text{all\_of}}(v)\iff\forall p\in P:\sigma(p)=\texttt{executed}$。这是"依赖满足式执行"的自然 join。

**Def 7.2（any_of join）**：覆盖候选集 $C$——(1) **派发**：所有依赖已满足的候选按**确定性全序**（如节点 id 升序）派发，顺序由 DAG 结构定、不依赖运行时状态；(2) **成功传播**：只要任一候选 $c^*$ 达 executed，后继 $v$ 即可进就绪集（$ready_{\text{any\_of}}(v)\iff\exists c\in C:\sigma(c)=\texttt{executed}$）；(3) **兄弟跳过**：$c^*$ 成功后，其余仍在 pending/ready/running/failed_retryable 的候选转 `skipped`（已终止的保留终止态）；(4) **失败传播**：若所有候选都终止失败且无一 executed，则 join 失败、$v$ 转 failed。这是"备选路径执行"的自然 join——并发试所有备选、取先成功者。

**Def 7.3（first_of join，被排除）**：first_of 会让 $v$ 在**任一候选完成（无论成败）**就进就绪集、并**中途取消所有其他候选**——即"投机执行/竞争性并行"（launch 多个竞争方案、取第一个成功、砍掉其余）。

**§7.3 排除 first_of 的两条理由（Why 设计层）**：
1. **败者取消（loser cancellation）**：中途砍掉不会被用的节点，需要**补偿协议**处理部分结果（回滚副作用、抢救部分输出）——LLM 生成是有状态的，中途取消很棘手。
2. **提交点歧义（commit-point ambiguity）**：若 A 先成功，B、C 还该不该执行？万一 B 会产出更高质量？这些都把非确定性引进执行语义。
作者强调这两个问题**并非不可解**（分布式事务系统有补偿协议/共识机制），但它们引入显著实现复杂度和自身失败模式（补偿本身可能失败），故在 P1（可控性优先）下**当前版本排除**，留作未来带"显式承认可控性代价"的扩展（§7.3、Table 14）。

**表达力边界（Table 11）**：Graph Harness 支持 顺序 / 并行 / 备选路径（✓），**不支持** 竞争性并行(first_of) / 递归子图展开 / 动态拓扑改 / 父链回滚（—）。每个被砍能力都对应一条原则违背（Table 14）。作者给了 first_of 的**变通**：用 all_of 并发跑 A/B/C，下游节点检视全部输出取最优——牺牲"取消效率"，保住选择逻辑（§10.1）；据调查**仅 8% 任务真需要 first_of**（多为"竞速求解"基准），多数任务条件选择足矣。

**附录两条 join 性质**：Prop A.1（all_of 就绪集**单调**——加入更多 executed 前驱只会让更多节点就绪，绝不会让已就绪节点反悔）；Prop A.2（any_of 就绪集**非单调**——一旦选定并 skip 兄弟，剩余候选被移出就绪集，这是"一个就够"语义的刻意结果）。

## §11　可归因实验协议：七组，逐组只加一个特征

> 主讲提示：这是本文的"实验设计"——但注意它**只是协议、没跑**。讲清它想把哪个增益单独摘出来。

**七组设计（Table 12，每组比上一组只多一个特征）**：

| 组 | 名称 | 调度器 | 结构 | 恢复 |
|---|---|---|---|---|
| G0 | SOTA Loop（如 Claude Code / Codex agent 模式） | 单就绪单元 | planner 提示+反思 | inline replan |
| G1 | Naive Loop | 单就绪单元 | 无 | 上下文续跑 |
| G2 | Planner Loop | 单就绪单元 | 无 | 上下文+replan |
| G3 | Structured Loop | 单就绪单元 | 脚手架 | 脚手架恢复 |
| G4 | GH-Core | **多就绪单元** | 静态 DAG | 仅 retry |
| G5 | GH+Patch | 多就绪单元 | 静态 DAG | retry+patch |
| G6 | GH+Replan | 多就绪单元 | 静态 DAG | retry+patch+replan |

**Def 8.1（可归因增益）**——每个相邻差摘出一个设计决策的贡献：
$$G_{plan}=\text{Perf}(G2)-\text{Perf}(G1),\quad G_{scaffold}=\text{Perf}(G3)-\text{Perf}(G2),$$
$$G_{graph}=\text{Perf}(G4)-\text{Perf}(G3),\quad G_{patch}=\text{Perf}(G5)-\text{Perf}(G4),\quad G_{replan}=\text{Perf}(G6)-\text{Perf}(G5),$$
$$G_{total}=G_{plan}+G_{scaffold}+G_{graph}+G_{patch}+G_{replan}.$$
其中 $G_1-G_0$ 专门用来隔离"**信息 vs 结构**效应"（G0/G1 都是循环，但 G0 提示更富；负值说明 G1–G6 的改进来自**结构**而非信息，§8.1）。

**受控变量**（§8.1）：同一批 50 个精选任务、同一底座模型（同温度/同 max token）、同工具集、同超时、同 token 预算。**测量变量**：成功率、执行时间、token 成本、节点数、恢复动作数、计划版本数。任务按依赖复杂度分三层（Table 13：Simple 1–3 步 / Medium 4–8 步（主评层）/ Complex 9+ 步）。

**作者自己列的偏差与缓解**（§8.1）：任务选择偏差（若过采样可并行任务会抬高 $G_{graph}$）→ 报告任务级结果并分析相关；实现偏差（G4–G6 可能有 bug）→ 单测 + 增量验证；LLM 非确定 → 每任务跑 10 次报均值±标准差；工具延迟偏差、顺序效应 → mock 工具/随机任务序。**加性分解的局限**（§8.4）：若规划与结构**交互**（好计划从图结构获益更多），$G_{graph}$ 会把纯结构效应与"计划×结构"交互混在一起；可在不同规划质量下估 $G_{graph}$ 来检测交互，若稳定则交互可忽略、若变动大则应改用乘法模型。

> **读出什么（Θ5 收紧）**：这套协议设计得相当讲究（逐组隔离 + 偏差清单 + 加性局限自曝），但**一次都没跑**（§8 Note、§9.5）。三个理论预测——$G_{graph}>0$（多就绪单元本身带来可测收益，独立于规划质量）、$G_{graph}$ 随任务复杂度增大、$G_{replan}>0$ 仅在高失败率任务上——都是"框架假设的逻辑推论，可能在实践中不成立"（§3.4 Important caveat）。这正是本篇作为 position paper 的诚实底色。

## §12　讨论：静态图什么时候够用、什么时候是负担

> 主讲提示：这一页是判断力的高地——不要把"静态图 > 循环"讲成绝对真理（Θ5）。

**调度器连续谱作为设计透镜（§9.1）**：给任意 agent 系统问三句话——每步 $|\mathcal{U}|$ 是多少？$\mathcal{P}$ 多显式？这直接约束它的表达力/可控性/可实现性画像。作者用它观察 2025–2026 文献的一个趋势：**系统都在向图结构执行收敛，但方向各异**——TDP(Li 2026) 加 DAG 减上下文纠缠、DynTaskMAS(Yu 2025) 加动态任务图求并行、GPTSwarm(Zhuge 2024) 加可优化图求自动改进——它们都到了多就绪单元区间，但**都不提供**执行承诺与有界恢复，作者argue这不是疏忽而是"优先表达力而非可控性"的后果（§9.1）。

**静态 DAG 何时更优（§9.2–9.3，Θ5 的核心）**：
- **静态 DAG 胜出**当任务结构**执行前已知**（工程任务、依赖清晰）——三大优势：可审计（不可变计划版本可事后检视，对合规/调试/根因关键）、可预测的失败语义（失败模式有界：缺依赖、契约违规、节点失败）、工程工具链兼容（天然接 CI/CD、监控、部署）。
- **动态图胜出**当结构**在执行中才浮现**（探索式研究、未知失败模式的调试、创意工作流）。作者明列 Graph Harness **不适用**的三类任务（§9.7、§10.2）：开放式探索（"研究这个主题写篇综述"，子任务集要看中间结果）、动态目标演化（"排查故障并修好"，修法依赖诊断、无法预先全规划）、创意生成（"写个故事再按反馈改"，修订结构依赖已生成内容）。对这些，LangGraph 式动态图或带 inline replan 的 Agent Loop 更合适。
- **权衡非零和**：Graph Harness 的设计点"刻意选来服务一类特定任务（可验证工程任务），代价是不服务另一类（探索/动态目标）"（§9.2 末）。

**一个致命的自曝：SGH 可能退化为单就绪单元（§9.8）**：**只有当 planner 生成的 DAG 含独立路径时，Graph Harness 才是多就绪单元调度器**。若 planner 产出线性链（每节点恰一前驱一后继），则处处 $|\mathcal{U}|\le 1$，**Graph Harness 退化为单就绪单元调度器**，相对 Agent Loop 的优势缩到只剩三条（有界恢复、计划版本可审计、上下文隔离），**拿不到多就绪单元的完整结构红利**。据调查**仅约 30–40% 的 agent 任务展现自然并行**（多文件读、并行 API、备选方案），其余 60–70% 本质是线性链或并行难识别（§9.3.2）。→ **SGH 的价值被 planner 质量上界**：planner 识别不出并行，图就白搭（§9.3.1、§9.8）。

> **读出什么（Θ5，本篇的判断力护城河）**："静态图 > 循环"是**分 regime** 的：
> - **图主导**：任务依赖执行前可articulate、有并行结构、要审计/合规 → 静态 DAG 拿满红利。
> - **图退居**：任务结构涌现（探索/动态目标/创意）、或本质线性、或 planner 识别不出并行 → 图退化为带护栏的循环，红利只剩恢复/审计/隔离。
> 这与标杆 Harness-Bench 的 regime 结论**同构**：那边说"强模型更不挑 harness、纯语言任务几乎不吃 harness"，这边说"线性/涌现任务几乎不吃图结构"。两篇从实证与理论两侧夹出同一个诚实边界。

## §13　局限与批判（论文 §9.5 + §10 + 我的补充）

**论文自陈的局限（罕见地坦诚，逐条引）**：
- **L1 无实验验证（最大局限）**：七组协议需要原型实现 + 精选基准，二者都是"进行中"；在实验做完前，多就绪单元调度、升级协议、不可变计划版本的性能收益**都是理论预测而非实证主张**（§9.5）。
- **L2 静态 DAG 假设**：对探索/动态目标/创意三类任务失效——静态承诺反成负担（§9.5、§9.7）。
- **L3 LLM 错误传播**：多就绪单元把风险放大——单就绪单元循环里 LLM 第 $t$ 步错了可在 $t{+}1$ 自我纠正；图里 planner 把 DAG 生错（如漏依赖），错误会传播到多个并行执行、浪费时间和 token。恢复协议靠"契约验证抓错误输入"缓解，但代价更高（§9.5）。
- **L4 冷启动**：新任务无现成 DAG 模板时，planner 须从零构造（识别子目标、定依赖、选 join、生成每节点契约），任一步错都触发恢复/replan；一次性任务里冷启动开销可能盖过并行收益（§9.5）。
- **L5 无复杂度理论保证**：未证"预算约束下算最优调度是 NP-hard"（虽由经典 DAG 调度归约看很可能是），未刻画默认拓扑策略的近似比（§9.5、§3.5）。作者仅给了一个理论连接：拓扑策略 = 零通信成本的 list scheduling，经典上对 $m$ 台同构处理器有 $2-1/m$ 近似比（Graham 1969，§2.7），暗示"若 LLM 执行时间近似恒定，默认策略有可证性能保证"。
- **L6 实现复杂度**：最小 SGH 约需 **3300–6500 行**生产级代码（DAG 校验引擎 1000–2000 + 并发调度器带限流 800–1500 + 状态持久化 WAL 500–1000 + 恢复引擎 600–1200 + 契约验证 400–800），比朴素 Agent Loop（300–500 行）复杂一个数量级，但比 Airflow(50000+)/Prefect(30000+) 简单（§9.5）。作者注：这些代码量估计本身也基于对同类系统的分析、待实测校准。

**我的补充批判**：
- **调查是全篇经验基座，但作者亲手拆了它的台**：附录 A.4 明写"70 项目调查**非同行评审**、应作**定性证据**而非定量证明"，项目选择主观未系统抽样、失败循环等分类靠人工 issue 检视（附录 A.5 报了 $\kappa=0.84$ 双人一致性，算是补强）、只反映 2026-04 现状。于是四原则的"survey rationale"证据强度都被这条自曝**打了折**——它们是"有观察支撑的设计直觉"，不是"被证明的规律"。
- **"验证间隙"(Thm 6.3) 是自曝的天花板**：契约验证只保证语法，语义验证的 $p_v$ 可能远小于 1；一个"全 executed"的 DAG 完全可能全是语义错的产物。作者诚实地只承诺"可观测+可控"、不承诺消除——但这意味着 **Graph Harness 的可靠性上界仍握在"谁来验证验证者"手里**，与全库收口隐忧同源。
- **例子的"理想性"被作者自己标注**：Figure 1 脚注、§3.6、§9.6 反复提醒 worked example 假设 planner 完美；而 §9.8 又承认 planner 常识别不出并行。**把"理想例子的红利"当成"真实红利"是最大的误读风险**——本文的诚实恰恰在于它到处贴这个提醒。
- **单作者、个人邮箱、无机构、无代码**：与标杆 Harness-Bench（北大+奇元、有 repo、5194 条轨迹）相比，本文的**权威性来源是"论证质量"而非"机构/实证背书"**。读它要按"一份严谨的设计提案"而非"一个已验证系统"对待。

## ★ 对我们的启发（Inspires Us）

> 这一节是组会高潮，也是 B 组的天然优势：**我们（Claude Code / 本课的 env-audit 验证器）本身就活在一个控制循环里**——
> 我们的 `_run_runbook` 就是一台 `subprocess.run(..., timeout)` 驱动的**单就绪单元、非确定、非幂等重试**循环。
> 本文这套"把循环写成调度器、把恢复写成电梯"的处方，能**逐条打到我们自己的 loop 上**。

➤ **a. 可直接借用的招（method/trick we can reuse）**：**执行上下文 / 诊断上下文隔离**（Def 5.3）+ **三级恢复电梯**（Def 6.3、Prop 6.4）。前者说"别把上次的失败报错塞进下一步的输入"；后者说"重试→打补丁→重规划只能逐级升，且用一个 `recovery_state∈{pristine,retried,patched}` 计数器**机械强制**、不许跳级"（§6.2）。这两招是纯工程处方，不依赖任何实验结论就能抄。

➤ **b. 可迁移到我们的模块（transfer）**：把 §6 的四原则对账表（Table 8）+ 七组归因协议（Table 12）搬来评我们自己的 harness 改动——每加一个循环特征（比如给 env-audit 的 runbook 跑加"失败自动重试"），就照 $G_{patch}=\text{Perf}(\text{加了}) - \text{Perf}(\text{没加})$ 的思路，用同一批模块做**逐特征归因**，而不是笼统说"新循环更好"。迁移前提：我们得先把每次 runbook 跑的结果**结构化记录**（现在 `RunResult` 已有 status/PASS/FAIL/TIMEOUT，是现成的可归因底座）。

➤ **c. 它暴露的开放问题 = 我们的机会（open problems → our opportunity）**：本文把 **first_of（竞争性并行）刻意砍掉**（§7.3），理由是"败者取消要补偿协议、LLM 生成有状态难回滚"。但我们的很多验证任务**恰恰是幂等只读或可安全取消的**（跑 `--help`、跑 smoke 训练到 exit）——**这正是本文留的口子**：在**副作用为零/低**的节点子集上，安全地重新引入 first_of / 竞速执行（比如"两种 smoke 配置并发跑、取先绿的"）。可下手第一步：给 runbook 条目加一个 `side_effect: none|low|high` 标注（对齐 P4），只对 `none` 的允许竞速。

➤ **d. 与本库其它论文/模块的连接（connect the dots）**：与标杆 **Harness-Bench(2605.27922)** 是**理论↔实证的一对**——它测出"换 harness 摆 23.8 分"，本文解释"因为循环层有这些可调自由度"；两者的 **regime 结论同构**（强模型/纯语言 不吃 harness ↔ 线性/涌现任务 不吃图结构）。与 **H 组失败恢复类论文**正面互补——本文的三级电梯是"恢复协议"的骨架，H 组测的是"每一级恢复的成功率"。**Thm 6.3 的"验证间隙"** 与全库"谁来验证验证者"的收口隐忧、以及我们 memory 里 `fix-algo-to-match-paper` 那条"代码要对得起论文"的独立校验精神，是同一种警惕。

➤ **e. 如果我来做下一步（my next move，第一人称，落到我们自己 harness 的具体组件）**：**我们当前的 `scripts/eric_3080ti_env_audit.py::_run_runbook` 正面临本文点名的非幂等重试问题**——它用 `subprocess.run(..., timeout=timeout)` 串行跑每条命令，TIMEOUT 只是记一个状态、**没有任何有界重试协议**；一旦某条 smoke 训练偶发挂了（CUDA 抢占、瞬时 OOM），当前要么整轮红、要么手动重跑，正是 §4.4 那个"要么无限重试要么过早放弃"的病态。**我的下一步**：给 `_run_runbook` 加一个**最小三级恢复层**——Level-1 `local_retry`（对 TIMEOUT/瞬时非零退出，按每条目 `smoke` 预算重试至多 $b_v$ 次，$b_v$ 从 runbook.yaml 读，默认 1）；Level-2 `local_patch`（退化到更小 smoke 预算再跑一次，等价于"改节点配置不改结构"）；Level-3 才 `request_replan`（标记该模块需人工介入）。并按 Prop 6.4 用一个 per-command `recovery_state` 计数器**机械禁止跳级**。这样我们自己的验证循环就从"祈祷不超时"升级成"有界、可审计、可归因"，且**先不引入图结构、零表达力代价**——正好落在本文 §9.8 说的"退化单就绪单元也仍拿得到的那三条红利（有界恢复/可审计/上下文隔离）"上，是投入产出比最高的第一步。

## §14　版图定位（canon/前沿坐标 + 在本库的位置）

- **时间坐标（Θ4）**：**2026 前沿 position paper**。相对基石的推进——**ReAct(Yao 2023)** 定义了 observe-reason-act 朴素循环，本文把它**形式化为"非确定单就绪单元调度器"**并指出其三处结构病；相对 **Plan-and-Act(2025)/TDP(2026)/LangGraph** 的推进——它们都到了"计划分离"或"多就绪单元"，但本文**额外**给出：① 计划不变式（P2）+ 三层分离；② 带证明的终止性(Thm 6.2)与条件可靠性(Thm 6.3)；③ 机械可强制的三级恢复电梯(Prop 6.4)。**它不证伪谁，而是给这一片"图化 agent"工作补了一副形式化的骨架和一把诚实的尺子。**
- **L 层归属（Θ1）**：本篇是 **B 组 / L 层（控制循环）** 的理论范文——全部机械都在"派发/恢复/计划可变性"的循环逻辑上；强依赖 **C 层**（$\mathcal{C}_{exec}/\mathcal{C}_{diag}$ 隔离）、接口 **V 层**（契约 $\kappa$ 把关 executed）。
- **回扣 `Agent = Model + Harness`（Θ2）**：本文的贡献是**把 harness 的"循环层"从黑箱 `while True:` 解剖成一台有 5 个可调自由度的调度器**（就绪集基数 / 策略显式度 / 策略确定性 / 计划可变性 / 恢复协议）。这从**理论侧**支撑了全库命题——"换 harness 换出能力差"之所以发生，是因为这些自由度的取值**本就决定**了可并行性、可恢复性、可审计性。它与标杆 Harness-Bench 的实证（同模型换 harness 摆 23.8 分，Table 2）**互为表里**：那边给数字，这边给"为什么会有这个数字"的结构解释。
- **在本库的位置**：B 组 ⭐ 前沿设计锚点。读完它，回看任何一篇 agent 系统论文都能问三句：它每步 $|\mathcal{U}|$ 是几？$\mathcal{P}$ 是函数还是关系（确定还是拍脑袋）？它的恢复有没有上界与升级序？——这三问就是本文留给全库的**通用体检探针**。

## §16　一页速记

- **命题**：Agent Loop 本质是**非确定的单就绪单元调度器**（$|\mathcal{U}|\le 1$，$\mathcal{P}$ 是关系），有三病——依赖隐式、恢复无界、计划静默改写（§1、§3.2）。
- **地基**：执行系统 = 五元组 $\mathcal{E}=(\mathcal{S},\mathcal{U},\mathcal{P},\mathcal{O},\Delta)$（Def 3.1）；沿"就绪集基数 / 策略显式度 / 确定性"排成调度器连续谱（Figure 2）。**加 planner 只右移显式度、不升就绪集基数——只有图执行器能到多就绪单元**（§3.4）。
- **药方（Graph Harness）**：把控制流抬升为**静态 DAG**；三承诺——① 计划版本内**不可变**（Def 5.2）；② 规划/执行/恢复**三层分离** + 执行/诊断上下文隔离（§5.3、Def 5.3）；③ 恢复走**三级电梯 retry→patch→replan、机械禁跳级**（Def 6.3、Prop 6.4）。
- **四原则**：可控性优先 / 稳定承诺 / 有界恢复 / 副作用分级（§4，Table 8，均由 70 系统调查佐证但作者自陈定性）。
- **两条定理**：Thm 6.2 有界终止（有界执行下概率 1 终止，上界 $\sum\tau_v(b_v{+}1)+\sum T_{human}$）；Thm 6.3 条件可靠性/验证间隙（$\Pr[\text{全对}]\ge\prod_v p_v$，语义验证 $p_v$ 可远小于 1、只能observable/controllable、消不掉）。
- **join**：支持 all_of（单调）/ any_of（非单调、先成功者胜、兄弟 skip）；**砍 first_of**（败者取消需补偿、提交点歧义），仅 8% 任务真需要（§7、Table 11/14）。
- **实验**：七组逐特征归因（Table 12）+ 五增益分解（Def 8.1）——**但一次没跑，全是理论预测**（§8 Note、§9.5）。
- **诚实（Θ5）**：静态图**分 regime**——依赖可预知/有并行/要审计 → 图主导；结构涌现（探索/动态/创意）或本质线性或 planner 识别不出并行 → 图退化为单就绪单元、红利只剩恢复/审计/隔离（§9.8）。与 Harness-Bench 的 regime 结论同构。
- **对我们**：搬"上下文隔离 + 三级恢复电梯"到我们的 `_run_runbook`——它正用 `subprocess.run(timeout)` 做无界非幂等重试，加一个**最小三级恢复层 + recovery_state 禁跳级**即可，零图结构、零表达力代价，先拿"有界终止/可审计"这三条稳红利。
