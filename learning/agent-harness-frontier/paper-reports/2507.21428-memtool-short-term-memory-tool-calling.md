# MemTool：为多轮对话里的动态工具调用做短期记忆管理（自主/工作流/混合三模式）

> **本篇定位**：这是一篇**工程取舍**很重的前沿论文，读它的价值不在某个惊艳数字，而在它把一个我们天天遇到的问题——
> "给 agent 挂了一大堆工具/MCP，多轮之后上下文里工具越堆越多"——**拆成一个可度量的短期记忆管理问题**，并给出三种"谁负责增删工具"的架构选择。
> 本文严格对齐库内标杆 [Harness-Bench](2605.27922-harness-bench-measuring-harness-effects.md) 的密度与诚实度：公式前先给直觉、符号先定义、指标给定义式、数字标 §/Table/Eq 出处、区分**宣称 vs 批判**。
> 分层上它是本库 **C 组（T 层为主、C 层为辅）**的样本；`## ★ 对我们的启发` 会把它直接打到 Claude Code 这个 harness 自己的工具层上。

## §1　TL;DR（一页讲清这篇在干嘛）

> 主讲提示：开场先讲清痛点（工具塞爆上下文），再抛出"三种谁来决定增删工具"的骨架，最后给一句最关键的实验分野（推理模型行、中小模型崩）。

**一句话**：LLM agent 现在可以从上千个工具/MCP 服务器里**动态搜索、装备、使用**工具（像人在应用商店里装 App）；但在**多轮对话**里，一个致命问题浮现——**用过的工具没人负责卸载，上下文里的工具越堆越多，最终撑爆上下文窗口 / 逼近 API 的工具数上限 / 让模型在一堆无关工具里选错**。MemTool 把这件事**框定为"短期记忆管理"**（§1、§2.1），并给出三种架构（Figure 1）：

- **Autonomous Agent Mode（自主模式）**：给 LLM 两个工具 `Search_Tools` 和 `Remove_Tools`，让它**在回答用户的同一个 while 循环里，自己决定**什么时候加工具、什么时候删工具（Algorithm 1）。**最大自主权**。
- **Workflow Mode（工作流模式）**：**不给 agent 增删自主权**；在每个用户 query 之前，用**两个固定的、独立的 LLM 调用**先"剪枝（删无关工具）"、再"搜索（加新工具）"，然后才让 agent 用最终工具集回答（Algorithm 2）。**最确定、最省心**。
- **Hybrid Mode（混合模式）**：**删由工作流固定负责（一个独立 prune LLM 调用先删），加由 agent 自主负责**（agent 只拿到 `Search_Tools`，可自己再检索补工具）（Algorithm 3）。**折中**。

**最关键的实验分野（Abstract + §4.2）**：在**自主模式**下，**推理型 LLM**（如 GPT-o3、Gemini 2.5 Pro、Claude Opus 4）能把工具移除效率做到**90–94%**（3 轮滚动平均，Table 1）；而**中小模型**（GPT-4o mini、Claude 3.5 Sonnet、LLaMA 3 70B、GPT-4.1 Nano）效率**暴跌到 0–60%**——它们只顾回答用户、**忘了删工具**，工具数一路爬到 128 上限（Figure 2）。而**工作流 / 混合模式**因为把"删"这件事**从模型手里拿走、交给固定流程**，**几乎所有模型**的移除率都稳定在 **>90%**（Table 1）——**代价**是牺牲了 agent 自主探索的灵活度。

- **属于 harness 的哪一层（Θ1）**：本篇主打 **T（Tools/工具接口）层**——它管的是"工具集这个可变资源如何在多轮里增删"；同时**深度牵涉 C（Context/上下文）层**，因为"工具定义"本身就占上下文 token，工具管理即短期记忆管理（作者在 §2.1 明确把它归到 short-term memory）。它**不碰** L（控制循环骨架）/ V（评测协议）/ E（环境沙箱）的本体，但用到了一套自定义评测指标（§4.1）。
- **回扣全库论点（Θ2）**：这篇是 `Agent = Model + Harness` 的一个**双向证据**——同一个 harness（自主模式）下，换模型（推理 vs 中小）分数摆动巨大（90% vs 0%，Table 1）**⟹ Model 很重要**；但同一批模型，换 harness（自主→工作流）能把最差模型的移除率从 0–24% 抬到 90%+ **⟹ Harness 也很重要**。**harness 的作用是"给弱模型兜底"**：把模型做不好的"删工具"外包给确定性流程。
- **够新够权威（Θ4）**：**2025-07 预印本**，出自**普华永道（PwC）**商业技术与创新办公室——工业界视角，作者同组还产出了 ScaleMCP 基准（本文实验基座）。它是"**短期工具记忆**"这条细分线上较早做系统化"多模式 + 多模型"评测的工作之一（§2.1 自述 "MemTool is the first study to address and evaluate short-term memory for dynamic tool-use across multi-turn conversations"）。

## §2　问题与动机：为什么"工具的短期记忆"值得单独拿出来做

> 主讲提示：这一段用 Why 三连的"问题层"，把"上下文压缩"这条老路和"工具管理"这条新路划清界限——这是全文立论的根。

**Why（问题层）——不解决会卡住什么？**

先摆事实。作者引 Karpathy 的比喻开篇（§1 题记）：**"The LLM is the CPU, and its context window is the RAM"**——上下文窗口就是 agent 的"内存"，而内存是有限的。旗舰模型上下文普遍在 **10 万–100 万 token**（§2.1.1）。动态工具调用（dynamic tool calling / function calling）让 agent 可以从**上千个工具或 MCP 服务器**里检索并装备工具，突破"固定工具集"的限制（§1）。**但**——

现有短期记忆研究的**盲区**（§1 原文点名）：主流做法**只压缩"助手–用户之间的对话文本"**（靠 summarization / truncation，引 Wu et al. 2025b、Breunig 2025）。**这些方法压的是对话，不是工具。** 在**动态工具检索**的多轮场景里，真正膨胀的是**工具定义**：agent 每轮可能检索出几个新工具塞进上下文，**用完却没人负责卸载**。作者明确指出（§1）："LLM agents can discover and add hundreds of new tools or MCP servers to its context window, **but need to reduce the number of available tools** in its context upon solving the user question and no longer needing them." 也就是说——**"加"有人管（检索/RAG 已经很成熟），"删"没人管。**

**为什么"删"这么关键？** 三重后果（§1、§2.2、§4.1）：
1. **撑爆上下文 / 触发 API 硬限**：工具定义占 token；而且 LLM 提供商对"单次请求能挂多少工具"有硬限制——OpenAI GPT 系列约 **128 个工具**，Gemini 2.5 Pro / Claude Opus 4 可到 **256–512**（§4.1，引 OpenAI 2024、Google 2025a）。工具堆到上限，API 直接报错。
2. **选择变差**：工具越多，模型在无关工具里**选错/选乱**的概率越高（这是 §2.2 引出的 tool-selection 难题）。
3. **多轮才暴露**：前人证明"一次性塞相关工具进去、单轮问答"精度不掉（RAG for tools，引 Lumer et al. 2025c、Chen et al. 2024）；**但没人评过"工具如何跨 100+ 轮同一 session 持久与管理"**（§1 原文 "there needs to be thorough evaluation of how tools persist through multi-turn conversations"）。

> **读出什么**：这篇的动机不是"再造一个更强 agent"，而是发现了一个**被现有记忆研究漏掉的维度**——**工具集本身也是短期记忆的一部分，而且是"只进不出"的那部分**。这正是它归到本库 **T 层（工具）× C 层（上下文）交叉**的原因：它把"工具接口设计"和"上下文/记忆管理"焊在了一起。

> **术语对照**：**MCP** = Model Context Protocol（模型上下文协议，一种让 agent 标准化接入外部工具/服务的开放协议）；**short-term memory（短期记忆）** = session 内、上下文窗口内的信息；**long-term memory（长期记忆）** = 跨 session 持久化的信息（本文只做短期，§2.1.2 只作背景交代）；**dynamic tool calling（动态工具调用）** = agent 不被固定工具集束缚、运行时检索装备工具。

## §3　三个核心贡献（论文 §1 + §3.1 提炼）

1. **问题的重新框定**：首次把"多轮动态工具场景下工具集的膨胀"**明确框定为短期记忆管理问题**（而非上下文文本压缩问题），并主张 agent 应能像人逛应用商店那样**自主装备与卸载**工具（§3.1）。
2. **三种架构（Figure 1，本文主体）**：Autonomous / Workflow / Hybrid，**核心差异是"增删工具的决策权归谁"**——模型自主、固定工作流、还是混合。每种都给了**可复现的算法伪码**（Algorithm 1/2/3）。
3. **跨模型的系统评测**：在 **ScaleMCP 5,000 工具 × 100 轮多轮对话 × 13+ LLM**（Table 1/2）上，定义并测量**工具移除效率（短期记忆维度）**与**任务完成度（能力维度）**，得出"**推理模型才配拿自主权、中小模型必须靠工作流兜底**"的可操作结论与推荐（每个模式末尾的 Recommendations）。

## §4　三种模式的骨架：谁来决定"增/删工具"？（本篇最该讲透的地方）

> 主讲提示：这是全场核心。别急着念伪码，先用一张"决策权归属表"把三种模式的**唯一本质差异**钉死，再逐个走算法。反复问一句："这一步的『删工具』是模型自己想删，还是流程强制它删？"

**先给直觉（一句话区分三种模式）**：三种模式解决的是**同一件事**（多轮里保持工具集精简），差异**只在两个动作——"加工具（Search）"和"删工具（Remove）"——分别由谁触发**：

| 模式 | 「删工具」由谁决定 | 「加工具」由谁决定 | 自主度 | 一句话画像 |
|---|---|---|---|---|
| **Autonomous 自主** | **模型自己**（调 `Remove_Tools`） | **模型自己**（调 `Search_Tools`） | 最高 | "你全权负责，边答边管家" |
| **Workflow 工作流** | **固定流程**（独立 prune LLM 调用） | **固定流程**（独立 search LLM 调用） | 最低 | "管家是外包的，你只管答题" |
| **Hybrid 混合** | **固定流程**（独立 prune 先删） | **模型自己**（agent 拿 `Search_Tools` 自己补） | 中 | "删由外包托底，加你可以自己来" |

**共同的输入与约定**（三个算法的 `Input` 完全一致）：LLM、用户 query `q`、历史消息 `msgs_prev`、既有工具集 `T_prev`、向量库 `V`（存 5000 个工具的 embedding）、检索 top-k=5、**工具数上限 L=128**、历史管理器 `H ∈ {TRUNCATEHISTORY, SUMMARIZEHISTORY}`（用于在 token 超限时裁剪/摘要**对话历史**——注意它裁的是历史文本，**不动工具**，§3.1.2 明确）。

### §4.1　Autonomous Agent Mode（自主模式）——把管家职责交给模型

**直觉**：给 agent 两个"元工具"——`Search_Tools(keywords)`（去向量库检索并装备新工具）和 `Remove_Tools(tool_names)`（把点名的工具从上下文卸载）——然后让它在**回答用户的同一个 while 循环里**，自己判断该搜该删（§3.1.1）。系统提示（Figure 5 全文给出）反复叮嘱它："用户问完、当前工具不相关就**立刻删**，别让它堆积""128 是上限，务必保持工具数低"。

**算法走一遍（Algorithm 1）**：
1. `messages ← H(msgs_prev) + q`；把工具集初始化为 `T_prev ∪ {Search_Tools, Remove_Tools}`（这两个元工具**永远在**）。
2. 进入 `while true`：调 `LLM(messages, tools=T)` 得到 `reply`。
3. 若 `reply` 是 **tool_call**，`switch` 它调了哪个：
   - **`Remove_Tools`**：拿到要删的名单 `R`，`T ← T \ (R \ {Search_Tools, Remove_Tools})`（**从工具集减掉 R，但保留两个元工具**）。
   - **`Search_Tools`**：拿查询 `Q`，`S ← VectorSearch(Q, V, k=5)`，`T ← T ∪ S`；**若 `|T| > L`（超 128），`Raise Error("Hit L limit, remove tools.")`**——把错误抛回给 agent，逼它去删。
   - **otherwise（普通业务工具）**：`ExecuteTool(reply)`，把结果 append 回 messages。
4. 若 `reply` 不是 tool_call（模型决定直接作答），`a ← reply.content`，`return a`。

**Why（设计层）——为什么要给自主权，而不是一上来就固定流程？**
> 朴素替代：直接写死"每轮先删再加"（就是后面的工作流）。→ 但那样 agent **无法在答不出时"回头再搜一次"**——它被钉在初始化好的工具集上（§3.1.2 明确指出工作流"once initialized... there is no mechanism to revert back to searching new tools"）。本文给自主模式的理由正是：**保留 agent 检索–装备的探索闭环**，让它"第一次没搜到合适工具时能再搜"（§4.2.2 "re-searching the tool knowledge base if it cannot find one the first time"）。**代价**：把"删"这件模型不擅长的事也压给了模型（见 §5 的实验，中小模型在这里全线崩盘）。

**作者给自主模式的落地经验（Recommendations，§3.1.1，很实用）**：
- 有些模型**就是不肯删**工具，数量一路涨到上限；
- **系统提示要写清"何时删、何时加"的细则**（Figure 5 就是范例）；
- **把上限错误（128）抛回去能"催"模型删工具**（"Upper limit tool errors 'encourage' LLMs to remove tools"）；
- **把"当前工具数"作为动态变量塞进系统提示**（Figure 5 里的 `<YOUR CURRENT TOOL COUNT>`）——这条**极其关键**：作者发现**若系统提示里不告诉模型现在有几个工具，模型就意识不到工具已经堆积、于是不删**（§4.2.2 "when the system prompt did not include the current number of tools... caused the LLM agent to not remove tools"）。

### §4.2　Workflow Mode（工作流模式）——把管家外包成两次固定 LLM 调用

**直觉**：**不给 agent 任何增删自主权**。在 agent 回答之前，用**两个专门的、无状态的（fresh）LLM 调用**把工具集整理好：**第一个调用只干"删"**（看当前工具+用户 query，吐出该删的工具名单），**第二个调用只干"加"**（看用户 query+现有工具，判断够不够，不够就生成检索关键词去搜）。整理完，才把最终工具集交给 agent 在 while 循环里答题——**而 agent 手里已经没有 Search/Remove 工具了**（§3.1.2）。

**算法走一遍（Algorithm 2）**：
1. `messages ← H(msgs_prev) + q`；`T ← T_prev`。
2. **剪枝（删）**：`R ← LLM_prune(messages, T)`；`T ← T \ R`。（`LLM_prune` 用 Figure 7 的 Remove 提示，要求**逐字匹配工具名**，否则删除失败。）
3. **搜索（加）**：`Q ← LLM_search(messages, q)`（用 Figure 6 的 Search 提示生成关键词）；**若 `Q ≠ ∅`**：`S ← VectorSearch(Q, V, k=5)`，`T ← T ∪ S`；**若加完 `|T| > L`**，进入一个 `while |T|>L` 的**补救剪枝循环**：再调 `LLM_prune` 删，若删不动（`R=∅`）就 `Raise Error("T>L")`，否则 `T ← T \ R` 直到降到上限内。
4. `while true`：`LLM(messages, tools=T)`——此时 agent **只能执行业务工具或直接作答**（`ExecuteTool` 或 `return reply.content`），**无权再增删**。

**Why（设计层）——为什么把增删拆成两个独立 fresh LLM 调用，而不是让答题的那个 agent 顺手做？**
> 朴素替代：就用自主模式，让答题 agent 自己删。→ 实验证明中小模型**一心答题、忘了删**（§4.2.2）。工作流的设计哲学是**职责分离**：**"删"和"加"各由一个只做这一件事的无状态调用负责**，把记忆优化的负担从"要同时答题的主 agent"身上卸掉（§4.3.2 "removes the burden of memory optimization from the model, allowing even smaller models to perform well"）。**代价**：agent 被"蒙住眼"（blinded / abstracted from tool management，§3.1.2），一旦初始化就**回不到"再搜一次"的探索**；且**每轮固定多花两次 LLM 调用**（成本上升）。

**落地经验（§3.1.2）**：工作流的**简单性**是它被推荐为"首选"的理由；但"限制自主 = 阻止了任何自我纠错（self-correctness）"——作者提到前人（Lumer et al. 2025d）想给工作流加"自校正"步骤，但"这类模式会变得过于复杂和死板"，所以本文的工作流**刻意保持简单**。

### §4.3　Hybrid Mode（混合模式）——删外包托底、加放权给模型

**直觉**：作者观察到一个**很干净的分工线索**（§3.1.3 原文）——**"LLMs tend to struggle with tool removal (剪枝) but perform well in searching and adding tools"**。既然模型**不擅长删、擅长搜**，那就**把删外包给固定流程（一个独立 prune LLM 调用先删掉无关工具），把加放权给 agent（给它 `Search_Tools`，让它自己按需补工具）**。这样既避免了自主模式"中小模型忘了删"的坑，又保留了自主模式"能回头再搜"的灵活。

**算法走一遍（Algorithm 3）**：
1. `messages ← H(msgs_prev) + q`；`T ← T_prev`。
2. **固定剪枝（删，外包）**：`R ← LLM_prune(messages, T)`；`T ← T \ R`。
3. **给 agent 只挂 `Search_Tools`**：`T ← T_prev ∪ {Search_Tools(keywords)}`（**注意：只给搜、不给删**）。
4. `while true`：`LLM(messages, tools=T)`；若 agent 调 `Search_Tools`，则 `VectorSearch → T ← T ∪ S`，同样带 `while |T|>L` 的补救剪枝循环（超限时再自动 prune）；否则执行业务工具或作答。

**Why（设计层）——为什么混合恰好把"删/加"这样劈开？**
> 这是全文最漂亮的一个设计论证：它**不是拍脑袋折中，而是对准了"模型能力的非对称性"**——删（剪枝）需要模型**主动意识到"某工具已无关"并逐字点名删除**，这对中小模型是弱项（§4.2.2 反复出现的失败）；搜（检索）只需模型**根据当前 query 判断"我还缺啥工具"**，这对模型是强项。所以混合**把弱项交给确定性流程、把强项留给模型**。作者明说混合"solves the limitation found in (Lumer et al. 2025d) of complex self-correcting workflows"（§3.1.3）。**代价 / 隐患**：agent 有搜的自由却没删的权，**可能一口气搜太多、触发工具上限、引发工作流复杂化**（§3.1.3 Recommendations 明确 "It could add too many tools and trigger a tool limit and cause complexity in the workflow"）——这时靠第 4 步的补救剪枝兜底。

**一句话前瞻（作者观点，§3.1.3）**："随着模型和提示技术进步，混合模式可以过渡到自主模式"——即混合是**当前模型能力下的务实折中**，不是终态。

> **读出什么（三模式取舍总结）**：三种模式是一条**"自主度 vs 可靠性/成本"的连续谱**——
> - **自主**：灵活度最高、额外 LLM 调用最少（增删都在主循环里顺手做），但**把命押在模型会不会删工具上**（→ 只适合推理模型）；
> - **工作流**：最可靠、最省心、连小模型都能用，但**每轮固定多两次 LLM 调用、且 agent 回不了头**（→ 成本敏感 + 追求确定性时首选）；
> - **混合**：卡在中间，**删托底、加放权**，对准模型能力的非对称性（→ 想要"确定性的删 + 灵活的加"时用）。
> 这条谱系正是本篇作为 **T 层设计样本**的最大价值：它把"工具接口该给模型多少自主权"这个抽象问题，落成了三个可直接抄的算法。

## §5　指标：怎么量"工具的短期记忆管好了没"

> 主讲提示：这页把 5 个公式讲透。核心思想——**用"加了的工具有没有被及时删掉"来度量短期记忆管理，用"任务答对没"来度量能力**，两者分开报。每个式子先给直觉再给符号。

**评测设置（§3.2、§4.1，先交代 setting）**：
- **数据**：从 ScaleMCP 的 **5,000 个工具/MCP 服务器**里，**分层抽样**出 **100 个多轮工具使用对话实例**（按"每轮工具调用数"分层）；ScaleMCP 平均**每轮 5.0 次工具调用**（§3.2）。
- **多轮结构**：每次实验 = **100 个顺序用户 query**，平均每轮 5 次工具调用；**相邻 query 可能相关也可能无关**（§4.1 "Each query can be related or unrelated to the immediate prior query"）——这正是制造"工具堆积"的关键：无关的新 query 会引入新工具，旧工具就该被删。
- **模型**：**13 个 LLM**，闭源+开源都有（Table 1/2 列全）。
- **工具知识库 embedding**：Azure OpenAI `text-embedding-ada-002`（§4.1，因为 ScaleMCP 已测过 5 种 embedding 差异极小）。
- **工具数硬上限**：`L = 128`（§4.1；为统一所有模型而设，即便 Gemini/Claude 能挂更多）。
- **裁判**：任务完成度用 **LLM-as-judge**，裁判模型是 **OpenAI GPT-4o mini**（§4.1，引 Confident AI 2025 的 DeepEval）。

**五个指标（§4.1，Eq 1–5）**。先讲两个"短期记忆维度"的：

**① Removal Ratio（移除率，Eq 1）**——直觉：**"我这一路加进来的工具，最后有多大比例被删掉了？"** 比例越高 = 工具没积压 = 短期记忆管得好。
$$\mathrm{RemovalRatio} = \frac{\sum_{t=1}^{T} R_t}{\sum_{t=1}^{T} A_t} \le 1$$
符号（先定义后用）：`t` = 轮次（turn），`T` = 总轮数（=100）；`R_t` = 第 t 轮**删掉的工具数**；`A_t` = 第 t 轮**加进来的工具数**。分子是累计删除、分母是累计添加，比值 ≤ 1（删的不会比加的多）。

**② Average Removal Ratio 3T（3 轮滚动平均移除率，Eq 2）**——直觉：**Eq 1 是"一整局的总账"，太粗；这个改成"每 3 轮一算、再平均"，能反映"模型是不是持续地、及时地在删"**，把偶发的大删/大加抹平。
$$\mathrm{AvgRemovalRatio}_{3T} = \frac{1}{T-2}\sum_{t=3}^{T}\frac{\sum_{i=t-2}^{t} R_i}{\sum_{i=t-2}^{t} A_i}$$
符号：对每个从第 3 轮起的窗口 `[t-2, t]`（3 轮宽），算这窗口内的删/加比，再对所有 `T-2` 个窗口取平均。**这是 Table 1 的主排序键**，也是 Abstract "90–94% over a 3-window average" 那个数。

**③ Average Residual +3（3 轮后平均残留，Eq 3）**——直觉：**"每次工具数冲到一个峰值后，过 3 轮还剩多少工具没删干净？"** 残留越低 = 峰值后清理越彻底。
$$\mathrm{AvgResidual}_{+3} = \frac{1}{K}\sum_{k=1}^{K}\left(\frac{1}{3}\sum_{i=1}^{3} T_{\cdot,\,p_k + i}\right)$$
符号：`p_k` = 第 k 个**峰值（peak tool count）**出现的轮次，`K` = 峰值总数；`T_{·, p_k+i}` = 峰值后第 i 轮（i=1,2,3）的工具数；内层对峰后 3 轮取平均、外层对所有峰取平均。**读它的方式**：这个数**越小越好**（残留少）；Table 1 里 GPT-4.1 Nano 自主模式残留 0.00（因为它压根不加也不删=退化），Claude 3.5 Sonnet 残留 124.00（几乎贴着 128 上限躺平=灾难）。

再讲两个"能力维度"的：

**④ Task Completion（任务完成度，Eq 4）**——直觉：**"agent 最后给用户的答案，和基准里的期望答案对不对得上？"**
$$\mathrm{TaskCompletion} = \mathrm{Alignment}(\mathrm{Task}, \mathrm{Output})$$
用 GPT-4o mini 做 LLM-as-judge，对"agent 最终回复"与"期望答案"打对齐分（§4.1）。

**⑤ Tool Correctness（工具正确率，Eq 5）**——直觉：**"agent 调的每一次工具，是不是该调的那个？"**
$$\mathrm{ToolCorrectness} = \frac{\text{Num. Correct Calls}}{\text{Total Calls}}$$
把每次工具调用与"期望工具调用"比对，算正确比例（§4.1，同样引 Confident AI 2025）。

> **读出什么**：这套指标的巧思在于**把"记忆管理好不好"（①②③）和"任务做得好不好"（④⑤）解耦开报**。一个模型完全可以"工具删得很干净但答不对题"（如 Gemini 2.5 Flash 自主模式：移除率 90.5%、完成度只 65%），也可以"答得还行但工具堆成山"（如 Claude 3.5 Sonnet 自主模式：完成度 59%、残留 124）。**这种解耦正是本篇能得出"哪种模型配哪种模式"结论的前提**——见 §6。

> **一处诚实的批判（我的补充）**：④⑤ 都依赖 **GPT-4o mini 这一个裁判**，属于"用一个 harness 里的模型去评另一个 harness"，会引入裁判自身偏好的系统偏差（与库内标杆 Harness-Bench §14 "谁来 judge the judge" 同一隐忧）。原文**未给**裁判可靠性的人类校验或多裁判投票，这是外推时要打的折扣。

**Why（结果层）——为什么自主模式会"两极分化"？**
看自主模式（Table 1 上段 + Figure 2 的工具数轨迹）：
- **顶部（推理模型）**：GPT-o3（0.941）、Gemini 2.5 Pro（0.924）、Claude Opus 4（0.878）——移除率≈90%+、残留低（6–14）。**机制**：这些经过大规模 RL 推理训练的模型，能"看懂"系统提示里"工具无关就删"的指令、**主动**调 `Remove_Tools`；Figure 2 里它们的工具数曲线**上蹿后能压回来**（尖峰后紧跟移除），全程远低于 128。
- **底部（中小模型）**：Claude 3.5 Sonnet（**0.062**）、GPT-4.1 Nano（**0.000**）、LLaMA 3 70B（0.244）、GPT-4o Mini（0.449）——**几乎不删**。Figure 2 里它们的工具数**一路爬到 128 上限就贴着躺平**（作者原文：LLaMA 3 70B、Claude 3.5 Sonnet 在第 20 轮就顶到 128）。**机制**（§4.2.2 给了三条诊断）：① 它们**只顾答用户、忘了删**；② 若系统提示**不告诉它当前工具数**，它就意识不到工具已堆积（"若不含 current tool count，模型不删"）；③ 它们**误以为一次只能删一个**、不理解"可以一次删掉 100 个无关工具"（"do not comprehend that it can remove 100 irrelevant tools at a time"）。

**再看工作流/混合如何"抹平"（Table 1 中/下段 + Figure 3/4）**：
- **工作流**：**所有模型**移除率**>90%**（最低 GPT-4.1 Nano 也有 0.904），残留全在 7–9（Figure 3 所有曲线都压在 128 下方低位）。**因为"删"被外包给固定 prune 调用，模型删不删已经无所谓了**。
- **混合**：同样**所有模型 >86%**，残留 6–10（Figure 4）。
- **翻天覆地的对比**：**Claude 3.5 Sonnet 从自主模式的 0.062 → 工作流的 0.917**；**GPT-4.1 Nano 从 0.000 → 0.904**。**这就是 harness 给弱模型兜底的实锤**：同一个模型，只因把"删工具"从它手里拿走交给流程，短期记忆管理从"灾难"变"优秀"。

**反直觉的成本/轨迹观察（§4.2.1、§4.3.2）**：
- **不是"删得多就答得好"**：Gemini 2.5 Flash 自主模式移除率 90.5%（小模型里唯一 >90%），但**任务完成度只 65%**——它"太专注删工具/加工具，反而答不好题"。**记忆管理 ≠ 任务能力**。
- **工作流里"能力"成了区分点**：既然工作流下所有模型移除率都差不多（§4.3.2），**选哪个模型就看 Tool Correctness / Task Completion**——GPT-o3 工具正确率 88%、完成度 84% 领跑；成本敏感时可用 GPT-4.1 Mini / Gemini 2.5 Flash 当"记忆控制器"、另配强模型答题。

> **读出什么（Θ2）**：这张表是 `Agent = Model + Harness` 的**双向证据**。**横看一行不同模式**（同模型换 harness）：Claude 3.5 Sonnet 0.062→0.917，**harness 决定了这个模型能不能用**。**竖看一列同模式不同模型**（同 harness 换模型）：自主模式里 0.941（o3）vs 0.000（Nano），**model 决定了能不能给它自主权**。**结论落地成一句可操作的话**：**"推理模型 → 给自主/混合；中小模型 → 必须上工作流。"** 这正是本库反复强调的——不是 harness 单方面赢，而是**harness 的价值在于"给能力不足的模型补上确定性的地基"**（呼应 Harness-Bench §9"强模型更不挑 harness"）。

## §7　三种模式的推荐与适用场景（论文各模式 Recommendations 汇总）

> 主讲提示：这页是"给工程师的选型卡"。把三张 Recommendations 合成一张决策流程。

**自主模式，什么时候用（§3.1.1）**：
- 只在**用推理模型**时用（否则中小模型不删、爆上限）；
- 系统提示**必须**写清增删细则、**必须**注入"当前工具数"动态变量、**建议**用 128 上限错误去"催"模型删；
- 优点：**能回头再搜**（探索闭环完整）、额外 LLM 调用最少。

**工作流模式，什么时候用（§3.1.2、§4.3.2）**：
- **成本效率 + 可靠性优先**时的**首选**（simplicity 被作者点名推荐）；
- **连小模型都能用**（删被外包，不依赖模型能力）；
- 选型看下游能力：要工具正确率高 → GPT-o3 / GPT-4.1；要任务完成度 → Claude 3.5/3.7 Sonnet；要低成本记忆控制器 → GPT-4.1 Mini / Gemini 2.5 Flash；
- 缺点：agent 被"蒙眼"、**回不了头再搜**；**每轮固定多两次 LLM 调用**；无自我纠错。

**混合模式，什么时候用（§3.1.3、§4.4.2）**：
- 想要"**确定性的删 + 灵活的加**"时用——删托底、加放权；
- 对准模型能力的非对称性（不擅删、擅搜）；
- 隐患：agent 可能**搜太多工具触发上限**、引发工作流复杂化（靠补救剪枝兜底）；
- 前瞻：随模型变强可**过渡到自主模式**。

> **读出什么**：这三张卡合起来是一条清晰的选型决策——**先看模型档次**（推理 vs 中小）→ 推理模型再看"要不要确定性的删"（要→混合/工作流，不要→自主）；中小模型**直接工作流**。这就是本篇作为工业界（PwC）论文的落地气质：不追 SOTA，追**"在真实生产里该怎么配"**。

## §8　局限与批判（论文 Limitations + 我的补充）

**论文自陈的局限（§Limitations，诚实）**：
- **自主模式对非推理模型不可靠**：删工具这件事在中小模型上做不好；缓解办法是改用**确定性的工作流模式**。
- **工作流模式限制了探索**：它"constrains the agent's capacity to loop back and explore additional tools"——一旦初始化就回不了头；所以**混合模式**成为折中。
- **混合模式仍可能爆上限**：agent 探索太多工具会突破预设上限；缓解策略是**触发一个专门的 prune LLM**，但"这可能误删必要上下文"，暗示有时需要**临时退回自主模式**。
- **工作流/混合都需谨慎选模型**：本文靠给出 Tool Correctness / Task Completion 指标，帮使用者按模型强项（删 vs 搜 vs 答）来选型/混搭。

**我的补充批判**：
- **裁判单一**：任务完成度/工具正确率全靠 **GPT-4o mini 一个 LLM 裁判**，无人类校验、无多裁判投票（原文未给），评分可能带裁判偏好——与库内标杆 Harness-Bench §14 同一隐忧。
- **成本没进指标**：工作流/混合**每轮固定多花 1–2 次 LLM 调用**，这在 100 轮 × 生产规模下是真金白银；但论文的指标里**没有 token/成本/延迟维度**（对比 Harness-Bench 把 Tok/Turns 单列），无法回答"工作流的可靠性溢价值不值"。**原文未给出**成本对比。
- **上限 L=128 是人为对齐**：为统一所有模型把上限钉死在 128，但 Gemini/Claude 实际能挂 256–512（§4.1 自承）。**"爆上限"这个失败模式部分是被这个人为约束制造出来的**——放宽到 512，中小模型的"贴顶躺平"曲线会不会缓解？原文未测。
- **"相关/无关 query"的分布无 ground truth**：制造工具堆积的关键是"无关 query 引入新工具"，但 100 个 query 里相关/无关的比例、以及它是否代表真实用户行为，**原文未量化**。
- **Removal Ratio 的度量盲区**：GPT-4.1 Nano 自主模式**移除率 0.000、残留也 0.00**——因为它**既不加也不删**（退化行为）。此时"移除率=0"看似最差，但残留也最低。**单看移除率会误判**，必须和残留、完成度合看（这也反向印证了 §5 指标解耦的必要）。
- **外推性**：这是 2025-07 的模型快照（GPT-o3、Claude Opus 4、Gemini 2.5…）；"哪些模型配自主权"的名单会随模型迭代过期。作者的**方法（三模式 + 解耦指标）**比**具体排名**更耐久。

## ★ 对我们的启发（Inspires Us）

> 这一节是组会高潮，也是本库的独门优势：**我们（Claude Code / 本课 m9.* 的 agent）本身就是一个 harness，而且我们正大量使用 MCP + 一堆工具**——MemTool 处理的"多轮后工具堆积"痛点**几乎是照着我们的使用场景写的**。所以下面每条都能直接"打到自己身上"。

➤ **a. 可直接借用的招（method we can reuse）**：MemTool 最能拆下来即用的是**混合模式的分工哲学**——**"删（剪枝）交给确定性流程、加（检索）放权给模型"**，因为它对准了一个被实证的事实：**模型不擅长主动删工具、但擅长按需搜工具**（§3.1.3）。落到我们身上：给 Claude Code 的工具/MCP 层加一个**独立的"工具剪枝器"（一个只做删的无状态 LLM 调用或纯规则）**，在每个用户 turn 开始时，把上一 turn 里"已完成、本 turn 无关"的 MCP 工具从上下文里摘掉；而"要不要新挂某个 MCP"仍由主 agent 自己判断。**再加两条自主模式的实操 trick**（§3.1.1、§4.2.2，几乎零成本）：**(1) 把"当前工具数"作为动态变量写进系统提示**——作者实测"不写它，模型就不删"；**(2) 用工具上限错误去催删**——挂满时抛一个可读错误逼 agent 收缩。

➤ **b. 可迁移到我们课题的思路（transfer to our m9.\* / harness）**：把 MemTool 的**三种"增删决策权归属"当成一个可切换的开关**接到我们的 harness 上——不同任务档位用不同模式：**长程自治任务 + 强模型 → 自主**（保留探索）；**成本敏感 / 用弱子模型的批量子任务 → 工作流**（外包删、稳）；**默认 → 混合**（删托底、加放权）。**迁移时要改什么**：MemTool 的"工具"粒度是 MCP server；我们的粒度更杂（内置工具 + MCP + skill），剪枝器要能**按"本 turn 意图"给工具打相关性分**，而不是靠逐字名单匹配（Figure 7 那种逐字匹配在我们这里太脆）。**什么前提不再成立**：MemTool 假设工具定义是主要的上下文占用；我们的上下文里还有大量文件/历史，所以**"删工具"只是我们上下文压缩的一个子模块**，不能替代对话/文件的 compaction。

➤ **c. 它暴露的开放问题 = 我们的机会（open problem → opportunity）**：MemTool **只报了"删没删干净"和"答没答对"，完全没报成本/延迟**（§8 批判）——但工作流/混合**每轮固定多 1–2 次 LLM 调用**，这在我们这种长会话里是实打实的开销。**机会**：设计一个**"自适应剪枝触发器"**——不是每轮都跑剪枝器，而是**只在"工具数超过阈值 τ"或"上下文占用超过预算"时才触发**，把 MemTool 的"每轮固定剪枝"变成"按需剪枝"，在可靠性和成本间找平衡。**可下手的第一步**：在我们的 harness 里加一个工具计数器 + 一个阈值 τ（比如挂载工具数 > 30 或工具定义 token > 上下文的 15%），超阈值才调剪枝器，量化它相对"每轮剪枝"省了多少 LLM 调用、又不牺牲移除率。

➤ **d. 与本库其它论文/模块的连接（connect the dots）**：与库内标杆 **[Harness-Bench](2605.27922-harness-bench-measuring-harness-effects.md)** 强呼应——MemTool 的"同模型换模式，Claude 3.5 Sonnet 0.062→0.917"正是 Harness-Bench "Agent=Model+Harness、harness 给弱模型兜底"命题的一个**具体机制级案例**（Harness-Bench 是宏观统计，MemTool 是微观机制）；与 **D 组（Context/上下文，如 MemGPT / AgentFold / IterResearch）** 互补——它们压的是**对话/状态**这条记忆线，MemTool 压的是**工具**这条被漏掉的记忆线，二者拼起来才是完整的"上下文工程"；与 auto-research 库的 `m9.6`（评测沙箱）可对接——MemTool 的**解耦指标（移除率/残留/完成度/工具正确率）**可作为我们评 agent"工具管理"的现成 rubric。

➤ **e. 如果我来做下一步（my next move，第一人称、可执行）**：我会先在 Claude Code 的 MCP 工具层做一个**最小混合模式原型**——加一个"turn 级工具剪枝器"（规则版：把上一 turn 用过、本 turn 意图无关的 MCP 工具移出上下文）+ 在系统提示里注入"当前挂载工具数"，然后**在一个挂了 20+ MCP 工具的多轮会话上跑一遍**，用 MemTool 的 AvgRemovalRatio3T + 残留两个指标量化"工具堆积"降了多少；**若移除率能从'基本不删'抬到 >90%、且不掉任务完成度**，再上"自适应触发器"（c 条）把成本压下来。这条路的赌注很明确：**我们既然天天挂一堆 MCP，就该有人负责'卸载'，而 MemTool 已经把'谁来卸载'的三种答案摆在桌上了。**

## §9　版图定位（canon/前沿坐标 + 在本库的位置）

- **时间坐标（Θ4）**：**2025-07 前沿**，工业界（**PwC**）出品。它在"**短期工具记忆**"这条细分线上是**较早系统化的尝试之一**（§2.1 自述为该问题的 first study）。相对基石的推进：**RAG-for-tools（Toolshed / ScaleMCP / Graph RAG-Tool Fusion 等，同作者组 2024–2025）解决了"从海量工具里检索并装备相关工具"（即"加"）**；MemTool 补上了它们都没管的另一半——**"多轮里如何删、如何不堆积"**（§1、§2.2 反复强调 prior work "do not address the removal and management of tools... within a 100+ multi-turn session"）。
- **E/T/C/L/O/V 归属（Θ1）**：本篇坐 **T（工具接口）层为主、C（上下文/短期记忆）层为辅**——它管的是"工具集这个可变资源在多轮里的增删接口设计"，而工具即上下文，故与 C 层深度耦合。**不碰** L（循环骨架）/ V（评测协议）/ E（沙箱）本体。
- **回扣 Agent = Model + Harness（Θ2）**：贡献了一个**机制级双向证据**——同模式换模型（0.941 vs 0.000）证 Model 重要；同模型换模式（0.062→0.917）证 Harness 重要；**且揭示 harness 的具体作用是"把弱模型不擅长的删工具外包给确定性流程"**。
- **regime 诚实（Θ5，不绝对化）**：**"该给 agent 多少自主权"是分 regime 的**——**强推理模型 regime 下**，自主模式的灵活（能回头再搜）是净收益，harness 的确定性剪枝边际价值下降（甚至作者说混合可"过渡到自主"）；**中小模型 / 成本敏感 / 追求确定性的 regime 下**，工作流的"外包删工具"是刚需，harness 主导。**不能说"动态增删工具永远该用某一种模式"**——本篇的价值恰恰是给出了"什么模型/什么诉求 → 配哪种模式"的分界。
- **在本库的位置**：**C 组（工具接口）前沿样本**，且是**离我们自己 harness 最近的一篇之一**——因为我们正大量用 MCP + 工具，"动态增删工具"是可直接落地的工程点（见 Inspires-Us）。

## §10　组会讨论问题（留给大家吵）

1. MemTool 只报"删没删干净 + 答没答对"，**完全没报成本**。工作流每轮多 1–2 次 LLM 调用，在生产规模下这笔"可靠性溢价"到底值不值？你会怎么设计一个含成本的消融？
2. 混合模式赌的是"模型不擅删、擅搜"这个**能力非对称性**。随着模型变强（作者说混合可"过渡到自主"），这个非对称性会不会消失？如果消失，混合模式就没意义了吗？
3. 上限 `L=128` 是人为对齐（Gemini/Claude 实际能到 512）。**"爆上限"这个失败模式有多少是被这个人为约束制造出来的**？放宽上限，中小模型的"贴顶躺平"会缓解吗？
4. GPT-4.1 Nano 自主模式"移除率 0、残留也 0"（既不加也不删）——**单看移除率会把"退化"误判成"最差"**。你会怎么设计一个"既惩罚堆积、又惩罚不作为"的复合指标？
5. 把"工具剪枝"接到我们自己的 harness 上，**规则剪枝 vs LLM 剪枝**哪个更适合我们这种"内置工具+MCP+skill"的混合工具生态？逐字名单匹配（Figure 7）在我们这里为什么会脆？

## §11　一页速记

- **痛点**：多轮动态工具场景里，**"加工具"有人管（RAG 成熟）、"删工具"没人管**，工具越堆越多 → 爆上下文 / 撞 API 上限（128）/ 选择变差。现有短期记忆只压"对话文本"，漏了"工具"这条线。
- **三模式（核心=增删决策权归谁）**：**自主**（模型自己搜自己删，Alg 1，最灵活）｜**工作流**（两个固定 LLM 调用先删后加、agent 无权增删，Alg 2，最稳最省心）｜**混合**（删外包托底、加放权给 agent，Alg 3，对准"模型不擅删、擅搜"）。
- **设置**：ScaleMCP 5000 工具 × 100 轮 × 13+ 模型；工具上限 L=128；裁判 GPT-4o mini。
- **指标（解耦报）**：移除率(Eq1)/3轮滚动移除率(Eq2)/峰后残留(Eq3) 量**记忆管理**；任务完成度(Eq4)/工具正确率(Eq5) 量**能力**。
- **铁证（Table 1）**：**自主模式两极分化**——推理模型 90–94%（o3 0.941），中小模型 0–60%（Claude 3.5 Sonnet **0.062**、GPT-4.1 Nano **0.000**，工具贴 128 躺平）；**工作流/混合把差距抹平**——所有模型 >86–90%，Claude 3.5 Sonnet **0.062→0.917**、Nano **0.000→0.904**。
- **规律**：**推理模型才配拿自主权；中小模型必须靠工作流兜底**（harness 给弱模型补确定性地基）。删工具需注入"当前工具数"+上限错误催删（否则模型不删）。删得多≠答得好（记忆管理≠能力）。
- **诚实**：裁判单一、无成本维度、L=128 是人为对齐、移除率单看会误判退化；"给多少自主权"**分 regime**（强模型偏自主、中小/成本敏感偏工作流）。
- **对我们（我们正大量用 MCP+工具）**：搬混合模式的"删外包、加放权"给 Claude Code 工具层加"turn 级剪枝器"+注入工具数；再上"自适应触发器"按阈值剪枝省成本；用 MemTool 解耦指标量化工具堆积降了多少。
