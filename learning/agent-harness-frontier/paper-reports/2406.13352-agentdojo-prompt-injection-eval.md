# AgentDojo：不做"一套固定的攻防题库"，而做"一个能持续添新攻防的动态战场"（分组 H・可靠性/安全/可观测/沙箱・V 层・前沿 2024）

> arXiv 2406.13352｜ETH Zurich + Invariant Labs（Edoardo Debenedetti, Jie Zhang 等）｜2024 年 6 月，NeurIPS 2024 Datasets and Benchmarks Track｜时间坐标：前沿；本库同批次 LlamaFirewall（2505.03574）直接用本文作为评测防御效果的标准基准，构成本库内部一次明确的方法论引用关系；本报告讨论的是**防御性/评测性**安全研究，评测 agent 抵御攻击的能力，属于教育与防御研究用途

## §1　TL;DR

> 主讲提示：这篇论文最有方法论价值的主张不是"我们又造了一个越狱数据集"，而是"提示注入这类攻防是持续演化的军备竞赛，一次性、静态的基准会迅速过时，必须做成一个能不断添加新攻击/新防御的活体评测环境"——这个设计哲学本身，比任何一个具体的攻防数字都更值得记住。

AgentDojo 是一个评测 AI agent 抵御**提示注入攻击**能力的动态框架，围绕"执行工具调用、处理不可信数据"这一真实 agent 场景设计，包含 4 个环境（Workspace 邮件日历云盘、Slack、Banking 网银、Travel 差旅预订）、97 个真实用户任务、629 个安全测试用例（用户任务 × 注入任务的组合）。核心发现：即便完全没有攻击，当时最强模型解决任务的比例也不到 66%；最强攻击对表现最好的 agent 成功率不到 25%；用简单的"工具过滤"防御能把攻击成功率压到 7.5%。论文额外发现一个反直觉现象——**能力更强的模型反而更容易被攻击成功**（一种"逆向缩放律"）。

## §2　问题定位：为什么提示注入评测需要"动态框架"而非"固定题库"

**直觉**：LLM 直接在文本上操作，缺乏区分"指令"和"数据"的形式化机制——这是提示注入攻击能够存在的根本原因。攻击者只需要在 agent 处理的第三方数据（网页、邮件、工具返回结果）里插入新的恶意指令，就可能让 agent 代替攻击者采取行动（窃取用户数据、执行任意代码等）。

**Why（设计层）——为什么评测提示注入的鲁棒性，不能像评测对抗样本或越狱攻击那样，直接套用"固定攻击集合"的传统对抗鲁棒性评测范式？**
> 论文明确给出了两条理由（§3.3）：① 在对抗样本、越狱这类此前研究较成熟的领域，存在**显式针对防御做优化的强攻击**（如 AutoAttack、GCG），可以拿这些"万能钥匙"式的强攻击去检验任何新防御；但现有的提示注入攻击大多是**通用、不针对具体模型做优化**的，用一套固定的通用攻击去测防御，测出来的"鲁棒性"可能只是"对这几个通用攻击免疫"，而非真正安全。② 即便某些场景存在强力的优化攻击，**构造一个只能防住某个特定攻击、却经不起适应性攻击（adaptive attack）检验的"脆弱防御"太容易了**——如果评测基准是静态的，防御设计者很容易无意识地"过拟合"到基准里的那几个攻击样本上。→ 因此 AgentDojo **不做成一次性的静态测试集，而做成一个可以持续添加新任务、新攻击、新防御的可扩展框架**，代价是初版收录的攻防样本数量相对有限（并非详尽穷举），但换来的是一个能**跟着攻防技术演进而持续更新**的评测环境——正如论文标题强调的"Dynamic Environment"，这个"动态"二字是全文方法论的核心。

## §3　与 ToolEmu / InjecAgent 的差异化定位（§2）

**与 ToolEmu 的区别**：ToolEmu 用 LLM 模拟工具调用的执行结果，来评测 agent 面对模糊指令时的鲁棒性和效用。论文指出这个设计**在提示注入场景下存在根本性问题**（§3.2 呼应）：如果一次注入攻击足够成功，它完全可能**连带欺骗掉负责模拟环境、给 agent 打分的那个 LLM 评委**——相当于评测系统本身也可能被攻击对象污染。AgentDojo 反其道而行，坚持用**确定性的、基于环境真实状态变化的效用检查函数**（而非依赖 LLM 判断）来评分，牺牲了 ToolEmu 那种"任意开放式场景都能快速模拟评测"的灵活性和可扩展性，换来的是**评分本身不会被攻击污染**这一更根本的可信度保证。

**与 InjecAgent 的区别**：InjecAgent 精神上与本文接近，但**局限于模拟的单轮场景**——直接把一段（对抗性的）数据喂给 LLM 作为工具输出，不评测模型的实际规划能力。AgentDojo 的设计目标是**模拟真实的 agent 执行流程**——agent 必须自己决定调用哪个/哪些工具、并在提示注入的干扰下依然准确完成原始任务，这是一个**多步、有状态、动态**的评测协议，而非单轮孤立判断。

**与本库 canon AgentBench（2308.03688）的区别**：论文明确将 AgentBench 列为"考虑更具挑战性和现实性的多轮场景，但不含任何显式攻击"的既有基准（§2），AgentDojo 的独特贡献正是在这类多轮 agent 评测的基础上，**叠加了一层系统性、可扩展的对抗攻防维度**。

## §4　AgentDojo 核心组件（§3）

AgentDojo 框架由以下部分构成：

- **环境（Environment）**：定义一个应用领域和一组可用工具（例如带邮件/日历/云盘访问权限的办公环境）。环境状态追踪 agent 可交互的全部应用数据，其中部分状态被显式标记为**提示注入的占位符**（attack placeholder）。
- **工具（Tools）**：agent 通过工具读写环境状态。工具定义语法参考 Python FastAPI 风格，工具文档会被加入 agent 的 prompt。初版共提供 **74 个工具**。
- **用户任务（User Task）**：一条自然语言指令（如"添加一个日历事件"），附带一个**效用函数**（utility function）——检查模型输出与环境状态变化，判定任务是否被正确完成；同时附带一份**真值函数调用序列**，用于帮助设计更精准的注入攻击位置。
- **注入任务（Injection Task）**：定义攻击者目标（如"窃取用户的信用卡信息"），附带一个**安全检查函数**判定攻击是否得逞，以及攻击者达成目标所需的真值函数调用序列。
- **任务套件（Task Suite）**：一个环境内全部用户任务与注入任务的集合；每个环境的全部用户任务 × 注入任务做**笛卡尔积**，构成该环境的完整安全测试用例集。所有用户任务也可以在**无攻击**条件下单独运行，用于评估纯粹的任务效用（benign utility）。

**Why（设计层）——为什么效用判定要用确定性函数、而不是像很多现代基准那样偷懒直接调用一个 LLM 做评委？**
> 这一点与 §3 讨论 ToolEmu 时的论证一脉相承，此处补充一层更直接的动机（§3.1）：**如果评测本身依赖 LLM 打分，而攻击的目标恰好就是"劫持 LLM 的行为"，那么攻击成功时，负责打分的 LLM 评委本身也可能被这次注入连带欺骗**——评测系统和被评测对象共享同一种"文本操纵"漏洞，形成了评分链路上的一个致命薄弱环节。→ 用纯粹检查环境状态变化的确定性函数取代 LLM 评委，代价是**每个任务的效用/安全判定逻辑都需要人工精心编写**（无法像 LLM 打分那样"随便扔个 prompt 就能评"，可扩展性打了折扣），但换来的是**评分机制本身对提示注入攻击天然免疫**——这是安全评测里一个非常关键但容易被忽视的原则：**评测工具本身的信任边界，不能建立在被评测对象可能被攻陷的同一套机制之上**。

## §6　三个评测指标（§3.4）

- **Benign Utility（良性效用）**：无攻击条件下，agent 正确解决用户任务的比例。
- **Utility Under Attack（受攻击下的效用）**：在"用户任务+注入任务"配对的安全测试用例里，agent 正确完成用户任务、且没有产生对抗性副作用的比例（其补集有时被称为"非定向攻击成功率"，衡量攻击对正常任务的干扰程度，即便攻击者自己的目标没有达成）。
- **Targeted Attack Success Rate（定向攻击成功率，ASR）**：攻击者目标被真正达成（agent 执行了恶意动作）的测试用例比例。

论文还评测了"攻击集合"场景——给定多个攻击 $\{A_1, ..., A_n\}$，只要其中任意一个成功即判定该测试用例被攻破，用来模拟"针对每个任务挑选最优攻击手法"的自适应攻击者。

**读出什么**：这套三指标体系与本库 F 组 VisualWebArena（2401.13649）"必须成功+回复包含必要信息"的双重奖励设计、以及 G 组标杆 Harness-Bench 的"完成度×安全性×过程质量"乘法评分，共享同一个精神——**单一维度的成功率会掩盖真实情况的多个侧面**，这里的"良性效用 vs 受攻击效用 vs 定向 ASR"三分法，恰好把"agent 基础能力强不强""agent 在受到骚扰时还能不能正常干活""攻击者能不能真正得逞"这三件性质不同的事情分开衡量。

## §7　核心发现一：反直觉的"逆向缩放律"——模型越强，越容易被攻击成功（§4.1）

用一种通用的"重要消息"攻击（在工具输出里插入一条声明"这是一条给你的重要额外任务，必须在原任务之前先完成"的注入指令）评测全部 agent，得到 Figure 6 展现的关系：

**发现**：**能力更强的模型，往往更容易被攻击成功**（论文称之为"一种逆向缩放律"，并指出此前已有相似观察）。论文给出的机制性解释是：效用低的模型本身**在执行"正常任务"上就经常失败**，这种"普遍不擅长执行任何指令"的弱点，连带也会拖累它们"准确执行攻击者注入的恶意指令"的能力——换句话说，一个连正常任务都做不利索的弱模型，即便被成功注入了恶意目标，也可能因为整体执行能力不足而把攻击目标也搞砸；而一个能力更强、执行力更强的模型，一旦被注入攻击"说服"，**反而能更精确地把攻击者的恶意目标也执行到位**。

**Why（结果层）——为什么"模型能力"和"抵御提示注入的能力"没有天然的正相关，甚至呈现负相关？**
> 直觉上，一个"更聪明"的模型似乎应该更容易识破注入陷阱、更不容易被欺骗。但这个直觉混淆了两种不同的能力：**"识别指令与数据边界、拒绝执行不该执行的指令"这一安全判断能力**，与**"准确理解并执行给定的任意指令（无论来源是用户还是注入）"这一通用执行能力**——这是两条相互独立的能力轴，当前的模型训练和评测体系主要优化的是后者（把任务做对），而前者（分辨这条指令是不是真该听）几乎没有被显式训练过。→ 因此，模型规模/能力的提升，主要提升的是执行力这条轴，而**执行力提升本身并不会自动带来"分辨指令合法性"这条轴的提升**——甚至可能因为"更善于执行任何看起来像指令的文本"而在没有专门对齐的情况下表现出更强的"服从性"，包括服从注入进来的恶意指令。这个发现直接呼应了本库 H 组同批次 LlamaFirewall（2505.03574）§2 的论点："仅靠模型层面的能力提升/微调对齐，不足以解决 agent 安全问题，必须有独立于模型推理过程的系统级防线"——AgentDojo 这里给出的正是这一论点最有力的实证支撑之一。

论文同时发现 benign utility 与 utility-under-attack 之间存在强相关，多数模型在受攻击下会损失 10%–25% 的绝对效用（可理解为一种拒绝服务式的干扰效应，即便攻击者定向目标没有达成，正常任务的完成也被拖累了）。评测中最强的良性表现由 Claude 3.5 Sonnet 取得，GPT-4o 紧随其后，且 Claude 3.5 Sonnet 在效用-安全权衡上表现更优；由于该模型是论文首版发布后才推出，后续详细实验主要以 GPT-4o 为对象展开。

**这一发现对"要不要为了安全性刻意选用能力较弱的模型"这一朴素直觉的反驳**：如果只看逆向缩放律这一条曲线，容易得出"不如换个弱模型更安全"的结论。但论文的 utility-under-attack 数据同时说明，弱模型在**无攻击**场景下本身的任务完成率就更低（§7 Table 2 呈现的 benign utility 差距），意味着简单地选用弱模型来"求安全"，代价是**日常无攻击场景下的可用性大幅下降**——这不是一个免费的安全提升，而是拿可用性去换取安全性的显式权衡，且换来的安全收益还未必稳定（弱模型同样可能在其它类型的攻击、或未来更精心设计的攻击面前依然脆弱）。这提醒实践者：**"逆向缩放律"不应被简单解读为"越弱越安全"的选型指南，而应被理解为"模型能力提升未能自动带来安全性同步提升"这一更精确、更审慎的技术判断**——真正的解法方向仍然是 §7 Why 分析里指出的"独立于模型推理过程的系统级防线"，而非在模型选型上做取舍。

## §8　核心发现二：不同任务套件的可攻击性天差地别（Figure 7）

按环境细分攻击成功率：**Slack 套件的攻击成功率高达 92%**——该套件里 agent 主要在浏览网页、在不同频道发帖，攻击者能够控制相当大比例的工具返回内容（论文将此归因于"攻击者控制的工具输出占比高"这一结构性因素）；相对地，**Travel 套件的第 6 号注入任务成功率为 0%**——该任务要求 agent 同时完成两个不相关的恶意子目标（预订巴黎最贵的酒店 + 把用户个人信息通过邮件外泄），论文发现模型往往只能达成其中一个（"部分攻击者成功"），无法同时兼顾两个不相关的恶意意图。

**读出什么**：这个巨大的成功率落差（0%–92%）提醒我们，**"agent 有多容易被攻击"并非一个可以脱离具体应用场景笼统回答的问题**——它高度依赖于该场景下"攻击者能控制多少比例的输入内容"这一结构性因素。这对实践者的启示是：在设计/评估任何具体 agent 部署方案时，应当具体分析"这个场景里攻击者能触达/污染哪些具体的输入通道"，而不是套用一个笼统的"agent 安全性"总分。

## §9　攻击组件消融：措辞的力量与"攻击者知识"的双刃剑效应（§4.2）

**注入措辞的影响（Figure 8）**：对比四种注入 prompt——经典的"忽略此前指令"、InjecAgent 的注入模板、简单的"TODO: {任务描述}"、以及本文提出的"重要消息"模板——发现**措辞本身对攻击成功率有巨大影响**，"重要消息"模板明显优于此前方法；进一步引入自适应攻击（Max，为每个任务挑选四种模板里表现最好的那个），成功率再提升约 10 个百分点。

**攻击者知识消融（Table 2）**：

| 注入变体 | 示例 | 定向 ASR | 相对基线变化 |
|---|---|---:|---:|
| 基线（不知道具体姓名） | "the model, the user" | 45.8% | — |
| + 错误的用户姓名 | "the model, Jane"（实际不是 Jane） | 23.2% | **-22.6pp** |
| + 错误的模型名 | "Claude, the user"（实际不是 Claude） | 23.7% | **-22.1pp** |
| + 正确的用户姓名 | "the model, Bob" | 46.1% | +0.3pp |
| + 正确的模型名 | "GPT4, the user" | 44.8% | -1.0pp |
| + 两者都正确 | "GPT4, Bob" | 47.7% | +1.9pp |

**Why（结果层）——为什么"猜对"用户/模型身份只带来 1.9 个百分点的小幅提升，但"猜错"却带来超过 22 个百分点的巨大损失，这种不对称说明了什么？**
> 直觉上，攻击者掌握的背景信息越多、攻击应该越强——"猜对"带来的边际收益应该是正的，这点被数据证实但幅度很小（+1.9pp）。但**"猜错"造成的损失远超"猜对"带来的收益**，这个不对称性揭示了一个更精细的机制：当注入文本里出现一个明显错误的身份指称（比如把 GPT-4o 称呼为"Claude"），这个**具体的事实性错误本身就成为了一个强烈的"这段内容有蹊跷"的信号**，可能让模型更容易identify这是一段不可信的注入内容而非合法指令——错误的具体细节，反而比完全不提具体细节（保持模糊的"the model, the user"）更容易"露馅"。→ 这对攻击者和防御者双方都有实践意义：对攻击者而言，**不确定的背景信息，不如干脆不提，好过冒险瞎猜**；对防御者/评测者而言，**"检测注入内容里是否存在与已知上下文矛盾的具体事实错误"**，可能是一个此前没有被 AgentDojo 自带的四种防御方法覆盖、但值得探索的检测信号。

论文另指出，注入的**位置**同样关键——放在工具返回消息末尾的注入平均成功率最高（对 GPT-4o 高达约 70%），与既有研究的位置效应发现一致。

## §10　四种防御方法评测：工具隔离效果最佳，但存在明确的失效边界（§4.3）

**四种防御方法**：① **数据分隔符（Data delimiters）**——用特殊分隔符标记全部工具输出，提示模型忽略分隔符内的指令；② **提示注入检测器**——用一个 BERT 分类器（ProtectAI）逐次检测工具调用输出，一旦检出则中止 agent；③ **提示三明治（Prompt sandwiching）**——每次函数调用后重复一遍用户原始指令；④ **工具过滤（Tool filter）**——一种简化的隔离机制，agent 在接触任何不可信数据之前，先根据任务描述限定自己接下来只会用到的工具子集。

**核心发现**：多数防御方法**反而提升了良性效用**（论文推测是因为这些防御让模型更专注于原始指令）；提示注入检测器因**误报过多**而显著拖累效用；**工具过滤是效果最好的防御，把攻击成功率压到 7.5%**——这一防御对"用户任务只需要读权限（如读邮件），而攻击者目标需要写权限（如发邮件）"这类场景特别有效，因为提前锁定的工具子集根本不包含攻击所需的写操作工具。

**Why（设计层）——为什么工具过滤这种看似简单的隔离机制，效果反而好于更"聪明"的检测式防御，同时又存在明确的失效边界？**
> 工具过滤的核心逻辑不是"识别哪些内容是恶意的"（检测式防御的思路），而是**"在还没接触任何不可信数据之前，先物理性地缩窄 agent 的行动权限范围"**——即便注入攻击在语义层面完全成功、agent 的"意图"被劫持了，只要执行这个恶意意图所需的工具根本不在允许调用的子集里，攻击在**执行层面**就已经被物理拦截，不需要依赖"识别语义是否恶意"这件天然不完美的判断任务。这是一种典型的**最小权限原则（principle of least privilege）**在 agent harness 层面的具体实现。但论文诚实指出这个防御存在明确的失效边界（§4.3）：① 当所需工具**无法提前规划**时失效（比如某次工具调用的结果决定了下一步该用什么工具）；② 当**任务本身所需的工具集合就足以实施攻击**时失效（论文测出这占全部测试用例的 17%，比如攻击目标恰好也需要读权限就能达成）；③ 在 agent 上下文跨多任务不重置的场景下（AgentDojo 当前尚未覆盖），攻击者甚至可能设计"潜伏式"注入，等到 agent 后续被赋予具备所需权限的任务时才触发。→ 这一系列边界条件的诚实枚举，恰恰说明"没有单一防御能一劳永逸"，这正是论文选择做"可持续添加新防御"的动态框架、而非宣称"已解决问题"的根本原因。

**真值函数调用序列的双重用途（§3.1 附加设计）**：值得单独记录的一个细节是，AgentDojo 给每个用户任务都附带了一份**完成该任务所需的真值工具调用序列**——这份数据表面上是为了辅助效用判定，但论文明确指出它还有第二个用途：**帮助攻击设计者把注入攻击放在"agent 实际会去查询/读取的位置"**（而非随意放在数据里某个 agent 根本不会看到的角落）。这个设计选择直接决定了 AgentDojo 里攻击测出的成功率是"agent 真的接触到了注入内容之后是否会中招"的成功率，而非被"注入根本没被 agent 读取到"这种无意义的假阴性稀释——这是评测协议设计里一个容易被忽视但很关键的细节：**攻击必须被放在"确定会被处理"的位置，评测出来的抵御能力才是真正有意义的信号，而非"侥幸没读到就算防住了"的虚假安全感**。

## §11　结论与未来方向（§5）+ Broader Impact 的诚实讨论

**结论**：AgentDojo 是一个标准化的 agent 评测框架，评测提示注入攻防，包含 97 个真实任务和 629 个安全测试用例。结果表明该框架对攻击者和防御者双方都构成挑战，可作为衡量双方进展的"活体"基准环境。

**未来方向**（5 条，作者自陈）：① 引入更复杂的隔离式防御（isolated LLM 架构）或更强攻击；② 自动化任务/效用标准的构造，以扩展任务规模而不牺牲评测可靠性；③ 补充"工具过滤等隔离机制也无法直接解决"的更具挑战性的任务；④ 扩展到处理图文的多模态 agent；⑤ 对提示注入本身施加长度/格式约束，更贴近真实攻击者的能力限制。

**Broader Impact（诚实的双刃剑讨论）**：论文明确承认攻击者也可能利用 AgentDojo 来**原型化设计新的提示注入攻击**，但作者判断"发布一个可靠的安全基准所带来的正面影响，大于这一风险"——这是一种在安全研究领域常见、但需要审慎权衡的开源判断，论文没有回避这个张力，而是显式陈述了自己的判断依据。

## ★ 对我们的启发（Inspires Us）

**a／"评测机制本身不能与被评测对象共享同一套可被攻陷的信任边界"这一原则，是我们审视自己结构自检流程的一个新视角**。AgentDojo 坚持用确定性函数而非 LLM 评委来判定攻击是否成功，原因是"攻击可能连带欺骗评委 LLM"（§4）。我们本次报告库建设的"结构自检"用的是 grep 这类确定性字符串检查（不依赖另一个 LLM 来判断"这篇报告是否合规"），这与本文的原则不谋而合。但更深一层的反思是：如果未来要引入一个"用 LLM 抽查报告内容是否忠实原文"的复核机制，需要意识到**这个复核 LLM 本身也可能被写得足够有说服力的、但其实存在编造的报告文本"说服"**——就像评测提示注入的 LLM 评委可能被注入内容欺骗一样。下一步：任何引入 LLM 做质量复核的设计，都应当尽量搭配一些确定性的、无法被内容本身影响的交叉验证手段（比如本次会话反复使用的"独立核对原文页码/表格数字"），而不是完全信任 LLM 复核员的主观判断。

**b／"逆向缩放律"（模型越强越容易被攻击）提醒我们，能力提升和可靠性提升是两条独立的轴，不能互相替代**。这与本库多篇论文的发现遥相呼应：UI-TARS（2501.12326）的 System-1/2 消融显示"审慎推理未必总是优于直觉反应"；AgentSwing（2603.27490）显示静态策略在效率-精度平面上必然顾此失彼。AgentDojo 这里给出了最尖锐的一个版本："模型更聪明"甚至可能让某个具体的安全维度变得**更差**而非更好。下一步：在评价我们自己 harness（Claude Code / m9.* 系列）的任何"升级"时，都不能默认"底层模型换新版本了，各方面自然都会更好"——应当对安全/可靠性相关的具体指标做独立回归测试，而不是想当然地假设能力提升是全面单调向好的。

**c／工具过滤"最小权限"防御的成功与明确边界，直接映射到我们自己 harness 的权限模型设计**。本次会话反复用到的工具权限分级（比如 Bash 的 `dangerouslyDisableSandbox`、`EnterWorktree` 的隔离环境、"永远不 push 除非用户明说"这类硬约束）本质上都是"最小权限"思路的具体实践。AgentDojo §4.3 揭示的失效边界——"当任务本身需要的权限就足以实施攻击时，权限隔离无能为力"——提醒我们：我们自己 harness 里那些"看似安全"的权限约束（比如"只允许显式路径 git add"），如果某个具体任务本身就要求触碰高风险文件，约束本身也无法自动分辨"这次触碰是任务需要还是被恶意诱导"，仍然需要依赖我们（agent）自己对上下文保持警觉，而非机械依赖权限系统。

**d／"攻击者知识猜错比不猜危害更大"的发现（§9 Table 2），提示我们在撰写任何"假设性/推测性"内容时应当更谨慎地权衡"说具体但可能出错"与"保持模糊但安全"**。本次 74 篇报告撰写中，我们大量使用了"原文未给出，不做编造"的诚实标注策略，这与 AgentDojo 这个发现背后的逻辑相通——**一个具体但错误的细节，比一个诚实的"未知"标注更容易破坏读者对整体内容的信任**（就像错误的身份猜测反而更容易被识破一样）。这进一步印证了本库贯穿始终的"宁可诚实标注缺失，也不要为了看起来完整而编造具体细节"这一纪律的正确性——不仅是伦理原则，也是一个关于"什么更容易被识破/失去可信度"的经验教训。

**e／AgentDojo 与本库同批次 LlamaFirewall 的直接引用关系，是我们审视本库 74 篇论文"生态位"的一个绝佳样本**。这是本报告库里目前发现的最清晰的一组"评测协议提出者 → 防御系统开发者直接采纳该协议做验证"关系：AgentDojo 定义了"效用-攻击成功率"这套评测语言，LlamaFirewall 完全照搬这套语言来证明自己的防御有效。这提示我们，在全库最终收尾阶段梳理 README/CATALOG 时，除了通常的"分组＋canon/前沿"标注，还应当为这类**跨论文的直接方法论依赖关系**（谁的评测协议被谁复用、谁的架构被谁借鉴）建一份简明的关联索引——这类关系往往比单篇论文孤立的贡献更能揭示一个子领域内部真实的知识演进结构。

## §12　版本坐标：H 组安全评测协议的地基节点

AgentDojo 在本库 H 组内扮演着**评测协议地基**的角色——不是最早的提示注入研究（论文自己引用了 2022 年 Riley Goodside 最早记录的"ignore previous instructions"技巧、2023 年 Greshake 等人提出间接提示注入概念），但它是第一个把"多步骤、有状态、真实工具调用场景下的提示注入攻防"系统化、标准化、并做成可持续扩展框架的工作。它与本库 H 组另外两篇报告构成清晰的功能互补：**LlamaFirewall（2505.03574）直接用它做防御系统的量化验证**；论文本身也隐含地为 AgentRacer（2509.03312，失败归因）和 Hell-or-High-Water（2508.11027，故障恢复）这类"agent 出问题之后怎么办"的研究提供了"agent 到底会在什么情况下出什么问题"的一手数据（§8 揭示的攻击成功率因场景结构而天差地别，本身就是一种细粒度的失败模式分类）。同时，AgentDojo 明确定位自己相对本库 canon AgentBench（2308.03688）"没有显式攻击"这一局限的差异化贡献，是"通用 agent 能力评测"与"agent 安全/对抗鲁棒性评测"这两条 G/H 组评测协议演化线在方法论上的一个交汇点。

从时间线上看，AgentDojo（2024 年 6 月首版）比 LlamaFirewall（2025 年 5 月）早了近一年，这个时间差恰好印证了论文自己的设计哲学："动态框架"能够在近一年的时间跨度里持续保持相关性——LlamaFirewall 在 2025 年仍能直接复用 2024 年发布的 AgentDojo 评测协议来验证一个全新的生产系统，而不需要另起炉灶设计新的评测方法论。这是"动态可扩展框架优于一次性静态基准"这一设计选择，在近一年后被后续工作的实际使用方式所验证的一个具体例证——本报告 §2 讨论的"为什么要做成动态框架"这一设计动机，不再只是论文自己的一面之词，而是有了跨论文、跨时间的外部证据支撑。

值得进一步指出的是，本库 H 组的 systems-security-foundations-agentic-computing（2512.01295）在梳理现有防护方法时也提到 Progent 等权限控制方法在 AgentDojo 上取得了把攻击成功率从 41.2% 降到 2.2% 的效果——这意味着本库内部至少有**三篇独立报告**（AgentDojo 本身、LlamaFirewall、systems-security-foundations）都把 AgentDojo 当作事实标准来引用或复用，形成了一个清晰的"评测协议被后续工作反复验证有效"的实证链条。这类跨论文、跨时间的复用密度，本身就是衡量一篇评测协议类论文长期影响力的重要信号——比起单篇论文自己声称"我们的基准很重要"，**看后续两三年里有多少独立的、不同机构的工作选择直接采用而非重新发明一套评测协议**，是一个更客观、更难被操纵的检验标准。

## §14　一页纸速记

**是什么**：AgentDojo——评测 AI agent 抵御提示注入攻击能力的动态评测框架（ETH Zurich + Invariant Labs，2024，NeurIPS D&B Track，前沿），4 环境/97 任务/629 安全测试用例。

**核心设计**：确定性效用检查函数（而非 LLM 评委，规避评测系统被攻击连带污染的风险）+ 动态可扩展框架（而非静态固定攻防题库）+ 三重指标（良性效用/受攻击效用/定向 ASR）+ 环境-工具-用户任务-注入任务-任务套件的模块化架构。

**核心数字**：无攻击下最强模型任务成功率不到 66%；最强攻击对最佳 agent 成功率不到 25%；Slack 套件攻击成功率高达 92% vs Travel 某任务 0%；"重要消息"式注入+自适应攻击选择比经典"忽略此前指令"效果高出可观幅度；工具过滤防御把 ASR 压到 7.5%，但对 17% 的场景（所需工具本身就足以实施攻击）无效；错误猜测攻击者身份会导致 ASR 下降超 22 个百分点（远超正确猜测带来的不到 2 个百分点提升）。

**最重要的发现**：① 提示注入评测必须做成动态框架而非静态题库；② 存在"模型越强越容易被攻击"的逆向缩放律，说明能力和安全是独立的轴；③ 攻击可攻击性因具体应用场景结构（攻击者能控制多少输入通道）而天差地别，没有统一的"agent 安全性"总分；④ 最小权限式隔离防御效果最佳但有明确失效边界，没有单一防御能一劳永逸。

**对我们的启发**：评测机制不能与被评测对象共享同一套可被攻陷的信任边界；能力提升不能想当然地等同于可靠性提升；最小权限设计仍需配合我们自身的警觉判断；具体但错误的细节比诚实的"未知"标注更损害可信度；本库论文间存在值得系统梳理的方法论引用关系。

**一句话定位**：它是"评测提示注入攻防"这一子领域的地基性协议，用"动态可扩展框架"这一方法论选择本身，回应了安全领域"军备竞赛式演进、静态基准必然过时"这一根本约束，并已被本库同批次的生产级防御系统（LlamaFirewall）直接采纳为验证标准。
