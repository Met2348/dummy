# UI-TARS：把 GUI Agent 从"框架"炼成"模型"（分组 F・Web/GUI Agent・E/T 层・前沿 2025）

> arXiv 2501.12326｜ByteDance Seed + 清华大学（Yujia Qin, Yining Ye 等）｜2025 年 1 月｜时间坐标：前沿，本库 F 组内 canon 节点 Mind2Web/WebArena/VisualWebArena 之后的原生端到端路线代表作

## §1　TL;DR

> 主讲提示：这篇论文的核心主张浓缩成一句话——"你不需要一个用提示词精心编排出来的 GPT-4o 工作流，你需要一个直接在数百万条 GUI 交互轨迹上训练出来的原生模型"。这是本库"Agent = Model + Harness"论点里少见的一篇**主张把 harness 的职责收编进 model 内部**的论文，值得和其它强调"harness 独立价值"的论文对照着读。

UI-TARS 是一个**只吃屏幕截图、输出类人操作**（键盘/鼠标）的原生 GUI agent 模型，在 Qwen-2-VL 7B/72B 基础上用约 500 亿 token 持续训练而成。核心数字（摘要）：OSWorld 上 50 步预算得分 24.6、15 步预算 22.7，均超过 Claude Computer-Use 的 22.0/14.9；AndroidWorld 上 46.6，超过 GPT-4o（+Aria-UI 框架）的 34.5。论文的方法论贡献分四块：**增强感知**（五类 GUI 专属数据任务）、**统一动作建模**（跨平台标准化动作空间）、**System-2 推理注入**（"思考"先于"动作"）、**基于在线轨迹的迭代精炼**（数百台虚拟机自动采集+反思调优+DPO）。

## §2　问题定位：Agent Framework 还是 Agent Model？

**直觉**：2023–2024 年主流 GUI agent 方案是"框架式"——用 GPT-4o 这类商用模型做推理核心，外挂专门的 grounding 模块、记忆模块、prompt 工程，多个组件拼装成一条流水线。这类方案能快速搭出 demo，但论文认为它存在结构性弱点。

**Θ2 关联（Agent=Model+Harness 论点的一次"反向"案例）**：UI-TARS 明确论证的是"框架式 harness 的收益有上限，原生端到端模型能超越精心设计的 harness"——摘要直接写"UI-TARS is an end-to-end model that outperforms these sophisticated frameworks"。这与本库标杆 Harness-Bench（2605.27922）"harness 差异能带来 23.8 分的巨大波动"看似矛盾，实则互补：**Harness-Bench 测的是"同一个模型换不同 harness"的方差；UI-TARS 论证的是"把 harness 该做的事，用数据训练进模型参数里"这条路线本身能跑赢"模型不变、只优化 harness"这条路线**。两篇论文合起来给出了一个更完整的图景：harness 工程和端到端训练是解决同一问题的两条互补路径，谁更优取决于数据规模、任务稳定性、以及你是否有能力做规模化训练。

**Why（问题层）——为什么框架式方案会有"结构性弱点"，而不仅仅是"暂时做得不够好"？**
> 论文列出框架式方案的三条内生缺陷（§2.2）：① **脆弱性与维护开销**——任务/界面一变，人工编写的 prompt/规则就要跟着重写，容易出错且耗费人力；② **学习范式割裂**——框架方法很少把新经验数据反馈回底层 LLM/VLM 参数，只能靠离线的 prompt 工程调整，一旦任务偏离原始设计域就容易失效；③ **模块不兼容**——复杂任务需要多个模块（视觉解析、记忆存储、长程规划）协同，任何一个模块的不一致或错误都可能拖垮整条流水线，且排查问题通常需要领域专家介入。→ 这三条缺陷的共同根源是：**框架式方案是"设计驱动"的（design-driven），依赖开发者预见并手工编码所有未来变化**；而原生模型是"数据驱动"的（data-driven），可以通过大规模数据和迭代反馈自动学习、自动适应。代价是原生模型路线需要一开始就解决"上哪儿找这么多高质量交互轨迹数据"这个更难的问题（这正是全文后半部分 §8-9 要解决的核心难题），但换来的是**不需要为每一种新场景重新设计 workflow**——一次训练，跨场景泛化。

## §3　GUI Agent 演化四阶段（§2）

论文把 GUI agent 的发展史划分为四个阶段，划分依据是"需要多少人工介入"与"泛化能力有多强"：

| 阶段 | 名称 | 特征 | 代表 |
|---|---|---|---|
| Stage 1 | 规则式 agent（RPA） | 靠预定义规则匹配指令、调用 API；无法从环境或历史经验中学习；需要 API/系统权限访问 | DART、WoB、Roscript、FLIN |
| Stage 2 | Agent 框架 | 用基础模型（GPT-4/GPT-4o）做推理核心，外挂记忆/grounding 等模块，靠 prompt 工程与工作流设计增强能力 | AutoGPT、LangChain、Cradle、SeeAct、AWM |
| Stage 3 | 原生 Agent 模型 | 感知、推理、记忆、动作统一进单一端到端模型，工作流知识内嵌在模型参数中，数据驱动 | Claude Computer-Use、Aguvis、ShowUI、OS-Atlas、**UI-TARS（本文）** |
| Stage 4 | 主动/终身 agent（展望） | agent 主动提出任务、执行、自我评估、自我奖励，持续自主学习，最小化人工介入 | 尚属前瞻方向 |

**Why（设计层）——为什么要把演化史分成四个阶段，而不是简单说"我们比之前的方法更好"？**
> 朴素做法是像多数论文一样，直接对比几个最新 baseline 证明自己更强。→ 但这样读者只能看到"点"的优劣，看不到"线"的方向。→ 论文选择先建立一个**以"人工介入程度"和"泛化能力"为轴的演化框架**，代价是要花大量篇幅（§2 整节）做文献综述式的铺垫，但换来的是能够**把 UI-TARS 自身定位在这条演化线的具体坐标上**（Stage 3，且主动指向 Stage 4），并让读者理解"为什么原生模型是下一步"而不只是"这个模型这次跑分更高"——这是一种**用历史演化逻辑而非单纯 benchmark 数字来论证技术路线正确性**的写作策略，在工业界技术报告中比学术论文更常见。

## §4　原生 Agent 模型的四大核心能力（§3.1）

论文把原生 agent 模型应当具备的能力拆成四块，UI-TARS 的每一个技术模块都能对应回其中一块：

1. **感知（Perception）**：不仅要"看懂"静态截图，还要动态跟踪界面随时间的变化（如加载动画）。论文明确指出：文本化表征（HTML/DOM/accessibility tree）存在隐私顾虑和平台特定噪声，**原生模型应像人一样直接通过视觉感知外部环境**，这是"纯视觉输入"这一设计选择的正式论证（对应 §6）。
2. **动作（Action）**：需要统一、多样、可跨平台迁移的动作空间；核心难点是坐标 grounding——不同设备分辨率/宽高比不同，精确定位元素坐标本身就是一个需要专门优化的子问题（对应 §7）。
3. **推理（System 1 & 2 思维）**：借用心理学的双系统理论——System 1 是快速、直觉式的反应（如点击熟悉的按钮），System 2 是缓慢、审慎的分析式思考（如规划整体工作流、遇错反思）。System 2 又进一步拆成五种具体推理模式：任务分解、长程一致性、里程碑识别、试错、反思（对应 §8）。
4. **记忆（Memory）**：短期记忆（当前任务的动作历史与执行轨迹）+ 长期记忆（跨任务的经验积累）。论文特别指出：**原生模型与框架式方案的关键区别在于，原生模型把长期经验编码进内部参数（隐式、参数化存储），而不是靠外挂的显式记忆模块**（对应 §9）。

## §5　架构形式化：从 ReAct 式 (o,a) 到 UI-TARS 式 (o,t,a)

**符号与形式化**（§4.1）：

$$(\text{instruction}, (o_1, a_1), (o_2, a_2), \ldots, (o_n, a_n)) \quad (1)$$

其中 $o_i$ 是第 $i$ 步的设备截图观测，$a_i$ 是 agent 执行的动作。UI-TARS 在每个动作前插入一个显式的"思考" $t_i$，把序列升级为：

$$(\text{instruction}, (o_1, t_1, a_1), (o_2, t_2, a_2), \ldots, (o_n, t_n, a_n)) \quad (2)$$

模型在给定指令、历史交互、当前观测的条件下，迭代式地预测下一步的思考与动作：

$$P(t_n, a_n \mid \text{instruction}, t_1, a_1, \ldots, (o_{n-i}, t_{n-i}, a_{n-i})_{i=1}^{N}, o_n) \quad (3)$$

为控制 token 预算（论文提到典型 32k 序列长度约束），只保留最近 $N$ 步观测作为输入上下文，但完整的历史动作与思考仍作为短期记忆保留。

**读出什么**：这个形式化直接说明 UI-TARS 的"System-2 推理"不是一个独立的规划模块，而是**内嵌进每一步的自回归生成序列里**——模型在吐出动作 $a_n$ 之前，先吐出一段思考 $t_n$，两者由同一个模型、同一次前向计算生成。原文明确将其与 ReAct（Yao et al., 2023）对比："受 ReAct 启发，但我们的思考整合更结构化、更有目标导向"——ReAct 交替"推理-行动"，UI-TARS 把这个交替循环**直接烧进训练数据和模型权重里**，而不是靠 prompt 模板临时拼接。

## §6　感知能力增强：五类 GUI 专属数据任务（§4.2）

论文指出 GUI 感知面临两个独特挑战：① **截图数据稀缺**（相比通用场景图片，GUI 专属截图少得多）；② **信息密度高、精度要求高**（一张 1920×1080 的截图里可能有几百个元素，其中不少是 10×10 像素的小图标）。为此构建了五类感知训练任务（数据来自"截图+元数据"的结构化采集，采用**自下而上**的构建策略——先学会理解单个元素，再逐步整合到整体界面理解）：

1. **元素描述（Element Description）**：对每个可见元素生成结构化描述，覆盖元素类型、视觉外观、空间位置、功能四个维度。
2. **密集字幕（Dense Captioning）**：把所有元素描述与图片说明整合成一段连贯、详尽、保留 GUI 布局结构的整体描述，训练时只给图片、要求模型输出对应字幕（强调"最小化幻觉"）。
3. **状态转移字幕（State Transition Captioning）**：给一对连续截图，训练模型识别并描述两者的差异，判断是否发生了用户动作（也包含非交互式变化，如动画、后台更新）。
4. **问答（QA）**：覆盖界面理解、图片解读、元素识别、关系推理，把感知与推理能力结合起来训练。
5. **Set-of-Marks（SoM）**：借鉴本库同组 VisualWebArena（2401.13649）验证过的 SoM 提示法，为解析出的元素画上带形状/颜色/大小变化的标记，训练模型将标记与对应元素关联。

**Why（设计层）——为什么要设计五种不同的感知任务，而不是直接训练"看图选元素"这一个最终任务？**
> 朴素做法是既然最终目标是"根据截图预测正确动作"，直接端到端训练"截图→动作"就够了，中间的感知能力应该会自动涌现。→ 但 GUI 截图信息密度极高（几百个元素、大量微小图标），如果没有专门的中间监督信号，模型很容易在"看清楚每个元素是什么、在哪里"这一步就出错，进而污染后续所有推理和动作预测。→ UI-TARS 设计五个独立的感知子任务分别监督"元素识别""整体布局理解""动态变化感知""关系推理""标记关联"这五种不同粒度/性质的视觉能力，代价是数据构建流水线复杂了很多（需要专门的解析工具提取元数据 + 用 VLM 二次合成），但换来的是**把"看得清"和"想得对"这两件事解耦分别训练**，降低了端到端训练在感知这一环节吸收噪声、逐层放大误差的风险——这与本库多篇 D 组记忆论文里"先专门训练检索/压缩能力，再训练下游推理"的两阶段思想是同构的，只是在这里两个阶段被融合进同一个模型的不同训练任务里，而非拆成两个物理上独立的模块。

## §7　统一动作建模与 Grounding（§4.3，Table 1）

**统一动作空间**：把 Windows 的"click"和移动端的"tap"这类跨平台但语义等价的动作标准化为共享操作集（`Click(x,y)`、`Drag`、`Scroll`、`Type`、`Hotkey`、`LeftDouble`、`RightSingle`、`LongPress` 等），同时保留部分平台专属的可选动作。特别设计了两个**终止动作**：`Finished()` 表示任务完成，`CallUser()` 表示需要用户介入（如登录/身份验证场景）。

**Why（设计层）——为什么要专门设计一个 `CallUser()` 终止动作，而不是让模型在遇到需要人类权限的场景时随便输出点什么？**
> 朴素做法是不特殊处理这类场景，指望模型在真的卡住时自然停止或反复重试。→ 但登录、支付验证码等场景，agent 本来就不该（也通常不能）代替用户完成，如果没有一个明确的"求助"退出通道，模型要么在这类场景里死循环瞎试，要么编造一个看似完成但实际无效的动作。→ 显式设计 `CallUser()`，代价是需要专门标注"什么时候该调用它"的训练数据，但换来的是**把"知道自己做不到、该交还控制权给人类"这件事，变成动作空间里一个和其它动作平权的、可被训练和评测的具体选项**——这与本库 VisualWebArena（2401.13649）§9 的"不可达任务"设计、Mind2Web（2306.06070）Inspires-Us 里提到的"止损能力应作为独立评测维度"是完全一致的设计哲学，且 UI-TARS 把它更进一步落到了动作空间的具体实现层面。

**动作轨迹与 Grounding 数据**（Table 2）：论文用两类数据源构建训练集——① 自建标注工具在真实 PC 环境采集的动作轨迹（经严格质量过滤）；② 整合开源数据集（MM-Mind2Web、GUIAct、AITW、AITZ、AndroidControl、GUI-Odyssey、AMEX 等）并统一格式。Grounding 数据额外整合了 SeeClick、UGround、Rico 系列等开源资源，训练模型直接预测元素中心点的归一化坐标（相对屏幕尺寸标准化，保证跨设备分辨率一致性）。

## §8　System-2 推理注入：从"看图教程"到"逆向标注思考"（§4.4）

这是全文方法论最精细的一节，分两步走：

**第一步・用 GUI 教程构建推理基础知识**：从 MINT 和 OmniCorpus 两个图文交错预训练数据集出发，用三阶段过滤管道提炼高质量 GUI 教程：① **粗筛**——用人工标注的高质量教程正例 + 随机负例训练一个 fastText 分类器做初筛；② **精筛**——用 LLM 识别并剔除粗筛后残留的假阳性；③ **去重与精炼**——用 URL 和局部敏感哈希（LSH）去重，再用 LLM 改写文本内容、剔除广告和低质内容。最终提炼出约 **600 万条**高质量 GUI 教程（平均每条含 510 个文本 token + 3.3 张图片）。

**第二步・给动作轨迹"倒装"思考过程**：原始动作轨迹只有 $(o,a)$ 序列，没有显式的推理过程。论文用两阶段标注解决：

- **ActRe**：给定已知的目标动作 $a_n$，用 VLM 在已有上下文条件下反向生成一段"逻辑上导向这个动作"的思考 $t_n$（公式 4）。
- **Thought Bootstrapping（思考自举）**：ActRe 的问题在于，反向标注时模型已经知道正确答案，生成的思考可能只是"表面上凑巧对得上动作"，而非真实反映决策过程中的因果逻辑（论文原话："thought to conform to the action rather than reflect the actual decision-making process"）。解决方案是**不告诉标注模型目标动作**，让早期版本的 UI-TARS（$\text{UI-TARS}_{\text{early}}$）多次独立采样候选的思考-动作对，再从中挑出恰好命中真值动作的那个思考（公式 5）。

**Why（设计层）——为什么不能只用更简单的 ActRe，非要再加一层更复杂的 Thought Bootstrapping？**
> 朴素做法（ActRe）是"已知答案倒推理由"，实现简单、成本低，一次 VLM 调用就能给每一步配上一段看似合理的思考。→ 但这种"倒推"存在一个隐藏风险：生成的思考可能只是**事后为动作找补的合理化叙述**，而不是模型如果真的自主决策会走的那条推理路径——用这种数据训练，模型学到的可能只是"如何把一个已知动作包装成一段听起来有逻辑的文字"，而非真正学会"如何通过推理导出正确动作"。→ Thought Bootstrapping 反其道而行——不给答案、多次独立采样、事后筛选命中真值的样本，代价是需要多次采样、标注流程更复杂（需要一个早期训练好的模型做采样源，形成"先有鸡还是先有蛋"式的迭代依赖），但换来的是**思考与动作之间存在真实的因果关系，而不只是表面的文字对应**——这是本报告读过的所有论文里，对"如何避免为思维链生成'合理化叙述'而非'真实推理轨迹'"这一常见陷阱给出的最具体、最可操作的工程解法之一，值得在本库其它涉及"生成 CoT 训练数据"的论文（如 B 组 Reflexion、Self-Refine）对照阅读。

论文进一步用五种具体推理模式引导 VLM 生成思考：**任务分解、长程一致性、里程碑识别、试错、反思**——这五种模式与 §4 "核心能力"里定义的 System-2 推理组成部分完全对应，形成了"能力定义→训练数据设计"的闭环。

## §9　在线自举与反思调优：让模型从自己的错误里学习（§4.5）

**在线轨迹自举（Online Trace Bootstrapping，Figure 7）**：这是一个持续迭代的自我改进循环，形式化为：

1. 第 $n$ 轮的 agent $M_n$ 在虚拟机环境里执行任务指令集 $I_n$，产生原始轨迹集 $T_{\text{raw},n}$；
2. 三级过滤函数 $\text{Filter}(T_{\text{raw},n}, I_n) \to T_{\text{filtered},n}$：① **规则奖励**剔除明显异常（如不改变环境状态的冗余动作）；② **VLM 打分**剔除低于阈值的轨迹；③ **人工复核**，标注者定位错误发生的具体步骤，丢弃该步之后的所有动作，只保留有效前缀；
3. 用过滤后的轨迹微调出下一代模型：$M_{n+1} = \text{FineTune}(M_n, T_{\text{filtered},n})$；
4. 标注者基于新一轮轨迹精炼/扩展指令集：$I_{n+1} = \text{HumanRefine}(I_n, T_{\text{filtered},n})$。

这个循环在**数百台虚拟机**上跑多轮，让每一代模型都用来生成更好的下一轮数据。

**反思调优（Reflection Tuning）**：论文指出一个关键盲点——多数离线数据集只包含"理想化、无错误"的轨迹（标注者会确保每一步都对），这类数据没法教会模型"如何从错误中恢复"。解决方案是：对在线轨迹中真实出现错误的步骤 $\tau$（原动作 $a_\tau^-$ 无效或次优），让标注者① 标出修正后的思考与动作 $(t_\tau^+, a_\tau^+)$（**错误修正**对，公式 6）；② 进一步模拟"错误已经发生"这个既成事实下，下一步该如何补救（**事后反思**对，公式 7）——例如"原本想把网页加入书签，误点了关闭按钮，那么下一步应该是重新打开刚关闭的网页，再点书签按钮"。训练时**只对修正后的步骤计算损失**，错误步骤本身不参与梯度更新。

**Agent DPO**：反思调优只用了"正例"（修正后的动作），丢弃了大量自然产生的"负例"（错误动作）。论文引入 Direct Preference Optimization，基于 Bradley-Terry 模型定义偏好似然：

$$P(a^+ \succ a^- \mid s_\tau) = \frac{\exp r(s_\tau, a^+)}{\exp r(s_\tau, a^+) + \exp r(s_\tau, a^-)}$$

并推导出标准 DPO 损失（对参考 SFT 策略的 KL 约束下的最优策略），显式地让模型"提高偏好动作的似然、压低错误动作的似然"，而不仅仅是"从没见过错误动作"。

**Why（设计层）——为什么反思调优不能只用 SFT，非要再加一层 DPO？**
> 朴素做法是反思调优阶段收集的"错误修正对"直接用标准监督微调（SFT）训练——模型看到"这样是错的、这样是对的"配对数据里对的那部分，学着模仿。→ 但 SFT 只利用了正例（修正后的动作），错误动作 $a_\tau^-$ 虽然被标注出来了，却完全没有参与训练信号，等于白白浪费了在线自举过程中自然产生的大量负例。→ 引入 DPO 后，代价是训练流程从单纯的"模仿学习"变成了需要构造偏好对、调超参数 $\beta$（控制与 SFT 策略的偏离程度）的更复杂优化过程，但换来的是**显式地把"避免犯这个错误"编码进训练目标**，而不只是"没被教过这个错误"——这是本库反复出现的"用负例做对比学习/偏好优化，比单纯正例模仿学习信号更丰富"这一原则（DPO 本身的核心思想）在 GUI agent 反思能力训练上的具体应用。

## §10　三阶段训练流程（§4.6）

为与 Aguvis、OS-Atlas 等既有工作公平对比，UI-TARS 采用相同的 VLM 骨架 **Qwen-2-VL**，约 **500 亿 token** 总训练量，分三阶段：

1. **持续预训练阶段**：用 §4 描述的全部数据（不含反思调优数据），恒定学习率，让模型学习自动化 GUI 交互所需的全部基础知识（感知、grounding、动作轨迹）。
2. **退火阶段**：从感知、grounding、动作轨迹、反思调优数据中挑选高质量子集做退火训练，让学习动态更聚焦、决策策略优化更精细——此阶段产出的模型记为 **UI-TARS-SFT**。
3. **DPO 阶段**：用在线自举产生的反思对做 DPO 训练——产出最终模型 **UI-TARS-DPO**。

论文最终训练出三个规模变体：UI-TARS-2B / 7B / 72B。

## §12　离线与在线 Agent 能力评测（§5.3-5.4）

**离线评测**（Table 7-8：Multimodal Mind2Web、AndroidControl、GUI-Odyssey）：论文报告 UI-TARS-72B 在关键指标上取得 SOTA，UI-TARS-7B 尽管参数量更小，仍超过 Aguvis-72B 与 Claude Computer-Use。在 AndroidControl 和 GUI-Odyssey 上，UI-TARS-7B/72B 相比此前 SOTA（OS-Atlas-7B）取得约 25 个百分点的绝对提升（原文表述，具体逐格数字因表格排版复杂本报告不做精确复述）。一个值得记录的定性发现：**Claude Computer-Use 在网页任务上表现强劲，但在移动端场景明显吃力**，作者将其解读为"Claude 的 GUI 操作能力没有很好地迁移到移动端领域"——而 UI-TARS 在网页和移动端都保持了优秀表现，论文将其归因于统一动作空间训练带来的跨平台泛化能力。

**在线评测**（Table 9：OSWorld、AndroidWorld）——**这是全文最重要的一组数字，摘要引用的核心结论均来自这里**：

| 模型 | OSWorld（15 步） | OSWorld（50 步） | AndroidWorld |
|---|---:|---:|---:|
| Claude Computer-Use | 14.9 | 22.0 | 原文未给出精确逐格数字 |
| GPT-4o + Aria-UI（框架式最佳） | — | — | 44.8 |
| UI-TARS-72B-SFT | 18.8 | — | **46.6** |
| UI-TARS-72B-DPO | **22.7** | **24.6（SOTA）** | — |

**读出什么**：UI-TARS-72B-DPO 用 15 步预算（22.7）就已经追平 Claude 用 50 步预算才达到的成绩（22.0）——原文称这体现了"出色的执行效率"（great execution efficiency）；用满 50 步预算则进一步刷新 SOTA 到 24.6。AndroidWorld 上 46.6 vs 框架式最佳方案 44.8，验证了摘要的核心宣称。

**Why（结果层）——为什么"UI-TARS 用更少步数就能追平 Claude 用更多步数的成绩"这件事值得专门强调？**
> 完成同一个任务所需的步数，本质上反映了 agent 每一步决策的"信息密度"和"纠错能力"——步数越少达到同等效果，意味着**更少的无效探索、更少走弯路、更少需要事后纠正的错误动作**。这与"逐步都要对"的严格评分哲学（本库 mind2web 报告 §15 讨论过的 Step SR/SR 合取评分）是同一个精神的另一种度量方式：**效率本身就是能力的一个维度**，而不仅仅是"最终能不能做完"。这也间接印证了 §9 反思调优 + Agent DPO 的设计目标——减少无效试错步骤，正是这套训练方法论试图直接优化的目标量。

DPO 阶段的效果对比也值得记录：**DPO 相比纯 SFT 在 OSWorld 上带来明显提升**（原文§5.4第3点），验证了"引入负例做偏好优化"的有效性；而 72B 相比 7B 的差距在**在线任务上比离线任务上更大**，作者认为这说明"扩大模型规模显著提升 System-2 推理能力"，且**纯离线基准可能无法准确捕捉模型在真实动态环境里的能力差异**——这是一个对本库其它 harness 评测论文（G 组）也适用的方法论提醒：**离线评测和在线评测衡量的可能是部分不重叠的能力维度，仅用离线基准做模型选型有盲区风险**。

## §13　System-1 vs System-2 推理消融：一个反直觉的发现（§5.5）

论文专门设计了一组对照实验，用同一个 UI-TARS-7B 模型，仅通过 prompt 工程切换"直接输出动作（System-1）"与"先思考再输出动作（System-2）"两种推理模式，在域内基准（Mind2Web/AndroidControl/GUI-Odyssey，均有对应训练数据）和域外基准（AndroidWorld，无对应训练数据）上分别测试。

**发现一（反直觉）：在 Best-of-1（N=1，只采一个样本）设置下，域内任务上 System-2 推理反而略逊于 System-1**。原文的解释是：System-2 推理引入的额外思考步骤有时会包含不相关或错误的中间推理（比如提到不存在的对象、做出错误推断），在**只有一次采样机会**的情况下，一旦模型陷入某条有缺陷的推理路径，就没有其它候选可以纠正，反而拖累了最终动作的正确率。

**发现二：当采样数增加到 N=16、N=64（Best-of-N）时，System-2 的优势开始显著超过 System-1**。更多候选输出带来了更大的决策多样性，即便部分中间推理步骤不够理想，正确动作出现在采样候选中的概率也会显著提高——System-2 推理的"深度"能在有更多尝试机会时被充分兑现。

**发现三：在域外任务（AndroidWorld）上，即便只用 Best-of-1，System-2 也明显优于 System-1**。原文的解释是：域内任务里 System-1 靠"记住见过的模式"就能应付，但**面对训练时没见过的新场景，System-2 的深层推理能力提供了更强的泛化能力**——这与 §12 的"UI-TARS 在效率与规模效应上都优于纯离线学到的模式匹配"是一致的机制。

**Why（结果层）——为什么"更审慎的推理，在单次尝试下反而不如直觉反应"这件事重要？**
> 这个发现挑战了一个常见的朴素假设——"给模型更多思考步骤总是更好"。真实情况是：**审慎推理的价值高度依赖于"是否有多次尝试/多个候选做冗余"这个条件**。在域内、单样本场景下，审慎推理反而因为"更容易产生一步错、步步错的幻觉性中间步骤"而略逊于直觉反应；但一旦允许多次采样，或者场景本身超出训练分布（域外泛化），审慎推理的价值才充分显现。这对我们设计任何"要不要让模型先想后做"的 prompt 策略都是重要的参考：**"要不要用思维链"不该是一个非黑即白的默认选择，而要考虑任务是域内还是域外、是否有条件做多候选采样/投票**。论文也明确提出了后续方向——用强化微调等技术，让 System-2 推理即便在单样本条件下也能高置信度地产出正确动作，减少对多采样的依赖。

## §14　结论（§6）

作者将贡献总结为：一个整合感知、动作、推理、记忆的可扩展、自适应原生 GUI agent 模型，在 OSWorld 等高难度基准上取得 SOTA，超越 Claude 和 GPT-4o；同时系统性回顾了 GUI agent 从规则式系统到自适应原生模型的演化路径。展望方向明确指向 §2.3 的 Stage 4——**主动、终身学习的 agent**，让 agent 通过持续的真实世界交互自主驱动自己的学习过程，而不再依赖当前阶段仍然大量存在的人工标注与训练指导。

## ★ 对我们的启发（Inspires Us）

**a／"把 harness 该做的事训练进模型参数"是一条值得我们持续关注、但目前不适合直接照搬的路线**。UI-TARS 用大规模数据训练把"感知+推理+记忆+动作"整合进单一端到端模型，正面挑战了框架式 harness 工程的天花板（§2）。但这条路线的前提是**能拿到数百亿 token 级别的专属训练数据、数百台虚拟机做在线自举**，这远超我们当前 harness（Claude Code / m9.* 系列）能承担的规模。更现实的启发是**局部借鉴其方法论**：比如 §9 的"反思调优"思想（记录自己的错误+修正过程，而不只是记录成功路径）——我们完全可以在不做模型训练的前提下，把这个思想应用到 prompt 层面：在 agent-harness 报告库这类长任务的 PROGRESS.md 账本里，不只记录"完成了什么"，也系统性记录"哪里出过错、怎么修正的"，形成可供未来任务参考的"错误修正对"知识库。

**b／Thought Bootstrapping（§8）揭示的"倒推理由 vs 真实推理路径"的区别，是我们审视自己生成的任何解释性文字时该用的标准**。本次agent-harness 报告库写作过程中，我们大量生成"Why（设计层/结果层）"这类论证性文字——UI-TARS 的发现提醒我们：**如果这些论证是"先知道论文的结论，再倒着编一套听起来合理的因果解释"，本质上就是 ActRe 类型的"表面合理化"，而不是真实的推理过程**。下一步的自我要求：撰写 Why 三连时，应当先独立推演"如果不知道论文最终选了这个设计，我会怎么分析利弊"，再对照论文实际给出的理由，而不是直接从结论反推一套顺理成章的叙述——这与本报告一贯坚持的"先直觉、后定义、再读出什么"的顺序性要求是同一个精神的具体化。

**c／§13 的 System-1/System-2 消融结果，直接影响我们对自己 harness 里"要不要用更长思维链"这一类问题的判断**。这次续写/撰写 mind2web、visualwebarena、ui-tars 这几篇长报告时，我们始终依赖单次生成（没有 Best-of-N 采样、没有投票机制）。UI-TARS 的发现提示：**在单样本条件下，过度审慎的多步推理未必总是优于更直接的判断**，尤其是在"域内"任务（我们已经反复练习过的、有成熟套路的任务类型，比如撰写标准结构的论文报告）上。但对于"域外"任务（比如这次因子代理配额耗尽而被迫切换的、此前没有直接经验的"用 pdftotext 分段提取 PDF 再手写长文档"这一新工作模式），更审慎的规划反而更重要——这与实际发生的情况相符：我们确实在切换策略时先规划了提取粒度、核对了 Table 2/3 的数字对齐问题，而不是直接生成。

**d／"CallUser() 终止动作"（§7）应该成为我们自己子代理派发协议里的标准组件**。本次会话反复遭遇"API 配额耗尽"这类"agent 自己无法解决、需要人类或更高权限介入"的场景，我们的应对是靠对话中的显式状态报告（更新 PROGRESS.md 为🟡、如实汇报"未做完"）来实现事实上的"呼叫用户"。UI-TARS 把这个概念显式定义成动作空间里的一等公民，这提示我们：如果未来要设计更自动化的子代理派发协议（比如 Workflow 脚本化的批量论文写作流水线），应该显式定义一个"停止并请求人工介入"的标准退出码/状态，而不是让每个子代理各自用不同方式含糊地表达"我卡住了"。

**e／"离线评测可能无法捕捉在线动态环境里的真实能力差距"（§12）是审视我们自己 74 篇报告库质量的一个警示**。UI-TARS 发现 72B 相比 7B 的优势在在线评测中比离线评测中大得多，说明只看静态基准分数会低估某些能力差距。这提醒我们：本库的"结构自检"（字符数、Inspires-Us 标题格式、Why 三连计数）本质上都是**离线、静态的检查**，能确保格式合规，但不能替代"真正被读者使用、被验证是否真的讲透了论文核心贡献"这种"在线"意义上的质量检验。下一步：在全库 74 篇完成后的最终审计阶段，应当抽样做一次"假装自己完全不懂这篇论文，只读报告能否完整重建论文核心论证"的沙盘验证，而不能只满足于结构自检全部通过。

## §15　版本坐标：F 组内的路线分野

UI-TARS 是本库 F 组内**原生端到端路线**的代表，与 Mind2Web（2306.06070，纯文本+两阶段候选生成）、WebArena（2307.13854，可执行环境+accessibility tree）、VisualWebArena（2401.13649，SoM 提示工程+多模态但仍是"框架式"用现成 VLM 加提示技巧）形成清晰对照：**前三者都是"用现成模型 + 精心设计的 harness/提示技巧"路线，UI-TARS 是"用大规模数据把该学的东西直接训练进模型参数"路线**。这个分野恰好对应 VisualWebArena §14 发现的"SoM 提示法的收益高度模型相关，不能假设 harness 层技巧模型无关"——如果一个 harness 层的技巧本身就对底层模型高度敏感、不易迁移，那么"干脆把这个能力练进模型里"就成了一条更根本、更不依赖巧妙提示工程的解法，这正是 UI-TARS 在方法论选择上给出的回应。同时，UI-TARS 明确将 Claude Computer-Use 列为"原生 Agent 模型"阶段（Stage 3）的代表之一，与自己归为同一路线的竞争对手而非上一代框架式方案，说明这条端到端路线在 2024-2025 年之交已经形成了初步的阵营认同。

## §17　一页纸速记

**是什么**：UI-TARS——纯视觉输入、原生端到端的 GUI agent 模型（ByteDance Seed + 清华，2025 年 1 月，前沿），基于 Qwen-2-VL 用约 500 亿 token 持续训练，2B/7B/72B 三个规模。

**核心设计**：五类感知数据任务（元素描述/密集字幕/状态转移字幕/QA/SoM）+ 统一跨平台动作空间（含 `Finished()`/`CallUser()` 终止动作）+ System-2 推理注入（6M GUI 教程 + ActRe + Thought Bootstrapping）+ 在线自举（数百虚拟机+三级过滤）+ 反思调优 + Agent DPO + 三阶段训练。

**核心数字**：OSWorld 50 步 24.6（SOTA，超 Claude 22.0）、15 步 22.7（超 Claude 14.9）；AndroidWorld 46.6（超 GPT-4o+Aria-UI 框架的 44.8）；VisualWebBench 82.8（超 GPT-4o 78.5）；ScreenSpot Pro 38.1（SOTA）。

**最重要的发现**：① 原生端到端模型能超越精心设计的框架式方案；② System-2 推理的价值高度依赖采样次数与任务是否域内/域外，并非无条件"越审慎越好"；③ 离线基准可能低估在线动态环境里的真实能力差距，72B 相比 7B 的优势在线上比线下大得多。

**对我们的启发**：局部借鉴"反思调优"的错误修正记录思想、警惕"倒推理由"式的表面合理化论证、审慎推理并非在所有条件下都优于直觉反应、"呼叫用户"应作为标准化的退出机制、离线结构自检不能替代真实使用价值的验证。

**一句话定位**：它是 F 组"框架式 harness 工程 vs 原生端到端训练"这条路线分野的前沿代表，用具体的 SOTA 数字论证了"把该学的东西直接训练进模型参数"这条路线在 2025 年初已经能够跑赢精心设计的商用模型+提示工程组合。
