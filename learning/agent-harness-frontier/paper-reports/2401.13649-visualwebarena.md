# VisualWebArena：视觉 grounding 版的 WebArena（分组 F・Web/GUI Agent・E/T 层・⭐canon 邻接节点）

> arXiv 2401.13649｜CMU（Jing Yu Koh, Robert Lo*, Lawrence Jang* 等，*同等贡献）｜ACL 2024｜时间坐标：2024 年初，属本库 canon 脊柱在 F 组的直接延伸（继承 WebArena 2307.13854 的环境框架）

## §1　TL;DR

> 主讲提示：一句话先立住——这篇论文回答的问题是"如果任务本身就需要看图才能做，纯文本 agent 会输成什么样"。

VisualWebArena（下称 VWA）在 WebArena 的可执行、可复现、自托管网页环境基础上，引入**910 个必须依赖视觉信息才能完成**的任务，覆盖 Classifieds（全新站点）、Shopping、Reddit 三个环境，并提出一种受 Set-of-Marks（SoM）提示启发的视觉 agent 表征。核心数字：人类基线成功率 88.7%（Table 3），当时最强的 VLM agent（GPT-4V + SoM）只有 16.37%——**70 个百分点的鸿沟**，且论文明确指出纯文本 LLM agent（最好的 GPT-4 也只有 7.25%）在视觉必需任务上遭遇结构性天花板。

---

## §2　问题：为什么"文本 agent 基准"不够用

**直觉**：人类操作电脑时大量依赖视觉——图标颜色、商品图片、页面版式——但 2023–2024 年之交的主流 web agent 基准（Mind2Web、WebArena 本身）评测的任务大多可以只看 HTML/accessibility tree 文本就做完，图片信息即便存在也不是完成任务的必要条件。这与"数字世界的界面主要是为人眼设计的"这一现实脱节（§1 引言）。

**Why（问题层）——为什么不能直接拿 WebArena 的任务改一改当视觉任务用？**
> 朴素做法是从 WebArena 现有任务里挑一些"看起来涉及图片"的任务重新标注。→ 但 WebArena 的任务设计之初就没有强制"必须看图才能做"这个约束，很多看似有图的任务其实文本线索（如商品名称、价格文本）已经足够完成，纯文本 agent 一样能蒙对。→ VWA 选择从头设计一批**视觉 grounding 是完成任务的必要条件**的新任务（§4.2 "Visually Grounded Tasks"：颜色、图案等仅靠视觉可辨识的属性，任务描述里故意不用文字点破），代价是标注成本更高（需要 6 名研究生撰写意图模板 + 人工扩展 + 手写奖励函数），但换来的是一个**能真正把"看不懂图"和"看得懂图"的 agent 分开**的基准，而不是一个文本 agent 靠猜也能刷到不错分数的基准。

## §3　与 WebArena 的继承关系：环境复用 + 视觉专属扩展

VWA **直接建立在 WebArena（Zhou et al., 2024，本库同组 2307.13854）之上**（§1），复用其"可复现、自托管、执行式评测（execution-based evaluation）"的核心哲学——即评测不是比较字符串输出，而是**真的在浏览器里执行动作、检查网页最终状态**。Shopping、Reddit 两个站点直接继承自 WebArena；**Classifieds 是 VWA 的全新贡献**（§4.1），仿照 Craigslist/Facebook Marketplace 用开源 CMS 系统 OSClass 搭建，包含 65,955 条真实分类广告。

**Θ2 关联**：这是"Agent = Model + Harness"论点在 benchmark 设计层面的一个具体例证——WebArena 提供的是**环境层（E 层）harness**（可复现的自托管网站基础设施），VWA 在完全不改动这套环境 harness 的前提下，只通过**换一批任务 + 加一层视觉评测原语**，就把整个基准的性质从"文本可解"改造成"视觉必需"，说明同一个 harness 底座上叠加不同的任务/评测层，能测出模型完全不同维度的能力。

## §4　环境形式化：POMDP 定义（§3）

**符号表**：

| 符号 | 含义 |
|---|---|
| $E = (S, A, \Omega, T)$ | 环境的部分可观测马尔可夫决策过程（POMDP）形式化 |
| $S$ | 状态集合（网页所处的具体状态） |
| $A$ | 动作集合（§3.2，Table 1） |
| $\Omega$ | 观测集合（§3.1，四种网页表征方式之一） |
| $T: S \times A \to S$ | 转移函数，**确定性**——同一状态下同一动作总是转移到同一后继状态 |
| $R: S \times A \to \{0,1\}$ | 奖励函数（§3.3），任务成功=1，否则=0 |

**读出什么**：转移函数被显式声明为**确定性**（deterministic），这是"自托管、可复现"这条设计主线在数学形式化层面的落地——如果环境是随机的，同一个 agent 同一个动作序列每次跑出来的结果都可能不同，评测数字就没有可比性。这与本库 G 组 harness 评测论文反复强调的"可复现执行环境是可信评测的地基"是同一个原则的更早期、更具体的实例。

## §5　观测空间：四种网页表征方式的对比（§3.1）

VWA 明确列出网页内容可以用四种方式喂给 agent：

1. **原始 HTML DOM 树**——早期 web agent 工作（如 Mind2Web）使用。
2. **Accessibility tree**——为辅助技术设计的结构化简化表示，是 **WebArena 基线 agent 的主要表征**。
3. **网页截图（RGB 数组）**——此前已有工作证明有效，但通常缺乏元素级别的精确定位能力。
4. **Set-of-Marks（SoM）表征**——VWA 自己提出的第四种：对网页上每个可交互元素画出**边界框 + 唯一 ID**，生成一张带标注的截图，让视觉 agent 直接引用元素 ID 来定位（Fig. 2）。

**Why（设计层）——为什么要在截图之外再发明一种"SoM 截图"，而不是直接让 VLM 看原始截图操作？**
> 朴素做法是让 VLM 直接看原始网页截图，输出要点击的 (x, y) 坐标。→ 这对当时（2023–2024）的主流 VLM 是一个不自然的任务：多数 SOTA VLM/LLM 没有专门针对"在像素图上做精细坐标定位"这类细粒度视觉定位任务训练过（§3.2："这些模型没有为如此精细粒度的元素引用而显式训练过"），直接输出坐标的误差会很大。→ VWA 借用 Set-of-Marks 提示法，把"输出连续坐标"这个困难的**回归**问题，转化成"从截图上已经标好号的一组候选里选一个 ID"这个**判别**问题（Fig. 2 示例：`[7][A][Comments]`），代价是需要额外一步 JavaScript 预处理来自动标注每个可交互元素，但换来的是**把高层推理和低层控制解耦**——模型只需要做"选哪个"的语义判断，不需要做"具体点在哪个像素"的精细定位。这与本库 C 组 SWE-Agent 的 ACI（Agent-Computer Interface）设计哲学——"给模型一个专门设计过的、容易正确使用的接口，而不是让它直接操作底层原语"——是同一个思想在视觉领域的应用。

## §6　动作空间（Table 1）

VWA 的动作集合 $A$ 与 WebArena 基本一致，包括：`click[elem]`、`hover[elem]`、`type[elem][text]`、`press[key_comb]`、`new_tab`、`tab_focus[index]`、`tab_close`、`goto[url]`、`go_back`、`go_forward`、`scroll[up|down]`、`stop[answer]`。动作参数是**当前观测里的唯一元素 ID**（accessibility tree 场景下是树里的元素 ID，SoM 场景下是标注框的 ID）——这个设计选择直接呼应了 §5 的"判别而非回归"哲学：无论用哪种观测表征，模型输出的都是一个可枚举、可校验的离散 ID，而不是自由格式的坐标或选择器字符串。

## §7　评测原语：从"字符串比对"扩展到"视觉比对"（§3.3）

VWA 继承 WebArena 的文本类评测原语，并新增两个视觉专属原语：

**继承自 WebArena 的文本原语**：
- `exact_match`：$\mathbb{1}\{\hat a = a\}$，逐字精确匹配，用于数值类答案。
- `must_include`：集合 $a$ 里的所有元素都必须出现在输出 $\hat a$ 里才给 1 分，用于无序列表类输出。
- `must_exclude`：`must_include` 的反义——集合 $a$ 里任何一个元素出现在 $\hat a$ 里就判 0 分（例：$\hat a$="\$1.99, \$2.50, \$10.00"，$a$={"1.50","2.00"}，因为禁止项都没出现，奖励为 1）。
- `fuzzy_match`：查询一个 LLM（原文用 GPT-4-Turbo）判断语义是否等价，输出"correct/incorrect/partially correct"，只有"correct"记 1 分——用于更开放式的语义匹配场景。

**VWA 新增的视觉原语**：
- `eval_vqa`：查询一个能做视觉问答（VQA）的 VLM（原文用 BLIP-2-T5XL），给一张图和一个问题，如果 VLM 的回答包含真值答案就记 1 分。用于"买一件低于 \$10 的绿色帽衫"这类**存在多个满足条件的正确答案、无法穷举 ID** 的开放式任务。
- `eval_fuzzy_image_match`：用结构相似性指数（SSIM）判断查询图片与真值图片是否相似，超过阈值 $t \in [0,1]$ 记 1 分。

**Why（设计层）——为什么要专门发明 `eval_vqa` 而不是让标注者手写"检查图片里有没有 XX"这种规则？**
> 朴素做法是像 `must_include` 一样，为每个视觉相关任务手写一套基于图片元数据/文件名的规则化检查。→ 但真实世界的视觉判断（"这件衬衫是不是 polo 衫""是不是绿色的"）用规则几乎无法穷举所有变体，且任务描述本身故意不写死"polo 衫"这个词（§4.2 例子），规则化检查会系统性漏判。→ VWA 用另一个 VLM 作为"评委"来做视觉判断，代价是引入了**评委模型自身的误差**（BLIP-2-T5XL 不是完美的 VQA 模型，评测本身就带噪声），但换来的是**能够评测那些本质上就是"开放式视觉语义判断"的任务**，而不必为每个任务单独设计规则——这正是本库 G 组 Harness-Bench 用 LLM-judge 做过程质量判分（Process Score）时面对的同一类权衡：judge 本身的误差 vs. 评测覆盖面的扩大。

Table 2（原文）用真实任务示例展示了这套原语如何组合使用，例如"把这个人穿的衣服类似的东西加入愿望单"对应 `eval_vqa(s, "Is this a polo shirt?", "yes")` + `eval_vqa(s, "Is this shirt green?", "yes")` 两次视觉判断的合取。

## §9　任务构造流程与规模（§4.2）

**流程**：6 名研究生（论文共同作者）先撰写意图模板（如"帮我找 {{属性}} 的 {{物品}}，价格在 {{区间}} 之间"），再由标注者手工扩展成具体任务实例（如"帮我找最便宜的红色 Toyota，价格在 \$3000 到 \$6000 之间"），并为每个任务手写基于 §7 原语组合的奖励函数。输入图片来自免版权来源与 MS-COCO 数据集。

**核心统计**：
- 共收集 **314 个独立模板**，平均每个模板扩展出 **2.9 个任务**，最终得到 **910 个任务**。
- 其中 **46 个任务（5.1%）是"不可达"任务**——故意设计成无法完成，用来测试 agent 能否正确识别"这个任务做不到"并提前终止，而不是无休止地瞎试。对于不可达任务，agent 必须输出**做不到的原因**，用 `fuzzy_match` 评测。
- **25.2% 的任务包含输入图片**，要求理解图文交错的信息。

**Why（设计层）——为什么要专门设计 5.1% 的"不可达"任务，而不是让全部 910 个任务都可解？**
> 朴素做法是让所有任务都存在一个可执行的成功路径，这样评测更"干净"（100% 任务都有正确答案）。→ 但真实世界里用户经常提出模糊、过度约束、或客观上无法满足的请求（比如要求的商品根本不存在），一个只会不断尝试、永不放弃的 agent 在真实部署中是危险且低效的。→ VWA 保留 5.1% 的不可达任务，代价是评测协议变复杂了（需要额外判断"agent 是否正确识别出不可达"而不仅仅是"任务是否成功"），但换来的是**同时测试"能不能做"和"知不知道做不到就该停"这两种能力**——这是一种非常朴素但极其重要的 harness 层安全设计理念：**知道何时止损，本身就是能力的一部分**，与本库 H 组关于 agent 可靠性/失败恢复的多篇论文（如 hell-or-high-water）在精神上是相通的。

## §10　任务难度分级：动作难度 × 视觉难度（§4.2）

VWA 把每个任务同时标注两个独立的难度轴：

- **动作难度**：按完成任务所需的动作步数分档——Easy（≤3 步）、Medium（4–9 步）、Hard（≥10 步）。
- **视觉难度**：按所需视觉理解的复杂程度分档——Easy（基础颜色/形状/高层物体识别，如"看出画面里有猫"）、Medium（模式辨别、语义理解、或对较短文本做 OCR）、Hard（多图理解、对细小或冗长文本做 OCR、或需要辨认精细细节）。

**Why（设计层）——为什么要用两条独立的难度轴，而不是一个综合难度等级？**
> 朴素做法是像多数基准一样，只给一个整体的 easy/medium/hard 标签。→ 但"动作步数多"和"视觉理解难"是两种性质完全不同的困难来源：一个任务可能只需要 2 步操作，但要求精细 OCR（低动作难度、高视觉难度）；另一个任务可能需要 12 步操作，但每一步的视觉判断都很直白（高动作难度、低视觉难度）。合并成一个标签会掩盖这种区别。→ VWA 用两条独立轴，代价是标注工作量翻倍（每个任务都要打两次标），但换来的是后续 Appendix C.2（Figure 6）能做**动作难度 × 视觉难度的交叉分析**，精确定位"agent 到底是不会做还是看不懂"——这与本报告 mind2web 篇（2306.06070）里"三级泛化切分"能拆解"到底是没见过任务、网站、还是领域"的设计哲学如出一辙：**用更细粒度的独立标注轴，把一个笼统的失败原因拆解成可诊断的多个维度**。

## §11　人类基线：88.7%（§4.3）

作者让 7 名参与过任务创建的大学生（但确保不测试自己创建的任务，避免数据泄漏）完成 230 个抽样任务（每个模板抽 1 个），测得整体成功率 **88.7%**（Table 3 最后一行；Classifieds 91.07% / Reddit 87.10% / Shopping 88.39%）。原文分析了人类犯错的常见原因：任务理解偏差（如把商品加入购物车而非愿望单）、放弃穷举式搜索（"在 5–10 分钟内找不到就以为任务不可能完成"）、电商场景下没有遍历所有候选页面来找最便宜/评分最高的商品。作者特别指出，这些人类失误恰恰是**强 agent 有潜力反超人类**的地方（agent 不会因为"懒得翻页"而漏掉更优解）。

## §12　Baseline 家族：从纯文本到多模态 + SoM（§5）

VWA 用统一的 3-shot in-context 提示（每个环境各一个示例，与测试任务无重叠）评测四类 agent：

1. **纯文本 LLM agent**（§5.1）：用 Chain-of-Thought 提示 + accessibility tree 作为观测。测试 GPT-4-Turbo、GPT-3.5-Turbo、Gemini-Pro、LLaMA-2-70B、Mixtral-8x7B。
2. **图片字幕增强 LLM agent**（§5.2）：用图片字幕模型（BLIP-2-T5XL 或 LLaVA-v1.5-7B）为网页上每张图生成 alt-text，拼进 accessibility tree 一起喂给纯文本 LLM——相当于给文本模型"间接装了眼睛"。
3. **多模态 agent**（§5.3，Image + Captions + Acc. Tree）：直接把网页截图 + 字幕 + accessibility tree 一起喂给原生支持图文输入的 VLM（GPT-4V、Gemini-Pro、IDEFICS-80B-Instruct、CogVLM）。
4. **多模态 SoM agent**（§5.3，Image + Captions + SoM）：本文作者自己提出的方案——用 SoM 标注截图替代原始截图 + accessibility tree。

**Why（设计层）——为什么要保留"字幕增强"这个中间档位，而不是直接对比纯文本 vs 多模态两端？**
> 朴素做法是只测最朴素的纯文本 baseline 和最强的原生多模态 baseline，形成一个简单的二元对照。→ 但这样无法区分"性能提升到底来自模型本身有没有视觉能力，还是仅仅因为多了一点点视觉相关的文字信息"。→ VWA 保留字幕增强这一档，代价是要多训练/多跑一组实验，但换来的是能够回答一个更精细的问题：**"给纯文本模型一点视觉线索的文字摘要"能在多大程度上逼近"真正看图"的效果**——结果（§13）显示字幕增强确实有帮助（GPT-4 从 7.25%→12.75%），但仍明显低于真正的多模态（15.05%），说明视觉信息里有相当一部分内容是**字幕这种有损压缩无法保留**的。

| 多模态 SoM | Gemini-Pro | 3.42% | 3.81% | 7.73% | 5.71% |
| 多模态 SoM | **GPT-4V** | 8.12%→9.83% | 12.38%→17.14% | 19.74%→19.31% | **16.37%** |

**读出什么·发现一：纯文本 LLM 全面失效，即便是当时最强的 GPT-4**。最强纯文本 agent（GPT-4）Overall 只有 7.25%，多数开源模型（LLaMA-2-70B、Mixtral）在 1–2% 附近，**在需要视觉理解的任务集合上，"会推理"和"看得懂图"是两种几乎不相关的能力**——这直接验证了 §2 的问题假设：如果任务本身要求视觉 grounding，纯文本 agent 无论推理能力多强都过不去这一关。

**读出什么·发现二：多模态相比字幕增强有实质提升，但字幕增强也不是没用**。GPT-4：7.25%（纯文本）→12.75%（字幕增强，+5.5pp）→15.05%（原生多模态，再 +2.3pp）。这是一个**递进关系**：给模型任何形式的视觉线索（哪怕只是一句字幕）都有帮助，但"真正看到图"仍然比"读一句关于图的描述"更有效——字幕是有损压缩，会丢失原始图像里的细节信息（这一点在 §15 的 OCR 任务分析中体现得尤其明显）。

**读出什么·发现三：人类 88.7% vs 最强 agent 16.37%，鸿沟高达 72.3 个百分点**。这个差距远大于同类型的纯文本基准（比如 Mind2Web 上微调模型与理论上限的差距）——说明视觉 grounding 在 2024 年初这个时间点，对 agent 而言是比"网页文本理解"更难跨越的能力鸿沟。

## §14　SoM 为什么只对 GPT-4V 有效（§6 "SoM Improves Navigability"）

SoM 表征把 GPT-4V 的成绩从 15.05% 进一步推高到 **16.37%**，且在 Classifieds（12.38%→17.14%）和 Reddit（8.12%→9.83%）上提升尤其明显——原文将其归因于这两个站点包含**更密集、排列紧凑的小图**（accessibility tree 难以区分空间上紧挨着的元素），SoM 的边界框标注恰好解决了这个"元素消歧"问题。

但作者诚实地报告了一个重要的**负面/中性结果**：SoM 对 IDEFICS-80B-Instruct、CogVLM、Gemini-Pro **没有带来显著提升**（Gemini-Pro 甚至从 6.04%→5.71% 略微下降），只有 GPT-4V 从中明显受益。

**Why（结果层）——为什么 SoM 只对 GPT-4V 有效，而不是对所有 VLM 普遍有效？**
> 原文将这一现象归因于 Yang et al. (2023a)（SoM 提示法的原始论文）已经发现的结论：**只有 GPT-4V 表现出"SoM grounding 能力"**，作者推测这可能与模型的规模或训练数据构成有关（§6，"perhaps due to scale or training data"）——原文对此没有给出更深入的机制性解释，属于诚实标注的"原文未给出"确切原因，只给出了假设性推测。这是一个值得警惕的发现：**一个在某个模型上验证有效的 harness 层设计（SoM 这种提示格式创新），不能想当然地认为可以直接迁移到另一个模型上**——harness 与底层模型能力之间存在隐藏的耦合，换模型可能让同一套 harness 设计完全失效。这对本库反复强调的"Agent = Model + Harness"论点是一个重要的补充和警示：**harness 的收益不是模型无关的，必须针对目标模型重新验证**。

## §15　细粒度任务子集分析：Table 4（GPT-4V + SoM）

| 任务子集 | 占总任务比例 | 成功率 |
|---|---:|---:|
| 需要 OCR | 17.1% | 13.4% |
| 不需要 OCR | 82.9% | 16.9% |
| 精确图片匹配 | 8.7% | 18.9% |
| 非精确图片匹配 | 91.3% | 16.2% |
| 含图片输入 | 25.2% | 19.0% |
| 不含图片输入 | 74.8% | 14.9% |

**读出什么**：三组对比揭示了三个不同方向的信号——① **OCR 是当前最强 VLM agent 的短板**（13.4% vs 16.9%，原文明确称"OCR may be a bottleneck for current agents"）；② **精确图片匹配反而不是瓶颈**（18.9% 高于均值），说明"看两张图是否是同一张"这种相对直接的视觉比对任务，SoM+GPT-4V 反而处理得不错；③ **含图片输入的任务成功率更高**（19.0% vs 14.9%），这一点看似违反直觉（更多信息输入难道不该更难？），原文解读为"一旦视觉内容被正确理解，这类任务反而更容易变得可解"——即图片输入提供的是**消除歧义的额外约束**而非额外负担。

**Why（结果层）——为什么"含图片输入的任务"反而比"不含图片输入的任务"成功率更高？**
> 朴素直觉是"输入模态越多，任务越复杂，成功率该更低"。→ 但作者的解释指向了另一种机制：VWA 里"不含图片输入"的任务，其视觉要求来自于**必须自己在网页上找到并理解相关图片**（比如"这件商品是不是绿色的"，需要 agent 自己定位到商品图片再判断颜色），这本身包含"定位"+"理解"两步；而"含图片输入"的任务直接把参考图片作为任务的一部分给出，**减少了"定位正确的视觉参照物"这一步的难度**，agent 只需要做图片间的比对/匹配。这个反直觉发现提示我们：**评价一个任务"难不难"，不能只看输入的模态数量，还要看每种模态承担的功能——是提供额外约束（更容易），还是需要额外的定位/搜索开销（更难）**。

**补充信号：Gemini-Pro 在这三组对比上呈现相反模式**（Appendix C.3）。原文额外指出，Gemini-Pro 系列在"含图片输入"任务上的成功率反而**低于**不含图片输入的任务，与 GPT-4 系列的模式相反，作者推测这可能反映 Gemini-Pro 处理**图文交错（interleaved image-text）输入**的能力弱于 GPT-4V（§C.3："这可能意味着该模型处理多个交错图文输入的能力不如预期"）。这是又一处印证 §14 "harness 收益模型相关"的证据——不仅 SoM 表征的收益因模型而异，连"给不给图片输入"这种最基础的信息增量，在不同模型家族上的边际效应方向都可能相反。

## §16　难度交叉分析与后续强模型验证（Appendix B/C）

**动作难度 × 视觉难度交叉表**（Figure 6，GPT-4V+SoM overall 行）：Easy/Easy 组合成功率最高，Hard 动作难度整体拖累最明显（Overall 行里 Easy 21.4% → Medium 12.9% → Hard 10.5%，态势符合直觉：步数越多，累计出错概率越高）。原文额外指出，多模态模型**在高视觉难度任务上优势尤其明显**：GPT-4V+SoM 在 Hard 视觉难度子集上均值 12.4%，显著高于字幕增强（8.0%）和纯文本（4.8%），说明视觉能力的边际收益在"视觉本身就很难"的任务上体现得最充分。

**Appendix B：后续用更强模型补测**（ACL 投稿截止后追加）：作者用 SoM 方案额外测了 GPT-4o、Gemini-Pro-1.5、Gemini-Flash-1.5、Llama-3-70B-Instruct（Table 5）。**GPT-4o 达到 19.78%，正式超过 GPT-4V 的 16.37%**——证明 SoM 这套 harness 设计本身能随着底层模型能力提升而"水涨船高"地继续受益（呼应 §14 的警示：虽然 SoM 不是对所有模型普适，但至少在 GPT 系列的连续迭代上是稳定有效的）。Llama-3-70B-Instruct（字幕增强）达到 9.78%，相比其前代 Llama-2-70B（0.66%）有巨大飞跃，展示了开源模型代际进步对视觉/文本联合理解能力的显著拉动。

**Appendix C.1：in-context 示例数量的边际效应**（Table 6，Gemini-Pro，Image+Caption+Acc.Tree 表征）：0-shot 2.86% → 1-shot 3.63% → 3-shot 6.04%——原文特别指出"从 1 条示例到 3 条示例有显著跃升"，并推测**如果在网页交互轨迹上做微调，成绩可能进一步显著提升**（§C.1，留作未来方向，非本文实验范围）。这组数字虽小，但提供了一个重要的方法论提醒：**本文报告的所有主结果都只用了 3-shot 提示**——也就是说 Table 3 里的全部数字都还没有触及"in-context 示例数量"这个维度的天花板，真实的模型能力上限可能比表中数字更高。

**Why（结果层）——为什么 0→1 shot 的提升（2.86%→3.63%，+0.77pp）远小于 1→3 shot（3.63%→6.04%，+2.41pp）？**
> 从 0 到 1 条示例，模型第一次看到"输出该长什么格式"的具体范例，但只有一个环境的示例，容易被误认为对其它两个环境也要生搬硬套同一种解法模式；从 1 到 3 条示例（论文设置下覆盖三个环境各一条），模型才第一次同时看到"同一套动作空间在不同环境里如何被灵活使用"的对比样本。这提示：**in-context 示例的边际价值不只取决于数量，还取决于示例集合本身是否覆盖了任务空间的关键变异维度**（这里是"环境类型"）——这与本库 B 组 Plan-and-Solve、C 组 ToolLLM 等论文里反复出现的"few-shot 示例要覆盖代表性场景而非任意堆量"的设计原则完全一致。

## §17　结论与 Ethical/Broader Impacts（§7、§8）

作者将贡献总结为：① 910 个视觉必需任务的基准，覆盖 3 个环境；② 一个受 SoM 启发的视觉 agent 表征，证明其在简化动作空间、提升视觉复杂网站上的表现方面的价值；③ 详尽的 baseline 评测，证明 VLM 相比纯文本 agent 有明显优势，但与人类相比仍有巨大差距。

**Ethical and Broader Impacts（§8）** 部分明确讨论了三方面考量：① **现实影响**——强 agent 有潜力提升计算机任务的可及性（帮助残障人士/技术能力有限的用户），但同时需要认真考虑对就业的宏观经济和社会影响；② **偏见与安全**——需要确保部署的 agent 不会无意中排斥或损害特定群体，agent 相比普通 LLM 有更大的现实危害潜力，需要审慎的安全防护；③ **预期用途**——明确声明 VWA 是研究基准，论文中的模型是研究原型，**不适合在当前状态下部署到实际应用**（尤其是高风险领域）。这段自我约束的表态与 Mind2Web（2306.06070）§6 的安全部署讨论一脉相承，也早于本库 H 组系统性 agent 安全论文（AgentDojo/LlamaFirewall 等 2024–2025 年工作）——是同一条"benchmark 作者主动提醒下游部署风险"的时间线延伸。

## ★ 对我们的启发（Inspires Us）

**a／"必须依赖某种能力才能完成"应该是我们设计能力测试任务时的硬约束**。VWA 最有力的方法论贡献不是"加了图片"，而是**确保视觉 grounding 是完成任务的必要条件**（§2 的 Why 三连），排除了"文本线索足够、图片只是装饰"的伪视觉任务。这提醒我们：如果未来要给自己 harness 的某个能力（比如 m9.* 系列的长上下文压缩能力）设计验证任务，必须先问"如果去掉这个能力，任务还能不能做"，而不是简单地"看起来涉及长文本"就算数。

**b／SoM 的"判别优于回归"设计（§5）可以直接迁移到我们自己的工具调用接口设计**。把连续、易错的坐标预测转化为离散、可校验的 ID 选择，这与本库 C 组 Toolformer/SWE-Agent 反复出现的"给模型一个专门设计过的、容易正确使用的接口"是同一个原则。我们在设计任何需要模型"定位到某个具体目标"的 prompt（不管是网页元素、文件里的某一行、还是知识库里的某个片段）时，应当优先考虑能否先枚举候选、再让模型选择，而不是让模型直接生成一个自由格式的定位描述。

**c／§14 的"SoM 只对 GPT-4V 有效"是我们评估任何 harness 层技巧迁移性时必须做的功课**。这次续写 mind2web（2306.06070）报告时，我们已经在 Inspires-Us 里记录过"harness 收益不是模型无关的"这一原则；VisualWebArena 提供了一个更具体、更让人意外的反例：**同一套提示格式创新（SoM），换一个 VLM 就可能完全失效**。下一步：我们在自己的 harness 里如果验证了某个 prompt 技巧（例如某种 few-shot 示例格式、某种输出结构约束）在当前默认模型上有效，切换模型版本后应当重新做一次小规模验证，而不是默认它继续有效。

**d／"不可达任务"（§9，5.1%）的设计，值得我们在自己的任务/工具执行流程里加入对应的"止损"训练信号**。本次 agent-harness 报告库建设过程中，我们自己也反复遇到"任务客观上做不到"的情况（比如本篇写作过程中撞上的"api key 日限额已用完"）——正确的行为不是无休止重试，而是**识别出"当前路径不可行"并及时切换策略**（本次实际应对：改为主线程直接续写而非死等子代理配额恢复）。VWA 把这种能力显式纳入评测（"agent 能否正确输出任务不可达的原因"），这提示我们：如果未来要评价自己 harness 的鲁棒性，"知道何时该停/该换路子"应该是一个独立打分的维度，而不是模糊地并入"任务完成率"。

**e／人类基线的错误分析（§4.3）提醒我们：基准测的是"agent 相对哪个参照系的能力"，人类基线本身也不完美**。VWA 记录了人类在这些任务上的具体失误模式（放弃穷举搜索、理解偏差），并指出这恰恰是 agent 有潜力超越人类的地方。这对我们审视自己 74 篇 agent-harness 报告库、100 个 Linux bash 函数等后续任务的"完成质量"标准有启发：**"人类专家会犯的错误"不应该被设为我们自己质量门槛的天花板**——比如本库的 PROGRESS.md 账本机制、"原文未给出"标注纪律，本质上都是在系统性地消除"人类容易犯但机器不该犯"的那类错误（遗漏、臆造、格式不一致），这是我们完全可以比未经系统训练的人类标注者做得更好的地方。

## §19　延伸讨论问题

1. §14 指出 SoM 提示法只对 GPT-4V 显著有效，对其他 VLM 无效甚至轻微负面。如果你要在一个新发布、你完全没有先验知识的 VLM 上决定"要不要用 SoM"，你会如何低成本地验证，而不是重新跑一遍完整的 910 任务基准？

## §20　一页纸速记

**是什么**：VisualWebArena——建立在 WebArena 环境基础设施上、910 个视觉必需任务的多模态 web agent 基准（Classifieds 全新站点 + Shopping/Reddit 继承），外加一个 SoM 启发的视觉 agent 方案。

**核心设计**：四种网页表征对比（DOM/accessibility tree/screenshot/SoM）+ 视觉专属评测原语（eval_vqa/eval_fuzzy_image_match）+ 双轴难度标注（动作难度×视觉难度）+ 5.1% 不可达任务。

**核心数字**：人类基线 88.7%；最强纯文本 agent（GPT-4）7.25%；字幕增强 GPT-4 12.75%；多模态 GPT-4V 15.05%；多模态 SoM GPT-4V 16.37%（当时最佳）；后续 GPT-4o + SoM 达 19.78%。OCR 任务子集成功率 13.4% vs 非 OCR 16.9%，OCR 是明确短板。

**最重要的发现**：① 纯文本 agent 在视觉必需任务上结构性失效；② SoM 表征的收益高度模型相关（只对 GPT-4V 显著有效），不能假设 harness 层技巧模型无关；③ 人类 88.7% vs 最强 agent 16.37%，鸿沟远大于同期纯文本基准。

**对我们的启发**：能力测试任务必须确保"该能力是完成任务的必要条件"、判别优于回归的接口设计、harness 技巧迁移性需要逐模型重新验证、"知道何时止损"应作为独立评测维度、人类基线的已知缺陷不该是我们质量门槛的天花板。

**一句话定位**：它是 WebArena 环境哲学在视觉维度上的直接延伸，用一个"70 个百分点鸿沟"的具体数字，把"web agent 需要视觉理解"这件事从直觉变成了可测量、可追踪的研究问题。
