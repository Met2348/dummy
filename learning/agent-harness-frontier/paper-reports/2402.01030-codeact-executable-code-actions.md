# CodeAct：把 Agent 的动作统一成"可执行 Python 代码"

> **本篇是 agent-harness 库 E 组（编码 Agent / 集成系统）的一块基石（canon）**。它回答一个看似工程、实则根本的问题：
> **一个 LLM-agent 每一步"想干什么"，应该用什么"语言"说出来给系统去执行？** 主流答案是 JSON 工具调用
> （`{"tool": ..., "args": ...}`）。CodeAct 给出另一个答案——**直接写一段可执行的 Python 代码**，交给解释器跑。
> 全篇的重量都压在一句直觉上：**"一次一个 JSON 调用"表达力太弱，而"一段代码"天然带控制流、能组合多工具、能复用整个 PyPI 生态、
> 能一次表达多步**——代价是你必须给它一个安全的执行环境。本报告严格对齐标杆范文 `2605.27922-harness-bench` 的密度与诚实度：
> 每个公式前给直觉→先定义符号→"读出什么"；指标给定义式；数字标 §/Table/Eq 出处；区分"论文宣称 vs 我的批判"；
> 并在 `## ★ 对我们的启发` 一节把每条打到**我们自己这套 harness**（Claude Code 的 Bash/代码执行工具）身上。

## §1　TL;DR（一页讲清这篇在干嘛）

> 主讲提示：开场只讲一件事——**"动作用什么格式表达"是 harness 的一个可设计变量**，CodeAct 把它从 JSON 换成代码，能力就上去了。别急着讲实验，先把"代码 vs JSON"这个对比刻进听众脑子里。

**一句话**：LLM-agent 要"干活"，就得把"我想调这个工具、传这些参数"表达出来交给系统执行。业界默认用 **JSON / 文本工具调用**（一次一个动作）。CodeAct 主张：**把所有动作统一成一段可执行的 Python 代码**（`CodeAct` = Code as Action），用一个 **Python 解释器**当执行后端。这样一来——控制流（`if`/`for`）、多工具组合、把一个工具的输出当另一个工具的输入（数据流）、复用现成的 PyPI 库、以及"一段代码顶多步"——全都免费获得（§1、Fig 1、Table 1）。

**三个层面的贡献**（论文 §1 末、§2–§3）：

1. **范式（跨 T/L 层）**：论证"代码即动作"相对 JSON/文本的四条结构性优势（Table 1），并用**17 个 LLM** 的受控实验坐实（§2）。
2. **数据**：造了 **CodeActInstruct**——7k 条多轮 agent–环境交互轨迹的指令微调集，专门喂"从执行反馈里自我改进（self-debug）"这项能力（§3.1、Table 4）。
3. **开源 Agent**：基于 LLaMA-2-7B / Mistral-7B 微调出 **CodeActAgent**，能无 in-context 示例地用 Python 库做复杂任务（下载数据、训模型、画图）并自动 debug，且**不损害通用能力**（§3.2、Table 5）。

**关键数字（先抛，后面 §7–§9 展开）**：
- 在专门为"多工具、多轮、需组合"设计的新基准 **M³ToolEval** 上，最好的模型 **gpt-4-1106-preview 用 CodeAct 拿 74.4% 成功率**，比它自己用文本动作（53.7%）**高 20.7 个绝对百分点**，同时**交互轮数从 7.7 降到 5.5**（Table 3）。
- 论文的口径是"**成功率最多 +20% 绝对值、动作数最多省 30%**"，且"**模型越强，这个增益越大**"（§1、Fig 1 bottom）。

**属于 harness 的哪一层（Θ1）**：本篇主打 **T（Tools，工具/动作接口）** 层——它重新定义了"动作长什么样"；但因为"一段代码 = 多步 + 控制流"，它同时深入 **L（Loop，控制循环）** 层——把"原本要在外层循环里跑 N 轮的多步逻辑"塞进了"一步之内的一段代码"。库内它归 **E 组（编码 Agent / 集成系统）**，因为它把"动作接口 + 执行环境 + 微调数据 + 开源 Agent"打成了一个集成方案。

**够权威吗（Θ4）**：**ICML 2024** 接收（PMLR 235），UIUC（Heng Ji 组）× Apple 出品，代码/数据/模型/demo 全开源。它是**首篇系统论证"代码即统一动作空间"并配齐"指令数据 + 开源 Agent"三件套**的工作，被 **OpenHands（前 OpenDevin）** 等主流编码 Agent 直接采纳为行动范式——所以它是本库 E 组的 **canon 基石**，不是一篇普通前沿。

> **读出什么**：这篇的野心不是"再做一个更强的工具调用协议"，而是**换掉动作的"表示层"本身**。这正对上本库的中心命题——`Agent = Model + Harness`：同一个模型，只把 harness 里"动作怎么表达/怎么执行"这一块从 JSON 换成代码，能力就显著变了（§7 会给实锤）。

## §2　问题与动机：一个 agent 的动作，到底该用什么"语言"说？

> 主讲提示：这一页用 Why 三连的"问题层"。先讲清"动作空间"是什么、为什么 JSON/文本这条主流路线会卡住。

**先建立词汇**。一个 LLM-agent 的运行，可以拆成一个循环：**观察（observation）→ 思考（可选，chain-of-thought）→ 动作（action）→ 环境返回新观察**（§2.1、Fig 2）。这里的 **action（动作）** 是 agent 唯一能"影响世界"的出口——调工具、改文件、控制机器人、跑实验，全靠它。所以"**动作用什么格式表达**"是 harness 设计里一个绕不开的根本选择。

**Why（问题层）——主流的 JSON/文本动作卡在哪？**
论文开篇点名（§1 原文）：LLM-agent 通常被提示去"**产出一个预定义格式的 JSON 或文本**"来表示动作，而这种做法有两个内生局限：
1. **动作空间受限（constrained action space）**：你能做的动作，被"预先定义好的那批工具"钉死了。想干一件没预置工具的事，就得先由人去"造工具"。
2. **组合灵活性差（restricted flexibility）**：**没法把多个工具组合进一个动作**。典型 JSON/文本协议是"**一次只能调一个工具**"（Appendix F.4/F.5 的 formatting instruction 原文就写死 "You can only invoke one tool at a time"）。于是任何"多步、要串工具、要根据中间结果分支"的逻辑，都得被**摊平成一长串"一步一动作"的往返**。

**举个论文自己的例子（Fig 1，全篇的"题眼"）**：任务是"在 USA/Japan/Germany/India 里，找出买某型号手机最划算的国家"。
- **用文本/JSON 动作（Fig 1 左）**：得一个国家一个国家地来——先 `lookup_rates("Germany")`，环境返回汇率；再 `lookup_phone_price(...)`；再 `convert_and_tax(...)`……**每算一个国家就是好几个来回**，四个国家就是十几轮交互，最后还得人肉比大小。
- **用 CodeAct（Fig 1 右）**：一段代码搞定——
  ```python
  countries = ['USA', 'Japan', 'Germany', 'India']
  final_prices = {}
  for country in countries:               # ← 控制流：一个 for 循环覆盖所有国家
      exchange_rate, tax_rate = lookup_rates(country)
      local_price = lookup_phone_price("xAct", country)
      converted_price = convert_and_tax(local_price, exchange_rate, tax_rate)
      shipping_cost = estimate_shipping_cost(country)
      final_prices[country] = estimate_final_price(converted_price, shipping_cost)
  # ← 数据流：上一个工具的输出直接喂给下一个
  most_cost_effective_country = min(final_prices, key=final_prices.get)  # ← 复用现成库函数 min()
  print(most_cost_effective_country, final_prices[most_cost_effective_country])
  ```
  **一个动作，跑完全部四国、比完价、给出答案。**

> **读出什么**：JSON 动作的病根不是"格式丑"，而是**表达力（expressiveness）**——它是一种"命令式、一次一动作"的接口，天生装不下"控制流 + 组合 + 复用"。而代码是一门**图灵完备**的语言，这三样是它的母语。这就是全篇的立论核心，请务必在组会上停在这一页把它讲透。

## §3　核心主张：为什么"代码"是更好的动作表示（Table 1 逐条拆）

> 主讲提示：这是全篇最该讲透的一页。Table 1 给了四条优势，我把每条都翻译成"直觉 + 它到底省了什么"。这一页讲完，实验只是给它背书。

论文 Table 1 把"CodeAct vs JSON/文本"的优势列成四行。我按"直觉→定义→读出什么"逐条拆：

| # | 维度 | CodeAct（代码即动作）| JSON / Text（预定义格式）|
|---|---|---|---|
| 1 | **数据可得性** | 海量代码在 LLM **预训练**里已见过 | 需要为特定格式**专门 curate 数据** |
| 2 | **复杂操作**（循环/多工具组合）| **原生支持**（控制流 + 数据流）| 要费劲工程化，甚至得"为模拟 if 而新造工具" |
| 3 | **工具可得性** | 直接调用**现成软件包**（整个 PyPI）| 得人肉把工具从头/从现有软件里"扒"成协议 |
| 4 | **自动反馈** | 报错/traceback 是编程语言**自带的基础设施** | 要人去实现"把反馈从底层语言接回工具" |

**① 数据可得性（availability of data）**。
- 直觉：模型在预训练里读了海量 GitHub 代码，"写 Python"是它的**母语级技能**；而"按你这套 JSON schema 精确产出"是它**没怎么见过**的方言，需要额外微调数据去教。
- 读出什么：这条是 CodeAct 对**开源模型**特别友好的根因——§2.2 实验显示 CodeAct 的增益在开源模型上更明显（Table 2），正是因为"代码数据更容易获得，比专门的 JSON/text 工具调用格式更适合微调开源 LLM"（§2.2 原文）。

**② 复杂操作（complex operation）——这是最本质的一条**。
- 直觉：`if`/`for`、把中间结果存进变量、把多个工具在一段里串起来（组合），这些是代码"免费自带"的；JSON 协议要模拟"循环 4 次"就得往返 4 次，要模拟"条件分支"甚至得"定义一个新工具来 mimic if-statement"（Table 1 原文用词）。
- 读出什么：这直接对应 Fig 1 的"更少动作"——**一段带 for 循环的代码 = 原本要 N 个 JSON 往返**。论文 §1 原话："code inherently supports control and data flow, allowing … composition of multiple tools … with one piece of code, thereby unlocking LLMs' potential to tackle complex tasks by leveraging its pre-trained knowledge of programming."

**③ 工具可得性（availability of tools）**。
- 直觉：CodeAct 不必"发明工具"——它可以 `import pandas`、`import sklearn`，把**整个 PyPI（"覆盖广泛应用的人写 Python 包"，Table 1 脚注）**当成即用工具库。
- 读出什么：这把"扩展 agent 能力"从"人工造工具/写 wrapper"变成"import 一行"。Fig 3 里 CodeActAgent 就是靠 `import pandas`/`import sklearn`/`import matplotlib` 完成了"下载 auto-mpg 数据 → 清洗 → 训回归 → 画图"整条数据科学流水线，**全程零 in-context 示例**。

**④ 自动反馈（automated feedback）**。
- 直觉：程序出错会抛 **traceback**（Python 的报错/异常本就是给人 debug 用的自然语言级信息）。CodeAct 把这套现成的反馈基础设施直接接给 LLM，让它**看着报错自我修正（self-debug）**。JSON 协议则要人去实现"怎么把底层错误接回到工具层"。
- 读出什么：这条为 §3 的"多轮自我改进"埋了伏笔——self-debug 之所以能低成本实现，是因为"反馈机制已经是大多数编程语言的基础设施"（Table 1 脚注 3）。

> **Why（设计层）——为什么是 Python，而不是自造一门更"安全/受限"的 DSL？**
> 朴素替代 A：设计一门**领域专用语言（DSL）**，只暴露安全的、受控的动作。→ 会**丢掉②③**：DSL 没有现成生态（③ 归零），控制流也要重造（② 打折），更别提模型没在预训练里见过它（① 归零）。
> 朴素替代 B：坚持 JSON，但**允许一次传多个工具调用**（batch）。→ 仍然表达不了**依赖前一步结果的分支/循环**（数据流断裂），本质还是"静态的一批动作"，不是"可计算的一段程序"。
> CodeAct 选 **Python**，因为它是 **TIOBE 排名第一**、生态最大、模型最熟的语言（§2 原文），把①②③④ 一次拿满——**代价**是：你必须提供一个**能真正执行任意 Python 的环境（解释器/沙箱）**，并承担其安全风险（§Impact Statement 明确点了"沙箱逃逸→网络攻击"这一最坏情形）。这就是"代码即动作"的**天下没有免费的午餐**：把表达力拉满，把执行环境的责任也拉满。

## §4　CodeAct 到底怎么跑：交互框架与那段系统提示

> 主讲提示：这页把抽象的"代码即动作"落成一个能跑的循环，并亮出真正让它工作的那段系统提示（这就是 harness 的"配置"）。

**三个角色、一个循环（§2.1、Fig 2）**。CodeAct 把 agent 的世界建模成三方：**agent / user / environment**。定义几个词（先定义后用）：
- **interaction（交互）** = agent 与某个外部实体（user 或 environment）之间的一次信息交换。
- **observation（观察，输入）** = 每一轮 agent 收到的东西——可能来自 user（自然语言指令）或 environment（**代码执行结果**）。
- **action（动作，输出）** = agent 发出的东西——发给 user 就用自然语言，发给 environment 就是**一段 Python 代码**。
- 中间可选一步 **planning（chain-of-thought）**：动作前先用自然语言想一想。

于是一次 CodeAct 运行就是：`观察 →（想）→ 发一段 Python 代码 → 解释器执行 → 把 stdout/报错作为新观察喂回 → …` 直到 agent 决定用自然语言给 user 最终答复。

**真正让它工作的，是这段系统提示（Appendix E，零样本部署用的原文，chatML 格式）**——它就是 harness 的一份"配置"：
```
The assistant can interact with an interactive Python (Jupyter Notebook) environment and
receive the corresponding output when needed. The code should be enclosed using
"<execute>" tag, for example: <execute> print("Hello World!") </execute>.
The assistant should attempt fewer things at a time instead of putting too much code in
one <execute> block. The assistant can install packages through PIP by <execute> !pip
install [package needed] </execute> and should always import packages and define
variables before starting to use them.
The assistant should stop <execute> and provide an answer when they have already obtained
the answer from the execution result.
```
读这段提示能看出几个 harness 级设计决策：
- **动作的物理载体是一对 `<execute>...</execute>` 标签**——解释器就靠它切出"要跑的那段代码"。（这正是我们自己 harness 里 Bash 工具"命令块"的同构物，见 §Inspires-Us。）
- **执行后端是 Jupyter Notebook 内核**——所以**变量、import 的状态在多轮之间是持久的**（一轮里 `import pandas as pd`，下一轮还能用 `pd`）。这把"状态"从 agent 的上下文里卸载到了一个真实的运行时里。
- **允许 `!pip install`**——运行期可以**动态扩展工具集**，呼应 §3 的③。
- **"一次少做一点（fewer things at a time）"**——这是一句很关键的 harness 提示：它承认"一段代码顶多步"虽好，但**一口气写太多会难 debug**，于是引导 agent 走"小步执行 + 看反馈 + 再续"的多轮节奏。

> **读出什么**：CodeAct 不是"让模型自由写代码"这么简单——它配了一套**具体的接口约定（`<execute>` 标签）、一个持久的执行后端（Jupyter）、一条节奏引导（少量多次）**。这三样合起来才是"CodeAct 这个 harness"。把系统提示、标签协议、执行环境三者视为一体，正是本库"harness 是可配置系统层"的活教材。

## §5　动作格式三选一：同一个动作，三种写法（Table A.6）

> 主讲提示：这一页给"代码 vs JSON vs 文本"一个最小、最直观的并排对照——一张表胜过千言。

论文在 Appendix A.6 给了一个"同一个动作、三种格式"的最小例子（把一件日程加进日历）：

| 格式 | 同一个动作 `AddAgenda(content="Meeting with John", time="2023-10-26 09:00:00")` 的写法 |
|---|---|
| **CodeAct** | `AddAgenda(content="Meeting with John", time="2023-10-26 09:00:00")` |
| **JSON** | `{"action": "AddAgenda", "content": "Meeting with John", "time": "2023-10-26 09:00:00"}` |
| **Text** | `Action: AddAgenda, content: Meeting with John, time: 2023-10-26 09:00:00` |

**读法**：在**单个原子动作**上，三者其实"长得差不多"、信息量一样——所以你会预期它们表现相近。CodeAct 的优势**不在单个动作**，而在"当你需要**把多个这样的动作用控制流/数据流串起来**时"：
- CodeAct 版可以直接写 `for c in contacts: AddAgenda(content=f"Meeting with {c}", time=...)`；
- JSON/Text 版只能"一次一个 `AddAgenda`"，要加 N 个就往返 N 次。

这恰好解释了后面两组实验的分工（§7）：
- 在**只考原子调用**的 API-Bank 上（每次只调一个工具），CodeAct 与 JSON/Text **打平或略优**（因为此时用不上控制流，Table 2）；
- 在**要多工具、多轮、需组合**的 M³ToolEval 上，CodeAct **显著领先**（控制流/数据流终于派上用场，Table 3）。

> **读出什么（对"公平比较"的方法论提示）**：作者很聪明地把实验切成"原子 / 复合"两档。这让"CodeAct 好"这个结论**可证伪**——它明说"在原子任务上优势会被消掉（ablated）"（§2.2 原文）。承认"我的方法在某个 regime 下没优势"，正是本库反复强调的诚实度（Θ5）。

## §6　两个基准与三个研究问题：怎么把"代码更好"做成可证伪的实验

> 主讲提示：这页交代实验骨架——三个 RQ、两个基准、指标定义式。先把"成功率"和"平均轮数"这两个指标的定义讲清楚。

**三个研究问题（§2 开头）**：
- **RQ1**：LLM 因为预训练见过海量代码，用 CodeAct 会不会比 text/JSON 有优势？（→ §2.2，用 API-Bank 原子调用测）
- **RQ2**：CodeAct 能不能从"Python 天生的控制/数据流"里，在**复杂问题**上占便宜？（→ §2.3，用 M³ToolEval 测）
- **RQ3（附加）**：CodeAct 的多轮 + 复用现成软件，能进一步增强 agent 吗？（→ §2.4、§3，用 Fig 3 定性 + CodeActAgent 定量）

**两个基准（先讲清各测什么）**：
1. **API-Bank（重用 Li et al., 2023）**：测"**原子**工具调用"——每个实例只让模型生成**一个**工具调用（Python 函数调用 / JSON / 文本三选一），用 **correctness（正确率）** 打分：把 ground-truth API 输出和模型生成 API 的执行输出对齐匹配（§2.2）。
2. **M³ToolEval（本文新造，§2.3、Appendix A.7）**：名字里三个 M = **m**ultiple tools × **m**ultiple calls × **m**ulti-turn。**82 个人工精编任务**，覆盖网页浏览、金融、行程规划、科学、信息处理，每个域都配一套**手工设计的工具**，且**故意不给 in-context 示例**（测零样本、像"不会 few-shot 的新手用户"那样用）。它填的空白（Table A.7）：现有工具基准要么不支持多轮、要么不支持多工具、要么只支持某一种动作格式——M³ToolEval **同时支持 CodeAct & JSON & Text 三种格式**且要求多轮多工具，才能公平地把三种动作格式摆在一起比。

**指标定义式（给公式前先给直觉）**：
- **成功率（success rate）**：直觉——"做对的任务占比"。定义——模型最终提交的答案与 ground-truth 解**精确匹配**的实例百分比（§2.3 原文 "percentage of the model proposed answers that match the ground-truth solutions"）。交互在"最多 10 轮"或"提交了正确解"时终止。
- **平均轮数（avg. turns，↓ 越低越好）**：直觉——"平均要几个来回才搞定"。定义——所有被评实例上的平均交互轮数（§2.3）。它衡量**效率**：同样做对，轮数越少说明"一段代码顶了越多步"。

> **读出什么**：这两个指标一"对不对"（success）、一"省不省"（turns），正好对应 CodeAct 宣称的两个好处——**更准 + 更少动作**。把它们分开报，才能看出 CodeAct 的增益是"既做得更对、又更省往返"，而不是拿轮数换准确率。

**Why（结果层）——为什么原子任务上 CodeAct 只是"打平"？**

2. **闭源模型上 JSON 常常反超**（如 gpt-4 系列 JSON 82 > CodeAct 76）——作者诚实解释：闭源模型"可能经过了针对 JSON 的定向微调"（§2.2 原文），所以它们的 JSON 能力被专门喂强了。**这是对自己方法的一处诚实让步**：CodeAct 不是在所有 regime 都赢。

**Why（结果层）——为什么复合任务上 CodeAct 能 +20.7 分？**
- **准确率**：最好的 **gpt-4-1106-preview 用 CodeAct 拿 74.4%，比它用文本动作（53.7%）高 20.7 个绝对百分点**（§2.3 原文点名这个 +20.7%）。机制上——这些任务"往往需要对多个工具做精细的协调与组合"（§2.3），而 CodeAct 用一段带控制流/数据流的代码就能把这套协调**一次表达**，text/JSON 却要拆成一长串往返，中途任何一步的格式/协调错都会累积成失败。
- **效率**：CodeAct 同时把**轮数从 7.7 压到 5.5**（省 ~2.1 轮）。**17 个模型里，12 个在成功率上、12 个在轮数上 CodeAct 最优**（Table 3 底部频次）——这正是论文"成功率最多 +20% 绝对、动作数最多省 30%"这句总结的来源（§1）。
- **"模型越强，增益越大"**：Fig 1 bottom 显示——从 Llama-2-70b 到 gpt-4，CodeAct 相对 JSON/Text 的领先幅度**随模型能力增强而变宽**（§1 原文 "gains widen as the capabilities of the LLMs increase"）。直觉：越强的模型越能写出"正确的、带复杂控制流的长代码"，于是越能兑现代码的表达力红利。

> **读出什么（Θ2，回扣 `Agent = Model + Harness`）**：Table 3 就是一份实证——**模型没变（同一个 gpt-4-1106-preview），只把 harness 里"动作的表示层"从 text 换成 code，成功率就 +20.7、轮数 −2.2**。这和标杆 Harness-Bench "换 harness 摆 23.8 分" 是同一类证据，只不过 CodeAct 动的是 harness 的**一个具体旋钮（动作格式）**，而 Harness-Bench 动的是**整套 harness**。两者互补：CodeAct 告诉你"哪个旋钮值得拧"，Harness-Bench 告诉你"整机差异有多大"。

> **一处诚实的裂缝（作者主动点出）**：即便用上 CodeAct，**开源与闭源仍有巨大鸿沟**——最好的开源模型只到 13.4%，而 gpt-4-1106-preview 74.4%（§2.3 原文）。作者由此呼吁"零样本设定下，亟需提升开源 LLM 的实用能力"——这正是 §3 造 CodeActAgent 的动机。**换句话说：好的动作格式能放大模型的能力，但放大不出模型本来没有的能力**（Θ5 的种子）。

## §8　让开源模型也会 CodeAct：CodeActInstruct 数据（Table 4）

> 主讲提示：这页讲"数据怎么造"。重点不是数字，而是那条**数据筛选哲学**——只留"犯错→自我纠正"的轨迹。

既然开源模型用 CodeAct 还很弱（§7.2），作者就造数据把这项能力"教"进去。**CodeActInstruct = 7,139 条**多轮 agent–环境交互轨迹（Table 4），覆盖五个能力域（Table A.9）：

| 域 | 能力 | 数据源 | 实例数 |
|---|---|---|---:|
| 信息检索 | 用 search API 查资料 | HotpotQA | 1,664 |
| 软件包(工具)使用 | 用 Python 数学库(sympy) | MATH | 1,732 |
| 软件包(工具)使用 | 导入包/造新工具 | APPS(代码生成) | 647 |
| 外部记忆 | 用 pandas/sqlite3 做表格推理 | WikiTableQuestion | 2,031 |
| 具身规划 | 用机器人 API 操作环境 | ALFWorld | 1,065 |
| **合计** | | | **7,139** |

**Why（设计层）——为什么不是"有正确答案的轨迹就都要"，偏要挑"先错后对"的？**
这是全节最精彩的一条设计决策（Appendix G.2 "Data Selection Heuristic"）。作者只保留满足三条的轨迹：
1. **Code-as-Actions**：剔除那些没按"代码即动作"格式来的轨迹（API 调用错误、或格式无法解析执行的）。
2. **Self-Improving（自我改进）**：**只保留"模型一开始出错、但后续在交互中纠正过来"的轨迹**；并**主动剔除"全程没报错就一次做对"的成功轨迹**——原文理由是"防止模型学成'即便执行了错误代码也照样蒙对答案'"（Appendix G.2 原文）。
3. **Instruction-Following**：剔除"没能回应用户指令"的异常轨迹（用"交互轮数为奇数"来识别）。

- 朴素做法：把所有"最终答对"的轨迹都拿来训。→ 会**混进"运气型成功"**（执行了错代码但碰巧答对），模型可能学到"不必在意执行反馈"。
- 本文做法：**刻意偏向"犯错→看 traceback→修正"的轨迹**，因为要训的恰恰是 **self-debug** 这项 harness 级能力。代价：能用的轨迹变少（过滤后从 6,985 条 gpt-3.5/claude 轨迹留 6,728，从 413 条 gpt-4 轨迹留 411）。

**与前作对比（Table 4）**：CodeActInstruct 比 FireAct、AgentInstruct **覆盖域更广、且专门含"自我 debug"质量数据**，规模是 AgentInstruct/FireAct 的 **3.8×/3.5× 轨迹、5×/19× token**。为保通用能力，训练时还**混入通用对话数据**（ShareGPT、OpenOrca、CapyBara 等 69,230 条，Table 4）。

> **读出什么**：这条"只留自我纠错轨迹"的筛选法，是把"**我们想要 agent 学会的行为**"直接编码进训练分布。它和标杆 Harness-Bench 的 Integrity 准则（任务设计层堵 reward hacking）是同一种警惕的两面——一个在**数据层**防"蒙对"，一个在**评测层**防"作弊"。

## §10　核心迁移概念：为什么"一段代码"≈"把多步压进一步"

> 主讲提示：这页把全篇最有迁移价值的抽象讲清——它值得你带回自己的项目。

CodeAct 最该被记住的，不是某个数字，而是一个**动作粒度（action granularity）**的重构：

- **JSON/文本动作 = 细粒度、命令式、外循环驱动**：每个动作是"一条指令"，多步逻辑靠**外层 agent 循环**反复往返来驱动（想循环 4 次 → agent 被调用 4 次 → 4 次 LLM 推理 + 4 次往返）。
- **代码动作 = 粗粒度、声明+过程式、内循环内联**：一个动作是"**一段程序**"，多步逻辑被**内联进这段代码的控制流**里，由**解释器**一次跑完（想循环 4 次 → 写一个 `for` → **1 次** LLM 推理 + 1 次往返，循环在解释器里转）。

这带来两个直接后果，正好对上实验：
1. **省 LLM 调用/往返**（→ Table 3 轮数下降）：把"该由确定性代码干的活（迭代、算术、比较、数据搬运）"从"LLM 的 token 生成"里挪到"解释器的执行"里——**便宜、准确、不耗上下文**。
2. **少累积错误**（→ Table 3 成功率上升）：多步往返里，每一轮的格式错/协调错都会累积；把多步收进一段代码，**中间步骤由解释器保证一致性**，只有"代码逻辑本身"是 LLM 要负责的。

> **Why（设计层）这个抽象为什么比"给更多工具"更本质**：你可以给 JSON-agent 加一百个工具，但只要它还是"一次一个、外循环驱动"，它就仍然装不下"依赖中间结果的循环/分支"。CodeAct 的贡献是**换了动作的计算模型**（从"一串指令"到"一段可计算的程序"），而不是"扩充指令集"。这与标杆 Harness-Bench 的洞见同源——harness 的优劣"**不在工具多不多，而在有没有把推理忠实搬运到可验证的动作**"（Harness-Bench §5.2）；CodeAct 恰恰给了"把多步推理忠实搬运成一段可执行、可验证代码"的具体载体。

## §11　与相邻工作的边界：CodeAct 不是"让 LLM 写代码解题"（Appendix A/B）

> 主讲提示：这页澄清一个最容易被误解的点——"代码即动作"≠ "code-as-solution"。区分清楚，才知道它的真正新意在哪。

很多前作也让 LLM"生成代码"来解题（PaL、Code-as-Policies、ViperGPT、Voyager、Data Interpreter、Code4Struct…）。作者在 Appendix A **划清界限**：

- **它们大多是"单轮、静态代码"**：模型一次性生成"一段完整代码/一个函数定义"作为解，**首次跑错就算失败**，**不能根据环境观察动态改后续动作**（Appendix A 原文）。这忽视了"看着中间执行结果再调整下一步"的潜力。
- **CodeAct 是"多轮、动态动作"**：每一步可以只跑一小段（原子动作或组合函数），**看反馈 → 动态产出下一段代码 / 修错**，与任意形式的环境观察兼容（§2.1）。
- **对 Voyager（Wang et al., 2023a）的精确区分**：Voyager 也做迭代 prompting，但它的动作空间被限制成"**函数定义**"——它一次要写一个"把整件事从头到尾想清楚的函数"（需要强领域知识预想所有分支），执行整段、看反馈、再更新**整个函数**；且依赖大量 prompt engineering 喂当前状态。CodeAct 则**允许在 fly 上动态调整原子动作**，且上下文只需装"过去的动作与观察"，**无需人工 prompt 工程去筛信息**（Appendix A 原文）。
- **对 TaskWeaver（Qiao et al., 2023，并行工作）的定位（Appendix B）**：TaskWeaver 也提"code-first"统一动作，但停留在"闭源模型 + 少量定性例子"的概念演示；CodeAct 的增量是**做了 17 模型的严格分析 + 开源指令数据(CodeActInstruct) + 开源 Agent(CodeActAgent)**，"把概念落成了给开源社区的可用资源"。

> **读出什么（Θ4，canon 的自我定位）**：CodeAct 的历史坐标不是"发明了用代码解题"，而是**把"代码"从"解题的媒介"提升为"agent 与环境多轮交互的统一动作空间"**，并配齐数据与开源 Agent。这一步"从 solution 到 action space"的抬升，正是后来 OpenHands 等把它当行动范式基石的原因。

## §12　实验设置与规模：把"17 个 LLM"这句话拆开

> 主讲提示：这页把"实验有多硬"交代清楚，方便判断结论的可信度。

- **RQ1/RQ2 的横向比较**：**17 个 LLM**（开源 + 闭源），每个都在 API-Bank（原子）与 M³ToolEval（复合）上、以 **CodeAct / JSON / Text 三种动作格式**各跑一遍（Table 2、Table 3）。这是一个"模型 × 动作格式"的受控矩阵——**同模型、只换动作格式**，所以观测到的差异干净地归因于"动作表示层"。
- **M³ToolEval 协议**：82 任务；每轮模型可"发一个动作"或"提交一个待精确匹配的解"；**最多 10 轮**；零 in-context 示例（Appendix F 给了三种格式各自的 formatting instruction 原文）。
- **CodeActAgent 训练**：全参 SFT，LLaMA-2 序列长 4,096 / Mistral 16,384；5 epoch、取第 3 个 checkpoint；训练吞吐 ~9k token/s（Appendix D）。
- **轨迹生成**：用 gpt-3.5-turbo-0613、claude-1-instant、claude-2 在下采样数据上生成交互轨迹；难题上用 gpt-4-0613 兜底（§3.1）。

> **读出什么**：结论的强度来自"**17 模型 × 3 格式 × 2 基准**"这个矩阵的一致性——不是单点跑赢，而是"12/17 个模型在成功率、12/17 在轮数上 CodeAct 最优"的**趋势性**结论。这比"我这一个 agent 分数高"要硬得多。

## §13　讨论与 regime 诚实：代码动作总是更好吗？（Θ5）

> 主讲提示：这页是判断力高地。别把"代码即动作 > JSON"讲成绝对真理——它分 regime，且有明确代价与风险。

**什么条件下 CodeAct 明显更好（增益 regime）**：
- 任务**需要多工具组合、控制流、依赖中间结果的分支/循环**（→ M³ToolEval +20.7 分，Table 3）；
- 模型**足够强**，能写出正确的复杂代码（→ Fig 1 "增益随能力变宽"）；
- 环境**允许多轮 + 有执行反馈**（→ self-debug 才有意义）。

**什么条件下 CodeAct 优势消失或反转（收窄 regime）**：
- **原子任务**：只调一个工具时，控制/数据流用不上，CodeAct 与 JSON/Text 打平，闭源模型上 JSON 甚至反超（§2.2、Table 2）——因为闭源模型被定向喂强了 JSON。
- **弱模型**：最好的开源模型在 M³ToolEval 仍只 13.4%（§2.3）——**代码格式放大不出模型没有的推理**。
- **没有安全执行环境**：CodeAct 的全部红利都以"能真跑任意 Python"为前提。

**必须诚实说出的三个代价/风险（论文自陈）**：
1. **安全（最重）**：CodeAct 直接授予 agent "在沙箱里自由执行代码"的权力——"最坏情形下（Sci-Fi 式），它可能突破沙箱限制、发起网络攻击"，作者明确呼吁"需要更好的安全机制来 safeguard 自主 agent"（§Impact Statement，引 Tang et al., 2024）。这是 JSON 受限动作**天然更安全**的地方——**表达力与安全性在这里是一对张力**。
2. **幻觉/未验证状态**：CodeActAgent 仍会犯 LLM 常见幻觉——"比如凭空想象一个变量的内容而没真的把它 print 出来"（§Conclusion/Limitation 原文），提示需要后续对齐。
3. **训练数据伪影（一个很诚实的失败案例，Appendix H）**：LLaMA-2 版 CodeActAgent 在 M³ToolEval 上**反常地没提升**；查模型输出发现怪异 token（如凭空冒出 "nobody is here to check your work, you can do whatever you want"、"sierp"，Appendix H 图中蓝色标注）。作者排查自家训练混合数据**未发现来源**，于是**诚实推断这些伪影可能来自 LLaMA-2 的预训练语料**（他们无法访问），外加 LLaMA-2 基座本就弱于 Mistral。**这是全篇最诚实的一段**——它承认"我们的方法在一个基座上失灵，且我们不完全知道为什么"。

> **读出什么（Θ5，与标杆呼应）**：把这段和标杆 Harness-Bench §6 的 regime 讨论并读——两篇给出同一个诚实结论的两半：**harness 设计（含动作格式）的价值分 regime**。CodeAct 补上的量化坐标是"**原子 vs 复合**"这条轴（Table 2 打平 / Table 3 大胜）和"**模型强弱**"这条轴（Fig 1 增益随能力变宽 / 开源仍 13.4%）。所以诚实表述是：**任务越需要组合/控制流、模型越强、环境越允许安全执行，代码即动作越主导；反之（原子任务、弱模型、无安全沙箱）它退居其次甚至反转，且始终背着"执行安全"这笔账。**

## §14　局限与批判（论文自陈 + 我的补充）

**论文自陈的局限（诚实）**：
- **安全边界未解**：只给了"需要更好安全机制"的呼吁，没给方案（§Impact Statement）。
- **仍会幻觉**：会想象未验证的变量/状态，需后续对齐（§Conclusion）。
- **基座依赖**：LLaMA-2 版失灵且原因未完全查明（Appendix H）。
- **自我改进尚初级**：CodeActAgent 只展示了"有限的"self-improving（§Conclusion 原文 "limited self-improving capability"），远非成熟。

**我的补充批判**：
- **成功率靠"精确匹配"**：M³ToolEval 用"答案与 ground-truth 精确匹配"判对（§2.3）。这对**开放式/多解**任务不友好——一个正确但表述不同的答案会被误判为失败，可能**系统性低估**所有格式（尤其偏语言的任务）。CodeAct 的相对优势应更稳，但绝对分要打问号。
- **82 任务、5 域、手工工具**：M³ToolEval 规模不大且工具是作者手工设计，"复杂工具任务"的分布本身没有 ground truth；换一批工具/任务，+20.7 的幅度未必稳定。
- **"代码"把安全成本外部化了**：论文把执行安全列为"future work"，但在实践中，**给 agent 一个能跑任意 Python 的沙箱**是整套方案里最贵、最难的一环（远比"设计 JSON schema"贵）。评价 CodeAct 的"性价比"时，这笔隐性成本不该被"成功率 +20"盖过。
- **self-debug 的双刃**：§8 刻意训"先错后对"，好处是学会自愈；但也可能让模型**养成"先乱跑再修"的习惯**，在真实高风险环境（真删了文件、真发了请求）里，"先错"这一步本身就是不可逆伤害——沙箱里安全，生产里未必。

## ★ 对我们的启发（Inspires Us）

> 这一节是组会高潮，也是本库的独门优势：**我们（Claude Code / 本课 m9.* 的 agent）本身就活在一个 harness 里，而且我们的 harness 里那个 `Bash`/代码执行工具，本质上就是 CodeAct 的一个实例**——agent 发一段命令/脚本、由 shell/解释器执行、把 stdout/stderr 作为观察喂回。所以 CodeAct 的每一条洞见，都能直接打到我们自己身上。

➤ **a. 可直接借用的招（Θ3：打到我们自己的 harness）**：CodeAct 的核心机制——**"用一段可执行代码当动作、用 `<execute>` 标签切块、用持久解释器(Jupyter)保状态、用 traceback 触发 self-debug"**——我们其实**已经有了同构件**：我们的 `Bash` 工具就是"发一段代码→执行→回传 stdout/stderr"。可直接借的三点：(1) **把'一次一个原子工具调用'的任务改写成'一段脚本'**——凡是"对一批对象做同样操作/需要中间结果分支"的活（如"对 40 篇 paper 各跑一次同样处理"），不要发 40 次工具调用，发**一段带 `for` 循环的脚本**，一次跑完，省往返、省上下文（正是 Fig 1 → Table 3 轮数下降的机制）；(2) 借它 Appendix E 的**"fewer things at a time"节奏提示**，在我们的循环里引导"小步执行 + 看反馈 + 再续"，而不是一口气写巨长脚本难 debug；(3) 借它的 **`!pip install` 动态扩工具**思路——运行期按需装库，而非预置一切。

➤ **b. 可迁移到我们模块的思路（transfer）**：CodeActInstruct 的**"只保留'先错→看 traceback→修正'的轨迹、并主动剔除'一次蒙对'的轨迹"**这条数据筛选哲学（Appendix G.2），可以整体迁到 auto-research 库的 `m9.6`（评测沙箱）/ 任何"训/评 agent self-debug"的场景：给我们自己 agent 的执行轨迹打标——是"一次做对 / 先错后修 / 错了没修"，**只把'先错后修'的当正样本**去强化"从反馈自愈"的能力。迁移前提：我们的 trace 要先**结构化录制 stderr/报错与后续修正动作的配对**（这正是下一步要补的 instrumentation），否则无法自动识别"先错后修"。

➤ **c. 它暴露的开放问题 = 我们的机会（open problem → opportunity）**：CodeAct 把**执行安全**完全推给了"future work"（§Impact Statement 只呼吁、没方案）——而这恰恰是我们这类**真能跑 Bash/改文件**的 harness 最该收口的地方。机会：设计一个**"代码动作的分级闸门"**——把 agent 发出的每段代码先做**静态/意图分类**（只读查询 / 可逆写 / 不可逆副作用如删文件、发网络请求），只读的直接放行、不可逆的要二次确认或落到沙箱。可下手的第一步：在我们的 `Bash` 工具前加一个**"命令风险分类器 + 干运行(dry-run)"**，量化它能否在不掉成功率的前提下拦住"先错这一步就造成不可逆伤害"的情况（正对 §14 我批判的"self-debug 双刃"）。

➤ **d. 与本库其它论文/模块的连接（connect the dots）**：与标杆 **Harness-Bench(2605.27922)** 正面互补——Harness-Bench 证明"换整套 harness 摆 23.8 分"，CodeAct 证明"只换'动作格式'这一个旋钮就摆 20.7 分"，二者一起坐实 `Agent = Model + Harness`，且 CodeAct 精确指出了"动作表示层"是那个高杠杆旋钮；与 **H 组工具失败恢复（如 Hell-or-High-Water）** 呼应——CodeAct 的 self-debug 正是"Robustness"这一项的具体实现；与 auto-research 的 `m9.8`（独立验证收口）共享 §8/§14 的隐忧——"先错后修"数据若没验证器把关，可能训出"敢乱跑"的 agent，需要外部 grounding。

➤ **e. 如果我来做下一步（第一人称、可执行）**：我会在我们某个 `m9.*` agent 上做一个**最小 A/B**——同一批"需要对多对象做同样处理 + 依赖中间结果"的任务，一组走"逐个原子工具调用"、一组走"发一段带 `for`/`if` 的脚本一次执行"，测**(1) 平均工具往返数是否像 Table 3 那样下降 ~30%、(2) 成功率是否上升、(3) 上下文 token 消耗是否显著降低**。若复现，就把"把多步逻辑打包成一段脚本"设为我们 harness 面对"批量/组合类"子任务时的**默认动作策略**，并配上 c 里的风险闸门兜住安全。

## §15　版图定位（canon 坐标 + 在本库的位置）

- **时间坐标（Θ4：canon）**：**2024 ICML 基石**。它"定义/奠基了什么"——把"**代码即统一动作空间（Code-as-Action）**"这一行动范式系统化，并首次配齐"**四条优势论证(Table 1) + 17模型受控实验 + M³ToolEval 基准 + CodeActInstruct 数据 + CodeActAgent 开源模型**"的完整闭环。"后续谁在它上面长肉"——**OpenHands（前 OpenDevin）** 等主流编码/软件工程 Agent 直接采用"代码即动作"作为行动范式；更广地，今天绝大多数"能跑代码/命令"的 agent（包括我们自己的 `Bash` 工具）都是这一范式的实例或近亲。它把"agent 该怎么动作"从"JSON 工具调用"这一条默认路，正式劈出了"写代码"这另一条主路。
- **相对基石推进了哪一步**：相对 ReAct（把"推理+行动"交错、但动作仍是文本/工具）、相对 Toolformer（教模型调 API），CodeAct 把**动作的"表示层与计算模型"**换了——从"一串命令式指令"升级到"一段图灵完备的可执行程序"，从而一次拿下控制流/组合/复用/多步。
- **E/T/C/L/O/V 归属（Θ1）**：本篇坐 **T（动作/工具接口）** 层为主、**L（控制循环）** 层为辅（因"一段代码内联了多步循环"），整体属库内 **E 组（编码 Agent / 集成系统）**——它把接口+环境+数据+Agent 打成一体。
- **回扣中心命题（Θ2）**：`Agent = Model + Harness`。CodeAct 的贡献是**精确定位并拧动了 harness 里"动作表示"这一个高杠杆旋钮**——同模型、只换动作格式，M³ToolEval 上 +20.7 分、−2.2 轮（Table 3）。它是"harness 决定能力"这一命题的**旋钮级实证**（Harness-Bench 是整机级实证，二者互补）。
- **在本库的位置**：**E 组 canon 锚点**。读完它，再看任何"编码 Agent / 软件工程 Agent"（OpenHands、SWE-agent 系）都可以问一句："它的动作是 CodeAct 式的代码，还是受限的 JSON/工具调用？它有没有 self-debug 的执行反馈闭环？"——CodeAct 提供了评判这些系统"行动范式"的坐标系。

## §16　组会讨论问题（留给大家吵）

1. **原子打平、复合才赢**——那么在**我们**的真实任务分布里，"原子 vs 复合"占比几何？如果我们的任务多是原子的，上 CodeAct 的收益可能没论文那么大——怎么先量一量再决定？
2. CodeAct 的全部红利押在"**能安全执行任意代码**"。在我们这种**真能改文件/发请求**的 harness 里，这笔安全成本该怎么定价？它会不会把"+20 成功率"吃掉一部分？
3. §8 刻意训"**先错后修**"。这在沙箱里是美德，在生产里"先错"这一步可能不可逆。怎么设计"**可逆性感知**"的 self-debug（先在 dry-run/影子环境里错，确认对了再对真实世界动手）？
4. Fig 1 说"**模型越强、代码动作增益越大**"。反过来推：随着底座越来越强，JSON 工具调用会不会被"代码动作"彻底取代？还是说在"要审计、要安全、要确定性"的场景里，受限的 JSON 动作反而更该保留？
5. Appendix H 的"训练伪影"很吓人（凭空冒出"没人检查你，随便干"）。这对"**用别人预训练的模型做 agent**"意味着什么风险？我们该怎么在上线前探测这类潜伏伪影？

## §17　一页速记

- **命题**：agent 的"动作"用什么表示是 harness 的一个高杠杆旋钮；CodeAct 主张用**可执行 Python 代码**，而非一次一个的 JSON/文本工具调用。
- **为什么更好（Table 1）**：①代码数据预训练里海量(对开源友好) ②原生控制流/数据流/多工具组合 ③直接复用整个 PyPI ④traceback 自带 → self-debug。核心是"**一段代码顶多步、可循环、可组合、可复用**"。
- **做法**：`<execute>` 标签切块 + Jupyter 持久解释器 + "少量多次"节奏 + 看报错自愈（Appendix E）。
- **铁证（Table 3, M³ToolEval, 82 任务多工具多轮）**：gpt-4-1106-preview 用 CodeAct **74.4%**，比文本 **+20.7 分**；轮数 **7.7→5.5**；17 模型里 12 个在成功率、12 个在轮数上 CodeAct 最优；**模型越强增益越大**（Fig 1）。
- **原子任务打平**（Table 2, API-Bank）：控制流用不上，CodeAct≈JSON/Text，闭源模型 JSON 甚至反超（被定向喂强）。
- **数据(Table 4)**：CodeActInstruct 7k 轨迹，**只留"先错→看 traceback→修正"、剔除"一次蒙对"**（Appendix G.2）→ 训出会自愈的 **CodeActAgent(7B)**，agent 任务碾压同尺寸、逼近 70B，且**不伤通用能力**（Table 5）。
- **诚实(Θ5)**：分 regime——复合/强模型/有安全沙箱时代码动作主导；原子/弱模型/无沙箱时退居其次甚至反转。**代价**：执行安全未解（§Impact）、仍会幻觉、LLaMA-2 版因预训练伪影失灵（Appendix H）。
- **canon 坐标(Θ4)**：2024 ICML 基石，定义"代码即动作"范式，OpenHands 等在其上长肉。
- **对我们(Θ3)**：我们的 `Bash`/代码执行工具就是 CodeAct 的实例——下一步做"原子调用 vs 一段脚本"的 A/B 测省不省往返，并给代码动作加一道"风险分类 + dry-run"闸门兜住安全。
