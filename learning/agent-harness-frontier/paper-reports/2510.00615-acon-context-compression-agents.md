# ACON：为长程 LLM 智能体做「无梯度」上下文压缩优化

> **本篇属 agent-harness 库 D 组（上下文，C 层）**。它回答一个我们自己每天都在踩的痛点：agent 跑长了，历史和观察越滚越大，既烧内存又把模型带歪。ACON 的解法有两层新意——① **不训练模型**，而是把「怎么压」写成一段自然语言 **压缩指南 (compression guideline)**，用「成功轨迹 vs 失败轨迹」的对比反馈去**无梯度地**迭代优化这段指南；② 再把「带着最优指南的大模型压缩器」**蒸馏进一个小模型**，让压缩这件事既便宜又快。
>
> 写作对齐库内 v2 标杆 [`2605.27922-harness-bench`](2605.27922-harness-bench-measuring-harness-effects.md) 的密度与诚实度：公式前先给直觉、先定义符号；指标给定义式；数字标 §/Table/Eq 出处；区分「论文宣称」与「批判」；Why 三连至少补齐设计层；强制 `## ★ 对我们的启发` 打到我们自己 harness。

## §1　TL;DR（一页讲清这篇在干嘛）

> 主讲提示：开场先把「长程 agent 的上下文炸了」这个共同痛点摆出来，再说 ACON 的两把刷子——「无梯度优化压缩指南」+「蒸馏到小压缩器」。这两点是全篇的命门，后面所有内容都在为它们服务。

**一句话**：长程 LLM agent 在真实环境里干活，会不断累积「观察 (observation) + 交互历史 (history)」，导致上下文**无界增长 (unbounded context growth)**（论文 Figure 2 红线：某任务 30 步内输入 token 从 ~7,200 峰值飙到 ~22,643）。这带来两个瓶颈（§1）：**① 推理内存爆炸**——Transformer 的 KV cache 随上下文线性膨胀，长程任务算力上不可行（Vaswani et al., 2017）；**② 推理质量退化**——过长上下文**稀释了相关信息**，用过时/无关细节把模型带偏（Shi et al., 2023，"distracted by irrelevant context"）。ACON（Agent Context Optimization）用一个统一框架同时压 history 和 observation，做到"压得狠又不掉分"。

**它属于 harness 的哪一层（Θ1）**：本篇正打 **C（Context）层**——上下文/记忆管理。它是一个**可插到任意 ReAct agent 上的上下文压缩组件**，不改工具层（T）、不改控制循环（L）的骨架，但直接决定循环每一步"喂给模型看什么"。因为它**模型无关 (model-agnostic)**、纯在提示词层面工作，所以对 E（环境）也没有耦合。

**回扣全库论点（Θ2）Agent = Model + Harness**：ACON 是"**同一个模型、只换 harness 的上下文组件，能力就摆动**"的又一实证。最刺眼的一组数字（§4.4）：把上下文压缩组件从"无"换成 ACON，**Qwen3-14B 这个小模型在 AppWorld 上从 25.6% 跳到 33.9%（+32.4% 相对提升）**、在 8-objective QA 上从 0.158 EM 跳到 0.23 EM（**+45.6%**）。模型权重一字没动，动的只是"上下文这层脚手架怎么管"。

**三条带走的结论**：
1. **无梯度也能优化压缩**：把"压缩策略"写成一段可读的自然语言指南，用**对比式任务反馈 (contrastive task feedback)**（成功 vs 失败轨迹）当"自然语言梯度"去迭代它——完全不碰模型权重，因此**能用在闭源 API 模型上**（这是 RL 类方法做不到的，见 §8 / Table 7）。
2. **压缩不只省钱，还能纠错**：压缩不是单纯截断，好的压缩把"真正决定成败的依赖"提炼出来，反而**降低了 distraction**——小模型受益尤其大（AppWorld +32.4%、QA +46%）。
3. **压缩能力可蒸馏**：把"大模型 + 最优指南"的压缩行为**蒸馏进一个小模型**（Qwen3-14B/8B、Phi-4），**保留 >95% 教师性能**，把每次压缩的 API 成本从 gpt-4.1 的 $0.045 降到 **$0.0004（降 99.1%）**，让昂贵大模型只留给 agent 本体（§4.3 / §B.3）。

**够新够权威（Θ4）**：2026 年 ICML（PMLR 306）录用，**Microsoft** 出品（一作 KAIST 博士生 Minki Kang 在 MSR 实习完成）。它相对本库已有的上下文/记忆工作（MemGPT、AgentFold、MEM1/ReSum）的**关键增量**是：把压缩从"要么靠脆弱手工启发式、要么靠 RL 训模型"这两条老路，切到**第三条路——纯提示词层的无梯度优化 + 蒸馏**（§2 "In contrast"）。

## §2　问题与动机（why）：为什么"上下文压缩"值得为 agent 单独做

> 主讲提示：这页用 Why 三连的"问题层"。要讲清两件事：(1) 为什么长程 agent 的上下文问题**特别**难（不是普通长文本压缩）；(2) 为什么现有三类做法都不够。

**Why（问题层）——不解决会卡住什么？**
LLM 正越来越多地以 **agent** 形态部署：在动态环境里调工具、改工作区、维护演化中的状态（§1）。这类任务"跨越 dozens~hundreds 步"（§2），要求 agent 维护一份**连续的**观察/工具输出/状态记录。此时上下文**不是可有可无的附属，而是基础性的**（"not auxiliary but foundational"，§1 原文）——丢掉一个文件路径或一个 API 参数这种"单个细节"，整条工作流就可能脱轨。随着交互累积，上下文无界增长（Figure 2），两个瓶颈同时压过来：

1. **推理内存不可行**：KV cache 随上下文线性增长，长程下算力"prohibitive"（§1，引 Vaswani et al., 2017）。
2. **决策质量退化**：过长上下文"稀释相关信息"，模型被过时/无关内容分心（§1，引 Shi et al., 2023）。

而且 agent 的上下文**和普通对话/文档压缩不是一回事**（§1 productivity 场景）：它要保留**异质信号**——事实历史、动作-结果关系、成功前置条件（如"需要先登录"）、未来决策线索（如"截止日期"）。所以"简单截断 (token truncation) 或通用摘要 (generic summarization) 都不够"（§1 原文），因为它们很容易把这些关键细节丢掉。这就要求压缩必须**在激进缩减与精确保留任务相关状态之间取得平衡**（§1）。

**Why（设计层）——为什么现有三类压缩都不够（这是最该讲的一层）？**
论文 §2 把已有做法归成三/四类，逐一说明它们"会怎样失败"：

| 现有路线 | 代表 | 为什么不够用（论文 §2 论证） |
|---|---|---|
| **对话记忆摘要** | MemGPT (Packer 2023)、Wang 2025a | 面向"对话连贯性"，不是"状态追踪"——丢的正是 agent 要的状态 |
| **文档中心压缩** | RECOMP (Xu 2024)、LLMLingua (Jiang 2023) | 假设"单步推理，用完即弃"；agent 是多步，信息**相关性随进度频繁漂移** |
| **agent 专用启发式** | Deng 2023、Smith 2025（OpenHands 凝练） | **脆弱、窄域专用 (brittle & narrowly specialized)**，鲁棒性差 |
| **把压缩当 agent 动作、用 RL 训** | MEM1 (Zhou 2025)、Yu 2025、Ye 2025、ReSum (Wu 2026) | **把压缩和推理耦合进一个模型、要更新权重**，因此**对闭源/大规模 API 模型不可行**（拿不到梯度）|

**读出什么（问题的真正靶心）**：现有方法要么"压得不对"（丢了 agent 要的状态），要么"改不动"（RL 要权重，API 模型给不了）。ACON 的动机就精准卡在这个缝里——它要一个**既能针对 agent 状态定制、又完全不碰模型权重**的压缩优化框架。这也解释了为什么它偏要走"自然语言指南优化"这条看似绕的路（下节详述）。

**一张图看清"压了值不值"（Figure 1，摘要首图）**：在 AppWorld 上，ACON 相对 naive prompting baseline——**gpt-4.1（大模型）省 26% peak token、同时 acc 不降反升 +30%**；**Qwen3-14B（蒸馏后的小模型）省 31% peak token、acc +26%**。这两个"既省又涨"的点，就是全篇要证明的核心断言的浓缩：**好的压缩不是"用少一点分换省一点钱"的取舍，而是在长程 agent 里同时改善效率与质量**（因为省掉的多是害人的 distraction）。

### §2.5　相关工作定位：它站在谁肩上、和谁不同（一张对比表）

> 主讲提示：这页把 ACON 放进"上下文压缩"与"提示优化"两条谱系里，说清它的独特坐标。

ACON 同时踩在两条技术谱系上：**(1) 上下文压缩**（它要解决的问题）与 **(2) LLM 提示优化 / 文本梯度**（它用的工具）。

| 谱系 | 代表工作 | ACON 的关系 |
|---|---|---|
| 上下文压缩·文档抽取 | LLMLingua/LongLLMLingua (Jiang 2023/24)、RECOMP (Xu 2024) | ACON 的 baseline；它们**单步、无状态**，被 ACON 在多步 agent 上超越 |
| 上下文压缩·对话记忆 | MemGPT (Packer 2023)、Wang 2025a | 面向对话连贯，ACON 面向 agent **状态保真** |
| 上下文压缩·KV cache | Zhang 2025 (LightThinker)、Willette 2025、Epicache (Kim 2025) | 低层 KV 级；ACON 是 **token/文本级**，可与之正交组合 |
| 上下文压缩·agent 专用(RL) | MEM1 (Zhou 2025)、Yu 2025、Ye 2025、ReSum (Wu 2026) | **最直接对手**；它们**RL 训模型**，ACON **无梯度改提示词**（Table 7 逐条对比）|
| 提示优化 / 文本梯度 | APO (Pryzant 2023)、OPRO (Yang 2024a)、**TextGrad (Yüksekgönül 2025, Nature)**、DSPy (Khattab 2024)、Choi 2026 | ACON **借用**其"LLM 当优化器、NL 反馈当梯度"的机制，**首次把它用到 agent 上下文压缩** |

> **读出什么（增量在哪）**：ACON 的新意不在发明新优化器，而在**跨界组合**——把成熟的"文本梯度提示优化"（TextGrad 谱系）**首次搬到"long-horizon agent 的 history+observation 压缩"这个新问题上**，并补上"蒸馏降本"这一工程闭环。它相对最近的 agent 压缩工作（MEM1 等 RL 路线）的**方法论增量**是"**无梯度、model-agnostic、agent/压缩器可解耦**"（§2 "In contrast" 段）。

## §3　核心 intention 与形式化：把"压缩"写成一个带成本的优化问题

> 主讲提示：这页把问题形式化。先给 POMDP 直觉，再给成本函数，最后给那个"最大化奖励-最小化成本"的核心目标 Eq 7。每个符号都在式子前定义好。

**方法总览（一图流，先看骨架再抠数学）**：ACON = 一个"离线优化 + 在线压缩"的两段式管线。离线阶段把压缩指南优化好、并蒸馏出小压缩器；在线阶段 agent 每步按阈值触发压缩、用压完的上下文继续 ReAct 循环。

```mermaid

      D0["训练任务 D_train"] --> R2["跑 无压缩 vs 有压缩<br/>两遍轨迹"]
      R2 --> DC["对比子集 D_cont<br/>(基线成功·压缩失败)"]
      DC --> UT["UT 步：对比反馈=NL梯度<br/>拉高奖励·别丢关键信息 (Eq8,9)"]
      UT --> CO["CO 步：只看成功轨迹<br/>压低token·别留冗余"]
      CO --> PS["最优指南 P*"]
      PS --> DIS["LoRA 蒸馏 (Eq10)<br/>大模型压缩器→小学生 φ_S"]
    end
    subgraph ON["在线：agent 每步压缩"]
      OBS["最新观察 o_t"] -->|"|o_t|>T_obs"| OC["观察压缩 o'_t (Eq4)"]

      OC --> ACT["agent M 用压缩上下文<br/>生成动作 a_t (Eq1)"]
      HC --> ACT
      ACT --> ENV["环境返回新观察"]
      ENV --> OBS
    end

    DIS -. .-> HC
```

**读出什么**：左上"离线"块产出两样东西——**最优指南 $\mathcal{P}^*$**（回答"怎么压"）和**蒸馏小压缩器 $\phi_S$**（回答"谁来压"）；右下"在线"块是任意 ReAct agent，只是在每步按阈值调一下这个小压缩器。整个设计**对 agent 本体零侵入**——这就是它能"插到任意 LLM agent 上"的结构原因。

### 3.1　任务的形式化：POMDP + 成本函数

**直觉**：agent 干活是"看一眼环境→想→动手→再看"的循环，环境只给部分可见信息，所以是**部分可观察马尔可夫决策过程 (POMDP)**。我们要在"把任务做成"和"别让上下文太贵"之间权衡。

**符号（先定义，后用式）**（§3.1）：
- $\mathcal{E}=\langle\mathcal{S},\mathcal{A},\mathcal{O},\mathcal{T},\mathcal{R}\rangle$：环境；$\mathcal{S}$ 状态空间、$\mathcal{A}$ 动作空间、$\mathcal{O}$ 观察空间；
- $\mathcal{T}(s,a)\to(s',o)$：**确定性**转移函数，执行动作 $a$ 返回下一状态 $s'$ 与观察 $o$；
- $\mathcal{R}$：奖励函数，只在终止状态 $s_T$（agent 发出 `finish_task` 这类特殊动作后到达）给出奖励；
- $\mathcal{M}(\cdot;\theta,\mathcal{P}_{\text{agent}})$：LLM agent，参数 $\theta$（预训练权重，**固定**），$\mathcal{P}_{\text{agent}}$ 是 agent 的系统提示（环境描述、工具、输出格式、few-shot，**固定**）；
- $\bm{h}_{t-1}=(o_0,a_0,o_1,a_1,\dots,o_{t-1},a_{t-1})$：到第 $t$ 步的**交互历史**；$o_t$：第 $t$ 步的**最新观察**。

agent 每步的动作生成（Eq 1）：

$$\mathcal{M}(o_t,\bm{h}_{t-1};\theta,\mathcal{P}_{\text{agent}})\mapsto a_t \tag{1}$$

**成本函数（Eq 2）**。直觉：我们要量化"每步喂进去这么多 token 有多贵"。Transformer 解码 $n$ 个输入 token 的成本约 $\mathcal{O}(n)$，所以定义每步成本 $\mathcal{C}(\bm{h}_{t-1},o_t)$，全任务总成本：

$$C(\bm{H})=\sum_{t=1}^{T}\mathcal{C}(\bm{h}_{t-1},o_t) \tag{2}$$

其中 $\bm{H}=\{\bm{h}_{t-1},o_t\}_{t=1}^{T}$ 是整条历史+观察序列。$C$ 通常正比于每步 action/observation 的 token 长度之和。**关键观察**：系统提示成本是**静态**的，但**交互历史的成本是无界的 (unbounded)**（§3.1 原文）——这就把用户逼到只有两条烂路：要么**提前终止任务**，要么**启发式截断到最大长度**。ACON 要问的核心问题就是：*怎么比这些启发式更聪明地压上下文？*

### 3.2　两个压缩算子：history compression 与 observation compression

**直觉**：不是每步都压——短的没必要压。所以设两个阈值，超了才压。压缩器本身也是个 LLM $f(\cdot;\phi,\mathcal{P})$，参数 $\phi$（固定）、带一段**压缩指南** $\mathcal{P}$（自然语言提示，这是要优化的东西）。

**历史压缩（Eq 3）**：只在历史长度超过阈值 $T_{\text{hist}}$ 时压：

$$\bm{h}'_t=f(\bm{h}_t,\phi,\mathcal{P}_{\text{hist}})\ \text{if}\ |\bm{h}_t|>T_{\text{hist}},\quad \bm{h}_t\ \text{otherwise} \tag{3}$$

压出来的 $\bm{h}'_t$ 替换掉 Eq 1 里的原始历史。

**观察压缩（Eq 4）**：只在最新观察超过阈值 $T_{\text{obs}}$ 时压：

$$o'_t=f(o_t,\bm{h}_{t-1},\phi,\mathcal{P}_{\text{obs}})\ \text{if}\ |o_t|>T_{\text{obs}},\quad o_t\ \text{otherwise} \tag{4}$$

> **Why（设计层）——为什么分成 history 和 observation 两路，而不是一把梭？**
> 朴素做法是"把所有上下文当一坨字符串压"。→ 会**抹掉两者不同的时效结构**：观察是"当下这一步产生的、可能极长的原始 API 输出"（如 Spotify 打印 100 个 API 的 JSON，见 D.4），压它是为了**降当前峰值**且**不影响 KV cache 复用**（观察压缩发生在 cache 之前）；历史是"跨多步累积的叙事"，压它是为了**控内存增长**但会**使已有 KV cache 失效、需重算**（§A "Latency and KV-cache dynamics"）。两者的成本收益结构不同，所以分开、各配阈值、各配指南。这一点在库内与 **AgentFold（历史折叠）/ ReSum（历史摘要）** 呼应，但 ACON 额外把**观察压缩**也纳入了统一框架（§2 自述这是它区别于同类的一点）。

**一个隐患（埋批判线）**：在两种压缩里，压缩器都是"凭它自己学到的先验知识"去选择保留什么，**没有任何保证**关键细节被留下（§3.2 原文 "no guarantee"）。因为 agent 上下文其实是一个**环境的世界模型 (world model)**——含因果关系（邮件留了草稿）、演化状态（账户余额）、前置条件（需登录）、决策线索（截止日期）。压缩若把这些"世界模型"的关键位丢了，任务就崩。**这正是为什么要优化压缩指南**——把"该保留什么"从模型的模糊先验，变成被反馈校准过的明确规则。

### 3.3　核心优化目标（Eq 7）：最大化奖励，最小化上下文成本

**直觉**：我们想要一段压缩指南，使 agent 用压完的上下文既能**把任务做成**（高奖励），又**省 token**（低成本）。

**符号**：压缩器参数 $\psi\triangleq(\phi,\mathcal{P})$（实际只优化 $\mathcal{P}$，$\phi$ 固定）；压完的上下文集合与其总成本（Eq 5、Eq 6）：

$$\bm{H}'(\psi)=\{\bm{h}'_t,o'_t\}_{t=1}^{T},\qquad C(\bm{H}'(\psi))=\sum_{t=1}^{T}\mathcal{C}(\bm{h}'_{t-1},o'_t) \tag{5,6}$$

agent 在压完的上下文上跑出轨迹 $\tau(\psi)$ 与终止状态 $s_T(\psi)$。**学习目标（Eq 7）**：

$$\max_{\psi}\ \underbrace{\mathbb{E}[\mathcal{R}(s_T(\psi))]}_{\text{maximize 任务奖励}}\ -\ \lambda\underbrace{\mathbb{E}[C(\bm{H}'(\psi))]}_{\text{minimize 上下文成本}},\quad\lambda\ge 0 \tag{7}$$

其中 $\lambda$ 是权衡系数，期望对任务分布取。

**读出什么**：这个式子把"压缩优化"讲成一个**受成本正则的奖励最大化**。但它**难以直接优化**（§3.3 "Challenges"）：① 奖励**稀疏**，只在轨迹末端出现；② 成本定义在**离散量**（token 数）上，**没法直接求梯度**。这两条"自然地指向 RL"——但 RL 又有三座大山：(a) 用 RL 更新 LLM 的 $\phi$ 算力上 prohibitive；(b) 环境 rollout 极贵（每次奖励都要 agent+压缩器多步执行）；(c) 策略梯度**方差高**（压缩质量只能通过"最终任务是否成功"间接评估）。**正是这三座山，逼出了下一节的无梯度做法。**

## §4　方法核心之一：无梯度的「压缩指南优化」（把 NL 提示当梯度）

> 主讲提示：这是本篇要"讲透"的第一块。核心比喻——**用一个 LLM 当优化器，把"成功轨迹 vs 失败轨迹"的差异写成自然语言反馈（"自然语言梯度"），拿去改压缩指南这段提示词**。全程不碰权重。要把 UT 步和 CO 步的分工讲清楚。

### 4.1　直觉与设计动机

**Why（设计层）——为什么优化"自然语言指南"而不是"微调压缩器权重"？**
- **朴素替代 A：微调压缩器 $\phi$。** → 就是 §3.3 的 RL 三座山：贵、方差高、且**对闭源 API 模型根本做不了**（拿不到梯度）。
- **朴素替代 B：手工写死压缩规则（启发式）。** → §2 已说，脆弱、窄域、鲁棒性差。
- **ACON 的选择：把压缩策略编码成自然语言提示 $\mathcal{P}$，用 LLM-as-optimizer 迭代它。** 依据是近年 prompt optimization 工作（Pryzant 2023 "gradient descent" for prompts；Yang 2024a LLM as optimizer；Yüksekgönül 2025 *Nature* 的 TextGrad；Khattab 2024 DSPy；Choi 2026）。它把"怎么压"变成一段**人类可读、可迁移、可蒸馏**的规则，且**model-agnostic**——这是它相对 A、B 的三重胜势（§3.3）。

**关键素材：轨迹提供的是"稠密信号"而非"稀疏奖励"。** ACON 在训练集 $\mathcal{D}_{\text{train}}$ 上，让 agent **同时**跑"无压缩"和"有压缩"两遍，得到基线上下文 $\bm{H}$ 和压缩上下文 $\bm{H}'$。它专挑一类任务：**agent 用 $\bm{H}$ 成功、但用 $\bm{H}'$ 失败**——这构成一个对比子集 $\mathcal{D}_{\text{cont}}$。这类"成功→失败"的对比，比"二值奖励"信息量大得多（§3.3 原文 "richer feedback than scalar rewards"）：它直接说明"**压缩恰恰弄丢了完成任务的关键信息**"。

### 4.2　两步交替：UT（效用最大化）与 CO（压缩最大化）

ACON 把 Eq 7 的两项拆成两步交替优化（这是"alternating optimization"的思路，§3.3）：

**第一步：Utility maximization（UT，效用最大化）——先把奖励顶上去。**
对 $\mathcal{D}_{\text{cont}}$ 里每个任务，拿"压缩前 $\bm{H}$"和"压缩后 $\bm{H}'$"去问一个 optimizer LLM，产出对比反馈（Eq 8）：

$$\text{Feedback}=\text{LLM}(\text{Feedback Instruction},\bm{H},\bm{H}') \tag{8}$$

这段反馈就是**自然语言梯度 (natural language gradient)**（§3.3，引 Yüksekgönül 2025）——它指出"压缩指南 $\mathcal{P}$ 该往哪改"。把多条任务的反馈**拼接**起来，做一次"批量优化步"（Eq 9）：

$$\mathcal{P}^{(1)}=\text{LLM}(\text{Update Inst.},\mathcal{P}^{(0)},\|_{i=1}^{n}\text{Feedback}_i) \tag{9}$$

其中 $\|$ 是各任务反馈的拼接（对应 textual gradient descent 的一次 batch step）。它还会**采样 $K$ 个候选指南 $\{\mathcal{P}^{(1)}_k\}$**，在 $\mathcal{D}_{\text{cont}}$ 上评估、挑最好的。这步主要顶起 Eq 7 的**第一项（任务奖励）**。

**第二步：Compression maximization（CO，压缩最大化）——再把 token 压下来。**
只优化奖励会忽略成本（Eq 7 第二项）。于是做第二轮迭代，**只在压缩后仍成功的任务上**，让 LLM 反馈"执行时到底用上了哪些信息"，把 $\mathcal{P}^{(1)}\to\mathcal{P}^{(2)}$，鼓励**更短但仍充分**的压缩。这步压 Eq 7 的**第二项（上下文成本）**。

> **读出什么（UT vs CO 的分工，这是理解全篇的钥匙）**：
> - **UT** 回答"**别丢关键信息**"（保命，拉高成功率）；
> - **CO** 回答"**别留冗余信息**"（省钱，压低 token）。
> 两者顺序是 **UT→CO**（先保命再省钱）。实验里两个变体：**ACON^UT**（只做 UT）和 **ACON^UTCO**（UT 后再 CO）。经验规律（§4.2）：**信息高保真任务（OfficeBench、QA）用 ACON^UT 更安全**（过度压缩会删掉定案所需的细节）；**冗余噪声多的环境（AppWorld，观察里全是 API dump 和 distractor）用 ACON^UTCO 更好**（激进裁剪能同时提效+聚焦）。这条"该用哪个变体"的经验，本身就是给我们自己 compaction 策略的直接指导。

**为什么"NL 反馈=梯度"这个类比成立（把抽象讲实）**：传统数值梯度下降是"损失对参数求偏导 → 沿反方向更新参数"。ACON 的对应关系是（§3.3，承接 TextGrad 的框架 Yüksekgönül 2025）：
- **参数** ↔ 压缩指南文本 $\mathcal{P}$（可读的自然语言，而非浮点向量）；
- **损失** ↔ "压缩后 agent 失败"这一事实（$\mathcal{D}_{\text{cont}}$ 里的 success→fail 对）；
- **梯度** ↔ optimizer LLM 产出的对比反馈文本（"$\bm{H}'$ 相比 $\bm{H}$ 丢了 access_token 这条状态，导致 401 循环"这种句子）；
- **更新步** ↔ Eq 9 用"Update 指令 + 拼接反馈"改写 $\mathcal{P}$；
- **batch** ↔ Eq 9 里的 $\|_{i=1}^n$（把 $n$ 个任务的反馈拼起来做一次批量更新，而非逐任务贪心）；
- **多候选 + 选优** ↔ 采样 $K$ 个 $\mathcal{P}^{(1)}_k$ 在 $\mathcal{D}_{\text{cont}}$ 上评估挑最好（对应"沿多个方向试探、取下降最多者"，也部分缓解了无收敛保证的问题）。
> **读出什么**：这套映射让"优化一段提示词"变得像"训练一个参数"一样有章法——但它是**离散、无梯度、靠 LLM 语义理解**的。代价是**没有形式收敛保证**（§A 承认），好处是**完全不需要模型内部权重**，因此能优化"闭源 API 模型 + 任意压缩指南"这种梯度方法碰不到的组合。这正是 §2 那句"model-agnostic"的技术底座。

### 4.3　Algorithm 1 全流程（含伪代码骨架）

论文 Algorithm 1（§23）把上面两步串成一个可复现流程。核心骨架（我按原文重述关键行）：

```text
输入: 训练索引 I; 固定 agent M(·;θ,P_agent); 压缩器 f(·;φ,P); 初始指南 P^(0);
      权衡 λ≥0; 轮数 R; 候选数 K
输出: 优化后的指南 P*

# 第0步：收集无压缩基线
for i in I:  跑 M(无压缩) 得 H_i 和成功标记 r_i^base
I+ ← {i | r_i^base = 1}          # 只在"基线能成功"的任务上做对比

for r = 0,2,...,2R-2:            # r 只取偶数
  # ===== Stage A：UT（reward-first，用 H vs H' 对比）=====
  for i in I: 跑 M(压缩 P^(r)) 得 H'_i(P^(r)), r_i, C(H'_i)
  D_cont ← {(H_i, H'_i) | i∈I+ 且 r_i=0}      # 基线成功但压缩失败
  for (H,H') in D_cont:
     Feedback ← LLM(FeedbackInstr, H, H')      # 对比反馈=自然语言梯度
     累加到 F_util
  {P^(r+1)_k}_{k=1..K} ← LLM(UpdateInstr, P^(r), ‖ F_util)   # 采样K个候选
  k*_util ← argmax_k SuccessRate({r_i(P^(r+1)_k)})           # 按奖励选
  P^(r+1) ← P^(r+1)_{k*_util}

  # ===== Stage B：CO（cost-minimizing，只用成功的 H'）=====
  for i in I: 用 P^(r+1) 得 H'_i, r_i, C(H'_i)
  D_succ ← {H'_i | r_i=1}                       # 只看成功轨迹
  for H' in D_succ:
     CompFeedback ← LLM(CompressInstr, H')      # 找 H' 里的冗余段
     累加到 F_comp
  {P^(r+2)_k} ← LLM(UpdateInstr, P^(r+1), ‖ F_comp)
  k*_comp ← argmax_k ( SuccessRate({r_i}) − λ·NormCost({C(H'_i)}) )  # 奖励-成本联合选
  P^(r+2) ← P^(r+2)_{k*_comp}
  if 早停条件满足: break
return P* = P^(r+2)
```

**实验里只跑一轮**（一个 UT + 一个 CO），就已取得主结果（§4.1 "single step per each step"；§C.2 还发现"多跑一轮 UT 反而掉分"——单轮足够，见 Table 14 的 ACON^UTCOUT）。

### 4.4　优化后的指南长什么样（这是最有"手感"的一页）

光讲机制抽象。论文 Appendix D 给了 AppWorld 上**优化前 vs 优化后**的真实指南（Prompt D.5→D.6→D.7），一看就懂 UT/CO 各干了啥：

- **优化前（Prompt D.5，naive）**：泛泛让压缩器"总结 REASONING、列 COMPLETED、保持 concise、别丢信息"——**没有结构、没有硬约束**。
- **UT 优化后（Prompt D.6）**：变成一个**强结构化的 `<HISTORY_SUMMARY>` 模板**，含 5 个固定小节：
  1. **REASONING**（关键进展/决策/前后影响）；
  2. **VARS**（一张 `name|value|purpose` 表，"记录下一 session 必须**重新声明**的每个运行时值：token、id、list、last page_index/page_limit"）；
  3. **TODO**（待办，细到可直接执行）；
  4. **COMPLETED**（已完成子任务）；
  5. **GUARDRAILS**（防重复错误的短提醒，如"memory reset 后先重建 VARS""paginate 到空页""调用前校验 API 参数""别重复登录"）。
- **CO 进一步压后（Prompt D.7）**：加了一堆**"压缩规则"**——"多 bullet 叙事压成 ≤2 句""重复步骤日志压成一句""长 token/credential 串截成 `<token>`""GUARDRAILS 缩到一条""VARS 只留仍被引用的变量""目标字数 well under 1500 字符"——**同时保留一条铁律**："Always keep evidence-driven items required next session (tokens, ids, emails, amounts, lists, paths…)"。

> **读出什么**：这三段指南的演化，把"无梯度优化"从抽象讲成了**看得见的规则增量**。UT 把"随便总结"变成"带 VARS 状态表 + GUARDRAILS 护栏的结构化摘要"（保命）；CO 再往里塞"压多长、怎么截、留哪些证据"的量化裁剪规则（省钱）。**这套 VARS/GUARDRAILS 模板完全可以直接抄进我们自己的 compaction 提示词**（见 Inspires-Us a）。

**观察压缩指南同理演化（Prompt D.8→D.9→D.10）**：naive 版（D.8）只说"生成 Reasoning + Refined Observation，保留最小充分信息"；UT 优化后（D.9）加了一串**"Always preserve / Never"清单**——"preserve every endpoint that may be called + 其完整参数列表和默认值（尤其 page_limit/page_index/auth token）""preserve 后续会引用的 response-schema 字段（play_count, release_date, ids…）""Never omit defaults that affect behaviour / Never hallucinate endpoints / Never replace machine-readable data with vague prose"；CO 再压时（D.10）加"Reasoning ≤40 词""description ≤4 词""压 JSON 为一行一对象""长均匀列表用 range/pattern 概括"。**这组"preserve endpoints/schema"规则，正是案例 D.4 里 ACON 能保住 `play_music` API、而 naive 压缩漏掉它的直接原因**——规则被反馈校准后，把"该留什么"从模型模糊先验变成了硬清单。

**优化的批量步是"跨任务聚合"而非"逐任务贪心"（易被忽略的细节）**：Eq 9 的 $\|_{i=1}^n$ 把 $n$ 个任务的反馈**拼接**后一次性更新，Prompt D.2 的更新指令里也明确写"Ground all changes in the PER-SAMPLE REDUCTION SIGNALS … Do not aggregate across samples; use the patterns and rules as-is"——即让 optimizer **看到每条 per-sample 信号但产出一份通用指南**。这避免了"改指南去迎合某一个失败任务、却牺牲其它任务"的过拟合，也是为什么"单轮 UT+CO 就够、多加一轮反而掉分"（§C.2 / Table 14）。

## §5　方法核心之二：把压缩能力「蒸馏进小压缩器」

> 主讲提示：这是要"讲透"的第二块，也是 Why 三连"设计层"的核心命题——**训专用压缩模型太贵，用 NL 指南无梯度优化便宜可迁移，但每步还要调大模型压缩仍然贵；于是把「大模型+最优指南」蒸馏到小模型，一次训练、永久提速降本**。

**Why（设计层）——为什么优化完还要蒸馏？**
UT/CO 优化出的最优指南 $\mathcal{P}^*$，用起来仍要**反复调用大 LLM 当压缩器**——每步压缩都花大模型的钱（§3.4 原文 "repeatedly invoking the large LLM … adds substantial overhead"）。三条朴素替代都不行：
- **A：干脆不压** → 回到上下文爆炸；
- **B：一直用大模型压** → 成本高（gpt-4.1 压一次 ~$0.045）；
- **C：从头训一个专用压缩模型** → 又回到"训练贵 + 要梯度"的老问题（也就是本篇一开始要绕开的坑）。
- **ACON 的选择**：把"大模型 $\phi_T$ 带着最优指南 $\mathcal{P}^*$"当**教师**，生成压缩输出去**监督一个小学生模型 $\phi_S$**——一次性蒸馏，之后小模型独立压缩，既快又便宜。

**蒸馏目标（Eq 10）**。直觉：让学生模仿"教师在最优指南下的压缩输出"。教师从输入 $\bm{x}$ 生成压缩 $\bm{y}$，学生用交叉熵学这个 $(\bm{x},\bm{y})$ 映射：

$$\min_{\phi_S}\ \mathbb{E}_{(\bm{x},\bm{y})\sim\mathcal{D}^+_{\text{train}}}\Big[-\sum_{n=1}^{L_y}\log f(\bm{y}_n\mid \bm{x},\bm{y}_{<n};\phi_S,\mathcal{P}^*)\Big] \tag{10}$$

符号：$(\bm{x},\bm{y})=(\bm{h}_t,\bm{h}'_t)$ 对应历史压缩（Eq 3），或 $((\bm{h}_{t-1},o_t),o'_t)$ 对应观察压缩（Eq 4）；$L_y$ 是目标输出长度；$\mathcal{D}^+_{\text{train}}$ 是**教师用压缩上下文仍成功**的那些任务（只学"好的压缩"）。

**两阶段总流程（Eq 11）**：先提示词优化、再蒸馏，最终得到一个**参数量远小于教师**（$|\phi_T|\gg|\phi_S|$）的压缩器：

$$f(\cdot;\phi_T,\mathcal{P})\xrightarrow{\ \text{prompt optim}\ }f(\cdot;\phi_T,\mathcal{P}^*)\xrightarrow{\ \text{distill}\ }f(\cdot;\phi_S,\mathcal{P}^*) \tag{11}$$

**读出什么**：训练完，**学生在推理时替换教师**，从而"把决策与压缩解耦"（§3.4）——大模型只负责当 agent（决策），小模型专职压缩。这就是标题"Optimizing Context Compression"的完整含义：**先无梯度优化"怎么压"（改指南），再有监督蒸馏"谁来压"（换小模型）**。

**蒸馏用的是 LoRA**（§B.3）：rank 16、$\alpha=32$、lr $10^{-4}$、3 epochs、batch 4、max seq len 10,000，linear warmup 5%、weight decay 0.01、AdamW，单张 A100 80GB，**没做任何超参调优**（作者明说留给未来）。学生模型：Qwen3-14B / Qwen3-8B / Phi-4。数据只用 **100 例/域**（§A 承认这可能是蒸馏 gap 的来源）。

**Why（结果层）——为什么"蒸小"还能保 >95% 教师、甚至 gpt-4.1-mini 不蒸就够用？**
关键在于**压缩是个"比推理更窄的子任务"**：optimizer 已经把"该保留什么、该压掉什么"写进了指南 $\mathcal{P}^*$，学生模型不需要自己"想清楚任务",只需**照着结构化模板做抽取式改写**（填 VARS 表、按规则截断）。这类"有明确 schema 的转写"远比"多步推理决策"容易学，所以小模型足够胜任（§4.3 原文 "small models are sufficient for compression, enabling the expensive LLM to be only reserved for the agent"）。这也解释了 §C.2 里一个反直觉发现：**gpt-4.1-mini 作为压缩器（不蒸馏）在 OfficeBench/QA 上就已够用**——因为最优指南把难度从"模型"转移到了"提示词"。

**蒸馏的性能账（Figure 4 history / Figure 7 observation，gpt-4.1 当 agent）**（灰虚线=gpt-4.1 教师）：

| 学生压缩器 | AppWorld(history) | OfficeBench(history) | QA(history) |
|---|---:|---:|---:|
| Qwen3-14B | 50.00 | 75.79（**超教师 70.53**）| 0.371 |
| Qwen3-8B | 47.00 | 73.68 | 0.386 |
| Phi-4 | 44.60 | — | 0.370 |
| gpt-4.1-mini（不蒸）| 42.30 | 77.89 | 0.379 |
| *（教师 gpt-4.1 参考）* | *~Teacher 线* | *70.53* | *0.386* |

观察压缩（Figure 7）同样"蒸馏后基本保住"：Qwen3-14B 在 AppWorld obs 上达 56.50、OfficeBench 71.58。**读出什么**：小压缩器不但没垮，某些环境（OfficeBench）甚至**超过教师**——因为教师 gpt-4.1 的通用性在"照指南压"这个窄任务上并非优势，小模型专注反而稳。

**蒸馏后能玩"大 agent + 小压缩器"这种异构组合（Table 13）**：Table 13 直接给了 **gpt-4.1 当 agent + Qwen3-14B 蒸馏压缩器**的组合，AppWorld history 达 **56.5**（对比 gpt-4.1-mini 压缩器 47.6、"先 naive 微调再优化"的 fine-tune 基线 50.0）——**说明"优化指南再蒸馏"比"直接拿 naive 输出微调小模型"更强**（Table 13 的 "Fine-tune" 行就是后者，全面落后 ACON 行）。这正是 §8 Table 7 里"agent 与 compressor 可用不同尺寸模型"这一卖点的实证。

| $f(\cdot;\phi,\mathcal{P})$ | 压缩器 LLM，权重 $\phi$、压缩指南 $\mathcal{P}$ | §3.2 |
| $\bm{h}_{t-1}$ / $o_t$ | 交互历史 / 最新观察 | §3.1 |
| $\bm{h}'_t$ / $o'_t$ | 压缩后的历史 / 观察 | Eq 3,4 |
| $T_{\text{hist}}$ / $T_{\text{obs}}$ | 触发历史/观察压缩的长度阈值 | Eq 3,4 |
| $\mathcal{P}_{\text{hist}}$ / $\mathcal{P}_{\text{obs}}$ | 历史/观察的压缩指南（被优化对象） | §3.2 |
| **UT**（utility max） | 效用最大化步：拉高任务奖励、别丢关键信息 | §3.3 |
| **CO**（compression max） | 压缩最大化步：压低 token、别留冗余 | §3.3 |
| **contrastive task feedback** | 用"成功 vs 失败轨迹"差异生成的自然语言反馈=NL 梯度 | §3.3, Eq 8 |
| $\mathcal{D}_{\text{cont}}$ | 对比子集（基线成功、压缩失败的任务） | §3.3 |
| $C(\bm{H})$ | 上下文总成本（≈token 长度和） | Eq 2 |

## §7　实验设置：数据集、指标定义式、baseline、超参、成本

> 主讲提示：这页是 v1 硬规范的重头。三个 benchmark 都要 ≥10 步（长程），指标要给**定义式**。

### 7.1　三个长程 agent benchmark（均 10+ 步）

| Benchmark | 是什么 | 规模/划分 | 性能指标 |
|---|---|---|---|
| **AppWorld**（Trivedi 2024，主 benchmark）| 9 个模拟 app（Spotify/Venmo/SimpleNote…）经 457 个 API、~100 模拟用户；每任务平均 **42.5 次 API 调用** | 90 训练任务（用于指南优化+蒸馏）/ 168 test-normal | task completion score |
| **OfficeBench**（Wang 2024b）| Word/Excel/PDF/Email/Shell/Calculator 办公自动化，跨多 app | 1:1 随机切 → 92 训 / 95 测（去 OCR、去歧义、testbed 不跨集共享）| accuracy（1/2/3-app 分档）|
| **8-objective QA**（Zhou 2025）| 深研究式：**一个任务里 8 个不同问题**，末尾要一次性全答（BM25 检索 2018 Wikipedia）| 100 训 / 100 测 | EM 与 F1 的平均 |

**统一用 ReAct agent**（Yao 2023），工具格式随各 benchmark 约定（§4.1）。

### 7.2　效率指标的定义式（这是 benchmark 类论文的重点）

除任务分外，报三个效率指标（§B.2，沿用 LightThinker/MEM1）：

- **Steps**：每任务平均交互步数。
- **Peak Tokens**：整条轨迹里**任意单序列**观察到的**最大** token 数（不含系统提示）——推理内存的代理，对应 Figure 2 的峰值。
- **Dependency（Eq 12）**。直觉：不仅看峰值，还看"整条轨迹累计对历史 token 依赖了多少"（Figure 2 曲线下面积）。设第 $t$ 步输入 token $n_i^{(t)}$、输出 token $n_o^{(t)}$：

$$\sum_{t\in[T]}\frac{(n_i^{(t)}+2n_o^{(t)})\times n_o^{(t)}}{2} \tag{12}$$

它近似"整条轨迹里动作生成累积的计算量"。**读出什么**：Peak 管"最贵那一步的内存"，Dependency 管"整程的累计算力"——两个一起看，才不会被"峰值低但每步都长"骗过。

### 7.3　baseline、优化器、超参、成本

- **Baselines**（§4.1）：① No Compression（全量上下文）；② **FIFO**（只留最近 $k$ 轮，丢旧的，Yang 2024b）；③ **Retrieval**（用 `text-embedding-3-large` 检索最相似的 $k$ 轮）；④ **LLMLingua**（编码器抽取式压缩，keep rate 30%，Jiang 2023）；⑤ **Prompting**（naive 通用压缩指令，Smith 2025 / Lee 2025，人工做过 prompt engineering）。
- **主 agent/压缩器**：gpt-4.1（也测 gpt-4.1-mini、gpt-5-chat）。评测用 API snapshot `gpt-4.1-2025-04-14`，temperature 0、seed 42（仍有非确定性，§B.3）。
- **压缩阈值**（§B.3）：$T_{\text{hist}}=4096$（AppWorld/OfficeBench）、2048（QA）；$T_{\text{obs}}=1024$（AppWorld）、512（OfficeBench）、400（QA）。
- **Prompt optimizer**：OpenAI **o3**（做分析+更新），update 阶段采样 5 个候选选最优（§B.3）。
- **成本（很关键的卖点）**：指南优化步 **< $2/benchmark**（§4.5）；而训练集轨迹 rollout（数据收集）才是大头，AppWorld ~$20（一次性、可摊销）——但仍**远低于 GRPO 这类 RL**（§4.5，引 Shao 2024）。

## §8　主要结果：压 26–54%、不掉分，小模型大涨

> 主讲提示：这是全场停留最久的两页。先报"压多少+掉不掉分"，再报"小模型涨多少"，最后报"蒸馏保多少+省多少钱"。每个数字都要给机制解读，别只贴数。

### 8.1　总体效率-性能（Table 1 AppWorld / Table 2 OfficeBench+QA）

**AppWorld（Table 1，gpt-4.1 agent+压缩器，Average over 168）**节选（Acc↑ / Peak↓（×10³）/ Dep↓（×10⁶）；蓝底=ACON）：

| 方法 | Acc | Peak | Dep | 备注 |
|---|---:|---:|---:|---|
| No compression | 56.0 | 9.93 | 5.96 | 上界，但峰值高 |
| FIFO | 45.8 | 6.73 | 5.69 | 丢旧上下文→中/难任务崩 |
| Retrieval | 27.4 | 8.39 | 6.68 | 检索错关键步→最差 |
| LLMLingua | 39.3 | 7.50 | 6.37 | 抽取式，掉分多 |
| Prompting | 43.5 | 6.93 | 5.29 | naive 指令 |
| **ACON^UT** | **51.2** | 7.17 | 4.49 | 只保命 |
| **ACON^UTCO** | **56.5** | 7.33 | 4.69 | **反超无压缩上界！** |

**Why（结果层）——为什么 ACON^UTCO 的 56.5 能反超"无压缩"的 56.0？**
不是"压缩魔法般提升了能力"，而是**无压缩的长上下文本身在害 agent**（distraction，§1）。ACON 把 42.5 次 API 调用里的冗余 dump 和过时状态压掉、把关键依赖（VARS 表里的 token/id/page_index）显式留住，于是"信息更聚焦"抵消并超过了"少喂了一些原文"。对比之下，FIFO/Retrieval/LLMLingua 全**掉到 45.8 以下**——它们丢的恰是关键状态，中/难任务（跨更多步）崩得最狠（Table 1 的 Hard 列，No-comp 39.7 → FIFO 15.9 → Retrieval 7.9）。

**按难度拆开看（Table 1 的 Easy/Medium/Hard 分档，最有信息量的一张切片）**——压缩的价值随任务变长而放大：

| 难度档 | No-comp Acc | ACON^UTCO Acc | FIFO Acc | Retrieval Acc | 读出什么 |
|---|---:|---:|---:|---:|---|
| Easy(57) | 84.2 | **86.0** | 84.2 | 61.4 | 短任务差别小，ACON 微超 |
| Medium(48) | 47.9 | **56.2** | 39.6 | 12.5 | 中任务 ACON 明显领先 |
| Hard(63) | 39.7 | 30.2（UTCO）/ ACON^UT 更看 peak | 15.9 | 7.9 | 难任务里 baseline **崩盘**，ACON 仍稳住 |

**读出什么**：Easy 档大家都行（上下文没长到害人）；一进 Medium/Hard，**FIFO 和 Retrieval 断崖式下跌**（Retrieval 在 Hard 只剩 7.9），因为它们靠"丢旧/检索"必然弄丢跨越几十步才用到的关键状态；ACON 靠"结构化保状态 + 压冗余"维持住。这条曲线量化了一个直觉——**任务越长，"压得对"的边际价值越大**，也说明为什么 ACON 在长程 agent（而非短对话）上才是刚需。

**OfficeBench（Table 2a）**：ACON 把 peak 压 >25% 且维持 acc >74%（No-comp 76.84 → ACON^UT 74.74）。**8-objective QA（Table 2b）**：ACON^UTCO 在 EM/F1 上**超过 no-compression**，同时**peak 降 54.5%、dependency 降 61.5%**（No-comp peak 10.35→4.65、dep 3.32→1.50）——这就是摘要里"reduces peak token usage by 26–54%"的来源。观察压缩上，ACON 也**稳定压过所有 baseline**（Table 1/2 的 Observation Compression 块）。

### 8.2　小模型作为 agent：ACON 是"均衡器"（Figure 5，§4.4）

无压缩时，小模型（Qwen3-14B）在中/难任务上常因 distraction 挂掉。加 ACON 后（Figure 5）：
- **AppWorld**：Qwen3-14B **25.6% → 33.9%（+32.4% 相对）**；
- **8-objective QA**：**0.158 → 0.23 EM（+45.6%）**；
- 摘要还提到 OfficeBench +20%、QA 全场 +46%。

**读出什么（Θ2 实锤）**：这就是 `Agent = Model + Harness` 的直接证据——**模型没换（还是 Qwen3-14B），只换了"上下文这层 harness 组件"，能力就跳三到四成**。作者原话："ACON acts as an equalizer, enabling smaller agents … to approach the performance of much larger models"（§4.4）。小模型"更吃"上下文压缩，因为它们**对 distraction 更脆弱**——这与库内 Harness-Bench §9"强模型更不挑 harness"是同一枚硬币的两面。

### 8.3　蒸馏结果：保 >95% 教师、省 99% 成本（Figure 4 / Table 4）

- **性能保持**：把 UT 后的最优指南蒸进 Qwen3-14B/8B/Phi-4，**保留 >95% 的 gpt-4.1 教师性能**（Figure 4 灰虚线为教师）。更妙：**gpt-4.1-mini 不蒸馏、直接当压缩器**在 OfficeBench/QA 上就已够用（§4.3）——说明"小模型足以胜任压缩，大模型可专供 agent"。
- **成本骤降（§B.3 / §4.5）**：以 gpt-4.1 压缩为基线 $0.045/例；换 gpt-4.1-mini → $0.014（**降 69.2%**）；换**蒸馏后 Qwen3-14B → $0.0004（降 99.1%）**。端到端（Table 4）：AppWorld 每任务 API 成本 No-comp $0.331 → ACON(history) $0.285 → ACON(obs) $0.272；代价是**延迟增加**（多一次压缩器调用：73.24s → 87.68s / 101.92s，本地 A100 实测）。

> **读出什么**：这张账把"无梯度优化 + 蒸馏"的经济性讲圆了——**一次性花 ~$20 收集轨迹 + <$2 优化指南 + 一次 LoRA 蒸馏，之后每步压缩几乎免费（$0.0004）**。这正是它敢说"便宜可迁移可提速"的底气。

### 8.4　与 RL 类方法（MEM1）的正面对比（Table 7）：无梯度路线的护城河

> 主讲提示：这张表是"为什么不用 RL"的最强论据，也是回答组会追问"你和 MEM1 有啥不同"的标准答案。

MEM1（Zhou 2025）及并发工作（Sun 2025、Ye 2025）把压缩当 agent 动作、用 RL **和 agent 联合训练**——单一可训模型、需全权重访问+梯度。ACON 逐维度对比（Table 7）：

| 维度 | ACON（本文）| MEM1 类（RL）|
|---|---|---|
| 需要训 agent 模型吗？ | ✓ **不需要**任何 agent 训练/权重更新 | ✗ 要对 agent 模型做 RL |
| 无模型权限能做吗？ | ✓ **开源+闭源 API 都行** | ✗ 需全权重+梯度 |
| agent 与压缩器可不同模型吗？ | ✓ **支持解耦**（如 gpt-4.1 agent + Qwen3-14B 压缩器）| ✗ 推理与压缩耦合进同一模型 |
| 能"大 agent + 小压缩器"吗？ | ✓ 支持 | ✗ 同一模型既当 agent 又当压缩器 |
| 避开 GPU RL 成本吗？ | ✓ **<$2 指南优化、无需 GPU** | ✗ RL 要多轨迹 rollout + GPU |

> **读出什么（Θ2 + 判断力）**：这不是"谁分高"的比较，而是**"可部署性"的比较**。RL 路线在能拿到权重的开源模型上或许更优，但**对绝大多数只能调 API 的团队（包括我们用 Claude/GPT 的场景）根本不可行**。ACON 的核心价值主张就是——**用"改提示词"换掉"改权重"，把上下文压缩优化的门槛从"要 GPU 集群+梯度"降到"调几次 API、花 <$2"**。这条护城河比它的绝对分数更重要，也是它值得进 D 组精读库的根本理由。

### 8.5　跨模型/跨环境泛化：不止 GPT、不止结构化工具（Table 6/12）

- **换更强模型仍成立**：gpt-5-chat 当 agent+压缩器（Table 12），趋势与 gpt-4.1 一致——ACON^UT/UTCO 在保分的同时降 peak/dependency，说明"泛化到更强闭源模型"（§C.1）。gpt-4.1-mini（Table 9/10/11）也复现同样趋势，且**对小模型 agent 的增益更明显**（呼应 §4.4）。
- **换 web-agent 环境仍成立（WebVoyager，Table 6）**：额外在 **WebVoyager**（He 2024，观察是网页 accessibility tree、极其冗长）上测——50 训练任务优化指南、70 held-out 测。结果：history 压缩把 acc 从 prompting 的 42.9 提到 **48.6**（peak 相当、dependency 更低）；observation 压缩把 acc 从 45.7 提到 **47.1**、dependency 从 0.85 降到 0.80。**读出什么**：ACON 学到的压缩指南**不只在结构化工具环境有效，在"观察啰嗦且动态变化"的 web-agent 场景也有效**（§C.2）——这补上了"是否只对 AppWorld 这类结构化 API 有效"的泛化质疑。

## §9　消融与分析：阈值、优化器、变体、失败→成功案例

> 主讲提示：这页挑四个最有信息量的消融讲，别贴满。

**① 压缩阈值：中庸最好（Figure 6，§4.5）**。阈值太小→压得勤、token 省但掉分；太大→保分但省得少。**history 4096 / observation 1024 是甜点**——接近无压缩的分，同时显著降 peak。这条直接给我们"compaction 触发阈值该设多少"提供了先例。

**② 优化器选择：o3 + 对比反馈最好（Table 3）**。默认 **o3 + contrastive feedback = 51.2 acc**；去掉对比反馈（只用失败轨迹）→ 50.6（-0.6）；换 gpt-4.1 当优化器 → 47.6（-3.6）；换 gpt-5 → 50.6。**读出什么**：(a) **对比反馈**（成功 vs 失败）确实比"只看失败"更有用，印证 §4.1 的设计；(b) 优化器可降级到 gpt-4.1 仍出"足够好"的指南——**优化器不必是最强推理模型**，利于降本。

**③ UT vs CO 的取舍（§4.2 / Table 14）**：AppWorld 这类"噪声多"的环境，UTCO（激进压）常更优；OfficeBench/QA 这类"要精确留事实"的，UT（保守压）更安全。**且"UT 后再多加一轮 UT"（UTCOUT）反而掉分**（Table 14：47.0 vs UTCO 44.6 …细看各档），说明**单轮优化已足够**，多轮过拟合反噬。

**④ 案例研究：压缩把"失败"变"成功"（§C.2，Example D.2→D.3，全篇最生动）**。任务："删掉 ~/downloads 下所有 .pdf"。
- **无压缩（D.2）**：gpt-4.1-mini **反复无效尝试**——它调 `file_system` API 却不知道要"每次调用都显式传 access_token"，于是陷入 401 循环（试 set_access_token→不存在、试 logout→login→仍 401、猜 API…），最后放弃、`complete_task(status="fail")`。**病根**：长而乱的历史里，"login 返回了 token 但没机制自动复用"这条**关键依赖被淹没**了。
- **压缩后（D.3）**：ACON 的 `<HISTORY_SUMMARY>` 把历史提炼成结构化状态——**VARS 表**里明确记着 `file_system_token = "eyJ...sBA8nTOB..."`；**GUARDRAILS** 写着"Access tokens must be explicitly handled or passed""Validate authorization before calling protected APIs to avoid 401"；**TODO** 直指"determine how to pass the access token"。于是压缩后的 agent 直接把 token 作为参数传进 `show_directory`/`delete_file`，一次成功删完所有 .pdf。

> **读出什么**：这个案例把 ACON 的"世界模型保真"讲活了——**压缩不是缩短历史，而是把"决定成败的隐藏依赖"（token 要手动传）从噪声里拎出来、写成显式护栏**。作者原话："compression does not merely shorten history but clarifies critical dependencies, turning a failure trajectory into a successful one"（§C.2）。**观察压缩**也有同类效果（Example D.4）：Spotify 打印 100 个 API 的原始 JSON，naive prompting 压完**丢了结构、还漏掉关键的 `play_music` API**（任务恰恰要播放），而 ACON 优化后的观察压缩**既保 JSON 结构又保住 `play_music`**——因为它的指南硬性要求"preserve every endpoint that may be called + response-schema fields referenced later"。

## §10　局限与批判（论文 §A + 我的补充）

> 主讲提示：这页是判断力高地。既讲作者诚实承认的，也补库内视角的质疑。

**论文自陈的局限（§A，诚实）**：
- **评测范围偏 GPU/预算限制**：主要在 **GPT 系模型**上验证；虽设计为 model-agnostic，但对 **Gemini/Claude** 的泛化"尚未充分验证"；DeepSeek-R1、Qwen3-235B 这类巨模型因 GPU 受限没跑。
- **仍是受控 benchmark**：三个环境虽贴近真实，但"仍受控"；迁移到 in-the-wild 生产（任务随机、环境约束更不可测）是 open challenge。
- **无梯度优化没有收敛保证**：NL 空间优化靠"采样多候选选最优"部分缓解，但**缺形式化收敛理论**（§A "lacks a formal convergence guarantee"）。
- **蒸馏仍有性能 gap**：学生 vs 教师有残差 gap，怀疑是训练数据只有 100 例/域、多样性不足（§A "Distillation gap"）。
- **压缩本身引入延迟 + KV-cache 重算**：history 压缩会使已有 KV-cache 失效、需重算整段（§A "Latency and KV-cache dynamics"）——这是几乎所有上下文压缩框架的通病。

**我的补充批判（库内视角）**：
- **压缩器/优化器都是 LLM，"谁来保证压缩器不删错"未解**：§3.2 明说压缩"no guarantee"关键信息被留下——优化指南把这个概率降低了，但**没有硬保证**。案例 D.4 里 naive 压缩漏掉 `play_music` 就是活例；ACON 靠"指南里写死 preserve every endpoint"来堵，但这仍是"提示词层的软约束"，不是可验证的硬闸门。**这正好与库内 Harness-Bench 的"contract/格式失败占 36.4%"呼应**——压缩错就是一种 execution drift。
- **"成功→失败对比子集"依赖基线能成功**：Algorithm 1 只在 $I^+$（无压缩能成功）的任务上做 UT 对比。若某类任务**无压缩本来就失败**，ACON 学不到针对它的压缩规则——优化信号有盲区。
- **阈值/变体选择仍需人工经验**：UT vs UTCO 该用哪个、阈值设多少，目前是**按环境经验**定（§4.2），没有自动选择机制——留了个"自动挑变体"的口子（正好是我们的机会，见下）。
- **延迟与省钱的张力**：Table 4 显示省了 API 钱（$0.331→0.272）却**加了 ~40% wall-clock 延迟**（多一次压缩调用）。在延迟敏感的交互场景，这笔账不一定划算。

## ★ 对我们的启发（Inspires Us）

> 这一节是组会高潮，也是本库的第一人称优势：**我们（Claude Code / 本课 m9.* 的 agent）本身就有一套 compaction/上下文压缩机制**——ACON 压的 history+observation，正是我们循环里每一步喂给模型的东西。所以下面每条都"打到自己身上"。

## §11　版图定位（canon/前沿坐标 + 在本库的位置）

- **时间坐标（Θ4）**：**2026 前沿**，ICML 2026、Microsoft 出品。它相对基石的推进——**把"上下文压缩"从两条老路（脆弱手工启发式 / RL 训模型）切到第三条路：纯提示词层的无梯度优化 + 蒸馏**。这是"上下文压缩"方法论谱系里的一次范式扩充（§2 "In contrast"）。
- **E/T/C/L/O/V 归属（Θ1）**：本篇坐 **C（Context）层**——上下文/记忆管理。它是一个**可插到任意 ReAct agent 上的上下文压缩组件**，对 T/L 骨架无侵入、对 E 无耦合（model-agnostic）。它依赖 L 层提供"每步历史/观察"的钩子，依赖 V 层的"任务成功判定"来生成对比反馈。
- **回扣 Agent = Model + Harness（Θ2）**：本篇是"**同模型、只换上下文这层 harness 组件，能力就摆动**"的实证——Qwen3-14B 在 AppWorld 上因换上 ACON 而 +32.4%、QA +45.6%（§4.4）。这把"harness 决定能力"从"换整个 scaffold"细化到"仅换上下文压缩子模块"的粒度。
- **regime 诚实（Θ5）**：ACON 的增益**分 regime**——**小模型/长程/噪声多的任务，压缩增益大**（AppWorld 小模型 +32%）；**强模型/纯语言/信息高保真任务，增益小甚至要保守压**（OfficeBench/QA 宜用 ACON^UT，激进压会掉分）。所以"上下文压缩总是好"是**不成立**的；它是"在特定 regime 下"的强杠杆。这与库内"harness>model 分 regime"的判断一致，不绝对化。
- **在本库的位置**：**D 组 ⭐⭐**，与 AgentFold/ReSum（F 组·历史管理）、MemGPT（记忆）并列为"上下文/记忆"支柱之一，且是其中**唯一走"无梯度提示词优化 + 蒸馏"**的代表。读完它，再看任何一个 agent 系统，都能问一句："它的上下文压缩是**截断/检索/RL/还是可优化的 NL 指南**？能不能蒸馏降本？"

## §12　组会讨论问题（留给大家吵）

1. ACON^UTCO 在 AppWorld 上 **56.5 反超无压缩 56.0**——你信"压缩本身提分"多少、信"长上下文本来在害 agent"多少？怎么设计消融把这两者拆开？
2. §3.2 说压缩"no guarantee"关键信息被留住，ACON 靠"指南写死 preserve"来堵。这是**软约束**。要把它变成**可验证的硬闸门**（压完自动检查该留的 token/id 都在），最小可行方案是什么？会不会又把成本加回去？
3. Table 3 说优化器从 o3 降到 gpt-4.1 只掉 3.6 分。那"用弱优化器 + 多采样候选"能不能追平"强优化器 + 少候选"？这对我们省成本有何启示？
4. ACON 的对比子集只取"无压缩能成功"的任务（$I^+$）。对"无压缩本来就失败"的难任务，它学不到压缩规则——这个盲区怎么补？能不能用"部分进度"当弱奖励？
5. UT vs UTCO 现在靠人工按环境选。**做一个自动选择器**（在小验证集上按 acc-peak 前沿自动挑），你会怎么定"选择准则"？（这正是 Inspires-Us c 的第一步。）
6. ACON 省了 API 钱但加了 ~40% 延迟（Table 4）。在**延迟敏感**的实时 agent 里，这笔账怎么算才划算？观察压缩（不废 KV-cache）是否比历史压缩更适合实时场景？
7. 把 ACON 迁到我们 m9.* 的**开放研究任务**上，最大障碍是"没有硬 oracle 判成功"。没有可验证成功信号时，"对比反馈"这套还能怎么造？

## §13　一页速记 takeaways

- **命题（C 层）**：长程 agent 上下文无界增长 → 烧内存 + distraction 掉分；要**压得狠又不掉分**。
- **两把刷子**：① **无梯度优化压缩指南**——把"怎么压"写成 NL 指南 $\mathcal{P}$，用"成功 vs 失败轨迹"的对比反馈当"自然语言梯度"迭代（UT 保命 + CO 省钱，Eq 7–9，Algorithm 1），**不碰权重 → 能用在闭源 API 模型**；② **蒸馏到小压缩器**——把"大模型+最优指南"蒸进 Qwen3-14B/Phi-4（Eq 10/11），**保 >95% 教师、压缩成本 $0.045→$0.0004（-99%）**。
- **符号**：$\bm{h}_t$/$o_t$ 历史/观察 → $\bm{h}'_t$/$o'_t$ 压缩后；阈值 $T_{\text{hist}}=4096$、$T_{\text{obs}}=1024$；UT=utility max、CO=compression max。
- **指标**：Peak tokens（内存代理）、Dependency（Eq 12，累计算力）、task acc/EM/F1。
- **主结果**：peak 降 **26–54%** 不掉分（AppWorld ACON^UTCO **56.5 反超无压缩 56.0**；QA peak -54.5%、dep -61.5% 且 EM 反超）；baseline（FIFO/Retrieval/LLMLingua）全掉到 <46。
- **小模型是最大赢家（Θ2 实锤）**：Qwen3-14B 只换上下文组件 → AppWorld **+32.4%**、QA **+45.6%**——`Agent=Model+Harness`。
- **机制**：压缩不是截断，是**把决定成败的隐藏依赖（如 access_token 要手动传）从噪声里拎成显式护栏**（案例 D.2 失败→D.3 成功；D.4 观察压缩保住 `play_music`）。
- **诚实（Θ5）**：增益**分 regime**——小模型/长程/噪声多→压缩增益大；强模型/纯语言/高保真→宜保守（用 ACON^UT）。局限：无收敛保证、蒸馏有 gap、压缩加延迟+废 KV-cache、无硬保证不删错、只学"基线能成功"的任务。
- **对我们**：搬 **VARS+GUARDRAILS 五段模板**进我们 compaction 提示词；用 **UT→CO 对比反馈**无梯度优化我们自己的压缩提示词；做一个 **UT/UTCO+阈值的自动选择器**；下一步先跑"自由摘要 vs ACON 结构化摘要"的 A/B，量成功率与 peak。
