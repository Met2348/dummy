# FuncBenchGen：把多步工具调用建模成『隐藏函数依赖 DAG 遍历』的无污染评测生成器

> **本篇是 agent-harness 库 C 组（工具接口 / 评测）的前沿精读**。对齐标杆 [Harness-Bench](2605.27922-harness-bench-measuring-harness-effects.md) 的密度与诚实度：公式前先给直觉、符号先定义、指标给定义式、数字标 §/Table/Figure 出处、区分「论文宣称 vs 我的批判」、缺失写「原文未给出」、并强制打满 Θ1–Θ5 + Why 三连 + `## ★ 对我们的启发`。
>
> 与标杆的**分工**：Harness-Bench 是「固定任务、只换 harness，度量 harness 效应」——它把 harness 当**被测对象**；FuncBenchGen 反过来——它是一台**任务发生器**，固定住模型/harness，**自动生成难度可拧、且没被训练集见过的多步工具任务**，用来把「模型 + harness」在工具用这一层的能力曲线**逼出来**。两者互为对偶：一个测「脚手架」，一个造「压力源」。

## §1　TL;DR（一页讲清这篇在干嘛）

> 主讲提示：开场把两件事说死——(1) 它把「多步工具调用」这件模糊的事，重写成一个**有精确定义**的图论问题；(2) 因为图是**评测时现场生成**的，所以天然无污染、且难度能拧。别一上来堆数字，先把「DAG 遍历」这个建模讲清，后面所有结果都挂在它上面。

**一句话**：现有的工具用（tool-use / function-calling）基准有两个老毛病——① **难度不可控**（要几步、依赖多深、有没有干扰项，都是固定的、说不清的）；② **易被污染**（API 集是公开的、任务-答案对可能被训练集或测试期联网搜到，见 §2 引 Han et al. 2025）。FuncBenchGen 的解法是：**把多步函数调用建模成对一张隐藏的「函数依赖 DAG」的遍历**——给 agent 一堆函数 schema、一组已知输入变量、一个目标变量，但**不给依赖图本身**；agent 必须自己推断出「先调哪个、后调哪个」的正确序列，一路把中间变量算出来，最终算出目标变量的值（Abstract、§3）。因为这张图是**评测时才合成**的（随机函数名、随机三位数变量值），训练集不可能见过 → 无污染；因为图的**节点数 / 深度 / 干扰函数数量与连接方式**都是生成参数 → 难度精确可控（Figure 1 的四个旋钮 `d, n^core, n^conn, n^dis`）。

**这篇打 harness 的哪一层（Θ1）**：主坐 **T（Tools / 工具接口）** 层——它压测的正是「agent 在一堆不熟悉的函数接口里，能不能选对、串对、把参数值正确地在多步之间传递」；同时它本身是一台 **V（Validation / 评测）** 层的**生成器**——不测某个 harness 好坏，而是给「工具用能力」立一把**可拧刻度、且拧完无污染**的尺。对其它层的依赖：它需要一个 **O（Observability）** 层的执行器把每一步调用/返回/状态录成 trace（§4.6 的「in-house executor」），失败分析才做得下去。

**回扣全库论点（Θ2）**：`Agent = Model + Harness`。这篇的实验固定了 harness（一个统一的 in-house 执行协议）、只换**模型**与**任务难度**，因此它量的是 `Agent` 里 **Model** 那一半在「多步工具用」上的天花板——结论很硬：**再强的推理模型也扛不住深依赖**（GPT-5 在 20 步序列上只剩 15% 成功率，§1/Table 2）。但真正呼应本库命题的是它的**缓解实验**：一个**纯 harness 侧**的小改动（每步把已知变量值重述给 agent），**模型一个字没改**，GPT-5 就从 62.5% 跳到 81.3%（Abstract/§4.6.1/Figure 8）——这正是「换 harness、分数大摆」的又一个受控铁证，且它落在 **C（Context / 上下文）** 层。

**够新够权威（Θ4）**：**ICLR 2026 正式接收**（顶栏 "Published as a conference paper at ICLR 2026"），Megagon Labs 出品，代码开源（github.com/megagonlabs/FuncBenchGen）。相对本库已有的 BFCL、ToolLLM、API-Bank、LongFuncEval、ComplexFuncBench 这批「静态真实 API 集」基准，它在 Table 1 里把「**Contamination-Free**」与「**Controllability**（可控的 Required Func. Size / Dependency Depth / Irrelevant Func. Type）」**两列一起打勾**——而那五个前作在这两列上几乎全是叉。

**论文自陈的三条贡献（§1 末，先摆明再逐一验证）**：
1. **提出 FuncBenchGen**——一个为 TaLM（tool-augmented language models，工具增强语言模型）设计的评测框架，**自动生成无污染的函数调用任务，且难度按参数可控**（所需调用数、输入/输出变量的数量与类型、无关函数的数量与连接方式）。
2. **用它做了大规模实证**——7 个 SOTA 开/闭源模型；主要发现：推理优化模型稳压通用模型，但连 GPT-5 在长序列上都撑不住（20 步只剩 15%）；**"连到解题 DAG 上的、类型兼容的无关函数（CIN）"对所有模型都严重掉分**；任务成功强烈受**图结构**影响（浅而串行的链更好解）；多数模型语法正确却**在多步间传错/传旧参数值**。
3. **提出并验证一个极简缓解招**——每步重述先前变量值，**跨模型显著提升成功率**（GPT-5：62.5%→81.3%）。

> **一句话点明权威性来源（ΔC）**：**顶会（ICLR 2026）正式接收 + 工业界研究室（Megagon Labs）出品 + 开源可复现**——不是 arXiv 预印占坑，是经过同行评审的正式工作，且给了可跑的生成器代码，结论的可信度与可复用度都高。

## §2　问题与动机：为什么工具用基准需要「无污染 + 可控」

> 主讲提示：这页用 Why 三连的「问题层」。讲清现有工具用基准的两个系统病，并且要引原文的动机句，别空谈。

**Why（问题层）——不解决会卡住什么？**
LLM 装上外部工具（tool use）后成了很多真实应用的核心：能与外部环境交互、取到最新信息（§1 引 Luo et al. 2025）。围绕这件事有一大堆子方向——处理大规模 API 集、多步函数调用、长程工具用、工具技能的广义评测（§1 引 Kwak/Zhong/Song/Kate/Li/Patil/Sun 等）。**但进展被两个系统性问题卡住**（§1 原文 "impeded by two key issues"）：

1. **污染（contamination）**：前作确实curate了真实 API 集、构造了任务来测函数调用能力（API-Bank、ToolLLM 等），但这些基准「函数集多样性有限、curation 成本高，且**易被污染**」——训练集与之重叠、或**测试期联网搜索**就能搜到答案，因为「benchmark 的问答对可能是公开的」（§1 原文，引 Han et al. 2025 的 "search-time data contamination"）。一旦污染，你测到的是「记住了没」，不是「会不会推」。
2. **难度不可控（little fine-grained control）**：现有基准「对任务难度几乎没有细粒度控制」——比如**需要几个函数、函数依赖有多深、有没有类型兼容的无关变量在场**。有些基准（Patil/Qin/Kate 等）能控某些侧面（如所需函数数量），但**缺乏对『函数依赖深度』和『无关函数的存在/类型兼容性』的综合控制**（§1 原文）。后果：经验结论的普适性打折，也**没法回答「到底哪个因素最影响性能」**（§1 末 "prevent us from understanding which factors most significantly impact model performance"）。

**这两病合起来的后果**：我们缺一台能「**在评测时现场造题、且每个难度旋钮都能独立拧**」的发生器，去把工具用能力做**受控的因子分析**（isolate each factor，§3.2）。

**它站在哪几条研究线的交汇处（§2 Related Work 四条线）**：
- **多步推理**（§2 Multi-step Reasoning）：LLM 在数学、多文档、常识上都能多步推理（引 Davoodi/Maekawa/Yu 2025），FOLIO/Multi-LogiEval 还用一阶逻辑测其严谨性——但**"当模型必须与外部工具交互时，多步推理还稳不稳"仍不清楚**（§2 原文）。FuncBenchGen 正是把"多步推理"放到"必须调外部工具"的场景里压测。
- **代码生成与推理**（§2 Code Generation）：CRUXEval 等测"对代码执行的推理"；FuncBenchGen 的差异是——**把代码/函数当成黑盒外部工具来用**（"using code as black-box external tools"），而非推理其内部实现。
- **LLM 函数调用**（§2 Function Calling in LLMs）：ToolLLM/Gorilla/ToolACE 强调"接海量 API"，Tool-Planner/ToolDial/HammerBench 加入规划与多轮对话，LongFuncEval/ComplexFuncBench 测长上下文与多步——**但这些数据集"往往缺乏普适性与可控性"**（§2 原文），限制了系统分析。
- **污染与鲁棒性**（§2 Contamination and Robustness）：GSM-Symbolic、Illusion-of-Thinking 揭示了推理评测的污染问题，LiveBench 用"定期刷新"来限污染——**但它们没专门针对"多步函数调用"场景**。FuncBenchGen 的定位就是**填这个空**：给多步函数调用提供一个"有原则的、自动化的、支持受控压测与鲁棒分析"的框架（§2 末原文）。

> **读出什么**：这篇的动机不是「再造一个更强 agent」，而是「给工具用评测补一台**无污染的、可调难度的合成风洞**」。它站在"多步推理 × 函数调用 × 抗污染"三条线的交汇点上——把前两条的能力放进第三条的严格性里测。这与标杆 Harness-Bench「给领域补一把游标卡尺」是同一种方法论洞见——只不过 Harness-Bench 造的是「量 harness 的尺」，FuncBenchGen 造的是「压 model+harness 的风洞」。参见本库 [C 组工具/评测坐标](README.md)。

## §2.5　与前作的正面对比：为什么现有工具用基准补不上这两块（Table 1）

> 主讲提示：这页用一张对照表把「增量」讲死——它不是又一个 benchmark，而是把前作集体缺的两列一次补齐。这也兑现了 Θ4「诚实标定相对已有工作的增量」。

**Table 1（Comparison of previously proposed function-calling testbeds with FuncBenchGen）** 把 5 个代表性前作与本篇在**两个维度**上对齐——① 是否 **Contamination-Free**；② **Task Complexity 的可控性**又拆成三列：**Required Func. Size**（能否控所需函数数）、**Dependency Depth**（能否控依赖深度）、**Irrelevant Func. Type**（能否控无关函数的类型）：

| 基准 | Contamination-Free | 控·所需函数数 | 控·依赖深度 | 控·无关函数类型 |
|---|:---:|:---:|:---:|:---:|
| API-Bank (Li et al. 2023) | ✗ | ✗ | ✗ | ✗ |
| BFCLv4 (Patil et al. 2025) | ✗ | ✓ | ✗ | ✗ |
| ToolBench (Qin et al. 2024) | ✗ | ✓ | ✗ | ✗ |
| ComplexFuncBench (Zhong et al. 2025) | ✗ | ✗ | ✗ | ✗ |
| LongFuncEval (Kate et al. 2025) | ✗ | ✓ | ✗ | ✗ |
| **FuncBenchGen (ours)** | **✓** | **✓** | **✓** | **✓** |

**读出什么**：三点值得强调——
1. **「Contamination-Free」这一列，5 个前作全是 ✗**。因为它们都建立在**公开的真实 API 集**上（ToolLLM 号称覆盖 16000+ 真实 API），一旦公开就无法排除训练集重叠或测试期联网搜到答案（§2 引 Han et al. 2025 的 search-time contamination）。FuncBenchGen 靠「评测时现场合成」拿到唯一的 ✓。
2. **「依赖深度可控」这一列，也是 5 个前作全 ✗**。有些前作（BFCL/ToolBench/LongFuncEval）能控**所需函数数**（✓），但**没一个能独立地拧「依赖深度」**——而 §7 会证明，**深度恰恰是最狠的难度旋钮**。这正是本篇能做出「深度→性能」清晰曲线（Figure 5）的前提。
3. **「无关函数类型可控」这一列同样全 ✗**：前作要么没有系统的干扰项，要么把干扰项笼统处理；只有 FuncBenchGen 能把干扰按类型兼容性分成 CIN/DIN（§3.3）——这才让 §8「CIN 最毒」的结论成为可能。

> **Why（设计层）——为什么"补齐这两列"本身就是贡献？**
> 朴素反驳是「已经有一堆工具用基准了，再加一个有什么意义？」→ 但那些基准**都缺乏对难度因子的独立控制、且都可能被污染**，导致「我们无法知道到底哪个因素最影响性能」（§1 末原文）。FuncBenchGen 的贡献不是"又一个题库"，而是**一台能把每个难度因子单独拧、且拧完不会被记忆作弊的因子分析仪**——它让「工具用能力」第一次可以像做物理实验一样**控制变量**地研究（§3.2 "isolating each factor"）。

## §3　把「多步工具调用」建模成「隐藏 DAG 遍历」（本篇最该讲透的一节）

> 主讲提示：**这是全篇的地基，也是任务点名要我讲透的部分**。按「直觉 → 符号定义 → 读出什么」的顺序走。先让听众在脑子里立起一张图，再上符号，最后说清「节点/边/三个旋钮各自映射到多步工具用的什么」。

### 3.1　直觉：把一次多步工具任务想成「在暗图里找路」

**直觉**。想象你要算出某个目标数 `v_T` 的值，但你手上只有：一堆**函数说明书**（每个函数吃某些类型的输入、吐一个类型的输出，附一句自然语言描述）、几个**已知的输入数**（比如 `mfmjsy = 731`）。你**看不到**「哪个函数的输出能喂给哪个函数」这张连线图——这张图是**藏起来**的。你要做的，是像在一座暗着灯的管道网络里摸索：试着调一个函数、看它认不认你给的输入（认就吐正确中间值，不认就吐一个随机错值），再用吐出来的中间值去喂下一个函数……直到某条路径把你带到 `v_T`。**"多步工具调用" = "在这张隐藏的函数依赖图上，找出并执行一条从已知变量通往目标变量的正确调用序列"**（§3 原文："frame multi-step function calling as a graph traversal problem defined over a directed acyclic graph (DAG) of function dependencies"）。

### 3.2　符号定义（先定义，后用）

给公式前把每个符号钉死（§3.1 Task Definition）：

- **函数集** $\mathcal{F} = \{f_1, f_2, \dots, f_n\}$：agent 能调用的全部函数。
- 每个函数 $f_i$ 吃一组**输入变量** $\mathcal{V}_i^{\text{in}}$，产出**单个输出变量** $v_i^{\text{out}}$（§3.4 明确 "we set the number of output variables for each function to one"，即每个函数只吐一个输出，简化建模）。
- **已知输入变量集** $\mathcal{V}_{input} = \{v_1, v_2, \dots, v_k\}$：一开始就给了值的变量（如 prompt 里的 `Variable mfmjsy = 731`）。
- **目标变量** $v_T$：要求出它的值。
- **函数依赖图** $G = (\mathcal{F}, \mathcal{E})$：一张 **DAG**（有向无环图），节点是函数，一条有向边 $(f_i, f_j) \in \mathcal{E}$ 表示「$f_j$ **可以消费** $f_i$ 的输出」（即 $f_i$ 的输出类型/子类型能匹配 $f_j$ 的某个输入）。

**任务**：给定 $\mathcal{F}$、$\mathcal{V}_{input}$、$v_T$，agent 要**迭代地执行一串函数调用**来定出 $v_T$ 的值。**关键约束**：agent **只拿到函数列表和输入变量，拿不到依赖图 $G$ 本身**（§3 原文 "LLMs are given only the list of generated functions and input variables, rather than the dependency graph itself, requiring them to infer the correct call sequence"）。每一步 agent 可以调任意多个函数，系统按 agent 给的输入值返回函数输出；直到 agent 不再调用，就把它的最终输出解析成答案（§3.1 末）。

> **一句话锚定**：$G$ 是**隐藏的地图**，$\mathcal{F}$ 的 schema 是**散落的路牌**，agent 要靠「类型能不能对上」这个线索，把路牌拼回地图、走到 $v_T$。

**为什么一定是 DAG（有向无环图），不是任意图？** 三个约束缺一不可：**有向**——依赖有方向（A 的输出喂给 B，不能反过来），对应「先算 A 才能算 B」的时序；**无环**——若有环（A 依赖 B、B 又依赖 A），就会陷入「先算谁都算不出」的死锁，任务无解，所以生成时**强制无环**（附录 A.1「若加边会成环就丢弃重试」）；**图（而非链/树）**——因为一个函数可能吃**多个**输入变量（多个父节点），一个中间变量也可能被**多个**函数消费（多个子节点），所以是一般的 DAG 而非简单链或树。这三条一起，把「多步工具调用」的**结构本质**——「带前序约束的、可汇聚可分叉的、无死锁的动作偏序」——精确地刻画了出来。

**一个最小具体例子（照 Figure 1 的 `d=4, n^core=5, n^conn=3, n^dis=1`）**：图里有 **5 个核函数**（蓝色，解题必用）、**3 个 CIN**（绿色，类型能接上但无关）、**1 个 DIN**（黄色，孤立噪声），从若干蓝色输入方块出发，经 4 层依赖，最终汇到红色 Target 方块。agent 拿到的是**打乱顺序的 8 份函数 schema（5 核 + 3 CIN + 1 DIN）**（Figure 1 中间 "Shuffled Function Schemas / Graph structure hidden"）——它必须自己判断：这 8 个里哪 5 个是解题需要的、它们的先后是什么、哪 3 个绿的是诱饵、那 1 个黄的是噪声。这就是「难度可控」的直观来源：**把 `n^core` 从 5 拧到 20，正确链就从 5 步变 20 步；把 `d` 从 4 拧到 1，图就从长链塌成星形；把 `n^conn` 从 3 拧到 40，诱饵就从 3 个变 40 个。**

### 3.3　四个难度旋钮：节点 / 深度 / 两类干扰（旋钮如何映射到「多步工具用」）

图的形状由**四个生成参数**决定（Figure 1 的输入框写着 `d=4, n^core=5, n^conn=3, n^dis=1`；Figure 2 给了 5 核节点下 `d=1/3/4` 的三张对照图）：

| 旋钮 | 符号 | 定义（原文） | **映射到「多步工具用」的什么** |
|---|---|---|---|
| **核节点数** | $n^{\text{core}}$ | 解题**必须**用到的函数个数（number of functions required to solve the task） | **要串多少步工具**——序列长度。$n^{\text{core}}$ 越大，正确调用链越长。 |
| **依赖深度** | $d$ | 解题路径上的依赖层数（dependency depth） | **推理链有多深/多串行**——$d=1$ 是「星形」（所有核函数各自独立喂给终点，几乎可并行）；$d$ 大是「长链」（一步依赖上一步，强串行）。见 §4.4。 |
| **连接型无关节点** | $n^{\text{conn}}$（CIN） | 与解题**无关**、但**连到**了解题 DAG 上的干扰函数（connected irrelevant nodes）。「连到」= 它与解题 DAG 里至少一个函数之间存在依赖边（§3.2 定义） | **像不像的诱饵**——CIN 的变量类型与核节点**兼容**，看起来「能接上」，诱导 agent 走岔路。这是最毒的干扰（§4.3）。 |
| **断开型无关节点** | $n^{\text{dis}}$（DIN） | 与解题无关、且**不连**到解题 DAG 的干扰函数（disconnected irrelevant nodes） | **明显的噪声**——孤立函数（DIN 之间最多加 $\lfloor n^{\text{dis}}/2 \rfloor$ 条边模拟彼此依赖，§附录 A.1），理论上一眼可弃，但仍占据 schema 列表、消耗注意力。 |

> **Why（设计层）——为什么把 CIN 和 DIN 分开，而不是笼统叫「干扰项」？**
> 朴素做法是「随便塞几个无关函数当 distractor」。→ 这样你**分不清** agent 是被「看起来能接上的诱饵」骗了，还是被「一眼假的噪声」拖累了——两种失败的机制完全不同。FuncBenchGen 用「**是否与解题 DAG 有依赖边**」把干扰切成 **CIN（连接型，类型兼容、能接上）** 与 **DIN（断开型，孤立）**，于是能做一个漂亮的因子分析：结果发现 **CIN 对所有模型都严重掉分，DIN 的影响则因模型而异**（§4.3，见 §8）——这个区分正是「可控性」的价值兑现处。

### 3.4　图是怎么生成的（两步：结构 → schema）

生成分两步（§3.2 Benchmark Generation Process）：

**① 图结构生成（Graph Structure Generation）**，又分两阶段（§3.2 + Figure 10）：
- **核节点创建**：先造一条**长度为 $d$ 的节点链**，保证「从输入变量到目标变量存在一条含所需函数数的合法路径」；然后**迭代地**给已有核节点随机加父节点，把核节点补到 $n^{\text{core}}$ 个，同时**保持无环**、**不超过最大深度 $d$**（若新加会超深就丢弃重试）。为增加多样性，还会在核节点间**随机加至多 $2 n^{\text{core}}$ 条边**（保持无环）（附录 A.1）。
- **无关节点添加**：CIN——从已有核节点里随机选若干，把新的无关节点**挂为它们的孩子**；DIN——直接加孤立节点，不连任何核节点。

**② 函数 schema 创建（Function Schema Creation）**：把每个图节点转成一份函数 schema，含 (1) **随机函数名**（如 `func_yep`）、(2) **带随机类型+子类型标注的输入变量**、(3) **单个输出变量**、(4) **自然语言描述**（§3.2）。这个「on-the-fly（现场）」生成正是**无污染的来源**。

> **Why（设计层）——为什么靠「类型 + 子类型」连函数，而不是靠变量名？**（§3.2 + 附录 B.2，这是全篇最精巧的一处设计）
> 朴素做法是「函数 A 的输出变量名 = 函数 B 的输入变量名」就算连上。→ 作者实测发现这样「**问题变得太简单**」（B.2 原文 "the problem became too easy"）——agent 只要做字符串匹配就能拼出图，根本不用「推理」。而真实世界里，即便 B 消费 A 的输出，两者的**变量名几乎不会一样**（B.2 原文），真正把它们连起来的是**语义含义 + 数据类型**。所以 FuncBenchGen 改用**类型匹配**连接：A 输出 `type_beo/subtype_dej`，B 的某个输入恰好要 `type_beo/subtype_dej`，二者才连（附录 B.2 的 "Linking based on types" 例子）。**子类型（subtype）** 的作用是**消歧**：很多函数都吃 `Employee` 类型，但 `num_managed_employees` 只认子类型 `Manager` 的 `Employee`（B.2 原文举例）——子类型让「谁能接谁」变得精确而非一对多乱接。这一步把「字符串匹配」升级成了「**对不熟悉 API 的语义类型推理**」，正是它想压测的核心能力。

### 3.5　执行语义：函数如何「奖惩」agent

实现细节（§3.4 Implementation Details）——这决定了「走错路会怎样」：
- 每个变量被赋一个**三位随机整数**值。
- 函数实现**确定性逻辑**：**只有当 agent 喂进的输入值恰好是「正确的期望输入」时，才返回正确输出值**；否则**返回一个随机的错误输出**——以此模拟真实 API 的「参数不对 → 静默失败 / 返回无用响应」（§3.4 原文 "return random incorrect outputs when given invalid inputs, simulating realistic API behavior where invalid parameters lead to silent failures"）。
- agent 与执行环境的交互协议：每步可发**多个**函数调用，收到各自输出，直到不再调用为止（§3.4）。模型输入长这样（附录 B.3）：User Prompt 给出目标变量 + 已知变量值（`Variable mfmjsy = 731`），Tool Prompt 给出 OpenAI-style 的函数 schema（`strict: True`, `additionalProperties: False`）。

> **Why（设计层）——为什么函数要"喂对才吐真、喂错吐随机"，而不是"喂错就报错"？**
> 朴素做法是「参数不对就抛一个明确的 error」。→ 这样等于**免费给了 agent 一个"你错了"的信号**，agent 可以据此二分搜索、试错逼近——任务就退化成"带反馈的搜索"，不再考验"一次想对"的规划能力。FuncBenchGen 改用**"喂错默默吐一个随机错值"**（§3.4 明说这是模拟真实 API 的 "silent failures"）——**错误不被显式标记**，agent 只能凭"这个值看起来对不对/后续能不能接上"自己判断。这把任务从"有反馈的搜索"逼回"**无反馈的规划 + 自检**"，也更贴近真实：很多真实 API 参数错了不会报错，只会返回一个语义上无用的响应。

> **读出什么**：这套「喂对才吐真、喂错吐随机」的语义**极其关键**——它意味着**一步错、步步错**：agent 一旦在某步用了错的中间值，该步返回随机数，后续全被污染，且**没有任何显式报错提醒它"这里错了"**。这就把「多步状态追踪」变成了一道**脆弱且盲飞的接力**，也预告了 §9 的主失败模式：**用了「尚未算出来」的变量值**。

### 3.6　它真能对上真实世界吗？（Real-world Applicability, §3.3）

作者用一个**真实例子**论证 DAG 建模不是玩具（§3.3 + Figure 3，例子借自 ComplexFuncBench 的 Taxi Reservation）：用户让 agent「订一辆明天游览结束后从酒店送我的出租车」。agent 必须：1) 用 **Hotel Name** 查 `get_hotel_location` → 2) 用位置查 `find_tours` → 3) 用 **Time** + 位置 `filter_tours` → 4) `reserve_taxi` 得 `Reservation`。这天然是一张 DAG——「没有酒店位置就查不了周边游、没选定游览就订不了车」。FuncBenchGen 把这种真实复杂度**抽象成可控评测**，同时**保留了核心挑战**：模型必须**只凭函数签名**推断这些依赖，就像它在部署时面对陌生 API 那样（§3.3 原文）。

> **读出什么（这就是我们每天在做的事）**：这个抽象之所以对**我们（Claude Code）** 特别有意义——**我们每次面对一个陌生代码库/陌生 MCP 服务器，本质上就是在解一道 FuncBenchGen 题**：一堆不熟悉的工具/函数摆在面前、依赖关系没人明说、要靠"这个的输出能不能喂给那个"把调用链拼出来。FuncBenchGen 的贡献是把这件"我们天天做、却从没被干净量化过"的事，做成了**旋钮可拧、答案已知、无污染**的实验台。所以它不是一个"别人领域的 benchmark"，而是**一台可以直接拿来给我们自己称重的测力计**（详见 §11）。

> **建模的边界（诚实）**：这个 Taxi 例子里其实还有 FuncBenchGen **尚未建模**的东西——比如 `filter_tours` 可能返回**空结果**需要 agent 改日期重试（条件分支），或"订不到就换一家酒店"（回溯）。当前 DAG 是**纯前向数据流、无条件分支、无循环**（§5 承认这是未来方向）。所以它抽象出的是真实工具用里**"依赖顺序推断 + 值传递"** 这条主干，把 **"控制流"** 那部分留给了后续工作。

## §4　评测协议与指标：怎么判「对」、怎么量「效率」

> 主讲提示：这页把「一次评测怎么跑、怎么打分」讲清。指标给定义式。

**成功判据（§4.1 Evaluation Metrics）**：一条函数调用序列被判**正确**，当且仅当它**产出了任务期望的目标值**（"produces the expected output for the task"）。这是个**端到端的、确定性的**判据——不看过程漂不漂亮，只看 $v_T$ 的值对不对。据此定义：

- **成功率（Success Rate）**：在一批随机生成的图上，最终算对 $v_T$ 的比例。这是主指标。
- **平均调用次数（ACs, Average number of function Calls）**：模型实际发出的函数调用数，用来量**效率**。论文**分开报** ACs(Succ.)（成功轨迹的平均调用数）和 ACs(Fail.)（失败轨迹的），因为二者揭示不同行为（§4.2）。
- **预算上限**：把最大调用数**封顶在「最少所需调用数的 2 倍」**（§4.1 "we cap the maximum number of calls at twice the minimum required number"）——这是一个 **L（Loop）层**的预算约束，防止 agent 无限试错。

**执行器与失败归因（§4.6）**：论文用一个**自研执行器（in-house executor）**记录**所有调用、参数、返回、状态更新**，做到「确定性、免标注」的失败归因。执行器强制**四条顺序的、机器可检的谓词**（four sequential, machine-checked predicates）——这四条是「任何一次合法函数执行的层级前提」，按 1→4 顺序用 if-branch 检查，任何一步出错就归到对应类别（§4.6）：

1. **name resolution（名字解析）**：函数存在吗？
2. **schema conformance（schema 符合）**：参数个数/结构对吗？
3. **dataflow availability（数据流可用）**：用到的变量值**已经**通过前序调用或初始输入**被建立**了吗？
4. **value consistency（值一致）**：用的变量值**对**吗（等于前序确立的值）？

> **读出什么（Θ1，指向 O 层）**：这台执行器是本篇的 **O（Observability）** 层支柱——正因为它把每步录成结构化 trace 并用四谓词自动打标，§9 的失败分析（Table 3）才成为**确定性统计**而非人工判读。这也是「无污染合成任务」的一个隐性红利：**因为是我们造的图，我们精确知道每一步的正确答案**，所以归因可以完全自动化。

## §5　实验设置：7 个模型、四个难度维度、5 个随机图取平均

> 主讲提示：把「变量矩阵」讲清——变的是什么、固定的是什么。

**被测模型 × 7**（§4.1 Models + 附录 B.1 Table 4）：分**推理优化**与**通用**两类——
- 推理模型：**GPT-5**、**GPT-5-mini**、**Gemini-2.5-Pro**、**Gemini-2.5-Flash**、**Qwen3-235B-A22B**（简称 Qwen3）。
- 通用模型：**GPT-4.1**、**GPT-4.1-mini**。
- 细节（Table 4）：GPT-5/GPT-5-mini 上下文 400k；Gemini-2.5-Pro/Flash 上下文 1M；Qwen3 是 235B、128k、Apache-2.0（唯一开放权重、经 HuggingFace 访问）；GPT-4.1 系 1M。**推理模型的思考预算取各自默认**（GPT-5 取 medium）。**温度=0.0、top-p=1.0**（不支持温度的模型用默认，脚注 1）。

**难度维度矩阵**（§4.1 Setup）：
- **核节点数** $n^{\text{core}} \in \{5, 10, 20\}$。
- **无关节点数** $\in \{0, 10, 20, 40\}$。
- **连接类型 × 3**：**Connected**（全部无关节点连到核）/ **Disconnected**（全部不连）/ **Half & Half**（一半连一半不连）。
- **依赖深度**：对 5、10 核节点，$d$ 在 $1 \dots n^{\text{core}}-1$ 间取；对 20 核节点，每 10% 递增取一档（即 $d=1,3,\dots,17,19$）。
- **每种配置生成 5 张随机图取平均**（§4.1 "we generate 5 different graphs for each setting"），5 trials/config。

**五个研究问题**（§4）：**RQ1** 核节点数如何影响性能？**RQ2** 无关函数如何影响？**RQ3** 依赖深度如何影响？**RQ4** 更大函数集 + 思考预算对最强模型的影响？**RQ5** 常见失败类型是什么、怎么缓解？

> **Θ2 提示——固定了什么**：这里的 harness（执行协议、预算、判据、四谓词归因器）是**固定**的；变的是**模型**和**任务难度**。所以本节量的是 `Agent = Model + Harness` 里 **Model** 在工具用维度上的能力曲线。唯一动 harness 的是 §10 的缓解实验（改上下文呈现）。

## §6　主结果 RQ1：核节点一多，连 GPT-5 都崩

> 主讲提示：这是第一个「震撼数字」。先报成功率随序列长度的塌陷，再点「通用模型基本躺平」。

**Table 2（跨全部配置聚合：无关函数 {0,10,20} × 三种连接 × 各深度，5 trials）** 主结果节选：

| 模型 | 5 核 成功率 | 5 核 ACs(S/F) | 10 核 成功率 | 10 核 ACs(S/F) | 20 核 成功率 | 20 核 ACs(S/F) |
|---|---:|---:|---:|---:|---:|---:|
| **GPT-5** | **72.5%** | 5.4 / 7.6 | **38.2%** | 11.5 / 13.0 | **15.0%** | 22.0 / 23.7 |
| GPT-5-mini | 16.0% | 5.0 / 7.9 | 7.6% | 10.0 / 14.2 | 4.2% | 20.0 / 23.4 |
| Gemini-2.5-Pro | 46.5% | 5.5 / 6.3 | 14.4% | 10.6 / 13.2 | 6.0% | 22.6 / 24.4 |
| Gemini-2.5-Flash | 31.5% | 5.1 / 1.1 | 13.8% | 10.3 / 1.3 | 7.2% | 20.0 / 1.3 |
| Qwen3 | 11.0% | 5.4 / 6.1 | 11.2% | 11.2 / 11.1 | 3.8% | 24.1 / 19.3 |
| GPT-4.1 | 12.0% | 5.2 / 4.1 | 2.2% | 10.3 / 7.1 | 0.2% | 21.0 / 12.0 |
| GPT-4.1-mini | 11.0% | 5.2 / 5.0 | 0.0% | – / 9.0 | 0.0% | – / 18.9 |

**Why（结果层）——为什么会这样塌？**
1. **推理模型碾压通用模型，但也扛不住长序列**（§4.2 原文 "GPT-5 outperforms all other models by a significant margin, but it reaches only 15% on extended function-call sequences"）。GPT-5 在**每个**核节点规模都领先，但成功率随序列变长**急剧下滑**：72.5%（5 核）→ 38.2%（10 核）→ **15.0%（20 核）**；Gemini-2.5-Pro 46.5% → 6.0%。机制上，这反映**有限的有效规划深度**和**仅靠反思做的、很有限的自我纠错**（§4.2 "limited effective planning depth and only modest self-correction via reflection"）——序列越长，一步错就全错的概率越高（呼应 §3.5 的脆弱接力）。
2. **通用模型（GPT-4.1 系）在 10 核以上几乎归零**（GPT-4.1-mini 10/20 核都是 0.0%）——说明「不带强推理」的模型在多步工具用上基本不可用。
3. **即便成功，GPT-5 也多调了 ~10% 的冗余调用**（§4.2 "GPT-5 invokes 10% more calls than necessary even when successful"）：10 核和 20 核任务（所需调用数即 10、20）里，GPT-5 成功轨迹平均调了 **11.5** 和 **22.0** 次——最强模型也**没法把调用序列优化到最省**。
4. **失败轨迹的调用数模式因模型而异**（§4.2）：GPT-5、GPT-5-mini、Gemini-2.5-Pro **失败时比成功时调得更多**（找不到路就反复试）；而 **Gemini-2.5-Flash 失败时调得远少于成功时**（ACs(Fail.)≈1.1–1.3）——**说明它一旦找不到路就直接放弃**（§4.2 "it gave up when it could not find the correct path"）。

**从 ACs 读"行为画像"（把效率指标当性格测试）**：ACs(Succ.) 与 ACs(Fail.) 的**对比**，能读出模型面对困境时的两种"性格"——
- **反复试错型（thrash）**：GPT-5、GPT-5-mini、Gemini-2.5-Pro **失败时调用数 ≥ 成功时**（如 GPT-5 在 5 核：失败 7.6 > 成功 5.4）——找不到正确路径时，它们**在预算内反复尝试**，直到撞上 2× 上限（§4.2）。
- **提前放弃型（give up）**：Gemini-2.5-Flash **失败时调用数远少于成功时**（5 核：失败 1.1 ≪ 成功 5.1）——它一旦判断"找不到路"就**直接停手、不再挣扎**（§4.2 "it gave up when it could not find the correct path"）。

> **读出什么（对 harness 设计的启示）**：这两种性格对**预算/重试策略**的含义完全相反。对 thrash 型，harness 该**更早地打断无效试错**（否则烧光预算还是错）；对 give-up 型，harness 反而该**鼓励它多试一步 / 提供 recovery 提示**（否则它过早认输）。**同一套重试策略不可能对两类模型都最优**——这本身就是 `Agent = Model + Harness` 里"harness 要因 model 而调"的一个微观例证。

> **读出什么（Θ2）**：这张表是「**Model 那一半也有硬天花板**」的直接证据。Harness-Bench 证明「换 harness 摆 23.8 分」，FuncBenchGen 补上另一半——**在固定 harness 下，纯把任务加长，最强模型也会从 72.5% 崩到 15%**。两者合起来才是完整的 `Agent = Model + Harness`：能力既受脚手架约束，也受模型的规划/状态追踪深度约束。

## §7　RQ3：依赖深度是最狠的旋钮——星形好解，长链要命

> 主讲提示：这页讲「深度」这个旋钮为什么比「节点数」更本质。配合 Figure 2 的星形 vs 长链讲。

**结论：深度越浅越好解**（§4.4）。以 10 核节点为例（Figure 5，跨所有无关节点设置取平均）：
- **$d=1$（星形结构，见 Figure 2）时，GPT-5 成功率近 90%**——星形意味着所有核函数各自独立地喂向终点，几乎可以**并行**决策，每步之间没有状态依赖。
- **深度升到 4–8 时，GPT-5 骤降到 <30%**（§4.4 原文 "drops sharply to less than 30% when the dependency depth increases, i.e., between 4 and 8"）。这个模式在 Gemini-2.5-Pro、GPT-5-mini 上一致——**高依赖深度对 LLM 是普遍的硬骨头**。

**一个反直觉的补充：分支少反而好**（§4.4 "Fewer branches in the function call sequence are less harmful"）。GPT-5、Gemini-2.5-Pro、Qwen3 在 $d=8,9$（图呈**路径/链状**）时，反而比 $d=5\text{–}7$ 略有回升——因为**分支少 → 每步的决策空间小 → 决策更容易**。但这个好处**只在更大更强的模型上出现**，GPT-5-mini、Gemini-2.5-Flash 上看不到（§4.4）。附录 Figure 13/14 用 95% 置信区间和 20 核复现了同一趋势。

> **Why（结果层）——为什么深度比节点数更本质？**
> 节点数决定「要走几步」，但**深度决定「有多少步是强串行、不可并行」**。星形（$d=1$）虽然也可能有很多核节点，但每步独立，等于把一个难题拆成很多**互不依赖**的易题；长链（$d$ 大）则每一步的输入都**锁死**在上一步的正确输出上——这正是 §3.5「脆弱接力」最致命的形态。所以**深度是「状态追踪压力」的真正代理变量**，而非节点数。

> **读出什么**：如果你想用 FuncBenchGen 造一道「专门压状态追踪」的题，**拧 $d$**（造长链）比拧 $n^{\text{core}}$（加节点）更精准。这对 §11「怎么无污染地测我们自己 agent」是直接的旋钮选择指导。

## §8　RQ2：连接型干扰（CIN）对所有模型都致命

> 主讲提示：这页是「可控性」价值的兑现——因为把干扰分成了 CIN/DIN，才看得出「诱饵」和「噪声」的天壤之别。

**核心发现：CIN 严重掉分，DIN 影响因模型而异**（§4.3 + Figure 4）：
- **Connected（全 CIN）设置对几乎所有模型的负面影响最大**（§4.3 原文 "Connected irrelevant nodes (CINs) severely degrade performance for all models"）。作者的机制解释：**CIN 与核节点共享变量（类型兼容），使模型难以把「相关的函数路径」从「无关的」里分辨出来**（§4.3）。这与其它推理任务里「无关上下文干扰」的现象一致（§4.3 引 Shi et al. 2023 "LLMs can be easily distracted by irrelevant context"）。
- **DIN（断开型）的影响因模型而异**（§4.3）：有意思的是，**GPT-5 在 "Disconnected" 设置下反而比 "No Extra"（无任何干扰）更好**——作者推测 DIN 的存在可能**促使 GPT-5 更谨慎地考虑无关函数**，反而带来增益；而 **Gemini-2.5-Pro 在 Disconnected 下更差**，说明它**即便干扰是断开的也难以忽略**。
- **Half & Half 居中**（§4.3）：一半 CIN 的效果介于 No Extra 与 Connected 之间——说明**只要有一些 CIN 在场，就足以显著迷惑模型**。

> **Why（设计层）回收——为什么这个发现只有 FuncBenchGen 能干净地做出来？**
> 因为它在生成期就**用「是否连到解题 DAG」把干扰精确二分**（§3.3）。如果像旧基准那样把 distractor 混作一谈，你根本没法把「类型兼容的诱饵（CIN）严重掉分」和「孤立噪声（DIN）影响不一」这两件事分开——而这恰恰是最有工程价值的结论：**危险的不是「工具多」，而是「有看起来能接上、其实无关的工具」**。

> **读出什么（打到我们自己身上）**：这条直接映射到我们 agent 的**工具集设计**——真正拖垮多步工具用的，不是工具总数，而是**类型/描述上与目标任务"貌合"的干扰工具**。见 §11 Inspires-Us b。

## §8.5　RQ4：更大函数集 + 更小思考预算，双重压垮最强模型

> 主讲提示：这页专讲「就算是 GPT-5，把函数集加大、把思考预算调小，也会塌」。两个结论都对我们上 MCP、调预算有直接警示。

RQ4 只拿**最强的 GPT-5** 做深入压测，变两件事：**函数集规模**与**思考预算**（§4.5）。

**发现一：函数集越大，越难——即便干扰是"断开的"**（§4.5 + Figure 6，跨 {0,10,20,40} 无关节点取平均）。把核节点从 5 加到 **40**，GPT-5 的成功率随之显著下滑；**关键在于**——**即使在 Disconnected 设置下**（无关函数与核函数**类型不兼容**、理论上最容易忽略），**40 核节点时 GPT-5 也掉到 <10%**（§4.5 原文 "GPT-5's performance degrades significantly (lower than 10% for 40 core nodes)"）。作者由此下了一记冷水：「**这说明连最好的模型都还没准备好用在大函数集上——而这正是现有 MCP server 在发生的事**」（§4.5 引 Anthropic 2025 MCP）。

**发现二：思考预算不足会严重限制多步工具用**（§4.5 + Figure 7，GPT-5、5 核、跨所有无关节点设置取平均）。对比 **medium thinking budget**（其它实验的默认）与 **minimal thinking budget**：从 medium 降到 minimal，成功率**大幅下滑**（§4.5 "a significant drop"）。具体地——**只有在「无任何额外无关节点」的设置下，minimal 预算才能勉强超过 50% 成功率；其它设置全部 <20%**（§4.5 原文）。而 medium 预算即便在有干扰时也稳得多。

> **Why（结果层）——为什么"预算"对多步工具用这么关键？**
> 因为 §7/§9 已经证明多步工具用的瓶颈是**规划深度 + 状态追踪**——这两件事都要"想"：想清依赖顺序、想清哪些值在手。**thinking budget 就是模型"想"的额度**，砍掉它等于让模型在最吃推理的地方裸奔。所以 minimal 预算下一有干扰就崩（<20%），而干扰恰恰是最需要"多想一步去分辨"的场景。

> **读出什么（对我们上 MCP / 调预算的双重警示）**：(1) **别无脑给 agent 挂满 MCP 工具**——即便那些工具"类型上接不上"（Disconnected），大函数集本身就会把成功率拖到 <10%；任务开始时**按需裁剪工具集**是刚需（呼应 §11 b）。(2) **多步工具任务别省思考预算**——这是最不该省 token 的地方；省下的预算会以成功率暴跌的形式加倍还回来。

## §9　RQ5 与失败分析：主病是「用了还没算出来的值」

> 主讲提示：这页是全篇最有迁移价值的诊断。把四类失败讲清，重点砸在压倒性的那一类。

**四类失败（§4.6 定义，对应 §4 的四谓词）**：
1. **Function Not Found**：调了一个**不存在**的函数（违反谓词 1）。
2. **Wrong Number of Inputs**：给的参数个数**多于或少于** schema 允许（违反谓词 2）。
3. **Value Not Yet Known**：试图用一个**尚未通过前序调用或初始输入建立**的变量值（违反谓词 3）。
4. **Incorrect Value**：用了一个**已建立、但值不对**的变量（违反谓词 4）。

**Table 3（跨 {5,10,20} 核 + 全部无关节点设置取平均）失败分布**：

| 失败类型 | GPT-5 | GPT-5-mini | Gemini-2.5-Pro | Gemini-2.5-Flash | Qwen3 | GPT-4.1 | GPT-4.1-mini |
|---|---:|---:|---:|---:|---:|---:|---:|
| Function Not Found | 0.0% | 0.0% | 2.4% | 0.0% | 0.0% | 0.0% | 0.0% |
| Wrong Number of Inputs | 0.0% | 0.0% | 0.2% | 0.9% | 0.1% | 0.0% | 0.0% |
| **Value Not Yet Known** | **79.6%** | **66.8%** | **69.1%** | **81.3%** | **74.0%** | **73.2%** | **66.8%** |
| Incorrect Value | 20.4% | 33.2% | 28.3% | 17.9% | 25.8% | 26.8% | 33.2% |
| *Total errors* | *6,054* | *13,180* | *5,756* | *235* | *9,472* | *3,597* | *8,560* |

**Why（结果层）——为什么「Value Not Yet Known」压倒性地占大头（每个模型 >66%）？**
- 这意味着 **agent 最普遍的错，是想调用一个函数、却给它喂一个「还没被任何前序调用算出来」的变量值**（§4.6 原文 "the most common failure mode across all models is attempting to use variable values that are not yet known, accounting for over 66% of failures in every model"）。换句话说，**agent 没搞清「此刻哪些值已经在手、哪些还得先去算」——这是一种状态追踪 / 计划顺序的失败**，不是「不会用工具」的失败。
- 佐证「不是不会用工具」：**Function Not Found 和 Wrong Number of Inputs 几乎全为 0**（§4.6 "models rarely attempt to call non-existent functions or provide too many inputs, suggesting that they generally understand the function schemas provided"）。即：模型**看得懂 schema、会填参数**，栽的是**多步之间的值传递与顺序**。
- **反直觉的一点**：即便是「长上下文值检索」很强的模型（§4.6 引 Hsieh et al. 2024 RULER），在**多步函数调用里维护上下文**仍然吃力——**"能从长文里捞出一个值" ≠ "能在多步动作里把正确的值传到正确的地方"**。

**案例（§4.7 + Figure 9/17）**：Figure 9a 里 GPT-5 在第 7、8 步调了**无关函数**（`func_pbb`、`func_ped`，都是 CIN），又在第 9 步给核函数 `func_nss` 喂了**错的输入值**（用了 `func_puf`/`func_eni` 的正确值，却又混进 `func_hoj` 的**错值**，尽管 `func_hoj` 第 5 步已成功调过）；Figure 9b 里 GPT-5 **漏掉了必需的核函数**（`func_rgy`、`func_sqo` 全程没调），导致后面只能拿无关值去喂目标函数。

**逐步拆解 Figure 17（`d=5, n^core=10, n^conn=10, n^dis=0` 的完整 14 步轨迹，目标是求变量 `zjy`）**——这是「一步错、步步塌」最清楚的活标本，值得逐帧看：
- **第 1–6 步（全对，蓝色核函数）**：GPT-5 顺利调了 `func_ssc`、`func_cdl`、`func_puf`、`func_eni`、`func_hoj`、`func_zip`，一路把中间变量正确算出来——比如第 5 步 `func_hoj` 正确产出 `xfz=152`（subtype `sxwyrg`）。**前半程说明它读懂了 schema、会填参数、会串前几步**。
- **第 7–8 步（走了诱饵，绿色 CIN）**：GPT-5 调了两个**无关函数** `func_pbb`、`func_ped`（都是 CIN）——它们类型能接上、看着相关，其实是干扰。虽然这两步本身"无错"（返回了值），但**浪费了预算、并往上下文里塞了无关的中间值**，为后面埋雷。
- **第 9 步（首个致命错，Incorrect Value）**：GPT-5 调核函数 `func_nss`，第二个参数**本应是 152（即第 5 步 `func_hoj` 产出的 `xfz`，subtype `sxwyrg`）**，但它**喂成了 932**（Figure 17 用橙框标出 "The second variable is expected to be 152"，并标 "Error: Incorrect Value"）——**它把某个错值当成了 `sxwyrg` 类型的值**。于是 `func_nss` 返回一个**随机错值**（§3.5 的惩罚语义）。
- **第 10–14 步（级联坍塌，连续 Value Not Yet Known）**：从第 10 步 `func_ywt` 开始，GPT-5 试图使用 subtype `tiozz` 的值——但 Figure 17 用青框标出 "No function has returned a variable in subtype 'tiozz' yet"（**还没有任何函数产出过 `tiozz` 类型的值**）。此后第 10、11、12、13、14 步**全部报 "Value Not Yet Known"**：因为第 9 步产出的是错值、且它需要的上游值从没被正确建立，agent 进入「拿着不存在/未确立的值反复试」的死循环，直到撞上预算上限。**目标 `zjy` 始终没算出来 → 判失败。**

> **这个轨迹把三种病一次演全了**：走诱饵（CIN，第 7–8 步）→ 用错值（Incorrect Value，第 9 步）→ 级联用未确立值（Value Not Yet Known，第 10–14 步）。它也直观解释了为什么 Table 3 里 "Value Not Yet Known" 占比这么高——**因为它常常不是"首因"，而是首个错误（用错值/漏调）之后的"并发症"**：一旦某步的正确中间值缺失，后续每一步都会因为"缺料"而反复触发"值尚未确立"。

> **读出什么**：这张失败表把「工具用能力」精准解剖开了：**瓶颈不在"接口理解"，而在"多步状态追踪 / 计划顺序"**。这直接催生了下一节那个惊人有效的缓解招。

## §10　缓解策略：每步重述已知变量值 → GPT-5 62.5%→81.3%（纯 harness 侧、模型不动）

> 主讲提示：**这是本篇对本库最重要的一页**——一个纯上下文层（C 层）的 harness 改动，不碰模型，就把主失败模式压下去。讲清「为什么这么轻的改动这么有效」。

**招法（§4.6.1 Mitigation Strategy）**：既然主病是「用了尚未算出来的值」（Value Not Yet Known，>66%）和「用了错值」（Incorrect Value），那就**在每一步，让每个函数除了返回自己的输出，还额外返回「当前所有已知变量值的清单」**（§4.6.1 原文 "each function ... also returns the list of all known variable values as well"）。

- **它不提供任何新信息**——只是把「模型自己已经发现过的值」**显式重述**一遍（包括那些通过错误调用得到的错值），给模型一份「**哪些值现在可用于函数调用**」的明确上下文（§4.6.1 "This provides no extra information to the model - it simply restates the values ... the model has already discovered"）。
- **它不依赖 FuncBenchGen 框架**，真实场景里**用一个包在函数外面的简单 wrapper 就能实现**（§4.6.1 末）——这点对可迁移性极重要。

**效果（Figure 8，5 核节点、跨 {0,10,20,40} CIN、5 trials）**：
- **GPT-5：62.5% → 81.3%**（Abstract 与 §4.6.1 反复强调的招牌数字，**+18.8 个百分点**）。
- **Gemini-2.5-Pro、GPT-5-mini、GPT-4.1、GPT-4.1-mini** 都有明显提升（Figure 8 橙 > 蓝）。
- **例外**：**Gemini-2.5-Flash 和 Qwen3 只有微小提升、甚至略降**（§4.6.1 "have a small improvement or a slight performance drop"）——作者推测这些模型**未必能有效利用额外提供的上下文**。附录 Figure 16 在 10 核上复现了同样趋势。

> **Why（设计层）——为什么这么轻的改动这么有效？**
> 朴素直觉是「要提升多步工具用，得让模型更聪明/思考更久」。→ 但 §9 的诊断说主病是**状态追踪**（忘了哪些值在手），不是**推理力**。所以对症的不是「加算力」，而是「**把状态外置到上下文里、别让模型靠脑子记**」——把「工作记忆」从模型内部脆弱的隐状态，搬到 prompt 里一份**每步刷新的显式清单**。这是一个典型的「**用 harness 的上下文管理，补 model 的状态追踪短板**」的操作，代价几乎为零（就是每步多打印一段文本）。

> **读出什么（Θ2 铁证 + Θ1 定位到 C 层）**：**模型一个参数没改，只改了 harness 呈现上下文的方式，GPT-5 就涨了 18.8 分**。这是 `Agent = Model + Harness` 又一个受控铁证，且它精确落在 **C（Context）层**——和 Harness-Bench「换整个 harness 摆 23.8 分」形成互补：那个是「换脚手架」，这个是「只动脚手架里的上下文策略」，粒度更细、更可直接抄。**这条几乎是为我们量身定制的**（见 §11 e）。

## §11　★ 对我们的启发（Inspires Us）

> 这一节是组会高潮，也是本库相对 auto-research 的独门优势：**我们（Claude Code / 本课 m9.* 的 agent）本身就是一个 harness**——有真实的 ReAct 循环、工具预算、上下文压缩、子代理编排。FuncBenchGen 压测的「陌生 API 上的多步工具用」，正是我们每天在做的事。所以下面每条都能「打到自己身上」。

➤ **a. 可直接借用的招（method/trick we can reuse）**——**「每步重述已知状态」是一个近乎免费的 harness 补丁**。§10 那招（每次工具返回时，附上「当前所有已确立变量/事实值的清单」）可以**原样搬进我们的工具执行层**：在我们的 ReAct 循环里，每次工具调用的 observation 后面，自动拼一段结构化的「**已知状态摘要**」（已获取的文件路径/内容摘要、已确认的配置值、已完成的子步骤产物）。它不给新信息、只把 agent 自己发现过的东西显式重述，成本就是多打印一段文本——但 §9 说主病正是「用了还没确立的值」，这招直击要害。第二个可借的招是 **§4.6 的四谓词失败归因器**（name-resolution / schema-conformance / dataflow-availability / value-consistency）——这是一套**确定性、免标注**的工具调用错误分类法，可直接接到我们的 trace 上给每次失败自动打标。

➤ **b. 可迁移到我们课题的思路（transfer）**——**把 CIN/DIN 的区分用到我们的工具集卫生（tool hygiene）上**。§8 的核心结论是「拖垮多步工具用的不是工具**数量**，而是**类型/描述上『貌合』的干扰工具（CIN）**」。迁移到我们身上：当我们给一个 agent 挂几十个 MCP 工具时，真正危险的不是「工具多」，而是「有若干工具的**描述/入参 schema 与当前任务貌似相关、其实无关**」。**可迁移的动作**：给我们的工具注册层加一个「**CIN 审计**」——量化「有多少已挂载工具的 type/描述与当前任务目标类型兼容但实际无关」，并在任务开始时**按相关性裁剪工具集**（呼应 Harness-Bench 提到的 Vercel「砍工具 80%→100%」）。迁移前提：需要给工具描述打上机器可读的「类型/领域」标签，否则「貌合」无从度量。

➤ **c. 它暴露的开放问题 = 我们的机会（open problems → our opportunity）**——**在线检测「值尚未确立」**。§9 说 66%+ 的失败是「用了还没算出来的值」，但论文是**事后**从 trace 归纳的；它**没给一个「在线拦截器」**。机会：在我们的工具层加一个**「参数溯源校验器」**——在把一次工具调用真正发出去**之前**，检查它的每个参数值是否**能在当前已确立的状态里找到来源**（要么是初始输入、要么是某次前序工具返回）；找不到来源就**拦下来、提示 agent「这个值你还没获取，先去取」**。这等于把 §4.6 的谓词 3（dataflow availability）从「事后归因」升级成「**事前拦截**」。可下手的第一步：在我们某个多步工具 pipeline 里，对每次调用做一次「参数值 ∈ 已知值集合?」的检查，量化它能否把类似 Figure 17 那种「一步错、步步塌」的级联失败提前掐断。

➤ **d. 与本库其它论文/模块的连接（connect the dots）**——与标杆 **[Harness-Bench](2605.27922-harness-bench-measuring-harness-effects.md)** 是**对偶关系**：Harness-Bench 固定任务、量 harness；FuncBenchGen 固定 harness、造可控任务量 model——两者拼起来才是完整的 `Agent = Model + Harness` 二维扫描。它的**主失败模式「状态/值追踪」**正好接上 Harness-Bench Table 3 的 "state/continuation" 失败线，也与本库 **F 组 AgentFold/IterResearch（上下文折叠/状态重建）** 直接呼应——那些工作攻的正是「多步里怎么把状态留住」，而 FuncBenchGen 的 §10 缓解招是同一问题的极简解（外置状态到上下文）。它的**无污染合成**思路则与 auto-research 库 `m9.6`（评测沙箱）互补：`m9.6` 给沙箱补**真任务**，FuncBenchGen 给沙箱补**可拧难度的合成任务**——两者结合可做「真任务打底、合成任务扫难度曲线」。

➤ **e. 如果我来做下一步（my next move，第一人称、可执行）**——**我会先在我们 `m9.*` 的一个多步工具 agent 上，把 §10 那招（每步重述已知状态）做成一个可开关的 harness flag**，用 FuncBenchGen 自己（开源在 github.com/megagonlabs/FuncBenchGen）生成一组 $n^{\text{core}}\in\{5,10\}$、$d\in\{1,3,5\}$、带 10 个 CIN 的合成任务当**离线回归集**，跑「开/关」对照，看我们自己的 agent 是不是也复现「主失败=Value Not Yet Known」以及「开了重述就涨」。**如果复现**，我就把「每步已知状态摘要」设为我们工具层的**默认行为**，并顺手加上 (c) 的「参数溯源事前拦截器」，量化两者叠加能把多步工具任务的成功率抬多少——这是一个下周就能起步、且有现成开源题库的最小验证。

## §12　局限与批判（论文自陈 + 我的补充）

**论文自陈的局限（诚实）**：
- **只覆盖「纯数据流」的控制流**：§5 Conclusion 明说，**把更复杂的控制流（如条件逻辑 conditional logic、迭代 iteration）纳入是未来方向**（"Incorporating more complex control flows such as conditional logic and iteration would be our interesting future direction"）。当前的 DAG 是**无环、无分支条件、无循环**的——真实 agent 任务常有 if/循环，这层尚未建模。
- **缓解招对部分模型无效/略降**：Gemini-2.5-Flash、Qwen3 用了「每步重述」反而没涨甚至略降（§4.6.1），说明该招**不是普适万灵药**，依赖模型「能用好额外上下文」。
- **大函数集下最强模型也不堪用**：§4.5 承认 GPT-5 在 40 核节点、disconnected 设置下成功率 **<10%**，并直言「**最好的模型都还没准备好用在现有 MCP server 那种大函数集上**」（§4.5 引 Anthropic 2025 MCP）——这是对当前 agent 工具用能力的一记冷水。

**我的补充批判**：
- **「类型/子类型匹配」是真实语义推理的一个**代理**，未必等价**：§3.2 用类型+子类型连函数，是为了避开「变量名字符串匹配太简单」。但真实世界的「谁能接谁」还掺杂**业务语义、单位、时序约束**等，光靠 type/subtype 匹配可能**既漏（语义相关但类型不同）也误（类型相同但语义无关）**。所以 FuncBenchGen 测的是「**类型推理**」这一个切面，不能直接外推成「全部工具用能力」。
- **随机三位数变量值 + 随机函数名，可能低估了「语义先验」的作用**：真实 API 的函数名/参数名是有意义的（`get_hotel_location`），模型能借语义先验猜依赖；FuncBenchGen 刻意抹掉语义（`func_yep`、`mfmjsy=731`），**这诚然保证了无污染，但也把任务变成了「纯符号推理」**——可能比真实工具用**更难**（缺语义线索）也**更纯**（不掺记忆）。这是「无污染」与「生态效度」之间的一个 trade-off，原文未量化这个差距。
- **只报成功率/调用数，未报 token/时延成本**：与 Harness-Bench 同时报 token/turns 相比，FuncBenchGen 主要报成功率与 ACs，**没有报 token 消耗或思考预算的具体开销**（§4.5 只定性讨论 thinking budget）——对「多步工具用的**性价比**」这一实践关切覆盖不足。
- **判据是「最终值对不对」的端到端二值**：一条**过程全错但侥幸蒙对最终值**的轨迹会被判成功，一条**过程几乎全对只差最后一步**的会被判失败。这种粗粒度判据**不区分「真会」与「蒙对」**（虽然随机三位数值让蒙对概率很低，但四谓词的过程信号其实可以支撑更细的部分credit，论文没这么打分）。

- **「无关但连接」的 CIN 结论，可能部分是"分心"、部分是"图变复杂"的混淆**：§8 把「Connected 掉分」归因于「CIN 与核节点类型兼容、难分辨」（分心假说）。但加入 CIN 也**客观上让整张图的边更多、结构更密**（附录 A.1：CIN 挂为核节点的孩子）——所以掉分里有多少是"被诱饵分心"、有多少是"图本身变复杂/待判分支变多"，**论文未做进一步消融拆分**。一个干净的消融是：固定图的总边数，只在"CIN 类型兼容"与"CIN 类型不兼容"间切换，看掉分是否仍显著——这正是 §14 讨论题 3 的由来。

- **缓解招"重述已知值"可能把错值也一起固化**：§10 明说重述的清单**包含通过错误调用得到的错值**（§4.6.1）。这是一把双刃剑——它固然帮 agent 记住"有哪些值可用"，但也可能**把一个早期的错值反复摆在眼前、强化 agent 继续用它**（正是 Figure 17 第 9 步那种"把 932 当成 152"的错，如果被重述固化，可能更难自我纠正）。论文只报了净效应（GPT-5 +18.8），**未拆分"帮记正确值"与"固化错误值"这两股相反力量**——这也解释了为什么它对 Flash/Qwen3 可能净为负。

## §13　版图定位（canon/前沿坐标 + 在本库的位置）

- **时间坐标（Θ4）**：**2026 前沿**，**ICLR 2026 正式接收**。它相对基石推进的一步很清楚——把工具用基准的两块短板「**无污染**」与「**难度可控**」**一次补齐**：Table 1 显示 API-Bank / BFCLv4 / ToolBench / ComplexFuncBench / LongFuncEval 在「Contamination-Free」列**全是叉**，在「可控性」三列（Required Func. Size / Dependency Depth / Irrelevant Func. Type）也大面积是叉，而 **FuncBenchGen 全勾**。它不是要取代那些「真实 API 集」基准（那些有生态效度），而是给领域补一台**合成风洞**，用来做「哪个因素最影响性能」的**受控因子分析**（§1 末动机）。
- **E/T/C/L/O/V 归属（Θ1）**：主坐 **T（Tools）** 层（压测工具接口 / 多步工具用），身份是 **V（Validation）** 层的**任务生成器**；实现上依赖 **O（Observability）**（四谓词执行器）；预算约束触及 **L（Loop）**（调用数封顶 2×）；而它最亮的缓解招落在 **C（Context）** 层（每步重述状态）。是一篇「以 T/V 为主、串起 O/L/C」的综合型评测论文。
- **回扣 `Agent = Model + Harness`（Θ2）**：贡献了**两类证据**。① **Model 侧天花板**：固定 harness、只加任务难度，GPT-5 从 72.5% 崩到 15%（Table 2）、深依赖 <30%（§4.4）——证明「模型那一半」有硬上限，脚手架再好也扛不住深链。② **Harness 侧杠杆**：模型不动、只改上下文呈现（每步重述已知值），GPT-5 涨 18.8 分（62.5%→81.3%，Figure 8）——证明「脚手架那一半」有巨大且廉价的操作空间。**两条合起来，正是本库中心命题最干净的一次「双向」佐证**。
- **regime 诚实（Θ5，不把结论绝对化）**：本篇的结论**分 regime**——(i) **任务越深/越串行（$d$ 大），Model 的短板越主导**，缓解招也越吃力；(ii) **任务越浅/越星形（$d=1$），成功率近 90%**，此时瓶颈不在能力而在干扰（CIN）；(iii) **缓解招（harness 上下文侧）的增益因模型而异**——GPT-5/Gemini-Pro 大涨，Flash/Qwen3 无感甚至略降。所以**不能笼统说「重述状态一定有用」或「模型一定不行」**：它取决于**深度 × 模型是否善用上下文**这两维。这与 Harness-Bench §9「强模型更不挑 harness」是同一种 regime 诚实——**harness/上下文操作的边际价值，随模型变强、任务变浅而衰减**。

> **在本库的位置**：C 组（工具 / ACI / 评测）的一块**前沿压舱石**，与 G 组标杆 Harness-Bench 构成「**造压力源 ↔ 量脚手架**」的对偶。读完它，再回看任何一篇工具用 / 上下文 / 循环的论文，都能追问一句：「**它改的东西，是在降 FuncBenchGen 的哪一类失败（Value-Not-Yet-Known / Incorrect-Value / CIN 干扰）？**」

## §14　组会讨论问题（留给大家吵）

1. **无污染 vs 生态效度**：FuncBenchGen 用随机函数名/变量值换来「无污染」，代价是抹掉了真实 API 的语义先验。**这让任务更难还是更容易？** 如果给函数名/描述注入真实语义（但仍随机组合结构），无污染还保得住吗？成功率会怎么变？
2. **缓解招为什么对 Flash/Qwen3 无效？** §10 那招在 GPT-5 上 +18.8，在 Gemini-Flash/Qwen3 上却无感甚至略降。是「这些模型不善用长上下文」，还是「重述的文本反而挤占了它们的有效上下文」？怎么设计消融去分辨？
3. **深度 vs 分支**：§4.4 发现「深度大掉分，但同样深度下分支少反而好，且只在大模型上」。**「有效规划深度」和「每步决策分支数」谁是更本质的难度来源？** 能不能用 FuncBenchGen 拧一组「等深度、变分支」的图把它俩解耦？
4. **把它接到我们自己身上**：如果用 FuncBenchGen 的合成题去测**我们（Claude Code）** 的多步工具用，你预期主失败模式还是「Value Not Yet Known」吗？我们的**上下文压缩/compaction** 会不会**加剧**这个失败（把已确立的中间值压没了）？

## §15　一页速记

- **命题重述**：把「多步工具调用」= **在一张隐藏的函数依赖 DAG 上，从已知变量找路到目标变量**（§3）。agent 只拿函数 schema + 已知输入，**拿不到图**，要靠「类型/子类型匹配」推断依赖（§3.2）。
- **无污染怎么来**：图在**评测时现场合成**（随机函数名、随机三位数值）→ 训练集没见过（§3.2）。
- **难度怎么拧**：四旋钮 `d`(深度) / `n^core`(核节点/序列长) / `n^conn`(CIN 诱饵) / `n^dis`(DIN 噪声)（Figure 1/2）。
- **主结果**：GPT-5 随序列变长崩塌——72.5%(5核)→38.2%(10核)→**15%(20核)**（Table 2）；深依赖 $d$=4–8 时 <30%（§4.4）；通用模型（GPT-4.1 系）10 核以上≈0。
- **最毒干扰**：**CIN（类型兼容的诱饵）对所有模型都严重掉分**；DIN（孤立噪声）影响因模型而异（§4.3）。
- **主失败**：**「Value Not Yet Known」（用了还没算出来的值）占每个模型 >66%**（Table 3）——瓶颈是**多步状态追踪**，不是接口理解（Function-Not-Found≈0）。
- **缓解铁证**：**模型不动，每步把已知变量值重述给 agent → GPT-5 62.5%→81.3%（+18.8）**（Figure 8）——纯 **C 层** harness 改动，`Agent=Model+Harness` 的又一受控证据。
- **诚实边界**：只建模纯数据流 DAG（无 if/循环，§5）；缓解招非普适；大函数集下最强模型 <10%（§4.5，"还没准备好用现有 MCP server"）。
- **对我们**：搬「每步重述已知状态」进我们工具层 + 加「参数溯源事前拦截器」（把谓词 3 从事后归因升级为事前拦截）+ 用它的合成题当离线回归集扫我们 agent 的工具用难度曲线。
