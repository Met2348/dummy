# MemGPT：把 LLM 当操作系统——用虚拟内存分页给有限窗口装上无界记忆

> **本篇属 agent-harness 库 D 组（上下文/记忆），层 = C。** 它奠定了一个此后被反复引用的核心隐喻：
> **LLM 的上下文窗口 = 操作系统里的物理内存（RAM），外部存储 = disk**；当"内存"装不下时，就像 OS 那样在两者之间**分页（paging）**，
> 而且——关键创新——**让模型自己用函数调用来决定何时换入/换出**。读它时请始终带着一个对照：
> 我们自己的 harness（Claude Code / 本课 m9.* 的 agent）里那套 **compaction（上下文压缩）+ 记忆文件**，
> 本质上就是 MemGPT 分页机制的一个"现实版实现"——只不过我们的换入/换出目前多由外部规则触发，而 MemGPT 让模型**自主**触发。
> 这条对照会一路贯穿到 `## ★ 对我们的启发`。

---

## §1　TL;DR（一页讲清这篇在干嘛）

> 主讲提示：开场先把"痛点—隐喻—机制—证据"四拍打完。这是一篇 2023 的 canon，价值不在某个 SOTA 数字，而在它**定义了一整套看待 LLM 记忆的语言**。

**一句话**：LLM 的上下文窗口是有限的（论文写作时最常用的开源模型只有 2k–8k token，Table 1），一旦对话变长或文档变大，窗口就装不下——朴素做法要么**截断**（丢历史）、要么靠**外部检索（RAG）**临时塞几段进来。MemGPT 提出 **virtual context management（虚拟上下文管理）**：把窗口当作 OS 的**物理内存**、把外部数据库当作 **disk**，当"内存压力"上来时，**由 LLM 自己发起函数调用**把信息在两层之间**分页（page in / page out）**，从而在**固定窗口**的模型上营造出**无界上下文**的错觉（§1、§2）。

- **属于 harness 的哪一层（Θ1）**：本篇打的是 **C（Context / 记忆）层**——它不改模型权重、不换底座，只在模型**外面**重新组织"什么该在窗口里、什么该睡到磁盘上、谁来搬"。这正是 harness 六层（E/T/C/L/O/V）里的 **C**，也是库内 **D 组**的核心议题。它对 **T（Tools）层有强依赖**：整套分页动作是通过 LLM 的**函数调用（function calling）**能力实现的（§1 明说"leverage the recent progress in function calling abilities"）；也用到 **L（Loop）层**——靠事件驱动的控制流与函数链把多步检索串起来（§2.4）。
- **回扣全库论点（Θ2）**：MemGPT 是 `Agent = Model + Harness` 里 **Harness 一侧对"记忆/上下文"这块的经典诠释**。它给出最锋利的一条证据（Table 2）：**同一个底座模型 GPT-4，套上 MemGPT 后，深度记忆检索（DMR）准确率从 32.1% 跳到 92.5%**——模型没变，变的只是"外面这层怎么管记忆"。
- **够权威（Θ4）**：UC Berkeley（Stoica / Gonzalez 组，即 Spark、Ray、vLLM、Gorilla 的同一支）2023 年 10 月放出、2024 年 2 月更新的工作。它是**记忆型 agent 领域被引用最广的奠基工作之一**，其开源实现后来发展为 **Letta**；库内 D 组后续所有"给 agent 装长期记忆"的论文（Mem0、A-MEM 等）都能追到这个源头。**本篇是 D 组的 canon 压舱石。**

> **读出什么**：不要把 MemGPT 读成"又一个 RAG 变体"。它的真正贡献是**把'记忆管理'形式化成一个由模型自己求解的决策问题**——"这条信息，我该现在搬进窗口、还是留在磁盘上？"。这一步"让模型自管理"，是它区别于所有"外部规则驱动的检索/压缩"的分水岭。

## §2　问题与动机：有限窗口到底卡住了什么

> 主讲提示：这一段用 Why 三连的"问题层"，把"窗口有限"为什么是个**结构性**难题讲清，而不是"买张更大的卡就行"。

**Why（问题层）——不解决会卡住什么？谁受影响？**

论文 §1 给出的动机链条很硬：

1. **窗口天生有限，且扩不动**。Transformer 的自注意力对序列长度是**二次**（quadratic）的计算与内存开销，所以直接把上下文拉长"代价高昂"，是一个"迫切的研究难题"（§1 原文 "incurs a quadratic increase in computational time and memory cost"）。Table 1 给了当年的现实：最常用的开源模型 Llama-2 只有 **4k** token（约 60 条消息），GPT-4 release 版 **8k**，即便当年"最长"的 Yi-34B-200k 也就 200k（约 4000 条消息）——对"跨月、跨年的对话"或"百万 token 级年报"依然杯水车薪。
2. **就算能拉长，也用不好**。作者引 Liu et al. 2023a（"Lost in the Middle"）指出：长上下文模型对**放在中间**的信息利用率很差——"the model is more capable of recalling information at the beginning or end of its context window, vs tokens in the middle"（§3.2）。即注意力分布不均，长窗口的边际收益递减。
3. **谁受影响 / 两个受灾场景**：
   - **对话智能体**：受限窗口的模型缺乏"上下文感知、人设一致性、长期记忆"，聊久了就"失忆、串戏"（§1、§3.1）。
   - **文档分析**：标准文本文件的长度很容易超过现代 LLM 的输入容量；真实的金融文档（如 SEC Form 10-K 年报）"轻松突破百万 token"，且很多任务要**跨多篇长文档**连线（§3.2）。

**Why（设计层）——为什么不用两个"显而易见的替代"？**

> **Why（设计层）**：
> - **朴素替代 A：把窗口一直做大（长上下文模型）**。→ 会因为①二次开销、②"Lost in the Middle"式的利用率坍塌而**边际收益递减**（§1 明说 "diminishing returns of context scaling"）。作者据此下判断："given the considerable resources needed … and diminishing returns …, there is a **critical need for alternative techniques**"。
> - **朴素替代 B：朴素 RAG（固定检索一批文档塞进窗口）**。→ 检索是**一次性、被动**的：检索器（retriever）召回什么就只能看什么，召不回关键那段就永远看不到（§3.2.1 的实验直接证明这点——固定上下文基线的表现被检索器天花板**卡死**，"capped roughly at the performance of the retriever"）。
> - **MemGPT 的选择**：不改窗口大小、也不满足于一次性检索，而是**借鉴 OS 的虚拟内存**——让系统能**反复、主动**地把数据在"内存↔磁盘"间搬进搬出，而且**让 LLM 自己决定搬什么、什么时候搬**（§1 "iteratively modify what is in its context for a single task"）。

> **读出什么**：这三条替代方案的对照，正是理解 MemGPT 定位的钥匙——它站在"长上下文"与"朴素 RAG"**中间**，用"可迭代 + 自主决策"补上两者各自的缺口。库内 D 组的另一侧（"Less-Context / 更少上下文反而更好"的反方，见 §13）也是从"替代 A 的失败"出发，但走向了相反的药方，这个张力后面会专门拆。

## §3　核心隐喻：LLM as OS，逐词对照

> 主讲提示：这一页是全篇的"世界观地基"。**必须把 OS 类比讲透**——每一个 OS 概念都精确对应 MemGPT 的一个部件。我用一张"术语对照表"钉死映射，再逐条落地。

传统 OS 用**虚拟内存（virtual memory）**制造"内存比实际大"的错觉：把溢出的数据**分页（paging）**到磁盘，等应用访问时再通过**缺页（page fault）**把它换回内存（§2 原文复述了这套机制）。MemGPT 把这套整体搬到 LLM 上，作者称之为 **"LLM OS"**（§2）。逐词对照如下：

| OS 概念（传统操作系统） | MemGPT 里的对应物 | 直觉/作用 |
|---|---|---|
| **物理内存 / 主存 / RAM** | **主上下文（main context）= prompt tokens** | 模型推理时**唯一能直接读到**的东西；容量 = 窗口大小，有限 |
| **磁盘 / disk（二级存储）** | **外部上下文（external context）** | 窗口**之外**的所有信息；模型看不到，除非被显式搬进来 |
| **分页（paging，内存↔磁盘搬运）** | **page in / page out**（函数调用搬数据） | 把外部信息搬进窗口（换入）、把窗口里不常用的信息写出去（换出） |
| **缺页中断（page fault）** | **函数调用检索**（如 `recall_storage.search(...)`） | 当前窗口缺某条信息 → 模型主动发起检索把它"换回" |
| **中断 / interrupt** | **memory pressure 系统警告 + 事件（events）** | "内存快满了"这类系统消息，打断模型、催它做记忆管理（Fig 1） |
| **系统调用 / syscall** | **MemGPT 提供的记忆管理函数** | 模型通过调用这些函数来读写自己的内存（唯一合法途径） |
| **MMU / OS 内核（谁来管分页）** | **LLM 自己（self-directed）** | ★ 最关键差异：搬运策略不是外部硬编码，而是**由模型在推理中自行决定** |

**Why（设计层）——为什么"让模型自己当内核"，而不是写一套外部规则去搬？**

> **Why（设计层）**：
> - **朴素做法**：外部写死一套启发式（heuristics）——"最近 N 条留窗口、更早的自动摘要、检索固定 top-k"。→ 这套规则**不懂任务语义**：它不知道"生日是 2 月 7 日"这条对当前对话至关重要、必须留在窗口，而某句寒暄可以被换出。规则是**任务无关**的，一定会在某些场景丢关键信息。
> - **MemGPT 的选择**：把"搬什么"这个决策**交给 LLM**——因为只有模型本身"读懂了当前在聊什么/在做什么"，才能判断哪条信息该 page in、哪条该 page out（§2.3 "Memory edits and retrieval are entirely self-directed: MemGPT autonomously updates and searches through its own memory **based on the current context**"）。代价：模型可能**决策失误**（该搬的没搬、不该搬的搬了），这也是它 §3.2.1 里"often stop paging before exhausting the retriever database"这类失败的根源（见 §9 批判）。

> **读出什么（贯穿全篇）**：MemGPT 的一句话内核 = **"把记忆管理从'外部固定策略'升级为'模型自己求解的决策问题'"**。整篇论文的所有部件，都是为了让这个"模型当内核"能真正跑起来：给它内存分层（§4）、给它函数去搬（§5）、给它中断去催（§6）、给它控制流去把多步搬运串成一次多跳检索（§7）。

## §4　主上下文（RAM 那半）：三段式窗口

> 主讲提示：这页讲"窗口里到底装了什么"。窗口不是一坨自由文本，而是被切成**三段有明确读写权限**的区域——这本身就是 OS 式内存管理的味道。

MemGPT 把 prompt tokens（= 主上下文 = "in-context，能被模型直接读到"的部分）切成**三个连续段**（§2.1）：

1. **系统指令（system instructions）**——**只读、静态**。里面写死了 MemGPT 的**控制流说明、各记忆层的用途、以及如何调用记忆函数**（例如"怎么检索 out-of-context 数据"）。相当于 OS 内核代码：模型读它来知道"我有哪些内存、该怎么管"。
2. **工作上下文（working context）**——**固定大小、可读写，只能经函数调用改写**。这是一块结构化的"便签本"：在对话场景里，用来存**用户关键事实、偏好、以及 agent 正在扮演的人设**（persona），让对话流畅一致（§2.1）。它对应 OS 里"常驻内存的关键工作集"。
3. **FIFO 队列（FIFO queue）**——**滚动的消息历史**。存 agent↔user 的往返消息、系统消息（如内存警告）、以及函数调用的输入输出。**队列的第 0 位很特殊**：它存一条**递归摘要（recursive summary）**，是对"已被挤出队列的那些消息"的压缩记录（§2.1）——相当于"被换出到磁盘的历史"的一个索引/摘要。

> **读出什么**：这三段的**读写权限差异**是 harness 设计的精髓——只读的系统指令保证控制逻辑不被模型自己改坏；working context 只能经函数改写，保证关键事实的修改是**可追踪、走正规接口**的（对照 Fig 4：改用户状态是靠 `working_context.replace("Boyfriend named James", "Ex-boyfriend named James")` 这样的函数调用，而不是模型随口一改）。这与我们 harness 里"系统提示只读 / 记忆文件经工具写"的分工完全同构（见 Inspires-Us）。

## §5　外部上下文（disk 那半）+ 谁来搬：队列管理器 & 函数执行器

> 主讲提示：这页把 Figure 3 的架构图讲活。核心是三个搬运工——Queue Manager、Function Executor——和两块磁盘——Recall Storage、Archival Storage。

**外部上下文（external context）= out-of-context 数据**：任何在窗口**之外**的信息。它**必须被显式搬进主上下文**，才能在推理时被模型读到（§2 开头强调这条铁律）。外部存储分**两块磁盘**（Fig 3）：

- **Recall storage（回溯存储）**：**MemGPT 的消息数据库**——完整的历史消息都落在这里。相当于"对话的完整录像带"，被换出窗口的消息不会丢，只是睡到这块盘上（§2.2）。
- **Archival storage（归档存储）**：一块**读写的通用数据库，存任意长度的文本对象**（§2.2）。它是 agent 的"长期笔记本 / 外挂知识库"——文档分析场景里，整个待分析文档集就被灌进 archival storage（§3.2.1）。

**两个搬运工**（Fig 3 的读写箭头把权限标得很清楚）：

- **队列管理器（Queue Manager，§2.2）**：管 FIFO 队列与 recall storage。收到新消息时，它把消息**追加进队列**、拼好 prompt tokens、**触发一次 LLM 推理**拿到输出（completion tokens），并把"进来的消息 + 生成的输出"都写进 recall storage。当模型请求检索 recall 里的旧消息时，队列管理器把命中的消息**追加回队列尾部**，重新进窗口。—— 它还负责**上下文溢出控制**（这就是下一节的"内存压力"，放到 §6 讲）。
- **函数执行器（Function Executor，§2.3）**：处理 completion tokens。模型的输出被**解析为函数调用**，执行器负责真正搬数据——在主上下文与外部存储之间移动。**记忆的编辑与检索完全自导（self-directed）**：MemGPT 依据当前上下文**自主**更新、搜索自己的记忆（§2.3）。

> **读出什么**：把 Fig 3 读一遍就会发现，MemGPT 的架构是一台**冯·诺依曼式的小机器**：`System Instructions | Working Context | FIFO Queue`（三段主存）→ 送进 LLM processor 推理 → 输出 buffer → 被解析成函数 → Function Executor / Queue Manager 去改主存或读写两块磁盘 → 反馈再进主存。**这就是"把 LLM 当 CPU、把 harness 当 OS"的字面实现。**

## §6　内存压力与中断：窗口快满了怎么办

> 主讲提示：这是 OS 类比里最精彩的一环——**中断（interrupt）**。窗口不是"满了就崩"，而是像 OS 一样，在快满时**发系统警告打断模型**，催它主动腾地方。把这段的两个阈值讲清。

队列管理器负责**上下文溢出控制**，用一套**基于阈值的驱逐策略（eviction policy）**（§2.2）：

- **警告线（warning token count，例：窗口的 70%）**：当 prompt tokens 触到这条线，队列管理器**往队列里插一条系统消息**，警告模型"内存压力来了、队列即将被驱逐"（a 'memory pressure' warning）。这条系统消息就是 **Fig 1 里那条红色的 `System Alert: Memory Pressure`**——它给模型一个"最后的机会"，去把 FIFO 队列里的重要信息**主动搬**到 working context 或 archival storage 保存起来（§2.2）。
- **冲刷线（flush token count，例：窗口的 100%）**：当 prompt tokens 触顶，队列管理器**冲刷（flush）队列**：用"现有递归摘要 + 被驱逐的消息"**生成一条新的递归摘要**，把队列腾出一大块空间（例：清到窗口的 50%）。被冲掉的消息**不再在窗口里可见，但永久保存在 recall storage**，随时能靠函数检索回来（§2.2）。

**Why（设计层）——为什么要"先警告、再冲刷"两级，而不是满了直接砍？**

> **Why（设计层）**：
> - **朴素做法**：窗口满了直接**硬截断**尾部（大多数聊天应用就这么干）。→ 关键信息（可能正好在被砍的那段）**无声丢失**，模型甚至不知道自己丢了东西。
> - **MemGPT 的两级设计**：先在 70% 处**中断并通知模型**"快满了，你有重要的东西赶紧存"——给模型一个**自主保存**的窗口期；实在到 100% 才冲刷，且冲刷也**不是丢弃而是转存 + 摘要**（信息进了 recall storage，还能捞回来）。这样"截断"就从"无声丢失"变成了"**有序换出 + 可召回**"。这正是 §2.2 说的设计意图——让 self-editing 机制"有效工作"，前提是模型对自己的 token 限制**有感知**，所以系统"prompts the processor with warnings regarding token limitations"。

> **读出什么**：这条"内存压力中断"是 MemGPT 里**最该被我们抄的机制之一**。我们 harness 的 compaction 目前多是"到阈值自动压缩"，模型往往**是被动的**（压缩由外部触发、模型不一定知道自己即将被压）。MemGPT 的做法是**先给模型一个'你快被压缩了，快把要紧的东西写进长期记忆'的中断**——把"要保住什么"的决策权交还给最懂任务的那一方（模型）。这条直接进 Inspires-Us e。

## §7　控制流与函数链：把"多步搬运"串成一次多跳检索

> 主讲提示：这页讲 MemGPT 怎么把"一次搬一页"升级成"连续搬好几页、边搬边想"。关键词两个：**事件驱动（event-driven）** 和 **函数链（function chaining）**。

**事件驱动的控制流（§2.4）**：在 MemGPT 里，**事件（events）触发 LLM 推理**。事件是被泛化的输入，可以是：用户消息（聊天）、系统消息（如内存压力警告）、用户交互（如"用户刚登录""用户上传了文档"）、以及**定时事件**（按计划触发，让 MemGPT 能"无人催促地自己动"——unprompted）。MemGPT 用一个解析器把这些事件转成纯文本消息，喂进 LLM。—— 这对应 OS 里"由中断/事件驱动的调度"。

**函数链（function chaining，§2.4）——为什么需要它**：很多任务要**连续调用多个函数**才能完成，比如"翻好几页检索结果"或"从多篇文档分别取数据再汇总"。为此 MemGPT 设计了一个特殊标志：**`request_heartbeat=true`**（Fig 3 标注）。

- 当模型在某次函数调用里**带上 `request_heartbeat=true`**：函数执行完后，控制权**立刻还给模型**，把函数结果加进主上下文，**紧接着链式发起下一次 LLM 推理**——于是模型可以"看到这一页结果 → 决定再翻下一页 / 换个查询"，把多步串起来。
- 若**不带**这个标志（称为 **yield / 让出**）：MemGPT **不再立即推理**，而是**等下一个外部事件**（如用户下一条消息、或定时中断）才继续。

> **读出什么（Θ2 呼应 L 层）**：`request_heartbeat` 本质是一个**"我还要继续、别把 CPU 让出去"的自愿信号**。它让 MemGPT 的控制循环既能"多步自主推进"（链式），又能在无事可做时"礼貌让出、等事件"（yield）——这正是一个成熟 agent **控制循环（L 层）**该有的两种模式。文档 QA 里 MemGPT 能"翻到第 2 页检索结果再答"（Fig 6：`archival_storage.search("nobel physics", page=2)`），靠的就是这条函数链。

**自我修正的反馈闭环（§2.3）**：每个推理周期，LLM processor 拿主上下文（拼成一个字符串）当输入、生成输出串；输出被 MemGPT 解析校验，若函数参数合法就执行。执行结果——**包括任何运行时错误**（比如"想往已满的主上下文里加东西"）——都被**喂回处理器**。这个反馈闭环让系统"从自己的动作里学习并调整行为"（§2.3 原文 "learn from its actions and adjust its behavior"）。

## §8　实验一：对话智能体——同模型 +MemGPT，DMR 从 32% 到 92%

> 主讲提示：这是 Θ2 的"实锤页"。停在 Table 2 上多讲两句——**模型没变，记忆层一换，深度记忆检索几乎从零翻到满分**。

**评什么（§3.1）**：对话 agent 要满足两条——**Consistency（一致性）**：新事实/偏好/事件要和此前一致，别串戏；**Engagement（参与度）**：能主动调用长期知识把消息个性化。为此设两个任务，都建在 **Multi-Session Chat（MSC）数据集**上（Xu et al. 2021，人工标注、每人扮一个固定人设、每段多会话）。

### §8.1　深度记忆检索 DMR（测一致性）

**任务定义（§3.1.1）**：向 agent 提一个**只能靠回忆此前某次对话才能答对**的问题，答案范围很窄。QA 对由一个独立 LLM 生成——它被要求写一个"user A 问 user B、且**只有真参与过此前对话**才答得出"的问题（附录 6.1.3 给了完整生成 prompt，还反复强调"绝不能出能从 persona 直接推出的题——那算作弊"）。评分用 **ROUGE-L 的 recall（R）** 对齐 gold 答案 + 一个 **LLM judge** 判"生成答案与 gold 是否一致"（§3.1.1，用 ROUGE-L recall 是因为生成答案通常比 gold 啰嗦，附录 6.1.2 给了 judge 的完整 prompt）。

**结果（Table 2，"Deep memory retrieval (DMR) performance"）**：

| 模型 | Accuracy ↑ | ROUGE-L (R) ↑ |
|---|---:|---:|
| GPT-3.5 Turbo | 38.7% | 0.394 |
| **+ MemGPT** | **66.9%** | **0.629** |
| GPT-4 | 32.1% | 0.296 |
| **+ MemGPT** | **92.5%** | **0.814** |
| GPT-4 Turbo | 35.3% | 0.359 |
| **+ MemGPT** | **93.4%** | **0.827** |

**Why（结果层）——为什么是这么大的跳变？**

基线（不带 MemGPT）的处理方式是：把此前 5 段会话做成**一份递归摘要**塞进 prompt，让模型据摘要作答（§3.1 说明）。MemGPT 则**不看摘要**，而是通过**分页检索**去 recall storage 里把**真正相关的原始消息**捞进窗口再答。机制上：**摘要是有损压缩**——DMR 问的又恰恰是"窄答案、藏在某次具体对话里的细节"，这类细节最容易在摘要里被抹掉；而 MemGPT 能"按需精准召回原文"，所以准确率从 32.1% 飙到 92.5%（§3.1 原文 "clear drop in both accuracy and ROUGE scores when going from MemGPT to the corresponding LLM baselines"）。

> **读出什么（Θ2）**：这一行就是 `Agent = Model + Harness` 对**记忆层**最干净的证明——**底座是同一个 GPT-4**，唯一区别是"外面这层用摘要压缩 vs 用分页按需召回原文"，DMR 就差了 **60 个百分点**。把它并到全库其它"同模型换 harness"的证据里（HarnessBench 的 23.8 分极差、Cursor 46%→80%），MemGPT 提供的是**记忆维度**的那一记。

### §8.2　对话开场白（测参与度）

**任务定义（§3.1.2）**：让 agent 主动写一句**能勾起用户、且取材于此前对话所积累知识**的开场白。用与 gold persona 的相似度 **SIM-1/SIM-3** 衡量"是否取材于人设知识"，用与人类写的开场白的相似度 **SIM-H** 衡量"像不像人写的"（§3.1.2）。

**结果（Table 3，"Conversation opener performance"）**：

| 方法 | SIM-1 ↑ | SIM-3 | SIM-H |
|---|---:|---:|---:|
| Human（人类基准） | 0.800 | 0.800 | 1.000 |
| GPT-3.5 Turbo | 0.830 | 0.812 | **0.817** |
| GPT-4 | **0.868** | **0.843** | 0.773 |
| GPT-4 Turbo | 0.857 | 0.828 | 0.767 |

> **读出什么**：MemGPT 生成的开场白在 SIM-1/SIM-3 上**能达到甚至超过人类**（0.868 vs 0.800），说明"把用户信息存进 working context"确实能让开场白更有料、更个性化（§3.1.2 原文 "storing information in working context is key to generating engaging openers"）。这一项虽不如 DMR 那样戏剧化，但补上了"记忆不仅让 agent **记得对**，还让它**说得更走心**"这半边。

## §9　实验二：文档分析——把窗口"卡死在检索器"的天花板顶穿

> 主讲提示：这页讲 MemGPT 在"长文档 + 多跳"上怎么赢固定窗口基线。两个任务：多文档 QA、嵌套 KV 检索。核心机制都是"**可迭代分页**"。

### §9.1　多文档问答（Fig 5）

**设置（§3.2.1）**：基于 retriever-reader 的文档 QA（数据来自 NaturalQuestions-Open）。基线与 MemGPT **用同一个检索器**（按 OpenAI `text-embedding-ada-002` 的余弦相似度取 top-K）。MemGPT 把**整个文档集灌进 archival storage**（PostgreSQL + pgvector + HNSW 索引，亚秒级向量检索），检索器"自然涌现"为 archival storage 的搜索功能。随着召回文档数 **K** 增大评准确率。

**结果（Fig 5，"Document QA task performance"）**：

- **固定上下文基线的表现被检索器天花板卡死**——"capped roughly at the performance of the retriever"（§3.2.1）：它们只能看窗口里那 K 篇；一旦检索器没把 gold 文档排进 top-K，基线**永远不可能答对**。而 NaturalQuestions 标注的黄金文档"常常掉在前几十篇之外，甚至更远"（§3.2.1），基线因此吃亏。
- **MemGPT 不受召回文档数增大的影响**（Fig 5 里 MemGPT 的曲线基本平），因为它能**反复向检索器发查询、翻页**（page through），把可用文档数**从"窗口能装几篇"解绑**。GPT-4 / GPT-4 Turbo 版 MemGPT 在此任务上结果相当；**GPT-3.5 版明显退化**，因其函数调用能力弱（§3.2.1）。

> **Why（结果层）**：为什么 MemGPT 反而不怕文档多？因为它把"看文档"从**一次性被动接收**（基线）改成了**主动、可迭代的检索**——"actively retrieves documents … and can iteratively page through results"（§3.2.1）。总可用文档数不再受窗口大小限制。**这正是 §2 里说的"朴素 RAG 的缺陷（一次性）被'可迭代'补上"的实证。**

### §9.2　嵌套 KV 检索（Fig 7）——多跳的压力测试

**任务定义（§3.2.2）**：在合成的 Key-Value 检索基础上，作者提出 **nested KV retrieval（嵌套键值检索）**——一个 key 的 value **本身又是另一个 key**，需要**顺着链一路查下去（multi-hop）**。固定 140 对 UUID（约 8k token），把嵌套层数从 0 变到 4（即最多 4 次 KV 查找才到最终值），采 30 种不同排列。

**结果（Fig 7，"Nested KV retrieval task performance"）**：

- **GPT-3.5**：在**1 层**嵌套就直接掉到 **0% 准确率**（它的典型失败是"干脆返回原始 value"，不肯继续跳）。
- **GPT-4 / GPT-4 Turbo（作为基线）**：比 3.5 好，但也在 **3 层**掉到 0%。
- **MemGPT（GPT-4）**：**不受嵌套层数影响、能稳定完成**——因为它能"对存在主上下文里的 KV 对**反复发函数查询**"，把多跳查找一路做完（§3.2.2 原文 "able to perform the nested lookup by repeatedly querying … through function queries"）。一个有意思的细节：**MemGPT (GPT-4 Turbo) 反而不如 MemGPT (GPT-4)**（Fig 7）——说明"更新的底座 ≠ 在这套 harness 上一定更强"，二者交互复杂（§3.2.2）。

> **读出什么**：嵌套 KV 是"**多跳推理 = 多次分页 + 边查边想**"的最小可复现证明。MemGPT 能过、固定窗口过不了，差别不在"能不能一次看到所有 KV"（它们本来就都在 8k 窗口里！），而在"**肯不肯、能不能顺着链持续发起多步查询**"——即函数链（§7）+ 自主检索的组合。

## §10　把宣称与批判分清：MemGPT 说了什么、我补充什么

> 主讲提示：canon 也要挑刺。这页严格区分"论文自己的宣称/局限"与"我的独立批判"。

**论文自己的宣称（faithful）**：
- 在两个"受限窗口重伤表现"的域（文档分析、对话）上，MemGPT **都超过了现有 LLM 方法**（§2 结尾、§5 结论）。
- 机制上，MemGPT 用"记忆分层 + OS 式控制流 + 事件驱动"，让**有限窗口的 LLM 处理无界上下文**（§2）。
- 结论（§5）把功劳明确归给"**OS 技术**（分层记忆管理 + 中断）"，并把 MemGPT 定位为"用 OS 架构思想最大化 LLM 潜能"的一个**有前景的新方向**，列了若干后续方向（接入不同存储层如数据库/缓存、改进控制流与记忆管理策略）。

**论文自陈的局限 / 边界（诚实之处）**：
- **强依赖函数调用能力**：GPT-3.5 版在文档 QA、嵌套 KV 上都明显退化，作者归因于"limited function calling capabilities"（§3.2.1、§3.2.2）。→ **MemGPT 的上限被底座的函数调用能力锁死**。
- **模型会"提前收手"**：文档 QA 里，"MemGPT will often **stop paging through retriever results before exhausting** the retriever database"（§3.2.1）——即模型的**自主决策会失误**，该多翻几页时它自己停了。这恰恰是"让模型当内核"的代价（§3 的 Why 已埋下伏笔）。
- **底座与 harness 的交互不单调**：GPT-4 Turbo 版嵌套 KV 反不如 GPT-4 版（§3.2.2），说明"配置级"效应复杂，不能简单认为"底座越新越好"。

**我的独立补充批判**：
- **没有直接对比"长上下文模型"**。实验里 MemGPT 打的是"固定短窗口 + 检索/摘要"的基线，但论文动机里点名的**替代 A（把窗口做大）**并没有被拉进来做**端到端对照**——"MemGPT vs 一个 128k/200k 长窗口模型直接读全文"这个最该做的头对头，原文未给出。这让"OS 式分页 > 长上下文"这个隐含主张缺一块实证。
- **评测都在 2023 的短窗口模型上**。Table 1 的窗口尺寸（2k–8k 为主）是 2024/01 的快照；随着底座窗口暴涨到 100k–1M，MemGPT 相对"直接塞全文"的**边际价值可能收窄**——这正是 §13 regime 之辩的要害。
- **记忆的"垃圾回收"缺位**。论文讲了怎么 page in / page out、怎么摘要，但对"working context / archival storage 里**过期或错误的记忆如何被清理/纠正**"着墨少（Fig 4 展示了 `replace` 改一条事实，但没有系统的一致性维护/去重机制）。长跑之下，记忆库会不会积累矛盾条目？**原文未给出**系统性方案。这是后续 A-MEM 等工作要补的洞（见 Inspires-Us）。

## §11　相关工作与坐标：MemGPT 站在哪

> 主讲提示：这页给它在文献里的经纬度。论文 §4 自己把家谱画得很清楚，我按"它继承谁 / 区别于谁 / 谁在它之后"三问整理。

**它继承 / 区别于谁（§4）**：
- **长上下文 LLM 一支**（稀疏注意力 Child/Beltagy、低秩 Wang、神经记忆 Lee、位置插值 Chen、ALiBi Press…）：这些是**从模型内部**扩窗口。MemGPT 明确说自己**建立在这些进步之上**，但**核心贡献正交**——它是"用一个长上下文 LLM 当**主存的实现**"的**分层记忆**，管的是主存**之外**（§4 原文 "Our main contribution is a hierarchical tiered memory that uses a long-context LLM as the implementation of main memory"）。**这句是理解 MemGPT 定位的关键：它不与长上下文竞争，而是把长上下文当作自己的一层零件。**
- **检索增强模型一支**（RAG / REALM / RETRO / DPR / FLARE / IRCoT…）：MemGPT 的外部记忆**建立在 RAG 的传统上**，但不同于一次性检索——尤其点名 **FLARE（Jiang et al.）**"主动决定何时检索、检索什么"与自己相近（§4）。区别在 MemGPT 把这套"主动检索"嵌进了**OS 式的分层内存 + 自管理控制流**里。
- **LLM-as-agent 一支**（Park et al. 生成式智能体、WebGPT 的分页、ReAct 的"边想边做"、AgentBench…）：MemGPT 说自己与 ReAct 一样能"plan out loud"，但**聚焦点不同**——"our work focuses on tackling the problem of equipping agents with **long-term memory of user inputs**"（§4 原文）。即在"给 agent 装长期记忆"这条线上定位自己。

**谁在它之后长肉（Θ4，超出原文、我补充版图）**：MemGPT 提出的"自管理分层记忆 + 记忆函数"成了**记忆型 agent 的事实模板**。它的开源实现演进为 **Letta**；后续 D 组工作（如 **Mem0** 的分层记忆抽取/巩固、**A-MEM** 借鉴 Zettelkasten 让记忆自组织成网）都可视作在 MemGPT 骨架上**继续深挖"怎么让记忆更结构化、更自我组织、更可纠错"**——补的正是 §10 里"记忆垃圾回收缺位"那个洞。

## §12　把 OS 类比再收一次口：为什么这个隐喻"好用"

> 主讲提示：这是判断力的一页。别把 OS 类比当花哨的比喻——它是一套**可迁移的工程范式**，我逐条说清它"迁移过来"带来了什么。

MemGPT 把 OS 的三样东西整体搬了过来，每样都对应一个具体工程收益：

1. **分层存储（memory hierarchy）→ 无界容量的错觉**：正如 OS 用"内存慢盘快盘"层级制造"内存比实际大"的错觉，MemGPT 用"主上下文/recall/archival"层级制造"上下文无界"的错觉。收益：**固定窗口的模型也能处理任意长的历史/文档**（§2、§5）。
2. **中断（interrupt）→ 让系统对资源有"自省"**：内存压力警告让模型**知道自己快满了**并主动应对（§2.2、§6）。收益：把"资源耗尽"从"无声崩溃/截断"变成"**可感知、可提前处置**的事件"。
3. **事件驱动控制流 + 函数链（syscall/中断驱动的调度）→ 多步自主**：靠 `request_heartbeat` 把单步搬运串成多跳（§2.4、§7）。收益：**多跳检索/多文档汇总这类需要"连续搬好几趟"的任务成为可能**。

> **读出什么（Θ2 收口）**：把这三条合起来看，MemGPT 其实是在论证一件更大的事——**"能力"很多时候不在模型里，而在'把模型组织起来的那套系统抽象'里**。OS 用了半个世纪的分层/中断/调度，被证明**能整体迁移到 LLM 之外的编排层**并释放新能力。这就是 `Agent = Model + Harness` 在**记忆维度**的最强论证：把 GPT-4 从 32% 抬到 92%（§8.1）的，不是更聪明的权重，而是**一套更聪明的"内存操作系统"**。

## §13　regime 诚实：MemGPT 会被"更大的窗口/更少的上下文"淘汰吗？（Θ5）

> 主讲提示：这页是本篇最需要判断力、也最容易讲错的地方。**不要把"自管理记忆 > 一切"讲成绝对真理。** MemGPT 是 2023 的 canon，要诚实评估它在 2026 语境下的适用边界。

MemGPT 的隐含主张是"**与其把窗口做大，不如让模型自己管理一个分层记忆**"。这个主张**分 regime 成立**，两侧都要讲：

**支持 MemGPT 的 regime（它确实赢）**：
- **信息量远超任何窗口**（跨年对话、百万 token 年报、需跨多篇长文档连线）：无论窗口多大都装不下时，"外挂磁盘 + 分页"是**结构性刚需**（§3.2 的 SEC 10-K 例子）。
- **需要持久、可纠错的长期状态**（人设一致、用户偏好跨会话保持）：这类"跨越单次上下文生命周期"的记忆，长窗口天生给不了——窗口一清就没了；必须落到 external storage（§8.1）。
- **多跳、需迭代检索**（嵌套 KV、多文档汇总）：一次性把全部塞进窗口也未必答得对（Fig 7：KV 全在 8k 窗口里，基线照样多跳失败），"能持续发起多步查询"才是关键（§9.2）。

**削弱 MemGPT 的 regime（反方，连接 D 组"Less-Context"一侧）**：
- **底座窗口暴涨**：论文写作时主流 2k–8k（Table 1），如今动辄 100k–1M。当"直接把全文塞进去"变得可行且便宜时，MemGPT 那套"分页开销 + 模型可能提前收手（§10）"的**成本/风险**就未必划算——**"OS 式分页 > 长上下文"这个头对头，MemGPT 原文并未直接给出（§10 批判）**，随窗口变大更需重估。
- **"上下文并非越多越好"（D 组 Less-Context 反方）**：库内另一侧论点是——**给模型更少、更精的上下文往往比塞满更好**（噪声上下文会干扰推理、"Lost in the Middle"式的利用坍塌）。这和 MemGPT 其实**同源但殊途**：两者都承认"塞满窗口是坏主意"，但 MemGPT 的药方是"**用分层记忆把不相关的换出去**"，Less-Context 的药方是"**一开始就别放那么多进来**"。**有意思的是，MemGPT 的 page-out（把不相关信息换出主上下文）本身就是一种"主动做减法、保持窗口精简"的机制**——所以它并不与 Less-Context 对立，反而可看作"**用自管理去实现 Less-Context**"的一条路径。
- **自主决策的可靠性存疑**：让模型当内核，好处是懂语义，坏处是**会失误**（§10 的"提前停止翻页"）。在"决策失误代价高"的场景，一套**更保守的外部策略**可能反而更稳——这与 Θ5 的核心一致：**harness 的哪套机制主导，取决于 regime**。

> **读出什么（Θ5 收口）**：诚实的表述是——**MemGPT 定义的"自管理分层记忆"是记忆维度的一个奠基范式，而非终局答案。** 它在"信息量爆表 + 需持久状态 + 多跳"时几乎是刚需；在"窗口够大 + 单轮短任务 + 决策失误代价高"时，其收益会被长上下文或更保守的策略稀释。把它和 D 组 Less-Context 反方并读，得到的不是"谁对谁错"，而是一张**"什么时候该给模型更多记忆、什么时候该给它更少上下文"的决策地图**。

## §14　实现与实验细节（便于复现的一页）

- **底座模型（§3 Implementation details）**：'GPT-4 Turbo' 指 `gpt-4-1106-preview`（128k 窗口）、'GPT-4' 指 `gpt-4-0613`（8,192）、'GPT-3.5 Turbo' 指 `gpt-3.5-turbo-1106`（16,385）。实验对 GPT-4 / GPT-4 Turbo / GPT-3.5 都跑 MemGPT，以观察"底座模型质量如何影响 MemGPT"。
- **对话数据**：Multi-Session Chat（MSC，Xu et al. 2021），5 段会话/人、每段约十几条消息；作者新增第 6 段做一次性 QA。**公开释放**了增强版 MSC、嵌套 KV 数据集、以及 **20M 维基百科文章的 embedding**（§3）。
- **文档 QA**：NaturalQuestions-Open；采 50 题子集；用**LLM judge** 判"答案是否确实从检索文档推出（避免非精确串匹配误判）"（§3.2.1）。
- **嵌套 KV**：140 对 UUID（约 8k token）、嵌套 0–4 层、30 种排列（§3.2.2）。
- **存储实现**：archival storage = PostgreSQL + **pgvector** 扩展 + HNSW 近似索引，向量检索亚秒级（§3.2.1）。
- **提示/指令**：附录 6.1 给了 DMR/文档 QA 的**完整 persona 指令、baseline 系统提示、LLM judge 提示、以及 DMR 数据自动生成提示**（6.1.1–6.1.5，作者注明"prompts have been edited for brevity"，完整版见项目主页）。

> **读出什么**：MemGPT 是一份**工程可复现度很高**的 canon——存储栈（pg + pgvector + HNSW）、数据、prompt 都开源了。这也是它能迅速成为社区模板、演进为 Letta 的原因之一。

## §15　局限与批判（汇总，论文 §3/§5 + 我的补充）

论文侧（诚实）：
- **上限受底座函数调用能力锁死**（GPT-3.5 版在文档 QA、嵌套 KV 明显退化，§3.2）。
- **自主决策会失误**（"提前停止翻页"，§3.2.1）——"让模型当内核"的固有代价。
- **配置级效应复杂**（GPT-4 Turbo 版嵌套 KV 反不如 GPT-4 版，§3.2.2），不能简单外推"底座越新越好"。

我的补充（超出原文）：
- **缺"MemGPT vs 长上下文模型"的头对头**（§10）——最该做的对照原文未给出。
- **记忆一致性/垃圾回收缺系统方案**（§10）——长跑下记忆库的矛盾/过期条目如何清理，原文未给出；这是 A-MEM 等后续工作要补的。
- **评测锚在 2023 短窗口**——随窗口暴涨，其相对"直接塞全文"的边际价值需重估（§13）。
- **"OS 隐喻"的边界**：OS 的分页是**确定性、由硬件/内核保证**的；MemGPT 的"分页"是**概率性、由 LLM 决策**的。把 OS 的可靠性直觉直接套到"模型当内核"上要小心——**一个会幻觉、会提前收手的内核，不是传统意义上可靠的内核**。这是隐喻最容易误导人的地方，需在工程上用"确定性兜底 + 中断催促"去补（正是 §6 两级阈值在做的事）。

## ★ 对我们的启发（Inspires Us）

> 这一节是组会高潮。**我们自己就活在一个 harness 里**：Claude Code / 本课 m9.* 的 agent 有真实的 **compaction（上下文压缩）+ 记忆文件（MEMORY.md 等）+ ReAct 循环 + 子代理编排**。
> 而 **MemGPT 讲的"主上下文↔外部存储的分页"，就是我们这套"活动上下文↔记忆文件"的理论原型**——只不过我们的换入/换出目前多由**外部规则**触发，MemGPT 让它由**模型自主**触发。下面每条都往我们自己身上打。

➤ **a. 可直接借用的招（method we can reuse）**：**"内存压力中断 + 两级阈值"（§6）** 是最该整体搬过来的机制。我们的 compaction 现在多是"到 token 阈值就自动压缩"，模型往往**被动**（压缩由外部触发，模型不一定知道自己即将被压、更没机会决定"保住什么"）。照 MemGPT 的做法：**在压缩前（如 70% 处）先给模型注入一条系统消息**——"你即将被 compaction，请把当前任务里最关键的状态/决策/未完成项，主动写进记忆文件"，到 100% 才真正压缩。这把"要保住什么"的决策权，交还给最懂当前任务的一方。**具体机制：一个 `warning_token_count` 钩子 → 注入"page-out 提示" → 观察模型是否主动落盘关键状态。**

➤ **b. 可迁移到我们模块的思路（transfer）**：把 MemGPT 的**三段式主上下文（只读系统指令 / 可经函数改写的 working context / 滚动 FIFO）**映射到我们的上下文布局——我们其实已有雏形（只读系统提示 / 可写记忆文件 / 滚动对话历史），但**缺"working context"这一段被结构化管理**。迁移动作：在我们某个 m9.* agent 里，显式开一块**固定大小、只能经工具改写的"工作便签"**（存当前任务目标、关键约束、已确认事实），并规定它**只能经工具 replace/append**（对照 Fig 4）。前提改变：这要求我们的记忆写入**全部走工具接口**（可追踪），而非模型在自由文本里随手记——这本身就是一次可观测性（O 层）的升级。

➤ **c. 它暴露的开放问题 = 我们的机会（open problem → opportunity）**：MemGPT **没解决"记忆的一致性维护 / 垃圾回收"**（§10、§15）——它能 page in/out、能摘要，但对"过期/矛盾记忆如何被清理纠正"没有系统方案。这正是我们的机会：给我们的记忆文件加一个**"记忆一致性巡检"子过程**——定期扫描记忆库，检测矛盾条目（如"任务已完成"与"任务进行中"并存）、过期条目，并触发 `replace`/删除。**可下手的第一步：在一个长跑 agent 上，给记忆写入打时间戳 + 来源，跑一个最小的"矛盾检测器"（同一实体的互斥断言），量化它能否减少"记忆自相矛盾导致的行为漂移"。**

➤ **d. 与本库其它论文/模块的连接（connect the dots）**：
- **与 D 组"Less-Context / 更少上下文"反方**：不是对立而是互补——MemGPT 的 **page-out（把不相关信息换出主上下文）本身就是"主动做减法"**，可看作"用自管理去实现 Less-Context"（详见 §13）。
- **与 HarnessBench（本库 G 组标杆）**：MemGPT 提供的是"同模型换记忆层，DMR 32%→92%"这一记**记忆维度**的 `Agent=Model+Harness` 证据，正好补进 HarnessBench 那张"换 harness 摆 23.8 分"的大图里。
- **与 F 组上下文折叠/状态重建（AgentFold / IterResearch 一类）**：它们攻的是 HarnessBench 里"state/continuation"那条失败线，与 MemGPT 的"递归摘要 + 分页召回"是**同一战场的不同兵种**——一个重"折叠压缩"，一个重"分层召回"，可组合。
- **与 auto-research 库的 m9.* 记忆实践**：我们此前在 auto-research 里已经用过"记忆文件当外部存储"，MemGPT 给这套实践补上了**理论命名（page in/out、内存压力中断）**和**自主触发**这半边。

➤ **e. 如果我来做下一步（my next move，第一人称·可执行）**：**我会先在我们某个 m9.* 的长跑 agent 上，把 compaction 从"静默自动压缩"改成"MemGPT 式两级：70% 处注入 page-out 提示、100% 才压缩"，跑 10 个长任务，对照"改造前 vs 改造后"——量一个指标：**compaction 之后，agent 是否还记得压缩前定下的关键约束/未完成项**（可用一组"压缩后追问关键状态"的探针题打分）。如果 page-out 提示确实让"压缩后失忆率"显著下降，就把它固化成我们 harness 的默认 compaction 策略，并接着做 (c) 的记忆一致性巡检。**一句话：把 MemGPT 的"让模型自己决定换出什么"这半步，真正接到我们自己的 compaction 上。**

## §16　版图定位（canon 坐标 + 在本库的位置）

- **时间坐标（Θ4）**：**canon（2023-10 初版 / 2024-02 更新）**。它**定义了**一整套此后被反复沿用的语言与范式——"LLM as OS"、"virtual context management"、"主上下文↔外部存储的分页"、"内存压力中断"、"自导记忆函数"。**后续谁在它上面长肉**：其开源实现演进为 **Letta**；D 组后续的 **Mem0（分层记忆抽取/巩固）**、**A-MEM（Zettelkasten 式自组织记忆）** 等都在 MemGPT 骨架上继续深挖"记忆更结构化、更自组织、更可纠错"，补的正是它 §10 留下的"记忆垃圾回收"洞。它相对更早的 **RAG**（一次性检索）推进了"**可迭代 + 自主决策**"；相对**长上下文**一支，它不竞争而是"把长上下文当自己的主存实现"（§4）。
- **E/T/C/L/O/V 归属（Θ1）**：本篇坐 **C（Context/记忆）层**为主，强依赖 **T（函数调用）** 与 **L（事件驱动控制流 + 函数链）**；对应库内 **D 组**（上下文/记忆）。
- **回扣全库命题（Θ2）**：MemGPT 是 `Agent = Model + Harness` 在**记忆维度**的经典诠释与最锋利证据之一——**同一 GPT-4，换上"分页召回"的记忆层，DMR 从 32.1% 到 92.5%（Table 2）**。它把"记忆管理"从"外部固定策略"升级为"模型自己求解的决策问题"，是理解"harness 如何塑造能力"的一块基石。
- **在本库的位置**：**D 组 ⭐ canon 压舱石**。读完它，再看 D 组任何一篇"给 agent 装记忆"的论文，都能问一句："它在 MemGPT 的哪个部件上动了刀——是把'分页策略'做得更聪明（谁 page in/out）、把'外部存储'做得更结构化（怎么存/组织）、还是把'记忆纠错'补上（怎么清理矛盾）？"

## §17　组会讨论问题（留给大家吵）

1. MemGPT 让**模型自己当内核**决定 page in/out——可模型会"提前收手"（§10）。到底该给模型多大自主权？"模型决策 + 确定性兜底"的最佳分界在哪？
2. 随着底座窗口涨到 100k–1M，"OS 式分页"相对"直接塞全文"的边际价值还剩多少？在哪些任务（多跳？持久状态？）它**不可替代**，哪些任务它**被淘汰**？（呼应 §13）
3. MemGPT 的 **page-out** 与 D 组 **Less-Context** 反方是"同一枚硬币两面"吗？如果"主动换出不相关信息"就是"保持窗口精简"，那"自管理记忆"和"少给上下文"是不是本可以统一成一个目标？
4. 我们要把 (a) 的"内存压力中断"接到自己 harness 的 compaction 上——最小可行方案是什么？用什么探针题量化"压缩后失忆率"？
5. MemGPT 没解决"记忆垃圾回收"（§10）。一个能自我纠错/去矛盾的记忆库，最小实现长什么样？谁来判定两条记忆"矛盾"——规则、还是又一个 LLM judge（那"谁来 judge the judge"）？

## §18　一页速记

- **命题**：把 LLM 上下文窗口当 **RAM**、外部数据库当 **disk**，用 OS 式**分页**在两者间搬信息，让**固定窗口**营造**无界上下文**（§1–2）。
- **内核创新**：**让 LLM 自己用函数调用决定何时换入/换出**——把"记忆管理"从外部固定策略升级为**模型自求解的决策问题**（§3）。
- **架构**：主上下文三段（只读系统指令 / 可写 working context / 滚动 FIFO）↔ 两块磁盘（recall 存全量消息 / archival 存任意文本）；Queue Manager + Function Executor 当搬运工（§4–5，Fig 3）。
- **中断**：**内存压力两级阈值**——70% 警告（催模型主动保存）、100% 冲刷（转存 + 递归摘要，不丢弃）（§6）。
- **多步**：**事件驱动 + 函数链（`request_heartbeat`）**把单步搬运串成多跳检索（§7）。
- **铁证（Θ2）**：**同一 GPT-4，+MemGPT 后 DMR 32.1%→92.5%**（Table 2）；文档 QA 顶穿检索器天花板（Fig 5）；嵌套 KV 唯一能稳定多跳（Fig 7）。
- **诚实（Θ5）**：上限受**函数调用能力**锁死、模型会**提前收手**、缺"vs 长上下文"头对头、缺记忆垃圾回收；"自管理记忆 > 一切"**分 regime**，与 D 组 Less-Context 反方**同源殊途**（§13）。
- **坐标（Θ4）**：**canon（2023，UC Berkeley）**，定义了 LLM-as-OS / 自管理记忆范式，演进为 Letta，Mem0 / A-MEM 在其上长肉。
- **对我们（Θ3）**：我们的 **compaction + 记忆文件就是 MemGPT 分页的现实版**；下一步——把 compaction 改成"**70% 注入 page-out 提示、100% 才压缩**"，量"压缩后失忆率"是否下降。
