# FlowSteer：用强化学习让智能体自己设计智能体工作流（渐进式画布编辑）

> **本篇属 agent-harness 库 B 组（控制循环 · L 层）**。它示范一件很"打在我们自己身上"的事：
> 我们（Claude Code / 本课 m9.* 的 agent）的控制循环——什么时候 plan、什么时候 verify、什么时候收口——
> 目前**大多是人手写死的**。FlowSteer 问的正是这个：**这套控制流能不能不写、而是用 RL 学出来？**
> 全文按 Θ 规范：二级标题=幻灯片 + `> 主讲提示`；每个公式先直觉→定义符号→"读出什么"；区分宣称 vs 批判。

## §1　TL;DR（一页讲清这篇在干嘛）

> 主讲提示：开场先把全库中心命题 `Agent = Model + Harness` 摆上，再说这篇是"把 harness 的 L 层从手写升级为可学习"的一次直接尝试。

一句话：**把"设计一条 agent 工作流"本身变成一个强化学习问题**——不再靠人拖拽节点、写规则模板、或一次性 prompt 生成图，而是训练一个**轻量策略 agent（Flow-Director）**，让它像画家改画一样，**一次提交一个原子编辑**（加一个算子 / 删 / 改 / 配 prompt / 加控制结构 / 收口），每一步都拿到**图级（graph-level）的、经过语法检查的执行反馈**，据此决定下一步怎么改。这个"边改边看反馈"的循环叫 **Reinforced Progressive Canvas Editing（强化式渐进画布编辑）**，用改造过的 GRPO 端到端训练。

- **属于 harness 的哪一层（Θ1）**：本篇正打 **L（Loop / 控制循环）** 层——它的产物就是一条**控制流**（先 plan → 再 solve → 再 verify → 再 format → finish）。区别在于：传统 harness 的这条流是**工程师写死的**；FlowSteer 的这条流是**策略在环境反馈里学出来、且逐步搭出来的**。它对 **T（Tools，算子库当工具）**、**C（Context，交互历史当状态）**、**V（Validation，画布语法检查 + Verify 算子）** 都有强依赖。
- **回扣全库论点（Θ2）**：全库命题是 `Agent = Model + Harness`（§3 的问题形式化里，执行后端 $\mathcal{M}_{\text{exec}}$ 与策略 $\pi_\theta$ 显式分离，正是 Model 与 Harness 的分离）。FlowSteer 是这条命题的**延伸**：如果 harness（这里=编排控制流）如此重要，那**"能不能把 harness 学出来、而不是写出来"**就是下一个自然问题。它给的答案是"能，而且在同后端下能把弱模型的分数抬上去"（§5.2，Table 3：GPT-4o-mini 后端上 IID 平均 +22.65 EM）。
- **够新够权威（Θ4）**：2026-05 预印本，CUHK-Shenzhen / NTU / NUS 出品。它是**首批**把"工作流编排"设成**多轮 RL 决策问题、并用图级执行反馈闭环**的工作之一（§2 把自己相对 AFlow/GPTSwarm/静态选择/SFT 的位置画得很清楚，见 Figure 2 的四范式对比）。

**三条带走的结论**：
1. **命题**：`最大化 J(θ)=E[R(τ)]`，其中 τ 是一条"编辑轨迹"、R 只在**结构合格后**才放开答案奖励——这一"条件放行（conditional release）"是防"抄近道直接出答案"的关键（Eq. 40）。
2. **机制**：状态=当前工作流画布 + 交互历史；动作=8 类原子编辑（add/delete/modify/set_prompt/finish + parallel/conditional/loop）；奖励=结构多样性(4 项检查) × 答案正确性，闸门式。用 **canvas-masked GRPO（CWRPO）** 训练，token 级 mask 只对**策略生成的 token** 回传梯度、屏蔽环境反馈 token（Eq. 12）。
3. **证据**：12 个数据集（6 IID + 6 OOD）、6 个后端。GPT-4o-mini 后端上 IID 平均 EM/F1/Acc 全面超 GPT-4o-mini 与 AFlow/Router-R1/Orchestrator 等 baseline（Table 3）；且**训练后 token 更省、交互轮数更少**（Figure 5c/d）。

## §2　问题与动机：为什么"编排"值得用 RL 来学，而不是继续手写

> 主讲提示：这页用 Why 三连的"问题层"。核心是三个卡点——人依赖、缺图级反馈、长程不可修复。把它们讲成"手写控制流的三宗罪"。

**Why（问题层）——不解决会卡住什么？**
agentic workflow（把算子组织成一张可执行的工作流图，用来解复杂任务）已经很普及，好处是可控、可调试、可复用（§1，引 Zeng 2023 / Qian 2024）。但**图是怎么来的**这件事，现实里主要靠三种方式，每种都有硬伤（§1 + §2 归纳为三个 challenge）：

1. **(i) 人依赖的构造（Human-dependent construction）**：工作流图主要靠**人手拖拽、规则模板、或一次性 LLM prompt** 产生（引 Zhang 2024 / Wang 2024）。换新任务、新算子库、新后端、新领域，就得**重新维护**——"严重限制可扩展性与泛化"（§2 原文 "severely limits scalability and generalization"）。
2. **(ii) 缺图级执行反馈（Lack of graph-level execution feedback）**：现有 agent 环境大多工作在**文本生成或外部工具调用**的粒度（引 Yao 2022 / Schick 2023），**没有一个能对"工作流图本身"给出结构化反馈的可执行抽象**。于是策略**无法在构造过程中诊断/修复结构错误**——它看不到"我刚加的这条边让图坏了没有"。
3. **(iii) 长程构造中不能在环修复（Inability to repair errors in-loop）**：一次性生成、或靠外部搜索来搭图（引 Yu 2025 / Guo 2025），**没法在几十步编辑里边搭边改**。后果是**学习信号稀疏、延迟**，诱发**抄近道行为**（例如过早 finish、把图简化到只剩一步），以及**长程 credit assignment 不稳**。

**后果**：我们缺一个范式，让**一个 agent 端到端地把工作流设计出来**、并在**可检查、可修复**的粒度上迭代。

> **读出什么**：这篇的野心不是"再造一个更强的执行 agent"，而是**把"设计 agent 的那个环节"自动化**——作者称之为 *Agent Designing Agentic Workflows*（智能体设计智能体工作流）。这正是 Θ2 里"把 harness 从手写升级为可学习"的字面实现。

## §3　四种编排范式对比：FlowSteer 站在谁的肩上、跟谁不同（相关工作定位）

> 主讲提示：这页对着 Figure 2 讲。四张小图=四代范式，FlowSteer 是第四代"交互式编排"。强调前三代的共同盲区是"没有 in-loop 的图级反馈闭环"。

论文 Figure 2 把编排范式分成四类，FlowSteer 属第四类：

| 范式 | 代表 | 工作方式 | 硬伤（本文视角） |
|---|---|---|---|
| **(a) 静态工作流选择** | Wang 2024 | 按任务相似度，从**预定义库**里检索一条现成工作流 | 库要人建、覆盖不了新任务；不产生新结构 |
| **(b) 离线工作流生成** | AFlow(Zhang 2024) 的 SFT/GRPO 变体 | 用 SFT 或 GRPO **一次性生成**整张工作流再执行 | 一次成图，**执行反馈不进环**；错了只能整条重来 |
| **(c) 自动工作流优化** | GPTSwarm(Zhuge 2024) / RobustFlow(Xu 2025) / LATS(Zhou 2023) | 在**搜索空间**里用搜索 + 执行反馈迭代改图 | 靠**外部搜索**（如 MCTS），非端到端学习；随图复杂度算力爆炸 |
| **(d) 交互式编排（本文 FlowSteer）** | — | 一个 agent 与**可执行画布**多轮交互，一次一原子编辑，图级反馈驱动，端到端 RL | 代价：要设计画布环境 + 奖励 + 承担训练成本 |

**Why（设计层）——为什么必须是 (d) 而不是把 (b)/(c) 调好就行？**
> **Why（设计层）**：朴素做法一是 (b) **一次性生成整图**（SFT/GRPO 直接吐一张图）→ 会因为"图错了没有中途反馈、只能事后从最终答案拿稀疏信号"而**长程 credit assignment 崩**、诱发抄近道（§2 challenge iii）。朴素做法二是 (c) **外部搜索**（MCTS 在算子组合里搜）→ 能用执行反馈，但**不是端到端学习**、无法把"跨问题的编排经验"沉淀进一个策略，且**随图规模算力上升**（§E.3 对 AFlow 的评述）。FlowSteer 改用 (d)：把编排**建模成一个 MDP**，让策略在**每一步原子编辑后**就拿到图级反馈——既有反馈闭环（补 c 的端到端短板），又把长程构造**拆成一串可检查的局部决策**（补 b 的稀疏信号短板）。

> **读出什么（Θ4 增量）**：本篇相对本库已有的 canon（ReAct 定义了"推理+行动"的单步循环）与前沿（AFlow 做离线图生成、GPTSwarm 做可优化图）的**增量**是：把"**图的构造过程**"本身变成**被 RL 优化的多轮策略**，且反馈粒度提升到**graph-level**。

## §4　把问题形式化：状态=画布/工作流，动作=编辑，奖励=结构×正确（核心）

> 主讲提示：这是全篇的"地基页"。务必先给直觉，再逐个定义符号，最后读出结论。把 RL 三要素（S/A/R）一次讲透。

### 4.1　三个基础定义（§3 Preliminaries）

**直觉**：先把"工作流"和"一次编排"这两个对象钉死，后面所有 RL 记号才有依托。

**定义 1：工作流图（Workflow Graph）**。一张有向无环图（DAG）
$$\mathcal{G} = (V, E, \text{attr})$$
符号（先定义后用）：
- $V=\{v_1,\dots,v_n\}$：$n$ 个**算子节点**（operator node）；
- $E \subseteq V\times V$：编码**数据依赖与执行顺序**的有向边；
- $\text{attr}(v)=(\text{op}(v),\ \text{param}(v),\ \text{prompt}(v))$：每个节点 $v$ 的属性——它是哪个算子类型 $\text{op}(v)\in\mathcal{O}$、参数配置、以及**执行 prompt**。

**定义 2：编排轨迹（Orchestration Trajectory）**。一条**唯一决定一张工作流图**的、长度为 $T$ 的交互序列：
$$\tau = \{(a_t^{\text{think}},\ a_t,\ o_t^{\text{exec}})\}_{t=1}^{T}\ \Rightarrow\ \mathcal{G}_\tau \tag{1}$$
符号：
- $T$：终止前的总交互步数；
- $a_t^{\text{think}}$：第 $t$ 步的**推理反思**（reflection，先想再做）；
- $a_t=(\alpha_t,\ a_t^{\text{out}})$：第 $t$ 步的**编辑动作**，含**动作类型** $\alpha_t\in\mathcal{A}_{\text{type}}$ 与**内容** $a_t^{\text{out}}$；
- $o_t^{\text{exec}}$：画布环境返回的**执行反馈**。

> **读出什么**：定义 2 是全篇的枢纽——它把"一张图"重新表述成"一串带反馈的编辑动作"。这一步**转写**（graph → editing sequence）正是让 RL 能上手的关键：MDP 要的是"状态-动作-反馈"的序列，而不是一张静态图。

**问题陈述（Problem Statement）**。给定任务分布 $\mathcal{D}_Q$ 里的任务 $q$、算子库 $\mathcal{O}$、以及**可插拔的后端 LLM** $\mathcal{M}_{\text{exec}}$，目标是学一个策略 $\pi_\theta$ 生成轨迹 $\tau$，对应工作流 $\mathcal{G}_\tau$ 被执行产出答案：
$$y_q = \text{Execute}(\mathcal{G}_\tau,\ q,\ \mathcal{M}_{\text{exec}}) \tag{2}$$
总学习目标（这就是全篇要最大化的东西）：
$$\max_\theta\ \mathcal{J}(\theta)=\mathbb{E}_{q\sim\mathcal{D}_Q}\ \mathbb{E}_{\tau\sim P_\theta(\cdot|q)}\big[R(\tau)\big] \tag{3}$$
符号：$P_\theta(\tau\mid q)$ 是策略诱导的轨迹分布；$R(\tau)$ 是**轨迹级奖励**，同时衡量**结构质量**与**答案正确性**（§4.3 展开）。

> **读出什么（Θ2 关键）**：Eq. 2 里 $\pi_\theta$（设计者/harness）与 $\mathcal{M}_{\text{exec}}$（执行者/model）**显式解耦**。训练时**只优化 $\pi_\theta$，$\mathcal{M}_{\text{exec}}$ 固定**。这正是 `Agent = Model + Harness` 的可操作版本：**把 harness 那一半拿出来单独学**。

### 4.2　RL 三要素落地（§4.1 的 Canvas / Action / State）

**① 状态空间 $\mathcal{S}$（State）**：agent 状态 $H_t$ 由**完整交互历史**定义。初始状态是任务 $q$、算子库描述 $d^{\text{lib}}$、模板 $a^{\text{tmpl}}$ 的拼接：$H_0=[q\oplus d^{\text{lib}}\oplus a^{\text{tmpl}}]$（$\oplus$=序列拼接）。状态转移：
$$H_t = H_{t-1}\oplus(a_t^{\text{think}},\ a_t,\ o_t^{\text{exec}}) \tag{5}$$
> **读出什么**：状态**不是"当前那张图"本身**，而是"到目前为止想过什么、改过什么、画布回了什么"的**全历史**。图 $\mathcal{G}_t$ 是历史的一个函数。这让策略能基于"反馈—反应"的因果链决策，而非只看静态快照。

**② 动作空间 $\mathcal{A}$（Action）**：策略每步先出反思 $a_t^{\text{think}}$，再出编辑动作 $a_t=(\alpha_t,a_t^{\text{out}})$，$\alpha_t\in\mathcal{A}_{\text{type}}$。**8 类动作类型**（Table 1 / Table 6 / Eq. 15）：

| 动作 $\alpha$ | 作用 | 对图的更新 |
|---|---|---|
| `add` | 加一个算子 | $V_t\leftarrow V_{t-1}\cup\{v\}$ |
| `delete` | 删一个节点 | $V_t\leftarrow V_{t-1}\setminus\{v\}$ |
| `modify` | 改算子类型 | $\text{op}(v)\leftarrow o'$ |
| `set_prompt` | 配某节点的 prompt | $\text{prompt}(v)\leftarrow p$ |
| `finish` | 收口/终止 | 触发执行得答案 $y_q$ |
| `parallel` | 加并行分支 | 分支合并进 $E_t$ |
| `conditional` | 加条件分支 | 分支合并进 $E_t$ |
| `loop` | 加循环结构 | back-edge 合并进 $E_t$ |

前 5 类是**局部编辑**，后 3 类专门**生成长程控制结构**（分支/条件/循环）。

**③ 算子库 $\mathcal{O}$（当"工具"用，T 层）**：12 个功能算子，分 6 类（Table 1 / Table 5）——**Planning**(Plan, Decompose)、**Solving**(Programmer, Custom, AnswerGenerate)、**Verification**(Test, Review, Verify)、**Revision**(Revise)、**Ensemble**(ScEnsemble, Aggregate)、**Formatting**(Format)。**关键设计：算子库 plug-and-play**——策略只通过**标准化动作接口 + 结构化输出**与算子交互，所以算子可以自由增删换而不改框架（§4.1）。Programmer 算子在**沙箱**里跑 Python，30 秒超时（§A.2）。

**④ 画布环境 $\mathcal{C}$（Workflow Canvas）**：维护当前图状态 $\mathcal{G}_t$ 并在每步给可执行反馈：
$$\mathcal{C}=(\mathcal{G}_t,\ \mathcal{O},\ \mathcal{M}_{\text{exec}},\ d^{\text{lib}}) \tag{4}$$
画布内部是个**两状态有限状态机**（§A.3.1）——**BUILDING**（可 add/delete/modify/finish）与 **AWAITING_PROMPT**（加完算子后必须先 `set_prompt` 才能回 BUILDING）。

> **Why（设计层）——为什么要"加算子"和"配 prompt"分两步？**
> **Why（设计层）**：朴素做法是一步就把"加哪个算子 + 用什么 prompt"一起定。→ 对**小模型**来说，一次要同时做**结构决策**（选算子）和**内容决策**（写 prompt）认知负荷太高，容易两头都做不好。本文用**两步机制**（add 后强制进 AWAITING_PROMPT、再 set_prompt），把结构与内容**解耦**，"降低策略模型的认知负荷……提升小模型的有效性"（§A.3.1 原文）。这是"用环境状态机去规训策略"的巧劲——**harness 的状态机在替小模型分解任务**。

## §6　方法核心一：设计者–执行者解耦（Agent Designing Agentic Workflows）

> 主讲提示：这页回答"为什么一个小策略能'指挥'一个强后端、还能跨后端不重训"。这是本篇最有 harness 味的一节。

**Designer–Executor Decoupling（§4.2）**。给定 $q$，策略 $\pi_\theta$ 采样一条编排轨迹 $\tau$，解码成图 $\mathcal{G}_\tau$；答案由执行后端跑图得到：
$$y_q=\text{Execute}(\mathcal{G}_\tau,q,\mathcal{M}_{\text{exec}}),\quad \mathcal{G}_\tau=\text{Decode}(\tau),\quad \tau\sim\pi_\theta(\cdot\mid q) \tag{6}$$
符号：$\text{Decode}(\cdot)$ 把编辑轨迹**确定性地**回放成最终图（顺序施加每个原子编辑）；$\text{Execute}(\cdot)$ 在后端上跑图。**训练全程 $\pi_\theta$ 只负责产 $\tau$，$\mathcal{M}_{\text{exec}}$ 冻结**。

**对工作流设计的边缘化（Marginal Likelihood，Eq. 7）**：FlowSteer 诱导的答案分布是对"所有可能工作流设计"的边缘化——
$$P(y_q\mid q;\pi_\theta,\mathcal{M}_{\text{exec}})=\mathbb{E}_{\tau\sim\pi_\theta(\cdot|q)}\big[P_{\text{exec}}(y_q\mid\mathcal{G}_\tau,q,\mathcal{M}_{\text{exec}})\big] \tag{7}$$
> **读出什么**：决策的**单位是"一张工作流"**（$\tau$ 在算子组合层面），不是 token。因此 $\pi_\theta$ 学到的是**编排策略**，它**跨算子实现与后端 LLM 迁移**——这正是 §5.4（RQ3）在 6 个后端上都涨的理论依据。

**解耦后的梯度（Eq. 8）**：
$$\mathcal{J}(\theta)=\mathbb{E}_{q\sim\mathcal{D}_Q,\ \tau\sim\pi_\theta(\cdot|q)}[R(\tau)],\qquad \nabla_\theta\mathcal{J}(\theta)=\mathbb{E}_{q,\tau\sim\pi_\theta}[\nabla_\theta\log\pi_\theta(\tau\mid q)\cdot R(\tau)] \tag{8}$$
关键观察：$\mathcal{M}_{\text{exec}}$ **只通过它对 $R(\tau)$ 的标量贡献**（经由 $y_q$）影响梯度。因此**换后端、改算子库、或重写某算子的 prompt，都保持 Eq. 8 的形式不变**，只改"某张图的已实现奖励"。

> **Why（设计层）——为什么不干脆把执行后端也一起微调？**
> **Why（设计层）**：朴素做法是把整个 agent（策略 + 执行）一起 fine-tune，端到端最优。→ 会**绑死后端**：换个更强/更便宜的后端就得重训，且大后端微调贵、还可能损失其通用能力。本文选择**冻结后端、只学一个小策略**——好处是**一个小策略"放大"一个强后端**（§5.2 原文 "amplify a strong backend without backend fine-tuning"），且 Proposition 2 断言这种解耦让 $\pi_\theta$ **无需重训即可跨算子库与后端泛化**（§4.2）。代价：策略只能通过"选算子/配 prompt"间接影响结果，天花板受后端能力约束。

## §7　方法核心二：渐进式画布构造与分层编辑策略

> 主讲提示：这页讲"怎么一步步把图搭出来"。重点是"每步一原子编辑"和"编辑动作被分解成 类型+内容"两件事。

**渐进式图构造（Progressive Graph Construction，Eq. 9）**：从空图 $\mathcal{G}_0=(\emptyset,\emptyset,\emptyset)$ 起，$T$ 次原子编辑后图是累积复合——
$$\mathcal{G}_T=\mathcal{G}_0\oplus a_1\oplus a_2\oplus\cdots\oplus a_T,\qquad \oplus:\mathcal{G}\times\mathcal{A}\to\mathcal{G} \tag{9}$$
即"每回合提交一个原子编辑，把长程设计拆成一串**可检查、可修复**的局部决策"。

**分层编辑策略（Hierarchical Editing Policy，Eq. 10）**：每步 Flow-Director 把编辑动作**因式分解**成"反思 + 动作类型 + 结构化输出"，条件于历史 $H_{t-1}$：
$$
\log\pi_\theta(a_t^{\text{think}},a_t\mid H_{t-1})=
\begin{cases}
\log\pi_\theta(a_t^{\text{think}}\mid H_{t-1})+\log\pi_\theta(\alpha_t\mid a_t^{\text{think}},H_{t-1})+\log\pi_\theta(a_t^{\text{out}}\mid\alpha_t,a_t^{\text{think}},H_{t-1}), & \alpha_t\neq\texttt{finish}\\[4pt]
\log\pi_\theta(a_t^{\text{think}}\mid H_{t-1})+\log\pi_\theta(\texttt{finish}\mid a_t^{\text{think}},H_{t-1}), & \alpha_t=\texttt{finish}
\end{cases}\tag{10}
$$
符号：先采**反思** $a_t^{\text{think}}$，再采**动作类型** $\alpha_t$，非终止时还要采**内容** $a_t^{\text{out}}$；终止（finish）时只需类型。

> **读出什么**：这个分解与 §4.2 的"两步机制"呼应——策略的每次输出被**结构化**成"先想 → 定类型 → 填内容"，天然把"结构决策"与"内容决策"分层。这降低了小模型一次要拿的决定数，是"用 harness 结构降低策略负担"的又一处。

**画布反馈驱动的轨迹（Eq. 11）**：给定 $a_t$，画布执行并返回反馈 $o_t^{\text{exec}}$，形成"诊断—编辑—验证"闭环：(i) 画布做**语法解析 + 约束检查**，采样 $o_t^{\text{exec}}\sim\mathcal{C}_{\text{exec}}(\cdot\mid\mathcal{G}_{t-1},a_t)$，携带**成功状态、失败原因、修复建议**；(ii) 图更新 $\mathcal{G}_t=\text{Update}(\mathcal{G}_{t-1},a_t,o_t^{\text{exec}})$，状态按 Eq. 5 演进；(iii) 直到 $\alpha_t=\texttt{finish}$ 或 $t=T_{\max}$。整条轨迹分布：
$$P_\theta(\tau\mid q)=\underbrace{\pi_\theta(a_1^{\text{think}},a_1\mid H_0)}_{\text{第一步编辑}}\cdot\prod_{t=2}^{T}\underbrace{\big(\mathcal{C}_{\text{exec}}(o_{t-1}^{\text{exec}}\mid\mathcal{G}_{t-2},a_{t-1})\cdot\pi_\theta(a_t^{\text{think}},a_t\mid H_{t-1})\big)}_{\text{反馈条件下的后续编辑}} \tag{11}$$
其中**只有 $\pi_\theta$ 被优化**（$\mathcal{C}_{\text{exec}}$ 是环境，不含可学参数）。

> **读出什么（Θ1 L 层）**：Eq. 11 把"控制循环"写成了概率图——策略的每一步都**显式条件于上一步的画布反馈**。这就是把 ReAct 那种"think→act→observe"单步循环，**升级成一条可被 RL 端到端优化的、图级的编辑轨迹**。这是本篇打 L 层的核心式子。

## §8　方法核心三：奖励设计——"条件放行"防抄近道（本篇最该讲透的一节）

> 主讲提示：这是全场停留最久的公式页。奖励的**符号**和**为什么长这样**都要说清。核心洞察：先逼它把结构搭对，再放开答案奖励。

**直觉**：如果奖励只看"最终答案对不对"，策略会发现一条捷径——**跳过搭工作流、直接吐个答案**，照样可能蒙对，于是永远学不会编排。所以奖励要设计成："**结构不合格 → 无论答案对错都给负分；结构合格 → 才把答案奖励放出来**。"

**轨迹级奖励（Eq. 40，Appendix B.3 / C.1）**：
$$R(\tau)=-1+R_{\text{diversity}}(\tau)+\mathbb{I}\{R_{\text{diversity}}(\tau)=1\}\cdot R_{\text{answer}}(\tau) \tag{40}$$
符号（先定义后用）：
- $R_{\text{diversity}}(\tau)\in[0,1]$：**结构多样性奖励**，由 4 项结构检查加总、**上限截到 1**；
- $\mathbb{I}\{\cdot\}$：指示函数，**当且仅当结构满分（=1）时取 1**，否则 0——这就是"**条件放行（conditional release）**"闸门；
- $R_{\text{answer}}(\tau)\ge 0$：**答案奖励**，衡量最终执行输出与 ground truth 的匹配；
- 常数 $-1$：基线偏移，制造"结构不合格必为负"的符号缺口。

**$R_{\text{diversity}}$ 的 4 项检查（§C.1，各项命中才计分，合计封顶 1.0）**：
- $R_{\text{checker}}$：**含验证算子**（Test/Review/Verify 至少一个）才 fire；
- $R_{\text{format}}$：**末步是 Format 算子**（保证答案被规整）才 fire；
- $R_{\text{operator}}$：**至少 3 个不同算子**（结构不能退化成一步）才 fire；
- $R_{\text{control}}$：**含至少一个控制结构**（parallel/conditional/loop）才 fire。

**符号缺口的数学后果（§B.3，Eq. 43–46）**：设可行集 $\mathcal{F}=\{\tau:R_{\text{diversity}}(\tau)=1\}$。
- 对 $\tau\in\mathcal{F}^c$（结构不合格）：指示函数=0，$R(\tau)=-1+R_{\text{diversity}}(\tau)\in[-1,0)$——**严格为负**；
- 对 $\tau\in\mathcal{F}$（结构合格）：$R(\tau)=-1+1+R_{\text{answer}}(\tau)=R_{\text{answer}}(\tau)\ge 0$——**非负**。

于是产生**两阶段学习**（§B.3(ii)）：训练早期，绝大多数轨迹落在 $\mathcal{F}^c$，学习信号主要是"**别违反结构约束**"（把可行概率 $p_\theta=\mathbb{P}(\tau\in\mathcal{F})$ 推高）；随 $p_\theta$ 上升，越来越多轨迹进入 $\mathcal{F}$，信号**自然切换到"优化答案正确性"**。

> **Why（设计层）——为什么用"减 1 + 闸门"而不是简单加权和？**
> **Why（设计层）**：朴素做法是 `R = 0.5·结构 + 0.5·答案` 这种加权和。→ 会漏网："**结构烂但答案蒙对**"仍能拿中等正分，策略照样能靠**抄近道**（跳过搭图、直接出答案）刷分，永远学不会编排（§2 challenge iii 点名的 shortcut behavior）。本文用**减 1 基线 + 条件放行指示函数**，制造**严格符号分离**（不合格必负、合格才非负），使"高奖励 ⟺ 结构合格 **且** 答案对"。这在数学上等价于**先学会搭合格骨架、再学答案**的课程（§B.3 证明 $\mathbb{E}[\hat A\mid\mathcal{F}]>0>\mathbb{E}[\hat A\mid\mathcal{F}^c]$，Eq. 50——优势符号把概率质量从 $\mathcal{F}^c$ 推向 $\mathcal{F}$）。代价：需要人为设定"什么算结构合格"（4 项检查本身是启发式，见 §15 批判）。

> **读出什么**：这套"减基线 + 硬闸门 + 分项结构检查"和标杆 Harness-Bench 的"乘法 + Security 硬闸门"是**同一种防漏网哲学**——都拒绝"某一维刷高就放行"。区别是 HB 用乘法、FlowSteer 用"加法 + 指示函数闸门"，效果类似。

## §9　训练算法：CWRPO = canvas-masked GRPO（把环境 token 的梯度屏蔽掉）

> 主讲提示：这页讲"怎么把上面的奖励真的训进去"。重点是那个 token mask——它是本篇相对 vanilla GRPO 的关键改造。

**端到端 RL（Eq. 12）**。采用标准 GRPO（Shao 2024），**加一处画布特定改造：token 级 mask**。给任务 $q$ 在旧策略 $\pi_{\theta_{\text{old}}}$ 下采 $N$ 条轨迹 $\{\tau_i\}_{i=1}^N$，masked 目标：
$$
\mathcal{J}(\theta)=\mathbb{E}_{q\sim\mathcal{D}_Q,\{\tau_i\}\sim\pi_{\theta_{\text{old}}}}\!\left[\frac{1}{N}\sum_{i=1}^N\frac{1}{|\tau_i|_{\text{mask}}}\sum_{t=1}^{|\tau_i|}\text{mask}_t^{(i)}\cdot\min\!\big(\rho_\theta^{(i,t)}\hat A_i,\ \text{clip}(\rho_\theta^{(i,t)},1\pm\epsilon)\hat A_i\big)-\beta D_{\text{KL}}(\pi_\theta\|\pi_{\text{ref}})\right]\tag{12}
$$
符号（逐个定义）：
- $\text{mask}_t^{(i)}\in\{0,1\}$：**canvas token mask**——**1=Flow-Director 自己生成的 token，0=画布反馈 token**；$|\tau_i|_{\text{mask}}=\sum_t\text{mask}_t^{(i)}$ 只数策略 token；
- $\rho_\theta^{(i,t)}=\pi_\theta(w_t^{(i)}|\cdot)/\pi_{\theta_{\text{old}}}(w_t^{(i)}|\cdot)$：第 $i$ 条第 $t$ 个 token 的**重要性比**；
- $\hat A_i$：由**轨迹级奖励** $R(\tau_i)$ 算出的**组标准化优势**（$\hat A_i=(R(\tau_i)-\bar R)/(\sigma+\epsilon)$，Algorithm 1 第 23 行）；
- $\epsilon$=clip 范围(=0.20)，$\beta$=KL 强度(=0.005)，$\pi_{\text{ref}}$=参考（初始 SFT）策略。

**为什么要这个 mask（§B.3(iv)，Eq. 57–62，理论最干净的一段）**：一条轨迹的 token 分两类——**策略 token**（反思+动作文本）与**环境 token**（画布返回）。环境 token 的生成分布**不含 $\theta$**，故其梯度为 0：
$$\nabla_\theta\log p_\theta(w_{1:|\tau|})=\sum_{t\in\mathcal{T}_\pi}\nabla_\theta\log\pi_\theta(w_t\mid w_{<t}) \tag{58/59}$$
用 mask 只选策略 token $\mathcal{T}_\pi$，**梯度估计是无偏的**（Eq. 60），且因为 $|\mathcal{T}_\pi|<|\tau|$、环境 token 只贡献与 $\theta$ 无关的噪声，**方差更小**（$\text{Var}_{\text{mask}}<\text{Var}_{\text{no-mask}}$，Eq. 61–62）。

> **Why（设计层）——不 mask 会怎样？**
> **Why（设计层）**：朴素做法是把整条轨迹（含画布反馈）所有 token 一起算 policy gradient。→ 环境反馈 token 的"概率"其实由**画布**决定、不由策略决定，对它们求梯度是在**对不可控量施力**——引入**有偏且高方差**的噪声梯度，训练震荡（§5.6 点名这是 GRPO/DAPO 把轨迹当"结构不透明序列"时的失败模式）。本文的 token mask **只对策略生成的 token 回传梯度**，无偏且降方差。本文把"GRPO + 这个 canvas mask + 条件放行奖励"的组合称作 **CWRPO**（图示 $\mathcal{J}_{\text{CWRPO}}(\theta)$，Figure 3）。

**训练总流程（Algorithm 1，三阶段 A→B→C）**：
- **Stage A（初始化）**：空图 $\mathcal{G}_0$、建 system prompt、第一步 think+action、画布首反馈、建 $H_1$；
- **Stage B（多轮协作搭图）**：$t=2\dots T$，每步 think→action→画布反馈→更新 $H_t$；遇 `finish` 且**约束检查通过**(`CheckConstraints()`) 才 break；末步执行图得答案 $y$；
- **Stage C（RL 优化）**：采 $N$ 条轨迹，按 Eq. 40 算 $R$、算组标准化优势，按 Eq. 12 的 masked GRPO 更新 $\theta$。
- **复杂度**：训练 $O(NT+T)$、推理 $O(T)$（§C 复杂度分析）；内存随历史长度线性增长，靠 windowing/summary 控制。

**评测指标（§5.1 / §F，给定义式）**：
- **QA 用 Exact Match / F1**：$\text{EM}=\frac{1}{N}\sum_i\mathbb{I}(\text{norm}(y_i)=\text{norm}(y_i^*))$（Eq. 63，norm=小写+去标点+去冠词+压空格）；$\text{F1}=\frac{2\cdot P\cdot R}{P+R}$，$P=\frac{|y\cap y^*|}{|y|}$，$R=\frac{|y\cap y^*|}{|y^*|}$（token 级，Eq. 64–66）。
- **数学用 Accuracy**：$\text{Acc}=\frac{1}{N}\sum_i\mathbb{I}(|y_i-y_i^*|<\epsilon)$，$\epsilon=10^{-6}$；符号答案用 SymPy 判等（Eq. 67–68）。
- **代码用 Pass@1**：$\text{Pass@1}=\frac{1}{N}\sum_i\mathbb{I}(\text{execute}(y_i,\text{tests}_i)=\text{pass})$（单次生成跑测试，Eq. 69–70）。

**关键超参（Table 9）**：策略=Qwen3-8B + LoRA(64/64)，bf16；GRPO clip 0.20 / KL 0.005 / 每组 N=36；生成 temp 0.6 / top-p 0.95 / top-k 20 / max 2048 / 开 thinking mode；**交互 $T_{\max}$=20**、max context 16,384；**执行后端=GPT-OSS-120B（temp 0，超时 600s）**；硬件 **2×A100 80GB**。

## §12　消融与分析（RQ4/RQ5）：每个部件都在出力，token mask 尤其关键

> 主讲提示：对着 Table 4 + Figure 5 讲。分两组：组件消融 vs RL 目标消融。强调"去掉 token mask 掉得最狠"。

## §13　案例研究：一条工作流是怎么"边搭边验"长出来的（§H）

> 主讲提示：这页把抽象机制落到一个具体例子，最能让听众"看见"渐进编辑。用 AIME coin 题走一遍 9 轮。

**AIME 2025 coin 题**（§H.1，ground truth 610）：贪心硬币何时=最优硬币，数 $N\in[1,1000]$。FlowSteer 用 **9 轮**搭出 `Plan → Programmer → Verify → Format → finish`（4 算子）：
- **Round 1-2（Plan）**：加 Plan 算子（进 AWAITING_PROMPT）→ set_prompt 让它产出"贪心 vs DP 对比"的 6 步策略；
- **Round 3-4（Programmer）**：加 Programmer，沙箱跑 Python，`greedy_coins` vs DP `min_coins`，执行得 **610**；
- **Round 5-6（Verify）**：策略反思"竞赛题不能只信一条计算路径"，加 Verify 独立复核 → 列 test case 表（N=42/30 贪心失败、N=25/100/1000 成功），返回 `is_correct=True, confidence=HIGH`；
- **Round 7-8（Format）**：加 Format 抽出干净答案 "610"；
- **Round 9（finish）**：四算子链完整 → `finish`，最终 `Plan→Programmer→Verify→Format`，答案 610 ✓。

> **读出什么**：这条轨迹**同时命中 $R_{\text{diversity}}$ 的多项检查**——含 Verify（$R_{\text{checker}}$）、末步 Format（$R_{\text{format}}$）、≥3 算子（$R_{\text{operator}}$）。它是"条件放行"想要的**合格骨架**：结构达标后答案奖励才被放出来。§H.2 的多跳 QA 案例则展示 `Decompose → [Custom×3 并行] → Verify → Aggregate`——**parallel 控制结构**被用在需要并行子问题的任务上。

## §14　理论主张（Prop.1–3，诚实标注"实验+证明"性质）

> 主讲提示：三条命题都是"直觉可信、证明依赖三条温和假设"。要区分"证明"与"经验验证"。汇报时点明假设，别当定理硬卖。

论文给了 3 条命题（§4 声明，§B 证明），都基于 3 条假设：**A1 认知可分解**（任务能拆成认知原语的有限组合）、**A2 画布反馈有信息量**（$I(Z;o^{\text{exec}}|H_{t-1})>0$）、**A3 可修复**（存在把非法图推向可行集的编辑，势函数 $\Phi$ 递减）。

- **Prop.1（算子-动作认知完备）**：算子库 $\mathcal{O}$（功能覆盖 7 类认知原语 P/D/E/M/R/I/O）+ 动作空间 $\mathcal{A}_{\text{type}}$（结构完备：能构造顺序/分支/循环/并行）**能拼出所有目标图**（§B.1，Eq. 16–23 给出满射 $\psi:\mathcal{O}\to\mathcal{C}$）。
- **Prop.2（解耦可泛化）**：设计者–执行者解耦让 $\pi_\theta$ **跨算子库与后端无需重训**即可泛化（经验证据=RQ3 六后端，§5.4）。
- **Prop.3（多轮单调改进）**：在 A2–A3 下，多轮画布交互的**贝叶斯风险势 $V(H_t)=1-\max_z\pi_t(z)$ 期望单调不增**（Eq. 27–34，用鞅性质 + Jensen 证 $\mathbb{E}[V(H_t)|H_{t-1}]\le V(H_{t-1})$），故成功率单调不降；且**多轮至少不差于单轮**（单轮=第一步就 finish，是多轮策略空间的特例，Eq. 39）。

> **读出什么（批判预埋）**：这些命题**优雅但假设强**。A1（任务真能拆成这 7 原语）、A2（反馈总有正信息）、A3（总存在修复动作）都是"良性假设"，作者也说它们"不要求真实系统完美，只够支撑理论"（§B）。所以更准确的定位：**这是"为什么该 work"的解释性框架，不是"一定 work"的保证**——真正的证据仍是 §5 的实验。

## §15　局限与批判（原文承认的 + 我的补充）

> 主讲提示：这页是判断力高地。既要认它强，也要指出它把哪些成本藏进了"训练"和"启发式奖励"。

**原文较少显式列局限**（这本身是个可批评点），但从设计里能读出边界；以下**区分"论文自述倾向" vs "我的独立质疑"**：

论文侧承认/隐含：
- **天花板受后端约束**：解耦意味着策略只能"选算子/配 prompt"，最终答案质量**上限由冻结的 $\mathcal{M}_{\text{exec}}$ 决定**（§4.2）；弱后端上编排再好也有限。
- **理论靠 3 条温和假设**（A1–A3），非无条件成立（§B）。

我的补充批判：
- **奖励里的"结构合格"是人手定的启发式**：$R_{\text{diversity}}$ 的 4 项检查（要有 Verify、要 Format 收尾、≥3 算子、要有控制结构）**本身就是一套"人对好工作流的先验"**——这等于**把一部分"手写控制流"从代码搬到了奖励函数里**。所以"完全不手写控制流"是被**打了折扣**的：控制流的**形状约束**仍是人给的，只是**具体编排**交给了 RL。这点原文未直面。
- **训练成本 vs 手写成本的账没算清**：2×A100、300 步、10,778 训练样本、每步采 36 轨迹×最多 20 轮×后端调用——**编排是省了人手，但换来了不小的训练算力与后端调用开销**。什么 regime 下"学"比"写"划算，原文未给盈亏平衡分析（Θ5：别把"学编排>手写"绝对化）。
- **$T_{\max}=20$、算子库=12 是小世界**：真实 harness 的工具/控制流规模远大于此；Figure 5e 只测到 14 算子就涨，**外推到几十上百工具时策略是否还稳**未知。
- **执行后端换成 GPT-OSS-120B 训练、GPT-4o-mini 评测**：训练/评测后端不完全一致（Table 9 vs Table 3），**分布偏移**对结论的影响原文未消融。
- **匿名仓库 + 预印本**：代码可得性与可复现性待正式版核验。

## ★ 对我们的启发（Inspires Us）

> 这一节是组会高潮，也是本库相对 auto-research 的第一人称优势：**我们（Claude Code / m9.* 的 agent）本身就是一个 harness，我们的控制循环大多是人手写的**。FlowSteer 正面回答"这套控制流能不能学出来"——每条都能打到我们自己身上。

➤ **a. 可直接借用的招（reuse）**：
1. **条件放行奖励（Eq. 40）** `R = -1 + R_struct + 𝟙{R_struct=1}·R_answer`——凡是我们要"先满足硬约束、再优化质量"的场景，都能抄这个"减基线 + 指示函数闸门"的形状：结构/安全不合格必负、合格才放开质量奖励，从数学上逼出"先合规再优化"的两阶段课程，且天然堵住抄近道。
2. **canvas token mask（Eq. 12）**——任何"策略 token 与环境/工具返回 token 交错"的多轮 RL（我们的 ReAct 循环恰是），都该**只对策略生成的 token 回传梯度、mask 掉工具返回 token**。这是无偏 + 降方差的免费午餐（Eq. 60–62），我们若要 RL 微调自己的循环策略，这是第一个要加的开关。
3. **两步机制（add → 强制 set_prompt）**——用**环境状态机**替小模型分解"结构决策 vs 内容决策"。我们给子代理派活时也可借鉴：先定"派哪个子代理"（结构），再单独一步定"给它什么 prompt/预算"（内容），降低单步认知负荷。

➤ **b. 可迁移到我们课题的思路（transfer）**：把 FlowSteer 的 $\pi_\theta$/$\mathcal{M}_{\text{exec}}$ 解耦映射到我们——**Flow-Director ≈ 我们的"控制循环/编排层"，$\mathcal{M}_{\text{exec}}$ ≈ 我们实际干活的模型调用**。核心 idea："**把编排层拿出来单独学，冻住执行层**"，可迁移到 auto-research 的 `m9.*`：例如把"何时调 Plan、何时调 critic、何时收口"这段**目前写死的编排**，改成一个可学的小策略。**迁移前提会变的**：我们的算子库比 12 个大、工具有真实副作用（不像沙箱可随意重跑），所以"可修复假设 A3"未必成立——真实工具的编辑不总能撤销。

➤ **c. 它暴露的开放问题 = 我们的机会（open problem → opportunity）**：本文把"结构合格"写成了 4 个**人手启发式检查**（要有 Verify、要 Format 收尾…），等于**把控制流的形状先验又搬回了奖励函数**。**机会**：能不能让"什么算合格结构"也**从数据里学**（例如用一个判别器/从人类好轨迹里蒸馏 $R_{\text{diversity}}$），而不是人写死 4 条？**可下手的第一步**：在我们的循环里，先用"是否包含一次自检 / 是否落盘产物"这 2 条**极简结构奖励**做闸门，量化它能否在不手写完整控制流的前提下压低"跳过自检直接交付"的比例。

➤ **d. 与本库其它论文/模块的连接（connect the dots）**：
- 与**标杆 Harness-Bench（2605.27922）正面呼应**：HB 证明"换 harness 分数摆 23.8 分"、把 harness 当**被测变量**；FlowSteer 更进一步，把 harness 的 L 层当**可学变量**——一个测、一个学，正好接力。HB 的"乘法 + Security 闸门"与本文"减 1 + 条件放行闸门"是同一种**防漏网**哲学。
- 与 **AFlow / GPTSwarm（离线生成 / 可优化图）对立又互补**：它们把"图"当优化对象但**反馈不进环 / 靠外部搜索**，FlowSteer 把"图的构造过程"当 RL 策略——是对同一目标的**端到端化**。
- 与 auto-research 的 `m9.8`（独立验证收口）共享隐忧：FlowSteer 靠 Verify 算子自检，但**"谁来 verify the verifier"**同样悬空（Verify 也是同后端在跑）。

➤ **e. 如果我来做下一步（my next move，第一人称、可执行）**：我会在我们**自己的控制循环**上做一个**最小可行 RL 改造**——把"何时插入一次 verify / 何时 finish"这两个**目前手写的决策点**抽出来，用 FlowSteer 的 **canvas token mask + 条件放行奖励**训一个极小策略（冻住执行模型），只在 5–10 个任务上跑，先验证两件事：(1) 加 token mask 后训练是否比不加更稳（复现 Eq. 61–62 的降方差）；(2) 条件放行闸门能否把"跳过 verify 直接收口"的轨迹比例压下来。**若这两点复现**，再考虑把更多手写编排点逐个"让给"策略。

## §18　一页速记（汇报当天速览）

- **命题**：把"设计一条 agent 工作流"变成 RL 问题——`max J(θ)=E[R(τ)]`，τ 是"编辑轨迹"。
- **RL 三要素**：**状态**=交互历史 $H_t$（图是其函数，Eq.5）；**动作**=8 类原子编辑（add/delete/modify/set_prompt/finish + parallel/conditional/loop）；**奖励**=结构多样性(4 检查) 闸门放行答案正确性（Eq.40）。
- **画布（Workflow Canvas）**：可执行图环境，每步原子编辑后回"语法检查 + 失败原因 + 修复建议"，两状态机（BUILDING/AWAITING_PROMPT）逼小模型分解结构 vs 内容。
- **解耦**：$\pi_\theta$(小策略/harness) 与 $\mathcal{M}_{\text{exec}}$(冻结后端/model) 分离（Eq.6）→ 跨算子库/后端不重训（Prop.2 / RQ3 六后端）。
- **训练 CWRPO**：GRPO + **canvas token mask**（只对策略 token 回梯度，无偏降方差 Eq.60–62）+ **条件放行奖励**（减 1 基线制造符号缺口，先合规再优化 Eq.43–46）。
- **铁证（Θ2）**：同后端 GPT-4o-mini，IID 平均 **EM +22.65**（Table 3）；OOD 平均 EM +17.72；AIME +16.67。
- **消融**：去掉 **token mask** 掉最狠（MATH 81.25→67.97）；去条件放行/多样性奖励 → 抄近道回潮；FlowSteer > GRPO > DAPO（§5.6）。效率上 token 更省、轮数更少。
- **诚实（Θ5）**："学编排"的增益集中在多步/保状态/串工具任务、弱后端受益大；但天花板受冻结后端约束、训练有算力成本、"结构合格"4 条仍是**人手启发式**（控制流形状先验没真消掉，只是从代码搬进了奖励）。
- **对我们（Θ3）**：搬 **token mask + 条件放行奖励** 到我们自己的 ReAct 循环；把"何时 verify / 何时 finish"这两个**手写决策点**抽出来训个极小策略，先复现降方差 + 压低"跳过自检直接收口"。
