# Less Context, Better Agents：少即是多，给长程工具型 Agent 的高效上下文工程

> **本篇是 D 组（上下文/记忆）里的"反方压舱石"。** 当本组一大半论文（MemGPT、A-MEM、Mem0、AgentFold……）都在讲"怎么把记忆存得更多、更结构化、更可检索"时，这篇微软的论文反过来问一句冷静的话：**对工具重的单会话任务，你真的需要那一整套外部记忆库/检索器吗？** 它的答案是——**不需要**；一个"近期窗口 + 一段紧凑运行摘要"就够，而且更省、更准。读它的正确姿势不是"信或不信",而是学它**把自己的 regime 划得干干净净**："我只对这一类工作流成立，别外推成普适真理。"这正是本组最缺的判断力。

## §1　TL;DR（一页讲清这篇在干嘛）

> 主讲提示：开场先把"反潮流论点"抛出来，再用一组极端数字砸实，最后点明"它只声称一类工作流"。

一句话：在一个**真实企业任务**（微软 Dynamics 365 Finance & Operations 里的"酒店发票逐条录入 / expense itemization"，经 MCP 代理调用）上，作者比较了四种**上下文策略**，发现——**把上下文裁到"最近 5 个工具调用/响应对" + 对被逐出的旧内容做"一遍式摘要"（摘要窗口 W=3）**，可以在**同一个模型（GPT-5）** 上，把完整任务完成率从"保留全历史"的 **71.0%** 抬到 **91.6%**，同时把总 token 从 **1,480,996** 砍到 **553,374**（约 −62.7%）、把跑完 50 个任务的墙钟时间从 **14.56 小时**压到 **5.79 小时**（Abstract、§4.3–4.4、Table 2）。**模型没换，换的只是"上下文里留什么"。**

- **属于 harness 的哪一层（Θ1）**：本篇是纯 **C（Context 上下文）层**的干预——它不新增工具、不改模型权重、不重训（作者反复强调"inference-time only, portable across LLM backends"，§1 贡献 1、§6.4）。它动的唯一旋钮是**"每步推理前，把哪些历史消息放进上下文"**。
- **回扣全库论点（Θ2）**：这是 `Agent = Model + Harness` 的又一记实证——**Model 固定为 GPT-5，只调 Harness 的上下文策略，完成率就摆动了 71.0 → 79.0 → 91.6（Table 2）**，且跨到第二个模型族（Claude Sonnet 4.5）上排序依旧（88.0 → 92.0 → 94.5，Table 8）。它和 Harness-Bench（2605.27922，只换 harness 摆 23.8 分）是同一命题的两个切片。
- **在本组里的角色 = 头号反方（Θ5 预告）**：它给"复杂记忆库/检索器"泼了一盆**有数据支撑的冷水**——但**只对"工具重、单会话、有硬完成判据"这一 regime**。作者自己把话说得很死："strong evidence for **one class** of enterprise tool-use workflow rather than a proof of universal generalization"（Abstract 末句）。这份 regime 诚实，本身就是范本。
- **够新够权威（Θ4）**：2026-06 预印本，**微软**出品，用的是**自家 ERP 产品 + 前沿模型（GPT-5 / Claude Sonnet 4.5）**的真实工作流，与本库 harness 标杆 Harness-Bench 同批次（2026 上半年）。**权威性来源**：不是学术玩具任务，而是**大厂在自家生产级 ERP（Dynamics 365 F&O）上、经真实 MCP 代理**跑出来的——加上严格统计（Wilson + Student-t 双区间）、跨模型复现（Sonnet 4.5）、跨类目泛化（Travel/Meals）。这让它的"少即是多"不是一句口号，而是**带误差带、带机制验证、可被别人在自己 MCP 栈上复刻**的结论。
- **一条能带走的经济账（为什么老板会关心）**：C2→C4 把单个 50-任务基准的 token 从 148 万砍到 55 万（2.68×）、墙钟从 14.56h 砍到 5.79h（2.51×）。**在"每月跑成千上万张费用报销"的规模上，这是直接的推理成本节省 + 吞吐/延迟改善**（§4.5、Appendix J）——而且**是在完成率反而更高（71%→91.6%）的前提下拿到的省**。"更省"和"更准"在这里不是取舍，是同向的。

## §2　问题与动机：为什么"上下文会撑爆"值得单独治

> 主讲提示：这一段用 Why 三连的"问题层"，讲清"工具重的企业 agent"到底卡在哪。别急着讲方法。

**Why（问题层）——不解决会卡住什么？**
LLM 以 **agent** 形态接入企业系统时，走的是"调工具→读工具返回→再决策"的多步循环。问题出在**工具返回太啰嗦**：企业 ERP（这里是 D365 F&O）通过 MCP 暴露的工具，每次返回的是**完整表单状态快照**——字段值、嵌套表单状态、导航面包屑、系统元数据。作者给的量级是：**单条工具响应就有 500–3,000 token；一个复杂的逐条录入任务要 15–30 次工具交互，累计吃掉 50,000–150,000+ token**（§3.2）。于是两个后果（§1 Background）：

1. **上下文溢出**：多步工作流里，哪怕是"扩容后的"大窗口也会在任务完成前被塞满。
2. **成本与"context rot（上下文腐烂）"**：处理成本随上下文长度**线性**增长，生产规模下贵到不可行；更糟的是，业界已观察到一种失败模式——**模型的有效召回会随 token 数增长而退化，远在触到硬上限之前就开始"记不住重点"**（§1 引 [4] Anthropic 的"context rot"说法）。

**这个任务为什么"代表一大类"**：expense itemization 要把一张收据总额拆成 4–22 个行项，每项都要"建行→填子类目→填金额→核对"，且有一条**硬判据**——剩余未分配金额必须**正好等于 \$0.00**，任何非零残差在生产 ERP 里都算失败（会触发合规复审）。作者说这代表"一大类企业 agent 挑战"：多步拆解 + 严格数值约束 + 逐步耗尽 token（§3.1）。

**把"上下文撑爆"算成一笔账（直觉先行）**：设一次工具响应平均 $r$ 个 token、一个任务要 $k$ 次工具交互。若**保留全历史**，那么第 $j$ 步请求模型时，上下文里累积的工具返回约为 $\sum_{i=1}^{j} r_i$；跑完整个任务，输入侧被读进模型的 token 大致是**二次量级** $\mathcal{O}(k^2 \bar r)$（因为每一步都把此前所有响应重喂一遍）。代入本文量级（$r\approx$ 500–3,000、$k\approx$ 15–30），单任务就能到 50,000–150,000+ token（§3.2）——这还只是**一个**任务。而"只留最近 $N$ 个工具对"把每步上下文压到常数 $\mathcal{O}(N\bar r)$，累积就退回**线性** $\mathcal{O}(kN\bar r)$。§7 会看到，这条"二次 → 线性"的差别在实测里就是 **1,481K vs 535K token** 的差别。

> **读出什么**：这篇的敌人**不是"事实记不住"**，而是"**工作记忆被啰嗦的旧状态淹没**"。这条区分——"工作记忆/陈旧状态"问题 vs "事实召回"问题——是全篇的**题眼**，后面 §12、§14 会反复回到它。请先在脑子里记住这把尺子：**如果旧信息会『过期』，它就是噪声不是资产；如果旧信息是一条要长期记住的『事实』，那才需要记忆库。** 本文全程只谈前者。

## §3　它站在谁的对立面：四类上下文管理路线（Related Work）

> 主讲提示：这页是本篇"反方身份"的来源。用 Table 1 一张图把"它和别人不同在哪一层单位"讲清——单位不同，问题就不同。

作者把已有工作按"**操作单位**"分成四类，并明确"我不做前两类、和第三类互补、比第四类更简单"（§2，Table 1）：

| 路线 | 操作单位 | 代表 | 与本文的关系 |
|---|---|---|---|
| Token 级提示压缩 | token | LLMLingua [2]、Selective Context [3] | **不做**：在"工具交互"之下操作，会**破坏 ERP agent 必须逐字读的结构化表单状态**（控件名、数值余额）——删几个 token 就把表单读错了 |
| 外部/长期记忆 | 记忆条目 | **MemoryBank [5]、LongMem [6]** | **不做**：为"跨会话**事实召回**"设计（多会话对话）；而本文治的是"单会话里的**工作记忆/陈旧状态**"，不是"检索一条事实" |
| 学习式压缩 / trajectory | 轨迹 | **ACON [9]**、long-horizon SWE 的 context 管理 [10]、厂商原生 compaction [4] | **互补但更简单**：ACON 学一个失败驱动的压缩器；本文用**固定近期窗口 + 可选一遍摘要**，不学压缩器 |
| 工具用广度基准 | — | MCP-Bench [11] | 广而浅（测跨服务器工具广度）；本文**窄而深**（单一高价值工作流 + 硬判据 + 同时测成本） |

作者把自己的定位写成一行（Table 1 末行）："**This work — 单位=Tool pair（工具对）— 近期逐出 + 对被逐出对做摘要。**"

> **Why（设计层）——为什么"操作单位"是关键分水岭？**
> 朴素想法是"上下文治理就治理呗，管它按 token 还是按轮"。→ 但**单位决定了你会不会误伤**：按 token 压缩（LLMLingua）可能把 ERP 必须逐字读的字段值删坏；按"记忆条目"存取（MemoryBank）解的是"跨会话找回一条事实"，跟"单会话里旧表单状态过期"根本是两个问题。本文把单位**钉在"整个工具调用/响应对（semantic level）"**：要么完整保留、要么整体逐出/摘要，**从不切开一个工具对内部**（§2 原文 "only ever evicting or summarizing complete units"）。这保证了"留下来的表单状态永远是完整、可逐字信任的"。

> **读出什么（Θ2 呼应）**：这张表就是它"反方"身份的户口本——**它明确拒绝了 MemoryBank/LongMem 那条"外部记忆库"路线**，理由不是"那条路线错",而是"**那条路线解的是另一个问题（事实召回），不是我这个问题（工作记忆/陈旧状态）**"。这正是 §14「与多存派对立」一节要展开的核心。

## §3.5　环境与任务体：MCP + ERP 的"啰嗦"到底啰嗦在哪

> 主讲提示：这页把"敌人"具象化——让大家看清"一条工具返回长什么样、为什么难"，否则后面"裁掉旧状态就变好"会显得像魔法。这也是本篇作为"真实企业任务"权威性的来源。

**工具体（Appendix B 的工具清单）**：agent 通过 D365 F&O 的 MCP 代理，拿到 **21 个工具，分三类**——(i) **form tools（表单工具，13 个）**：UI 级交互（表单/标签页导航、菜单项与控件发现、读/设控件值、开查找、网格过滤/排序/选行、点控件、保存）；(ii) **data tools（数据工具，6 个）**：实体级 OData 访问（实体类型发现、元数据检索、对 F&O 实体的增删改查）；(iii) **API tools（2 个）**：发现并调用自定义服务端动作菜单项（§3.2、Appendix B）。**注意：工具集在四个配置里完全一样**——这保证了 T 层不变，任何差异只能来自 C 层（上下文策略）。

**为什么工具返回会"淹没"上下文**：MCP 工具返回**按设计就啰嗦**——每条响应携带**完整表单状态快照**：所有字段值、元数据、导航面包屑、系统状态（§3.2 原文 "verbose by design"）。这就是 §2 那笔账里的 $r\approx$ 500–3,000 token 的来源：agent 每点一个控件，ERP 就把"整张表单现在长什么样"塞回来一次。

**任务体（Figure 1 的样例 + Appendix E 的结构注释）**：每个任务是**两段式提示**——`#Task` 段说清 D365 动作（在 `TrvExpenseLines` 下建费用行），`#Data` 段给一份结构化收据（公司、商户、日期、总额、类目，外加一个 `PurchasedItems` 清单当**真值**）；C2–C4 里 user model 额外收到一份 `user_context`（定义"完成 = 残差 0.00 且表单关闭"的判据，§3.3）。Figure 1 的样例收据总额 \$333.05，含 5 行，其中**两笔 "Hotel Tax"（\$6.82 与 \$10.23）金额不同**、且 "Entertainment External"→*Business entertainment*、"Room Service & Meals"→*Room service* 需要**非平凡子类目映射**。

**两种"故意埋的坑"（§3.6）**——这正是失败的主要来源：
- **同名不同额的重复行**：agent 若因为"看到子类目已存在"就跳过第二笔 "Hotel Tax"，会留下 \$10.23 未分配。
- **间接的行→子类目映射**：把 "Entertainment External" 错映到别的子类目，就会引入一条无法核销到零的错误行。

**数据集结构梯度（Appendix E、Table 6）**：主基准是 **50 个酒店任务**（中位 8 行、范围 4–23、23 项子类目词表、有 per-night 重复子类目）；扩展评测另加 **Travel（Car rental 15 + Flight 15 = 30 任务）** 与 **Meals & Gifts（Business meal 20 + Gift 12 = 32 任务）**。作者按**结构复杂度排序：Hotel > Travel > Meals & Gifts**——酒店最难（多晚住宿、同类目逐晚重复、词表最大且含近义词如 Room tax vs Non-Room tax、Restaurant vs Room service vs Loungebar），Meals & Gifts 最简单（1–3 行、单一子类目、无重复）。这条梯度在 §4.8 用来测"策略是否随复杂度稳健"。

> **读出什么**：把这页记牢，后面所有"机制"才立得住——**"裁掉旧状态就变好"不是玄学**，而是因为"旧状态"具体就是"上一次点控件时 ERP 返回的那张**已经被后续操作改掉**的表单快照"。agent 若回头引用它（"表单里还差 \$240"），就会据一个**过时读数**做错动作。§8/§12 的 stale-state 失败正是这么来的。

## §4　方法：四个配置 = 一条"从最少到刚好"的上下文阶梯

> 主讲提示：这页把 C1→C2→C3→C4 讲成一条"逐步加东西"的阶梯，并**特别说清 C1 是『动机消融』不是『上下文阶梯的一环』**——这是作者最容易被误读、也最诚实的一处设计。

**系统架构（§3.2）**：四个部件——(1) **GPT-5 agent**（主 LLM，带一份详细系统提示，讲清完整录入流程、合法子类目、子类目映射）；(2) **User model（仅 C2–C4 有）**：一个次级 LLM（GPT-4.1）扮"用户"，回答 agent 中途的澄清/确认请求，并被 `user_context` 约束"任务只有在剩余=0.00 且表单已关闭才算完成"；(3) **D365 F&O MCP server**：把 ERP 表单交互暴露成离散工具（表单导航、读字段、设值、点按钮）；(4) **内部评估 harness**：非交互式编排框架，管循环、上下文工程、指标采集，**执行期无人参与**。

四个配置（§3.3）：

- **C1 — GPT-5，无 User Model（动机消融 / motivating ablation）**：只有系统提示，没有"用户"。结果 agent 会中途停下来问澄清问题，而 harness 是非交互的、没人回答，于是工作流卡死、任务半途而废。**C1 的唯一作用是量化"为什么在非交互 harness 里需要一个 user model"，它不是上下文阶梯的一环。**（下面 §6 会看到 C1 只有 8.0% 完成率。）
- **C2 — GPT-5 + User Model（全上下文 / Full Context）**：加入 user model 解决 C1 的"卡死"，并**保留全部对话历史**。这是"标准 agent 实践"，也是**上下文工程比较的基线**。
- **C3 — GPT-5 + User Model（保留最近 5 个工具调用 / Last-5 Tool Calls）**：把上下文裁到**最近 5 个工具调用/响应对**，更早的**直接丢弃、不摘要**。为什么是 5？作者做了任务结构分析：一个录入行大约要 2–3 次工具调用（建行、设值、可选核对），所以**5 个工具调用≈两个完整录入周期的工作记忆**（§3.3；鲁棒性在 §4.7 验证）。
- **C4 — GPT-5 + User Model（Last-5 + 摘要，窗口 W=3 / Last-5 + Summarization）**：在 C3 的裁剪之上，对**被逐出的更早历史**做**自动摘要**；摘要窗口 W=3 表示——用"逐出边界之前最近的 3 次交互"去生成那段摘要。摘要抓的是"打开了哪些表单、操作了哪些控件、点了哪些按钮、录入了哪些数据"，给 agent 一份**任务级情境**，但不付全历史的 token 代价（§3.3）。

**关键的"隔离设计"（§3.3 原文，务必讲清）**：作者坦白 **C1 一次改了两件事**——既去掉 user model、又去掉任何上下文策略——所以"C1→C2 的巨大跳升**不能**归给上下文工程"。因此他们把话挑明："**user model 在 C2/C3/C4 里被固定为『存在且不变』；我们所有的上下文工程主张，都只建立在 C2(全上下文)→C3(裁剪)→C4(裁剪+摘要) 这个只变上下文策略的比较上。**"C1 只是个动机消融。

> **Why（设计层）——为什么非要用一个"User Model"当陪跑？**
> 朴素做法是让 agent 拿一份静态系统提示自己一路做到底（就是 C1）。→ 在**非交互 harness**里，GPT-5 有时会"想确认一下"就停住问问题，没人答→卡死→任务不完整（§3.3、§4.1）。加一个 user model 扮用户来回话，把工作流"推着走"。**但这引入一个混淆变量**，所以作者用"C1 只做动机、上下文主张只靠 C2–C4"的方式把它隔离掉——**注意：这个"卡死"是 GPT-5 特有的**：跨模型研究里 Claude Sonnet 4.5 **不卡**，其 no-context 基线已经到 88.0%（§4.9、Table 8），反证"C1→C2 的大 gap 是模型特异的停顿行为，不是上下文工程的功劳"。这处自我拆台，是全篇诚实度的高点。

## §5　核心算法：CONSTRUCTCONTEXT（Algorithm 1）逐行读

> 主讲提示：这是全篇唯一的形式化，务必先给直觉、再定义符号、再逐行走一遍。它其实很朴素——朴素正是卖点。

**直觉**：每次要请求模型前，先按一条规则重建上下文——"**留最近 N 个工具往返；更早的，如果开了摘要就压成一句话塞回最前面，否则直接扔**"。

**符号先定义（§3.4）**：
- **H**：完整消息历史（full message history）。
- **N**：要保留的"最近工具调用/响应对"的个数（**pruning window，裁剪窗口**）。C3 用 N=5, W=0；C4 用 N=5, W=3；C2 相当于 N=∞（不逐出）。
- **W**：**summary window（摘要窗口）**——用"被逐出内容里最近的 W 条消息"来生成那段摘要。W=0 表示不摘要（纯裁剪，即 C3）；W=−1 是一个对照变体，表示"摘要**全部**被逐出内容"（无固定近期窗口的全历史压缩，见 §4.7 敏感性）。

**逐行走一遍（Algorithm 1）**：

```
输入：历史 H；保留窗口 N；摘要窗口 W
1  c ← H 中"工具消息"的条数
2  d ← max(0, c − N)                 # 需要逐出的"工具对"个数
3  若 d = 0：直接 return H            # 没超窗，原样返回
4  K ← [] (保留集)， E ← [] (逐出集)
5  按顺序遍历 H 的每条消息 m：
6      若 m 是工具消息 且 还要继续逐出(d>0)：
7          d ← d − 1；把 m 放进 E
8          若 K 的最后一条是"assistant 的工具调用消息"：把它也从 K 移到 E   # 成对逐出：调用与其响应一起走
9      否则：把 m 追加进 K
10 若 W≠0 且 E 非空：
11     E_W ← (W=−1 时取 E 全部，否则取 E 的最后 W 条)
12     s ← SUMMARIZE(E_W)
13     把 "Summary of previous tool calls: s" 插到 K 里"最早被逐出的位置"
14 return K
```

（上面是对论文 Algorithm 1 第 1–23 行的忠实转写；配对逐出的关键在第 11–13 行"if last(K) is an assistant tool-call msg then move it to E"。）

**走一个具体例子（帮助读懂"成对逐出"）**：设 N=2、W=1，历史里已有 3 个工具往返：
`[sys][A:call₁][T:resp₁][A:call₂][T:resp₂][A:call₃][T:resp₃]`（A=assistant 的工具调用，T=工具响应）。工具消息（T）共 3 条，c=3，d=max(0,3−2)=1，要逐出 1 个工具对。遍历时遇到 `T:resp₁` 逐出它（d→0），并把它前面那条 `A:call₁` 也一起搬进 E（第 8 行"成对"）——**调用和它的响应必须同进同出，否则上下文里会留一条"孤儿调用"，模型会困惑**。逐出集 E=`[A:call₁, T:resp₁]`；W=1 只摘要 E 里最后 1 条，生成一句 `s`；最终上下文=`[sys]["Summary of previous tool calls: s"][A:call₂][T:resp₂][A:call₃][T:resp₃]`。时间线仍连贯：一句话概括最早发生的事，后面跟着最近 2 个**完整**往返。

> **读出什么**：这个算法**朴素到近乎"没有算法"**——没有学习式压缩器、没有向量库、没有相关性打分、没有 embedding。它只做两件事：**(1) 按"工具对"这个语义单位、成对地逐出旧的；(2) 可选地把逐出的最近 W 条压成一句话插回去**。作者把"简单"当成**卖点而非短板**：inference-time、零训练、可移植到任何后端（§1 贡献 1）。这正是"反方"的姿态——**别人加机制（分页调度/结构化互链/学习式压缩），我减机制**。在 §11 的 Table 7 里，作者还会拿"N=10 只比 N=5 多不到 1 点却贵 53% token"来证明：**加复杂度在这个 regime 里边际收益递减**。

> **Why（设计层）——为什么摘要只插"一条"、还放在最早逐出的位置？**
> 朴素替代是"每逐出一批就追加一段摘要"，摘要会越滚越多、又变成一种啰嗦。→ 本文只维护**单条**滚动摘要（每次逐出事件重算、替换），且**放在被逐出内容原本的时间位置**（最早逐出处），让模型读到的时间线仍然连贯：`[一句话摘要旧事] → [最近 5 个完整工具对]`。代价：摘要是"一遍式、生成式"的，可能漏信息——作者在 §5 失败分析里诚实地把这算作 C4 的残余失败来源之一。

## §6　评测设置与指标：一条"钉死判据"的严标准

> 主讲提示：这页讲清"怎么算完成"和"钱怎么算"。重点讲主指标为什么是"完整录入率"而非"部分分"。

**数据集（§3.6、Appendix E）**：50 个酒店发票逐条录入任务，经 MCP 代理在 D365 F&O 真环境执行。酒店发票"故意难"——(1) 常有**同名不同额**的行（比如两笔 "Hotel Tax" 金额不同）；(2) **子类目要非平凡映射**到 D365 的 23 项固定词表（如 "Room Service & Meals" → *Room service*、"Entertainment External" → *Business entertainment*）；(3) **严格零残差**。每个任务行数 4–23（中位 8）。50 个任务在四个配置上**完全一致**以保证公平。

**指标（§3.5、Appendix C，均给定义式）**。先给直觉：主指标必须反映"生产 ERP 里算不算真做完"——差一分钱都不行，所以用"完整录入率"。每个任务由**独立回读**（读 D365 里保存后的表单，**不信 agent 自报**）算出四个量：`total_amount / itemized_amount / remaining_amount / num_itemized`，再算：

$$\textsc{CompletelyItemized}=\mathbb{1}\big[\texttt{remaining\_amount}=0.00\big]\quad(\textbf{主指标})$$
$$\textsc{LessThan10\%}=\mathbb{1}\Big[\tfrac{\texttt{remaining\_amount}}{\texttt{total\_amount}}\le 0.1\Big]$$
$$\textsc{AtLeastOne}=\mathbb{1}\big[\texttt{num\_itemized}>0\big]$$
$$\%\textsc{AmountItemized}=100\cdot\tfrac{\texttt{itemized\_amount}}{\texttt{total\_amount}}$$

外加两个效率量：**Total Token Usage**（输入+输出，跨 50 任务合计）与 **Execution Time**（跑完 50 任务的墙钟）。所有基准级数字是"先对 50 任务取均值、再对 5 次独立运行取均值"（§3.5、Appendix C）。

- **主指标为什么是"完整录入率"（§3.5 原文）**：因为它反映**真正的业务完成**——在生产 ERP 里，非零残差 = 失败，无论你分配对了多少。"部分分好看但残差非零"在业务上一文不值，所以不能用"平均分配比例"当主指标。

> **读出什么**：这套"独立回读 + 零残差硬判据"和 Harness-Bench 的"评委看不到答案"是同一种警惕——**成功必须被机器客观验证，不能靠 agent 自报**。这也是它敢下强结论的底气：主指标不是软的 rubric，而是"剩余金额是不是恰好 0"。

## §7　主结果一：全上下文(C2) = 性能不错，但成本灾难

> 主讲提示：先立 C2 这个"标准做法"的靶子——它不差，但贵得离谱。把"贵"量化清楚，后面 C3/C4 的"又好又省"才有冲击力。

**Table 2 关键行**（50 任务 × 5 次运行均值；Comp.=完整录入率=主指标）：

| 配置 | Comp. Item.(主) | <10% Rem. | ≥1 Item. | %Amt Item. | Total Tok(K) | Input Tok(K) | Output Tok(K) | Time(hrs) |
|---|---:|---:|---:|---:|---:|---:|---:|---:|
| C1: 仅 GPT-5（无 user） | 8.0% | 37.2% | 99.6% | 58.89% | 532.6 | 531.3 | 1.3 | 3.08 |
| **C2: +User（全上下文）** | **71.0%** | 74.0% | 100.0% | 92.03% | **1,481.0** | 1,478.5 | 2.5 | **14.56** |
| **C3: +User（Last-5 TC）** | **79.0%** | 87.0% | 100.0% | 96.92% | **535.3** | 532.8 | 2.5 | **5.39** |
| **C4: +User（Last-5 + 摘要）** | **91.6%** | 99.6% | 100.0% | 99.64% | 553.4 | 550.8 | 2.6 | 5.79 |

**C2 读数（§4.2）**：加了 user model + 全历史，主指标到 **71.0%**、平均录入额到 92.03%、"至少录一行"到 **100%**（说明任务指令能可靠地把 agent 带上路）。**但代价惊人**：总 token **1,480,996**（相对 C1 基线 +177.9%，几乎全来自输入 token 1,478,509），墙钟 **14.56 小时**（≈C1 的 4.73×）。作者点出一个刺眼比值：**输入:输出 token = 594.7 : 1**——即成本几乎**全压在输入侧**，也就是"喂进去的历史"。

> **Why（结果层）——为什么 C2 的成本几乎全在输入侧？**
> 因为工具响应是**啰嗦的表单快照**，而"全上下文"意味着**每一步都把此前所有啰嗦快照重新喂一遍**。随着交互变多，输入侧的 token 累积成主导项（1,478,509 / 1,480,996 ≈ 99.83%）。→ 这直接推出全篇的战略判断（§4.2 原文）：**"针对输入侧做削减的上下文管理策略，将带来最大的效率收益。"** 这句话就是 C3/C4 存在的理由。

**顺带看清 C1 这个"动机消融"（§4.1）**：C1（无 user model）只有 **8.0%** 完整录入，但"至少录一行"高达 **99.6%**——这个巨大反差（99.6% 能起步、只有 8.0% 能收尾）说明：**GPT-5 完全能从 MCP 工具描述 + 系统提示里"发起"录入，但在非交互 harness 里没法可靠地把工作流『推到完成』**——它常在录了一两行后就停下来问"要不要继续/是不是这样"，没人答就半途而废（§4.1）。C1 的 token 只有 532.6K（其中输出仅 1,331），正是"任务被过早截断"的表现。**记住：C1 的 8.0% 不是"上下文工程失败"，而是"缺一个能回话的 user"**——所以它被排除在上下文阶梯之外，只当动机（§3.3、§4.9 用 Sonnet 反证）。

> **读出什么**：C2 不是"差方案",而是"**标准但不可持续**"的方案——71% 的完成率在企业里已经能用，但 148 万 token / 14.6 小时的账单让它"在规模上不可部署"（§4.2 原文 "impractical for production deployment at scale"）。那个 **594.7:1 的输入:输出比**是全篇的战略路标：它告诉你**"省钱的杠杆几乎 100% 在输入侧"**——凡是能砍输入历史的策略，回报最大。靶子立稳了：C3/C4 要证明的，就是"砍掉输入侧的旧历史，不但省钱，还更准"。

## §8　主结果二：只裁剪(C3) = 性能与效率"同时"变好（反直觉点）

> 主讲提示：这页是第一个"反直觉"高潮。多数人以为"少给上下文=性能下降换省钱"，这里是**又省又好**。务必把机制讲透。

**C3 读数（§4.3、Table 2）**：把上下文裁到最近 5 个工具对（**丢弃更早的、不摘要**）——

- **主指标 71.0% → 79.0%**：绝对 **+8 个百分点**、相对 +11.3%，**比全上下文还高**。
- **总 token 1,481K → 535.3K**：**−63.9%**，基本回到 C1 基线的 token 预算。
- **墙钟 14.56h → 5.39h**：**−63.0%**。
- 输出 token 反而从 1,331 升到 **2,515**（§4.3）——因为任务被更完整地执行了、agent 推理更充分。

**先破一个直觉误区（为什么大多数人会猜错）**：常见的心智模型是"上下文=信息，信息越多决策越好，裁剪只是拿性能换成本"。这个模型在**"旧信息一直有效"**的世界里对（比如"记住用户三个月前说过的偏好"）。但在 ERP 录入这种世界里，它**错在一个隐含假设**——它假设"历史里的每条工具返回都仍然为真"。实际上，**一张早期的表单快照，在 agent 后续改了表单之后就『作废』了**——它不再描述"现在"，只描述"曾经"。于是"更多上下文"= 更多**作废的、但看起来仍像真的**的状态读数混进视野。

**这为什么反直觉、又为什么解释得通（§4.3 原文）**：直觉上"少给信息该更差";但这里 **"更少上下文"同时买到了"更好性能"**。机制是——**旧的工具交互描述的是"已被后续操作取代的、过期的表单状态"**。保留这些陈旧表单状态会给 agent 对"当前系统状态"的理解**注入噪声**，导致字段填错或导航错误。把上下文限制到最近 5 个工具对，就把 agent 的注意力**钉在"当前表单状态 + 最近动作"**上——这恰恰是"下一步决策最相关的信息"。**一句话：裁剪不是在『丢信息』，而是在『丢已经变成谎言的旧信息』。**

> **Why（结果层）——"少即是多"的确切机制是什么？**
> 不是"上下文越少越好"这种玄学，而是一个**具体的失败通道被堵住了**：全上下文里，agent 会**引用早期快照**（"表单里还差 \$240"这种**已经过时**的读数），据此做错误动作 → 残差。裁剪把这些过时快照移出视野，**过期状态无从引用**。§5 的失败分类学正是为验证这条机制设计的——**stale-state reference（引用过期状态）从 C2 的 34 次 → C3 的 6 次**（Table 5），直接量化了"裁剪堵住了哪条通道"。

> **读出什么**：这一节把"context rot（上下文腐烂）"从"模型召回退化"这个模糊说法，**落地成一个可测的、任务级的具体错误**（引用过期表单状态）。这是本篇最扎实的贡献之一：**它没有停在"长上下文不好"的口号，而是指出"坏在哪一步、并证明裁掉后那一步的错误就消失"**。

## §9　主结果三：裁剪+摘要(C4) = 最优，且摘要几乎不要钱

> 主讲提示：这页收官——C4 是最优点。关键要讲清"摘要补回了 C3 丢掉的什么"，以及"补这一块只花了 3.4% 的额外 token"。

**C4 读数（§4.4、Table 2）**：在 C3 的裁剪上，对被逐出历史做一遍式摘要（W=3）——

- **主指标 79.0% → 91.6%**：比"只裁剪"再 **+12.6 个百分点**；比"全上下文"高 **+20.6 个百分点**。
- **<10% 剩余 = 99.6%**：几乎消灭了"大额未分配"——50 个任务里只有 1 个残留 >10%（C3 有 6 个/13%，C2 有 13 个/26%，§4.4）。
- **代价极小**：总 token 535.3K → 553.4K（**仅 +3.4%**）、墙钟 5.39h → 5.79h（+7.4%）。摘要每次逐出事件只多**一次** LLM 调用（§3.4）。

**摘要补回了什么（§4.4、§6.3 原文）**：纯裁剪(C3)会让 agent **丢掉任务级情境**——"已经录了几行、总额分配到哪了、整体是否接近核销"。丢了这个，C3 会**过早终止**（agent 以为做完了就提交，其实还差钱）。C4 的那句摘要提供了**两条互补信息流**：**最近 5 个工具对供给"当前局部状态"，摘要供给"全局任务进度"**——两者合起来，就凑齐了"可靠完成任务所需的一切"。

> **Why（设计层）——为什么"一句泛泛的摘要"就够，不用显式算余额？**
> 朴素替代是"让摘要器精确计算并写明剩余金额"。→ 但那样摘要器就得内建业务逻辑、还可能算错。本文的摘要器**故意做成"泛型"的**——它只描述"开了哪些表单、点了哪些控件、录了哪些数据"，**不显式算运行余额**（Appendix F 原文 "intentionally generic ... rather than computing a running balance explicitly"）。就这点信号，已经**足以把 agent 推过 C3 的『过早终止』阈值**。代价：它不保证数值精确，残差类算术错误仍偶发（Table 5 的 residual mismatch，C4 仍有 4 次）。

**"两条互补信息流"这个抽象值得单独记（§6.3）**：作者把 C4 为什么成立提炼成一句——agent 可靠完成任务需要**两种**信息，而 C3/C4 恰好各补一半：
- **局部状态（local state）**：当前表单长什么样、上一步做了什么、有没有成功、当前剩多少——由**最近 5 个完整工具对**供给（recency 窗口的职责）。
- **全局进度（global task progress）**：一共要录几行、已经录了几行、整体是否接近核销——由**那一句滚动摘要**供给（summary 的职责）。

C2（全上下文）两者都有，但**把"局部状态"埋在一堆过期快照里**（→ stale-state）；C3（纯裁剪）有干净的局部状态但**丢了全局进度**（→ premature termination）；**只有 C4 同时拥有『干净的局部』和『压缩的全局』**——这就是它 91.6% 的来源。

> **Why（结果层）——为什么"局部要精确、全局要粗略"这种非对称设计是对的？**
> 因为两种信息对"精确度"的需求天然不同：**下一步该点哪个控件、填什么值**，要求**逐字精确的当前表单状态**（差一个字段名就报错）→ 所以局部用"完整工具对"、不压缩；而**"我是不是快做完了、还该不该继续"**只需要一个**粗略的进度感**（"已录 2 行、大概还差一半"）→ 所以全局用"一句泛型摘要"就够，甚至**不必精确算余额**。本文的非对称设计（局部精确 + 全局粗略）正好匹配了这两种需求的精度要求——这也是为什么"精确计算余额的摘要器"是过度设计（§9 的 Why 设计层）。

> **读出什么**：C4 的美学在于**"用最便宜的东西补最贵的洞"**——过早终止是 C3 最致命的失败（Table 5 里 C3 的 premature termination 高达 18 次），而修它只需一句 40 词的摘要、+3.4% token。这是"少即是多"的完整形态：**不是一味地减，而是『减到只留决策相关的局部，再用一句话补回全局感』**。把这条抽象出来——**"局部完整 + 全局摘要"是一个可复用的上下文设计模式**，远比"酒店发票"这个具体任务有迁移价值（见 §15 Inspires-Us a）。

## §10　统计严谨性：作者怎么防"是不是运气好"

> 主讲提示：这页快讲，但要点出"它没有满足于报一个均值"——这正是它比很多同类工作诚实的地方。

作者把每个任务在主指标上的结果当成一次**伯努利试验**，用两种互补方式报散布（§4.6、Appendix D）：

- **Run-level（5 次运行）**：把每次运行当作"基准级成功率"的一次观测，报均值±样本标准差 + 95% 置信区间（Student-t，自由度 4，$t_{0.975,4}=2.776$）。
- **Pooled（汇总 50×5=250 个任务-运行）**：报 **Wilson 分数区间**（近 0/1 时比 Wald 近似更可靠）。

**Table 3**：

| 配置 | 5 次运行 均值±SD | Wilson 95% CI |
|---|---|---|
| C2 | 71.0 ± 4.4 | [65.1, 76.3] |
| C3 | 79.0 ± 8.2 | [73.5, 83.6] |
| C4 | 91.6 ± 1.7 | [87.5, 94.4] |

**读数（§4.6）**：C4 的 Wilson 区间 [87.5, 94.4] 与 C3 的 [73.5, 83.6] **完全不重叠**——"摘要改善性能"有强的汇总试验证据。C2 与 C3 区间略有重叠（[65.1,76.3] vs [73.5,83.6]），但 run-level 均值差 +8 点方向一致。一个诚实的观察：**C3 的运行间波动最大（±8.2）**——纯裁剪把 C2 吸收掉的方差**暴露了出来**；而 **C4 最稳（±1.7）**——摘要把这份方差又吸收回去。作者不做配对假设检验，理由是"效应量（+8、+12.6 点）相对每轮散布已经足够大 + C3→C4 的 Wilson 区间不重叠"，两者合起来使"排序是运气"不太可能（§4.6 "Effect sizes vs. noise"）。

**为什么偏偏 C3 方差最大（±8.2）——一个有意思的解读**：C3（纯裁剪）把 agent 卡在一个**微妙的临界点**上——它有干净的局部状态，但**没有全局进度感**，于是"会不会过早提交"高度依赖**每次运行里 agent 恰好在哪一步失去了『还差多少』的线索**。这是一个**接近阈值的、对随机性敏感的**行为，所以运行间抖得厉害。C2 的方差被"全上下文"掩盖（信息冗余到掩盖了随机性），C4 的方差被"那句进度摘要"钉住（每次都有稳定的全局锚点）。**换句话说：方差本身也在讲机制故事——最不稳的配置，恰是那个"信息刚好不够、全靠运气补齐"的配置。** 这也解释了为什么作者说"summarization absorbs the variance that pruning alone exposes"（§4.6）。

**统计方法为什么这么做（§4.6 直觉）**：为什么要 run-level 和 pooled **两种**区间？因为它们回答不同问题——run-level t 区间问"**如果我重跑 5 次实验，基准级成功率会落在哪**"（对"实验可复现性"负责）；pooled Wilson 问"**在这 250 个任务-运行的证据下，真实成功率的置信范围**"（对"单个任务的成功概率"负责，且近 0/1 时比常见的 Wald 正态近似更可靠）。两把尺子都指向同一结论（C4>C3>C2），才敢下强判断。

> **读出什么**：这套"run-level t + pooled Wilson 双报 + 明说不做 p 值检验并解释为什么"的做法，本身就是一份可抄的**小样本 agent 评测统计模板**（见 §15 Inspires-Us b）。它没有把 91.6% 当成一个孤零零的数字，而是给了它误差带和"是否稳"的判断。

## §11　敏感性 & 跨模型：N=5/W=3 是不是碰巧？换个模型还成立吗？

> 主讲提示：这页回答两个"最该被追问"的问题——超参是不是调出来的、结论是不是只对 GPT-5。作者都正面答了。

**(A) 超参敏感性（§4.7、Appendix H，Table 7）**：分别扫 N 和 W，另一个固定——

| 扫描 | 设置 | Comp.(%) | Tok(K) |
|---|---|---:|---:|
| 裁剪 N | N=3,W=0 | 74.0 | 425 |
|  | **N=5,W=0（C3）** | **79.0** | **535.3** |
|  | N=10,W=0 | 80.0 | 820 |
|  | N=∞（C2） | 71.0 | 1,481 |
| 摘要 W | N=5,W=1 | 86.4 | 540 |
|  | **N=5,W=3（C4）** | **91.6** | **553.4** |
|  | N=5,W=5 | 92.0 | 575 |
|  | 全历史摘要(W=−1) | 92.0 | 615 |

**读数（§4.7 "Interpretation"）**：完整录入率在 **N=5 附近见顶**——N=3 掉约 5 点（窗口太小，连"两个录入周期的工作记忆"都装不下），N=10 只多不到 1 点却多烧约 53% token（多留的 5 个工具对里，大多是**已过期的旧状态**，帮不上忙还占地方），而 N=∞(C2) 在两个轴上都严格更差。摘要侧在 **W=3 以上就平了**——W=5 和全历史摘要(W=−1) 只多 0.4 点却更贵，W=1 差约 5 点（摘要太短，进度感不足）。结论：**(N=5, W=3) 正好坐在两条曲线的"膝盖"上**——再多买不到性能、更小丢掉防过早终止的记账。

**这条"膝盖曲线"本身就是一个论点（对本组的意义）**：它不只是"调参调对了"，而是**对"多多益善"的定量反驳**——N 从 5 加到 10（上下文翻倍），完整率只从 79.0 动到 80.0（+1 点），token 却从 535K 涨到 820K（+53%）。**换句话说：在这个 regime 里，"给更多上下文"的边际收益几乎为零、边际成本却是线性的。** 这正是本篇对 AgentFold/ACON 那类"更聪明地保留更多"路线的隐含质问：如果连"简单地多留 5 个"都不值，那"复杂地多留很多"凭什么值？（§14/§15 展开）

> **Why（设计层）——recency 窗口的一个已知短板，及作者留的口子**：
> 固定 recency-N 有个明显盲区——**如果任务需要回头引用"很久以前"的某条工具结果**（它早被逐出、又没进那句摘要），recency 就丢了它。作者诚实地把这条列为**未来工作**：§3.7 提到一个正交方向 **importance-pruning**——保留"最近**被 agent 引用过**的 N 个工具对"，而非"时间上最近的 N 个"。本文没做（只在酒店这种"旧状态基本都过期、很少回头引用"的任务上，recency 已经够好），但明说它是"a candidate for future work"。**这正是我们可以接手的开放问题（§15 Inspires-Us c）。**

**(B) 跨模型（§4.9、Appendix I，Table 8）**：把整套换成 **Claude Sonnet 4.5** 当 agent（其余不变）——

| 配置（Sonnet 4.5） | Comp. | <10% | ≥1 | %Amt | Tok(K) | Time |
|---|---:|---:|---:|---:|---:|---:|
| 无上下文工程（全上下文） | 88.0 | 96.0 | 100.0 | 98.23 | 3,562 | 6.20 |
| 裁剪（Last-5 TC） | 92.0 | 96.5 | 100.0 | 98.78 | 2,161 | 10.70 |
| 裁剪+摘要 | 94.5 | 97.5 | 100.0 | 99.20 | 2,235 | 11.30 |

**两个发现（§4.9）**：**其一**，Sonnet 4.5 在非交互 harness 里**不卡死**——它的"无上下文工程"基线已到 **88.0%**（对比 GPT-5 C1 的 8.0%），这**反证** GPT-5 那个巨大的 C1→C2 gap 是**模型特异的停顿行为**，与上下文工程无关（呼应 §4 的隔离设计）。**其二**，**上下文工程的排序依旧成立**：加摘要把完成率从 92.0 抬到 94.5，只多约 5.6% 墙钟——和 GPT-5 上观察到的"摘要小幅时间开销"一致。

> **读出什么（Θ2）**：这就是 `Agent = Model + Harness` 的跨模型印证——**换了模型（Model），上下文策略（Harness 的 C 层）带来的相对排序不变**（裁剪>全上下文，裁剪+摘要>裁剪）。同时它诚实地指出**绝对水平和"需不需要 user model"是模型特异的**（GPT-5 需要、Sonnet 不需要）。一处小反差值得记：Sonnet 上裁剪后**时间反而变长**（6.20h→10.70h），token 却降了——原文未展开解释，我在 §13 批判里把它列为存疑点。

## §12　失败分类学：把"少即是多"拆成可验证的机制

> 主讲提示：这页是本篇的"显微镜"。它没有只报总数，而是把每一类失败按配置数出来，用来**验证"每个上下文策略到底修好了什么、又没修好什么"**。

作者给 C2–C4 的每一个"未完成"任务贴上六类失败标签之一（据工具调用错误日志 + "保存后表单 vs 真值"的回读差异，§5、Appendix C/F）。**Table 5（5 次运行汇总计数）**：

| 失败模式 | C2 | C3 | C4 | 含义 |
|---|---:|---:|---:|---|
| **Stale-state reference（引用过期状态）** | **34** | **6** | 4 | agent 依据已被取代的旧表单快照行动 |
| Wrong subcategory mapping（子类目映射错） | 8 | 9 | 6 | 把行项映到 D365 词表里的错类目 |
| Duplicate / skipped repeat item（重复/漏掉重复行） | 12 | 11 | 5 | 多晚住宿里同类目行重复或漏录 |
| **Premature termination（过早终止）** | 9 | **18** | **3** | 还没核销就提交、以为做完了 |
| Tool / form navigation error（工具/导航错） | 6 | 5 | 2 | 遇到枚举不匹配等报错后无法恢复 |
| Residual amount mismatch（残差不符/算术错） | 4 | 4 | 4 | 四舍五入/税行算错导致差几分钱 |
| **未完成总数** | **73** | **53** | **21** | C2→C4 未完成数 **−71%** |

**这张表怎么"证明机制"（§5 原文预测→验证）**：作者事先给出三条**可证伪的预测**，Table 5 逐条兑现——

1. **"全上下文(C2)会过度产生 stale-state 错误"** → 兑现：C2 的 34 次是压倒性主因；裁剪把它打到 C3 的 6 次（**47%→11%**）。**这直接坐实了 §8 "裁剪为什么反而更好" 的机制。**
2. **"纯裁剪(C3)会减少 stale-state，但引入过早终止（因为看不到运行余额了）"** → 兑现：C3 的 premature termination **飙到 18**（C2 只有 9），成为 C3 的头号失败。**这直接坐实了 §9 "摘要补回了什么" 的必要性。**
3. **"摘要(C4)会压住过早终止、又不重新引入 stale-state"** → 兑现：C4 把过早终止 **18→3**（六倍降），且不反弹 stale-state（6→4），未完成总数 **73→21，整体降 71%**。

**四条真实 trace（Appendix F 节选，让失败"看得见"）**——A=agent，T=工具响应：
- **Stale-state reference（C2）**：agent 先 `get_form_state()` 看到 `total_added:$180, remaining:$420`，加了 RoomTax(\$40)，又 `get_form_state()` 看到 `total_added:$220, remaining:$380`……几步之后它**引用了第一张快照**、又 `add_line(Room,$180)` 加了一条重复行 → 残差 **−\$180（超额，重复了 Room 行）**。**病因清清楚楚：它信了一张过时的表单读数。**
- **Premature termination（C3）**：agent `get_form_state()` 见 `total_added:$340, remaining:$240`，加了 Parking(\$45)；**裁剪窗口把那条"remaining:$240"的旧消息挤出去了**，于是 agent **再也看不到运行余额**，直接 `submit_form()` → 残差 **\$195 仍欠，表单不完整**。**这正是 §9 说的"C3 丢了全局进度感"的活标本。**
- **Tool / form navigation error（C2）**：agent `add_line(category=Hotel-Resort_Fee)` 触发 `ERROR: invalid category enum`（期望 Hotel-ResortFee/Hotel-Parking…），换了个写法又错，最后直接提交 → 残差 \$25，**枚举不匹配后无法恢复**。
- **Residual mismatch（C4）**：收据 \$580.42，agent 逐行加 Room/Tax/Parking 后 `submit_form()` → 残差 **\$0.37（税行算术错）**。**注意：这类算术错在 C4 依然发生**——摘要器"故意泛型、不算余额"，管不了这种细粒度算术。

**哪些失败上下文工程"管不了"（§5 诚实收尾）**：wrong subcategory mapping、duplicate/skipped、residual mismatch 这三类**大体与上下文策略无关**——它们反映的是**模型级的推理/工具绑定错误**（尤其子类目错，被 D365 那 23 项含近义词的词表放大：Room tax vs Non-Room tax、Restaurant vs Room service vs Loungebar，Appendix E）。这些"策略不变项"**框定了裁剪/摘要类方法的头顶天花板**：把 C4 的 21 个未完成拆开看，stale-state(4)+premature(3)+navigation(2)=9 是"上下文相关"的残余，而 mapping(6)+duplicate(5)+residual(4)=15 是"上下文无关"的——**说明再优化上下文策略，天花板也就在这 15 个附近**，要再往上得靠别的技术（§6 未来工作点名 importance-pruning）。

> **读出什么**：这是全篇方法论上最漂亮的一手——**先下预测（§5 明写三条"C2 应过量 stale-state / C3 应减 stale 但增 premature / C4 应压 premature 不反弹 stale"），再用分类学逐条验证**。这让"少即是多"不是一个总分上的巧合，而是"**裁剪堵住 A 通道(stale-state) → 副作用暴露 B 通道(premature termination) → 摘要再堵 B 且不重开 A**"的**可追溯、可证伪的因果链**。这套"失败分类学当作机制显微镜"的做法，和 Harness-Bench 的五类失败症状是同一种工具，可直接搬来体检我们自己的 agent（见 §15）。

## §12.5　跨类目泛化：换了发票类型，那条"结构梯度"预测也兑现

> 主讲提示：这页回答"是不是只对酒店发票成立"。作者不是简单地"再测两类都过了就完事"，而是**事先用结构复杂度预测了增益大小**，再验证——这比"泛化 OK"有力得多。

作者把策略搬到另两类发票——**Travel（Car rental + Flight）** 与 **Meals & Gifts（Business meal + Gift）**——它们与酒店在**结构复杂度**上不同（Hotel > Travel > Meals & Gifts）。**Table 4 主指标（Comp. Item.，5 运行均值）**：

| 类目 | n | C1(无user) | C2(全上下文) | C3(裁剪) | C4(裁剪+摘要) | C1→C2 增益 | C2→C4 增益 |
|---|---:|---:|---:|---:|---:|---:|---:|
| **Hotel** | 50 | 8.0 | 71.0 | 79.0 | **91.6** | +63.0 | +20.6 |
| **Travel** | 30 | 20.0 | 76.0 | 86.6 | **95.0** | +56.0 | +19.0 |
| **Meals & Gifts** | 32 | 26.9 | 75.6 | 89.4 | **96.1** | +48.7 | +20.5 |

**两条被兑现的预测（§4.8 原文）**：
1. **C1→C2→C3→C4 单调上升在每一类都成立**，且 **C2→C4 的增益跨类目高度一致（+19.0 ~ +20.6 点）**——说明"裁+摘"的收益**不挑发票类型**。
2. **结构梯度预测**：C1(无 user model) 的完成率**随复杂度降低而升高**（Hotel 8.0 < Travel 20.0 < Meals 26.9）——因为**越简单的发票，给"无督导 agent"留的『卡住』机会越少**（§4.8、Appendix E 的 "structural gradient" 论证）。反过来，**越复杂的发票，C1→C2 的 gap 越大**（Hotel +63.0 > Travel +56.0 > Meals +48.7）。这条"gap 随复杂度收缩"的定量规律被精确预测并命中，是对"机制解释"的强佐证。

> **读出什么**：泛化实验的"含金量"不在"又测了两类也过了"，而在**它先用一个结构性变量（复杂度）预测了增益的『方向和大小』，再验证**。这把"我们的方法能泛化"从一句空话变成了"**我们理解为什么它在何处增益多、何处增益少**"——这正是 §14 regime 诚实的量化底座。

## §13　局限与批判（论文 §6.4 + 我的补充）

> 主讲提示：这页别护短。本篇最大的价值恰恰是"它敢说自己不适用于哪里"——把这份诚实讲足，就是本组最需要的判断力。

**论文自陈的局限（§6.4、Abstract、§6 Generalizability，非常克制）**：

- **只测了一类工作流**：酒店 expense itemization（虽扩到 Travel、Meals & Gifts 两类和 Sonnet 4.5，§4.8/4.9）。作者反复声明这是"**one class** of enterprise tool-use workflow 的强证据，**不是普适泛化的证明**"。
- **(N=5, W=3) 是为"比较清晰"选的单点**；联合调参、按任务自适应窗口是未来工作。
- **摘要器是单次一遍式**；结构化/学习式压缩器（如 ACON [9]）、厂商原生 compaction [4] 可以和它做 head-to-head，本文没做。
- **有一整类失败它治不了**：wrong subcategory / duplicate / residual mismatch 是模型级错误，上下文工程不针对（§5）——需要 importance-pruning（按"最近被引用"而非"最近时间"保留，§3.7 提到但列为未来工作）等互补技术。

**我的补充批判**：

- **规模有点小**：50 个任务 × 5 次运行。虽然统计做得规矩（Wilson/t 双报），但**50 个任务上 8 点的差**、以及"扩展类目"里 Travel/Meals 也各只有 30/32 任务（Table 4），外推到"整个企业 agent 世界"仍需更大样本。
- **"user model 作为陪跑"是把双刃剑**：C2–C4 里那个 GPT-4.1 扮的 user 会**主动提示 agent"该核对了/该继续了"**（§3.3）。这意味着**被测系统里混进了一个"隐性督导"**——C4 的高分里有多少来自"好的上下文"、多少来自"user model 的助推"，论文没有进一步消融（它只隔离了 C1 vs C2–C4，没隔离"user model 助推强度"）。这是"谁在真正把任务推完"的未解问题。
- **Sonnet 上"裁剪后时间反而变长"未解释**（§11：6.20h→10.70h，token 却降）。可能是裁剪后 agent 多走了几轮工具，但原文未给 turns 数据佐证，是个悬着的反常点。
- **摘要器本身也是个会出错的 LLM**：C4 的残差/重复类失败里，有多少是"摘要漏了或写错了"导致的？Appendix F 只给了一条"看起来对"的摘要样例，没有摘要错误率的统计。
- **残余错误率的生产含义（作者自己也点了，§6.5）**：即便 C4 到 91.6%，**仍有 8.4% 的任务未完整核销**——作者诚实地说"生产部署应对被标记的 case 保留人工复核"。这意味着"少即是多"**降低了成本、没有消除风险**：它把 agent 从"71% 且贵"推到"91.6% 且省"，但**最后那一段（尤其是 §12 那 15 个"上下文无关"的模型级错误）还得靠人或别的技术兜底**。别把 91.6% 读成"可以无人值守"。

## §14　regime 诚实：它在哪成立、在哪不成立（Θ5，本篇的灵魂）

> 主讲提示：这页是本组的判断力高地，也是本篇最该被学的地方。**把它的适用边界画到最清楚**，让全组以后引用它时不会外推过头。

本篇是"**regime 诚实的教科书**"——它主动、反复地把自己的适用边界钉死（Abstract、§2、§6.4、§6 Generalizability）。诚实的表述如下：

**它明确成立（"少即是多"为真）的 regime——四个条件同时满足**：
1. **工具重（tool-heavy）**：工具响应啰嗦（本例 500–3,000 token/条），上下文主要被"工具返回的状态快照"占据。
2. **单会话（single-session）**：任务在一次会话内完成，**不需要跨会话记住用户/历史事实**。
3. **状态会过期（stale-state-prone）**：早期工具返回描述的状态会被后续操作取代——**旧上下文不是"信息"而是"过期噪声"**。
4. **有硬完成判据（hard completion criterion）**：成功能被客观机器验证（本例：残差=0）。

**它明确不声称成立的 regime（作者自己划出去的）**：
- **跨会话的"事实召回"任务**：需要把某条事实**长期、跨会话**记住并检索——这正是 **MemoryBank/LongMem** 那条外部记忆路线要解的（§2）。本篇**不与它们竞争**，只说"我这个 regime 不需要它们"。
- **旧上下文本身就是"资产"的任务**：如果历史里的早期信息**长期相关**（不会过期），那么裁剪会真的丢信息——本篇的"少即是多"就**不再成立**。
- **对 GPT-5 之外绝对水平/是否需 user model**：这些是**模型特异**的（§4.9 用 Sonnet 4.5 亲自反证）。

**一句话把"工作记忆 vs 事实召回"这把尺子说透（本篇题眼）**：本篇治的是"**工作记忆被过期状态淹没**"（working-memory / stale-state 问题），**不是**"某条事实想不起来"（factual-recall 问题）。**判断一个任务该不该上复杂记忆库/检索器，就问一句：我需要的是『别被旧状态带偏』（→ 用本篇的裁+摘），还是『把某条事实长期记住并找回』（→ 才需要外部记忆/检索）？** 这条区分，就是本篇留给全组最有价值的思维工具。

**做成一张可操作的判别表（把题眼变成流程）**：

| 问题 | 若"是" | 若"否" |
|---|---|---|
| ① 旧的历史信息会**过期/被取代**吗？ | 偏"工作记忆"问题 → 继续问② | 偏"事实召回"问题 → **考虑外部记忆/检索**（MemoryBank/MemGPT/A-MEM 路线） |
| ② 任务在**一次会话**内完成吗？ | → 继续问③ | 跨会话 → 需要**持久化记忆**，本篇不适用 |
| ③ 工具返回**啰嗦**（大块结构化状态）吗？ | → 继续问④ | 上下文压力小 → 可能**无需任何策略**（全上下文即可） |
| ④ 有**客观硬判据**能验证完成吗？ | ✅ **四条全中 → 用本篇：recency-N + 一句摘要，别上重记忆** | 判据软 → 结论的可信度下降，需谨慎 |

**这张表的用法**：从①走到④，**四条全"是"才落到本篇的"少即是多"**；任何一条"否"都把你**推向别的路线**。它把 §14 的 regime 边界从"一段话"变成"一个可以照着走的判断流程"——这正是我们（§15/§16）要拿去给自己 harness 分流用的东西。

> **读出什么（与本组"多存派"并读）**：把本篇和 MemGPT/A-MEM/AgentFold 并排，得到的**不是"谁对谁错"，而是一张 regime 地图**——
> - 任务**工具重+单会话+状态会过期** → 本篇赢（少即是多，别上重记忆）。
> - 任务**跨会话+要事实召回+旧信息长期相关** → 多存派赢（MemGPT/A-MEM 的结构化记忆有价值）。
>
> 全组以后引用本篇时，**必须带着这张地图**——脱离 regime 说"少即是多"或"多多益善"都是错的。这正是 Θ5 要的判断力：**别把任一侧写成绝对真理。**

## §17　一页速记

- **论点（反潮流）**：对**工具重 + 单会话 + 状态会过期 + 有硬判据**的企业工作流，**"最近 5 个工具对 + 一句滚动摘要"就够**——不需要外部记忆库/检索器。**少即是多。**
- **做法**：纯 C 层。`CONSTRUCTCONTEXT`（Algorithm 1）= 按"工具对"成对逐出最近 N 之外的历史 + 对逐出内容维护**单条**摘要（W=3）。inference-time、零训练、可移植。
- **任务**：D365 F&O 酒店发票逐条录入（经 MCP），50 任务 × 5 运行，硬判据=残差恰为 \$0.00，独立回读打分。
- **铁证（同一 GPT-5，只换上下文）**：全上下文 **71.0% / 1,481K tok / 14.56h** → 裁剪 **79.0% / 535K(−63.9%) / 5.39h** → 裁剪+摘要 **91.6% / 553K / 5.79h**（Table 2）。摘要只 +3.4% token 却 +12.6 点。
- **机制（可验证）**：裁剪堵住 **stale-state 引用**（34→6，Table 5）→ 但暴露 **过早终止**（9→18）→ 摘要再堵过早终止（18→3），未完成总数 **73→21（−71%）**。
- **统计**：run-level t + pooled Wilson 双报；C4[87.5,94.4] 与 C3[73.5,83.6] 不重叠。超参 (N=5,W=3) 坐在两条曲线膝盖上（Table 7）。
- **跨模型**：Sonnet 4.5 排序不变（88.0→92.0→94.5，Table 8），且**不需要 user model**——反证 GPT-5 的 C1→C2 大 gap 是模型特异停顿，非上下文工程之功。
- **regime 诚实（灵魂）**：只声称"**one class** of workflow"；**不与 MemoryBank/MemGPT/A-MEM 的"事实召回/跨会话记忆"竞争**——它治"工作记忆/陈旧状态"，不治"事实召回"。**这把尺子是它给全组最大的礼物。**
- **对我们**：把 `CONSTRUCTCONTEXT` 搬进我们上下文压缩层做 A/B（先只对工具重单会话子集启用）；把双区间统计+6 类失败标注接进 m9.6；下一步实现 importance-pruning。**先判断任务属工作记忆还是事实召回，再决定上不上重记忆。**
