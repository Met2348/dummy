# Inner Monologue：用文本反馈给具身 LLM 规划闭环

> **本篇属 agent-harness 库 B 组（控制循环）、L 层 canon。** 它回答一个今天看来理所当然、但 2022 年才被系统论证的问题：
> **LLM 规划器该不该、能不能把"环境反馈"读进来，边做边改？** Inner Monologue 的答案是——把成功/失败、场景描述、人类回答都**转成自然语言塞回 prompt**，
> 让语言模型自己"内心独白"式地消化反馈、决定下一步。这正是我们自己 harness 里 ReAct 循环"工具→观察→再思考"的**具身版原型**，
> 也是 Harness-Bench 用来度量 harness 优劣的核心抽象 **execution alignment**（推理有没有被反馈忠实修正）的**历史源头**。

## §1　TL;DR（一页讲清这篇在干嘛）

> 主讲提示：开场先把"开环 vs 闭环"的对立摆出来，再点明这篇打的是 L 层，且是 canon。

**一句话**：给"用 LLM 做高层规划"的具身 agent 装上**闭环**——不再是一次性吐出整个计划然后盲目执行，而是**每执行一步，就把环境反馈（这步成没成功？现在场景里有什么？人类怎么说？）用自然语言写回 prompt**，让 LLM 读着反馈接着规划。作者把这个"行动—反馈—再规划"的循环叫 **Inner Monologue（内心独白）**（Abstract；§3.2）。

- **打 harness 的哪一层（Θ1）**：**L 层（Loop / 控制循环）**。这篇不造新工具、不改上下文压缩，它动的是**控制循环的形状**——把开环（open-loop）的"规划一次→执行到底"改造成闭环（closed-loop）的"规划→执行一步→读反馈→再规划"。它对 **T 层（工具/技能，π_k）** 和 **O 层（可观测，成功检测器、场景描述器）** 有强依赖：反馈从哪来，决定了循环能闭多紧。
- **回扣全库论点（Θ2）**：`Agent = Model + Harness`。本篇是一个**极干净的"只改 harness、模型冻结"消融**——LLM（InstructGPT-1.3B / PALM-540B）**完全不微调**，只靠 few-shot prompt；变的只是"反馈要不要回注、回注哪几种"。结果：真实厨房里加对抗扰动时，无闭环的 SayCan 抽屉任务成功率≈**0%**，加了闭环反馈的 Inner Monologue 拉到 **44.4%**、移动操作从 0%→**75%**（Table 3）。**同一个模型，装不装反馈闭环，差出一个"能用/不能用"。**
- **canon 坐标（Θ4）**：**2022 CoRL 奠基作**，出自 Google Robotics / Everyday Robots，是 SayCan（affordance grounding）的**闭环后继**、ReAct（同为 2022）的**具身孪生**。它奠基了"把非文本环境信号翻译成语言、回灌进 LLM 规划器"这一范式，后续大量具身 agent（PaLM-E、VoxPoser、语言反馈类 agent）在它上面长肉。

## §2　问题与动机：为什么"开环规划"在真实世界会崩

> 主讲提示：这一段用 Why 三连的"问题层"，把"一次性生成计划"的脆弱讲透。这也是全库 L 层的立命之本。

**Why（问题层）——不解决会卡住什么？**
智能、灵活的具身交互，要求机器人能**按情况把基本技能排成长程序列，并在某个行为失败时识别出来、换个招**（§1 首段原文）。而 2022 年主流的"LLM 当规划器"做法（如 Huang et al. 的 zero-shot planner [20]、SayCan [21]）有一个共同的**结构缺陷**：它们是**单向、开环**的——

> 原文点名（§3.2）：*"LLMs used in this context have thus far remained one-directional — providing a list of skills, without making corrections or leveraging opportunities to replan accordingly."*
> （此前 LLM 只是**单向**吐一串技能，既不纠错，也不利用重规划的机会。）

后果很直接（§2 "Task Planning with Language Models" 原文批评 [20,21]）：这类方法**假设每一步都会被成功执行**，因此"在动态环境里、或低层策略较弱时，可能扛不住中途失败（intermediate failures）"。真机上失败是常态——抓空了、物体被遮挡没进初始计划、策略被噪声带偏——开环计划一旦某步挂了，后面全错，且**无法恢复**。

**这就是 L 层（控制循环）存在的理由**：一个 agent 的鲁棒性，不只取决于模型多聪明（能不能生成一个好计划），更取决于**循环能不能吸收失败、就地重规划**。

> **读出什么**：这篇的动机不是"造一个更强的规划器"，而是"给规划器接上**感觉神经**"——让它感知到自己刚才做的事成没成，再决定下一步。这与我们自己 harness 里 ReAct 的核心信念完全同源：**没有观察回灌的行动是盲目的**。

## §3　核心概念：什么是 Inner Monologue

> 主讲提示：这页给全篇的定义骨架。先直觉、再定符号、最后"读出什么"。

**直觉**：想象你要开一扇锁着的门（§1 用的正是这个例子）——"我得先拿钥匙 → 试着插进锁孔 → *不对，插不进* → 我换一把再试 → 这把行了，转开。" 你脑子里这串**夹杂着"观察结果"和"临场纠正"的自言自语**，就是 inner monologue。作者主张：**把这种'边做边碎碎念'搬给 LLM——只要能把环境反馈写成语言，LLM 就能像人一样在独白里消化它、纠正计划。**

**问题设定与符号（§3.1，先定义后用）**：
- **指令 $i$**：一条高层自然语言指令（如 "把桌上的饮料拿给我"）。
- **技能库 $\Pi$ 与技能 $\pi_k \in \Pi$**：一组**短程**、预训练好的低层策略（RL 或行为克隆得到），每个技能带一句**语言描述 $\ell_k$**（如 "pick up the coke"）。**关键前提：LLM 本身不动手，它只会从 $\{\ell_k\}$ 里挑技能名。**
- **规划器 = 一个预训练 LLM**：负责把 $i$ 分解成一串可执行技能。
- **观察 / 反馈 $o$**：可被追加进 prompt、或由规划器主动索取的**文本**环境反馈——可以是成功检测、物体识别、场景描述、VQA、乃至人类回答。

**Inner Monologue 的形式化操作（§3.2）**：**不断把来自各反馈源的信息注入 LLM 的规划语言 prompt**，随机器人与环境交互而更新。注意作者的一个诚实边界——

> 原文（§3.2）：*"our aim is to provide a case study on the value of incorporating different types of feedback into closed-loop LLM-based planning."* 且明确：**不绑定某一种"LLM 与低层控制融合"的具体算法，也不绑定某一种"从环境抽取反馈"的具体方法**——它是一个**框架/范式**，在三个不同系统里各自实例化（§4）。所有实例的 LLM **都不微调，只 few-shot prompt**。

> **读出什么（Θ1）**：Inner Monologue **不是一个模型、也不是一个工具，而是一种控制循环的组织方式**——把"感知→语言→再规划"焊进循环。这就是它坐 **L 层**的原因：它规定的是"反馈如何在循环里流动"，而把"反馈从哪来"（O 层）和"手怎么动"（T 层）当作可插拔的黑盒。

## §4　三类文本反馈：这篇的"输入词表"（§3.3, Fig 2）

> 主讲提示：这是本篇的分类学核心。三类反馈 = 三种把世界翻译成语言的方式。每类都先给直觉，再说它答什么问题。

Inner Monologue 把"能回灌的环境反馈"分成三类（Fig 2 三栏图对应）：

| 反馈类型 | 触发方式 | 答什么问题 | 典型实现 |
|---|---|---|---|
| **Success Detection（成功检测）** | 每步执行后自动 | "我刚才这步**成了没有**？" | 二分类器（真机上可学习：foresight/hindsight 检测器，见 §9） |
| **Passive Scene Description（被动场景描述）** | 每步自动、无需询问 | "现在场景里**有什么 / 进度到哪**？" | 物体识别（*Object*，如 MDETR/ViLD）；任务进度子目标（*Scene*） |
| **Active Scene Description（主动场景描述）** | LLM **主动发问**才触发 | "关于场景/任务/人的偏好，我**想问一个具体问题**" | VQA 模型答，或**人类**答（本文只做人类答，记作 *Human* 反馈） |

**逐类直觉**：
- **Success（成功检测）**：最基础的闭环信号——一步做完给个 True/False。它让 agent 知道"要不要重试这一步"。作者把检测器输出**转成语言**（如 "Action was successful." / "not successful."）再回灌（§3.3）。
- **Passive Scene（被动场景）**：*Object* 告诉规划器"场景里有哪些物体"（防止把没看见的物体漏出计划）；*Scene* 在仿真里进一步给"任务进度型子目标描述"（哪些子目标已达成），让规划器在**组合爆炸**的多步任务里跟得住全局进度。**被动 = 每步都自动喂，不用问。**
- **Active Scene（主动场景）**：前两类都是"结构化、窄"的；主动场景是**开放式**的——LLM 自己决定"要不要问、问什么"，人类给**开放式**回答（可以是确认事实、也可以是修改计划、甚至是"我改主意了"）。本文只研究**人类**作为回答者（记作 *Human* 反馈）。

> **Why（设计层）——为什么要区分"被动"与"主动"，而不是只用一种？**
> 朴素做法一：**全被动**（每步把场景一股脑塞进去）。→ 信息虽全，但**冗余、且无法针对性获取"prompt 里没有的关键信息"**（如人的偏好、模糊指代的消歧）。
> 朴素做法二：**全主动**（凡事都问）。→ 打断人、低效，且很多信息（成功与否、场景物体）本可自动获得，不必问。
> 本文的两分法让 agent **该自动的自动、该问的才问**：结构化窄反馈走被动通道，开放式/主观信息走主动通道。这正对应我们 harness 里"工具观察自动回灌 vs 主动向用户澄清"的分工。

> **读出什么（Θ3 预告）**：这张三类表，本质上就是一份 **"harness 该给规划器接哪些反馈线"的清单**。我们自己的 agent 也正好有这三根线：工具执行结果（≈Success）、环境/文件状态回读（≈Passive Scene）、向用户提问（≈Active/Human）。见 §14 Inspires-Us。

## §5　系统总览：一个循环、三处实例化（Fig 1 & Fig 3）

> 主讲提示：这页讲"同一个循环骨架，怎么长在三个不同的机器人系统上"。强调"骨架不变、零件可换"。

**闭环骨架（Fig 1）**，用一句话串起来：人给指令 → LLM 规划器出**下一步技能** → 低层策略 $\pi_k$ 执行 → 各反馈源把结果**写成语言**追加进 prompt → LLM 读着新 prompt 出**再下一步**（或宣布 "done" / 主动发问）。图 1 的例子极典型：

```
Human: 把桌上的饮料拿给我
Robot: "go to the table"                          ← LLM 出技能
Scene Descriptor: "I see: coke, water, chocolate bar."  ← 被动场景反馈回灌
Robot: "Do you want water or coke?"               ← LLM 主动发问
Human: "Coke please."                             ← Human 反馈回灌
Robot: "pick up the coke"
Success Detector: "Action was not successful."    ← 成功检测回灌（失败！）
Robot: "pick up the coke"                         ← 读到失败 → 重试同一步
Success Detector: "Action was successful."        ← 成功
Robot: "bring it to you"
```

**三处实例化（Fig 3 + 附录 Table 4）**——同一循环，换不同 LLM、不同技能、不同反馈源：

| 维度 | ① 仿真桌面整理 | ② 真机桌面整理 | ③ 真机厨房移动操作 |
|---|---|---|---|
| **LLM（冻结）** | InstructGPT-1.3B | InstructGPT-1.3B | **PALM-540B** |
| **低层技能** | CLIPort + Transporter Nets | MDETR + 启发式抓放 | SayCan 的学习/脚本策略 |
| **Success 反馈** | ground-truth + 启发式 | MDETR + 启发式 | **微调 CLIP**（可学习检测器） |
| **Object 反馈** | ground-truth + 启发式 | MDETR | 人类提供（结构化） |
| **Scene 反馈** | ground-truth + 启发式 | — | — |
| **Human 反馈** | — | — | 人类提供（非结构化） |

> **读出什么（Θ2）**：这张表本身就是 `Agent = Model + Harness` 的活标本——**LLM 从 1.3B 到 540B 变了，反馈源的实现从"仿真真值"到"学习检测器"变了，但让系统 work 的那个"闭环回灌"骨架没变**。作者反复强调 Inner Monologue "不依赖某个特定 LLM 或某种 grounding 反馈"（§4 首段）——即：**能力来自 harness 结构（闭环），可跨模型/跨实现迁移。**

## §6　实验一：仿真桌面整理（§4.1, Table 1）

> 主讲提示：这页的看点是"闭环把泛化能力从 0 拉起来"。先报数字，再解释机制。

**设置（§4.1）**：Ravens 仿真里的积木-碗整理任务；4 个 seen + 4 个 unseen 任务；测试时注入**高斯扰动**（像素观测 $\mathcal N(0,0.5)$、pick-place 热图 $\mathcal N(0,2.5)$、放置坐标 $\mathcal N(0,0.02m)$）以模拟真实干扰。上限 $k{=}15$ 步。对比对象：多任务 **CLIPort**（不用 LLM 规划的单步策略基线，+oracle 表示给了"终止" oracle）与 Inner Monologue 的三个变体（*Object* / *Object+Success* / *Object+Scene*）。

**Table 1 节选（50 episodes 平均成功率 %）**：

| | 任务 | CLIPort | +oracle | IM: *Object* | *Object+Success* | *Object+Scene* |
|---|---|---:|---:|---:|---:|---:|
| **Seen** | "Pick and place" | 24.0 | 74.0 | 80.0 | 90.0 | **94.0** |
| | "Stack all the blocks" | 2.0 | 32.0 | 4.0 | 10.0 | **26.0** |
| | "Put all blocks in the [x] bowl" | 32.0 | 94.0 | 52.0 | 46.0 | **56.0** |
| **Unseen** | "Put blocks in their matching bowls" | 0.0 | 0.0 | 56.0 | 70.0 | **82.0** |
| | "Put blocks on mismatched bowls" | 0.0 | 0.0 | 62.0 | 76.0 | **86.0** |
| | "Put blocks in different corners" | 0.0 | 0.0 | 20.0 | 20.0 | **26.0** |

**Why（结果层）——为什么 Object+Scene 全面最好、且 unseen 上把 0% 拉到 80%+？**
两个机制叠加：
1. **闭环让它能从扰动导致的失败里重试/重规划**——CLIPort 是单步、不会自发终止、也不感知"这步没成"，扰动一来就崩（unseen 全 0，且 seen 也要靠 oracle 才能终止）。
2. **Scene 反馈（任务进度子目标）让它在组合爆炸里跟住全局**——"哪些子目标已达成"每步回灌，规划器不会重复已完成的、也不会漏掉未完成的。作者补充：*Object+Scene* 变体因为推理更复杂，**加一段 chain-of-thought 总结（"Robot thought: ..."）能提升"推断目标 vs 已达成目标"的一致性**（§4.1 原文）。
3. **泛化红利来自冻结的预训练 LLM**——unseen 任务 CLIPort 需要重训才可能会，而 LLM 靠语义知识 zero-shot 迁移（§4.1 Analysis 原文："performance can be directly translated to unseen tasks without further training"）。

> **读出什么**：这一版最漂亮的对照是 **unseen 那几行的 `0.0 → 82.0`**——不是模型换了，而是**把"规划+闭环反馈"这层 harness 加上去**，泛化能力凭空出现。这是 L 层价值的第一记实锤。

## §7　实验二：真机桌面整理（§4.2, Table 2）

> 主讲提示：这页把结论从仿真搬到真机噪声下。看点：两种反馈"互补"。

**设置（§4.2）**：UR5e 机械臂 + 腕载 RealSense；两个任务族——3 积木堆叠（2 个已预堆、需补第 3 个）、按类别把食物/瓶子分盘。**故意注入高斯噪声**（$\sigma{=}4\text{mm}$，clip 到 $2\sigma$）压测失败恢复。对比：*Object*（开环，只在开头跑一次物体识别）vs *Object+Success* vs *Object+Success* 的组合。

**Table 2（真机，10 次运行平均成功率 %）**：

| 任务族 | LLM *Object*（开环） | +IM *Object* | +IM *Success* | +IM *Object+Success* |
|---|---:|---:|---:|---:|
| 3 积木堆叠 | 20 | 40 | 40 | **100** |
| 瓶装食物分类 | 20 | 50 | 40 | **80** |
| **合计** | **20** | **45** | **40** | **90** |

**Why（结果层）——为什么两个反馈缺一不可、合起来 20%→90%？**
作者拆得很清楚（§4.2 Analysis）：
- **只 Object 开环的致命伤**：3 积木任务里**底层积木初始被遮挡**，开头那次场景描述看不全，规划器只会做一次 pick-place 就以为完事，**无法从"其实还没堆完"里恢复**（这正是 Harness-Bench 里 "artifact-commit / state-continuation" 类失败的具身版）。→ **闭环场景描述**让它每步重看、把新暴露的物体纳入计划。
- **只缺 Success 的致命伤**：噪声让 pick-place 失败时，没有成功检测就**不会重试**。→ **成功检测**专门补"失败重试"这条线。
- 二者**互补**（complementary）：Object 管"看全场景、别漏物体"，Success 管"这步没成就重来"。合起来才 90%。

> **读出什么**：这页把 §4 的三类反馈落到"每一类各堵哪种失败"——**Object 堵'漏看/状态跟丢'，Success 堵'失败不重试'**。这是一张可直接迁移的"反馈-失败对照表"。

## §8　实验三：真机厨房移动操作（§4.3, Table 3, Fig 4）—— canon 的高光

> 主讲提示：这是全篇最有说服力的一页，也是"闭环 vs 开环"差距最大的一页。基线是同门 SayCan（开环）。

**设置（§4.3）**：Everyday Robots 移动机械臂，办公厨房，5 个命名位置 × 15 个物体；LLM 用 **PALM-540B**；低层用 SayCan 的策略 + affordance 值函数 grounding；**Success 用学习的 CLIP 检测器**、Object 用人类提供。120 次评测覆盖 3 个任务族（操作 / 抽屉灵巧操作 / 长程移动操作）。**关键压测**：加**对抗扰动**故意让策略执行失败（有时重试即可，有时需换新技能重规划）。基线 **SayCan = 开环**（有 affordance grounding，但**反馈不回传给语言模型**——§4.3 原文点明 "affordance functions ... which do not communicate back to the language model"）。

**Table 3（120 次评测平均成功率 %）**：

| 任务族 | SayCan（开环） | IM *Success* | IM *Object+Success* |
|---|---:|---:|---:|
| **无扰动** 操作 | 50.0 | 62.5 | **75.0** |
| 无扰动 移动操作 | 50.0 | 50.0 | **75.0** |
| 无扰动 抽屉 | 83.3 | 83.3 | **100.0** |
| **有扰动** 操作 | 12.5 | 25.0 | **33.3** |
| 有扰动 移动操作 | **0.0** | 25.0 | **75.0** |
| 有扰动 抽屉 | **0.0** | **44.4** | 44.4 |
| **合计** | **30.8** | 48.7 | **60.4** |

**Why（结果层）——为什么有扰动时 SayCan 掉到≈0、而 Inner Monologue 还能站住？**
机制上一句话：**SayCan 没有"高层重试/重规划"行为**（§4.3 Analysis 原文："Without any LLM-informed feedback SayCan has success rate close to 0% since it does not have explicitly high-level retry behavior"）。一旦被扰动逼到策略失败，开环计划无路可退。Inner Monologue 因为**把失败/场景写回 LLM**，能"**invoke appropriate recovery modes depending on the environment feedback**"——重试、或换一个技能。Fig 4 的失败归因进一步显示：**只有 Inner Monologue 变体在加扰动（红）后仍能持续完成指令**；且 Success + Object 反馈能有效**减少 LLM 规划失败**这一类错误（代价是引入一些新失败模式，§4.3 结尾诚实指出）。

> **读出什么（Θ2 实锤）**：**同一套低层策略、同一个 PALM-540B、同一批任务**——唯一区别是"反馈回不回灌给语言模型"。SayCan（开环）有扰动移动操作 **0.0%** → Inner Monologue（闭环）**75.0%**。这是本库 `Agent = Model + Harness` 里**最干净的一次"只换 L 层"消融**：能力的天壤之别，全来自控制循环的形状，而非模型。

## §9　真机反馈是怎么"学"出来的（附录 A.3, Fig 7, Table 5）

> 主讲提示：这页补上"反馈从哪来"的工程实感——反馈不是白来的，O 层要投入。别把"oracle 反馈"当理所当然。

前面的实验里，最理想的反馈来自"真值/人类"。作者在附录里**认真研究了把反馈换成学习模型**的可行性（这决定了系统能不能真自动化）：

- **学习的成功检测器（Fig 7）**：训了两个互补模型。**Foresight**——输入初始观测 $o_0$、终止观测 $o_f$、技能 $\ell_k$，用 CLIP 图像编码器 embed 两帧、融合后过 MLP，输出"这次成功的概率"（二元交叉熵训练，CLIP 一并微调），推理时概率低于阈值就输出 `[success: no]`。**Hindsight**——给首末观测，输出"在所有技能上最可能达成了哪个"的分布（CLIP 式对称对比损失），用来**压低 foresight 的假阳性**。二者用阈值 $\tau$ 串联：foresight 说成、且 hindsight 的 argmax 正是目标技能，才判成功。
- **学习的物体识别（Table 5，10 个代表性厨房 episode 上）**：

| | ViLD | MDETR |
|---|---:|---:|
| Precision | 85.7% | 39.6% |
| Recall | 72.0% | 87.5% |
| Accuracy | **88.9%** | 68.2% |

> **Why（设计层）——为什么要 foresight + hindsight 两个检测器，而不是一个？**
> 朴素做法是只训一个"成功/失败"二分类器。→ 它的**假阳性**（把没成的判成成了）会让 agent 误以为完事、提前收手；**假阴性**（把成了的判成没成）则触发无谓重试。作者用 hindsight"这到底像哪个技能的结果"去**交叉验证** foresight 的判断，专治假阳性——这是一种"双检测器互检"的可靠性设计。

> **读出什么（Θ5 诚实）**：ViLD **仍有 28% 的物体漏检**（§附录原文），学习的反馈**明显不如人类/真值**。作者由此把主实验的漂亮数字**诚实地限定**在"较理想反馈"下（§5 Limitations：主实验"assume access to oracle scene descriptors ... or scripted systems"）。**这条边界很重要——它提醒我们：闭环的收益上限，受制于 O 层反馈的质量。**

## §10　涌现能力：闭环反馈"顺手"带出的四种行为（§4.4, Fig 5）

> 主讲提示：这页是"意外之喜"，但要讲清它们都是**从反馈闭环里自然长出来**、并非专门 prompt 教的。

作者发现，一旦把环境反馈接进冻结 LLM，会**涌现**出 prompt 里没示范过的行为（§4.4）：

- **a. 持续适应新指令（Continued Adaptation）**：人类在任务**中途改高层目标**（Fig 5a：先改一次目标、又说"先完成上一个任务"——连改两次），规划器能正确切换；甚至没被教过，人说 "please stop" 它也会 generalize 出 "done"。
- **b. 不可行时自提新目标（Self-Proposing Goals under Infeasibility）**：任务 "把任意两个积木放进紫碗"，紫积木**故意做得太重抓不起**；给一句提示 "the purple block is too heavy" 后，规划器**自己提出 "find a lighter block"** 并最终解决（Fig 5b）。
- **c. 多语言交互（Multilingual）**：人类用**中文**下新指令，PALM **能正确理解、用英文重述成具体目标**并据此重规划（Fig 5c）；作者说偶尔连符号/emoji 都能处理。
- **d. 交互式场景理解（Interactive Scene Understanding）**：任务执行完后**追问关于场景的问题**（prompt 里没这种结构），它能**结合过去的动作与反馈**答对需要时序+具身推理的问题（Fig 5d）。

另有两条鲁棒性（§4.4 末 + 附录 Fig 9）：**对反馈顺序鲁棒**（把新的人类指令插在 prompt 中间、不按训练约定的 `[action, scene, thought]` 顺序，仍能识别并生成新的 "Goal state is..."）、**对错别字鲁棒**（继承自 LLM 底座）。

> **Why（设计层 / 机制）——这些能力为什么"免费"出现？**
> 因为**反馈是用自然语言回灌的**，而底座是一个**通用语言模型**。于是"改目标""消歧""换语言""答问"这些**本就属于语言模型强项**的能力，一旦被放进"能读到环境反馈"的循环里，就**自动可用**了——无需为每种能力单独训练或 prompt。这正是"**用语言当统一接口**"（§3.2 引 Socratic Models [19]）的复利。

> **读出什么（Θ5 诚实）**：作者没吹过头——明确说这些涌现能力在"没有相似示例时**一致性参差**，受当前 LLM 能力所限"（§4.4 末原文 "varying levels of consistency"）。这是 canon 的克制。

## §11　与前作的精确差分：ReAct / SayCan / Huang-planner（版图定位前置）

> 主讲提示：这页帮听众把三篇 2022 名作在脑子里排好队。一句差分讲清各自补了什么。

- **vs Huang et al. zero-shot planner [20]**：那篇**开环**——一次性把指令翻成技能序列，不读执行反馈。Inner Monologue = 给它**接上反馈闭环**。
- **vs SayCan [21]**：SayCan 用 **affordance 值函数**给动作"可行性"打分做 grounding，但**这些分数不回传给语言模型**（§4.3 原文）。Inner Monologue = 把"执行结果/场景"**用语言回传给 LLM**，让 LLM 自己重规划。二者正交且互补——本篇的厨房实验就是**直接搭在 SayCan 之上**（附录致谢明说 "construct our Kitchen ... upon [21]"）。
- **vs ReAct（2022，同期）**：ReAct 在**纯文本 / 网页 QA** 里交错 "Reason + Act(+Observation)"。Inner Monologue 是它的**具身版**——把 "Observation" 具体化为**成功检测 / 场景描述 / 人类反馈**这些**物理世界信号**。两者共享同一信念：**把观察回灌进推理循环**。可以说 Inner Monologue 是 ReAct 循环在机器人身上的孪生。

> **读出什么**：这三篇合起来定义了 2022 年"LLM-as-agent"的地基。Inner Monologue 的独特贡献，是**第一个系统论证"把非文本环境反馈翻译成语言、回灌进 LLM 规划器"能显著提升具身任务完成度**，并给出**反馈的分类学**（Success/Passive/Active）。

## §12　局限与批判（论文 §5 + 我的补充）

**论文自陈的局限（诚实）**：
- **依赖理想反馈**：主实验假设有 **oracle 场景描述器 / 脚本系统 / 人类**提供文本反馈（§5）；学习版反馈（§9）明显更弱（ViLD 漏检 28%）。
- **新失败模式**：反馈本身会出错——**假阴性**成功检测→多余重试；**假阳性**→给环境加了"对抗性的部分可观测"（§5 原文）。有时**LLM 干脆无视环境反馈**，仍去规划场景里根本不存在的物体（§5 原文点名的一种失败）。
- **受限于低层策略**：LLM 推理再强，**任务范围与完成度仍被低层控制策略能做到什么卡死**（§5 "Limitations of results" 原文）——高层聪明救不了底层手笨。
- **仍需人在环**：完全自动化（去掉人类 oracle）要等 image/video captioning、VQA 进一步成熟（§5 future work）。

**我的补充批判**：
- **"评的是完整系统、不是纯 harness 增益"**：三个实例里 LLM、技能、反馈源**同时在变**（Table 4），严格说每个数字是"某模型+某技能+某反馈"的联合效应；不过厨房实验（§8）把 LLM 和技能都对齐 SayCan、只改"反馈回不回灌"，**是最接近纯 L 层消融的一组**，可信度最高。
- **样本量偏小**：真机 10 次/任务、厨房 120 次总评测——真机常态，但**方差未报**，个别 40%/44.4% 这种数字的置信区间存疑。
- **反馈"该信多少"没建模**：所有反馈被当作"可信文本"直接回灌，**没有对反馈本身的不确定性建模**（作者在 future work 里也承认可"用文本描述反馈的不确定性"）——这恰是后续工作的空档。

## ★ 对我们的启发（Inspires Us）

> 这一节是组会高潮，也是本库相对 auto-research 的独门优势：**我们（Claude Code / 本课 m9.* 的 agent）本身就是一个 harness**，
> 而 Inner Monologue 恰是我们 ReAct 循环的"具身祖先"。它把"反馈回不回灌"做成了可测的成败变量——这正是 **Harness-Bench 的 execution-alignment**（推理有没有被工具反馈忠实修正）**这条问题线的鼻祖**（Θ3）。下面每条都落到我们自己 harness 的具体组件。

➤ **a. 可直接借用的招（method we can reuse）**：**三类反馈通道 = 三根必须显式接好的线**。把 Inner Monologue 的 `Success / Passive-Scene / Active-Human` 直接映射到我们 harness 的三根线——(1) **工具执行结果**（≈Success：这次工具调用成没成、报错没报错）、(2) **环境/工作区状态回读**（≈Passive Scene：文件差异、目录列表、测试输出，每步自动回灌）、(3) **主动向用户澄清**（≈Active/Human：模糊指代/偏好才问）。可直接抄的机制：**成功检测器要"转成语言"再回灌**——把 exit code / stderr 摘成一句 `"tool X failed: <reason>"` 放进上下文，而不是塞原始日志。

➤ **b. 可迁移到我们的模块（transfer）**：把 §7 的"**反馈-失败对照表**"接到 auto-research 的 `m9.6`（评测沙箱）上——给每条失败轨迹按"缺哪根反馈线"打标：是**缺 Success**（失败了没重试）、还是**缺 Passive**（漏看了状态/文件、在过时快照上继续）、还是**缺 Active**（该问没问、猜错了意图）。迁移前提要改的是：我们的 trace 得**结构化记录"每步之后哪根反馈线被读进了下一步 prompt"**——这与 Harness-Bench 的 instrumentation 要求完全一致，一步到位。

➤ **c. 它暴露的开放问题 = 我们的机会（open problem → opportunity）**：论文明说 **agent 有时会"无视环境反馈、继续规划不存在的物体"**（§5），但**没有在线检测/纠正这种"反馈被忽略"的机制**——它是事后从失败里看出来的。这**正是** Harness-Bench 里 **execution-drift** 的具身先声，也是我们 execution-alignment 的直接抓手。机会：在我们的 ReAct 循环里加一个 **"上一条工具观察是否被这一步引用"探针**——若这一步的推理里没有任何 token grounding 到上一条 observation，就判为"反馈脱钩"，触发一次强制"复述观察再规划"。**第一步（本周可做）**：在 m9.* 某个 agent 上加这个探针，度量它能否降低"在过时状态上蛮干"（contract/state 类）失败。

➤ **d. 与本库其它论文/模块的连接（connect the dots）**：**与 Harness-Bench（2605.27922）正面呼应且互为源流**——Harness-Bench 把 "工具反馈是否被纳入下一步" 抽象成 execution-alignment 并**度量**它，Inner Monologue 则是**第一个把这件事做出来**的系统；本篇的 §8（SayCan 开环 0% → 闭环 75%）就是 Harness-Bench "Robustness / 失败恢复" 那一列的**历史原型**。**与 ReAct 是同信念孪生**（观察回灌进推理），**与专测工具失败恢复的 H 组（如 Hell-or-High-Water）互补**——那类工作正是在把本篇的 "Success 反馈 + 重试" 这条线做深、做成基准。

➤ **e. 如果我来做下一步（my next move，第一人称）**：我会先在我们某个 m9.* agent 的**控制循环**里，把"工具观察"从"原始日志直接塞进上下文"改成 **Inner-Monologue 式的"先摘成一句自然语言状态/成败，再回灌"**，并加上 (c) 的"反馈脱钩"探针；跑 10 个任务，看**两件事**：① 是否复现本篇"闭环 > 开环"的趋势（即回灌摘要版是否比塞原始日志更少在过时状态上蛮干）；② "反馈脱钩"探针触发率与失败率是否正相关——若正相关，就把它升级成一个 harness 级的**在线 execution-alignment 指标**。

## §13　版图定位（canon 坐标 + 在本库的位置）

- **时间坐标（Θ4）**：**2022 CoRL canon（奠基作）**。它**定义/奠基了**"把非文本环境反馈翻译成语言、回灌进冻结 LLM 规划器以形成闭环"这一范式，并给出**反馈分类学（Success / Passive-Scene / Active-Human）**。**谁在它上面长肉**：PaLM-E（把感知直接编码进 LLM）、VoxPoser、以及各类"语言反馈 / 自我反思"具身 agent，都可视作"更强 grounding / 更强底座"下的 Inner Monologue；而 **Harness-Bench 把它隐含的核心问题（反馈有没有被纳入下一步）抬升为可度量的评测主轴**。相对基石 ReAct/SayCan：它把 ReAct 的"observation"**具身化**、把 SayCan 的 grounding**从"打分不回传"补成"语言回传可重规划"**。
- **E/T/C/L/O/V 归属（Θ1）**：本篇坐 **L 层（控制循环）**——它改的是"反馈如何在循环里流动、循环能否从失败重规划"；强依赖 **T 层**（技能 π_k）与 **O 层**（成功检测器 / 场景描述器 / VQA，反馈的产地）。
- **regime 诚实（Θ5）**：本篇给出的是 **"闭环 > 开环"** 在**具身、真机、有扰动、低层策略会失败**这一 regime 下的强证据（§8 差距最大）。但它**不主张 harness 永远压过 model**——恰恰相反，它诚实指出**收益上限被两头夹**：上受制于 O 层反馈质量（§9 学习反馈明显更弱），下受制于 T 层策略能力（§5 "高层再聪明救不了底层手笨"）。换言之：**任务越需要"动手、会失败、要恢复"，闭环这层 harness 越主导；反馈越理想、底层策略越强，其边际收益越依赖这两端**——这与 Harness-Bench 的 regime 结论（动手/状态类任务最吃 harness）严丝合缝。
- **在本库的位置**：**B 组（控制循环）L 层 canon**，是"为什么 agent 需要闭环"这一整条论证线的**起点**。读完它再回看任何一篇"工具反馈 / 自我反思 / 失败恢复"的论文，都可以问一句："它是不是在把 Inner Monologue 的三根反馈线（Success/Passive/Active）之一做深、或换更强的底座与更可靠的反馈？"

## §15　一页速记
