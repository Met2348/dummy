# Hell or High Water：Agent 明明知道"工具坏了"，却依然想不出备用方案（分组 H・可靠性/安全/可观测/沙箱・L/O 层・前沿 2025）

> arXiv 2508.11027｜Johns Hopkins University（Andrew Wang, Sophia Hager, Adi Asija, Daniel Khashabi, Nicholas Andrews）｜COLM 2025 会议论文｜时间坐标：前沿；本库 H 组全部 6 篇报告的收官之作，与 systems-security-foundations（2512.01295）的"系统安全"视角、AgentRacer（2509.03312）的"失败归因"视角互补，聚焦"故障发生后 agent 能不能重新规划"这一具体能力维度

## §1　TL;DR

> 主讲提示：论文标题"Hell or High Water"（不论遇到什么阻碍都要完成）来自一句英语成语，指"无论环境有多恶劣，义务都必须履行"。这篇论文问的问题很朴素：用户交给 agent 一个任务，中途某个关键工具突然不可用了，但换一条路依然能完成任务——**agent 能不能自己想到"换一条路"这件事？**答案是：**不太行，即便是当时最强的商用模型也不行**。

Hell-or-High-Water 是首个系统性研究"外部故障"（agent 自身没有犯错，但环境突然让原计划走不通）对语言 agent 规划能力影响的基准。它包含 830 个问题、4450 个候选函数，每个问题保证至少存在两条互不重叠的正确解法；基准会在 agent 尝试第一条解法时人为让所需函数"不可用"，同时**保证任务始终可解**——理想情况下，一个鲁棒的 agent 应该完全不受这种人为故障影响。核心发现：**全部被测模型在遭遇外部故障后性能都出现 32%–45% 的相对下降**，即便是环境反馈已经非常明确（"函数 987 当前不可用，请尝试其它函数"）；**模型规模增大并不能挽救这种脆弱性**，抗故障能力不像是一种会随参数量涌现的能力；不同模型"知道该放弃"的意愿差异巨大。

## §2　问题定位：内部错误 vs 外部错误——一个被长期忽视的区分（§1）

**直觉**：agent 执行计划失败的原因可以分两类——**内部错误**（agent 自己犯的错，比如给函数传错了参数）和**外部错误**（原因超出 agent 控制范围，比如订机票时发现某个航空公司 API 没有满足条件的航班，agent 必须尝试其它航司的订票 API）。

**Why（问题层）——为什么"外部错误恢复能力"值得单独作为一个基准维度来研究，而不是笼统地归入"agent 鲁棒性"？**
> 论文指出（§1）：此前的工作大量聚焦于**语言 agent 纠正自己错误的能力**（Reflexion、Self-Refine 等自我反思方法），但**外部错误的影响此前一直未被充分研究，尽管它们在现实世界里普遍存在**。→ 这两类错误对 agent 提出的要求本质不同：纠正内部错误主要考验"agent 能不能发现自己的逻辑/参数错误并修正"，本质上是一种**自我校验**能力；应对外部错误则要求 agent 在**原计划的第一步就被证明走不通**的情况下，重新搜索、评估、并执行一条**结构不同**的替代路径——这更接近于**在约束突变后的重新规划**能力，考验的是"能不能跳出第一次想到的那个方案框架"。→ 论文选择专门为这第二类此前被忽视的能力设计一个干净的基准，代价是需要精心设计"保证至少存在两条互不重叠解法"这一较为苛刻的构造约束（详见 §4），但换来的是能够**把"重新规划能力"从"自我纠错能力"里剥离出来单独测量**，而不是让两种能力的信号混杂在一起。

## §3　与已有基准的差异化定位（§2）

论文将自己与几类既有工作对照：**API-bank、ToolQA** 只测试单步的工具规划-检索-调用；**ToolSandbox、AppWorld** 引入反馈机制，支持多轮交互对话；**TravelPlanner** 要求 agent 安排多日行程，天然需要处理多步骤任务和输出组合。Hell-or-High-Water 的独特之处在于：**为每个问题提供至少两种不同复杂度的解法**——一种单步直接解决，另一种需要 agent 把任务拆解成子任务、为每个子任务调用正确的函数。论文明确指出，"提供需要不同程度创造性/推理的多重解法，从而能够研究 LLM 根据反馈调整方案的能力"，据其调研**此前的基准都不具备这种设计**。

论文同时区分了自己与"LLM 创造力"研究文献（逃脱密室谜题、真实世界物体使用等）的关系——这些工作同样要求创造性思维和工具组合，但局限于**物理工具使用**这一狭窄类别，无法评测需要 API 调用的虚拟工具场景；Hell-or-High-Water 聚焦 API 请求场景，既保持了真实世界设定，又天然具备**错误消息这类内在反馈机制**（而非依赖人工/LLM 编写的虚构场景描述）。

## §4　基准设计的四个核心属性（§1，作者自陈设计目标）

论文明确列出了让这个基准"适合研究外部错误现象"的四个属性：① **评测过程始终把 agent 置于"不用备用方案就无法完成任务"的处境**，同时保证任务依然可解；② 任务要**足够有挑战性以要求真正的规划**，但又足够简单，使得**通过与环境交互确实能找到成功的备用方案**（不是无解的难题）；③ 任务相对 LLM 预训练数据是**新颖的**，避免 agent 靠记忆背诵蒙对答案而非真实推理；④ 解法基于**功能正确性**、容易客观评测——为满足这最后一点，论文特意**避免依赖 LLM 做评委**（担心潜在偏见，也为了保证可复现性），转而使用人工整理的、可自动核验功能正确性的数据集。

**读出什么**：这四条设计属性里，第④条"避免用 LLM 评委"与本库 H 组同批次 AgentDojo（2406.13352）的核心设计原则**完全一致**——两篇论文都独立得出了"评测机制不应依赖可能被污染/带偏的 LLM 判断，应该用确定性的功能正确性检查"这一结论，尽管应用场景（对抗鲁棒性评测 vs 故障恢复评测）完全不同。这提示"用确定性检查取代 LLM 评委"正在成为 agent 评测协议设计里一条越来越被广泛认可的最佳实践。

## §5　基准构造：从 Text-to-SQL 数据集自动生成函数库（§3.2，Figure 2）

**核心构造思路**：论文没有人工设计函数库，而是**从 text-to-SQL 数据集（Spider）自动、确定性地生成**——这类数据集天然把自然语言指令与对应的 SQL 查询配对，且 SQL 可以针对真实数据库执行，从而天然获得可供验证的期望输出。

**具体流程**（Figure 2）：① **SQL 参数化**——把 SQL 查询里的字面量（人名、数字等具体值）替换成占位符变量，通过把 SQL 命令解析成语法树（用 SQLGlot 库）、对树做深度优先搜索定位字面量节点来自动完成；② **函数包装**——把参数化后的 SQL 查询包装成接受输入参数、返回 JSON 记录列表的 Python 函数；③ **函数组合**——对于 SQL 里含有子查询的问题，用解析器定位并提取子查询，同样转换成独立的函数；这类问题至少能产生 3 个函数（原始参数化查询 1 个 + 子查询相关函数 2 个），构成**两条等价但结构不同的解法**——一条直接调用原始查询函数，另一条需要先调用子查询函数、再把结果传给下一个函数（即 Figure 1 示例里的 Func A 单步 vs Func B+Func C 两步组合）；④ **函数文档生成**——用语言模型为每个 Python 函数生成文档，遵循 OpenAI function-calling 的标准文档格式。

**Why（设计层）——为什么要从现成的 text-to-SQL 数据集"倒推"出函数库，而不是像大多数工具使用基准那样直接人工设计一套 API？**
> 朴素做法是人工设计一批贴近真实业务场景的 API（就像本库 G 组 τ-bench 的零售/航空 API），这样场景更逼真、更贴近实际部署。→ 但人工设计的代价是**规模天然受限**——人工设计并维护数千个高质量、彼此存在有意义组合关系的 API 极其昂贵。→ 论文选择从 Spider 这类已有大规模、已验证正确性的 text-to-SQL 数据集出发，用确定性程序化流程自动"倒推"出函数库，代价是牺牲了一部分"贴近具体业务场景"的真实感（终归是从 SQL 派生的抽象函数，而非真实世界某个具体产品的 API），但换来的是**能够以近乎零边际成本规模化生成 4450 个函数、830 个高质量问题，且天然自带"至少两种解法"这一关键属性**（因为子查询结构本身就提供了"直接查询 vs 分解成两步查询"这两条数学上等价的路径）——这是一种巧妙地把"如何低成本构造大规模、有保证性质的基准"这一工程难题，转化为"如何从已有结构化数据里挖掘出所需性质"这一数据变换问题的设计思路。

**质量控制**：剔除查询空数据库、结果为空、或结果超过 100 条（避免上下文过长/输出难以阅读）的问题；进一步过滤出所有真正含子查询的指令（保证"至少两条解法"这一核心性质），初筛后剩 413 个问题，经数据增强后扩充到 **922 个问题**。

## §6　数据集规模与 CodeAct 交互框架（§3.1，§3.2）

**最终规模**：验证集 92 例（用于 prompt 工程），测试集 **830 例**，每例至少有 2 种已知的函数调用组合可得到正确答案；总计自动生成 **4450 个函数**（其中 2160 个零参数、1661 个一参数、568 个两参数、46 个三参数、15 个四参数）。

**交互框架**：agent 通过 **CodeAct**（ReAct 的扩展，本库 canon 2402.01030）与环境交互——动作被具体定义为可执行代码块，工作方式类似 Jupyter Notebook（内核逐块执行代码），每个动作前都有一段口头化的推理与规划。面对超过 4000 个候选函数的规模，论文提供了一个**语义搜索工具**——agent 生成查询、执行搜索，获得一组检索到的候选函数及其文档；agent 可以选择执行观察到的函数，或用修改后的查询继续搜索。

**制造外部错误的机制**：通过在 agent 尝试执行第一条解法路径所需的函数时，**人为让该函数报错不可用**，从而强制诱导需要备用方案的场景，同时保证第二条解法路径始终完好可用。

## §7　核心结果：Table 1——全部模型性能大幅下降（§4.2）

**开放世界设置**（agent 须自主搜索工具，最难版本）核心数字：

| 模型 | 无外部错误 | 有外部错误 | 相对下降幅度 |
|---|---:|---:|---:|
| **Gemini 2.0 (Flash)** | 71.4% ± 1.6 | 41.1% ± 1.7 | **-42.4%** |
| GPT-4o | 60.5% ± 1.7 | 38.4% ± 1.7 | -36.5% |
| Llama 3.3 (70B, Instruct) | 64.0% ± 1.7 | 38.9% ± 1.7 | -39.2% |
| Llama 3.1 (70B, Instruct) | 42.3% ± 1.7 | 23.3% ± 1.5 | -44.9% |
| Qwen 2.5 (72B, Instruct) | 64.1% ± 1.6 | 35.3% ± 1.7 | -44.9% |

**读出什么**：即便是无外部错误条件下表现最好的 Gemini 2.0 Flash（71.4%，比第二名 GPT-4o 高出 11 个百分点），一旦引入外部错误，相对下降幅度反而是全部模型里**第二高**的（-42.4%）——这说明"良性条件下表现好"和"面对突发故障时的韧性"是两个不完全相关的能力维度，与本库 H 组 AgentDojo（2406.13352）"逆向缩放律"（模型越强越容易被攻击）在方法论精神上遥相呼应：**单一维度的领先，不能自动预示其它维度也领先**。

**Why（结果层）——为什么 Llama 3.3 和 Llama 3.1（同一模型家族，架构相近）在这个基准上的表现差距如此悬殊（64.0% vs 42.3% 无错误条件下）？**
> 论文给出了一个具体、可验证的解释（§4.2 Discussion）：Llama 3.3 相对 Llama 3.1 在**指令遵循能力、编码能力、推理能力**上有针对性的提升（IFEval 从 87.5 提升到 92.1，HumanEval 从 80.5 提升到 88.4，MATH 从 67.8 提升到 77.0），这些正是完成本基准任务所需要的核心能力组合——本基准要求 agent 既要读懂并遵循函数文档（指令遵循）、又要正确生成调用代码（编码）、还要在复合任务里做正确的多步推理（数学/逻辑推理）。→ 这提示：本基准衡量的表现差异，很大程度上反映的是**通用能力的组合，而非某种单独针对"故障恢复"训练出来的专项能力**——目前没有任何被测模型接受过专门针对"外部故障恢复"的训练，它们的相对排名基本跟随通用能力排名波动，这也为 §11 讨论的"规模无法挽救故障恢复能力下降"这一发现埋下了伏笔：如果连通用能力提升都无法针对性地改善这个具体短板，说明这个短板可能需要专门的训练干预才能解决。

## §8　封闭世界消融：搜索空间大小的独立影响（§4.2，Figure 3）

为把"寻找相关函数"和"制定备用方案"这两种能力分开考察，论文设计了一个**封闭世界**消融实验——去掉语义搜索工具，直接把所有必需函数连同数量可变的"干扰函数"（对解题无用但混入 prompt 的额外函数）一起放进上下文。

**核心发现**：封闭世界设置下整体准确率明显更高，但**随着外部错误的引入，性能依然会退化到与开放世界搜索场景相近的水平**（干扰函数数量为 81 时，Qwen2.5 准确率跌破 40%）。关键的对照组是：**当完全没有干扰函数（只有 3 个函数可选）时，Qwen2.5 面对外部错误只出现轻微性能下降**——这有力地说明**搜索空间的规模本身，是决定这个基准难度的一个关键独立变量**，而不仅仅是"要不要主动搜索工具"这一表面差异。

## §9　错误分析：搜索失败是最大的单一瓶颈（§5，Table 2/3）

论文把出错的 agent 轨迹归纳成四类失败模式：① **搜索失败**——正确工具从未出现在搜索结果里；② **识别失败**——正确工具出现在搜索结果里，但 agent 从未使用它；③ **链接失败**——agent 识别出其中一个工具但漏掉另一个，或调用顺序错误；④ **工具使用失败**——agent 尝试了正确的工具，但没能正确使用它就放弃了。

**Table 2 核心数字**（占全部失败案例的百分比）：

| 模型 | 总失败数 | 搜索失败 | 识别失败 | 链接失败 | 工具使用失败 |
|---|---:|---:|---:|---:|---:|
| Gemini 2.0 (Flash) | 489 | **57.1%** | 25.5% | 12.3% | 6.14% |
| GPT-4o | 511 | **52.6%** | 24.6% | 8.61% | 14.1% |
| Llama 3.3 (70B) | 496 | **66.3%** | 11.9% | 13.9% | 7.86% |
| Llama 3.1 (70B) | 495 | **59.3%** | 18.0% | 11.7% | 10.9% |
| Qwen 2.5 (72B) | 637 | **60.5%** | 26.4% | 7.06% | 6.12% |

**读出什么**：在开放世界（需要主动搜索）设置下，**搜索失败（52.6%–66.3%）是遥遥领先的最大失败来源**，其次才是识别失败——这意味着当前 agent 面临故障时最大的短板，甚至不在于"知道正确工具后如何组合使用它们"，而在于**从数千候选里检索出正确工具这一步就先掉了链子**，很可能源于 agent 没能构造出恰当反映所需信息的搜索查询这一更基础的能力短板。

**Table 3（封闭世界，Gemini，随工具数量变化的错误构成）**：工具数量从 3 增到 81 时，**识别失败占比从 32.7% 攀升到 54.8%**，而链接失败占比基本稳定在 15%–24% 区间。论文据此得出结论：**"识别正确工具"的困难度随搜索空间增大而急剧上升，而"把工具正确串联起来"这一步的难度相对没有随搜索空间变化而明显加剧**——也就是说，"找到该用的工具"比"用对的顺序调用已经找到的工具"是一个更依赖搜索空间大小的能力瓶颈。

## §10　模型规模的（无）效应：抗故障能力不是涌现能力（§5，Figure 4）

论文对 Llama-3.1-Instruct（2 个规模）和 Qwen-2.5-Instruct（4 个规模）做了规模扫描实验。**核心发现**：绝对准确率随模型规模增大而提升，但**"面对外部错误时的性能下降幅度"并没有随规模增大而收窄**——论文明确指出这看起来**不是语言模型的一种涌现能力**（呼应 Wei et al. 2022 关于涌现能力的讨论）。

**Why（结果层）——为什么"把模型做大"这条在其它很多 LLM 能力维度上屡试不爽的捷径，在"抗故障恢复能力"这个维度上失灵了？**
> 涌现能力通常指某种能力在模型规模跨过某个阈值后突然大幅提升——这类能力往往与"更精细的模式识别、更长的隐式推理链"相关，规模增大直接扩充了模型的表征/推理容量，能力自然水涨船高。→ 但"面对突发外部约束变化、主动重新规划"这件事，在论文观察下**似乎并不是"模型不够聪明想不出来"的问题，而更像是一种"训练偏置压过了明确的环境反馈"的行为模式问题**（呼应 §11 讨论）——如果瓶颈在于模型的某种训练偏好/习惯性行为模式（比如倾向于坚持第一次生成的方案、或不擅长把错误消息转化为具体的下一步搜索策略），那么单纯扩大模型规模、给它更强的"通用智力"，并不会自动矫正这种**特定的行为习惯**，正如一个人再聪明，如果没有专门被要求/训练过"遇到路障要主动绕路"这个具体反应模式，也未必会自动表现出这种行为。→ 论文进一步观察到一个更细粒度的例外：Qwen 从 30B 到 70B 出现性能提升（而非规模单调关系里的异常下降），检查执行轨迹发现 30B 版本有**忽略输出格式规范**这一具体缺陷，这个具体缺陷在放大到 70B 后被修复——这个细节进一步印证"规模能修复的是具体的、离散的能力短板，而不是笼统地让所有维度都同步变好"。

## §11　"知道何时该放弃"：模型间行为差异巨大（§5，Figure 5）

在此前的实验里，任务在原则上始终是可解的。这个实验专门测试相反的情形——**故意把 agent 能访问的函数集合限制成只剩干扰函数**，让任务变得客观上不可能完成，理想情况下 agent 应该识别出这一点并主动放弃，而非无休止地尝试。

**核心发现**：**是否具备"识别不可行、主动放弃"这一能力，因模型而异、差异巨大**——Llama-3.3-Instruct 表现出明显更强的放弃意愿，会在耗尽计算预算前主动认输；**Qwen-2.5-Instruct 则普遍无法识别出"当前给定的函数根本解决不了这个任务"**，倾向于持续尝试直到轮次耗尽。此外，论文还观察到一个有意思的辅助证据：在**能够**正确解决问题的场景下，agent 给出正确答案所用的轮次通常**少于**给出错误答案的轮次——这暗示**"多花几轮继续尝试"本身并不会帮助 agent 找到正确答案**，很可能是 agent 在错误路径上无谓地打转、而非在朝着解法逐步收敛。

**读出什么**：这个发现与本库 F 组 VisualWebArena（2401.13649）"5.1% 不可达任务"的设计精神一脉相承——**"知道何时止损"是一种独立于"任务完成能力"的、值得单独测量的能力维度**。本文的贡献是提供了这一维度上**跨模型行为差异极大**的直接实证，说明这不是一个"模型普遍已经具备、只是程度略有差异"的能力，而是一个**部分模型完全不具备**的能力盲区。

## §12　指令欠规约的影响（§5，Table 4）

真实场景里用户经常表述不完整（比如问选举结果，没说清是全国选举还是地区选举）。论文通过控制每个 prompt 里包含的额外消歧信息量，专门研究这一变量：

| 模型 | 无外部错误 | 有外部错误 | 相对下降 |
|---|---:|---:|---:|
| Llama 3.3 (70B) | 54.2% ± 1.7 | 36.5% ± 1.7 | -32.7% |
| Qwen 2.5 (72B) | 71.1% ± 1.6 | 47.7% ± 1.7 | -32.9% |

**读出什么**：与 §7 未加消歧信息的结果对照（Llama 3.3: 64.0%→38.9%；Qwen 2.5: 64.1%→35.3%），加入消歧信息后 **Qwen 2.5 的无错误准确率显著提升（64.1%→71.1%），但 Llama 3.3 反而下降（64.0%→54.2%）**——同一个干预（补充消歧信息）对不同模型产生了**方向相反**的效果。论文没有对这一反直觉现象给出机制性解释（**原文未给出**具体归因，只客观报告了这一发现），但明确指出：无论是否补充消歧信息，**引入外部错误后的性能相对降幅始终保持在 32%–33% 这一稳定区间**——这提示"指令欠规约"和"外部故障恢复"是两个相对独立的干扰变量，缓解前者不能自动缓解后者带来的性能损失。

## §13　讨论：为什么语言 agent 不能有效吸收环境反馈？（§6）

论文提出这一发现"从两个角度看都令人意外"（§6 Discussion）：① 环境反馈本身相当明确——错误消息形如"函数 987 当前不可用，请尝试其它函数"，并非隐晦或需要复杂推断的信号；② 参与对比的模型里包括**明确宣传自己擅长这类场景**的产品——Google Gemini 2.0 的官方介绍强调其"行动能力、长上下文理解、复杂指令遵循与规划、组合式函数调用"，且它确实是无错误条件下表现最好的模型（比 GPT-4o 高出 11 个百分点），**但一旦被迫探索备用方案，准确率依然出现大幅滑坡**。

**未来方向的猜想（作者自陈，非定论）**：论文推测，由于基准涉及全新函数（模型此前未见过），**语言 agent 的训练偏置可能对新颖情境的适应造成了相当大的损害**——例如 Figure 5 里两个能力接近的模型表现出的巨大行为差异（放弃倾向的巨大分野），可能暗示"训练偏置压过了模型对环境明确反馈的响应"，即便反馈已经足够清晰。论文还提到用**思维/推理模型**（thinking models）做的初步实验显示它们可能需要不同的提示策略，但由于成本限制未能做更详细分析——这是明确标注为"初步、未充分展开"的方向，本报告不做过度引申。

**Memory 的定位（§6）**：论文特别指出，**上下文窗口本身就构成了 agent 的一种情景记忆**（过去的尝试、错误、反思都保留在上下文里，供当前决策参考）；而**跨情节持续存在的记忆**（服务终身学习）是另一个重要但独立的研究方向（Reflexion 等工作已有涉及），论文明确表态**将持久化记忆的探索留给未来工作**，本基准的定位是聚焦"备用方案规划能力"这一单一维度。

**局限性（作者自陈）**：① 出于成本考虑未评测最新的商用模型；② 数据集构建基于 Spider（一个存在已知不完美之处的 text-to-SQL 基准，需要人工过滤空表、数据质量问题等），但论文指出这是**依赖人工生成源数据、以此保证答案验证不需要对解法本身做限制性假设**（呼应 §4 的第④条设计原则）这一取舍下的合理代价；③ 已知模型对 prompt 的具体措辞较为敏感，论文采用了基于 ReAct 的成熟提示框架，但承认针对性的、逐模型定制的 prompt 设计可能进一步改善具体模型的表现；④ 出于跨模型公平对比和实现可行性的考虑，论文统一使用各模型默认的指令遵循能力，而非各家专属、互不统一的工具调用模板。

## ★ 对我们的启发（Inspires Us）

**a／"内部错误 vs 外部错误"这一区分，精确刻画了本次会话反复经历的两类完全不同的中断场景，提醒我们对症下药**。本次 agent-harness 报告库建设中，我们遭遇过两类性质不同的问题：**内部错误**类似"数字对齐搞错了、表格提取列错位"（我们自己的判断/执行失误，靠交叉核对纠正）；**外部错误**则是"API 5小时/日限额耗尽、上层进程重启导致子代理被中断"——这完全是超出我们控制范围的环境故障。论文的核心发现——**"备用方案规划能力"和"自我纠错能力"是两种不同的能力，需要分别评估、分别设计应对策略**——精确解释了为什么本会话应对这两类问题时用了完全不同的策略：应对内部错误靠"用 raw 模式重新提取核对数字"这类**自我校验**手段；应对外部错误靠"切换到主线程直接撰写""用 SendMessage 恢复被中断的子代理"这类**重新规划备用路径**的手段——这两套应对逻辑不能互相替代，我们本次实际上是无意识地、正确地区分并应对了论文定义的这两类不同故障。

**b／"搜索失败是最大的单一瓶颈"（§9，52.6%–66.3% 的失败源于连正确工具都没被搜到）是我们审视自己文献检索/PDF 内容定位习惯的一记警钟**。本次会话在处理复杂表格数字歧义时反复用到"先 -layout 提取探路、发现问题再用 -raw 交叉核对"这套流程——这本质上是一种主动的、迭代式的"重新搜索"策略。论文这个发现提示：**"知道正确答案在哪"和"能不能正确使用它"是两个独立的能力关卡，且前者往往是更大的瓶颈**。下一步：在后续任务（比如即将开始的 100 个 Linux bash 函数任务）中，如果某个知识点的具体细节难以在第一次检索/回忆中确认，应当优先投入精力"扩大搜索范围、换用不同角度重新定位信息源"，而非急于在信息可能不完整的情况下直接组合/使用手头已有的碎片信息。

**c／"知道何时放弃"的模型间巨大差异（§11），呼应了本会话对"5 小时限额 vs 日限额"两种不同严重程度中断的差异化应对**，并进一步启示我们细化止损判断的粒度。本次会话面对"日限额耗尽"时选择了"停止重试、ScheduleWakeup 长间隔后再试"；面对"5 小时限额耗尽"时同样选择了类似的止损策略；但当发现"上层进程重启导致 13 篇批次几乎全部中断"这一更严重的情况时，我们选择了完全不同的应对——**用 SendMessage 恢复而非等待**。这与论文揭示的"止损判断不该是一刀切的二元开关，而应该根据具体故障的性质做出不同响应"是同一个原则：Llama-3.3 的"愿意放弃"和 Qwen-2.5 的"死磕到底"都不是普适正确的策略，正确的策略取决于**具体是哪一类故障**——真正可行但暂时受阻 vs. 客观上确实不可能完成。下一步：在设计任何自动化的重试/止损逻辑时，应当先分类故障的性质（临时性 vs 结构性、可恢复 vs 不可恢复），而非用统一的重试次数上限来一刀切处理所有故障。

**d／同一模型家族（Llama 3.1 vs 3.3）因指令遵循/编码/推理能力提升而在故障恢复任务上表现悬殊（§7），提示我们评估"版本升级是否真的解决了某个具体短板"时要有针对性的测试，而不能只看笼统的能力宣传**。本次会话不止一次依赖对不同模型/子代理能力的隐含假设（比如假设某次重新派发会比上次更可靠）。论文这个发现提醒：**通用能力的提升（哪怕是官方数据支撑的、可验证的提升）不能自动保证某个具体的、细分的能力维度也同步提升**——正如 §10 揭示的"模型规模增大不能挽救故障恢复能力"。下一步：如果我们要评估"切换到某个新模型/新工具版本是否真的能解决当前遇到的具体问题"，应当针对这个具体问题设计一个小规模的验证测试，而不是仅凭"新版本理论上更强"这一笼统假设就直接大规模切换依赖。

**e／论文对"训练偏置压过明确环境反馈"这一假说的诚实标注（§13，明确说这是猜测、未充分验证），是本库贯穿全部报告的诚实标注纪律在收官之作上的又一次印证**。这篇论文和本库此前的 SWE-Bench-CL（2507.00014，核心实证"进行中"）、LlamaFirewall（2505.03574，AlignmentCheck 标注"实验性"）一样，在最有意思、最吸引人深挖的发现面前，选择诚实地说"我们不确定，这需要未来工作验证"，而不是为了论文的完整性把猜测包装成结论。本库 74 篇报告走到这最后一篇，这个反复出现的模式已经足以称为一条经验规律：**优秀的前沿研究论文，普遍把"我们观察到了什么"和"我们认为这是为什么"清楚地分层表述，且对后者保持应有的克制**——这是本次报告库建设全程贯彻的"原文未给出不编造"纪律最终被证明并非我们自己额外加的负担，而是与整个领域最佳实践相符的合理标准，值得作为一条固定原则延续到未来任何类似的深度阅读/报告任务中。

## §14　版本坐标：H 组收官——从"防御"到"评测"到"恢复"的完整闭环

Hell-or-High-Water 是本库 H 组的最后一块拼图，与此前 5 篇报告共同构成一个完整的"agent 可靠性问题处理链条"（呼应 LlamaFirewall 报告 §12 提出的框架）：**LlamaFirewall（预防）→ AgentDojo（检测评估）→ systems-security-foundations（理论化系统性梳理）→ AgentRacer（归因，定位具体是哪一步出的错）→ Hell-or-High-Water（本文，恢复：知道错在哪之后，能不能想出新方案）**。这篇论文回答的问题恰好承接在"归因"之后——即便一个系统已经完美地知道"是哪个环节失败了"（本文的实验设置里，故障信息以清晰的错误消息直接呈现给 agent，相当于归因已经解决），**恢复本身依然是一个远未解决的独立难题**。这提示：agent 可靠性工程不能只满足于"检测到故障"或"定位到故障根因"，**"定位到根因"和"基于根因成功重新规划"之间存在实质性的能力鸿沟**，本文用扎实的实证数据首次系统性地刻画了这道鸿沟的大小（32%–45% 的相对性能损失）。同时，本文与本库 canon CodeAct（2402.01030）、本库 B 组 Reflexion（2303.11366）也构成方法论对话——CodeAct 提供了本文的交互框架基础，Reflexion 代表的"自我反思"范式被本文明确排除在讨论范围之外（因其针对的是内部错误而非外部错误），这一明确的范围划分本身就是本文对既有文献做出的一项方法论贡献。

从更宏观的视角看，本文也与本库 G 组标杆 Harness-Bench（2605.27922）的核心论点形成呼应——Harness-Bench 提出的 TaskScore 公式把过程质量拆解为"稳健性（Robustness）"这一独立维度，而"稳健性"的一大组成部分正是"面对意外情况能否保持有效执行、不崩溃、不放弃"，这与 Hell-or-High-Water 用整整一篇论文的篇幅去深挖的"外部故障恢复能力"，本质上是同一种能力在不同评测框架里的不同刻画粒度：Harness-Bench 把它压缩成 TaskScore 公式里的一个乘法因子，Hell-or-High-Water 则把它单独摘出来、用一整套专门的基准和错误分类学去精细测量。这种"整体评分框架里的一个分量 vs 独立成篇深挖的专项研究"的关系，恰好体现了本库 G 组（评测协议）与 H 组（可靠性/安全专项研究）之间分工互补的整体结构——G 组关心"如何给 agent 的综合表现打一个公允的总分"，H 组关心"这个总分里某个具体的分量到底是怎么回事、为什么会失分"。

## §15　延伸讨论问题

1. §10 显示模型规模增大无法挽救故障恢复能力的下降，论文推测这"可能不是一种涌现能力"。如果未来有人专门用"外部故障+备用方案"这类数据对模型做针对性微调（而非单纯扩大规模），你预期这种专项训练能在多大程度上弥补规模扩大做不到的事？
2. §11 揭示"给出正确答案所需的轮次通常少于给出错误答案的轮次"，暗示"更多尝试并不必然带来更好结果"。这对我们设计任何"允许 agent 多次重试"的系统（包括本次会话里子代理的重试/恢复机制）有什么直接的启发或警示？
3. 论文的评测环境把外部错误的反馈设计得非常清晰明确（"函数 987 当前不可用，请尝试其它函数"），但即便如此模型依然表现挣扎。如果把错误反馈换成更贴近真实世界的、含糊不清的报错信息（比如只返回一个 HTTP 500 而没有具体原因），你预期性能下降幅度会进一步扩大多少？这对"如何设计工具/API 的错误信息格式来帮助 agent 更好地恢复"这件事有什么启发？

## §16　一页纸速记

**是什么**：Hell-or-High-Water——首个系统研究语言 agent 应对"外部故障"（非自身失误、但任务依然可解）能力的基准（Johns Hopkins，COLM 2025，前沿），830 问题+4450 函数，基于 Spider text-to-SQL 数据集自动构造。

**核心设计**：内部错误 vs 外部错误的明确区分 + 每问题保证至少两条互不重叠解法（通过 SQL 子查询结构自动生成）+ 语义搜索工具应对大规模函数库 + CodeAct 交互框架 + 确定性功能正确性评测（不依赖 LLM 评委）。

**核心数字**：全部 5 个模型引入外部错误后相对下降 32.7%–44.9%（Gemini 2.0 Flash -42.4%、GPT-4o -36.5%、Llama 3.3 -39.2%、Llama 3.1 -44.9%、Qwen 2.5 -44.9%）；搜索失败占全部错误的 52.6%–66.3%，是最大单一瓶颈；封闭世界搜索空间从 3 增到 81 时识别失败占比从 32.7% 攀升到 54.8%；模型规模增大不能挽救故障恢复能力下降；不同模型"知道何时放弃"的意愿天差地别。

**最重要的发现**：① "备用方案规划能力"是独立于"自我纠错能力"的一种能力，此前被严重低估研究；② 即便环境反馈已经足够明确，语言 agent 依然难以有效吸收并据此重新规划；③ 抗故障能力不是一种随模型规模涌现的能力，可能需要专项训练干预；④ "知道何时该放弃"是一种部分模型完全不具备、而非普遍具备但程度不同的能力盲区。

**对我们的启发**：内部错误与外部错误需要用完全不同的应对策略（自我校验 vs 重新规划）；搜索失败往往是比"用对工具"更大的瓶颈，值得优先投入精力；止损判断应针对故障性质分类而非一刀切；通用能力提升不能自动保证特定短板同步改善；诚实标注"观察 vs 推测"的分层表述贯穿本库全部 74 篇报告，是被前沿研究反复验证的最佳实践。

**一句话定位**：它是本库 H 组"agent 可靠性处理链条"的收官环节——揭示了即便预防、检测、归因都已到位，"基于已知故障成功制定新方案"这最后一步依然是一道远未解决的、独立的能力鸿沟。
