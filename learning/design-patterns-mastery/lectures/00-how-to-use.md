# Design Patterns Mastery 使用说明

> **先读 `01-textbook.md`**:那是老手要求补充的本科课件式完整教材(七章、连贯教科书叙述,仿照GoF原书写法),本页只是DeepPoint/ScenarioPoint追问链的使用说明,建议在读完教材之后再来这里练习追问链。

## 三层怎么练

1. **Tier 1(浅,`tier1_shallow/`)**:先把GoF 23种经典模式里最常见的12种(5种创建型+7种结构型)的核心动机和结构记熟。这一层的重点不是背UML图,而是理解"这个模式解决的具体工程问题是什么"——比如Builder解决的是"构造函数参数过多导致可读性和可选参数组合爆炸"这个具体问题,不是"因为教材里有这个模式"。
2. **Tier 2(深,`tier2_deep/`)**:前两个文件补完剩下11种行为型模式,第三个文件跳出GoF 23种的范围,讲"设计模式在现代工程实践里怎么演化"——依赖注入容器解决的是大型项目手动组装依赖图的复杂度爆炸,CQRS是"读写分离"这个朴素想法在数据模型层面的正式化,函数式编程的一等函数经常能用一个参数就替代掉整个Strategy模式的类层次。练习时建议留意"这个模式在什么语言/范式下会显得多余"这类反直觉的追问。
3. **Tier 3(社招级别,`tier3_social_hire/`)**:无标准答案的资深判断题。第一个文件考"这个场景该用什么模式"的综合判断,第二个文件考"什么时候不该用模式"——过度设计边界判断。这两个文件合起来练的是同一种资深能力的两面:知道什么时候该引入抽象,也知道什么时候该拒绝抽象。

## 一个容易被忽视的陷阱

很多人对设计模式的理解停留在"记住23种模式的类图",但面试追问链的第二三层经常考"这个模式和另一个模式的区别在哪"(比如Strategy vs State、Bridge vs Adapter、Facade vs Mediator),或者"这个模式在现代语言特性(高阶函数/装饰器语法)下是否还有必要"。`tier3_social_hire` 里反复强调的"过度设计"也是同一个陷阱的另一面——知道模式的存在,不代表任何场景都该用上,`trap` 字段里指出的正是"为了用上设计模式而生搬硬套"这个最容易犯的错误。
