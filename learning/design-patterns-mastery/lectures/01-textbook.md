## 第一章 创建型模式

### 1.1 引言:为什么"创建对象"本身也是一个值得设计的问题

初学面向对象编程的人,几乎都是从 `new SomeClass(...)` 这样一行代码开始的。在小程序里,这样写完全没有问题:类是谁、参数是什么、什么时候创建,一目了然。但当系统规模变大、协作方增多之后,直接在业务代码里散落地 `new` 具体类,会在至少四个方向上带来麻烦,而这四个方向恰好对应本章要讲的五种创建型模式所要解决的核心问题。

第一个方向是**耦合具体类**。一旦一段业务逻辑里写死了 `new WindowsButton()` 这样的语句,这段代码就和 `WindowsButton` 这个具体类牢牢绑在了一起。将来如果要支持 macOS 风格的按钮,或者要在测试中把真实按钮替换成一个假的实现,唯一的办法就是回头修改这段业务代码本身。面向接口编程的一个基本诉求,是让业务逻辑只依赖抽象类型,而把"具体实例化哪一个类"这个决定推迟、转移到别处去做。这正是工厂方法模式和抽象工厂模式要解决的问题。

第二个方向是**无法控制实例的数量**。`new` 关键字本身没有任何机制去限制一个类能被实例化多少次——你可以毫无阻拦地对同一个类调用一百次 `new`,得到一百个各自独立、互不知晓的对象。但现实中有一些资源天然应该是唯一的:一个负责管理全局配置的对象、一个负责协调所有线程写日志顺序的记录器、一个代表当前硬件设备状态的句柄。如果这些角色被无意间创建出多份,轻则浪费内存和初始化开销,重则出现状态互相矛盾、行为无法预测的bug。这正是单例模式关注的问题:如何在语言层面没有提供"限制实例数量"这个原生能力的前提下,由类自己保证全局只存在一个实例。

第三个方向是**构造过程本身太复杂**,复杂到用一个构造函数塞满所有参数已经不现实。当一个对象有大量属性、其中很多是可选的,而且这些属性之间还可能存在"设置了A就不该设置B"这样的相互约束时,简单的构造函数已经无法优雅地表达这种复杂性,调用方也很容易在传参时出错。这正是建造者模式想要解决的问题:把"构造"这个动作从一次性的函数调用,拆解成一系列可以按需组合的步骤。

第四个方向是**创建成本高、但已有一个现成实例可以参考**。有些对象的构造过程本身涉及大量计算(比如需要读取磁盘、访问网络、执行复杂的初始化逻辑),如果已经有一个配置好的实例摆在那里,重新走一遍完整的构造流程往往是不必要的浪费——更经济的做法是直接复制这个已有实例,在此基础上做局部调整。这正是原型模式的着眼点:通过克隆而不是重新构造来获得新对象。

这四类问题看似互不相关,但它们共享一个共同的解题思路:**把"要不要创建对象、创建哪个对象、怎样创建这个对象"这些决策,从对象的使用者手里剥离出来,交给专门的机制去负责**。这正是"创建型模式"这个分类名称的含义——GoF在《设计模式》一书里把这一组模式统称为创建型(Creational)模式,强调的正是它们共同关注"实例化过程本身"这件事,而不是对象创建完成之后的行为或者对象之间怎么协作。理解了这个共同出发点,再去看Singleton、Factory Method、Abstract Factory、Builder、Prototype这五个具体模式,就不会觉得它们是五个孤立的知识点,而能看出它们分别是在"数量控制""类型延迟决定""产品族一致性""分步构造""克隆复用"这五个不同侧面上,对同一个母题给出的具体解法。

### 1.2 单例模式(Singleton)

单例模式要解决的问题最容易用一句话概括:**保证一个类在整个程序生命周期里只有一个实例,并且提供一个全局可访问的入口来获取这个实例**。它的结构非常简单——类自己把构造函数标记为私有或者受保护,不允许外部代码直接调用;同时提供一个类级别的方法(通常命名为 `getInstance`),内部持有一个静态字段保存已经创建好的实例,外部想要获取实例,只能通过这个方法,而不能绕过它:

```
class Singleton:
    _instance = None            # 类级别的静态字段,保存唯一实例

    def __init__(self):
        pass                     # 实际工程中会限制外部不要直接调用

    @classmethod
    def get_instance(cls):
        if cls._instance is None:
            cls._instance = cls()
        return cls._instance
```

围绕"什么时候创建这个唯一实例",工程实践里分化出两种做法。**饿汉式(Eager Initialization)** 在类被加载的那一刻就立即完成实例的创建,不管程序运行过程中会不会真的用到它;它的好处是实现极其简单,而且由于类加载动作本身通常由运行时环境保证只执行一次,天然不存在多线程重复创建的风险。它的代价是,如果这个实例的构造开销很大而程序实际上未必会用到,这份开销就白白付出了。**懒汉式(Lazy Initialization)** 则把创建动作推迟到第一次真正被请求的时候,按需创建、避免浪费,但代价是必须自己处理好"会不会有多个线程同时判断实例还不存在、于是各自创建出好几份实例"这个并发问题——懒汉式并不是天然线程安全的,这一点恰恰是懒汉式最容易被误解的地方。

要在懒汉式下正确处理多线程安全,同时又不想让每一次获取实例都要付出加锁的性能代价,工程实践里常见的做法是**双重检查锁定(Double-Checked Locking)**:方法先在没有加锁的情况下检查一次实例是否已经存在,如果已经存在,直接返回,完全不需要进入加锁逻辑;只有判断实例还不存在时,才进入加锁的代码块,而进入之后要再检查一次(第二次检查),因为在等待锁的过程中,可能已经有另一个线程抢先完成了创建。这里有一个经常被忽略的细节:仅仅写出"检查两次"的代码结构还不够,用来保存实例的字段还必须声明为线程可见性有保障的(在Java里是 `volatile`,其他语言有各自对应的机制)。原因在于,构造一个对象其实包含"分配内存""执行初始化逻辑""把引用赋值给字段"这几个步骤,如果没有额外的内存屏障保证,这些步骤的执行顺序在编译器或处理器优化下可能被重新排列,导致另一个线程观察到字段已经不是空值,但它指向的对象其实还没有完成初始化——也就是拿到了一个"半成品"。这提醒我们,并发安全的单例实现,考验的不只是"要不要加锁"这么表面的问题,还牵涉到内存模型这一层更底层的保证。

尽管单例模式是GoF原书里正式收录的模式之一,它在现代工程实践里却常常被认为是一种应当谨慎使用、甚至应该尽量避免的反模式。核心争议在于:单例本质上是在程序里安插了一份**隐藏的全局可变状态**。任何代码只要能引用到这个类,就可以随时通过 `getInstance` 拿到同一份实例并修改它的内部数据,而这种依赖关系完全不会体现在调用方的函数签名或者类的字段声明里——阅读一个方法的参数列表,根本看不出它其实间接依赖着某个单例的内部状态。这种"看不见的耦合"对单元测试尤其不友好:如果一个类的内部逻辑直接写死调用某个单例的 `getInstance()`,测试这个类的时候就没有办法把这个单例替换成一个可控的、行为可预测的替身对象,因为代码里没有给外部留下"注入替代实现"的接口。更麻烦的是,如果多个测试用例共享同一个单例的状态,测试之间还可能相互污染,出现只有按特定顺序运行才会通过、单独运行反而失败的诡异现象。正因如此,现代工程实践里更提倡用**依赖注入(Dependency Injection)**的思路来替代直接调用单例:把"整个程序只维护一份实例"这个职责交给外部的组装代码或者容器去承担,而具体使用这个实例的类,通过构造函数或者参数把依赖接收进来,而不是自己主动去找单例要。这样一来,测试时只需要传入一个模拟实现,就能把这个类的逻辑单独隔离测试,不再需要牵动真实的全局状态。

### 1.3 工厂方法模式(Factory Method)

如果说单例模式关注的是"限制实例数量",工厂方法模式关注的则是另一个维度的问题:**当一段通用逻辑需要用到某个产品对象,但具体应该实例化哪一个产品类,只有等到运行时、根据具体的上下文才能确定,该怎么写这段代码才不会因为将来新增产品类型而被迫反复修改?**

工厂方法模式给出的答案是:在负责通用逻辑的父类里,把"创建产品对象"这一步单独抽出来,定义成一个方法(即工厂方法),这个方法本身在父类里只有签名、没有具体实现,真正"该实例化哪个产品类"这个决定,交给不同的子类通过覆写这个方法各自给出。父类里其余不涉及"该用哪个具体产品"的通用逻辑保持不变、不需要重复:

```
class Logistics:                     # Creator(创建者)基类
    def plan_delivery(self):
        transport = self.create_transport()   # 调用工厂方法,自己不关心具体返回什么
        transport.deliver()

    def create_transport(self):
        raise NotImplementedError    # 工厂方法,把决定权留给子类

class TruckLogistics(Logistics):
    def create_transport(self):
        return Truck()

class ShipLogistics(Logistics):
    def create_transport(self):
        return Ship()
```

`plan_delivery` 这段通用逻辑完全不知道、也不需要知道最终用的是卡车还是轮船,新增一种运输方式(比如航空)只需要新增一个子类并覆写 `create_transport`,不需要触碰 `Logistics` 里原有的任何代码——这正是工厂方法模式相比"把 `new` 语句直接写在业务逻辑里"的优势所在:后者每新增一种产品都要回头修改已有的业务代码,前者只需要新增代码,完全符合"对扩展开放、对修改关闭"的开闭原则。

这里有必要澄清工厂方法模式和一个经常被混淆、但并不属于GoF23种模式的编程技巧——**简单工厂(Simple Factory)**——之间的区别。简单工厂通常是一个独立的工厂类,提供一个静态方法,内部用条件分支(`if-else` 或 `switch`)根据传入的参数决定实例化并返回哪个具体产品类,好处是把原本分散在各处的 `new` 语句集中到了一个地方管理,但它并没有真正解决"新增产品类型需要修改已有代码"这个问题——每新增一种产品,依然要回去给这个工厂类的条件分支里加一个新分支。工厂方法模式则是通过"子类继承 + 覆写"这个机制,让新增产品类型这件事只需要新增子类、完全不触碰已有代码,这才是两者最本质的差别:一个是"要改已有代码",一个是"只需新增代码"。

当然,工厂方法模式并不是任何创建对象的场景都值得引入。它的代价是引入了额外的类层次结构:每一种产品理论上都要配一个对应的creator子类,如果产品种类本身很少、也几乎不会再扩展,强行套用工厂方法模式只会让代码平添一堆只有一个方法不同的子类,增加阅读和维护成本,这时候简单工厂甚至直接的条件分支反而更实用。工厂方法模式真正划算的场景,是产品类型未来大概率会持续增加、且每种产品对应的创建逻辑差异明显,值得用继承体系分别封装。另外要注意,工厂方法通常只负责创建单一等级结构里的某一个产品;如果需要的是一组互相关联、必须保持风格一致的产品,工厂方法就不够用了,这就引出了下一节的抽象工厂模式。

### 1.4 抽象工厂模式(Abstract Factory)

抽象工厂模式要解决的问题,和工厂方法模式看起来很像,但关注的粒度不同:**当系统需要创建一系列相关或者相互依赖的对象——这些对象天生应该配套出现、风格必须保持一致、不能随意混搭——该如何保证客户端拿到的永远是搭配正确的一整套,而不需要客户端自己费心去核对搭配规则?**

最经典的例子是跨平台UI组件库。假设一个GUI工具包要同时支持Windows风格和macOS风格,每种风格都需要一整套按钮(Button)、复选框(Checkbox)、滚动条(Scrollbar)。如果用户选择了Windows风格,那么这一整套控件都必须是Windows风格,绝不能出现按钮是Windows风格、复选框却是macOS风格这种错配。抽象工厂模式的做法是定义一个抽象工厂接口,里面声明一组创建方法(`create_button`、`create_checkbox`、`create_scrollbar`),然后为每一种风格各自提供一个具体工厂实现:

```
class GUIFactory:                       # 抽象工厂
    def create_button(self): ...
    def create_checkbox(self): ...

class WindowsFactory(GUIFactory):       # 具体工厂之一
    def create_button(self):
        return WindowsButton()
    def create_checkbox(self):
        return WindowsCheckbox()

class MacFactory(GUIFactory):           # 具体工厂之二
    def create_button(self):
        return MacButton()
    def create_checkbox(self):
        return MacCheckbox()
```

业务代码只需要在启动时拿到某一个具体工厂的实例,之后所有创建控件的动作都通过这个工厂完成,得到的产品必然是风格统一的一整族,搭配是否正确这件事完全由具体工厂内部保证,业务代码自己不需要操心。

抽象工厂和工厂方法的本质区别,不在于"复杂程度"高低,而在于**创建的对象数量和维度不同**:工厂方法关注单一等级结构里的一个产品,通过"子类继承"这一种机制来决定产出哪个具体类型;抽象工厂关注的是多个属于不同等级结构、但彼此必须风格一致的产品族,通过"把一组创建方法打包进同一个工厂接口、约束同一个具体工厂的所有产出必须风格一致"来实现族内一致性。从实现结构上看,一个具体工厂类内部的 `create_button`、`create_checkbox` 这些方法,单独拿出来看都可以说是一个个工厂方法,抽象工厂可以理解为把多个工厂方法组合、打包在同一个工厂对象里,再加上"同一个工厂的产出必须配套"这一层约束。

抽象工厂模式也有它自己的代价:如果要往已有产品族里新增一种产品类型(比如在按钮、复选框、滚动条之外再加一个下拉菜单),就必须修改抽象工厂接口本身、给它新增一个方法声明,并且让所有已经存在的具体工厂实现类都跟着补上这个新方法的实现——这意味着抽象工厂模式在"新增产品种类"这个维度上恰恰违反了开闭原则。但反过来,在"新增一整套新的产品族风格"这个维度上(比如再加一个Linux风格的具体工厂),抽象工厂模式是符合开闭原则的,只需要新增一个具体工厂类去实现已有的抽象接口,不需要改动任何已有代码。这一个模式在两个不同维度上呈现出"开放"与"封闭"两种相反表现,是理解抽象工厂时容易被忽视、但非常值得记住的一点。

### 1.5 建造者模式(Builder)

建造者模式关注的是和前两个模式都不同的另一类问题:**当一个对象拥有大量属性、其中许多是可选的、这些属性还可能需要按特定顺序逐步设置,该如何避免用一个塞满所有参数的构造函数来表达这种复杂性?**

这个痛点在面向对象编程里有一个专门的名字,叫"望远镜构造函数反模式"(telescoping constructor anti-pattern)。设想一个 `House` 对象,只有地基和墙体是必需属性,屋顶、车库、游泳池、花园都是可选的。为了覆盖各种参数组合,类的作者往往会写出一串重载构造函数,从只传必需参数的版本,到依次追加1个、2个、3个可选参数的版本,层层叠加,参数列表像望远镜一节一节拉长。调用方使用这些构造函数时,常常要在自己不关心的可选参数位置传入占位的默认值,一旦相邻参数类型相同(比如连续几个布尔值或整数),非常容易在顺序上传错,而这种错误编译器通常无法帮忙检查出来。

建造者模式的解法是把"构造"这个动作从对象本身剥离出来,交给一个独立的Builder对象负责。Builder提供一系列语义清晰的方法,分别用来设置某一个属性,调用方只需要调用自己关心的那几个方法,不需要为了跳过某个可选参数而传占位符,最后调用一个 `build` 方法才真正产出目标对象:

```
class HouseBuilder:
    def __init__(self):
        self._house = House()

    def set_walls(self, n):
        self._house.walls = n
        return self                # 返回自身,支持链式调用

    def set_roof(self, has_roof):
        self._house.roof = has_roof
        return self

    def build(self):
        return self._house

house = HouseBuilder().set_walls(4).set_roof(True).build()
```

这里的链式调用(每个设置方法返回 `self`)是建造者模式常见但并非必需的写法,它的价值在于提升代码的可读性——调用方一眼就能看出这次构造设置了哪些属性、值是什么,不再是一串没有名字的位置参数。但需要澄清的是,链式调用只是一种配套的实现技巧,而不是建造者模式的定义性特征:即便每个设置方法都老老实实返回 `void`、调用方分成好几行调用,只要具备"构造过程从对象本身分离出来、通过一系列步骤逐步设置属性、最后统一产出对象"这个核心结构,依然是建造者模式。

GoF原书里建造者模式还包含一个可选角色——**Director**,它持有一个Builder的引用,内部封装一个或多个固定的构建流程(比如"组装一栋最小配置的房屋"、"组装一栋豪华配置的房屋"),按预先设定好的顺序调用Builder的各个设置方法,把"怎样组装出某种典型配置"这个知识封装起来,调用方不需要了解具体的步骤顺序,只需要选择调用Director提供的某个预设流程。在现代实践里,尤其是搭配链式调用的场景下,Director角色经常被省略,由调用方自己按需组合Builder暴露出来的方法——这在只存在一种典型配置、或者不同调用方需要完全定制化组合的场景下反而更灵活。只有当系统里确实存在几种反复被使用的"标准配置"时,把这些标准配置的组装步骤序列封装进Director,才能真正避免重复代码、保证一致性。

建造者模式和抽象工厂模式虽然都在"构建出比较复杂的对象结果",但两者的关注点不同:抽象工厂关注的是"创建哪一族产品"——一次调用直接返回一个完整可用的产品,不同产品之间是并列、彼此独立可用的关系;建造者模式关注的是"如何分步骤组装出一个内部结构复杂、由多个部件组成的单一对象"——同一个目标对象需要经过多个先后步骤才能拼装完整。前者是横向的产品族配套问题,后者是纵向的单个对象分步组装问题,这是区分两者最可靠的角度。

### 1.6 原型模式(Prototype)

原型模式解决的问题和前面四个模式都不太一样:**当构造一个新对象的常规流程开销很大,而系统里已经存在一个配置好的实例可以参考时,能不能直接复制这个已有实例来获得新对象,而不用重新走一遍完整的构造过程?**

原型模式的结构很简单:让需要被克隆的类实现一个统一的克隆接口(通常叫 `clone`),调用方拿到某个已有实例后,不通过构造函数创建新对象,而是调用这个实例自身的克隆方法得到一份拷贝,再在这份拷贝的基础上做局部修改。这个思路特别适合"已有一个精心配置好的对象、只需要在细节上做微调"的场景,比如一个包含大量默认参数的配置对象,或者一个初始化开销很高的复杂对象。

原型模式最容易踩的坑,集中体现在**浅拷贝(shallow copy)和深拷贝(deep copy)的区别**上。浅拷贝只复制对象本身的字段,如果某个字段本身是一个指向其他对象的引用(比如一个列表、一个字典,或者另一个自定义对象),浅拷贝只会复制这个引用本身,而不会递归复制它指向的内容,结果是原对象和拷贝出来的新对象,这个字段实际上指向同一份底层数据。如果之后修改了新对象里这个字段所指向内容,原对象对应的内容也会跟着一起改变——这在调用方以为"两个对象已经彻底独立"的场景下,是一个极其隐蔽、容易在测试里都发现不了的bug。深拷贝则会递归地对每一层引用都创建新的独立副本,保证拷贝出来的对象和原对象在任何嵌套层次上都不共享可变的底层数据,但代价是执行开销更大,如果对象内部存在循环引用(A引用B、B又引用回A),朴素的深拷贝实现还可能陷入无限递归,需要额外的机制(比如用一个"已处理对象"的记录表)来规避。在实际编码时,原型模式是否安全好用,几乎完全取决于这一层浅拷贝/深拷贝语义有没有被正确理解和正确选择,而不是"调用了克隆方法"这个动作本身。

### 1.7 场景走查:一个数据库连接管理器该不该用单例模式

把上面讲的原理落到一个具体场景里,能看得更清楚。假设要设计一个负责管理数据库连接池的组件,直觉上很容易觉得"连接池全局只应该有一份,这不就是单例模式的典型场景吗"。这个直觉分析到一半是对的,但完整的分析需要走得更远。

先看单例模式确实解决了什么问题:如果这个连接池被意外创建出多份,每一份各自维护自己的连接集合,系统实际同时打开的连接数就可能远超预期上限,数据库那边可能因为连接数过多而拒绝服务,而且多份连接池各自做连接的分配与回收决策,彼此不知道对方的存在,容易导致资源争用或者重复复用同一条连接这类难以复现的并发问题。从"保证只有一份、避免资源重复分配"这个角度看,这确实是单例模式想解决的核心问题。

但如果止步于"用单例模式实现连接池"就结束分析,会漏掉前面讨论过的代价。如果连接池类内部直接写死调用自己的 `getInstance()`,那么所有依赖数据库访问的业务代码,都在事实上依赖着这一个全局可变对象,却不会在任何函数签名或类型声明里体现出来。等到需要写单元测试、想要在测试里换成一个不会真正连接数据库的假连接池时,会发现业务代码根本没有给"传入替代实现"留出接口——因为它们从来不是"接收"一个连接池,而是自己主动伸手去 `getInstance()` 要一个。更现实的做法是,依然保证程序运行期间只组装出一份连接池实例,但这个"只有一份"的保证由外部的组装逻辑或者应用启动流程来承担,而各个需要用到连接池的业务类,通过构造函数参数接收这个连接池实例,自己完全不知道、也不关心这份实例是全局唯一还是每次都新建的。这样一来,"全局唯一"这个约束本身没有被放弃,只是承担这个约束的位置,从"业务类自己伸手去要"变成了"外部负责组装的人负责保证",测试的时候只需要把参数换成一个假的实现即可,不再需要碰到任何全局状态。这个走查过程说明,判断"该不该用单例"不能只看"这个东西是不是全局唯一"这一层表面特征,还要往下多问一层:这个唯一性约束,应该由谁来承担、以什么方式承担,才不会在带来"数量控制"这一个好处的同时,顺带引入"隐藏依赖、难以测试"这个隐患。

### 1.8 常见误区澄清

第一个常见误区是**把单例模式等同于"一个全局变量的高级写法"**。全局变量在程序启动或者第一次赋值时就已经存在,任何代码都可以直接读写、覆盖它,没有创建时机上的控制;而单例模式封装了实例化的时机和过程,可以做到懒加载、可以显式处理线程安全,并且外部代码始终只能通过统一的访问入口获取实例,不能绕过这个入口直接替换掉底层对象。两者看起来都表现为"全局能访问到同一份东西",但对创建过程的控制能力完全不同。

第二个常见误区是**认为工厂方法模式和抽象工厂模式没有本质区别,只是"复杂程度不同"**。真正的区别在于两者关注的创建对象的数量和维度:工厂方法关注单一等级结构里一个产品该由哪个子类决定;抽象工厂关注一组产品之间是否风格一致、能不能配套使用。抽象工厂内部往往正是由若干个工厂方法组合而成,但多加了"这些方法的产出必须来自同一个具体工厂、因此风格一致"这一层约束,这层约束才是抽象工厂真正的价值所在,而不是单纯的"方法数量变多了"。

第三个常见误区是**觉得建造者模式只是为了"少写几个参数"、图个方便**。建造者模式真正解决的是"可选参数按特定顺序设置、参数组合容易在顺序上传错、编译器又无法帮忙检查"这一类结构性问题,链式调用带来的书写便利只是这个模式的一个附加好处,不是它存在的根本理由。如果一个对象的构造逻辑本身并不复杂、参数也不多,即便用了链式写法,也谈不上是在运用建造者模式解决什么实质问题。

第四个常见误区是**认为浅拷贝和深拷贝只是"性能快慢"的区别,选哪个都差不多**。两者的差别不是性能,而是语义正确性:浅拷贝之后,原对象和新对象在嵌套引用的层次上依然共享同一份底层可变数据,一方的修改会影响到另一方,这在很多场景下会直接导致隐蔽的逻辑错误,而不只是"慢一点"或"快一点"的权衡。选择浅拷贝还是深拷贝,首先要看清楚要克隆的对象内部有没有嵌套的可变引用,而不是凭感觉挑一个看起来更简单的选项。

第五个常见误区是**认为只要用到"创建对象",就应该主动往设计模式上靠**。前面几节反复强调过,工厂方法、抽象工厂这些模式都有各自的适用边界和引入代价:产品种类稳定、几乎不会扩展时硬套工厂方法,只会平添不必要的类层次;产品之间彼此独立、并不需要保持一致风格时硬套抽象工厂,也是不必要的过度设计。设计模式提供的是解决特定结构性问题的现成方案,而不是"用得越多说明代码越高级"的加分项。

### 1.9 本章小结

创建型模式表面上分成五种不同的具体形态,但它们背后共享同一个主题:**把"要不要创建、创建哪一个、怎样一步步创建、能不能靠复制现有实例来创建"这些原本会散落在客户端代码里的决策与复杂性,收拢、封装到专门的角色或机制里,从而让使用对象的客户端代码,不需要再关心这些具体类是怎么被造出来的**。单例模式封装的是"数量与访问入口"的决策,工厂方法封装的是"具体子类型该由谁决定"的决策,抽象工厂封装的是"一组产品是否配套一致"的决策,建造者模式封装的是"复杂对象分几步搭建"的决策,原型模式封装的是"能否靠克隆代替重新构造"的决策。理解了这一条共同主线,再去分辨这五个模式两两之间的边界(工厂方法与简单工厂、工厂方法与抽象工厂、抽象工厂与建造者),就不再是死记硬背几条区别,而是能从"它们各自想封装什么决策"这个角度自然地推导出来。接下来建议对照本专题 `src/tier1_shallow/dp_pat_creational_patterns.py` 里对应的18个DeepPoint追问链,逐条检验自己是否真的能在被追问"为什么""区别在哪""这样做有什么代价"时给出经得起推敲的回答,把本章讲的系统性原理转化为能够应对连续追问的扎实理解。

## 第二章 结构型模式

### 2.1 引言:这一类模式究竟在回答什么问题

GoF 的 23 种设计模式通常被划分成三大类:创建型、结构型、行为型。这三大类不是随意的归档标签,而是分别对应软件设计里三个不同层次的问题。创建型模式回答的是"一个对象应该怎么被造出来"——工厂方法、抽象工厂、建造者、原型、单例,关心的都是把"决定创建哪个具体类的实例"这件事从客户端代码里剥离出去,让客户端不必和具体类名绑死。行为型模式回答的是"多个对象之间应该怎么协作、怎么分配职责、怎么传递控制流"——策略、观察者、状态、模板方法等等,关心的是对象之间动态的交互关系。而结构型模式回答的是介于两者之间的第三个问题:对象和类已经存在了,创建这件事已经解决,交互协作也不是眼下的重点,现在要问的是——**怎么把这些已经存在的类和对象组织、拼装成更大、更有用的结构**。

"组织成更大的结构"这句话听起来很抽象,拆开来看其实对应几种很具体的现实困境。第一种困境是"接口对不上":系统里已经有一个类能完成需要的功能,但它暴露的方法签名和当前代码期望的完全不是一回事,这时候需要某种方式把两者"接"起来,这是适配器模式要处理的问题。第二种困境是"继承体系撑不住需求的组合爆炸":如果通过定义子类来表达功能的各种组合,子类数量会随着可选功能的种类数指数增长,需要一种不通过继承也能动态叠加行为的手段,这是装饰器模式要处理的问题。第三种困境是"直接暴露一个对象不安全或不划算":对象的创建代价很高,或者不应该被任意访问,需要在客户端和真实对象之间插入一层受控的中介,这是代理模式要处理的问题。第四种困境是"子系统太复杂,客户端学不动":一整套类互相协作才能完成一件事,客户端如果要直接和它们打交道,学习成本和耦合面都太大,需要一个简化的统一入口,这是外观模式要处理的问题。第五种困境是"部分和整体要一视同仁":系统里有天然的树状层次结构(文件夹里有文件也有子文件夹),但客户端不想为"处理单个节点"和"处理一整棵子树"写两套不同的代码,这是组合模式要处理的问题。第六种困境是"一个抽象概念有多种实现方式,而且两者都会独立演化":如果把"是什么"和"怎么实现"揉在同一条继承链里,任何一边的变化都会带来类数量的组合爆炸,这是桥接模式要处理的问题。第七种困境是"细粒度对象数量太多,内存扛不住":当系统里需要存在的对象实例数量非常庞大,但这些对象之间又有大量可以共享的公共部分,需要把可共享和不可共享的部分拆开,这是享元模式要处理的问题。

这七个具体问题看起来互不相关,但它们有一个共同的方法论内核:都不是通过继承体系的纵向扩展来解决问题,而是通过"让一个对象持有另一个对象的引用,把请求转发出去,同时在转发的过程中做一些额外的事情"这种组合式的手法来解决问题。这也是为什么在学习结构型模式的时候,经常会发现好几个模式的类图长得极其相似——同样是一个类持有另一个类的引用并转发调用——却因为想解决的具体问题不同而被赋予了不同的名字。理解这一点,是学好整个结构型模式家族最关键的一把钥匙:结构型模式之间的区别,往往不在"代码长什么样",而在"为什么要这么写"。

### 2.2 适配器模式:让本来说不上话的两个接口对上话

适配器模式要解决的场景可以这样描述:客户端代码是按照某一套接口约定(称为目标接口,Target)写好的,它只知道调用某个名字的方法、传什么样的参数、期待什么样的返回值。与此同时,系统里已经存在一个类能够完成客户端真正需要的功能,但这个类暴露出来的方法名、参数形式、调用方式跟目标接口完全对不上——它可能是一个第三方库,可能是遗留系统里遗留下来的旧模块,总之客户端没办法拿着目标接口的调用方式直接去调它。适配器的做法很直接:新写一个类,这个类实现目标接口,内部持有(或者继承)那个"说不上话"的类,在目标接口要求的每一个方法里,把调用转换成对方能听懂的形式,转发过去,再把返回结果转换回客户端期待的形式。整个转换过程对客户端完全不可见,客户端仍然认为自己是在跟一个符合目标接口的普通对象打交道。

适配器有两种经典实现路径。一种是通过继承实现,称为类适配器:适配器类同时继承目标接口和被适配的类,因为是继承关系,可以直接复用被适配类已有的方法实现,甚至可以覆写某个具体方法插入自己的逻辑。这种做法依赖语言支持多重继承(或者接口加类的组合继承),而且从继承关系上把适配器和某一个具体的被适配类死死绑在了一起。另一种是通过组合实现,称为对象适配器:适配器不继承被适配类,而是在内部持有一个指向被适配对象的引用,所有转发都通过调用这个引用完成。用伪代码表示大致是这样:

```
interface Target:
    request()

class ObjectAdapter implements Target:
    adaptee: Adaptee          # 组合持有,而非继承

    request():
        result = self.adaptee.specific_request(...)
        return convert(result)
```

对象适配器因为持有的只是一个引用而不是通过继承锁死一个具体类,这个引用在运行期可以指向被适配类的任意子类实例,适配器的行为会随注入对象的变化而自动变化,不需要为每一种被适配的子类单独定义一个新的适配器子类。这正是"组合优于继承"这条通用设计原则在适配器场景下的具体体现:继承是编译期就锁定的强绑定关系,而组合是运行期可以灵活替换的引用关系。也正因如此,即便是像 Python 这样原生支持多重继承的语言,业界的普遍实践仍然是优先选择对象适配器,除非确实需要覆写被适配类内部某个方法的实现细节,类适配器才有一席之地。

需要提醒的是,适配器模式的适用边界是"接口形状不同,但语义相近"——也就是两边其实是在做同一件事,只是说法不一样。如果适配器内部为了把两边拉到一起,不得不塞进大量的业务判断和数据加工逻辑,这说明两边的语义鸿沟已经太大,继续用适配器只是在用一层"翻译"掩盖一个更深层的设计不匹配问题,这时候更值得做的是重新审视这两个接口是否真的应该被撮合到一起。

### 2.3 装饰器模式:用叠加组合代替继承扩展

装饰器模式解决的问题是:如何在不改动一个对象所属类的前提下,动态地给这个对象添加一些额外的职责。之所以要强调"不通过继承"来做这件事,是因为继承在表达"功能的任意组合"时有一个结构性的缺陷,通常称为子类爆炸:如果一个基础功能上可以任选叠加 k 种彼此独立的附加功能,理论上所有可能的组合数是 2 的 k 次方,如果用继承来实现每一种组合,需要定义的子类数量会随着可选功能种类数指数增长——一杯咖啡可以选择加奶、加糖、加摩卡,任选组合就足以产生一长串子类,新增第四种可选调料,子类数量还要再翻倍。

装饰器模式的解法是让"装饰器"和"被装饰的对象"实现同一个接口,装饰器内部持有一个指向被装饰对象的引用,调用接口方法时,先把调用转发给被装饰对象执行原有逻辑,再在转发前后插入自己这一层新增的行为。关键在于,装饰器本身也满足这个统一接口,所以装饰器可以被另一个装饰器继续包装,层层嵌套、按需组合。伪代码大致是:

```
interface Beverage:
    cost() -> float

class Espresso implements Beverage:
    cost(): return 2.0

class MilkDecorator implements Beverage:
    wrapped: Beverage
    cost(): return self.wrapped.cost() + 0.5

class SugarDecorator implements Beverage:
    wrapped: Beverage
    cost(): return self.wrapped.cost() + 0.2

drink = SugarDecorator(MilkDecorator(Espresso()))
```

每新增一种可选功能,只需要新增一个装饰器类,而不需要为它和已有功能的每一种组合都定义一个新类;想要什么样的组合,在运行时把对应的装饰器一层层套上去即可。这背后依赖的前提是接口一致性:正因为装饰器和被装饰对象实现同一个接口,客户端才感知不到自己面对的到底是原始对象还是被套了好几层的装饰器,叠加装饰器也才不会破坏客户端的调用方式。

装饰器模式并非没有代价。一是会产生大量结构相似的小对象,嵌套层数越深,对象数量和内存开销越大;二是调试变得更困难,一次方法调用要经过多层转发才能到达真正处理业务逻辑的那一层,追踪某个具体行为到底是哪一层装饰器加上去的,往往比在单一继承体系里排查问题更费力。此外,装饰器叠加的顺序通常会实质性地影响最终结果,模式本身并不保证顺序无关——顺序是否重要,完全取决于各个装饰器具体做的操作彼此之间是否满足交换律。一个常被拿来说明的例子是"先压缩再加密"和"先加密再压缩"效果完全不同:先压缩再加密,是对压缩后的数据加密;先加密再压缩,则是试图压缩已经加密、因而高熵、几乎没有可压缩结构的数据,压缩效果会大打折扣。装饰器模式在结构层面完全不会因为顺序错误而报错,因为每一层的接口都合法,顺序错误只会在运行结果里悄悄体现出来,这也是使用装饰器模式时容易被忽视的一个陷阱。

这里必须特别澄清一个经常被混淆的问题:Python(以及很多现代语言)里的 `@decorator` 语法糖,和 GoF 的 Decorator 模式思想同源但并不是一回事。两者共享"在不修改原始定义的前提下包装它、叠加新行为"这个核心思想,但落地方式不同。GoF 的 Decorator 是面向对象范式下的一种结构方案,包装的是运行时的对象实例,装饰关系可以在程序运行过程中动态决定、动态调整。Python 的 `@decorator` 语法本质上是一个高阶函数用法:它接收一个函数或类作为参数,返回一个新的可调用对象替换掉原来的名字,这个包装动作发生在模块加载、定义发生的那一刻,而不是运行时任意时刻,一旦包装完成,后续没有办法在运行时再动态地给这个函数追加或去掉这层行为。可以说两者是同一种"包装增强"思想在面向对象范式与函数式范式下各自独立长出的表达方式,而不是谁是谁的简化版本。

### 2.4 代理模式:控制访问,而不是增加职责

代理模式的类结构和装饰器模式表面上非常相似:代理类和真实对象(Real Subject)实现同一个接口,代理内部持有一个指向真实对象的引用,客户端始终面向这个共同接口编程。区别在于代理存在的目的不是"叠加职责",而是"控制访问"——具体来说是控制这次调用要不要被转发、什么时候转发、转发给谁。这种控制可以体现为几种典型变体:

远程代理(Remote Proxy)针对真实对象实际运行在另一个进程或另一台机器上的场景。代理把方法调用的参数序列化、通过网络发送给远程的真实对象执行,再把结果反序列化传回本地,客户端调用一个远程对象的写法和调用本地对象完全一样,感知不到网络通信的存在。很多 RPC 框架里的客户端 stub 扮演的正是这个角色。

虚拟代理(Virtual Proxy),也就是通常所说的懒加载(lazy loading),针对真实对象创建开销很大的场景,比如加载一张高分辨率大图或者初始化一个很重的资源。代理一开始只是一个轻量的占位壳子,不持有任何昂贵资源,只有当客户端第一次真正调用需要访问真实数据的方法时,代理才在这一刻去创建真实对象并转发调用,后续调用直接复用已经创建好的实例。

保护代理(Protection Proxy)针对访问权限控制的场景,代理在转发调用之前先检查调用者的身份或权限,足够才放行,不够则拒绝。把这部分校验逻辑集中放在代理这一层,而不是散落进真实对象内部,本质上是一种关注点分离——权限校验是横切关注点,不应该和真实对象的核心业务逻辑混在一起。

缓存代理是另一种常见的实践变体,代理记住此前调用的结果,如果后续调用的参数命中已缓存的结果就直接返回,不再转发给真实对象,用空间换时间减少重复计算或重复的远程调用开销。

回到代理和装饰器的区分问题:同样是"持有同接口对象、转发调用"的结构,判断到底该叫代理还是装饰器,要看这层包装存在的目的。如果目的是在原有行为基础上追加一项新的、独立的职责,并且允许多个这样的包装自由叠加组合,调用总是会被转发执行,只是多了一些副作用,这更接近装饰器的意图。如果目的是决定"要不要把调用转发下去、什么时候转发、转发给谁",甚至可能直接拒绝执行(权限不足时抛异常而不执行真实逻辑),这就是代理的意图。GoF 把"意图"列为区分设计模式的第一位标准而不是类图结构,原因也在这里:设计模式是对反复出现的设计问题及其解法的归纳和命名,同一种代码结构完全可以用来解决不同性质的问题,只按结构分类会把目的完全不同的两类场景错误地混为一谈。

### 2.5 外观模式:给复杂子系统一个简化的正门

外观模式(Facade)针对的场景是一个复杂子系统由多个相互协作的类组成,客户端如果要完成一项业务操作,需要了解这些类各自的职责、正确的调用顺序、彼此之间的依赖关系,学习和使用成本很高;而且一旦客户端代码直接依赖了这些具体类,子系统内部结构今后的任何调整,所有直接依赖它的客户端代码都要跟着改动,耦合面很大。外观模式的做法是新增一个 Facade 类,内部持有对子系统各个类的引用,针对客户端最常见的几种业务场景,把子系统内部原本需要客户端自己正确编排的多步骤调用顺序,封装成 Facade 对外提供的少数几个简单方法。客户端只需要调用这几个方法就能完成原本繁琐的操作,完全不需要知道子系统内部由哪些类构成:

```
class OrderFacade:
    inventory: InventoryService
    payment: PaymentService
    shipping: ShippingService

    place_order(cart, card):
        self.inventory.reserve(cart.items)
        self.payment.charge(card, cart.total)
        self.shipping.schedule(cart.items)
```

客户端只需要调用 `place_order`,而不必了解库存、支付、物流这三个子系统各自的接口和正确的调用顺序。这个效果和迪米特法则(最少知识原则,Law of Demeter)高度吻合——该法则要求一个对象应该尽量少了解其他对象的内部细节,只和"直接的朋友"打交道。外观模式把客户端原本需要认识的一整批子系统内部对象,压缩成只需要认识 Facade 这一个"直接的朋友",子系统内部的重构变化只要不影响 Facade 暴露的方法签名,就不会波及客户端。

需要澄清的是,外观模式不代表子系统内部类从此必须被完全屏蔽——有特殊精细化需求的高级客户端仍然可以绕过 Facade 直接访问内部类,Facade 提供的是覆盖大多数常见场景的默认便捷路径,而不是唯一合法的访问路径。也要提醒的是,外观模式的价值建立在子系统本身确实存在一定复杂度这个前提之上,如果子系统本来就只有一两个类、调用方式已经足够简单,强行包一层 Facade 不会带来任何耦合隔离或简化调用的收益,反而平白多了一层没有意义的间接调用。此外,随着业务场景增多,如果不加区分地持续往同一个 Facade 里堆方法,这个 Facade 本身会膨胀成方法数量庞大、边界模糊的"上帝对象",这时候正确的做法不是放弃外观模式,而是按客户端实际的使用场景把这一个臃肿的 Facade 拆分成多个更聚焦的 Facade,比如把下单和支付拆成两个各自独立的 Facade。

### 2.6 组合模式:让部分和整体被一视同仁地对待

组合模式(Composite)解决的问题是如何表示"部分-整体"的层次结构,并且让客户端可以用完全相同的代码统一处理单个对象和由多个对象组成的整体。最经典的例子是文件系统:一个文件夹里可以包含文件,也可以包含子文件夹,子文件夹里又可以递归地包含更多文件和文件夹。如果客户端要计算一个文件夹的总大小,理想情况下不应该写"如果是文件就怎样、如果是文件夹就怎样"这样的条件分支,而是应该能对文件和文件夹调用同一个方法,得到统一处理的结果。

组合模式的结构是定义一个抽象的 Component 接口(比如都拥有 `get_size()` 方法),叶子节点(文件)和组合节点(文件夹)都实现这个接口。叶子节点的实现直接返回自身的大小;组合节点内部持有一组子 Component 的引用(可能是文件,也可能是子文件夹),它的实现是遍历这些子节点,对每一个都调用同样的 `get_size()` 方法并把结果累加起来:

```
interface FileSystemNode:
    get_size() -> int

class File implements FileSystemNode:
    get_size(): return self.byte_size

class Folder implements FileSystemNode:
    children: list[FileSystemNode]
    get_size():
        return sum(child.get_size() for child in self.children)
```

因为 Folder 的 `get_size()` 内部调用的是同一个 `FileSystemNode` 接口,不关心某个子节点具体是文件还是文件夹,这个递归结构可以自然地处理任意深度的嵌套层次,客户端代码从头到尾只需要认识 `FileSystemNode` 这一个接口。这种设计上有一个经典的取舍:是否要在抽象接口里就声明"添加子节点""移除子节点"这类只有组合节点才有意义的方法。如果声明了(透明式组合),叶子节点也要实现这些方法(通常是抛出异常或者空实现),换来的是客户端可以对叶子和组合节点用完全一致的方式调用,不需要做类型判断;如果不声明(安全式组合),叶子节点的接口更干净,但客户端在调用"添加子节点"之前必须先判断当前节点是不是组合节点,牺牲了一部分透明性来换取接口的类型安全。这两种做法各有取舍,不存在绝对正确的答案。此外,过深的递归层次在某些语言和运行环境下需要留意栈溢出的风险,这是组合模式在实践中一个容易被忽略的边界问题。

### 2.7 桥接模式:把"是什么"和"怎么实现"从设计之初就分开

桥接模式(Bridge)解决的问题是:当一个抽象概念存在多个变体、这些变体又各自可能有多种实现方式时,如果把"抽象的变体"和"实现的变体"放进同一条继承体系,类的数量会随着两个维度的组合数相乘式增长。经典的例子是"图形"和"渲染方式":图形可以是圆形、方形,渲染方式可以是用矢量渲染、用光栅渲染,如果用继承同时表达图形种类和渲染方式,就需要 圆形-矢量、圆形-光栅、方形-矢量、方形-光栅四个类,每新增一种图形或一种渲染方式,类的数量都要跟着乘。

桥接模式的解法是把这两个维度拆成两条独立的继承体系:一条是抽象部分(Abstraction),比如 Shape 及其子类 Circle、Square;另一条是实现部分(Implementor),比如 Renderer 接口及其具体实现 VectorRenderer、RasterRenderer。抽象部分不通过继承去表达渲染方式的差异,而是内部持有一个指向 Renderer 接口的引用,把渲染这件事委托给这个引用:

```
interface Renderer:
    render_circle(x, y, r)

class VectorRenderer implements Renderer: ...
class RasterRenderer implements Renderer: ...

class Shape:
    renderer: Renderer     # 桥,连接抽象与实现

class Circle extends Shape:
    draw(): self.renderer.render_circle(self.x, self.y, self.r)
```

这样一来,图形的种类数和渲染方式的种类数分别独立增长,不再相乘,新增一种渲染方式只需要新增一个 Renderer 的实现类,不需要触碰 Shape 这条继承体系,反之亦然。这里持有 Renderer 引用的这条纽带就是"桥"这个名字的来源。

桥接模式和适配器模式在结构上有相似之处——都是一个类持有另一个接口的引用并转发调用——但两者的时机和意图截然不同。桥接模式是在设计之初就有意识地把抽象和实现分离开,目的是让两者今后能够独立演化、独立扩展;适配器模式则是事后的补救措施,针对的是已经存在、且接口形状对不上的两个既成事实的类,目的是让它们能够勉强协同工作。简单说:桥接是提前规划好的解耦,适配器是发现不兼容之后的补丁。

### 2.8 享元模式:用共享换内存

享元模式(Flyweight)解决的问题是当系统里需要存在的细粒度对象数量非常庞大时,如果每一个逻辑上的对象都对应一个独立的实例,内存开销会变得难以承受。享元模式的解法是把每个对象的状态拆成两部分:内部状态(intrinsic state)是可以在多个对象之间共享、不随具体使用场景变化的部分;外部状态(extrinsic state)是因具体使用场景而异、不能被共享的部分。享元对象只存储内部状态,并且对同样的内部状态只创建一份实例、缓存起来重复使用;外部状态则不存在享元对象内部,而是在每次需要使用这个享元对象时,由调用方以参数的形式传进去。

一个常被用来说明享元模式的场景是游戏里渲染大量的树:一片森林里可能有成千上万棵树,每一棵树都需要知道自己的种类、纹理贴图、三维网格模型这些数据,但同一种树(比如同一种橡树)的纹理和网格模型对所有这一种树的实例来说都是完全相同的,不同的只是每一棵具体的树各自所在的坐标位置、大小缩放比例、朝向角度。如果给森林里的每一棵树都创建一个独立对象、各自持有一份完整的纹理和网格数据,内存开销会随着树的数量线性增长,而其中绝大部分数据(纹理、网格)其实是完全重复的。用享元模式重新设计,把纹理和网格这些不随具体某一棵树而变化的数据识别为内部状态,封装进一个享元对象(比如一个 `TreeType` 对象),同一种树类型只创建一个 `TreeType` 实例并缓存复用;而每一棵具体树的坐标、缩放、朝向这些外部状态,不存在享元对象里,而是在渲染的时候由调用方(比如森林场景对象)作为参数传给共享的 `TreeType` 实例的渲染方法。这样一来,不管森林里实际有多少棵树,内存里只需要保存种类数量级的 `TreeType` 实例(通常远小于树的总数),再加上每棵树各自记录的那一小份外部状态坐标数据,总的内存开销从"和树的总数成正比"降到了"和树的种类数成正比加上外部状态本身占用的少量空间",这正是享元模式希望达到的效果。

这里要划清享元模式和另一个容易被混淆的对象池模式的边界:两者虽然都涉及"复用已有对象、避免重复创建"这个表面相似的行为,但语义完全不同。享元模式共享的是不可变的、多个使用者可以同时并发引用的对象(比如上面例子里的 `TreeType`,任意数量的具体树都可以同时引用同一个 `TreeType` 实例,互不冲突);对象池模式管理的则是有状态、同一时刻只能被一个使用者独占的资源(比如数据库连接),使用者用完之后必须显式归还,归还之前这个实例不能被其他使用者借用。一个是"共享只读的公共部分",一个是"排队借用独占的可变资源",不能把两者混为一谈。

### 2.9 场景走查:两个具体设计过程

**场景一:咖啡店的加料订单系统。** 假设要为一家咖啡店设计订单系统,每一杯饮品有一个基础价格,顾客可以在基础饮品上任选加奶、加糖、加摩卡、加香草这几种配料,每种配料对应固定的加价。如果一开始就用继承来设计,很自然会想到给每种基础饮品定义一个子类,再为每一种配料组合定义一个新的子类,比如 `EspressoWithMilk`、`EspressoWithMilkAndSugar`。但配料一旦增加到四五种,组合数很快超过十几甚至几十种,子类数量随之膨胀,而且这些子类之间几乎全是重复代码,只是加价数字不同。用装饰器模式重新设计,第一步是识别出所有的基础饮品和所有的配料都应该实现同一个接口(比如都有 `cost()` 和 `description()` 方法);第二步是把每种基础饮品实现成这个接口最朴素的实现类;第三步是把每种配料实现成一个装饰器类,装饰器内部持有一个指向被装饰对象(可能是某个基础饮品,也可能是另一个配料装饰器)的引用,`cost()` 的实现是"被装饰对象的价格加上自己这一层的加价",`description()` 类似地在被装饰对象描述的基础上追加自己这一层配料的名字。要下单一杯加奶又加糖的美式咖啡,只需要在运行时构造 `SugarDecorator(MilkDecorator(Americano()))`,不需要提前定义任何新类。这个走查过程说明了装饰器模式真正发挥价值的判断标准:当一组功能是可选的、彼此独立的、且组合数量会随功能种类数增长时,装饰器模式比继承更适合。同时也要留意前面提到的顺序问题——如果某些配料的加价涉及百分比折扣而不是固定加价,叠加顺序就可能实质性影响最终价格,需要在设计装饰器组合的组装逻辑时格外小心。

**场景二:大规模场景渲染中的内存优化。** 延续享元一节里森林渲染的例子,可以把完整的设计决策过程再走一遍。设计者最初的做法往往是直接给每一棵树定义一个包含全部字段(位置、缩放、朝向、纹理引用、网格数据)的对象,当森林规模扩大到数万棵树时,这种做法在内存上很快难以为继,尤其是纹理和网格数据本身体积可观、但同一树种下这些数据其实完全相同。诊断问题的第一步是逐个字段判断:哪些字段的值会因为具体是哪一棵树而不同(位置、缩放、朝向,这些是外部状态),哪些字段的值只取决于树的种类、和具体是哪一棵树无关(纹理、网格,这些是内部状态)。诊断清楚之后,把内部状态封装进一个独立的享元对象(树种),为每一个不同的树种只创建一个实例,用一个工厂或者缓存表按树种名字查找并复用已有实例,避免重复创建;森林场景对象本身则只保留一个轻量的列表,列表里每一项只包含外部状态(位置、缩放、朝向)以及指向对应享元实例的引用。渲染整片森林时,遍历这个轻量列表,对每一项取出外部状态,连同这份外部状态一起传给对应享元实例的渲染方法,由享元内部状态(纹理、网格)结合外部状态(具体位置朝向)共同完成这一棵具体树的绘制。这个走查过程说明享元模式生效的前提有两个,缺一不可:一是确实存在数量巨大、有共性的细粒度对象;二是能够清晰地把状态拆分成不随实例变化的内部状态和随实例变化的外部状态,如果这两个状态之间界限模糊、或者需要共享的部分本身占比很小,享元模式带来的内存收益就会很有限。

### 2.10 常见误区澄清

结构型模式的学习过程中,以下几种误解出现的频率相当高,值得专门澄清。

第一种误区是"Decorator 模式和 Python 的 `@decorator` 语法是一回事"。这一点在 2.3 节已经详细澄清:两者共享包装增强的思想,但 GoF Decorator 面向的是运行时对象实例、支持动态的层层叠加,Python 的 `@decorator` 语法面向的是定义时的函数或类、在模块加载那一刻一次性完成包装,不能把两者简单划等号,更不能认为写了几个 `@decorator` 就等于用上了设计模式意义上的装饰器模式。

第二种误区是"Adapter 和 Bridge 没有区别,反正都是持有另一个接口的引用转发调用"。区别的关键在于时机和意图:适配器是针对已经存在、事后发现接口不兼容的两个类做的补救,目标是让它们能凑合协同工作;桥接是在设计之初就有意识地把抽象和实现拆成两条独立的继承体系,目标是让二者能够各自独立扩展演化。一个是亡羊补牢,一个是未雨绸缪,不能因为结构相似就认为它们是同一件事。

第三种误区是"Facade 就是把很多方法或者很多类塞进同一个类里,图个方便"。真正的外观模式要求 Facade 暴露的方法对应客户端真实的、有意义的业务场景,是对子系统内部多步骤调用的重新组织和简化,而不是把子系统里每个类的方法不加选择地照搬透传一遍;如果不做取舍地持续堆方法,Facade 本身会变成臃肿的上帝对象,这恰恰是外观模式想要避免的结果,而不是它的正常形态。

第四种误区是"只要结构上是'一个类持有另一个接口的引用并转发调用',就都是同一种模式,模式的名字无所谓"。这是对整个结构型模式家族最根本的误解。适配器、装饰器、代理三者的类图可以画得几乎一模一样,但它们分别回答的是"接口形状对不上怎么办"、"怎么不用继承也能叠加职责"、"怎么控制访问"这三个完全不同的问题。GoF 明确把"意图"列为区分设计模式的第一位标准,而不是 UML 类图上的结构,原因就在于设计模式本质上是对反复出现的设计问题及其解法的归纳和命名,同一种结构可以承载不同的意图,脱离意图去死记结构,记住的只是空壳。

第五种误区是"享元模式和对象池模式是一回事,反正都是复用对象、节省资源"。2.8 节已经说明,享元共享的是不可变的、可以被多个使用者同时并发引用的对象,对象池管理的是有状态的、需要独占并显式归还的资源,两者的使用语义完全不同,不能因为都涉及"复用"就混为一谈。

### 2.11 本章小结

结构型模式关心的共同问题是:在对象的创建方式已经确定、对象之间的动态交互还不是重点的前提下,怎样把已经存在的类和对象组合、拼装成更大、更灵活的结构。这七种模式——适配器、装饰器、代理、外观、组合、桥接、享元——分别针对接口不兼容、继承扩展导致组合爆炸、访问需要受控、子系统过于复杂、部分与整体需要一视同仁、抽象与实现需要独立演化、细粒度对象数量过多这七类具体问题,给出了各自的解法,但方法论内核是相通的:通过让一个对象持有另一个对象的引用并在转发调用的过程中插入额外的逻辑,用组合的方式而不是继承的纵向扩展来达成设计上的灵活性。正因为好几个模式的类图结构非常相似,真正区分它们的从来不是"代码长什么样",而是"当初为什么要这么设计"——这也是为什么本章反复强调要回到每个模式的意图去理解它,而不是死记 UML 图。读完本章之后,建议对照 `learning/design-patterns-mastery/src/tier1_shallow/dp_pat_structural_patterns.py` 里对应的 17 个 DeepPoint 追问链逐条练习,把本章讲清楚的原理放到具体的追问场景里去检验是否真正吃透,尤其要留意其中关于模式相互区分、以及"什么时候不该用这个模式"的追问,那往往是从"知道模式"到"会用模式"之间最容易露怯的地方。

## 第三章 行为型模式(一)

### 3.0 引言：从"怎么搭"到"怎么互动"

前面讨论结构型模式的时候，我们关心的核心问题是"对象和类怎么组合成一个更大的、静态的结构"——一个对象由哪些部件构成、一个接口怎样被适配成另一个接口、一堆细粒度的小对象怎样被组织成一棵树。这些问题回答的都是"系统在某一个时刻长什么样子"。

行为型模式关心的是另一件事：系统运行起来之后，对象之间到底是怎么发消息、怎么协作、职责又是怎么在这些对象之间分配的。同样是"一个电商系统要支持多种促销折扣"这件事，结构型模式会问"这个折扣计算器应该挂在商品对象上还是订单对象上、要不要用装饰器一层层叠加优惠"；行为型模式问的则是"当订单结算的时候，究竟是谁负责决定用哪种折扣算法去算、算法本身应该被写成什么形状，才能在促销规则频繁变化时不用大动干戈"。一个偏静态结构，一个偏运行时协作，这是这两大类模式最根本的分野。

本章要系统讲解的五个行为型模式——策略、观察者、命令、模板方法、迭代器——虽然表面上分别处理"选算法""发通知""封请求""定流程""走元素"这五件不同的事情，但它们背后共享一个非常朴素的设计直觉：把"容易变化、需要被替换、或者需要在运行时被灵活组织"的那一小部分行为，单独抽出来做成一个独立的抽象（一个接口、一个对象），而不是把它硬编码在使用它的那段代码内部。掌握了这个共同直觉，再去看每个模式各自的结构细节，会发现它们其实是同一个思路在不同场景下的具体变体。

### 3.1 策略模式（Strategy）：把"选哪种算法"和"怎么用这个算法"分开

策略模式要解决的问题可以用一句话概括：当一个系统里存在一族可以互相替换的算法（比如根据订单金额、用户等级、时令活动计算出不同的折扣），而这些算法未来还可能继续增加或调整时，怎样组织代码才能让"新增一种算法"这件事，不必去改动已经写好、跑得好好的那些代码。

最朴素、也是大多数人上手时的第一反应，是把所有算法逻辑塞进一个方法里，用一长串 if-elif（或者 switch）按照类型分支挑出对应的计算逻辑去执行。这种写法在算法种类很少、且长期稳定的时候完全可以接受，问题在于一旦种类持续增长，这个方法会越改越长、分支之间彼此纠缠，任何一次修改都有可能牵连到看起来毫不相干的另一个分支，测试和排查的成本也随着分支数量线性甚至更快地上升。

策略模式的做法是：先声明一个策略接口，比如约定一个 `calculate(order)` 方法签名；然后把每一种具体算法各自实现成一个实现这个接口的具体策略类；使用算法的那一方（GoF 里称为 Context）不再自己写死某一种算法，而是内部持有一个指向策略接口的引用，把真正的计算工作委托给这个引用背后的具体对象去完成。用伪代码表达大致是：

```
interface DiscountStrategy:
    def calculate(order) -> amount

class NoDiscount(DiscountStrategy): ...
class PercentageOff(DiscountStrategy): ...
class FullReductionCoupon(DiscountStrategy): ...

class Order:                      # 相当于 Context
    def __init__(self, strategy: DiscountStrategy):
        self._strategy = strategy
    def final_price(self):
        return self._strategy.calculate(self)
```

新增一种促销规则，只需要新增一个实现 `DiscountStrategy` 的类，`Order` 和其他已有的策略类都不需要任何改动——这正是"对扩展开放、对修改关闭"的开闭原则在这里的体现。反过来说，如果折扣规则几乎不会再变、种类也很少，硬要引入一整套接口和类，反而是一种没有实际收益、只是徒增代码层次的过度设计；策略模式是否划算，最终要看算法变化的频率和数量，而不是无条件地认为"用了模式就一定更好"。

值得特别强调的一点是策略模式和依赖注入（Dependency Injection）之间的关系：Context 持有的那个策略引用，通常不是自己在内部 new 出来的，而是由外部——可能是调用方，也可能是一个专门的容器或工厂——在构造 Context 的时候"注入"进去的。也就是说，策略模式描述的是"算法可以被替换"这个结构，而依赖注入描述的是"这个可替换的依赖具体是怎么被组装、被交给需要它的对象"这件事，二者经常配合出现：策略接口负责定义可替换的契约，依赖注入负责在运行时把一个具体实现"喂"给 Context。策略模式本身并不回答"该在什么条件下选用哪个具体策略"——这个选择的职责通常仍然落在客户端或者一个额外的工厂/注册表身上；策略模式解开的只是 Context 和"算法怎么执行"之间的耦合，"该选哪个算法"这一层判断并没有消失，只是被转移到了另一个地方。

另外一个经常被问到的问题是：在支持一等函数的语言（函数可以像普通值一样被赋值、传递）里，是不是完全没必要写一整套策略类了？答案是：对于那些不需要携带额外状态、只是一个纯粹计算过程的算法，直接传一个函数（或者一个闭包）作为参数，本质上和策略模式解决的是同一个问题，只是省去了"声明接口、为每种算法定义一个类"这一层结构，是一种更轻量的等价实现；而当某种算法除了核心计算逻辑之外还需要维护绑定的状态、或者需要对外暴露不止一个方法时，用类来组织会更自然。这提醒我们：设计模式描述的是要解决的问题和角色分工这一层抽象结构，而不是必须用某种具体语法（比如类和接口）才算数。

### 3.2 观察者模式（Observer）：一对多的状态变化通知

观察者模式解决的是另一类常见问题：一个对象（我们称它为 Subject，或者"被观察者"）的状态发生变化时，可能有若干个其他对象都需要知道这件事、并据此做出反应，而且这些"需要知道"的对象的数量和具体是谁，在设计的时候往往无法提前固定下来。观察者模式的结构是：Subject 内部维护一个观察者列表，并提供"注册"（attach/subscribe）和"注销"（detach/unsubscribe）两个方法让观察者加入或退出这个列表；每个观察者都实现一个约定好的更新接口（比如 `update()`）；当 Subject 自身状态发生变化时，遍历这个列表，逐一调用每个观察者的更新方法，完成通知：

```
class Subject:
    def __init__(self):
        self._observers = []
    def attach(self, observer):
        self._observers.append(observer)
    def detach(self, observer):
        self._observers.remove(observer)
    def notify(self):
        for observer in self._observers:
            observer.update(self)
```

这里有一个经常被混用、但其实值得区分的细节：经典的 Observer 模式里，Subject 和 Observer 是直接互相知道的——Subject 持有观察者对象的直接引用，调用是同步的方法调用，两者通常运行在同一个进程、同一个对象图里。而我们日常挂在嘴边的"发布-订阅"（Publish-Subscribe）虽然解决的也是"一对多通知"这个问题，但它通常在发布者和订阅者之间插入了一个独立的中介——一个消息代理、事件总线或者消息队列。发布者只管把消息发给这个中介，完全不知道也不关心谁订阅了、订阅了多少个；订阅者也只和这个中介打交道，不直接持有发布者的引用。换句话说，经典 Observer 是"没有中介的直接观察"，发布-订阅是"引入了中介、把双方进一步解耦"的一种演化，中介的存在还顺带带来了跨进程、跨网络、异步投递等能力，这是经典 Observer 单靠语言内的对象引用做不到的。

观察者模式有一个非常容易在实际系统里踩中、而且初学者往往想不到根本原因的坑：内存泄漏。原理是这样的——Subject 要具备通知能力，就必须在自己内部的列表里保存对每一个已注册观察者的引用，而且这通常是一个强引用。如果某个观察者对象的生命周期本该比 Subject 短（典型场景是一个页面切换后就该销毁的组件，订阅了一个全局的、生命周期很长的数据源），但这个观察者在自己被销毁之前忘了调用一次"注销"，Subject 的列表里就会一直留着一份指向它的强引用。垃圾回收器判断一个对象能否被回收，依据的是"是否还存在指向它的可达引用"，只要 Subject 那份引用还在，即便程序里其他所有地方都已经不再使用这个观察者，它占用的内存也不会被释放——这块内存就这样被一份"被遗忘的订阅关系"悄悄占住，而且这类问题往往不会表现为明显的错误，只会在系统长时间运行后逐渐表现为内存缓慢上涨，比一般的对象引用泄漏更难定位。工程上常见的两种应对方式：一是把"销毁前必须调用取消订阅"写成组件生命周期里不可省略的一步、并通过约定或者代码检查去保证它真的被执行；二是让 Subject 用弱引用而不是强引用去持有观察者列表，这样即使观察者忘了主动注销，一旦程序其他地方不再持有它，垃圾回收器依然可以正常回收——不过这个方案也有代价，弱引用指向的观察者有可能在 Subject 真正发出下一次通知之前就已经被提前回收，导致这次通知被悄悄跳过，观察者错过了本该收到的更新，所以它是"用可能漏掉个别通知，换取不会内存泄漏"的一种权衡，而不是没有代价的万能解法。

### 3.3 命令模式（Command）：把"一次操作"变成一个可以传递的对象

命令模式的核心动作是：把一次请求——包括"要在谁身上执行""执行什么操作""需要什么参数"这些信息——打包封装成一个独立的对象，这个对象对外统一暴露一个 `execute()` 方法。发起请求的一方不再直接调用接收者身上的具体方法，而是构造（或者取出）一个命令对象，调用它的 `execute()`：

```
class Command:
    def execute(self): raise NotImplementedError
    def undo(self):    raise NotImplementedError

class InsertTextCommand(Command):
    def __init__(self, document, position, text):
        self._document, self._position, self._text = document, position, text
    def execute(self):
        self._document.insert(self._position, self._text)
    def undo(self):
        self._document.delete(self._position, len(self._text))
```

把"请求"从一次转瞬即逝的方法调用，变成一个实实在在、可以被保存下来的对象，这一步转变一旦完成，很多原本难以实现的能力就变得顺理成章。第一是撤销/重做：只要命令对象在 `execute` 之外，再额外实现一个 `undo` 方法（记录下执行前需要恢复的状态，或者直接实现一个反向操作），程序就可以把每一个执行过的命令对象压入一个历史栈；撤销就是弹出栈顶命令、调用它的 `undo`，重做就是把被撤销的命令挪到另一个栈里、需要时再调用回它的 `execute`。第二是排队和日志：因为命令是一个对象，它可以被放进一个列表里等待稍后统一执行（排队），也可以被序列化写进日志用于事后审计或者故障恢复时重放（日志与重放）——这两种能力共同的前提，都是请求已经从一次直接调用变成了一个可以被保存、传递的对象，如果请求还停留在"直接调用一个方法"的形式上，这些能力根本无从谈起。

命令模式还支持把多个具体命令组合成一个"宏命令"（Composite Command）：宏命令自己也实现同样的 `execute`/`undo` 接口，内部持有一个子命令的有序列表。它的 `execute` 按顺序依次调用每个子命令的 `execute`；但它的 `undo` 必须按照与执行相反的顺序去依次撤销——这是因为如果子命令之间存在先后依赖（后一步建立在前一步已经生效的基础上），撤销时必须先撤掉最后生效的那一步，再依次向前撤，顺序反了就可能让撤销作用在一个不符合预期的中间状态上。

和策略模式里的讨论类似，在支持一等函数的语言里，一个不需要支持撤销、也不需要被序列化的简单命令，完全可以直接用一个闭包代替，闭包捕获接收者和参数就相当于命令对象持有的那些字段，调用闭包就相当于调用 `execute`；但一旦需要撤销（闭包很难再自然地附带一个配套的反向操作和执行前状态），或者需要把命令持久化用于日志重放（闭包捕获的执行环境通常无法完整序列化），带有明确字段和方法的命令类仍然是更稳妥的选择。

### 3.4 模板方法模式（Template Method）：父类定骨架，子类填细节

模板方法模式处理的是这样一种情况：几种具体流程（比如若干种数据导入方式）在整体步骤和执行顺序上是完全一致的——都要"读取数据源→校验→转换→写入"，只是其中某一两个具体步骤的实现细节不同。模板方法模式把这个固定不变的整体流程写在父类的一个方法里（这个方法通常应当被声明为不可覆盖，或者至少约定不应该被子类覆盖），这个方法依次调用一系列子步骤；有些步骤在父类里已经给出了通用的默认实现，另一些步骤则被声明成抽象方法、或者是一个默认什么都不做的"钩子方法"（hook method），留给子类根据需要去覆盖：

```
class DataImportTemplate:
    def run(self):                    # 模板方法：整体骨架固定
        raw = self.read_source()      # 抽象步骤，子类必须实现
        cleaned = self.validate(raw)  # 父类给出默认实现
        self.before_load_hook()       # 钩子方法：子类可选择是否覆盖
        self.load(cleaned)            # 抽象步骤，子类必须实现

    def before_load_hook(self):
        pass                          # 默认空实现，不强制子类关心
```

钩子方法和抽象方法的区别在于：抽象步骤是子类必须提供实现的，父类要么不给实现、要么直接声明为抽象；钩子方法则给出了一个默认（通常是空的）实现，子类可以选择覆盖它来"钩入"流程的某个特定环节、施加一些可选的额外行为，也可以完全不管它、直接使用父类给的默认行为。钩子方法的价值在于给子类留了一个"如果需要就用、不需要就忽略"的可选扩展点，而不强迫每个子类都必须对这个环节表态。

模板方法模式体现的是所谓"好莱坞原则"（"别调用我们，我们会调用你"）：不是子类主动去编排调用父类或者其他步骤，而是父类的模板方法在自己执行到某个环节时，反过来调用子类已经实现好的具体步骤——控制权始终在父类手里，子类只是被动地在被调用的时机提供细节。

这里非常适合和策略模式做一次对照，因为两者的动机（让算法的某部分可以变化）看起来很像，但实现路径和可变粒度完全不同：模板方法用的是继承，子类通过覆盖父类里留出的步骤方法来定制细节，可变的粒度是"算法内部的某几个步骤"，整体流程结构本身固定不变，而且一个子类实例一旦创建，它对各步骤的实现在类定义时就已经确定，运行时不能只替换某一步、保留其他步骤不变；策略模式用的是组合，Context 持有一个指向策略接口的引用，可变的粒度是"整个算法本身"，替换策略就是把这个引用重新指向另一个策略对象，同一个 Context 在运行时可以先后使用不同策略，不需要像模板方法那样通过定义新的子类才能获得新的行为组合。一句话总结：模板方法固定流程、只变某几步，靠继承、粒度是步骤级；策略模式整体可换、粒度是算法级，靠组合、可以在运行时动态切换。

因为模板方法基于继承，它也天然继承了继承关系本身固有的"脆弱基类问题"：父类看似无关的实现调整（比如改变了模板方法内部调用步骤的顺序、给某一步加了新的前置条件）可能在子类完全不知情、代码一行没改的情况下，悄悄改变子类的实际运行行为，而这种变化通常不会有任何编译期提示。另一个具体的坑是：如果父类构造函数里直接或间接调用了模板方法或者某个会被子类覆盖的步骤，在很多语言里子类自己的字段这时候可能还没有完成初始化，被调用到的子类步骤如果依赖了这些字段，就会读到出乎意料的空值或默认值。缓解手段包括把父类模板方法的调用契约写清楚、当作可能影响所有子类的破坏性变更去谨慎对待，以及严格避免在构造函数里调用任何可能被子类覆盖的方法。

### 3.5 迭代器模式（Iterator）：遍历，但不暴露内部怎么存的

迭代器模式要解决的问题是：一个聚合对象（自定义的容器）内部可能用数组、链表、树或者哈希表这类完全不同的数据结构去组织它包含的元素。如果调用方遍历时直接访问这个聚合对象的内部结构——比如直接按下标访问内部数组，或者直接拿到链表头节点顺着指针走——调用方的代码就和这个聚合对象具体的实现方式绑死了：一旦内部实现从数组换成链表，所有这样写的调用方代码都要跟着改。

迭代器模式的做法是让聚合对象提供一个统一的方法（比如 `iterator()`）返回一个迭代器对象，这个迭代器对象内部封装了"怎么找到下一个元素"这套具体逻辑，对外只暴露"还有没有下一个"和"取下一个"这样统一、与内部实现无关的接口。调用方只面向这套统一接口写遍历代码，完全不需要关心聚合对象内部到底用了什么数据结构；哪怕内部实现将来发生变化，只要对应地提供一个新的迭代器实现，调用方的遍历代码可以完全不动。

这也解释了为什么几乎所有现代语言都把这个模式直接内建到语言层面：既然"遍历一个容器"是一件几乎所有程序都会反复做的事情，把迭代器协议做成语言或标准库的一部分（Python 的 `__iter__`/`__next__`，Java 的 `Iterable`/`Iterator` 接口），就能让所有容器类型共享同一套 `for ... in ...` 语法，不用每次自己实现遍历都从零发明一套接口。以 Python 为例，一个对象只要实现 `__iter__`（返回一个迭代器对象）和 `__next__`（每次返回序列中的下一个元素，没有更多元素时抛出 `StopIteration`），就自动获得了被 `for` 循环、内建 `iter()`/`next()`、以及各种推导式使用的能力；`for` 循环在底层做的事情，正是不断调用 `__next__` 直到捕获 `StopIteration` 为止。Python 选择用抛异常表达"遍历结束"，而不是像有些语言那样单独设计一个 `hasNext()`，是把"取值"和"判断是否结束"合并成了一次调用，减少了每一步都要多问一遍的开销。另外值得注意的是，可迭代对象（实现 `__iter__`）和迭代器对象（同时实现 `__iter__` 返回自己和 `__next__`）在概念上是分开的：每次调用聚合对象的 `__iter__` 都应该返回一个全新的、独立维护自己遍历进度的迭代器，这样同一个聚合对象才能被多次同时、互不干扰地完整遍历——如果聚合对象自己直接兼任迭代器、把遍历进度存成自己的一个内部状态，两次并发遍历就会共享同一份进度、互相打乱彼此的位置。Python 的生成器函数（用 `yield`）是实现这套协议一种非常轻量的写法：函数体写起来像一段普通的顺序代码，执行到 `yield` 就自动暂停并交出一个值，下次被 `next()` 调用时再从暂停处继续，遍历进度由运行时自动保存在这个暂停的调用帧里，不需要手写一个显式维护位置字段的类。

这里还有一个所有语言、所有容器实现都要小心的通用陷阱：在遍历一个聚合对象的同时，对这个聚合对象本身做增删这类结构性修改。绝大多数迭代器依赖某种内部位置指针或索引记录"当前走到哪了"，一旦遍历期间元素被增删，底层存储里元素的位置会整体偏移，迭代器持有的那个位置就可能和修改后的实际数据对不上，结果或者是漏掉本该遍历到的元素、或者是重复遍历到同一个元素，或者（很多语言和标准库会主动检测这种情况）直接抛出运行时异常提前暴露问题——这本质上是因为迭代器的正确性建立在"聚合对象在整个遍历期间结构保持稳定"这个隐含假设上，这个假设一旦被打破，各实现只是选择了不同的应对方式（静默产生不一致结果，还是主动报错），问题本身并不会消失。

### 3.6 场景走查：两个具体例子

**场景一：电商促销折扣的重构过程。** 设想一个下单结算流程，最初只有"无折扣"和"满减券"两种情况，写在一个 `calculate_price` 方法内部用 if-else 分支处理。后来运营陆续提出"按用户等级打折""限时百分比折扣""叠加优惠券再打折"等多种规则，这个方法逐渐膨胀成一长串相互嵌套的条件判断，每次新增一种规则都要在这个方法内部找准位置插入新分支，稍不留神就可能影响到旁边某个已经上线、工作正常的分支。这时候引入策略模式的重构路径是：先给"计算最终价格"这个行为定义一个策略接口，让"无折扣""满减券""等级折扣""限时折扣"分别成为实现这个接口的独立类，各自只需要关心自己那一种规则的计算逻辑；订单对象（Context）改成持有一个策略引用，结算时把计算工作委托给当前配置的策略对象。这样一来，运营再提出一种新规则，只需要新增一个策略类，订单对象和其它规则类都不必改动；至于订单在具体下单时到底该用哪个策略，可以由一个专门的策略选择器（根据用户等级、当前是否在活动期等条件挑出对应策略）来集中决定，避免让"该用哪个策略"这个判断逻辑重复散落在多处调用代码里。

**场景二：文本编辑器的撤销/重做栈。** 一个文本编辑器要支持插入文字、删除文字、查找替换等操作的撤销和重做。做法是把每一种用户操作都封装成一个实现了 `execute`/`undo` 的命令对象：插入操作的 `execute` 往文档指定位置插入文字，`undo` 则删除刚插入的那段文字；删除操作反过来，`execute` 删除一段文字前先记录下被删除的内容，`undo` 把这段内容重新插回原位置。编辑器维护一个历史栈，每执行一个命令就把它压入栈顶；用户按下撤销，就弹出栈顶命令调用它的 `undo`，同时把这个命令挪进一个"重做栈"；用户按下重做，就从重做栈里取出命令重新调用它的 `execute`。对于"查找替换全部"这种一次操作实际上是对文档多处做了同样修改的场景，可以把这次操作实现成一个宏命令，内部持有对每一处被替换文字分别生成的一个个体命令；宏命令的 `execute` 按文档顺序依次替换，`undo` 则必须按相反顺序把每一处替换逐一还原，这样才能保证多处替换互相之间如果存在位置依赖时，撤销不会撤出一个错乱的中间状态。

### 3.7 常见误区澄清

**误区一：观察者模式必须要用专门命名为 Observer/Subject（或者 Subscriber/Publisher）的接口，才算是用上了这个模式。** 这是把模式和某一种具体命名习惯搞混了。观察者模式描述的是"一对多依赖关系、状态变化时统一通知所有依赖者"这个结构性意图，只要某种机制实现了这个意图——不管它叫 `on_change` 回调注册、事件监听器列表，还是别的名字——在设计层面就是在应用观察者模式，接口叫什么名字并不是判断标准。

**误区二：策略模式和模板方法模式没有本质区别，反正都是把容易变的部分抽出来。** 两者的动机确实相似，但实现手段和可变粒度完全不同：模板方法靠继承、只让算法里的某几个步骤可变，整体流程骨架固定，一个子类的行为组合在实例创建后基本是定死的；策略模式靠组合、把整个算法作为可插拔的对象整体替换，同一个 Context 在运行时可以反复切换不同策略。混淆这两者，容易在该用组合实现灵活切换的场景里，错误地套用继承体系，导致每新增一种行为组合就要新增一个子类，反而失去了本该有的灵活性。

**误区三：命令模式就是把一次函数调用包一层类，纯属多此一举。** 如果一个操作只需要执行一次、不需要撤销、也不需要被排队或者记录，把它包装成一个命令对象确实是不必要的开销，直接调用或者传一个函数就够了。但命令模式真正的价值在于：一旦请求被对象化，就自然获得了撤销/重做（配合历史栈）、排队（命令对象可以被暂存到列表里延迟执行）、日志与重放（命令对象可以被序列化持久化）这些能力，这些能力都建立在"请求是一个可以被保存和传递的对象"这个前提上，是单纯的一次函数调用无法提供的。是否值得引入命令模式，取决于是否真的需要这些能力，而不是对"包一层"这件事本身的价值判断。

**误区四：Python 时代 for 循环底层自动处理了一切，自己写的容器类内部想怎么实现遍历都无所谓，不需要认真遵守 `__iter__`/`__next__` 这套协议。** 恰恰相反，正因为语言把迭代器模式内建成了统一协议，遵守这套协议才能让自定义容器无缝享受到 `for` 循环、内建函数、推导式等一整套语言设施；如果不遵守协议、只是自己另外发明一套遍历方法，这个容器就无法被这些通用语法直接使用，等于放弃了语言层面这套本该带来的一致性红利。同时前面提到的两个细节——遍历期间不要对容器做结构性增删、`__iter__` 每次都应返回一个独立的新迭代器——并不会因为语言帮你写好了协议就自动被满足，这两点仍然需要实现者自己在设计容器类的时候有意识地保证。

### 本章小结

策略、观察者、命令、模板方法、迭代器这五个模式，表面上分别处理"选算法""发通知""封请求""定流程""走元素"这五件不同的事，但共同的主题是一致的：把系统里"容易变化、需要被替换、或者需要在运行时被灵活组织"的那部分行为，从使用它的代码里剥离出来，做成一个独立的抽象——一个接口、一个可以被传递的对象——从而让对象之间协作的具体方式可以在不改动大量既有代码的前提下被灵活替换或扩展。策略模式替换的是整个算法，模板方法固定骨架只替换步骤，命令把请求本身变成可以被保存和重放的对象，观察者让状态变化的通知关系可以动态增减，迭代器让遍历方式与容器内部实现解耦——它们是同一个设计直觉在五个不同问题场景下的具体投影。理解到这一层，再去看每个模式各自的结构细节、以及它们彼此之间容易混淆的边界，会比单纯背诵类图更容易记牢，也更容易判断一个具体场景到底该不该用、该用哪一个。掌握了这些系统性的原理之后，接下来可以去做 `tier2_deep` 里对应的 15 个 DeepPoint 追问链练习，把这里讲的"是什么、为什么"进一步落实到面试场景里"追问三层还能不能接得住"的具体考验上。

## 第四章 行为型模式(二)

### 4.1 引言:从"一对多通知"到"多对多协调"

上一章我们学习了行为型模式的第一组成员——观察者、策略、模板方法、命令、迭代器。回顾那一组模式的共同关切,大致可以概括为三类问题:一个对象状态变化后如何通知一批依赖它的对象(观察者)、一段算法里哪些步骤应该允许替换或者延迟到子类去实现(策略与模板方法)、以及如何把"一次请求"或者"一次遍历"本身封装成一个独立的对象来操作(命令与迭代器)。这些问题有一个共同特征:它们处理的大多是相对静态的结构——算法族是固定的、请求是一次性的、遍历的对象结构在遍历期间通常不发生变化。

本章要处理的六个模式——状态、职责链、中介者、备忘录、访问者、解释器——面对的是更复杂的动态场景。状态模式要处理的是"同一个对象在不同时间点表现出不同行为"这种随时间演化的问题;职责链要处理的是"一个请求应该交给谁处理,在编写代码的这一刻还说不准"这种不确定性;中介者要处理的是一组对象之间纠缠不清的相互引用关系;备忘录要处理的是"如何让时间倒流"这个听起来有点反直觉的需求;访问者要处理的是"如何在不惊动原有类结构的前提下,持续给它叠加新操作"这个开放式的扩展问题;解释器则处理一个更加专门化的问题——如何用面向对象的方式表达一门语言的文法规则。

把这六个模式放在一起看,可以发现它们共享一条隐藏的主线:都是在对象与对象之间的"关系"和"过程"上做文章,而不是在某个对象内部的数据结构上做文章。学习这一章时,建议不要孤立地记住每个模式的类图,而要不断追问一个问题——这个模式到底改变了对象之间原来是什么样的依赖关系,变成了什么样的依赖关系。这条主线贯穿了下面六节的讲解。

### 4.2 状态模式:让对象的行为随内部状态改变

先看一个几乎每个工程师都写过的代码形态:一个对象有一个表示当前状态的字段(比如订单的status),它对外暴露的每一个方法内部,都先判断一遍这个字段的取值,再决定具体执行哪一段逻辑。随着状态种类增多、每种状态下的行为差异变大,这一段条件判断会越写越长,而且同一个状态判断逻辑往往会在这个类的好几个方法里重复出现——凡是需要区分"现在是什么状态"的地方,都要再判断一次。更麻烦的是,当需要新增一种状态时,必须回头修改这个类里所有包含该判断的地方,遗漏一处就会留下一个隐蔽的行为缺口。

状态模式(State)给出的解法是:把"处于某个状态时应该表现出的行为"整体搬到一个独立的类里去,让所有状态类实现同一个抽象接口,而原来那个承载业务对象的类(称为上下文,Context)自己不再判断状态取值,只保留一个指向"当前状态对象"的引用,把外部调用统一转发给这个引用所指向的对象。简化的结构大致如下:

```
interface State:
    handle(context)

class PendingPaymentState implements State:
    handle(context):
        # 完成支付这件事发生后,由状态对象自己决定下一个状态是谁
        context.setState(PaidState())

class Context:
    state: State
    request():
        state.handle(self)
```

这里有两个关键设计决策值得强调。第一,状态切换的动作——也就是"决定下一个状态是谁、并把Context持有的引用替换掉"——通常写在具体状态类内部,而不是写在Context里。这样一来,状态的迁移规则是随着状态类一起演化的,新增一种状态只需要新增一个类,不需要触碰已有的任何代码,天然满足开闭原则。第二,正因为状态转换的知识被分散到各个状态类里,状态类之间难免要相互了解"下一个该轮到谁",这是状态模式引入之后必须承受的一点耦合,不是没有代价的免费午餐。

一个经常被问到的问题是:状态模式和策略模式(上一章已经学过)在类图上几乎画不出区别——都是Context持有一个接口引用,具体行为委托给引用背后的实现类。真正的分野不在结构,而在于"谁来决定用哪个实现"以及"这个决定在对象生命周期里是否会自己发生变化"。策略模式里,具体策略之间通常是平级、互不知情的,选用哪个策略是外部调用方在使用前一次性注入的决定,策略对象自己不会在运行过程里主动把自己换成另一个策略;状态模式里,状态之间存在明确的先后转换关系,而且这个转换往往由状态对象自己在处理完一次请求之后主动触发,不需要外部每次重新指定。换句话说,策略是"外部挑一个算法来用",状态是"对象自己知道它已经变成了另一种东西"。这也解释了为什么不能简单地把状态模式看成策略模式的一个特例然后共用一套代码——一旦具体实现类需要承担"触发向另一个实现切换"这个职责,它就必然要持有对Context的引用或者了解其他实现类的存在,这是纯粹的策略模式设计中本不需要、也应当避免引入的耦合。

最后要提醒的是,状态模式并非任何带状态字段的类都值得引入。如果状态种类很少(两三种)并且转换规则长期稳定,一个清晰的条件分支往往比拆出一堆状态类更直观、维护成本更低。状态模式真正划算的场景,是状态数量较多、每种状态下的行为差异明显、并且状态会随着业务演化被频繁地增加或调整——这时候用多态替换条件分支带来的收益,才能覆盖引入额外类所付出的复杂度成本。

### 4.3 职责链模式:请求沿着一条链寻找归宿

再看另一类常见写法:一个请求进来,需要按优先级依次尝试若干个处理逻辑,直到某一个逻辑接受它为止,这段代码往往被写成一长串if-elif,每个分支对应一种处理方式。这种写法的问题在于,发起请求的一方必须在代码里明确知道所有候选处理逻辑及其顺序,新增、删除或者调整某个处理环节都要直接改动这一串分支代码。

职责链模式(Chain of Responsibility)把这些候选处理者做成一组对象,每个对象持有指向"链上下一个处理者"的引用,请求从链的头部进入后,每个处理者依次判断自己能否处理:能处理就处理,不能处理就转交给自己持有的下一个处理者,如此沿着链条传递下去,直到某个处理者接手或者传到链尾。结构大致如下:

```
abstract class Handler:
    next: Handler

    setNext(h):
        self.next = h
        return h

    handle(request):
        if self.canHandle(request):
            return self.process(request)
        elif self.next is not None:
            return self.next.handle(request)
        else:
            return None
```

这个结构真正带来的收益,是把"发起请求的一方"和"具体由谁来处理"这两件事彻底解耦。发起方只需要拿到链头部第一个处理者的引用,调用一次即可,完全不需要知道链上究竟有多少个处理者、分别是谁、按什么顺序排列。新增一个处理者时,只需要调整"组装这条链"的代码,把新处理者插入到合适的位置、接好前后引用,发起方的代码不需要任何改动。这一点和一长串if-elif的本质区别不在"是否用了链表这种数据结构",而在于处理者的集合是否可以在运行期动态调整、且不牵连发起方的代码。

这里必须提醒一个容易被忽视、但确实属于该模式固有的风险:如果一个请求沿着整条链传递下去,恰好没有任何一个处理者认为自己能够处理,请求就会在链尾悄无声息地消失。这不同于抛出异常——异常至少会让问题在第一时间暴露出来,而"没有人处理"这件事,如果链的设计者没有在链尾专门放一个兜底处理者(记录日志、报告一个明确的"未处理"结果),发起方既不会得到结果,也不会得到任何错误提示,只会在下游业务对不上账的时候才被追溯发现,排查成本远高于一般的异常。因此一条设计完整的责任链,通常需要在链尾显式地放置一个兜底节点,或者在链的入口处对"是否有人处理"做一次独立于链内部实现的校验。

### 4.4 中介者模式:把网状依赖收拢成星形依赖

设想一组需要互相协作的对象——比如一个界面上若干个互相联动的控件:选中一个下拉框会让另一个输入框变为可编辑,勾选一个复选框会让第三个控件隐藏。如果这些对象之间直接持有彼此的引用、直接调用对方的方法来实现联动,那么随着控件数量增加,两两之间可能存在的引用关系数量会按平方级增长,很快形成一张难以理清的网状依赖图——任何一个控件接口的变化,都可能波及所有直接引用它的其他控件。

中介者模式(Mediator)的思路是引入一个专门的中介对象,让所有参与协作的对象都只和这个中介者打交道:一个对象发生了某个动作后,把这件事通知给中介者,由中介者内部统一决定应该联动哪些其他对象、具体怎么联动。结构大致如下:

```
class Mediator:
    notify(sender, event):
        if sender is Dropdown and event == "changed":
            self.textInput.setEditable(True)
        if sender is Checkbox and event == "checked":
            self.thirdControl.hide()
```

引入中介者之后,每个参与协作的对象只需要持有指向中介者这一个对象的引用,不再相互直接引用,对象数量增加时新增的只是"新对象与中介者"之间的一条关系,而不是"新对象与所有已有对象"之间的多条关系,网状依赖被收拢成了以中介者为中心的星形依赖。但这份收益是有代价的:中介者模式并没有减少系统里交互逻辑的总量,只是把这些逻辑从分散在各个对象内部,转移集中到了中介者这一个类里。如果参与交互的对象数量本身很多、交互规则也很复杂,中介者类会迅速膨胀成一个塞满条件判断和调度逻辑的庞然大物,变成事实上难以阅读和维护的"上帝对象"。这也是为什么在实践中,一旦中介者开始变得臃肿,通常需要考虑按交互场景把它拆分成若干更小的协调单元,或者引入事件订阅的方式分散职责,而不能指望引入一个中介者对象就一劳永逸地解决所有耦合问题。

顺带一提,中介者模式很容易和外观模式(Facade,已在结构型模式部分学过)混淆,因为两者看起来都是"引入一个额外对象来简化一组对象之间的关系"。区分的关键在于调用方向是单向还是双向:外观对象通常只是调用方单向访问子系统的一个简化入口,子系统内部对象不会反过来通过外观对象主动协调彼此;中介者对象则要持续接收各个对象发来的状态变化通知,并主动决定如何联动调用其他对象,是交互过程里的积极参与者,而不只是一层简化过的静态外壳。

### 4.5 备忘录模式:在不破坏封装的前提下时间回溯

许多应用都需要提供撤销(undo)或者重做(redo)功能,这本质上要求把一个对象过去某个时间点的内部状态保存下来,以便之后能够恢复。直接的想法是让一个专门负责管理历史记录的对象去读取业务对象的内部字段、存一份副本,但这样做的问题是:管理历史记录的对象必须获得读取业务对象私有状态的权限,这就破坏了业务对象原本的封装性,而且一旦业务对象的内部结构调整,负责保存状态的代码也要跟着改。

备忘录模式(Memento)引入了三个角色来解决这个矛盾:发起人(Originator)就是需要被保存和恢复状态的业务对象;备忘录(Memento)是一个专门用来盛放某一时刻内部状态的对象;管理者(Caretaker)负责持有和调度这些备忘录,但并不关心备忘录里具体装的是什么。协作方式是:由Originator自己创建Memento并把自己当前的内部状态写进去(因为只有Originator自己知道内部状态由哪些字段构成,也只有它有权限直接访问这些私有字段),再把创建好的Memento交给Caretaker去保管;等到需要撤销时,Caretaker把此前保存的某个Memento还给Originator,由Originator自己读取其中的数据恢复状态。对Caretaker而言,每一个Memento都是不透明的黑盒,它只做存储和调度,不做读取和修改。结构大致如下:

```
class Originator:
    save() -> Memento:
        return Memento(self._state)

    restore(m: Memento):
        self._state = m.getState()

class Caretaker:
    history = []

    backup(originator):
        history.append(originator.save())

    undo(originator):
        originator.restore(history.pop())
```

这里的设计精髓在于:状态的读写权限完全留在Originator手里,Caretaker只承担存储职责,Originator的内部实现细节不会因为要支持撤销功能而泄露出去。工程上还有一处常被忽略的细节——如果备忘录需要持久化到磁盘,序列化这个动作同样需要访问内部的完整状态数据,这个职责应该继续留给Originator或者一个被它信任的辅助逻辑来完成,而不能交给通用的存储层直接读取Memento内部字段,否则就等于让存储层绕开了原本设计给它的窄接口,重新获得了访问内部状态的权限,封装边界又被打破了。

值得对比的是,上一章学过的命令模式(Command)同样常被用来实现撤销功能,但两者实现撤销的思路并不相同。命令模式的撤销通常依赖每个具体命令对象自己知道如何"反向执行"——要么记住做这件事之前需要的参数以便反向调用,要么直接实现一个逻辑相反的操作,关注点在"动作"本身能否被逆转;备忘录模式的撤销则完全依赖"状态快照能否被完整保存和恢复",关注点在"状态"本身,不要求业务逻辑本身具有可逆性。如果一个操作很难写出清晰的反向逻辑(比如一次涉及大量字段联动修改的复杂编辑),备忘录模式往往比命令模式更直接;反过来,如果状态数据量巨大、频繁保存完整快照的开销难以接受,命令模式"只记录动作而不记录整个状态"的方式则更节省资源。两者并不互斥,不少撤销系统会把两种思路结合使用。

### 4.6 访问者模式:对稳定结构反复叠加新操作

假设已经有一组类结构相对稳定的元素类——比如一棵语法树里的各种节点类型——而现在需要不断地给这组类新增各种操作,比如打印、求值、类型检查。如果每新增一种操作就去修改每一个元素类、往里面塞一个新方法,元素类会因为承担越来越多不相关的职责而变得臃肿,而且每次修改都要触碰已经稳定、经过测试的既有代码。

访问者模式(Visitor)把"操作"从元素类中抽取出来,定义成一个独立的访问者接口,每种具体操作对应一个具体访问者类;元素类只需要保留一个统一的accept方法,用来接纳一个访问者对象。结构大致如下:

```
class Element:
    accept(visitor):
        visitor.visit(self)

class NumberNode(Element):
    accept(visitor):
        visitor.visitNumberNode(self)

class AddNode(Element):
    accept(visitor):
        visitor.visitAddNode(self)

class Visitor:
    visitNumberNode(node): ...
    visitAddNode(node): ...
```

这里能够生效的核心机制是双分派(double dispatch)。普通的方法调用只根据一个对象的实际类型决定执行哪个方法,这叫单分派;而在访问者模式里,首先根据元素对象的实际类型决定调用它自己的accept实现(第一次分派),accept内部再根据传入的这个访问者对象的实际类型,决定最终调用访问者身上哪一个具体的visit方法(第二次分派)。最终真正执行的代码,是由"元素类型"和"访问者类型"两者共同决定的,而不是仅由其中一个决定。正因为如此,新增一种操作只需要新写一个访问者类、为每种已有元素类型各实现一个对应的visit方法,完全不需要改动任何一个已有的元素类,新增操作的成本被压到了最低。

代价也随之而来,而且是结构性的、无法回避的代价。首先,新增一种新的元素类型会变得很麻烦——访问者接口里必须为每一种元素类型都声明一个对应的visit方法,新增元素类型意味着要修改访问者接口本身,并且所有已经写好的具体访问者类都要补上针对这个新类型的实现,这与"新增操作很容易"正好相反,是访问者模式在另一个维度上的固有代价。其次,访问者的visit方法要对元素做有意义的处理,往往需要读取元素内部相当多的状态,这就要求元素类把原本可以完全私有的实现细节以public方法的形式暴露出来,在一定程度上削弱了封装性。因此,访问者模式适用的前提是:这组元素类型的集合已经相对稳定、不会被频繁扩展,而针对它们需要执行的操作确实会持续增长且来自多个不同的关注点——只有当"操作增长"这件事发生的频率远高于"类型增长",引入访问者模式带来的收益才会大于它的代价。

### 4.7 解释器模式:用类层次表达一门语言的文法

最后一个模式针对的场景比前面五个都要专门:已经存在一门相对简单的语言或者文法规则(比如一个四则运算表达式),需要给这门语言的每一种文法规则设计一个对应的类,并通过这些类的组合来解释执行具体的句子。这些类通常分成两类:终结符表达式(不能再往下拆分的基本单元,比如一个数字字面量)和非终结符表达式(由更小的子表达式递归组合而成的复合结构,比如一个加法表达式由左右两个子表达式构成)。每个表达式类实现同一个interpret方法,非终结符表达式的实现内部递归调用它持有的子表达式的interpret方法,从表达式树的根节点调用一次interpret,就能递归地把整棵表达式树求值出来。结构大致如下:

```
class Expression:
    interpret(context)

class Number(Expression):
    interpret(context):
        return self.value

class Add(Expression):
    interpret(context):
        return self.left.interpret(context) + self.right.interpret(context)
```

这套机制的本质,是把一段按照文法规则解析出来的语法树,用同一套面向对象的递归调用方式解释执行,每一种文法规则对应一个类,规则之间的组合关系直接体现为类与类之间的组合关系。它有两个明显的规模问题:一是文法规则种类一多,表达式类的数量会随之线性增长;二是表达式树如果很深,递归求值带来的调用栈深度会随树的深度线性增加,存在栈溢出的风险,每一层的函数调用开销在需要频繁求值的场景下也会变成明显的性能负担。正因如此,解释器模式只适合规则相对简单、结构不太深的场景,不适合用来实现一门语法复杂的通用编程语言。

这也是解释器模式在现代工程实践里应用范围明显比其他行为型模式窄的根本原因。绝大多数应用场景根本不需要自己发明一门新语言,复用配置文件格式加上常规的代码逻辑就足以表达所需要的规则;即便确实需要解析和执行某种表达式或规则语言,工程上也更倾向于直接使用成熟的解析器生成工具或者现成的表达式引擎、规则引擎,而不是从零手写一整套终结符/非终结符表达式类——这些成熟工具已经解决了词法分析、语法解析、性能优化等大量细节问题,重新手写一遍解释器模式的完整实现,很难说是划算的选择。理解这个模式的价值,更多在于它提供了一种把文法结构映射到面向对象类层次、再用递归调用完成解释执行的思维框架,这套框架有助于理解编译器、模板引擎、规则引擎等工具内部的实现原理,而不在于鼓励在日常业务代码里手写一套解释器。

### 4.8 场景走查

#### 4.8.1 订单状态机:用状态模式设计一个订单生命周期

考虑一个电商订单对象,它在生命周期里会经历待支付、已支付、已发货、已完成、已取消这几种状态,每种状态下,"用户能否申请取消"、"能否确认收货"、"支付超时会发生什么"这些行为都完全不同。如果这些判断都写在订单类的各个方法里,每个方法开头都要重复判断一遍当前状态,而且新增一种状态(比如"部分退款")时,需要在好几个方法里同时补充新的分支,很容易遗漏。

用状态模式重新设计时,第一步是把"待支付""已支付""已发货""已完成""已取消"分别设计成实现同一个订单状态接口的类,接口里声明"支付""发货""确认收货""申请取消"这几个订单可能收到的动作。第二步是把判断逻辑下放到每个具体状态类内部:比如"待支付"状态类实现"支付"动作时,执行完扣款确认逻辑后,把订单持有的状态引用替换成"已支付"状态对象;而"待支付"状态类实现"确认收货"动作时,应该直接拒绝或者抛出一个明确的业务异常,因为一个还没支付的订单不可能被确认收货。第三步是订单对象(Context)本身只保留一个指向当前状态对象的引用,对外暴露的方法一律委托给这个引用,不再包含任何和状态相关的判断。这样设计之后,如果未来需要新增"部分退款"这一状态,只需要新增一个实现了状态接口的类、并且在相关状态类里补上转向它的转换逻辑,不需要触碰订单类本身,也不需要在已有状态类中重新梳理条件分支。这正是状态模式相对于集中式条件判断的核心收益所在。

#### 4.8.2 请假审批流程:用职责链模式设计多级审批

再看一个请假审批的场景:请假天数在一天以内的申请,直属主管就可以批准;超过一天但不超过三天,需要部门经理批准;超过三天则需要报到人力资源部门备案,天数再长则需要总经理审批。如果把这套规则写成一段集中的条件判断——先判断天数再决定去调用哪个审批人的处理方法——那么审批规则本身或者审批链条的层级一旦调整(比如新增一层"跨部门借调审批"),这段集中判断的代码就要跟着重写,而且这段代码天然地要同时知道所有审批人的存在。

用职责链模式设计时,把"直属主管""部门经理""人力资源""总经理"分别实现成职责链上的处理者对象,每个处理者知道自己能处理的天数范围以及链上的下一个处理者是谁。一个请假申请从"直属主管"这个节点进入链条:直属主管先判断申请天数是否在自己权限范围内,如果是就直接批准并结束流程;如果超出权限,就把申请转交给自己持有的下一个处理者——部门经理,由部门经理重复这个判断和转交的过程,依此类推直到某一层级批准这次申请。这套设计的好处是,提交申请的员工或者审批系统入口,完全不需要知道审批链条一共有几层、分别是谁,只需要把申请交给链条的第一个节点;当组织架构调整、需要在链条中插入或者移除某一层审批时,只需要调整组装这条链的代码,不需要改动发起申请这一侧的任何逻辑。同时,这个场景也提醒我们必须在链条末尾设计一个兜底处理:如果出现一个天数远超所有已定义审批层级上限的极端申请,链条不能让它在传到链尾后被悄悄丢弃而不给员工任何反馈,而应该在链尾放置一个专门处理"超出所有审批权限"情形的节点,给出明确的提示或者升级路径。

### 4.9 常见误区澄清

**误区一:状态模式和策略模式类图相同,所以本质上是同一个模式。** 类图相同不代表设计意图相同。策略模式描述的是"外部一次性选定一个算法来使用",各个具体策略之间通常互不知情、不会自行切换;状态模式描述的是"对象自己知道该在什么时候变成另一种东西",状态之间的转换往往由状态对象自己在运行过程中主动触发。判断该用哪个模式,应该看"切换的决定权在谁手里、切换是否会在对象生命周期内部自然发生",而不是看画出来的类图长什么样子。

**误区二:职责链模式里,只要一个处理者处理了请求,就必须停止往下传递。** GoF对职责链的经典描述确实以"某个处理者处理后传递终止"为典型形态,但这只是最常见的一种实现选择,并不是这个模式唯一允许的行为。是否在处理完之后继续往下传递,是"要不要继续传递"和"要不要处理这次请求"两个相互独立的决策,一些工程实现(比如更贴近洋葱模型的中间件链)会选择即使已经处理过,也继续把控制权交给链上后面的节点,以便实现请求和响应两个方向上的叠加处理。设计一条职责链时,应该先想清楚自己的场景到底需不需要"处理完就终止",而不是想当然地认为这是模式规定的固定行为。

**误区三:访问者模式既然能不修改元素类就新增操作,那么用它来管理一组还在快速演化、经常新增新类型的元素结构也没问题。** 恰恰相反,访问者模式的核心适用前提是元素类型集合长期稳定、操作种类持续增长。一旦元素类型本身经常需要新增,访问者接口和所有已经写好的具体访问者类都要跟着修改,新增类型的改动成本会被放大到所有下游的操作实现上,访问者模式反而会变成改动成本最高的设计选择。选择访问者模式之前,必须先判断清楚系统里真正频繁变化的维度到底是类型还是操作。

**误区四:备忘录模式里,管理者(Caretaker)既然持有了备忘录对象,读取里面存的具体状态数据应该无伤大雅。** 备忘录模式能够保持封装性,恰恰依赖Caretaker把每一个备忘录当作不透明的黑盒来对待——只做存储和调度,不读取也不修改其中的数据。一旦允许Caretaker读取备忘录内部的具体字段,就等于让原本只应由发起人(Originator)掌握的内部实现细节泄露给了外部的管理者,这个模式想要解决的"保存状态但不暴露内部结构"这一初始问题也就不再成立了。

### 4.10 本章小结

这一章的六个模式表面上覆盖的场景相差很大,但都在回答同一类问题:当对象之间的交互变得复杂——涉及状态随时间演化、请求归属不确定、多个对象相互牵连、需要回到过去的状态、需要对稳定结构反复叠加新能力、或者需要表达一门语言的文法——应该把哪一部分"关系"或者"过程"抽取成独立的对象,从而让原来纠缠在一起的代码各自变得简单。状态、职责链、中介者三个模式处理的是对象之间"如何联系、如何转交、如何协调"的问题,备忘录处理的是"如何回溯",访问者和解释器处理的则是"如何在一个相对固定的结构上,反复施加新的操作或者新的解释"。掌握这一组模式,不能止步于记住每个模式的角色名字,而要能够清楚说出每个模式相对于"不用这个模式会怎么写"这个基线,到底改变了哪些具体的依赖关系、又付出了什么代价。建议接下来对照 `tier2_deep/dp_pat_behavioral_patterns_two.py` 中对应的15个DeepPoint追问链,逐条检验自己是否能在被追问第二层、第三层问题时依然说清楚这些取舍,而不是只停留在"知道模式叫什么名字"这个层面。

## 第五章 现代模式与反模式深水

### 5.1 引言：从"一套类图"到"一整套工程实践"

1994年,《设计模式:可复用面向对象软件的基础》由"四人组"(Gang of Four)出版,书里总结的23种模式,基本上都是在C++和Smalltalk这样的静态类型、以类为核心组织单位的面向对象语言背景下诞生的。彼时函数还不能像整数或字符串一样被随手传来传去当作一个值使用,想要把"一段可替换的行为"当参数交给另一个对象,唯一的办法就是先把这段行为包一层接口和实现类。也是在那个背景下,系统的部署形态大多是单体应用,一个进程内部对象之间的调用关系相对扁平,还没有出现"读请求量是写请求量的几十上百倍""服务之间只能靠网络调用通信"这类现在司空见惯的规模问题。

三十多年过去,软件工程实践发生了几处对模式使用方式产生实质影响的变化。第一,几乎所有主流语言(不论本身是不是以"函数式"自居)都吸收了把函数当作一等值来传递的能力,原本必须靠类和接口才能表达的"可替换行为"这个诉求,很多时候可以直接靠传一个函数完成,不必再绕这层弯子。第二,依赖注入的思想从一种手写的编码习惯,演化成了一整套自动化的框架基础设施——现在很少有工程师会说"我自己手写工厂类来管理依赖",而是直接说"这个项目用了某某依赖注入容器"。第三,当系统开始向微服务、分布式方向演化,数据访问、事务边界、读写负载这些原本在单体应用里可以被经典模式简单覆盖的问题,暴露出新的复杂度层次,催生了Repository、Unit of Work、CQRS这类专门针对"应用层与持久化层如何组织"的新模式——它们不在GoF的23种之列,却是当下工程实践里同样高频出现的知识点。

但历史的另一面同样值得警惕:当一整套模式变成了"教科书知识"、变成了可以在简历上罗列的关键词,一种新的反面倾向也随之出现——为了显得"会用设计模式"而不顾场景是否真的需要,把模式的整套结构生搬硬套进本来几行代码就能解决的问题里。这种"因为学过所以要用"的心态,和三十年前"因为语言限制不得不用类去模拟行为传递"是完全不同的两回事:前者是脱离约束的形式主义,后者才是真实约束催生出的解决方案。本章要建立起的核心视角是:模式所指向的那类设计问题的本质,并不会因为语言特性或者工程范式的演化而消失,变化的只是"用什么载体去解决它",以及"值不值得为了解决它而引入这份载体带来的额外复杂度"。带着这个视角,本章依次梳理依赖注入容器、Repository与Unit of Work、CQRS、函数式编程对部分经典模式的替代、YAGNI原则与过度设计、以及God Object反模式这六个主题,并在最后给出两个完整的场景走查。

### 5.2 依赖注入容器:把"组装对象图"这件事自动化

在没有依赖注入这个概念之前,一个对象如果需要用到另一个对象提供的能力,最直接的写法就是在自己内部直接new出这个依赖来使用。这样写的问题,只有当依赖关系图变得复杂之后才会充分暴露:一个业务对象往往依赖好几个协作对象,这些协作对象自己又依赖更下层的对象,如果每一层对象都在内部自己创建自己的依赖,那么"某个底层对象应该用哪种具体实现""这个底层对象的构造需要传入什么参数"这些决策,会分散在整个代码库的各个角落——想要统一替换某个底层实现(比如把访问真实数据库换成访问一个测试用的内存实现),需要挨个找到所有直接或间接用到它的地方逐一修改。

依赖注入(Dependency Injection, DI)这个手法的核心动作其实很简单:一个对象不再自己创建它所依赖的对象,而是把这些依赖作为外部参数接收进来,由调用方或者某个专门的组装过程负责创建好之后"注入"给它。单独看这个动作,和前面章节学过的工厂模式(Factory)有着明显的家族相似性——工厂模式解决的也是"把创建对象这件事,从使用对象的代码里剥离出来"这个问题。可以说,依赖注入容器(DI容器)本质上是把"工厂"这件事做到了系统级别、做成了通用化的基础设施:一个具体的工厂类通常只知道如何创建某一种或者某一组相关的对象,而DI容器知道整个应用里所有已注册类型之间的依赖关系,在应用启动时一次性把整张依赖关系图解析清楚,按正确的顺序把所有对象创建、组装完毕。可以把DI容器理解成"一个知道如何创建应用里几乎所有对象、并且自己推导构造顺序的超级工厂",区别在于普通工厂类的创建逻辑是工程师手写出来的,而DI容器的创建逻辑是通过读取类型声明(构造函数签名、注解或者配置文件)自动推导出来的。

依赖具体"注入"进一个对象,常见有三种方式,各有取舍。构造函数注入(constructor injection)把依赖作为构造函数参数传入,对象一旦被创建出来,它需要的所有依赖就已经就绪,不可能处于"半初始化"的中间状态,而且只要看这个类的构造函数签名,就能一目了然知道它依赖哪些类型——这是目前工程实践里公认的首选方式,因为它把依赖关系变成了显式的、在对象创建那一刻就能被检查的契约。属性注入(property/setter injection)先用无参构造函数把对象创建出来,再通过设置某个公开属性或者调用一个setter方法把依赖赋值进去,这种方式的问题在于对象存在一段依赖还没有被赋值的中间状态,如果有代码在依赖被注入之前就调用了这个对象的方法,会因为依赖为空而出错,而且从构造函数签名上完全看不出这个对象实际依赖了什么,依赖关系变得不透明;它比较合适的场景是依赖本身是可选的,对象在缺少这个依赖时仍然能表现出合理的默认行为。方法注入(method injection)把依赖作为某个具体方法调用时的参数传入,而不是绑定在对象整个生命周期上,适合"这个依赖只在执行某一次具体操作时才需要、且每次调用可能需要不同实例"的场景,比如一个处理请求的方法需要接收当次请求相关的上下文对象。三种方式并不是非此即彼的选择:占绝大多数、生命周期和对象本身一致的核心依赖用构造函数注入,少数可选的、有合理默认值的依赖用属性注入,只在个别方法调用时才需要变化的依赖用方法注入,这是比较务实的组合方式。

无论具体选用哪种注入方式,DI容器带来的核心工程价值是一致的:它把"由谁负责创建对象、这些对象之间该按什么顺序组装"这件事,从散落在业务代码各处的手写逻辑里彻底剥离出来,变成了一份声明式的、可以被自动化处理的配置。这正是它相比手写工厂模式更进一步的地方——手写工厂类通常只覆盖系统里的一小部分对象创建逻辑,而DI容器把这份职责推广到了整个应用的对象图规模上。

### 5.3 Repository与Unit of Work:让业务逻辑不再关心"数据存在哪里"

业务逻辑代码里最常见的耦合来源之一,是直接在业务规则中间插入数据访问的具体细节——一段计算订单折扣的逻辑里,穿插着拼接查询语句、处理数据库连接、处理查询结果集这些和"折扣该怎么算"毫无关系的代码。Repository模式要解决的正是这种耦合:把针对某一类领域对象的数据存取操作,封装进一个对外暴露"看起来像是在操作一个内存集合"的接口里。示意结构大致如下:

```
class OrderRepository:
    add(order): ...
    findById(orderId) -> Order: ...
    findByCustomer(customerId) -> list[Order]: ...
```

业务逻辑代码调用`orderRepository.findById(...)`时,感觉上就像在向一个装满Order对象的集合索要一个元素,不需要关心这个Order对象背后到底是从关系型数据库的一张表查出来的,还是从某个文档型数据库、甚至从内存缓存里查出来的。真正知道这些细节的,是Repository接口背后具体的实现类,业务逻辑只依赖这个抽象接口。这样设计换来两个具体好处:第一,更换底层存储技术时只需要新写一个满足同一接口的实现类,业务逻辑代码不需要任何改动;第二,单元测试时可以用一个基于内存字典实现的假Repository替换掉真实数据库实现,测试跑起来不需要连接任何真实数据库,速度和稳定性都远好于端到端测试。

Repository解决的是"某一类领域对象各自的存取方式该怎么封装"这个问题,而Unit of Work(工作单元,常缩写为UoW)解决的是另一个经常和它一起出现、但概念完全不同的问题:一次业务操作里,如果需要同时新增一个订单、扣减若干条库存记录、还要更新一条积分记录,这些变更必须作为一个整体一起成功或者一起失败,不能出现"订单已经创建但库存扣减失败"这种半途而废的中间状态。UoW对象的职责,就是在一次业务事务的范围内,统一跟踪这次操作里涉及到的所有新增、修改、删除,业务逻辑代码执行过程中不需要每做一步小改动就立刻写入数据库,而是先把这些变更登记给UoW,等这次业务操作里该做的事情全部完成之后,再统一调用一次提交,由UoW把登记过的所有变更打包成一次数据库事务批量写入,任何一步失败都可以整体回滚。

值得强调的是,Repository关心的是"某一个领域对象怎么被存取",UoW关心的是"这次业务事务里一批对象的变更如何被统一提交或回滚",两者是不同维度上的关注点,只是因为绝大多数业务场景里,一次跨多个Repository的变更总是需要保证事务性,才让这两个模式经常被打包在一起使用,常见的组合方式是让多个Repository共享同一个UoW所持有的数据库连接或者会话,由UoW统一控制提交的时机。把两者混为一谈,是初学者最容易犯的一个概念错误——见到Repository就以为它自带事务保证,或者以为UoW只是Repository的一个内部实现细节,都是对这两个模式各自职责边界理解不清楚的表现。

### 5.4 CQRS:承认"写"和"读"是两件不同的事

绝大多数应用在最初设计数据模型时,都会本能地让"写"和"读"共用同一套结构——毕竟从关系型数据库的表结构来看,增删改查本来就是针对同一张表进行的。但随着系统复杂度上升,这套共用模型会暴露出两类操作在本质诉求上的差异:写操作(Command)关心的是业务规则是否被正确执行、数据的一致性约束是否得到满足,它需要的模型是围绕业务不变量组织的、相对规范化的结构,处理过程往往还需要校验、需要拒绝不合法的输入;读操作(Query)关心的是尽可能快、尽可能贴合调用方展示需要地把数据取出来,它不需要重新校验任何业务规则,反而经常需要把原本分散在好几张规范化表里的数据提前拼接、冗余存储成一个更贴近界面展示形状的结构,才能避免每次查询都做成本高昂的多表联合。

CQRS(Command Query Responsibility Segregation,命令查询职责分离)提出的做法,就是承认这两类操作的诉求本质不同,不再强求它们共用同一套模型,而是分别设计一套写模型和一套读模型,各自朝着最适合自己的方向优化。这么做主要针对的是读写负载特征差异明显、或者读模型和写模型的结构诉求长期不一致的系统——比如一个系统的读请求量是写请求量的几十甚至上百倍,读请求又需要频繁地跨多个业务实体拼接展示,这时候让读路径完全独立于写路径的表结构做优化,收益会非常明显。反过来,如果一个系统的读写模型差异很小、请求量也不大,额外维护两套模型、处理两套模型之间如何保持数据同步,这份复杂度本身就是不划算的过度设计,不是什么系统都应该无条件引入CQRS。

一个需要特别澄清的误解是,CQRS经常被和事件溯源(Event Sourcing)捆绑在一起讲解,导致不少人误以为"用CQRS就必须同时上事件溯源"。两者其实是相互独立的概念:CQRS谈的是"要不要把读写模型拆开",事件溯源谈的是"如何存储状态"——把每一次状态变化都作为一条不可变的事件持久化下来,需要时通过重放事件序列还原出当前状态,而不是像常规做法那样只存储最新的状态快照。CQRS的写模型完全可以继续用最常规的方式,把状态直接存成一张常规的表,只是把它和给读请求用的另一套物化视图分开而已,事件溯源不是CQRS成立的必要前提。两者经常被一起提及,是因为在需要事件溯源的系统里,天然会需要一个专门的读模型来提供高效查询能力(直接在原始事件流上做查询几乎不可行),所以两者搭配使用的场景确实存在交集,但交集不等于依赖关系,不能倒果为因地认为其中一个是另一个成立的前提条件。

### 5.5 函数式编程如何接管部分经典模式的意图

策略模式(Strategy)在GoF原始定义里,需要一个抽象策略接口,加上若干个实现这个接口的具体策略类,业务代码通过持有接口引用、在运行时决定具体使用哪一个实现来实现"可替换的算法"这个诉求。这套结构之所以需要绕这么大一圈,根本原因是彼时主流的面向对象语言不支持把一个函数本身当作一个可以被赋值、被传递的值——想要把"一段行为"当作参数交出去,唯一的办法就是先把这段行为包进一个对象里,再靠多态在运行时决定调用哪个对象身上的方法体。

在支持把函数当作一等公民(first-class function)的语言里,这层包装很多时候就不再必要了。函数本身可以直接被赋值给变量、作为参数传递、作为返回值返回,这意味着策略模式想要表达的"传入一段可替换的算法逻辑"这个需求,可以直接传一个函数进去实现:

```
# 面向对象写法:需要接口 + 多个实现类
interface DiscountStrategy:
    apply(price) -> price

class PercentageDiscount implements DiscountStrategy: ...

# 一等函数写法:直接传函数
def apply_discount(price, discount_fn):
    return discount_fn(price)

apply_discount(100, lambda p: p * 0.9)
```

两种写法要解决的设计问题完全一样——都是把"具体用哪种折扣计算方式"这个决定权从调用方硬编码的分支判断里剥离出来,让它可以被灵活替换——区别只在于承载这个"可替换的算法"的载体,一个是对象,一个是函数。这也是本节想强调的核心观点:语言特性的演化,替代掉的是模式的某种具体实现形态,而不是模式所指向的设计问题本身。策略模式想解决的"如何让算法可以独立于使用它的代码而变化"这个问题,在任何语言里都依然存在、依然需要被认真对待,只是解决它所需要的代码量,在支持一等函数的语言里大幅减少了。

需要提醒的是,这种"退化成传函数"并不是在所有场景下都适用。如果一种策略除了要执行的具体逻辑之外,还需要携带一些配置状态、或者需要在被调用的过程中访问某些共享的资源,单纯一个函数就没有地方存放这些额外的状态,这时候用一个类把状态和行为封装在一起,反而比东一个闭包变量、西一个全局配置更清晰。判断的标准并不是"这门语言支不支持一等函数",而是"这个策略本身是不是无状态的纯逻辑"——纯逻辑用函数替代足够,带状态的复杂策略仍然适合用一个类来承载。

类似的替代关系也发生在装饰器模式(Decorator)身上,虽然替代得不如策略模式彻底。GoF的装饰器模式针对的是给一个对象动态叠加职责,做法是定义一个和被装饰对象实现同一接口的装饰器类,内部持有指向被装饰对象的引用,在转发调用前后插入额外逻辑。很多语言里提供的函数装饰器语法,本质上是一个接收函数、返回新函数的高阶函数,新函数在调用原始函数前后插入额外逻辑——这解决的是同一类"不修改原始逻辑本身、给它叠加额外行为"的问题,只是作用对象从"一个拥有多个方法的对象"缩小到了"一个独立的函数入口"。如果需要叠加行为的不是一个函数,而是一个有好几个方法、且这几个方法都需要保持同一套对外接口的对象,单个函数装饰器就不够用了,这时候仍然需要回到经典装饰器模式的完整结构。可见,函数式编程范式接管的只是这些模式在"单一函数级别"就能满足的那部分场景,一旦问题的复杂度超出单个函数的粒度,回到面向对象的模式结构依然是合理甚至必要的选择。

### 5.6 YAGNI原则:不为想象中的未来预付成本

设计模式一旦被系统性地学习过,很容易带来一种副作用:看到任何一段将来"也许"需要变化的代码,都忍不住想提前抽出一个接口、预留一个扩展点,仿佛不这样做就是"设计得不够专业"。这种倾向如果不加节制,就会滑向过度设计(over-engineering)这个反模式——为了应对目前还没有出现、甚至可能永远不会出现的未来需求,提前引入额外的抽象层、配置开关或者灵活性机制,而这些额外复杂度在当前真实存在的需求下完全用不上。

YAGNI(You Aren't Gonna Need It,你不会需要它)这条原则,是敏捷开发实践里针对这种倾向提出的具体准则:只为当前明确存在、已经被真实需求确认的场景编写代码和设计抽象,不要为了纯粹假设的、还没有被业务验证会发生的未来场景提前买单。之所以要这样克制,原因不只是"省事",而是有两层扎实的工程理由。第一层理由是认知负担:任何一层提前引入的抽象,都需要维护成本——多一层间接调用,读代码的人要多理解一层不必要的结构,而如果只有一种具体实现,这份抽象接口本身既不能带来任何实际的多态收益,也没有为读者提供任何有用的信息,纯粹增加了理解这份代码需要花费的脑力。第二层理由更隐蔽,却往往更重要:提前设计出来的抽象接口形状,是基于对未来需求的一种猜测,而这种猜测和真正到来的需求之间,几乎总是存在偏差——等到真的出现第二种实现的那一刻,才发现最初设计的抽象接口的参数、方法划分方式和真实需要的扩展点对不上,这时候原本以为"提前做好了准备"的抽象层,反而成了阻碍,需要先花力气把它推翻重来,再按真实需求重新设计,成本比一开始没有这层抽象、等真正需要时才从零设计要更高。

这里需要划出一条清楚的界线,不能把YAGNI简单理解成"什么都不要提前考虑"。YAGNI真正反对的,是为一个具体的、尚未被验证的未来需求,提前搭建一套具体的抽象结构和灵活性机制;它并不反对把当下已经存在的需求实现得职责清晰、耦合度低、容易被修改。换句话说,"为已知的当下需求做合理设计"和"为假设的未来买单"之间的边界,可以用一个具体的问题来检验:这个需求现在是否已经确定存在、有没有具体的业务场景或者产品决策作为依据。如果答案是肯定的,把代码写得结构清晰、边界明确,这属于合理设计,不是过度设计;如果答案只是"我觉得将来可能会需要",那么现在就去搭建应对它的抽象层,就属于YAGNI要提醒警惕的过度设计。保持代码整体质量(职责单一、低耦合、易读)本身就是最好的前瞻性——它不针对任何具体假设的未来场景,却能让代码在任何一种真实需求到来时都相对容易被修改和扩展,这和"提前为某个具体的假设需求量身定制一套暂时用不上的结构"是两回事。

### 5.7 God Object反模式:职责失控的反面教材

如果说过度设计是"提前做了太多不需要的抽象",God Object(上帝对象)反模式则处在相反的一端——一个类逐渐积累了远超它应该承担的职责,变成了系统里几乎所有其他模块都要直接依赖或者调用的庞然大物。这种类往往同时负责数据存储、业务规则校验、外部服务调用、日志记录,甚至还夹杂着一部分界面展示逻辑,方法数量和代码行数随着功能迭代持续膨胀。

God Object的成因,通常不是一次性设计出来的,而是逐步"长成"的。最初这个类可能只承担一个边界清晰的职责,但每当有一个新功能需要实现、而工程师又找不到(或者懒得去找)一个更合适的落脚点时,最方便的做法就是往这个已经存在、大家都很熟悉、随处可以访问到的类里再加一个方法——这个决策在每一次单独发生的时候,增量成本看起来都很小,但长期累积下来,职责边界模糊、图省事把新逻辑堆进已有大类的选择,会不断复合放大,最终让这个类彻底失控。

它带来的危害是多方面的,而且相互加强。首先是难以测试:一个类如果同时耦合了数据库访问、外部网络调用和复杂业务规则,想要单独测试其中的业务规则,不得不同时应付和数据库、网络相关的一大堆无关设置,单元测试的隔离性无从谈起。其次是难以并行开发:当系统里几乎所有功能都要触碰这同一个类,多个工程师同时基于它做修改时,几乎必然产生冲突,团队的开发节奏被这一个类拖慢。第三是修改风险的扩散:因为多个互不相关的职责共享着同一批字段和内部状态,针对其中一个职责做的改动,很可能意外影响到看起来完全不相关的另一个职责的行为,而这种影响往往不会在编译期被发现,只能靠覆盖尽可能广的回归测试才能兜住,随着类越来越大,回归测试的范围和成本也随之越滚越大。

从原则层面看,God Object本质上是对单一职责原则(Single Responsibility Principle, SRP)最直接、最典型的违反。SRP要求一个类应该只有一个引起它发生变化的原因,也就是说一个类应该只服务于一类相关的关注点;而God Object恰恰同时因为好几个完全不相关的原因随时都可能需要被修改——业务规则调整要改它、更换日志方案要改它、接入新的外部服务也要改它,这些原本应该分别属于不同类的、彼此独立的变化原因,被强行捏合进了同一个类里。理解了这一层关系之后,识别一个类是否正在滑向God Object,并不需要等到它已经膨胀到几千行才后知后觉——一个更早期、更可操作的信号是:每次要修改这个类,能不能清楚说出"这次修改属于它承担的哪一类职责";一旦发现同一个类里越来越频繁地因为完全不相关的原因被打开修改,这就是职责边界已经开始模糊、需要考虑拆分的预警信号。

### 5.8 场景走查

#### 5.8.1 电商下单流程:用Repository与Unit of Work组织数据访问层

考虑一个电商系统里"提交订单"这个业务操作,它需要完成几件相互关联的事情:创建一条新的订单记录、扣减对应商品的库存数量、以及给下单用户增加相应的积分。如果不做任何数据访问层面的组织,最直接的写法是在这个业务方法内部直接编写针对订单表、库存表、积分表的查询和更新语句,业务规则(比如"库存不足时应该拒绝下单""积分按订单金额的固定比例计算")会和大量数据库操作细节混杂在同一段代码里,难以单独阅读、也难以单独测试。

引入Repository模式之后,第一步是针对订单、库存、积分这几类领域对象,分别设计出各自的Repository接口,比如`OrderRepository`暴露`add`和`findById`,`InventoryRepository`暴露`findBySkuId`和`decreaseStock`,`PointRepository`暴露`increasePoints`。提交订单这个业务方法本身,不再直接和任何数据库API打交道,而是依次调用这几个Repository接口:先通过`InventoryRepository`查询库存是否充足,充足的话调用`decreaseStock`预留库存;再构造一个订单对象交给`OrderRepository`的`add`方法;最后调用`PointRepository`给用户增加积分。业务方法读起来完全是业务语言——检查库存、创建订单、增加积分——不再夹杂任何查询语句或者连接管理的细节。

第二步是意识到这几步操作必须作为一个整体一起成功或者一起失败:如果库存扣减成功但订单创建失败,或者订单创建成功但积分增加失败,系统就会进入一个业务上不一致的状态。这正是Unit of Work要负责解决的问题。具体做法是让这几个Repository共享同一个UoW所管理的数据库会话,业务方法在依次调用完`InventoryRepository`、`OrderRepository`、`PointRepository`之后,不是让每一次调用各自独立提交,而是统一调用一次UoW的`commit`,由UoW把这次事务里涉及到的所有变更打包在一次数据库事务里提交,任何一步失败都会触发整体回滚,不会留下"库存扣了但订单没建"这种半途而废的中间状态。

这个场景也提示了一个容易被忽略的设计细节:如果提交订单这个流程里,除了操作本地数据库,还需要调用一个外部的支付网关发起扣款,这次外部调用没有办法被纳入本地这一个UoW管理的数据库事务里——UoW的提交回滚机制只能覆盖它所管理的这一个数据库连接范围内的变更,不能自动把一次跨系统的外部API调用也当成同一个原子操作的一部分。真实场景下这类跨系统一致性问题,通常需要额外引入"先在本地记录一条待处理的支付意图、再通过后续核对或者补偿逻辑保证最终状态一致"这类机制,而不能简单指望Unit of Work把本地事务和外部调用一起打包提交。这也是为什么理解Repository和UoW各自的能力边界如此重要——搞清楚它们各自能保证什么、不能保证什么,才能在跨系统场景下提前设计好补充的一致性手段,而不是想当然地认为引入了这两个模式就自动解决了所有一致性问题。

#### 5.8.2 一个类如何长成God Object,又是如何被拆解重构的

假设一个订单管理系统最初只有一个`OrderService`类,职责很单一:接收下单请求、创建订单记录。项目早期这个类只有几个方法,清晰易懂。随着产品需求不断增加,团队在若干次迭代里陆续做了这些决定:因为下单时需要发一条通知短信,最快的实现方式是直接在`OrderService`里加一个发短信的方法;后来又需要在下单后同步更新一个用于报表统计的汇总数字,同样是直接在`OrderService`里加了对应逻辑;再后来因为要接入一个新的优惠券系统,校验优惠券是否可用的逻辑也被顺手放进了这个类。每一次这样的决定,单独看都是局部最优的选择——毕竟这个类已经在处理下单流程,顺手加一个和下单相关的方法看起来影响范围很小、改动量也不大。

但几轮迭代之后,`OrderService`变成了一个同时负责订单创建、短信通知、报表统计更新、优惠券校验的类,这几类职责彼此之间原本没有任何必然联系,却因为都恰好和"下单"这个动作沾边,被堆进了同一个类里。这时候团队开始感受到具体的疼痛:想要单独测试"下单时优惠券校验逻辑是否正确",却因为这个方法所在的类同时依赖短信网关和报表数据库连接,写一个单元测试要处理大量和优惠券本身无关的初始化工作;两名工程师分别要修改报表统计逻辑和短信通知逻辑,却因为改动的是同一个文件而在合并代码时频繁冲突;一次只是想调整短信文案的小改动,却在回归测试时因为共享了某个内部状态字段,意外影响到了报表统计的计算结果。

面对这样一个已经成型的God Object,直接一次性把它拆开重写,风险很高——这类类内部往往已经积累了大量彼此纠缠、没有被明确记录下来的隐式依赖关系,贸然大规模重构很容易在拆分过程中遗漏某个依赖,引入新的缺陷。更稳妥的路径是渐进式的:先为这个类当前的实际行为(不管当前设计是否合理)尽可能补齐测试覆盖,把这些测试当作后续重构过程中的安全网;然后按照职责边界,一次只挪动一类相对独立的职责——比如先把短信通知相关的逻辑整体搬到一个新建的`NotificationService`里,`OrderService`里原本调用发短信的地方,改成调用`NotificationService`,跑一遍已有测试确认行为没有变化;确认这一步稳定之后,再挪动报表统计逻辑到一个`ReportingService`,最后再挪动优惠券校验逻辑到一个`CouponService`。每完成一次职责的搬迁,都跑一遍测试确认没有破坏已有行为,再进行下一步。这样拆解之后,`OrderService`重新收敛回它最初应该承担的单一职责——协调下单这个流程本身,而不是同时充当短信、报表、优惠券这几个完全不相关领域的执行者;这也是识别与修复God Object时最值得记住的方法论:测试先行、小步搬迁、每步验证,而不是指望靠一次大重构一步到位。

### 5.9 常见误区澄清

**误区一:CQRS必须搭配事件溯源(Event Sourcing)使用,不上事件溯源就不算真正的CQRS。** 这是把两个相互独立的设计决策混为一谈。CQRS谈的是"要不要把读模型和写模型拆开设计",事件溯源谈的是"用什么方式存储状态变化"。写模型完全可以继续用最常规的方式把最新状态存成一张普通的表,只是把它和专门服务查询场景的读模型分开维护,这就已经是完整的CQRS。两者经常被放在一起讲,是因为在确实采用了事件溯源的系统里,几乎必然需要一个独立的读模型来提供高效查询能力,但这只是"经常搭配"不等于"必须搭配",不能倒果为因。

**误区二:函数式编程流行起来之后,GoF的设计模式已经过时,不再需要系统学习。** 函数式编程接管的,只是策略、装饰器这类模式里"用对象包装一段可替换行为"这部分实现细节,在支持一等函数的语言里可以退化成直接传一个函数;但这只发生在"这段行为本身是无状态的纯逻辑、且只涉及单一函数入口"的场景下。一旦这段行为需要携带状态、或者需要同时覆盖一个对象身上多个方法组成的完整接口,仍然需要回到面向对象的类结构来承载。更重要的是,像组合结构的递归处理、通过中介者收拢网状依赖、通过状态对象承载随时间演化的行为这类模式所指向的设计问题,和"要不要有一等函数"这件事关系不大,并没有因为函数式特性的普及而消失。

**误区三:只要项目达到一定规模,就应该无条件引入依赖注入容器,DI容器是现代工程的必需品。** DI容器本身并非没有成本:它引入了一定的隐式性(对象具体在哪里、何时被创建,不再能从调用代码里直接读出来,需要理解框架自身的注册和解析规则),依赖关系图里的错误配置(比如生命周期不匹配、遗漏注册)往往要等到应用启动甚至运行时才会暴露,而不是在编译期就能被发现。如果一个项目的依赖关系图很浅、对象数量不多,手写几行清晰的构造函数组装代码,可能比引入一整套DI框架的学习成本和调试成本更低。是否引入应该视依赖关系图的规模和演化速度而定,而不是默认"现代项目就理所当然要用"。

**误区四:为了灵活性,提前把所有可能变化的地方都抽象出接口,总是更专业、更负责任的做法,不会有什么坏处。** 提前抽象的坏处并不是想象出来的:一方面,只要目前只有一种具体实现,这层抽象接口就不能带来任何真实的多态收益,反而让读代码的人多理解一层没有信息量的间接结构;另一方面,凭空猜测出来的抽象接口形状,和真正到来的未来需求之间几乎总有偏差,等到第二种实现真的出现时,往往需要先推翻重来,成本比等到那时候再设计更高。值得追求的"专业"不是提前为某个具体的假设需求量身定制一套暂时用不上的抽象,而是把当下已经存在的需求实现得职责清晰、耦合度低,这才是真正廉价而且始终有效的前瞻性。

### 5.10 本章小结

这一章的六个主题看起来分属几个不同的领域——依赖管理、数据访问、读写架构、语言范式、工程原则、代码质量——但它们共同回答的是同一个问题:当经典设计模式诞生时所处的语言特性和工程规模假设发生了变化,原本的模式该如何被继承、被改造,又该在哪里保持警惕。依赖注入容器、Repository与Unit of Work、CQRS,是现代工程实践针对经典模式没有覆盖到的新问题(自动化的对象图组装、持久化细节隔离、读写负载差异)所发展出的新答案;函数式编程对策略、装饰器等模式的部分替代,提醒我们模式的意图和它的实现载体是两回事,载体会随语言特性变化,意图未必会变;YAGNI原则和God Object反模式则分别从"抽象加得太早"和"职责堆得太多"这两个相反的方向,标注出脱离真实设计问题去使用或者不使用模式所带来的具体代价。贯穿全章的判断准则始终如一:是否真实存在某个模式想要解决的那类问题,以及愿不愿意为了解决它去承担这个模式本身带来的代价——只有这两个问题都有清楚的答案,引入或者放弃一个模式才是一次经得起推敲的工程决策,而不是跟风套用或者盲目回避。建议接下来对照`tier2_deep/dp_pat_modern_patterns_antipatterns.py`中对应的15个DeepPoint追问链,逐条检验自己是否能在被追问第二层、第三层问题时依然说清楚这些取舍,而不是只停留在"知道这个名词是什么意思"这个层面。

## 第六章 架构选型判断

### 6.1 引言:选型是一种预测,不是一种记忆

前面几章分别讲完了创建型、结构型、行为型三大类模式,每一节的落脚点都是"这个模式解决什么具体问题、付出什么具体代价"。如果只停留在这个层面,读者很容易形成一种错觉:只要把23种模式的定义、类图、适用场景背熟,遇到实际问题时把对应的模式"套"上去就是好的设计。这一章要澄清的,恰恰是这种错觉——把模式选型当成一道"看到关键词就匹配答案"的填空题,是初学者最容易掉进去的陷阱,也是资深工程师和初学者之间最明显的分野所在。

真正的架构选型判断,做的是一件更难、也更本质的事情:预测。一个系统在写下第一行代码的时候,几乎不可能同时在所有维度上都做到"面向未来"——面向未来意味着增加抽象层次、增加间接调用、增加代码量,而这些投入只有当"未来真的朝这个方向变化"时才会兑现收益,如果预测错了方向,这些投入不仅是浪费,甚至会成为下一次真正需要改动时的绊脚石。所以选型判断的核心工作,从来不是"这个模式的定义是不是符合当前代码的结构",而是先回答一个更根本的问题:这个系统在可预见的将来,最可能沿着哪个方向发生变化?回答清楚这个问题之后,模式的作用才浮现出来——用一个恰当的模式,把"沿着这个最可能的变化方向去修改系统"这件事的成本降到最低,而不是试图让系统在所有方向上都同样容易改动(那是不可能做到的,任何一种结构在获得某个方向上的灵活性时,几乎总是在其他方向上付出僵化的代价)。

理解了这一点,就能理解为什么这一章不会给出"什么场景该用什么模式"这样一份对照表。这样的对照表在教学的第一步(认识每个模式)是有用的,但在真实工程决策里几乎从不存在可以直接套用的标准答案——同一个需求描述,放在一个三人团队和一个三十人团队里、放在一个刚起步的项目和一个已经稳定运行五年的系统里、放在一个对可靠性要求极高和一个允许快速试错的场景里,合理的选型结论完全可能是相反的。这一章要教的,是做出这类判断时应该依据的几条分析维度,读者练完这一章之后,应该具备的不是"记住了几条选型规则",而是拿到一个新的、从未见过的场景描述时,知道自己应该先问哪几个问题、按什么顺序问、每个问题的答案会把结论推向哪个方向。

### 6.2 变化维度分析法:先问"什么会变",再问"用什么模式"

几乎每一种设计模式,拆解到最后都可以归纳为同一句话:找出系统里最可能变化的那个概念,把它从其余相对稳定的部分中分离出来,单独封装。这句话看起来简单,但落地时最容易被忽略的一步,是"找出最可能变化的概念"这个动作本身——很多人会跳过这一步,直接凭直觉或者凭"这段代码看起来像哪个模式"去选择,结果选中的模式恰好隔离了一个并不常变化的维度,而真正频繁变化的维度反而没有被照顾到。

做变化维度分析,建议按下面几个层次逐一检查,而不是笼统地问"这里以后会不会变"。第一层要区分的是变化的对象:变化的是"用什么算法/策略去处理同一份输入"(比如折扣计算方式、排序规则、渠道对接方式),还是"要构造出的对象本身的种类或者产品族在增加"(比如同一套UI组件要出多个主题风格的变体),还是"同一个对象在生命周期里表现出的行为会随内部状态或者外部时间推移而改变"(比如订单在不同阶段允许的操作不同),还是"多个独立对象之间需要相互感知对方的变化但又不想耦合"(比如界面控件之间的联动)。这几类变化在直觉上似乎都是"会变",但它们对应的隔离手段完全不同,把某一类问题误判成另一类,即使模式本身用得"技术上正确",也解决不了真正困扰这个系统的问题。

第二层要区分的是变化的确定性:这个变化是已经发生过至少一次的、有真实历史证据支撑的变化,还是纯粹出于"未来可能会需要"的想象。这一点极其重要,却经常被选型讨论忽略。已经真实发生过的变化(比如支付渠道从一个变成两个、导出格式从一种变成两种)提供的是可观察的、有具体形状的证据,可以据此判断新增变体大概率长什么样、变化频率大概率是多高;而单纯基于想象的"未来可能要支持插件化""未来可能要国际化成十种语言"这类描述,由于还没有真实实例出现,工程师这时候设计出来的抽象往往是猜出来的,猜错方向的概率并不低。变化维度分析法要求把"已经发生的变化"和"想象中可能发生的变化"分开对待,对前者可以相对积极地引入相应的隔离结构,对后者则应该保持警惕,因为为一个尚未真实出现、形状还不确定的变化预先设计抽象,本身就是这一章后面要反复强调的"过度工程"最常见的来源。

第三层要区分的是变化的量级:这个系统同时存在好几个可能变化的维度时,应该优先隔离影响面最大、变化频率最高、或者一旦发生变化而缺乏隔离所导致的返工成本最高的那一个维度,而不是试图一次性把所有能想到的变化维度都用模式包起来。一份需求描述里同时出现"以后可能要支持更多支付渠道"和"以后可能要支持批量支付"和"以后可能要多币种"这类线索时,资深判断力体现在能不能排出这几个变化维度的优先级,而不是不假思索地为每一个线索都各自安排一层抽象——那样做的直接后果是模式叠模式,系统的间接层次迅速膨胀,可读性和可维护性反而下降。

这三层检查合起来,构成了这一章反复使用的分析工具:变化维度分析法。后面几节讲的"什么信号提示该引入哪一类模式",本质上都是这套分析法在创建型、结构型、行为型三个大类下的具体展开;而"团队能不能维护""什么时候该引入"这两节,则是在变化维度分析法给出"应该隔离哪个维度"这个结论之后,进一步追问"值不值得现在就为此投入"。

### 6.3 什么信号提示应该引入创建型模式

创建型模式关注的变化维度,始终围绕着"构造一个可用对象"这件事本身的复杂度,而不是对象造出来之后如何被使用。日常写代码时,直接调用构造函数或者`new`是默认的、最省力的做法,只有当构造这个动作本身开始表现出下面几类信号时,才值得考虑引入专门的创建型模式。

第一类信号是构造逻辑本身开始承载业务规则,而不再是单纯的字段赋值。如果构造一个对象需要先做一些校验、需要根据传入参数的不同组合决定用哪种初始化路径、需要在构造完成之后紧接着做一次额外的准备工作(比如注册到某个全局注册表),这些逻辑一旦分散地写在每一个调用`new`的地方,就会导致同一段构造规则被复制多份,任何一处遗漏都会产生不一致的对象。这时候把构造过程本身收敛到一个专门的地方(工厂方法、静态构造函数、构建器)不是"更优雅",而是消除了重复和不一致的真实风险。

第二类信号是需要同时构造出一组互相配套、必须保持一致性的对象,也就是产品族的概念开始出现。当一个系统只需要造出孤立的单个对象时,不存在"族"的问题;但当系统需要保证"造出来的这一组对象必须互相匹配"(比如同一套风格的多个UI控件、同一种数据库驱动下配套的连接对象和游标对象),如果这组对象的选配逻辑散落在各处,很容易出现"某个地方不小心把两个不同风格的组件搭配在了一起"这种难以察觉的错误。这正是抽象工厂这一类模式真正要解决的问题:不是"造对象"本身复杂,而是"保证一组对象的一致性"这件事需要被集中管理。

第三类信号是参数数量和可选组合本身已经让直接构造变得容易出错,尤其是在构造过程还夹杂着分步校验、参数互斥关系等额外规则时。这里要提醒一个容易被忽视的前提:参数多这件事本身,在支持具名可选参数的现代语言里,往往不需要专门的建造器模式就能解决,只有当构造过程存在超出"填几个字段"的额外编排逻辑时,建造器模式的收益才真正体现出来,这属于语言特性和构造过程复杂度两个维度共同决定的判断,不能只看参数数量这一个指标。

第四类信号是对象的构造成本显著高于使用成本,例如构造过程涉及昂贵的初始化(网络连接、大量计算、加载大体积资源),而系统里又存在大量需要使用相近配置对象的场景。这种场景下,与其每次都重新承担构造成本,不如复用已经构造好的实例或者以一个已有实例为模板去克隆,这是原型模式和各类对象池、单例式共享设计背后的共同动机——判断依据是"构造这件事本身的开销"和"复用的现实需求"是否同时成立,而不是单纯"这个对象看起来只需要一份"。

概括起来,创建型模式该不该引入,判断的落脚点始终是构造过程本身:它是否已经承载了超出简单赋值的规则、是否需要保证一组对象的一致性、是否存在语言特性无法自然表达的额外编排逻辑、以及构造成本是否高到值得被复用。任何时候,如果构造逻辑仍然只是"把参数原样塞进字段",引入创建型模式换来的只是多一层间接调用,没有实际收益。

### 6.4 什么信号提示应该引入结构型模式

结构型模式关注的变化维度,和创建型模式不同,它面对的通常不是"内容会变",而是"两个本来应该协作的部分,形状对不上"或者"内部结构已经复杂到调用方难以正确驾驭"。判断该不该引入结构型模式,核心是找准这两类"对不上"或者"太复杂"具体表现在哪里。

第一类信号是接口不兼容:系统需要接入一个自己无法修改的外部对象——可能是第三方SDK,也可能是遗留系统里的老接口——它的方法命名、参数形状、调用约定和当前系统内部的统一约定不一致。这类信号的关键特征是"一对一的形状转换":需要转换的是一个具体接口对接另一个具体接口,转换逻辑相对固定,不涉及编排多个步骤。这正是适配器类模式要解决的问题,而且要认识到它解决的仅仅是"形状不匹配",不应该顺便把其他不相关的职责也塞进同一个转换层。

第二类信号是子系统内部复杂度已经外溢,导致调用方为了正确使用这个子系统,不得不了解一堆本该属于内部实现细节的知识——需要按特定顺序调用好几个内部组件、需要正确处理各个组件之间的依赖关系、一旦顺序或者参数传错就会产生难以定位的错误。这类信号的关键特征和第一类不同,它通常是"多对一的流程收敛":把一整套多步骤的内部编排,包装成一个简单的对外入口,让调用方不需要关心内部到底调用了几个组件、以什么顺序。识别这类信号时要注意和第一类区分开——如果同一个场景里同时存在"接口形状不兼容"和"内部流程需要收敛"两个问题,通常意味着需要两层结构分别解决各自的问题,而不是用同一个类同时承担转换和编排两种职责,那样做会让这个类的修改原因变得不单一,一旦其中一个方面需要调整,另一个方面的代码也会被牵连着重新走查一遍。

第三类信号是需要给一组已有对象动态地、可自由组合地叠加额外能力,而这些能力之间没有严格的先后依赖或互斥关系。如果尝试用继承来表达"每种能力开关组合"这件事,子类数量会随着能力种类数指数级增长,这是继承在表达"横向可选叠加"这类需求时的结构性短板,因为继承层次天然适合表达"是什么"这种单一维度的特化,而不适合表达"叠加了哪几种正交能力"。当能力之间确实相互独立、需要按需自由组合时,把每种能力实现成一层独立的包装、通过嵌套的方式按需组合,是比继承更合适的选择;但如果这些"能力"之间实际组合方式非常有限、或者存在明显的依赖或互斥关系,自由组合本身就不成立,更简单的参数开关往往已经够用,不必上升到一整套包装层的设计。

第四类信号是需要用统一的方式处理"单个对象"和"由若干同类对象组成的整体",尤其是这批对象本身呈现出天然的树形层次结构,调用方希望不必区分自己面对的到底是一个叶子节点还是一整棵子树。这类信号在需要递归处理层次结构的场景里(比如文件系统式的容器和叶子对象、界面组件的嵌套)比较典型,核心价值在于让调用方可以用同一套接口统一操作,而不需要在每处调用点都写一次"如果是叶子节点就……,如果是容器就递归……"这样的判断。

结构型模式这一大类的共同特点是:它们要解决的问题很少来自"某个算法或者行为本身会变",而是来自"部件之间的形状、边界、层次关系"出了问题。判断该不该引入,要先确认清楚复杂度到底出在哪个具体环节——是接口形状不兼容、是内部流程该收敛却没收敛、是需要自由组合的能力被继承强行拧成了固定分支、还是需要对层次结构做统一处理——找准环节之后,对应的结构型模式几乎是自然而然浮现出来的,而不是反过来先想着用哪个模式再去凑问题。

### 6.5 什么信号提示应该引入行为型模式

行为型模式关注的变化维度,落在"过程"和"交互"上:一个对象的行为如何随状态或者类型变化、一个请求应该由谁处理这件事是否能在写代码的当下就确定下来、多个对象之间要不要维持某种协调一致但又不想彼此硬编码引用。

最典型也最常见的信号,是代码里出现了大段的条件分支,而这些分支判断的本质是"根据某个类型或者某个状态,决定接下来执行哪一段完全不同的逻辑"。这种条件分支如果只出现在一个地方、逻辑本身不复杂,直接写清楚反而比拆出模式更容易读;但如果同样的判断逻辑在代码库的好几个地方重复出现、每新增一种类型或状态就要回头在好几处分别补充分支、遗漏某一处就会产生难以察觉的行为缺口,这就是行为随类型或状态变化这件事已经分散到失控的信号,值得考虑用多态替换掉这些重复的条件判断——把"每种类型/状态下应该怎么做"收拢到各自独立的实现单元里,新增一种类型或状态时只需要新增一个实现,不需要触碰已有代码里所有做过判断的地方。

第二类信号是一个请求最终应该交给谁来处理,在设计这段代码的当下无法确定、或者这份"由谁处理"的名单本身会随时间调整。如果处理者的候选集合和顺序是固定不变的,直接写清楚清晰的调用序列反而更直观;但当这份名单需要在运行期动态增删、调用方不应该被迫知道所有候选处理者的存在时,把处理者组织成一条可以在运行期拼装的链路,能让"发起请求"和"具体谁来处理"彻底解耦。

第三类信号是需要把"做一次什么操作"本身封装成一个独立的对象,而不是一次即时的方法调用——通常出现在需要把请求排队、记录下来以便追溯、支持撤销、或者支持在不同时间点重新执行的场景。如果一个操作发起之后立刻执行、不需要被记录或者延后处理,直接调用方法已经足够;只有当"这次操作发生过"这件事本身需要被保留、传递、重放,才值得把操作包装成对象。

第四类信号是多个对象之间需要保持某种数据或者状态的一致,但都不希望在代码里直接持有对方的引用、不希望知道对方的具体实现。这类信号的核心不是"行为复杂",而是"耦合关系不该是直接的双向或者网状引用",需要引入一层间接的通知或者协调机制,让对象之间只依赖一个抽象的通知契约,而不是具体的另一个对象。

行为型模式内部彼此之间也存在不少表面相似、内核不同的情形——这一点本章不再展开,前面几章已经逐一比较过策略与状态、职责链与简单分支、备忘录与命令等相近模式的区别,这里要强调的是一个更高层的判断顺序:先确认问题落在"过程/交互"这个大类里,再去确认具体是哪一种过程性质(类型行为差异、处理者不确定、操作需要被封装、多对象需要间接协调),这个顺序颠倒过来、直接从模式名称去反推问题,是行为型模式选型时最容易走弯路的地方。

### 6.6 架构层面的选型与团队驾驭复杂度的能力

前面几节讨论的都是单个模式该不该引入,落到架构层面的选型——比如整个服务要不要采用命令查询职责分离(CQRS)、要不要引入事件溯源、要不要在微服务之间强制通过异步消息通信而不是同步调用——判断的复杂度会再上一个台阶,因为这类决策影响的不是一个类、一个模块,而是整个团队日常开发、排查问题、上线变更的方式。

架构级选型判断,变化维度分析法仍然适用,但需要多问一层:读操作和写操作对数据模型、性能特征、扩展需求的要求是否已经明显分道扬镳。如果一个系统的写入路径需要严格的事务一致性和复杂的业务校验,而读取路径需要支持远比写入侧复杂得多的聚合查询、且读写的流量量级差异很大,这确实是CQRS这类读写分离架构原本要解决的核心矛盾——用同一个数据模型同时伺候差异巨大的读写需求,会让这个模型两头都不讨好。但只有当这个矛盾已经真实存在、已经在current系统里造成了具体可观察的困难(查询性能瓶颈、写入模型被迫为了迁就查询需求而变得畸形)时,引入CQRS才是在解决一个真实问题;如果读写模式并没有实质性分化,业务体量也远没有到需要独立扩展读写路径的程度,引入这类架构只是提前背上了双模型同步、最终一致性处理、更复杂的调试链路这些实打实的运维负担。

而这里最容易被技术选型讨论忽略的一个维度,是团队维护能力和团队规模。任何一种架构级别的复杂度投入,不只是设计上"值不值",更是团队"扛不扛得住"的问题——CQRS引入之后,系统里会同时存在两个数据模型、一段用来把写模型的变化同步到读模型的管道,一旦这条同步管道出问题,团队需要有能力在最终一致性的语境下排查"为什么读到的数据和刚写入的不一样"这类问题,这对团队在分布式系统、异步消息、数据一致性方面的经验有明确的要求。一个规模很小、缺乏相应经验积累的团队,即使业务体量确实出现了读写分化的信号,贸然引入这类架构也可能是在拿系统的稳定性做赌注——出问题时没有足够的人力和经验去快速定位和恢复,反而让团队把大量精力耗费在维护架构本身,而不是持续交付业务价值。反过来,一个成熟度较高、有专职团队负责基础设施稳定性的组织,即使当前体量还没有严格到"不用CQRS就活不下去"的地步,提前布局也可能是合理的,因为这个团队本身有能力把这份额外复杂度的运维成本控制住。

由此可以得到一条这一章反复强调的原则:技术选型的判断维度里,"团队能不能长期维护住这份复杂度"必须和"这个架构在技术上是否更优雅、更符合大厂通用做法"放在同等重要的位置上一起权衡,而不能只谈后者。脱离团队实际情况去谈论"更先进的架构",是架构选型讨论里一种常见但危险的思维方式。

### 6.7 引入的时机:现在就抽象,还是等真实需求出现

除了"该不该引入"之外,判断的另一个维度是"现在就引入"还是"先用最简单直接的写法,等到真实需求反复出现之后再重构成模式"。这两种时机选择背后,分别对应两种不同的风险:过早抽象的风险,和过晚抽象的风险。

过早抽象的风险,在于工程师是在只见过一个具体实例、甚至还没见过任何具体实例、纯粹凭猜测的情况下,就试图设计出一套通用的抽象结构去覆盖"未来可能出现的各种情况"。这时候设计出来的抽象,形状是猜出来的,而现实世界里需求的真实变化方向,很大概率和最初的猜测对不上——等真正的第二个、第三个具体需求出现时,往往会发现当初设计的抽象层根本装不下新的变化,不但要重新设计,还要先想办法拆掉已经嵌入系统各处、被到处调用的旧抽象,这份返工成本通常比"一开始老老实实写具体实现,后来再重构"要高得多。

正因为如此,工程实践里一个被反复验证有效的经验法则,是等到出现第二次、第三次具体的、真实的重复需求实例之后,再考虑把共同的部分抽象出来,而不是在第一次遇到某类需求的时候就急于设计一个通用结构。第一次实现时,直接针对这一个具体场景写最直接的代码,即使明知道"以后大概率还会有类似的需求"也不必急于抽象——因为此时能看到的实例只有一个,任何抽象都只是在为想象中的其他实例做设计;等到第二个具体实例出现,才有了两个真实样本可以对比,这时候可以开始观察这两个实例到底在哪些地方相同、哪些地方不同,但通常还不急着动手,因为两个样本还不足以确定"变化的形状"具有稳定的规律,很可能只是巧合式的相似或者差异;真正等到第三个实例出现,三个真实样本已经足以让人比较有把握地判断出哪部分是稳定不变的公共结构、哪部分才是真正需要被抽象隔离开的变化点,这时候动手重构成一个显式的模式,抽象出来的形状才更可能贴合真实的变化规律,而不是凭空想象出来的规律。

需要提醒的是,这条经验法则不是一条不可违背的教条,而是一种在"信息不足"时用来控制风险的默认策略。如果工程师对这类需求未来会如何演化确实拥有扎实的、来自过往经验或者行业公开认知的把握——比如某个变化点属于行业里已经被反复验证过的、几乎确定会持续增长的开放集合——在第一次实现时就适度预留一定的隔离设计,也是合理的判断,只是这种"提前判断"要求工程师对判断依据有清晰的自觉,而不是单纯出于"感觉以后可能会用得上"的模糊直觉。这也再次呼应本章开头的立场:选型判断本质上是一种带着不确定性的预测,时机的选择同样要基于"现在掌握的信息,支不支持我对未来变化形状做出有把握的判断"这个标准,而不是简单套用"能不能用某条经验法则"这一层面。

### 6.8 场景走查

#### 6.8.1 一个报表系统面对"支持多种导出格式"的需求

设想这样一个场景:一个内部报表系统目前只支持导出PDF,产品经理提出希望能同时支持导出Excel和CSV,并且提到"以后可能还要支持导出到某个第三方BI工具"。有工程师提议引入Strategy模式,把每种导出格式实现成一个独立的策略类,报表模块只依赖一个抽象的导出接口。这个判断该怎么走查,而不是直接给出"该用"或者"不该用"的结论?

第一步要做的是变化维度分析:先弄清楚"导出格式变化"具体变化的是什么。如果三种格式的导出逻辑,本质上都是"把同一份已经查询、聚合好的表格数据模型,序列化成不同的文件格式",每种格式的处理逻辑之间没有共享的初始化状态、不需要依赖注入、彼此互不知情,那么这里真正变化的只是"最后一步序列化用哪种规则",这种情况下,一个"格式名->序列化函数"的字典分发,已经能提供和Strategy类结构同样的开闭原则收益——新增一种格式只需要新增一个函数和一条字典映射,不需要修改已有代码,而且实现比引入接口和类要直接得多。

第二步要检查的是这几种格式的导出逻辑是否真的像表面看起来那么"平级"。如果深入看会发现,导出Excel需要维护一个跨多个sheet共享的工作簿对象、需要处理单元格样式和公式,导出到第三方BI工具需要维护一个和该工具的API会话、可能涉及鉴权和重试逻辑,这些格式各自的处理过程不再是无状态的纯函数,而是需要携带状态、需要多个步骤共享同一个客户端或者会话对象——这时候简单的字典分发已经不能优雅地表达这种"跨步骤共享状态"的需求,引入一个带接口的类结构、把状态和依赖收敛到构造阶段统一注入,才开始体现出实际收益。

第三步是应用引入时机的判断:产品经理提到的"以后可能还要支持第三方BI工具导出"目前还只是一句设想,并非确定会做、也没有具体的对接细节。按照等到出现真实的第三个实例再抽象的原则,现在只有CSV和Excel两个明确要做的新格式(加上原有的PDF一共三种),是否已经构成了"该抽象"的证据,取决于这三种格式内部的处理逻辑差异有多大——如果三种格式确实都只是纯粹的序列化差异,即使数量到了三个,复杂度维度上仍然不构成非抽象不可的理由;而"以后可能还有BI工具"这类还没发生的变化,不应该现在就为它单独设计预留接口。

把这三步走查合起来,可以看到,这个场景真正需要判断的,不是"支持多种格式是不是Strategy模式的经典场景"这种表面匹配,而是"每种格式内部的处理逻辑,复杂到什么程度、是否需要携带状态"这个具体的技术细节,以及"未来可能增加的格式"这个信息本身,只提示了必须避免继续堆叠if-else分支,却并不能单独决定该停留在字典分发这一级,还是应该升级到带接口的类结构。

#### 6.8.2 一个电商后台是否要为报表查询引入读写分离架构

再看一个架构层面的场景:一个电商系统的订单服务,同时承担着"处理下单、支付、发货"这类写入为主的交易流程,以及"运营人员每天查看销售趋势、库存周转、多维度交叉统计"这类读取为主的报表需求。随着报表查询的维度越来越多、聚合逻辑越来越复杂,运营人员反馈报表页面经常很慢,甚至偶尔会拖慢下单接口的响应时间,团队里有人提出应该引入CQRS,把读模型和写模型彻底分开,读模型单独用一份为查询优化过的数据结构,通过异步管道从写模型同步过来。

走查这个提议,首先要确认的是,读写分化的信号是否真实存在、且已经造成了具体可观察的后果——这里已知的线索是"报表查询变慢""偶尔拖慢下单接口",这已经是实打实的信号,而不是纯粹的假设,这一点和上一个场景里"以后可能要支持BI工具导出"这种尚未发生的假设性变化不同,值得认真对待。其次要确认清楚问题真正卡住的环节:是查询本身的聚合计算量太大,还是查询和交易写入在同一份数据表、同一个数据库实例上抢占资源,还是两者都有。这一步很重要,因为如果问题主要来自"缺少合适的索引"或者"报表查询没有做结果缓存"这类相对局部的优化空间,完整引入CQRS这种架构级别的改造就显得远超问题本身的量级,一些更轻量的手段(读写分离的数据库实例、给报表单独配一个只读副本、对报表结果做定期物化)可能已经能解决大部分问题,不需要在整个服务里引入两套数据模型和一条异步同步管道。

再往下要走查的,是团队能不能承受引入CQRS之后的运维复杂度:团队里是否有人对"最终一致性"这件事的排查有实际经验,系统里是否已经具备可靠的消息或者事件基础设施来支撑读写模型之间的同步,团队规模是否能匀出精力去维护这条新增的同步链路并在它出问题时快速定位。如果团队从来没有运维过异步一致性系统、当前也没有成熟的消息中间件可用,即使读写分化的信号已经很明确,直接一步跳到完整的CQRS,大概率会把团队的精力从业务功能开发转移到应付这套新架构本身的稳定性上,这时候更稳妥的路径,可能是先用一个只读副本这类风险和复杂度都低得多的手段缓解当前的具体问题,把"要不要走向完整CQRS"作为业务和数据规模进一步增长之后再重新评估的选项,而不是因为"CQRS理论上是解决这类问题的正规架构"就直接采用。

这个场景想要说明的是,架构级别的选型判断,不能只回答"这个架构模式是否对症",还必须同时回答"用更小的代价能不能缓解同样的问题"以及"团队现在有没有能力稳定地运维这份新增复杂度",三个问题任何一个没有仔细走查,都可能得出草率的结论。

### 6.9 常见误区澄清

**误区一:用的模式越多,说明设计水平越高。** 模式的价值来自于它精确隔离了系统里真正存在的某个变化维度,而不是来自于"用了多少种模式"这个数量本身。一个系统里如果同时叠加了大量模式,却有相当一部分并没有对应任何真实存在或者已经验证的变化需求,这不是设计水平高的体现,而是不必要的间接层次堆积——阅读和维护这样的代码,反而需要在多层抽象之间来回跳转才能搞清楚一件简单的事情究竟是怎么发生的。判断设计水平,应该看引入的每一层抽象是否都能对应说清楚"它隔离的是哪个具体变化维度、这个变化是否真实存在",而不是数模式的种类。

**误区二:应该在项目一开始就把所有可能用到的模式设计进去。** 项目起步阶段,对系统未来会朝哪个方向变化的判断依据最少、不确定性最高,这时候投入大量精力设计一套试图覆盖各种可能性的抽象体系,本质上是在用最模糊的信息做最长远的赌注。真实的变化方向往往在项目跑起来、积累了若干具体需求实例之后才逐渐清晰,提前设计的抽象十有八九和真实的变化方向对不上,后续要么被废弃、要么被生硬地拉伸去凑合并不真正匹配的新需求,两种结果都比等真实需求出现后再重构的成本更高。

**误区三:每种场景都有唯一正确的模式选择。** 同一段需求描述,在参数是否携带状态、团队是否熟悉某个模式、项目生命周期长短、语言本身是否已经提供某种能力(比如具名可选参数、一等函数)等具体条件发生变化时,合理的选型结论完全可能不同,甚至截然相反。资深的选型判断力,恰恰体现在能不能说清楚"在什么条件下我会选择A,在什么条件下我会转而选择B",而不是记住一个放之四海而皆准的固定答案。本章反复强调的判断维度框架,存在的意义正是帮助在不同条件组合下推导出适合当前具体情境的结论,而不是提供另一份可以直接查表的标准答案。

**误区四:不用模式就是设计能力差。** 很多结构清晰、运行良好的代码,从来没有出现过任何模式的名字,却已经在实际效果上做到了"变化维度被合理隔离"——比如一个回调列表配合注册函数,本质上已经具备了观察者模式的核心收益,只是没有用抽象接口的形式表达出来。是否需要把这类代码重构成"更显式""更规范"的模式结构,应该看当前的简化实现是否已经暴露出具体的问题,而不是看它是否在命名和结构上符合教科书写法。反过来,一段代码即使一个模式的名字都没提到,只要它准确地把真正易变的部分隔离开、让最常发生的那类改动足够便宜,就已经体现了良好的设计判断力;而另一段代码即使工工整整地写出了某个模式的完整类图,如果隔离的根本不是这个系统实际会变化的维度,也谈不上是好设计。

**误区五:模式选型应该主要参照"业界标准做法"或者"大厂通用架构"。** 业界的通用经验值得参考,但它是在别的团队、别的业务体量、别的团队构成下总结出来的,直接套用到当前系统,相当于把别人的变化维度分析结论、别人的团队维护能力,当成了自己系统的结论。同样一个架构选型,在一个团队里是经过深思熟虑之后匹配自身情况的合理选择,照搬到另一个规模、阶段、团队能力都不同的项目里,完全可能变成不匹配当前情境的过度设计。参照业界经验的正确方式,是学习别人分析问题的思路和判断维度,而不是直接搬运别人得出的具体结论。

### 6.10 本章小结

这一章没有给出任何一道"什么场景用什么模式"的标准答案,因为这本来就不是一件靠背诵就能做好的判断。贯穿全章的核心工具是变化维度分析法:先弄清楚系统未来最可能沿着哪个方向变化、这个变化是已经被验证过还是仅仅出于想象、多个可能的变化方向里哪一个的影响面和发生概率最高,再考虑用什么样的隔离手段去降低"沿着这个方向改动"的成本。在这个基础上,创建型、结构型、行为型模式各自对应着一类特征鲜明的信号——构造过程本身变复杂、部件之间形状或边界出了问题、行为随状态或类型分化、请求归属和多对象协调变得不确定——识别信号比记住模式定义更重要。架构层面的选型还必须叠加团队维护能力这个维度:任何复杂度投入,除了问"技术上是否更优",还要问团队能不能长期扛住这份复杂度。而所有这些判断的落地时机,则要参照"等到真实需求反复出现再抽象"这条默认策略,在信息不足时control住过早抽象的风险。建议接下来结合 `tier3_social_hire/sc_pat_architecture_selection_judgment.py` 中对应的14道ScenarioPoint场景判断题逐条练习,检验自己能不能在具体场景里独立走一遍这套分析流程,而不是直接去看每道题给出的分析示例——这一章要培养的判断力,只有在自己动手分析、并且能够为自己的结论说出清晰依据的过程中才能真正建立起来。

## 第七章 过度设计边界判断

### 7.1 引言:两种相反的错误

前面六章讲的都是"这个模式解决什么问题、结构是什么样、和相邻模式的区别在哪",这些内容对应的是初级工程师最容易犯的一类错误——**设计不足**:该抽象的地方没有抽象,导致一段逻辑散落在好几个地方各自维护,新增一种变化就要满世界找地方改代码;该解耦的地方没有解耦,导致一个类的职责越滚越大,谁都不敢动它。这类错误的表现形式很直观,后果也容易被看见——代码改不动、改一处炸三处、新人接手要花很长时间才能理清楚一个类到底在干什么。正因为后果直观,大多数工程师在积累了一定经验之后,都能比较自然地学会"看到重复就想抽象、看到职责混杂就想拆分"这一套本能反应。

但工程判断力真正被拉开差距的地方,恰恰出现在这套本能反应被训练得足够熟练之后。一名工程师一旦尝到了"提前抽象带来的好处",很容易走向另一个同样有害、但更隐蔽的极端——**过度设计**:为了应对一个尚不存在、也没有任何证据表明会出现的需求,提前搭建起一整套灵活的抽象机制。这类错误的后果不像"设计不足"那样直观,它不会让代码立刻报错、立刻改不动,它带来的是一种缓慢积累的认知税——每一个读这段代码的人,都要多花时间去理解一层本不需要存在的接口、一个从未被真正调用过的分支、一套只服务于假想未来的配置项,而这些额外的理解成本换回来的"面向未来的灵活性",很多时候永远不会被兑现,因为想象中的那个"未来的需求"根本没有到来,或者到来时的真实形态和当初设计者的猜测完全对不上。

这就是本章要处理的核心问题:资深工程判断力的一大体现,恰恰是能够在"设计不足"和"过度设计"这两个相反的极端之间找到一个恰当的平衡点。但必须提前说明——这一章不会告诉你一条可以套用的公式或者一个可以照抄的清单,因为**这个平衡点本身没有放之四海而皆准的答案**,同一段代码在不同的业务背景、不同的团队规模、不同的变化确定性下,恰当的答案可能完全相反。本章真正要建立的,是一套**用来分析具体情境、做出这类判断的思维框架**——识别过度设计症状的几个信号、YAGNI原则真正划定的边界、判断"要不要拒绝一个模式"的复杂度守恒视角、rule of three这个经验法则背后的逻辑,以及简单性与灵活性权衡的本质。掌握这套框架之后,面对一个具体场景,你应该能够自己论证出一个有理有据的判断,而不是去背一条"某种场景该怎么做"的标准答案——因为这类场景本来就没有标准答案,只有站得住脚和站不住脚的论证。

### 7.2 过度设计的典型症状:如何识别信号

在建立判断框架之前,先要练就一双能看出"过度设计嫌疑"的眼睛。以下几种代码形态,是过度设计最常见的表现症状,值得在阅读或评审代码时留意——但请注意,它们只是"嫌疑信号",不是"过度设计的充分证据",看到这些信号之后仍然需要结合具体情境去论证,而不能一看到信号就直接下结论。

第一种症状,是**只有唯一实现的接口/抽象层**。当一段代码里出现了一个接口、一个抽象基类,但翻遍整个代码库只找到一个具体实现类,这本身不是罪过——正如后面会讲到的,如果这个接口的存在是为了对接一个已经明确排期的第二种实现,那这层抽象是合理的前瞻。但如果这个"以后可能有第二种实现"只是设计者自己的猜测,没有任何业务方或路线图作背书,这层只服务于单一实现的接口,就是在没有任何收益的情况下,单纯地让调用方多了一次跳转、多了一层需要理解的抽象契约。

第二种症状,是**为"将来可能的扩展"预留的大量配置项**。一个函数或者一个类暴露出十几个可选参数,但真实调用它的地方永远只传固定的那几个值,其余参数从代码写下的第一天起就没有被任何调用方用不同的值调用过。这种为假想的灵活性预留的参数表面(surface area),不但没有带来真实的灵活性,还让每一个需要理解这个函数的人都要多问一句"这个参数是干什么用的、什么情况下需要改它",增加的是纯粹的认知负担而非实际能力。

第三种症状,是**类的层级很深,但每一层只做很薄的转发**。当追踪一次调用需要跳过五六层类,而每一层实际做的事情只是把参数原封不动地传给下一层、或者做一个可以忽略不计的包装,这种层级往往不是真实职责划分的结果,而是"为了显得有层次感"堆出来的形式主义抽象。判断的标准很简单:如果拿掉中间某一层,直接调用它下面那一层,代码的行为、可测试性、可替换性没有任何实质性损失,这一层大概率就是多余的。

需要强调的是,这三种症状都只是提示"这里值得多问一句为什么",而不是"看到就应该立刻删除"的信号灯。有的深层级确实对应着真实存在、边界清晰的职责划分;有的只有一个实现的接口确实是为了明确排期的需求做准备。识别症状只是分析的起点,后面几节要建立的框架,才是用来判断这个症状背后到底是"合理"还是"过度"的真正工具。

### 7.3 YAGNI原则的正确边界:预留扩展点,还是提前做了改动

YAGNI(You Aren't Gonna Need It)是讨论过度设计时被引用最频繁,也是被误解得最严重的一条原则。它的本意是:不要为了一个尚不确定、纯属假设的未来需求,在当下投入额外的复杂度。这句话经常被简化理解成"任何时候都不应该考虑扩展性、任何预留都是错的",但这是一种走向另一个极端的误读——YAGNI反对的是"投入"这个动作本身的正当性依据是否成立,而不是反对"考虑未来"这件事本身。一个工程师如果因为担心违反YAGNI而完全不去想任何一行代码将来可能怎么变化,写出来的东西同样会失衡,只是失衡的方向从"过度设计"倒向了"设计不足"。

真正划分"合理预留扩展点"与"过度设计"这两者边界的,不是"有没有考虑未来",而是一个更精确的问题:**这个预留,是让将来的改动变得容易,还是提前把将来的改动做掉了。** 这句话值得反复咀嚼,因为它是整章最核心的一条判断标准。

"让改动变得容易"指的是:遵循开闭原则,把代码写得在真实需求出现的那一刻,能够以较小的代价去扩展,而不需要动大手术。这通常意味着把当前唯一确定要做的事情做干净、做清楚、避免不必要的耦合和隐式依赖——一段结构清晰、职责单一的代码,天然地比一团缠绕不清的代码更容易在未来被扩展,这种"容易扩展"的性质是当下投入的一部分自然产物,不需要额外的复杂机制去实现。

"提前做了改动"指的是:在真实需求还没有出现之前,就已经把应对这个假想需求的复杂抽象机制、接口层次、配置系统完整地搭了出来。这不是"让未来容易",而是"把未来的一部分工作提前搬到了现在",而且是在没有真实需求校验的情况下凭猜测搬过来的。这里的关键风险在于:没有人能够在需求出现之前,准确猜到它真实的形态。设计者往往是依据自己的经验和想象去构造一个"看起来通用"的抽象,但真实需求出现时,这个抽象的具体细节——参数应该是什么形状、扩展点应该开在哪个位置、异常处理应该遵循什么约定——很可能与提前设计好的版本对不上。这时候,提前搭建的这套机制不仅没有省下工作量,反而会成为绊脚石:要么被迫削足适履,把新需求硬塞进不合适的旧抽象里;要么干脆推翻重来,而推倒重来的成本,往往比等到真实需求出现时才从零开始设计更高,因为这时候还要额外承担"理解并拆除原有的错误抽象"这一层负担。

因此,判断一次预留是否合理,应该追问的核心问题是:**支撑这次预留的,是已经被确认、有具体依据的已知需求,还是工程师自己基于主观预测的假设性需求。** 已知需求的典型标志是:有明确的路线图、有业务方或客户的书面/口头承诺、有可预期的时间节点;假设性需求的典型标志是:找不到任何外部依据,唯一的支撑就是"以后大概率会用到"这种直觉判断。当支撑是已知需求时,提前把可扩展的骨架搭起来,是对已知信息的合理利用;当支撑只是假设时,任何形式的提前投入,不论描述得多么"面向未来",本质上都是在为一个可能永远不会发生的场景支付确定发生的当下成本。

这里还有一个经常被忽视的细节:即便是应对已知需求的合理预留,也应该只做"预留接口/预留位置"这一步,而不应该进一步去写只为了凑数、没有任何真实调用者的空实现骨架。空实现骨架比单纯定义一个接口危害更大,因为它会给团队制造一种"这个能力已经具备"的错觉,一旦被误用于生产环境就是实实在在的事故隐患;而且这类占位代码还需要跟随主逻辑的每一次修改被同步维护,是在为一个尚未真正存在的能力持续支付维护成本。合理的做法是把"预先做技术预研、验证某个方案是否可行"和"把没有真实调用者的代码合并进主干长期存在"这两件事分开——前者可以作为独立的实验或原型存在,后者不应该发生。

### 7.4 什么时候该拒绝引入一个设计模式:复杂度守恒

设计模式本身是中性的工具,它们的价值取决于使用的场景是否满足它们各自的适用前提。第四、五章讲过的每一个模式,都有一段"这个模式在什么条件下才能兑现价值"的论述,而这一节要讲的是它的反面——当这些前提没有被满足时,应该如何有理有据地拒绝引入一个模式,而不是因为"觉得没必要"这种缺乏说服力的直觉去反对。

判断该不该引入一个模式,一个很有效的检验标准是:**引入这个模式之后,当前的代码是变得更简单、更容易理解了,还是仅仅看起来更"专业""更符合某种教科书范式"了。** 如果答案是后者——也就是说,这段代码的可读性、可测试性、可维护性并没有因为这个模式的引入而获得实质提升,唯一变化的是"看上去更像一段有设计感的代码"——这通常就是过度设计正在发生的明确信号。工程决策应该服务于降低真实的认知成本和变更成本,而不是服务于制造某种观感上的专业印象。

要理解为什么"看起来更专业"是一个危险的理由,需要引入一个更根本的视角——**复杂度守恒**。任何一个设计模式,本质上做的事情都是把复杂度从系统的一个位置转移到另一个位置,而不是凭空消灭复杂度。比如把一段if-else形式的条件分支改写成策略模式,分支判断的逻辑并没有消失,它只是从调用方的代码里搬到了一组具体策略类和一个负责选择策略的地方;把直接的对象创建改写成工厂模式,创建的判断逻辑并没有减少,只是从调用现场搬进了工厂内部。这种转移,只有在满足以下条件时才是划算的:被转移走的复杂度原本会在多个调用点重复出现(转移之后不用重复写),或者转移之后能让调用方专注于自己真正关心的事情、不必了解不必要的细节(转移之后关注点更清晰)。而当一个场景里根本没有"多个调用点"、也没有"调用方需要被保护起来的细节"——比如全篇只有唯一一处调用、创建逻辑本身毫无复杂度可言——引入模式所做的转移就没有对应的收益,只是单纯地在系统里多设置了一层间接跳转,复杂度不但没有减少,阅读和调试的路径反而变长了。

拒绝一个模式的提议时,最有说服力的做法不是笼统地说"我觉得不需要",而是给出具体的、可以被验证的代价对比:这个模式引入之后,调用链路会从直接调用变成什么样的间接形式?出问题时排查路径会变得多长?当前场景里,真的存在支撑这个模式兑现价值的前提条件(多个实现、多个独立调用方、频繁变化的需求)吗?如果暂时无法确定,是否可以提出一种"可逆决策"的折中方案——先用最简单的方式实现,同时说明清楚在什么条件出现之后(比如订阅者数量涨到某个规模、或者出现明确的动态注册需求)可以随时重新引入这个模式?这种基于具体代价与前提条件的论证,远比"面向对象设计的直觉""这样更优雅"这类主观表态更容易在评审中站得住脚,也更容易把讨论的焦点拉回到"是否真的有收益"这个问题上,而不是滑向"谁更懂设计模式"的面子之争。

### 7.5 Rule of Three(三次法则):不要在第一次就抽象

前一节讲的是"要不要引入模式"的判断,这一节要讲的是一个同样重要、但角度不同的问题——"什么时候引入"。即便某种抽象未来大概率是需要的,过早去做这件事本身也是有风险的,而rule of three(三次法则)正是应对这个风险的一条经验法则。

这条法则的具体内容是:第一次实现某个功能时,应该用最直接、最朴素的方式去写,不去考虑任何抽象;当出现第二个相似的需求时,可以先容忍一定程度的重复,但开始留意两者之间哪些部分是真正共通的;直到第三个相似的需求出现时,才真正动手把共性部分抽象出来,设计成可复用的模式或者组件。

这条法则背后的逻辑并不是"凑够三次"这个数字本身有什么魔力,而是在说明一个更根本的认识论问题:**只有一个具体案例的时候,任何从它出发做的抽象都只是对这一个案例的重新包装,而不是真正意义上的抽象。** 设计一个抽象,本质上是要回答"这一类事情里,哪些部分是必然如此、哪些部分只是这一个实例的偶然特征"这个问题,而回答这个问题需要至少两个、通常需要更多个真实样本作为参照,才能把"必然"和"偶然"分辨清楚。仅凭一个样本设计出来的所谓通用抽象,其实只是这一个样本的形状,一旦真正出现第二个样本,而这个样本恰好在"被当作偶然特征"的地方和第一个样本不一样,这个提前设计好的抽象就会立刻显得不合适,需要推倒重来——而这种返工的成本,通常比一开始就不做抽象、等到样本足够时再动手,要更高。

这里有两个需要额外澄清的地方。第一,"三"这个数字不是必须死守的教条,它的精神实质是"要有足够多的真实样本才能分辨出真正稳定的公共逻辑",在有的场景下两个样本已经足以看出明显一致的模式,在另一些场景下,即便出现了两个样本,它们的相似也可能只是表面上的巧合,内在语义完全不同。判断的关键始终落在具体分析这些样本之间的相似到底是本质相同还是表面凑巧,而不是简单地数数字。第二,如果两个案例之间的差异其实相当大,勉强把它们塞进同一个抽象里,常见的后果是这个抽象不得不通过大量可选参数、条件分支去分别迁就两边的差异,这种"看起来消除了重复,实际上制造出一堆迁就性代码"的抽象,阅读和维护的难度往往比保留两份各自清晰、稍有重复的代码更高。遇到这种情况,一个更稳妥的做法是只对确实明确无歧义的那一小部分公共逻辑做局部提取,而不必强行做一次大而全的整体抽象。

### 7.6 简单性与灵活性的权衡本质

把前几节的讨论收拢起来看,可以发现过度设计判断题的绝大多数情形,并不是非黑即白的对错问题,而是一种权衡:一端是"当下写起来简单、当下理解成本低",另一端是"未来如果发生变化,改起来会更省力,但当下就要为这份潜在的省力预先支付更高的认知负担"。几乎每一个具体场景里的判断分歧,拆开来看都能归结到这条权衡线上的取舍问题。

资深工程判断力真正体现的地方,不是记住了某条规则该怎么套用,而是对这条权衡线上"未来变化"这一端有更准确的估计能力——这个系统的这一部分,真实发生变化的可能性到底有多大?如果发生变化,大概会以什么样的形态出现?现在为它多付出的这部分复杂度代价,值不值得?这几个问题的答案,高度依赖对业务领域、团队规模、产品所处阶段的具体理解,同一段代码放在一个需求高度不确定、快速试错的早期项目里,和放在一个需求已经相对稳定、有明确演进路线的成熟系统里,恰当的答案可能完全相反。这也是为什么本章反复强调"没有放之四海而皆准的公式"——因为权衡的两端本身会随着具体情境剧烈变化,任何试图给出一条固定规则(比如"超过三个参数就要拆成配置对象""所有Service都要有接口")的做法,都是把一个需要具体分析的权衡问题,错误地简化成了一个可以无脑套用的机械规则,而机械规则恰恰是过度设计最常见的制度性根源之一。

需要指出的是,这种权衡判断本身是可以随着更多信息的出现而修正的,并不是一次性的、不可逆的决策。工程实践中很多决策的代价,恰恰在于是否可逆——如果选择"现在从简"之后,未来发现真的需要更灵活的设计,这种情况下的重构成本是否可控;如果选择"现在从繁"之后,未来发现假想的需求根本没有发生,这种沉没成本能否被及时止损。把这一层可逆性也纳入权衡的考量,往往能让判断变得更稳健——当不确定该往权衡线的哪一端靠时,优先选择更容易在未来调整方向的那一个选项,通常是更稳妥的策略。

### 7.7 场景走查

#### 7.7.1 一个工具函数的插件框架之殇

设想这样一段常见的经历:团队里有一个用于清洗和标准化文本字段的小工具函数,一开始只需要处理去除首尾空白、统一大小写这两种最基本的操作。负责这块代码的工程师认为,数据清洗这种需求以后大概率会越来越复杂,于是没有停留在写一个简单的函数上,而是设计了一整套"清洗规则插件框架"——定义了一个抽象的清洗规则接口,一个负责按顺序调度各个规则的执行引擎,一套通过配置文件声明启用哪些规则、以什么顺序执行的机制,甚至预留了规则之间可以声明依赖关系、由框架自动解析执行顺序的能力。这些设计在提出的时候,听起来都是"面向未来、可扩展"的合理考虑。

几个月之后,真实的需求变化到来了,但它的形态和当初设计这套插件框架时设想的完全不同——新的清洗需求不是"再插入几条独立的规则",而是需要在清洗过程中引用同一条记录里的其他字段做联合判断,这意味着"规则"这个抽象本身的输入输出契约就需要彻底调整,原本设计的"每个规则只处理单个字段、规则之间通过声明依赖解析顺序"这套机制,完全无法容纳这种跨字段联合处理的需求。到这一步,团队面临的选择变成了两难:要么继续在已经不合适的框架上打补丁,让规则接口承载越来越多为了迁就新需求而添加的特殊参数,把原本设计成通用的接口拖成一个专门为这一个特殊情况开后门的畸形版本;要么承认最初的抽象形状猜错了,把整套插件框架推倒,重新从最初那两个最简单的清洗操作出发去设计。无论选择哪一条路,付出的代价都明显超过了当初直接写一个朴素函数、把插件框架推迟到真正出现第二、第三个具体清洗需求之后再设计的做法。

回过头看这个场景,问题的根源不在于"考虑未来"这件事本身有错,而在于在只有两种清洗操作、还没有出现任何真实的多规则组合需求时,就凭空猜测了一整套抽象的形状,而这个猜测被验证是不准确的。如果当初的做法是先把两个操作实现清楚,直到确实出现了第三个、第四个真实的清洗需求,再回头观察这些真实需求之间到底有哪些共性,那么设计出来的抽象会建立在真实样本之上,更有机会贴合实际,而不是建立在对未来的想象之上。

#### 7.7.2 一次code review中的可扩展性判断

再看一个评审场景:一名工程师提交了一段处理用户消息通知的PR,里面引入了一个通知渠道接口,以及一个目前唯一的邮件通知实现。评审者需要判断这是一次合理的可扩展性设计,还是一次过度设计。

第一步要问的是:这个通知渠道接口对应的"未来可能新增短信、站内信等渠道"这个预期,有没有真实依据?如果产品侧已经在需求文档里写明短信通知会在下一个迭代上线,这个预期就属于已知需求,提前把接口定义清楚、让当前的邮件实现遵循这个接口,是对已知信息的合理利用,应该被接受。如果这个预期纯粹是工程师自己的推测,没有任何产品侧或路线图的依据,评审者就应该追问:去掉这个接口、直接实现一个邮件通知函数,当前的代码会变得更难理解吗?通常答案是不会,因为当前只有一种真实需要处理的渠道,接口在此刻没有任何调用方需要依赖它的多态性。

第二步要看提交的代码里有没有掺杂"占位实现"——比如为短信、站内信也一并写了空的实现类,即使这两种渠道完全没有被提出过需求。如果有,这是比单纯定义接口更值得警惕的信号,因为它不仅没有被真实需求验证过,还可能让后续维护者误以为这些渠道已经具备可用能力。

第三步,即便评审者倾向于认为这属于过度设计,给出反馈时也不应该止步于"我觉得没必要"这种主观表态,而应该说清楚具体的判断依据:当前是否存在支撑这个抽象的已知需求?去掉这层抽象,当前代码的可读性、可维护性是否会因此受损?如果暂时无法达成一致,是否可以建议先合入不带抽象的简单实现,同时在PR描述或者任务跟踪系统里记录"如果短信渠道需求被正式排期,再引入这层抽象"这样一个明确的、可执行的后续动作。这种基于具体依据、并且给出可执行替代方案的反馈方式,比单纯地"批准"或者"拒绝"更能帮助团队校准对这类判断的共识。

### 7.8 常见误区澄清

**误区一:提前设计好扩展性总没有坏处,顶多是暂时用不上。** 这个说法忽视了提前设计本身是有实打实成本的:每一层提前搭建的抽象都需要被阅读、被理解、被维护,即便从未被真正用到,它依然会持续占用后来者的认知带宽;而且这个提前设计的抽象形状是基于猜测得出的,真实需求出现时形状对不上、需要推倒重来的情形并不少见,这时候"暂时用不上"就变成了"白白付出成本还要承担返工的代价",并不是一个只赚不赔的选择。

**误区二:接口越多、抽象层次越丰富就代表设计越灵活、水平越高。** 面向接口编程带来的收益——替换实现、支持mock测试、兼容多种实现——都是有前提条件的,只有当这些前提真实存在时,接口才能兑现对应的价值。给每一个类都套一层没有任何多实现需求、没有任何mock测试需求的接口,不会带来任何实际收益,只会让每一次修改都要同步改动接口和实现两个地方,让每一次阅读都要多一次跳转才能找到真正生效的逻辑。用接口数量或者抽象层数来衡量设计水平,是一种表面化的评价标准,真正的设计水平应该用"这段代码的变更成本和认知成本是否因此降低"来衡量。

**误区三:强调YAGNI就意味着完全不能考虑任何未来的可能性,一切从简就对了。** 这是把YAGNI理解反了。YAGNI反对的是没有依据、纯属假设的投入,不是反对所有形式的前瞻性设计。当一项未来需求已经被业务方或路线图明确确认时,提前做一些让代码更容易被扩展的准备,恰恰是对已知信息负责任的利用,不在YAGNI反对的范围之内。关键的判断始终是"这个预留有没有真实依据",而不是"是否考虑了未来"这件事本身。

**误区四:一段代码用了某种知名设计模式,就说明这段代码质量更高。** 设计模式是用来解决特定重复出现问题的手段,不是评价代码质量的目的本身。把"用了几种模式"当作代码质量的指标,容易诱使工程师为了凑数在不需要抽象的地方生搬硬套模式,代码里模式的数量确实增加了,但实际的可读性、变更成本、缺陷率却可能不降反升。衡量代码质量应该看变更是否容易、缺陷是否可控、新人是否容易读懂,而不是数模式的种类。

**误区五:rule of three里的"三"是一个必须严格凑够的数字,不到三次绝对不能抽象。** 三次法则的精神是"需要足够多真实样本才能分辨出稳定的公共逻辑",而不是三这个数字本身有特殊意义。有些场景下两个样本已经足够看出清晰的公共模式,值得提前抽象;另一些场景下即便凑够三个样本,它们的相似也可能只是表面巧合,贸然抽象反而会制造出为了迁就差异而堆砌参数的蹩脚代码。判断的落脚点应该是具体分析样本之间的相似是否本质相同,而不是数数字是否达标。

### 7.9 本章小结

这一章没有给出任何一条"某种场景该怎么做"的标准答案,因为过度设计边界判断这件事本身的性质就是没有标准答案、需要具体问题具体分析。真正应该带走的是一套分析框架:先学会识别过度设计的几个典型症状——只有唯一实现的抽象层、服务于假想需求的过量配置项、层级很深却只做薄转发的类结构;再用"是否有已知需求背书"这条标准去区分YAGNI真正反对的假设性投入和合理的前瞻性预留;用"复杂度守恒"的视角去判断一次模式引入到底是把复杂度转移到了值得的地方,还是单纯制造了没有收益的间接层;用rule of three提醒自己不要在只有一个样本时就急于抽象,而要等真实案例积累到足以分辨出稳定共性;最终把几乎所有具体判断都还原成"当下的简单"与"未来的灵活"之间的权衡,并把自己对"这部分系统未来变化的可能性有多大"这一问题的判断力,当作真正需要长期打磨的核心能力。建议接下来对照 `tier3_social_hire/sc_pat_overengineering_boundary_judgment.py` 中的14个ScenarioPoint场景判断练习,逐条尝试自己先给出论证,再对照参考的论证要点,检验自己是否已经把这套框架内化成了可以在真实场景里灵活调用的判断力,而不是记住了几条听起来正确的口号。
